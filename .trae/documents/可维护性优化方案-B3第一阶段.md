### 第一阶段选择依据

选择 **A. 路径/目录** 和 **I. API 响应助手** 作为第一阶段拆分目标:

1. **零外部依赖**:两个模块都不依赖 fs-utils.ts 内部其他函数,拆分后不会产生循环依赖
2. **被广泛引用**:paths.ts 被 C/H/K/L/M/O 等几乎所有模块使用,拆分后可验证 re-export 兼容性
3. **API 助手极简**:仅 2 个函数,作为 re-export 模式的最小验证单元
4. **风险最低**:纯工具函数,无状态,无副作用,易于测试

## 拆分方案

### 模块 1: `paths.ts`(路径与目录管理)

**目标文件**: `packages/author-site/src/lib/paths.ts`

**提取函数**(17 个,约 100 行):

```
findProjectRoot, getDataDir, getProjectsDir, getTemplatesDir,
getSnapshotsDir, getSessionsDir, getWorkspacesDir, ensureDirsExist,
getProjectPath, getTemplatePath, getSnapshotPath, getSessionPath,
findSessionPath, projectExists, sessionExists, getDemosDir, getDemoPath
```

**关键细节**:
- `findProjectRoot`、`DATA_DIR`、`PROJECTS_DIR` 等模块级常量需一并迁移到 paths.ts
- `SESSION_EXPIRY_MS` 常量保留在 fs-utils.ts(属于 session 生命周期域)
- `getDemosDir`、`getDemoPath` 从行 1899-1907 提取(逻辑上属于路径域)
- `projectExists`、`sessionExists` 从行 206-217 提取(纯路径检查,属于路径域)

**模块内部依赖**:
- `getSessionPath` 调用 `findSessionPath` → 同模块内,无需跨文件导入
- `projectExists` 调用 `getProjectPath` → 同模块内
- `sessionExists` 调用 `getSessionPath`、`findSessionPath` → 同模块内

**对外依赖**:仅 `path`、`fs`,无内部依赖

### 模块 2: `api-helpers.ts`(API 响应助手)

**目标文件**: `packages/author-site/src/lib/api-helpers.ts`

**提取函数**(2 个,约 25 行):

```

```

**关键细节**:
- 依赖 `ErrorCodeType`、`ERROR_MESSAGES`(从 `@workbench/shared` 导入)
- 无内部依赖,完全独立

### fs-utils.ts 改造

**保留内容**:
- 其余 82 个导出函数(约 2228 行)
- 从 paths.ts 和 api-helpers.ts 导入所需函数
- 通过 `export * from "./paths"` 和 `export * from "./api-helpers"` re-export 保持向后兼容

**关键改动**:
```ts
// fs-utils.ts 顶部
import {
  findProjectRoot, getDataDir, getProjectsDir, /* ...其他路径函数 */,
  projectExists, sessionExists, getDemosDir, getDemoPath,
} from "./paths";
import { createApiError, createApiSuccess } from "./api-helpers";

// 文件末尾 re-export 保持向后兼容
export * from "./paths";
export * from "./api-helpers";
```

**预期行数变化**:
- fs-utils.ts: 2353 → ~2228 行(减少约 125 行)
- paths.ts: 新建 ~100 行
- api-helpers.ts: 新建 ~25 行

## 假设与决策

### 假设

1. **向后兼容**: 所有调用 fs-utils.ts 中 paths/api-helpers 函数的代码无需修改,通过 re-export 保持兼容
2. **常量初始化时机**: `DATA_DIR` 等模块级常量在 paths.ts 中初始化,导入 paths.ts 的代码会触发常量求值,行为与原 fs-utils.ts 一致
3. **测试覆盖**: 现有 author-site 测试应覆盖 paths/api-helpers 的核心场景,无需新增测试

### 决策

1. **不拆分 `SESSION_EXPIRY_MS`**: 该常量属于 session 生命周期域,后续拆分 session 域时一并迁移
2. **不拆分 `getDemosDir`/`getDemoPath` 之外的 demo 管理函数**: `demoExists`、`listDemos`、`createDemo`、`deleteDemo` 涉及文件系统写操作,属于 demo 生命周期域,留待后续阶段
3. **使用 `export *` 而非命名 re-export**: 减少维护成本,新增函数时无需同步更新 re-export 列表

## 风险与回滚

### 风险

1. **循环依赖**: paths.ts 不依赖 fs-utils.ts,无循环依赖风险;api-helpers.ts 同样无循环依赖
2. **常量初始化顺序**: `DATA_DIR` 在 paths.ts 中初始化,需确认被 fs-utils.ts 中其他函数使用时已正确求值
   - 缓解:`DATA_DIR` 是模块级常量,在 import 时即求值,顺序正确
3. **测试覆盖不足**: 若现有测试未覆盖 paths/api-helpers,可能漏掉回归
   - 缓解:re-export 保证 API 不变,行为应与原 fs-utils.ts 一致

### 回滚

如拆分后出现问题,直接 `git revert` 相关 commit 即可恢复原 fs-utils.ts 单文件状态。无数据迁移,无配置变更,回滚成本极低。
