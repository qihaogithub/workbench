- **Phase A 收尾**：完成基础设施配置（修复 lint error + CI 增加 lint 检查）
- **Phase B**：拆分 4 个超大文件，降低单文件复杂度

### Phase B：大文件拆分

#### B1：拆分 `packages/project-core/src/service.ts`（4798 行）

**当前结构**：整个文件定义一个 `ProjectAdminService` 类，包含所有项目管理方法。

**拆分策略**：门面模式（Facade Pattern），保持对外 API 不变

**目标文件结构**：
```
packages/project-core/src/
├── service.ts                    # 门面，保留 ProjectAdminService 类，委托给各子服务
├── services/
│   ├── project-service.ts        # 项目 CRUD：createProject, updateProject, deleteProject, listProjects
│   ├── page-service.ts           # 原型页：updatePrototypePage, createPrototypePage
│   ├── asset-service.ts          # 资源：uploadAsset, replaceAsset, getAssetSummary
│   ├── version-service.ts        # 版本控制：createPageVersion, getPageVersionHistory, restorePageVersion
│   ├── validation-service.ts    # 验证：validateProject, validatePage
│   ├── config-service.ts         # 配置：getProjectConfig, updateProjectConfig
│   └── knowledge-service.ts      # 知识库：indexTemplateSnapshot, getKnowledgeIndex
└── index.ts                      # 保持现有导出
```

**实施步骤**：
1. 识别 `ProjectAdminService` 类的所有方法，按业务域分组
2. 为每个业务域创建独立的服务类（如 `ProjectService`、`PageService` 等）
3. 将对应方法实现迁移到子服务类
4. `ProjectAdminService` 改为门面，持有各子服务实例，方法委托调用
5. 保持对外导出的 API 不变

**预期效果**：
- 主文件降至 200-300 行
- 每个子服务 300-800 行
- 业务逻辑边界清晰

#### B2：拆分 `packages/sketch-react/src/index.tsx`（6962 行）

**当前结构**：22 个导出，包含 6 个组件、3 个 hook、13 个类型。

**拆分策略**：按功能模块拆分

**目标文件结构**：
```
packages/sketch-react/src/
├── index.tsx                     # 仅做 re-export
├── types.ts                      # 所有类型定义（PreviewSize, SketchTool 等 13 个）
├── hooks/
│   ├── use-sketch-selection.ts   # useSketchSelection
│   ├── use-sketch-history.ts     # useSketchHistory
│   └── use-sketch-editor-state.ts # useSketchEditorState
└── components/
    ├── sketch-page-preview.tsx   # SketchPagePreview
    ├── sketch-editor-toolbar.tsx # SketchEditorToolbar
    ├── sketch-layer-panel.tsx   # SketchLayerPanel
    ├── sketch-property-panel.tsx # SketchPropertyPanel
    ├── sketch-editor-canvas.tsx  # SketchEditorCanvas
    └── sketch-page-editor.tsx   # SketchPageEditor
```

**实施步骤**：
1. 先提取所有类型定义到 `types.ts`
2. 逐个提取 hook 到 `hooks/` 目录
3. 逐个提取组件到 `components/` 目录（按依赖顺序：低层组件优先）
4. `index.tsx` 改为纯 re-export，保持对外 API 不变

**预期效果**：
- 主文件降至 50 行（re-export）
- 每个组件文件 500-2000 行
- 类型集中管理，便于引用

#### B3：拆分 `packages/author-site/src/lib/fs-utils.ts`（2353 行）

**当前结构**：100+ 导出函数，涵盖路径处理、项目、会话、工作区、Demo、配置、版本、资源等多个业务域。

**拆分策略**：按业务域拆分

**目标文件结构**：
```
packages/author-site/src/lib/
├── fs-utils.ts                   # 保留核心路径处理和 API helper（createApiError, createApiSuccess, findProjectRoot, getDataDir 等）
├── project-meta.ts               # 项目元数据：listProjects, readProjectMeta, writeProjectMeta, createProject, deleteProject
├── session-meta.ts               # 会话：createSession, getSessionMeta, deleteSession, isSessionExpired, sessionExists
├── workspace.ts                  # 工作区：getWorkspacePath, findWorkspacePath, getWorkspaceMeta, updateWorkspaceFiles
├── demo-pages.ts                 # Demo 页：generatePageSlug, isValidRouteKey, readDemoPageMeta, listDemoPages, AppGraph 相关
├── version-history.ts            # 版本：generateVersionId, cleanupOldVersions, createProjectVersionSnapshot, getVersionHistory, restoreVersion
├── session-assets.ts             # 资源：getSessionAssetsPath, saveSessionAsset, deleteSessionAsset, listSessionAssets
├── project-config.ts             # 配置：getProjectConfigSchema, saveProjectConfigSchema, getProjectConfigValues
```

**实施步骤**：
1. 先识别所有函数的依赖关系（避免循环引用）

3. 逐组迁移到对应文件
4. `fs-utils.ts` 保留核心函数和 API helper，从新文件 re-export 关键函数以保持向后兼容
5. 后续可逐步更新调用方使用新路径

**预期效果**：

#### B4：拆分 `packages/demo-ui/src/PreviewCanvas.tsx`（2970 行）

## 风险和注意事项

### 风险

1. **循环依赖**：B3 拆分 fs-utils.ts 时，多个业务域可能互相引用。需识别依赖关系，必要时通过参数传递而非直接导入。

2. **import 路径变更**：B1-B4 都会改变内部文件结构，需更新所有调用方的 import 路径。

3. **行为变化**：拆分过程中可能意外改变函数行为。每个拆分任务完成后必须运行完整验证。

4. **测试覆盖不足**：部分包可能没有完整测试，拆分后问题可能潜伏。优先在有测试的包上执行。

### 注意事项

1. **保持对外 API 不变**：所有拆分任务的对外导出（包的 `index.ts` 或 `package.json` 的 `main`）必须保持不变。

2. **增量提交**：每个子任务（如 B1 的一个子服务迁移）完成后提交一次，便于回滚。

3. **先类型检查再运行测试**：类型错误更容易定位，先 `typecheck` 再 `test`。

4. **不重构逻辑**：拆分只移动代码，不修改函数实现。如发现需要优化的逻辑，记录待办，拆分任务完成后再单独处理。

## 假设和决策

1. **门面模式 vs 直接拆分**：B1 选择门面模式，保持 `ProjectAdminService` 类的对外 API 不变，降低调用方变更成本。

2. **re-export vs 更新调用方**：B2 和 B3 的 `index.tsx`/`fs-utils.ts` 选择 re-export 模式，保持向后兼容，后续可逐步迁移调用方。

3. **不修改 lint warning**：agent-service 有 69 个 `@typescript-eslint/no-explicit-any` warning，本次只修复 error，warning 留待后续处理。

4. **CI 不增加 test:e2e**：E2E 测试需要浏览器和真实服务，不适合 CI 环境，本次只增加 `lint:all`。
