## 六步分析法

### 第一步：信息收集

**目标**：建立问题的完整上下文

**必做事项**：
- 记录问题的精确描述（谁、什么、何时、何地）
- 获取错误信息原文（堆栈跟踪、日志片段、错误码）
- 确认复现步骤与条件
- 识别影响范围（哪些用户/功能/环境受影响）
- 记录预期行为与实际行为的差异

**关键提问**：
- 问题是否可稳定复现？还是偶发的？
- 问题首次出现的时间点？是否有相关变更？
- 是否存在环境差异（开发/测试/生产）？

### 第二步：系统性调查

**目标**：追踪代码路径与系统行为，定位问题发生位置

**调查维度**：
1. **代码路径追踪**：从入口点到出错点，追踪完整调用链
2. **数据流分析**：检查数据在各层之间的传递与转换
3. **状态检查**：审查相关组件的状态变化时序
4. **配置审查**：检查相关配置项与环境变量
5. **依赖分析**：检查第三方依赖版本与已知问题

**工具使用**：
- 使用 `SearchCodebase` 追踪代码路径
- 使用 `Grep` 搜索错误信息与关键函数
- 使用 `Read` 查看具体实现细节
- 使用 `GetDiagnostics` 获取类型与编译错误

### 第三步：证据采集

**目标**：从代码、日志与系统状态中收集具体证据

**证据类型**：
- **代码证据**：具体的代码行、函数实现、类型定义
- **日志证据**：错误日志、运行时输出、调试信息
- **配置证据**：环境变量、配置文件、部署参数
- **行为证据**：系统响应、时序关系、状态转换

**证据记录要求**：
- 标注证据来源（文件路径:行号）
- 保留原始内容，不做推断性修改
- 区分直接证据与间接证据

### 第四步：根因定位

**目标**：区分表面现象与深层原因

**核心原则**：
- 现象 ≠ 根因。报错信息是现象，导致报错的代码逻辑才是根因
- 使用"5个为什么"技术逐层深入
- 验证根因假设：如果修复该根因，现象是否必然消除？

**常见根因模式**：
| 模式 | 典型表现 | 根因层次 |
|------|---------|---------|
| 类型不匹配 | 运行时 TypeError | 数据流中某环节缺少类型转换 |
| 竞态条件 | 偶发错误 | 异步操作缺少同步机制 |
| 状态污染 | 功能间歇性异常 | 共享状态被意外修改 |
| 配置漂移 | 环境间行为不一致 | 配置未统一管理 |
| 依赖冲突 | 构建或运行时错误 | 依赖版本不兼容 |

### 第五步：方案制定

**目标**：制定直接解决已识别根因的方案

**方案评估维度**：
- **有效性**：是否直接解决根因
- **影响范围**：修改涉及多少模块/文件
- **风险等级**：是否可能引入新问题
- **实现复杂度**：所需工作量

**方案类型**：
1. **精确修复**：仅修改导致问题的具体代码行
2. **防御性修复**：在问题点添加保护逻辑
3. **架构调整**：修改导致问题的设计模式
4. **配置修正**：调整错误的配置项

### 第六步：报告撰写

**目标**：按结构化格式整理分析结果

详见 [report-template.md](report-template.md)

## 证据分级标准

| 级别 | 定义 | 示例 |
|------|------|------|
| A 级 | 直接代码证据，精确定位到行 | `file.ts:42` 的类型转换缺失 |
| B 级 | 日志或运行时行为证据 | 错误日志显示 null 引用 |
| C 级 | 配置或环境差异证据 | 生产环境缺少环境变量 |
| D 级 | 推断性证据，需进一步验证 | 基于代码逻辑的推断 |

**要求**：根因结论至少需要 A 级或 B 级证据支撑，不可仅凭推断。

## 根因识别技术

### 5 个为什么

对每个现象连续追问"为什么"，直到找到不可再分的根因：

```
现象：API 返回 500 错误
→ 为什么？数据库查询超时
→ 为什么？查询未使用索引
→ 为什么？WHERE 条件字段未建索引
→ 为什么？迁移脚本遗漏了该索引
→ 根因：数据库迁移脚本不完整
```

### 差异分析法

对比正常与异常场景的差异：

- 代码版本差异
- 配置差异
- 数据差异
- 环境差异
- 时序差异

### 回归分析法

如果问题是新出现的，检查最近的变更：

- 最近的代码提交
- 最近的配置变更
- 最近的依赖升级
- 最近的环境变更

## 质量保证检查清单

分析报告完成前，逐项检查：

### 完整性
- [ ] 问题背景是否包含预期行为与实际行为对比
- [ ] 是否记录了所有错误信息原文
- [ ] 是否说明了问题发生的环境与条件

### 证据充分性
- [ ] 每个根因结论是否有 A/B 级证据支撑
- [ ] 代码路径是否追踪完整（从入口到出错点）
- [ ] 是否区分了直接证据与推断

### 准确性
- [ ] 文件路径与行号是否准确
- [ ] 代码引用是否与实际代码一致
- [ ] 是否区分了现象与根因

### 客观性
- [ ] 是否存在未经验证的假设
- [ ] 是否考虑了其他可能的根因
- [ ] 结论是否基于证据而非直觉

### 可操作性
- [ ] 解决方案是否具体、可执行
- [ ] 是否说明了每种方案的利弊
- [ ] 是否标注了实现复杂度与风险
