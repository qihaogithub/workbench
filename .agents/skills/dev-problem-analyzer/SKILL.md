# 开发问题分析器

系统性分析开发问题，基于证据定位根因，产出结构化分析报告。

**核心原则：仅分析与报告，绝不修改任何代码。**

## CLI-First 诊断原则

**始终优先使用 `ops-cli` 工具进行诊断，再用代码搜索补充。**

ops-cli 位于 `OPS/CLI/`，运行方式：`cd OPS/CLI && pnpm dev <command> --json`

所有命令支持 `--json` 全局标志，输出结构化 JSON 供程序化解析，节省 token。

### CLI 命令速查表

| 命令                            | 用途               | 典型场景                                       |
| ------------------------------- | ------------------ | ---------------------------------------------- |
| `ops-cli system --json`         | 一键环境诊断       | Step 1: 收集运行时、服务状态、端口、后端可用性 |
| `ops-cli health --json`         | 服务健康检查       | 确认 agent-service 是否运行正常                |
| `ops-cli diagnose <sid> --json` | 会话错误诊断       | Step 2: 自动化 4 步诊断（健康→会话→测试→分析） |
| `ops-cli sessions --json`       | 列出所有会话       | 查看错误状态会话、资源泄漏                     |
| `ops-cli session <sid> --json`  | 会话详情           | 查看特定会话状态、后端、消息数                 |
| `ops-cli logs [sid] --json`     | 日志采集           | Step 3: 按级别/关键字过滤日志                  |
| `ops-cli destroy <sid>`         | 销毁会话           | 清理错误会话释放资源                           |
| `ops-cli stream <sid> "msg"`    | WebSocket 流式测试 | 验证流式响应是否正常                           |
| `ops-cli send <sid> "msg"`      | HTTP 消息测试      | 验证非流式响应是否正常                         |

### CLI 诊断决策树

```
问题报告
  ├─ 涉及 Agent Service? ─── ops-cli system --json
  │   ├─ 服务未运行 ─── 建议启动 pnpm dev:agent
  │   ├─ 服务异常 ─── ops-cli health --json → ops-cli logs --json
  │   └─ 服务正常 ─── ops-cli sessions --json
  │       ├─ 有 error 状态会话 ─── ops-cli session <sid> --json
  │       └─ 需要测试消息 ─── ops-cli diagnose <sid> -m "测试" --json
  └─ 不涉及 Agent Service ─── 走代码搜索流程
```

## 分析流程

问题分析按以下六步顺序执行：

1. **信息收集** — 建立问题完整上下文
2. **系统性调查** — 追踪代码路径与系统行为
3. **证据采集** — 从代码、日志与系统状态中收集证据
4. **根因定位** — 区分现象与根因
5. **方案制定** — 制定直接解决根因的方案
6. **报告撰写** — 按标准格式输出分析报告

每一步完成后再进入下一步。若信息不足，向用户请求补充。

## 第一步：信息收集

从用户描述中提取以下要素：

- **问题现象**：精确描述异常行为
- **预期行为**：系统应该怎样
- **实际行为**：系统实际怎样
- **错误信息**：完整的错误原文、堆栈跟踪
- **复现条件**：何时/何种操作触发
- **环境信息**：开发/测试/生产、浏览器/Node版本等

**CLI 快速采集**：立即运行 `ops-cli system --json`，一次获取：

- Node/pnpm/TypeScript 版本
- Agent Service 运行状态、PID、端口
- CLI 后端可用性（opencode/claude/codex 等）
- 项目结构完整性
- 端口占用情况
- 自动检测到的环境问题

若用户描述模糊，主动提问澄清。重点关注：是否可稳定复现、首次出现时间、是否有相关变更。

## 第二步：系统性调查

### 优先使用 CLI 工具

1. **服务层诊断**：`ops-cli diagnose <sessionId> --json` — 自动执行健康检查→会话状态→测试消息→错误分析
2. **会话状态扫描**：`ops-cli sessions --json -s error` — 快速定位错误状态会话
3. **特定会话详情**：`ops-cli session <sessionId> --json` — 获取会话完整状态

### 补充代码搜索

仅当 CLI 诊断不足以定位问题时，使用代码搜索工具：

1. **搜索错误信息**：用 `Grep` 搜索错误信息关键字，定位出错代码
2. **追踪调用链**：用 `SearchCodebase` 从入口点追踪到出错点的完整路径
3. **检查相关实现**：用 `Read` 查看关键函数的具体实现
4. **审查配置**：检查相关配置文件与环境变量
5. **检查诊断信息**：用 `GetDiagnostics` 获取类型错误与编译问题

## 第三步：证据采集

### 优先使用 CLI 采集

- **日志证据**：`ops-cli logs <sessionId> --json -l error` — 按级别过滤错误日志
- **日志搜索**：`ops-cli logs <sessionId> --json -p "关键字"` — 按关键字搜索日志
- **会话状态证据**：`ops-cli session <sessionId> --json` — 获取会话状态作为行为证据

### 证据记录标准

为每个发现记录：

- **证据内容**：具体代码行、日志片段、配置项
- **证据来源**：文件路径:行号
- **证据级别**：A（直接代码证据）、B（日志/行为证据）、C（配置证据）、D（推断）

**根因结论至少需要 A 级或 B 级证据支撑。**

## 第四步：根因定位

区分现象与根因：

- **现象**：用户看到的外在表现（报错信息、异常行为）
- **根因**：导致现象的底层代码逻辑或设计缺陷

使用"5个为什么"逐层深入，直到找到不可再分的根因。验证方法：如果修复该根因，现象是否必然消除？

若存在多种可能根因，列出每种假设及其支撑与矛盾证据。

## 第五步：方案制定

为每个根因提供解决方案：

- 至少提供一个**推荐方案**
- 每个方案说明：具体做法、为何有效、影响范围、风险、复杂度
- 若信息不足以确定方案，提出后续调查步骤

## 第六步：报告撰写

按标准报告格式输出，报告必须包含四个部分：

1. **问题背景** — 问题描述、发生场景、预期/实际行为、错误信息
2. **根因分析** — 调查过程、证据链、根本原因、代码执行路径
3. **解决方案** — 具体方案、利弊分析、后续建议
4. **相关代码路径** — 涉及文件、调用链、相关配置

**详细方法论**：参见 [references/methodology.md](references/methodology.md)
**报告模板与示例**：参见 [references/report-template.md](references/report-template.md)

## 报告保存

分析报告保存至项目目录 `docs/plans/进行中/`，文件名格式：`{问题简述}-分析报告.md`

## 质量底线

报告完成前逐项检查：

- 每个根因结论有 A/B 级证据支撑
- 文件路径与行号准确
- 区分了现象与根因
- 解决方案具体可执行
- 未包含未经验证的假设性断言
- 已使用 CLI 工具进行环境验证（如适用）
