# Doc Maintainer

按照统一的目录结构、DRY 原则和跨文档引用规范，维护项目文档库。

## 核心工作流

收到文档操作请求后，按以下三步执行：

### 第一步：现状评估

1. 分析请求，识别涉及哪个【功能模块】
2. 若涉及全新模块，先规划目录结构（见 [directory-structure.md](references/directory-structure.md)）
3. 检查目标模块是否已有 INDEX.md 和现有文档

### 第二步：拆分与撰写

1. 梳理需要新建或更新的「需求文档」与「实现文档」
2. 严格遵守内容边界规则（见 [content-rules.md](references/content-rules.md)）
3. 若模块过于复杂，**不要一次性输出全部**，主动将实现文档拆分为多个小文件
4. 修改需求文档前，**必须先备份**（见 [content-rules.md](references/content-rules.md) 的备份策略）

### 第三步：建立引用

1. 更新对应模块的 `INDEX.md`
2. 更新全局 `docs/INDEX.md`
3. 确保所有文档处于无孤儿（Orphan）的网状链接中

## 关键规则速查

| 规则         | 要求                                                    |
| ------------ | ------------------------------------------------------- |
| DRY 原则     | 同一信息在文档库中只存在一次，全局背景不在局部复述      |
| 跨文档引用   | 必须使用 Markdown 相对路径链接，禁止重复描述            |
| 需求文档备份 | 修改前必须创建 `-backup.md`，确认备份成功后再修改       |
| 实现文档     | 可直接修改，无需备份                                    |
| INDEX.md     | 可直接修改，无需备份                                    |
| 需求文档内容 | 只写"做什么"和"为什么"，禁止技术实现细节                |
| 实现文档内容 | 只写"怎么做"，顶部必须有 YAML covers 映射，禁止贴源代码 |
| 实现文档表达 | 面向非纯技术人员，多用自然语言和比喻                    |

## 输出格式

每轮文档操作完成后，必须以如下摘要格式结尾：

```
✅ **文档操作完成**
📝 **新建/更新**：
- `[需求]` docs/.../xxx_需求文档.md
- `[技术]` docs/.../技术/xxx_实现设计.md
- `[索引]` docs/INDEX.md
💾 **备份记录**：[如无备份填"无"；若修改了需求文档，必须列出生成的 `xxx-backup.md`]
⏳ **下一步建议**：[指出是否需要用户确认、是否有遗漏的文档待补充，或询问下一个要处理的模块]
```
