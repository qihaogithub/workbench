# 内容边界与安全规则

## 一、需求文档编写规范

**文件位置**：`docs/[模块]/*_需求文档.md`

### ✅ 必须包含

- 模块定位：该模块在整个系统中的角色
- 功能需求清单：从用户视角出发，描述用户能做什么
- 业务规则与约束：前置条件、限制条件

### ✅ 可选包含

- 业务背景与目标
- UI/UX 设计要求

### ❌ 绝对禁止

- 任何技术实现细节（如表名、框架选型、代码字段、API 路径等）
- 系统架构描述
- 数据库设计

## 二、实现文档编写规范

**文件位置**：`docs/[模块]/技术/*.md`

### ✅ 必须包含

- 技术架构与选型理由
- 数据流程处理
- 组件/模块协作关系
- 接口契约说明

### 📌 顶部 YAML 映射

每个实现文档顶部必须保留如下格式，注明关联的代码文件路径：

```yaml
---
covers:
  - src/api/user.ts
  - src/components/LoginBtn.tsx
---
```

### 🧠 表达方式要求

- 面向非纯技术人员
- 多用自然语言和比喻讲解业务流转
- 示例："数据在 A 处打包，发到 B 处验证"

### ❌ 绝对禁止

- 贴真实的源代码块或长篇伪代码
- 只需引用实际代码的相对路径
- 禁止重复需求文档里的宏观背景

## 三、DRY 原则与跨文档引用

### 绝对禁止重复描述

同一信息在整个文档库中只能存在一次：

- 全局背景只在 `全局总需求文档.md` 中描述，模块文档不重复
- 技术文档不重复罗列业务需求
- 多个模块共用的概念，在最先定义的文档中描述，其他位置引用

### 强制相对路径跳转

提及其他文档的概念或功能时，必须使用 Markdown 相对路径链接：

**正确写法**：

```markdown
关于鉴权失败的交互，请见 [基础登录需求](../基础登录需求_需求文档.md)
```

**错误写法**：

```markdown
关于鉴权失败的交互，当用户输入错误密码三次后，系统将锁定账户30分钟...（重复描述）
```

### 引用路径计算参考

| 当前文件位置 | 引用目标 | 相对路径示例 |
|-------------|---------|-------------|
| `docs/创作端/01-用户鉴权/技术/01_架构设计.md` | 同模块需求文档 | `../用户鉴权_需求文档.md` |
| `docs/创作端/01-用户鉴权/技术/01_架构设计.md` | 同目录其他技术文档 | `./02_数据库设计.md` |
| `docs/创作端/01-用户鉴权/INDEX.md` | 模块内需求文档 | `./用户鉴权_需求文档.md` |
| `docs/INDEX.md` | 模块目录 | `./创作端/01-用户鉴权/` |

## 四、修改策略

### 需求文档

修改 `*_需求文档.md` 时可直接编辑原文件，无需创建 `-backup.md` 或其他备份文件。

修改前仍应读取原文件完整内容，确认现有需求边界、跨文档引用和模块索引，避免覆盖未理解的用户改动。

### 实现文档 & 索引文档

- `docs/[模块]/技术/*.md` — 可直接修改
- `INDEX.md` — 可直接修改
