# Issue Builder

将用户口语化或零散的需求、Bug 描述改写为专业、清晰、结构化的 Issue Markdown 文档，并检索代码仓库列出相关文件路径。

**核心原则：只列举事实和现象，不分析根因，不给出解决方案，不做结论或推测。**

## 使用方式

用户输入粗略问题描述，Issue Builder 输出：

1. 结构化 Markdown Issue 文档 → 保存至 `issues/inbox/`
2. 可选更新 JSON 索引 → `issues/index.jsonl`

## 工作流程

### Step 1：信息提取

从用户描述中提取：

- 问题现象或需求要点
- 涉及模块、页面、组件
- 关键词用于代码检索

若描述过于模糊，向用户确认最小必要信息（不要过度追问）。

### Step 2：代码仓库检索

**必须使用工具实际检索代码，严禁编造文件路径。**

检索策略（按需组合，并行执行）：

| 检索方式     | 工具                  | 用途                         |
| ------------ | --------------------- | ---------------------------- |
| 语义搜索     | `SearchCodebase`      | 按功能意图查找相关模块       |
| 关键字搜索   | `Grep`                | 搜索错误信息、函数名、变量名 |
| 文件路径查找 | `Glob`                | 查找特定目录或文件模式       |
| 符号定义     | `LSP workspaceSymbol` | 定位类、函数、接口定义位置   |
| 引用追踪     | `LSP findReferences`  | 追踪调用链上下游             |

**文件分级标准：**

- **高相关**：直接涉及问题核心逻辑的文件（最多 8 个）
- **中相关**：与问题有间接关联、可能需要了解的文件（最多 10 个）
- **低相关**：背景上下文、配置文件等（可选，不限数量）

每个文件必须记录：

- 关联原因（为什么与 Issue 相关）
- 证据（从哪个函数/类/导出发现的关联）
- 置信度（High / Medium / Low）

### Step 3：生成 Issue 编号

检查 `issues/index.jsonl`（如存在）获取已有最大编号，新 Issue 编号 = 最大编号 + 1。

若索引文件不存在，从 `ISSUE-001` 开始。

编号格式：`ISSUE-{三位数字}`，如 `ISSUE-001`、`ISSUE-042`。

### Step 4：生成 Markdown 文档

使用下方模板，保存至 `issues/inbox/ISSUE-{编号}-{简短描述}.md`。

文件名中的简短描述：小写字母、数字、连字符，最长 40 字符，概括问题核心。

示例：`ISSUE-001-fix-image-stretch.md`、`ISSUE-015-add-dark-mode-toggle.md`

### Step 5：更新 JSON 索引（可选）

向 `issues/index.jsonl` 追加一行（文件不存在则创建）：

```json
{
  "id": "ISSUE-001",
  "title": "修复图片拉伸问题",
  "status": "Todo",
  "priority": "High",
  "type": "Bug",
  "files": [
    "packages/author-site/src/components/image-viewer.tsx",
    "packages/shared/src/types.ts"
  ]
}
```

每行一条 Issue，字段：`id`、`title`、`status`、`priority`、`type`、`files`（高相关文件路径列表）。

## Issue 模板

```markdown
# ISSUE-{编号}: {标题}

## 1. 元信息

- 类型：{Bug / Feature / Task / Documentation}
- 优先级：{High / Medium / Low}
- 状态：Todo
- 创建来源：Issue Builder
- 是否需要代码修改：{Yes / No}

## 2. 用户原始描述

{保留用户原话，不做任何改写}

## 3. 任务背景

{说明为什么需要做这个任务，提供必要的上下文}

## 4. 当前问题 / 当前需求

{客观描述当前存在的问题或需要新增的能力}

## 5. 期望结果

{描述完成后应达到的状态}

## 6. 影响范围

{列出可能受影响的模块、页面、组件或功能}

## 7. 相关代码文件路径

### 高相关

- {path/to/file}
  - 关联原因：{为什么与 Issue 相关}
  - 证据：{具体函数/类/导出/配置项}
  - 置信度：High

### 中相关

- {path/to/file}
  - 关联原因：{为什么与 Issue 相关}
  - 证据：{具体函数/类/导出/配置项}
  - 置信度：Medium

### 低相关

- {path/to/file}
  - 关联原因：{为什么与 Issue 相关}
  - 证据：{具体函数/类/导出/配置项}
  - 置信度：Low

## 8. 执行约束

- {列出执行此任务时必须遵守的约束条件}
- {如：不能破坏现有 API、必须兼容某浏览器、需要保留某配置项}

## 9. 不要做的事

- {明确列出禁止行为，防止后续执行模型越界}
- {如：不要重构无关模块、不要修改数据库 Schema}

## 10. 验收标准

- [ ] {具体的、可验证的验收条件 1}
- [ ] {具体的、可验证的验收条件 2}

## 11. 建议验证方式

{如何验证任务完成，如：运行哪个测试、访问哪个页面、执行哪个命令}

## 12. 不确定信息

{如有不确定信息，如实列出，标注不确定原因；若无则写"暂无"}

## 13. 完成后必须输出

- 修改文件列表
- 如何验证
- 遗留风险
```

## 填写规范

### 元信息优先级判定

| 优先级 | 判定条件                                         |
| ------ | ------------------------------------------------ |
| High   | 核心功能不可用、数据丢失、安全漏洞、阻塞其他任务 |
| Medium | 功能异常但有规避方案、体验问题、非核心功能缺失   |
| Low    | 优化建议、文档补充、代码整洁、边缘场景           |

### 任务类型判定

| 类型          | 适用场景                         |
| ------------- | -------------------------------- |
| Bug           | 现有功能行为不符合预期           |
| Feature       | 新增功能或能力                   |
| Task          | 重构、迁移、配置变更、技术债清理 |
| Documentation | 文档新增或更新                   |

### 禁止行为

- **禁止编造文件路径**：所有路径必须来自实际代码仓库检索结果
- **禁止分析根因**：只描述现象，不推测为什么出问题
- **禁止给出解决方案**：不写"建议修改 xxx"、"可以通过 xxx 解决"
- **禁止做结论性判断**：不写"这是 xxx 导致的"、"问题出在 xxx"
- **禁止遗漏用户原话**：用户原始描述必须完整保留

## 输出完成后

向用户报告：

1. Issue 文件路径
2. 检索到的相关文件数量（高/中/低）
3. 是否需要补充信息
