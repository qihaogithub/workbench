# React组件与事件：状态传递与回调链路

> 从AI对话区暂停按钮无效问题分析报告中提取的React组件状态传递与事件处理经验

---

## 一、核心定位与设计哲学

**解决的最核心问题**：父组件定义的回调函数如何正确传递给深层子组件，以及状态变更如何触发正确的UI更新。

**铁律**：组件间的事件传递链路必须完整且显式。任何缺失的链路都意味着功能失效。

---

## 二、架构机制与正确方向

### 2.1 Context驱动的状态分发

`PromptInput`组件通过React Context向子组件分发状态和回调：

```typescript
interface PromptInputContextValue {
  text: string;
  setText: (text: string) => void;
  files: File[];
  addFiles: (files: File[]) => void;
  clearFiles: () => void;
  onSubmit: (data: { text: string; files: File[] }) => void;
  status: 'idle' | 'loading' | 'streaming';
  onCancel?: () => void;  // 可选的取消回调
}
```

ContextValue中的字段即为子组件可访问的全部状态和回调。

### 2.2 状态驱动的UI渲染

按钮根据`status`状态渲染不同图标：

```typescript
{status === 'loading' ? (
  <Loader2 className="h-4 w-4 animate-spin" />
) : status === 'streaming' ? (
  <Square className="h-3 w-3 fill-current" />  // 暂停图标
) : (
  <Send className="h-4 w-4" />
)}
```

**关键**：UI状态与业务状态必须一一对应，不存在"显示暂停图标但点击无响应"的状态。

### 2.3 回调链路的三层传递

```
AIChat (定义 handleCancel)
    ↓ 通过 onCancel={handleCancel} 传递给
PromptInput (通过 Context 分发)
    ↓ 子组件通过 usePromptInput() 获取
PromptInputSubmit (调用 context.onCancel())
```

## 三、反模式与历史避坑

### 3.1 缺少streaming状态的事件处理

**❌ 错误写法**（prompt-input.tsx:430-435）：
```typescript
const handleClick = () => {
  if (status === 'idle' && (context.text.trim() || context.files.length > 0)) {
    context.onSubmit?.({ text: context.text.trim(), files: context.files })
    context.setText('')
    context.clearFiles()
  }
  // streaming 状态时，没有任何处理逻辑！
}
```

**根因**：按钮的UI已经根据status渲染为暂停图标，但handleClick只处理idle状态。streaming状态的点击被完全忽略。

**✅ 正确写法**：
```typescript
const handleClick = () => {
  if (status === 'idle' && (context.text.trim() || context.files.length > 0)) {
    context.onSubmit?.({ text: context.text.trim(), files: context.files })
    context.setText('')
    context.clearFiles()
  } else if (status === 'streaming') {
    context.onCancel?.()
  }
}
```

### 3.2 Props接口与ContextValue不匹配

**❌ 错误做法**：在props接口中定义回调，但ContextValue中未包含

```typescript
interface PromptInputProps {
  onCancel?: () => void;  // 定义了
  // ...
}

// 但实际ContextValue中缺少onCancel
```

**根因**：Props接口定义了属性，但ContextValue未同步更新，导致子组件无法通过usePromptInput获取。

**✅ 正确做法**：Props接口、ContextValue、handleClick三个位置必须同步更新。

### 3.3 回调链路中间断裂

**❌ 错误做法**：父组件定义了回调，但未传递给中间组件

```typescript
// AIChat中定义了handleCancel
<PromptInput />  // 未传递 onCancel
```

**根因**：调用链路断裂，回调永远无法到达目标组件。

**✅ 正确做法**：
```typescript
<PromptInput onCancel={handleCancel} />
```

## 四、核心指标与安全边界

### 4.1 事件处理响应时间

| 场景 | 预期响应 | 说明 |
|:-----|:---------|:-----|
| 按钮点击（idle→loading） | < 16ms | 下帧即可见loading态 |
| 按钮点击（streaming→idle） | < 16ms | 流中断，消息保存 |
| 配置面板变更（UPDATE_CONFIG） | ~16ms | 下一帧渲染 |

### 4.2 状态机完整性检查

每次实现状态变更时，必须检查：
1. **UI层**：该状态下应该显示什么图标/文字？
2. **业务层**：该状态下点击应该触发什么逻辑？
3. **链路层**：回调是否能从触发点到达处理函数？

任何一层缺失都会导致功能bug。

---

## 五、关键教训

### 5.1 UI与业务必须同步

按钮显示什么图标是UI层的事，点击后做什么是业务层的事。两者必须对同一status状态有一致处理。**禁止**：UI显示暂停图标但点击什么都不做。

### 5.2 Context字段即子组件全部能力

子组件通过Context获取的所有字段，就是它能做的全部事情。如果子组件需要新的能力（如取消），必须在ContextValue中添加对应字段，并在父组件中提供实现。

### 5.3 状态驱动的组件必须穷举所有状态

React的条件渲染如果使用`if...else if`，**必须**穷举所有可能的状态。遗漏的状态会成为"僵尸状态"——UI可能显示，但交互无响应。
