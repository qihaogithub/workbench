## 快速开始

### 方式 A：在服务器 Mac 本机构建部署（推荐）

该机用 **OrbStack** 作为容器引擎，`docker` CLI 未挂 Compose v2 插件（`docker compose` 报 unknown command），因此用 **`docker-compose`（连字符）**；并把 `.env.docker` 复制成 `.env` 让 compose 自动读取变量，规避 `--env-file`。

SSH 登录 `10.131.75.39` 后：

```bash
cd /Users/jojo/Documents/workbench
git pull                                   # 拉取最新代码

# 1) 确认 OrbStack 引擎在跑、docker 能连上 daemon
orb start 2>/dev/null || open -a OrbStack   # 启动 OrbStack
docker context use orbstack 2>/dev/null || true
docker info >/dev/null 2>&1 && echo "docker ok" || echo "OrbStack 未就绪，等几秒重试"

# 2) 数据目录 + 让 compose 自动加载变量
mkdir -p /Users/jojo/Documents/workbench/data
cp .env.docker .env

# 3) 构建并启动全部服务（含截图服务，默认已开启）
docker-compose up -d --build
docker-compose ps
```

`.env.docker`（复制为 `.env`）里 `APP_DATA_DIR` 已设为 `/Users/jojo/Documents/workbench/data`（macOS 友好路径），截图服务默认随 compose 启动。日后重新部署重复 `git pull` +（必要时 `cp .env.docker .env`）+ `docker-compose up -d --build` 即可。

> 若该机的 `docker` 之后挂上了 Compose v2 插件（`docker compose version` 正常），可改用 `docker compose --env-file .env.docker up -d --build`，无需复制 `.env`。

### 方式 B：从开发机远程推送（deploy-fast.sh）

在能连通 `10.131.75.39` 的开发机上，从项目根目录执行（走 SSH+rsync 推送、本地构建镜像上传）：

```bash
# 日常前端部署
scripts/deploy-fast.sh author viewer
# 只重新部署 agent-service
scripts/deploy-fast.sh agent
# 部署核心服务（Agent + 创作端 + 浏览端）
scripts/deploy-fast.sh core
# 完整部署（deploy.sh 默认已含截图服务）
scripts/deploy.sh
```

流程：代码同步 → 本地构建 Docker 镜像 → 上传镜像 → 重启服务 → 自动健康检查。注意目标机也需能跑 `docker compose`。

## 部署命令速查

### deploy-fast.sh 快捷名

| 短名 | 等价服务 | 适用场景 |
| --- | --- | --- |
| `author` | author-site | 只改了创作端页面/接口/依赖 |
| `viewer` | viewer-site | 只改了浏览端页面/依赖 |
| `agent` | agent-service | 只改了 Agent 后端 |
| `core` | agent-service + author-site + viewer-site | 前后端同时改动 |
| `shot` | screenshot-service | 只改了截图服务（构建耗时较长） |

可组合使用：`scripts/deploy-fast.sh author viewer`

### 常用选项

| 选项 | 说明 |
| --- | --- |
| `--dry-run` | 只打印部署范围，不真正执行 |
| `--full-sync` | 完整同步所有包（默认只同步目标服务需要的包） |
| `--remote-build` | 在服务器上构建镜像（兜底模式，需服务器内存 ≥ 3GB） |

### 部署前预检

```bash
scripts/deploy-fast.sh author viewer --dry-run
```

## 截图服务

截图服务（依赖 Chromium）**默认已随部署一起启动**：compose 已移除 `screenshot` profile，`deploy.sh` 默认 `DEPLOY_SERVICES` 也含 `screenshot-service`。compose 不再强制 `linux/amd64`，跟随宿主机原生平台构建（Apple 芯片下为 arm64，避免模拟拖慢/崩溃）。

如需临时**跳过**截图服务：

```bash
# 方式 A（本机 compose）：显式列出不含截图的服务
docker compose --env-file .env.docker up -d --build agent-service author-site viewer-site
# 方式 B（deploy.sh）：
INCLUDE_SCREENSHOT_SERVICE=false DEPLOY_SERVICES="agent-service author-site viewer-site" scripts/deploy.sh
```

## data 目录

- 数据目录由 `.env.docker` 的 `APP_DATA_DIR` 指定，当前为 `/Users/jojo/Documents/workbench/data`（macOS 需在 `/Users` 下，Docker 默认不共享 `/opt`）
- 方式 A（本机 compose）会自动挂载该目录；首次执行前用 `mkdir -p` 建好即可
- 方式 B（deploy.sh）日常**不会**覆盖 `data/`，若 `APP_DATA_DIR` 目录不存在会中断（防止切到空数据），首次可用 `ALLOW_CREATE_APP_DATA_DIR=true scripts/deploy.sh`

### 本地 data → 覆盖正式环境

> 高风险操作，会替换正式数据。

```bash

scripts/deploy-author-with-data.sh --dry-run --overwrite-data --confirm-overwrite-production-data
# 只备份正式 data
scripts/deploy-author-with-data.sh --backup-only
# 执行覆盖并验证
scripts/deploy-author-with-data.sh --deploy-author --overwrite-data --confirm-overwrite-production-data --verify
```

## 常见问题

**代码改了但线上没变** — 确认执行了构建部署命令（不是只重启容器）。如果改动不属于默认目标包，使用 `--full-sync`。

**加速部署提示缺少 packages** — 目标服务 Dockerfile 需要某个包但加速列表未包含。临时用 `--full-sync`，后续更新 `deploy.sh` 中的 targeted package 列表。

**部署后项目列表变空** — `data/` 可能被切到空目录。在服务器检查：`du -sh /Users/jojo/Documents/workbench/data`。确认后执行 `deploy-author-with-data.sh --overwrite-data --confirm-overwrite-production-data --verify` 恢复。

**SSH 连接失败** — 确认已安装 `sshpass`（`which sshpass`），并手动验证连接 `sshpass -p 123456 ssh jojo@10.131.75.39 "echo ok"`。注意 `10.131.75.39` 是局域网地址，需在能连通该内网的机器上执行部署。

**`.env.docker` 缺少配置** — 脚本要求 `INTERNAL_API_TOKEN`，缺失会中断部署。补齐后重试。
