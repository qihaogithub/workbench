# 正式环境部署指南

> 适用版本: Workbench 最新版本  
> 更新日期: 2026-07-16  
> 适用场景: 将本地代码部署到局域网正式服务器

## 概述

Workbench 使用一键部署脚本部署到局域网服务器。日常部署只需要执行一条命令,脚本会完成以下步骤:

1. **代码同步** - 使用 Rsync 将代码上传到服务器
2. **本地构建镜像** - 在本机构建 Docker 镜像（默认）
3. **镜像上传** - 将镜像归档上传并加载到服务器
4. **服务启动** - 使用 Docker Compose 重启服务
5. **部署后自检** - 验证容器运行状态、健康检查和 API 端点

默认正式环境服务器:

| 项目 | 默认值 |
| --- | --- |
| 服务器 | `root@10.130.33.131` |
| 远端目录 | `/opt/workbench` |
| SSH 私钥 | `~/.ssh/figma-mirror-deploy-key` |
| 数据目录 | `/opt/opencode-workbench/data` |
| 创作端 | `http://10.130.33.131:3200` |
| Agent 服务 | `http://10.130.33.131:3201` |
| 截图服务 | `http://10.130.33.131:3202` |
| 浏览端 | `http://10.130.33.131:3300` |

## 部署前检查

在项目根目录执行命令:

```bash
cd /Users/qh2/Documents/PGM/1·Work/workbench
```

确认以下内容已经准备好:

1. 本地存在 `.env.docker`,并且包含正式环境需要的配置。
2. `.env.docker` 中设置了 `INTERNAL_API_TOKEN`。
3. 本地存在 SSH 私钥 `~/.ssh/figma-mirror-deploy-key`。
4. 当前机器可以通过 SSH 连接正式服务器。
5. 本地 Docker 运行正常（默认使用本地构建模式）。

可以先做一次只读预览:

```bash
scripts/deploy-fast.sh author viewer --dry-run
```

看到类似输出即表示部署目标解析正常:

```bash
DEPLOY_BUILD_MODE="local" DEPLOY_SYNC_MODE="targeted" DEPLOY_SERVICES="author-site viewer-site" scripts/deploy.sh
```

## 最常用部署

### 重新部署创作端和浏览端

日常前端代码变更通常执行这个命令:

```bash
scripts/deploy-fast.sh author viewer
```

这个命令会:

1. 只部署创作端 `author-site` 和浏览端 `viewer-site`。
2. 默认使用加速同步,只同步目标服务构建需要的代码包。
3. 默认使用本地构建模式，在本机构建 Linux 镜像后上传到服务器。
4. 不覆盖正式环境的 `data/`。
5. 不重建截图服务。

### 只重新部署创作端

```bash
scripts/deploy-fast.sh author
```

适用于只改了创作端页面、接口或创作端依赖包的情况。

### 只重新部署浏览端

```bash
scripts/deploy-fast.sh viewer
```

适用于只改了浏览端展示页面或浏览端依赖包的情况。

### 重新部署核心服务

```bash
scripts/deploy-fast.sh core
```

`core` 等价于部署:

- `agent-service`
- `author-site`
- `viewer-site`

适用于同时改了 Agent 后端、创作端和浏览端相关代码的情况。

## 加速同步说明

`scripts/deploy-fast.sh` 默认启用 `targeted sync`。它会根据部署目标只同步相关 workspace 包,减少局域网传输和远程构建上下文准备时间。

常见场景可以直接使用默认加速模式:

```bash
scripts/deploy-fast.sh author viewer
```

如果不确定本次改动是否被目标服务覆盖,可以改用完整同步:

```bash
scripts/deploy-fast.sh author viewer --full-sync
```

建议使用完整同步的情况:

1. 新增或移动了 workspace 包。
2. 修改了部署脚本、Dockerfile 或 workspace 依赖关系。
3. 修改范围很大,不确定属于哪个服务。
4. 加速部署失败,并提示缺少某个 `packages/...` 包。

## data 目录规则

日常部署不会覆盖正式环境 `data/`:

```bash
scripts/deploy-fast.sh author viewer
```

脚本同步代码时会排除本地 `data/`,正式服务器上已有的数据会保留。

普通部署还会检查 `.env.docker` 中的 `APP_DATA_DIR`。如果这个目录在正式服务器上不存在,部署会直接中断,避免服务切到一个新建的空 `data/` 目录。

只有在明确执行 `deploy-author-with-data.sh` 的覆盖命令时,才会用本地 `data/` 覆盖正式环境。

首次部署到一台全新服务器时,如果确认就是要创建空 `data/` 目录,可以显式允许:

```bash
ALLOW_CREATE_APP_DATA_DIR=true scripts/deploy.sh
```

## 使用本地 data 覆盖正式环境

这个操作会替换正式环境数据,风险较高。执行前请确认本地 `data/` 就是希望上线的数据。

先查看将要操作的环境:

```bash
scripts/deploy-author-with-data.sh --dry-run --overwrite-data --confirm-overwrite-production-data
```

只备份正式环境 data:

```bash
scripts/deploy-author-with-data.sh --backup-only
```

部署创作端、覆盖正式 data,并验证:

```bash
scripts/deploy-author-with-data.sh --deploy-author --overwrite-data --confirm-overwrite-production-data --verify
```

这个命令会:

1. 备份远端正式环境 `data/`。
2. 上传本地 `data/` 到远端临时 staging 目录。
3. 停止共享 data 的服务。
4. 用本地 `data/` 覆盖正式环境 data。
5. 重新启动服务。
6. 验证容器健康和 HTTP 端点。

远端备份默认保存在:

```bash
/opt/workbench-data-backups
```

## 使用正式环境 data 覆盖本地

这个操作会替换本地 `data/`。执行前脚本会先读取正式环境 `APP_DATA_DIR`,并在覆盖前备份当前本地 `data/`。

先查看将要操作的环境:

```bash
scripts/sync-production-data-to-local.sh --dry-run --overwrite-local-data --confirm-overwrite-local-data
```

只备份本地 data:

```bash
scripts/sync-production-data-to-local.sh --backup-only
```

拉取正式 data,覆盖本地 data:

```bash
scripts/sync-production-data-to-local.sh --overwrite-local-data --confirm-overwrite-local-data
```

这个命令会:

1. 检查 SSH 连接和正式环境 `APP_DATA_DIR`。
2. 拉取正式环境 `data/` 到本地 staging 目录。
3. 备份当前本地 `data/`。
4. 用 staging 中的正式数据覆盖本地 `data/`。
5. 输出覆盖后的本地 data 摘要。

本地备份和 staging 默认保存在项目目录外:

```bash
../workbench-data-backups
../workbench-data-staging
```

## 部署后验证

脚本会自动做基础健康检查。部署完成后,也可以手动检查:

```bash
curl -I http://10.130.33.131:3200
curl -I http://10.130.33.131:3300
curl http://10.130.33.131:3201/health
curl http://10.130.33.131:3202/health
```

预期结果:

- 创作端 `3200` 返回 HTTP 200 或可访问页面。
- 浏览端 `3300` 返回 HTTP 200 或可访问页面。
- Agent 服务 `3201/health` 返回健康状态。
- 截图服务 `3202/health` 返回健康状态。

如果需要检查截图服务的深度健康状态:

```bash
SCREENSHOT_HEALTH_URL=http://10.130.33.131:3202/health?deep=1 scripts/docker-screenshot-deep-health.sh
```

## 截图服务部署

默认部署不会重建截图服务,因为截图服务依赖 Chromium,构建耗时较长,也更容易受服务器资源和系统依赖影响。

只有明确需要更新截图服务代码时,再部署截图服务:

```bash
scripts/deploy-fast.sh shot
```

如果要在完整部署中包含截图服务:

```bash
INCLUDE_SCREENSHOT_SERVICE=true scripts/deploy.sh
```

注意: 截图服务部署会在本地构建包含 Chromium 的镜像，镜像体积较大，上传时间可能较长。

## 常见问题

### 代码改了,但线上没有变化

先确认执行的是重新构建部署命令,不是只重启容器。推荐使用:

```bash
scripts/deploy-fast.sh author viewer
```

如果改动不属于创作端或浏览端依赖包,需要部署对应服务,或使用完整同步:

```bash
scripts/deploy-fast.sh author viewer --full-sync
```

### 加速部署提示缺少 packages

说明目标服务的 Dockerfile 需要某个包,但加速同步列表没有包含它。临时处理可以使用完整同步:

```bash
scripts/deploy-fast.sh author viewer --full-sync
```

后续再更新 `scripts/deploy.sh` 中对应服务的 targeted package 列表。

### 部署后项目列表变空

先确认正式环境 `data/` 是否被切到了空目录。可以在服务器上检查:

```bash
du -sh /opt/opencode-workbench/data
find /opt/opencode-workbench/data/projects -maxdepth 1 -mindepth 1 | wc -l
```

如果确认要用本机 `data/` 恢复正式环境,执行:

```bash
scripts/deploy-author-with-data.sh --overwrite-data --confirm-overwrite-production-data --verify
```

该命令会先备份当前正式 `data/`,再用本机 `data/` 覆盖正式环境并验证服务。

### SSH 连接失败

检查私钥是否存在:

```bash
ls -l ~/.ssh/figma-mirror-deploy-key
```

检查是否能连接服务器:

```bash
ssh -i ~/.ssh/figma-mirror-deploy-key root@10.130.33.131
```

### `.env.docker` 缺少配置

部署脚本会从 `.env.docker` 生成远端环境文件。如果缺少 `INTERNAL_API_TOKEN`,部署会中断。请补齐 `.env.docker` 后重试。

### 想先确认命令会部署哪些服务

使用 `--dry-run`:

```bash
scripts/deploy-fast.sh author viewer --dry-run
scripts/deploy-fast.sh core --dry-run
scripts/deploy-fast.sh shot --dry-run
```

## 推荐操作顺序

日常代码部署:

```bash
scripts/deploy-fast.sh author viewer --dry-run
scripts/deploy-fast.sh author viewer
curl -I http://10.130.33.131:3200
curl -I http://10.130.33.131:3300
```

需要覆盖正式 data:

```bash
scripts/deploy-author-with-data.sh --backup-only
scripts/deploy-author-with-data.sh --deploy-author --overwrite-data --confirm-overwrite-production-data --verify
```

需要用正式 data 覆盖本地:

```bash
scripts/sync-production-data-to-local.sh --dry-run --overwrite-local-data --confirm-overwrite-local-data
scripts/sync-production-data-to-local.sh --overwrite-local-data --confirm-overwrite-local-data
```
