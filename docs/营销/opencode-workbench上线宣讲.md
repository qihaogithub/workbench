## 01. 标题页

### 页面标题

Opencode Workbench

### 副标题

AI 辅助组件创作与使用平台：把组件从想法、代码、配置到预览嵌入，收敛到一个可协作的工作台。

### 页面辅助信息

- 内部上线宣讲
- 2026.07
- 面向：开发者、设计师、产品、运营、集成方

### 讲述重点

这不是单纯的 AI 写代码工具，也不是只给开发使用的后台。它的定位是把组件生产、配置验收和外部嵌入这条链路打通，让团队能围绕同一个项目工作台协作。

## 02. 为什么现在需要它

### 页面标题

组件从“能写出来”到“能被别人稳定使用”，中间有三段断点。

### 页面内容

#### 断点一：代码生成和人工修改割裂

需求变化后，代码、预览、文件状态和历史回溯经常分散在不同工具里。

#### 断点二：非技术同事难以参与配置

组件参数、配置 Schema、预览效果如果都依赖开发解释，协作成本会持续增加。

#### 断点三：交付之后缺少统一入口

组件浏览、版本、嵌入、截图和回归验证需要一条能被团队复用的标准链路。

### 讲述重点

项目上线不是为了“多一个系统”，而是解决组件协作中反复出现的断点：开发能做，但业务难参与；组件能跑，但难配置；页面能交付，但难复用。

## 03. 一句话定位

### 页面标题

一个把“创作、使用、集成”打通的组件工作台。

### 页面内容

Opencode Workbench 面向组件化开发，提供从组件设计、AI 辅助编码、配置管理、实时预览到 iframe 嵌入的完整工作流。

核心链路：

- 创作端：AI 对话、代码修改、Schema、预览、版本
- 使用端：浏览项目、调整配置、验收效果
- 嵌入集成：iframe + postMessage 接入外部系统
- 底层语义：项目、工作区、Session、快照、模板、CLI 复用同一套项目读写规则

### 讲述重点

强调“同一套项目语义”。创作端、使用端、外部嵌入不是三个割裂入口，而是围绕同一个项目当前态和配置协议展开。

## 04. 谁会使用

### 页面标题

不是只给开发用，也不是另一个低代码壳。

### 页面内容

#### 开发者 / 设计师

负责把想法变成可维护组件。

- 用自然语言推进组件生成和修改
- 维护配置 Schema 和预览状态
- 用快照和版本保护关键变更

#### 产品 / 运营

负责看得懂、调得动、验得准。

- 在使用端浏览组件项目
- 通过表单调整配置参数
- 用真实预览参与上线验收

#### 外部系统集成方

负责把组件放进业务现场。

- iframe 方式快速嵌入
- postMessage 动态传递配置
- 减少重复开发和一次性页面

### 讲述重点

讲清楚每类同事的入口，不要让大家误解成“这是研发内部工具”。项目的价值在于让更多角色能在自己的边界内参与组件交付。

## 05. 端到端核心流程

### 页面标题

从一句需求，到一个可配置、可预览、可嵌入的组件。

### 页面内容

1. 创建项目  
   初始化项目基准工作区，定义组件交付边界。

2. AI 对话  
   Agent 根据自然语言修改工作区文件。

3. 配置 Schema  
   把可变参数沉淀为表单和组件 Props。

4. 实时预览  
   动态编译后立即看效果，配置修改能实时反馈到组件渲染。

5. 快照版本  
   关键动作同步当前态并生成可回溯历史。

6. 发布嵌入  
   使用端验收后，通过 iframe 集成到外部系统。

### 讲述重点

这一页可以作为主流程页。不要展开太多技术细节，只让同事建立“从需求到嵌入”的完整路径感。

## 06. 创作端价值

### 页面标题

给开发同事的是一个“能落地”的 AI 工作区。

### 页面内容

它不只回答问题，而是围绕项目工作区、文件变更、实时预览和版本快照形成可追踪的生产过程。

#### AI 辅助编码

- 自然语言指令生成或修改组件
- 文件读写、截图、控制台等工具协同

#### 配置与预览

- Schema 解析为可操作表单
- TSX 动态编译后实时渲染

#### 工作空间语义

- Session 不拥有文件
- 项目工作区承载当前态
- 快照承载历史回溯

#### 知识库上下文

- 工作空间参考文档可被 AI 查阅
- 降低重复说明和上下文丢失

### 讲述重点

强调“AI 修改必须被项目语义约束”。这能帮助同事理解为什么要做工作区、快照、诊断、工具权限，而不是只接一个聊天框。

## 07. 使用端价值

### 页面标题

让非技术同事不用等开发，也能参与验收。

### 页面内容

使用端提供低门槛入口：

- 项目列表：浏览组件项目，按项目维度寻找可用资产
- 配置面板：通过 JSON Schema 生成表单，让参数调整可视化
- 实时预览：配置修改后直接看到组件效果
- 独立部署：使用端可独立运行，服务非开发角色

### 建议展示画面

可以放一个抽象的“预览窗口 + 配置表单”视觉：

- 左侧是组件预览画布
- 右侧是配置表单
- 下方标注“配置驱动真实组件渲染”

### 讲述重点

让产品和运营知道：他们不是只能提需求和等结果，而是可以直接参与配置、预览和验收。

## 08. Agent 服务层价值

### 页面标题

AI 能力独立出来，前端专注 UI，服务层负责业务和工具边界。

### 页面内容

#### Pi Agent 单后端

采用进程内嵌入式 Agent 后端，不依赖外部 OpenCode Server 或外部 CLI 子进程。

#### HTTP + WebSocket

- REST 管理项目、Session、文件和模型配置
- WebSocket 推送流式响应与文件变更

#### 工具权限与变更捕获

覆盖文件读写、截图、控制台、计划审批、页面删除和子 Agent 委派等能力。

#### Session Guard + Snapshot

通过文件边界校验、变更比较和快照，把 AI 修改变成可审计的项目行为。

### 讲述重点

这一页面向稍微技术一点的同事，重点解释为什么 Agent 是独立服务层：为了边界、权限、状态、事件和后续扩展。

## 09. 平台工程能力

### 页面标题

能上线，也要能持续维护。

### 页面内容

#### 共享项目语义

`project-core`、`project-scaffold`、`project-cli` 复用同一套项目读写和脚手架能力，避免 Web、CLI、模板各自实现。

#### 诊断与回归

创作端诊断日志、CLI 查询、Jest、Vitest、Playwright 共同形成问题闭环。

#### 部署边界清晰

Docker Compose 覆盖 author-site、agent-service、screenshot-service、viewer-site 四个服务。

#### 截图与预览治理

截图服务支持同步单页截图、异步批量截图、编译缓存和文件系统截图缓存，帮助项目预览和验收。

### 讲述重点

这一页传达“不是 Demo 工程”。系统已经围绕上线后的维护、排查、部署和回归建立了工程支撑。

## 10. 项目特色总结

### 页面标题

它把组件交付需要的关键能力放在同一条链路里。

### 页面内容

#### AI 辅助编码

基于 Pi Agent，自然语言指令生成或修改组件代码。

#### 配置驱动

JSON Schema 驱动表单生成，非技术人员可以调整配置。

#### 实时预览

动态编译方案，轻量快速，配置变更能即时反馈到预览区。

#### 版本管理

快照备份和版本历史支持回退。

#### 项目工作区

项目工作区承载当前态，分支工作区用于显式隔离事务。

#### 知识库

工作空间内管理参考文档，为 AI 提供上下文依据。

#### 嵌入集成

iframe + postMessage 协议支持外部系统动态嵌入组件。

#### 项目工具链

Web API、CLI 和脚手架转换复用同一项目读写语义。

### 讲述重点

这一页适合作为阶段性总结。如果时间紧，可以压缩成 4 个关键词：AI、配置、预览、嵌入。

## 11. 同事可以怎么参与

### 页面标题

这不是“发布一个系统”，而是启动一条新的组件交付方式。

### 页面内容

#### 试用

用一个真实组件跑完整链路，从 AI 修改、配置、预览到快照，看每一步是否符合团队实际习惯。

#### 沉淀

把常用组件做成模板，让后续项目不用重复搭结构、补说明、复制配置。

#### 补充

提供知识库材料，把规范、品牌、业务说明变成 AI 可查阅的可靠上下文。

#### 集成

选择一个外部场景嵌入，验证 iframe + postMessage 在业务系统里的真实交付效果。

### 讲述重点

让听众知道下一步能做什么。宣讲结束后最好能带走一个明确试点组件或试点场景。

## 12. 建议落地节奏

### 页面标题

先用小范围真实项目验证，再把可复用经验沉淀为模板和规范。

### 页面内容

#### 第一步：宣讲与 Demo

让团队先理解定位、角色入口和端到端链路。

#### 第二步：选择试点组件

挑一个真实业务组件，验证 AI 修改、配置预览和嵌入交付。

#### 第三步：固化模板与知识

把稳定流程沉淀成模板、Schema、知识库和验收标准。

#### 第四步：接入更多业务

通过使用端和嵌入 API，把组件能力扩散到更多系统。

### 讲述重点

不要一上来追求全量推广。先用真实组件跑通一个闭环，再把可靠做法固化。

## 13. 结束页

### 页面标题

从会写代码的工具，到能交付组件的工作台。

### 页面内容

Opencode Workbench 的价值不在于单点功能更酷，而在于把团队围绕组件生产、配置验收和外部集成的协作链路统一起来。

### 行动号召

找一个真实组件试起来。

最好的宣传方式，是让同事看到它怎样把一个组件从需求推进到可用交付。

### 讲述重点

结尾要收回到“真实组件试点”。避免变成纯功能介绍，要推动同事参与试用和反馈。
