# OneFlow 上线宣讲大纲

> 用途：上线前给同事做内部宣讲，用于先确认整体叙事结构和章节顺序。
> 当前状态：面向全员的协作叙事大纲，不展开每一页的具体内容。
> 宣讲对象：活动策划、产品经理、UI 设计师、运营设计师、开发者，以及后续需要使用活动页面和活动配置能力的业务同事。
> 核心主张：OneFlow 不是多一个工具入口，而是一次活动协作方式升级。它的目标是让产品经理和 UI 设计师的经验沉淀为活动模板、功能知识和资源规范，让活动策划借助 AI 自助生成专业可交互原型，让运营设计师直接查看配置项、上传资源并实时预览效果；需要开发时，开发者可以通过 CLI 获取项目代码、配置协议、高保真页面代码和上下文。

---

## 一、全员宣讲主线

### 1. 从“工具上线”讲成“协作方式上线”

宣讲开场不先讲功能列表，而是先建立共识：现在的活动交付不是某一个角色低效，而是知识、规范、原型、资源和代码分散在不同角色与工具里。

### 2. 从“每个活动都靠人咨询”讲到“经验沉淀成模板”

过去每个活动都要反复问产品功能、问 UI 规范、问后台配置、问开发边界。OneFlow 的目标不是让产品和 UI 更频繁地跟进活动，而是让他们把经验沉淀成活动模板、功能知识和资源规范，让 AI 可以被策划和运营设计直接调用。

### 3. 从“交付设计稿和切图”讲到“交付可运行的高保真页面代码”

在 AI 参与开发之后，只交付 Figma、切图和标注已经不是最优协作方式。UI 设计师或运营设计师可以在 OneFlow 中通过 Vibe Coding 产出高保真页面代码，这份代码本身就是开发者 AI 精确还原视觉的上下文。

### 4. 从“一个真实活动闭环”讲清平台价值

全场最好围绕一个真实活动案例展开：

活动目标进入 OneFlow -> AI 结合知识库和活动模板给出方案建议 -> 活动策划自助生成专业可交互原型图 -> 原型自动沉淀页面、配置项和资源规范 -> UI 或运营设计通过 Vibe Coding 形成高保真页面代码 -> 运营设计查看配置项、上传资源并实时预览效果 -> 如需开发，开发者通过 CLI 获取项目代码、高保真页面代码和上下文 -> 验收发布 -> 后续复用。

## 二、推荐宣讲结构

### 01. 标题与定位

#### 本段目的

让听众立刻知道 OneFlow 是什么，以及为什么它不是普通的 AI 工具或活动后台。

#### 建议表达

- OneFlow 是面向活动策划、活动原型、配置资源、视觉实现和页面交付的 AI 工作流平台。
- 它的终极目标是让活动策划借助 AI 独立产出专业、可交互、可配置、可预览的活动原型图。
- 它让运营设计师直接看到本次活动有哪些配置项、每个配置资源有什么规范，并上传资源实时预览效果。
- 它把产品功能知识、活动模板、UI 资源规范和历史活动经验沉淀为 AI 可查阅的知识库。
- 它让 UI 设计师和运营设计师不只交付 Figma、切图和标注，也可以通过 Vibe Coding 直接产出高保真页面代码。
- 它让开发者通过 CLI 获取项目代码、配置协议、高保真页面代码和上下文，用自己的 AI 更精确地还原视觉和实现新功能。

#### 需要避免

- 不要把 OneFlow 讲成“AI 自动生成页面”。
- 不要把 OneFlow 讲成替代产品、UI、运营设计或开发。
- 要强调角色变化：产品和 UI 从具体活动跟进者，转为模板、功能知识和资源规范维护者。

---

### 02. 当前协作痛点

#### 本段目的

建立共识：现在的成本主要来自活动协作链路太长，关键知识分散，原型和资源规范难以一次到位。

#### 过去的典型流程

1. 活动策划提出活动方案和业务目标。
2. 产品经理根据策划方案，尽量使用当前活动系统已有功能来实现。
3. 如果已有功能无法满足活动目标，产品经理再判断是否需要开发新功能。
4. 活动策划不清楚当前活动系统的功能细节，只能依赖产品经理判断。
5. 产品经理通常只熟悉自己做过的功能，涉及其他项目组能力时还需要咨询其他产品经理。
6. 产品经理确定方案后，还需要和 UI 设计师确认具体功能页面的资源规范。
7. 活动策划、产品经理和 UI 设计师共同协作，才能产出准确的活动原型图。
8. 运营设计师期望拿到一份完善原型图：能看到所有活动页面、正式文案、每个页面需要制作的资源清单和资源规范。
9. 但这份原型图依赖活动策划、产品经理和 UI 设计师三方共同协作，任何一方支持不足，原型图都会不完整。
10. 过去 UI 资源规范主要沉淀在钉钉文档里，长期会膨胀成一个巨大的规范文档库。
11. 运营设计师制作资源时，通常还需要 UI 额外整理一份配置项列表，里面写明每个配置项对应哪一篇规范文档。
12. 由于这些规范文档按具体功能维护，而不是按本次活动结构维护，运营设计师可能需要打开几十个文档才能完成一次活动设计。
13. 运营设计师再根据原型图、资源规范和本次活动资源清单完成设计。
14. 如果不需要开发，设计师提供切图，业务方再到后台配置；如果需要开发，还要继续对接开发者。

#### 核心痛点

- 活动策划最了解业务需求，但不了解产品功能，无法独立产出准确原型图。
- 产品经理了解功能，但不是需求定义者，在策划阶段更像咨询顾问。
- UI 设计师掌握资源规范，但规范没有和原型图、配置项自然绑定。
- 运营设计师依赖完善原型图开展设计，但现实中产品经理和 UI 设计师不一定每次都能给到足够支持。
- 运营设计师要查阅大量分散文档，无法在一个地方看到页面、配置项、规范和资源清单全局。
- UI 资源规范以钉钉文档库维护时，文档按功能逻辑组织，不按活动结构组织；UI 只能用最低成本维护通用规范，无法每次活动都定制一套活动视角的规范说明。
- 传统规范文档是静态的，阅读者只能看文字和图片，不容易理解配置项在真实页面里的效果。
- 开发者需要重新理解策划意图、原型结构、资源规范、视觉要求和配置边界。

#### 过渡问题

如果产品功能、UI 规范、活动模板、原型图、资源清单、高保真页面代码和开发上下文都在同一个项目里流动，流程会怎样变化？

### 03. 未来协作图景

#### 本段目的

让全员先看到 OneFlow 最终要形成的协作方式。

#### 建议表达

- 产品经理不再逐个跟进具体运营活动，而是维护活动系统功能知识、活动类型模板、可复用能力边界和验收规则。
- UI 设计师不再逐个确认具体活动的资源规范，也不再维护一个越来越大的独立钉钉规范文档库，而是在不同活动模板项目中维护页面结构、配置项资源规范、配置项备注、功能状态、品牌标准和体验规则。
- 活动策划借助 AI 和知识库，自助生成专业、可交互、带配置资源清单和资源规范的活动原型图。
- 运营设计师不再依赖多份分散文档或不完整原型，而是在项目里直接看到有哪些配置项、每个配置项需要什么资源，并上传资源实时预览效果。
- 模板项目本身承载 UI 资源规范：每个配置项都可以有备注、规范说明和预览效果；规范不再只是静态文档，而是可预览、可交互、AI 可读取的项目知识。
- UI 设计师或运营设计师可以通过 Vibe Coding 产出高保真页面代码，让视觉、间距、状态和动效直接变成开发 AI 可读取的上下文。
- 开发者需要接入时，通过 CLI 获取项目代码、页面结构、配置协议、高保真页面代码和上下文。

#### 一句话总结

OneFlow 要把“每次活动都靠人补上下文”，变成“模板和知识先沉淀，策划和运营设计自助推进，开发只在必要时精准接入”。

### 04. 角色变化

#### 本段目的

让每类听众知道自己未来不是被替代，而是工作重心发生变化。

#### 活动策划

从“提出需求、等待产品和 UI 支持”，变成“借助 AI 生成可落地的活动方案和专业可交互原型图”。

#### 产品经理

从“每个活动的功能咨询顾问”，变成“活动模板、功能知识、能力边界和验收规则的维护者”。

#### UI 设计师

从“每个活动的资源规范确认者”，变成“活动模板视觉规范、页面结构、资源规范和高保真页面代码能力的维护者”。

#### 运营设计师

从“拿着不完整原型和分散规范做设计”，变成“在 OneFlow 里查看完整配置项和资源规范、上传资源实时预览，并在需要时通过 Vibe Coding 直接影响最终页面效果”。

#### 开发者

从“重新理解需求、标注、切图和上下文”，变成“通过 CLI 读取项目代码、配置协议、高保真页面代码和上下文，用 AI 精确实现新增能力”。

---

### 05. 核心工作流

#### 本段目的

用一条完整链路建立全局地图，这是整场宣讲的主干。

#### 建议流程

1. 产品经理和 UI 设计师先维护不同类型活动的项目模板、功能知识、页面结构和资源规范。
2. 活动策划提出活动目标、玩法、节奏和投放诉求。
3. AI 读取活动模板、活动系统功能、历史活动、资源规范和配置规则，给出可落地方案建议。
4. 活动策划和 AI 一起生成专业、可交互的活动原型图。
5. 原型图自动沉淀所有功能页面、正式文案、配置资源列表和每个资源的规范。
6. UI 设计师或运营设计师在需要时通过 Vibe Coding 产出高保真页面代码，直接表达视觉、间距、动效和页面状态。
7. 运营设计师查看本次活动全部配置项和资源清单，上传图片、动效、音效等资源。
8. OneFlow 根据配置项和上传资源提供实时预览。
9. 如果不需要开发，策划和运营设计师可基于预览完成配置和验收。
10. 如果需要新功能开发，开发者通过 CLI 获取项目代码、页面结构、配置协议、高保真页面代码和上下文。
11. 活动上线后，将新增能力、页面结构、配置项、资源规范和高保真页面经验反哺到模板或知识库。

#### 讲述重点

这页只讲顺序，不讲页面细节。让听众记住：OneFlow 的价值不是“多一个协作工具”，而是让活动从策划、原型、视觉、资源、预览、开发到复用都在同一个项目里连续流转。

### 06. 中心产物：可交互活动原型

#### 本段目的

解释为什么原型图是全员协作的中心，而不是只给产品或设计看的中间文件。

#### 原型图应该承载

- 所有活动页面。
- 正式文案。
- 每个页面的功能状态。
- 每个页面的配置项。
- 需要制作的资源清单。
- 每个资源的尺寸、格式、状态和使用规范。
- 每个配置项的备注、使用说明和关联规范。
- 真实配置和资源上传后的预览效果。

#### 讲述重点

运营设计师真正需要的不是“差不多的草图”，而是一份可执行原型。OneFlow 要让原型图成为活动策划、运营设计、产品模板、UI 规范和开发接入之间的共同语言。模板项目承载资源规范以后，规范不再是分散的钉钉文档，而是跟配置项、页面结构和预览效果绑定在一起。

---

### 07. AI 的定位：全员的知识顾问

#### 本段目的

澄清 AI 的角色：AI 不是替代所有人，而是读取被沉淀下来的模板、规范和知识，帮助不同角色减少重复咨询。

#### 建议表达

- AI 帮助活动策划理解当前活动系统能力，提供更符合运营目标和系统边界的方案建议。
- AI 基于产品经理维护的活动模板、功能知识和配置规则，替代产品经理在具体活动中的常规咨询角色。
- AI 基于 UI 设计师维护的页面结构和资源规范，替代 UI 在具体活动中的常规规范解释角色。
- AI 帮助运营设计师查阅资源规范，解释每个配置资源的尺寸、格式、状态和使用方式。
- AI 可以读取模板项目内的规范文档、配置项备注和预览状态，回答使用者关于资源制作和配置规则的问题。
- AI 帮助把活动目标、文案、页面结构、配置项和资源规范整理成专业、可交互的活动原型图。
- AI 帮助 UI 设计师或运营设计师用 Vibe Coding 生成高保真页面代码，把视觉要求变成开发 AI 可直接读取的上下文。
- 开发者的 AI 读取高保真页面代码后，可以更精确地还原视觉、间距、状态和动效。

#### 需要避免

- 不要讲成 AI 自动完成所有活动。
- 要强调 AI 的质量来自模板、知识库、资源规范、配置协议和真实预览约束。

---

### 08. Vibe Coding：AI 时代的设计交付方式

#### 本段目的

说明为什么在 AI 开发链路中，高保真页面代码比单纯 Figma、切图和标注更适合作为开发上下文。

#### 建议表达

- 过去设计交付依赖 Figma、切图、标注和口头解释。
- 现在开发人员也在使用 AI，高保真页面代码本身就是给开发 AI 的最佳提示词。
- UI 设计师可以在 OneFlow 中用 Vibe Coding 实现完全符合视觉要求的高保真页面。
- 在部分活动中，页面视觉更多由运营设计决定，运营设计师也可以加入 Vibe Coding，直接影响页面开发出来的效果。
- 开发者不需要反复确认切图、间距和视觉还原度，而是让 AI 读取高保真页面代码，再结合业务代码和配置协议实现最终页面。

#### 讲述重点

这不是要求设计师变成传统开发者，而是让设计侧可以把视觉判断转化为 AI 可执行、开发可读取的高保真代码上下文。

### 09. 平台能力边界

#### 本段目的

给出能力边界，帮助大家形成合理预期。

#### 当前可以重点讲

- 活动模板：沉淀不同类型活动的页面结构、功能组合、配置项、配置项备注和资源规范。
- 项目内规范文档：把原本分散在钉钉文档库里的 UI 资源规范沉淀到模板项目中，按活动结构和配置项组织。
- 知识库：沉淀活动系统功能、历史活动、资源规范、配置后台规则和业务说明，并让 AI 可以读取后回答使用者问题。
- 活动原型图：承载页面结构、正式文案、功能页面、配置资源列表和资源规范。
- AI 对话：辅助活动策划生成可交互原型，辅助运营设计师理解配置项和资源规范。
- Vibe Coding：支持 UI 设计师或运营设计师产出可运行的高保真页面代码。
- 配置系统：把活动可变内容沉淀成可操作参数。
- 资源上传与预览：让运营设计师上传素材后直接看到活动效果。
- 实时预览：支持运营设计师上传资源后立即查看活动效果。
- CLI 开发接入：让开发者获取项目代码、页面结构、配置协议、高保真页面代码和上下文。
- 快照与版本：保护关键变更和回溯能力。

#### 暂不展开

- 每个页面具体怎么操作。
- 每个技术模块的实现细节。
- 完整后台、部署和诊断机制。

### 10. 试点与参与方式

#### 本段目的

把宣讲从“听完理解”推进到“下一步参与”。

#### 推荐试点

优先选择一个真实、范围可控、需要多角色协作的活动。

#### 试点要验证什么

- 活动策划能否借助 AI 自助生成可交互原型。
- 原型是否能完整呈现活动页面、正式文案、配置项、资源清单和资源规范。
- UI 资源规范能否从分散钉钉文档，迁移为模板项目内可备注、可预览、可交互、AI 可读取的规范。
- 运营设计师能否基于配置项上传资源并实时预览。
- 产品经理和 UI 设计师维护的模板与规范，能否减少具体活动跟进成本。
- UI 或运营设计通过 Vibe Coding 产出的高保真页面代码，能否成为开发者 AI 精确还原视觉的上下文。
- 开发者能否通过 CLI 获取项目代码、配置协议、高保真页面代码和上下文。

#### 各角色如何参与

- 活动策划提供一个真实活动目标、玩法和文案方向。
- 产品经理提供并维护当前活动系统可复用功能、限制条件、活动模板和验收规则。
- UI 设计师提供并维护页面结构、资源规范和体验标准。
- 运营设计师提供素材、文案、动效、音效和配置诉求。
- UI 设计师或运营设计师在需要时通过 Vibe Coding 产出高保真页面代码。
- 开发者在需要新功能时，通过 CLI 接入项目代码、配置协议、高保真页面代码和上下文。

#### 讲述重点

试点不是为了证明“AI 能不能生成页面”，而是验证 OneFlow 能不能让产品和 UI 的经验转化为模板与规范，让策划和运营设计自助推进具体活动，让开发在必要时精准接入。

---

### 11. 上线节奏

#### 本段目的

给宣讲一个可执行落点。

#### 建议节奏

1. 内部宣讲：统一 OneFlow 的定位和协作方式。
2. 选择试点：确定一个真实活动。
3. 跑通闭环：完成活动目标、AI 可交互原型、配置资源清单、资源上传、实时预览、Vibe Coding 高保真页面代码、开发接入、验收和发布。
4. 复盘流程：记录哪些环节节省成本，哪些环节仍然卡住。
5. 沉淀模板：把产品功能、页面结构、资源规范、高保真页面经验和可复用配置项转成活动模板和知识库。
6. 扩展场景：逐步接入更多活动、运营模块和业务系统。

### 12. 结束收束

#### 本段目的

把宣讲主题收回到 OneFlow 的长期愿景。

#### 建议结尾

OneFlow 的上线不是多了一个工具入口，而是把产品经理和 UI 设计师的经验沉淀为活动模板、功能知识和资源规范，让活动策划可以借助 AI 自助生成专业可交互原型，让运营设计师可以直接查看配置项、上传资源并实时预览效果，让设计侧通过高保真页面代码影响最终页面，让开发者用 AI 精确读取上下文并接入新能力。
