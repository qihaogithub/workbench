# OneFlow 上线宣讲 PPT 逐页规划

> 用途：上线前内部宣讲 PPT 制作与现场讲解。本文按 PPT 每一页组织，每页包含“宣讲目的”“PPT 页面内容”“宣讲逐字稿”。
> 宣讲对象：活动策划、产品经理、UI 设计师、运营设计师、开发者，以及后续需要使用活动页面和活动配置能力的业务同事。
> 建议时长：30-40 分钟。

---

## 第 1 页：OneFlow 是什么

### 宣讲目的

让听众在开场就建立清晰定位：OneFlow 不是一个单点 AI 工具，也不是普通活动后台，而是围绕活动策划、原型、配置、资源、预览和开发接入形成的一套 AI 工作流平台。

### PPT 页面内容

页面标题：

- OneFlow：面向活动交付的 AI 工作流平台

页面主文案：

- 让活动从“策划想法”连续流转到“可交互原型、配置资源、实时预览和开发接入”
- 把产品功能知识、活动模板、UI 资源规范和历史经验沉淀为 AI 可读取的项目上下文
- 帮助策划自助生成专业原型，帮助运营设计查看配置项并实时预览，帮助开发精准获取实现上下文

页面视觉建议：

- 用一条横向流程表示：活动目标 -> AI 方案 -> 可交互原型 -> 配置资源 -> 实时预览 -> 开发接入 -> 复用沉淀。
- 页面底部放一句定位语：不是多一个入口，而是活动协作方式的升级。

### 宣讲逐字稿

大家好，今天这次宣讲主要介绍 OneFlow。

我先用一句话说清楚它是什么：OneFlow 是面向活动交付的 AI 工作流平台。它不是单独的 AI 聊天入口，也不是传统意义上的活动后台，而是把活动从策划目标、方案生成、可交互原型、资源配置、实时预览，到必要时的开发接入，放到同一个项目工作流里连续推进。

它背后的核心变化是：过去很多活动经验分散在产品经理、UI 设计师、运营设计师、开发者以及各种文档里。OneFlow 希望把这些经验沉淀为活动模板、功能知识、资源规范和项目上下文，让 AI 可以读取，让不同角色可以在同一个项目里协作。

所以今天我不会只讲有哪些按钮和页面，而是重点讲 OneFlow 会怎样改变我们做活动的协作方式。

## 第 2 页：为什么需要 OneFlow

### 宣讲目的

建立共识：当前活动交付的主要问题不是某个角色效率低，而是功能知识、UI 规范、资源要求、原型和开发上下文分散，导致每个活动都需要反复沟通和补齐信息。

### PPT 页面内容

页面标题：

- 现在的问题：每个活动都在重新补上下文

页面左侧：当前协作链路

1. 策划提出活动目标和玩法
2. 产品判断系统能力和可复用功能
3. UI 确认页面结构和资源规范
4. 运营设计查规范、做素材、等反馈
5. 开发重新理解需求、视觉、配置和边界

页面右侧：核心成本

- 活动策划不了解完整产品能力
- 产品经理反复承担功能咨询角色
- UI 资源规范分散在文档中
- 运营设计要在多个文档和原型之间切换
- 开发接入时需要重新理解大量上下文

页面结论：

- 问题不是“谁不够努力”，而是上下文没有在项目里连续流动。

### 宣讲逐字稿

我们先看现在为什么需要 OneFlow。

一次活动从提出想法到上线，中间会经过很多角色。策划最清楚业务目标和活动玩法，但通常不清楚当前活动系统到底有哪些能力、哪些功能可以复用、哪些地方需要开发。产品经理熟悉系统能力，但每次活动都需要介入判断方案是否可落地。UI 设计师掌握页面结构和资源规范，但这些规范往往分散在文档里。运营设计师真正需要的是完整原型、资源清单和规范说明，但现实中经常要在原型、配置后台、规范文档和沟通记录之间来回切换。

到了开发环节，开发者又需要重新理解策划意图、原型结构、资源要求、视觉还原和配置边界。

所以这里的核心问题不是某一个角色不够高效，而是活动上下文没有在一个项目里连续流动。每个活动都要重新问、重新找、重新解释、重新确认。OneFlow 要解决的就是这个问题。

## 第 3 页：过去的典型活动交付流程

### 宣讲目的

把大家熟悉但没有系统化表达的旧流程具象化，让后续的 OneFlow 新流程有对比基础。

### PPT 页面内容

页面标题：

- 过去：活动原型依赖多角色手工拼齐

流程图内容：

1. 活动策划提出目标、玩法和文案方向
2. 产品经理判断可用功能、组合方式和是否需要新增能力
3. 产品经理咨询其他项目组或历史能力
4. UI 设计师确认页面结构、资源尺寸、状态和规范
5. 策划、产品、UI 共同补齐活动原型
6. 运营设计师查规范、做素材、输出切图
7. 业务方配置后台或开发者接入新功能

页面强调框：

- 原型不是自然生成的，而是靠多角色把知识临时拼在一起。
- 资源规范不是跟活动绑定的，而是分散在独立文档中。
- 开发上下文不是项目自带的，而是靠沟通补齐的。

### 宣讲逐字稿

过去一次活动原型的形成，实际上非常依赖多角色手工拼齐。

策划先提出活动目标、玩法和文案方向。产品经理再判断这个目标能不能用当前系统已有功能实现，如果不能实现，还要判断是否需要新增能力。很多时候，某个产品经理只熟悉自己做过的功能，如果涉及其他项目组能力，还需要继续咨询其他产品同事。

方案大致确定后，还要找 UI 设计师确认页面结构和资源规范，比如某个模块需要哪些图片、尺寸是多少、有哪些状态、哪些地方可以配置。运营设计师拿到原型之后，还要继续查规范、做素材、输出切图，或者等待进一步确认。

这个流程里，原型不是自然从项目里长出来的，而是策划、产品和 UI 把各自掌握的知识临时拼到一起。资源规范也不是围绕本次活动组织的，而是散落在很多通用文档里。开发接入时需要的上下文，也不是项目天然自带的，而是靠沟通一点点补齐。

## 第 4 页：OneFlow 的目标工作方式

### 宣讲目的

让听众看到 OneFlow 想要形成的新协作方式：模板和知识先沉淀，策划和运营设计自助推进，开发只在必要时精准接入。

### PPT 页面内容

页面标题：

- 未来：让活动上下文在同一个项目里流动

主流程：

1. 产品和 UI 维护活动模板、功能知识、页面结构、资源规范
2. 策划输入活动目标，AI 给出可落地方案建议
3. AI 生成专业、可交互、带配置资源清单的活动原型
4. 运营设计查看配置项和资源规范，上传素材并实时预览
5. UI 或运营设计通过 Vibe Coding 产出高保真页面代码
6. 开发通过 CLI 获取项目代码、配置协议和完整上下文
7. 上线后把新增经验沉淀回模板和知识库

页面结论：

- 从“每次活动都靠人补上下文”
- 变成“模板、规范、原型、资源、预览和代码上下文在项目内连续沉淀”

### 宣讲逐字稿

OneFlow 想要形成的目标工作方式，是让活动上下文在同一个项目里流动。

产品经理和 UI 设计师不再只是每次活动的临时支持者，而是先把活动模板、功能知识、页面结构和资源规范维护好。策划提出活动目标之后，AI 可以读取这些模板和知识，给出更符合系统能力和业务目标的方案建议。

接着，AI 帮助策划生成专业、可交互的活动原型。这个原型不只是页面草图，它还应该包含正式文案、功能页面、配置项、资源清单和每个资源的规范。

运营设计师进入项目后，不需要再打开很多分散文档，而是直接看到本次活动有哪些配置项、每个配置项需要什么素材、规范是什么，并且上传资源后可以实时预览效果。

如果需要更高保真的视觉实现，UI 设计师或运营设计师可以通过 Vibe Coding 产出高保真页面代码。开发者需要接入时，通过 CLI 获取项目代码、配置协议、高保真页面代码和上下文。

这就是 OneFlow 的核心变化：从每次活动都靠人补上下文，变成上下文在项目里连续沉淀和流转。

## 第 5 页：角色变化总览

### 宣讲目的

降低听众对“AI 替代角色”的误解，让每类角色知道自己的工作重心如何变化，以及 OneFlow 对自己有什么直接价值。

### PPT 页面内容

页面标题：

- OneFlow 不是替代角色，而是改变每个角色的发力点

角色卡片：

- 活动策划：从提出需求、等待支持 -> 借助 AI 生成可落地方案和可交互原型
- 产品经理：从每个活动的功能咨询顾问 -> 活动模板、功能知识、能力边界和验收规则维护者
- UI 设计师：从逐个活动确认资源规范 -> 模板视觉规范、页面结构和高保真代码能力维护者
- 运营设计师：从查分散规范、做素材、等反馈 -> 查看配置项、上传资源、实时预览并影响最终页面效果
- 开发者：从重新理解需求和视觉 -> 读取项目代码、配置协议、高保真页面代码和上下文

页面底部：

- 人的经验不会消失，而是变成 AI 和项目都能读取的资产。

### 宣讲逐字稿

这里需要特别强调一点：OneFlow 不是要替代某个角色，而是改变每个角色的发力点。

对活动策划来说，变化是从提出需求、等待产品和 UI 支持，变成可以借助 AI 生成更可落地的活动方案和可交互原型。

对产品经理来说，变化是从每个活动都做功能咨询顾问，变成维护活动模板、功能知识、能力边界和验收规则。也就是说，产品经验不再只存在于人的解释里，而是沉淀到系统里。

对 UI 设计师来说，变化是从逐个活动确认资源规范，变成维护模板视觉规范、页面结构、资源标准和高保真页面代码能力。

对运营设计师来说，变化是不用再拿着不完整原型和分散文档做设计，而是在 OneFlow 里看到完整配置项和资源规范，上传素材后直接预览效果。必要时，运营设计师也可以通过 Vibe Coding 更直接地影响最终页面效果。

对开发者来说，变化是不用从零重新理解需求、标注、切图和口头上下文，而是直接读取项目代码、配置协议、高保真页面代码和完整上下文。

人的经验不会消失，而是变成 AI 和项目都能读取的资产。

---

## 第 6 页：中心产物是可交互活动原型

### 宣讲目的

说明为什么 OneFlow 的中心不是文档、聊天记录或后台配置，而是可交互活动原型。让大家理解原型在协作中的承载范围。

### PPT 页面内容

页面标题：

- 可交互活动原型：所有角色的共同语言

原型应该包含：

- 所有活动页面
- 正式文案
- 页面功能状态
- 页面配置项
- 本次活动资源清单
- 每个资源的尺寸、格式、状态和使用规范
- 每个配置项的备注和使用说明
- 上传真实资源后的预览效果

页面视觉建议：

- 中间放“可交互活动原型”
- 四周连接：策划方案、产品能力、UI 规范、运营资源、开发上下文

页面结论：

- 原型不是草图，而是活动交付的执行说明书。

### 宣讲逐字稿

OneFlow 里面最重要的中心产物，是可交互活动原型。

这里的原型不是我们过去理解的“差不多画一下页面结构”。它应该是一份活动交付的执行说明书。它要包含所有活动页面、正式文案、页面功能状态、页面配置项、本次活动资源清单，以及每个资源的尺寸、格式、状态和使用规范。

对策划来说，它是活动方案的可视化表达。对产品来说，它是系统能力和活动玩法的组合结果。对 UI 来说，它承载页面结构和资源规范。对运营设计来说，它告诉我们到底要做哪些素材、每个素材如何使用、上传后效果如何。对开发来说，它是理解页面结构、配置协议和视觉要求的重要上下文。

所以 OneFlow 不是只生成一张原型图，而是让原型成为所有角色都能围绕它协作的共同语言。

## 第 7 页：模板和知识库如何发挥作用

### 宣讲目的

解释 AI 输出质量的来源：不是凭空生成，而是读取产品和 UI 持续沉淀的模板、功能知识、资源规范、配置规则和历史活动经验。

### PPT 页面内容

页面标题：

- AI 的质量来自被沉淀的模板和知识

左侧：沉淀内容

- 活动类型模板
- 系统功能说明
- 可复用功能边界
- 页面结构和组件规则
- UI 资源尺寸、格式和状态规范
- 配置项备注和验收规则
- 历史活动经验

右侧：AI 能做什么

- 判断活动方案是否符合系统能力
- 推荐可复用玩法和页面结构
- 生成带配置项和资源清单的原型
- 解释资源规范和使用方式
- 帮助整理验收和开发上下文

页面结论：

- AI 不是替代经验，而是放大已经沉淀下来的经验。

### 宣讲逐字稿

很多人听到 AI 生成方案或原型，会担心它是不是凭空编出来的。OneFlow 的思路不是让 AI 凭空发挥，而是让 AI 读取已经沉淀好的模板和知识。

这些知识包括活动类型模板、系统功能说明、可复用功能边界、页面结构、组件规则、UI 资源尺寸、格式和状态规范、配置项备注、验收规则以及历史活动经验。

当这些内容被沉淀在项目和知识库里之后，AI 才能更准确地判断一个活动方案是否符合系统能力，推荐哪些玩法可以复用，生成哪些页面和配置项，告诉运营设计师每个素材应该怎么做，也可以帮助开发者整理实现上下文。

所以 AI 在这里不是替代经验，而是放大已经沉淀下来的经验。产品和 UI 沉淀得越清楚，策划和运营设计自助推进的质量就越高。

---

## 第 8 页：项目内资源规范取代分散文档查找

### 宣讲目的

突出运营设计师的直接收益：不再围绕大量钉钉规范文档查找信息，而是在项目中按本次活动结构查看配置项、资源要求和预览效果。

### PPT 页面内容

页面标题：

- 资源规范从“文档库”变成“项目内配置上下文”

过去：

- 规范按功能分散在多个文档中
- 运营设计需要打开多份文档查尺寸和状态
- UI 需要额外整理本次活动配置项列表
- 规范说明和真实页面效果分离

OneFlow：

- 规范按活动页面和配置项组织
- 每个配置项有备注、尺寸、格式、状态和使用说明
- 上传资源后可以在真实页面中预览
- AI 可以直接读取并解释资源规范

页面结论：

- 运营设计要看的不是一堆通用规范，而是本次活动要制作什么、怎么做、效果怎样。

### 宣讲逐字稿

对运营设计师来说，一个非常直接的变化是资源规范的使用方式。

过去很多 UI 资源规范沉淀在钉钉文档里，而且这些文档通常是按具体功能或模块维护的。这个方式对长期维护是有价值的，但对一次具体活动来说，运营设计师真正关心的是：本次活动到底有哪些页面、有哪些配置项、需要制作哪些素材、每个素材尺寸是多少、有哪些状态、上传之后页面效果是什么。

如果规范分散在很多文档里，运营设计师就需要打开很多文档查找信息。UI 也经常需要额外整理一份配置项列表，告诉大家每个配置项对应哪篇规范。

OneFlow 希望把这些资源规范放到项目内，按活动页面和配置项组织。每个配置项可以有备注、尺寸、格式、状态和使用说明。上传资源之后，可以直接在真实页面里预览效果。AI 也可以读取这些规范，回答运营设计师的问题。

这样规范就不再只是静态文档，而是本次活动的配置上下文。

## 第 9 页：配置、资源上传与实时预览

### 宣讲目的

让听众理解运营设计师在 OneFlow 中的核心操作闭环：查看配置项、上传资源、实时预览、调整并验收。

### PPT 页面内容

页面标题：

- 从“交付切图”到“上传资源并实时验证效果”

页面流程：

1. 查看本次活动全部页面和配置项
2. 查看每个资源的规范说明
3. 上传图片、动效、音效等素材
4. 在真实页面中实时预览
5. 根据预览调整素材或配置
6. 完成配置验收

页面展示建议：

- 左侧放“配置项清单”
- 中间放“资源上传区域”
- 右侧放“实时预览页面”

页面结论：

- 设计交付不只停在素材文件，而是可以直接验证页面效果。

### 宣讲逐字稿

OneFlow 对运营设计师的另一个关键价值，是把素材交付变成可以实时验证的闭环。

过去很多时候，运营设计师完成素材之后，需要把切图交给业务方或配置同学，再到后台配置，之后才能看到页面效果。如果发现尺寸、状态或视觉效果不符合预期，还要再来回调整。

在 OneFlow 里，运营设计师可以先看到本次活动全部页面和配置项，查看每个资源的规范说明，然后上传图片、动效、音效等素材。上传之后，不是等别人再配置，而是可以在真实页面中实时预览。

如果预览效果不理想，可以马上调整素材或配置，再次验证。这样设计交付就不只停在素材文件，而是可以直接确认这些素材放进活动页面后是否真的可用、是否符合预期。

过去设计交付：

AI 时代的补充交付：

### 宣讲逐字稿

过去设计交付主要依赖 Figma、切图、标注和口头解释。这些方式仍然有价值，但在开发者也大量使用 AI 的情况下，它们并不一定是最适合 AI 读取的上下文。

高保真页面代码的价值在于，它本身就是可运行的页面表达。视觉层级、间距、状态、动效和交互，不只是停留在标注里，而是可以变成开发 AI 能直接读取的上下文。

这不是要求 UI 设计师或运营设计师变成传统开发者。它更像是一种新的设计表达方式：设计侧可以通过 Vibe Coding 把视觉判断转成可运行页面。开发者再结合业务代码、配置协议和系统边界，完成最终实现。

所以 Vibe Coding 的重点不是炫技，而是减少视觉还原和上下文解释成本。

这就是 OneFlow 最重要的闭环：一次活动，不只是完成一次交付，还会反哺下一次活动。
