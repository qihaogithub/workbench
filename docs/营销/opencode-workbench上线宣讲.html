ONEFLOW LAUNCH NARRATIVE

OneFlow 上线宣讲

不是多一个工具入口，而是一次活动协作方式升级。

让活动策划、产品经理、UI 设计师、运营设计师、开发者在同一个平台、同一个工作流里协作。

内部宣讲大纲
目标

统一 OneFlow 的定位、角色变化和试点节奏

对象

策划、产品、UI、运营设计、开发与业务同事

主线

从人肉咨询，走向模板、知识、原型、预览和代码上下文

固定 16:9 HTML 演示稿，可直接浏览器播放。

核心主张

OneFlow 上线的不是工具，是协作方式。

过去的问题不是某个角色不够努力，而是功能知识、UI 规范、活动原型、资源清单和开发上下文散落在不同工具与人身上。

全员共识
01

产品和 UI 的经验要沉淀为模板、知识和规范。

02

策划和运营设计要能基于 AI 自助推进具体活动。

03

开发只在必要时带着完整上下文精准接入。

01 / 定位
过去流程

一次活动要靠多轮咨询才能拼出可执行原型。

协作链路

活动策划

提出活动目标、玩法、节奏和业务诉求。

产品经理

判断现有活动系统能力，必要时评估新开发。

UI 设计师

确认功能页面资源规范与体验标准。

运营设计师

基于原型、规范和清单制作图片、动效、音效。

业务 / 开发

不开发就配置后台，需要新能力则继续对接开发。

策划懂需求，但不熟功能边界。
产品像咨询顾问，跨组还要再咨询。
UI 规范不和原型、配置项绑定。
开发重新理解需求、视觉和配置协议。
02 / 当前协作痛点
痛点不是单点

运营设计真正需要的是完整原型，但完整原型天然依赖三方协作。

原型缺口
A

需求和功能分离

活动策划最了解业务目标，却不清楚活动系统已有功能；产品经理了解功能，却不是需求定义者。

B

原型质量不稳定

运营设计期望看到所有页面、正式文案、资源清单和规范，但产品和 UI 不一定每次都能给到足够支持。

C

资源规范分散

规范按功能维护，不按本次活动结构组织；一次活动可能需要打开几十个钉钉文档。

02 / 当前协作痛点
未来图景

把“每次都靠人补上下文”，变成“模板和知识先沉淀”。

OneFlow Loop

一个项目承载一条活动工作流

模板与知识

产品维护功能边界、活动类型模板和验收规则。

资源规范

UI 在模板项目中维护页面结构、配置项备注和体验标准。

AI 可交互原型

策划借助 AI 生成专业原型、页面、文案、资源清单。

上传与预览

运营设计查看配置项、上传资源并实时预览效果。

CLI 接入

开发获取项目代码、配置协议和高保真页面上下文。

03 / 未来协作图景
角色变化

OneFlow 不替代角色，而是改变每个人的工作重心。

全员协作
策

活动策划

从：提出需求，等待产品和 UI 支持

到：借助 AI 生成可落地方案和专业原型

产

产品经理

从：具体活动功能咨询顾问

到：模板、能力边界、验收规则维护者

UI

UI 设计师

从：每次确认资源规范

到：模板视觉规范和高保真页面能力维护者

运

运营设计师

从：拿不完整原型和分散规范设计

到：查看配置项、上传资源、实时预览

Dev

开发者

从：重新理解需求、标注、切图

到：用 CLI 和 AI 精确读取上下文

04 / 角色变化
中心产物

可交互活动原型，是所有角色共同阅读的项目事实。

Interactive Prototype
活动原型 / 配置预览OneFlow
资源上传后实时预览Live

所有活动页面

原型不只是首页草图，而是完整页面地图。

正式文案

业务文案进入原型，减少后续反复确认。

配置项

每个页面有哪些可配置资源，一眼可见。

资源清单

尺寸、格式、状态、使用规范随页面沉淀。

配置备注

每个配置项都能说明作用、限制和关联规范。

真实预览

上传图片、动效、音效后立即看到活动效果。

05 / 中心产物
规范沉淀

模板项目本身，就是下一代 UI 资源规范。

Spec as Project

过去：钉钉文档库

按功能维护，长期膨胀，活动视角不清晰。

运营设计要做资源

需要 UI 额外整理配置项列表和规范链接。

结果：查阅成本高

一次活动可能打开几十份文档，还看不到真实预览。

现在：规范进入模板项目

按活动结构组织，绑定页面、配置项、备注和预览。

配置项备注

说明用途、边界、状态和制作注意事项。

规范说明

尺寸、格式、切图、动效、音效要求。

交互预览

不是静态文档，而是可操作、可理解的效果。

AI 可读取

AI 可以基于项目知识回答资源制作和配置问题。

06 / 资源规范
AI 定位

AI 是全员的知识顾问，不是替代所有人的黑盒。

Knowledge Advisor
策

辅助策划

读取模板、历史活动和功能边界，给出符合运营目标的方案建议。

产

替代常规咨询

基于产品维护的能力说明和配置规则，回答活动能否落地。

UI

解释资源规范

读取模板内规范、配置项备注和预览状态，减少重复答疑。

Dev

增强开发上下文

读取高保真页面代码、配置协议和项目结构，更精确还原视觉与状态。

AI 的质量来自模板、知识库、资源规范、配置协议和真实预览约束。

07 / AI 的定位
Vibe Coding

AI 时代，设计交付可以从“看图还原”升级为“读取高保真页面代码”。

Design to Code
过去

Figma、切图、标注和口头解释。

开发者需要理解视觉意图、间距、状态、切图和资源边界。

运营设计决定视觉时，也很难直接影响最终页面实现。

AI 参与开发后，图片和标注不是最强提示词。

现在

高保真页面代码成为开发 AI 的可执行上下文。

UI 可以用 Vibe Coding 表达视觉、间距、状态和动效。

运营设计也可以加入，直接影响最终页面效果。

开发者让自己的 AI 读取高保真代码，再结合业务代码和配置协议实现。

08 / 设计交付方式
核心工作流

一条完整链路：从活动目标，到原型、资源、预览、开发、复用。

End-to-End
01

维护模板

产品/UI 沉淀活动模板、功能知识、页面结构、资源规范。

02

提出目标

策划输入目标、玩法、节奏、投放和文案方向。

03

AI 生成原型

读取知识库，形成可交互原型和配置资源清单。

04

设计资源

运营设计查看配置项和规范，上传图片、动效、音效。

05

实时预览

OneFlow 根据配置项和上传资源立即展示效果。

06

开发接入

需要新功能时，开发用 CLI 获取项目代码和上下文。

上线后，新增能力、页面结构、配置项、资源规范和高保真页面经验继续反哺模板与知识库。

09 / 核心工作流
能力边界

先讲平台能承载什么，不把宣讲变成系统菜单演示。

Scope

活动模板

页面结构、功能组合、配置项、备注和资源规范。

项目内规范文档

把分散钉钉规范转为按活动结构组织的项目知识。

知识库

活动系统功能、历史活动、配置规则和业务说明。

活动原型图

承载页面、文案、功能、配置资源列表和规范。

AI 对话

辅助策划生成原型，辅助运营设计理解配置项。

Vibe Coding

支持 UI 或运营设计产出可运行高保真页面代码。

资源上传与预览

素材上传后直接看到活动页面效果。

CLI 开发接入

让开发者获取项目代码、配置协议和上下文。

10 / 平台能力边界
试点验证

选择一个真实、范围可控、多角色协作的活动。

Pilot
推荐试点

不是证明 AI 能不能生成页面，而是验证协作闭环能不能跑通。

目标是减少产品和 UI 的具体活动跟进成本，让策划与运营设计自助推进，让开发在必要时精准接入。

策划能否借助 AI 自助生成可交互原型。
原型是否完整呈现页面、文案、配置项、资源清单和规范。
UI 资源规范能否迁移为项目内可备注、可预览、AI 可读取的规范。
运营设计能否基于配置项上传资源并实时预览。
Vibe Coding 高保真页面代码能否成为开发 AI 上下文。
开发者能否通过 CLI 获取代码、协议和项目上下文。

扩展场景

逐步接入更多活动、运营模块和业务系统。

12 / 上线节奏
结束收束

第一步，不是欢迎体验，而是一起跑完一条真实工作流。

从策划方案，到 AI 可交互原型、配置资源清单、设计资源上传、Vibe Coding 高保真页面代码、实时预览、配置验收和开发接入。

2

3
