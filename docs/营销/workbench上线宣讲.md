# OneFlow 上线宣讲 PPT 逐页规划

> 用途：上线前内部宣讲 PPT 制作与现场讲解。本文按 PPT 每一页组织，每页包含“宣讲目的”“PPT 页面内容”“宣讲逐字稿”。
> 宣讲对象：活动策划、产品经理、UI 设计师、运营设计师、开发者，以及后续需要使用活动页面和活动配置能力的业务同事。
> 建议时长：35-45 分钟。

## 第 1 页：OneFlow 是什么

### 宣讲目的

让听众在开场就建立清晰定位：OneFlow 的核心不是“多了一个 AI 聊天框”，而是让 AI 接管重复执行、整合分散知识、承接活动系统客服式问答，并围绕活动策划、原型、配置、资源、预览和开发接入形成一套新工作流。

### PPT 页面内容

页面标题：

- OneFlow：面向活动交付的 AI 工作流平台

页面主文案：

- 让活动从“策划想法”连续流转到“可交互原型、配置资源、实时预览和开发接入”
- 把产品功能知识、活动模板、UI 资源规范和历史经验沉淀为 AI 可读取的项目上下文
- AI 接管过去大量人工体力活：方案整理、原型绘制、配置清单生成、规范查找和上下文整理
- AI 整合分散在产品、UI、运营设计和开发之间的经验，让不同角色不再靠会议和文档反复对齐
- AI 承接活动系统的大量常规咨询，让策划和运营设计可以自助推进大部分不需要开发的运营活动

页面视觉建议：

- 用一条横向流程表示：活动目标 -> AI 方案 -> 可交互原型 -> 配置资源 -> 实时预览 -> 开发接入 -> 复用沉淀。
- 页面底部放一句定位语：不是多一个入口，而是把 AI 变成活动交付的执行者、知识整合者和自助顾问。

### 宣讲逐字稿

大家好，今天这次宣讲主要介绍 OneFlow。

我先用一句话说清楚它是什么：OneFlow 是面向活动交付的 AI 工作流平台。它不是单独的 AI 聊天入口，也不是传统意义上的活动后台，而是把活动从策划目标、方案生成、可交互原型、资源配置、实时预览，到必要时的开发接入，放到同一个项目工作流里连续推进。

它背后的核心变化是 AI 的角色变了。过去很多体力活必须靠人完成，比如整理方案、画原型、列配置项、查规范、把会议结论整理成文档。现在这些工作 AI 可以完成，而且只要给到足够的模板和知识，AI 可以比人工临时拼出来更完整、更稳定。

还有一类变化，是知识整合。过去经验分散在产品经理、UI 设计师、运营设计师、开发者以及各种文档里，信息对齐需要会议、文档和反复确认。OneFlow 希望把这些经验沉淀为活动模板、功能知识、资源规范和项目上下文，让 AI 直接读取和整合。

第三类变化，是自助咨询。过去在大量不需要开发的运营活动里，产品经理和 UI 设计师经常像活动系统的售后客服，回答“这个功能能不能做”“这个资源尺寸是多少”“这个配置项怎么用”。OneFlow 要让 AI 承接这类常规客服工作，让策划和运营设计可以自助推进。

所以今天我不会只讲有哪些按钮和页面，而是重点讲 OneFlow 会怎样改变我们做活动的协作方式。

## 第 2 页：在线、低门槛、面向团队协作的 Agent 平台

### 宣讲目的

说明 OneFlow 的平台级价值：它不是要求所有岗位去安装 Claude Code、Codex、Trae、OpenClaw（龙虾）等编程工具，也不是要求大家学习 Git，而是把接近前沿的 Agent 能力放进一个在线、项目化、可协作的平台里，让所有相关岗位都能参与 Vibe Coding 和活动交付。

### PPT 页面内容

页面标题：

- 不用安装编程工具，也不用学习 Git，就能参与 AI 创作和协作

传统 Agent 工具的特点：

- Claude Code、Codex、Trae、OpenClaw 等工具更偏个人提效
- 需要本地环境、账号配置、命令行或 IDE 使用习惯
- 通常默认使用 Git、分支、提交、合并等开发协作方式
- 对非开发岗位学习门槛高
- 多人围绕同一个活动项目协作时，上下文和产物不容易自然共享

OneFlow 的平台价值：

- 在线使用，不要求下载安装本地编程工具
- 用项目承载对话、页面、配置、资源、预览和高保真代码
- 不要求策划、产品、UI、运营设计先学习 Git
- 多角色可以在同一个项目里查看上下文、产物和版本
- 把个人 Agent 能力变成团队可协作的活动交付能力

页面结论：

- OneFlow 不是把个人编程工具搬进浏览器，而是把 Agent 能力产品化成团队协作平台。

### 宣讲逐字稿

在开场之后，我想先讲 OneFlow 一个非常核心的平台价值。

现在很多前沿 Agent 工具，比如 Claude Code、Codex、Trae、OpenClaw 这类产品，个人使用时确实能大幅提效。但它们大多是围绕个人开发者工作流设计的。使用者通常需要安装工具、配置环境、理解命令行或 IDE，也需要理解 Git、分支、提交、合并这些开发协作方式。

这对开发者来说可以接受，但对活动策划、产品经理、UI 设计师、运营设计师来说，门槛就很高。更关键的是，这些工具解决的是个人提效，不天然解决团队协作。一个人用 Agent 做得很快，不代表团队能围绕同一个活动项目顺畅共享上下文、页面、配置、资源、预览和版本。

OneFlow 的价值，是把接近前沿的 Agent 能力放到一个在线平台里。大家不需要下载 Claude Code、Codex、Trae 或其他编程工具，也不需要先学习 Git，就可以在浏览器里围绕同一个项目使用 AI 能力。

这里的重点是团队协作。策划的活动目标、AI 生成的原型、产品维护的功能边界、UI 维护的资源规范、运营设计上传的素材、Vibe Coding 产出的高保真页面代码、开发接入需要的上下文，都沉淀在同一个项目里。

所以 OneFlow 不是把个人编程工具搬进浏览器，而是把 Agent 能力产品化成团队协作平台。它让不懂编程工具、不熟悉 Git 的岗位，也能进入 AI 创作和活动交付链路。

## 第 3 页：为什么需要 OneFlow

### 宣讲目的

建立共识：当前活动交付的主要问题不是某个角色效率低，而是大量重复劳动仍由人工完成，功能知识、UI 规范、资源要求、原型和开发上下文又分散在不同角色中，导致每个活动都需要反复沟通和补齐信息。

### PPT 页面内容

页面标题：

- 现在的问题：每个活动都在重新做体力活、重新补上下文

页面左侧：当前协作链路

1. 策划提出活动目标和玩法
2. 人工整理方案、绘制原型、列页面和配置项
3. 产品判断系统能力和可复用功能
4. UI 确认页面结构和资源规范
5. 运营设计查规范、做素材、等反馈
6. 开发重新理解需求、视觉、配置和边界

页面右侧：核心成本

- 原型绘制、配置清单整理、文档同步等体力活占用大量时间
- 活动策划不了解完整产品能力
- 产品经理反复承担功能咨询角色
- UI 资源规范分散在文档中
- 运营设计要在多个文档和原型之间切换
- 开发接入时需要重新理解大量上下文

页面结论：

- 问题不是“谁不够努力”，而是重复执行靠人、知识整合靠会、常规咨询靠产品和 UI 人肉响应。

### 宣讲逐字稿

我们先看现在为什么需要 OneFlow。

一次活动从提出想法到上线，中间会经过很多角色。这里面有三类成本过去都很重。

第一类是重复执行成本。比如整理活动方案、绘制原型图、列出页面和配置项、整理资源清单，这些工作并不一定都需要人的创造性判断，但过去必须人工完成。

第二类是知识整合成本。策划最清楚业务目标和活动玩法，但通常不清楚当前活动系统到底有哪些能力、哪些功能可以复用、哪些地方需要开发。产品经理熟悉系统能力，但每次活动都需要介入判断方案是否可落地。UI 设计师掌握页面结构和资源规范，但这些规范往往分散在文档里。运营设计师真正需要的是完整原型、资源清单和规范说明，但现实中经常要在原型、配置后台、规范文档和沟通记录之间来回切换。

第三类是客服式支持成本。在大量不需要开发的运营活动里，产品经理和 UI 设计师做的不是重新设计系统，而是在回答系统能力、功能边界、资源规范和配置方式。这类工作很像活动系统的售后客服。

到了开发环节，开发者又需要重新理解策划意图、原型结构、资源要求、视觉还原和配置边界。

所以这里的核心问题不是某一个角色不够高效，而是重复执行靠人、知识整合靠会、常规咨询靠产品和 UI 人肉响应。OneFlow 要解决的，就是把这些工作尽可能交给 AI 承接。

## 第 4 页：过去的典型活动交付流程

### 宣讲目的

把大家熟悉但没有系统化表达的旧流程具象化，让后续的 OneFlow 新流程有对比基础，并突出过去哪些环节其实适合由 AI 承接。

### PPT 页面内容

页面标题：

- 过去：体力活和咨询工作都依赖人工

流程图内容：

1. 活动策划提出目标、玩法和文案方向
2. 策划或产品手工整理方案、页面结构和原型草稿
3. 产品经理判断可用功能、组合方式和是否需要新增能力
4. 产品经理咨询其他项目组或历史能力
5. UI 设计师确认页面结构、资源尺寸、状态和规范
6. 策划、产品、UI 共同补齐活动原型和资源清单
7. 运营设计师查规范、做素材、输出切图
8. 业务方配置后台或开发者接入新功能

页面强调框：

- 原型绘制、配置清单、资源清单等体力活过去依赖人工。
- 产品和 UI 在大量活动中承担“活动系统客服”角色。
- 分散知识需要靠会议和文档临时对齐。
- 开发上下文不是项目自带的，而是靠沟通补齐。

### 宣讲逐字稿

过去一次活动原型的形成，实际上非常依赖人工。

策划先提出活动目标、玩法和文案方向。然后有人要把这些内容整理成方案、页面结构和原型草稿。这个过程里有很多体力活，比如把活动流程画出来，把页面状态列出来，把配置项和资源清单整理出来。过去这些事情只能靠人做。

接着，产品经理要判断这个目标能不能用当前系统已有功能实现，如果不能实现，还要判断是否需要新增能力。很多时候，某个产品经理只熟悉自己做过的功能，如果涉及其他项目组能力，还需要继续咨询其他产品同事。

方案大致确定后，还要找 UI 设计师确认页面结构和资源规范，比如某个模块需要哪些图片、尺寸是多少、有哪些状态、哪些地方可以配置。运营设计师拿到原型之后，还要继续查规范、做素材、输出切图，或者等待进一步确认。

所以过去产品经理和 UI 设计师在很多不需要开发的运营活动里，实际上承担了大量客服式工作：解释系统能做什么、资源怎么做、配置项怎么用。这个流程里，原型不是自然从项目里长出来的，而是策划、产品和 UI 把各自掌握的知识临时拼到一起。OneFlow 要改变的，就是让这些重复执行、知识整合和常规问答交给 AI 承接。

## 第 5 页：OneFlow 的目标工作方式

### 宣讲目的

让听众看到 OneFlow 想要形成的新协作方式：AI 承接重复执行、知识整合和常规咨询；模板和知识先沉淀，策划和运营设计自助推进，开发只在必要时精准接入。

### PPT 页面内容

页面标题：

- 未来：让 AI 承接重复工作，让人维护判断和标准

主流程：

1. 产品和 UI 维护活动模板、功能知识、页面结构、资源规范
2. 策划输入活动目标，AI 整合产品能力、UI 规范和历史经验
3. AI 给出可落地方案建议，自动生成专业、可交互原型
4. AI 生成页面结构、配置项、资源清单和规范说明
5. 运营设计向 AI 查询配置和资源规范，上传素材并实时预览
6. UI 或运营设计通过 Vibe Coding 产出高保真页面代码
7. 开发通过 CLI 获取项目代码、配置协议和完整上下文
8. 上线后把新增经验沉淀回模板和知识库

页面结论：

- 从“人画原型、人查规范、人回答常规问题”
- 变成“AI 生成原型、整合知识、承接活动系统自助咨询”

### 宣讲逐字稿

OneFlow 想要形成的目标工作方式，是让 AI 承接那些过去必须由人完成、但本质上可以被知识和规则驱动的工作。

产品经理和 UI 设计师不再只是每次活动的临时支持者，而是先把活动模板、功能知识、页面结构和资源规范维护好。策划提出活动目标之后，AI 可以读取这些模板和知识，把分散在不同岗位和文档里的信息整合起来，给出更符合系统能力和业务目标的方案建议。

接着，AI 不只是回答问题，还会直接完成过去很多体力活：生成专业、可交互的活动原型，整理页面结构、正式文案、功能页面、配置项、资源清单和每个资源的规范。

运营设计师进入项目后，不需要再打开很多分散文档，也不需要每个问题都找 UI 确认，而是可以直接向 AI 查询本次活动有哪些配置项、每个配置项需要什么素材、规范是什么，并且上传资源后可以实时预览效果。

如果需要更高保真的视觉实现，UI 设计师或运营设计师可以通过 Vibe Coding 产出高保真页面代码。开发者需要接入时，通过 CLI 获取项目代码、配置协议、高保真页面代码和上下文。

这就是 OneFlow 的核心变化：从人画原型、人查规范、人回答常规问题，变成 AI 生成原型、整合知识、承接活动系统自助咨询。

## 第 6 页：角色变化总览

### 宣讲目的

降低听众对“AI 替代角色”的误解，让每类角色知道 AI 接管哪些重复工作，自己需要转向哪些更高价值的判断、标准和沉淀工作。

### PPT 页面内容

页面标题：

- OneFlow 不是替代角色，而是把人从重复执行和客服式支持中释放出来

角色卡片：

- 活动策划：从提出需求、等待支持 -> 借助 AI 生成可落地方案、活动原型和配置资源清单
- 产品经理：从活动系统功能客服 -> 活动模板、功能知识、能力边界和验收规则维护者
- UI 设计师：从资源规范客服 -> 模板视觉规范、页面结构、资源标准和高保真代码能力维护者
- 运营设计师：从查分散规范、反复确认 -> 向 AI 查询配置规范、上传资源、实时预览并影响最终页面效果
- 开发者：从重新理解需求和视觉 -> 读取项目代码、配置协议、高保真页面代码和上下文

页面底部：

- AI 承接执行和问答，人负责沉淀标准、判断边界和处理例外。

### 宣讲逐字稿

这里需要特别强调一点：OneFlow 不是要替代某个角色，而是把人从重复执行和客服式支持中释放出来。

对活动策划来说，变化是从提出需求、等待产品和 UI 支持，变成可以借助 AI 生成更可落地的活动方案、可交互原型、配置项和资源清单。

对产品经理来说，变化是从每个活动都做功能咨询顾问，变成维护活动模板、功能知识、能力边界和验收规则。过去很多“这个活动能不能做、应该用哪个功能、哪些地方需要开发”的问题，需要产品经理反复回答。未来这类常规客服工作应该由 AI 承接，产品经理重点维护 AI 可以依据的规则和边界。

对 UI 设计师来说，变化是从逐个活动确认资源规范，变成维护模板视觉规范、页面结构、资源标准和高保真页面代码能力。过去很多“这个图多大、这个状态怎么做、这个配置项对应哪个规范”的问题，需要 UI 反复解释。未来这些说明应该沉淀到项目里，由 AI 直接回答。

对运营设计师来说，变化是不用再拿着不完整原型和分散文档做设计，而是在 OneFlow 里看到完整配置项和资源规范，也可以直接问 AI：这个素材怎么做、上传到哪里、页面效果怎样。上传素材后还能直接预览效果。必要时，运营设计师也可以通过 Vibe Coding 更直接地影响最终页面效果。

对开发者来说，变化是不用从零重新理解需求、标注、切图和口头上下文，而是直接读取项目代码、配置协议、高保真页面代码和完整上下文。

人的经验不会消失，而是变成 AI 和项目都能读取的资产。AI 承接执行和问答，人负责沉淀标准、判断边界和处理例外。

## 第 7 页：中心产物是可交互活动原型

### 宣讲目的

说明为什么 OneFlow 的中心不是文档、聊天记录或后台配置，而是 AI 生成并持续完善的可交互活动原型。强调过去人工绘制原型很耗时，AI 现在可以承担这部分体力活，并因为能读取模板和规范而生成得更完整。

### PPT 页面内容

页面标题：

- AI 生成可交互活动原型：从体力活变成标准化产物

原型应该包含：

- 所有活动页面
- 正式文案
- 页面功能状态
- 页面配置项
- 本次活动资源清单
- 每个资源的尺寸、格式、状态和使用规范
- 每个配置项的备注和使用说明
- 上传真实资源后的预览效果

AI 带来的变化：

- 不再依赖人手工补齐页面、配置项和资源清单
- 不再把原型和资源规范分散在不同文档里
- AI 可以基于模板和知识生成更完整、更一致的原型

页面视觉建议：

- 中间放“AI 生成的可交互活动原型”
- 四周连接：策划方案、产品能力、UI 规范、运营资源、开发上下文

页面结论：

- 原型不再只是人工画出来的草图，而是 AI 整合知识后生成的活动执行说明书。

### 宣讲逐字稿

OneFlow 里面最重要的中心产物，是 AI 生成并持续完善的可交互活动原型。

过去画原型是一件非常典型的体力活。要把策划想法转成页面，要把活动流程画出来，要把每个页面状态列出来，还要补齐配置项和资源清单。这个过程很耗时间，而且完整度取决于当时参与的人是否记得足够多的信息。

现在 AI 可以承担这部分工作，而且只要它能读取活动模板、产品功能知识、UI 资源规范和历史活动经验，就不只是“帮忙画图”，而是可以生成更完整、更一致的原型。

这里的原型不是我们过去理解的“差不多画一下页面结构”。它应该是一份活动交付的执行说明书。它要包含所有活动页面、正式文案、页面功能状态、页面配置项、本次活动资源清单，以及每个资源的尺寸、格式、状态和使用规范。

对策划来说，它是活动方案的可视化表达。对产品来说，它是系统能力和活动玩法的组合结果。对 UI 来说，它承载页面结构和资源规范。对运营设计来说，它告诉我们到底要做哪些素材、每个素材如何使用、上传后效果如何。对开发来说，它是理解页面结构、配置协议和视觉要求的重要上下文。

所以 OneFlow 不是只让 AI 生成一张图，而是让 AI 把原型、配置项、资源规范和开发上下文整合成所有角色都能使用的共同语言。

如果规范分散在很多文档里，运营设计师就需要打开很多文档查找信息。UI 也经常需要额外整理一份配置项列表，告诉大家每个配置项对应哪篇规范。

OneFlow 希望把这些资源规范放到项目内，按活动页面和配置项组织。每个配置项可以有备注、尺寸、格式、状态和使用说明。上传资源之后，可以直接在真实页面里预览效果。AI 也可以读取这些规范，回答运营设计师的问题。

这样规范就不再只是静态文档，而是本次活动的配置上下文。

所以 Vibe Coding 的重点不是炫技，而是减少视觉还原和上下文解释成本。

这个试点最好是真实业务活动，不是纯演示样例。范围要可控，页面数量和配置项不要一开始就过大。同时，它要能覆盖多个角色协作，比如需要策划、产品、UI、运营设计和开发共同参与。活动本身也要有明确目标、玩法和上线时间。

我们希望通过这个试点验证几件事：策划能不能借助 AI 自助生成可交互原型；AI 能不能把产品能力、UI 规范和历史经验整合成可执行方案；原型能不能完整呈现页面、文案、配置项、资源清单和规范；AI 能不能承接常规功能咨询和资源规范问答；多岗位能不能不安装本地编程工具、不学习 Git，也能围绕同一个项目顺畅协作；运营设计能不能基于配置项上传资源并实时预览；产品和 UI 维护的模板与规范，能不能减少具体活动跟进成本；高保真页面代码能不能帮助开发者更准确地实现页面。

试点不是为了证明 AI 能不能生成页面，而是验证 AI 能不能真正承担执行、整合和客服三类工作，并让 OneFlow 跑通活动交付闭环。

### 宣讲逐字稿

这一页讲试点里每个角色具体怎么参与。

产品经理需要提供当前系统可复用功能、限制条件、活动模板和验收规则，同时把功能知识和能力边界维护到 OneFlow 可以读取的上下文里。这样 AI 才能承接“这个活动能不能做、应该用哪个功能”这类常规咨询。产品经理维护的是项目上下文和判断标准，不需要把协作强行搬到开发工具链里。

UI 设计师需要提供页面结构、资源规范、体验标准和模板视觉规则，并把资源规范说明维护到项目里。这样 AI 才能承接“这个资源怎么做、这个状态怎么处理”这类常规规范问题。必要时，UI 设计师也可以通过 Vibe Coding 产出高保真页面代码，让复杂视觉和页面状态表达得更准确。这个过程是在线协作，不要求 UI 设计师先成为本地开发工具用户。

开发者在需要新功能时接入，通过 CLI 获取项目代码、配置协议和上下文，完成新增能力实现和发布前验收。

这样每个角色的参与方式都是具体的，而不是一句“大家一起协作”。

## 第 17 页：上线节奏

### 宣讲目的

### 宣讲逐字稿

最后讲一下上线节奏。

第一步是内部宣讲，先统一 OneFlow 的定位、协作方式和试点目标。第二步是选择试点，确定一个真实、范围可控的活动。第三步是跑通闭环，也就是完成 AI 方案、AI 原型、在线协作、配置资源、实时预览、必要时的开发接入和最终验收。

跑完之后，我们要复盘流程，记录 AI 接管了哪些体力活、整合了哪些分散知识、承接了哪些常规咨询，在线协作是否降低了工具和 Git 门槛，以及哪些环节仍然卡住。接着把产品功能、页面结构、资源规范、高保真页面经验和可复用配置项沉淀回模板和知识库。

最后再逐步扩展到更多活动、运营模块和业务系统。

这里的推进原则是先跑通，再扩展。不要一开始追求覆盖所有场景，而是用一个真实活动验证 OneFlow 的完整闭环。

### 宣讲逐字稿

最后我们回到这次宣讲的主题。

OneFlow 不是多一个工具入口。它希望把产品能力、UI 规范、活动原型、配置资源、实时预览和开发上下文放进同一个项目，让这些上下文可以被人读取，也可以被 AI 读取。

更重要的是，它把 AI 放到活动交付的核心位置。过去原型绘制、配置清单、资源规范查找和上下文整理这类体力活靠人完成，现在 AI 可以承接。过去产品能力、UI 规范、运营经验和开发边界分散在不同同事和文档里，现在 AI 可以整合。过去产品经理和 UI 设计师在大量不需要开发的运营活动中承担售后客服式支持，现在这些常规咨询可以由 AI 承接。

同时，OneFlow 还提供了一个在线、低门槛、面向团队协作的 Vibe Coding 和 Agent 平台。它不是要求所有岗位去下载 Claude Code、Codex、Trae 或其他编程工具，也不是要求策划、产品、UI 和运营设计先学习 Git。它让大家在同一个在线项目里共享上下文、产物、预览和版本。

它让策划可以借助 AI 自助生成专业、可交互的活动原型；让运营设计师可以直接查询配置项和资源规范、上传资源并验证页面效果；让设计侧通过高保真页面代码更准确地表达视觉判断；让开发者在必要时精准接入；也让每次活动上线后的经验，继续反哺模板和知识库。

所以接下来最重要的行动，就是选择一个真实活动，用 OneFlow 跑完从策划方案、AI 可交互原型、配置资源清单、设计资源上传、实时预览、Vibe Coding 高保真页面代码、开发接入、验收发布到复盘沉淀的第一条完整工作流。

只要这条闭环跑通，一次活动的经验就不再只是一次交付结果，而会成为下一次活动的起点。
