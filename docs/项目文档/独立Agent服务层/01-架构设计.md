## 一、背景与目标

### 1.1 问题背景

当前项目中，Agent 相关逻辑直接嵌入在 Next.js 的 API Routes 中，存在以下问题：

| 问题         | 影响                                       |
| :----------- | :----------------------------------------- |
| **耦合度高** | Agent 逻辑与 Web 框架紧耦合，难以独立演进  |
| **扩展性差** | 只支持 opencode 后端，无法切换其他 AI 后端 |
| **复用性低** | Agent 能力与前端紧耦合，难以独立测试和维护 |
| **部署受限** | 必须与前端一起部署，无法独立扩缩容         |

### 1.2 设计目标

1. **解耦**：将 Agent 服务层从 Next.js 应用中剥离，作为独立服务运行
2. **抽象**：定义统一的 Agent 接口，支持多后端切换（opencode、Claude、Codex 等）
3. **高性能**：支持流式响应、WebSocket 实时推送
4. **易维护**：AI 编程友好，代码结构清晰

## 三、核心设计原则

### 3.1 工厂模式

支持多种 AI 后端，通过工厂方法创建对应的 Agent 实例。工厂负责注册和创建不同类型的 Agent，实现后端的可插拔式扩展。

**实现文件**：[agent-factory.ts](../packages/agent-service/src/core/agent-factory.ts)

**核心思路**：

- 使用 Map 存储后端类型与创建函数的映射
- `register()` 方法注册新后端类型
- `create()` 方法根据配置创建对应 Agent 实例

### 3.2 观察者模式

Agent 与消费层通过事件解耦，Agent 通过回调函数向外部发送流式内容、错误和完成事件。

**实现文件**：[agent.ts](../packages/agent-service/src/core/agent.ts)

**核心思路**：

- 继承 Node.js EventEmitter 实现事件发射
- 定义统一的事件类型：`stream`、`error`、`finish`、`status`
- 消费者通过 `on()`/`off()` 订阅事件

### 3.3 单例模式

AgentManager 作为全局单例，管理所有 Agent 实例的生命周期。

**实现文件**：[agent-manager.ts](../packages/agent-service/src/core/agent-manager.ts)

**核心思路**：

- 使用 Map 缓存 Agent 实例，避免重复创建
- `getOrCreate()` 方法实现懒加载和缓存复用
- 提供 `destroy()` 和 `destroyAll()` 方法清理资源

### 3.4 策略模式

不同后端使用不同的通信策略，通过统一的接口抽象。

**接口定义**：[backends/base.ts](../packages/agent-service/src/backends/base.ts)

**核心思路**：

- 定义 `IBackendAdapter` 接口规范所有后端行为
- 每个后端实现 `connect()`、`sendMessage()`、`disconnect()` 等方法
- 后端可独立演进，互不影响

## 七、安全设计

### 7.1 三层防护隔离机制

Agent 服务采用**三层防护**实现安全隔离：

| 层级   | 机制                                   | 强度 | 说明                   |
| :----- | :------------------------------------- | :--- | :--------------------- |
| **L1** | 权限系统（opencode.json 白名单）       | 强   | 配置文件级别的访问控制 |
| **L2** | 行为约束（AGENTS.md）                  | 软   | AI 行为规范和约束      |
| **L3** | 后端校验（session-guard 文件变更检测） | 强   | 运行时校验和自动回滚   |

**L3 后端校验实现**：[session/session-guard.ts](../packages/agent-service/src/session/session-guard.ts)

**核心思路**：

- 定义允许修改的文件白名单
- 每次文件变更前校验路径是否在白名单内
- 违规时自动回滚并记录日志
- 防止路径遍历攻击和符号链接攻击

### 7.2 认证（可选）

支持 API Key 和 Bearer Token 两种认证方式，通过 Fastify 中间件实现。

### 7.3 速率限制

使用 `@fastify/rate-limit` 插件实现请求限流，防止滥用。

## 八、性能考量

### 8.1 连接池

使用 undici 的连接池管理 HTTP 连接，提升并发性能。

**实现位置**：[backends/opencode-http.ts](../packages/agent-service/src/backends/opencode-http.ts)

### 8.2 Agent 缓存

AgentManager 内部使用 Map 缓存 Agent 实例，避免重复创建和初始化。

**实现位置**：[core/agent-manager.ts](../packages/agent-service/src/core/agent-manager.ts)

### 8.3 流式响应优化

使用 Transfer-Encoding: chunked 实现流式响应，直接写入底层 socket，减少内存占用。

## 九、扩展性设计

### 9.1 添加新后端

1. 在 `backends/` 目录创建新的后端适配器文件
2. 继承 `BaseBackendAdapter` 基类
3. 实现必要的接口方法
4. 在服务启动时注册到 AgentFactory

**后端基类**：[backends/base.ts](../packages/agent-service/src/backends/base.ts)

### 9.2 插件系统（未来）

预留插件接口，支持在消息处理前后插入自定义逻辑。

## 十、与现有系统集成

### 10.1 迁移策略

```
阶段 1：并行运行
├── Agent 服务独立部署
├── Web 前端同时支持新旧 API
└── 逐步切换流量

阶段 2：完全迁移
├── 删除 Web 前端中的 Agent 逻辑
├── 所有请求走 Agent 服务
└── 清理遗留代码
```

### 10.2 共享类型

类型定义统一放在 `packages/shared/src/` 目录，供 Web 前端和 Agent 服务共同使用。

## 十二、总结

本架构设计遵循以下核心原则：

1. **解耦**：Agent 服务独立部署，消费层轻量化
2. **抽象**：统一接口，支持多后端
3. **高性能**：流式响应、连接池、缓存
4. **可扩展**：工厂模式、插件系统
5. **易维护**：清晰的分层结构、完善的文档
