## 一、模块总览

当前 `agent-service` 是 Pi Agent 单后端服务，不再包含 OpenCode、Claude、Codex、Gemini 等历史后端实现。模块关系如下：

```text
Agent Service
├── server.ts
│   ├── Fastify / CORS / WebSocket / rate-limit
│   ├── 注册 pi-agent 后端
│   └── 注册 routes/
├── routes/
│   ├── agent.ts              Agent 会话、文件变更、工作空间 API
│   ├── websocket.ts          WebSocket 消息入口
│   ├── ws-event-router.ts    Pi Agent 事件到前端事件的转换
│   ├── projects.ts           项目、编辑会话、版本 API
│   ├── models.ts             Pi Agent 模型探测 API
│   ├── validate.ts           demo 代码和 schema 校验 API
│   └── internal-config.ts    author-site 到 agent-service 的内部配置同步
├── services/
│   ├── model-catalog-service.ts  `/models` 模型探测编排
│   └── viewer-ai-context.ts      使用端只读 AI 上下文构建
├── core/
│   ├── agent.ts              BaseAgent 生命周期抽象
│   ├── backend-agent.ts      IBackendAdapter 包装层
│   ├── agent-manager.ts      Session 到 Agent 实例缓存
│   ├── agent-factory.ts      固定创建 pi-agent
│   └── types.ts              Agent、事件、结果、权限类型
├── backends/
│   ├── base.ts               后端适配接口
│   ├── pi-agent.ts           Pi Agent 后端适配器
│   └── pi-tools/             工作台工具集
├── session/
│   ├── session-store.ts      Session 元数据
│   ├── session-guard.ts      路径安全校验
│   ├── snapshot-service.ts   Git/snapshot 双模式变更检测
│   ├── console-buffer.ts     页面控制台日志缓存
│   └── run-log-store.ts      运行事件日志
└── workspace/
    ├── workspace-manager.ts          临时/用户工作空间
    ├── project-workspace-manager.ts  项目工作空间和版本
    └── utils.ts
```

## 二、Core 层

### 2.1 类型定义

文件：`packages/agent-service/src/core/types.ts`

核心事实：

- `AgentType` 固定为 `"pi-agent"`。
- `AgentConfig` 记录 `sessionId`、`workingDir`、`demoId`、`model`、`toolVersion`、`permissions`、`backendProviders` 和 Pi Agent 配置。
- 事件类型包含 `stream`、`thought`、`tool_call`、`tool_call_update`、`plan`、`permission_request`、`user_choice_request`、`finish`、`error`、`status`。
- `AgentResult` 统一承载最终文本、文件变更、错误和 metadata。

### 2.2 BaseAgent

文件：`packages/agent-service/src/core/agent.ts`

`BaseAgent` 负责 Agent 生命周期抽象和事件机制。它维护状态、创建时间、最后活动时间和消息计数，并要求具体实现提供启动、发送消息、取消和销毁能力。

### 2.3 AgentFactory

文件：`packages/agent-service/src/core/agent-factory.ts`

`AgentFactory` 仍保留注册表结构，但 `create()` 当前固定使用 `"pi-agent"`。这个工厂的价值是隔离创建过程和配置装配，不表示当前系统支持多后端动态选择。

### 2.4 AgentManager

文件：`packages/agent-service/src/core/agent-manager.ts`

`AgentManager` 是全局单例，以 `sessionId` 为键缓存 Agent 实例。路由层通过 `getOrCreate()` 获取 Agent；销毁 Session 时调用 `destroy()`，服务关闭时调用 `destroyAll()`。

### 2.5 BackendAgent

文件：`packages/agent-service/src/core/backend-agent.ts`

`BackendAgent` 将 `IBackendAdapter` 包装为 `BaseAgent`，负责：

- 启动后端并转发后端事件。
- 防止同一 Agent 并发发送消息。
- 暴露模型切换、模型信息、当前 Pi Agent Session ID、文件变更、超时和取消能力。
- 支持 `updateSystemPrompt()` 将 author-site 构建的系统提示词注入 Pi Agent。

## 三、Pi Agent 后端层

### 3.1 后端接口

文件：`packages/agent-service/src/backends/base.ts`

`IBackendAdapter` 是服务层和 Pi Agent 后端之间的稳定边界。当前只有 `PiAgentBackend` 实现它。

### 3.2 PiAgentBackend

文件：`packages/agent-service/src/backends/pi-agent.ts`

职责：

- 动态导入 `@earendil-works/pi-agent-core`、`@earendil-works/pi-agent-core/node` 和 `@earendil-works/pi-ai`。
- 创建 `AgentHarness`、`NodeExecutionEnv` 和内存 Session 仓库。
- 合并全局 backend providers、Session 级模型配置和环境变量，确定当前模型。
- 创建工作台工具集，并把工具执行、权限确认、文件操作、计划和子 Agent 事件转换为统一事件。
- 捕获 `writeFile`、`editFile`、页面删除等工具带来的文件变更。
- 对图片附件执行必要的预描述或多模态传递。
- 维护 Pi Agent 当前 Session ID，支持恢复和模型切换。

配置来源：

| 配置 | 默认值 | 环境变量 |
| :--- | :--- | :--- |
| 服务端口 | `3201` | `PORT` |
| 监听地址 | `0.0.0.0` | `HOST` |
| 日志级别 | `info` | `LOG_LEVEL` |
| 截图服务地址 | `http://localhost:3202` | `SCREENSHOT_SERVICE_URL` |
| Pi Agent provider | `anthropic` | `PI_AGENT_PROVIDER` |

| 子 Agent 开关 | 开启 | `PI_AGENT_SUBAGENTS_ENABLED` |
| 子 Agent timeout | `120000` | `PI_AGENT_SUBAGENT_TIMEOUT` |
| 网页读取开关 | 开启 | `PI_AGENT_WEB_READ_ENABLED` |
| 网页读取 timeout | `10000` | `PI_AGENT_WEB_READ_TIMEOUT_MS` |

## 四、Pi Tools 工具层

文件目录：`packages/agent-service/src/backends/pi-tools/`

当前工具能力由 `createWorkbenchTools()` 统一创建，`WORKBENCH_TOOL_VERSION = 13`。`GET /api/tools/capabilities` 会返回工具版本和工具名，供前端或调试入口识别能力集。

| 工具类型 | 代表工具 | 说明 |
| :--- | :--- | :--- |
| 文件读取 | `readFile`、`readFileWithLines`、`listFiles` | 读取工作空间内文件和目录 |
| 文件写入 | `editFile`、`writeFile` | 精确替换或写入文件，变更会被后端捕获 |
| 命令执行 | `bash` | 受权限白名单限制的命令执行 |
| Schema 校验 | `schemaValidate` | 校验配置 schema |
| 图片资源 | `saveImage`、`listImages` | 保存图片并查询项目图片清单 |
| 页面观测 | `getConsoleLogs`、`captureScreenshot` | 读取前端控制台日志和截图 |
| 知识报告 | `knowledgeReport` | 按访问上下文检索系统知识和项目知识，返回带来源、可信等级和风险提示的结构化报告 |
| 网页读取 | `webRead` | 读取公开 HTTP/HTTPS 文本页面；默认启用，可通过 `PI_AGENT_WEB_READ_ENABLED=false` 关闭，并拒绝本机、内网、保留地址、凭证型 URL、非文本内容和过大响应 |
| 联网搜索 | `webSearch` | 使用 Brave Search API 查询公开互联网搜索结果；默认关闭，只在 `PI_AGENT_WEB_SEARCH_ENABLED=true` 时注册 |
| 外部工具 | `figmaMcp`、`dingtalk` | 使用用户授权后的外部工具能力 |
| 画布整理 | `arrangeCanvasPages` | 自动整理画布页位置 |
| 计划与确认 | `requestPlanApproval`、`requestUserChoice`、`updatePlan` | 计划审批、需求澄清单选卡片和计划状态更新 |
| 页面删除 | `listPages`、`previewDeletePages`、`executeDeletePagePlan`、`deletePage`、`deletePages` | 页面删除计划、预览和执行 |
| 子 Agent | `delegateTask` | 将独立任务委派给短生命周期子 Agent；子 Agent 不再继续委派 |

权限控制由 `pi-tools/permissions.ts`、删除计划 store、权限确认事件和 `SessionGuard` 共同完成。

## 五、Session 与快照层

### 5.1 SessionStore

文件：`packages/agent-service/src/session/session-store.ts`

`SessionStore` 管理 Session 元数据，包括 `sessionId`、`demoId`、`workingDir`、模型配置、工作空间类型、快照模式、状态和消息计数。HTTP 和 WebSocket 路由都会同步更新它，避免前端状态和后端 Agent 生命周期脱节。

### 5.2 SessionGuard

文件：`packages/agent-service/src/session/session-guard.ts`

`SessionGuard` 提供路径校验和工作空间边界保护。所有用户输入的目标路径都要解析到工作空间内，防止 `../`、绝对路径或符号链接绕过边界。

### 5.3 SnapshotService

文件：`packages/agent-service/src/session/snapshot-service.ts`

详见 [05-快照服务](./05-快照服务.md)。核心职责是建立基准、比较变更、暂存文件、丢弃文件和清除快照。Git 仓库使用 Git 模式，普通目录使用内存/文件快照模式。

### 5.4 ConsoleBuffer 与 RunLogStore

文件：

- `packages/agent-service/src/session/console-buffer.ts`
- `packages/agent-service/src/session/run-log-store.ts`

`console-buffer` 接收前端通过 WebSocket `console_data` 上报的日志，供 `getConsoleLogs` 工具读取。`run-log-store` 记录 Agent 运行事件，服务于运行进度和事件排查。

## 六、工作空间与项目管理层

### 6.1 WorkspaceManager

文件：`packages/agent-service/src/workspace/workspace-manager.ts`

负责普通 Agent 会话的工作空间创建和清理。不指定 `workingDir` 时创建临时工作空间；指定路径时按用户工作空间处理并规范化路径。

### 6.2 ProjectWorkspaceManager

文件：`packages/agent-service/src/workspace/project-workspace-manager.ts`

负责正式项目数据：

- 创建、读取和删除项目。
- 打开编辑会话并复制正式工作空间到临时编辑空间。
- 保存编辑变更，生成版本快照，再覆盖正式工作空间。
- 放弃编辑并清理临时空间。
- 查询版本历史和恢复指定版本。

该模块支撑 `routes/projects.ts`，也是创作端和使用端读取项目数据的服务边界。

## 七、路由与事件层

### 7.1 HTTP 路由

路由注册入口：`packages/agent-service/src/routes/index.ts`

| 文件 | 职责 |
| :--- | :--- |
| `agent.ts` | Agent 消息、Session、工作空间、文件变更、工具能力 |
| `projects.ts` | 项目 CRUD、编辑会话、保存/放弃、版本历史、恢复版本 |
| `models.ts` | `/models` HTTP 适配，具体模型探测由 `ModelCatalogService` 编排 |
| `validate.ts` | demo 代码与 schema 校验 |
| `internal-config.ts` | 内部 backend providers 同步 |
| `websocket.ts` | WebSocket 连接和客户端消息处理 |

### 7.2 WebSocketEventRouter

文件：`packages/agent-service/src/routes/ws-event-router.ts`

负责把 Pi Agent 后端事件整理成前端可消费的稳定消息。它会聚合工具调用、记录 finish/error、转发权限确认和需求确认、处理计划事件，并避免前端直接依赖 Pi Agent 原始事件结构。

WebSocket 主流程的边界是：`websocket.ts` 只负责连接、消息解析和协议适配；`ws-event-router.ts` 负责把 agent 事件转成稳定的前端消息；Pi Agent 原始事件映射、工具结果、文件变更、权限请求和需求确认请求由后端管理器处理。后续修改 agent message 或 stream event 时，不应把 Pi Agent 原始事件结构直接暴露给 WebSocket route。

### 7.3 Route/Service 分层

HTTP route 只负责请求参数、状态码和响应结构适配；可脱离 Fastify 测试的业务编排应进入 `services/`。当前 `/models` 已采用该模式：`routes/models.ts` 只映射成功、503 和 500 响应，`ModelCatalogService` 负责创建临时 agent、启动探测、转换模型分组、识别连接错误，并在 finally 中清理临时 agent。

错误响应必须保持 `{ success: false, error: { code, message, details? } }` 结构。新增或重构 route 时优先使用 `routes/api-response.ts` 的 `sendApiSuccess()` 和 `sendApiError()`，避免在 route 中手写多个不一致的响应形状。

### 7.4 服务入口

文件：`packages/agent-service/src/server.ts`

启动流程：

1. 从根目录 `.env` 加载环境变量。
2. 初始化 backend providers 管理器。
3. 创建 Fastify 实例并注册 CORS、WebSocket、rate-limit。
4. 在 `AgentFactory` 注册 `pi-agent` 创建器。
5. 注册 HTTP 和 WebSocket 路由。
6. 暴露 `/health`。
7. 监听 `SIGTERM`，销毁 Agent、清空 SessionStore 并关闭 Fastify。

## 九、总结

当前核心模块的主线是：

```text
HTTP/WebSocket 请求
  -> routes
  -> AgentManager
  -> BackendAgent
  -> PiAgentBackend
  -> Pi Tools / AgentHarness
  -> 统一事件与文件变更
  -> WebSocketEventRouter 或 HTTP 响应
```

扩展服务能力时应优先复用这条链路。新增模型供应商走 backend providers 配置；新增 Agent 能力优先扩展 `pi-tools/`；新增项目读写能力优先进入 `project-workspace-manager.ts` 或独立 `project-core`，不要恢复历史多后端适配器。
