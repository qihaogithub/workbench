### 2.2 Agent 基类模块

**文件位置**：[core/agent.ts](../packages/agent-service/src/core/agent.ts)

**核心职责**：定义 Agent 的基础行为和生命周期

**设计思路**：

1. **继承 EventEmitter**：利用 Node.js 原生事件机制实现观察者模式
2. **状态管理**：内部维护 `_status` 状态，通过 `setStatus()` 方法变更并触发事件
3. **抽象方法**：定义 `start()`、`sendMessage()`、`cancel()`、`kill()` 为子类必须实现的方法
4. **生命周期追踪**：记录 `createdAt`、`lastActivityAt`、`messageCount` 等元信息

**核心属性**：

| 属性             | 类型        | 说明         |
| :--------------- | :---------- | :----------- |
| `config`         | AgentConfig | Agent 配置   |
| `_status`        | AgentStatus | 当前状态     |
| `messageCount`   | number      | 消息计数     |
| `createdAt`      | Date        | 创建时间     |
| `lastActivityAt` | Date        | 最后活动时间 |

**核心方法**：

| 方法            | 说明                     |
| :-------------- | :----------------------- |
| `start()`       | 启动 Agent（抽象方法）   |
| `sendMessage()` | 发送消息（抽象方法）     |
| `cancel()`      | 取消当前操作（抽象方法） |
| `kill()`        | 销毁 Agent（抽象方法）   |
| `getInfo()`     | 获取 Agent 信息          |
| `setStatus()`   | 设置状态并触发事件       |

### 2.3 Agent 工厂模块

**文件位置**：[core/agent-factory.ts](../packages/agent-service/src/core/agent-factory.ts)

**核心职责**：根据配置创建对应类型的 Agent 实例

**设计思路**：

1. **注册机制**：使用 Map 存储类型与创建函数的映射
2. **运行时扩展**：支持在运行时动态注册新的 Agent 类型
3. **单例访问**：提供 `getAgentFactory()` 获取全局工厂实例

**核心方法**：

| 方法                      | 说明                      |
| :------------------------ | :------------------------ |
| `register(type, creator)` | 注册 Agent 类型和创建函数 |
| `create(config)`          | 根据配置创建 Agent 实例   |
| `has(type)`               | 检查类型是否已注册        |
| `getRegisteredTypes()`    | 获取所有已注册类型        |

**使用方式**：

1. 服务启动时，调用 `register()` 注册各后端类型
2. 收到请求时，调用 `create()` 创建 Agent 实例
3. 工厂内部根据 `config.backend` 字段选择对应的创建函数

---

### 2.4 Agent 管理器模块

**文件位置**：[core/agent-manager.ts](../packages/agent-service/src/core/agent-manager.ts)

**核心职责**：管理所有 Agent 实例的生命周期

**设计思路**：

1. **实例缓存**：使用 Map 缓存 Agent 实例，避免重复创建
2. **懒加载**：首次访问时创建 Agent，后续复用缓存
3. **资源清理**：提供销毁单个或全部 Agent 的方法
4. **消息代理**：封装消息发送逻辑，自动处理初始化

**核心方法**：

| 方法                                       | 说明                |
| :----------------------------------------- | :------------------ |
| `getOrCreate(sessionId, config)`           | 获取或创建 Agent    |
| `get(sessionId)`                           | 获取指定 Agent      |
| `has(sessionId)`                           | 检查 Agent 是否存在 |
| `destroy(sessionId)`                       | 销毁指定 Agent      |
| `destroyAll()`                             | 销毁所有 Agent      |
| `sendMessage(sessionId, content, options)` | 发送消息            |
| `list()`                                   | 列出所有 Agent 信息 |
| `count()`                                  | 获取 Agent 数量     |

**生命周期管理**：

1. `getOrCreate()` 检查缓存，不存在则通过工厂创建
2. `sendMessage()` 自动调用 `start()` 初始化未就绪的 Agent
3. `destroy()` 调用 Agent 的 `kill()` 方法并从缓存移除

## 五、BackendAgent 模块

### 5.1 BackendAgent 适配器

**文件位置**：[core/backend-agent.ts](../packages/agent-service/src/core/backend-agent.ts)

**核心职责**：统一封装后端接口，提供模型切换、忙闲状态等高级功能

**设计思路**：

1. **继承 BaseAgent**：复用事件机制和生命周期管理
2. **封装 IBackendAdapter**：将后端适配器包装为统一接口
3. **忙闲状态管理**：通过 `busy` 标志防止并发请求
4. **模型切换支持**：可选的 `setModel()` 和 `getModelInfo()` 方法
5. **事件转发**：将后端事件转发到 Agent 层

**核心属性**：

| 属性          | 类型                    | 说明             |
| :------------ | :---------------------- | :--------------- |
| `backend`     | BackendWithModelSupport | 后端适配器       |
| `busy`        | boolean                 | 是否正在处理请求 |
| `initialized` | boolean                 | 是否已初始化     |

**核心方法**：

| 方法                             | 说明                                       |
| :------------------------------- | :----------------------------------------- |
| `start(options?)`                | 启动 Agent，调用后端的 start 或 initialize |
| `sendMessage(content, options?)` | 发送消息，自动处理初始化和忙闲检查         |
| `cancel()`                       | 取消当前请求                               |
| `kill()`                         | 销毁 Agent                                 |
| `isBusy()`                       | 检查是否繁忙                               |
| `setModel(modelId)`              | 切换模型（可选）                           |
| `getModelInfo()`                 | 获取模型信息（可选）                       |
| `getCurrentSessionId()`          | 获取当前 Session ID（可选）                |

**事件转发机制**：

```typescript
this.backend.onStream((event) => {
  this.emit(event.type, event);
});
```

### 6.2 Session 守卫

| `getChangedFiles(workingDir)`               | 获取变更文件列表         |

### 6.3 快照服务

**文件位置**：[session/snapshot-service.ts](../packages/agent-service/src/session/snapshot-service.ts)

**核心职责**：文件变更检测和回滚支持

**设计思路**：

1. **双模式支持**：Git 仓库使用 git-repo 模式，普通目录使用 snapshot 模式
2. **变更检测**：对比当前状态与基准状态
3. **变更回滚**：支持丢弃变更恢复到基准状态

**核心方法**：

| 方法                                           | 说明                           |
| :--------------------------------------------- | :----------------------------- |
| `init(workingDir)`                             | 初始化快照，检测 Git 仓库      |
| `compare(workingDir)`                          | 对比变更，返回 staged/unstaged |
| `getBaselineContent(workingDir, filePath)`     | 获取基准内容                   |
| `stageFile(workingDir, filePath)`              | 暂存文件（Git 模式）           |
| `discardFile(workingDir, filePath, operation)` | 丢弃变更                       |
| `clearSnapshot(workingDir)`                    | 清除快照                       |

**快照模式**：

| 模式       | 说明                  |
| :--------- | :-------------------- |
| `git-repo` | 使用 Git 命令检测变更 |
| `snapshot` | 使用内存快照检测变更  |

**启动流程**：

4. 注册后端到工厂
5. 注册路由
6. 启动监听

**优雅关闭**：

1. 监听 `SIGTERM` 和 `SIGINT` 信号
2. 销毁所有 Agent 实例
3. 关闭 Fastify 服务
4. 退出进程

## 十三、总结

本模块设计遵循以下原则：

1. **单一职责**：每个模块只负责一个功能
2. **依赖注入**：通过工厂和管理器解耦依赖
3. **接口抽象**：使用接口定义契约，便于扩展
4. **事件驱动**：通过事件总线解耦组件
5. **工作空间隔离**：每个 Session 绑定独立工作空间，确保安全隔离
