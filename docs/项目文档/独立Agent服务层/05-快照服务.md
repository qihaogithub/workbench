## 一、模块定位

快照服务（SnapshotService）是 Agent 服务层中**工作空间层**的核心组件，负责文件变更检测和回滚支持。它为用户提供了"AI 修改了哪些文件"的可视化能力，以及"撤销 AI 修改"的回滚能力。

快照服务与 [Session 守卫](./03-核心模块设计.md#62-session-守卫) 协同工作：Session 守卫负责**安全边界**（文件能否被访问），快照服务负责**变更追踪**（文件被改成了什么样）。

---

## 二、双模式架构

快照服务的核心设计是**双模式策略**：根据工作空间是否为 Git 仓库，自动选择最优的变更检测方式。

```
工作空间目录
    │
    ├── 检测：是否为 Git 仓库？
    │       │
    │   ┌───┴───┐
    │   ▼       ▼
    │ Git 仓库  非 Git 目录
    │   │       │
    │   ▼       ▼
    │ git-repo  snapshot
    │ 模式      模式
    │   │       │
    │   ▼       ▼
    │ 委托 Git  内存快照
    │ 原生命令  逐文件对比
    │
    └──► 统一输出：CompareResult（staged + unstaged）
```

### 2.1 模式判定

| 步骤 | 方法 | 说明 |
|:-----|:-----|:-----|
| 1 | 执行 `git rev-parse --git-dir` | 成功则为 Git 仓库 |
| 2 | 获取分支名 `git rev-parse --abbrev-ref HEAD` | 仅 Git 模式有效 |
| 3 | 若非 Git 仓库 | 退化为内存快照模式 |

### 2.2 两种模式对比

| 能力 | git-repo 模式 | snapshot 模式 |
|:-----|:-------------|:-------------|
| 变更检测 | `git status --porcelain` | 重新扫描目录，逐文件对比内容和 mtime |
| 暂存（stage） | `git add` / `git add -A` | **不支持**（staged 始终为空数组） |
| 取消暂存 | `git reset HEAD` | **不支持** |
| 丢弃变更 | `git checkout HEAD --` 或删除文件 | 从内存快照恢复或删除文件 |
| 获取基线内容 | `git show HEAD:<file>` | 从内存 Map 中读取 |
| 数据持久性 | Git 仓库自带持久化 | **进程内存**，服务重启即丢失 |
| 分支信息 | 有 | 无（branch 为 null） |

## 三、核心流程

### 3.1 初始化流程

```
init(workingDir)
    │
    ├── isGitRepository(workingDir)?
    │       │
    │   ┌───┴────┐
    │   ▼        ▼
    │  是 Git    非 Git
    │   │        │
    │   ▼        ▼
    │ 返回       createFileSnapshot()
    │ {           │
    │   mode:     ├── scanDirectory() 递归扫描
    │   'git-repo',│   ├── 跳过 '.' 开头文件（.session.json 除外）
    │   branch    │   ├── 跳过 node_modules
    │ }          │   ├── 跳过 opencode.json
    │            │   └── 读取每个文件内容 + mtime
    │            │
    │            ▼
    │          存入内存 Map<workingDir, SnapshotData>
    │            │
    │            ▼
    │          返回 { mode: 'snapshot', branch: null }
    │
    └──► SessionMeta 记录 snapshotMode 和 snapshotBranch
```

### 3.2 变更比较流程

```
compare(workingDir)
    │
    ├── isGitRepository(workingDir)?
    │       │
    │   ┌───┴────┐
    │   ▼        ▼
    │  Git 模式   Snapshot 模式
    │   │        │
    │   ▼        ▼
    │ compareWithGit()   compareWithSnapshot()
    │   │                │
    │   ▼                ▼
    │ 解析 git status    重新扫描目录
    │ --porcelain 输出   对比内存快照
    │   │                │
    │   ▼                ▼
    │ 区分 staged/      仅产生 unstaged
    │ unstaged 变更     （staged 始终为空）
    │   │                │
    │   ▼                ▼
    │ 操作映射：          操作映射：
    │ A/? → create       新文件 → create
    │ D → delete         内容/mtime 变 → modify
    │ 其他 → modify      快照有但磁盘无 → delete
    │
    └──► 返回 CompareResult { staged, unstaged }
```

### 3.3 丢弃变更流程

```
discardFile(workingDir, filePath, operation)
    │
    ├── isGitRepository(workingDir)?
    │       │
    │   ┌───┴────┐
    │   ▼        ▼
    │  Git 模式   Snapshot 模式
    │   │        │
    │   ▼        ▼
    │ operation?   operation?
    │   │           │
    │   ├── create  ├── create → 删除文件
    │   │   └── rm  │
    │   │           ├── modify → 从内存快照恢复
    │   └── 其他    │   └── writeFile(baseline.content)
    │       └── git │
    │       checkout ├── delete → 从内存快照恢复
    │       HEAD --  │   └── writeFile(baseline.content)
    │
    └──► 文件恢复到基线状态
```

### 3.4 回滚全流程（API 层组合）

回滚操作由路由层组合调用多个快照服务方法：

```
POST /api/agent/:sessionId/rollback
    │
    ├── 1. compare() ──► 获取所有变更文件
    │
    ├── 2. 对每个变更文件调用 discardFile() ──► 逐个恢复
    │
    ├── 3. clearSnapshot() ──► 清除旧快照
    │
    └── 4. init() ──► 重新创建快照（以当前状态为新基线）
```

## 四、共享类型定义

快照服务使用的核心类型定义在 `@opencode-workbench/shared` 包中，供前后端共用。

| 类型 | 字段 | 说明 |
|:-----|:-----|:-----|
| `SnapshotInfo` | `mode: 'git-repo' \| 'snapshot'` | 当前快照模式 |
| | `branch: string \| null` | Git 分支名（仅 git-repo 模式） |
| `FileChangeInfo` | `path: string` | 变更文件相对路径 |
| | `operation: 'create' \| 'modify' \| 'delete'` | 变更操作类型 |
| | `status: 'staged' \| 'unstaged'` | 暂存状态 |
| `CompareResult` | `staged: FileChangeInfo[]` | 已暂存的变更列表 |
| | `unstaged: FileChangeInfo[]` | 未暂存的变更列表 |

## 五、API 方法一览

| 方法 | 参数 | 返回值 | Git 模式 | Snapshot 模式 |
|:-----|:-----|:-------|:---------|:-------------|
| `init(workingDir)` | 工作目录 | `Promise<SnapshotInfo>` | 返回分支信息 | 创建内存快照 |
| `compare(workingDir)` | 工作目录 | `Promise<CompareResult>` | git status 解析 | 逐文件对比 |
| `getBaselineContent(workingDir, filePath)` | 目录+文件 | `Promise<string \| null>` | git show HEAD | 内存 Map 读取 |
| `stageFile(workingDir, filePath)` | 目录+文件 | `Promise<void>` | git add | **空操作** |
| `stageAll(workingDir)` | 工作目录 | `Promise<void>` | git add -A | **空操作** |
| `unstageFile(workingDir, filePath)` | 目录+文件 | `Promise<void>` | git reset HEAD | **空操作** |
| `discardFile(workingDir, filePath, operation)` | 目录+文件+操作 | `Promise<void>` | git checkout / rm | 快照恢复 / rm |
| `resetFile(workingDir, filePath, operation)` | 同 discardFile | `Promise<void>` | 委托 discardFile | 委托 discardFile |
| `clearSnapshot(workingDir)` | 工作目录 | `void` | 清除内存记录 | 清除内存快照 |

## 六、路由层调用关系

快照服务被 HTTP 路由和 WebSocket 路由调用，以下是完整的调用映射：

### 6.1 HTTP API 调用

| API 端点 | 调用方法 | 触发场景 |
|:---------|:---------|:---------|
| `POST /api/agent/:sessionId/message` | `init()` | 首次创建会话时初始化快照 |
| `DELETE /api/agent/:sessionId` | `clearSnapshot()` | 销毁会话时清理快照 |
| `GET /api/agent/:sessionId/files` | `compare()` | 获取文件变更列表 |
| `POST /api/agent/:sessionId/rollback` | `compare()` + `discardFile()` + `clearSnapshot()` + `init()` | 回滚所有变更 |
| `PUT /api/agent/:sessionId/workspace` | `init()` | 切换工作空间时重新初始化 |
| `POST /api/agent/:sessionId/files/stage` | `stageFile()` | 暂存指定文件 |
| `POST /api/agent/:sessionId/files/discard` | `discardFile()` | 丢弃指定文件变更 |

### 6.2 WebSocket 调用

| 场景 | 调用方法 | 说明 |
|:-----|:---------|:-----|
| 首次消息创建会话 | `init()` | 初始化快照 |
| WebSocket 连接关闭（临时工作空间） | `clearSnapshot()` | 清理快照 |

## 七、扫描过滤规则

内存快照模式在扫描目录时，遵循以下过滤规则：

| 规则 | 说明 |
|:-----|:-----|
| 跳过 `.` 开头的文件/目录 | 隐藏文件不纳入快照 |
| **例外**：`.session.json` | 会话配置文件需要追踪 |
| 跳过 `node_modules` | 依赖目录不纳入快照 |
| 跳过 `opencode.json` | 项目配置文件不纳入快照 |

## 八、Git Status 解析规则

Git 模式下，`git status --porcelain` 的每行输出格式为 `XY PATH`，解析规则如下：

| 条件 | 分类 | 操作映射 |
|:-----|:-----|:---------|
| `X`（indexStatus）不为空且不为 `?` | staged | `A` → create, `D` → delete, 其他 → modify |
| `Y`（workTreeStatus）不为空 | unstaged | 同上映射 |
| `X` 为 `?`（未跟踪文件） | unstaged | → create |

## 九、设计考量与限制

### 9.1 内存存储的权衡

Snapshot 模式将文件内容存储在进程内存中，这意味着：

- **优势**：无需额外依赖，实现简洁，读写速度快
- **限制**：服务重启后快照丢失，需重新 `init()`；大工作空间可能占用较多内存

### 9.2 暂存功能的不对称

Snapshot 模式下 `stageFile`、`stageAll`、`unstageFile` 均为空操作，`CompareResult.staged` 始终为空数组。这是因为"暂存"概念源自 Git 的索引区，在纯文件系统中没有对应机制。

### 9.3 全局单例模式

快照服务以全局单例导出（`export const snapshotService = new SnapshotService()`），内部使用 `Map<workingDir, SnapshotData>` 管理多个工作空间的快照数据。每个工作空间独立维护自己的快照。

## 十、与 Session 生命周期的关系

快照服务与 Session 的生命周期紧密绑定：

```
Session 创建 ──► snapshotService.init() ──► 记录 snapshotMode/snapshotBranch 到 SessionMeta
    │
    │  （AI 对话进行中，文件可能被修改）
    │
    ├── 用户查看变更 ──► snapshotService.compare()
    ├── 用户暂存文件 ──► snapshotService.stageFile()
    ├── 用户丢弃变更 ──► snapshotService.discardFile()
    ├── 用户回滚全部 ──► compare() → discardFile() → clearSnapshot() → init()
    ├── 用户切换工作空间 ──► snapshotService.init()（重新初始化）
    │
Session 销毁 ──► snapshotService.clearSnapshot() ──► 释放内存
```
