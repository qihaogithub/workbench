# 知识库服务整体架构设计

## 一、分层定位

独立知识库采用领域包、服务进程和集成客户端三层：

| 层级 | 包 | 职责 |
|:-----|:---|:-----|
| 领域层 | `@workbench/knowledge-core` | 定义知识条目、来源、可信等级、访问上下文、权限、阅读地图、报告和索引任务状态 |
| 服务层 | `@workbench/knowledge-service` | 运行 Fastify 服务、模板项目发现、SQLite FTS5 索引、检索、原文读取、状态和在线备份 |
| 集成层 | author-site / project-core / agent-service | 切换模板身份、触发立即同步、调用知识报告与原文读取 |

`knowledge-service` 是独立 Docker 服务，但与其他服务共享只读项目数据目录。SQLite 只有该服务进程写入，避免多个业务服务直接竞争数据库写锁。

## 二、集成方

- author-site：维护工作空间知识，并在模板身份发生变化后请求立即同步。
- project-core：维护 `Project.projectType` 和模板元数据，项目工作空间始终是事实源。
- agent-service：先生成系统/当前项目报告，再通过服务查询其他模板项目；需要全文时使用来源引用回读。
- viewer-site：当前不开放跨模板检索，避免使用端越过公开范围。

## 三、数据落点

服务使用共享数据目录：

```text
data/
├── knowledge/
│   ├── knowledge.db
│   ├── knowledge.db-wal
│   └── knowledge.db-shm
├── backups/knowledge/knowledge-{timestamp}.db
└── projects/{projectId}/
    ├── project.json
    └── workspace/
```

工作空间用户知识继续保存在 `workspace/knowledge/manifest.json` 和 Markdown 文件中。数据库只是可重建索引，不替代项目文件；备份使用 SQLite Online Backup API 生成一致性单文件副本。

## 四、调用方向

模板保存链路：

```text
project-core 把项目标记为 template
  → author-site 请求立即 reconcile
  → knowledge-service 扫描 project.json
  → 按 revision + rootHash 判断是否变化
  → 读取并切分 workspace 文档
  → 短事务替换该项目的 documents/chunks/FTS 索引
```

服务启动时先完整协调，之后默认每 5 秒扫描一次。立即同步失败不会回滚项目操作，周期协调会自动补偿。

AI 报告链路：

```text
Pi Agent 调用 knowledgeReport
  → 本地 Basic 检索系统规则和当前项目知识
  → knowledge-service FTS5 检索其他模板项目
  → 返回摘要、knowledge:// 引用、revision 和 rootHash
  → 必要时调用 readKnowledgeSource 读取精确原文
```

## 五、边界原则

- 创作端 UI 不拥有知识来源优先级和权限过滤规则。
- Agent Service 不直接定义知识来源优先级，只承载工具调用。
- SQLite 服务保持单实例且不映射宿主机端口，只允许 Compose 内部网络调用。
- 当前项目事实始终优先于其他模板项目参考。
- 检索后端可以替换，但报告格式和权限模型不应被后端反向定义。
