# 知识库服务整体架构设计

## 一、分层定位

独立知识库服务在当前 v1 中采用轻量包形态，不单独部署进程：

| 层级 | 包 | 职责 |
|:-----|:---|:-----|
| 领域层 | `@workbench/knowledge-core` | 定义知识条目、来源、可信等级、访问上下文、权限、阅读地图、报告和索引任务状态 |
| 服务层 | `@workbench/knowledge-service` | 提供 Basic 检索、报告生成、模板阅读地图、文件型索引任务和异步整理扩展点 |
| 集成层 | author-site / project-core / agent-service | 管理知识文档、保存模板、调用知识报告 |

这种形态先保证服务边界清晰，后续可独立拆为 `knowledge-service` 进程和后台 worker。

## 二、集成方

- author-site：维护工作空间用户知识，并在保存模板时触发阅读地图生成。
- project-core：保存模板时触发模板阅读地图生成。
- agent-service：通过 `knowledgeReport` 工具向主 AI 提供结构化知识报告。
- viewer-site：未来通过只读访问上下文使用公开知识。

## 三、数据落点

当前 v1 使用已有数据目录：

```text
data/
├── knowledge/
│   ├── index-jobs/
│   └── templates/{templateId}/reading-map.json
├── templates/{templateId}/workspace/
└── users.db
```

工作空间用户知识继续保存在 `workingDir/knowledge/manifest.json` 和 Markdown 文件中。创作端管理后台不再维护内置知识库配置；跨端系统规则后续应通过独立知识库服务边界接入。

## 四、调用方向

模板保存链路：

```text
author-site / project-core 保存模板
  → knowledge-service 创建索引任务
  → 读取模板 workspace
  → 生成 reading-map.json
```

AI 报告链路：

```text
Pi Agent 调用 knowledgeReport
  → 收集系统知识快照和工作空间 manifest
  → BasicRetrievalBackend 检索与权限过滤
  → 输出结构化报告
```

## 五、边界原则

- 创作端 UI 不拥有知识来源优先级和权限过滤规则。
- Agent Service 不直接定义知识来源优先级，只承载工具调用。
- 检索后端可以替换，但报告格式和权限模型不应被后端反向定义。
