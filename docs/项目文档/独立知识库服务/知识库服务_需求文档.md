# 知识库服务需求文档

## 模块定位

独立知识库服务是面向 AI 的知识基础设施，不是单纯的文档 CRUD。它负责统一管理和组织系统规则、当前项目事实、模板知识、模板库参考和会话上下文，让创作端、使用端、Agent Service 和未来系统都能在同一套规则下检索、读取和引用知识。

创作端知识库 UI 的具体交互见 [创作端知识库需求](../创作端/09-知识库/知识库_需求文档.md)。

## 功能需求清单

### 1. 知识来源分层

- 系统规则知识必须作为最高优先级的硬约束。
- 当前项目知识必须作为当前项目事实。
- 关联模板知识只能作为默认参考。
- 全量模板知识只能作为参考样本。
- 会话上下文只能作为临时事实，不能覆盖系统规则和当前项目事实。

### 2. 权限过滤

- 服务必须根据调用主体、入口、用途和能力请求过滤知识。
- 创作端作者和创作端 AI 可使用作者私有知识。
- 使用端访问者和使用端 AI 只能使用公开允许的知识。
- 管理员和系统任务可管理索引任务与运维状态。
- 低权限主体不能通过搜索、关联、报告或缓存推断不可见知识的存在性。

### 3. 模板项目入库

- 项目被标记为模板后必须自动进入独立知识库，不要求额外复制模板快照。
- 模板项目后续仍可正常编辑和发布；内容修订后索引必须自动刷新。
- 项目取消模板身份或被删除后，必须立即从跨项目检索结果中失效。
- 入库内容至少覆盖项目总览、页面源码、项目配置和工作空间知识文档。
- 索引只保存检索副本，项目工作空间仍是事实源。

### 4. 索引任务状态

- 知识索引任务必须暴露 pending、running、ready 和 failed 状态。
- 索引必须按项目修订号与根哈希幂等更新，未变化的项目不得重复重建。
- 文档切分和文件读取应在事务外完成，SQLite 写事务应保持短小。
- 服务必须提供完整性、最近同步时间、失败任务和索引规模等运维状态。

### 5. 知识报告

- AI 可请求知识报告，而不是把大量检索过程放进主对话上下文。
- 知识报告必须包含结论摘要、相关资料、来源路径、可信等级、适用范围、建议继续读取的原文、未找到的信息和风险提示。
- 报告必须明确区分系统规则、当前项目事实、模板参考和模板样本。
- 创作端 Agent 可以搜索除当前项目外的全部模板项目，并获得不可伪造的来源引用。
- Agent 必须能使用来源引用读取被索引文档的完整原文。
- 涉及配置、Schema、系统规则和业务规则时，报告必须提示主 AI 继续读取原文确认。

### 6. 扩展边界

- 当前局域网部署使用 SQLite FTS5 作为全文索引，服务以单实例独立进程运行。
- 中文检索必须在不依赖外部云服务的前提下支持连续文本命中。
- SQLite 数据库必须使用 WAL、短事务和在线备份；备份不能通过直接复制活动中的数据库文件完成。
- LLM 异步整理只能作为增强能力，不应改变业务 API。
- 向量库、图数据库或 PostgreSQL 等只作为规模扩大后的可替换后端，不定义业务协议。

## 业务规则与约束

- 系统规则优先级高于当前项目知识。
- 当前项目知识优先级高于模板参考。
- 其他模板资料不能被表述为当前项目事实。
- 跨模板检索默认排除当前项目，避免同一事实以两种信任级别重复出现。
- 检索结果必须携带模板项目 id、修订号和根哈希，便于判断来源是否过期。
- 报告缓存键必须包含访问上下文。
- 摘要和阅读地图不得扩大原文权限。
- 服务调用方只声明访问上下文，不自行复制权限过滤规则。
