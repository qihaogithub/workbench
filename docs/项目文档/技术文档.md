# UI Demo 可配置生成系统 - 技术文档

## 文档说明

本文档描述系统的技术架构和实现细节，按功能模块组织内容。文档侧重说明设计思路、架构决策和模块交互，而非实现细节。

---

## 1. 系统架构

### 1.1 Monorepo 架构

项目采用 Monorepo 架构，使用 pnpm workspaces 管理多个包：

```
opencode工作台/
├── packages/
│   ├── web/                    # Web 工作台前端
│   ├── figma-plugin/           # Figma 插件（后续接入）
│   └── shared/                 # 共享代码（类型、工具、组件）
├── demos/                      # Demo 存储目录
├── sessions/                   # 草稿工作区目录
├── package.json                # 根 package.json
├── pnpm-workspace.yaml         # pnpm 工作区配置
└── turbo.json                  # Turborepo 配置
```

#### 1.1.1 模块说明

| 包 | 职责 |
|:---|:-----|
| `packages/web` | Web 工作台前端界面 |
| `packages/shared` | 共享类型定义、工具函数、SDK 组件 |
| `demos/` | Demo 持久化存储目录 |
| `sessions/` | 草稿工作区目录（所有编辑会话的父目录） |

#### 1.1.2 Monorepo 优势

- **统一管理**：Web 工作台、Figma 插件、共享代码在同一仓库管理
- **代码复用**：共享类型定义、工具函数、组件可跨项目复用
- **统一依赖**：统一管理依赖版本，避免版本冲突
- **原子提交**：跨项目修改可在一个 PR 中完成

### 1.2 技术栈选型

| 类别 | 技术栈 | 说明 |
|:-----|:-------|:-----|
| 前端框架 | Next.js 14 | React 全栈框架（App Router） |
| 样式方案 | Tailwind CSS | 原子化 CSS 框架 |
| UI 组件库 | shadcn/ui | 基于 Radix UI 的高质量组件库 |
| 代码预览 | Sandpack | 在线代码编辑器和预览器 |
| JSON Schema 表单 | react-json-schema-form / formily | 根据 Schema 自动生成表单 |
| AI 引擎 | opencode server | 单进程共享架构，通过权限白名单实现 Demo 隔离 |
| 包管理 | pnpm | 高效的包管理器，支持 Monorepo |
| 构建工具 | Turborepo | Monorepo 构建加速工具 |
| 部署方式 | Docker | 容器化部署 |

## 2. 架构设计原则

### 2.1 核心理念

**"插件产出骨架，AI 注入灵魂，SDK 托底能力"**

### 2.2 设计原则

#### 2.2.1 单进程共享架构

所有 Session 共享一个 opencode serve 进程，通过权限系统实现隔离。

- **工作目录**：`/sessions`（所有 session 的父目录）
- **权限控制**：每个 session 目录包含独立的权限配置和行为约束
- **参考性**：AI 可以读取其他 Demo 的代码作为参考

#### 2.2.2 隔离机制设计

采用三层防护实现安全隔离：

| 层级 | 机制 | 作用 | 强度 |
|:-----|:-----|:-----|:-----|
| L1 | 权限系统 | opencode.json 细粒度权限，物理上阻止 AI 编辑非目标文件 | 强 |
| L2 | 行为约束 | AGENTS.md 告诉 AI 不要尝试访问其他目录 | 软 |
| L3 | 后端校验 | 文件变更检测 + 自动回滚 | 强 |

#### 2.2.3 配置优先级

Session 规则 > Demo 规则 > 全局规则 > 全局配置

### 2.3 决策依据

**为什么选择单进程共享而非多进程隔离？**

多进程方案（每个 Demo 启动独立 opencode 进程）虽然隔离性最强，但存在以下问题：

- 资源开销大（每个进程占用内存）
- 进程管理复杂（端口分配、超时清理、异常恢复）
- **AI 无法参考其他 Demo 的实现** — 这是设计师的重要需求

单进程方案通过 opencode 原生权限系统实现强约束，在保证安全性的同时保留 AI 参考其他 Demo 的能力。

## 3. 存储模块

### 3.1 Demo 存储结构

#### 3.1.1 目录结构

```
demos/                          # Demo 存储目录（正式版本）
├── demo-1/
│   ├── index.tsx            # React 组件代码
│   ├── config.schema.json   # 配置 Schema
│   └── AGENTS.md            # Demo 级规则（可选）
├── demo-2/
│   └── ...

sessions/                       # 草稿工作区目录（临时编辑）
├── session-uuid-1/          # 用户 A 的编辑会话
├── session-uuid-2/          # 用户 B 的编辑会话
└── ...
```

#### 3.1.2 文件说明

| 文件 | 职责 |
|:-----|:-----|
| `index.tsx` | React 组件代码，通过 props 接收配置值 |
| `config.schema.json` | JSON Schema 格式的配置元数据定义 |
| `AGENTS.md` | Demo 级规则，保存用户要求等记忆 |

### 3.2 草稿工作区设计

#### 3.2.1 核心原则

用户和 AI 不直接修改原始 Demo 文件，而是在独立的草稿工作区中编辑。

#### 3.2.2 工作流设计

| 阶段 | 操作 | 说明 |
|:-----|:-----|:-----|
| 进入编辑（Clone） | 复制 `/demos/demo-1/` → `/sessions/{session_id}/` | 为当前编辑会话创建独立副本 |
| 注入配置（Config） | 写入 `opencode.json`（权限白名单）+ `AGENTS.md`（行为约束） | 确保 AI 只能编辑当前 session 的目标文件，但可读取其他 Demo 参考 |
| AI 隔离（Mount） | 单进程 opencode serve 以 `/sessions` 为工作目录启动 | 所有 session 共享同一进程，通过权限系统实现隔离 |
| 实时预览（Sync） | 前端通过 `/file/content` API 读取 session 目录代码渲染到 Sandpack | 用户实时看到修改效果 |
| 保存（Merge） | 复制 `/sessions/{session_id}/` → `/demos/demo-1/`，删除 session 目录 | 将修改合并回原文件 |
| 放弃（Drop） | 直接删除 `/sessions/{session_id}/` | 零成本回滚，原文件不受影响 |

#### 3.2.3 Session 管理策略

| 策略 | 说明 |
|:-----|:-----|
| Session ID | 使用 UUID 或用户 ID + 时间戳生成唯一标识 |
| 存储位置 | `/sessions/{session_id}/` |
| 过期清理 | 用户关闭页面超时（如 2 小时无操作）自动销毁 session 目录 |
| 并发编辑 | 多用户同时编辑同一 Demo 时，各自在独立 session 目录下互不干扰，最后保存者覆盖 |

#### 3.2.4 方案优势

- **极其简单**：只需用到 Node.js 的文件系统操作（复制和删除）
- **完美契合 opencode**：AI 拥有绝对干净、独占的工作区，没有任何并发冲突
- **天然回滚**：用户取消编辑或超时，直接删除 session 目录，原文件完全不受影响
- **多人协作友好**：各用户在独立 session 中编辑，互不干扰

## 4. AI 隔离模块

### 4.1 权限配置设计

每个 session 创建时自动写入 `opencode.json`，实现白名单式权限控制：

| 权限类型 | 配置 | 说明 |
|:---------|:-----|:-----|
| edit | 白名单 | AI 只能修改指定文件（index.tsx、config.schema.json、AGENTS.md） |
| read | 全开 | AI 可以读取任何文件（包括其他 Demo），用于参考实现 |
| bash | 受限 | 禁止删除/移动/复制/创建目录等破坏性操作 |
| external_directory | 允许 | 允许读取 SDK 和其他 Demo |

### 4.2 行为约束设计

每个 session 创建时自动写入 `AGENTS.md`，约束 AI 行为：

- 明确告知 AI 当前工作目录
- 列出可修改和只读的文件范围
- 禁止破坏性操作
- 说明如何参考其他 Demo

### 4.3 后端校验机制

作为兜底防护，在 Next.js API 路由层增加文件变更校验：

| 校验项 | 处理方式 |
|:-------|:---------|
| 变更文件白名单校验 | 校验变更文件是否在白名单内 |
| 非法操作回滚 | 自动从原始 Demo 目录恢复文件 |
| 告警记录 | 记录告警日志，通知管理员 |

## 5. SDK 集成模块

### 5.1 问题背景

Sandpack 运行在浏览器中，无法直接访问本地磁盘的 SDK 包。

### 5.2 解决方案

将本地 SDK 代码作为 Demo 的一部分注入 Sandpack：

| 步骤 | 说明 |
|:-----|:-----|
| 读取 SDK 源码 | 页面加载时使用 Node.js fs 模块递归读取 SDK 目录 |
| 注入 Sandpack | 将 SDK 源码注入到 Sandpack 的 files 属性中 |
| 虚拟路径 | SDK 放到虚拟的 `/sdk/` 目录 |

### 5.3 优势

- **零维护成本**：SDK 更新后刷新页面即可生效，无需构建或发布
- **高灵活性**：AI 生成代码时使用相对路径，符合大模型直觉

### 5.4 注意事项

- 需过滤 SDK 目录中的非必要文件（测试文件、README 等）
- SDK 文件较大时会影响首屏加载速度

## 6. Schema 校验模块

### 6.1 问题背景

AI 修改代码时可能出现：

- 修改了 `index.tsx` 的 props 但忘记更新 `config.schema.json`
- 生成的 JSON Schema 格式损坏

### 6.2 校验规则

| 校验项 | 说明 |
|:-------|:-----|
| JSON Schema 语法 | 验证 JSON 格式是否合法 |
| Props 与 Properties 对应 | 组件 props 与 Schema properties 字段一致 |
| Required 字段存在性 | required 中的字段在 properties 中存在 |

### 6.3 校验时机

| 时机 | 处理方式 |
|:-----|:---------|
| 前端实时校验 | 用户编辑时即时反馈语法错误 |
| 保存时校验 | 校验失败则阻止保存，提示错误信息 |
| AI 修改后校验 | 校验失败则自动触发 AI 修复（最多重试次数） |

### 7.4 AI 编辑工作台模块

#### 7.4.1 布局设计

三栏布局：

| 区域 | 位置 | 功能 |
|:-----|:-----|:-----|
| 用户交互区 | 左侧 1/4 | AI 对话区 / 代码编辑区 Tab |
| 预览区 | 中间 1/2 | Sandpack 实时预览 |
| 配置面板 | 右侧 1/4 | 动态配置表单 |

#### 7.4.2 用户交互区设计

| Tab | 功能说明 |
|:-----|:---------|
| AI 对话区 | 用户输入自然语言指令，AI 通过 opencode server 修改当前 Demo 代码 |
| 代码编辑区 | 用户直接编辑 Demo 代码（统一分隔符格式） |

#### 7.4.3 代码编辑区设计

采用统一的分隔符格式展示，背后自动管理两个文件：

| 文件 | 说明 |
|:-----|:-----|
| index.tsx | React 组件代码 |
| config.schema.json | 配置 Schema |

**编辑区工作流程：**

| 操作 | 处理逻辑 |
|:-----|:---------|
| 从 Figma 粘贴 | 自动识别分隔符格式，无需额外处理 |
| 用户手动编辑 | 实时解析内容，分别更新到对应文件 |
| 保存时 | 按分隔符拆分，分别保存到对应文件 |
| 加载时 | 读取两个文件，合并为分隔符格式展示 |

**设计优势：**

- **统一体验**：编辑区与 Figma 导出格式一致，用户无需切换思维
- **一键粘贴**：从 Figma 复制后直接粘贴即可，无需手动拆分
- **直观编辑**：用户可以在同一视图中编辑代码和配置
- **自动同步**：编辑区内容与文件系统自动同步

### 7.5 AI 对话区工作原理

| 步骤 | 说明 |
|:-----|:-----|
| 1. 用户输入 | 用户输入自然语言指令（如"把 banner 改成轮播"） |
| 2. 请求转发 | Web 工作台通过 Next.js API 路由转发请求至 opencode server |
| 3. AI 处理 | 调用同步阻塞接口，AI 在当前 session 目录下读取并修改代码 |
| 4. 读取结果 | Web 工作台通过文件内容接口读取修改后的文件 |
| 5. 后端校验 | 后端校验文件变更合法性 |
| 6. 刷新预览 | 刷新预览区展示修改效果 |

**关键设计点：**

- 不使用 SSE 事件流：接口为同步阻塞调用
- Session 复用：进入编辑页时创建一次 opencode session，整个对话期间复用
- 消息历史：由 opencode server 自动维护

## 9. 数据规范模块

### 9.1 JSON Schema 扩展约定

| 扩展字段 | 说明 | 示例 |
|:---------|:-----|:-----|
| enumNames | 枚举值显示名称 | ["浅色", "深色"] |
| ui:widget | 表单控件类型 | file, richtext, color |
| ui:options | 控件额外配置 | { accept: "image/*" } |
| ui:options.preview | 预览面板尺寸配置 | { width, height, minHeight, maxHeight } |

### 9.2 表单控件映射

| JSON Schema 类型 | format / ui:widget | 表单控件 |
|:-----------------|:--------------------|:---------|
| string | - | 文本输入框 |
| string | uri + ui:widget: file | 文件上传 |
| string | uri | URL 输入框 |
| string | enum | 下拉选择框 |
| string | ui:widget: richtext | 富文本编辑器 |
| string | ui:widget: color | 颜色选择器 |
| number / integer | - | 数字输入框 |
| boolean | - | 开关 |
| array | - | 数组编辑器（动态增删） |
| object | - | 嵌套对象表单 |

### 9.3 代码与 Schema 的关联

- React 组件通过 props 接收配置值
- JSON Schema 的 properties 键名与组件 props 一一对应
- 工作台根据 Schema 生成默认值，传入组件渲染

## 10. 部署方案

### 10.1 环境变量

| 变量名 | 说明 | 必填 |
|:-------|:-----|:-----|
| OPENCODE_SERVER_URL | opencode server 地址 | 是 |
| DEMOS_DIR | Demo 存储目录路径 | 否（默认 `./demos`） |
| SESSIONS_DIR | Session 草稿目录路径 | 否（默认 `./sessions`） |

### 10.2 Docker 部署配置

部署时需配置以下挂载卷：

| 挂载路径 | 说明 |
|:---------|:-----|
| opencode-config | opencode 全局配置 |
| demos | Demo 存储目录 |
| sessions | 草稿工作区目录 |
| packages/shared | 公共 SDK |

**关键约束：**

- opencode serve 的工作目录必须是 sessions 根目录（而非单个 session）
- 每个 session 通过 `opencode.json` 实现权限隔离
- AI 可以读取所有 session 和 demo 目录（用于参考），但只能编辑白名单文件
