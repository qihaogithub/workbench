# 认证架构设计

## 1. 架构概览

用户鉴权模块采用**纯 JWT 无状态认证方案**，整个认证流程不依赖服务端会话存储，通过加密签名的方式验证用户身份。

### 1.1 核心设计理念

| 理念 | 说明 |
|:-----|:-----|
| **无状态** | 服务端不存储会话信息，所有认证状态通过 JWT Token 传递 |
| **自包含** | Token 本身包含用户身份信息，无需查询数据库即可验证 |
| **安全传输** | 使用 httpOnly Cookie 传输 Token，防止 XSS 攻击 |
| **自动过期** | Token 设置 7 天有效期，过期后需重新登录 |

## 2. 认证流程

### 2.1 注册流程

```
用户提交注册信息
  ↓
API 层验证用户名和密码格式
  ↓
检查用户名是否已存在
  ↓
使用 bcrypt 加密密码
  ↓
创建用户记录（生成 UUID）
  ↓
签发 JWT Token
  ↓
设置 httpOnly Cookie
  ↓
返回用户信息（自动登录）
```

**关键点**：
- 注册成功后立即签发 Token，实现自动登录
- 密码在入库前已完成加密，数据库中不存储明文密码
- 用户 ID 使用 UUID，避免自增 ID 带来的安全风险

### 2.2 登录流程

```
用户提交登录信息
  ↓
API 层验证输入非空
  ↓
根据用户名查询用户记录
  ↓
使用 bcrypt 验证密码
  ↓
签发 JWT Token
  ↓
设置 httpOnly Cookie
  ↓
返回用户信息
```

**关键点**：
- 密码验证使用 bcrypt.compare，自动处理加盐逻辑
- 登录失败时统一返回"用户名或密码错误"，不暴露具体原因
- Token 签发后立即设置到 Cookie，前端无需额外处理

### 2.3 身份验证流程

```
用户访问受保护资源
  ↓
中间件拦截请求
  ↓
从 Cookie 中提取 Token
  ↓
验证 Token 签名和有效期
  ↓
解析用户身份信息
  ↓
允许访问或重定向到登录页
```

**关键点**：
- 中间件在路由层面拦截，无需每个页面单独处理
- Token 验证失败时自动重定向到登录页
- 已登录用户访问登录页时自动重定向到首页

### 2.4 登出流程

```
用户点击登出按钮
  ↓
前端清除 Cookie（设置过期时间为过去）
  ↓
跳转到登录页
```

**关键点**：
- 纯 JWT 方案无需调用后端接口
- 通过设置 Cookie 过期时间实现登出
- 多设备登录时，登出仅影响当前设备

## 5. 安全机制

### 5.1 密码安全

| 机制 | 实现方式 | 作用 |
|:-----|:---------|:-----|
| 单向加密 | bcrypt.hash() | 密码不可逆，只能验证不能还原 |
| 自动加盐 | bcrypt 内置 | 每个密码使用不同的盐值 |
| 迭代加密 | 10 轮 | 增加破解难度 |

### 5.2 Token 安全

| 机制 | 实现方式 | 作用 |
|:-----|:---------|:-----|
| 签名验证 | HS256 算法 | 防止 Token 被篡改 |
| 有效期限制 | 7 天过期 | 限制 Token 有效时间 |
| httpOnly Cookie | 服务端设置 | 防止 XSS 攻击读取 Token |
| sameSite: lax | Cookie 属性 | 防止 CSRF 攻击 |

### 5.3 传输安全

| 机制 | 实现方式 | 作用 |
|:-----|:---------|:-----|
| HTTPS | 生产环境强制 | 防止中间人攻击 |
| Cookie secure 属性 | 生产环境启用 | 仅通过 HTTPS 传输 |

## 6. 技术选型理由

### 6.1 JWT vs Session

| 对比项 | JWT | Session |
|:-------|:-----|:---------|
| 服务端存储 | 不需要 | 需要存储会话数据 |
| 扩展性 | 天然支持分布式 | 需要会话共享机制 |
| 性能 | 无需查询数据库 | 需要查询会话存储 |
| 安全性 | Token 泄露风险 | Session ID 泄露风险 |
| 适用场景 | 无状态 API、微服务 | 传统 Web 应用 |

**选择 JWT 的原因**：
- 本项目为前后端分离架构，适合无状态认证
- 简化服务端实现，无需维护会话存储
- 支持多设备同时登录，无需额外处理

### 6.2 bcrypt vs 其他加密算法

| 算法 | 特点 |
|:-----|:-----|
| bcrypt | 专为密码设计，自动加盐，可调整计算成本 |
| SHA-256 | 通用哈希算法，需要手动加盐 |
| MD5 | 已不安全，不推荐使用 |

**选择 bcrypt 的原因**：
- 业界标准，专门用于密码存储
- 自动处理加盐，避免开发者错误
- 可调整计算成本，适应硬件性能提升

### 6.3 SQLite vs 其他数据库

| 数据库 | 特点 |
|:-------|:-----|
| SQLite | 轻量级，无需独立服务，适合小型应用 |
| PostgreSQL | 功能强大，适合大型应用 |
| MySQL | 流行度高，适合中型应用 |

**选择 SQLite 的原因**：
- 本项目用户量预期较小，SQLite 足够使用
- 无需部署独立的数据库服务，降低运维成本
- 数据存储在文件中，便于备份和迁移

## 7. 扩展性设计

### 7.1 支持多种认证方式

当前架构可扩展支持以下认证方式：

| 认证方式 | 扩展点 |
|:---------|:-------|
| 第三方登录 | 在用户表添加第三方账号关联字段 |
| 双因素认证 | 在登录流程中添加二次验证步骤 |
| API Token | 签发长期有效的 API Token |

### 7.2 支持权限系统

当前架构可扩展支持基于角色的权限控制：

| 扩展点 | 说明 |
|:-------|:-----|
| 用户表 | 添加 role 字段（admin、user 等） |
| 中间件 | 添加权限检查逻辑 |
| API 层 | 添加权限验证装饰器 |

## 8. 性能优化

### 8.1 Token 验证优化

| 优化措施 | 说明 |
|:---------|:-----|
| 无数据库查询 | Token 自包含用户信息，验证无需查询数据库 |
| 快速签名验证 | HS256 算法验证速度快 |

### 8.2 数据库优化

| 优化措施 | 说明 |
|:---------|:-----|
| WAL 模式 | 启用 Write-Ahead Logging，提升并发性能 |
| 索引优化 | 用户名字段建立唯一索引，加速查询 |
