## 一、项目定位

**Opencode Workbench** 是一个面向组件化开发的 **AI 辅助创作与使用平台**，提供从组件设计、AI 辅助编码、配置管理到预览嵌入的完整工作流。

### 1.1 核心价值

| 维度 | 说明 |
|:-----|:-----|
| **创作端** | 开发者通过 AI 对话生成/修改组件代码，实时预览效果，管理配置 Schema |
| **使用端** | 非技术人员浏览项目、调整配置参数、预览组件效果，无需编码 |
| **Agent 服务** | 独立的 AI 代理服务层，支持多后端切换（OpenCode、Claude、Codex 等） |
| **嵌入集成** | 通过 iframe + postMessage 协议，将组件嵌入外部系统使用 |

### 1.2 目标用户

| 角色 | 使用场景 | 使用端 |
|:-----|:---------|:-----|
| **开发者/设计师** | AI 辅助编码、配置定义、版本管理 | 创作端 |
| **产品/运营人员** | 浏览组件库、调整配置参数、预览效果 | 使用端 |
| **外部系统集成** | 通过 iframe 嵌入组件，动态传递配置 | 嵌入 API |

## 三、核心功能模块

### 3.1 用户鉴权（创作端）

**设计理念**：纯 JWT 无状态认证，安全简洁，多设备友好

| 功能 | 实现方式 |
|:-----|:---------|
| **注册/登录** | 用户名密码 → bcrypt 哈希 → SQLite 存储 |
| **Token 管理** | JWT (jose) → httpOnly Cookie → 7 天有效期 |
| **路由守卫** | Next.js middleware.ts → `/demo`, `/projects` 重定向到 `/login` |
| **会话管理** | 2 小时过期 (`SESSION_EXPIRY_MS`) |
| **多设备支持** | 无状态 Token，不限制登录设备数 |

**数据流**：
```
用户登录 → 验证密码 → 生成 JWT → 写入 httpOnly Cookie
         ↓
访问受保护路由 → middleware 验证 Token → 放行/重定向
```

### 3.2 项目管理（创作端）

**核心设计理念**：
```
"一个项目 = 一个工作空间，编辑在临时空间，保存才合并"
```

#### 3.2.1 核心概念

| 概念 | 说明 |
|:-----|:-----|
| **项目 (Project)** | 一个项目对应一个正式工作空间，持久化存储 |
| **工作空间 (Workspace)** | 文件状态的独立载体，可被多个 Session 共享 |
| **会话 (Session)** | AI 对话上下文，通过 `workspaceId` 关联工作空间，不直接持有文件 |
| **临时空间 (Temp Workspace)** | 编辑时的隔离空间，不影响原项目文件 |
| **版本快照 (Snapshot)** | 保存时创建的版本备份，支持回退（最多 50 个版本） |
| **代理配置 (.opencode)** | Workspace 中的 AI 代理配置，自动注入，保存时清理 |

#### 3.2.2 项目生命周期

```
创建项目 → 初始化正式工作空间
    ↓
打开编辑 → 创建临时工作空间 → 复制项目文件 → 注入 .opencode → 创建 Session
    ↓
AI 对话/手动编辑 → 修改临时空间文件
    ↓
保存变更 → 创建版本快照 → 临时空间覆盖正式空间 → 清理临时空间 → 记录版本
    ↓
恢复版本 → 备份当前状态 → 从快照恢复 → 创建新版本记录
```

#### 3.2.3 冲突处理策略

**后保存覆盖**：采用最简单策略，后保存覆盖先保存，但通过版本历史保障可回退

```
用户 A 保存 → 创建 v1
用户 B 保存 → 创建 v2（覆盖 A 的修改）
A 的修改可通过 v1 恢复
```

### 3.3 配置与预览系统（创作端 & 使用端）

#### 3.3.1 配置系统架构

```
Figma 导出文本 / AI 生成代码
    ↓
Schema 解析器 (parser.ts) → 提取代码 + JSON Schema
    ↓
表单生成器 (RJSF) → 根据 Schema 自动生成表单控件
    ↓
配置验证器 (validator.ts) → 实时验证配置值
    ↓
预览面板 → 配置值作为 Props 传入组件渲染
```

**技术选型**：
- **表单引擎**：react-jsonschema-form (RJSF)，JSON Schema 驱动
- **自定义控件**：颜色选择器、图片上传、富文本编辑器、代码编辑器
- **备注功能**：TipTap 富文本编辑器，备注存储在 `$demo.note` 字段

#### 3.3.2 实时预览机制

**创作端预览**（直接渲染）：

| 机制 | 说明 |
|:-----|:-----|
| **服务端编译** | `/api/compile` 将 TSX 编译为 CommonJS |
| **前端执行** | `new Function` 执行编译后代码，注入项目依赖 |
| **直接渲染** | `React.createElement` 直接渲染，无需 iframe |
| **编译缓存** | 相同代码不重复编译 |

**配置联动**：
```
用户修改配置表单 → ConfigForm.onChange(formData)
    ↓
父组件更新 configData 状态
    ↓
PreviewPanel 接收新 configData → React.createElement(compiledComponent, configData)
    ↓
预览区更新渲染（无需重新编译，延迟 < 16ms）
```

**AI 编辑快照驱动**（snapshotVersion）：

```
AI 编辑完成（SSE session.idle）
    ↓
useChatStream.onFinish → 获取完整文件快照（drain 机制保障）
    ↓
applyDemoSnapshot() 原子化更新：
  • code / schema / editorContent
  • 合并 configData 默认值
  • snapshotVersion++（驱动 PreviewPanel/ConfigForm 强制重渲染）
    ↓
PreviewPanel 编译 effect 触发 → compileCode → executeComponent → 渲染
ConfigForm key 变化 → 强制重新挂载 → 应用新 schema 默认值
```

**使用端预览**（iframe 渲染）：

```
使用端通过 iframe 嵌入创作端 /embed/{projectId}/{demoId}
    ↓
postMessage(UPDATE_CONFIG, configFormData) → 传递配置
    ↓
创作端接收配置 → 更新预览 → 实时反馈
```

#### 3.3.3 宫格预览（GridView）

用于同时展示多个 Demo 页面，采用 **iframe 渲染 + CSS Grid 布局**：

| 特性 | 实现 |
|:-----|:-----|
| **按行 Grid 渲染** | 列宽基于卡片宽高比动态计算（`gridTemplateColumns: r₁fr r₂fr`） |
| **垂直对齐** | ResizeObserver 监听容器高度，动态切换 `justify-content: center | flex-start` |
| **懒加载** | Intersection Observer 实现可见区域加载 |
| **活跃页面规则** | 活跃页面使用配置面板尺寸，非活跃页面使用自身尺寸 |

### 3.4 AI 对话系统（创作端）

#### 3.4.1 组件架构

```
编辑页（Demo/Project）
  │
  ├── 管理状态：sessionId, workspaceId, code, schema
  │
  └── AIChat（组合层，~190 行）
        │
        ├── Hooks 层
        │     ├── useChatMessages    → 消息/流式/当前消息状态
        │     ├── useChatStream      → 流式通信、事件处理、文件操作防抖
        │     └── useChatModels      → 模型列表获取、模型切换
        │
        ├── UI 子组件
        │     ├── ChatMessages       → 消息列表展示
        │     ├── ChatPlan           → Plan 折叠展示
        │     ├── ChatInput          → 输入区域（模型选择、附件、历史按钮）
        │     └── ModelSelectWithGuard → 模型选择器（含图片附件守卫）
        │
        ├── HistoryDialog           → 历史对话对话框
        └── PermissionDialog        → 权限确认对话框
```

#### 3.4.2 流式通信

**SSE / WebSocket 双模式**：

| 模式 | 场景 | 说明 |
|:-----|:-----|:-----|
| **SSE** | 实时推送 AI 响应 | Server-Sent Events，单向流式传输 |
| **WebSocket** | 双向通信、文件变更推送 | 支持消息发送、文件操作、权限请求 |

**Drain 机制**（保障文件快照完整性）：

```
AI 响应完成 → session.idle 事件
    ↓
等待 session.diff 事件（最多 2s）
    ↓
确保 finish 时文件快照完整 → 返回 result.files
    ↓
兜底：fetchSessionFiles() HTTP 获取
```

#### 3.4.3 工作空间与对话解耦

**核心原则**：切换对话不影响工作空间文件

| 操作 | 行为 |
|:-----|:-----|
| **新建对话** | 复用当前 workspaceId，仅更新 sessionId，不清空代码/Schema |
| **切换对话** | 仅切换 Session 元数据，加载消息历史，不影响 Workspace 文件 |
| **workspaceId 传递** | Session API → 编辑页 → AIChat → HistoryDialog 透传 |

### 3.5 独立 Agent 服务层

**设计理念**：
```
"Agent 服务层负责业务逻辑和 AI 交互，前端消费层只负责 UI 渲染和用户交互"
```

#### 3.5.1 架构模式

| 模式 | 应用 |
|:-----|:-----|
| **工厂模式** | AgentFactory 支持多种 AI 后端，运行时动态注册 |
| **观察者模式** | Agent 继承 EventEmitter，通过事件解耦（stream/error/finish） |
| **单例模式** | AgentManager 全局单例，管理所有 Agent 实例生命周期 |
| **策略模式** | 不同后端使用不同通信策略（HTTP API / stdio） |

#### 3.5.2 分层职责

| 层级 | 职责 | 关键模块 |
|:-----|:-----|:---------|
| **API Gateway** | 请求路由、认证、限流 | Fastify 路由、中间件 |
| **Agent Manager** | Agent 生命周期、Session 管理 | AgentManager、SessionStore |
| **Agent Factory** | Agent 创建、后端注册 | AgentFactory、BackendRegistry |
| **Backend Adapter** | 与具体 AI 后端通信 | OpenCodeBackend、ClaudeBackend |

#### 3.5.3 接口规范

**REST API**：

| 端点 | 方法 | 说明 |
|:-----|:-----|:-----|
| `/api/agent/:sessionId/message` | POST | 发送消息 |
| `/api/agent/:sessionId` | GET / DELETE | 获取状态 / 销毁 Agent |
| `/api/agent/:sessionId/files` | GET | 获取变更文件 |
| `/api/agent/:sessionId/rollback` | POST | 回滚文件变更 |
| `/api/sessions` | GET | 列出所有 Session |
| `/health` | GET | 健康检查 |

**WebSocket**：

```
ws://host/api/agent/:sessionId/stream

客户端发送：{ "type": "message", "content": "..." }
服务端推送：
  { "type": "stream", "content": "..." }
  { "type": "error", "message": "..." }
  { "type": "finish", "files": [...] }
```

#### 3.5.4 安全设计（三层防护）

| 层级 | 机制 | 强度 | 说明 |
|:-----|:-----|:-----|:-----|
| **L1** | 权限系统（opencode.json 白名单） | 强 | 配置文件级别的访问控制 |
| **L2** | 行为约束（AGENTS.md） | 软 | AI 行为规范和约束 |
| **L3** | 后端校验（文件变更检测 + 自动回滚） | 强 | 运行时校验和自动恢复 |

**L3 后端校验**：
- 定义允许修改的文件白名单
- 每次文件变更前校验路径是否在白名单内
- 违规时自动回滚并记录日志

### 3.6 嵌入 API（创作端）

**设计理念**：支持外部系统通过 iframe 嵌入组件，动态传递配置

#### 3.6.1 嵌入架构

```
外部系统
    ↓
<iframe src="http://author-site:3200/embed/{projectId}/{demoId}">
    ↓
创作端嵌入端点 → 加载组件代码 + Schema
    ↓
postMessage 通信协议：
  • UPDATE_CONFIG → 外部系统传递配置
  • READY → 嵌入端点就绪通知
  • ERROR → 错误信息推送
    ↓
实时更新预览 → 反馈给外部系统
```

| 端点 | CORS 配置 | 说明 |
|:-----|:---------|:-----|
| **Agent Service** | `.env` 的 `CORS_ORIGINS` | 控制跨域访问 |
| **创作端** | `middleware.ts` | author → viewer 跨域 |

### 4.2 核心数据结构

```typescript
interface Project {
  id: string                    // 项目唯一标识
  name: string                  // 项目名称
  workspacePath: string         // 正式工作空间绝对路径
  versions: VersionInfo[]       // 版本历史（最多 50 个）
  createdAt: number
  updatedAt: number
}
```

**EditSession（编辑会话）**：
```typescript
interface EditSession {
  sessionId: string             // 会话唯一标识
  projectId: string             // 关联的项目 ID
  workspaceId: string           // 关联的工作空间 ID
  username: string              // 当前编辑者用户名
  basedOnVersion: string        // 基于的版本号
  status: 'editing' | 'archived' | 'saved' | 'discarded'
  opencodeSessionId: string | null
  createdAt: number
  expiresAt: number             // 2 小时过期
}
```

**VersionInfo（版本信息）**：
```typescript
interface VersionInfo {
  versionId: string             // 版本号（"v1", "v2", ...）
  savedAt: number               // 保存时间戳
  savedBy: string               // 保存者用户名
  sessionId: string             // 关联的编辑会话 ID
  snapshotPath: string          // 快照备份绝对路径
  fileCount: number
  note?: string                 // 用户备注
}
```

## 五、关键技术与创新点

### 5.1 动态编译预览

**传统方案**：Sandpack / WebContainers（重、慢、资源占用高）

**本项目方案**：
```
TSX 代码 → /api/compile (sucrase) → CommonJS → new Function 执行 → React.createElement 渲染
```

| 优势 | 说明 |
|:-----|:-----|
| **轻量** | 无需完整构建工具链，仅编译 TSX 语法 |
| **快速** | 编译缓存 + 按需编译，首次加载 < 500ms |
| **灵活** | 可注入自定义依赖（项目组件、工具函数） |

### 5.2 SnapshotVersion 驱动机制

**问题**：AI 同时修改代码和 Schema 时，配置面板与预览区状态不同步

**解决方案**：
- `snapshotVersion` 作为 UI 更新的单一可信源
- `applyDemoSnapshot()` 原子化更新所有状态（code/schema/configData）
- 版本号递增驱动 PreviewPanel 重新编译、ConfigForm 强制重新挂载
- 配置合并策略：`mergeSchemaDefaults(existingConfig, newSchema)` 智能合并

### 5.3 Workspace 与 Session 解耦

**传统方案**：Session 直接持有文件，切换对话导致文件状态混乱

**本项目方案**：
- Workspace 是独立的文件载体
- 多个 Session 通过 `workspaceId` 共享同一 Workspace
- 切换对话仅切换 Session 元数据，不影响 Workspace 文件
- 新建对话复用 Workspace，保持代码/Schema 连续

### 5.4 Agent 多后端适配

**设计模式**：工厂模式 + 策略模式

```
AgentFactory.register('opencode-http', createOpenCodeAgent)
AgentFactory.register('claude', createClaudeAgent)
AgentFactory.register('codex', createCodexAgent)
    ↓
AgentFactory.create(config) → 根据 backend 类型创建对应 Agent 实例
    ↓
Agent 通过 IBackendAdapter 接口与后端通信（统一抽象）
```

**后端适配器**：
- OpenCode HTTP（默认后端，连接池优化）
- OpenCode ACP（兼容旧版，stdio 子进程）
- Claude API（stdio 通信）
- Codex API（stdio 通信）
- Gemini API（stdio 通信）
- Custom Backend（可扩展）

### 5.5 Drain 机制（SSE 时序竞争解决）

**问题**：`session.idle` 事件先于 `session.diff` 到达，导致 finish 时文件快照不完整

**解决方案**：
```
session.idle 到达 → 启动 drain 计时器（2s）
    ↓
等待 session.diff 到达 → 累积文件变更
    ↓
drain 超时或 diff 到达 → 返回完整 result.files
    ↓
兜底：HTTP 获取 session 文件
```

class CustomBackend extends BaseBackendAdapter {

## 十一、总结

Opencode Workbench 是一个面向组件化开发的 **AI 辅助创作与使用平台**，通过创作端、使用端、Agent 服务三层架构，实现了从组件设计、AI 辅助编码、配置管理到预览嵌入的完整工作流。

**核心设计理念**：
- 创作端：`"一个项目 = 一个工作空间，编辑在临时空间，保存才合并"`
- Agent 服务：`"Agent 服务层负责业务逻辑和 AI 交互，前端消费层只负责 UI 渲染和用户交互"`
- 使用端：`"只读浏览 + 配置调整，无 AI 对话、无代码编辑，开箱即用"`

**技术创新**：
- 动态编译预览（轻量快速，替代 Sandpack）
- SnapshotVersion 驱动机制（保障 AI 编辑后状态一致性）
- Workspace 与 Session 解耦（切换对话不影响文件状态）
- Agent 多后端适配（工厂模式 + 策略模式，支持扩展）
- Drain 机制（解决 SSE 时序竞争，保障文件快照完整性）

**适用场景**：组件库开发、低代码平台、营销页面搭建、第三方集成、设计系统管理。
