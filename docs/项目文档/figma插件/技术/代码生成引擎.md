# 代码生成引擎

> 负责将 Figma 节点树转换为可运行的代码（HTML 或 Tailwind/React）

---

## 架构概述

代码生成引擎采用**双引擎架构**，根据用户选择的框架输出不同格式的代码：

```
Figma 节点树
    │
    ▼
┌─────────────────┐
│  节点预处理层    │  ← 过滤不可见节点、处理 #prompt 等特殊标记
│ (code.ts)       │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐  ┌────────────┐
│ HTML   │  │ Tailwind   │  ← 并行双引擎
│ 引擎   │  │ /React 引擎 │
└───┬────┘  └─────┬──────┘
    │             │
    ▼             ▼
  HTML代码    React/Tailwind代码
```

---

## 核心流程

### 1. 入口调度 (code.ts)

`run()` 函数是整个代码生成的入口，负责协调各个阶段：

1. **节点选择**：获取 Figma 当前选中的图层
2. **节点转换**：将 Figma 原生节点转换为 AltNode JSON 格式
3. **代码生成**：调用 `convertToCode()` 生成目标代码
4. **预览生成**：生成 HTML 预览用于插件内展示

```
run(settings)
  ├── nodesToJSON(selection)     → 转换为可序列化的节点树
  ├── convertToCode(nodes)       → 生成代码
  └── generateHTMLPreview(nodes) → 生成预览
```

### 2. 节点转换层 (altNodes)

Figma 的节点对象包含循环引用（如 parent 指针），无法直接序列化。`nodesToJSON()` 负责：

- 提取节点的核心属性（位置、尺寸、样式等）
- 处理颜色变量和样式引用
- 递归处理子节点
- 生成纯 JSON 格式的 AltNode 树

### 3. 双引擎实现

#### HTML 引擎

输出纯 HTML + CSS 代码，适合快速预览和静态页面生成。

**核心组件**：
- `htmlMain.ts` - 入口，协调生成流程
- `htmlDefaultBuilder.ts` - 通用节点构建器
- `htmlTextBuilder.ts` - 文本节点专用构建器
- `builderImpl/*` - 各类样式实现（布局、颜色、阴影等）

**代码格式配置**：

通过 `tailwindGenerationMode` 设置控制输出格式：

| 模式 | 值 | 输出格式 | 用途 |
|------|-----|----------|------|
| React (JSX) | `jsx` | TSX 组件代码 | React 项目（默认） |
| HTML | `html` | 纯 HTML + 内联样式 | 静态页面预览 |

用户在插件的"更多"面板中可以切换代码格式。

#### Tailwind/React 引擎

输出 Tailwind CSS 类名的 React 组件代码，是 DSLP 的主要输出格式。

**核心组件**：
- `tailwindMain.ts` - 入口，包含 DSLP 标记拦截逻辑
- `tailwindDefaultBuilder.ts` - 通用节点构建器
- `tailwindTextBuilder.ts` - 文本节点构建器
- `builderImpl/*` - Tailwind 类名生成器

**DSLP 标记拦截**：

引擎在生成代码时会检查节点名称，识别特殊标记：

| 标记 | 处理方式 |
|------|----------|
| `#ignore` | 完全跳过该节点 |
| `#prompt` | 转换为 JSX 注释 `{/* AI Instruction: ... */}` |
| `#slot:type:id` | 替换为 `<SdkImage />` / `<SdkText />` / `<SdkVideo />` 组件 |
| `#list:id` | 替换为 `<SdkList />` 组件并递归渲染子节点 |
| `#static` | 导出为图片，渲染为 `<img>` 标签 |

## Props 自动生成（新增）

为了解决“插件输出代码无法直接驱动工作台配置面板”的断点，Tailwind/React 引擎新增了 Props 自动生成能力。

### 设计目标

- 插件在生成 JSX 的同时，自动生成 `interface Props`
- 每个字段附带元数据注释（`@title/@format/@widget/@group/@order`）
- AI 工作台可直接编译配置面板，减少对 AI 二次补全的依赖

### 关键实现

- `tailwindMain.ts`：在生成代码时创建 `PropsCollector`，收集 `#slot/#list` 字段并在输出顶部注入 `interface Props`
- `propsGenerator.ts`：负责字段名转换、分组推断、顺序分配、字段去重、注释生成
- 收集结果会输出日志（`[PropsCollector] collected props:`）用于排查和验收

### 字段映射规则

| Figma 标记 | Props 字段 | 元数据 |
|------------|------------|--------|
| `#slot:img:hero_banner` | `heroBanner: string` | `@format uri @widget image-upload` |
| `#slot:text:title` | `title: string` | `@format string @widget input` |
| `#slot:video:hero_video` | `heroVideo: string` | `@format uri @widget video-upload` |
| `#list:product_grid` | `productGrid: Array<object>` | `@title Product Grid 列表` |

### 分组规则

- 默认：使用最近的非标记父节点名称作为 `@group`
- 手动覆盖：支持语法 `#slot:img:banner[Banner区域]`（`#list` 同样支持）

### 向后兼容

- 若代码中缺失 `interface Props` 或元数据不完整，工作台仍按原有推断逻辑降级处理
- JSX 主体结构保持不变，Props 作为增强信息注入，不破坏既有渲染链路

### 运行注意事项（开发/验收）

- 插件主线程代码由 `apps/plugin/plugin-src/code.ts` 打包到 `apps/plugin/dist/code.js`。
- 若后端 `tailwindMain.ts` 逻辑已更新但 Figma 仍表现旧行为，通常是插件未重载最新产物。
- 推荐验收步骤：
  1. 运行 `pnpm --filter plugin build:main`
  2. 在 Figma 中 Reload 插件
  3. 再次生成并确认代码顶部出现 `interface Props`
  4. 控制台确认存在 `[PropsCollector] collected props:` 日志

## 关键算法

### 样式转换流程

```
Figma 节点
    │
    ├── 尺寸 (width/height) → Tailwind 类名 (w-10 h-20)
    ├── 位置 (x/y)          → absolute / relative 定位
    ├── 填充 (fills)        → bg-color / background-image
    ├── 边框 (strokes)      → border-color border-width
    ├── 圆角 (radius)       → rounded-lg
    ├── 阴影 (effects)      → shadow-md
    └── 布局 (layoutMode)   → flex / grid 布局
```

### 自动布局 (Auto Layout) 转换

Figma 的 Auto Layout 转换为 CSS Flexbox：

| Figma 属性 | CSS/Tailwind 映射 |
|------------|-------------------|
| `layoutMode: "HORIZONTAL"` | `flex flex-row` |
| `layoutMode: "VERTICAL"` | `flex flex-col` |
| `primaryAxisAlignItems: "CENTER"` | `justify-center` |
| `counterAxisAlignItems: "CENTER"` | `items-center` |
| `itemSpacing` | `gap-x` / `gap-y` |
| `paddingLeft/Right/Top/Bottom` | `pl/pr/pt/pb` |

## 性能优化

### 缓存机制

- **执行缓存**：`previousExecutionCache` 存储已处理的样式，避免重复计算
- **资源缓存**：上传后的图片资源通过 hash 缓存，避免重复上传

### 并发控制

- 资源上传使用队列 + 并发限制（默认 5 个并发）
- 防止大量图片同时导出导致内存溢出
