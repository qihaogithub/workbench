## 1. 业务背景与目标

### 1.1 痛点分析

传统设计交付中，UI 设计规范仅以文档和静态图形式存在。运营设计师在制作具体活动资源时，无法预知资源在真实代码环境（如手机端）下的渲染效果，导致：

1. **理解偏差**：无法直观感知交互逻辑和动态布局。
2. **返工率高**：资源上线后发现尺寸或遮挡问题，需重新设计。

### 1.2 系统目标

打造一套**"所见即所得"**的可配置 Demo 生成系统。

1. **设计侧**：通过 AI 将 Figma 稿自动转换代码，经过 AI 工作台完善为交互的 Demo。
2. **运营侧**：提供无需代码的 Web 配置台，支持实时替换素材、调整文案、预览动态效果并导出配置包。
3. **系统侧**：设计规范 CMS 和活动管理系统可通过接口调用 Demo，实现设计规范的直观展示和活动页面的快速配置。

## 2. 项目边界

| 模块 | 责任方 | 说明 |
| :--- | :--- | :--- |
| **Web 工作台** | 本项目 | AI 工作台前端界面开发 |

| **SDK** | 后续按需补充 | 提供给 AI 使用的组件库 |

## 5. Demo 存储结构

### 5.1 目录结构

每个 Demo 以独立目录形式存储，包含组件代码和配置 Schema 两个文件：

```

│   └── config.schema.json   # 配置 Schema
├── demo-2/
│   ├── index.tsx
│   └── config.schema.json
├── demo-3/
│   ├── index.tsx
│   └── config.schema.json
└── ...
```

### 5.2 文件说明

| 文件 | 说明 |
| :--- | :--- |
| `index.tsx` | 纯粹的 React 组件代码，通过 props 接收配置值 |
| `config.schema.json` | JSON Schema 格式的配置元数据定义 |

### 5.3 AI 编辑隔离机制

**核心原则：进入某 Demo 编辑页时，opencode server 仅能编辑当前 Demo 相关的代码。**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   用户进入 Demo-1 编辑页                                                      │
│                    ↓                                                        │
│   Web 工作台通知 opencode server                                             │
│                    ↓                                                        │
│   opencode server 工作目录切换至 demos/demo-1/                                │
│                    ↓                                                        │
│   AI 只能读写 demos/demo-1/ 下的文件                                         │
│   ├── index.tsx                                                             │
│   └── config.schema.json                                                    │
│                                                                             │
│   ✅ AI 可以修改：index.tsx、config.schema.json                               │
│   ❌ AI 无法访问：其他 Demo 目录、系统配置文件                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

这种隔离机制确保：
1. **安全性**：AI 不会误删或修改其他 Demo 的代码
2. **专注性**：AI 上下文仅限于当前 Demo，提高编辑准确性
3. **可追溯**：每个 Demo 的修改历史独立

## 8. 数据规范

### 8.1 Demo 文件格式

#### 8.1.1 index.tsx（React 组件代码）

纯粹的 React 组件代码，通过 props 接收配置值：

```tsx
import React from 'react';

interface DemoProps {
  banner: string;
  title: string;
  description: string;
}

export default function Demo({ banner, title, description }: DemoProps) {
  return (
    <div className="container">
      <img src={banner} alt="banner" />
      <h1>{title}</h1>
      <p>{description}</p>
    </div>
  );
}
```

```json
    "banner": {
      "type": "string",
      "format": "uri",
      "title": "顶部 Banner 图",
      "description": "建议尺寸: 750x400px",
      "default": "https://example.com/default-banner.png"
    },
    "title": {
      "type": "string",
      "title": "活动标题",
      "default": "活动标题",
      "maxLength": 20
    },
    "description": {
      "type": "string",
      "title": "活动描述",
      "default": "活动描述文案"
    },
    "countdown": {
      "type": "boolean",
      "title": "显示倒计时",
      "default": false
    },
    "theme": {
      "type": "string",
      "title": "主题颜色",
      "enum": ["light", "dark", "colorful"],
      "enumNames": ["浅色", "深色", "多彩"],
      "default": "light"
    }
  },
  "required": ["banner", "title"]
}
```

### 8.2 JSON Schema 扩展约定

为支持更丰富的表单控件，定义以下扩展字段：

| 扩展字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| `enumNames` | 枚举值显示名称，与 `enum` 一一对应 | `["浅色", "深色"]` |
| `ui:widget` | 指定表单控件类型 | `file`, `richtext`, `color` |
| `ui:options` | 控件额外配置 | `{ accept: "image/*" }` |

```json
{
  "banner": {
    "type": "string",
    "format": "uri",
    "title": "顶部 Banner 图",
    "ui:widget": "file",
    "ui:options": {
      "accept": "image/*"
    }
  }
}
```

### 8.3 支持的表单控件映射

| JSON Schema 类型 | format / ui:widget | 表单控件 |
| :--- | :--- | :--- |
| `string` | - | 文本输入框 |
| `string` | `uri` + `ui:widget: file` | 文件上传 |
| `string` | `uri` | URL 输入框 |
| `string` | `enum` | 下拉选择框 |
| `string` | `ui:widget: richtext` | 富文本编辑器 |
| `string` | `ui:widget: color` | 颜色选择器 |
| `number` / `integer` | - | 数字输入框 |
| `boolean` | - | 开关 |
| `array` | - | 数组编辑器（动态增删） |
| `object` | - | 嵌套对象表单 |

### 8.4 代码与 Schema 的关联

- React 组件通过 `props` 接收配置值
- JSON Schema 的 `properties` 键名与组件 `props` 一一对应
- 工作台根据 Schema 生成默认值，传入组件渲染

## 9. 项目架构

### 9.1 Monorepo 架构

项目采用 Monorepo 架构，便于统一管理多个子项目：

```
opencode工作台/
├── packages/
│   ├── web/                    # Web 工作台（本项目）
│   │   ├── src/
│   │   ├── package.json
│   │   └── ...
│   ├── figma-plugin/           # Figma 插件（后续接入）
│   │   ├── src/
│   │   ├── package.json
│   │   └── ...
│   └── shared/                 # 共享代码
│       ├── types/              # 共享类型定义
│       ├── utils/              # 共享工具函数
│       └── components/         # 共享组件
├── demos/                      # Demo 存储目录
├── package.json                # 根 package.json
├── pnpm-workspace.yaml         # pnpm 工作区配置
└── turbo.json                  # Turborepo 配置
```

### 9.2 Monorepo 优势

| 优势 | 说明 |
| :--- | :--- |
| **统一管理** | Web 工作台、Figma 插件、共享代码在同一仓库管理 |
| **代码复用** | 共享类型定义、工具函数、组件可跨项目复用 |
| **统一依赖** | 统一管理依赖版本，避免版本冲突 |
| **原子提交** | 跨项目修改可在一个 PR 中完成 |
| **简化协作** | 团队成员只需克隆一个仓库 |

### 9.3 子项目规划

| 子项目 | 状态 | 说明 |
| :--- | :--- | :--- |
| `packages/web` | ✅ 当前开发 | Web 工作台前端 |
| `packages/figma-plugin` | 🔜 后续接入 | Figma 插件，由其他项目组开发 |
| `packages/shared` | ✅ 当前开发 | 共享代码（类型、工具、组件） |
