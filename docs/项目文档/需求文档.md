# UI Demo 可配置生成系统 - 需求文档

## 1. 业务背景与目标

### 1.1 痛点分析

传统设计交付中，UI 设计规范仅以文档和静态图形式存在。运营设计师在制作活动资源时，无法预知资源在真实代码环境下的渲染效果，导致：

1. **理解偏差**：无法直观感知交互逻辑和动态布局
2. **返工率高**：资源上线后发现尺寸或遮挡问题，需重新设计

### 1.2 系统目标

打造一套**"所见即所得"**的可配置 Demo 生成系统：

1. **设计侧**：通过 AI 将 Figma 稿自动转换代码，经过 AI 工作台完善为交互 Demo
2. **运营侧**：提供无需代码的 Web 配置台，支持实时替换素材、调整文案、预览动态效果并导出配置包
3. **系统侧**：设计规范 CMS 和活动管理系统可通过接口调用 Demo，实现设计规范的直观展示和活动页面的快速配置

## 2. 项目边界

| :--- | :--- | :--- |
| **Web 工作台** | 本项目 | AI 工作台前端界面开发 |

## 5. Demo 存储结构

### 5.1 目录结构

每个 Demo 以独立目录形式存储，编辑会话存储在独立的 sessions 目录：

```
demos/                          # Demo 存储目录（正式版本）
├── demo-1/
│   ├── index.tsx            # React 组件代码
│   ├── config.schema.json   # 配置 Schema
│   └── AGENTS.md            # Demo 级规则（可选）
├── demo-2/
│   └── ...

sessions/                       # 草稿工作区目录（临时编辑）
├── session-uuid-1/          # 用户 A 的编辑会话
│   ├── index.tsx
│   ├── config.schema.json
│   └── AGENTS.md
├── session-uuid-2/          # 用户 B 的编辑会话
│   └── ...
```

### 5.2 文件说明

| 文件 | 说明 |
| :--- | :--- |
| `index.tsx` | React 组件代码，通过 props 接收配置值 |
| `config.schema.json` | JSON Schema 格式的配置元数据定义 |
| `AGENTS.md` | Demo 级规则，保存用户要求等记忆 |

### 5.3 AI 编辑隔离机制

**核心原则：单进程共享 + 权限白名单 + 后端兜底校验，实现安全隔离的同时允许跨 Demo 参考代码。**

隔离机制采用三层防护：

| 层级 | 机制 | 作用 | 强度 |
|------|------|------|------|
| **L1: 权限系统** | `opencode.json` 细粒度权限 | 物理上阻止 AI 编辑非目标文件 | 🔒 强 |
| **L2: AGENTS.md** | 行为约束指令 | 告诉 AI 不要尝试访问其他目录 | ⚠️ 软 |
| **L3: 后端校验** | 文件变更检测 + 自动回滚 | 检测并回滚非法修改 | 🔒 强 |

**设计决策：为什么不用多进程隔离？**

多进程方案（每个 Demo 启动独立 opencode 进程）虽然隔离性最强，但存在以下问题：
1. 资源开销大（每个进程 ~50-100MB 内存）
2. 进程管理复杂（端口分配、超时清理、异常恢复）
3. **AI 无法参考其他 Demo 的实现** — 这是设计师的重要需求

单进程方案通过 opencode 原生权限系统实现强约束：
- `edit: { "*": "deny", "index.tsx": "allow", "config.schema.json": "allow" }` — 白名单模式
- `read: { "*": "allow" }` — 允许读取所有文件（包括其他 Demo），供 AI 参考
- `bash: { "rm *": "deny", "mv *": "deny" }` — 禁止破坏性操作

隔离机制确保：
- **安全性**：AI 无法修改其他 Demo 的代码（权限系统强制拦截）
- **参考性**：AI 可以读取其他 Demo 的代码作为参考，提升生成质量
- **专注性**：AI 上下文仅限于当前 Demo 的编辑目标
- **可追溯**：每个 Session 的修改历史独立
- **兜底保障**：后端检测文件变更，非法修改自动回滚

### 5.4 草稿工作区（Workspace）

**核心原则：用户和 AI 不直接修改原始 Demo 文件，而是在独立的草稿工作区中编辑。**

当用户点击"编辑"时，系统为其创建一个临时副本，所有修改都在副本中进行，保存时才合并回原文件。

#### 5.4.1 工作流设计

| 阶段 | 操作 | 说明 |
|------|------|------|
| **进入编辑（Clone）** | 复制 `/demos/demo-1/` → `/sessions/{session_id}/` | 为当前编辑会话创建独立副本 |
| **注入配置（Config）** | 写入 `opencode.json`（权限白名单）+ `AGENTS.md`（行为约束） | 确保 AI 只能编辑当前 session 的目标文件，但可读取其他 Demo 参考 |
| **AI 隔离（Mount）** | 单进程 opencode serve 以 `/sessions` 为工作目录启动 | 所有 session 共享同一进程，通过权限系统实现隔离 |
| **实时预览（Sync）** | 前端通过 `/file/content` API 读取 session 目录代码渲染到 Sandpack | 用户实时看到修改效果 |
| **保存（Merge）** | 复制 `/sessions/{session_id}/` → `/demos/demo-1/`，删除 session 目录 | 将修改合并回原文件 |
| **放弃（Drop）** | 直接删除 `/sessions/{session_id}/` | 零成本回滚，原文件不受影响 |

#### 5.4.2 Session 管理

| 策略 | 说明 |
| :--- | :--- |
| Session ID | 使用 UUID 或用户 ID + 时间戳生成唯一标识 |
| 存储位置 | `/sessions/{session_id}/` |
| 过期清理 | 用户关闭页面超时（如 2 小时无操作）自动销毁 session 目录 |
| 并发编辑 | 多用户同时编辑同一 Demo 时，各自在独立 session 目录下互不干扰，最后保存者覆盖 |

#### 5.4.3 优势

- **极其简单**：只需用到 Node.js 的 `fs.cpSync`（复制）和 `fs.rmSync`（删除）
- **完美契合 opencode**：AI 拥有绝对干净、独占的工作区，没有任何并发冲突
- **天然回滚**：用户取消编辑或超时，直接删除 session 目录，原文件完全不受影响
- **多人协作友好**：各用户在独立 session 中编辑，互不干扰

### 5.5 OpenCode 配置方案

#### 5.5.1 设计原则

**单进程共享架构：** 所有 Session 共享一个 opencode serve 进程，通过权限系统实现隔离。

- 工作目录：`/sessions`（所有 session 的父目录）
- 每个 session 目录包含独立的 `opencode.json`（权限配置）和 `AGENTS.md`（行为约束）
- 通过 `external_directory` 允许 AI 访问公共资源（如 SDK）和其他 Demo（只读参考）

**配置优先级：** Session 规则 > Demo 规则 > 全局规则 > 全局配置

#### 5.5.2 配置目录结构

```
opencode工作台/
├── demos/                          # Demo 存储目录
├── sessions/                       # 草稿工作区（所有 session 共享）
│   ├── session-uuid-1/
│   │   ├── index.tsx
│   │   ├── config.schema.json
│   │   ├── AGENTS.md               # Session 级行为约束
│   │   └── opencode.json           # Session 级权限配置（动态注入）
│   └── session-uuid-2/
│       └── ...
├── packages/
│   ├── web/                        # Web 工作台前端
│   └── shared/sdk/                 # 公共 SDK，AI 可访问
└── opencode-config/                # 全局配置目录
    ├── opencode.json               # 全局配置（模型、MCP 等）
    ├── agents/demo-engineer.md     # AI 代理配置
    └── AGENTS.md                   # 全局规则
```

#### 5.5.3 Docker 部署配置

```yaml
services:
  opencode-server:
    image: opencode/opencode
    volumes:
      - ./opencode-config:/root/.config/opencode
      - ./demos:/app/demos
      - ./sessions:/app/sessions
      - ./packages/shared:/app/shared
    ports:
      - "4096:4096"
    working_dir: /app/sessions      # 工作目录为 sessions 根目录
    command: ["opencode", "serve", "--hostname", "0.0.0.0"]
```

#### 5.5.4 Session 级权限配置（动态注入）

每个 session 创建时自动写入 `opencode.json`，实现白名单式权限控制：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "edit": {
      "*": "deny",
      "index.tsx": "allow",
      "config.schema.json": "allow",
      "AGENTS.md": "allow"
    },
    "read": {
      "*": "allow",
      "*.env": "deny",
      "*.env.*": "deny"
    },
    "bash": {
      "*": "ask",
      "ls *": "allow",
      "cat *": "allow",
      "grep *": "allow",
      "rm *": "deny",
      "mv *": "deny",
      "cp *": "deny",
      "mkdir *": "deny"
    },
    "external_directory": {
      "**/packages/shared/sdk/**": "allow",
      "**/demos/**": "allow"
    }
  }
}
```

**权限说明：**
- `edit` 白名单：AI 只能修改 `index.tsx`、`config.schema.json`、`AGENTS.md`
- `read` 全开：AI 可以读取任何文件（包括其他 Demo），用于参考实现
- `bash` 受限：禁止删除/移动/复制/创建目录等破坏性操作
- `external_directory`：允许读取 SDK 和其他 Demo

#### 5.5.5 Session 级 AGENTS.md（动态注入）

每个 session 创建时自动写入 `AGENTS.md`，约束 AI 行为：

```markdown
# UI Demo 工程师

## 当前工作目录
你正在编辑的 Demo 位于当前 session 目录。

## 核心约束
1. **你只能修改以下文件：**
   - `index.tsx` — React 组件代码
   - `config.schema.json` — 配置 Schema
   - `AGENTS.md` — 本规则文件

2. **你可以读取但不能修改：**
   - 其他 Demo 目录下的文件（`../demos/*/`）— 用于参考实现
   - SDK 目录（`../../packages/shared/sdk/`）— 用于了解组件 API

3. **禁止操作：**
   - 禁止删除任何文件
   - 禁止创建新文件或目录
   - 禁止运行 `rm`、`mv`、`cp`、`mkdir` 等命令
   - 禁止修改其他 Demo 的任何文件

## 参考其他 Demo
如果需要参考其他 Demo 的实现，可以：
1. 使用 `read` 工具读取其他 Demo 的 `index.tsx`
2. 学习其实现后，在当前 Demo 的 `index.tsx` 中应用
3. **不要复制粘贴其他 Demo 的代码**，而是理解后重写
```

#### 5.5.6 AI 代理配置

全局代理 `demo-engineer.md` 定义专门负责 Demo 代码生成的 AI 代理，包含：
- 角色定义：UI Demo 工程师
- 核心职责：生成/修改 React 组件和 JSON Schema
- 技术栈：React 18 + TypeScript + Tailwind CSS
- 工作约束：仅修改当前 session 目录下的目标文件（`index.tsx`、`config.schema.json`）

### 5.6 SDK 集成方案

**问题：** Sandpack 运行在浏览器中，无法直接访问本地磁盘的 SDK 包。

**解决方案：** 将本地 SDK 代码作为 Demo 的一部分注入 Sandpack。

**实现流程：**

1. 页面加载时（`getServerSideProps` 或 Server Component），使用 Node.js `fs` 模块递归读取 `packages/shared/sdk` 目录
2. 将 SDK 源码注入到 Sandpack 的 `files` 属性中，放到虚拟的 `/sdk/` 目录
3. AI 生成的代码通过相对路径 `import { xxx } from './sdk'` 引用

**优势：**
- **零维护成本**：SDK 更新后刷新页面即可生效，无需构建或发布
- **高灵活性**：AI 生成代码时使用相对路径，符合大模型直觉

**注意事项：**
- 需过滤 SDK 目录中的非必要文件（测试文件、README 等）
- SDK 文件较大时会影响首屏加载速度

### 5.7 Schema 与 Code 一致性校验

**问题：** AI 修改代码时可能出现：
1. 修改了 `index.tsx` 的 props 但忘记更新 `config.schema.json`
2. 生成的 JSON Schema 格式损坏

**解决方案：** 在 Web 工作台增加校验层。

**校验规则：**

| 校验项 | 说明 |
|--------|------|
| JSON Schema 语法 | 验证 JSON 格式是否合法 |
| Props 与 Properties 对应 | 组件 props 与 Schema properties 字段一致 |
| Required 字段存在性 | required 中的字段在 properties 中存在 |

**校验时机：**

| 时机 | 处理方式 |
|------|----------|
| 前端实时校验 | 用户编辑时即时反馈语法错误 |
| 保存时校验 | 校验失败则阻止保存，提示错误信息 |
| AI 修改后校验 | 校验失败则自动触发 AI 修复（最多重试 3 次） |

### 5.8 后端文件变更校验（兜底防护）

**问题：** 即使有权限系统约束，仍需防范权限配置被 AI 篡改或权限系统本身存在 bug。

**解决方案：** 在 Next.js API 路由层增加文件变更校验。

**校验流程：**
1. AI 完成修改后，后端通过 `GET /file/status` 获取变更文件列表
2. 校验变更文件是否在白名单内（`index.tsx`、`config.schema.json`、`AGENTS.md`）
3. 如果发现非法变更，自动从原始 Demo 目录恢复文件
4. 记录告警日志，通知管理员

**白名单文件：**
- `index.tsx` — React 组件代码
- `config.schema.json` — 配置 Schema
- `AGENTS.md` — 行为规则

**非法操作处理：**
- 如果 AI 创建了白名单外的新文件 → 自动删除
- 如果 AI 修改了其他 Demo 的文件 → 自动从原始 Demo 恢复
- 如果 AI 删除了文件 → 自动从原始 Demo 恢复

### 6.4 AI 编辑工作台

**布局：** 三栏布局
- 左侧 1/4：用户交互区（AI 对话区 Tab / 代码编辑区 Tab）
- 中间 1/2：Demo 预览区（Sandpack 实时预览）
- 右侧 1/4：配置面板（动态配置表单）

**用户交互区功能：**

| Tab | 功能说明 |
| :--- | :--- |
| AI 对话区 | 用户输入自然语言指令，AI 通过 opencode server 修改当前 Demo 代码 |
| 代码编辑区 | 用户直接编辑 Demo 代码（统一分隔符格式） |

**代码编辑区设计：**

代码编辑区采用统一的分隔符格式展示，用户看到的是一个完整的编辑器，但背后自动管理两个文件（`index.tsx` 和 `config.schema.json`）。

**编辑器内容示例：**

```
=== DEMO CODE ===
import React from 'react';

export default function Demo({ title }) {
  return <h1>{title}</h1>;
}
=== DEMO SCHEMA ===
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Demo 配置",
  "type": "object",
  "properties": {
    "title": { "type": "string", "title": "标题" }
  }
}
=== END ===
```

**编辑区工作流程：**

| 操作 | 处理逻辑 |
| :--- | :--- |
| 从 Figma 粘贴 | 自动识别分隔符格式，无需额外处理 |
| 用户手动编辑 | 实时解析内容，分别更新到对应文件 |
| 保存时 | 按分隔符拆分，分别保存 `index.tsx` 和 `config.schema.json` |
| 加载时 | 读取两个文件，合并为分隔符格式展示 |

**优势：**

- **统一体验**：编辑区与 Figma 导出格式一致，用户无需切换思维
- **一键粘贴**：从 Figma 复制后直接粘贴即可，无需手动拆分
- **直观编辑**：用户可以在同一视图中编辑代码和配置
- **自动同步**：编辑区内容与文件系统自动同步

**AI 对话区工作原理：**

1. 用户输入自然语言指令（如"把 banner 改成轮播"）
2. Web 工作台通过 Next.js API 路由转发请求至 opencode server
3. 调用 `POST /session/{id}/message`（同步阻塞调用，直接返回 AI 回复）
4. AI 在当前 session 目录下读取并修改代码（受权限白名单约束）
5. Web 工作台通过 `GET /file/content` 读取修改后的文件
6. 后端校验文件变更合法性，刷新预览区

**关键 API 调用方式：**
- **不使用 SSE 事件流**：`/session/{id}/message` 是同步阻塞调用，直接返回 `{ info, parts }`
- **不使用 `/session/{id}/event`**：该端点不存在于官方 API 中
- **Session 复用**：进入编辑页时创建一次 opencode session，整个对话期间复用
- **消息历史**：由 opencode server 自动维护，前端无需传递完整历史

### 6.5 历史记录功能

**功能说明：**

| 功能 | 说明 |
| :--- | :--- |
| 自动保存 | 每次代码变更自动创建历史记录快照 |
| 撤销/重做 | 快捷键 `Ctrl+Z` / `Ctrl+Y` |
| 历史浏览 | 点击"历史"按钮查看完整历史记录列表 |
| 版本回退 | 点击历史记录项，回退到指定版本 |
| 历史清理 | 自动清理超过 30 天的历史记录 |

**历史记录触发时机：**

| 触发场景 | 说明 |
| :--- | :--- |
| AI 修改 | AI 完成一次代码修改后自动保存快照 |
| 手动编辑 | 用户在代码编辑区完成编辑后自动保存快照 |
| 手动保存 | 用户点击"保存"按钮时创建快照 |
| 定时保存 | 每隔 5 分钟自动保存一次（可配置） |

### 6.6 核心使用流程

**UI 设计师创建 Demo：**

1. 首页点击"新建 Demo" → 系统创建新目录
2. 进入 AI 编辑工作台 → 后端克隆 Demo 到 session 目录，注入权限配置

8. 满意后点击"保存"，后端合并 session 到 Demo 目录并清理

**运营设计师使用 Demo：**

1. 首页找到目标 Demo，点击"使用"
2. 进入 Demo 使用页面
3. 在配置面板修改配置项，预览区实时展示效果
4. 满意后导出配置包

## 7. 数据规范

### 7.1 Demo 文件格式

#### index.tsx（React 组件代码）

```tsx
import React from 'react';

interface DemoProps {
  banner: string;
  title: string;
  description: string;
}

export default function Demo({ banner, title, description }: DemoProps) {
  return (
    <div className="container">
      <img src={banner} alt="banner" />
      <h1>{title}</h1>
      <p>{description}</p>
    </div>
  );
}
```

#### config.schema.json（配置 Schema）

使用行业标准 [JSON Schema](https://json-schema.org/) 定义配置项：

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "活动 Demo 配置",
  "type": "object",
  "properties": {
    "banner": {
      "type": "string",
      "format": "uri",
      "title": "顶部 Banner 图",
      "description": "建议尺寸: 750x400px",
      "default": "https://example.com/default-banner.png"
    },
    "title": {
      "type": "string",
      "title": "活动标题",
      "default": "活动标题",
      "maxLength": 20
    },
    "theme": {
      "type": "string",
      "title": "主题颜色",
      "enum": ["light", "dark", "colorful"],
      "enumNames": ["浅色", "深色", "多彩"],
      "default": "light"
    }
  },
  "required": ["banner", "title"]
}
```

### 7.2 JSON Schema 扩展约定

| 扩展字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| `enumNames` | 枚举值显示名称，与 `enum` 一一对应 | `["浅色", "深色"]` |
| `ui:widget` | 指定表单控件类型 | `file`, `richtext`, `color` |
| `ui:options` | 控件额外配置 | `{ accept: "image/*" }` |

### 7.3 表单控件映射

| JSON Schema 类型 | format / ui:widget | 表单控件 |
| :--- | :--- | :--- |
| `string` | - | 文本输入框 |
| `string` | `uri` + `ui:widget: file` | 文件上传 |
| `string` | `uri` | URL 输入框 |
| `string` | `enum` | 下拉选择框 |
| `string` | `ui:widget: richtext` | 富文本编辑器 |
| `string` | `ui:widget: color` | 颜色选择器 |
| `number` / `integer` | - | 数字输入框 |
| `boolean` | - | 开关 |
| `array` | - | 数组编辑器（动态增删） |
| `object` | - | 嵌套对象表单 |

### 7.4 代码与 Schema 的关联

- React 组件通过 `props` 接收配置值
- JSON Schema 的 `properties` 键名与组件 `props` 一一对应
- 工作台根据 Schema 生成默认值，传入组件渲染

### 7.5 Figma 导出格式

Figma 插件导出时，将 React 组件代码和配置 Schema 使用分隔符封装成单一文本，方便用户一键复制粘贴到工作台。

#### 导出格式示例

```
=== DEMO CODE ===
import React from 'react';

interface DemoProps {
  title: string;
  description: string;
}

export default function Demo({ title, description }: DemoProps) {
  return (
    <div className="container">
      <h1>{title}</h1>
      <p>{description}</p>
    </div>
  );
}
=== DEMO SCHEMA ===
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Demo 配置",
  "type": "object",
  "properties": {
    "title": {
      "type": "string",
      "title": "标题",
      "default": "Hello World"
    },
    "description": {
      "type": "string",
      "title": "描述"
    }
  },
  "required": ["title"]
}
=== END ===
```

#### 分隔符规范

| 分隔符 | 说明 |
| :--- | :--- |
| `=== DEMO CODE ===` | React 组件代码开始标记 |
| `=== DEMO SCHEMA ===` | JSON Schema 配置开始标记 |
| `=== END ===` | 内容结束标记（可选，用于校验完整性） |

## 10. 部署方案

### 10.1 环境变量

| 变量名 | 说明 | 必填 |
|--------|------|------|
| `OPENCODE_SERVER_URL` | opencode server 地址 | 是 |
| `DEMOS_DIR` | Demo 存储目录路径 | 否（默认 `./demos`） |
| `SESSIONS_DIR` | Session 草稿目录路径 | 否（默认 `./sessions`） |

### 10.2 opencode server 启动方式

```bash
# 以 sessions 目录为工作目录启动
opencode serve --hostname 0.0.0.0 --port 4096
# 工作目录 = sessions 根目录（所有 session 的父目录）
```

**关键约束：**
- opencode serve 的工作目录必须是 sessions 根目录（而非单个 session）
- 每个 session 通过 `opencode.json` 实现权限隔离
- AI 可以读取所有 session 和 demo 目录（用于参考），但只能编辑白名单文件
