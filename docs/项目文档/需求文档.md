# UI Demo 可配置生成系统 - 需求文档

## 1. 业务背景与目标

### 1.1 痛点分析

传统设计交付中，UI 设计规范仅以文档和静态图形式存在。运营设计师在制作活动资源时，无法预知资源在真实代码环境下的渲染效果，导致：

1. **理解偏差**：无法直观感知交互逻辑和动态布局
2. **返工率高**：资源上线后发现尺寸或遮挡问题，需重新设计

### 1.2 系统目标

打造一套**"所见即所得"**的可配置 Demo 生成系统：

1. **设计侧**：通过 AI 将 Figma 稿自动转换代码，经过 AI 工作台完善为交互 Demo
2. **运营侧**：提供无需代码的 Web 配置台，支持实时替换素材、调整文案、预览动态效果并导出配置包
3. **系统侧**：设计规范 CMS 和活动管理系统可通过接口调用 Demo，实现设计规范的直观展示和活动页面的快速配置

## 2. 项目边界

| 模块 | 责任方 | 说明 |
| :--- | :--- | :--- |
| **Web 工作台** | 本项目 | AI 工作台前端界面开发 |
| **Figma 插件** | 其他项目组 | 导出 React 代码 + Schema，需对接本项目定义的 Schema 格式 |
| **SDK** | 后续按需补充 | 提供给 AI 使用的组件库 |

## 3. 用户角色

| 角色 | 职责描述 | 核心诉求 |
| :--- | :--- | :--- |
| **UI 设计师** | Demo 生产者 | 仅需简单标记 Figma 图层，即可生成高还原度 Demo，无需编写代码 |
| **运营设计师** | Demo 使用者 | 在 Web 端配置图片/文案/视频，实时预览效果，导出配置包交付上线 |
| **研发/管理员** | 系统维护者 | 维护通用 SDK 组件库，监控 AI 生成质量，管理活动 Demo 库 |

## 4. 系统架构

### 4.1 核心理念

**"插件产出骨架，AI 注入灵魂，SDK 托底能力"**

### 4.2 模块说明

| 模块 | 说明 |
| :--- | :--- |
| **Figma 插件** | 负责导出高还原度的 React 组件代码 + 配置 Schema |
| **用户** | 作为搬运工和决策者，将 React 代码粘贴至 AI 工作台，并可输入自然语言指令 |
| **AI Agent** | 利用 opencode server 作为 AI 引擎，本项目为其开发定制的前端界面 |
| **Web 工作台** | 包含首页（Demo 管理）、Demo 使用页面、AI 编辑工作台 |
| **SDK** | 提供现成的组件与能力底座，降低 AI 生成代码难度（后续按需补充） |

## 5. Demo 存储结构

### 5.1 目录结构

每个 Demo 以独立目录形式存储：

```
demos/
├── demo-1/
│   ├── index.tsx            # React 组件代码
│   ├── config.schema.json   # 配置 Schema
│   └── AGENTS.md            # Demo 级规则（可选）
├── demo-2/
│   └── ...
```

### 5.2 文件说明

| 文件 | 说明 |
| :--- | :--- |
| `index.tsx` | React 组件代码，通过 props 接收配置值 |
| `config.schema.json` | JSON Schema 格式的配置元数据定义 |
| `AGENTS.md` | Demo 级规则，保存用户要求等记忆 |

### 5.3 AI 编辑隔离机制

**核心原则：进入某 Demo 编辑页时，opencode server 仅能编辑当前 Demo 相关的代码。**

隔离机制确保：
- **安全性**：AI 不会误删或修改其他 Demo 的代码
- **专注性**：AI 上下文仅限于当前 Demo，提高编辑准确性
- **可追溯**：每个 Demo 的修改历史独立

### 5.4 OpenCode 配置方案

#### 5.4.1 设计原则

每个 Demo 目录作为独立的项目级别，包含自己的 `AGENTS.md`，通过 `external_directory` 权限允许 AI 访问公共资源（如 SDK）。

**配置优先级：** Demo 规则 > 全局规则 > 全局配置

#### 5.4.2 配置目录结构

```
opencode工作台/
├── demos/                          # Demo 存储目录
├── packages/
│   ├── web/                        # Web 工作台前端
│   └── shared/sdk/                 # 公共 SDK，AI 可访问
└── opencode-config/                # 全局配置目录
    ├── opencode.json               # 全局配置（模型、MCP 等）
    ├── agents/demo-engineer.md     # AI 代理配置
    └── AGENTS.md                   # 全局规则
```

#### 5.4.3 Docker 部署配置

```yaml
services:
  opencode-server:
    image: opencode/opencode
    volumes:
      - ./opencode-config:/root/.config/opencode
      - ./demos:/app/demos
      - ./packages/shared:/app/shared
    ports:
      - "4096:4096"
```

#### 5.4.4 全局配置示例

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "edit": "auto",
    "bash": "deny",
    "external_directory": {
      "/app/shared/sdk": "allow"
    }
  }
}
```

#### 5.4.5 AI 代理配置

全局代理 `demo-engineer.md` 定义专门负责 Demo 代码生成的 AI 代理，包含：
- 角色定义：UI Demo 工程师
- 核心职责：生成/修改 React 组件和 JSON Schema
- 技术栈：React 18 + TypeScript + Tailwind CSS
- 工作约束：仅修改当前 Demo 目录下的文件

## 6. 页面设计

### 6.1 页面导航关系

| 页面 | 路由 | 用户 | 功能说明 |
| :--- | :--- | :--- | :--- |
| **首页** | `/` | 所有用户 | Demo 列表管理，支持创建、删除、搜索 Demo |
| **Demo 使用页面** | `/demo/[id]` | 运营设计师 | 预览区 + 配置面板，用于配置和测试 Demo |
| **AI 编辑工作台** | `/demo/[id]/edit` | UI 设计师 | 三栏布局：AI 对话区 + 预览区 + 配置面板 |

### 6.2 首页（Demo 管理页面）

**功能清单：**

| 功能 | 说明 |
| :--- | :--- |
| Demo 列表 | 卡片形式展示所有 Demo，显示预览图、名称、更新时间 |
| 搜索 | 按 Demo 名称搜索 |
| 新建 Demo | 创建空白 Demo 目录，跳转至 AI 编辑工作台 |
| 使用 Demo | 跳转至 Demo 使用页面 |
| 编辑 Demo | 跳转至 AI 编辑工作台 |

### 6.3 Demo 使用页面

**布局：** 左侧 2/3 为预览区（Sandpack 实时渲染），右侧 1/3 为配置面板

**功能清单：**

| 功能 | 说明 |
| :--- | :--- |
| 预览区 | Sandpack 实时渲染 Demo，响应配置变更 |
| 配置面板 | 根据 JSON Schema 自动生成的表单 |
| 配置联动 | 修改配置项实时更新预览区 |
| 编辑入口 | 点击"编辑"跳转至 AI 编辑工作台 |

### 6.4 AI 编辑工作台

**布局：** 三栏布局
- 左侧 1/4：用户交互区（AI 对话区 Tab / 代码编辑区 Tab）
- 中间 1/2：Demo 预览区（Sandpack 实时预览）
- 右侧 1/4：配置面板（动态配置表单）

**用户交互区功能：**

| Tab | 功能说明 |
| :--- | :--- |
| AI 对话区 | 用户输入自然语言指令，AI 通过 opencode server 修改当前 Demo 代码 |
| 代码编辑区 | 用户直接编辑 Demo 代码（统一分隔符格式） |

**代码编辑区设计：**

代码编辑区采用统一的分隔符格式展示，用户看到的是一个完整的编辑器，但背后自动管理两个文件（`index.tsx` 和 `config.schema.json`）。

**编辑器内容示例：**

```
=== DEMO CODE ===
import React from 'react';

export default function Demo({ title }) {
  return <h1>{title}</h1>;
}
=== DEMO SCHEMA ===
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Demo 配置",
  "type": "object",
  "properties": {
    "title": { "type": "string", "title": "标题" }
  }
}
=== END ===
```

**编辑区工作流程：**

| 操作 | 处理逻辑 |
| :--- | :--- |
| 从 Figma 粘贴 | 自动识别分隔符格式，无需额外处理 |
| 用户手动编辑 | 实时解析内容，分别更新到对应文件 |
| 保存时 | 按分隔符拆分，分别保存 `index.tsx` 和 `config.schema.json` |
| 加载时 | 读取两个文件，合并为分隔符格式展示 |

**优势：**

- **统一体验**：编辑区与 Figma 导出格式一致，用户无需切换思维
- **一键粘贴**：从 Figma 复制后直接粘贴即可，无需手动拆分
- **直观编辑**：用户可以在同一视图中编辑代码和配置
- **自动同步**：编辑区内容与文件系统自动同步

**AI 对话区工作原理：**

1. 用户输入自然语言指令（如"把 banner 改成轮播"）
2. Web 工作台发送请求至 opencode server
3. AI 在当前 Demo 目录下读取并修改代码
4. Web 工作台检测文件变更，刷新预览区

### 6.5 历史记录功能

**功能说明：**

| 功能 | 说明 |
| :--- | :--- |
| 自动保存 | 每次代码变更自动创建历史记录快照 |
| 撤销/重做 | 快捷键 `Ctrl+Z` / `Ctrl+Y` |
| 历史浏览 | 点击"历史"按钮查看完整历史记录列表 |
| 版本回退 | 点击历史记录项，回退到指定版本 |
| 历史清理 | 自动清理超过 30 天的历史记录 |

**历史记录触发时机：**

| 触发场景 | 说明 |
| :--- | :--- |
| AI 修改 | AI 完成一次代码修改后自动保存快照 |
| 手动编辑 | 用户在代码编辑区完成编辑后自动保存快照 |
| 手动保存 | 用户点击"保存"按钮时创建快照 |
| 定时保存 | 每隔 5 分钟自动保存一次（可配置） |

### 6.6 核心使用流程

**UI 设计师创建 Demo：**

1. 首页点击"新建 Demo" → 系统创建新目录
2. 进入 AI 编辑工作台，opencode server 工作目录切换
3. 从 Figma 插件复制分隔符格式代码，粘贴到代码编辑区
4. 系统自动解析并创建 `index.tsx` 和 `config.schema.json`
5. 用户通过 AI 对话或直接编辑修改代码
6. 在配置面板测试配置项，预览区实时展示效果
7. 满意后点击"保存"，返回首页

**运营设计师使用 Demo：**

1. 首页找到目标 Demo，点击"使用"
2. 进入 Demo 使用页面
3. 在配置面板修改配置项，预览区实时展示效果
4. 满意后导出配置包

## 7. 数据规范

### 7.1 Demo 文件格式

#### index.tsx（React 组件代码）

```tsx
import React from 'react';

interface DemoProps {
  banner: string;
  title: string;
  description: string;
}

export default function Demo({ banner, title, description }: DemoProps) {
  return (
    <div className="container">
      <img src={banner} alt="banner" />
      <h1>{title}</h1>
      <p>{description}</p>
    </div>
  );
}
```

#### config.schema.json（配置 Schema）

使用行业标准 [JSON Schema](https://json-schema.org/) 定义配置项：

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "活动 Demo 配置",
  "type": "object",
  "properties": {
    "banner": {
      "type": "string",
      "format": "uri",
      "title": "顶部 Banner 图",
      "description": "建议尺寸: 750x400px",
      "default": "https://example.com/default-banner.png"
    },
    "title": {
      "type": "string",
      "title": "活动标题",
      "default": "活动标题",
      "maxLength": 20
    },
    "theme": {
      "type": "string",
      "title": "主题颜色",
      "enum": ["light", "dark", "colorful"],
      "enumNames": ["浅色", "深色", "多彩"],
      "default": "light"
    }
  },
  "required": ["banner", "title"]
}
```

### 7.2 JSON Schema 扩展约定

| 扩展字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| `enumNames` | 枚举值显示名称，与 `enum` 一一对应 | `["浅色", "深色"]` |
| `ui:widget` | 指定表单控件类型 | `file`, `richtext`, `color` |
| `ui:options` | 控件额外配置 | `{ accept: "image/*" }` |

### 7.3 表单控件映射

| JSON Schema 类型 | format / ui:widget | 表单控件 |
| :--- | :--- | :--- |
| `string` | - | 文本输入框 |
| `string` | `uri` + `ui:widget: file` | 文件上传 |
| `string` | `uri` | URL 输入框 |
| `string` | `enum` | 下拉选择框 |
| `string` | `ui:widget: richtext` | 富文本编辑器 |
| `string` | `ui:widget: color` | 颜色选择器 |
| `number` / `integer` | - | 数字输入框 |
| `boolean` | - | 开关 |
| `array` | - | 数组编辑器（动态增删） |
| `object` | - | 嵌套对象表单 |

### 7.4 代码与 Schema 的关联

- React 组件通过 `props` 接收配置值
- JSON Schema 的 `properties` 键名与组件 `props` 一一对应
- 工作台根据 Schema 生成默认值，传入组件渲染

### 7.5 Figma 导出格式

Figma 插件导出时，将 React 组件代码和配置 Schema 使用分隔符封装成单一文本，方便用户一键复制粘贴到工作台。

#### 导出格式示例

```
=== DEMO CODE ===
import React from 'react';

interface DemoProps {
  title: string;
  description: string;
}

export default function Demo({ title, description }: DemoProps) {
  return (
    <div className="container">
      <h1>{title}</h1>
      <p>{description}</p>
    </div>
  );
}
=== DEMO SCHEMA ===
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Demo 配置",
  "type": "object",
  "properties": {
    "title": {
      "type": "string",
      "title": "标题",
      "default": "Hello World"
    },
    "description": {
      "type": "string",
      "title": "描述"
    }
  },
  "required": ["title"]
}
=== END ===
```

#### 分隔符规范

| 分隔符 | 说明 |
| :--- | :--- |
| `=== DEMO CODE ===` | React 组件代码开始标记 |
| `=== DEMO SCHEMA ===` | JSON Schema 配置开始标记 |
| `=== END ===` | 内容结束标记（可选，用于校验完整性） |


## 8. 项目架构

### 8.1 Monorepo 架构

```
opencode工作台/
├── packages/
│   ├── web/                    # Web 工作台（本项目）
│   ├── figma-plugin/           # Figma 插件（后续接入）
│   └── shared/                 # 共享代码（类型、工具、组件）
├── demos/                      # Demo 存储目录
├── package.json                # 根 package.json
├── pnpm-workspace.yaml         # pnpm 工作区配置
└── turbo.json                  # Turborepo 配置
```

### 8.2 Monorepo 优势

| 优势 | 说明 |
| :--- | :--- |
| 统一管理 | Web 工作台、Figma 插件、共享代码在同一仓库管理 |
| 代码复用 | 共享类型定义、工具函数、组件可跨项目复用 |
| 统一依赖 | 统一管理依赖版本，避免版本冲突 |
| 原子提交 | 跨项目修改可在一个 PR 中完成 |

### 8.3 子项目规划

| 子项目 | 状态 | 说明 |
| :--- | :--- | :--- |
| `packages/web` | ✅ 当前开发 | Web 工作台前端 |
| `packages/figma-plugin` | 🔜 后续接入 | Figma 插件，由其他项目组开发 |
| `packages/shared` | ✅ 当前开发 | 共享代码（类型、工具、组件） |
