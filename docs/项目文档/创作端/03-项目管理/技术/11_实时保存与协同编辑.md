# 实时保存与协同编辑

> 更新日期：2026-07-22

## 定位

实时协同层服务于 [项目工作区](./03_项目工作区_v2.md)：多人默认编辑项目级共享当前态，而不是用户私有临时副本，也不是直接操作历史快照。业务需求见 [实时协作保存需求](../实时协作保存_需求文档.md)，历史记录由 [版本管理](./04_版本管理.md) 在自动检查点、命名版本、发布快照和恢复快照场景生成。

## Workspace Mutation Authority

活动 Workspace 采用单一 Authority 提交模型。Authority 核心（shared mutation contract、资源注册表、每 Workspace 串行队列、事务日志、receipt 幂等、启动恢复、外部漂移检测、health/status、projection ack 和事件 WebSocket）已实现。协同 flush、AI 文本/Sketch/画布工具、页面新增/复制/删除/恢复/重命名/排序/运行时切换、文件夹 API、Session 文件保存、单页面文件保存、项目级共享配置定义和运行值、画布布局保存、图片资产 staging、知识文档新增/编辑/删除和页面资源版本恢复，都已携带资源基线 hash 提交 mutation，并以 durable receipt 作为“已提交”的唯一依据。剩余入口（完整 CLI Authority client、离线 IndexedDB 草稿、编辑页状态机和后台 coalesce canonical materializer）仍在迁移中。不同入口在同一数据目录下共享串行队列、committed 事件总线与跨进程 lease；重复请求按 mutation ID 返回原 receipt。

Authority 将状态、journal、prepared manifest、receipt、内容寻址 committed backup、projection ack 与 lease 放在 Workspace 目录外。agent-service 在监听端口前扫描已注册 live Workspace：无 receipt 的 prepared mutation 恢复业务文件和上一份 revision state；receipt 与 state 已一致时只完成 prepared/staging 清理，不回滚已提交内容。reconcile restore 的中断记录也使用相同判定收敛。任何 stale lease、receipt/state 不一致、Authority state 找不到对应 live Workspace 或恢复失败都会阻止服务 ready；未注册历史 Workspace 仅记为 skipped，不会在启动时静默 bootstrap/adopt。

受管资源由 project-core 的统一注册表定义，覆盖 React 页面源码、HTML/CSS 原型与 meta、页面 Schema、Sketch scene 与 meta、项目配置 Schema/运行值、页面树、画布布局、知识 Markdown 与 manifest、二进制 assets。注册表同时负责安全路径规范化、文本/JSON/页面树/Sketch/二进制内容校验、单资源 hash 和按路径稳定排序的 root manifest；未注册路径或不符合对应内容契约的资源不能进入 mutation。

每个 Workspace 的 mutation 在进程级共享队列中串行执行，并在写前获取持久化 Workspace lease。事务先校验全部 operation 与目标 hash，再持久化 prepared 记录和 backup，随后应用全部文件，最后依次持久化 Authority state、receipt 与 journal；只有这些 durable 证明完成后才发布 committed event。任一 operation 失败会用 prepared backup 恢复全部已触碰资源和旧 state，因此调用方不会看到“前半个操作成功、后半个失败”的已提交版本。旧 revision 只有在目标资源 hash 仍匹配时才能安全 rebase；目标资源已变化时必须返回冲突。

检测到受管文件被旁路修改时会 fail-closed。显式 `reconcile adopt` 把当前磁盘接纳为新 revision；`reconcile restore` 从最后 committed backup 恢复磁盘。服务启动和普通读取都不会静默采用外部变化。

Workspace Authority 提供只读 health/status 诊断入口。该入口在通过 Session/Workspace 校验后返回 ready、revision/rootHash、实际 rootHash、external drift、queue depth、active lease、prepared/recovery pending 事务数、recovery state、持久化 mutation 冲突数、committed event 订阅者数、staging、committed backup 及缺口、receipt、journal 和 projection ack 计数；它不获取写 lease、不触发 bootstrap，也不修改业务文件。冲突数从 Authority journal 的 `conflicted` 记录派生，进程重启后仍保留；订阅者数表示当前进程、同一 `DATA_DIR` 下的 committed-event 监听器。agent-service 的全局 `/health` 另外返回本次启动扫描、skipped、pending、rolled back 和 committed cleanup 计数。部署 preflight、诊断 CLI 或 reconcile 工具应优先消费这些只读状态，再决定是否继续关键动作或要求显式修复。

关键动作把 live Workspace 物化到项目基准工作区前，会先得到协同 flush 后的 committed revision，再通过 `ensureCanonicalRevision` 读取 Authority snapshot 并确认 snapshot 至少覆盖目标 revision；同 revision 下如果 root hash 不一致会 fail-closed。校验通过后，项目元数据会同时记录 `canonicalSyncedWorkspaceId`、`canonicalSyncedRevision`、`canonicalSyncedRootHash` 和 `canonicalSyncedAt`；自动检查点等项目级历史记录也会保存本次消费的 Workspace ID、revision 和 root hash。后续发布、导出、模板和历史动作必须把这些字段视为 canonical 物化进度，而不能只根据时间戳推断同步完成。无 Session 发布和版本创建这类只消费项目基准工作区的动作，也必须要求 revision/rootHash 存在后才认为 active Workspace 已被安全物化。若旧 active Workspace 因 `baseVersion` 过期被替换，新 live Workspace 会清空这些 canonical 同步字段，直到它重新通过同步边界。

协同房间已直接消费 `workspace_mutation_committed` receipt 中的确切资源路径，更新自身 committed baseline；这条新链路不依赖旧 `file_operation` 的短路径猜测。干净房间会把 committed 内容投影回 Y.Doc；非协同写入提交到 Authority 前，会先通过协同草稿 provider flush 同 Workspace、同目标资源的活跃房间。若 AI、author-site 或 Pi 工具基于旧 hash 提交，草稿会先成为 committed 内容，随后旧 hash mutation 被冲突拒绝，避免覆盖用户未落盘草稿。编辑器重连可读取 committed text snapshot；预览是否应用某个 revision 使用独立 projection ack，不能反向改变已经成功的 mutation。Authority 对 applied/failed ack 分别写诊断，ack revision 落后当前 committed revision 时另写 gap 事件，供重连 catch-up 和预览故障排查。

Authority API 在 Session、项目与 Workspace 归属校验后提供 state、完整文本 snapshot、单资源 read、mutation、二进制 staging、health 与显式 reconcile。事件 WebSocket 接受客户端最后应用的 revision，先按持久化 receipt 补发缺失 committed event，再订阅新 committed event 与 projection ack；若 receipt 链无法从期望 revision 连续恢复，则发送结构化 revision gap，客户端必须读取 snapshot 收敛，不能拿旧内存状态补写。projection ack 另有独立查询接口，预览 applied/failed 状态不会改写 mutation receipt。HTTP 与 WebSocket 只暴露 shared 定义的稳定错误码，未知内部异常统一收敛为 `WORKSPACE_MUTATION_FAILED`，客户端不得通过错误消息文本猜测分支。

author-site 的服务端 client 使用内部 agent-service 地址；浏览器 client 只请求同源 `/api/workspace-authority/...` 代理。代理先核验登录用户、Session、项目与 Workspace 的绑定关系，再转发到受限的 Authority endpoint 集合，不提供任意内部路径转发。project-cli 也提供同一协议的 typed client，连接不可用时返回稳定 `WORKSPACE_AUTHORITY_NOT_READY`，不会退回为本地 live Workspace 写入。

知识文档 live 写入现在由 author-site 先校验 Session 与 Workspace，再把 Markdown 正文和 `knowledge/manifest.json` 元数据合并成一个 Authority mutation；live Workspace 下缺少有效 Session 时 fail-closed，不回退到本地文件写入。知识列表和内容读取在 live Workspace 下只读现有 manifest 与正文，不再借读取触发 builtin knowledge 初始化。

AI Workspace 上下文扫描同样是只读路径：它只读取当前 `knowledge/manifest.json` 并过滤系统条目，不再创建 knowledge 目录、重写 manifest、清理旧 system 文档或把旧 `references/` 自动迁移为 `knowledge/`。

单页面文件保存接口仍保留既有的鉴权、Schema 冲突检测、原型页运行时校验和手绘 patch 基线校验；通过校验后，live Workspace 中的 React 源码、页面 Schema、HTML/CSS 原型文件、原型 meta、手绘 scene 和手绘 meta 会组成同一个 Authority mutation。项目页文件兼容保存入口和旧 Session 文件保存入口在 live Workspace 下也改为 Authority mutation；非 live 工作区继续使用原有隔离文件写入语义。

页面新增接口在 live Workspace 下会把新页面初始文件和 `workspace-tree.json` 追加记录组成同一个 Authority mutation；页面复制会把源页面文件副本和 `workspace-tree.json` 追加记录合并提交；页面删除会把受管页面文件删除和 `workspace-tree.json` 移除记录合并提交；页面恢复会从删除快照读取受管页面文件，并与 `workspace-tree.json` 恢复记录合并提交，提交成功后清理已消费的删除快照。页面运行时切换会把目标运行时文件、可选页面 Schema 与 `workspace-tree.json` 的运行时类型更新放入同一个 mutation。页面资源版本恢复在 live Workspace 下只读取历史资源版本内容，先 flush 当前协同草稿，再把目标运行时文件和页面 Schema 作为 Authority mutation 提交；它不再调用 Project Core 的直接工作区恢复，也不立即推进项目基准工作区。文件夹 API 的新增、更新和删除会提交 `workspace-tree.json`，其中删除文件夹并删除内容时会同时删除被包含页面的受管文件。页面重命名、单页排序、移动父级和批量页面/文件夹排序都只更新 `workspace-tree.json`，live Workspace 下也通过 Authority 提交。页面结构 Web API 已完成单写者收敛；`project-core` 会在发现事务或 `.workspace.json` 指向 live Workspace 时拒绝直接写入并返回 `WORKSPACE_AUTHORITY_REQUIRED`，防止 CLI 或共享层误把 live Workspace 当作普通分支事务写入。Pi Agent 的 bash 工具在 live Workspace 下进入只读模式，拒绝 node/npm/npx、重定向、heredoc、管道、命令连接和命令替换等可能绕过 Authority 的写副作用；`delegateTask` 子 Agent 在 live Workspace 下直接禁用并返回同一错误码，直到子 Agent 写入接入受管工具、actor identity 和 receipt 汇总。完整 CLI Authority client、canonical 物化和全量副作用门禁仍在迁移中。

项目级共享配置定义 `project.config.schema.json` 和运行值 `project.config.values.json` 都是 Authority 受管文本资源。live Workspace 下配置面板更新或删除项目级配置定义时会提交 `update_project_config_schema` 或 `delete_project_config_schema` mutation；写入运行值时会提交 `update_project_config_values` mutation。非 live 工作区继续使用原有文件写入和 workspace timestamp 语义。

本迁移尚未完成 CLI、离线草稿、编辑页状态机和 canonical 后台物化，因此其余章节中描述的旧链路仅代表这些未迁移入口的当前兼容行为，不能被用于新增 live Workspace 写入口。

## 协作边界

- 协同房间以项目、Workspace、资源路径为边界，不同页面文件彼此隔离。
- 连接必须携带 Session、Workspace 和资源信息，服务端校验 Session 属于当前项目和 Workspace 后才允许加入。
- 项目工作区是共享编辑边界。默认编辑页会把不同用户的 Session 绑定到同一个 `scope=live` Workspace；服务端不再用单一 workspace owner 阻断跨用户协作。
- 资源范围限制在 React 页面代码、HTML/CSS 原型源码、页面 Schema、手绘页面 scene、项目级 Schema、页面树、画布布局和工作空间知识文档，避免协同通道写入任意文件。
- 知识文档协同只放行 `knowledge/*.md`、`knowledge/*.markdown` 和 `knowledge/*.mdown` 这类单层用户文档；阅读地图、模板索引和嵌套知识服务产物不进入创作端协同写入范围。
- 本能力明确只支持单实例 agent-service 写入。服务启动会断言 single 模式与副本数为 1，Docker Compose 显式声明该策略，部署 preflight 会拒绝缺少声明或 replicas 大于 1 的配置；意外第二进程触达同一 `DATA_DIR` 时，Workspace lease 继续 fail-closed。多实例写入在实现带单调 fencing token 的持久化 lease 前禁止启用，不能把当前文件锁或进程内 queue 当作多实例协调方案。

协同房间的资源类型来自共享类型 `CollabResourceKind`，当前包括 `page-code`、`page-prototype-html`、`page-prototype-css`、`page-schema`、`page-sketch-scene`、`project-schema`、`workspace-tree`、`canvas-layout` 和 `knowledge-document`。资源类型和资源路径必须同时匹配允许规则，避免前端通过协同通道越权写入工作空间内的其他文件。原型页 HTML 与 CSS 分别只允许写入 `demos/{pageId}/prototype.html` 和 `prototype.css`；手绘 scene 只允许写入 `demos/{pageId}/sketch.scene.json`。三类非 React 页面源码都进入同一套房间基线、外部写入重载和 flush 保护。

## 房间与连接模型

协同房间基于 Hocuspocus（Yjs WebSocket 服务器）实现，前端使用 `@hocuspocus/provider` 的 `HocuspocusProvider` 客户端连接。房间标识通过 JSON 编码的 `documentName` 携带 `{projectId, workspaceId, resourcePath, kind}`，服务端在 Hocuspocus 扩展链中解码该标识完成鉴权与持久化。

前端通过 `useCollabDocument` 为某个资源构造房间描述，使用 `HocuspocusProvider` 连接 agent-service 的协同 WebSocket 端点，token 携带当前 Session ID。Hocuspocus 协议在握手阶段通过认证消息传递 token，而非 URL 查询参数。

协同状态必须严格区分“WebSocket 已连接”和“Yjs 初始同步已完成”。`connected` 只证明传输通道建立，前端仍保持连接中；只有 provider 的 `synced` 事件确认初始状态交换完成后，资源才进入已同步。若在两者之间把本地 REST 全文写入空 Y.Text，服务端同时播种的磁盘全文会被 Yjs 当成另一组独立插入，合并后形成完整模块的 2 倍、4 倍或 8 倍重复。

agent-service 的 `HocuspocusCollabServer` 配置了两个扩展（按优先级执行）：

| 扩展 | 优先级 | 钩子 | 职责 |
| :--- | :----- | :--- | :--- |
| `SessionAuthExtension` | 1000 | `onAuthenticate` | 解码 documentName，校验 Session 归属，返回连接上下文（含 workspacePath） |
| `AuthorityPersistenceExtension` | 100 | `onLoadDocument` / `onStoreDocument` | 首次加载时从文件播种 Yjs text；debounce 后将 Yjs text 通过 Authority 落盘 |

协同路由仍负责三类入口：

| 入口类型            | 作用                                                                                 |
| :------------------ | :----------------------------------------------------------------------------------- |
| WebSocket 房间      | 委托给 Hocuspocus 处理 Yjs 同步和 presence 通道                                      |
| 单资源 flush        | 触发指定 documentName 的 debounced `onStoreDocument` 立即执行                        |
| Workspace flush-all | 校验 Session 后，匹配并 flush 当前 Workspace 下所有活跃 documentName                |

服务端首次加载房间时，`onLoadDocument` 从 Workspace 文件读取初始内容放入 Yjs text。后续编辑产生的 Yjs 更新在房间内广播，Hocuspocus 按 debounce（默认 1 秒，最大 10 秒）触发 `onStoreDocument`，将 Yjs text 通过 Authority 落盘。Yjs-First 架构下 Yjs room 是唯一内容权威，Authority 退化为纯持久化后端，不再做 `expectedHash` 冲突检测，也不再执行 self-heal 全量替换。

## 数据流

用户打开可协同文件时，前端会连接 agent-service 的协同 WebSocket。服务端从 Workspace 文件读取初始草稿内容，放入协同文档。多个浏览器对同一文档的编辑会在内存中合并，再按短延迟写回 Workspace 文件。

当用户命名版本时，版本接口会先通过 author-site 的 flush coordinator 调用协同服务 flush 当前 Workspace，再由版本管理生成命名版本。前端仍可提前发起 flush 以优化交互反馈，但可靠边界在后端关键动作入口。flush coordinator 在调用 agent-service 前会续期仍处于编辑状态的 Session，避免长时间打开编辑页后本地仍可编辑、但协同服务因 Session 超过默认有效期而拒绝写回，导致 AI 修复停留在 live Workspace、项目基准工作区没有被推进。

发布项目时也会先执行 Workspace 级 flush，并在发布前生成发布快照。即使当前浏览器没有本地未同步标记，只要协同房间里还有未落盘内容，也会先写回 Workspace，再进入发布流程。自动检查点、保存/合并 Session、页面恢复和显式持久化同样使用这一后端前置协议；本地项目包获取由 CLI `project pull` 负责，不再通过创作端编辑页触发。

author-site 在关键动作推进项目基准工作区前，会先判断目标是否为 live Workspace；只有 live Workspace 会向 Workspace Authority 读取一次 snapshot。这个读取不用于替代后续 canonical 物化，但会让 Authority 先校验当前 live Workspace 的 root hash；如果检测到受管文件被旁路修改，关键动作返回过期/冲突类错误，并且不会继续把 live 磁盘内容同步到项目基准工作区。非 live / branch 工作区继续使用旧兼容同步路径，不强制依赖 Authority。这样可以避免“外部漂移内容”借发布、自动检查点或显式持久化进入 canonical，同时不破坏隔离分支工作流。

发布 API 是发布前同步的可靠边界。前端可以携带当前 Session 和 Workspace，但不需要先发起一次重复 flush；如果页面上的 Session 已过期、被归档或被清理，而 Workspace 元数据仍证明它属于当前用户和当前项目，发布 API 会先为这个 Workspace 续建有效 Session，再用新 Session 执行 Workspace 级 flush、同步项目工作区和发布。若 Workspace 本身不存在或归属不匹配，发布仍会被阻断。若请求没有携带 Workspace，只有项目基准工作区已确认追上项目工作区时才能发布；否则必须返回同步要求，避免发布落后的项目工作区。

编辑页不再把右上角“保存”作为防丢入口。顶部只展示自动保存与协同状态；手动标记历史节点的入口放在历史记录区，按钮文案为“命名此版本”。该操作会先 flush 当前内容，再创建页面版本和项目版本快照。用户产生本地编辑后，前端会把 Workspace 标记为待落盘，约 1.2 秒后通过 author-site 同源轻量接口触发 Workspace flush；flush 成功后还必须通过 `persist-workspace` 把 Session Workspace 同步到项目当前 Workspace，只有两个步骤都成功后才能展示“已自动保存”。手绘 scene 保存也遵守同一边界：先让 `page-sketch-scene` 协同草稿落到 Workspace 文件，再由 session 文件 API 做 scene 或 patch 校验。退出时若仍有待落盘编辑、flush 失败、项目当前 Workspace 同步失败、连接中、同步中、离线待同步或协同异常，才提示最新修改尚未确认同步完成。同源接口在服务端调用 agent-service，避免浏览器退出路径直接依赖 agent-service 的跨域地址和环境配置。

重新进入项目时，创作端必须优先复用当前用户、当前项目下的活跃 Session，并在复用前确保该 Session 已绑定项目工作区。没有活跃 Session 时，新 Session 也绑定项目工作区，而不是为当前用户创建私有副本。复用活跃 Session 前必须确认其 `workspaceId` 仍能定位到 Workspace 目录；如果目录已被清理或缺失，应把该 Session 标记为断链并跳过，避免编辑页继续加载一个无法读取文件、也无法 flush 的旧会话。若项目原 active live Workspace 的 `baseVersion` 已落后于最新项目版本，重新进入时会创建新的 live Workspace，并清空旧的 `canonicalSyncedWorkspaceId` / `canonicalSyncedAt` 指针；新 live Workspace 必须重新通过 canonical 同步边界确认，不能继承旧 Workspace 的“已同步到项目基准工作区”状态。

AI 或文件工具通过 Workspace Authority 提交前，会先触发目标资源的协同草稿 flush barrier；提交后，agent-service 以 durable receipt 中的确切资源路径作为唯一写入事实，并通知对应协同房间投影同一 Workspace 和资源路径下的 committed 内容。旧的 `tool_execution_end` 或 `file_operation` 事件不能承担持久化成功判断，也不能再通过磁盘回读推导写入结果。编辑页收到 Authority 派生的资源更新后，把 committed 内容应用到当前页面状态、页面文件映射、refs 和预览缓存；原型页的 `prototype.html` 与 `prototype.css` 还会通过各自协同房间广播给所有打开同一 Workspace 的客户端。这样，工具成功、当前预览状态和协同权威文本使用同一份内容，旧浏览器状态不能在后续自动保存中恢复旧文件。

页面切换或新增删除导致当前页面变化时，仍要同步维护 `codeRef` / `schemaRef`，避免后续保存基于旧引用。前端收到 Authority 派生的资源更新后还必须把 Workspace 标记为待同步，让既有自动保存链路执行 Workspace flush 和项目基准工作区同步；否则修复可能只停留在 live Workspace，重新打开项目时仍有机会从旧项目基准内容恢复。Yjs-First 架构下 Yjs room 是唯一内容权威：Pi 工具和 HTTP 路由都通过 Yjs room 写入（`writeToResource` / DirectConnection），不再直接调用 Authority mutate，因此不再存在"多路写入导致 baseline 过期"的竞态。

## 持久化策略

协同服务采用“内存合并、延迟落盘、关键动作强制落盘、项目当前态显式推进”的策略：

| 阶段                    | 行为                                                                                                                                                                                                                                                                                                                                   |
| :---------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 加入房间                | 读取 Workspace 中的目标资源作为初始文本                                                                                                                                                                                                                                                                                                |
| 复用空房间              | 空且未变更的内存房间会重新对齐非空 Workspace 文件                                                                                                                                                                                                                                                                                      |
| 实时编辑                | Yjs 更新广播给同房间连接，并标记房间 dirty                                                                                                                                                                                                                                                                                             |
| 延迟保存                | 默认约 1 秒后将 dirty 文本写回项目工作区文件；落盘前检查房间文本是否为严格的全文自拼接，命中时递归收敛到单份并同步修正房间                                                                                                                                                                                                               |
| 前端自动保存            | 页面代码、页面 Schema、项目 Schema、页面树、工作区文件、知识文档和 AI 文件回流会把 Workspace 标为待同步；前端约 1.2 秒后执行 Workspace flush，再调用 `persist-workspace` 推进项目当前工作区                                                                                                                                            |
| 手绘 scene 保存         | 手绘页面 scene 通过 `page-sketch-scene` 协同房间写入 `sketch.scene.json`；session 文件 API 负责 scene 或 patch 校验                                                                                                                                                                                                                    |
| 页面树操作              | live Workspace 下新增、复制、恢复和运行时切换会把页面文件与 `workspace-tree.json` 合并为 Authority mutation；删除页面会把受管页面文件删除和 tree 更新合并提交；文件夹 API 新增/更新/删除提交 `workspace-tree.json`，删除内容时同步删除受管页面文件；页面重命名、排序、移动父级和批量 reorder 入口只提交 `workspace-tree.json` mutation |
| 页面资源版本恢复        | live Workspace 下只把历史资源版本内容恢复到当前 Workspace 的受管页面文件，并以 Authority receipt 作为成功边界；项目基准工作区由后续物化/持久化链路推进                                                                                                                                                                                 |
| 共享配置运行值          | live Workspace 下写入 `project.config.values.json` 走 Authority mutation；非 live 工作区保留原文件写入和 timestamp 更新                                                                                                                                                                                                                |
| CLI / project-core 事务 | 分支事务继续写入 `scope=branch` 工作区；如果共享层检测到 live Workspace 元数据或非 branch 事务 scope，会 fail-closed，要求调用方改走 Authority                                                                                                                                                                                         |
| AI bash 工具            | live Workspace 中只允许简单只读命令；可能写文件或运行脚本的命令直接返回 `WORKSPACE_AUTHORITY_REQUIRED`，不执行 shell                                                                                                                                                                                                                   |
| AI 子 Agent 委派        | live Workspace 中 `delegateTask` 直接返回 `WORKSPACE_AUTHORITY_REQUIRED`，不启动子 Agent runner；branch 工作区保持原委派语义                                                                                                                                                                                                           |
| 画布布局                | 画布页面移动、自由文本节点和布局状态由画布 hook debounce 保存到 `/api/sessions/:sessionId/canvas-layout`；退出前会强制保存最新画布状态                                                                                                                                                                                                 |
| 当前会话撤回/重做       | 编辑页维护仅存在于当前浏览器会话内的 UI command history；它可撤回页面删除和画布内容性变更，但不跨刷新持久化，也不生成版本历史                                                                                                                                                                                                          |
| 外部文件写入            | AI 工具和 HTTP 路由通过 `HocuspocusCollabServer.writeToResource` 写入 Yjs room（DirectConnection + transact），写入后触发 `onStoreDocument` 落盘；不再直接调用 Authority mutate，避免多路写入竞态                                                                                                                                              |
| Yjs-First 单写者        | Yjs room 是每个 resource 的唯一内容权威；Authority 退化为纯持久化后端，不再做 `expectedHash` 冲突检测，不再返回 `WORKSPACE_RESOURCE_CONFLICT`，不再执行 self-heal 全量替换                                                                                                                                    |
| 显式 flush              | 命名版本、自动检查点、发布、保存/合并 Session 和恢复前立即写回 Workspace 下所有活跃房间                                                                                                                                                                                                                                                |
| 基准同步                | 关键动作或显式持久化时，将项目工作区推进到项目基准工作区                                                                                                                                                                                                                                                                               |
| 自动检查点              | 停止编辑 5 分钟或持续编辑 30 分钟且存在变更时，先推进项目基准工作区，再生成自动保存记录                                                                                                                                                                                                                                                |
| 房间清理                | 空闲房间被清理前先 flush，再销毁内存文档和 presence                                                                                                                                                                                                                                                                                    |

文件写入通过临时文件加重命名完成，避免中途失败留下半截内容。写入成功后会刷新 Workspace 元信息的更新时间，用于后续工作空间活跃状态判断。全文自拼接守卫优先按原始字符串严格切半，避免源码尾部换行让行级统计产生奇数项而漏检；连续四份、八份等二次幂重复会递归收敛。守卫只处理两个半段完全相同的内容或既有 JSON 重复兼容形态，不把局部相似代码误判为整文件重复。

agent-service 的协同持久化默认读取仓库根 `data/`，而不是当前包目录下的 `data/`。本地开发时 author-site 与 agent-service 可以从不同工作目录启动，但两者必须指向同一份 `data/sessions` 和 `data/workspaces`；生产或 Docker 环境仍以显式 `DATA_DIR` 为准。若两端数据目录不一致，Workspace flush 会因找不到 Session 或 Workspace 被拒绝，编辑页应展示同步失败而不能误报已自动保存。

项目工作区的标准形态是 `data/workspaces/projects/{projectId}/{workspaceId}`，并带有 `.workspace.json` 元数据。旧的 `data/workspaces/{userId}/{projectId}/{workspaceId}` 仅作为分支工作区或 legacy Workspace 兼容读取。协同持久化优先读取元数据校验归属；若历史数据或异常写入导致 `.workspace.json` 缺失，服务端可按嵌套目录名定位 `workspaceId`，并用目录中的 `projectId` 段兜底校验项目归属。这样可以避免已有 Workspace 内容存在但 flush 误报 `WORKSPACE_NOT_FOUND`。

自动检查点由编辑页调度。调度器监听当前页面内容、Schema、页面数量和画布脏状态；当内容签名变化后重新计算 5 分钟空闲窗口，同时保留本轮持续编辑的开始时间。触发检查点前会先 flush 当前 Workspace 下的协同房间和画布布局，再把项目工作区同步到项目基准工作区，随后从项目基准工作区复制历史快照。自动检查点不会结束 Session，也不会清理 Workspace；若基准同步失败，检查点接口必须返回失败，前端不能把状态标记为已自动保存。自动检查点、保存/合并 Session 和发布快照从已同步 live Workspace 生成新的项目级版本后，项目级版本创建函数必须同步推进该 Workspace 元数据里的 `baseVersion`，否则下一次自动保存、命名版本后的合并或发布前同步会把同一个活跃 Workspace 误判为过期。

如果历史数据中已经出现“最新项目版本由当前 Session 生成，但 live Workspace 的 `baseVersion` 仍停留在上一版本”的不一致状态，关键同步入口只能在最新版本 `sessionId` 与当前请求 Session 完全一致、且 Workspace 仍是项目 active Workspace 时修复基线并重试同步。其他来源的新版本仍按过期 Workspace 处理，避免旧工作区覆盖他人或系统生成的新项目当前态。

## 权限与路径安全

协同写入前必须完成三层校验：

| 校验           | 目的                                                   |
| :------------- | :----------------------------------------------------- |
| Session 校验   | 确认会话存在、未过期，且归属于当前项目和 Workspace     |
| Workspace 校验 | 确认 Workspace 存在，并与项目关系一致                  |
| Resource 校验  | 确认资源路径没有越级片段，并匹配对应资源类型的允许规则 |

校验失败时，WebSocket 使用策略错误关闭连接；flush 接口返回标准错误响应。这样可以把“协作连接失败”和“草稿写入失败”都归因到明确的项目、会话或资源边界，而不是静默丢弃用户编辑。

Workspace 级 flush 不接收资源路径，只校验调用者 Session 对项目和 Workspace 的访问关系。校验通过后，它会刷新该 Workspace 下所有活跃协同房间，包括其他协作者用不同 Session 打开的房间；没有活跃房间时返回 no active room，关键动作可以继续使用磁盘上的 Workspace 当前态。agent-service 不可达或 flush 返回错误时，author-site 关键动作必须阻断，避免生成落后于协同草稿的快照、发布产物或导出包。

## 状态与呈现

前端展示连接中、已自动保存、同步中、离线待同步和协同异常状态。协作者信息通过 presence 同步，用于展示在线用户和远端光标。编辑页顶部展示项目级在线状态，文件编辑弹窗展示当前资源的协同状态。

编辑页和文件编辑弹窗读取可协同资源时，必须区分“连接尚未完成时的默认空值”和“真实远端内容”。当本地或 REST 接口已经拿到非空页面代码、Schema、手绘 scene、项目配置、页面树或画布布局，而协同同步返回空文本时，前端应先用非空内容回填协同文档，不能直接把编辑器覆盖为空。用户在编辑器中主动清空内容属于显式编辑行为，可按正常协同更新继续处理。手绘 scene 的协同回流必须先通过 `SketchSceneDocument` 解析，解析失败时不能覆盖本地 scene。

编辑页复用 `workspace-tree.json` 协同房间的 presence 展示项目在线用户，避免为在线状态额外打开独立房间。页面树新增、复制、删除、恢复、重命名、排序、移动父级、批量 reorder，以及文件夹 API 新增/更新/删除仍由既有项目 API 承接，但 live Workspace 下这些 API 会通过 Authority 提交资源变更；操作成功后前端会重新读取 `workspace-tree.json` 并写入协同文档，其他打开同一 Workspace 的编辑页会据此刷新页面与文件夹结构。

画布布局同时具备协同观察和独立持久化两层。编辑页仍打开 `.canvas-layout.json` 协同房间，用于状态聚合、presence 和远端布局应用；但本地画布编辑的可靠保存路径由 `useCanvasWorkspace` 维护：页面移动、自由文本节点和布局状态变化后，hook 使用最新状态引用 debounce 调用 `/api/sessions/:sessionId/canvas-layout`。该 API 会把 `{ version, projectId, updatedAt, state }` 写入当前 Session 目录作为恢复缓存；目标是 live Workspace 时，`.canvas-layout.json` 通过 Authority mutation 提交，不直接写 Workspace 文件，也不直接覆盖项目基准工作区；目标是非 live / branch Workspace 时，继续写入该隔离 Workspace 文件并保留 Session 缓存。重新打开项目创建新 Workspace 时可以从项目工作区或 Session 缓存恢复上次直接退出前的画布布局。退出编辑页时会先强制保存画布布局；只有页面代码、Schema、页面树等非画布工作区内容仍有未保存改动时，协同 Workspace flush 才会阻塞退出。

全局撤回/重做是编辑页本地交互层能力。快捷键在非输入焦点下拦截 `Cmd/Ctrl+Z`、`Cmd/Ctrl+Shift+Z` 和 `Cmd/Ctrl+Y`，输入框、代码编辑器和富文本区域继续保留自身原生撤回。撤回栈只保存当前会话内的命令闭包：页面删除会依赖删除前快照恢复原页面 ID；画布布局只记录页面布局、画布节点、图层和隐藏文档等内容性变化，视口平移和缩放不进入历史。页面刷新、重新打开项目、远端替换或版本恢复后，撤回栈清空，长期追溯仍以自动检查点、命名版本、页面版本和发布快照为准。

知识文档编辑复用同一协同通道。知识库弹窗中的 Markdown 富文本编辑器把当前文档正文序列化到 `knowledge/{fileName}` 对应的 Yjs 文本资源；保存时先 flush 协同草稿，再调用知识文档 API。live Workspace 中，正文与 manifest 元数据必须在同一个 Authority mutation 中提交，避免正文与知识索引元信息分离；非 live 工作区继续使用原有隔离文件写入语义。

## 异常与降级

协作异常分为连接异常、权限异常和落盘异常。连接异常时，前端应展示离线或错误状态，避免用户误以为远端已经收到更新。权限异常通常来自会话过期、项目不匹配、Workspace 不匹配或资源路径不被允许，此时应阻止继续加入房间。落盘异常发生在 flush 或延迟保存阶段，需要保留当前编辑态并提示用户重试关键动作。

协同状态不替代历史记录状态。内容即使已经自动保存，也只是 Workspace 当前状态；自动检查点、命名版本、发布快照和恢复快照才形成可在历史记录中恢复的快照。

## 与既有能力的关系

- Session 生命周期仍由 [生命周期设计](./02_生命周期设计_v2.md) 管理。
- `scope=live` 的项目工作区不随 Session 清理；分支工作区和 legacy Workspace 清理仍以活跃 Session 引用和过期策略为准。
- AI 或非协同文件写入仍可继续使用原工作区文件 API；写入成功后必须让服务端协同房间对齐磁盘当前态，避免旧内存文本在后续 flush 中回写。
- 配置与预览如何消费协作草稿，见 [协同草稿驱动预览](../../04-配置与预览/技术/08_协同草稿驱动预览.md)。

## 回归覆盖

创作端 E2E 的 `canvas-autosave-reopen-regression.spec.ts` 是实时保存主回归脚本。该脚本创建 `__e2e__` 项目后，覆盖页面代码、页面 Schema、项目 Schema、页面新增、复制、删除、重命名、排序、文件夹新增、删除、重命名、知识文档、`memory.md`、画布页面移动和画布文本节点。除画布交互外，脚本通过 `persist-workspace` 验证自动保存的项目当前态推进，而不是通过命名版本保存；画布交互则验证退出前强制保存后重新打开仍能读回布局。协作底层的多用户 Workspace 归属、Hocuspocus 扩展鉴权、外部文件写入回流和知识文档路径边界由 agent-service 单元测试覆盖；`authority-persistence.test.ts` 额外固定带尾部换行的两份模块、四份与八份模块递归收敛，以及局部相似源码不误判等边界。

## 连接限流策略

协同编辑通道和 Agent 流式响应通道都是长连接 WebSocket。项目编辑页会为页面代码、页面 Schema、手绘 scene、项目 Schema、页面树和画布布局等资源同时建立多个协同房间连接；这些连接属于单次编辑会话的正常工作负载，不进入 agent-service 的普通 HTTP 全局请求限流桶。

普通 HTTP API 仍保留全局 rate limit，用于保护消息发送、项目版本、flush 等短请求接口。WebSocket 握手路由只豁免这层按请求数统计的限流，房间参数校验、Session/Workspace 归属校验、连接清理和持久化边界仍由协同服务自身负责。

## 2026-06-27 同步状态抖动约束

`useCollabDocument` 将 HocuspocusProvider 的短暂 `disconnected` 状态视为自动重连过程，不立即把可见状态降级为“离线待同步”。只有断连持续超过前端阈值且仍未恢复时，才展示“离线待同步”；进入真实离线后，自动重试的 `connecting` 状态不再反复覆盖离线状态，直到连接恢复为止。该约束用于避免项目编辑页同时聚合多个协同房间时，在“连接中”和“离线待同步”之间反复抖动。
