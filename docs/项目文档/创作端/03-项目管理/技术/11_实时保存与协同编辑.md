# 实时保存与协同编辑

> 更新日期：2026-07-02

## 定位

实时协同层服务于 [项目工作区](./03_项目工作区_v2.md)：多人默认编辑项目级共享当前态，而不是用户私有临时副本，也不是直接操作历史快照。业务需求见 [实时协作保存需求](../实时协作保存_需求文档.md)，历史记录由 [版本管理](./04_版本管理.md) 在自动检查点、命名版本、发布快照和恢复快照场景生成。

## 协作边界

- 协同房间以项目、Workspace、资源路径为边界，不同页面文件彼此隔离。
- 连接必须携带 Session、Workspace 和资源信息，服务端校验 Session 属于当前项目和 Workspace 后才允许加入。
- 项目工作区是共享编辑边界。默认编辑页会把不同用户的 Session 绑定到同一个 `scope=live` Workspace；服务端不再用单一 workspace owner 阻断跨用户协作。
- 资源范围限制在页面代码、页面 Schema、项目级 Schema、页面树、画布布局和工作空间知识文档，避免协同通道写入任意文件。
- 知识文档协同只放行 `knowledge/*.md`、`knowledge/*.markdown` 和 `knowledge/*.mdown` 这类单层用户文档；阅读地图、模板索引和嵌套知识服务产物不进入创作端协同写入范围。
- 本能力先面向单实例部署；多实例部署需要额外的共享协调层。

协同房间的资源类型来自共享类型 `CollabResourceKind`，当前包括 `page-code`、`page-schema`、`project-schema`、`workspace-tree`、`canvas-layout` 和 `knowledge-document`。资源类型和资源路径必须同时匹配允许规则，避免前端通过协同通道越权写入工作空间内的其他文件。

## 房间与连接模型

前端通过 `useCollabDocument` 为某个资源构造房间描述。房间描述由项目、Workspace、Session、资源路径和资源类型组成；资源路径会被编码进 WebSocket 房间名，实际鉴权仍以查询参数里的房间描述为准。

agent-service 的协同路由负责三类入口：

| 入口类型 | 作用 |
|:---------|:-----|
| WebSocket 房间 | 建立 Yjs 同步和 presence 通道 |
| 单资源 flush | 将指定房间的内存草稿立即写回 Workspace |
| Workspace flush-all | 调用者用有效 Session 证明可访问该 Workspace，服务端写回当前 Workspace 下所有活跃房间 |

服务端创建房间时，会先从 Workspace 文件读取初始内容，放入 Yjs 文本。后续编辑产生的 Yjs 更新在房间内广播，同时标记房间为 dirty，等待短延迟保存或显式 flush。由于 Yjs 会把不同文档里的同文本插入视为不同操作，服务端在收到非服务端同步后会检查房间文本是否变成当前磁盘文件的整数倍重复；若命中，立即以磁盘文件重置房间并取消保存，避免“重新打开项目后同一页面源码被拼接成 2 倍、4 倍”的协同合并事故。

如果房间已经在服务进程内存在，服务端优先复用内存房间，但房间必须记录自己最近一次加载或写入的文件哈希。复用房间和写回文件前都要重新核对磁盘资源：如果磁盘文件已经由 AI 工具或其他非协同写入更新，且内容不同于房间当前文本，磁盘文件是新的权威当前态，服务端需要把磁盘内容重新灌入协同文档，并取消旧房间的待保存状态。

## 数据流

用户打开可协同文件时，前端会连接 agent-service 的协同 WebSocket。服务端从 Workspace 文件读取初始草稿内容，放入协同文档。多个浏览器对同一文档的编辑会在内存中合并，再按短延迟写回 Workspace 文件。

当用户命名版本时，版本接口会先通过 author-site 的 flush coordinator 调用协同服务 flush 当前 Workspace，再由版本管理生成命名版本。前端仍可提前发起 flush 以优化交互反馈，但可靠边界在后端关键动作入口。flush coordinator 在调用 agent-service 前会续期仍处于编辑状态的 Session，避免长时间打开编辑页后本地仍可编辑、但协同服务因 Session 超过默认有效期而拒绝写回，导致 AI 修复停留在 live Workspace、项目基准工作区没有被推进。

发布项目时也会先执行 Workspace 级 flush，并在发布前生成发布快照。即使当前浏览器没有本地未同步标记，只要协同房间里还有未落盘内容，也会先写回 Workspace，再进入发布流程。自动检查点、保存/合并 Session 和页面恢复同样使用这一后端前置协议；本地项目包获取由 CLI `project pull` 负责，不再通过创作端编辑页触发。

发布 API 是发布前同步的可靠边界。前端可以携带当前 Session 和 Workspace，但不需要先发起一次重复 flush；如果页面上的 Session 已过期、被归档或被清理，而 Workspace 元数据仍证明它属于当前用户和当前项目，发布 API 会先为这个 Workspace 续建有效 Session，再用新 Session 执行 Workspace 级 flush、同步项目工作区和发布。若 Workspace 本身不存在或归属不匹配，发布仍会被阻断。若请求没有携带 Workspace，只有项目基准工作区已确认追上项目工作区时才能发布；否则必须返回同步要求，避免发布落后的项目工作区。

编辑页不再把右上角“保存”作为防丢入口。顶部只展示自动保存与协同状态；手动标记历史节点的入口放在历史记录区，按钮文案为“命名此版本”。该操作会先 flush 当前内容，再创建页面版本和项目版本快照。用户产生本地编辑后，前端会把 Workspace 标记为待落盘，并通过 author-site 同源轻量接口触发 Workspace flush；flush 成功后还必须把 Session Workspace 同步到项目当前 Workspace，只有两个步骤都成功后才能展示“已自动保存”。退出时若仍有待落盘编辑、flush 失败、项目当前 Workspace 同步失败、连接中、同步中、离线待同步或协同异常，才提示最新修改尚未确认同步完成。同源接口在服务端调用 agent-service，避免浏览器退出路径直接依赖 agent-service 的跨域地址和环境配置。

重新进入项目时，创作端必须优先复用当前用户、当前项目下的活跃 Session，并在复用前确保该 Session 已绑定项目工作区。没有活跃 Session 时，新 Session 也绑定项目工作区，而不是为当前用户创建私有副本。复用活跃 Session 前必须确认其 `workspaceId` 仍能定位到 Workspace 目录；如果目录已被清理或缺失，应把该 Session 标记为断链并跳过，避免编辑页继续加载一个无法读取文件、也无法 flush 的旧会话。

AI 或文件工具直接写入 Workspace 文件后，agent-service 的工具结果 hook 会通知协同房间管理器重载同一 Workspace 和资源路径下的活跃房间；服务端是这类外部写入回流到协同房间的权威入口。编辑页收到 AI 文件变更后只把文件系统最新内容应用到当前页面状态、页面文件映射、refs 和预览缓存，不再主动把同一份完整文本写入 Yjs 文本；否则多个浏览器或多个会话同时执行“删除旧文本再插入全文”时，Yjs 会把这些全文插入视作独立操作并合并成重复内容。页面切换或新增删除导致当前页面变化时，仍要同步维护 `codeRef` / `schemaRef`，避免后续保存基于旧引用。前端收到 AI 文件变更后还必须把 Workspace 标记为待同步，让既有自动保存链路执行 Workspace flush 和项目基准工作区同步；否则修复可能只停留在 live Workspace，重新打开项目时仍有机会从旧项目基准内容恢复。若工具事件返回的是当前页相对路径，例如 `index.tsx` 或 `config.schema.json`，编辑页要先按当前页面归一成 Workspace 内页面路径，再判断是否需要刷新页面状态和触发同步；服务端在处理外部文件变更通知时也会对这类短路径做保守回退：只有 Workspace 根部不存在同名文件时，才匹配活跃页面协同房间，避免把真正的根目录文件误判成页面资源。后端的文件哈希基线是兜底保护：即使前端刷新或工具通知缺失，旧协同房间也不能覆盖一个已被外部写入推进过的磁盘文件。

## 持久化策略

协同服务采用“内存合并、延迟落盘、关键动作强制落盘”的策略：

| 阶段 | 行为 |
|:-----|:-----|
| 加入房间 | 读取 Workspace 中的目标资源作为初始文本 |
| 复用空房间 | 空且未变更的内存房间会重新对齐非空 Workspace 文件 |
| 实时编辑 | Yjs 更新广播给同房间连接，并标记房间 dirty |
| 延迟保存 | 默认约 1 秒后将 dirty 文本写回项目工作区文件 |
| 外部文件写入 | AI 工具写入 Workspace 文件后，服务端按 Workspace 和资源路径重载匹配协同房间；`index.tsx` / `config.schema.json` 这类当前页短路径在根部无同名文件时会回退匹配活跃页面房间 |
| 旧房间保护 | flush 前发现磁盘文件哈希不同于房间基线时，放弃旧文本写回并以磁盘文件重置房间 |
| 重复内容保护 | 非服务端同步导致房间文本等于当前磁盘文件的 2 倍或更多倍完整重复时，立即重置为磁盘文件，不调度保存；若重复对象是完整 JSON Schema，也会先归一化为单份可解析 JSON 再落盘 |
| 显式 flush | 命名版本、自动检查点、发布、保存/合并 Session 和恢复前立即写回 Workspace 下所有活跃房间 |
| 基准同步 | 关键动作或显式持久化时，将项目工作区推进到项目基准工作区 |
| 自动检查点 | 停止编辑 5 分钟或持续编辑 30 分钟且存在变更时，先推进项目基准工作区，再生成自动保存记录 |
| 房间清理 | 空闲房间被清理前先 flush，再销毁内存文档和 presence |

文件写入通过临时文件加重命名完成，避免中途失败留下半截内容。写入成功后会刷新 Workspace 元信息的更新时间，用于后续工作空间活跃状态判断。

agent-service 的协同持久化默认读取仓库根 `data/`，而不是当前包目录下的 `data/`。本地开发时 author-site 与 agent-service 可以从不同工作目录启动，但两者必须指向同一份 `data/sessions` 和 `data/workspaces`；生产或 Docker 环境仍以显式 `DATA_DIR` 为准。若两端数据目录不一致，Workspace flush 会因找不到 Session 或 Workspace 被拒绝，编辑页应展示同步失败而不能误报已自动保存。

项目工作区的标准形态是 `data/workspaces/projects/{projectId}/{workspaceId}`，并带有 `.workspace.json` 元数据。旧的 `data/workspaces/{userId}/{projectId}/{workspaceId}` 仅作为分支工作区或 legacy Workspace 兼容读取。协同持久化优先读取元数据校验归属；若历史数据或异常写入导致 `.workspace.json` 缺失，服务端可按嵌套目录名定位 `workspaceId`，并用目录中的 `projectId` 段兜底校验项目归属。这样可以避免已有 Workspace 内容存在但 flush 误报 `WORKSPACE_NOT_FOUND`。

自动检查点由编辑页调度。调度器监听当前页面内容、Schema、页面数量和画布脏状态；当内容签名变化后重新计算 5 分钟空闲窗口，同时保留本轮持续编辑的开始时间。触发检查点前会先 flush 当前 Workspace 下的协同房间和画布布局，再把项目工作区同步到项目基准工作区，随后从项目基准工作区复制历史快照。自动检查点不会结束 Session，也不会清理 Workspace；若基准同步失败，检查点接口必须返回失败，前端不能把状态标记为已自动保存。自动检查点、保存/合并 Session 和发布快照从已同步 live Workspace 生成新的项目级版本后，项目级版本创建函数必须同步推进该 Workspace 元数据里的 `baseVersion`，否则下一次自动保存、命名版本后的合并或发布前同步会把同一个活跃 Workspace 误判为过期。

如果历史数据中已经出现“最新项目版本由当前 Session 生成，但 live Workspace 的 `baseVersion` 仍停留在上一版本”的不一致状态，关键同步入口只能在最新版本 `sessionId` 与当前请求 Session 完全一致、且 Workspace 仍是项目 active Workspace 时修复基线并重试同步。其他来源的新版本仍按过期 Workspace 处理，避免旧工作区覆盖他人或系统生成的新项目当前态。

## 权限与路径安全

协同写入前必须完成三层校验：

| 校验 | 目的 |
|:-----|:-----|
| Session 校验 | 确认会话存在、未过期，且归属于当前项目和 Workspace |
| Workspace 校验 | 确认 Workspace 存在，并与项目关系一致 |
| Resource 校验 | 确认资源路径没有越级片段，并匹配对应资源类型的允许规则 |

校验失败时，WebSocket 使用策略错误关闭连接；flush 接口返回标准错误响应。这样可以把“协作连接失败”和“草稿写入失败”都归因到明确的项目、会话或资源边界，而不是静默丢弃用户编辑。

Workspace 级 flush 不接收资源路径，只校验调用者 Session 对项目和 Workspace 的访问关系。校验通过后，它会刷新该 Workspace 下所有活跃协同房间，包括其他协作者用不同 Session 打开的房间；没有活跃房间时返回 no active room，关键动作可以继续使用磁盘上的 Workspace 当前态。agent-service 不可达或 flush 返回错误时，author-site 关键动作必须阻断，避免生成落后于协同草稿的快照、发布产物或导出包。

## 状态与呈现

前端展示连接中、已自动保存、同步中、离线待同步和协同异常状态。协作者信息通过 presence 同步，用于展示在线用户和远端光标。编辑页顶部展示项目级在线状态，文件编辑弹窗展示当前资源的协同状态。

编辑页和文件编辑弹窗读取可协同资源时，必须区分“连接尚未完成时的默认空值”和“真实远端内容”。当本地或 REST 接口已经拿到非空页面代码、Schema、项目配置、页面树或画布布局，而协同同步返回空文本时，前端应先用非空内容回填协同文档，不能直接把编辑器覆盖为空。用户在编辑器中主动清空内容属于显式编辑行为，可按正常协同更新继续处理。

编辑页复用 `workspace-tree.json` 协同房间的 presence 展示项目在线用户，避免为在线状态额外打开独立房间。页面树的新增、复制、删除、重命名和排序仍通过既有项目 API 落盘，操作成功后前端会重新读取 `workspace-tree.json` 并写入协同文档，其他打开同一 Workspace 的编辑页会据此刷新页面与文件夹结构。

画布布局使用 `.canvas-layout.json` 协同房间同步。远端布局变更会以非脏状态应用到当前画布，不标记页面代码、Schema 或页面树等非画布工作区内容未保存。本地画布编辑产生未保存画布变更时，会把布局封装为 `{ version, projectId, updatedAt, state }` 写入协同草稿，由服务端 debounce 落盘。画布状态保存以 hook 内部的最新状态引用为准，定时保存和退出前强制保存都读取同一份最新状态，避免移动页面、编辑文本后立刻退出时落盘到上一帧布局。通过 `/api/sessions/:sessionId/canvas-layout` 保存画布布局时，会同步写入当前 Session 目录、当前 Workspace 和项目基准工作区的 `workspace/.canvas-layout.json`；重新打开项目创建新 Workspace 时可以从项目工作区恢复上次直接退出前的画布布局。退出编辑页时会先强制保存画布布局；只有页面代码、Schema、页面树等非画布工作区内容仍有未保存改动时，协同 Workspace flush 才会阻塞退出。

知识文档编辑复用同一协同通道。知识库弹窗中的 Markdown 富文本编辑器把当前文档正文序列化到 `knowledge/{fileName}` 对应的 Yjs 文本资源；保存时先 flush 协同草稿，再调用知识文档 API 更新描述和 manifest 元数据，避免正文与知识索引元信息分离。

## 异常与降级

协作异常分为连接异常、权限异常和落盘异常。连接异常时，前端应展示离线或错误状态，避免用户误以为远端已经收到更新。权限异常通常来自会话过期、项目不匹配、Workspace 不匹配或资源路径不被允许，此时应阻止继续加入房间。落盘异常发生在 flush 或延迟保存阶段，需要保留当前编辑态并提示用户重试关键动作。

协同状态不替代历史记录状态。内容即使已经自动保存，也只是 Workspace 当前状态；自动检查点、命名版本、发布快照和恢复快照才形成可在历史记录中恢复的快照。

## 与既有能力的关系

- Session 生命周期仍由 [生命周期设计](./02_生命周期设计_v2.md) 管理。
- `scope=live` 的项目工作区不随 Session 清理；分支工作区和 legacy Workspace 清理仍以活跃 Session 引用和过期策略为准。
- AI 或非协同文件写入仍可继续使用原工作区文件 API；写入成功后必须让服务端协同房间对齐磁盘当前态，避免旧内存文本在后续 flush 中回写。
- 配置与预览如何消费协作草稿，见 [协同草稿驱动预览](../../04-配置与预览/技术/08_协同草稿驱动预览.md)。

## 连接限流策略

协同编辑通道和 Agent 流式响应通道都是长连接 WebSocket。项目编辑页会为页面代码、页面 Schema、项目 Schema、页面树和画布布局等资源同时建立多个协同房间连接；这些连接属于单次编辑会话的正常工作负载，不进入 agent-service 的普通 HTTP 全局请求限流桶。

普通 HTTP API 仍保留全局 rate limit，用于保护消息发送、项目版本、flush 等短请求接口。WebSocket 握手路由只豁免这层按请求数统计的限流，房间参数校验、Session/Workspace 归属校验、连接清理和持久化边界仍由协同服务自身负责。
## 2026-06-27 同步状态抖动约束

`useCollabDocument` 将 y-websocket 的短暂 `disconnected` 事件视为自动重连过程，不立即把可见状态降级为“离线待同步”。只有断连持续超过前端阈值且仍未恢复时，才展示“离线待同步”；进入真实离线后，自动重试的 `connecting` 事件不再反复覆盖离线状态，直到连接恢复为止。该约束用于避免项目编辑页同时聚合多个协同房间时，在“连接中”和“离线待同步”之间反复抖动。
