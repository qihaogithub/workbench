# 实时保存与协同编辑

> 更新日期：2026-06-27

## 定位

实时协同层服务于 [草稿工作区](./03_草稿工作区_v2.md)：多人编辑的是同一个 Workspace 草稿，而不是直接修改项目正式工作区。业务需求见 [实时协作保存需求](../实时协作保存_需求文档.md)，项目版本仍由 [版本管理](./04_版本管理.md) 在用户手动创建版本时生成。

## 协作边界

- 协同房间以项目、Workspace、资源路径为边界，不同页面文件彼此隔离。
- 连接必须携带 Session、Workspace 和资源信息，服务端校验三者归属一致后才允许加入。
- 资源范围限制在页面代码、页面 Schema、项目级 Schema、页面树、画布布局和工作空间知识文档，避免协同通道写入任意文件。
- 知识文档协同只放行 `knowledge/*.md`、`knowledge/*.markdown` 和 `knowledge/*.mdown` 这类单层用户文档；阅读地图、模板索引和嵌套知识服务产物不进入创作端协同写入范围。
- 本能力先面向单实例部署；多实例部署需要额外的共享协调层。

协同房间的资源类型来自共享类型 `CollabResourceKind`，当前包括 `page-code`、`page-schema`、`project-schema`、`workspace-tree`、`canvas-layout` 和 `knowledge-document`。资源类型和资源路径必须同时匹配允许规则，避免前端通过协同通道越权写入工作空间内的其他文件。

## 房间与连接模型

前端通过 `useCollabDocument` 为某个资源构造房间描述。房间描述由项目、Workspace、Session、资源路径和资源类型组成；资源路径会被编码进 WebSocket 房间名，实际鉴权仍以查询参数里的房间描述为准。

agent-service 的协同路由负责三类入口：

| 入口类型 | 作用 |
|:---------|:-----|
| WebSocket 房间 | 建立 Yjs 同步和 presence 通道 |
| 单资源 flush | 将指定房间的内存草稿立即写回 Workspace |
| Workspace flush-all | 在保存、创建版本或发布前写回当前 Workspace 下所有匹配房间 |

服务端创建房间时，会先从 Workspace 文件读取初始内容，放入 Yjs 文本。后续编辑产生的 Yjs 更新在房间内广播，同时标记房间为 dirty，等待短延迟保存或显式 flush。

## 数据流

用户打开可协同文件时，前端会连接 agent-service 的协同 WebSocket。服务端从 Workspace 文件读取初始草稿内容，放入协同文档。多个浏览器对同一文档的编辑会在内存中合并，再按短延迟写回 Workspace 文件。

当用户点击保存项目时，前端先要求协同服务 flush 当前草稿，随后继续走原有 Session 保存流程。这样版本快照捕获到的是最新草稿，而实时编辑本身不会频繁生成版本。

发布项目时也会先执行 Workspace 级 flush。即使当前浏览器没有本地未保存标记，只要协同房间里还有未落盘内容，也会先写回 Workspace，再进入发布流程。

编辑页不再把右上角“保存”作为防丢入口。顶部只展示自动保存与协同状态；手动生成版本的入口放在版本历史区，按钮文案为“创建版本”。该操作会先 flush 协同草稿，再创建页面版本和项目版本快照。退出时若存在尚未创建版本的草稿，只提示用户是否创建一个可回退版本，而不是提示草稿可能丢失。

## 持久化策略

协同服务采用“内存合并、延迟落盘、关键动作强制落盘”的策略：

| 阶段 | 行为 |
|:-----|:-----|
| 加入房间 | 读取 Workspace 中的目标资源作为初始文本 |
| 实时编辑 | Yjs 更新广播给同房间连接，并标记房间 dirty |
| 延迟保存 | 默认约 1 秒后将 dirty 文本写回 Workspace 文件 |
| 显式 flush | 创建版本、保存页面版本或发布前立即写回相关房间 |
| 房间清理 | 空闲房间被清理前先 flush，再销毁内存文档和 presence |

文件写入通过临时文件加重命名完成，避免中途失败留下半截内容。写入成功后会刷新 Workspace 元信息的更新时间，用于后续工作空间活跃状态判断。

## 权限与路径安全

协同写入前必须完成三层校验：

| 校验 | 目的 |
|:-----|:-----|
| Session 校验 | 确认会话存在、未过期，且归属于当前项目 |
| Workspace 校验 | 确认 Workspace 存在，并与项目和用户关系一致 |
| Resource 校验 | 确认资源路径没有越级片段，并匹配对应资源类型的允许规则 |

校验失败时，WebSocket 使用策略错误关闭连接；flush 接口返回标准错误响应。这样可以把“协作连接失败”和“草稿写入失败”都归因到明确的项目、会话或资源边界，而不是静默丢弃用户编辑。

## 状态与呈现

前端展示连接中、草稿已实时保存、同步中、离线待同步和协同异常状态。协作者信息通过 presence 同步，用于展示在线用户和远端光标。编辑页顶部展示项目级在线状态，文件编辑弹窗展示当前资源的协同状态。

编辑页复用 `workspace-tree.json` 协同房间的 presence 展示项目在线用户，避免为在线状态额外打开独立房间。页面树的新增、复制、删除、重命名和排序仍通过既有项目 API 落盘，操作成功后前端会重新读取 `workspace-tree.json` 并写入协同文档，其他打开同一 Workspace 的编辑页会据此刷新页面与文件夹结构。

画布布局使用 `.canvas-layout.json` 协同房间同步。远端布局变更会以非脏状态应用到当前画布，本地画布编辑产生未保存画布变更时，会把布局封装为 `{ version, projectId, updatedAt, state }` 写入协同草稿，由服务端 debounce 落盘。

知识文档编辑复用同一协同通道。知识库弹窗中的 Markdown 富文本编辑器把当前文档正文序列化到 `knowledge/{fileName}` 对应的 Yjs 文本资源；保存时先 flush 协同草稿，再调用知识文档 API 更新描述和 manifest 元数据，避免正文与知识索引元信息分离。

## 异常与降级

协作异常分为连接异常、权限异常和落盘异常。连接异常时，前端应展示离线或错误状态，避免用户误以为远端已经收到更新。权限异常通常来自会话过期、项目不匹配、Workspace 不匹配或资源路径不被允许，此时应阻止继续加入房间。落盘异常发生在 flush 或延迟保存阶段，需要保留当前编辑态并提示用户重试关键动作。

协同状态不替代版本状态。草稿即使已经实时保存，也只是 Workspace 当前状态；只有创建版本后，才形成可在版本历史中恢复的快照。

## 与既有能力的关系

- Session 生命周期仍由 [生命周期设计](./02_生命周期设计_v2.md) 管理。
- Workspace 清理策略仍以活跃 Session 引用为准。
- AI 或非协同文件写入仍可继续使用原工作区文件 API；后续可在这些写入点追加协同房间刷新或冲突提示。
- 配置与预览如何消费协作草稿，见 [协同草稿驱动预览](../../04-配置与预览/技术/08_协同草稿驱动预览.md)。

## 连接限流策略

协同编辑通道和 Agent 流式响应通道都是长连接 WebSocket。项目编辑页会为页面代码、页面 Schema、项目 Schema、页面树和画布布局等资源同时建立多个协同房间连接；这些连接属于单次编辑会话的正常工作负载，不进入 agent-service 的普通 HTTP 全局请求限流桶。

普通 HTTP API 仍保留全局 rate limit，用于保护消息发送、项目版本、flush 等短请求接口。WebSocket 握手路由只豁免这层按请求数统计的限流，房间参数校验、Session/Workspace 归属校验、连接清理和持久化边界仍由协同服务自身负责。
## 2026-06-27 同步状态抖动约束

`useCollabDocument` 将 y-websocket 的短暂 `disconnected` 事件视为自动重连过程，不立即把可见状态降级为“离线待同步”。只有断连持续超过前端阈值且仍未恢复时，才展示“离线待同步”；进入真实离线后，自动重试的 `connecting` 事件不再反复覆盖离线状态，直到连接恢复为止。该约束用于避免项目编辑页同时聚合多个协同房间时，在“连接中”和“离线待同步”之间反复抖动。
