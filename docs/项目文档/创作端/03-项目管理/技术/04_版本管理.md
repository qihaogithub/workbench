# 会话与版本管理 - 版本管理

> 版本：v3.0
> 更新日期：2026-07-06

---

## 一、版本管理概述

版本管理负责把当前工作空间在关键时刻固化为不可变快照，为历史查看、回退、发布追溯提供依据。用户日常编辑依赖 [实时协作保存](../实时协作保存_需求文档.md) 保持当前工作状态；版本历史只记录需要回退和追溯的节点。

### 1.1 核心功能

| 功能 | 说明 |
|:-----|:-----|
| 自动保存记录 | 系统按策略沉淀的普通历史检查点 |
| 命名版本 | 用户主动标记的重要阶段 |
| 发布快照 | 发布前自动生成，用于线上状态追溯 |
| 恢复快照 | 从历史恢复后自动生成，保留操作轨迹 |
| 资源级历史 | 页面和知识文档拥有独立资源版本，可单独查看和恢复 |
| 内容图提交 | 项目级时间线记录一次操作影响了哪些资源，并作为发布和物化追溯点 |
| 版本清理 | 自动清理超出限制的旧记录，优先淘汰自动保存记录 |

### 1.2 设计原则

| 原则 | 说明 |
|:-----|:-----|
| 当前状态优先 | 自动保存负责防丢失，版本历史负责回退与追溯 |
| 快照不可变 | 历史记录创建后不修改内容，只新增后续记录 |
| 发布可追溯 | 每次发布必须绑定一个发布快照 |
| 恢复可审计 | 恢复不会覆盖旧记录，而是新增恢复快照 |
| 单资源回退 | 恢复单个页面或知识文档只推进该资源指针，不改写无关资源历史 |
| 内容图唯一 | 页面和知识文档资源历史只写入内容图，旧 `pageVersions` 不再作为持久化入口 |
| 先同步再快照 | 关键动作必须先 flush 项目工作区，并推进项目基准工作区，再生成历史节点或发布产物 |
| 基线随版本推进 | 项目级版本创建函数负责把已同步 live Workspace 的 `baseVersion` 提升到新版本 |
| 数量受控 | 最多保留 50 条，清理时优先移除自动保存记录 |

## 二、历史记录数据结构

项目历史分为两层：

- 项目级快照继续沿用 `VersionInfo`，用于整体项目恢复和发布快照。
- 资源级历史使用内容图：资源版本记录单个对象的不可变内容，项目提交记录一次操作把哪些资源指向了哪些版本。

项目尚未上线，不保留旧页面历史兼容层。页面历史不再写入 `project.json.pageVersions`，旧 `demos/{demoId}/versions` REST 路由和旧 `page version-*` CLI 命令也不再作为入口。

| 字段 | 说明 |
|:-----|:-----|
| `versionId` | 全局递增版本号，例如 `v1`、`v2` |
| `type` | `auto_checkpoint`、`named_version`、`publish_snapshot`、`restore_snapshot` |
| `savedAt` | 快照创建时间 |
| `savedBy` | 创建者，自动任务可使用系统或触发用户 |
| `sessionId` | 关联会话或系统动作标识 |
| `snapshotPath` | 快照目录 |
| `fileCount` | 快照文件数量 |
| `note` | 用户备注或系统说明 |

页面历史在前端仍可投影为页面版本列表，但其真实 ID 来自页面 `ResourceVersion`，不占用项目级 `v1`、`v2` 快照序列。

### 2.1 内容图结构

内容图存放在项目目录的 `content/` 下：

| 子目录 | 说明 |
|:-----|:-----|
| `state.json` | 当前 head commit、物化状态和最后更新时间 |
| `commits/` | 项目提交日志，每个提交包含资源指针集合和变更摘要 |
| `resources/{kind}/{resourceId}/` | 单资源不可变版本 |
| `blobs/` | 按内容哈希存储页面代码、Schema、原型文件和知识文档正文 |
| `refs/references.json` | 当前可查询的资源引用关系 |
| `materialization/manifest.json` | 最近一次物化使用的 commit 和资源 hash 清单 |

首轮内容图覆盖 `page` 和 `knowledge_document`。`canvas`、`asset` 和 `project_config` 已在类型与目录结构中预留，后续可按同一模型扩展。

### 2.2 资源版本与项目提交

资源版本是单个对象的不可变内容。页面资源版本包含页面元数据、运行时类型、页面 Schema、高保真 React 代码或 HTML/CSS 原型文件；手绘页面在创建页面资源版本时还可以携带已验证的 patch 摘要，只记录操作数、是否有基线和前后节点数，不写入 scene 或 patch operations 内容。OpenPencil 保存成功后，前端只在当前 scene 仍匹配最近保存 draft 时，才会把这份摘要带入命名版本或自动检查点的页面资源版本创建请求；知识文档资源版本包含文档 manifest 元数据和 Markdown 正文。

项目提交是一次用户可理解的变化。它保存资源指针集合、变更资源列表、可见性、操作意图和审计 actor。恢复单个资源时，系统新增一个 `restore` 提交，把该资源指针指回目标资源版本；旧资源版本和旧提交都不被修改。

## 三、快照触发策略

| 触发源 | 条目类型 | 行为 |
|:------|:---------|:-----|
| 停止编辑 5 分钟且有变更 | 自动保存记录 | 先同步项目工作区，为当前页面写入资源版本，再从项目基准工作区生成普通检查点，不结束编辑会话 |
| 持续编辑超过 30 分钟且有变更 | 自动保存记录 | 先同步项目工作区，为当前页面写入资源版本，再从项目基准工作区生成普通检查点，不结束编辑会话 |
| AI 批量修改完成且有文件变更 | 自动保存记录 | 从项目工作区生成普通检查点 |
| 用户点击“命名此版本” | 命名版本 | 先同步项目工作区，再生成用户可识别的历史记录 |
| 用户点击发布 | 发布快照 | 先同步项目工作区，再生成发布快照并绑定发布结果 |
| 用户恢复历史记录 | 恢复快照 | 先同步项目工作区，再恢复内容并生成新的历史记录 |
| 用户保存页面历史 | 资源版本 | 写入页面资源版本并生成语义提交 |
| 用户保存知识文档 | 资源版本 | 写入知识文档资源版本、生成语义提交 |
| 用户删除知识文档 | Tombstone 提交 | 工作区移除文档，同时内容图把该文档标记为删除，历史仍可恢复 |

当前实现已落地命名版本、发布快照、恢复快照和定时自动检查点。自动检查点先同步项目工作区，把当前页面保存为资源版本，再从项目基准工作区复制历史记录；它不把会话标记为已保存，也不清理 `scope=live` Workspace。所有从已同步项目基准工作区创建项目级版本的入口，都由项目级版本创建函数统一推进当前 live Workspace 的版本基线；保存/合并 Session 会复用同一版本创建函数，发布快照也会在项目基准工作区已追上 live Workspace 时推进基线。发布前同步会在请求携带有效 Workspace 时先强制 Workspace 级 flush，再把项目工作区推进到项目基准工作区并生成发布快照。如果发布请求没有携带 Workspace，只有项目基准工作区已确认追上项目工作区时才能继续发布。

## 四、核心流程

### 4.1 命名版本

用户在历史记录区选择“命名此版本”时，系统先通过后端关键动作入口 flush 当前项目工作区下的协作内容，再推进项目基准工作区并复制为快照。快照写入历史记录后，用户可在历史面板中看到该命名版本，当前 Session 继续保持编辑状态。

命名版本会形成页面级历史节点，并随后通过保存/合并 Session 形成项目级历史节点。项目级历史节点创建成功后，当前 live Workspace 的 `baseVersion` 必须指向这个新版本，确保后续自动保存和发布前同步仍能识别它是项目最新版本上的连续编辑，而不是旧版本工作区。

如果当前页面是 OpenPencil 手绘页面，且最近一次“保存手绘”已经通过服务端 patch 回放校验，命名版本会在当前 scene 未被后续本地变更替换时附带 `sketchPatchSummary`。该摘要只用于资源版本 metadata 审计，不参与恢复内容，不携带 scene 或 patch operations。

### 4.2 发布快照

发布入口不要求用户先手动保存。若请求携带当前 Workspace，系统先 flush 该 Workspace 下所有活跃协同房间，再把项目工作区同步回项目基准工作区，并记录发布快照；随后发布流程从项目基准工作区生成发布产物，发布产物中的 `publishedVersion` 指向该发布快照。

发布结果还会绑定一个 protected 内容图提交 `commitId`。发布产物和发布状态因此既能追溯整体项目快照，也能追溯当时每个资源指针和资源 hash 清单。发布快照不改变页面内容，但会成为新的项目级最新版本；因此当项目基准工作区已确认追上 active live Workspace 时，发布快照也必须推进该 Workspace 的 `baseVersion`。若请求没有可 flush 的 Workspace 且项目基准工作区落后于项目工作区，服务端必须阻断发布。

发布产物中的图片资源本地化规则见 [12_发布资源本地化.md](./12_发布资源本地化.md)。发布快照负责确定发布内容版本，资源本地化负责保证发布后的图片由本项目服务器稳定提供。

### 4.3 恢复快照

用户恢复任一历史记录时，系统先确认当前项目工作区已同步并保存恢复前检查点，再从目标快照恢复项目基准工作区和项目工作区，然后新增一条恢复快照。旧记录保持不变，因此用户可以继续追溯“恢复到哪个版本”和“恢复后形成的新状态”。

单资源恢复不走整体项目覆盖。页面恢复会从目标页面资源版本读取页面闭包，只写回该页面目录和页面 Schema，并生成语义 restore commit；知识文档恢复只写回对应 Markdown 文件和 manifest 条目，并生成语义 restore commit。两类恢复都不会修改无关页面、画布布局或其他知识文档。

页面资源恢复在有编辑会话时必须使用前端请求携带的当前 `workspaceId` 作为同步目标；`sessionId` 只负责会话归属、过期校验和协同 flush。这样用户切换过对话或历史会话后，恢复动作仍落在当前 live Workspace 上，不会因为旧 Session 元数据里的 workspace 已不再是项目 active workspace 而误报过期。服务端写回 Session Workspace 前会校验该 Workspace 仍属于当前项目，恢复成功后再把该 Workspace 的 `baseVersion` 推进到恢复生成的新项目版本，并同步项目基准工作区。

创作端单页面预览工具栏提供资源历史入口。当前预览对象是页面时，历史按钮打开该页面的资源版本列表并支持只恢复该页面；如果页面仍有未保存改动，前端会先同步工作区、写入一个页面资源版本，再打开历史列表，避免用户刚编辑完看到空历史。当前预览对象是文档时，按钮会解析文档节点中正在展示的知识文档 ID，并只恢复该知识文档。恢复完成后前端刷新当前页面或知识文档内容，保持单页面预览与内容图恢复结果一致。

### 4.4 内容图物化

内容图是资源历史的权威记录，项目基准工作区是预览、发布和 CLI 操作的运行缓存。物化会读取目标 commit 的资源指针，把页面和知识文档写回 `workspace/`，并记录 `materialization/manifest.json`。检查模式只验证资源版本与 blob 是否齐全，不写文件。

### 4.5 清理策略

项目历史最多保留 50 条。超过上限时，系统优先删除最旧的自动保存记录；如果自动保存记录不足，再删除最旧的其他记录。命名版本、发布快照、protected 内容图提交和仍被资源版本引用的 blob 拥有更高保留优先级。内容图 blob 清理先提供 dry-run，只有确认无资源版本、发布或受保护提交引用时才可删除。

## 五、与相关能力的关系

- 当前工作状态的持续落盘由 [实时保存与协同编辑](./11_实时保存与协同编辑.md) 负责。
- 工作空间、会话和对话切换语义见 [会话管理需求](../会话管理_需求文档.md)。
- 发布流程会消费版本历史中的发布快照，发布状态仍由发布管理模块维护。
