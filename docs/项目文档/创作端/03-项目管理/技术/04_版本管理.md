# 会话与版本管理 - 版本管理

> 版本：v2.2
> 更新日期：2026-06-30

---

## 一、版本管理概述

版本管理负责把当前工作空间在关键时刻固化为不可变快照，为历史查看、回退、发布追溯提供依据。用户日常编辑依赖 [实时协作保存](../实时协作保存_需求文档.md) 保持当前工作状态；版本历史只记录需要回退和追溯的节点。

### 1.1 核心功能

| 功能 | 说明 |
|:-----|:-----|
| 自动保存记录 | 系统按策略沉淀的普通历史检查点 |
| 命名版本 | 用户主动标记的重要阶段 |
| 发布快照 | 发布前自动生成，用于线上状态追溯 |
| 恢复快照 | 从历史恢复后自动生成，保留操作轨迹 |
| 版本清理 | 自动清理超出限制的旧记录，优先淘汰自动保存记录 |

### 1.2 设计原则

| 原则 | 说明 |
|:-----|:-----|
| 当前状态优先 | 自动保存负责防丢失，版本历史负责回退与追溯 |
| 快照不可变 | 历史记录创建后不修改内容，只新增后续记录 |
| 发布可追溯 | 每次发布必须绑定一个发布快照 |
| 恢复可审计 | 恢复不会覆盖旧记录，而是新增恢复快照 |
| 先同步再快照 | 关键动作必须先确认 Workspace 当前态已同步，再生成历史节点或发布产物 |
| 数量受控 | 最多保留 50 条，清理时优先移除自动保存记录 |

## 二、历史记录数据结构

历史记录沿用 `VersionInfo`，新增 `type` 区分条目来源。旧数据可能没有 `type`，读取时按普通命名版本兼容。

| 字段 | 说明 |
|:-----|:-----|
| `versionId` | 全局递增版本号，例如 `v1`、`v2` |
| `type` | `auto_checkpoint`、`named_version`、`publish_snapshot`、`restore_snapshot` |
| `savedAt` | 快照创建时间 |
| `savedBy` | 创建者，自动任务可使用系统或触发用户 |
| `sessionId` | 关联会话或系统动作标识 |
| `snapshotPath` | 快照目录 |
| `fileCount` | 快照文件数量 |
| `note` | 用户备注或系统说明 |

页面级历史记录复用同一类型字段，并额外记录页面 ID 和页面名称。项目级版本号和页面级版本号共享递增序列，避免同一项目中出现重复版本号。

## 三、快照触发策略

| 触发源 | 条目类型 | 行为 |
|:------|:---------|:-----|
| 停止编辑 5 分钟且有变更 | 自动保存记录 | 先同步当前 Workspace，再从当前会话工作空间生成普通检查点，不结束编辑会话 |
| 持续编辑超过 30 分钟且有变更 | 自动保存记录 | 先同步当前 Workspace，再从当前会话工作空间生成普通检查点，不结束编辑会话 |
| AI 批量修改完成且有文件变更 | 自动保存记录 | 从当前工作空间生成普通检查点 |
| 用户点击“命名此版本” | 命名版本 | 先同步当前 Workspace，再生成用户可识别的历史记录 |
| 用户点击发布 | 发布快照 | 先同步当前 Workspace，再生成发布快照并绑定发布结果 |
| 用户恢复历史记录 | 恢复快照 | 先同步当前 Workspace，再恢复内容并生成新的历史记录 |

当前实现已落地命名版本、发布快照、恢复快照和定时自动检查点。自动检查点只复制当前会话工作空间形成历史记录，不把会话标记为已保存，也不清理工作空间；发布前同步会先强制 Workspace 级 flush，再把当前会话工作空间同步到项目工作空间并生成发布快照。

## 四、核心流程

### 4.1 命名版本

用户在历史记录区选择“命名此版本”时，系统先通过后端关键动作入口 flush 当前 Workspace 下的协作内容，再把当前工作空间复制为快照。快照写入历史记录后，用户可在历史面板中看到该命名版本。

### 4.2 发布快照

发布入口不要求用户先手动保存。若存在当前编辑会话，系统先 flush 当前 Workspace 下所有活跃协同房间，再把会话工作空间同步回项目工作空间，并记录一条自动保存记录；随后发布流程从项目工作空间生成发布快照，发布产物中的 `publishedVersion` 指向该发布快照。

### 4.3 恢复快照

用户恢复任一历史记录时，系统先确认当前 Workspace 已同步，再从目标快照恢复工作空间，然后新增一条恢复快照。旧记录保持不变，因此用户可以继续追溯“恢复到哪个版本”和“恢复后形成的新状态”。

### 4.4 清理策略

项目历史最多保留 50 条。超过上限时，系统优先删除最旧的自动保存记录；如果自动保存记录不足，再删除最旧的其他记录。命名版本和发布快照因此比普通自动保存记录拥有更高保留优先级。

## 五、与相关能力的关系

- 当前工作状态的持续落盘由 [实时保存与协同编辑](./11_实时保存与协同编辑.md) 负责。
- 工作空间、会话和对话切换语义见 [会话管理需求](../会话管理_需求文档.md)。
- 发布流程会消费版本历史中的发布快照，发布状态仍由发布管理模块维护。
