# 会话与版本管理 - 草稿工作区

> 版本：v4.0
> 更新日期：2026-07-01
> 变更说明：默认编辑态升级为项目级共享 active Workspace，用户隔离工作区仅保留为显式 branch/transaction 能力

## 一、工作空间分层

草稿工作区采用三层模型：

| 层级 | 角色 | 说明 |
|:-----|:-----|:-----|
| 项目级 active Workspace | 当前可编辑草稿 | 默认编辑页、实时协同、预览和自动保存共同使用的当前态 |
| canonical Workspace | 物化正式工作区 | 位于项目目录下，供发布、导出、快照和旧工具读取 |
| Snapshot | 不可变历史 | 版本、发布、恢复等关键节点的只读快照 |

Session 不再拥有文件副本。Session 只记录用户、项目、AI 对话和绑定的 `workspaceId`；文件状态由 Workspace 承载。

## 二、目录结构

### 2.1 项目级 active Workspace

```text
workspaces/
└── projects/
    └── {projectId}/
        └── live-{timestamp}-{random}/
            ├── .workspace.json
            ├── workspace-tree.json
            ├── project.config.schema.json
            ├── demos/
            ├── assets/
            └── knowledge/
```

`.workspace.json` 标识该工作区为项目级共享当前态：

| 字段 | 说明 |
|:-----|:-----|
| `workspaceId` | Workspace 唯一标识 |
| `projectId` / `demoId` | 归属项目 |
| `scope` | `live` 表示项目级共享当前态 |
| `status` | `active` 表示当前可编辑 |
| `baseVersion` | 创建当前态时基于的项目版本 |
| `updatedAt` | 最近内容更新时间 |

### 2.2 canonical Workspace

```text
projects/
└── {projectId}/
    ├── project.json
    └── workspace/
```

canonical Workspace 是 active Workspace 的物化结果。关键动作会先同步 active Workspace，再从 canonical Workspace 生成发布产物或历史快照。

### 2.3 branch Workspace

```text
workspaces/
└── {actorId}/
    └── {projectId}/
        └── cli_edit-.../
```

branch Workspace 用于 CLI、AI 大改或实验性事务。它不进入默认实时协同房间，提交前必须校验基准版本，避免覆盖别人已经同步的 active Workspace。

## 三、核心流程

### 3.1 打开项目

用户打开项目时，系统读取项目元数据中的 `activeWorkspaceId`：

- 如果 active Workspace 存在，当前用户的新 Session 直接绑定它。
- 如果不存在，系统从 canonical Workspace 创建一个 `scope=live` 的 active Workspace，并写回项目元数据。
- 如果存在旧的用户私有 Workspace，首次迁移时可选择最近仍有效的旧 Workspace 作为 active Workspace 初始内容。

### 3.2 实时编辑

实时协同房间以项目、Workspace、资源路径为边界。多个用户只要进入同一项目，就会绑定同一个 active Workspace，因此对同一资源的修改会进入同一个协同上下文。

实时保存先写入 active Workspace。自动保存状态表示当前态已经落盘；是否形成历史版本由 [版本管理](./04_版本管理.md) 决定。

### 3.3 关键动作同步

命名版本、自动检查点、发布、导出和恢复前必须执行统一同步：

1. flush active Workspace 下的协同房间。
2. 将 active Workspace 原子同步到 canonical Workspace。
3. 再生成快照、发布产物或恢复记录。

如果协同服务不可达或同步失败，关键动作必须失败，不能消费落后的 canonical Workspace。

### 3.4 Session 生命周期

归档、放弃或过期 Session 只影响对话上下文，不删除 `scope=live` 的 active Workspace。需要回退文件内容时，通过历史恢复重置 active Workspace。

## 四、与分支事务的关系

`project-core` 的编辑事务继续使用隔离 Workspace，但该能力被定义为 branch transaction：

- 适合 CLI、批量维护、AI 沙盒和实验性改动。
- 不作为普通编辑页默认入口。
- 提交时比较事务 `baseVersion` 和项目当前版本。
- 存在冲突时拒绝整体覆盖，要求重新基于最新 active Workspace 开启事务。
