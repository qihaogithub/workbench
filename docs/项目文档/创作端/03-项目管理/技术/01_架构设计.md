# 会话与版本管理 - 架构设计

> 版本：v4.0
> 更新日期：2026-07-03
> 变更说明：统一项目工作区、项目基准工作区、分支工作区、历史快照和发布产物清理边界

---

## 一、整体架构

### 1.1 核心概念

| 概念 | 说明 |
|:-----|:-----|
| **项目 (Project)** | 一个业务实体，持有项目工作区、项目基准工作区和版本历史 |
| **项目工作区 (active Workspace)** | 默认编辑页、实时协同、预览和自动保存共同使用的项目级共享当前态 |
| **项目基准工作区 (canonical Workspace)** | `projects/{id}/workspace/` 下的项目当前态的基准文件区，供发布、导出、快照和旧工具读取 |
| **分支工作区 (branch Workspace)** | CLI、AI 大改或实验性事务使用的显式隔离工作区 |
| **会话 (Session)** | AI 对话上下文，通过 workspaceId 关联工作空间，不直接持有文件 |
| **历史快照 (Snapshot)** | 自动检查点、命名版本、发布和恢复时创建的不可变备份 |
| **代理配置 (.opencode)** | Workspace 中的 OpenCode AI 代理配置，自动注入 |

## 二、目录结构

### 2.1 整体结构

由于本项目是 monorepo 架构，用户数据目录放置于 `packages/author-site/data/` 下，与源代码同属 author-site 包，职责清晰。

```
packages/author-site/
├── data/                        # 用户数据目录
├── data/                        # 用户数据目录
│   ├── projects/                # 项目数据
│   │   └── proj_abc123/
│   │       ├── project.json     # 项目元数据（包含版本历史）
│   │       └── workspace/       # 项目基准工作区
│   │           ├── index.tsx
│   │           ├── config.schema.json
│   │           └── ...
│   │
│   ├── workspaces/              # 项目工作区与分支工作区
│   │   ├── projects/{projectId}/
│   │   │   └── live-{timestamp}-{random}/
│   │   │       ├── .workspace.json  # scope=live
│   │   │       ├── workspace-tree.json
│   │   │       ├── project.config.schema.json
│   │   │       └── demos/
│   │   └── {actorId}/{projectId}/
│   │       └── cli_edit-.../        # scope=branch
│   │
│   ├── sessions/                # 对话元数据（不含文件）
│   │   └── {userId}/{projectId}/
│   │       └── session-{timestamp}-{random}/
│   │           └── .session.json    # 含 workspaceId 关联字段
│   │
│   └── snapshots/               # 版本快照
│       └── proj_abc123/
│           ├── v1/
│           ├── v2/
│           ├── ...
│           └── v50/             # 最多保留 50 个版本
│
├── data/published/               # 使用端可读取的发布产物
│   ├── projects-index.json       # 已发布项目索引
│   └── proj_abc123/
│       ├── project.json
│       ├── app.graph.json
│       ├── preview-runtime/
│       └── demos/
│
├── src/                         # 源代码
├── package.json
└── ...
```

### 2.2 项目目录

```
packages/author-site/data/projects/
└── proj_{id}/
    ├── project.json             # 项目元数据
    └── workspace/               # 项目基准工作区
        ├── index.html
        ├── styles.css
        └── ...
```

### 2.3 工作空间目录

```
packages/author-site/data/workspaces/
├── projects/{projectId}/
│   └── live-{timestamp}-{random}/   # 项目工作区
│       ├── .workspace.json          # scope=live
│       ├── workspace-tree.json
│       ├── project.config.schema.json
│       ├── demos/
│       └── .opencode/               # AI 代理配置（自动注入）
└── {actorId}/{projectId}/
    └── cli_edit-.../                # 分支工作区
        └── .workspace.json          # scope=branch
```

**说明**：
- 项目工作区是默认编辑态，与 Session 解耦
- 多个 Session 可通过 `workspaceId` 共享同一个项目工作区
- 分支工作区只用于显式事务，不进入默认协同房间
- `.opencode` 目录在创建 Workspace 时自动注入
- Session 归档、放弃或过期不会清理项目工作区
- 项目删除成功后，需要同步移除对应的发布产物目录并重建发布索引，避免使用端继续展示已删除项目

### 2.4 会话目录

```
packages/author-site/data/sessions/
└── {userId}/{projectId}/
    └── session-{timestamp}-{random}/
        └── .session.json            # 会话元数据（含 workspaceId）
```

**说明**：
- Session 目录仅存储 `.session.json`，不包含代码文件
- 通过 `workspaceId` 字段关联到 Workspace
- 切换对话仅切换 Session，不影响 Workspace 文件

### 2.5 快照目录

```
packages/author-site/data/snapshots/
└── proj_{projectId}/
    ├── v1/                      # 版本 1 快照
    ├── v2/                      # 版本 2 快照
    └── ...
```

## 三、数据结构

### 3.1 Project（项目）

```typescript
interface Project {
  id: string                    // 项目唯一标识
  name: string                  // 项目名称
  description?: string          // 项目描述
  workspacePath: string         // 项目基准工作区绝对路径
  activeWorkspaceId?: string    // 项目工作区 ID
  activeWorkspaceUpdatedAt?: number
  canonicalSyncedWorkspaceId?: string
  canonicalSyncedAt?: number
  versions: VersionInfo[]       // 版本历史（最多 50 个）
  createdAt: number             // 创建时间戳
  updatedAt: number             // 最后更新时间戳
}
```

### 3.2 VersionInfo（版本信息）

```typescript
interface VersionInfo {
  versionId: string             // 版本号（"v1", "v2", ...）
  savedAt: number               // 保存时间戳
  savedBy: string               // 保存者用户名
  sessionId: string             // 关联的编辑会话 ID
  snapshotPath: string          // 快照备份绝对路径
  fileCount: number             // 文件数量
  note?: string                 // 用户备注（可选）
}
```

### 3.3 EditSession（编辑会话）

```typescript
interface EditSession {
  sessionId: string             // 会话唯一标识
  projectId: string             // 关联的项目 ID
  workspaceId: string           // 关联的工作空间 ID
  username: string              // 当前编辑者用户名
  basedOnVersion: string        // 基于的版本号
  status: 'editing' | 'archived' | 'saved' | 'discarded'
  opencodeSessionId: string | null  // OpenCode Agent 会话 ID
  createdAt: number             // 创建时间戳
  expiresAt: number             // 过期时间戳
}
```

**说明**：
- `workspaceId` 关联到 Workspace，多个 Session 可共享同一 Workspace
- `archived` 状态表示用户切换到其他对话，该对话被归档
- Session 不直接持有文件，文件存储在关联的 Workspace 中

## 四、核心流程

### 4.1 打开项目编辑

```
用户打开项目
      │
      ▼
读取项目信息，获取最新版本号
      │
      ▼
创建或复用项目工作区
路径：/workspaces/projects/{projectId}/{workspaceId}
      │
      ▼
若项目工作区不存在，从项目基准工作区初始化
      │
      ▼
注入 .opencode AI 代理配置
      │
      ▼
创建编辑会话记录（关联 workspaceId）
      │
      ▼
返回：{ sessionId, workspaceId, workspacePath, tempWorkspace, basedOnVersion }
说明：tempWorkspace 仅为旧调用方兼容别名。
```

### 4.2 保存变更

```
用户点击保存
      │
      ▼
读取会话信息和项目信息
      │
      ▼
生成新版本号 v{N+1}
      │
      ▼
同步项目工作区到项目基准工作区，并创建历史快照
从：{project.workspacePath}
到：/snapshots/{projectId}/{versionId}
      │
      ▼
记录版本信息到 project.versions
      │
      ▼
清理旧版本（保留最近 50 个）
      │
      ▼
返回：{ success: true, version: "v{N}" }
```

### 4.3 恢复版本

```
用户选择恢复版本
      │
      ▼
读取项目信息和指定版本信息
      │
      ▼
生成新版本号 v{N+1}
      │
      ▼
备份当前项目工作区
（创建新版本，备注："从 {versionId} 恢复"）
      │
      ▼
从快照恢复项目基准工作区和项目工作区
从：/snapshots/{projectId}/{versionId}
到：{project.workspacePath}
      │
      ▼
更新项目版本历史
      │
      ▼
返回：{ newVersionId: "v{N+1}" }
```

保存时正式 workspace 的目录替换需要兼容 Windows 文件句柄释放延迟。实现上应对“删除旧 workspace 后重命名临时 workspace”做短间隔重试，并在重命名持续失败时使用复制兜底；不能依赖单次删除和重命名在所有平台都立即成功。

### 4.4 核心流程回归验证

项目新建、编辑会话、工作空间写入、手动保存和再次读取正式项目，是创作端项目管理的主链路。仓库通过 `test/创作端E2E回归测试/author-core-flow-regression.spec.ts` 维护该链路的 Playwright 回归用例。

该用例从真实登录和首页新建入口开始，随后验证编辑会话能写入页面文件、保存动作能生成版本，并通过重新创建编辑会话读取项目工作区内容，确认保存结果已经从项目工作区同步到项目基准工作区。测试结束时会删除本次创建的项目，避免回归数据长期残留。

专用运行命令为 `pnpm test:e2e:core-flow`。运行前需要启动 author-site，并准备可登录的测试账号；完整 E2E 回归仍使用 `pnpm test:e2e`。

## 五、协作冲突处理策略

### 5.1 实时草稿协作

多人协作不再采用“各自用户私有副本、后保存覆盖”的策略。多个协作者进入同一个项目工作区后，页面代码、页面配置、项目配置、页面树、画布布局和知识文档会按资源进入独立协同房间；同一资源的编辑先合并到项目工作区，再按实时保存策略落盘。

实时草稿协作的业务边界见 [实时协作保存需求](../实时协作保存_需求文档.md)，协同房间、presence、flush 和路径安全设计见 [11_实时保存与协同编辑](./11_实时保存与协同编辑.md)。

### 5.2 安全保障

| 保障措施 | 说明 |
|:---------|:-----|
| 实时草稿 | 高频编辑先保存到 Workspace 草稿，减少手动保存丢失风险 |
| 手动版本 | 用户需要回退节点时再创建版本快照 |
| 显式 flush | 创建版本或发布前先写回协同房间内存草稿 |
| 可回退 | 已创建的版本可通过版本历史恢复 |
