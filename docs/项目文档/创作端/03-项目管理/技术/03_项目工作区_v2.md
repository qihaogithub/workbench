# 会话与版本管理 - 项目工作区

> 版本：v4.4
> 更新日期：2026-07-11
> 变更说明：增加 Authority 系统不变量参考

## 一、统一术语

当前项目管理链路只使用以下四个主概念：

| 中文名称       | 实现名                                           | 角色                             | 说明                                                                                                                         |
| :------------- | :----------------------------------------------- | :------------------------------- | :--------------------------------------------------------------------------------------------------------------------------- |
| 项目工作区     | active Workspace / `scope=live`                  | 当前可编辑态                     | 默认编辑页、实时协同、预览和自动保存共同使用的项目级共享当前态                                                               |
| 项目基准工作区 | canonical Workspace / `projects/{id}/workspace/` | 项目当前态的基准文件区与物化缓存 | 供发布、导出、历史快照、模板保存、CLI 和脚手架转换等既有项目读写流程读取，由项目工作区同步推进，也可从内容图 commit 物化重建 |
| 分支工作区     | branch Workspace / `scope=branch`                | 显式隔离事务                     | 仅用于 CLI、AI 大改或实验性修改，不进入默认多人协作房间                                                                      |
| 历史快照       | Snapshot                                         | 不可变历史                       | 版本、发布、恢复等关键节点的只读记录                                                                                         |

旧称处理规则：

- “草稿工作区”只作为历史文档或旧文件名保留；当前文档语义统一为“项目工作区”。
- “正式工作空间”容易误导为浏览器直接编辑的唯一源头，当前统一称为“项目基准工作区”。
- “临时工作空间”只用于旧 API 字段、历史临时目录或明确的分支事务，不描述默认编辑态。

## 二、Authority 系统不变量

项目工作区的所有写入都经过 Workspace Mutation Authority，以下不变量是系统核心约束（详见计划文档 `docs/plans/已完成/创作端Workspace写入一致性-单写者事务改造方案.md`）：

| 编号   | 不变量                             | 说明                                                                       |
| :----- | :--------------------------------- | :------------------------------------------------------------------------- |
| INV-1  | 活动 Workspace 单写者              | `scope=live` 激活后只有 Authority 可改受管资源                             |
| INV-2  | 资源级冲突检测                     | 每个变更携带 `expectedHash`，hash 不匹配返回 `WORKSPACE_RESOURCE_CONFLICT` |
| INV-3  | 提交后才发事件                     | 只有 durable receipt 写入后才广播 `workspace_mutation_committed`           |
| INV-4  | 旧投影无权回写                     | 协同房间 flush 时若目标资源已被其他 mutation 推进，只能进入冲突状态        |
| INV-5  | 草稿预览即时，可靠消费有版本       | 编辑态预览消费本地/Yjs 草稿；截图、发布、导出只消费 committed snapshot     |
| INV-6  | 项目版本与 Workspace revision 分离 | `baseVersion` 和 `revision` 不得互相替代                                   |
| INV-7  | 关键动作绑定确切 revision          | 命名版本、发布、导出、模板、恢复必须记录 `workspaceId` + `revision`        |
| INV-8  | 成功状态由系统生成                 | UI 状态由 receipt 和 revision ack 生成，不从模型自然语言推断               |
| INV-9  | 自动保存不等待 canonical           | “已自动保存”只要求 mutation durable commit；canonical 异步追赶             |
| INV-10 | 实时热路径不进入 Authority         | 本地输入、Yjs 广播、草稿预览不等待 mutation journal 或 canonical           |

## 三、工作空间分层

Session 不再拥有文件副本。Session 只记录用户、项目、AI 对话和绑定的 `workspaceId`；文件状态由 Workspace 承载。

## 四、目录结构

### 3.1 项目工作区

```text
workspaces/
└── projects/
    └── {projectId}/
        └── live-{timestamp}-{random}/
            ├── .workspace.json
            ├── workspace-tree.json
            ├── project.config.schema.json
            ├── demos/
            ├── assets/
            └── knowledge/
```

`.workspace.json` 标识该工作区为项目级共享当前态：

| 字段                   | 说明                       |
| :--------------------- | :------------------------- |
| `workspaceId`          | Workspace 唯一标识         |
| `projectId` / `demoId` | 归属项目                   |
| `scope`                | `live` 表示项目工作区      |
| `status`               | `active` 表示当前可编辑    |
| `baseVersion`          | 创建当前态时基于的项目版本 |
| `updatedAt`            | 最近内容更新时间           |

### 3.2 项目基准工作区

```text
projects/
└── {projectId}/
    ├── project.json
    └── workspace/
```

项目基准工作区保存项目目录下的官方基准文件。关键动作会先同步项目工作区，再从项目基准工作区生成发布产物或历史快照。

项目目录下的 `content/` 是资源级历史的长期记录。`workspace/` 作为预览、发布和 CLI 的运行缓存，不再承载页面或知识文档历史事实。内容图物化会把指定 commit 的资源指针写回 `workspace/`，并记录物化 manifest；物化失败时保留 commit，状态标记为 failed，后续可重试。

### 3.3 分支工作区

```text
workspaces/
└── {actorId}/
    └── {projectId}/
        └── cli_edit-.../
```

分支工作区用于 CLI、AI 大改或实验性事务。它不进入默认实时协同房间，提交前必须校验基准版本，避免覆盖别人已经同步的项目工作区。

## 五、核心流程

### 4.1 打开项目

用户打开项目时，系统读取项目元数据中的 `activeWorkspaceId`：

- 如果项目工作区存在且 `baseVersion` 等于项目最新版本，当前用户的新 Session 直接绑定它。
- 如果不存在，或现有项目工作区基于旧版本，系统从项目基准工作区创建一个 `scope=live` 的项目工作区，并写回项目元数据。
- 如果存在旧的用户私有 Workspace，首次迁移时可选择最近仍有效的旧 Workspace 作为项目工作区初始内容。

### 4.2 实时编辑

实时协同房间以项目、Workspace、资源路径为边界。多个用户只要进入同一项目，就会绑定同一个项目工作区，因此对同一资源的修改会进入同一个协同上下文。

实时保存先写入项目工作区。自动保存状态表示当前态已经落盘；是否形成历史版本由 [版本管理](./04_版本管理.md) 决定。

### 4.3 关键动作同步

命名版本、自动检查点、发布、导出和恢复前必须执行统一同步：

1. flush 项目工作区下的协同房间。
2. 将项目工作区原子同步到项目基准工作区。
3. 再生成快照、发布产物或恢复记录。

如果协同服务不可达或同步失败，关键动作必须失败，不能消费落后的项目基准工作区。

同步前必须确认传入的 `workspaceId` 仍是项目元数据中的当前项目工作区，并且该工作区 `baseVersion` 等于项目最新版本。过期工作区只能提示用户刷新或重新打开项目，不能覆盖项目基准工作区。

项目元数据中的 `activeWorkspaceId`、`activeWorkspaceUpdatedAt`、`canonicalSyncedWorkspaceId`、`canonicalSyncedRevision`、`canonicalSyncedRootHash` 和 `canonicalSyncedAt` 是创作端共享当前态的跨模块契约字段。`canonicalSyncedRevision` / `canonicalSyncedRootHash` 来自 live Workspace 的 Authority snapshot，表示项目基准工作区已经物化到哪个 committed revision 和根哈希；不能只用时间戳判断关键动作是否消费了最新 committed 内容。业务层不能直接调用底层 canonical 同步实现，必须通过 `canonical-materializer` 边界推进项目基准工作区；关键动作先用 `ensureCanonicalRevision` 校验目标 revision，再由 materializer 物化，物化完成后还要再次确认 live Workspace 仍停留在同一 revision/rootHash。若物化期间出现新的 committed revision，本次关键动作必须返回 `WORKSPACE_STALE`，并按本次写入的 workspaceId/revision/rootHash 条件清除刚写入的 stale canonical proof，不能继续用旧 canonical 结果创建发布、导出、模板或历史记录，也不能让后续无 Session 入口误用这组旧证明。`project-core` 在创建页面资源版本、写入内容图提交、发布状态或其他 ProjectAdminService 结果时，必须读取并原样保留这些字段；只有分支事务提交成功、明确以隔离工作区替换项目基准工作区时，才允许清空旧项目工作区指针。

三个版本轴使用独立语义类型，更新规则不能混用：

| 版本轴                    | 推进时机                                                                               | 禁止行为                                                                             |
| :------------------------ | :------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------- |
| 项目 `baseVersion`        | 创建或恢复项目历史版本，并确认当前 live Workspace 仍是该版本的有效当前态时             | 不能因单次 Workspace mutation 自动推进                                               |
| Workspace `revision`      | Authority bootstrap 建立 revision 1；此后每次 mutation 或显式 reconcile 成功后单调递增 | 不能由 canonical 同步结果回写或倒退                                                  |
| `canonicalSyncedRevision` | canonical materializer 成功消费某个 Workspace revision/rootHash，且后置校验仍一致时    | 不能代表 Authority 当前 revision，不能只凭时间戳生成，也不能跨 active Workspace 继承 |

无 Session 直接发布项目时，如果项目仍有 active Workspace，发布入口必须要求 `canonicalSyncedWorkspaceId` 指向当前 active Workspace，且 `canonicalSyncedRevision` / `canonicalSyncedRootHash` 已存在；缺少 revision/rootHash 的旧时间戳同步状态不能作为发布输入。创建项目版本时，只有具备这组 canonical 元数据的 active Workspace 才允许随版本创建推进自身 `baseVersion`，避免旧同步证明把未绑定 revision 的内容标记为已消费。项目包导出、CLI `project pull` 和保存为模板也消费项目基准工作区：当项目仍有 active Workspace 时，这些入口必须要求 canonical 指针已经绑定当前 active Workspace。导出包会把 `workspaceId`、`workspaceRevision` 和 `workspaceRootHash` 写入本地项目包 manifest 与同步状态；模板快照会记录 `sourceWorkspaceId`、`sourceWorkspaceRevision` 和 `sourceWorkspaceRootHash`。

### 4.4 Session 生命周期

归档、放弃或过期 Session 只影响对话上下文，不删除 `scope=live` 的项目工作区。需要回退文件内容时，通过历史恢复重置项目工作区。

## 六、与分支事务的关系

`project-core` 的编辑事务继续使用隔离 Workspace，但该能力被定义为分支事务：

- 适合 CLI、批量维护、AI 沙盒和实验性改动。
- 不作为普通编辑页默认入口。
- 提交时比较事务 `baseVersion` 和项目当前版本。
- 提交成功后，项目基准工作区和新版本快照成为最新权威状态，项目元数据中的旧项目工作区指针必须失效。
- 存在冲突时拒绝整体覆盖，要求重新基于最新项目工作区开启事务。

分支事务提交不会删除旧 `scope=live` 目录，但旧目录不再作为 active Workspace 复用。后续保存、自动修复、发布前同步或恢复动作如果携带旧 `workspaceId`，必须返回工作区过期错误，避免旧当前态反向覆盖新提交。
