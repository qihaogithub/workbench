# 实时协作保存 - 需求文档

> 版本：v1.0
> 更新日期：2026-06-27
> 变更说明：新增多人协作实时保存的业务语义、用户体验和保存边界

---

## 一、模块定位

实时协作保存用于保障多人同时编辑同一个项目工作空间时，草稿内容能持续保存、实时同步，并在需要回退节点时再由用户手动创建版本。

本能力补充 [会话与版本管理](./会话管理_需求文档.md) 中的多人协作部分：协作中的即时输入先成为 Workspace 草稿，不直接成为项目正式版本；项目版本仍由用户明确创建。

---

## 二、功能需求

### 2.1 实时草稿保存

**用户视角**：

- 用户编辑页面代码、页面配置、项目配置、页面树、画布布局或知识文档时，修改应自动保存到当前工作空间草稿。
- 用户不需要频繁点击保存来避免内容丢失。
- 页面刷新或重新进入编辑页后，应能恢复最近一次已同步的草稿内容。

**交互效果**：

- 编辑界面应展示当前草稿的保存状态，例如连接中、同步中、已保存、离线或异常。
- 自动保存失败时，应明确提示用户当前草稿可能没有同步完成。
- 用户离线或连接异常时，可以继续保留本地编辑状态，并在恢复连接后尝试同步。

### 2.2 多人实时协作

**用户视角**：

- 多个协作者打开同一个项目工作空间时，可以看到彼此对同一资源的编辑结果。
- 系统应展示当前在线协作者，帮助用户判断是否有人正在共同编辑。
- 协作者切换页面或编辑不同资源时，系统应尽量展示其活跃位置。

**交互效果**：

- 同一资源的远端修改应同步到当前编辑器或画布。
- 不同资源之间应互不干扰，例如页面 A 的代码编辑不应覆盖页面 B。
- 协作者离开后，其在线状态和光标信息应自动消失。

### 2.3 手动创建版本

**用户视角**：

- 自动保存只表示草稿已保存，不表示项目已经生成可回退版本。
- 用户需要阶段性留档时，可在版本历史区手动创建版本。
- 发布项目前，如果存在尚未创建版本的草稿，系统应先确保草稿已同步，并按发布规则生成必要版本。

**交互效果**：

- “创建版本”应表达为生成回退点，而不是防止内容丢失的唯一入口。
- 创建版本前应先同步当前协作草稿，避免版本快照落后于编辑器内容。
- 创建版本成功后，版本历史应展示新的版本记录。

### 2.4 退出与异常提醒

**用户视角**：

- 用户离开编辑页时，若草稿仍在同步中或同步失败，应收到风险提示。
- 若草稿已经实时保存但尚未创建版本，系统应提示用户是否需要创建可回退版本。
- 会话过期或权限失效时，应阻止继续写入协作草稿。

**交互效果**：

- 异常提示应区分“草稿未同步”和“草稿未创建版本”两类风险。
- 用户选择不创建版本离开时，草稿仍保留在工作空间，不应误导为内容丢失。
- 用户重新进入时，应继续基于当前工作空间草稿编辑。

## 三、业务规则与约束

| 规则 | 说明 |
|:-----|:-----|
| 草稿优先 | 实时编辑先保存为 Workspace 草稿 |
| 版本手动 | 自动保存不自动生成项目版本 |
| 同资源协同 | 同一项目、同一工作空间、同一资源进入同一个协作上下文 |
| 资源隔离 | 页面代码、页面配置、项目配置、页面树、画布布局和知识文档分别协作 |
| 会话校验 | 只有有效会话可以加入协作并写入草稿 |
| 权限收敛 | 协作保存只能写入允许的项目资源，不能写入任意工作空间文件 |
| 发布前同步 | 发布前必须确保当前工作空间协作草稿已同步 |
| 版本兜底 | 历史回退能力依赖用户创建的版本快照 |
