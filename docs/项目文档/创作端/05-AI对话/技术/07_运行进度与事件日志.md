# 运行进度与事件日志

> 更新日期：2026-07-10

本文描述 AI 对话在一次运行中的事件可视化和后端日志保存机制。业务侧需求见 [AI 对话需求文档](../AI对话_需求文档.md)。

## 设计目标

AI 运行过程可能包含模型输出、工具调用、子 Agent 委派和文件变更。第一版不推断百分比进度，也不显示等待秒数，只展示真实事件带来的反馈。

对普通创作用户，聊天区只需要知道 AI 正在处理，避免在长任务中反复看到“模型响应”“子 Agent 运行中”等容易误解的内部状态。具体产物仍由右侧预览区、页面列表和文件变更反馈承载。

对排查问题的开发者，后端需要保存本轮运行的事件日志，便于定位空回复、工具结果丢失、子 Agent 无结果和文件变更遗漏。

## 事件流转

agent-service 的 WebSocket 路由在收到用户消息后，为本轮消息创建运行日志，并把后端事件统一交给事件路由器。

事件路由器负责两件事：

- 继续向前端透传必要的实时事件，保证既有流式响应、工具展示和权限确认可用。
- 同步写入后端 JSONL 日志，保留模型、工具、子 Agent、文件和系统状态的关键事件。

WebSocket 消息不再把“长时间没有新 token 或工具事件”直接判定为 Agent 死亡。一次运行开始后，agent-service 会在等待期间持续发送 `processing` 状态心跳，表示服务端仍持有当前 run，前端据此维持处理中状态。只有调用方显式传入 timeout 时，后端才把它作为本轮运行的硬取消上限；默认运行依赖用户取消、连接生命周期和底层模型/工具真实返回，而不是按附件大小或上下文长度猜测死活。

同一个 Agent 会话只允许一个 Pi Agent harness 运行。WebSocket 和 HTTP 发送入口都需要在启动新 run 前检查后端 busy 状态；若上一轮仍在运行，不再调用 `harness.prompt`，而是记录一条 `AGENT_BUSY` 失败日志并向前端返回可重试错误。该错误不表示模型切换失败，也不表示网络断连；它表示当前会话仍被上一轮占用，前端应提示等待完成或先取消。

Pi Agent 事件映射需要保留工具入参、工具结果、details、错误和耗时。底层工具执行以 `tool_execution_start` 和 `tool_execution_end` 表示，后端需要把它们映射为应用层 `tool_call` 和 `tool_call_update`。`delegateTask` 这类子 Agent 工具不应只传状态，否则前端详情弹窗和后端日志都无法还原任务入参、输出摘要、文件变更与失败原因。

## 前端展示

聊天区在一次请求刚开始、尚未出现思考过程、工具调用或文本输出时，使用靠左的紧凑 5x5 点阵循环动效作为空窗期兜底提示，只表达“AI 正在处理”。当界面开始出现思考、工具调用或正文输出后，点阵动效收进 assistant 消息底部的操作栏，与复制等操作按钮处在同一控制区；流式输出期间该操作栏保持可见，避免运行反馈被正文增长挤出用户视野。界面不显示额外文字标签，但保留无障碍状态语义；它不是任务列表，也不承担进度估算。

历史执行过程仍可在 assistant 消息内折叠查看，用于展示思考、普通工具调用和子 Agent 摘要。子 Agent 条目可以打开详情弹窗，查看委派任务、补充上下文、输出摘要、文件变更、失败原因和原始事件详情。

前端不展示会话日志入口。日志是开发排查能力，不是普通用户界面的一部分。

## 后端日志

运行日志默认保存在数据目录下：

`data/agent-run-logs/<sessionId>/<messageId>.jsonl`

每条日志是一行 JSON，包含时间、级别、来源、事件类型、标题、摘要、关联工具调用 ID 和脱敏后的 payload。日志写入失败不应中断用户对话，只记录服务端 warning。

日志 payload 会对 key、token、authorization、password、secret 等敏感字段做脱敏；长文本会截断，避免把完整大段内容写入排查日志。

同时，agent-service 会把本轮运行的关键边界事件同步写入创作端结构化诊断事件流，供 [诊断与日志模块](../../11-诊断与日志/) 的 SQLite/JSONL 查询入口串联。同步事件只保存运行 ID、工具名、状态、耗时、文件路径摘要、数量和错误摘要；完整 prompt、模型正文、工具入参和工具结果仍只存在受控的 AI run log 中。

每轮结束时必须记录 finish 摘要，包括：

- 最终回复内容长度。
- 累计流式输出长度。
- 工具结果数量。
- 子 Agent 结果数量。
- 文件操作数量。
- 文件结果数量。
- 后端返回的错误或空回复调试信息。

当模型成功返回但最终回复为空时，finish 日志使用 warning 级别，并保留后端提取到的响应结构摘要，便于判断是模型真实无输出、响应结构变化，还是文本提取逻辑遗漏。

如果 Pi Agent 返回的 assistant message 带有错误摘要，例如供应商错误或模型调用失败，后端必须把本轮结果转成失败事件，而不是作为空内容的成功响应继续传给前端。这样用户看到的是明确错误，日志中的 finish 也能保留错误摘要和响应结构。

Pi Agent 发送链路捕获到底层异常时，服务端错误日志必须记录标准化后的错误字段，而不是直接依赖运行时对象的默认 JSON 序列化。标准化信息至少保留错误名称、消息、状态码、堆栈和最近一次 assistant response 结构摘要；对于 SDK 抛出的非标准对象，也要尽量提取 `message`、`error.message`、`status` 和 `statusCode`。这保证运行日志里不会只出现空对象，排查模型供应商、网络或响应结构问题时能直接看到可行动的失败原因。

如果运行失败时前端已经收到部分 assistant 正文、思考过程或工具调用，前端应把错误说明合并进当前 assistant 消息并固化该消息；只有在本轮完全没有可见内容时才单独追加错误气泡。这样导出记录和聊天区不会同时出现一条超时错误消息和另一条同轮半截 assistant 消息。

## 兼容性

`tool_call` 和 `tool_call_update` 在保留旧字段的基础上扩展参数、结果、详情和耗时字段。旧消费者仍可只读取原有状态字段；新消费者可以按需读取更完整的事件 payload。
