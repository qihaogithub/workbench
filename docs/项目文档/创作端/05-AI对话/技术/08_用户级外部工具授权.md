## 一、定位

用户级外部工具授权用于让创作端 AI 在当前登录用户授权范围内访问 Figma 与钉钉。平台不提供部署者全局账号兜底，也不要求用户在对话中粘贴 token、refresh token 或 dws 认证包。

该能力只影响当前用户自己的 AI 会话。外部系统能读写的内容由该用户在 Figma 或钉钉中的实际权限决定，创作端不会扩大外部系统权限。

## 二、授权入口

用户可以从设置弹窗进入“外部账号授权”，查看 Figma 与钉钉两类 provider 的连接状态、授权账号、过期或需重登提示，并进行连接、重新授权、断开和刷新状态操作。

当 AI 在对话中调用 Figma 或钉钉工具且发现当前用户未授权或授权过期时，工具会返回结构化的 `external_auth_required` 结果。聊天消息列表会直接渲染授权卡片，用户可在卡片中连接 Figma 或钉钉、打开浏览器授权页、查看钉钉 user code，并在完成授权后刷新状态。授权完成后，author-site 会把最新授权配置同步给该用户的活跃 agent session。

授权卡片会携带当前 agent sessionId 调用授权接口。这样授权完成后的状态不只依赖“扫描活跃会话”，还会直接热更新当前正在对话的 agent session。流式消息中的授权卡片也拥有稳定的 assistant 消息 id；当卡片检测到授权已连接时，会停止当前未授权流程并自动重发上一条用户请求。

如果部署环境返回外部授权不可用，聊天内授权卡片会显示平台配置原因，主连接按钮切换为不可用状态，只保留“重新检查配置”用于在平台补齐配置后刷新状态。

状态读取 API 只返回脱敏后的安全结构。浏览器端永远不会拿到 Figma access token、refresh token 或钉钉 dws 配置目录中的认证内容。

## 三、服务端存储

用户级授权状态保存在 `users.db` 的 `user_external_auth_configs` 表中，主键是 `user_id` 与 `provider`。敏感凭据复用用户模型配置的 AES-GCM 加密策略，密钥来源仍按 `MODEL_CONFIG_ENCRYPTION_KEY`、`JWT_SECRET`、开发兜底值的优先级选择。

Figma 保存加密后的 OAuth 凭据和过期时间。钉钉保存加密后的用户级 dws 配置目录路径，真实登录态由 dws 写入该用户隔离目录。

## 四、授权流程

Figma 使用浏览器 OAuth 授权。部署环境需要在 author-site 中配置 `FIGMA_OAUTH_CLIENT_ID`、`FIGMA_OAUTH_CLIENT_SECRET` 和与 Figma OAuth app 完全一致的 `FIGMA_OAUTH_REDIRECT_URI`；开发环境由 `packages/author-site/.env.local` 提供这些变量。默认授权范围为 `file_content:read`，Figma app 的 OAuth scopes 页必须选择同一 scope；Embed API 的 allowed origins 只影响 Figma Embed Kit，不决定 OAuth 授权是否可用。callback 完成后的页面回跳优先使用 `FIGMA_OAUTH_POST_AUTH_ORIGIN`，未配置时使用 `FIGMA_OAUTH_REDIRECT_URI` 的 origin；当 OAuth redirect 需要 Cloudflare Tunnel 等公网入口，但用户实际在本地 `http://localhost:3200` 使用创作端时，应把 `FIGMA_OAUTH_POST_AUTH_ORIGIN` 设置为本地创作端 origin。未配置 Figma OAuth 客户端时，状态会标记为不可用，AI 工具返回“未启用或需平台准入”，不回退到任何全局 Figma token。创建或恢复会话时，如果 Figma access token 已进入临期窗口，author-site 会用 refresh token 换取新的 access token，并将刷新后的凭据重新加密保存；刷新失败则标记为需重新授权，避免把过期 token 注入 agent-service。用户浏览器里已经登录 Figma 只代表授权页面可更顺畅完成登录，不等于创作端已经获得 Figma OAuth token。

钉钉使用 agent-service 容器内的 dws device flow。author-site 请求 agent-service 启动授权后，将授权链接和 user code 返回给浏览器。用户在浏览器完成钉钉授权后，状态刷新会查询 agent-service 中该用户隔离目录的 dws 登录态，成功后更新为已连接。

dws 的 `success:true` 只表示 `auth status` 命令本身执行成功，不表示账号已登录。系统优先使用 dws 明确返回的 `authenticated:true`、`loggedIn:true` 或等价已登录状态判断 connected；如果 `auth status` 返回未登录，但同一用户隔离目录下的只读 `doc list` 探测可以成功读取文档列表，也视为 connected。这个兜底用于兼容当前 dws CLI 已能访问文档但 `auth status` 仍返回 `authenticated:false` 的情况。若 status 与只读探测都失败，author-site 会把旧 connected 记录覆盖为需重新授权并清空当前 session 注入的钉钉权限。用户浏览器里已经登录钉钉不等于 dws CLI 已登录，必须完成 dws OAuth 并写入对应用户隔离 `DWS_CONFIG_DIR`。

断开授权时，author-site 删除对应数据库记录；钉钉还会请求 agent-service 删除该用户的 dws 配置目录。

## 五、会话注入

创建或恢复 AI session 时，author-site 会读取当前用户可用的外部授权配置，并通过内部接口推送给 agent-service 的同名 session。新建会话和复用活跃会话都会执行这一步，因此同一用户完成一次授权后，后续进入新的编辑对话或回到已有对话时不需要重新授权。agent-service 只在 session 级内存表中保存该配置；如果 Agent 已经存在，会立即热更新运行时配置。授权卡片触发的 start、status 和 Figma callback 也会带上当前 agent sessionId，确保用户完成授权后当前对话马上获得最新配置。

钉钉 dws 登录态保存在 `DATA_DIR/dws-auth/<user-hash>/` 这类用户隔离目录中；Docker 部署下 author-site 与 agent-service 共用 `/app/data` 持久卷，因此容器重启后仍能复用该目录。Figma OAuth 凭据保存在用户授权表中，access token 临期时由 author-site 用 refresh token 刷新后再注入 session。

这条链路保证同一用户后续会话可以复用授权，同时避免一个用户的外部授权进入其他用户的 session。

## 六、工具边界

Pi Agent 工具层新增两个受控工具：

| 工具 | 能力范围 | 关键限制 |
| :--- | :--- | :--- |
| `figmaMcp` | Figma MCP `listTools` 与 `callTool` | 仅使用当前用户 Figma token；未授权或过期时返回聊天授权卡片所需的结构化结果；MCP 准入失败时返回明确错误 |
| `dingtalk` | dws `doc`、`sheet`、`wiki` | 只允许三个产品；未授权时返回聊天授权卡片所需的结构化结果；强制 `--format json`；使用 `execFile` 参数数组和用户级 `DWS_CONFIG_DIR` |

钉钉群聊、通讯录、审批、待办等产品不在本期开放范围内。通用 bash 工具不允许作为 Figma 或 dws 的绕行入口。

## 七、写操作确认

外部写操作继续走创作端确认卡。Figma 的 create、update、delete、upload、import、write 等工具名，以及 dws 的 create、update、delete、move、rename、copy、permission、member、append、export 等命令都会触发 `permission_request`。

用户取消确认时，工具不会访问外部系统。读操作在已有授权范围内可以直接执行。
