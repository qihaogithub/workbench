# AI 页面删除工具

> 更新日期：2026-06-21

## 设计目标

AI 删除页面必须基于当前工作区的真实页面 ID，不能根据页面名称或显示顺序猜测。批量删除采用“先预览、再确认、最后执行”的事务流程；删除失败时必须保持失败状态，不能向用户汇报成功。

## 工具边界

- `listPages` 是删除前的事实来源，返回页面 `id`、`name`、`indexPath` 和 `schemaPath`。
- `deletePage` 只删除一个真实存在的页面 ID。
- `previewDeletePages` 根据页面名称片段、文件夹 ID 或显式页面 ID 生成删除计划，不修改文件。
- `executeDeletePagePlan` 只接受预览生成的 `planId`，执行前弹出聊天内确认卡，确认后按同一份计划删除页面。
- `deletePages` 作为兼容工具保留，但批量、全部、按条件删除的长期入口是 `previewDeletePages` 和 `executeDeletePagePlan`。
- 页面删除工具直接操作当前 Agent 会话的 `workingDir`，只允许删除 `demos/{pageId}/` 和更新 `workspace-tree.json`。
- 页面删除不调用创作端登录态 API，避免 agent-service 服务端请求缺少浏览器 Cookie 导致假成功。

批量删除调用形态：

```ts
previewDeletePages({
  mode: "nameIncludes",
  query: "副本",
});

executeDeletePagePlan({
  planId: "delete_plan_xxx",
});
```

## 失败规则

- 页面 ID 不存在时，工具返回失败；如果页面名称能匹配候选页面，只返回候选 ID，不自动删除。
- 页面名称对应多个候选页面时，工具返回歧义信息，由用户或下一轮指令明确 ID。
- 删除计划只在当前 Agent 会话内存中保存 5 分钟；过期后必须重新预览。
- 执行计划前会确认 `workspace-tree.json` 没有变化，避免用户或其他工具改动页面结构后继续按旧计划删除。
- 删除后会重新校验目录和 `workspace-tree.json` 记录；校验失败时返回失败。
- 删除请求会检查至少保留一个页面，禁止删除最后一个页面。

## 确认方式

删除计划执行时，后端通过权限通道发送确认请求，前端在聊天区域显示内联确认卡。确认卡展示删除数量、页面名称、页面 ID 和计划 ID。用户点击确认后，后端才会执行同一份计划；用户取消或请求超时不会删除任何页面。

## 能力握手

创作端发送消息前会向 Agent Service 查询当前真实工具能力。系统提示词中的删除规则会根据返回的工具清单生成：如果事务化工具齐全，就使用“预览计划再执行”的规则；如果后端版本较旧，则不会要求 AI 调用不存在的工具。

Agent Service 还会给工具集标记版本。已有会话中的 Agent 如果持有旧版本工具集，在下一轮非执行状态下会被重建，避免出现“提示词已经知道新工具，但底层 Agent 仍拿着旧工具”的错位。

当用户发起批量删除请求而后端缺少事务化删除工具时，前端会阻止发送，并提示重启 Agent Service 与刷新页面。这样系统不会再降级成逐个删除，也不会让 AI 汇报并不存在的执行能力。

## 前端同步

页面删除成功后，Agent 结果会包含：

- `demos/{pageId}/`：`deleted`
- `workspace-tree.json`：`modified`

编辑页收到这些变更后重新读取当前 Session 文件状态，刷新页面列表、文件夹列表、页面代码缓存和当前选中页面。如果当前页面已被删除，自动切换到剩余页面中的第一项。
