# 图片内容预描述

> 版本：v1.1
> 创建日期：2026-06-21
> 更新日期：2026-07-06

---

## 一、定位

图片内容预描述用于补齐文本模型的图片理解短板。它不改变用户选择的主模型，而是在主模型看不懂图片时，先请一个识图模型把图片翻译成文字，再把“图片文字描述 + 用户原始问题”交给主模型继续回答。

这套机制对应的业务能力见 [AI 对话需求文档](../AI对话_需求文档.md#219-图片内容预描述)。

## 二、整体协作方式

可以把这个流程理解成“翻译员模式”：

| 角色 | 职责 |
| :--- | :--- |
| 用户选择的主模型 | 继续负责最终回答和代码协作 |
| 识图模型 | 只负责把图片内容翻译成文字 |
| agent-service | 判断是否需要翻译、调用识图模型、拼接上下文 |

当用户发送图片时，后端先检查当前主模型是否支持图片输入：

- 支持图片输入：图片按原路径传给主模型。
- 不支持图片输入：进入图片预描述流程。
- 未配置识图模型：直接返回明确错误，避免静默失败。

## 三、数据流

图片预描述的数据流如下：

1. 前端通过 WebSocket 发送用户文字和图片附件。创作端聊天输入只接收图片文件；agent-service 在图片进入模型前会剥离 `data:image/...;base64,` 前缀，并校验最终数据是纯 base64 图片内容。
2. agent-service 创建或复用当前会话的 Pi Agent 后端。
3. 后端读取当前模型能力，判断主模型是否支持图片。对于管理后台配置的 OpenAI 兼容供应商，这一判断来自会话或全局 `BackendProvidersConfig.multimodalModels`。
4. 若主模型不支持图片，则调用 `ImageDescriber`。
5. `ImageDescriber` 根据图片内容生成或读取缓存描述。
6. 后端将消息改写为：

```text
【图片内容】图片描述文本

【用户问题】用户原始问题
```

7. 主模型只接收改写后的纯文本，并生成最终回答。

## 四、项目本地保存

图片附件有两种用途：一类是“理解图片内容”，一类是“把图片放进页面或配置”。前者走上面的预描述或多模态输入；后者必须通过 `saveImage` 工具保存成项目本地资源。

`saveImage` 不上传 OSS，也不写入全局 `/api/images/` 图床。它要求当前 Agent 会话绑定项目工作区，并把图片写入该工作区的 `assets/images/` 目录。工具返回工作区内路径和页面文件可直接引用的相对路径，页面代码通常使用从 `demos/{pageId}/` 指向根资源目录的相对路径。

保存后的图片会作为项目工作区文件参与预览、持久化和发布。发布阶段的资源本地化规则由 [发布资源本地化](../../03-项目管理/技术/12_发布资源本地化.md) 负责，不需要 AI 手动上传外部存储。

## 五、识图模型调用

当前实现优先面向 OpenAI 兼容接口。对于配置了 `baseURL` 的供应商，预描述服务直接调用该供应商的 `/chat/completions` 接口，并使用 `image_url` 形式传入图片。

这样做有两个原因：

- 图片预描述只需要一次轻量调用，不需要完整 Agent 工具链参与。
- 部分兼容接口对 AgentHarness 的图片格式适配不稳定，直连标准兼容接口更可靠。

对于没有 `baseURL` 的内置模型，后端仍保留通过 AgentHarness 调用的兜底路径。

## 六、配置与生效

当前可用配置由 agent-service 运行环境提供：

| 配置项 | 作用 |
| :--- | :--- |
| `IMAGE_DESCRIPTION_ENABLED` | 是否启用图片预描述 |
| `IMAGE_DESCRIPTION_MODEL` | 用于识图的模型 ID，推荐使用 `provider/model` 格式 |
| `IMAGE_DESCRIPTION_TIMEOUT` | 单次识图调用超时时间 |
| `IMAGE_DESCRIPTION_MAX_CACHE` | 内存缓存最大条目数 |
| `IMAGE_DESCRIPTION_PROMPT` | 自定义图片描述提示词 |

识图模型所需的供应商、baseURL、API Key 和模型列表由 agent-service 的供应商配置提供。平台全局供应商配置的管理方式见 [管理后台架构设计](../../08-管理后台/技术/01_架构设计.md)，用户个人模型配置的会话同步方式见 [用户模型配置](./04_用户模型配置.md)。

## 七、缓存与降级

图片描述会按图片 Base64 内容计算 SHA-256 哈希，并以内存 LRU 方式缓存。相同图片再次发送时，系统直接复用已有描述，减少识图模型调用次数和等待时间。

当识图模型调用失败、超时或返回空内容时，系统不会把原始图片继续交给文本模型，而是降级为图片元数据描述，例如图片名称和 MIME 类型。这样主模型至少能知道用户上传过图片，但不会误以为自己已经理解了图片细节。

## 八、边界与注意事项

- 图片预描述不会让文本模型真正具备视觉能力；它只是在上下文里补充一段图片文字说明。
- 保存图片和理解图片是两条不同链路：只有当 AI 需要在项目中引用该图片时，才调用 `saveImage` 写入项目本地资源。
- 识图模型的描述质量会直接影响主模型回答质量。
- 主模型可能仍会根据自己的行为习惯尝试查看项目文件；这属于主模型 Agent 行为，不是图片预描述本身。
- 当前能力已通过 WebSocket 端到端验证：文本主模型可收到由识图模型生成的图片描述，并基于该描述回答用户问题。
