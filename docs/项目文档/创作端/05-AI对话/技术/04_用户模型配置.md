## 一、定位

用户模型配置允许每个登录用户接入自己的 OpenAI 兼容大模型 API。它是平台全局模型配置之上的个人覆盖层：平台仍负责提供默认供应商和模型，用户个人配置存在时，会优先用于该用户自己的 AI 对话会话。

这套机制的目标不是让普通用户管理全站模型，而是让用户在不修改 `.env`、不重启服务的情况下，把自己的 baseURL、API Key 和模型列表接入当前创作端。

## 二、入口与交互

用户从创作端右上角设置入口进入“AI 模型配置”。设置弹窗提供以下字段：

| 字段 | 作用 |
| :--- | :--- |
| 供应商 ID | 作为模型 ID 前缀，默认是 `custom` |
| 供应商名称 | 设置中展示的可读名称 |
| baseURL | OpenAI 兼容 API 地址 |
| API Key | 私密凭证，输入时为密码框 |
| 模型列表 | 一行一个模型，形成该供应商下的可选模型 |
| 默认模型 | 可选；为空时使用模型列表第一项 |

读取配置时，服务端返回的是脱敏后的安全结构。API Key 不会回显到浏览器，前端只知道是否已经保存过 Key。用户保存时如果不输入新 Key，服务端会保留原 Key；用户清空配置时，会删除整条个人配置。

## 三、服务端存储

个人配置存储在 `users.db` 的 `user_model_configs` 表中，按 `user_id` 一人一条。配置内容保存为 JSON，API Key 不以明文写入 JSON，而是先加密后保存。

加密密钥来源按优先级选择：

| 优先级 | 来源 |
| :--- | :--- |
| 1 | `MODEL_CONFIG_ENCRYPTION_KEY` |
| 2 | `JWT_SECRET` |
| 3 | 开发兜底值 |

生产环境应显式配置稳定的 `MODEL_CONFIG_ENCRYPTION_KEY`，否则更换 `JWT_SECRET` 后，历史保存的 API Key 将无法解密。

## 四、配置校验与安全返回

保存个人配置时，服务端做基础校验：

| 校验项 | 规则 |
| :--- | :--- |
| baseURL | 必填，且只允许 http 或 https |
| 模型列表 | 至少一个非空模型 |
| 默认模型 | 如填写，必须存在于模型列表 |
| 供应商 ID | 会规范化为小写字母、数字、下划线和连字符 |
| API Key | 新输入时加密保存；留空且未要求清除时保留旧值 |

返回给前端的结构会把 `apiKey` 固定置空，并只返回 `hasApiKey`。这样浏览器可以提示“已保存”，但不会拿到明文 Key。

## 五、与全局配置的合并

个人配置会转换成 agent-service 能识别的 `backendProviders` 配置。合并规则如下：

| 顺序 | 说明 |
| :--- | :--- |
| 1 | 用户自己的 provider 放在第一位 |
| 2 | 管理员全局 provider 作为 fallback 追加 |
| 3 | 与用户 provider ID 相同的全局 provider 会被用户版本覆盖 |
| 4 | activeProviderId 使用用户 provider |
| 5 | activeModelId 使用用户默认模型；未设置时使用用户模型列表第一项 |

因此，当用户配置存在时，自己的模型会优先成为当前会话模型；当用户清空配置后，会话回到平台全局配置。

## 六、同步到 Agent 会话

用户模型配置需要进入 agent-service，AI 才能真正调用对应模型。同步发生在两个时机：

| 时机 | 行为 |
| :--- | :--- |
| 创建或复用编辑会话 | Session API 读取用户配置并推送给对应 agent-service session |
| 用户保存个人配置 | API 找到该用户的活跃会话并逐个推送最新配置 |

推送通过 author-site 与 agent-service 之间的内部接口完成，使用共享的 `INTERNAL_API_TOKEN` 鉴权。agent-service 收到后，将配置保存在 session 级内存表中；如果该 session 的 Agent 已经存在，则立即更新 Agent 配置。

推送失败不会阻止用户配置保存。这样可以避免 agent-service 短暂不可用时丢失用户设置，但用户可能需要刷新或重新打开编辑页才能让新配置进入对话。

## 七、消息发送时的模型确认

模型选择器维护的是用户当前希望使用的模型。为了避免“界面显示的模型”和“后端实际运行的模型”脱节，每条 AI 消息发送时都会把当前选中的完整模型 ID 一起带给 agent-service。

这条消息级模型 ID 是一次发送的明确指令：流式 WebSocket 和非流式 HTTP 兜底路径都必须携带它。agent-service 收到后会先忽略空字符串；如果请求带了有效模型 ID，就在本轮执行前把对应 session 切到该模型，再开始运行。这样即使模型列表连接中的 `set_model` 同步延迟或丢失，真正执行的消息仍以输入框当前选择为准。

后端日志里的运行模型应记录这次消息最终使用的完整模型 ID。若日志出现空模型，表示消息级模型没有正确进入运行链路，需要优先检查 AIChat 到 StreamService/AgentClient 的传参，而不是只检查管理后台配置。

## 八、运行时优先级

AI 对话使用模型配置时，整体优先级是：

1. 当前消息显式携带的模型 ID。
2. 当前 session 已切换到的模型。
3. 当前 session 的用户模型配置。
4. 管理后台推送到 agent-service 的全局供应商配置。
5. agent-service 启动时从环境变量读取的 fallback 配置。

这保证了个人配置只影响自己的会话，不会污染其他用户，也不会覆盖平台全局默认。
