## 一、概述

本系统通过**四层防护机制**约束 AI Agent 的行为,确保 AI 在安全、可控的范围内执行代码生成和文件操作任务。这种分层设计既保证系统安全性,又给予 AI 足够的自主性来完成开发任务。

### 约束架构总览

```
┌─────────────────────────────────────────────┐
│ L4: 用户确认层 (PermissionDialog)            │ ← 人工审核 ✅ 已实现
│   敏感操作需用户手动确认                       │
├─────────────────────────────────────────────┤
│ L3: 工作空间参考层 (references/)             │ ← 参考手册 ✅ 已实现
│   配置系统参考、UI 最佳实践等                 │
├─────────────────────────────────────────────┤
│ L2: 系统提示层 (System Prompt)               │ ← 软约束 ✅ 已实现
│   AI角色定义 + 怎么做 + 能做什么 + 不能做什么  │
├─────────────────────────────────────────────┤
│ L1: 文件系统权限层 (Permission Config)       │ ← 硬限制 ✅ 已注入
│   文件读写权限 + 命令执行权限                  │
└─────────────────────────────────────────────┘
```

**核心原则**: 外层约束优先于内层,硬限制无法被 AI 绕过,软约束依赖 AI 的自我遵守。

## 二、L1 - 文件系统权限层 (硬限制)

### 2.1 配置文件位置

**文件**: `packages/author-site/src/lib/templates/permission-config.ts`

该文件导出两个核心模板:

- `OPENCODE_CONFIG_TEMPLATE`: 文件系统权限配置
- `AGENTS_MD_TEMPLATE`: AI 行为规范模板

### 2.2 权限配置详解

`OPENCODE_CONFIG_TEMPLATE` 采用细粒度的模式匹配语法,定义三类权限:

```typescript
export const OPENCODE_CONFIG_TEMPLATE = {
  $schema: "https://opencode.ai/config.json",
  permission: {
    edit: {
      /* 编辑权限 */
    },
    read: {
      /* 读取权限 */
    },
    bash: {
      /* 命令执行权限 */
    },
    external_directory: {} /* 外部目录访问 */,
  },
};
```

#### 2.2.1 编辑权限 (edit)

| 规则                                    | 权限    | 说明                 |
| --------------------------------------- | ------- | -------------------- |
| `'*': 'deny'`                           | ❌ 禁止 | 默认禁止编辑所有文件 |
| `'index.tsx': 'allow'`                  | ✅ 允许 | 允许编辑页面组件     |
| `'config.schema.json': 'allow'`         | ✅ 允许 | 允许编辑配置 Schema  |
| `'project.config.schema.json': 'allow'` | ✅ 允许 | 允许编辑项目级配置   |
| `'.demo.json': 'allow'`                 | ✅ 允许 | 允许编辑页面元数据   |
| `'AGENTS.md': 'allow'`                  | ✅ 允许 | 允许编辑规则文件自身 |

**关键设计**: 采用**白名单模式**,只有明确允许的文件才能被修改,防止 AI 意外修改系统文件。

#### 2.2.2 读取权限 (read)

| 规则                | 权限    | 说明                         |
| ------------------- | ------- | ---------------------------- |
| `'*': 'allow'`      | ✅ 允许 | 默认允许读取所有文件         |
| `'*.env': 'deny'`   | ❌ 禁止 | 禁止读取环境变量文件         |
| `'*.env.*': 'deny'` | ❌ 禁止 | 禁止读取 `.env.local` 等变体 |

**设计意图**: AI 需要读取代码和配置来理解项目结构,但必须隔离敏感信息(数据库密码、API Key 等)。

#### 2.2.3 命令执行权限 (bash)

| 规则                | 权限    | 说明                   |
| ------------------- | ------- | ---------------------- |
| `'*': 'ask'`        | ⚠️ 询问 | 默认所有命令需用户确认 |
| `'ls *': 'allow'`   | ✅ 允许 | 允许列出目录           |
| `'cat *': 'allow'`  | ✅ 允许 | 允许查看文件内容       |
| `'grep *': 'allow'` | ✅ 允许 | 允许搜索文件内容       |
| `'rm *': 'deny'`    | ❌ 禁止 | **禁止删除文件**       |
| `'mv *': 'deny'`    | ❌ 禁止 | **禁止移动文件**       |
| `'cp *': 'deny'`    | ❌ 禁止 | **禁止复制文件**       |
| `'mkdir *': 'deny'` | ❌ 禁止 | **禁止创建目录**       |

**安全策略**:

- 只读命令自动允许(`ls`, `cat`, `grep`)
- 写操作命令默认需要用户确认(`ask`)
- 破坏性命令明确禁止(`rm`, `mv`, `cp`, `mkdir`)

### 2.4 配置修改指南

如需调整 AI 的文件操作权限,修改 `permission-config.ts`:

```typescript
// 示例：允许 AI 编辑 TypeScript 类型定义文件
edit: {
  '*': 'deny',
  'index.tsx': 'allow',
  'config.schema.json': 'allow',
  'project.config.schema.json': 'allow',
  '.demo.json': 'allow',
  'AGENTS.md': 'allow',
  '*.d.ts': 'allow',  // ← 新增：允许编辑类型定义
},

// 示例：允许 AI 安装 npm 包
bash: {
  '*': 'ask',
  'ls *': 'allow',
  'cat *': 'allow',
  'grep *': 'allow',
  'rm *': 'deny',
  'mv *': 'deny',
  'cp *': 'deny',
  'mkdir *': 'deny',
  'npm install *': 'allow',  // ← 新增：允许安装包
},
```

**⚠️ 注意事项**:

- 规则按顺序匹配,更具体的规则会覆盖通用规则
- 修改后需要新建 Session 才能生效
- 过于宽松的权限可能导致 AI 破坏项目结构

## 三、L2 - 系统提示层 (软约束)

### 3.1 L2 的定位

L2 是 AI 的**任务手册**，告诉 AI：

- **怎么做**：任务执行流程、代码质量标准、文件修改决策规则
- **能做什么**：允许的操作范围
- **不能做什么**：禁止行为清单

系统提示词分布在三个位置, 各自承担不同职责:

### 3.4 L3：工作空间参考层 (references/) ✅ 已实现

**实现方案**: 采用 `references/` 目录存放参考手册，AI 按需查阅。

**核心设计**:

- L2 (`demo-generator.template.md`) 定义**规则**：怎么做、能做什么、不能做什么
- L3 (`references/`) 提供**参考知识**：配置系统规范、控件类型、完整示例
- L2 是**不变的规则**, L3 是**按需查阅的字典**
- 类比: L2 是"员工手册", L3 是"技术字典"

**L3 文件**:

| 文件                          | 内容             | 用途                                           |
| :---------------------------- | :--------------- | :--------------------------------------------- |
| `references/config-system.md` | 配置系统参考手册 | 生成 schema 时查阅控件映射、扩展字段、完整示例 |

**注入流程**:

```
Workspace 创建
  ↓
injectOpencodeAgentConfig()
  ↓
injectReferences() — 复制 references/*.md 到工作空间
  ↓
AI 对话时，L2 提示：「生成 schema 前必须先读取 references/config-system.md」
  ↓
AI 主动查阅 → 生成高质量 schema
```

**隐藏配置**:

`references/` 目录在文件树中隐藏（`workspace-file-utils.ts` 的 `HIDDEN_ENTRIES`），用户无需看到。

**与 L2 的关系**:

```markdown
# demo-generator.template.md 中的引用指令

## 参考文件

生成或修改 `config.schema.json` 前，**必须先读取** `references/config-system.md`，
了解配置系统支持的控件类型、扩展字段和完整示例。
```

**为什么不用 AGENTS.md**:

原方案考虑将 L3 内容注入到 `AGENTS.md`（工作空间根目录），但该方案存在以下问题：

1. AGENTS.md 是静态文件，AI 每次对话都会加载，增加上下文负担
2. 参考知识量大（配置系统规范、控件速查表等），不适合作为 AGENTS.md 内容
3. `references/` 目录可扩展，后续可添加更多参考文件

**当前 L2 中保留的“工作空间结构”和“页面信息”章节**:

这些内容属于运行时动态上下文，仍在 L2 中通过占位符注入（`{{PAGE_LIST}}` 等），因为：

1. 这些信息是**当前工作空间的快照**，AI 每次对话都需要知道
2. 不属于“参考知识”，属于“当前上下文”

## 五、L4 - 用户确认层 (人工审核) ✅ 已实现

### 5.1 权限请求流程

```
AI 执行敏感操作
  ↓
触发 permission_request 事件
  ↓
前端收到事件，显示 PermissionDialog
  ↓
用户查看操作详情（文件路径、操作类型）
  ↓
用户选择：
  ├─ 允许（本次） → 发送 allow
  ├─ 始终允许 → 发送 allow_always
  └─ 拒绝 → 发送 deny
  ↓
AI 收到响应，继续或中止操作
```

### 6.2 权限检查伪代码

```typescript
// agent-service 侧的权限检查逻辑
async function checkPermission(
  operation: Operation,
): Promise<PermissionResult> {
  // L1: 文件系统权限检查（硬限制）
  const l1Result = await checkFileSystemPermission(operation);
  if (l1Result === "deny") {
    return { allowed: false, reason: "权限配置禁止此操作" };
  }

  // L2/L3: AI 自我约束（软约束，无法代码化检查）
  // AI 根据 System Prompt 和 AGENTS.md 自主判断

  // L4: 用户确认检查
  if (l1Result === "ask") {
    return {
      allowed: false,
      requiresUserConfirmation: true,
      operation: operation,
    };
  }

  return { allowed: true };
}
```

## 七、典型场景分析

### 场景 1: AI 修改页面样式

**用户请求**: "把首页标题改成红色"

**系统处理流程**:

1. **L2/L3 决策**: AI 根据"文件修改决策规则"判断需要编辑 `demos/home/index.tsx`
2. **L1 权限检查**: `index.tsx: allow` → 通过
3. **L4 用户确认**: 白名单文件,无需确认 → 直接执行
4. **执行结果**: AI 修改 `className` 中的颜色类

### 场景 2: AI 创建新页面

**用户请求**: "创建一个用户中心页面"

**系统处理流程**:

1. **L2/L3 决策**: AI 根据"创建页面"流程,生成 demoId,准备创建三个文件
2. **L1 权限检查**:

- `index.tsx: allow` → 通过
- `config.schema.json: allow` → 通过
- `.demo.json: allow` → 通过
- `mkdir *': deny` → ❌ **禁止创建目录**

3. **权限冲突**: AI 无法创建目录,但文件权限允许
4. **解决方案**:

- 方案 A: 用户在界面手动创建页面(推荐)
- 方案 B: 修改 L1 配置允许 `mkdir` (不推荐,安全风险)

### 场景 3: AI 尝试删除文件

**用户请求**: "删除这个没用的组件"

**系统处理流程**:

1. **L2/L3 决策**: AI 判断需要删除某个文件
2. **L1 权限检查**: `rm *: deny` → ❌ **直接拒绝**
3. **操作中止**: AI 返回提示"无法执行删除操作,请在界面中操作"

### 场景 4: AI 读取环境变量

**AI 行为**: 尝试读取 `.env` 获取 API Key

**系统处理流程**:

1. **L1 权限检查**: `*.env: deny` → ❌ **直接拒绝**
2. **L2 约束**: System Prompt 未告知 API Key 位置
3. **结果**: AI 无法获取敏感信息

### 场景 5: AI 执行危险命令

**AI 行为**: 尝试运行 `rm -rf node_modules && npm install`

**系统处理流程**:

1. **L1 权限检查**: `rm *: deny` → ❌ **直接拒绝**
2. **即使用户授权**: 硬限制无法被 `allow_always` 覆盖
3. **结果**: 命令被底层拦截,不会执行

## 八、安全性评估

### 8.1 防护层有效性

| 威胁类型          | L1          | L2/L3       | L4          | 综合评级 |
| ----------------- | ----------- | ----------- | ----------- | -------- |
| AI 误删文件       | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 泄露敏感信息   | ✅ 强制阻止 | ⚠️ 依赖遵守 | N/A         | **高**   |
| AI 执行危险命令   | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 修改系统配置   | ✅ 强制阻止 | ✅ 明确禁止 | N/A         | **高**   |
| AI 超出任务范围   | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |
| AI 使用不安全依赖 | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |

### 8.2 潜在风险点

1. **System Prompt 注入攻击**: 用户通过对话诱导 AI 忽略约束

- **缓解措施**: L1 硬限制无法被 Prompt 覆盖

2. **权限配置过于宽松**: 允许过多文件/命令

- **缓解措施**: 默认采用白名单 + 最小权限原则

3. **allow_always 滥用**: 用户习惯性授权

- **缓解措施**: Session 级别隔离,定期重新确认

4. **后端实现差异**: 三个后端的 `buildSystemPrompt()` 可能不同步

- **缓解措施**: 代码审查 + 单元测试验证一致性
