## 一、概述

本系统通过**四层防护机制**约束 AI Agent 的行为,确保 AI 在安全、可控的范围内执行代码生成和文件操作任务。这种分层设计既保证系统安全性,又给予 AI 足够的自主性来完成开发任务。

### 约束架构总览

```
┌──────────────────────────────────────────────────────────────────┐
│ L5: 用户确认层 (原 L4, PermissionDialog)                         │ ← 人工审核 ✅ 已实现
│   敏感操作需用户手动确认                                           │
├──────────────────────────────────────────────────────────────────┤
│ L4: 记忆层 (memory.md)                                           │ ← 会话首条注入 ✅ 已实现
│   用户偏好、关键决策、项目约定（跨会话长期记忆）                    │
├──────────────────────────────────────────────────────────────────┤
│ L3: 动态上下文层 (user message 前缀)                              │ ← 运行时注入 ✅ 已实现
│   工作空间页面列表 + 代码内容（≤2 页时嵌入）+ 知识库索引           │
├──────────────────────────────────────────────────────────────────┤
│ L2: 系统提示层 (System Prompt, 100% 静态)                         │ ← 软约束 ✅ 已实现
│   AI 怎么做 + 能做什么 + 不能做什么 + 记忆维护规则 + L5 权限确认说明│
├──────────────────────────────────────────────────────────────────┤
│ L1: 文件系统权限层 (pi-tools/permissions.ts)                      │ ← 硬限制 ✅ 已实现
│   路径白/黑名单 + 命令白/黑名单 + beforeToolCall 拦截             │
└──────────────────────────────────────────────────────────────────┘
```

**核心原则**: 外层约束优先于内层,硬限制无法被 AI 绕过,软约束依赖 AI 的自我遵守。

## 二、L1 - 文件系统权限层 (硬限制)

### 2.1 权限配置文件

**文件**: `packages/agent-service/src/backends/pi-tools/permissions.ts`

v3.2 后，权限检查不再通过 `opencode.json` 的 `permission` 字段，而是由 Pi Agent 的工具层直接执行。权限配置采用 `PermissionConfig` 接口：

```typescript
export interface PermissionConfig {
  allowedPaths: string[]; // 路径白名单（glob 模式）
  deniedPatterns: string[]; // 路径黑名单（glob 模式）
  allowedCommands: string[]; // 命令白名单
  deniedCommands: string[]; // 命令黑名单
}
```

### 2.2 默认权限配置

`DEFAULT_WORKSPACE_PERMISSIONS` 定义了开箱即用的安全策略：

#### 2.2.1 路径权限

| 规则                                      | 类型    | 说明                                        |
| :---------------------------------------- | :------ | :------------------------------------------ |
| `**`                                      | ✅ 允许 | 工作空间内通用访问（黑名单优先）            |
| `demos/*/index.tsx`                       | ✅ 允许 | 页面组件代码                                |
| `demos/*/config.schema.json`              | ✅ 允许 | 页面配置定义                                |
| `workspace-tree.json`                     | ✅ 允许 | 工作空间统一清单（页面元数据 + 文件夹结构） |
| `project.config.schema.json`              | ✅ 允许 | 项目级共享配置                              |
| `AGENTS.md`                               | ✅ 允许 | 规则文件                                    |
| `index.tsx` / `config.schema.json`        | ✅ 允许 | 根目录页面文件（兼容单页面模式）            |
| `**/*.env` / `**/*.env.*`                 | ❌ 禁止 | 环境变量文件                                |
| `**/.git` / `**/.git/**`                  | ❌ 禁止 | Git 目录                                    |
| `**/node_modules` / `**/node_modules/**`  | ❌ 禁止 | 依赖目录                                    |
| `**/packages` / `**/packages/**`          | ❌ 禁止 | 工作空间外源码                              |
| `**/.opencode` / `**/.opencode/**`        | ❌ 禁止 | Agent 配置目录                              |
| `**/.workspace.json` / `**/.session.json` | ❌ 禁止 | 系统元数据文件                              |
| `**/.canvas-layout.json`                  | ❌ 禁止 | 画布布局状态文件，只能由画布专用工具写入    |

**安全策略**: 黑名单优先于白名单检查，路径必须在 `workingDir` 内（防止目录穿越）。

#### 2.2.2 命令执行权限

| 类型    | 命令                                                                            | 说明                 |
| :------ | :------------------------------------------------------------------------------ | :------------------- |
| ✅ 允许 | `npm`, `node`, `npx`, `ls`, `cat`, `head`, `tail`, `grep`, `find`, `wc`, `echo` | 只读命令 + 包管理器  |
| ❌ 禁止 | `rm`, `rmdir`, `mv`, `cp`, `mkdir`, `sudo`, `chmod`, `chown`                    | 破坏性命令和权限操作 |

**安全策略**: 命令名提取后先检查黑名单再检查白名单，不在任一列表中的命令默认拒绝。

### 2.3 权限执行机制

权限检查在两个层级执行：

**1. `beforeToolCall` 钩子（pi-agent.ts）**:

```typescript
beforeToolCall: async (context) => {
  const toolName = context.toolCall.name;
  if (
    toolName === "readFile" ||
    toolName === "writeFile" ||
    toolName === "listFiles"
  ) {
    const args = context.args as { path?: string };
    if (args.path && !isPathAllowed(args.path, workingDir, permissions)) {
      return {
        block: true,
        reason: `Access denied: path "${args.path}" is not allowed`,
      };
    }
  }
  return undefined;
};
```

**2. 工具实现层（file-tools.ts、bash-tool.ts）**:

每个工具在 `execute` 函数内部再次检查权限：

- `readFile` / `writeFile` / `listFiles`：调用 `isPathAllowed()` 检查路径权限
- `bash`：调用 `isCommandAllowed()` 检查命令权限
- 权限检查失败时返回 `isError: true` 的错误结果

### 2.4 权限配置传递

权限配置通过 `AgentConfig.permissions` 字段传递，未提供时使用 `DEFAULT_WORKSPACE_PERMISSIONS` 默认值：

```typescript
const permissions = config.permissions ?? DEFAULT_WORKSPACE_PERMISSIONS;
```

### 2.5 配置修改指南

如需调整权限规则,修改 `permissions.ts` 中的 `DEFAULT_WORKSPACE_PERMISSIONS`：

```typescript
// 示例：允许 AI 编辑 TypeScript 类型定义文件
allowedPaths: [
  '**',
  'demos/*/index.tsx',
  // ... 现有规则 ...
  '*.d.ts',  // ← 新增：允许编辑类型定义
],

// 示例：允许 AI 执行 npm install
allowedCommands: [
  'npm', 'node', 'npx', 'ls', 'cat', 'head', 'tail', 'grep', 'find', 'wc', 'echo',
  // npm install 已通过 'npm' 基础命令允许
],
```

**⚠️ 注意事项**:

- 黑名单优先于白名单检查
- 路径必须在 `workingDir` 内（目录穿越保护）
- 修改后需重启 agent-service 生效

### 2.6 Schema 修改约束

`config.schema.json` 的写入仍受 L1 路径权限保护。具体字段是否应新增、删除或修改，由 L2 静态提示词和 L3 当前页面上下文约束：没有用户明确配置诉求时，AI 不得自行推断配置字段；有明确诉求时，AI 必须只修改目标页面 schema，并避免重复定义项目级配置字段。

项目知识库仍可为配置规范提供补充材料。AI 需要项目特有规则时，应从 L3 知识库索引选择相关文档，再通过 `readFile` 或 `readFileWithLines` 读取 `knowledge/{文件名}`。

## 三、L2 - 系统提示层 (软约束)

### 3.1 L2 的定位

L2 是 AI 的**任务手册**，告诉 AI：

- **怎么做**：任务执行流程、代码质量标准、文件修改决策规则
- **能做什么**：允许的操作范围
- **不能做什么**：禁止行为清单

### 3.2 v3.4 静态/动态拆分架构

v3.4 将 System Prompt 拆分为**静态部分**和**动态部分**，以优化 LLM API 缓存命中率：

| 部分                      | 注入位置            | 更新频率                    | 缓存表现           |
| :------------------------ | :------------------ | :-------------------------- | :----------------- |
| **L2 静态规则**           | `systemPrompt` 字段 | 应用启动后不变              | 100% 缓存命中      |
| **L3 动态上下文**         | user message 前缀   | 每次 sendMessage 前重新扫描 | 随工作空间状态变化 |

### 3.3 L2 模板：system-prompt.md

**文件**: `packages/author-site/src/lib/agent/prompts/system-prompt.md`

通过 webpack `asset/source` 构建时嵌入为 `SYSTEM_PROMPT` 常量，包含完整的 AI 行为规则：

- **页面内容编辑**：自然语言匹配页面名称，自主判断修改目标
- **页面管理操作**：创建/重命名/删除页面的完整流程
- **项目级配置管理**：新增/删除/修改项目配置的规则
- **代码质量标准**：TypeScript、DemoProps 接口、Tailwind CSS 等
- **React 版本约束**：React 18.3.1 兼容、依赖白名单
- **禁止行为清单**：访问工作空间外文件、修改系统文件、重复定义字段等
- **文件修改决策规则**：样式/配置/组件/项目配置/页面元数据的判断规则
- **知识库查阅**：查阅 `knowledge/` 目录中的参考文档
- **子 Agent 委派**：通过 `delegateTask` 将独立、可并行或重复性强的工作交给短生命周期子 Agent，并由主 Agent 汇总验收
- **画布管理**：整理画布页面位置和尺寸时调用 `arrangeCanvasPages`，禁止直接写 `.canvas-layout.json`
- **记忆维护规则**：AI 如何维护 `memory.md`（何时读取/更新、操作规范、更新频率）
- **L5 权限确认说明**：哪些操作需要用户确认

**v3.4 关键变化**: 模板不再包含 `{{PROJECT_NAME}}`、`{{PAGE_LIST}}` 等占位符。动态上下文已移至 L3（user message 前缀）。

### 3.4 新建页面配置项约束（v3.4.4）

新建页面属于页面管理操作，不等同于新增配置项。L2 静态提示词现在把这两件事拆开处理：

- AI 创建新页面时仍需创建 `index.tsx` 与 `config.schema.json`，但用户没有明确配置诉求时，`config.schema.json` 只能保留空属性集合。
- 标题、文案、图片、颜色、按钮、布局等页面内容默认直接写在 `index.tsx`，不得为了“可编辑”而自行抽成配置字段。
- 用户明确说“添加配置项”“这个要可配置”“加一个字段”等时，AI 才可以按用户要求修改页面级或项目级 schema。
- 没有配置字段时，页面 Props 声明也必须为空，避免 Props 示例反向诱导 AI 制造 `title`、`description` 等字段。

该约束是 L2 软约束，当前通过 System Prompt 单元测试锁定。若模型仍绕过规则，需要在 Agent 服务工具层增加基于用户意图的 schema 字段变更硬拦截。

### 3.5 画布整理工具约束（v3.4.5）

画布布局是工作区状态，不属于页面树元数据。左侧页面树顺序仍由 `workspace-tree.json` 的 `order` 表达；画布中的页面位置、尺寸和视口由 `.canvas-layout.json` 表达。

为了避免 AI 手写隐藏状态文件导致格式损坏，L1 权限层禁止通用文件工具访问 `.canvas-layout.json`。当用户要求“整理画布”“排列画布页面”“调整画布页面位置或大小”时，L2 提示词会要求 AI 使用 `arrangeCanvasPages` 工具。

`arrangeCanvasPages` 当前承担最小闭环能力：

- 默认保留画布现有大致分组，只做对齐、间距修正和视口适配。
- 可按页面树顺序重新铺成规则网格。
- 可选择保留当前画布尺寸，或按页面 `$demo.previewSize` 重置尺寸。
- 可只整理指定页面，页面 ID 必须来自页面清单工具。

System Prompt 由 `buildStaticSystemPrompt()` 根据 agent-service 返回的工具能力分支生成：当 `arrangeCanvasPages` 不存在时，AI 必须明确告知用户当前服务版本过旧或工具未加载，不能声称已完成画布整理。

### 3.6 子 Agent 委派说明（v3.4.3）

创作端的主 Agent 会在 L2 静态提示中看到 `delegateTask` 工具。这个工具来自独立 Agent 服务层的 Pi Agent 工具集，支持在同一轮中并行委派多个彼此独立的子任务；具体运行边界见 [独立 Agent 服务层的 Pi Agent 子 Agent 文档](../../../独立Agent服务层/06-Pi-Agent子Agent.md)。

在创作端语义中，子 Agent 适合承担“可拆出去的小包裹”：例如批量页面排查、重复布局修复、独立文件审查或先行收集候选方案。主 Agent 委派时需要说明任务范围、相关文件和验收标准；子 Agent 返回后，主 Agent 仍负责检查结果、继续收尾，并把最终结论回复给用户。

子 Agent 继承当前工作区、权限和文件工具，因此 L1 路径安全、知识库写保护、页面删除确认、配置字段约束仍然生效。子 Agent 自身不再拥有 `delegateTask`，避免任务无限递归。

前端展示层会将 `delegateTask` 识别为“委派子 Agent”，让用户在 AI 处理过程里能直接看出当前步骤不是普通命令，而是一次子 Agent 执行。流式处理中，单个子 Agent 返回后显示为“待主 Agent 汇总”；整轮消息结束后才显示为“已完成”。

### 3.7 构建流程

**文件**: `packages/author-site/src/lib/agent/system-prompt.ts`

```typescript
import SYSTEM_PROMPT from './prompts/system-prompt.md';

// 静态部分：应用启动时缓存（100% 不变）
export function buildStaticSystemPrompt(): string {
  return SYSTEM_PROMPT;
}

// 动态部分：每次 sendMessage 前渲染（L3 上下文）
export function buildDynamicContextPrefix(context: SystemPromptContext): string {
  return render(WORKSPACE_STATUS_TEMPLATE, {
    PROJECT_NAME: context.projectName,
    PROJECT_CONFIG_STATUS: context.projectConfigStatus,
    WORKSPACE_PATH: context.workspacePath,
    PAGE_COUNT: String(context.pageCount),
    PAGE_LIST: context.pageList || '（暂无页面）',
  });
}

// 记忆前缀（L4 注入时使用）
export function buildMemoryPrefix(content: string): string {
  const charCount = content.replace(/\s/g, '').length;
  return `\n\n## 项目记忆（跨会话长期记忆）\n\n${content}\n\n[系统：当前 memory.md 共 ${charCount} 字]\n`;
}

// 知识库索引前缀（L3 注入时使用）
export function buildKnowledgeIndexPrefix(content: string): string {
  return `\n\n${content}\n`;
}
```

**注入时机**：

1. **WebSocket 路径**（stream-service.ts）：`sendMessage()` 将静态 prompt 放入 `systemPrompt` 字段，L3 前缀 + 知识库索引 + L4 记忆拼接到 user content 头部；首条消息额外追加 L4 记忆内容
2. **HTTP 路径**（api/ai/chat/route.ts）：同样将静态 prompt 放入 `options.systemPrompt`，L3 前缀 + 知识库索引 + L4 记忆拼接到 `finalContent`；首条消息额外追加 L4 记忆内容

## 四、L3 - 动态上下文层 (user message 前缀) ✅ 已实现(v3.4)

### 4.5 knowledge/ 目录机制

`knowledge/` 目录取代旧的 `references/`，现在只存放用户项目知识文档。需要更高质量上下文时，AI 可调用 `knowledgeReport`，由知识库服务按访问上下文完成过滤并返回结构化报告。

#### 4.5.1 用户项目知识库

用户可手动添加 `.md` 文件到 `knowledge/` 目录，并更新 `manifest.json` 索引：

- **添加方式**：用户通过代码视图直接在 `knowledge/` 目录创建 `.md` 文件
- **索引更新**：用户文档元数据写入 `manifest.json`，历史 system 条目会被清理
- **AI 查阅**：L3 注入索引文本，AI 通过 `readFile` 工具主动查阅 `knowledge/{文件名}`

#### 4.5.2 知识库索引管理

`manifest.json` 只管理用户项目知识文档索引。模板知识由 `data/knowledge/templates/{templateId}/reading-map.json` 提供，报告层统一标注来源和可信等级：

```json
{
  "version": 1,
  "items": [
    {
      "id": "kb_user_001",
      "title": "React 组件规范",
      "source": "user",
      "description": "项目 React 组件编写规范",
      "fileName": "react-patterns.md"
    }
  ]
}
```

#### 4.5.4 知识报告工具

`knowledgeReport` 工具面向主 AI 提供“图书馆管理员式”的结构化报告。工具内部从系统知识快照和工作空间知识 manifest 建立 Basic 检索候选，按 `knowledge-core` 的访问上下文过滤可见范围，再输出结论摘要、相关资料、来源路径、可信等级、适用范围、建议继续读取的原文、未找到的信息和风险提示。创作端 AI 使用作者侧上下文，使用端只读 AI 使用 viewer 上下文，二者不能共享报告缓存。

知识来源分层、权限过滤、阅读地图和报告契约统一维护在 [独立知识库服务](../../../独立知识库服务/) 文档中；本节只说明 AI 行为层如何使用这些能力。

#### 4.5.5 自动迁移机制

`migrateReferencesToKnowledge()` 在工作空间创建时将旧 `references/` 迁移到 `knowledge/`：

- **触发条件**：`references/` 存在且 `knowledge/` 不存在
- **迁移内容**：所有 `.md` 文件，`source` 标记为 `"system"`
- **迁移后**：删除旧 `references/` 目录
- **文件保护**：AI 不得修改或删除 `knowledge/` 中的文件

## 五、L4 - 记忆层 (跨会话长期记忆) ✅ 已实现(v3.3)

### 5.1 设计目标

`memory.md` 是位于**项目根目录**的记忆文件，用于记录：

- **用户偏好**：写代码前先说明思路、拿不准时先问、改配置时要解释
- **关键决策**：首页用轮播 banner、详情页不加下拉刷新
- **项目约定**：自然语言描述，非技术人员看得懂、用户可直接编辑

### 5.2 与 AGENTS.md 的区别

|              | `AGENTS.md`（系统文件）        | `memory.md`（用户记忆文件）      |
| ------------ | ------------------------------ | -------------------------------- |
| **用途**     | 告诉 AI 项目技术架构和开发规范 | 记录用户偏好、项目约定、关键决策 |
| **读者**     | AI（技术人员）                 | 用户 + AI                        |
| **语言**     | 技术术语、代码结构             | 自然语言、日常表达               |
| **维护者**   | 开发者手动编辑                 | AI 自动维护、用户可编辑          |
| **内容风格** | 精确、结构化                   | 叙事化、易理解                   |

> **重要**：`memory.md` 绝不替代或修改 `AGENTS.md`。两者共存，互不干扰。

### 5.3 注入策略

**注入时机**：仅在**会话首条消息**时，读取 `memory.md` 并注入到消息内容之前。

```
首条消息：${L3前缀} + ${L4记忆内容} + ${用户消息}
后续消息：${L3前缀} + ${用户消息}  （不重复注入 L4）
```

**字数提示**：系统读取 `memory.md` 后，自动在末尾附加当前字数统计：

```
[系统：当前 memory.md 共 847 字]
```

这样 AI 无需手动计数，看到接近 1500 就知道该压缩了。该行仅注入时不写入文件。

**为什么只注入首条消息**：

- L3 每次刷新是因为页面列表可能变了；`memory.md` 在会话中几乎不会变
- 重复注入浪费 token，且可能让 AI 过度关注记忆内容而非用户当前消息

### 5.4 实现位置

| 文件                                   | 改动内容                                                                                                                                                                                                                                                                       |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `system-prompt.md`                     | L2 静态模板包含记忆维护规则（何时读取/更新、操作规范、更新频率）和 memory.md 文件模板                                                                                                                                                                                          |
| `system-prompt.ts`                     | 新增 `buildMemoryPrefix(content)` 函数（格式化 L4 注入内容，含字数统计）                                                                                                                                                                                                      |
| `scan-workspace.ts`                    | 新增 `readMemoryContent(workingDir)` 函数（服务端文件读取，容错返回 null）                                                                                                                                                                                                     |
| `api/ai/chat/route.ts`                 | HTTP 路径：在 L3 上下文后追加 `memory.md` 内容注入                                                                                                                                                                                                                             |

| `stream-service.ts`                    | 新增 `hasInjectedMemory` 标记，首条消息注入 L4 记忆（后续消息跳过）；`close()` 时重置标记；函数 `fetchDynamicContextPrefix` 重命名为 `fetchContextPrefix`，返回 `{ l3, memory, knowledgeIndex }`                                                                              |

### 5.5 AI 维护规则（L2 层注入）

将记忆维护规则注入到 System Prompt 的 L2 静态部分，告知 AI：

**何时更新**：

| 触发条件             | 用户典型发言                                     | 应更新的章节       |
| -------------------- | ------------------------------------------------ | ------------------ |
| **用户明确要求记住** | "请记住这个"、"以后都这样做"、"把这个记下来"     | 按内容放入对应章节 |
| **表达个人偏好**     | "我不喜欢……"、"我更习惯……"、"遇到这种情况先问我" | 我的偏好           |
| **做出关键决策**     | "那就用……吧"、"不用……了，换成……"、"我们决定……"   | 关键决策           |

**不应该记录什么**：

- 一次性操作（如"帮我调大这个按钮"）—— 不是长期记忆
- 讨论过程中的试探和犹豫 —— 只记最终决定
- 可以从代码里直接看到的信息
- 密码、密钥、Token 等敏感信息
- **系统提示词中已有的编码规范** —— memory.md 只记录对话中新产生的记忆

**操作规范**：

1. **先读后写**：更新前必须先用 readFile 读取当前 `memory.md` 的完整内容
2. **增量编辑**：只修改需要更新的章节，其他章节保持原样不动
3. **保留用户手写内容**：如果某个章节的内容看起来是用户手动写的，不要覆盖，只追加新内容
4. **去重检查**：新增内容前先确认是否已有类似信息，避免重复
5. **更新日期**：每次修改后在顶部更新「最后更新」日期
6. **极简表达**：每条决策一句话说清，用「——」分隔决定和原因。偏好每条不超过 15 字
7. **控制总长度**：建议控制在 1500 字以内。系统自动附字数提示，接近上限时 AI 压缩后再写入

├─ 发送时:
│   ├─ systemPrompt: STATIC_SYSTEM_PROMPT (L2)

    ↓

    │   ├─ isPathAllowed() → 路径权限

    │   └─ config.schema.json 写入路径权限校验
    │
    ├─ L2 检查: AI 根据 System Prompt 判断是否符合规则

## 八、典型场景分析

### 场景 1: AI 修改页面样式

**用户请求**: "把首页标题改成红色"

**系统处理流程**:

1. **L3 上下文**: user message 前缀包含当前页面列表，AI 知道「首页」对应 `demos/{homeId}/`
2. **L2 决策**: AI 根据"文件修改决策规则"判断需要编辑 `demos/{homeId}/index.tsx`
3. **L1 权限检查**: `isPathAllowed("demos/{homeId}/index.tsx")` → 白名单命中 → 通过
4. **L4 用户确认**: 白名单文件,无需确认 → 直接执行
5. **执行结果**: AI 修改 `className` 中的颜色类

### 场景 2: AI 创建新页面

**用户请求**: "创建一个用户中心页面"

**系统处理流程**:

1. **L2/L3 决策**: AI 根据"创建页面"流程,生成 demoId,准备创建文件并注册到 `workspace-tree.json`
2. **L1 权限检查**:

- `isPathAllowed("demos/{id}/index.tsx")` → ✅ 通过
- `isPathAllowed("demos/{id}/config.schema.json")` → ✅ 通过
- `isPathAllowed("workspace-tree.json")` → ✅ 通过
- 目录创建：`writeFile` 工具内部调用 `fs.mkdir(dir, { recursive: true })`，无需单独 mkdir 命令

3. **执行结果**: AI 通过 `writeFile` 工具创建目录并写入文件

### 场景 3: AI 尝试删除文件

**用户请求**: "删除这个没用的组件"

**系统处理流程**:

1. **L2 决策**: AI 判断需要删除某个文件
2. **L1 权限检查**: `isCommandAllowed("rm ...")` → `rm` 在 `deniedCommands` 中 → ❌ **直接拒绝**
3. **操作中止**: AI 返回提示"无法执行删除操作,请在界面中操作"

### 场景 4: AI 读取环境变量

**AI 行为**: 尝试读取 `.env` 获取 API Key

**系统处理流程**:

1. **L1 权限检查**: `isPathAllowed(".env")` → 命中黑名单 `**/*.env` → ❌ **直接拒绝**
2. **L2 约束**: System Prompt 未告知 API Key 位置
3. **结果**: AI 无法获取敏感信息

### 场景 5: AI 执行危险命令

**AI 行为**: 尝试运行 `rm -rf node_modules && npm install`

**系统处理流程**:

1. **L1 权限检查**: `isCommandAllowed("rm -rf node_modules ...")` → `rm` 在 `deniedCommands` 中 → ❌ **直接拒绝**
2. **硬限制**: 黑名单命令无法被任何方式覆盖
3. **结果**: 命令被工具层拦截，不会执行

### 场景 6A: AI 生成新页面但用户没有要求配置项

**用户请求**: "帮我生成一个活动首页"

**系统处理流程**:

1. **L2 决策**: AI 判断这是创建页面请求，不是配置项请求。
2. **页面内容生成**: 标题、文案、图片、颜色、按钮、布局等页面内容直接写入新页面的 `index.tsx`。
3. **默认 schema**: 新页面仍创建 `config.schema.json`，但保持空属性集合，不添加 `title`、`bannerImage`、`themeColor` 等字段。
4. **Props 约束**: 页面 Props 声明保持为空，不为页面内容制造 Props 字段。
5. **结果**: 用户得到完整页面，但配置面板不会出现 AI 自行推断出的配置项。

### 场景 6B: AI 修改配置 schema

**用户请求**: "给 Banner 图配置添加尺寸限制，要求宽度至少 750px"

**系统处理流程**:

1. **L2 决策**: 用户已明确提出图片尺寸限制，AI 可以修改目标页面 `config.schema.json`
2. **L3 上下文**: AI 根据当前页面列表确认目标页面路径
3. **按需知识查阅**: 如项目知识库存在配置规范，AI 先读取相关 `knowledge/{文件名}`，否则按静态规则执行
4. **L1 路径权限**: `isPathAllowed("demos/{id}/config.schema.json")` → ✅ 通过
5. **AI 生成配置**:

```json
{
  "bannerImage": {
    "type": "string",
    "format": "image",
    "title": "Banner 图",
    "ui:options": {
      "accept": "image/*",
      "maxSize": 5242880,
      "minWidth": 750,
      "maxWidth": 2048,
      "maxHeight": 1024
    }
  }
}
```

7. **结果**: 配置更新成功，用户上传 Banner 图时会自动校验尺寸

## 九、安全性评估

### 9.1 防护层有效性

| 威胁类型          | L1          | L2/L3       | L4          | 综合评级 |
| ----------------- | ----------- | ----------- | ----------- | -------- |
| AI 误删文件       | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 泄露敏感信息   | ✅ 强制阻止 | ⚠️ 依赖遵守 | N/A         | **高**   |
| AI 执行危险命令   | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 修改系统配置   | ✅ 强制阻止 | ✅ 明确禁止 | N/A         | **高**   |
| AI 超出任务范围   | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |
| AI 使用不安全依赖 | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |

### 9.2 潜在风险点

1. **System Prompt 注入攻击**: 用户通过对话诱导 AI 忽略约束

- **缓解措施**: L1 硬限制无法被 Prompt 覆盖

2. **权限配置过于宽松**: 允许过多文件/命令

- **缓解措施**: 默认采用白名单 + 最小权限原则

3. **权限响应取消**: 用户习惯性取消请求

- **缓解措施**: L1 硬限制始终生效，取消不会导致安全风险

4. **单后端故障**: Pi Agent 后端异常可能导致全部功能不可用

- **缓解措施**: 错误处理 + 日志记录，后端重启恢复

5. **memory.md 过度膨胀**: AI 过度记录导致文件变长

- **缓解措施**: 1500 字上限建议 + 系统自动附字数提示 + 接近上限时 AI 压缩后再写入

6. **AI 覆盖用户手写内容**: AI 更新时覆盖用户手动修改的记忆

- **缓解措施**: 操作规范明确要求「保留用户手写内容，只追加新内容」

### 11.2 常见错误

| :-------------------- | :---------------------------- | :------------------------------------------------------------ |
| AI 无法编辑文件       | L1 `isPathAllowed` 拒绝       | 检查 `permissions.ts` 的 `allowedPaths` 和 `deniedPatterns`   |

### 13.2 已实现功能

✅ **L1 - 文件系统权限层**

- `permissions.ts` 定义 `PermissionConfig` 接口和 `DEFAULT_WORKSPACE_PERMISSIONS`

- 路径黑名单：`.env`、`.git`、`node_modules`、`packages`、`.opencode` 等
- 命令黑名单：`rm`、`rmdir`、`mv`、`cp`、`mkdir`、`sudo`、`chmod`、`chown`
- 命令白名单：`npm`、`node`、`npx`、`ls`、`cat`、`head`、`tail`、`grep`、`find`、`wc`、`echo`

✅ **L2 - 系统提示层（任务手册）**

- `author-site/prompts/system-prompt.md` 包含完整的 AI 行为规则
- `system-prompt.ts` 的 `buildStaticSystemPrompt()` 返回静态模板
- 通过 `systemPrompt` 字段注入到 Pi Agent，100% 静态（LLM API 缓存命中）
- 包含知识库查阅指令：「查阅 `knowledge/` 目录中的参考文档」
- 包含新建页面配置项约束：无明确配置诉求时，新页面 schema 保持空属性集合，Props 也不声明页面内容字段

✅ **L3 - 动态上下文层**

- `workspace-status.template.ts` 定义 L3 模板（页面列表）
- `scan-workspace.ts` 的 `scanWorkspaceContext()` 扫描 `demos/` 目录
- 页面数 ≤ 2 时嵌入页面代码和配置内容（`MAX_INLINE_PAGES = 2`）

- 每次 sendMessage 前拼接到 user message 头部
- `scanKnowledgeIndex()` 扫描 `knowledge/manifest.json` 生成项目知识库索引，并清理历史 system 条目

✅ **L4 - 记忆层（跨会话长期记忆）**

- `memory.md` 位于项目根目录，记录用户偏好和关键决策
- 记忆维护规则写入 `system-prompt.md`（何时读取/更新、操作规范、更新频率）

- `buildMemoryPrefix()` 格式化 L4 注入内容（含字数统计）
