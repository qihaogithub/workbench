## 一、概述

本系统通过**四层防护机制**约束 AI Agent 的行为,确保 AI 在安全、可控的范围内执行代码生成和文件操作任务。这种分层设计既保证系统安全性,又给予 AI 足够的自主性来完成开发任务。

### 约束架构总览

```
┌─────────────────────────────────────────────┐
│ L4: 用户确认层 (PermissionDialog)            │ ← 人工审核 ✅ 已实现
│   敏感操作需用户手动确认                       │
├─────────────────────────────────────────────┤
│ L3: 工作空间现状层 (AGENTS.md)               │ ← 动态注入 ⚠️ 待实现
│   当前工作空间的目录结构、页面列表、配置现状   │
├─────────────────────────────────────────────┤
│ L2: 系统提示层 (System Prompt)               │ ← 软约束 ✅ 已实现
│   AI角色定义 + 怎么做 + 能做什么 + 不能做什么  │
├─────────────────────────────────────────────┤
│ L1: 文件系统权限层 (Permission Config)       │ ← 硬限制 ⚠️ 待实现
│   文件读写权限 + 命令执行权限                  │
└─────────────────────────────────────────────┘
```

**核心原则**: 外层约束优先于内层,硬限制无法被 AI 绕过,软约束依赖 AI 的自我遵守。

## 二、L1 - 文件系统权限层 (硬限制)

### 2.1 配置文件位置

**文件**: `packages/author-site/src/lib/templates/permission-config.ts`

该文件导出两个核心模板:

- `OPENCODE_CONFIG_TEMPLATE`: 文件系统权限配置
- `AGENTS_MD_TEMPLATE`: AI 行为规范模板

### 2.2 权限配置详解

`OPENCODE_CONFIG_TEMPLATE` 采用细粒度的模式匹配语法,定义三类权限:

```typescript
export const OPENCODE_CONFIG_TEMPLATE = {
  $schema: "https://opencode.ai/config.json",
  permission: {
    edit: {
      /* 编辑权限 */
    },
    read: {
      /* 读取权限 */
    },
    bash: {
      /* 命令执行权限 */
    },
    external_directory: {} /* 外部目录访问 */,
  },
};
```

#### 2.2.1 编辑权限 (edit)

| 规则                                    | 权限    | 说明                 |
| --------------------------------------- | ------- | -------------------- |
| `'*': 'deny'`                           | ❌ 禁止 | 默认禁止编辑所有文件 |
| `'index.tsx': 'allow'`                  | ✅ 允许 | 允许编辑页面组件     |
| `'config.schema.json': 'allow'`         | ✅ 允许 | 允许编辑配置 Schema  |
| `'project.config.schema.json': 'allow'` | ✅ 允许 | 允许编辑项目级配置   |
| `'.demo.json': 'allow'`                 | ✅ 允许 | 允许编辑页面元数据   |
| `'AGENTS.md': 'allow'`                  | ✅ 允许 | 允许编辑规则文件自身 |

**关键设计**: 采用**白名单模式**,只有明确允许的文件才能被修改,防止 AI 意外修改系统文件。

#### 2.2.2 读取权限 (read)

| 规则                | 权限    | 说明                         |
| ------------------- | ------- | ---------------------------- |
| `'*': 'allow'`      | ✅ 允许 | 默认允许读取所有文件         |
| `'*.env': 'deny'`   | ❌ 禁止 | 禁止读取环境变量文件         |
| `'*.env.*': 'deny'` | ❌ 禁止 | 禁止读取 `.env.local` 等变体 |

**设计意图**: AI 需要读取代码和配置来理解项目结构,但必须隔离敏感信息(数据库密码、API Key 等)。

#### 2.2.3 命令执行权限 (bash)

| 规则                | 权限    | 说明                   |
| ------------------- | ------- | ---------------------- |
| `'*': 'ask'`        | ⚠️ 询问 | 默认所有命令需用户确认 |
| `'ls *': 'allow'`   | ✅ 允许 | 允许列出目录           |
| `'cat *': 'allow'`  | ✅ 允许 | 允许查看文件内容       |
| `'grep *': 'allow'` | ✅ 允许 | 允许搜索文件内容       |
| `'rm *': 'deny'`    | ❌ 禁止 | **禁止删除文件**       |
| `'mv *': 'deny'`    | ❌ 禁止 | **禁止移动文件**       |
| `'cp *': 'deny'`    | ❌ 禁止 | **禁止复制文件**       |
| `'mkdir *': 'deny'` | ❌ 禁止 | **禁止创建目录**       |

**安全策略**:

- 只读命令自动允许(`ls`, `cat`, `grep`)
- 写操作命令默认需要用户确认(`ask`)
- 破坏性命令明确禁止(`rm`, `mv`, `cp`, `mkdir`)

- ❌ 未注入: 没有任何代码将此模板写入工作空间

### 2.4 配置修改指南

如需调整 AI 的文件操作权限,修改 `permission-config.ts`:

```typescript
// 示例：允许 AI 编辑 TypeScript 类型定义文件
edit: {
  '*': 'deny',
  'index.tsx': 'allow',
  'config.schema.json': 'allow',
  'project.config.schema.json': 'allow',
  '.demo.json': 'allow',
  'AGENTS.md': 'allow',
  '*.d.ts': 'allow',  // ← 新增：允许编辑类型定义
},

// 示例：允许 AI 安装 npm 包
bash: {
  '*': 'ask',
  'ls *': 'allow',
  'cat *': 'allow',
  'grep *': 'allow',
  'rm *': 'deny',
  'mv *': 'deny',
  'cp *': 'deny',
  'mkdir *': 'deny',
  'npm install *': 'allow',  // ← 新增：允许安装包
},
```

**⚠️ 注意事项**:

- 规则按顺序匹配,更具体的规则会覆盖通用规则
- 修改后需要新建 Session 才能生效
- 过于宽松的权限可能导致 AI 破坏项目结构

## 三、L2 - 系统提示层 (软约束)

### 3.1 L2 的定位

L2 是 AI 的**任务手册**，告诉 AI：

- **怎么做**：任务执行流程、代码质量标准、文件修改决策规则
- **能做什么**：允许的操作范围
- **不能做什么**：禁止行为清单

系统提示词分布在三个位置, 各自承担不同职责:

| 层级             | 文件位置                                                                              | 职责                                     | 更新频率       |
| ---------------- | ------------------------------------------------------------------------------------- | ---------------------------------------- | -------------- |
| **全局模板**     | `packages/author-site/src/lib/agent-prompts/demo-generator.template.md`               | 角色定义、任务流程、行为规则、禁止行为   | 低(架构变更时) |
| **后端适配**     | `packages/agent-service/src/backends/{claude,codex,gemini}.ts`                        | 注入技术栈、依赖白名单、代码规范         | 中(依赖更新时) |
| **工作空间现状** | `packages/author-site/src/lib/templates/permission-config.ts` 的 `AGENTS_MD_TEMPLATE` | 当前工作空间目录结构、页面列表、配置现状 | 低(需求变更时) |

## 五、L4 - 用户确认层 (人工审核) ✅ 已实现

### 5.1 权限请求流程

```
AI 执行敏感操作
  ↓
触发 permission_request 事件
  ↓
前端收到事件，显示 PermissionDialog
  ↓
用户查看操作详情（文件路径、操作类型）
  ↓
用户选择：
  ├─ 允许（本次） → 发送 allow
  ├─ 始终允许 → 发送 allow_always
  └─ 拒绝 → 发送 deny
  ↓
AI 收到响应，继续或中止操作
```

### 5.2 前端组件实现

**组件位置**: `packages/author-site/src/components/ai-elements/`

**关键文件**:

- `permission-dialog.tsx`: 权限确认对话框
- `use-chat-stream.ts`: 处理 `permission_request` 事件
- `stream-service.ts`: 发送权限响应

### 5.3 权限请求触发条件

| 操作类型 | 触发条件              | 用户可见信息       |
| -------- | --------------------- | ------------------ |
| 文件编辑 | 编辑非白名单文件      | 文件路径、操作类型 |
| 命令执行 | 运行 `ask` 级别的命令 | 命令内容、参数     |
| 文件删除 | 任何删除操作          | 待删除文件路径     |

### 5.4 `allow_always` 机制

用户可以选择"始终允许"某类操作,AI 会缓存该决策,后续同类操作不再询问。

**实现位置**: `packages/agent-service/src/acp/connection.ts`

**注意事项**:

- `allow_always` 状态保存在当前 Session 内
- Session 结束后重新询问
- 不会跨 Session 持久化

## 六、行为约束执行流程

### 6.1 完整请求处理链路 (实际执行情况)

```
用户发送消息
  ↓
前端 AIChat 组件
  ↓
WebSocket 连接 agent-service
  ↓
AgentManager.getOrCreate() 获取/创建 Agent
  ↓
BackendAgent.sendMessage()
  ↓
┌─ 如果是首次消息:
│   ├─ 注入 System Prompt (L2) ✅ 已生效 — 规则: 怎么做/能做/不能做
│   ├─ 加载 AGENTS.md (L3) ❌ 未注入 — 现状: 当前工作空间有什么
│   └─ 应用权限配置 (L1) ❌ 未注入 — 硬限制: 文件/命令权限
│
└─ AI 处理消息,生成响应
    ↓
    AI 请求执行操作(如编辑文件)
    ↓
    ├─ L1 检查: 权限配置是否允许? ❌ 未生效,跳过
    │
    ├─ L2/L3 检查: AI 自我判断是否符合规则? ✅ L2 生效 (规则), L3 未生效 (现状)
    │   ├─ 符合 → 继续
    │   └─ 不符合 → AI 自主调整策略
    │
    └─ L4 检查: 操作是否需要用户确认? ✅ 已生效
        ├─ 不需要(allow) → 直接执行
        └─ 需要(ask) → 触发 PermissionDialog
            ↓
            用户决策 → 执行/拒绝
```

**关键说明**: 当前 L1 权限配置未生效,AI 的文件操作**不受硬限制**,仅依赖 L2 软约束和 L4 用户确认。这意味着:

- AI 理论上可以修改任何文件(只要有工具权限)
- 实际安全性靠 AI 遵守 System Prompt 的软约束
- 敏感操作仍需用户确认(L4)

### 6.2 权限检查伪代码

```typescript
// agent-service 侧的权限检查逻辑
async function checkPermission(
  operation: Operation,
): Promise<PermissionResult> {
  // L1: 文件系统权限检查（硬限制）
  const l1Result = await checkFileSystemPermission(operation);
  if (l1Result === "deny") {
    return { allowed: false, reason: "权限配置禁止此操作" };
  }

  // L2/L3: AI 自我约束（软约束，无法代码化检查）
  // AI 根据 System Prompt 和 AGENTS.md 自主判断

  // L4: 用户确认检查
  if (l1Result === "ask") {
    return {
      allowed: false,
      requiresUserConfirmation: true,
      operation: operation,
    };
  }

  return { allowed: true };
}
```

## 七、典型场景分析

### 场景 1: AI 修改页面样式

**用户请求**: "把首页标题改成红色"

**系统处理流程**:

1. **L2/L3 决策**: AI 根据"文件修改决策规则"判断需要编辑 `demos/home/index.tsx`
2. **L1 权限检查**: `index.tsx: allow` → 通过
3. **L4 用户确认**: 白名单文件,无需确认 → 直接执行
4. **执行结果**: AI 修改 `className` 中的颜色类

### 场景 2: AI 创建新页面

**用户请求**: "创建一个用户中心页面"

**系统处理流程**:

1. **L2/L3 决策**: AI 根据"创建页面"流程,生成 demoId,准备创建三个文件
2. **L1 权限检查**:

- `index.tsx: allow` → 通过
- `config.schema.json: allow` → 通过
- `.demo.json: allow` → 通过
- `mkdir *': deny` → ❌ **禁止创建目录**

3. **权限冲突**: AI 无法创建目录,但文件权限允许
4. **解决方案**:

- 方案 A: 用户在界面手动创建页面(推荐)
- 方案 B: 修改 L1 配置允许 `mkdir` (不推荐,安全风险)

### 场景 3: AI 尝试删除文件

**用户请求**: "删除这个没用的组件"

**系统处理流程**:

1. **L2/L3 决策**: AI 判断需要删除某个文件
2. **L1 权限检查**: `rm *: deny` → ❌ **直接拒绝**
3. **操作中止**: AI 返回提示"无法执行删除操作,请在界面中操作"

### 场景 4: AI 读取环境变量

**AI 行为**: 尝试读取 `.env` 获取 API Key

**系统处理流程**:

1. **L1 权限检查**: `*.env: deny` → ❌ **直接拒绝**
2. **L2 约束**: System Prompt 未告知 API Key 位置
3. **结果**: AI 无法获取敏感信息

### 场景 5: AI 执行危险命令

**AI 行为**: 尝试运行 `rm -rf node_modules && npm install`

**系统处理流程**:

1. **L1 权限检查**: `rm *: deny` → ❌ **直接拒绝**
2. **即使用户授权**: 硬限制无法被 `allow_always` 覆盖
3. **结果**: 命令被底层拦截,不会执行

## 八、安全性评估

### 8.1 防护层有效性

| 威胁类型          | L1          | L2/L3       | L4          | 综合评级 |
| ----------------- | ----------- | ----------- | ----------- | -------- |
| AI 误删文件       | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 泄露敏感信息   | ✅ 强制阻止 | ⚠️ 依赖遵守 | N/A         | **高**   |
| AI 执行危险命令   | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 修改系统配置   | ✅ 强制阻止 | ✅ 明确禁止 | N/A         | **高**   |
| AI 超出任务范围   | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |
| AI 使用不安全依赖 | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |

### 8.2 潜在风险点

1. **System Prompt 注入攻击**: 用户通过对话诱导 AI 忽略约束

- **缓解措施**: L1 硬限制无法被 Prompt 覆盖

2. **权限配置过于宽松**: 允许过多文件/命令

- **缓解措施**: 默认采用白名单 + 最小权限原则

3. **allow_always 滥用**: 用户习惯性授权

- **缓解措施**: Session 级别隔离,定期重新确认

4. **后端实现差异**: 三个后端的 `buildSystemPrompt()` 可能不同步

- **缓解措施**: 代码审查 + 单元测试验证一致性

### 10.2 常见错误

| 错误现象              | 可能原因             | 解决方案                           |
| --------------------- | -------------------- | ---------------------------------- |
