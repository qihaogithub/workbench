## 一、概述

本系统通过**四层防护机制**约束 AI Agent 的行为,确保 AI 在安全、可控的范围内执行代码生成和文件操作任务。这种分层设计既保证系统安全性,又给予 AI 足够的自主性来完成开发任务。

### 约束架构总览

```
┌──────────────────────────────────────────────────────────────────┐
│ L5: 用户确认层 (原 L4, PermissionDialog)                         │ ← 人工审核 ✅ 已实现
│   敏感操作需用户手动确认                                           │
├──────────────────────────────────────────────────────────────────┤
│ L4: 记忆层 (新增, memory.md)                                      │ ← 会话首条注入 ✅ 已实现
│   用户偏好、关键决策、项目约定（跨会话长期记忆）                    │
├──────────────────────────────────────────────────────────────────┤
│ L3: 动态上下文层 (user message 前缀)                              │ ← 运行时注入 ✅ 已实现
│   工作空间页面列表、项目配置状态（每次 sendMessage 前扫描刷新）     │
├──────────────────────────────────────────────────────────────────┤
│ L2: 系统提示层 (System Prompt, 100% 静态)                         │ ← 软约束 ✅ 已实现
│   AI 怎么做 + 能做什么 + 不能做什么 + 记忆维护规则 + L5 权限确认说明│
├──────────────────────────────────────────────────────────────────┤
│ L1: 文件系统权限层 (pi-tools/permissions.ts)                      │ ← 硬限制 ✅ 已实现
│   路径白/黑名单 + 命令白/黑名单 + beforeToolCall 拦截             │
└──────────────────────────────────────────────────────────────────┘
```

**核心原则**: 外层约束优先于内层,硬限制无法被 AI 绕过,软约束依赖 AI 的自我遵守。

### v3.2 架构变更说明

v3.2 对约束机制进行了重大重构：

1. **单后端统一**：原三后端（claude.ts、codex.ts、gemini.ts）合并为单一 `PiAgentBackend`，System Prompt 不再由后端 `buildSystemPrompt()` 构建
2. **静态/动态拆分**：L2 + L4 组成 100% 静态 System Prompt（LLM API 缓存持续命中），L3 改为 user message 前缀动态注入
3. **L1 迁移**：权限检查从 `opencode.json` 的 `permission` 字段迁移到 `pi-tools/permissions.ts`，由 `beforeToolCall` 钩子和工具实现层直接执行
4. **模板迁移**：L2 模板从 `author-site/src/lib/agent-prompts/demo-generator.template.md` 迁移到 `packages/shared/src/agent-prompts/demo-generator.template.ts`

## 二、L1 - 文件系统权限层 (硬限制)

### 2.1 权限配置文件

**文件**: `packages/agent-service/src/backends/pi-tools/permissions.ts`

v3.2 后，权限检查不再通过 `opencode.json` 的 `permission` 字段，而是由 Pi Agent 的工具层直接执行。权限配置采用 `PermissionConfig` 接口：

```typescript
export interface PermissionConfig {
  allowedPaths: string[];     // 路径白名单（glob 模式）
  deniedPatterns: string[];   // 路径黑名单（glob 模式）
  allowedCommands: string[];  // 命令白名单
  deniedCommands: string[];   // 命令黑名单
}
```

### 2.2 默认权限配置

`DEFAULT_WORKSPACE_PERMISSIONS` 定义了开箱即用的安全策略：

#### 2.2.1 路径权限

| 规则 | 类型 | 说明 |
|:-----|:-----|:-----|
| `**` | ✅ 允许 | 工作空间内通用访问（黑名单优先） |
| `demos/*/index.tsx` | ✅ 允许 | 页面组件代码 |
| `demos/*/config.schema.json` | ✅ 允许 | 页面配置定义 |
| `workspace-tree.json` | ✅ 允许 | 工作空间统一清单（页面元数据 + 文件夹结构） |
| `project.config.schema.json` | ✅ 允许 | 项目级共享配置 |
| `AGENTS.md` | ✅ 允许 | 规则文件 |
| `index.tsx` / `config.schema.json` | ✅ 允许 | 根目录页面文件（兼容单页面模式） |
| `**/*.env` / `**/*.env.*` | ❌ 禁止 | 环境变量文件 |
| `**/.git` / `**/.git/**` | ❌ 禁止 | Git 目录 |
| `**/node_modules` / `**/node_modules/**` | ❌ 禁止 | 依赖目录 |
| `**/packages` / `**/packages/**` | ❌ 禁止 | 工作空间外源码 |
| `**/.opencode` / `**/.opencode/**` | ❌ 禁止 | Agent 配置目录 |
| `**/.workspace.json` / `**/.session.json` | ❌ 禁止 | 系统元数据文件 |

**安全策略**: 黑名单优先于白名单检查，路径必须在 `workingDir` 内（防止目录穿越）。

#### 2.2.2 命令执行权限

| 类型 | 命令 | 说明 |
|:-----|:-----|:-----|
| ✅ 允许 | `npm`, `node`, `npx`, `ls`, `cat`, `head`, `tail`, `grep`, `find`, `wc`, `echo` | 只读命令 + 包管理器 |
| ❌ 禁止 | `rm`, `rmdir`, `mv`, `cp`, `mkdir`, `sudo`, `chmod`, `chown` | 破坏性命令和权限操作 |

**安全策略**: 命令名提取后先检查黑名单再检查白名单，不在任一列表中的命令默认拒绝。

### 2.3 权限执行机制

权限检查在两个层级执行：

**1. `beforeToolCall` 钩子（pi-agent.ts）**:

```typescript
beforeToolCall: async (context) => {
  const toolName = context.toolCall.name;
  if (toolName === 'readFile' || toolName === 'writeFile' || toolName === 'listFiles') {
    const args = context.args as { path?: string };
    if (args.path && !isPathAllowed(args.path, workingDir, permissions)) {
      return { block: true, reason: `Access denied: path "${args.path}" is not allowed` };
    }
  }
  return undefined;
}
```

**2. 工具实现层（file-tools.ts、bash-tool.ts）**:

每个工具在 `execute` 函数内部再次检查权限：

- `readFile` / `writeFile` / `listFiles`：调用 `isPathAllowed()` 检查路径权限
- `bash`：调用 `isCommandAllowed()` 检查命令权限
- 权限检查失败时返回 `isError: true` 的错误结果

### 2.4 权限配置传递

权限配置通过 `AgentConfig.permissions` 字段传递，未提供时使用 `DEFAULT_WORKSPACE_PERMISSIONS` 默认值：

```typescript
const permissions = config.permissions ?? DEFAULT_WORKSPACE_PERMISSIONS;
```

### 2.5 配置修改指南

如需调整权限规则,修改 `permissions.ts` 中的 `DEFAULT_WORKSPACE_PERMISSIONS`：

```typescript
// 示例：允许 AI 编辑 TypeScript 类型定义文件
allowedPaths: [
  '**',
  'demos/*/index.tsx',
  // ... 现有规则 ...
  '*.d.ts',  // ← 新增：允许编辑类型定义
],

// 示例：允许 AI 执行 npm install
allowedCommands: [
  'npm', 'node', 'npx', 'ls', 'cat', 'head', 'tail', 'grep', 'find', 'wc', 'echo',
  // npm install 已通过 'npm' 基础命令允许
],
```

**⚠️ 注意事项**:

- 黑名单优先于白名单检查
- 路径必须在 `workingDir` 内（目录穿越保护）
- 修改后需重启 agent-service 生效

## 三、L2 - 系统提示层 (软约束)

### 3.1 L2 的定位

L2 是 AI 的**任务手册**，告诉 AI：

- **怎么做**：任务执行流程、代码质量标准、文件修改决策规则
- **能做什么**：允许的操作范围
- **不能做什么**：禁止行为清单

### 3.2 v3.2 静态/动态拆分架构

v3.2 将 System Prompt 拆分为**静态部分**和**动态部分**，以优化 LLM API 缓存命中率：

| 部分 | 注入位置 | 更新频率 | 缓存表现 |
|:-----|:-----|:-----|:-----|
| **L2 静态规则** + L4 说明 | `systemPrompt` 字段 | 应用启动后不变 | 100% 缓存命中 |
| **L3 动态上下文** | user message 前缀 | 每次 sendMessage 前重新扫描 | 随工作空间状态变化 |

### 3.3 L2 模板：demo-generator.template.ts

**文件**: `packages/shared/src/agent-prompts/demo-generator.template.ts`

导出 `DEMO_GENERATOR_TEMPLATE` 常量（TS 字符串），包含完整的 AI 行为规则：

- **页面内容编辑**：自然语言匹配页面名称，自主判断修改目标
- **页面管理操作**：创建/重命名/删除页面的完整流程
- **项目级配置管理**：新增/删除/修改项目配置的规则
- **代码质量标准**：TypeScript、DemoProps 接口、Tailwind CSS 等
- **React 版本约束**：React 18.3.1 兼容、依赖白名单
- **禁止行为清单**：访问工作空间外文件、修改系统文件、重复定义字段等
- **文件修改决策规则**：样式/配置/组件/项目配置/页面元数据的判断规则
- **参考文件引用**：「生成 schema 前必须先读取 `references/config-system.md`」

**v3.2 关键变化**: 模板不再包含 `{{PROJECT_NAME}}`、`{{PAGE_LIST}}` 等占位符。动态上下文已移至 L3（user message 前缀）。

### 3.4 L5 权限确认说明（静态注入）

**文件**: `packages/author-site/src/lib/agent/system-prompt.ts`

`USER_CONFIRMATION_NOTICE` 常量（原 `L4_NOTICE`）拼接到 L2 模板末尾，告知 AI 哪些操作需要用户确认：

```typescript
const USER_CONFIRMATION_NOTICE = `## 权限确认

以下操作需要用户确认（系统会自动发送确认请求给用户）：
- 创建新页面目录
- 删除页面文件
- 修改项目级共享配置（project.config.schema.json）

收到 permission_request 事件后等待用户授权，不要直接继续操作。`;
```

### 3.5 记忆维护规则（L2 静态注入）

**文件**: `packages/author-site/src/lib/agent/system-prompt.ts`

`MEMORY_MAINTENANCE_RULES` 常量（约 60 行）注入到 L2 静态 System Prompt 中，告知 AI 如何维护 `memory.md`：

- **何时读取**：每次对话开始时读取 `memory.md`（实际通过 L4 注入实现）
- **何时更新**：用户明确要求记住、表达个人偏好、做出关键决策时
- **不记录什么**：一次性操作、讨论过程、代码可见信息、敏感信息、系统已有规范
- **操作规范**：先读后写、增量编辑、保留用户手写内容、去重检查、更新日期、极简表达、控制 1500 字以内
- **更新频率**：同一条信息只更新一次，不是每轮对话都要更新

### 3.6 构建流程

**文件**: `packages/author-site/src/lib/agent/system-prompt.ts`

```typescript
// 静态部分：应用启动时缓存（100% 不变）
export function buildStaticSystemPrompt(): string {
  return [DEMO_GENERATOR_TEMPLATE, MEMORY_MAINTENANCE_RULES, USER_CONFIRMATION_NOTICE].join('\n\n---\n\n');
}

// 动态部分：每次 sendMessage 前渲染（L3 上下文）
export function buildDynamicContextPrefix(context: SystemPromptContext): string {
  return render(WORKSPACE_STATUS_TEMPLATE, { /* 变量 */ });
}

// 记忆前缀（L4 注入时使用）
export function buildMemoryPrefix(content: string): string {
  return `\n\n## 项目记忆（memory.md）\n\n${content}\n\n[系统：当前 memory.md 共 X 字]`;
}
```

**注入时机**：

1. **WebSocket 路径**（stream-service.ts）：`sendMessage()` 将静态 prompt 放入 `systemPrompt` 字段，L3 前缀拼接到 user content 头部；首条消息额外追加 L4 记忆内容
2. **HTTP 路径**（api/ai/chat/route.ts）：同样将静态 prompt 放入 `options.systemPrompt`，L3 前缀拼接到 `finalContent`；首条消息额外追加 L4 记忆内容

## 四、L3 - 动态上下文层 (user message 前缀) ✅ 已实现(v3.2)

### 4.1 v3.2 架构变化

v3.2 将原 L3（references/ 参考文件 + 工作空间状态占位符）重构为**动态上下文前缀**：

- **旧方案**：工作空间状态（页面列表、配置状态）通过 `{{PAGE_LIST}}` 占位符注入到 System Prompt
- **新方案**：工作空间状态作为 user message 前缀动态注入，System Prompt 保持 100% 静态
- **收益**：LLM API 缓存持续命中，每次 sendMessage 只需重新扫描工作空间状态

### 4.2 动态上下文内容

**模板**: `packages/author-site/src/lib/agent-prompts/workspace-status.template.ts`

```typescript
export const WORKSPACE_STATUS_TEMPLATE = `当前工作空间中的页面（系统自动扫描）：

{{PAGE_LIST}}
`;
```

**扫描器**: `packages/author-site/src/lib/agent/scan-workspace.ts` 的 `scanWorkspaceContext()`

扫描 `demos/` 目录，读取 `workspace-tree.json` 获取页面名称、顺序等元数据，也兜底扫描目录结构。生成 `SystemPromptContext`：

```typescript
export interface SystemPromptContext {
  projectName: string;
  projectConfigStatus: '已设置' | '未设置';
  pageCount: number;
  pageList: string;      // 格式化的页面列表
  workspacePath: string;
}
```

### 4.3 注入流程

```
用户发送消息
  ↓
stream-service.ts / api/ai/chat/route.ts
  ↓
fetchDynamicContextPrefix(workingDir) 或 scanWorkspaceContext(workingDir)
  ↓
buildDynamicContextPrefix(context) → 渲染 L3 前缀
  ↓
finalContent = `${L3前缀}${用户消息}`
  ↓
发送到 agent-service（systemPrompt 字段仅含 L2 + L4 静态内容）
```

### 4.4 references/ 参考文件机制

`references/config-system.md` 配置系统参考手册仍然通过 `injectReferences()` 注入到工作空间的 `references/` 目录：

- **注入时机**: Workspace 创建时（`workspace-manager.ts` 的 `injectReferences()`）
- **文件树隐藏**: `workspace-file-utils.ts` 的 `HIDDEN_ENTRIES` 包含 `"references"`
- **AI 查阅方式**: L2 模板中包含引用指令「生成 schema 前必须先读取 `references/config-system.md`」
- **与 L3 的关系**: references/ 是 AI 通过工具主动查阅的静态参考文件，L3 是每次 sendMessage 前自动注入的动态上下文

### 4.5 与其他层的协同

| 层级 | 内容 | 生效方式 | 覆盖范围 | 实现状态 |
|:---|:---|:---|:---|:---|
| L1 | 权限配置 | 强制执行（beforeToolCall + 工具层） | 文件/命令级别 | ✅ 已实现 |
| L2 | System Prompt | AI 自我约束（100% 静态） | 任务/行为级别 | ✅ 已实现 |
| L3 | 动态上下文 + references/ | user message 前缀 + AI 主动查阅 | 工作空间状态 + 知识/规范级别 | ✅ 已实现 |

**协同示例**: AI 修改某个页面的配置 schema

- L1 检查路径权限：`demos/*/config.schema.json` 在白名单中 → ✅ 通过
- L2 规则：「配置项修改 → 编辑 `demos/{demoId}/config.schema.json`」→ AI 自我约束
- L3 上下文：user message 前缀包含当前页面列表，AI 知道目标页面是否存在
- L3 参考：AI 读取 `references/config-system.md` 查阅控件映射规则
- L4 确认：如果是修改 `project.config.schema.json`，需要用户确认

## 五、L4 - 记忆层 (跨会话长期记忆) ✅ 已实现(v3.3)

### 5.1 设计目标

`memory.md` 是位于**项目根目录**的记忆文件，用于记录：

- **用户偏好**：写代码前先说明思路、拿不准时先问、改配置时要解释
- **关键决策**：首页用轮播 banner、详情页不加下拉刷新
- **项目约定**：自然语言描述，非技术人员看得懂、用户可直接编辑

### 5.2 与 AGENTS.md 的区别

| | `AGENTS.md`（系统文件） | `memory.md`（用户记忆文件） |
|---|---|---|
| **用途** | 告诉 AI 项目技术架构和开发规范 | 记录用户偏好、项目约定、关键决策 |
| **读者** | AI（技术人员） | 用户 + AI |
| **语言** | 技术术语、代码结构 | 自然语言、日常表达 |
| **维护者** | 开发者手动编辑 | AI 自动维护、用户可编辑 |
| **内容风格** | 精确、结构化 | 叙事化、易理解 |

> **重要**：`memory.md` 绝不替代或修改 `AGENTS.md`。两者共存，互不干扰。

### 5.3 注入策略

**注入时机**：仅在**会话首条消息**时，读取 `memory.md` 并注入到消息内容之前。

```
首条消息：${L3前缀} + ${L4记忆内容} + ${用户消息}
后续消息：${L3前缀} + ${用户消息}  （不重复注入 L4）
```

**字数提示**：系统读取 `memory.md` 后，自动在末尾附加当前字数统计：

```
[系统：当前 memory.md 共 847 字]
```

这样 AI 无需手动计数，看到接近 1500 就知道该压缩了。该行仅注入时不写入文件。

**为什么只注入首条消息**：

- L3 每次刷新是因为页面列表可能变了；`memory.md` 在会话中几乎不会变
- 重复注入浪费 token，且可能让 AI 过度关注记忆内容而非用户当前消息

| `api/ai/chat/route.ts` | HTTP 路径：在 L3 上下文后追加 `memory.md` 内容注入 |
| `api/agent/workspace-context/route.ts` | API 响应新增 `memoryContent` 字段，供 streaming 客户端获取 |
| `stream-service.ts` | 新增 `hasInjectedMemory` 标记，首条消息注入 L4 记忆（后续消息跳过）；`close()` 时重置标记；函数 `fetchDynamicContextPrefix` 重命名为 `fetchContextPrefix`，返回 `{ l3, memory }` |

### 5.5 AI 维护规则（L2 层注入）

将记忆维护规则注入到 System Prompt 的 L2 静态部分，告知 AI：

**何时更新**：

| 触发条件 | 用户典型发言 | 应更新的章节 |
|---|---|---|
| **用户明确要求记住** | "请记住这个"、"以后都这样做"、"把这个记下来" | 按内容放入对应章节 |
| **表达个人偏好** | "我不喜欢……"、"我更习惯……"、"遇到这种情况先问我" | 我的偏好 |
| **做出关键决策** | "那就用……吧"、"不用……了，换成……"、"我们决定……" | 关键决策 |

**不应该记录什么**：

- 一次性操作（如"帮我调大这个按钮"）—— 不是长期记忆
- 讨论过程中的试探和犹豫 —— 只记最终决定
- 可以从代码里直接看到的信息
- 密码、密钥、Token 等敏感信息
- **系统提示词中已有的编码规范** —— memory.md 只记录对话中新产生的记忆

**操作规范**：

1. **先读后写**：更新前必须先用 readFile 读取当前 `memory.md` 的完整内容
2. **增量编辑**：只修改需要更新的章节，其他章节保持原样不动
3. **保留用户手写内容**：如果某个章节的内容看起来是用户手动写的，不要覆盖，只追加新内容
4. **去重检查**：新增内容前先确认是否已有类似信息，避免重复
5. **更新日期**：每次修改后在顶部更新「最后更新」日期
6. **极简表达**：每条决策一句话说清，用「——」分隔决定和原因。偏好每条不超过 15 字
7. **控制总长度**：建议控制在 1500 字以内。系统自动附字数提示，接近上限时 AI 压缩后再写入

### 7.1 完整请求处理链路 (v3.3 架构)

```
用户发送消息
  ↓
前端 AIChat 组件
  ↓
WebSocket 连接 agent-service（或 HTTP fallback）
  ↓
┌─ 发送前（author-site 端）:
│   ├─ 构建静态 System Prompt (L2 + L4维护规则 + L5) ✅ 100% 静态，模块级缓存
│   ├─ 扫描工作空间 → 渲染 L3 上下文前缀 ✅ 每次 sendMessage 前重新扫描
│   ├─ 首条消息：读取 memory.md → 注入 L4 记忆内容 ✅ 仅首条消息，后续跳过
│   └─ finalContent = `${L3前缀}${L4记忆(仅首条)}${用户消息}`
│
├─ 发送时:
│   ├─ systemPrompt: STATIC_SYSTEM_PROMPT (L2 + L4维护规则 + L5)
│   └─ content: finalContent (L3 + L4记忆(仅首条) + 用户消息)
│
└─ agent-service 收到消息:
    ↓
    PiAgentBackend.sendMessage()
    ↓
    Agent 处理消息，调用工具
    ↓
    ├─ L1 检查: beforeToolCall 拦截 + 工具层权限检查
    │   ├─ isPathAllowed() → 路径权限
    │   └─ isCommandAllowed() → 命令权限
    │
    ├─ L2 检查: AI 根据 System Prompt 判断是否符合规则
    │   └─ 符合 → 继续；不符合 → AI 自主调整策略
    │
    ├─ L4 检查: AI 参考 memory.md 中的用户偏好和决策
    │   └─ 优先遵循已记录的约定
    │
    └─ L5 检查: 操作是否需要用户确认
```

## 九、安全性评估

### 9.1 防护层有效性

| 威胁类型          | L1          | L2/L3       | L4          | 综合评级 |
| ----------------- | ----------- | ----------- | ----------- | -------- |
| AI 误删文件       | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 泄露敏感信息   | ✅ 强制阻止 | ⚠️ 依赖遵守 | N/A         | **高**   |
| AI 执行危险命令   | ✅ 强制阻止 | ⚠️ 依赖遵守 | ✅ 用户确认 | **高**   |
| AI 修改系统配置   | ✅ 强制阻止 | ✅ 明确禁止 | N/A         | **高**   |
| AI 超出任务范围   | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |
| AI 使用不安全依赖 | N/A         | ⚠️ 依赖遵守 | ✅ 用户确认 | **中**   |

### 9.2 潜在风险点

1. **System Prompt 注入攻击**: 用户通过对话诱导 AI 忽略约束

- **缓解措施**: L1 硬限制无法被 Prompt 覆盖

2. **权限配置过于宽松**: 允许过多文件/命令

- **缓解措施**: 默认采用白名单 + 最小权限原则

3. **权限响应取消**: 用户习惯性取消请求

- **缓解措施**: L1 硬限制始终生效，取消不会导致安全风险

4. **单后端故障**: Pi Agent 后端异常可能导致全部功能不可用

- **缓解措施**: 错误处理 + 日志记录，后端重启恢复

5. **memory.md 过度膨胀**: AI 过度记录导致文件变长

- **缓解措施**: 1500 字上限建议 + 系统自动附字数提示 + 接近上限时 AI 压缩后再写入

6. **AI 覆盖用户手写内容**: AI 更新时覆盖用户手动修改的记忆

- **缓解措施**: 操作规范明确要求「保留用户手写内容，只追加新内容」

- 命令白名单：`npm`、`node`、`npx`、`ls`、`cat`、`head`、`tail`、`grep`、`find`、`wc`、`echo`

✅ **L3 - 动态上下文层**

- `workspace-status.template.ts` 定义 L3 模板（页面列表）
- `scan-workspace.ts` 的 `scanWorkspaceContext()` 扫描 `demos/` 目录

- `references/config-system.md` 仍通过 `injectReferences()` 注入到工作空间
- `references/` 在文件树中隐藏（`HIDDEN_ENTRIES`）

✅ **L4 - 记忆层（跨会话长期记忆）**
