# AI 对话模块 - 需求文档

> 版本：v2.15
> 更新日期：2026-07-21
> 变更说明：统一图片与文件附件入口为加号菜单，并统一双端点阵等待动画

---

## 一、模块定位

AI 对话模块是用户与 AI Agent 交互的核心通道，负责将用户的自然语言指令转化为代码修改操作，并实时展示 AI 的思考和执行过程。

---

## 二、功能描述

### 2.1 AI 对话功能

#### 2.1.1 发送对话消息

**用户视角**：

- 用户可在对话区输入自然语言指令，请求 AI 修改代码
- AI 响应应通过流式传输实时展示

**交互效果**：

- 支持流式响应，实时显示 AI 输出
- AI 可请求执行工具操作（如修改代码）
- AI 只有在修改已进入当前工作空间、同步到编辑状态并可被当前预览读取后，才能向用户报告修改成功
- 已写入的文件不得被同一编辑页或其他协作者持有的旧状态再次覆盖；同步失败时应明确报告失败，不能误报完成
- 敏感工具调用需用户确认

#### 2.1.2 对话历史

**用户视角**：

- 用户可查看当前会话的完整对话历史
- 对话历史应包含用户消息和 AI 响应

#### 2.1.3 系统发起任务

**用户视角**：

- 当系统自动发现可交给 AI 处理的问题时，对话区应明确展示这是系统发起的任务，而不是用户手动输入的消息
- 用户应能看到任务正在处理、处理完成或处理失败
- 用户可按需通过弹窗查看任务详情，但默认不被技术错误长文本打断

**交互效果**：

- 系统任务使用区别于用户气泡和 AI 回复的展示样式
- 自动修复类任务应展示轻量说明和状态，卡片不应被长错误详情撑开
- 查看详情应打开独立弹窗展示技术详情，并保留卡片在对话流中的紧凑布局
- 失败后应提供重新触发修复的入口
- 系统任务不应改写用户输入历史，也不应让用户误以为自己发送了该消息

#### 2.1.4 新建对话

**用户视角**：

- 用户可在编辑页面中新建对话，与 AI 开始全新交互
- 新建对话时，代码编辑器和预览区保持当前状态不变
- 新对话继承当前工作空间的文件状态
- 用户在当前编辑页手动选择过指定模型后，新建对话应继续使用该模型，不应回到平台默认模型

**交互效果**：

- AI 对话区清空，开始新的对话
- 代码/Schema/编辑器内容不重置
- 新对话与当前对话共享同一工作空间
- 模型选择器保持用户最近一次手动选择的模型和思考深度；如果该模型在新对话中不可用，则使用新对话可用模型列表的默认选择
- 用户在同一编辑上下文内切换 tab、切换预览视图或 AI 面板重挂载后，模型选择仍应保持

#### 2.1.5 切换对话

**用户视角**：

- 用户可在对话历史列表中切换到之前的对话
- 切换对话时，代码编辑器和预览区保持当前状态不变
- 切换仅影响 AI 的上下文，不影响工作空间文件

**交互效果**：

- AI 对话区显示目标对话的历史消息
- 代码/Schema/编辑器内容不重置
- AI 恢复该对话的上下文
- 同一对话中连续追问时，AI 应能引用最近对话内容；短对话不应因为面板重挂载、流式完成或 HTTP 降级而表现为没有上下文

#### 2.1.6 跨会话记忆

**用户视角**：

- AI 能记住之前对话中了解到的项目信息、用户偏好、架构决策等上下文
- 切换对话后 AI 仍能通过阅读记忆文件了解上下文
- 用户可直接阅读和编辑记忆文件（`memory.md`）
- AI 在对话中发现重要信息时自动更新记忆文件

**交互效果**：

- 每次新对话开始时，AI 自动读取项目根目录的 `memory.md`
- 用户表达个人偏好（如"我不喜欢……"）时，AI 自动记录
- 用户做出关键决策（如"那就用……吧"）时，AI 自动记录
- 用户可手动修改 `memory.md`，AI 会保留用户手写内容

**业务规则**：

- 记忆文件位于项目根目录：`{项目路径}/memory.md`
- 记忆文件已加入 `.gitignore`，不提交到版本控制
- 建议控制在 1500 字以内，接近上限时 AI 自动压缩
- AI 不记录：一次性操作、讨论过程、代码可见信息、敏感信息

#### 2.1.7 对话历史列表

**用户视角**：

- 用户可查看当前项目下的所有对话历史
- 对话列表显示对话创建时间和消息数量
- 当前活跃对话有高亮标识

**交互效果**：

- 过期对话标记为不可选
- 可删除不需要的对话

#### 2.1.8 用户自定义大模型 API

**用户视角**：

- 登录用户可在账号设置中配置自己的大模型 API
- 用户可填写供应商名称、OpenAI 兼容 baseURL、API Key、模型列表和默认模型
- 保存后，用户后续与 AI 对话时优先使用自己的模型配置
- 用户可清空个人配置，恢复使用平台管理员提供的默认模型配置

**交互效果**：

- API Key 输入框应以密码形式展示
- 已保存 API Key 时，再次打开设置不回显明文，只提示已保存
- 留空保存 API Key 时，应保留原有 Key
- 模型列表支持一行一个模型，默认模型必须来自列表内
- 配置不完整或格式不合法时，应给出明确错误提示

**业务规则**：

- 个人模型配置只影响当前登录用户
- 个人配置优先于平台全局模型配置
- 已有活跃对话应尽量同步最新个人配置；无法同步时不影响配置保存
- 平台仍可通过管理后台配置全局可用供应商和默认模型

#### 2.1.9 图片内容预描述

**用户视角**：

- 用户可在 AI 对话中上传图片，并继续使用自己选择的文本模型提问
- 当当前模型不支持图片输入时，系统可先将图片内容转换为文字描述，再交给当前文本模型回答
- 用户无需手动切换模型即可询问图片内容、截图问题、UI 参考图或设计稿信息

**交互效果**：

- 已配置识图能力时，用户发送图片后应收到与图片内容相关的正常回答
- 未配置识图能力时，系统应明确提示当前模型不支持图片处理，并引导联系管理员配置识图模型
- 多张图片应按顺序转换为可理解的文字上下文

**业务规则**：

- 图片预描述能力仅在当前模型不支持图片输入时触发
- 当前模型支持图片输入时，系统应保留原有图片直传体验
- 识图能力由平台侧配置，普通用户无需理解背后的模型拆分
- 图片内容描述只作为当前对话上下文使用，不应成为公开展示内容
- 当 AI 需要把用户上传的图片用于页面代码或配置时，图片应保存为当前项目的本地资源，而不是保存到外部 OSS 或全局图床
- 保存后的图片应跟随项目工作区、预览和发布流程流转，用户不需要单独管理外部图片地址

#### 2.1.10 需求确认卡片

**用户视角**：

- 当用户需求存在多个合理实现方向时，AI 可在对话中发起单选题卡片。
- 用户可直接选择一个预设选项，也可在允许时选择“其他”并输入自己的答案。
- 用户选择后，AI 应基于该选择继续处理当前任务。

**交互效果**：

- 确认卡片以内联形式出现在 AI 回复中，不应打断对话主流程。
- 卡片应展示问题、必要说明、选项和取消入口。
- 用户提交后，卡片应保留在对话历史中，并显示已选内容。

**业务规则**：

- 单选卡片只用于需求澄清，不用于敏感操作授权。
- 取消、超时或当前环境不支持卡片时，AI 应用普通文字继续说明或追问。

#### 2.1.11 执行计划审批卡片

**用户视角**：

- 当 Agent 判断任务复杂、影响范围较大或需要用户确认执行方向时，可先提交执行计划。
- 用户可在对话中的计划卡片上直接批准，也可先查看并编辑计划后再批准。
- 用户批准前，Agent 不应继续执行该计划内的代码修改、删除、验证或子 Agent 委派动作。

**交互效果**：

- 计划审批卡片以内联形式出现在对话中，展示“执行计划待确认”的状态。
- 卡片同时提供“查看计划”和“批准”入口；“批准”用于按当前计划继续执行，“查看计划”用于打开可编辑计划内容。
- 计划弹窗中仍提供取消和批准入口，取消后当前任务停止执行。
- 计划审批卡片出现后，输入区不应继续显示“停止生成”的按钮态；系统应等待用户审批，同时允许用户改问新问题。

**业务规则**：

- Agent 应自主判断是否需要计划审批；简单、低风险、目标明确的小任务不应频繁打断用户。
- 复杂、多步骤、跨文件/跨页面、需要排查后实施、需要子 Agent 或涉及关键产品决策的任务，才应提交审批计划。
- 计划审批不替代敏感操作授权；删除等高风险动作仍需使用对应确认流程。
- 用户在计划审批等待期间发送新问题时，应取消当前待审批计划并开始处理新问题，避免旧计划阻塞正常对话。

#### 2.1.12 回复期间消息排队

**用户视角**：

- AI 正在回复时，用户仍可继续输入下一条消息或添加图片、文件。
- 用户在回复期间提交的新消息应进入等待队列，不应打断当前回复。
- 当前回复结束、失败或被用户停止后，系统应自动按提交顺序发送下一条等待消息。
- 用户应能取消尚未发送的等待消息。

**交互效果**：

- 等待消息应吸附展示在输入框上方，并标记“等待发送”。
- 等待消息不应混入对话历史滚动流，避免让用户误以为 AI 已经开始处理下一条。
- 输入区有内容时，发送按钮优先提交当前输入；输入区为空且 AI 正在回复时，按钮才用于停止当前回复。

**业务规则**：

- 第一版只支持同一会话内串行发送，不并发执行多条 Agent 请求。
- 排队消息不替代权限确认、需求确认卡片、计划审批卡片或其他专用响应。
- 切换会话或组件卸载时，旧会话的等待队列不应发送到新会话。

#### 2.1.13 本地文件附件读取

**用户视角**：

- 用户可在 AI 对话中上传本地文件，让 AI 读取文件内容后回答问题或辅助修改页面。
- 图片继续保留原有图片理解体验；非图片文件作为只读参考资料进入当前 AI 对话。
- 图片和非图片文件统一从输入框的加号按钮添加；点击后通过气泡菜单选择“添加图片”或“添加附件”，不并列展示两个上传按钮。
- AI 开始处理但尚未输出正文时，创作端与使用端统一展示 5×5 点阵等待动画。
- 用户可上传文本、Markdown、JSON、CSV、常见代码文件、PDF 和 DOCX 文件。

**交互效果**：

- 已添加的文件应在输入区、用户消息气泡和等待队列中展示文件名、大小和类型提示。
- 对话历史和导出记录应保留文件附件元数据，至少包含文件名、大小和只读附件 ID，便于确认本轮消息确实携带了文件。
- 文件上传失败、类型不支持或超过大小限制时，应提示用户，并且不发送不完整消息。
- AI 回复时应能按需读取用户上传文件的文本内容；扫描版 PDF、图片型 PDF 或无法提取文字的文件应明确无法读取正文。

**业务规则**：

- 文件附件只作为当前 AI 会话的只读上下文，不自动写入项目工作区，不参与项目保存、发布或版本历史。
- 文件附件不替代项目素材管理；当用户希望页面引用图片或资源时，仍按图片本地资源保存规则处理。
- 第一版不支持 XLSX；表格类内容建议先使用 CSV。
- 会话切换或组件卸载时，未发送的文件附件队列不应转移到其他会话。

## 三、业务规则与约束

### 3.1 接口规范

| 方法 | 路径         | 说明         |
| :--- | :----------- | :----------- |
| POST | /api/ai/chat | 发送对话消息 |

### 3.2 安全规则

| 规则         | 说明                         |
| :----------- | :--------------------------- |
| 敏感操作确认 | 文件删除等敏感操作需用户确认 |
| 超时处理     | AI 响应超时应提示用户        |

### 3.3 对话切换规则

| 规则         | 说明                                       |
| :----------- | :----------------------------------------- |
| 切换不改代码 | 切换对话时，代码/Schema/编辑器内容保持不变 |
| 新建不改代码 | 新建对话时，代码/Schema/编辑器内容保持不变 |
| 共享工作空间 | 同一项目的多个对话共享同一个工作空间       |
| 保留模型偏好 | 新建对话时保留当前编辑页最近一次手动选择的模型；不可用时回退到新会话默认模型 |
| 友好错误提示 | 连接失败、超时、鉴权/配额、服务异常、请求取消和未知错误应显示普通用户可理解的中文提示 |
| 同轮上下文保持 | 同一对话的最近历史应稳定展示，并随下一轮请求作为上下文提供给 AI |
