## 一、背景与问题

### 1.1 问题描述

当 AI 通过对话修改 schema 时，需要将新的 schema 默认值与用户已修改的配置值进行合并。原有实现存在以下问题：

1. **直接清空配置**：代码变更时将 `configData` 清空为 `{}`，丢失用户已修改的值
2. **简单覆盖**：`{ ...prev, ...newDefaults }` 会导致新默认值覆盖用户值
3. **缺乏类型检查**：字段类型变化时仍使用旧值，可能导致运行时错误

### 1.2 设计目标

- **保留用户值**：用户已修改且类型未变的字段，保留其值
- **应用新默认值**：新增字段使用 schema 默认值
- **移除已删除字段**：schema 中不存在的字段从 configData 中移除
- **类型安全**：字段类型变化时使用新默认值

## 二、合并策略设计

### 2.1 核心函数

```typescript
function mergeSchemaDefaults(
  existingConfig: Record<string, unknown>,
  newSchemaStr: string,
): Record<string, unknown>;
```

**输入**：

- `existingConfig`：用户当前配置值
- `newSchemaStr`：新的 schema JSON 字符串

**输出**：

- 合并后的配置对象

### 2.2 合并规则

| 场景                               | 处理方式            |
| :--------------------------------- | :------------------ |
| 新增字段（不在 existingConfig 中） | 使用 schema default |
| 删除字段（不在 newSchema 中）      | 从 configData 移除  |
| 仍存在且用户修改过的字段           | 保留用户值          |
| 字段类型变化                       | 使用新 default      |
| 用户值为 null/undefined            | 使用新 default      |

### 2.3 类型兼容性检查

```typescript
function isTypeCompatible(oldValue: unknown, newType: string): boolean {
  if (oldValue === null || oldValue === undefined) return false;

  const actualType = typeof oldValue;

  switch (newType) {
    case "string":
      return actualType === "string";
    case "number":
      return actualType === "number";
    case "boolean":
      return actualType === "boolean";
    case "array":
      return Array.isArray(oldValue);
    case "object":
      return actualType === "object" && !Array.isArray(oldValue);
    default:
      return true; // 未知类型默认兼容
  }
}
```

### 2.4 实现逻辑

```typescript
export function mergeSchemaDefaults(
  existingConfig: Record<string, unknown>,
  newSchemaStr: string,
): Record<string, unknown> {
  try {
    const schema = JSON.parse(newSchemaStr);
    const properties = schema.properties || {};
    const result: Record<string, unknown> = {};

    for (const [key, propSchema] of Object.entries(properties)) {
      const defaultValue = (propSchema as any).default;
      const newType = (propSchema as any).type || "string";
      const existingValue = existingConfig[key];

      if (existingValue === undefined || existingValue === null) {
        // 场景 1：新增字段或空值 → 使用 default
        result[key] = defaultValue;
      } else if (!isTypeCompatible(existingValue, newType)) {
        // 场景 4：类型变化 → 使用 new default
        result[key] = defaultValue;
      } else {
        // 场景 3：仍存在且类型兼容 → 保留用户值
        result[key] = existingValue;
      }
    }

    // 处理 __order（展示元数据）
    if (schema.$demo?.orderable) {
      result.__order = schema.$demo.orderable;
    } else {
      result.__order = Object.keys(properties);
    }

    return result;
  } catch {
    // schema 解析失败 → 返回空配置
    return {};
  }
}
```

## 三、测试覆盖

### 3.1 测试用例

| 测试场景                        | 预期结果                                 |
| :------------------------------ | :--------------------------------------- |
| 空配置 + 新 schema              | 全部使用 schema default                  |
| 已有值类型匹配                  | 保留用户值                               |
| 新增字段                        | 使用 schema default                      |
| 删除字段                        | 从结果中移除                             |
| 字段类型变化（string → number） | 使用新 default                           |
| \_\_order 处理                  | 从 $demo.orderable 或 property keys 生成 |
| 无效 JSON schema                | 返回空对象                               |
| 无 properties 的 schema         | 返回空对象                               |
| null/undefined 值               | 使用 default                             |
| 数组类型字段                    | 保留用户数组值                           |

### 3.2 关键测试用例

```typescript
test("should preserve existing values that match schema type", () => {
  const existing = { title: "My Title", count: 42 };
  const schema = JSON.stringify({
    properties: {
      title: { type: "string", default: "Default Title" },
      count: { type: "number", default: 0 },
    },
  });

  const result = mergeSchemaDefaults(existing, schema);

  expect(result.title).toBe("My Title"); // 保留用户值
  expect(result.count).toBe(42); // 保留用户值
});

test("should use new default when field type changes", () => {
  const existing = { count: "not a number" };
  const schema = JSON.stringify({
    properties: {
      count: { type: "number", default: 0 },
    },
  });

  const result = mergeSchemaDefaults(existing, schema);

  expect(result.count).toBe(0); // 类型不匹配，使用 default
});
```

## 四、集成方式

### 4.1 在 applyDemoSnapshot 中使用

```typescript
const applyDemoSnapshot = useCallback(
  (params: { code?: string; schema?: string; source: string }) => {
    const { code: newCode, schema: newSchema, source } = params;

    // ... 更新 code ...

    if (newSchema !== undefined) {
      setSchema(newSchema);

      // 合并 config defaults
      const defaults = mergeSchemaDefaults(
        configDataMap[activeDemoId] ?? {},
        newSchema,
      );
      setConfigDataMap((prev) => ({
        ...prev,
        [activeDemoId]: defaults,
      }));
    }

    // ... 递增 snapshotVersion ...
  },
  [code, schema, sessionId],
);
```

const handleEditorChange = useCallback((value: string) => {

## 五、效果与收益

### 5.1 解决的问题

| 问题           | 解决方式                       |
| :------------- | :----------------------------- |
| 用户配置丢失   | 保留类型兼容的用户值           |
| 新字段无默认值 | 使用 schema default            |
| 已删除字段残留 | 按新 schema properties 过滤    |
| 类型不匹配错误 | 类型检查，不匹配时使用 default |

### 5.2 用户体验提升

- **配置不丢失**：AI 修改 schema 后，用户已调整的配置值仍然保留
- **智能合并**：只更新新增或类型变化的字段，减少用户重复操作
- **类型安全**：避免类型不匹配导致的运行时错误

### 5.3 性能影响

- **纯函数**：无副作用，易于测试和缓存
- **时间复杂度**：O(n)，n 为 schema properties 数量
- **空间复杂度**：O(n)，创建新的配置对象
