# 浏览器扩展截图增强远期方案

> 更新日期：2026-06-29  
> 适用模块：创作端配置与预览、画布模式、截图增强能力

## 1. 方案定位

浏览器扩展截图增强是远期可选能力，不替代现有 screenshot-service，也不作为普通 Web 用户的默认前置条件。

当前主链路仍应优先按照 [截图服务性能优化方案](./09_截图服务性能优化方案.md) 提升服务端截图速度。扩展方案只面向重度创作用户、内部环境或需要更强本地浏览器能力的场景，用于把“当前浏览器已经渲染好的页面”更快固化为截图。

普通网页不能直接调用浏览器内部截图能力。扩展的意义是经过用户安装和授权后，获得浏览器允许的增强权限，再把截图结果安全地交回创作端。

## 2. 用户体验目标

理想体验是：用户仍然打开 Web 创作端，不需要安装桌面客户端。安装扩展并授权后，画布截图加速自动生效。

用户感知流程：

1. 用户正常打开创作端并编辑页面。
2. 实时 iframe 先展示当前代码和配置的真实效果。
3. 页面加载稳定后，扩展从当前浏览器上下文导出该页面或目标区域截图。
4. 创作端把截图登记为当前内容版本的快照。
5. 用户滚动画布或查看多页面时，页面更快切到轻量截图态。
6. 如果扩展不可用，系统自动回到 screenshot-service 或实时 iframe，不阻断编辑。

扩展不应让用户频繁点授权弹窗，也不应把截图能力做成用户每次手动触发的操作。它的价值在于后台增强，而不是增加新的工作步骤。

## 3. 与系统截图的区别

系统截图像给显示器拍照，结果受窗口遮挡、屏幕缩放、多显示器和系统权限影响。浏览器扩展更适合 Web 产品，因为它面对的是浏览器内部的页面或当前 tab。

| 维度 | 系统截图 | 浏览器扩展截图 |
| :--- | :--- | :--- |
| 截图对象 | 屏幕像素 | 浏览器页面或目标区域 |
| 是否怕窗口遮挡 | 怕 | 通常不怕当前页面被其他窗口遮挡 |
| 权限方式 | 系统屏幕录制权限 | 浏览器扩展权限 |
| Web 产品适配 | 弱 | 较强 |
| 长页面与元素定位 | 难 | 更容易与 DOM 信息对齐 |
| 部署形态 | 更像本地工具 | 更像 Web 增强能力 |

## 4. 能力边界

浏览器扩展可以增强当前浏览器环境，但不能解决所有截图场景。

适合：

- 当前用户正在访问的创作端页面。
- 当前 tab 中已经渲染好的预览 iframe 或画布区域。
- 本地创作体验加速。
- 重度用户或内部用户可接受安装扩展的场景。

不适合：

- Docker、CI 或服务器端批量截图。
- 未打开浏览器页面的后台缩略图生成。
- 不愿安装扩展或企业环境禁止扩展的用户。
- 需要完全无用户授权的公网 SaaS 默认能力。

因此，扩展只能是增强通道，不能成为唯一通道。

## 5. 建议架构

扩展方案建议采用“三段式”协作：

| 角色 | 职责 |
| :--- | :--- |
| 创作端页面 | 识别需要截图的页面、内容 hash、尺寸和目标元素 |
| 浏览器扩展 | 获取当前 tab 的截图能力，按授权范围导出图片 |
| author-site 或 screenshot-service | 校验截图元数据，写入现有截图缓存或返回可读 URL |

数据流可以理解为：

```text
创作端页面
  -> 请求扩展截图当前预览区域
  -> 扩展导出图片
  -> 创作端提交图片与 hash 元数据
  -> 服务端写入截图缓存
  -> 画布按既有截图 URL 展示
```

扩展生成的图片不应绕过现有 hash 规则。创作端仍需确认截图对应当前代码、配置和尺寸，避免扩展返回的旧图片污染当前页面。

## 6. 安全与授权原则

扩展权限必须尽量收敛。

建议原则：

- 只对创作端域名启用，不申请无关网站权限。
- 只响应创作端页面发出的明确截图请求。
- 每次截图请求都携带页面 ID、内容 hash、尺寸和目标区域信息。
- 扩展不长期保存截图内容。
- 服务端不信任扩展返回的业务元数据，仍按当前用户、项目、页面和 hash 校验。
- 用户可以在创作端设置中关闭扩展加速。

扩展权限说明应对用户讲清楚：它用于导出当前创作端预览画面的截图，不读取其他网站内容，不作为通用录屏工具。

## 7. 与现有 screenshot-service 的关系

扩展增强后，截图来源会变成多通道，但消费规则保持一致。

| 来源 | 作用 |
| :--- | :--- |
| screenshot-service | 默认、稳定、可部署、可批量、可服务端运行 |
| 浏览器扩展 | 可选、本地加速、当前页面优先 |
| 实时 iframe | 截图不可用或版本不匹配时的真实预览兜底 |

前端展示时不应关心图片来自扩展还是服务端 Puppeteer。只要 hash、尺寸和 `renderBox` 满足规则，就按 [截图服务与预览快照机制](./07_截图服务与预览快照机制.md) 展示；不满足则继续使用实时 iframe。

## 8. 分阶段落地建议

### 8.1 概念验证

先做内部验证，不进入普通用户路径。

验证重点：

- 扩展能否稳定识别创作端 tab。
- 能否截取当前预览 iframe 或画布区域。
- 截图尺寸、DPR、滚动位置是否与画布显示一致。
- 截图结果能否写入现有缓存并被前端正常消费。
- 扩展不可用时是否能无感回退。

### 8.2 Beta 增强

面向内部用户或重度用户开放。

建议能力：

- 创作端检测扩展是否安装。
- 显示“浏览器截图增强已启用”状态。
- 当前视口页面优先走扩展截图。
- 失败后自动回到 screenshot-service。
- 收集扩展截图耗时、失败原因和回退次数。

### 8.3 产品化评估

只有当扩展明显改善体感，并且安装转化、权限解释、企业环境兼容性都可接受时，才考虑作为可推荐增强能力。

不建议把扩展设为默认必需项。普通用户应继续使用不安装任何东西也可工作的 Web 主链路。

## 9. 验收指标

| 指标 | 目标方向 |
| :--- | :--- |
| 扩展安装后当前视口截图耗时 | 低于 screenshot-service 严格截图 |
| 扩展截图成功率 | 稳定可接受 |
| 截图尺寸与画布布局一致性 | 无明显偏移 |
| 扩展不可用回退成功率 | 接近 100% |
| 用户额外操作次数 | 安装授权后接近 0 |
| 旧图误展示 | 必须为 0 |

扩展方案的核心验收不是“能不能截图”，而是用户安装一次后，画布能更快进入轻量截图态，同时失败时完全不影响主链路。

## 10. 决策建议

短期不建议优先做扩展。更稳的顺序是：

1. 先优化现有 screenshot-service，让所有用户无感受益。
2. 再用内部扩展验证“当前浏览器截图”是否显著快于服务端严格截图。
3. 若收益明确，再把扩展作为高级增强能力提供给重度用户。

这样可以避免普通 Web 产品过早背上安装门槛，同时保留未来接入浏览器原生渲染结果的空间。
