## 一、表单生成器职责

表单生成器负责根据 JSON Schema 自动生成可交互的表单控件，实现配置驱动的 UI 生成。

***

## 二、技术选型

| 文件                                    | 说明                                                         |
| :------------------------------------ | :--------------------------------------------------------- |
| `components/demo/ConfigForm.tsx`      | 表单组件                                                       |
| `components/demo/types.ts`            | 类型定义                                                       |
| `components/demo/widgets.tsx`         | 自定义控件（含 FileUploadWidget、ColorPickerWidget、RichTextWidget） |
| `components/demo/ImageListWidget.tsx` | 多图片上传控件                                                    |

**技术选型**：使用 **react-jsonschema-form (RJSF)** 作为表单引擎，支持 JSON Schema 驱动、丰富控件扩展、实时验证。

## 四、Schema 变更与表单重置

### 4.1 问题背景

RJSF 的 `Form` 组件在 `formData` prop 变更时**不会自动重置内部状态**。这意味着当 Schema 发生变更（如 AI 编辑更新了 `default` 值），即使传入了新的 `initialData`，表单控件仍可能显示旧值。

**典型场景**：

1. 用户打开配置面板，表单显示 Schema 中的默认值
2. AI 编辑代码，同时更新了 Schema 中的 `default` 值
3. 父组件传入新的 `initialData`，但 RJSF Form 内部状态未更新
4. 配置面板仍显示旧值，与预览区不一致

### 4.2 解决方案：key 强制重新挂载

通过在父组件中为 `ConfigForm` 添加 `key={schema}` 属性，当 Schema 内容变化时，React 会销毁旧组件实例并创建新实例，从而强制表单以新默认值重新初始化：

```typescript
// page.tsx — 父组件中
<ConfigForm
  key={schema}          // Schema 变化时强制重新挂载
  schema={schema}
  onChange={handleConfigChange}
  initialData={configData}
  sessionId={sessionId}
/>
```

### 4.3 工作原理

```
Schema 变更（AI 编辑 / 用户导入新配置）
        │
        ▼
page.tsx 中 schema 状态更新
        │
        ▼
React 检测 key={schema} 变化
        │
        ├─→ 旧 ConfigForm 实例被卸载（unmount）
        │
        └─→ 新 ConfigForm 实例被创建（mount）
                │
                ▼
        新实例以最新 initialData 初始化 RJSF Form
                │
                ▼
        表单显示最新默认值 ✅
```

### 4.4 设计考量

| 考量                     | 说明                                                |
| :--------------------- | :------------------------------------------------ |
| 为什么不用 `useEffect` 强制更新 | RJSF Form 是第三方组件，无法直接操控其内部状态                      |
| 为什么不用 ref              | RJSF 不暴露强制重置内部状态的 API                             |
| key 方案的代价              | 组件完全重建，丢失用户未保存的输入（但 Schema 变更本身就意味着"规则变了"，重建是合理的） |
| 与 configData 重置的配合     | key 方案解决表单 UI 层问题，configData 重置解决数据层问题，两者互补       |

### 4.5 与 AI 编辑联动的协作

本机制与 [02\_实时预览机制.md](./02_实时预览机制.md) 中描述的「AI 编辑联动」协同工作：

```
AI 编辑代码 + Schema
        │
        ├─→ configData 重置为 {}           ← 数据层：清空旧配置
        ├─→ 新默认值合并 { ...prev, ...newDefaults }  ← 数据层：新值优先
        └─→ key={schema} 触发表单重建      ← UI 层：表单重新挂载
                │
                ▼
        配置面板 + 预览区 同时更新 ✅
```

### 6.1 FileUploadWidget — 单图片上传

```yaml
---
covers:
  - packages/author-site/components/demo/widgets.tsx
---
```

**文件位置**：`components/demo/widgets.tsx`

支持点击上传和拖拽上传，提供图片尺寸实时预览、文件大小限制、尺寸校验等功能。

**可配置选项**：

| 选项        | 类型     | 默认值       | 说明       |
| :-------- | :----- | :-------- | :------- |
| accept    | string | `image/*` | 接受的文件类型  |
| maxSize   | number | 50MB      | 文件大小限制   |
| minWidth  | number | -         | 最小宽度（px） |
| minHeight | number | -         | 最小高度（px） |
| maxWidth  | number | -         | 最大宽度（px） |
| maxHeight | number | -         | 最大高度（px） |

**Schema 示例**：

```json
{
  "type": "string",
  "title": "头像图片",
  "ui:widget": "file",
  "ui:options": {
    "accept": "image/*",
    "maxSize": 5242880,
    "minWidth": 100,
    "minHeight": 100,
    "maxWidth": 2048,
    "maxHeight": 2048
  }
}
```

**交互流程**：

1. 用户点击上传区域或拖拽图片文件
2. 前端校验文件大小和图片尺寸
3. 若尺寸不符合要求，弹出确认对话框，用户可选择"取消"或"继续上传"
4. 上传成功后显示图片预览和删除按钮
5. 若替换新图片，自动删除旧的上传文件

## 七、组件排序控件

```yaml
---
covers:
  - components/demo/ConfigFormNew.tsx
---
```

### 7.1 功能概述

当 Schema 中声明 `$demo.orderable` 时，配置面板顶部显示"组件排序"分组，允许用户调整子组件的排列顺序。

**触发条件**：

| 条件                          | 说明          |
| :-------------------------- | :---------- |
| Schema 包含 `$demo.orderable` | 必须声明可排序属性列表 |
| `orderable` 至少 2 项          | 少于 2 项无排序意义 |

### 7.2 控件交互

排序控件同时支持**箭头按钮**和**拖拽排序**两种方式，两种方式共享同一排序状态（`__order`）：

| 交互方式 | 适用场景       | 实现方式                                  |
| :--- | :--------- | :------------------------------------ |
| 箭头按钮 | 精确移动、无障碍操作 | 原生按钮点击，交换相邻项                          |
| 拖拽排序 | 大跨度移动、高效操作 | `@dnd-kit/sortable`，`arrayMove` 计算新顺序 |

**技术选型**：拖拽排序使用 `@dnd-kit/core` + `@dnd-kit/sortable`，理由：

- 轻量级，无障碍支持良好（内置 `KeyboardSensor`）
- 与 React 生态集成度高，支持 `SortableContext`
- `PointerSensor` 配置 5px 激活距离，防止误触

### 7.3 组件结构

排序控件由两个组件协作实现：

| 组件             | 职责                                              |
| :------------- | :---------------------------------------------- |
| `OrderControl` | 排序区域容器，管理 `DndContext`、`SortableContext`，处理排序变更 |
| `SortableItem` | 单个可排序项，集成拖拽手柄 + 序号 + 标题 + 箭头按钮                  |

**SortableItem 渲染结构**：

```
[拖拽手柄 GripVertical] [序号] [组件标题] [↑ 按钮] [↓ 按钮]
```

| 元素    | 样式说明                                         |
| :---- | :------------------------------------------- |
| 拖拽手柄  | GripVertical 图标，颜色较淡，悬停时加深并变为 `cursor: grab` |
| 序号    | 等宽字体，居中对齐                                    |
| 组件标题  | 使用 Schema 中的 `title`，超长截断                    |
| 箭头按钮  | ChevronUp / ChevronDown，禁用态置灰                |
| 拖拽中状态 | 半透明（opacity: 0.5）+ 背景高亮                      |

### 7.4 排序状态

排序结果通过 `__order` 字段持久化：

```typescript
// 配置数据中的排序字段
{
  "header": "标题文本",
  "content": "内容文本",
  "__order": ["content", "header"]  // 排序结果
}
```

**状态管理流程**：

```
用户操作（箭头/拖拽）
    │
    ▼
OrderControl 计算新顺序
    ├─ 箭头：交换相邻项
    └─ 拖拽：arrayMove(order, oldIndex, newIndex)
    │
    ▼
onOrderChange(newOrder)
    │
    ▼
ConfigForm.handleOrderChange
    └─ { ...formData, __order: newOrder }
        │
        ▼
onChange(newData) → page.tsx setConfigData → PreviewPanel 更新
```

### 7.5 与表单的集成

组件排序控件与 RJSF 表单分离，单独渲染在表单顶部：

```typescript
// ConfigForm 渲染顺序
<>
  <OrderControl />       {/* 排序控件，条件渲染 */}
  <分隔线 />
  <字段分组 />           {/* RJSF 表单字段 */}
</>
```

**条件渲染逻辑**：

```typescript
const orderable = useMemo(() => getOrderable(schema), [schema]);
// 仅当 orderable 存在且至少 2 项时渲染
{orderable && orderable.length >= 2 && <OrderControl ... />}
```


### 8.3 备注数据解析

`parseSchemaToFields` 函数在解析每个属性时，提取 `$demo.note` 字段到 `FieldConfig`：

```typescript
// ConfigFormNew.tsx — parseSchemaToFields

  key,
  title: prop.title || key,
  type: prop.type || "string",
  // ...其他字段

};
```

### 8.4 备注状态管理

`ConfigForm` 内部维护 `noteDialogField` 状态，记录当前打开备注弹窗的配置项 key：

```typescript
const [noteDialogField, setNoteDialogField] = useState<string | null>(null);
const currentNoteField = useMemo(() => {
  if (!noteDialogField) return null;
  for (const group of fieldGroups) {
    const field = group.fields.find((f) => f.key === noteDialogField);
    if (field) return field;
  }
  return null;
}, [noteDialogField, fieldGroups]);
```

### 8.5 备注写入 Schema

通过 `updateSchemaNote` 和 `deleteSchemaNote` 两个回调操作 Schema JSON：

```typescript
const updateSchemaNote = useCallback((fieldKey: string, noteHtml: string) => {
  if (!onSchemaChange || !schema) return;
  const parsed = JSON.parse(schema);
  if (parsed.properties?.[fieldKey]) {
    if (!parsed.properties[fieldKey].$demo) {
      parsed.properties[fieldKey].$demo = {};
    }
    parsed.properties[fieldKey].$demo.note = noteHtml;
    onSchemaChange(JSON.stringify(parsed, null, 2));
  }
}, [schema, onSchemaChange]);

const deleteSchemaNote = useCallback((fieldKey: string) => {
  if (!onSchemaChange || !schema) return;
  const parsed = JSON.parse(schema);
  if (parsed.properties?.[fieldKey]?.$demo) {
    delete parsed.properties[fieldKey].$demo.note;
    if (Object.keys(parsed.properties[fieldKey].$demo).length === 0) {
      delete parsed.properties[fieldKey].$demo;
    }
    onSchemaChange(JSON.stringify(parsed, null, 2));
  }
}, [schema, onSchemaChange]);
```

**关键设计点**：

- 备注变更通过 `onSchemaChange` 传递完整 Schema 字符串，而非仅传递备注字段
- 删除备注时，若 `$demo` 对象为空则一并删除，保持 Schema 干净
- `updateSchemaNote` 和 `deleteSchemaNote` 均依赖稳定的 `schema` 和 `onSchemaChange`

### 8.6 备注按钮与摘要展示

`FieldRenderer` 接收 `onNoteClick` 回调，渲染 `NoteButton` 和 `NotePreview`：

## 九、验证机制

### 9.1 内置验证

RJSF 内置以下验证：

| 验证类型 | Schema 字段   |
| :--- | :---------- |
| 必填验证 | `required`  |
| 类型验证 | `type`      |
| 最小值  | `minimum`   |
| 最大值  | `maximum`   |
| 最小长度 | `minLength` |
| 最大长度 | `maxLength` |
| 格式验证 | `pattern`   |
| 枚举验证 | `enum`      |

### 9.2 自定义验证

可通过 `customValidate` 扩展验证逻辑：

```typescript
function customValidate(formData, errors) {
  if (formData.title && formData.title.length < 3) {
    errors.title.addError('标题至少需要 3 个字符')
  }
  return errors
}
```
