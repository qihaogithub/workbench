## 一、表单生成器职责

表单生成器负责根据已经确认的 JSON Schema 配置字段生成可交互的表单控件，实现“配置字段驱动表单控件生成”。它只消费字段类型、默认值、控件语义、备注、上传校验、排序和定位等配置声明，不负责表达页面结构、布局层级或 AI 生成页面的内容模型。

`config.schema.json` 是用户配置项的持久协议。没有用户明确配置诉求时，新建页面可以保留空 `properties`；页面标题、文案、图片、颜色、按钮和布局默认由页面源码或原型页文件承载。用户在单页属性面板中明确选择“设为配置项”后，才进入配置化写回链路：高保真 React 页把字段追加到页面 Schema 并补齐 Props 绑定；HTML/CSS 原型页把字段追加到页面 Schema 并写入 `data-bind-*` 或 `{{fieldKey}}` 绑定。写回链路详见 [02_实时预览机制.md](./02_实时预览机制.md) 和 [06_可视化批注与编辑机制.md](./06_可视化批注与编辑机制.md)。

复杂列表、组件组、布局结构、动态表达式和跨页面业务动作不由普通表单字段直接承载。此类修改应走页面源码、原型页、属性面板结构化上下文、AI 修改链路、画布布局或应用图协议，避免把 JSON Schema 扩展成页面结构 DSL。

## 四、Schema 变更与表单重置

### 4.1 问题背景

RJSF 的 `Form` 组件在 `formData` prop 变更时**不会自动重置内部状态**。这意味着当 Schema 发生变更（如 AI 编辑更新了 `default` 值），即使传入了新的 `initialData`，表单控件仍可能显示旧值。

**典型场景**：

1. 用户打开配置面板，表单显示 Schema 中的默认值
2. AI 编辑代码，同时更新了 Schema 中的 `default` 值
3. 父组件传入新的 `initialData`，但 RJSF Form 内部状态未更新
4. 配置面板仍显示旧值，与预览区不一致

### 4.2 解决方案：key 强制重新挂载与内部状态同步

通过在父组件中为 `ConfigForm` 添加 `key={schema}` 属性，当 Schema 内容变化时，React 会销毁旧组件实例并创建新实例，从而强制表单以新默认值重新初始化。

同时，`ConfigForm` 内部会在 Schema 或 initialData 进入组件时补齐缺失字段，但必须遵守一条约束：如果合并后的表单数据与当前状态等价，状态更新函数返回原对象，不创建新对象。这样父组件即使因预览、画布或配置面板联动传入等价的新对象引用，也不会把“没有变化的数据”变成新的渲染任务。

```typescript
// page.tsx — 父组件中
<ConfigForm
  key={schema}          // Schema 变化时强制重新挂载
  schema={schema}
  onChange={handleConfigChange}
  initialData={configData}
  sessionId={sessionId}
/>
```

### 4.3 工作原理

```
Schema 变更（AI 编辑 / 用户导入新配置）
        │
        ▼
page.tsx 中 schema 状态更新
        │
        ▼
React 检测 key={schema} 变化
        │
        ├─→ 旧 ConfigForm 实例被卸载（unmount）
        │
        └─→ 新 ConfigForm 实例被创建（mount）
                │
                ▼
        新实例以最新 initialData 初始化 RJSF Form
                │
                ▼
        表单显示最新默认值 ✅
```

### 4.4 设计考量

| 考量                            | 说明                                                                                     |
| :------------------------------ | :--------------------------------------------------------------------------------------- |
| 为什么状态同步要做等价判断      | 父层可能传入等价的新对象引用；若内部无条件返回新对象，React 会继续安排渲染，严重时触发最大更新深度错误 |
| 为什么 Schema 同步不依赖 initialData 引用 | Schema 默认值只由 Schema 内容决定；initialData 的补齐由独立同步流程处理，避免对象引用变化带来重复重置 |
| key 方案的代价                  | 组件完全重建，丢失用户未保存的输入（但 Schema 变更本身就意味着"规则变了"，重建是合理的） |
| 与 configData 重置的配合        | key 方案解决表单 UI 层问题，configData 重置解决数据层问题，两者互补                      |

### 4.5 与 AI 编辑联动的协作

本机制与 [02\_实时预览机制.md](./02_实时预览机制.md) 中描述的「AI 编辑联动」协同工作：

```
AI 编辑代码 + Schema
        │
        ├─→ configData 重置为 {}           ← 数据层：清空旧配置
        ├─→ 新默认值合并 { ...prev, ...newDefaults }  ← 数据层：新值优先
        └─→ key={schema} 触发表单重建      ← UI 层：表单重新挂载
                │
                ▼
        配置面板 + 预览区 同时更新 ✅
```

## 五、控件映射体系

### 5.1 设计原则：三层映射结构

表单控件的选择遵循**三层优先级**，职责清晰：

```
Layer 1: ui:widget 显式覆盖     → 最优先，用于非标准控件或强制指定
Layer 2: format 语义映射         → 根据 JSON Schema 标准 format 值推断
Layer 3: type 数据类型回退       → 根据基本数据类型推断
```

**设计原则**：`format` 是语义声明（「这个值是什么」），`ui:widget` 是控件覆盖（「我想用不同的控件」）。渲染层负责理解语义声明并给出合理的默认控件；只有需要偏离默认时，才需要 `ui:widget`。

### 5.2 Layer 1 — ui:widget 显式覆盖

通过 `ui:widget` 字段强制指定控件，用于 JSON Schema 标准 `format` 不覆盖的场景：

| ui:widget 值         | 控件                | 适用场景   |
| :------------------- | :------------------ | :--------- |
| `"file"` / `"image"` | FileUploadWidget    | 单文件或单图片上传 |
| `"imageList"`        | ImageListWidget     | 多图片上传 |
| `"richtext"`         | Textarea (richText) | 富文本编辑 |

### 5.3 Layer 2 — format 语义映射

根据 JSON Schema 标准 `format` 值推断控件，是最符合规范的写法：

| format 值 | 控件              | 说明       |
| :-------- | :---------------- | :--------- |
| `"image"` | FileUploadWidget  | 单图片上传 |
| `"color"` | ColorPickerWidget | 颜色选择器 |

> **说明**：当用户明确要求某个字段成为配置项时，`format: "image"` 是 JSON Schema 标准写法，会被自动映射为图片上传控件，无需额外写 `ui:widget`。`format: "color"` 的映射机制与此一致。AI 不应仅因为页面中出现图片或颜色，就自行把它们抽成配置字段。

### 5.4 Layer 3 — type 数据类型回退

根据基本数据类型推断控件，适用于无特殊 format 声明的场景：

| type 值                  | 控件            | 触发条件                        |
| :----------------------- | :-------------- | :------------------------------ |
| `"array"`                | ImageListWidget | 多图片上传                      |
| `"boolean"`              | Switch          | 开关                            |
| `"number"` / `"integer"` | Slider / Input  | 有 min/max 时滑块，否则数字输入 |
| `enum`                   | Select          | 下拉选择                        |
| `maxLength > 100`        | Textarea        | 长文本域                        |
| 默认                     | Input           | 文本输入                        |

### 5.5 控件优先级示例

同一字段同时声明 `format` 和 `ui:widget` 时，`ui:widget` 优先生效：

```json
{
  "type": "string",
  "format": "image",
  "ui:widget": "file" // ← Layer 1 生效，但与 Layer 2 结果相同
}
```

建议**不要重复声明**，按以下原则选择：

- 单图字段 → 优先使用 `format: "image"`（标准写法）
- 多图字段 → 使用 `type: "array"`（Layer 3）或 `ui:widget: "imageList"`（Layer 1）
- 富文本字段 → 使用 `ui:widget: "richtext"`（无对应 format 标准值）

### 5.6 字段分组与条件显示

`ConfigForm` 默认会根据字段名、`format` 和字段语义自动归组，例如图片字段进入“图片资源”，颜色字段进入“颜色配置”。当自动归组无法表达业务关系时，字段可以通过 `ui:options.group` 指定分组名称。配置面板会优先使用显式分组，让相关字段稳定展示在同一个折叠区域中。

| 配置项 | 说明 |
| :----- | :--- |
| `group` | 字段所在分组名称；为空或未声明时继续使用自动归组 |

`ConfigForm` 支持在字段的 `ui:options.visibleWhen` 中声明轻量条件显示规则，用来表达同一份配置中的二选一或依赖关系。例如媒体类型选择“图片”时只展示图片上传，选择“SVGA动画”时只展示 SVGA 文件上传。

| 配置项 | 说明 |
| :----- | :--- |
| `field` | 依赖的同层字段 key |
| `equals` | 目标值，支持字符串、数字、布尔值和 `null` |

条件显示只影响配置面板 UI，不会清空隐藏字段已保存的值。用户切回原选项时，之前填写或上传的资源仍会回显。未声明 `visibleWhen` 的字段始终显示；不支持嵌套表达式、跨页面字段、`oneOf` 或 `anyOf` 联动。

### 5.7 配置分类筛选

`PageConfigPanel` 在页面列表和页面详情顶部提供配置分类筛选。筛选只改变右侧配置面板的浏览范围，不修改 Schema、配置数据或隐藏字段的已保存值。

配置分类由用户在配置项元数据中显式设置，写入字段级 `ui:options.category`。内置分类包含“设计”“动效”“音效”“视频”“其他”，也允许用户输入自定义分类。没有设置分类的旧配置项只在“全部分类”下展示，不会被自动归入“其他”。

筛选选项由 `config-categories.ts` 从项目级 Schema 和页面级 Schema 中实际存在的分类动态聚合：未出现的内置分类不显示；自定义分类按出现顺序补在内置分类之后。页面列表数量和详情表单使用同一套分类统计规则，避免列表显示“有匹配项”但详情为空。筛选为空时，面板显示轻量空状态，并允许用户切回其他分类继续查看。

## 六、内置自定义控件详解

### 6.1 FileUploadWidget — 单图片上传

**文件位置**：`components/demo/widgets.tsx`

支持点击上传和拖拽上传，提供图片尺寸实时预览、文件大小限制、尺寸校验等功能。

**两种触发方式**：

| 方式            | Schema 写法         | 说明                 |
| :-------------- | :------------------ | :------------------- |
| Layer 2（推荐） | `format: "image"`   | JSON Schema 标准写法 |
| Layer 1         | `ui:widget: "file"` | 自定义控件显式指定，可用于 `.svga` 等非图片资源 |

**可配置选项**：

| 选项      | 类型   | 默认值    | 说明           |
| :-------- | :----- | :-------- | :------------- |
| accept    | string | `image/*` | 接受的文件类型 |
| maxSize   | number | 50MB      | 文件大小限制   |
| minWidth  | number | -         | 最小宽度（px） |
| minHeight | number | -         | 最小高度（px） |
| maxWidth  | number | -         | 最大宽度（px） |
| maxHeight | number | -         | 最大高度（px） |

当 `ui:options.accept` 指向 `.svga` 时，上传接口按文件扩展名允许浏览器上报的 `application/octet-stream`，并返回可被预览运行时读取的 session asset URL。SVGA 的播放能力不由表单控件承担，页面应把该 URL 交给预览运行时内置的 `SvgaPlayer`。

**Schema 示例**（Layer 2 — 推荐的标准写法）：

```json
{
  "type": "string",
  "format": "image",
  "title": "Banner图片",
  "ui:options": {
    "accept": "image/*",
    "maxSize": 5242880,
    "minWidth": 100,
    "minHeight": 100,
    "maxWidth": 2048,
    "maxHeight": 2048
  }
}
```

**Schema 示例**（Layer 1 — 等效的显式写法）：

```json
{
  "type": "string",
  "format": "image",
  "title": "Banner图片",
  "ui:widget": "file",
  "ui:options": {
    "accept": "image/*",
    "maxSize": 5242880
  }
}
```

**交互流程**：

1. 用户点击上传区域或拖拽图片文件
2. 前端校验文件大小和图片尺寸
3. 若尺寸不符合要求，弹出确认对话框，用户可选择"取消"或"继续上传"
4. 上传成功后显示图片预览和删除按钮
5. 若替换新图片，自动删除旧的上传文件

## 七、组件排序控件

```yaml
---
covers:
  - packages/shared/src/demo/ConfigForm.tsx
---
```

### 7.1 功能概述

当 Schema 中声明 `$demo.orderable` 或 `$demo.orderableHorizontal` 时，配置面板顶部显示排序分组，允许用户调整子组件的排列顺序。支持**垂直排序**和**水平排序**两种模式，可同时声明，互不干扰。

**触发条件**：

| 条件                                   | 说明                            |
| :------------------------------------- | :------------------------------ |
| Schema 包含 `$demo.orderable`          | 声明垂直可排序属性列表          |
| Schema 包含 `$demo.orderableHorizontal` | 声明水平可排序属性列表          |
| 至少 2 项                              | 两种模式各自独立检查，最少 2 项 |

### 7.2 控件交互

排序控件同时支持**箭头按钮**和**拖拽排序**两种方式，两种方式共享同一排序状态：

| 交互方式 | 适用场景             | 实现方式                                    |
| :------- | :------------------- | :------------------------------------------ |
| 箭头按钮 | 精确移动、无障碍操作 | 原生按钮点击，交换相邻项                    |
| 拖拽排序 | 大跨度移动、高效操作 | `@dnd-kit/sortable`，`arrayMove` 计算新顺序 |

**两种模式的 UI 差异**：

| 维度     | 垂直排序                    | 水平排序                              |
| :------- | :-------------------------- | :------------------------------------ |
| 标题     | "组件排序"                  | "横向排序"                            |
| 方向提示 | `ArrowDown` + "从上到下排列" | `ArrowRight` + "从左到右排列"         |
| 布局     | 全宽行列表，每个占一行       | 行内 Chip 标签，项间用 `→` 分隔       |
| 序号     | 圆形序号徽标，始终可见       | 无序号                                |
| 箭头按钮 | ↑↓ (始终可见)              | ←→ (hover 时才显示)                  |
| 拖拽策略 | `verticalListSortingStrategy` | `horizontalListSortingStrategy`       |

**技术选型**：拖拽排序使用 `@dnd-kit/core` + `@dnd-kit/sortable`，理由：

- 轻量级，无障碍支持良好（内置 `KeyboardSensor`）
- 与 React 生态集成度高，支持 `SortableContext`
- `PointerSensor` 配置 5px 激活距离，防止误触

### 7.3 组件结构

排序控件由两个组件协作实现：

| 组件           | 职责                                                             |
| :------------- | :--------------------------------------------------------------- |
| `OrderControl` | 排序区域容器，管理 `DndContext`、`SortableContext`，处理排序变更 |
| `SortableItem` | 单个可排序项，集成拖拽手柄 + 序号 + 标题 + 方向箭头按钮          |

**SortableItem 渲染结构**（垂直模式）：

```
[拖拽手柄 GripVertical] [序号] [组件标题] [↑ 按钮] [↓ 按钮]
```

**SortableItem 渲染结构**（水平模式）：

```
[拖拽手柄] [组件标题] [← hover才显示] [→ hover才显示]
```

| 元素       | 样式说明                                                     |
| :--------- | :----------------------------------------------------------- |

## 八、元素定位控件


### 8.1 功能概述

当 Schema 中声明 `$demo.positionable` 时，配置面板中排序区域下方显示"元素定位"分组，允许用户通过可视化的缩略画布拖拽或坐标输入框精确调整子组件位置。

**触发条件**：

| 条件                               | 说明                  |
| :--------------------------------- | :-------------------- |
| Schema 包含 `$demo.positionable`   | 必须声明可定位属性配置 |
| `positionable.items` 至少 1 项     | 至少 1 项才有定位意义 |

### 8.2 控件交互

定位控件提供两种操作方式，共享同一状态（`__positions`）：

| 交互方式     | 适用场景             | 实现方式                               |
| :----------- | :------------------- | :------------------------------------- |
| 画布拖拽     | 直观调整、大范围移动 | `onMouseDown`/`onMouseMove`/`onMouseUp` 事件 |
| 坐标输入     | 精确数值控制         | 数字输入框，限制 0 ~ 容器尺寸          |

**画布设计**：

- 缩略画布等比例缩放容器尺寸，最大 280×200px
- 画布显示网格辅助线（100px 间隔的 SVG `<line>`），不干扰交互
- 每个可定位元素以可拖拽标记块显示，拖拽时标记高亮（`bg-primary text-primary-foreground` + `z-10`）
- 标记块显示 `Move` 图标 + 组件名称截断文本（最长 60px）

### 8.3 组件结构

定位控件由单个组件实现：

| 组件              | 职责                                                       |
| :---------------- | :--------------------------------------------------------- |
| `PositionControl` | 定位区域容器，管理画布拖拽逻辑、坐标输入、重置功能         |

**核心状态**：

```typescript
const [draggingKey, setDraggingKey] = useState<string | null>(null);
```

### 8.4 画布尺寸计算

画布尺寸通过 `$demo.positionable.size`、`$demo.previewSize` 或回退值计算：

```typescript
const containerWidth =
  positionable.size?.width ??
  (typeof previewSize?.width === "number" ? previewSize.width : 800);
const containerHeight =
  positionable.size?.height ??
  (typeof previewSize?.height === "number" ? previewSize.height : 600);

// 等比缩放，确保不超过 MAX 280×200
const aspectRatio = containerWidth / containerHeight;
// 按宽高比计算 canvas 尺寸，保持容器比例
let CANVAS_WIDTH, CANVAS_HEIGHT;
if (MAX_CANVAS_WIDTH / aspectRatio <= MAX_CANVAS_HEIGHT) {
  CANVAS_WIDTH = MAX_CANVAS_WIDTH;
  CANVAS_HEIGHT = Math.round(MAX_CANVAS_WIDTH / aspectRatio);
} else {
  CANVAS_HEIGHT = MAX_CANVAS_HEIGHT;
  CANVAS_WIDTH = Math.round(MAX_CANVAS_HEIGHT * aspectRatio);
}

const scaleFactor = CANVAS_WIDTH / containerWidth; // 画布→容器坐标转换系数
```

### 8.5 定位状态

定位结果通过 `__positions` 字段持久化：

```typescript

  "button": "立即购买",
  "__positions": {
    "logo": { "x": 100, "y": 50 },
    "title": { "x": 200, "y": 150 },
    "button": { "x": 350, "y": 250 }
  }
}
```

**状态管理流程**：

```
画布拖拽 / 坐标输入
    │
    ▼
PositionControl 计算新坐标
    ├─ 画布：mousedown → mousemove → mouseup
    │       坐标转换：canvasX / scaleFactor → containerX
    │       边界限制：Math.max(0, Math.min(value, maxValue))
    └─ 输入：parseInt → 边界限制
    │
    ▼

    │
    ▼

    └─ { ...formData, __positions: newPositions }
        │
        ▼
```


### 9.5 备注写入 Schema

通过 `updateSchemaNote` 和 `deleteSchemaNote` 两个回调操作 Schema JSON：

```typescript
const updateSchemaNote = useCallback(
  (fieldKey: string, noteHtml: string) => {
    if (!onSchemaChange || !schema) return;
    const parsed = JSON.parse(schema);
    if (parsed.properties?.[fieldKey]) {
      if (!parsed.properties[fieldKey].$demo) {
        parsed.properties[fieldKey].$demo = {};
      }
      parsed.properties[fieldKey].$demo.note = noteHtml;
      onSchemaChange(JSON.stringify(parsed, null, 2));
    }
  },
  [schema, onSchemaChange],
);

const deleteSchemaNote = useCallback(
  (fieldKey: string) => {
    if (!onSchemaChange || !schema) return;
    const parsed = JSON.parse(schema);
    if (parsed.properties?.[fieldKey]?.$demo) {
      delete parsed.properties[fieldKey].$demo.note;
      if (Object.keys(parsed.properties[fieldKey].$demo).length === 0) {
        delete parsed.properties[fieldKey].$demo;
      }
      onSchemaChange(JSON.stringify(parsed, null, 2));
    }
  },
  [schema, onSchemaChange],
);
```

**关键设计点**：

- 备注变更通过 `onSchemaChange` 传递完整 Schema 字符串，而非仅传递备注字段
- 删除备注时，若 `$demo` 对象为空则一并删除，保持 Schema 干净
- `updateSchemaNote` 和 `deleteSchemaNote` 均依赖稳定的 `schema` 和 `onSchemaChange`
- `NoteDialog` 在编辑模式复用 `DocumentEditor` 的 HTML 模式，提供与知识文档一致的标题、列表、任务列表、引用、代码块、链接、分隔线和预览能力
- 保存备注时仍通过 `sanitizeNoteHtml` 过滤危险 HTML，仅保留编辑器允许生成的安全标签和属性

### 10.3 图片尺寸校验

**实现位置**：

- `components/demo/widgets.tsx` — FileUploadWidget（单图片上传）
- `components/demo/ImageListWidget.tsx` — ImageListWidget（多图片上传）

#### 10.3.1 核心函数

两个控件均使用相同的校验逻辑：

```typescript
// 1. 获取图片实际尺寸
function getImageDimensions(
  file: File,
): Promise<{ width: number; height: number }>;

// 2. 校验尺寸是否合规
function validateImageDimensions(
  actual: { width: number; height: number },
  options: { minWidth?; minHeight?; maxWidth?; maxHeight? },
): { valid: boolean; message: string };
```

#### 10.3.2 校验流程

```

用户选择/拖拽图片
    │
    ▼
1. 文件大小校验 (file.size > maxSize)
    ├─ 失败 → setError()，终止
    └─ 通过 ↓
    │
    ▼
2. 尺寸校验检查 (hasDimensionCheck)
    ├─ 未声明尺寸限制 → 跳过，直接上传
    └─ 已声明 ↓
    │
    ▼
3. getImageDimensions(file)
    ├─ 读取失败 → setError('无法读取图片尺寸')
    └─ 成功 ↓
    │
    ▼
4. validateImageDimensions(actual, options)
    ├─ valid: true → 继续上传
    └─ valid: false → setSizeWarning({ file, message })
        │
        ▼
    弹出 Dialog 警告
        ├─ 「取消上传」→ setSizeWarning(null)
        └─ 「继续上传」→ doUpload(file, true) // skipDimensionCheck=true
```

#### 10.3.3 关键实现细节

| 细节        | 说明                                                                                            |
| :---------- | :---------------------------------------------------------------------------------------------- |
| 触发条件    | `hasDimensionCheck = Object.values(dimensionOptions).some(v => typeof v === 'number')`          |
| 跳过机制    | `skipDimensionCheck` 参数，用户确认继续时传 `true`                                              |
| 错误提示    | 自动拼接所有不合规项：`"图片尺寸不符合要求：宽度小于 100px，高度大于 2048px（实际 80x3000px）"` |
| Dialog 组件 | 使用 shadcn/ui 的 `Dialog`，包含 `AlertTriangle` 警告图标                                       |
| 尺寸读取    | 使用 `URL.createObjectURL(file)` + `Image` 对象异步加载                                         |

#### 10.3.4 Schema 配置映射

```typescript
// widgets.tsx 中的配置提取逻辑
const rawOptions = (props.options || {}) as FileUploadWidgetOptions;

const dimensionOptions: DimensionOptions = {
  minWidth: rawOptions.minWidth,
  minHeight: rawOptions.minHeight,
  maxWidth: rawOptions.maxWidth,
  maxHeight: rawOptions.maxHeight,
};

const hasDimensionCheck = Object.values(dimensionOptions).some(
  (v) => typeof v === "number",
);
```

> **注意**：尺寸校验是**可选功能**，只有当 Schema 中声明了至少一个尺寸限制时才生效。
