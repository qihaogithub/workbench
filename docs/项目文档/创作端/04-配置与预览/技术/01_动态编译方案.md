# 预览系统 - 动态编译方案

> 版本：v3.0
> 创建日期：2026-04-19
> 更新日期：2026-06-30

## 一、架构概述

### 1.1 核心思路

采用**服务端编译 + 同源 Preview Runtime + iframe 沙箱渲染**架构，将 AI 生成的 TSX 代码编译为 ESM 模块，核心运行时和已登记依赖从同源 `preview-runtime` manifest 加载，在 iframe 沙箱中隔离执行。

```
┌───────────────────────────────────────────────────────────────┐
│  compiler.ts (服务端)                                          │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │ 1. sucrase 编译 TSX → ESM (保留 ESM import/export)      │  │
│  │ 2. 提取 import 语句中的 npm 包                          │  │
│  │ 3. rewriteImportsToCdn() 替换为同源 runtime URL          │  │
│  │    - React / ReactDOM / jsx-runtime → manifest URL       │  │
│  │    - lucide / framer / @preview/sdk → manifest URL       │  │
│  │ 4. 生成 moduleHash，登记只读 moduleUrl                   │  │
│  │ 5. 返回 { compiledCode, moduleHash, moduleUrl, ... }     │  │
│  └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌───────────────────────────────────────────────────────────────┐
│  iframe 沙箱 (浏览器)                                         │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │ 1. 固定 shell 或 inline shell 加载同源 runtime import map │  │
│  │ 2. 通过 UPDATE_MODULE 导入 moduleUrl                     │  │
│  │ 3. 旧 UPDATE_CODE / compiledCode 路径继续兼容             │  │
│  │ 4. ErrorBoundary 捕获运行时错误并回传父窗口               │  │
│  └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 架构演进

| 阶段 | 方案 | 说明 |
|:-----|:-----|:-----|
| v1.0 (2026-04) | CommonJS + new Function | 服务端编译为 CommonJS，前端通过 `new Function` 执行并注入 `requireMap` |
| v2.0 (2026-05) | ESM + iframe + CDN | 服务端保留 ESM 格式，依赖通过 esm.sh CDN 加载，iframe 沙箱隔离 |
| v2.1 (2026-06) | 阶段观测 + 首屏占位 | 保留 CDN 架构，补齐加载阶段 timing、截图 meta 预填、首屏截图生成避让和 CDN base 统一 |
| v3.0 (2026-06) | 同源 Preview Runtime + 内容寻址模块 | 核心运行时和白名单依赖本地化，固定 shell，`moduleHash/moduleUrl` 协议，发布包记录 runtime 版本 |

v2.0 核心改进：
- **iframe 沙箱隔离**：DOM/CSS/JS 完全隔离，消除样式污染和全局变量冲突
- **动态 CDN 依赖**：任意 npm 包自动通过 esm.sh 加载，无需前端预置依赖映射表
- **依赖一致性**：通过 `?deps` 参数 + Import Map 双重保障，确保 iframe 内所有模块共享同一 React 实例

v3.0 保留 sucrase 和 iframe 隔离边界，但把首屏关键依赖从远程 ESM CDN 移到同源 runtime。远程 CDN 只保留为紧急诊断回退，不再是默认加载路径。

## 二、服务端编译

### 2.1 编译入口

**函数**: `compileCode()` 位于 `packages/author-site/src/lib/compiler.ts`

**输入**:
```typescript
{
  code: string              // TSX 代码字符串
  lockedDependencies?: Record<string, string>  // 已锁定的依赖 URL
}
```

**输出**:
```typescript
{
  compiledCode: string      // 编译后代码（import 已替换为同源 runtime URL 或 CDN 回退 URL）
  dependencies: string[]    // 提取的依赖列表
  cssImports: string[]      // CSS 导入列表（单独处理）
  moduleHash: string        // 编译后代码 sha256
  moduleUrl?: string        // /api/preview-modules/{hash}.js
}
```

### 2.2 编译流程

```typescript
import { transform } from 'sucrase';

// 1. 自动包装：裸 JSX → export default function() { return (...) }
const wrappedCode = autoWrapIfNoDefaultExport(code);

// 2. sucrase 编译 TSX → ESM（保留 ES Module import/export）
const result = transform(wrappedCode, {
  transforms: ['typescript', 'jsx'],
  jsxRuntime: 'automatic',    // 自动添加 react/jsx-runtime 导入
  production: true,
});

// 3. 从编译后的代码提取 import 语句
const dependencies = extractImports(result.code);
// sucrase 会自动注入: import { jsx as _jsx } from 'react/jsx-runtime'

// 4. 分类处理
const cssImports = dependencies.filter(isCssImport);

// 5. 将 npm 包 import 路径替换为 manifest URL 或 CDN 回退 URL
const compiledCode = rewriteImportsToCdn(result.code, dependencies, lockedDependencies);
```

### 2.3 runtime URL 替换规则

`rewriteImportsToCdn()` 函数名保持兼容，但默认行为已经从 CDN URL 生成变为 preview runtime manifest 解析：

| import | 默认替换 | CDN 回退 |
|:-------|:---------|:---------|
| `react` | `/preview-runtime/vendor/react.js` | `https://esm.sh/react@18.3.1` |
| `react-dom/client` | `/preview-runtime/vendor/react-dom-client.js` | `https://esm.sh/react-dom@18.3.1/client` |
| `react/jsx-runtime` | `/preview-runtime/vendor/react-jsx-runtime.js` | `https://esm.sh/react@18.3.1/jsx-runtime` |
| `lucide-react` | `/preview-runtime/vendor/lucide-react.js` | `https://esm.sh/lucide-react@0.323.0?deps=react@18.3.1,react-dom@18.3.1` |
| `framer-motion` | `/preview-runtime/vendor/framer-motion.js` | `https://esm.sh/framer-motion@12.38.0?deps=react@18.3.1,react-dom@18.3.1` |
| `@preview/sdk` | `/preview-runtime/vendor/preview-sdk.js` | data URL 虚拟模块 |

`PREVIEW_RUNTIME_SOURCE=cdn` 或 `PREVIEW_RUNTIME_CDN_FALLBACK=1` 会启用 CDN 回退。该回退用于诊断或紧急止血，不作为默认路径。

### 2.4 编译器选型

| 维度 | sucrase | babel | swc |
|:-----|:--------|:------|:----|
| 编译速度 | ⭐⭐⭐⭐⭐ (~50ms) | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 包体积 | ~50KB | ~500KB | ~3MB |
| TSX 支持 | ✅ | ✅ | ✅ |
| 零原生依赖 | ✅ | ✅ | ❌ (需 wasm) |

**选择 sucrase 的理由**：纯 JS 实现、编译速度快、包体积小，满足实时预览需求。配置 `jsxRuntime: 'automatic'` 和 `production: true` 以获得最佳输出。

## 三、iframe 沙箱渲染

### 3.1 iframe 模板

编译后的代码通过 iframe 沙箱执行，模板由 `packages/demo-ui/src/iframe-template.ts` 和 `packages/shared/src/demo/iframe-template.ts` 的 `generateIframeHtml()` 生成。创作端默认使用固定 shell route；viewer 静态导出使用 inline shell，但会把 runtime base 显式绑定到当前站点 origin。

**模板结构**：

```html
<head>
  <!-- Import Map：同源 preview runtime -->
  <script type="importmap">
  {
    "imports": {
      "react": "/preview-runtime/vendor/react.js",
      "react-dom/client": "/preview-runtime/vendor/react-dom-client.js",
      "react/jsx-runtime": "/preview-runtime/vendor/react-jsx-runtime.js",
      "lucide-react": "/preview-runtime/vendor/lucide-react.js"
    }
  }
  </script>
</head>
<body>
  <div id="root"></div>

  <script type="module">
    import React from 'react';
    import ReactDOM from 'react-dom/client';

    // ErrorBoundary 捕获运行时错误 → postMessage 回传父窗口
    // 通过 moduleUrl 或 compiledCode 加载编译后 ESM 模块
    // postMessage 协议：UPDATE_MODULE / UPDATE_CODE / UPDATE_CONFIG
  </script>
</body>
```

### 3.2 渲染流程

```
父窗口发送 UPDATE_MODULE（含 moduleUrl + CSS 列表）
        │
        ▼
iframe 更新 <link> CSS 样式
        │
        ▼
fetch(moduleUrl) → Blob URL 创建 → dynamic import() 加载 ESM 模块
        │
        ▼
获取 module.default → currentComponent = module.default
        │
        ▼
ReactDOM.createRoot().render(
  <ErrorBoundary>
    <currentComponent {...configData} />
  </ErrorBoundary>
)
```

### 3.3 双通道 postMessage 协议

| 通道 | 触发时机 | 是否重新编译 | 是否重置组件状态 |
|:-----|:---------|:-------------|:----------------|
| `UPDATE_MODULE` | AI 修改代码且编译 API 返回 `moduleUrl` | ✅ | ✅（iframe 重新 import） |
| `UPDATE_CODE` | AI 修改代码 | ✅ | ✅（iframe 重新 import） |
| `UPDATE_CONFIG` | 用户修改配置 | ❌ | ❌（仅更新 props） |

- 配置变更不触发重新编译，~16ms 响应
- 代码变更优先通过内容寻址 `moduleUrl` 加载；旧 `compiledCode` 和 `compiledJsUrl` 继续走 `UPDATE_CODE` 兼容路径

### 3.4 错误处理

| 错误阶段 | 捕获方式 | 处理方式 |
|:---------|:---------|:---------|
| 语法错误 | sucrase 编译异常 | API 返回错误行号和列号 |
| runtime 加载失败 | iframe `error` 事件 | postMessage 回传错误详情 |
| React 运行时错误 | ErrorBoundary | 显示友好错误提示，不清空上次成功结果 |
| 模块加载失败 | dynamic import `.catch()` | 回传错误信息 |

### 3.5 依赖一致性保障

iframe 沙箱中可能存在多个 React 实例的问题，v3.0 通过同源 runtime manifest 从根上收敛：

| 层级 | 方案 | 作用 |
|:-----|:------|:-----|
| **Layer 1（核心）** | manifest 固定 specifier → vendor chunk | React、ReactDOM、jsx-runtime、lucide、framer 均使用同一批 ESM 文件 |
| **Layer 2（安全网）** | iframe Import Map | 兜底任何遗留的裸 specifier |
| **Layer 3（预防）** | AI Prompt 补充 React 版本约束 | 减少 AI 使用不兼容库的概率 |

CDN 回退仍保留 `?deps=react@18.3.1,react-dom@18.3.1`，仅在显式回退开关开启时生效。

`react/jsx-runtime` 和 `react/jsx-dev-runtime` 是同源 runtime 的强约束项。页面代码经 sucrase 自动 JSX 转换后，会从 `react/jsx-runtime` 读取 `jsx`、`jsxs` 和 `Fragment`；开发 JSX runtime 需要 `jsxDEV` 和 `Fragment`。`scripts/build-preview-runtime.mjs` 不能只对 React 的 CommonJS JSX runtime 做泛用 `export *` 包装，必须在构建入口里显式重导出这些 named exports。否则浏览器 dynamic import 页面模块时会因为 vendor 文件缺少 named export 而失败，iframe 不会发出 `LOADED`，单页预览和画布预览都会停留在加载态。

构建脚本更新后需要运行 `corepack pnpm build:preview-runtime`，让 author-site 和 viewer-site 的 `public/preview-runtime` 同步生成同一批 vendor 文件和 manifest。回归测试需要覆盖生成产物中 `react-jsx-runtime.js` 暴露 `jsx/jsxs/Fragment`，以及 `react-jsx-dev-runtime.js` 暴露 `jsxDEV/Fragment`。

## 四、性能与缓存

### 4.1 服务端编译缓存

```typescript
const compileCache = new Map<string, CompileResult>();
const MAX_CACHE_SIZE = 100;
```

- 缓存键：代码内容 MD5 hash + `lockedDependencies` JSON
- 上限 100 条，FIFO 淘汰
- 命中缓存直接返回，跳过 sucrase 编译

### 4.2 runtime 资源缓存

| 资源类型 | 缓存策略 | 说明 |
|:---------|:---------|:-----|
| React / ReactDOM | 同源静态资源 HTTP 缓存 | `public/preview-runtime/vendor/*.js` 随构建产出 |
| 第三方库 (lucide-react 等) | 同源静态资源 HTTP 缓存 | manifest 记录 hash 和 bytes，URL 稳定 |
| 编译后代码 | 内容寻址 module URL + Blob URL 兼容 | `/api/preview-modules/{hash}.js` 返回不可变缓存头 |

### 4.3 延迟预算

| 阶段 | 耗时 | 说明 |
|:-----|:-----|:-----|
| 服务端编译 (sucrase) | ~50ms | 纯 JS，与代码长度正相关 |
| runtime URL 替换 | ~5ms | 正则替换，开销极低 |
| 同源 runtime 加载 | ~10-150ms | 取决于本地服务和浏览器缓存 |
| CDN 回退下载依赖 | ~200-500ms | 仅显式回退时发生 |
| iframe 渲染 | ~50-100ms | 组件复杂度决定 |
| **首次冷启动** | **~0.3-0.8s** | 目标区间，取决于本地 runtime 缓存和页面复杂度 |
| **代码变更 (依赖已缓存)** | **~200ms** | 仅重新加载编译后代码 |

### 4.4 阶段耗时观测

预览加载过程分成父窗口和 iframe shell 两层观测：

| 位置 | 阶段 | 说明 |
|:-----|:-----|:-----|
| `PreviewPanel` | `compile_start` / `compile_done` / `compile_error` | 记录编译开始、完成、失败和缓存命中情况 |
| `PreviewPanel` | `parent_update_code_sent` / `parent_update_code_url_sent` | 记录父窗口发送页面模块或 module URL 的时刻、代码体积和 CSS 数量 |
| `PreviewPanel` | `iframe_ready` / `iframe_loaded` | 记录 shell READY 和页面 LOADED 回到父窗口的时刻 |
| iframe shell | `shell_start` / `ready_sent` | 记录 shell 模块脚本启动和 READY 发出 |
| iframe shell | `update_code_received` | 记录 shell 收到页面模块更新 |
| iframe shell | `module_import_start` / `module_import_done` | 记录 Blob URL 或 module URL 导入耗时 |
| iframe shell | `render_invoked` | 记录 React render 已调用 |
| iframe shell | `resource timing snapshot` | 在 `shell_start` 和 `module_import_done` 中附带 preview-runtime、CDN、Blob 模块资源耗时摘要 |
| compile API | `[PreviewRuntime][compile-api]` | 服务端日志记录请求类型、代码长度、依赖数量、CSS 数量和 API 耗时 |

iframe 内部 timing 通过 `CONSOLE_LOG` 回传父窗口，payload 使用 `source: "preview-runtime"` 标识。父窗口会同步打印 `[PreviewRuntime]` 到 DevTools，并继续通过 console buffer 转发给 agent-service。`LOADED` 仍表示页面组件模块已挂载，不等待图片、字体等外部资源全部完成。

排查加载慢时优先按这些日志定位：

- `compile_start` 到 `compile_done` 或服务端 `[PreviewRuntime][compile-api] elapsedMs` 高，说明慢在编译 API 或服务端文件读取。
- `iframe_html_created` 到 `iframe_load_event` 高，通常说明 iframe 文档、阻塞脚本或网络初始化慢。
- `iframe_load_event` 到 `iframe_ready` 高，通常说明 shell 的模块脚本或前置 CDN 资源阻塞。
- `module_import_start` 到 `module_import_done` 高，并且 resource timing 中 `preview-runtime` 项耗时高，说明慢在同源 runtime 静态资源或浏览器模块解析。
- `module_import_start` 到 `module_import_done` 高，并且 resource timing 中 `<cdn>` 项耗时高，说明系统处于 CDN 回退路径。
- resource timing 中 `<jsdelivr>/npm/tailwindcss-cdn...` 耗时高，说明 Tailwind CDN 脚本是首屏瓶颈。

### 4.5 首屏截图占位与生成避让

编辑页打开时，截图 hook 会按页面 id 查询截图 file API 的 `?meta=1` 模式。若本地已有 `currentHash`，hook 立即构造带 hash 的 `placeholderScreenshotUrl` 并填入 `pageScreenshots`，让单页预览在 iframe 真实内容加载前先显示旧截图。

批量截图生成不再随编辑页首次渲染立即启动，而是延后到两个时机之一：

- 单页预览首次收到 `LOADED`，说明主预览链路已经完成首轮加载。
- 用户主动切换到画布模式，画布需要截图占位。

这样截图服务、Puppeteer 和本机 Chrome 不会在首屏与 iframe 运行时同时抢 CPU、网络和浏览器资源。代码或配置变化后仍会按原有 debounce 机制再生单页截图。

### 4.6 runtime 来源与回退

默认 runtime 来源是同源 manifest。CDN base 仍保留给回退路径和 Tailwind CDN：

| 消费方 | 配置来源 | 说明 |
|:-------|:---------|:-----|
| author-site 服务端编译 | `preview-runtime/manifest.json` | `preview-dependency-policy` 默认生成同源 runtime URL |
| author-site 固定 shell | `/api/preview-runtime/shell` | 默认本地 runtime；`runtimeSource=cdn` 可诊断回退 |
| viewer-site | 构建产出的 `public/preview-runtime` | dev 可用固定 shell；production export 使用 inline shell + 当前 origin |
| screenshot-service iframe shell | author-site URL + `PREVIEW_RUNTIME_SOURCE` | module URL 和 runtime base 由 author-site 提供 |
| 发布包 | `/data/{projectId}/preview-runtime` | 发布时复制 runtime，并在 `project.json` 记录 manifest 版本 |

旧发布页面仍可使用其已写入的 iframe HTML 和 compiled JS。新发布页面默认使用随发布包固化的 runtime；如果开启 CDN 回退，则发布页记录 `previewRuntime.source = "cdn"`。

## 五、依赖版本锁定

### 5.1 锁定机制

`ProjectMeta.lockedDependencies` 是旧 CDN 依赖锁定机制，仍用于兼容历史发布和 CDN 回退路径。v3.0 默认依赖版本由 preview runtime manifest 固定：

```typescript
{
  "lockedDependencies": {
    "lucide-react": "https://esm.sh/lucide-react@0.344.0?deps=react@18.3.1,react-dom@18.3.1"
  }
}
```

- `compileCode()` 继续接收 `lockedDependencies` 参数，保持旧调用签名兼容。
- 默认路径优先使用 manifest 中的同源 URL，不再运行时解析 CDN 版本。
- 开启 CDN 回退时，依赖 URL 仍按固定版本和 `?deps` 策略生成。
- 项目发布时会在发布产物 `project.json` 中记录 `previewRuntime.version`、`previewRuntime.source` 和本地 runtime base path。

### 5.2 锁定流程

```
AI 编辑阶段 → compileCode 使用同源 preview runtime manifest
用户点击发布 → 复制 preview-runtime 到发布包 /data/{projectId}/preview-runtime
             → project.json 记录 previewRuntime.version/source/basePath
嵌入/查看阶段 → iframe shell 和 compiled.js 使用同一 runtime base
```

## 六、安全说明

### 6.1 iframe 沙箱隔离

| 隔离维度 | 实现方式 |
|:---------|:---------|
| DOM 隔离 | iframe 独立文档树 |
| CSS 隔离 | iframe 独立样式表，父窗口样式不穿透 |
| JS 隔离 | iframe 独立执行上下文，全局变量不共享 |
| 网络隔离 | `sandbox="allow-scripts allow-same-origin"`，禁止导航和弹窗 |

### 6.2 runtime 依赖安全

- 依赖版本由 `preview-runtime/manifest.json` 固定，manifest 记录 runtime 版本、specifier 映射、文件 hash 和依赖版本。
- runtime vendor chunk 由仓库依赖构建产出，不在预览首屏动态解析远程 ESM 服务。
- 未登记 npm import、相对 import 和非法 lucide named import 会在编译前被契约校验拒绝。
- CDN 依赖路径仅作为显式回退保留，继续使用固定版本和 `?deps` 参数降低多 React 实例风险。
