# 协同草稿驱动预览

> 更新日期：2026-07-11

## 定位

配置与预览模块读取的是当前 Workspace 草稿。实时协同编辑引入后，代码和 Schema 的变化可以先进入协同草稿，再同步到预览相关状态。

## 编辑入口

Workspace 文件编辑弹窗在打开页面代码、页面 Schema、项目级 Schema、页面树或画布布局文件时，会尝试进入协同模式。协同模式下，CodeMirror 不再只依赖本地输入状态，而是由协同文档驱动内容变化。

## 预览同步

当前页面代码或 Schema 发生协同变化时，编辑页通过文件保存回调刷新页面代码缓存、Schema 缓存和预览输入。这样另一个协作者保存的草稿可以反映到当前预览，而不需要等待项目版本保存。

页面树协同变化会刷新编辑页的页面和文件夹结构；当前活跃页面的代码与 Schema 仍由对应页面资源房间驱动，避免远端结构变更污染当前页面内容。

AI 写入或删除 `project.config.schema.json` 时，编辑页需要把它视为会影响配置与预览的 Workspace 变更：重新读取 Session 文件快照、用最新项目级 Schema 合并各页面默认配置，并把项目级 Schema 协同文档同步为文件内容。文件被删除时，协同文档也必须同步为空，避免旧的共享配置草稿重新拉回右侧配置面板。

画布布局协同变化会写入 `.canvas-layout.json` 草稿并更新当前画布状态。远端画布状态应用时不会标记本地画布脏状态；本地拖拽、缩放、节点调整等画布变更会在已有画布持久化之外同步写入协同草稿，供其他协作者实时恢复。

画布远端状态只能在协同文档文本本身变化时应用。编辑页不能因为本地 `canvasState` 更新就重新应用旧的 `.canvas-layout.json` 内容，否则本地缩放、抓手平移或节点拖拽刚写入状态就会被旧远端草稿覆盖，表现为画布无法缩放或拖动。

## 画布页面组

画布页面组是 `.canvas-layout.json` 中的可视化组织信息，用来把多个独立页面折叠成一个当前页面卡片查看。页面组保存自己的标题、目录项、当前激活页面、组布局和目录折叠状态；被收入目录的页面 ID 同时记录在隐藏列表中，避免它们继续作为独立卡片重复展示。

页面组布局只描述当前展示页面卡片的位置和尺寸，不包含目录宽度。目录以左侧悬浮层跟随页面卡片显示，可折叠为只显示页面数量的小按钮；因此合并多个页面后，画布不会出现一个包住多页面的大容器。

页面组在选择模型中按页面类对象处理：选择工具框选时会和普通页面一起进入页面对齐工具栏。对齐写回时，普通页面更新 `pages` 布局，页面组更新自身 `pageGroups` 布局，两者不会互相改写。页面组缩放使用当前激活页面的预览尺寸比例，外框与内容同步等比变化。

页面组不接管页面资源本体。页面代码、Schema、页面树、版本历史、发布数据、AI 上下文和 CLI 页面流程仍按原页面 ID 运转。页面组里的右侧预览只是在画布上复用当前激活页面的预览能力，因此 high-fidelity、HTML/CSS 原型页、草图页、截图层和 iframe 调度仍沿用页面级策略。

移动或缩放页面组时，只更新页面组自己的布局。组内页面原有布局继续保留，便于未来解除分组或从其他入口查看页面时仍使用原位置。

## 画布跨项目复制粘贴

画布支持通过 Ctrl/Cmd+C 和 Ctrl/Cmd+V 快捷键复制粘贴所有可见内容，包括自由节点（文档、图片、文字）、页面卡片和页面组。剪贴板数据通过 `localStorage`（key: `workbench:canvas-clipboard`）在同 origin 下跨项目传输，无需服务端中转。

复制时，`PreviewCanvas` 收集当前选中的节点数据、页面内容（`CanvasPageData` 中的 code/prototypeHtml/sketchScene 等）和页面组信息，序列化后写入 localStorage。粘贴时，自由节点在画布内部直接插入（生成新 ID、偏移位置），页面通过 `onRequestPastePages` 回调通知父组件，由父组件调用 `projectApiClient.createDemoPage` + `updateDemoPageFiles` 在目标项目中创建页面并写入内容，返回旧 ID 到新 ID 的映射供画布更新布局引用。

相关模块：

- `packages/demo-ui/src/canvas-clipboard.ts`：剪贴板数据结构、读写、边界计算
- `packages/demo-ui/src/PreviewCanvas.tsx`：复制/粘贴键盘事件处理
- `packages/author-site/src/app/demo/[id]/edit/page.tsx`：`handlePastePages` 回调，页面创建 API 调用

## 连接生命周期约束

`useCollabDocument` 的 WebSocket provider 生命周期只由协同房间字段决定：`projectId`、`workspaceId`、`sessionId`、`resourcePath` 和 `kind`。调用方可以传入临时构造的 descriptor 对象，但 hook 内部必须先按这些字段稳定 descriptor，再创建或销毁 Yjs provider。

这个约束用于避免父组件普通渲染触发协同连接重建。若 provider 初始化中的状态回写又导致父组件渲染，而 descriptor 仅因对象引用变化被视为新房间，就会形成“渲染 -> 重建 provider -> 状态回写 -> 再渲染”的循环，最终表现为 React `Maximum update depth exceeded`，并伴随大量 `y-websocket` 连接被关闭或限流日志。

## 页面级 Schema 保护

页面级 Schema 在编辑页内需要同时保存在“当前活跃 Schema”和“按页面 ID 索引的 Schema 缓存”中。切换单页预览、画布选中页面或配置面板详情页前，必须先把当前活跃页面的非空 Schema 回写到页面缓存；随后从页面文件或协同草稿加载到的空 Schema 不能覆盖已知非空 Schema。这样可以避免协同连接初始化时的临时空值把页面级配置清空，导致画布选中页面后只剩项目级共享配置。

## 保存语义

实时协同只代表草稿已写入 Workspace。项目版本、发布状态和使用端可见内容仍由项目保存与发布流程控制，避免每一次实时输入都产生历史版本。

在 Workspace Mutation Authority 架构下，编辑态预览允许消费本地或 Yjs 合并草稿，必须在输入和协同更新后立即响应（INV-5）；同时记录 `draftVersion` 和其基于的 `committedRevision`。截图、自动修复判定、历史、发布和导出只消费 Authority 返回的 committed snapshot。草稿预览不能被标记为"已自动保存"，也不能作为发布输入。

“已自动保存”只表示所有当前 dirty 协同房间已通过 mutation 提交，并且 author-site 已观察到对应 committed revision。canonical 物化不再是普通自动保存成功条件（INV-9）。预览投影不是 mutation 事务的一部分，使用独立 `WorkspaceProjectionAck`：预览 applied/failed 状态独立于 mutation receipt，已 committed 的修改不会因预览失败而回滚。

创作端编辑页将“创建版本”放在版本历史区，而不是放在右上角作为保存按钮。用户可以持续编辑并依赖草稿自动保存；需要可回退节点时再手动创建版本。发布前若存在尚未创建版本的草稿，发布按钮会先创建版本再发布。

## 后续扩展

页面配置面板的细粒度字段级协同仍可继续接入同一通道。接入时应保持“草稿实时保存，版本手动生成”的边界不变。
