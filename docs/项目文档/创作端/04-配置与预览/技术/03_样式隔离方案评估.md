> 版本：v1.0
> 创建日期：2026-04-19
> 状态：待评审

## 一、项目背景

### 1.1 当前架构

本项目采用**动态编译 + 直接渲染**方案（已从 Sandpack iframe 迁移）：

| 组件 | 说明 |
|:-----|:-----|
| [compiler-client.ts](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/lib/compiler-client.ts) | 调用 `/api/compile` 将 TSX 编译为 JS |
| [component-executor.ts](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/lib/component-executor.ts) | 通过 `new Function` 执行编译后代码，注入项目依赖 |
| [PreviewPanel.tsx](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/components/demo/PreviewPanel.tsx) | 渲染编译后的 React 组件，支持配置联动 |

### 1.2 核心问题

动态编译的组件直接在主 DOM 树中渲染，**继承了全局样式**（[globals.css](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/app/globals.css)）：

```css
:root {
  --background: 222.2 84% 4.9%;    /* 深色背景 */
  --foreground: 210 40% 98%;       /* 浅色文字 */
}

body {
  @apply bg-background text-foreground;  /* 全局应用 */
}
```

导致 AI 生成的组件样式被污染（如 `text-gray-900` 被覆盖为白色）。

### 1.3 评估目标

寻找最优的样式隔离方案，满足：
1. **完整性**：彻底解决全局样式污染
2. **性能**：配置联动延迟 < 100ms
3. **可维护性**：实现成本低，长期维护简单
4. **兼容性**：不影响现有功能（尺寸配置、错误提示、依赖注入等）

## 二、方案评估

### 方案 A：CSS Reset（all 属性）

#### 2.1 技术原理

```css
.preview-isolated {
  all: initial;  /* 将所有属性重置为 CSS 规范初始值 */
}

.preview-isolated * {
  all: revert;   /* 子元素恢复浏览器默认样式 */
}
```

**作用机制**：
- `all: initial`：切断所有从父元素继承的样式（CSS 变量、颜色、字体等）
- `all: revert`：恢复浏览器 User Agent 默认样式表（`<h1>` 恢复大字号，`<ul>` 恢复列表圆点）

#### 2.2 实现成本

| 维度 | 评估 |
|:-----|:-----|
| 代码改动 | 仅需修改 2 个文件（PreviewPanel.tsx + globals.css），约 15 行代码 |
| 学习成本 | 低，CSS `all` 属性为 W3C 标准，兼容性好 |
| 测试成本 | 低，回归测试现有 Demo 即可 |

#### 2.3 可行性分析

**✅ 优势**：

| 优势 | 说明 |
|:-----|:-----|
| **完整隔离** | 重置所有 CSS 属性，覆盖颜色、字体、列表、表单等全部污染场景 |
| **Tailwind 完全生效** | 重置后 Tailwind 类名具有最高优先级 |
| **零侵入** | 不修改任何 Demo 组件代码 |
| **性能无损** | 纯 CSS 方案，无额外运行时开销 |
| **配置联动保持毫秒级** | 组件仍在主 React 树中，props 变更直接触发渲染 |

**⚠️ 局限性**：

| 局限 | 影响 | 应对策略 |
|:-----|:-----|:---------|
| 全局 CSS 变量不可用 | 组件若使用 `var(--primary)` 将失效 | 组件应使用硬编码值或自行定义变量 |
| 全局字体不可继承 | 字体变为浏览器默认 | 组件应显式声明 `font-sans` 等类名 |
| 伪元素样式需手动处理 | `::before`/`::after` 可能被 reset | 在 `.preview-isolated *` 规则中补充 |

#### 2.4 长期可维护性

| 维度 | 评估 |
|:-----|:-----|
| 维护成本 | **低**，纯 CSS 规则，无需额外依赖 |
| 扩展性 | **中**，若未来需要主题透传，需额外开发 |
| 风险 | **低**，`all` 属性为 CSS 标准，浏览器兼容性 > 95% |

#### 2.5 适用场景

- ✅ 内部工具，AI 生成的组件代码可控
- ✅ 需要毫秒级配置联动响应
- ✅ 追求实现简单、维护成本低

### 方案 B：Shadow DOM

#### 2.6 技术原理

```tsx
class ShadowPreview extends HTMLElement {
  connectedCallback() {
    const shadow = this.attachShadow({ mode: 'open' });
    shadow.innerHTML = `
      <style>/* 独立样式 */</style>
      <div id="root"></div>
    `;
    // 使用 ReactDOM.render 挂载到 shadow root
  }
}
```

**作用机制**：
- Shadow DOM 创建独立的 DOM 子树，外部样式无法穿透
- 内部样式不影响外部文档

#### 2.7 实现成本

| 维度 | 评估 |
|:-----|:-----|
| 代码改动 | **高**，需重写 [component-executor.ts](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/lib/component-executor.ts)，改造 React 渲染逻辑 |
| 学习成本 | **中高**，需掌握 Shadow DOM API、React Portal、样式注入机制 |
| 测试成本 | **高**，需覆盖 Shadow DOM 边界场景 |

#### 2.8 可行性分析

**✅ 优势**：

| 优势 | 说明 |
|:-----|:-----|
| **最强隔离** | 浏览器原生隔离，样式 100% 不互相影响 |
| **样式可控** | 可在 Shadow Root 内注入 Tailwind CDN 或自定义样式 |

**⚠️ 局限性**：

| 局限 | 影响 | 严重程度 |
|:-----|:-----|:---------|
| **React 18 不支持 Shadow DOM** | `ReactDOM.createRoot` 不接受 Shadow Root 作为容器 | 🔴 阻塞 |
| **事件穿透问题** | Shadow DOM 内的事件冒泡到 Light DOM 时会丢失路径 | 🟡 需 workaround |
| **全局 API 失效** | `document.querySelector` 无法查询 Shadow DOM 内部 | 🟡 需适配 |
| **样式注入复杂** | 需在 Shadow Root 内重新注入 Tailwind、shadcn/ui 样式 | 🟡 增加包体积 |
| **DevTools 调试困难** | React DevTools 默认不显示 Shadow DOM 内的组件树 | 🟡 降低开发效率 |

**🔴 关键技术阻塞**：

React 18 的 `createRoot` API 明确要求容器必须是 `Element`（非 Shadow Root）。虽然可通过 `ReactDOM.render`（React 17 API）或自定义 Renderer 绕过，但：
- 放弃 Concurrent Mode 特性
- 未来 React 升级可能彻底移除旧 API
- 社区方案（如 `react-shadow`）成熟度低，维护不活跃

#### 2.9 长期可维护性

| 维度 | 评估 |
|:-----|:-----|
| 维护成本 | **高**，需持续维护 Shadow DOM 适配层 |
| 扩展性 | **低**，与 React 生态存在天然冲突 |
| 风险 | **高**，React 官方未提供 Shadow DOM 支持，未来不确定性大 |

#### 2.10 适用场景

- ❌ 不推荐用于 React 项目
- ✅ 仅适用于 Web Components 或原生 JS 项目

### 方案 C：iframe 沙箱

#### 2.11 技术原理

```tsx
<iframe
  style={previewStyle}
  sandbox="allow-scripts"
  srcDoc={`
    <!DOCTYPE html>
    <html>
      <head>
        <script src="https://cdn.tailwindcss.com"></script>
      </head>
      <body>
        <div id="root"></div>
        <script>
          // 注入编译后的代码并执行
        </script>
      </body>
    </html>
  `}
/>
```

**作用机制**：
- iframe 创建完全独立的浏览器上下文
- 通过 `sandbox` 属性限制权限
- 通过 `postMessage` 与父窗口通信

#### 2.12 实现成本

| 维度 | 评估 |
|:-----|:-----|
| 代码改动 | **极高**，需重写整个 [PreviewPanel.tsx](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/components/demo/PreviewPanel.tsx)，改造编译流程、通信机制 |
| 学习成本 | **中**，需掌握 iframe 安全策略、postMessage 通信、跨域限制 |
| 测试成本 | **极高**，需覆盖通信延迟、跨域错误、沙箱权限等场景 |

#### 2.13 可行性分析

**✅ 优势**：

| 优势 | 说明 |
|:-----|:-----|
| **最强隔离** | 完全独立的 JS 执行环境、样式、DOM |
| **安全性高** | `sandbox` 属性可限制脚本权限 |
| **样式自由** | 可在 iframe 内独立加载 Tailwind CDN 或任何样式方案 |

**⚠️ 局限性**：

| 局限 | 影响 | 严重程度 |
|:-----|:-----|:---------|
| **配置联动延迟高** | props 变更需通过 `postMessage` 传递，iframe 内需重新编译渲染 | 🔴 1-3s 延迟 |
| **首屏加载慢** | iframe 需独立加载 HTML、JS、CSS | 🔴 2-5s 白屏 |
| **通信复杂** | 需实现双向 `postMessage` 协议，处理消息序列化、错误重试 | 🟡 增加代码复杂度 |
| **依赖注入困难** | iframe 内无法直接使用项目的 shadcn/ui 组件，需重新打包注入 | 🟡 增加包体积 |
| **调试困难** | 需单独打开 iframe DevTools，无法使用主窗口 React DevTools | 🟡 降低开发效率 |
| **回退到 Sandpack** | 本质上与已废弃的 Sandpack 方案类似，重蹈覆辙 | 🔴 历史教训 |

**🔴 核心性能问题**：

iframe 方案的配置联动链路：
```
用户修改配置 → postMessage → iframe 接收 → 重新生成 entryCode → 
重新编译 → ReactDOM.render → 预览更新
```

单程 `postMessage` 延迟约 10-50ms，加上 iframe 内重新编译（sucrase 约 100-500ms），总延迟 **1-3 秒**，远超目标值 < 100ms。

#### 2.14 长期可维护性

| 维度 | 评估 |
|:-----|:-----|
| 维护成本 | **高**，需维护通信协议、沙箱权限、错误处理 |
| 扩展性 | **中**，可支持第三方组件预览，但成本高 |
| 风险 | **中**，iframe 为成熟技术，但性能瓶颈难以优化 |

#### 2.15 适用场景

- ✅ 第三方不可信代码预览（如插件市场）
- ✅ 需要 100% JS 执行隔离
- ❌ 不推荐用于内部工具（性能损耗过大）

## 四、推荐方案

### 4.1 首选：CSS Reset（all 属性）

**推荐理由**：

1. **完美契合当前架构**
   - 项目已从 Sandpack 迁移到动态编译方案，组件直接在主 React 树中渲染
   - CSS Reset 无需改动渲染流程，保持毫秒级配置联动

2. **实现成本最低**
   - 仅需修改 2 个文件，约 15 行代码
   - 无需引入新依赖或改造现有架构

3. **长期可维护性强**
   - 纯 CSS 方案，无运行时依赖
   - `all` 属性为 W3C 标准，浏览器兼容性 > 95%
   - 未来 React 升级不受影响

4. **覆盖全部污染场景**
   - 颜色、字体、列表、表单、边框等全部重置
   - Tailwind 类名完全生效

**实施步骤**：

详见 [预览区样式隔离方案.md](file:///e:/重要文件/Programming/1_Work/opencode工作台/docs/plans/预览区样式隔离方案.md)

### 4.2 备选：iframe 沙箱（特定场景）

**适用条件**：
- 未来需要支持第三方不可信代码预览
- 对配置联动延迟不敏感（> 1s 可接受）

**实施建议**：
- 保留 CSS Reset 作为默认方案
- 新增 `isolated="iframe"` 属性，按需启用 iframe 模式
- 参考已废弃的 Sandpack 方案，避免重复踩坑

### 4.3 不推荐：Shadow DOM

**原因**：
- React 18 不原生支持 Shadow DOM
- 社区方案成熟度低，维护成本高
- 与 React 生态存在天然冲突

## 五、风险评估

### 5.1 CSS Reset 方案风险

| 风险 | 概率 | 影响 | 应对措施 |
|:-----|:-----|:-----|:---------|
| 组件依赖全局 CSS 变量 | 低 | 中 | 文档说明，要求组件使用硬编码值 |
| 伪元素样式异常 | 低 | 低 | 在 CSS 规则中补充 `::before`/`::after` 处理 |
| 浏览器兼容性 | 极低 | 低 | `all` 属性兼容 Chrome 37+、Firefox 27+、Safari 9.1+ |

### 5.2 回退策略

如 CSS Reset 方案出现问题，可快速回退：

1. 移除 `.preview-isolated` 类名
2. 删除 globals.css 中的 CSS Reset 规则
3. 恢复原有样式（可能存在污染，但功能正常）

**回退耗时**：< 5 分钟
