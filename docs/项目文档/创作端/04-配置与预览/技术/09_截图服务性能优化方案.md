# 截图服务性能优化方案

> 更新日期：2026-06-29  
> 适用模块：创作端配置与预览、画布模式、项目缩略图、screenshot-service

## 1. 方案定位

本方案用于评估和优化现有 screenshot-service 主链路，让所有 Web 用户在不安装扩展、客户端或额外本地工具的前提下，更快拿到可信截图。

现有截图服务的基础契约仍以 [截图服务与预览快照机制](./07_截图服务与预览快照机制.md) 为准：截图必须对应当前代码、配置、运行时资源和预览尺寸；截图不可用时，具备实时预览能力的消费方继续使用 iframe。性能优化不能牺牲“最新真实优先”原则，也不能让旧截图冒充当前页面。

可以把截图链路理解成一条流水线：前端决定哪些页面需要截图，author-site 同源代理请求，screenshot-service 编译页面并驱动 Chromium 渲染，最后把 PNG 和元数据写入缓存。长期最佳方案不是单点加速某个动作，而是让这条流水线少做无效工作、优先服务用户正在看的页面，并在缓存和降级边界上保持可信。

## 2. 当前代码现状评估

当前代码已经具备一部分性能和一致性基础，后续方案应在这些事实之上演进，避免重复建设。

| 能力 | 当前状态 | 评估 |
| :--- | :--- | :--- |
| author-site 同源代理 | 浏览器只访问 `/api/screenshots/*`，author-site 转发到 screenshot-service | 合理，保留 |
| 前端版本过滤 | `useScreenshotGeneration` 用 `expectedHash` 过滤旧批次结果 | 合理，后端调度不能替代它 |
| 画布渲染调度 | `demo-ui` 已按页面数量、可见区域、选中页和截图可用性切换 iframe、截图、休眠 iframe 与 loading | 合理，性能优化不应重写画布显示规则 |
| 批量截图状态 | 批量任务后台执行，状态接口逐页返回结果，单页失败不阻塞其他页 | 合理，但内部仍按请求数组顺序执行 |
| 批量取消 | 新批次或卸载时会取消旧批次 | 有收益，但当前只能阻止后续队列，不能可靠中断已进入 Chromium 渲染的任务 |
| 浏览器复用 | `BrowserPool` 复用单个 Browser，单任务仍 `newPage` 后关闭 | 浏览器级复用已实现；page 池尚未实现，属于高风险后续项 |
| 队列限流 | `BrowserPool` 按 `maxConcurrentPages` 控制并发并记录 `queueWaitMs`，并已支持 priority 排队 | 合理，后续继续观察优先级分布 |
| 同 hash 合并 | 同一 `projectId/pageId/hash` 的并发请求复用同一个生成 Promise | 高 ROI，保留 |
| 编译缓存 | `CompileCache` 按 code 与 session scope 做 LRU | 已有基础；后续重点是命中率观测和缓存身份审计 |
| 截图缓存 | 文件按 `pageId.hash.png` 存储，hash URL 使用 immutable 缓存 | 合理，但快速/严格截图不能直接覆盖同一 hash 文件 |
| renderBox | 截图服务返回 fullPage 渲染盒，画布只消费兼容的 fullPage 截图 | 合理，是避免黑边和尺寸错位的关键 |
| 资源预热 | 画布侧已提取图片 URL 并预热图片 | 合理；后端可借鉴资源指纹，但不应重复构造复杂资源系统 |
| 指标 | 单页返回阶段耗时；批量状态已返回 `priorityStats` 和 `metrics` | 有基础，后续可接入长期监控与 P90/P99 聚合 |

### 2.1 2026-06-29 实施状态

首轮实施已经覆盖 P0/P1 中风险最低、收益最高的链路：

- 批量截图请求支持 `priority`，可表达 `active`、`visible`、`nearby`、`thumbnail` 和 `background`。
- screenshot-service 会按 priority 排序批量任务，并把 priority 继续传给 `BrowserPool`；浏览器池内部队列也按 priority 插队，同优先级保持原始顺序。
- 批量状态接口返回 `priorityStats` 和 `metrics`，用于观察各优先级完成情况、队列等待、阶段耗时、渲染数、截图缓存命中、编译缓存命中和 in-flight 合并次数。
- author-site 会透传截图 priority；编辑页把当前编辑/选中页标为 `active`，把画布当前可见页标为 `visible`，其余页面标为 `background`。
- `PreviewCanvas` 会把当前可见页面集合回传给 author-site，仅用于截图调度，不改变画布渲染规则。
- screenshot-service 新增 `SCREENSHOT_WARMUP=true` 可选启动预热，服务启动后提前拉起 Browser，默认关闭以保持兼容。
- 截图产物已经区分 strict 与 fast：strict 继续使用 `pageId.hash.png`，fast 使用 `pageId.hash.fast.png` 和 `variant=fast` URL，不覆盖 immutable strict 文件。
- fast 截图保留根节点等待和运行时错误检查，但缩短 network idle、字体和图片稳定等待；active/visible 页面优先请求 fast，其余页面继续请求 strict。
- 截图 hash 已纳入 session/resource scope、Preview Runtime 来源、runtime base URL 和 CDN base URL，避免不同运行时或资源作用域复用同一截图。
- 截图请求可携带 `measuredHeight`，fast 路径会把它作为 fullPage 高度提示；strict 路径仍以服务端测量为准，只在高度一致时减少稳定轮询。

这意味着“当前可见页优先”“性能基线”“缓存身份审计”和“fast/strict 产物分层”已经具备基础实现。后续重点转向 iframe shell 缓存收益评估，以及在指标证明有必要时评估 page/context 复用。

## 3. 方案合理性结论

原方向整体合理：先建立性能基线，再做当前可见页优先、等待策略分层、缓存优化和批量渐进完成。这些都符合现有架构。

需要调整的重点有四个：

1. Page 池不是第一优先级。当前已经复用 Browser，真正的 page 复用会引入 DOM、storage、请求监听器、service worker 和内存污染风险。它应排在观测、调度、缓存身份审计和等待策略之后。
2. 快速截图不能覆盖同一 immutable URL。当前 `?hash=` 文件按不可变资源缓存。如果快速截图与严格截图使用同一 hash 路径，浏览器或 CDN 可能长期保留快速图。快速/严格分层必须先拆开“内容版本 hash”和“截图产物 variant”。
3. 前端测量高度只能作为提示。画布 iframe 的高度可以帮助减少重复测量，但截图服务仍必须自行校验，因为服务端 Chromium 的字体、图片、DPR 和运行时状态可能与当前浏览器不同。
4. 文件写入异步化不是当前高 ROI 主线。当前写入已经使用临时文件再替换，并且清理旧文件在后台进行。除非指标证明 `writeMs` 占比高，否则不应优先改写存储层。

## 4. 长期设计原则

| 原则 | 说明 |
| :--- | :--- |
| 最新真实优先 | 页面变化后先回到实时 iframe；截图只在版本、尺寸和 renderBox 可信时替换 iframe |
| 内容身份与产物身份分离 | 内容 hash 表示代码、配置、资源作用域和尺寸；截图产物还应区分快速/严格、renderBox 版本和渲染策略 |
| 少做无效工作 | 取消旧批次、合并同 hash 请求、跳过已有可信缓存，优先处理当前可见页 |
| 先观测再优化 | 没有基线时不改等待策略和 page 复用，避免把偶发变快误判为整体收益 |
| 配置开关独立 | 调度、快速模式、预热、page 池都应可独立启停，便于灰度和回退 |
| 失败不阻断编辑 | 截图慢、失败或离线时，画布和单页预览继续使用实时 iframe |

## 5. 高 ROI 演进路线

### 5.1 P0：补齐性能基线和缓存可信度

第一阶段不改变用户行为，只让瓶颈可见。现有单页结果已经返回阶段耗时，后续应把这些数据聚合到批量任务和服务日志中。

需要补齐的观测维度：

- 单页与批量的 P50、P90、P99 总耗时。
- `compileMs`、`renderMs`、`writeMs`、`queueWaitMs` 占比。
- 编译缓存命中率、截图文件缓存命中率、同 hash in-flight 合并次数。
- 按 `fullPage`、尺寸、页面数量、批次大小统计耗时。
- `SELECTOR_TIMEOUT`、`RENDER_TIMEOUT`、`RUNTIME_ERROR`、`QUEUE_TIMEOUT` 的数量和占比。
- 批量任务中当前可见页完成时间，而不只是整批完成时间。

同时要做一次缓存身份审计。当前截图 hash 已包含代码、配置、宽高、fullPage、snapshotVersion 和 renderBox 版本，但还需要确认运行时来源、session 资源作用域、图片资源变化和 module URL 是否会导致“代码没变但真实渲染变了”。如果真实渲染会变，缓存身份也必须跟着变，否则提高缓存命中率会放大旧图风险。

### 5.2 P1：当前可见页优先，先减少排队浪费

这是短期最高 ROI 的优化。它不改变截图质量，也不触碰 Chromium 复用边界，只改变任务顺序。

前端已有画布可见区域和页面渲染模式判断，因此可以先用现有接口做请求排序：当前编辑页、当前视口页和即将进入视口的页面排在请求数组前面，远处页面排在后面。由于服务端 batch 当前按数组顺序取任务，这个改动可以立即降低用户感知等待。

后端再补可选优先级字段，让排序从“前端请求顺序”演进为显式契约。

| 优先级 | 使用场景 | 处理策略 |
| :--- | :--- | :--- |
| `active` | 当前编辑页或当前选中页 | 最先入队；旧批次同页任务可降级或取消 |
| `visible` | 当前视口内页面 | 高优先级，保障首屏体感 |
| `nearby` | 即将进入视口的页面 | 中优先级，滚动前预热 |
| `thumbnail` | 项目卡片或非画布缩略图 | 稳定优先，允许等待 |
| `background` | 远处页面补齐 | 空闲时执行，可被新批次打断 |

取消语义要写清楚：取消旧批次只保证未开始的任务不再执行；已进入 Chromium 渲染的任务允许自然结束，但结果仍由 `expectedHash` 和 batchId 过滤，不能回写到当前页面。

### 5.3 P1：快速截图与严格截图分层，但先修正产物模型

快速截图的价值是缩短当前可见页从 iframe 切到轻量截图态的时间；严格截图的价值是作为长期缓存和缩略图来源。两者可以共享同一个内容版本，但不能共享同一个 immutable 产物 URL。

建议模型：

| 概念 | 含义 | 稳定性 |
| :--- | :--- | :--- |
| 内容 hash | 代码、配置、尺寸、资源作用域和快照版本 | 用于判断是不是当前页面 |
| 截图 variant | `fast` 或 `strict`，代表等待策略和质量 | 用于区分产物 URL |
| renderBox 版本 | 测量规则版本 | 变化时强制失效 |
| 产物 URL | 内容 hash 加 variant 后得到的可缓存文件 | 可以继续使用 immutable |

接口兼容策略：

- 旧调用方不传 `renderMode` 时继续走严格截图，响应结构保持兼容。
- 新调用方传 `renderMode=fast` 时，响应增加 `quality`、`variant` 和 `assetUrl`，并返回能区分快速产物的 URL。
- 画布消费快速截图时仍校验内容 hash、variant 和 renderBox；项目卡片默认只消费严格截图。
- 后台严格截图完成后，可以通过新的严格产物 URL 替换快速产物；不要覆盖同一个 immutable 文件路径。当前实现采用“active/visible 先 fast、background 后 strict”的自然补齐策略。

快速模式可以减少等待，但不能跳过关键正确性检查：

- 必须等待根容器出现。
- 必须读取并上报运行时错误，不能用截图掩盖报错。
- fullPage 截图必须返回可信 renderBox；缺少 renderBox 时不能交给画布。
- 可以缩短 network idle 和图片等待，但资源未稳定时只能作为 `fast` 产物，不得冒充严格截图。

### 5.4 P1：浏览器预热，而不是马上 page 池

当前第一张截图可能承担 Chromium 启动成本。比 page 池更稳的短期优化是浏览器预热：

- 服务启动后可选预启动 Browser。
- 健康检查可按配置触发最小渲染，验证 Chromium 真能工作。
- 记录冷启动截图耗时与预热后首张耗时差异。

这类优化风险低，因为它不复用页面状态。只有当指标证明 `newPage` 成本占比高，且预热收益不足时，才进入 page 池。

### 5.5 P2：page 池和 context 复用

page 池属于长期能力，目标是减少 `newPage`、设置 viewport 和注入 HTML 的成本。它的前提是隔离策略可靠。

建议先评估 context 级复用，再评估 page 级复用。page 级复用必须具备：

- 每次任务前后清空 DOM、localStorage、sessionStorage、IndexedDB 和事件监听影响。
- 清理请求拦截、console 监听和错误监听。
- 每个 page 设置最大复用次数和最大存活时间。
- 监控内存、失败率和跨页面状态污染。
- 配置开关可以一键退回“每任务 newPage”。

如果 page 复用导致错误率上升或内存持续增长，应优先保留 Browser 级复用，不继续追求极限速度。

### 5.6 P2：iframe shell 与编译缓存增强

当前编译缓存已经按 code 与 session scope 复用编译结果。后续不应笼统地“再做编译缓存”，而应拆清楚缓存对象。

| 缓存对象 | 命中条件 | 收益 | 风险控制 |
| :--- | :--- | :--- | :--- |
| 编译结果 | code 与 session scope 一致 | 少调 author-site `/api/compile` | 已有基础，补命中率观测 |
| iframe shell | 编译结果、配置、runtime 来源、baseOrigin 一致 | 少做 HTML 拼装 | 必须包含 config 和 runtime 指纹 |
| 资源指纹 | 图片列表、资源作用域、页面尺寸一致 | 缩短等待资源稳定时间 | 只作为等待提示，不替代真实加载 |
| module URL | author-site 返回可复用 module URL | 少传大段 JS 字符串 | 当前编译结果已支持 module URL，重点是确认生命周期 |

缓存优化的目标不是让命中率无限高，而是在“不会复用错资源”的前提下减少重复编译和重复拼装。

### 5.7 P2：前端测量高度作为提示

画布实时 iframe 已经能上报内容高度。对于刚刚在当前浏览器里真实显示过的页面，前端可以把测量高度随截图请求传给服务端，作为 fullPage 高度候选。

服务端处理规则：

- 有前端高度时，仍做一次服务端测量。
- 两边差异在阈值内时，可以减少稳定轮询轮次。
- 差异超过阈值时，以服务端测量为准。
- 缺少前端高度、快速模式或资源不稳定时，沿用服务端完整测量。

这样可以减少重复等待，但不会把浏览器端测量结果直接当成截图事实。

### 5.8 P2：批量任务分片和动态提权

当前状态接口已经支持逐页返回，下一步是让批量任务更像“多个小片逐步完成”，而不是单个大队列。

建议策略：

- 创建批量任务时按优先级拆分为 active、visible、nearby、background 几个分片。
- 状态接口继续逐页返回，同时暴露各优先级分片的完成度。
- 用户滚动或选中远处页面时，可以提交一个高优先级小批次，把对应页面提到队列前面。
- 低优先级页面只在空闲时补齐，不能挤占当前可见页。

这能降低首屏截图等待，同时保留“最终整批补齐”的能力。

## 6. 接口扩展建议

后续实现应保持旧接口兼容，新增字段全部可选。

| 字段 | 方向 | 作用 | 兼容策略 |
| :--- | :--- | :--- | :--- |
| `priority` | 前端到截图服务 | 表示 active、visible、nearby、thumbnail 或 background | 不传时按当前数组顺序 |
| `renderMode` | 前端到截图服务 | 选择 strict 或 fast | 不传时 strict |
| `measuredHeight` | 前端到截图服务 | 提供 iframe 已测得高度提示 | 只作为 hint |
| `resourceScope` | 前端或 author-site 到截图服务 | 标识 session、workspace 或资源作用域 | 用于缓存身份审计后再落地 |
| `quality` / `variant` | 截图服务到前端 | 标记 fast 或 strict 产物 | 旧调用方可忽略 |
| `assetUrl` | 截图服务到前端 | 返回已包含 hash 与 variant 的实际图片 URL | 新快速模式调用方优先使用 |
| `retryAfterMs` | 截图服务到前端 | 队列繁忙时提示轮询节奏 | 不传时沿用固定轮询 |

旧调用方不传新字段时，服务继续执行当前严格截图路径，返回当前结构。

## 7. 验收指标

建议先收集基线，再设定目标。第一轮目标应以用户感知和正确性为主。

| 指标 | 目标方向 |
| :--- | :--- |
| 当前视口首张可用截图 P90 | 明显下降 |
| 当前视口全部截图完成 P90 | 明显下降 |
| 批量任务整批完成 P90 | 不上升，理想情况下下降 |
| 队列等待 P90 | 下降 |
| 编译缓存命中率 | 上升或保持稳定 |
| 截图文件缓存命中率 | 上升或保持稳定 |
| Chromium 冷启动首张耗时 | 预热后下降 |
| 截图错误率 | 不上升 |
| `RUNTIME_ERROR` 可见性 | 不下降，不能被快速截图吞掉 |
| 旧图误展示 | 必须为 0 |
| iframe 降级可用性 | 不下降 |

关键验收不是“某一次截图最快”，而是用户在画布中编辑、滚动、缩放时，当前区域更快进入轻量截图态，同时仍然能看到当前真实页面。

## 8. 推荐实施顺序

| 顺序 | 阶段 | ROI | 风险 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 性能基线与日志聚合 | 高 | 低 | 先知道慢在哪里 |
| 2 | 前端按可见性排序批量请求 | 高 | 低 | 利用现有后端数组顺序，最快见效 |
| 3 | 后端显式 priority 调度 | 高 | 中 | 让优先级成为稳定接口契约 |
| 4 | 缓存身份审计 | 高 | 中 | 防止优化放大旧图风险 |
| 5 | 浏览器预热 | 中 | 低 | 降低冷启动成本 |
| 6 | fast/strict 分层与 variant 产物 | 高 | 中 | 必须先修正 immutable URL 问题 |
| 7 | iframe shell 缓存与 measuredHeight hint | 中 | 中 | 收益依赖指标 |
| 8 | context/page 池 | 中 | 高 | 只有指标证明值得时再做 |

不建议把浏览器扩展截图作为短期性能优化主线。扩展仍属于 [浏览器扩展截图增强远期方案](./10_浏览器扩展截图增强远期方案.md) 的可选增强能力，不能替代主链路。

## 9. 风险与回退

| 风险 | 处理方式 |
| :--- | :--- |
| 快速截图被当成严格截图长期缓存 | 内容 hash 与产物 variant 分离，不覆盖同一 immutable URL |
| 优先级调度导致远处页面长期不生成 | 空闲时补齐 background 分片，并暴露分片完成度 |
| 取消旧批次后仍有旧任务完成 | 前端继续用 `expectedHash` 和 batchId 过滤，后端不承诺中断已渲染任务 |
| 前端测量高度不准 | 前端高度只作为 hint，最终以服务端测量为准 |
| page 复用造成状态污染 | 默认关闭 page 池，设置复用次数、生命周期和快速回退开关 |
| 缓存命中率提高但出现旧图 | 立即回退缓存扩展，优先修复缓存身份模型 |
| 指标变好但错误率上升 | 回退等待策略或关闭快速模式 |

每个阶段都应可以通过配置开关独立启停。上线顺序应先观测和调度，再做等待策略分层；最后才考虑 page 池这类状态复用能力。
