# 截图服务与预览快照机制

> 更新日期：2026-06-25  
> 适用模块：创作端配置与预览、项目卡片缩略图、画布模式、多页面截图缓存

## 1. 机制定位

截图服务是预览系统的通用快照能力，不属于画布专用模块。它为项目卡片缩略图、页面快照、画布多页面展示等场景提供一致的截图生成、缓存和读取链路。

截图服务输出的是某个内容版本的静态快照；实时预览仍由 iframe 链路负责。任何消费方在使用截图前，都必须先确认截图版本与当前代码、配置和预览尺寸一致。

### 1.1 最新真实优先原则

预览区域中的页面必须尽快展示当前代码和配置对应的真实效果。截图、缓存、占位和 iframe 调度都只能服务于这个目标，不能让用户误以为旧页面就是最新页面。

| 原则 | 约束 |
| :--- | :--- |
| 最新优先 | 页面代码或配置变更后，预览区域必须立即切到当前版本的实时 iframe，直到最新截图可用 |
| 真实一致 | 预览内容必须来自同一份编译链路和运行时配置，不能用简化渲染替代真实代码效果 |
| 快速可见 | 配置变更后优先用最快路径展示真实效果，性能优化不得延后最新页面可见时间 |
| 禁止旧图 | 截图 hash 与当前内容版本不一致时，必须移除或隐藏该截图，不能以旧图、旧图加状态或生成中占位替代真实页面 |

### 1.2 消费路径

需要展示页面效果的消费方通常具备两条显示路径：

| 显示路径 | 使用场景 | 用户感知 |
| :--- | :--- | :--- |
| 截图路径 | 截图服务在线，且截图 hash 与当前页面内容版本一致 | 多页面画布更轻、更稳定 |
| 实时 iframe 路径 | 正在编辑、截图未生成、截图版本不匹配、截图失败或需要立即查看最新效果 | 页面仍按真实代码效果预览和编辑 |
| 休眠 iframe 路径 | 页面近期加载过但暂时超出活跃 iframe 预算 | 保留 iframe DOM，唤醒时少一次完整重建 |
| 轻量加载态 | 截图不可用且同屏实时 iframe 数量已超过预算 | 暂时降低浏览器压力，等待用户靠近或截图补齐 |

这意味着“截图服务不可达”不能阻断实时预览使用。它只会让消费方从截图优化回退到实时 iframe 或不展示缩略图，不能显示过期截图，也不能用生成中状态替代真实页面。

## 2. 服务协作关系

浏览器不直接访问截图服务。所有截图相关请求都先发给 author-site 的同源 API，再由 author-site 服务端转发到 screenshot-service。

```
浏览器
  -> author-site /api/screenshots/*
    -> screenshot-service /api/screenshots/*
      -> Puppeteer/Chromium 渲染页面截图
      -> data/screenshots 写入截图缓存
```

这样做有三个目的：

- 避免浏览器端把 `localhost:3202` 解析成用户本机，导致局域网或 Docker 场景下不可达。
- 避免让截图服务暴露给前端跨域调用，降低 CORS 配置复杂度。
- 让 author-site 可以统一处理健康检查、错误响应和文件读取回退。

## 3. author-site 代理契约

author-site 提供以下同源入口：

| author-site 入口 | 转发目标 | 说明 |
| :--- | :--- | :--- |
| `GET /api/screenshots/health` | `GET /health` | 返回统一 `{ success: true, data }`，用于前端判断截图服务是否在线 |
| `POST /api/screenshots/generate` | `POST /api/screenshots/generate` | 单页截图生成 |
| `POST /api/screenshots/generate-batch` | `POST /api/screenshots/generate-batch` | 多页面批量截图生成 |
| `GET /api/screenshots/status/:projectId/:batchId` | `GET /api/screenshots/status/:projectId/:batchId` | 批量任务轮询 |
| `POST /api/screenshots/cancel/:projectId/:batchId` | `POST /api/screenshots/cancel/:projectId/:batchId` | 取消尚未完成的批量任务，常用于重新发起截图或离开页面 |
| `GET /api/screenshots/file/:projectId/:pageId` | `GET /api/screenshots/file/:projectId/:pageId` | 优先代理截图文件，404 或服务不可达时回退读取 author-site 本地缓存 |

截图服务不可达时，author-site 返回统一错误：

| 字段 | 值 |
| :--- | :--- |
| `success` | `false` |
| `error.code` | `SCREENSHOT_SERVICE_UNAVAILABLE` |
| `error.message` | `截图服务不可达` |

前端只需要识别该错误码，不需要判断底层是端口未启动、Docker DNS 不通，还是 Chromium 启动失败。

当 author-site 已经连上截图服务但下游请求超时时，会返回 `SCREENSHOT_PROXY_TIMEOUT`。前端同样按“截图服务暂不可用”处理：停止当前轮询、清除非当前版本截图。具备实时预览能力的消费方应回退到 iframe；不具备实时预览能力的消费方应停止展示该截图。

## 4. 前端状态流

`useScreenshotGeneration` 负责维护每个页面的截图状态：

| 状态字段 | 含义 |
| :--- | :--- |
| `screenshotUrl` | 当前内容版本可用截图地址 |
| `hash` | 当前 `screenshotUrl` 对应的内容哈希 |
| `expectedHash` | 本轮截图请求期望返回的内容哈希，用于过滤旧批次或旧请求结果 |
| `renderBox` | 截图服务实际捕获的渲染盒元数据，包含布局宽度、完整内容高度、初始视口和测量高度 |
| `loading` | 当前是否正在生成或轮询截图 |
| `error` | 当前页面最近一次截图失败原因 |
| `serviceAvailable` | 截图服务健康状态，`null` 表示尚未确认 |

关键规则：

- 页面内容版本变化、启动批量截图或单页重试时，必须清除与当前版本不一致的 `screenshotUrl`。具备实时预览能力的消费方立即回到 iframe；不具备实时预览能力的消费方不展示旧截图。
- 批量截图创建后，前端记录每页服务端返回的 `expectedHash`；轮询结果的 `hash` 不一致时不更新页面截图，避免旧批次晚返回覆盖新内容。
- 截图展示地址使用 `GET /api/screenshots/file/:projectId/:pageId?hash=<hash>`；`?t=` 时间戳只用于旧兼容路径，不作为主链路。
- 截图服务不可达时，将 `serviceAvailable` 标记为 `false`；是否回退 iframe 由消费方是否具备实时预览能力决定。
- 截图生成成功后，更新对应页面的 `screenshotUrl` 和 `renderBox`，并清除该页错误状态。
- 画布消费截图时，除了校验内容 `hash`，还必须校验 `renderBox` 与当前画布渲染基准兼容；缺少 `renderBox` 的旧截图不能替换实时 iframe。
- 批量任务通过 `status` 轮询更新每个页面，单页失败不会阻塞其他页面。
- 批量轮询存在最长等待时间和连续失败上限；达到上限后会停止轮询并标记服务离线。
- 重新发起批量截图前会取消旧批次，避免旧任务继续占用截图队列。

## 5. 截图消费规则

所有消费方都遵守以下通用规则：

1. 只有截图 hash 与当前内容版本一致时，才能展示截图。
2. 截图版本不匹配、尚未生成、加载中或生成失败时，必须移除该截图。
3. 旧截图不得作为过渡内容展示；一旦发现截图不是当前版本，必须从显示路径中移除。

不同消费方按自身能力选择后续动作：

| 消费方 | 截图不可用时的行为 |
| :--- | :--- |
| 单页面预览 | 继续使用实时 iframe |
| 画布页面对象 | 立即使用实时 iframe，保证用户最快看到当前配置和代码的真实效果 |
| 项目卡片或只读缩略图 | 不展示旧截图，可展示非页面内容的空态或缩略图缺省外框 |

画布组件在通用规则基础上按以下顺序决定每个页面如何显示：

1. 当前页面被选中：页面以白色描边高亮，并渲染实时 iframe；如果已有可用截图，截图先继续覆盖，直到 iframe 上报内容加载完成后再渐隐切换，降低用户感知。
2. 非选中页面有可用截图、截图已加载，且截图 hash 与当前页面内容版本一致：优先显示截图。
3. 非选中页面截图版本不匹配、尚未生成、加载中或生成失败：优先使用实时 iframe，保证用户最快看到当前配置和代码的真实效果。
4. 当页面总数少于 6 个时，画布不使用截图替代 iframe；即使已有可用截图，也直接让所有页面进入实时 iframe 路径。
5. 当页面总数达到 6 个及以上，且可见区域内需要实时 iframe 的页面过多时，画布最多保留 12 个活跃 iframe；超出活跃预算但近期加载过的页面最多保留 12 个休眠 iframe。
6. 休眠 iframe 保留 DOM，但父层不再推送配置更新或尺寸采集请求，iframe 内部也跳过主动测量；页面重新进入活跃态时，父层会补发最新代码与配置。
7. 既不在活跃预算内、也不在休眠保留池内的页面展示轻量加载态，待用户靠近、选中或截图补齐后再升级显示。

画布会为活跃候选页和休眠页预热图片资源。预热只处理图片：系统从页面配置和代码字符串中提取图片 URL，把会话工作区内的相对图片路径转成同源资源地址，并按页面代码、图片列表和预览尺寸生成资源指纹。相同指纹只预热一次；预热失败不会阻断 iframe 渲染。

画布页面初始宽高由页面 `config.schema.json` 中的 `$demo.previewSize` 决定。画布模式下，`$demo.previewSize.width` 是页面内容布局宽度；`$demo.previewSize.height` 只作为初始视口高度，不代表最终页面高度。当真实内容高度超过预览视口时，画布卡片高度会按 iframe 上报高度或截图服务返回的 `renderBox.height` 扩展到完整页面。

画布截图生成使用整页截图。截图服务会在资源和字体稳定后测量 body 与 document 的完整高度，并返回 `renderBox`。画布只在截图 `hash` 与当前内容版本一致、`renderBox.fullPage` 为 true、且 `renderBox.width` 与 `$demo.previewSize.width` 对齐时，才用截图替换实时 iframe。

截图图片加载完成只表示静态载体可以显示，不能再用图片 `naturalWidth` 或 `naturalHeight` 反向改写画布布局。画布布局高度必须来自 iframe 测量或截图服务 `renderBox`，避免截图态和 iframe 态使用不同尺寸口径。

截图在画布中按页面宽度铺满并从顶部开始显示；不能用居中缩放模式展示截图，否则当卡片高度与图片比例暂时不一致时会露出上下黑边，造成截图态与 iframe 态视觉不一致。

当截图服务离线时：

- `CanvasToolbar` 显示“截图服务离线，正在使用实时预览”。
- 工具栏提供重试入口，重试会先调用健康检查，再触发批量截图。
- `CanvasPageItem` 在页面右上角显示轻量状态，如“使用实时预览”和“重试”。
- 页面本身不显示阻断式错误卡片，用户仍可拖拽、缩放、选择页面并编辑配置。

## 6. screenshot-service 职责

screenshot-service 是独立 Fastify 服务，默认端口 `3202`。它负责：

- 接收单页或批量截图请求。
- 调用编译能力生成 iframe HTML。
- 使用 Puppeteer/Chromium 渲染 PNG。
- 按 `projectId` 与 `pageId` 写入截图缓存与元数据。
- 对外返回本次截图的内容 `hash`，让前端用 hash 精确读取图片，而不是读取“当前页面截图”的活动别名。
- 通过进程内队列限制同时打开的 Chromium 页面数；超过并发上限的任务会排队，而不是直接失败。
- 对同一页面同一内容 hash 的并发请求进行合并，避免重复编译和重复渲染。
- 使用临时文件写入截图，成功后再替换正式文件和元数据，避免半写入文件被读取。
- 提供 `/health` 用于 author-site 和运维检查；可通过深度检查验证 Chromium 是否真的能完成一次最小渲染。

浏览器渲染依赖本机或容器内 Chromium。服务启动时会按以下顺序寻找浏览器：

1. `PUPPETEER_EXECUTABLE_PATH`
2. macOS Chrome 默认路径
3. Linux Chromium/Chrome 默认路径
4. Windows Chrome/Edge 默认路径

本地 Windows 开发环境不需要额外配置时，应优先使用自动探测路径；如果机器安装路径特殊，再显式设置 `PUPPETEER_EXECUTABLE_PATH`。

## 7. Hash 寻址一致性契约

截图缓存同时保留两种读取方式：

| 读取方式 | 用途 | 缓存策略 |
| :--- | :--- | :--- |
| `GET /api/screenshots/file/:projectId/:pageId?hash=<hash>` | 主链路，读取指定内容版本 | 长缓存，可视为不可变资源 |
| `GET /api/screenshots/file/:projectId/:pageId` | 历史兼容，读取 meta 中记录的 current 版本 | 不长期缓存 |

关键约束：

- 带 `hash` 的请求只读取 `pageId.hash.png`；文件不存在时返回 404，不回退到 `currentHash`。
- author-site 在 screenshot-service 不可达时，本地文件回退也遵守同样规则，避免服务降级时显示旧图；如果无法确认 hash 属于当前版本，不返回该截图给前端显示。
- 前端拿到 hash 图片 404 或加载失败时，具备实时预览能力的消费方回退 iframe，不具备实时预览能力的消费方停止展示截图；都不能改读 current alias。
- `currentHash` 仍用于兼容旧工具和排查，不再作为判断“当前代码截图”的依据。

## 8. 本地开发与 Docker

本地开发建议使用：

```bash
pnpm dev:preview
```

该命令只启动 author-site 与 screenshot-service，适合验证创作端预览、截图生成和截图消费链路，不依赖历史 snapshot 包。

Docker 场景下，author-site 使用服务端环境变量访问容器内部服务：

| 变量 | 建议值 | 说明 |
| :--- | :--- | :--- |
| `SCREENSHOT_SERVICE_URL` | `http://screenshot-service:3202` | author-site 服务端访问截图服务 |
| `NEXT_PUBLIC_SCREENSHOT_SERVICE_URL` | 不建议依赖 | 浏览器端不再直连截图服务 |
| `PUPPETEER_EXECUTABLE_PATH` | `/usr/bin/chromium` | screenshot-service 容器内 Chromium 路径 |

关于 Docker 服务编排与环境变量，请见 [Docker部署方案](../../06-基础设施/技术/03_Docker部署方案.md)。

## 9. 排查顺序

当截图相关功能出现问题时，按以下顺序排查：

1. 打开 `GET /api/screenshots/health`，确认 author-site 代理是否返回成功。
2. 打开 `GET http://localhost:3202/health`，确认本地截图服务是否启动。
3. 检查 screenshot-service 日志，确认 Chromium 是否可启动。
4. 对具备实时预览能力的消费方，检查是否仍显示实时 iframe；如果 iframe 也不可用，应转向实时预览链路排查，参见 [02_实时预览机制](./02_实时预览机制.md)。
5. 如果健康检查成功但批量截图一直未完成，检查 `/api/screenshots/status/:projectId/:batchId` 中的 `errorsByCode`、`cancelled`、`updatedAt` 与截图服务日志中的 `requestId`。
6. 如果截图与代码不一致，检查页面状态中的 `expectedHash`、状态接口返回的 `hash` 和图片 URL 的 `hash` 是否一致；三者不一致时应优先排查旧批次回写或前端状态更新路径。
