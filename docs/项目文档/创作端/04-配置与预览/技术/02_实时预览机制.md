## 一、实时预览概述

### 1.1 核心机制

实时预览通过以下机制实现：

| 机制       | 说明                                             |
| :--------- | :----------------------------------------------- |
| 服务端编译 | 通过 `/api/compile` 将 TSX 编译为 CommonJS       |
| 前端执行   | 通过 `new Function` 执行编译后代码，注入项目依赖 |
| 直接渲染   | `React.createElement` 直接渲染，无需 iframe      |
| 编译缓存   | 相同代码不重复编译，提升加载速度                 |

### 1.2 更新流程

```
代码变更
    │
    ▼
触发 compileCode()
    │
    ▼
检查缓存 / 调用 API 编译
    │
    ▼
executeComponent() 执行
    │
    ▼
React.createElement 渲染
    │
    ▼
预览区更新
```

## 二、配置联动

### 2.1 联动机制

配置变更与预览的联动流程：

```
用户修改配置表单
        │
        ▼
ConfigForm.onChange(formData)
        │
        ▼
父组件更新 configData 状态
        │
        ▼
PreviewPanel 接收新 configData
        │
        ▼
React.createElement(compiledComponent, configData)
        │
        ▼
预览区更新渲染（无需重新编译）
```

**关键设计点**：

- 配置变更**不触发重新编译**
- 已编译的组件函数保持不变，仅更新 props
- 联动延迟 < 16ms（一帧内响应）

### 2.2 性能优化

| 优化措施    | 说明                                      |
| :---------- | :---------------------------------------- |
| React.memo  | PreviewPanel 使用 memo 避免不必要的重渲染 |
| useCallback | 配置变更回调使用 useCallback 缓存         |
| 编译缓存    | 相同代码不重复请求 API                    |

```
        │

```

### 3.2 AI 编辑快照驱动（snapshotVersion）

当 AI 通过对话编辑代码时，采用 **snapshotVersion 驱动**机制保证预览与配置的强一致性：

```
AI 编辑完成（SSE session.idle）
        │
        ▼
useChatStream.onFinish
        │
        ├─ 优先使用 result.files（drain 机制保证完整）
        ├─ 兜底 fetchSessionFiles() HTTP 获取
        └─ 触发 onSnapshotReady 回调
        │
        ▼
page.tsx — applyDemoSnapshot()
        │
        ├─ 原子化更新：code / schema / editorContent
        ├─ 合并 configData 默认值
        └─ snapshotVersion++
        │
        ▼
PreviewPanel 编译 effect 触发
  依赖: [code, sessionId, validCode, sendUpdateCode, effectiveVersion]
        │
        ▼
compileCode(code) → executeComponent() → 渲染
        │
        ▼
ConfigForm key 变化 → 强制重新挂载
  key={`${activeDemoId}-${snapshotVersion}`}
```

**关键设计点**：

- **snapshotVersion 是唯一驱动源**：所有 AI 编辑产生的变更（code + schema）通过一次原子更新应用，然后递增版本号
- **effectiveVersion 计算**：`const effectiveVersion = snapshotVersion ?? compileVersion`，向后兼容旧代码
- **编译 effect 依赖**：`[code, sessionId, validCode, sendUpdateCode, effectiveVersion]`，版本变化触发重新编译
- **ConfigForm 重建**：`key={\`${activeDemoId}-${snapshotVersion}\`}` 确保表单完全重建，应用新 schema 默认值

### 3.3 竞态处理

```typescript
useEffect(() => {
  let cancelled = false;

  compileCode(code)
    .then((result) => {
      if (cancelled) return; // 防止竞态
      const Component = executeComponent(result.compiledCode);
      setCompiledComponent(() => Component);
    })
    .catch((err) => {
      if (cancelled) return;
      setCompileError(err.message);
    })
    .finally(() => {
      if (!cancelled) setIsCompiling(false);
    });

  return () => {
    cancelled = true;
  }; // cleanup
}, [code]);
```

## 四、AI 编辑联动

### 4.1 背景与问题

AI 通过 WebSocket 编辑代码时，与用户手动编辑有本质区别：

| 维度     | 用户手动编辑     | AI 编辑                                  |
| :------- | :--------------- | :--------------------------------------- |
| 触发方式 | 编辑器 onChange  | WebSocket 消息                           |
| 编辑范围 | 通常只改代码     | 可能同时改代码 + Schema（含 default 值） |
| 状态感知 | 用户知道改了什么 | 系统需自动检测变更内容                   |

**典型问题场景**：AI 同时修改了 `index.tsx` 中的默认标题和 `config.schema.json` 中的 `default` 值，预览区更新了，但配置面板仍显示旧值。

### 4.2 根因分析

问题的根源在于 **configData 状态滞留**：

```
AI 编辑前：
  configData = { title: "旧标题" }  ← 用户之前在配置面板设置的值
  schema.default = "旧标题"

AI 编辑后：
  schema.default = "新标题"         ← AI 更新了默认值
  configData = { title: "旧标题" }  ← 但 configData 没变！

渲染时：
  props = { ...newDefaults, ...configData }
        = { title: "新标题", title: "旧标题" }
        = { title: "旧标题" }       ← 旧值覆盖了新默认值！
```

**为什么颜色变更能正常更新？** 因为颜色值通常只在代码中硬编码，不经过 configData，所以不存在覆盖问题。而标题等文本内容既在代码中有默认值，又在 configData 中有用户设置的值，两者冲突时旧 configData 胜出。

### 4.3 解决方案

#### 机制一：代码变更时重置 configData

当检测到代码内容变更（而非仅配置变更）时，将 configData 重置为空对象，让 Schema 中的新默认值生效：

```typescript
// page.tsx — handleCodeUpdate 中
if (schema) {
  const result = validateAll(newCode, schema);
  setValidationResult(result);

  // 代码变更时重置 configData 为空，让组件默认值生效
  setConfigData({});
}
```

**设计考量**：重置为 `{}` 而非保留旧值，是因为代码变更意味着组件的"基线"发生了变化，旧配置值可能不再适用。这是一种"信任新代码"的策略。

#### 机制二：默认值合并顺序

当 Schema 变更带来新的默认值时，确保新默认值优先于旧 configData：

```typescript
// ❌ 错误：旧 configData 覆盖新默认值
setConfigData((prev) => ({ ...newDefaults, ...prev }));

// ✅ 正确：新默认值覆盖旧 configData
setConfigData((prev) => ({ ...prev, ...newDefaults }));
```

**设计考量**：`{ ...prev, ...newDefaults }` 的含义是"以 Schema 的最新默认值为准，但保留用户之前设置过、且 Schema 未变更的配置项"。这样既能让 AI 的默认值更新生效，又不会丢失用户的其他配置。

### 4.4 AI 编辑完整数据流

```
AI 发送编辑指令（通过 WebSocket）
        │
        ▼
agent-service 执行编辑，写入文件
        │
        ▼
WebSocket 推送 file_operation 事件
        │
        ▼
ai-chat.tsx 接收事件，触发 onFileChange 回调
        │
        ▼
page.tsx — handleCodeUpdate(newCode, newSchema)
        │
        ├─→ 代码变了？ → setConfigData({})        ← 重置配置
        │
        ├─→ Schema 变了？ → 合并新默认值            ← 更新默认值
        │       setConfigData(prev => ({...prev, ...newDefaults}))
        │
        └─→ setCode(newCode) + setSchema(newSchema)
                │
                ▼
        PreviewPanel 重新编译 + 渲染
        ConfigForm 接收新 configData + 新 Schema
```

### 4.5 关键原则

| 原则         | 说明                                               |
| :----------- | :------------------------------------------------- |
| 信任新代码   | 代码变更时重置配置，以新默认值为准                 |
| 新默认值优先 | 合并时 `{ ...prev, ...newDefaults }`，新值覆盖旧值 |
| 保留用户配置 | 仅覆盖 Schema 中变更的字段，其他用户设置保留       |
| 双重保障     | 重置 configData + 合并顺序，两道防线确保新值生效   |

## 五、预览尺寸控制

### 5.1 类型定义

```typescript
interface PreviewSize {
  width?: string | number;
  height?: string | number;
  minHeight?: string | number;
  maxHeight?: string | number;
  scale?: number;
}
```

### 5.2 配置方式

在 JSON Schema 中通过 `$demo.previewSize` 字段配置：

```json
{
  "$demo": {
    "previewSize": {
      "width": "100%",
      "height": "600px",
      "minHeight": "400px",
      "maxHeight": "800px"
    }
  }
}
```

### 5.3 样式生成

```typescript
function buildPreviewStyle(size?: PreviewSize): React.CSSProperties {
  const effectiveSize = size ?? DEFAULT_PREVIEW_SIZE;

  const style: React.CSSProperties = {
    width: effectiveSize.width,
    height: effectiveSize.height,
    minHeight: effectiveSize.minHeight ?? "400px",
    margin: "0 auto",
    border: "1px solid #e5e7eb",
    borderRadius: "8px",
    overflow: "hidden",
  };

  if (effectiveSize.maxHeight !== undefined) {
    style.maxHeight = effectiveSize.maxHeight;
  }

  if (effectiveSize.scale !== undefined) {
    style.transform = `scale(${effectiveSize.scale})`;
    style.transformOrigin = "top center";
  }

  return style;
}
```

## 八、宫格预览机制

### 8.1 架构概述

宫格预览（GridView）用于在配置面板中同时展示多个 Demo 页面，每页以卡片形式排列。与单页预览（PreviewPanel）不同，宫格预览使用 **iframe 渲染 + CSS Grid 布局**，支持不同尺寸页面按比例并行展示。

**核心组件**：

| 组件              | 说明                                        |
| :---------------- | :------------------------------------------ |
| `PreviewGrid.tsx` | 宫格容器，管理布局渲染、对齐、懒加载        |
| `GridCard.tsx`    | 单张卡片封装，含 iframe、选中状态、变更提示 |
| `GridIframe.tsx`  | iframe 渲染封装，支持缩放和滚动条控制       |

**渲染结构**：

```
containerRef (div.h-full.overflow-y-auto + 隐藏滚动条)
  └─ div(min-h-full.p-4.flex.flex-col + justifyContent 动态)
       └─ div(display: flex.flex-col.gap-16)
            └─ row × N (display: grid; gridTemplateColumns: 按比例fr; gap: 16px)
                 └─ card × M (aspectRatio: 各自宽高比; GridIframe 懒加载)
```

### 8.2 按行 Grid 渲染

#### 8.2.1 分组算法

将页面列表按 `gridColumns` 分组为行：

```typescript
const rows: GridPageItem[][] = [];
for (let i = 0; i < demoPages.length; i += gridColumns) {
  rows.push(demoPages.slice(i, i + gridColumns));
}
```

#### 8.2.2 列宽比例计算

每行的 `gridTemplateColumns` 基于该行卡片的宽高比动态生成：

```typescript
const ratios = row.map((p) => {
  const size =
    p.id === activePageId ? previewSize : (p.previewSize ?? previewSize);
  return getAspectRatioValue(size); // width / height
});
const columnTemplate = ratios.map((r) => `${r}fr`).join(" ");
// 例如: "0.462fr 1.333fr" → 手机卡: iPad 卡 = 0.462:1.333
```

**数学原理**：同一行中，每张卡片宽度与其宽高比成正比。假设卡片宽高比分别为 r₁、r₂，则 `gridTemplateColumns: r₁fr r₂fr`。每行实际宽度 = Σ(cardHeight × ratio) = availableWidth。因此 cardHeight = availableWidth / Σratio，cardWidth = cardHeight × ratio = availableWidth × ratio / Σratio。

**示例**：同一行 iPad（1024×768，比值 1.333）和手机（375×812，比值 0.462）：

- `gridTemplateColumns: 1.333fr 0.462fr`
- iPad 卡片宽度占比：1.333 / (1.333 + 0.462) = 74.3%
- 手机卡片宽度占比：0.462 / (1.333 + 0.462) = 25.7%

#### 8.2.3 高度规则

同一行所有卡片自然等高，由 CSS Grid 默认的 `align-items: stretch` 保证。无需额外约束。

### 8.3 垂直对齐

#### 8.3.1 判断逻辑

使用 `useAlignmentMode` Hook，通过 `ResizeObserver` 同时观察滚动容器和 grid 元素高度：

```
if (gridScrollHeight + padding < containerClientHeight) {
  align = 'center'   // 居中显示
} else {
  align = 'top'      // 顶部对齐 + 滚动
}
```

#### 8.3.2 实现方式

```tsx
<div ref={containerRef} className="h-full overflow-y-auto preview-grid-scroll">
  <div
    className="min-h-full p-4 flex flex-col"
    style={{
      justifyContent: alignmentMode === "center" ? "center" : "flex-start",
    }}
  >
    <div
      ref={gridMeasureRef}
      style={{ display: "flex", flexDirection: "column", gap: "16px" }}
    >
      {/* rows mapping */}
    </div>
  </div>
</div>
```

- 外层 `flex-col` wrapper 提供垂直对齐能力
- `ResizeObserver` 同时观察 `containerRef` 和 `gridMeasureRef`
- 动态切换 `justify-content: center | flex-start`

### 8.4 滚动条隐藏

**外层容器**：

```tsx
<div ref={containerRef} className="h-full overflow-y-auto preview-grid-scroll"
  style={{ scrollbarWidth: "none", msOverflowStyle: "none" }}>
  <style>{`.preview-grid-scroll::-webkit-scrollbar { display: none; }`}</style>
```

**iframe 内部**：

```typescript
function disableIframeScrollbar(iframe: HTMLIFrameElement) {
  try {
    const doc = iframe.contentDocument;
    if (!doc) return;
    doc.documentElement.style.overflow = "hidden";
    doc.body.style.overflow = "hidden";
  } catch {}
}
```

### 8.5 活跃页面规则

```typescript
const effectiveSize =
  page.id === activePageId
    ? previewSize // 活跃页面：使用 grid-level 的当前配置尺寸
    : (page.previewSize ?? previewSize); // 非活跃页面：使用自身尺寸或回退到 grid-level
```

**设计考量**：

- `previewSize` 是用户在配置面板设置的当前尺寸，切换页面时更新
- 活跃页面使用配置面板尺寸，而非页面自身的 `previewSize`
- 非活跃页面使用各自独立尺寸，实现每页不同预览效果

### 8.6 懒加载

使用 `useVisiblePages` Hook 结合 Intersection Observer 实现：

| 机制     | 说明                                                    |
| :------- | :------------------------------------------------------ |
| 触发条件 | 卡片 DOM 元素进入视口                                   |
| 观察目标 | `containerRef` 的子元素 `[data-page-id]`                |
| 回退策略 | 不可见卡片使用占位 div 保持布局空间（aspectRatio 不变） |
| 共存性   | 与 `useAlignmentMode` 监听不同元素，互不干扰            |

### 8.7 卡片渲染

| 功能         | 实现                                                               |
| :----------- | :----------------------------------------------------------------- |
| iframe 缩放  | `cardWidth / iframeWidth` 动态 scale，确保卡片内内容适配           |
| 选中高亮     | `border-2 border-primary ring-2` + `scale-[1.02]`                  |
| 变更小橙点   | `hasChanges` 状态 → absolute `<span>` 定位在卡片右上角             |
| 预览尺寸解析 | `parseSizeValue` 处理字符串类型（`"375px"` → `375`）和 number 类型 |
