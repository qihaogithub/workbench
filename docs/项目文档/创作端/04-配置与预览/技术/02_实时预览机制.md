# 预览系统 - 实时预览机制

> 版本：v2.13
> 创建日期：2026-04-06
> 更新日期：2026-07-01
> 更新说明：补充 preview contract AST 化解析边界，避免 type-only import、字符串注释符号、helper 空返回和重复 var 造成系统误报；明确单页面预览与画布工作台的文档正文按需读取和首轮截图批处理范围，避免拖慢预览首屏；补充画布页面框尺寸语义与历史布局兼容规则

## 一、实时预览概述

### 1.1 核心机制

实时预览通过以下机制实现：

| 机制       | 说明                                             |
| :--------- | :----------------------------------------------- |
| 服务端编译 | 通过 `/api/compile` 将 TSX 编译为 CommonJS       |
| iframe 执行 | `PreviewPanel` 将编译结果通过 postMessage 发送到预览 iframe |
| 预览渲染   | iframe 模板在隔离文档中执行编译后代码并渲染组件 |
| 编译缓存   | 相同代码不重复编译，提升加载速度                 |
| 应用图执行 | 页面通过 `@preview/sdk` 触发业务动作，viewer shell 根据应用图切页、传参和更新运行时状态 |

### 1.2 预览区模式

当前预览区只有两种工作模式：

| 模式 | 承载组件 | 适用场景 | 关键边界 |
| :--- | :--- | :--- | :--- |
| 单页面预览 | `PreviewPanel` / `CanvasDocumentContent` | 查看当前 Demo 页面或文档，编辑当前 Demo 页面 | 页面目标承载实时 iframe、配置联动、可视化批注和局部编辑入口；文档目标只读展示 Markdown |
| 画布工作台 | `PreviewCanvas` | 多页面总览、整理、定位和布局调整 | 承载页面卡片、素材节点、画布视口、拖拽缩放、最新截图与 iframe 兜底 |

宫格预览不再作为独立预览模式存在。多页面查看统一进入画布工作台；需要对某一页做细节编辑时，再回到单页面预览。

单页面预览内部使用独立的目标状态区分“页面目标”和“文档目标”。页面目标继续以 `activeDemoId` 为核心，驱动当前代码、Schema、配置面板和 `PreviewPanel`；文档目标不改写 `activeDemoId`，只替换预览区正文为只读 Markdown 文档。这样用户可以在单页模式中阅读说明文档，同时避免文档阅读影响页面保存、配置联动和属性面板上下文。

单页面预览目录由预览区右上角的下拉选择承载。下拉项使用页面和文档两类目标值区分：页面条目点击复用现有页面加载流程并更新 `activeDemoId`；文档条目点击只切换单页目标，并清空当前 DOM 图层选择。当前目标是文档时，右侧配置和属性面板不渲染，图层按钮保持可见但禁用。

多页面预览的数据来源必须以页面 ID 为边界。页面尺寸、配置、代码和截图任务参数应从同一个页面身份解析，不能因为某页代码尚未加载，就借用当前活动页的编辑器代码。切页过程中，系统可以先使用页面自身已缓存的代码；若无法确认当前编辑器代码归属目标页，应保持该页为空等待态或使用截图占位，直到目标页代码加载完成。这样可以避免“页面 A 的尺寸展示页面 B 内容”的跨页串用。

创作端从单页模式进入画布工作台时，需要检查画布页面集合中哪些页面尚未加载代码，并按页面 ID 从当前 session 补齐代码、Schema、默认配置和预览尺寸。这个补齐过程只更新页面缓存，不切换当前活动编辑页，避免用户正在查看的单页编辑器被其他页面内容打断。若补齐前画布页面已经挂载，`PreviewPanel` 对空代码显示“等待页面代码加载”占位，而不是继续显示预览加载 spinner；只有拿到代码或模块 URL 后，预览加载态才代表 iframe 正在编译或等待 `LOADED` 回执。

`PreviewPanel` 复用固定 iframe shell 时，每一次代码或模块 URL 更新都必须生成新的请求版本。父窗口发送代码、配置和测量指令时带上当前请求版本；iframe 在回传加载完成、运行错误、内容高度和可定位元素尺寸时带回同一版本。父窗口只接受当前版本的运行时消息，旧页面晚到的 `LOADED`、`RESIZE` 或尺寸测量结果必须被丢弃。这样可以避免旧页面异步加载完成后把新页面的加载态解除，造成新页面尺寸里露出旧页面内容。

画布工作台中的页面卡片通过 `PreviewPanel` 的 `fillContainer` 模式承载实时 iframe。页面框由画布布局决定，iframe 内容仍按页面 `previewSize` 渲染后等比缩放到页面框内。`PreviewPanel` 必须使用当前容器的正宽高计算缩放，并在父级页面框变化后主动重新量测；在 `CanvasViewport` 使用 CSS transform 缩放整张画布时，容器量测必须使用未受 transform 影响的布局尺寸，不能把 `getBoundingClientRect()` 返回的屏幕尺寸再次用于 iframe 内部缩放。ResizeObserver 上报的 0 宽高不能覆盖已有正尺寸。宽高比不一致时，iframe 的居中偏移使用父容器坐标中的实际像素值，不能再按缩放比例二次折算。画布截图层和单页截图占位层也必须以页面框为容器进行 `object-contain` 贴合，避免截图按自然尺寸显示而和蓝色页面选中框脱节。

画布页面框默认代表当前页面真实预览画板尺寸，而不是历史任意容器尺寸。页面布局状态使用 `sizeMode` 和 `previewSizeKey` 区分尺寸来源：默认 `preview` 尺寸跟随当前 `$demo.previewSize`，用户手动 resize 后写入 `custom` 尺寸，右键重置大小恢复 `preview`。读取历史 `.canvas-layout.json` 时，如果页面布局没有尺寸语义标记且宽高与当前 `previewSize` 不一致，系统只保留位置和层级，重新使用当前 `previewSize` 作为页面框尺寸。内容高度超过设计高度时，非 custom 页面只允许在当前 `previewSize.width` 下拉长高度，不能用过期的保存宽度推导高度，避免错误尺寸被截图或 iframe 高度回传继续放大并持久化。

请求版本只能在确实存在可发送的预览源时创建。若当前 render 只是代码尚未补齐、模块 URL 尚未准备好或页面切换中的空代码过渡，`PreviewPanel` 必须进入 no-request 状态，而不能抢占新的请求版本。否则上一份真实代码对应的 iframe `LOADED` 会被误判为过期，画布卡片就会停留在预览加载态。

预览请求生命周期只允许由代码、模块 URL、session、页面 ID、URL 模式或外部 CSS 列表这类真实预览源变化驱动。错误处理、控制台采集等父级回调需要通过最新 ref 读取，不能作为编译 effect 的重启条件；否则编辑页切换焦点或右侧面板状态变化会在页面已经显示后重置 `contentLoaded` 并创建新请求版本，旧 iframe 的 `LOADED` 回执被过滤后会再次形成永久加载。

### 1.3 页面源码契约

创作端页面源码契约由 `@opencode-workbench/preview-contract` 统一维护，是 author-site 编译器、project-core/CLI 校验和 Agent prompt 的共同规则源。该包按运行环境拆分入口：`rules` 入口只包含依赖策略、契约版本和 Agent 创作规则，可被浏览器可达代码引用；`runtime` 入口包含页面源码校验、错误类型和 import 重写，依赖 TypeScript 静态解析器，只能在 Node 侧执行；`compiler` 入口包含 Sucrase 转换和内容 hash 生成，只能用于服务端编译链路。

author-site 仍然是最终预览入口，继续负责 runtime URL、manifest、CDN 与同源 vendor 的解析。共享 contract 只负责判断页面源码是否符合创作端可运行约束，并复用与 author-site 一致的 Sucrase TSX/JSX 转换策略生成编译产物。

页面 authoring 源码必须保持单文件 React 源码形态，不能提交已经手写 `react/jsx-runtime` 的编译产物。运行时相对源码 import、未登记 npm import、无可渲染默认导出和默认渲染入口直接返回空内容都会在 contract 阶段被报告为页面级 issue。`import type` 这类只服务 TypeScript 类型检查的声明会被编译器擦除，不参与运行时依赖判断；页面 helper 或嵌套函数内部的 `return null` 不代表默认组件空白，也不作为阻断项。编译产物模式允许出现 `react/jsx-runtime`，用于区分“源码契约”和“编译结果”。

页面写入后的 fast gate 分为三层：源码契约、TSX/JSX 编译转换、ESM 模块静态预检。源码契约以单个页面模块为边界，允许不同页面各自使用 `page`、`theme`、`accentMap` 等普通顶层变量名；只有同一个页面模块内部重复声明会触发浏览器导入阶段失败的同名值绑定时，才报告 `DUPLICATE_TOP_LEVEL_DECLARATION`。模块预检只解析编译产物并检查浏览器导入阶段必然失败的问题，例如多个默认导出、重复拼接块、词法绑定冲突或系统生成绑定冲突；重复 `var` 声明本身不属于 ESM 导入阶段错误。预检不通过动态 `import()` 或 iframe 执行用户页面代码。若冲突来自 JSX runtime 转换、编译器包装或 import 重写后的生成产物，诊断使用系统生成模块冲突语义，不能提示用户批量改不同页面里的普通变量名。诊断统一使用 `PreviewGenerationDiagnostic` / `RuntimeContractIssue` 结构，包含 `stage`、`code`、`message`、`instruction` 和页面定位信息，供创作端自动修复、CLI JSON 输出和 Agent 工具反馈复用。

preview contract 的源码识别应基于 TypeScript AST，而不是对源码做注释剥离和正则搜索。字符串中的 `//`、多行字符串、注释里的 import 文本、type-only import 和 JSX/TSX 语法都必须按语法树语义处理，避免把普通页面内容误当成依赖、默认导出或空渲染信号。

project-core 和 Project Admin CLI 在页面 create/update、事务校验、提交、项目包导出、模板产出和发布前复用同一套 preview contract。当前事务新增或修改页面的 runtime diagnostics 会作为 blocking issue；历史未改页面的问题在事务校验中降级为 warning，避免无关旧问题阻断当前编辑。CLI JSON 结果会在页面写入成功后附加 `runtimeValidation`，因此文件可以落盘供 Agent/Codex 继续修复，但命令结果不会被误判为预览正常。

`PREVIEW_DEPENDENCY_POLICY` 从 `rules` 入口导出，author-site 的 `preview-dependency-policy.ts` 只保留运行时 URL 解析和 SDK 虚拟模块生成职责，避免 CLI、Agent prompt 与预览编译器各自维护依赖白名单。浏览器可达的 Agent prompt 代码只能导入 `@opencode-workbench/preview-contract/rules`，不能通过 `runtime` 间接把 TypeScript 编译器对象带入客户端 bundle。

浏览器可达模块不得从 contract 总入口、`runtime` 入口或 `compiler` 入口导入能力。总入口会继续面向 Node 侧兼容导出完整能力；前端 bundle 只能使用 `@opencode-workbench/preview-contract/rules` 读取轻量规则，避免把 TypeScript、`node:crypto`、Sucrase 等服务端依赖带入 Webpack 客户端编译。

author-site 的 Next 配置需要把 `typescript` 作为服务端组件外部依赖处理，作为 Node 侧 runtime 被误纳入 RSC/client 分析时的防线。调整 preview contract 的解析器依赖或入口分层时，需要同步检查 Next dev server 的浏览器 console，确保没有 `runtime.ts` 指向的 TypeScript 导出警告或 hydration 阶段 dev overlay 运行时错误。

### 1.4 画布素材节点

画布工作台除 Demo 页面卡片外，还承载文档和图片两类素材节点。素材节点用于补充说明和图片素材，不改变 Demo 页面本身的代码与配置。外部网页插入功能已删除，画布不再提供 URL 输入、网页预设尺寸或外部 iframe 素材节点。

文档节点在画布中只负责浏览 Markdown 渲染结果；进入编辑时必须复用共享组件 `DocumentEditor` 的 Markdown 模式。文档不维护独立标题字段，保存时从 Markdown 第一行派生画布节点标题。这样文档节点、工作区 Markdown 文件、知识库文档和计划审批编辑都走同一套工具栏、序列化和预览能力，避免后续出现手写 textarea 与预览逻辑不一致的问题。

多个文档节点可以合并为一个聚合文档节点。聚合节点仍然是画布素材节点，只保存多个知识库文档的轻量引用和当前激活文档 ID；左侧目录来自这些引用的标题，右侧正文仍按当前目录项读取知识库 Markdown。合并不会创建新的工作区 Markdown 文件，也不会把多篇正文拼成一份长期内容。

画布和单页面文档目标共用 `CanvasDocumentContent` 渲染 Markdown 正文和聚合文档内部目录，避免两处维护不同的 Markdown 样式和目录切换逻辑。知识文档正文读取统一收敛到 `useCanvasDocumentMarkdown`，它按知识文档 ID 缓存 Markdown 内容；画布节点保存的 `activeDocumentId` 仍是聚合文档当前目录项的唯一来源。单页模式中来自画布节点的文档直接复用这些字段，未在画布节点中出现且未被隐藏的知识文档会被补成只读文档目标。单页模式默认查看页面时只构建文档目录元数据，不预取全部文档正文；只有用户切换到某个文档目标后，才按当前文档节点读取正文，避免文档网络请求与页面 iframe 编译、加载争抢首屏资源。画布工作台同样不能在入口读取所有文档正文，只读取当前视口内或用户选中的文档节点，离屏文档保留标题和目录元数据，等进入视口或被选中后再读取正文。

进入画布工作台时，页面卡片的实时 iframe、截图生成和文档正文读取共享同一段首屏资源预算。首轮截图批处理只提交 active、visible 和 nearby 页面，后台页面不参与首轮批处理；后台页面由后续显式截图再生、页面内容变更或维护任务补齐。这样可见页面优先获得真实预览或最新截图，不会被离屏页面截图队列拖慢。

本地文件导入统一从 `PreviewCanvas` 的画布根容器进入。拖入本地 Markdown 文件时，画布读取文本内容并创建文档节点；拖入本地图片时，画布读取图片数据并创建图片节点。拖放导入会把浏览器事件坐标换算为画布坐标，使新节点尽量落在用户释放鼠标的位置；键盘粘贴图片仍沿用视口中心落点。导入产生的是画布素材节点，不会自动写入项目工作区文件树，也不会改变 Demo 页面代码。

图片节点以图片资源填充卡片，可通过直接拖入画布或在画布中使用键盘粘贴添加。图片节点保存图片原始宽高，后续缩放时使用原始宽高比例约束节点外框，不允许把图片节点拉伸成与图片比例不一致的形状。图片节点不添加额外背景色，避免透明图或非矩形素材被底色干扰。

文档和图片节点不在卡片内部渲染头部，避免压缩内容浏览空间；文档编辑入口通过悬浮编辑按钮和双击提供。素材节点的布局随画布状态保存，和页面卡片一起参与拖拽、缩放、吸附与适应屏幕计算。素材节点不提供卡片内删除按钮，删除统一通过选中节点后按 Delete 或 Backspace 完成。

### 1.5 更新流程

```
代码变更
    │
    ▼
触发 preview contract fast gate
    │
    ├─ 失败：保留最近成功预览，回流诊断给自动修复
    │
    ▼
触发 compileCode()
    │
    ▼
检查缓存 / 调用 API 编译
    │
    ▼
向 iframe 发送 UPDATE_CODE
    │
    ▼
iframe 执行并渲染组件
    │
    ▼
预览区更新
```

## 二、配置联动

### 2.1 联动机制

配置变更与预览的联动流程：

```
用户修改配置表单
        │
        ▼
ConfigForm.onChange(formData)
        │
        ▼
父组件更新 configData 状态
        │
        ▼
PreviewPanel 接收新 configData
        │
        ▼
向 iframe 发送 UPDATE_CONFIG
        │
        ▼
iframe 使用新配置重新渲染
        │
        ▼
预览区更新渲染（无需重新编译）
```

**关键设计点**：

- 配置变更**不触发重新编译**
- 已编译结果保持不变，仅通过 `UPDATE_CONFIG` 更新 props
- 联动延迟 < 16ms（一帧内响应）

### 2.2 性能优化

| 优化措施    | 说明                                      |
| :---------- | :---------------------------------------- |
| React.memo  | PreviewPanel 使用 memo 避免不必要的重渲染 |
| useCallback | 配置变更回调使用 useCallback 缓存         |
| 编译缓存    | 相同代码不重复请求 API                    |

## 三、页面逻辑图运行时

多页面预览支持项目级应用逻辑图。页面仍然只负责展示和触发用户意图，不直接拼接 viewer URL，也不直接依赖随机页面 ID。页面代码从受控虚拟模块 `@preview/sdk` 使用 `trigger(event, payload)` 或 `PageAction` 发出语义动作；iframe 模板把动作发送为 `APP_ACTION` 消息。

`PreviewPanel` 只校验消息来自自身 iframe，并把动作连同当前 `pageId` 上报给外层 viewer。viewer shell 负责把 `pageId` 映射为页面 `routeKey`，在 `app.graph.json` 中查找 `from + event` 对应的 action。若 action 写入 `setState`，viewer 先更新轻量运行时 state；若 action 指向目标页面，viewer 切换 active page、同步 URL 中的 `route` 查询参数，并把 route params 与 app state 继续投递给目标页面 iframe。

运行时数据和配置数据保持分层。配置 Schema 仍通过组件 props 进入页面；route params 与 app state 通过 iframe 全局运行时和 SDK hook 读取。`useRouteParams()` 用于读取上一步 action 携带的参数，`useAppState()` 用于读取跨页面流程上下文。这样页面代码表达业务动作，viewer 统一解释流程，后续发布和本地项目包也能复用同一份应用图。

## 四、编译产物依赖 URL 约束

预览编译器会把页面代码中的受控依赖替换为 CDN URL 或内部虚拟模块 URL。`@preview/sdk` 属于内部虚拟模块，会以内联 `data:` URL 形式进入编译产物；该 URL 内部可能包含单引号、双引号或其他对 JavaScript 字符串有意义的字符。

因此编译器在生成 ESM import specifier 时，必须把最终 URL 序列化为合法 JavaScript 字符串字面量，不能手写单引号或双引号拼接。否则 iframe 的动态 import 会在浏览器语法解析阶段失败，页面不会发出 `LOADED` 消息，单页预览和画布实时 iframe 都会停留在加载态。

预览依赖策略版本参与服务端编译缓存键；浏览器端编译缓存也需要带上同一类策略版本。调整内部虚拟模块或 import URL 转义策略时，需要同步提升这些缓存版本，避免旧编译产物继续命中缓存。

同源 `react/jsx-runtime` vendor 还必须真实暴露 sucrase 自动 JSX 转换后的 named imports。若页面模块请求、编译 API 和 preview module 路由都返回成功，但 iframe 控制台出现 `react-jsx-runtime.js does not provide an export named 'jsx'`、`jsxs` 或 `Fragment`，根因在 preview runtime vendor 构建产物，而不是业务页面代码。此时 iframe 会停在 `module_import_start` 后，父窗口收不到 `LOADED`，`PreviewPanel` 和画布页面卡片会持续显示加载态。修复应更新 `scripts/build-preview-runtime.mjs` 并重新生成 author-site 与 viewer-site 的 `public/preview-runtime`。

## 五、运行错误回流与自动修复

预览 iframe、编译 API、module preflight 或画布页面卡片捕获到问题后，编辑页会把诊断归属到当前焦点页面，并写入 AI 当前视图上下文。这个上下文使用统一的 `previewDiagnostic` 字段承载来源、阶段、错误代码、页面、文件、错误信息和修复指引；旧的 `previewRuntimeError` 仅作为兼容字段保留。该上下文用于让 AI 判断要修复的页面和文件，但不直接作为用户消息展示。

用户侧只看到预览区的友好修复状态，以及 AI 对话区中的系统自动修复任务卡片。系统任务卡片展示修复进度和可展开详情；完整错误、文件路径和修复要求作为隐藏提示发送给 Agent。这样既保留自动修复所需的信息密度，又避免用户误以为自己发送了技术错误长文本。

同一页面的预览诊断按“项目 + 页面 + 诊断指纹”限流自动触发修复。诊断指纹优先使用编译或模块 hash；没有 hash 时退回到来源、阶段、错误代码、文件和错误信息组合。限流状态在浏览器本地短期持久化，因此重新打开同一项目时，同一份坏代码不会再次自动发起相同修复；页面代码变化产生新指纹后仍可自动修复。若自动修复失败，对话区的系统任务进入失败状态，用户可以查看详情并手动重新触发。

坏代码未通过 fast gate 时，`PreviewPanel` 不清空最近一次成功编译结果，也不把失败产物发送给 iframe。预览区可显示轻量修复状态，但 iframe 仍保留最近一次成功结果或截图占位，避免页面白屏或长期加载。编辑器和 workspace 保留真实坏代码，供用户和 Agent 检查、修复。

自动 checkpoint、页面命名版本、项目包导出、模板产出和发布属于 strict gate。执行这些关键动作前必须先 flush 协同草稿，再基于落盘 workspace 校验真实页面源码。strict gate 未通过时不创建正常版本或产物，接口返回 `VALIDATION_ERROR` 和结构化 runtime diagnostics。

### 6.2 AI 编辑快照驱动（snapshotVersion）

当 AI 通过对话编辑代码时，采用 **snapshotVersion 驱动**机制保证预览与配置的强一致性：

```
AI 编辑完成（SSE session.idle）
        │
        ▼
useChatStream.onFinish
        │
        ├─ 优先使用 result.files（drain 机制保证完整）
        ├─ 兜底 fetchSessionFiles() HTTP 获取
        └─ 触发 onSnapshotReady 回调
        │
        ▼
page.tsx — applyDemoSnapshot()
        │
        ├─ 原子化更新：code / schema / editorContent
        ├─ 合并 configData 默认值
        └─ snapshotVersion++
        │
        ▼
PreviewPanel 编译 effect 触发
  依赖: [code, sessionId, validCode, sendUpdateCode, effectiveVersion]
        │
        ▼
compileCode(code) → executeComponent() → 渲染
        │
        ▼
ConfigForm key 变化 → 强制重新挂载
  key={`${activeDemoId}-${snapshotVersion}`}
```

**关键设计点**：

- **snapshotVersion 是唯一驱动源**：所有 AI 编辑产生的变更（code + schema）通过一次原子更新应用，然后递增版本号
- **effectiveVersion 计算**：`const effectiveVersion = snapshotVersion ?? compileVersion`，向后兼容旧代码
- **编译 effect 依赖**：`[code, sessionId, validCode, sendUpdateCode, effectiveVersion]`，版本变化触发重新编译
- **ConfigForm 重建**：`key={\`${activeDemoId}-${snapshotVersion}\`}` 确保表单完全重建，应用新 schema 默认值

### 6.3 竞态处理

```typescript
useEffect(() => {
  let cancelled = false;

  compileCode(code)
    .then((result) => {
      if (cancelled) return; // 防止竞态
      const Component = executeComponent(result.compiledCode);
      setCompiledComponent(() => Component);
    })
    .catch((err) => {
      if (cancelled) return;
      setCompileError(err.message);
    })
    .finally(() => {
      if (!cancelled) setIsCompiling(false);
    });

  return () => {
    cancelled = true;
  }; // cleanup
}, [code]);
```

## 七、AI 编辑联动

### 7.1 背景与问题

AI 通过 WebSocket 编辑代码时，与用户手动编辑有本质区别：

| 维度     | 用户手动编辑     | AI 编辑                                  |
| :------- | :--------------- | :--------------------------------------- |
| 触发方式 | 编辑器 onChange  | WebSocket 消息                           |
| 编辑范围 | 通常只改代码     | 可能同时改代码 + Schema（含 default 值） |
| 状态感知 | 用户知道改了什么 | 系统需自动检测变更内容                   |

**典型问题场景**：AI 同时修改了 `index.tsx` 中的默认标题和 `config.schema.json` 中的 `default` 值，预览区更新了，但配置面板仍显示旧值。

### 7.2 根因分析

问题的根源在于 **configData 状态滞留**：

```
AI 编辑前：
  configData = { title: "旧标题" }  ← 用户之前在配置面板设置的值
  schema.default = "旧标题"

AI 编辑后：
  schema.default = "新标题"         ← AI 更新了默认值
  configData = { title: "旧标题" }  ← 但 configData 没变！

渲染时：
  props = { ...newDefaults, ...configData }
        = { title: "新标题", title: "旧标题" }
        = { title: "旧标题" }       ← 旧值覆盖了新默认值！
```

**为什么颜色变更能正常更新？** 因为颜色值通常只在代码中硬编码，不经过 configData，所以不存在覆盖问题。而标题等文本内容既在代码中有默认值，又在 configData 中有用户设置的值，两者冲突时旧 configData 胜出。

### 7.3 解决方案

#### 机制一：代码变更时重置 configData

当检测到代码内容变更（而非仅配置变更）时，将 configData 重置为空对象，让 Schema 中的新默认值生效：

```typescript
// page.tsx — handleCodeUpdate 中
if (schema) {
  const result = validateAll(newCode, schema);
  setValidationResult(result);

  // 代码变更时重置 configData 为空，让组件默认值生效
  setConfigData({});
}
```

**设计考量**：重置为 `{}` 而非保留旧值，是因为代码变更意味着组件的"基线"发生了变化，旧配置值可能不再适用。这是一种"信任新代码"的策略。

#### 机制二：默认值合并顺序

当 Schema 变更带来新的默认值时，确保新默认值优先于旧 configData：

```typescript
// ❌ 错误：旧 configData 覆盖新默认值
setConfigData((prev) => ({ ...newDefaults, ...prev }));

// ✅ 正确：新默认值覆盖旧 configData
setConfigData((prev) => ({ ...prev, ...newDefaults }));
```

**设计考量**：`{ ...prev, ...newDefaults }` 的含义是"以 Schema 的最新默认值为准，但保留用户之前设置过、且 Schema 未变更的配置项"。这样既能让 AI 的默认值更新生效，又不会丢失用户的其他配置。

### 7.4 AI 编辑完整数据流

```
AI 发送编辑指令（通过 WebSocket）
        │
        ▼
agent-service 执行编辑，写入文件
        │
        ▼
WebSocket 推送 file_operation 事件
        │
        ▼
ai-chat.tsx 接收事件，触发 onFileChange 回调
        │
        ▼
page.tsx — handleCodeUpdate(newCode, newSchema)
        │
        ├─→ 代码变了？ → setConfigData({})        ← 重置配置
        │
        ├─→ Schema 变了？ → 合并新默认值            ← 更新默认值
        │       setConfigData(prev => ({...prev, ...newDefaults}))
        │
        └─→ setCode(newCode) + setSchema(newSchema)
                │
                ▼
        PreviewPanel 重新编译 + 向 iframe 推送更新
        ConfigForm 接收新 configData + 新 Schema
```

### 7.5 关键原则

| 原则         | 说明                                               |
| :----------- | :------------------------------------------------- |
| 信任新代码   | 代码变更时重置配置，以新默认值为准                 |
| 新默认值优先 | 合并时 `{ ...prev, ...newDefaults }`，新值覆盖旧值 |
| 保留用户配置 | 仅覆盖 Schema 中变更的字段，其他用户设置保留       |
| 双重保障     | 重置 configData + 合并顺序，两道防线确保新值生效   |

function buildPreviewStyle(size?: PreviewSize): React.CSSProperties {

  const style: React.CSSProperties = {
    width: effectiveSize.width,
    height: effectiveSize.height,

## 十一、两种预览模式的边界

预览区的模式选择只在“当前页深度编辑”和“多页面空间管理”之间切换，不再维护第三种宫格视图。

### 11.1 单页面预览

单页面预览是编辑当前页面和阅读项目文档的统一入口。页面目标由 `PreviewPanel` 承载，文档目标由 `CanvasDocumentContent` 承载。只有页面目标进入配置、属性和 DOM 图层链路；文档目标不创建 iframe 预览请求，也不显示右侧页面面板。

| 能力 | 说明 |
| :--- | :--- |
| 实时渲染 | 代码变更后重新编译并向 iframe 发送 `UPDATE_CODE` |
| 配置联动 | 配置变更后发送 `UPDATE_CONFIG`，不重新编译 |
| 尺寸控制 | 使用当前页面的 `$demo.previewSize` 或用户选择的设备预设 |
| 可视化编辑 | 承载悬停框、选中框、批注标记、文本编辑和配置化入口 |
| 文档阅读 | 复用画布文档节点 Markdown 渲染；聚合文档通过内部目录切换当前文档 |

### 11.2 画布工作台

画布工作台由 `PreviewCanvas` 承载，是唯一的多页面总览入口。它用一个可平移、可缩放的工作区展示多个页面，并在截图版本与当前页面内容一致时优先使用截图；一旦没有当前版本截图，必须立即使用实时 iframe 保证最新效果可见。

| 能力 | 说明 |
| :--- | :--- |
| 多页面总览 | 所有页面以可拖拽、可缩放的画布对象展示 |
| 页面定位 | 支持选中页、聚焦页和页面标签，帮助用户在多页面项目中定位内容 |
| 自动排版 | 工具栏提供一键整理入口，基于当前布局轻量对齐页面并适应全部视图 |
| 布局持久化 | 页面位置、尺寸和画布视口保存到 session 的画布布局文件 |
| 渲染兜底 | 有最新截图时优先展示截图；无最新截图或截图失败时立即回退到实时 iframe |

### 11.3 模式切换规则

| 用户意图 | 使用模式 | 状态处理 |
| :--- | :--- | :--- |
| 调整当前页面代码、配置或可视化批注 | 单页面预览 | 保持当前 active page，预览 iframe 接收最新 code/config |
| 阅读项目文档或画布文档节点 | 单页面预览 | 保持当前 active page 不变，只切换单页文档目标并隐藏右侧页面面板 |
| 查看项目里的多个页面 | 画布工作台 | 读取页面列表、截图状态和已保存布局 |
| 整理多页面位置或尺寸 | 画布工作台 | 画布变更由 `useCanvasWorkspace` debounce 保存 |
| 对某个页面继续做局部编辑 | 单页面预览 | 以画布选中页作为 active page 回到 `PreviewPanel` |

## 十三、画布工作台机制

画布模式用于多页面总览、整理和定位。它不替代单页深度编辑，而是在共享组件 `PreviewCanvas` 中承载页面布局、视口、工具模式和截图兜底显示。

### 13.1 状态边界

| 层级 | 职责 |
| :--- | :--- |
| `useCanvasWorkspace` | 创作端画布状态入口，管理预览模式、选中页、聚焦页、布局加载、debounce 保存和保存状态 |
| `PreviewCanvas` | 共享画布工作台，接收页面数据、布局状态、截图状态和保存状态，负责画布渲染和交互编排 |
| `canvas-layout.ts` | 画布布局纯函数，负责页面尺寸解析、初始布局、自动排版和适应视口计算 |
| `CanvasPageItem` | 单页卡片，负责拖拽、缩放、截图/iframe 切换和截图状态覆盖层 |
| `canvas-kernel.ts` | 画布薄内核，收敛坐标转换、工具态解析、指针路由优先级、分层状态迁移和文本语义层保存适配 |
| `/api/sessions/:sessionId/canvas-layout` | 按 session 持久化画布布局，保存 `version`、`projectId`、`viewport`、`pages`、旧 `nodes` 兼容字段和新 `layers` 分层字段 |

### 13.2 布局持久化

画布布局保存到 session 目录下的 `.canvas-layout.json`。进入编辑页后优先加载已保存布局；如果没有保存数据，则由画布组件按页面预览尺寸计算初始布局。

画布布局读取需要同时兼容项目工作区、协同工作区和 session 目录中的 `.canvas-layout.json`。如果某个布局文件因历史并发写入或异常退出形成多个 JSON 文档直接拼接，读取接口应从文件中恢复所有顶层 JSON 文档，验证其中的画布状态，并选择 `updatedAt` 最新的有效布局返回；无效片段只被忽略，不能让编辑页停留在全局加载状态。写入布局时必须先写入临时文件再原子重命名到目标路径，避免写入过程中再次产生截断或拼接文件。

创作端编辑页从单页面预览首次切换到画布工作台时，会向 `PreviewCanvas` 传入一次性适应屏幕触发信号。`PreviewCanvas` 等页面和素材布局签名、容器尺寸都可用后，复用工具栏“适应屏幕”的同一套视口计算更新画布 `viewport`，并在成功后通知编辑页关闭该触发信号。该动作只覆盖本次打开编辑页后的首次进入画布；用户手动缩放、平移或再次从单页回到画布时，不会被自动适应屏幕重置。

自动排版属于用户主动触发的布局整理。点击工具栏入口后，系统读取当前页面位置和尺寸，先识别现有行，再在行内做顶部对齐、轻量网格吸附和最小间距修正；下方行只在接近上一行列位时对齐，不重新生成紧凑网格。整理后把视口调整到能看到全部页面，结果进入同一套 debounce 保存链路。

Agent 侧的画布整理通过 `arrangeCanvasPages` 完成。该工具读取工作区 `.canvas-layout.json` 后只替换 `pages` 和 `viewport`，写回时保留已有 `nodes`、`layers` 和隐藏知识文档状态，避免整理页面布局时删除画布文档、图片或文本语义节点。

画布工作台在编辑模式下默认进入选择工具，便于用户直接选中、移动和缩放页面或素材节点。选择工具在画布空白区拖拽时进入框选，框选优先命中 Demo 页面卡片；选中多个页面后，画布显示浮动对齐工具栏，支持左/中/右、上/中/下对齐，以及水平和垂直均分。拖动任意一个已选中页面时，其他已选中页面按相同位移一起移动；对齐、均分和多选拖动都会直接更新被选中页面的画布布局，并进入同一套 debounce 保存链路。

当框选未命中页面但命中文档节点时，画布进入文档多选状态；用户也可以通过 Shift、Command 或 Ctrl 点击文档节点累加选择。选中至少两个文档节点后，画布显示文档多选工具栏并提供合并入口。合并后的聚合节点使用所选节点外接矩形作为初始位置和尺寸，下限保持可读宽高；原节点从画布移除，对应知识库文档 ID 记录为已隐藏，避免后续知识库同步重新补回单独文档节点。

抓手工具仍保留在工具栏中；用户在选择工具下也可以按住空格临时切换为抓手平移画布，松开空格后回到选择工具。抓手工具下鼠标滚轮可直接按鼠标位置缩放视口；选择工具下普通滚轮不缩放，只有在按住 Ctrl 或 Cmd 时才缩放，避免普通滚轮误触导致画布比例变化。工具栏不提供重置布局入口，避免用户在整理多页面位置后误触丢失当前布局。

滚轮缩放由画布视口容器上的原生 `wheel` 事件处理，并使用非 passive 监听器；当事件会触发画布缩放时必须先阻止浏览器默认行为，避免 Ctrl/Cmd + 滚轮触发浏览器页面缩放。

保存触发由 `useCanvasWorkspace` 统一处理：

- 页面拖拽、缩放或视口变化后标记为 dirty。
- 通过 debounce 合并高频变化，避免拖拽过程中频繁写入。
- 保存状态不在画布工具栏常驻展示，避免用户在平移、缩放等高频操作中被后台保存过程打扰。
- 保存失败不阻断画布继续操作，失败详情写入浏览器日志，便于开发排查。

### 13.3 可见页面集合上报

画布会根据当前视口、容器尺寸和页面布局计算“当前可见页面集合”，供创作端决定截图生成优先级和相邻页面预热范围。该集合从 `PreviewCanvas` 向编辑页父组件上报。

上报必须带有等价状态保护：如果本次计算出的页面 id 列表与上一次上报内容完全一致，即使数组引用或父级回调函数在 React 渲染中发生变化，也不能再次触发父级状态写入。否则父级会因为收到新数组而重新渲染并重新生成 `pages` 数据，画布又重新计算同一组可见页，最终形成父子组件循环更新，在生产环境表现为 React 最大更新深度错误。

### 13.4 文本语义层与未来批注层边界

画布内核保持薄层职责，不直接渲染业务对象。它只提供共享能力：视口坐标转换、工具态解析、指针事件路由优先级、选择目标描述和分层保存适配。页面预览层仍然是默认优先级最高的图层；抓手工具和空格平移永远先交给内核处理。

画布当前正式节点只保留文档、图片和文本。旧 `nodes` 字段仍然读写，用于兼容已保存布局和协同草稿。新保存结构同步写入 `layers.annotations.nodes`，并由纯函数在读取时归一化。文档节点也会映射到 `layers.documents.nodes`，方便后续单独拆出文档图层。聚合文档节点在文档节点上额外保存 `documents` 目录项和 `activeDocumentId`，读取时仍兼容旧的单个 `knowledgeDocument` 字段。

文本节点属于 Agent 可读写的语义对象。动态工作区上下文会扫描 `.canvas-layout.json` 并注入文本节点摘要，包含节点 id、标题、文本内容、位置、关联页面线索和更新时间；短文本完整注入，过长文本截断但保留节点 id，避免 Agent 依赖截图 OCR 理解文字。当前阶段不建设待办、任务、分组、关系或评审等画布功能。

文本工具采用“先选工具、再点落点、直接输入”的交互。`CanvasViewport` 在文本创建态显示文本光标，只响应画布空白区域的点击落点；`PreviewCanvas` 使用该画布坐标作为文本节点左上起点创建空文本节点，并立即选中节点回到选择工具态；`CanvasFreeNodeItem` 在选中态渲染透明 textarea，不提供默认占位文案，也不为新文本节点写入默认背景色。

文本节点的选中态和编辑态必须分离。选择工具单击文本节点或框选覆盖文本节点时，只选中文本框并显示统一选中框，不出现输入光标；用户双击已选中的文本节点后才进入编辑态。进入编辑态时，输入光标默认放在已有文本末尾，便于用户继续追加内容，而不是从第一个字前开始输入。

文本节点默认态不显示卡片外框，保持接近原生文字输入的视觉；选中、拖拽或缩放时保留轻量蓝色描边，帮助用户识别当前操作目标。文字属性浮层使用紧凑工具条，文字颜色和背景色只显示当前色块，点击色块展开对应色板；背景色板第一项是“无背景”，不再提供独立背景按钮、独立橡皮擦按钮，也不提供会误删文本内容的清空按钮。

文字颜色和背景色色板应从属性工具条上方展开，并保持紧凑宽度，避免遮挡正在编辑的文字内容或覆盖下方页面卡片。

文字颜色与背景颜色共用一个颜色设置入口。颜色弹层内同时展示字体颜色和背景颜色，避免用户在两个相邻色块之间反复切换；背景颜色区域必须保留“无背景”选项。新建文本节点默认使用白色文字，保证在深色画布背景下创建后立即可见。

文本节点高度应跟随文字内容，而不是沿用文档或图片节点的通用最小高度。新建文本节点默认高度按 `fontSize * lineHeight` 计算；用户缩放文本节点时，最小高度按当前文本内容的视觉行数、字号、行高和节点宽度估算，允许单行文本背景框缩到单行文字等高，但不允许多行文本缩到小于当前内容高度。

文本节点宽度与高度应分别遵循不同下限。新建文本节点默认处于自动宽度模式：初始宽度约为一个字宽，输入时文本框宽度随当前最长文本行增长，高度保持单行；只有用户主动拖拽文本框边或角之后，文本节点才进入定宽模式并按当前宽度换行。用户缩放文本节点时，最小宽度按当前字号下一个字符宽度估算，允许用户主动把文本框变窄并触发换行。最小高度则按缩放后的当前宽度重新计算当前文本内容的实际视觉行数，确保文本框不能小于所有文本内容在当前宽度下所需的高度。视觉行数估算必须区分字符宽度：中文、日文、韩文和全角字符按接近一个字号宽度计算，英文和数字按较窄字符宽度计算，避免中文三行内容被误判为两行。

文本节点的边和角使用不同缩放语义。鼠标拖拽文本框四条边时，只调整文本框宽度或高度，文字字号不变；鼠标拖拽四个角时，以对角为锚点等比例缩放文本框，并同步按比例调整文字字号，用于快速改变整段文字大小。

文本节点高度通常接近单行文字，因此缩放热区不能沿用文档、图片节点的大尺寸边缘命中范围。选择工具下，用户在文本框内部拖拽应优先移动文本节点；只有鼠标非常贴近文本框边缘或四角时，才触发边拖拽缩放或角拖拽缩放。

文本编辑态下，文字属性工具条应与文字框保持足够垂直间距，不能遮挡首行内容。用户持续输入导致文本内容高度超过当前节点高度时，节点高度应自动增长以容纳内容；删除一行或多行文字导致内容高度降低时，节点高度也应自动缩小到当前内容所需高度。textarea 本身不显示滚动条。

画布选中态使用统一组件绘制。页面卡片、文档节点、图片节点和文本节点被选中、拖拽或缩放时，都显示 Figma 风格的蓝色 1px 选中边框和四角控制点；不再由各组件分别使用不同粗细、阴影或 ring 样式表达选中状态。

选择工具框选时，页面卡片和文本节点都参与命中判断。当前多选对齐工具只面向页面卡片；当框选区域没有命中页面但命中文本节点时，画布应选中该文本节点但不进入编辑态。

画布全局快捷键必须避开文本编辑目标。用户在 textarea、input、select 或 contenteditable 内输入时，`H`、`V`、空格等画布快捷键不得切换工具或触发平移。文本节点内容和样式修改必须同时写回旧 `nodes` 字段与 `layers.annotations.nodes`，否则分层归一化会用旧分层数据覆盖用户输入。

画笔、箭头、形状和高亮只作为未来批注模式评估。批注层可以被 Agent 通过画布截图作为视觉参考，但默认不可由 Agent 写入，也不能替代文本语义层。

### 13.5 页面标签与截图日志

画布使用截图降低多页 iframe 成本，但截图只是性能优化，不能优先于“最新页面效果可见”和“显示结果与真实代码一致”。页面代码或配置变化后，画布必须立即展示该变化对应的实时 iframe；只有当前版本截图生成并加载成功后，才允许用截图替换 iframe。

- 每个页面框外左上角常驻展示页面名，帮助用户像在 Figma 中一样识别画布对象。
- 截图开始、完成、失败和服务不可达信息写入浏览器日志。
- 点击画布页面只改变选中态和配置面板上下文；如果当前截图仍然匹配页面内容版本，不能因为选中页面就切换到 iframe，避免无变化时闪烁。
- 配置变更触发新一轮截图时，画布页面先清除上一版截图并继续显示实时 iframe；直到截图服务返回与当前配置一致的 hash，并且图片加载成功后，才允许切回截图显示。
- 画布页面初始宽高来自当前页面 `config.schema.json` 中的 `$demo.previewSize`；当真实内容高度超过预览视口时，画布卡片高度按 iframe 内容高度或截图自然高度扩展到完整页面。
- 画布截图生成使用整页截图；截图文件的自然高度用于校正画布卡片高度，使截图态和 iframe 态共享同一个画布实际尺寸。
- 截图在画布中按宽度铺满并顶部对齐展示，不能用居中 contain 产生上下留黑，否则会和 iframe 的顶部对齐渲染不一致。
- 无最新截图或截图失败时继续使用 iframe 实时预览兜底；不能为了控制性能而用生成中、需刷新或可进入实时预览的状态替代真实页面。
- 画布优化不能把旧截图伪装成当前页面效果，截图版本必须与当前代码和配置版本一致后才能作为稳定显示内容。

## 2026-06-27 positionable sizes 状态回写约束

- `PreviewPanel` 从 iframe 接收 `POSITIONABLE_SIZES_RESULT` 时，父级编辑页必须按元素 key、`width` 和 `height` 做等价去重；尺寸内容没有变化时应返回旧 state。
- 该约束适用于单页预览和画布实时 iframe 兜底两条入口，避免 iframe 尺寸采集产生的新对象引用触发父组件无意义重渲染。
- 尺寸真实变化时仍应正常更新 `ConfigForm` 的 `positionableItemSizes`，保证元素定位控件使用最新渲染尺寸。

## 2026-06-28 单页面预览尺寸来源

- 单页面模式的 `PreviewPanel` 尺寸以当前页面 `config.schema.json` 中的 `$demo.previewSize` 为准，不允许因异步切页、缓存状态或默认尺寸而沿用上一页尺寸。
- 创作端编辑页渲染单页预览时，应从当前页 schema 派生 `activePreviewSize`，再回退页面尺寸缓存和既有状态；没有声明 `$demo.previewSize` 的历史页面继续使用默认预览尺寸。
- 缩放后的单页面预览框应在预览可用区域内水平和垂直居中展示；只有页面实际显示尺寸超过可用区域时，才通过外层滚动容器查看溢出内容。
- 服务端 API、发布流程和前端预览共用 `getPreviewSize` 的解析语义，只接受对象形式的 `$demo.previewSize`，并对 `width`、`height`、`minHeight`、`maxHeight` 和 `scale` 做类型收敛。
