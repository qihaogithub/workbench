# 截图服务与画布降级机制

> 更新日期：2026-06-21  
> 适用模块：创作端配置与预览、画布模式、多页面截图缓存

## 1. 机制定位

截图服务是画布模式的性能优化层，不是画布预览的唯一渲染来源。画布中的页面始终具备两条显示路径：

| 显示路径 | 使用场景 | 用户感知 |
| :--- | :--- | :--- |
| 截图路径 | 截图服务在线且页面截图已生成 | 多页面画布更轻、更稳定 |
| 实时 iframe 路径 | 截图服务离线、截图生成失败、尚无截图或正在编辑页面 | 页面仍可预览和编辑 |

这意味着“截图服务不可达”不能阻断画布使用。它只会让画布从截图优化回退到实时预览。

## 2. 服务协作关系

浏览器不直接访问截图服务。所有截图相关请求都先发给 author-site 的同源 API，再由 author-site 服务端转发到 screenshot-service。

```
浏览器
  -> author-site /api/screenshots/*
    -> screenshot-service /api/screenshots/*
      -> Puppeteer/Chromium 渲染页面截图
      -> data/screenshots 写入截图缓存
```

这样做有三个目的：

- 避免浏览器端把 `localhost:3202` 解析成用户本机，导致局域网或 Docker 场景下不可达。
- 避免让截图服务暴露给前端跨域调用，降低 CORS 配置复杂度。
- 让 author-site 可以统一处理健康检查、错误响应和文件读取回退。

## 3. author-site 代理契约

author-site 提供以下同源入口：

| author-site 入口 | 转发目标 | 说明 |
| :--- | :--- | :--- |
| `GET /api/screenshots/health` | `GET /health` | 返回统一 `{ success: true, data }`，用于前端判断截图服务是否在线 |
| `POST /api/screenshots/generate` | `POST /api/screenshots/generate` | 单页截图生成 |
| `POST /api/screenshots/generate-batch` | `POST /api/screenshots/generate-batch` | 多页面批量截图生成 |
| `GET /api/screenshots/status/:projectId/:batchId` | `GET /api/screenshots/status/:projectId/:batchId` | 批量任务轮询 |
| `GET /api/screenshots/file/:projectId/:pageId` | `GET /api/screenshots/file/:projectId/:pageId` | 优先代理截图文件，404 或服务不可达时回退读取 author-site 本地缓存 |

截图服务不可达时，author-site 返回统一错误：

| 字段 | 值 |
| :--- | :--- |
| `success` | `false` |
| `error.code` | `SCREENSHOT_SERVICE_UNAVAILABLE` |
| `error.message` | `截图服务不可达` |

前端只需要识别该错误码，不需要判断底层是端口未启动、Docker DNS 不通，还是 Chromium 启动失败。

## 4. 前端状态流

`useScreenshotGeneration` 负责维护每个页面的截图状态：

| 状态字段 | 含义 |
| :--- | :--- |
| `screenshotUrl` | 最近一次可用截图地址 |
| `loading` | 当前是否正在生成或轮询截图 |
| `error` | 当前页面最近一次截图失败原因 |
| `serviceAvailable` | 截图服务健康状态，`null` 表示尚未确认 |

关键规则：

- 启动批量截图或单页重试时，不清空已有 `screenshotUrl`。
- 截图服务不可达时，将 `serviceAvailable` 标记为 `false`，页面继续使用 iframe 实时预览。
- 截图生成成功后，更新对应页面的 `screenshotUrl`，并清除该页错误状态。
- 批量任务通过 `status` 轮询更新每个页面，单页失败不会阻塞其他页面。

## 5. 画布降级表现

画布组件按以下顺序决定每个页面如何显示：

1. 当前页面正在编辑：始终使用实时 iframe。
2. 有可用截图且截图已加载：优先显示截图。
3. 无截图、截图加载中或截图失败：使用实时 iframe。

当截图服务离线时：

- `CanvasToolbar` 显示“截图服务离线，正在使用实时预览”。
- 工具栏提供重试入口，重试会先调用健康检查，再触发批量截图。
- `CanvasPageItem` 在页面右上角显示轻量状态，如“使用实时预览”和“重试”。
- 页面本身不显示阻断式错误卡片，用户仍可拖拽、缩放、选择页面并编辑配置。

## 6. screenshot-service 职责

screenshot-service 是独立 Fastify 服务，默认端口 `3202`。它负责：

- 接收单页或批量截图请求。
- 调用编译能力生成 iframe HTML。
- 使用 Puppeteer/Chromium 渲染 PNG。
- 按 `projectId` 与 `pageId` 写入截图缓存与元数据。
- 提供 `/health` 用于 author-site 和运维检查。

浏览器渲染依赖本机或容器内 Chromium。服务启动时会按以下顺序寻找浏览器：

1. `PUPPETEER_EXECUTABLE_PATH`
2. macOS Chrome 默认路径
3. Linux Chromium/Chrome 默认路径
4. Windows Chrome/Edge 默认路径

本地 Windows 开发环境不需要额外配置时，应优先使用自动探测路径；如果机器安装路径特殊，再显式设置 `PUPPETEER_EXECUTABLE_PATH`。

## 7. 本地开发与 Docker

本地开发建议使用：

```bash
pnpm dev:preview
```

该命令只启动 author-site 与 screenshot-service，适合验证创作端预览和画布截图，不依赖历史 snapshot 包。

Docker 场景下，author-site 使用服务端环境变量访问容器内部服务：

| 变量 | 建议值 | 说明 |
| :--- | :--- | :--- |
| `SCREENSHOT_SERVICE_URL` | `http://screenshot-service:3202` | author-site 服务端访问截图服务 |
| `NEXT_PUBLIC_SCREENSHOT_SERVICE_URL` | 不建议依赖 | 浏览器端不再直连截图服务 |
| `PUPPETEER_EXECUTABLE_PATH` | `/usr/bin/chromium` | screenshot-service 容器内 Chromium 路径 |

关于 Docker 服务编排与环境变量，请见 [Docker部署方案](../../06-基础设施/技术/03_Docker部署方案.md)。

## 8. 排查顺序

当画布出现截图相关问题时，按以下顺序排查：

1. 打开 `GET /api/screenshots/health`，确认 author-site 代理是否返回成功。
2. 打开 `GET http://localhost:3202/health`，确认本地截图服务是否启动。
3. 检查 screenshot-service 日志，确认 Chromium 是否可启动。
4. 检查画布是否仍显示实时 iframe；如果 iframe 也不可用，应转向实时预览链路排查，参见 [02_实时预览机制](./02_实时预览机制.md)。
