# 预览系统 - 需求文档

> 版本：v2.11
> 创建日期：2026-04-06
> 更新日期：2026-07-07

---

## 一、模块定位

预览系统是 Demo 可视化呈现的核心模块，负责在浏览器端实时渲染 AI 生成的 React 组件，提供所见即所得的预览体验。本模块采用**服务端编译 + iframe 预览执行**架构，确保用户配置的变更能够即时反映在预览区，缩短反馈循环。

---

## 二、功能描述

### 2.1 实时预览功能

#### 2.1.1 组件渲染

**用户视角**：
- 系统应在预览区实时渲染 AI 生成的 React 组件
- 渲染结果应与实际使用效果一致

**交互效果**：
- 预览区应支持 React 组件的完整渲染
- 预览区应支持 TypeScript/JSX 代码的编译执行
- 组件内部的交互行为（如点击、动画）应正常工作
- 预览组件应直接使用项目的样式系统和组件库（shadcn/ui）

#### 2.1.2 配置注入

**用户视角**：
- 用户在配置面板设置的参数应自动注入到 Demo 组件
- 配置变更后，预览应即时更新

**交互效果**：
- 配置值应作为组件 Props 传递
- 配置变更后，预览应在 < 16ms 内更新（一帧内响应）

#### 2.1.3 错误展示

**用户视角**：
- 当代码存在语法错误时，系统应在预览区显示错误信息
- 当组件运行时发生错误时，系统应显示错误边界提示
- 当运行错误可自动交给 AI 修复时，用户应看到“正在修复预览”等友好状态，而不是完整技术错误

**交互效果**：
- 错误信息应包含错误类型和位置
- 错误信息应使用用户友好的语言描述
- 应提供错误修复建议（如可能）
- 自动修复触发后，对话区应显示系统发起的修复任务，完整技术细节默认收起

---

### 2.2 预览尺寸控制功能

#### 2.2.1 自定义尺寸

**用户视角**：
- 用户可设置预览区的宽度和高度
- 预览区应支持响应式布局

**交互效果**：
- 支持设置固定尺寸（如 800x600 像素）
- 支持设置百分比尺寸（如 100%宽度）
- 支持设置最小/最大尺寸限制

#### 2.2.2 尺寸预设

**用户视角**：
- 系统应提供常用设备尺寸预设
- 用户可快速切换不同设备预览

**交互效果**：
- 预设尺寸应包括：手机、平板、桌面等常见设备
- 切换预设后，预览区应立即调整尺寸

---
### 2.3 预览刷新功能

#### 2.3.1 手动刷新

**用户视角**：
- 用户可手动触发预览刷新
- 刷新后，预览区应重新编译并渲染组件

**交互效果**：
- 提供刷新按钮
- 编译过程中显示加载状态

#### 2.3.2 自动刷新

**用户视角**：
- 代码变更后，预览应自动重新编译并刷新
- 配置变更不触发重新编译，直接通过 props 更新
- 自动刷新应支持缓存，相同代码不重复编译

**交互效果**：
- 代码变更后自动触发编译和刷新
- 编译结果应缓存，提升重复加载速度

---

### 2.4 预览模式切换功能

#### 2.4.1 单页面预览

**用户视角**：
- 系统应提供单页面预览，用于查看、配置和编辑当前选中的 Demo 页面
- 系统应在单页面预览中统一提供页面和文档查看入口，用户无需切到画布也能阅读文档说明
- 用户在单页面预览中应能使用尺寸预设、实时配置联动、刷新和错误提示
- 用户需要进行可视化批注、局部文本编辑或添加配置项时，应在单页面预览中完成
- 用户查看草图页时，应先看到只读页面效果；需要直接绘制和拖拽对象时，再通过顶部编辑入口进入草图编辑态

**交互效果**：
- 单页面预览只聚焦当前页面，避免多页面内容干扰编辑判断
- 切换当前页面后，预览区应立即展示新页面内容
- 单页面预览的目录应位于预览区右上角下拉入口，展开后同时提供页面和文档选择
- 单页面预览目录应能清晰标识当前查看对象；当前对象可以是页面，也可以是文档
- 用户切换到文档时，预览区应只读展示 Markdown 渲染结果；文档编辑仍通过画布文档弹窗或知识库入口完成
- 当前查看对象是文档时，右侧配置和属性面板应隐藏，避免展示与文档无关的页面配置或 DOM 属性
- 单页面预览应保留当前页面的交互能力，如点击、动画和滚动
- 当项目没有任何页面或当前没有可预览页面时，单页面预览应展示“暂无页面”的明确空状态，引导用户添加页面或让 AI 创建页面，不应停留在页面代码加载提示中。
- 当预览尚未生成独立截图或发布产物但已有页面源码时，预览区应继续走源码编译和实时渲染链路；加载态文案应表达正在加载页面预览，而不是让用户误以为预览产物永久生成中。
- 草图页编辑态应只作用于当前页面内部对象；退出编辑态后，草图工具栏应收起，预览恢复只读展示

#### 2.4.2 画布工作台

**用户视角**：
- 系统应提供画布工作台，用于查看项目中的多个 Demo 页面
- 用户可在画布中识别页面名称、选中页面、调整页面位置和尺寸
- 用户可在画布中添加文档说明、文本说明和图片素材
- 画布文本应作为 Agent 可读写的语义节点，而不是批注层里的像素文字
- 用户可通过直接拖入画布添加本地 Markdown 文档和图片
- 用户可通过工具栏图片按钮从本地选择图片添加到画布
- 用户可在画布中使用键盘粘贴添加图片
- 用户可在画布中选中多个文档节点，并合并为一个带目录的聚合文档节点
- 用户可在画布中选中多个页面，并合并为一个带目录的页面组，用于对比同一页面的不同状态
- 用户可一键基于当前摆放自动整理画布页面，让多页面布局在不破坏大致分组的前提下更规整
- 用户可从画布中的某个页面回到单页面预览继续细节编辑
- 系统不提供外部网页插入功能

**交互效果**：
- 画布应支持多页面总览、平移、缩放和页面聚焦
- 用户打开创作端项目编辑页后，首次切换到画布工作台时，画布应自动适应屏幕，效果与用户点击工具栏“适应屏幕”一致；后续再次进入画布不应覆盖用户已经调整过的视口
- 画布应支持自动排版，保留当前大致行列关系，仅对齐页面、修正过近间距并适应全部页面视图
- 画布应将拖入的本地 Markdown 文件创建为文档节点，将拖入的本地图片创建为图片节点，并尽量放置在用户释放鼠标的位置
- 画布应将工具栏添加的文本创建为可移动、可缩放、可编辑的语义文本节点
- 文字工具激活后鼠标应呈现文本输入光标；用户点击画布空白目标位置后，应在该位置直接出现可输入光标
- 新建文本节点不应预填默认文本，也不应默认添加白色背景，避免把语义文字误表现为卡片化批注
- 画布文字节点应支持在画布中快速编辑文字内容
- 画布中的文档节点应在画布内只浏览 Markdown 渲染结果，编辑时打开统一 Markdown 编辑弹窗
- 聚合文档节点应在左侧展示目录，用户可在同一个节点内切换查看不同文档
- 页面组应在当前展示页面左侧提供悬浮目录，用户可在同一个页面组内切换查看不同页面状态；目录可折叠，折叠后只显示组内页面数量
- 页面组不应用额外大框包裹多个页面；合并后的画布主体仍应表现为一个当前页面卡片，目录不占用页面卡片尺寸
- 页面组应能被选择工具框选，并可与普通页面一起进入批量对齐工具栏
- 页面组合并只改变画布上的可视化分组；被合并的页面仍应作为独立页面存在，不影响单页编辑、页面树、历史版本、发布数据、AI 上下文和 CLI 流程
- 页面组移动或缩放时只改变页面组在画布中的位置和尺寸，不应改写组内原页面的独立布局；缩放时页面外框和当前展示内容应按当前页面比例等比缩放
- 文档编辑弹窗不提供独立标题输入，文档第一行即作为画布节点标题
- 画布中的图片节点应直接展示图片内容，不显示内部头部，不添加额外背景色
- 画布中的图片节点调整尺寸时应始终保持图片原始比例，外边框比例不应被自由拉伸改变
- 画布中的页面应优先使用截图展示；截图不可用时仍应能展示实时预览
- 画布中的草图页应作为普通页面对象展示，不在画布工作台中暴露草图内部绘制工具
- 画布默认工具态必须优先服务页面选择、拖拽、缩放、聚焦和配置入口
- 画布文本语义对象应以结构化数据保存，并能被 Agent 稳定读取、引用和改写
- 用户选中画布中的页面后，该页面应以白色描边高亮，并尽快切换为可反映最新内容的真实预览
- 当截图尚未准备好时，画布应优先展示真实预览；当真实预览数量过多影响性能时，才对部分页面展示轻量加载态
- 原型页配置绑定缺少当前配置值时，应保留原型 HTML/CSS 中的兜底内容，不能把已有文案、图片或样式清空；只有用户显式配置为空值时，预览才显示为空
- 页面布局调整后应自动保存，下次进入时恢复用户整理过的位置
- 页面布局、文档、图片和文本语义节点应分层保存；语义层或扩展数据异常时不应阻断页面布局恢复

#### 2.4.3 模式边界

**用户视角**：
- 预览区只保留单页面预览和画布工作台两种模式
- 多页面查看统一使用画布工作台
- 当前页面的深度编辑统一使用单页面预览；文档在单页面预览中只承担阅读入口

**交互效果**：
- 系统不再提供独立的宫格预览入口
- 模式切换不应丢失当前选中页面
- 从画布回到单页面预览时，应以画布中选中的页面作为当前页面

### 2.5 可视化批注与局部编辑

#### 2.5.1 元素点选

**用户视角**：
- 用户可在单页预览中进入批注模式，直接点选页面上的可见元素
- 选中元素后，系统应显示当前元素的标签、文本、样式类名和位置线索
- 用户可将当前选区发送给 AI，让 AI 围绕该局部范围提出修改建议

**交互效果**：
- 鼠标悬停时显示元素外框和简短标签
- 点击元素后保留选中外框
- 选区信息应包含 DOM 路径、元素标签、文本摘要和 className 摘要

#### 2.5.2 页面批注

**用户视角**：
- 用户可对选中的预览元素添加文字批注
- 用户可一次性发送多条未处理批注给 AI
- 发送后，系统应把批注范围和内容作为上下文交给 AI，而不是让 AI 任意扩大修改范围

**交互效果**：
- 批注以浮层输入框完成
- 已添加批注在预览区以标记点展示
- 点击标记点可重新打开批注内容
- 批注为空且没有样式变更时，应自动丢弃

#### 2.5.3 预览内文本编辑

**用户视角**：
- 用户可对无子元素的文本节点进行快速文本修改
- 修改不会立即覆盖项目代码，而是先生成可审阅的修改建议
- 用户接受建议后，系统才将修改写回当前页面代码

**交互效果**：
- 双击可编辑的文本元素后输入新文本
- 系统生成待审修改卡片
- 用户可接受或拒绝该修改
- 当原始文本在代码中不存在或出现多次时，系统应拒绝自动写回并提示用户手动确认

#### 2.5.4 样式调整批注

**用户视角**：
- 用户可在属性面板中临时调整常见文本、外观、尺寸、间距和布局样式，作为给 AI 的可视化修改说明
- 样式调整应先用于预览表达意图，再随批注发送给 AI 处理
- 布局调整应使用面向非技术用户的设计工具式控件，而不是暴露代码缩写或 CSS 轴向概念

**交互效果**：
- 支持调整文本内容、文本颜色、字体、字号、字重、行高、文字对齐、背景、透明度、圆角、宽高、内外边距、布局方式、对齐和间距
- 图片分组只提供上传替换能力；图片地址和图片描述不作为属性面板中的直接编辑项
- 外观分组应提供不透明度和圆角的紧凑控件；圆角默认整体设置，并可展开为四角独立设置
- 背景分组在当前元素没有实际背景色时，只应展示添加入口；用户添加背景后，再展示颜色编辑项
- 边框分组在当前元素没有实际边框时，只应展示添加入口；用户添加边框后，再展示颜色和宽度等编辑项
- 阴影与模糊分组在当前元素没有实际阴影或滤镜时，只应展示添加入口；用户添加效果后，再展示阴影和滤镜编辑项
- 颜色属性应统一使用“色块 + Hex + 明度百分比”的紧凑控件展示；用户输入 Hex 色值时，带 `#` 或不带 `#` 都应可识别，明度默认 100% 表示原色，调低后应即时影响临时预览。
- 布局分组应参考 Figma Auto layout 的信息结构，依次提供排列方式、尺寸填充、九宫格对齐、元素间距、上下/左右内边距和裁剪内容等直观控件。
- 面板文案应避免使用 Flow、Gap、PT、PR、主轴、交叉轴等偏开发者术语
- 取消样式调整时，预览应恢复到调整前状态
- 确认样式调整后，系统应记录变更项、旧值和新值
- 当前版本不要求把样式调整直接转换为 className 写回，样式落地由 AI 根据批注上下文完成

#### 2.5.5 可视化添加配置项

**用户视角**：
- 用户可在单页预览中进入配置化模式，直接点选页面上的可见元素
- 用户可把属性面板中展示的任意可配置属性添加为配置面板中的配置项
- 用户确认后，新配置项应立即出现在配置面板，并能联动当前预览

**交互效果**：
- 鼠标悬停和点击沿用预览选区外框
- 系统应根据选中元素给出可配置内容候选，如文本内容、图片地址、文字颜色、背景颜色或边框颜色
- 用户可编辑配置项显示名称、字段 key、默认值、分类和配置范围
- 当系统无法安全定位代码中的唯一元素时，应拒绝自动写回并提示用户改用 AI 或手动方式确认

## 三、业务规则与约束

### 3.1 执行环境约束

| 约束 | 说明 |
|:-----|:-----|
| 依赖白名单 | AI 生成的组件只能使用项目已有依赖（react、lucide-react、clsx 等） |
| 代码来源 | 组件代码由 AI 生成，经过系统提示词约束和白名单校验 |
| 存储限制 | 预览环境无法访问本地存储 |

### 3.2 支持的技术栈

| 技术 | 版本要求 | 说明 |
|:-----|:---------|:-----|
| React | 18.x | 核心渲染框架 |
| TypeScript | 5.x | 类型安全的代码编写 |
| CSS | - | 支持 Tailwind CSS、CSS Modules |
| 组件库 | shadcn/ui | 项目内置组件库，可直接使用 |

### 3.3 性能要求

| 指标 | 要求 | 说明 |
|:-----|:-----|:-----|
| 首次渲染时间 | < 1s | 从加载到首次渲染完成（含编译） |
| 配置联动时间 | < 16ms | 配置变更到预览更新（无需重新编译） |
| 代码编译时间 | < 500ms | 代码变更到编译完成 |
| 错误恢复时间 | < 1s | 错误修复到预览恢复 |

## 四、预览尺寸配置规范

### 4.1 尺寸配置格式

在配置定义中通过扩展字段设置预览尺寸：

```json
{
  "$demo": {
    "previewSize": {
      "width": "100%",
      "height": "600px",
      "minHeight": "400px",
      "maxHeight": "800px"
    }
  }
}
```

### 4.2 预设尺寸列表

| 预设名称 | 宽度 | 高度 | 适用场景 |
|:---------|:-----|:-----|:---------|
| 手机竖屏 | 375px | 667px | 移动端应用 |
| 手机横屏 | 667px | 375px | 移动端游戏 |
| 平板竖屏 | 768px | 1024px | 平板应用 |
| 平板横屏 | 1024px | 768px | 平板应用 |
| 桌面 | 100% | 100% | 桌面应用 |
| 自定义 | 用户定义 | 用户定义 | 特殊需求 |

## 2026-06-27 画布工具态历史补充

- 文字和图片工具采用先选工具再落点的编辑体验，不在点击工具按钮时立即插入节点。
- 文字工具在用户点击画布空白目标位置后创建文本节点，并直接在画布中聚焦输入；图片工具在选择本地图片后等待用户点击画布目标位置再创建图片节点。
- 文本节点保存到画布布局，作为 Agent 可读写的语义文本；它不进入单页预览批注链路，也不直接写回 Demo 页面代码。

## 2026-06-28 Agent 友好画布语义层补充

- 画布继续以页面预览层为核心，不在正式工具栏暴露结构化图层或 Excalidraw 标注验证入口。
- 画布后续扩展优先建设 Agent 可读写的文本语义层和文档说明能力，不在当前阶段建设待办、分组、关系或评审等画布功能。
- 画笔、箭头、形状和高亮等白板类能力只作为未来批注模式评估，不作为当前阶段的正式工具。

## 2026-06-28 单页面预览尺寸要求

- 单页面模式下，预览区域的宽高必须匹配当前页面 `config.schema.json` 中声明的 `$demo.previewSize`，例如平板横屏页面应按 `1024x768` 等页面配置展示。
- 切换当前页面、加载会话文件或刷新页面列表后，预览尺寸应立即跟随当前页面 schema 更新；在新页面尺寸尚未解析前，不应沿用上一页尺寸误导用户。
- 页面按可用区域缩放后，应在预览区域内居中展示，避免横屏或平板页面贴近顶部或边缘。
- 未声明 `$demo.previewSize` 的历史页面可以继续使用系统默认尺寸，但这属于兼容回退，不应覆盖已声明尺寸的页面。

## 2026-06-29 单页预览属性抽屉要求

- 单页面预览的右侧栏默认展示“配置”面板；“选择”作为单页顶部工具栏按钮，点击后进入编辑模式：右侧以抽屉形式覆盖配置栏展示属性编辑，左侧以抽屉形式覆盖 AI 对话区展示当前页面图层。
- 左侧图层抽屉和右侧属性抽屉都不单独展示关闭按钮；用户再次点击顶部“选择”按钮退出编辑模式。退出编辑模式、切换页面、切换到文档对象或离开单页预览后，预览应恢复原本页面交互，不再拦截点击选择元素。
- 只有属性抽屉打开时，左侧预览才进入元素选择状态。用户不需要先进入批注模式或配置化模式；在属性抽屉打开状态点击页面位置后，系统应识别该位置命中的对象层级，并默认选中最具体、最可编辑的对象。
- 图层菜单应展示真实 DOM 树状结构，而不是面向开发者的扁平 DOM 清单。右键预览元素时，菜单只展示当前选中元素及其子级；点击“选择”按钮进入单页编辑模式时，当前页面图层树应在左侧抽屉中展开并遮住 AI 对话区。图层菜单不常驻在属性面板正文中，属性栏也不放置图层列表按钮。对象名称应使用非技术用户可理解的名称，避免直接显示 `div`、`span` 等标签名。用户悬停对象项时，预览区应高亮对应对象，点击对象项时，预览区保留选中框并同步右侧属性面板。图层菜单滚动条应采用项目内细滚动条样式，避免使用浏览器默认粗滚动条。
- 用户点击 iframe 内空白处或预览页面外层空白区域时，应取消当前选中对象并清空当前点击图层，右侧属性面板回到未选中提示态。
- 属性面板应采用类似设计工具 Inspector 的结构，按“位置、布局、外观、图片、文本、背景、边框、链接、阴影与模糊”组织当前对象属性，而不是一项一张卡片的表单。位置分组应以两行紧凑结构展示：第一行使用 X/Y 表示横向和纵向位置，第二行使用 W/H 表示宽度和高度。布局分组应采用接近 Figma Auto layout 的纵向组织，先展示排列，再展示尺寸、对齐与间距、内边距和裁剪。数值类控件应显示单位后缀，颜色类控件应同时展示色块、Hex 值和明度百分比，布局和对齐类控件应优先使用图标分段按钮。
- 属性名称、按钮提示和输入占位文案应使用中文，避免直接展示 Width、Height、Size、Color、Line、Spacing 等英文属性名。
- 属性面板应优先展示当前选中对象的常用编辑项，不额外展示选中对象概览卡片或底部发送区；背景、边框、阴影、滤镜等可选视觉效果在当前元素没有实际效果时只展示添加入口，避免非技术用户在首屏面对无意义字段。
- 颜色类属性应统一以 Hex 和明度百分比组合展示；用户输入 Hex 色值时，带 `#` 或不带 `#` 都应可识别，明度百分比应与 Hex 一起用于实时预览。
- 文本、图片、外观、尺寸、间距、布局和链接等高频属性应支持临时修改，预览区应立即展示临时效果。临时修改不直接写回源码、资源文件或配置 Schema。
- 本地图片替换应优先保存到当前会话资源目录，并使用返回的资源地址进行临时预览；当无法保存到资源目录时，可以退回临时 data URL 预览，但正式落地仍需由 AI 保存为项目资源。
- 属性面板正文顶部应展示 AI 修改说明输入区，便于用户在调整属性前先描述整体意图。
- 每个可配置属性的属性名本身应作为配置项入口，不再额外展示配置图标，也不依赖“更多”菜单。用户点击属性名后，应以弹窗方式填写配置名称、字段 key、默认值、分类和作用域；字段 key 与已有配置冲突时，应在弹窗中提示调整。属性名在悬停或聚焦时应有低噪声状态反馈，已设置为配置项时应保持明确的选中状态。
- 已设置为配置项的属性应在属性面板顶部展示为紧凑配置栏信息条；用户点击信息条可继续修改配置项设置，点击移除按钮可删除该配置标记。
- 属性面板顶部右侧应提供“清空”按钮，用于清空当前选中图层的全部属性修改和配置项标记；当前图层没有修改时按钮不可用。
- 左侧图层抽屉中，对象项右侧不展示类型标签；当该图层存在属性修改或配置项标记时，仅展示合并后的数字徽标。
- 左侧图层抽屉应支持临时隐藏或显示单个图层，用于排查遮挡关系和确认页面层次；该操作只影响当前预览会话，不写回源码、Schema 或发布数据。
- 左侧图层抽屉底部应展示所有图层的合并修改数量，并提供“发送给AI”按钮；没有新的属性修改、配置项标记或补充说明时，“发送给AI”不可用。
- 用户点击“发送给 AI”后，本次内容应进入已发送状态，按钮应变为“已发送给 AI”或“AI 处理中”并阻止重复发送同一批内容；用户继续修改属性时，新修改重新作为待发送草稿展示并允许再次发送。
- 已发送给 AI 的属性修改不再视为未发送草稿，但仍可保留在临时预览中，直到 AI 正式落地或用户继续调整。
- 用户可以在属性面板中填写补充说明；有明确属性变更时，AI 应以结构化属性变更为主，补充说明只作为上下文。没有属性变更但有选中元素和说明时，也可以发送给 AI。
- 用户再次点击顶部“选择”按钮退出编辑模式时，如果存在尚未发送的临时属性修改、配置项标记或补充说明，系统应立即提示用户确认是否丢弃；用户取消时继续停留在编辑模式，用户确认后丢弃未发送草稿并收起左右抽屉。
- 切换页面时属性抽屉应自动收起并停止元素点选；刷新或离开单页预览前，如果存在尚未发送的临时属性修改、配置项标记或补充说明，系统应要求用户确认是否丢弃；已经发送给 AI 的内容不触发该确认。

## 2026-07-01 单页面文档查看与下拉目录要求

- 单页面预览应能查看画布中可查看的文档和未隐藏的知识文档，形成与画布一致的文档阅读入口。
- 单页面预览右上角下拉目录应同时列出页面和文档，用户可在同一入口切换当前查看对象。
- 下拉目录应清晰区分页面和文档，并在收起态显示当前查看对象名称。
- 文档在单页面预览中为只读阅读对象，不进入页面配置、属性面板或 DOM 图层编辑链路。
