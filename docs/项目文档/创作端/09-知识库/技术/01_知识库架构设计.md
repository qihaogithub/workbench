# 知识库架构设计

## 一、整体架构

知识库采用“管理后台系统知识 + 工作空间用户知识”的双层结构。系统内置文档保存在 SQLite，由管理员统一维护；工作空间只保存用户项目知识。AI 每轮只获得轻量摘要索引，需要正文时再通过文件工具按需读取。

```

工作空间目录 (workingDir，仅用户知识)
  └── knowledge/
      ├── manifest.json        ← 用户文档元数据清单
      ├── 设计规范.md           ← 文档文件（.md 格式）
      ├── 组件说明.md
      └── ...

SQLite system_knowledge_documents
  └── 系统内置文档正文、摘要、关键词、启停状态、排序
```

## 二、数据模型

### 2.0 系统内置知识

系统内置知识由管理后台维护，核心字段包括稳定 ID、标题、描述、文件名、正文、分类、标签、启用状态、排序、版本、正文哈希、AI 摘要、AI 关键词和摘要状态。代码内置 Markdown 仅作为首次种子数据，缺失时插入，不覆盖管理员修改。

摘要状态用于判断索引质量：

- `ready`：已生成或由管理员手动确认
- `stale`：正文变化后等待生成
- `failed`：摘要生成失败，索引使用描述和正文摘录回退

### 2.1 Manifest 结构

`knowledge/manifest.json` 只管理用户文档。历史版本遗留的 `source: "system"` 条目会在目录确保流程中被剔除，系统文档由数据库动态合并。

```json
{
  "version": 1,
  "items": [
    {
      "id": "kb_1234567890_abc123",
      "title": "设计规范",
      "source": "user",
      "description": "组件库设计规范说明",
      "fileName": "设计规范.md",
      "addedAt": "2026-06-01T00:00:00.000Z",
      "updatedAt": "2026-06-01T00:00:00.000Z",
      "sizeBytes": 1024,
      "category": "配置与预览",
      "tags": ["config.schema.json", "配置项"]
    }
  ]
}
```

关键字段：
- `id`：由 `kb_` + 时间戳 + 随机字符串组成，保证全局唯一
- `source`：工作空间 manifest 中固定为 `user`
- `fileName`：依据标题自动生成的安全文件名，仅包含中文、英文、数字、下划线、连字符
- `category` / `tags`：可选索引字段，帮助 AI 在大规模知识库中筛选相关文档

### 2.2 文件命名规则

标题到文件名的转换逻辑：

1. 将非中英文、非数字、非下划线/连字符的字符替换为下划线
2. 合并连续下划线为单个
3. 去除首尾下划线
4. 如果文件已存在，自动追加 `_2`, `_3` 序号

## 三、API 设计

### 3.1 获取知识库列表

**GET** `/api/knowledge?workingDir={workingDir}`

读取启用的系统内置知识索引，并合并指定工作空间的用户 manifest。返回项按 `source` 区分：系统文档只读，用户文档可编辑。

### 3.2 添加文档

**POST** `/api/knowledge?workingDir={workingDir}`

请求体：`{ title, description, content }`

流程：
1. 验证 title 和 content 必填
2. 生成安全文件名（同名自动加序号）
3. 写入 `.md` 文件
4. 读取/初始化 manifest
5. 追加文档条目，设置 `source: "user"`
6. 写回 manifest

### 3.3 更新文档

**PUT** `/api/knowledge/{docId}?workingDir={workingDir}`

请求体：`{ title?, description?, content? }`

流程：
1. 查找 docId 对应的文档条目
2. 校验 `source !== "system"`（系统文档禁止修改）
3. 如有 content 则更新 `.md` 文件内容
4. 更新 manifest 中的元数据和更新时间

### 3.4 删除文档

**DELETE** `/api/knowledge/{docId}?workingDir={workingDir}`

流程：
1. 查找 docId 对应的文档条目
2. 校验 `source !== "system"`（系统文档禁止删除）
3. 删除对应的 `.md` 文件
4. 从 manifest 中移除条目

### 3.5 管理后台系统知识 API

Admin API 受管理后台鉴权保护：

- `GET /api/admin/knowledge`：列出全部系统知识，包括停用项
- `POST /api/admin/knowledge`：新增系统知识，保存后尝试生成摘要并推送到 agent-service
- `GET /api/admin/knowledge/{docId}`：读取单篇系统知识
- `PUT /api/admin/knowledge/{docId}`：更新系统知识；正文变化时标记摘要过期并尝试重生成
- `DELETE /api/admin/knowledge/{docId}`：删除系统知识
- `POST /api/admin/knowledge/{docId}/summary`：手动重新生成摘要

## 四、前端组件架构

```
编辑页（Demo/Project）
  │
  ├── KnowledgePanel        ← 知识库面板（侧边栏嵌入）
  │     │
  │     ├── 文档列表展示（标题、来源标签、时间）
  │     ├── 添加文档按钮 → 打开 KnowledgeDocDialog（新建模式）
  │     └── 点击文档 → 打开 KnowledgeDocDialog（查看模式）
  │
  └── KnowledgeDocDialog    ← 知识文档对话框
        │
        ├── 查看模式：Markdown 渲染展示
        ├── 编辑模式：Markdown 编辑器（memory-markdown-editor）
        └── 操作：保存/取消/删除
```

### 4.1 KnowledgePanel

展示工作空间知识库的所有文档，支持：
- 以列表形式展示文档标题、来源类型、时间
- 点击文档查看详情
- 添加新文档入口
- 系统文档和用户文档通过徽章区分

### 4.2 KnowledgeDocDialog

统一的文档查看/编辑对话框，支持：
- 查看模式：渲染 Markdown 内容
- 编辑模式：使用 Markdown 编辑器修改标题、描述、内容
- 系统文档仅可查看，用户文档可编辑和删除

## 五、与 AI 对话的集成

知识库在 AI 对话中的角色：

```
每轮对话前合并系统摘要索引和用户 manifest
       │
       ▼
轻量索引注入 L3 动态上下文
       │
       ▼
AI 根据任务自主选择相关文档
       │
       ▼
readFile / readFileWithLines 读取 knowledge/{文件名}
       │
       ▼
AI 参考文档正文完成代码或配置修改
```

知识库正文不会直接进入静态 System Prompt，也不会全部塞入用户消息前缀。创作端只把标题、管理员描述、AI 摘要、关键词、分类、标签和文件路径组成的索引注入 L3 动态上下文。AI 需要使用具体知识时，再通过只读文件工具读取对应 Markdown 文件。

系统文档不会复制到工作空间。author-site 在管理员保存后和构建 workspace context 前，把启用的系统知识快照推送给 agent-service。agent-service 收到快照后，在 `readFile` 和 `readFileWithLines` 中优先解析 `knowledge/{文件名}` 对应的虚拟系统文档；没有命中时再读取工作空间中的用户文档。

Agent Service 对 `knowledge/` 目录保留写保护：AI 可以读取知识库文档，但不能写入、编辑或删除知识库文件。
