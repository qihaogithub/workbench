# 创作端知识库架构设计

## 一、模块边界

创作端知识库模块只负责 UI、API 和现有工作空间数据接入：

- 工作空间用户知识文档 CRUD。
- 项目知识库文档与画布文档节点的同源引用。
- AI 对话前的项目知识轻量索引。
- 保存模板时调用项目模板流程，让独立知识库服务生成阅读地图。

知识来源优先级、权限过滤、阅读地图结构、索引任务状态和 `knowledgeReport` 报告契约见 [独立知识库服务](../../../独立知识库服务/)。

## 二、数据来源

### 2.1 工作空间用户知识

用户知识保存在项目工作空间：

```text
workingDir/
└── knowledge/
    ├── manifest.json
    ├── 设计规范.md
    └── 业务说明.md
```

`manifest.json` 只管理用户知识文档。历史遗留的系统知识 manifest 条目会在确保目录时被清理。

画布文档节点不再把正文作为长期来源。它保存知识库文档的轻量引用和画布布局，正文读取与保存仍通过工作空间知识文档 API 完成。聚合文档节点保存多份知识库文档引用和当前目录项 ID，用一个画布节点承载多篇原文档的查看入口，但不生成新的合并 Markdown 文件。历史布局中只有 `markdown` 正文的画布文档仍可显示；用户再次保存后会进入项目知识库并获得知识库引用。

用户知识文档同时进入项目内容图。保存或上传用户文档时，系统以 `docId` 为稳定资源 ID 创建 `knowledge_document` 资源版本；删除用户文档时，工作区移除 Markdown 和 manifest 条目，内容图写入 tombstone 提交，因此历史版本仍可用于恢复。`fileName` 只作为物化文件名，不作为长期身份。

### 2.2 模板知识

模板知识不由创作端知识 UI 直接维护。保存模板后，`project-core` 或 author-site 旧模板入口会调用独立知识库服务生成阅读地图。

## 三、API 设计

### 3.1 工作空间知识 API

- `GET /api/knowledge?workingDir={workingDir}`：返回用户知识 manifest。
- `POST /api/knowledge?workingDir={workingDir}`：新增用户知识文档。
- `GET /api/knowledge/{docId}?workingDir={workingDir}`：读取单篇用户知识元数据。
- `PUT /api/knowledge/{docId}?workingDir={workingDir}`：更新用户知识文档。
- `DELETE /api/knowledge/{docId}?workingDir={workingDir}`：删除用户知识文档。
- `GET /api/knowledge/content?workingDir={workingDir}&fileName={fileName}`：读取用户知识文档正文。

当请求同时携带 `projectId` 时，创建、更新和删除接口会把结果同步到项目内容图。对象历史与恢复通过项目资源 API 暴露：

- `GET /api/projects/{projectId}/resources/knowledge_document/{docId}/versions`：读取文档资源历史。
- `POST /api/projects/{projectId}/resources/knowledge_document/{docId}/versions`：为当前文档创建资源版本。
- `GET /api/projects/{projectId}/resources/knowledge_document/{docId}/versions/{versionId}`：读取指定文档版本。
- `POST /api/projects/{projectId}/resources/knowledge_document/{docId}/versions/{versionId}`：恢复单个文档到目标版本。

## 四、前端组件

### 4.1 KnowledgePanel

知识面板嵌入项目编辑页，负责展示文档列表、添加入口、查看入口和对象历史入口。列表只展示轻量字段，不加载所有正文。

### 4.2 KnowledgeDocDialog

文档对话框负责查看、编辑和删除用户知识文档。已有用户知识文档进入编辑模式时，会按 `knowledge/{fileName}` 接入实时协同通道；编辑器仍使用富文本界面编辑 Markdown 序列化内容，同一 Workspace 中的多个编辑页会共享同一 Yjs 文档。

保存已有知识文档时，前端先 flush 协同草稿，确保 `knowledge/*.md` 已落盘，再调用知识文档更新 API 写入描述、正文和 manifest 元数据。新增文档仍先走知识文档创建 API，创建成功后再具备可协同编辑的资源路径。

文档历史弹窗复用统一 `ResourceHistoryDialog`。用户从知识库菜单打开历史时，弹窗读取该文档的资源版本列表；恢复时只写回对应文档正文和 manifest 元数据，并触发知识库列表刷新，不影响页面或画布布局。

### 4.3 PreviewCanvas

画布接收项目编辑页传入的用户知识文档列表，并把列表同步为文档节点。每个文档节点只保存画布布局和知识库文档引用。画布需要展示正文时，通过项目编辑页提供的读取回调调用知识库内容 API；画布中新建或拖入 Markdown 文件时，通过创建回调先写入知识库，再把返回的文档元数据绑定到画布节点。

当知识库列表中出现新文档时，画布会补齐对应文档节点；当知识库文档被删除时，画布会移除引用该文档的节点，避免保留失效副本。聚合文档节点中的每个目录项都计入“画布已引用”的知识文档，因此同步时不会再为这些文档补单独节点；如果某个知识文档被删除，只移除聚合目录中的对应项，目录为空时才移除整个聚合节点。

画布文档节点还保存折叠状态和展开高度。折叠状态只影响画布展示：展开时渲染知识库正文，折叠时节点高度收缩到标题高度且只显示标题；再次展开时恢复折叠前的正文阅读高度。正文读写仍然走同一个知识库文档来源。

用户可以在画布中通过 Shift、Command 或 Ctrl 点击累加选择文档节点，也可以在未命中页面的框选中选中文档节点。合并后，原文档节点从画布语义层移除，聚合节点保留按画布位置排序后的目录项。画布记录这些原知识文档在当前布局中已隐藏，避免知识库同步重新生成单独卡片；知识库文件和 manifest 不受合并影响。

文档编辑弹窗复用共享 `DocumentEditor`。Markdown 文档进入编辑器时先转换为富文本结构；用户粘贴看起来像 Markdown 的纯文本时，编辑器会先按 Markdown 解析再插入当前光标位置，因此标题、列表、引用、代码块、链接和表格会立即以富文本形式呈现，保存时再序列化回 Markdown 正文。文档正文滚动区域使用细窄半透明滚动条，减少阅读和编辑时的视觉占用。

用户从画布删除知识库文档节点时，系统会记录该知识文档在当前画布中已隐藏。后续知识库列表同步会跳过这些已隐藏文档，避免删除后又被自动补回；知识库原文档仍保留在工作空间知识库中。

## 五、AI 对话接入

创作端构建 workspace context 时调用 `scanKnowledgeIndex()`，将用户知识 manifest 格式化为轻量文本。正文不直接进入每轮对话上下文。

AI 需要知识正文时：

```text
readFile / readFileWithLines
  → knowledge/{文件名}
  → 读取工作空间用户知识
```

AI 创建知识文档时，通过 `writeFile` 工具写入 `knowledge/{文件名}.md`。writeFile 检测到新建知识文档后会自动同步 `manifest.json`（追加条目并提交 Authority mutation），无需 AI 感知。工具返回 `knowledgeDocumentCreated` 标志，前端 `onToolUpdate` 回调检测到该标志后 dispatch `knowledge-updated` 事件，触发 KnowledgePanel 刷新和画布自动添加文档节点。

覆盖写入已存在的 `knowledge/*.md` 不会触发 manifest 同步（仅更新正文）。AI 编辑 `knowledge/manifest.json` 本身也不受限，但建议通过 writeFile 创建文档的方式间接维护 manifest。

需要结构化检索报告时，由 Agent Service 的 `knowledgeReport` 工具调用独立知识库服务能力，详见 [Agent 知识报告契约](../../../独立知识库服务/技术/04_Agent知识报告契约.md)。
