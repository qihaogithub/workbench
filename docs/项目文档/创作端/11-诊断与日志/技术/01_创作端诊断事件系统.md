# 创作端诊断事件系统

> 更新日期：2026-07-10

本文描述创作端结构化诊断事件的事件契约、写入链路、旧日志兼容和跨服务协作。业务侧目标见 [诊断与日志需求文档](../诊断与日志_需求文档.md)。

## 架构定位

诊断事件系统是创作端横向排查层，不拥有业务状态，也不参与业务决策。它像沿途的里程牌：记录关键动作发生过、在哪个上下文发生、最终停在成功还是失败，方便事后复盘。

主查询入口是 SQLite 事件库。JSONL 只作为 SQLite 不可用时的短期 spool、跨服务低耦合追加通道和导出包兜底片段。业务流程不能依赖事件库写入成功；诊断写入失败时，编辑、保存、AI 对话、预览和发布都继续执行。

## 统一事件契约

事件契约由 `packages/shared/src/diagnostics.ts` 维护。核心字段包括：

| 字段 | 说明 |
|:-----|:-----|
| `schemaVersion` | 事件契约版本，首版为 `1` |
| `ts` | ISO 时间字符串 |
| `source` | 事件来源，如 `frontend`、`author-api`、`agent-service`、`preview`、`ai-run`、`cli` |
| `level` | `debug`、`info`、`warn`、`error` |
| `eventGroup` | 事件组，如 `collab`、`autosave`、`ai`、`preview`、`project`、`publish` |
| `eventType` | 稳定事件名，如 `autosave.flush_succeeded` |
| `projectId`、`sessionId`、`workspaceId` | 项目、编辑会话和工作区关联字段 |
| `editorSessionId` | 单次打开编辑页的浏览器会话 ID |
| `traceId` | 串联一次短动作 |
| `operationId` | 串联一次长事务，如 AI 回复或发布 |
| `pageId`、`resourcePath` | 当前页面或资源 |
| `payload` | 白名单摘要，不保存大段正文或敏感信息 |

旧编辑页 JSONL 事件仍可按 `timestamp`、`category`、`name`、`details` 进入诊断层。诊断层负责把旧字段映射为新契约，业务调用方不需要一次性迁移。

## payload 治理

长期事件库的 payload 采用事件类型白名单。允许字段以 ID、路径、hash、长度、状态、错误码、耗时、计数和摘要为主。

默认禁止进入长期事件库的内容包括：

- 完整页面源码。
- 完整 AI prompt、完整模型回复和完整工具结果。
- token、cookie、Authorization、密钥和用户密码。
- 上传文件正文和数据库快照。

通用脱敏函数作为最后兜底：敏感 key 会被替换，大段文本会被摘要化，源码、schema、prompt、response、diff 和 toolResult 等字段只保留长度或结构摘要。

手绘保存链路通过 `page.sketch_patch_validated` 和 `page.sketch_patch_rejected` 区分 patch 校验成功和 patch 被拒绝。两类事件都只保留状态、操作数、是否存在基线 key、当前/目标节点数量、目标来源和失败原因摘要，不保存 scene、operations 或节点内容。目标来源只用于区分 `server_patch` 这类服务端回放，或 `client_scene` 这类客户端目标 scene 校验。

AI 发送前同步和自动保存 flush 链路需要保留可直接判定故障边界的摘要字段：`phase` 表示失败发生在当前页面落 session、协同房间 flush 还是工作区同步到 canonical；`errorCode` 和 `httpStatus` 保留服务端返回的机器可读错误；`message` 只保存短错误摘要。Codex 查询 `autosave.flush_before_ai_send_failed`、`autosave.flush_failed`、`autosave.exit_flush_failed` 和 `ai.before_send_failed` 时，应优先用这些字段判断是会话过期、workspace stale、权限拒绝还是服务不可达。

工作区物化到项目基准工作区时，author-site 记录 `workspace.canonical_materialization_started`、`succeeded` 和 `failed`。三类事件以 project/workspace/revision 组合作为 operation/trace，保留目标 revision、root hash 和耗时。`failed` 用 `reason` 细分 `WORKSPACE_STALE` 背后的权威链路状态：`missing_active_workspace` 表示项目缺少 active 指针，`active_workspace_mismatch` 表示请求工作区和项目 active 指针不一致，`workspace_not_found` 表示物化工作区缺失，`base_version_stale` 表示工作区基线版本落后，`canonical_write_failed` 表示写入项目基准工作区失败。payload 只保留工作区指针、版本、hash、阶段、耗时和错误码等元字段，不写入 `project.json` 全文或工作区文件内容。

Workspace Authority mutation 使用 `workspace.mutation_received`、`prepared`、`committed`、`conflicted`、`rolled_back` 和 `recovered` 组成完整生命周期。统一写入器强制每条事件包含 project ID、workspace ID、session ID、mutation ID、base revision、当前或 committed revision、actor、受管资源路径、trace ID 和耗时。mutation ID 同时作为 operation/trace 关联键；没有用户 Session 的启动恢复使用保留值 `workspace-authority`，避免持久化事件丢失 session 字段。received 阶段尚未得到 revision 时显式记为 `null`，而非省略字段。queue wait、commit latency 和稳定错误码作为阶段增量字段。请求中的文本、binary 字节、before backup 和 diff 不进入诊断 spool。external drift 还会记录独立 `workspace.external_drift_detected`，便于与普通资源 hash 冲突区分。

projection ack 按结果记录 `workspace.projection_applied` 或 `workspace.projection_failed`。如果客户端确认的 revision 低于 Authority 当前 revision，同时记录 `workspace.projection_gap_detected`，表示该预览面仍需要 catch-up。事件保留 mutation/trace、applied revision、current revision、client、surface、ack 时间、projection latency 和稳定运行时错误码；它只观察投影状态，不改变已提交 mutation。

## 写入链路

前端 `useEditorDiagnostics` 在浏览器内维护最近事件缓冲，并按短延迟批量提交到 author-site。author-site 正常路径只写入 SQLite 主库：

- `data/diagnostics/editor-events.db`

SQLite 写入开启 WAL 和 busy timeout，主表按项目、编辑会话、trace、operation、workspace 和事件类型建立索引。只有 SQLite 写入失败时，author-site 才把本批事件追加到 `data/editor-diagnostics/<editorSessionId>.jsonl` 作为 JSONL spool，并在诊断状态中标记 `jsonlFallbackUsed`、`dbUnavailable` 和 `eventGapDetected`。

agent-service 写入 AI run log 时，会同步追加结构化诊断摘要到 `data/editor-diagnostics/agent-service.jsonl`。Workspace Authority 的 mutation 生命周期和启动恢复也追加到这个 spool；recovered 事件保留统一关联字段、mutation 或 reconcile 模式以及 `rolled_back` / `committed_cleanup` 结果，不记录恢复的文件正文。该路径是跨服务低耦合 spool，不是长期主账本；CLI 读取时会将其标记为 JSONL fallback/spool。诊断 spool 写入失败不改变 mutation 或恢复结果，Authority journal/receipt/state 仍是业务与审计权威。

由于 author-site 事件进 SQLite，agent-service mutation/projection 事件进 JSONL spool，Workspace 全链路查询不能在查到 SQLite 后就停止读取 spool。诊断 CLI 对 autosave、collab、preview、project 和 export 查询会同时读取两类存储，按事件 ID 去重后统一按时间排序。这种合并只影响诊断视图，不改变 SQLite 主账本和 Authority journal/receipt 的权威边界。

## 事件分组

| 事件组 | 记录边界 |
|:-------|:---------|
| `project` | 项目打开、会话创建或复用、工作区绑定、基线版本变化 |
| `page` | 页面选择、资源加载、schema 加载、资源摘要变化和手绘 patch 校验 |
| `collab` | 协同连接、断开、快照接收、外部变更命中或遗漏 |
| `autosave` | 保存调度、flush、持久化、退出前保存和失败 |
| `ai` | 消息提交、run 开始、工具调用、文件变更、自动修复和 run 结束 |
| `preview` | 编译、iframe 加载、运行时错误和预览诊断 |
| `publish` | 发布前 flush、checkpoint、命名版本和发布结果 |
| `workspace` | Authority mutation/recovery、projection applied/gap/failed，以及 active 工作区 canonical materialization 的开始、成功和失败原因 |
| `ui`、`system` | 隐藏导出入口、快捷键、诊断能力启动和内部异常 |

高频事件默认不应无条件进入长期事件库。状态快照、资源 hash 变化和重复 runtime 诊断需要按 trace、资源路径、事件类型或 stack hash 合并、采样或只保留最近摘要。
