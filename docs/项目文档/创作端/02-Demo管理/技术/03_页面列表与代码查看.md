## 一、功能概述

### 1.1 功能定位

项目编辑页左侧栏提供**双 Tab 切换**机制：

- **AI 对话 Tab**：与 AI 代理交互，通过自然语言管理页面
- **页面列表 Tab**：可视化查看和管理项目中的所有页面，支持代码查看与页面操作

### 1.2 核心价值

| 价值点   | 说明                                                   |
| :------- | :----------------------------------------------------- |
| 双模操作 | 提供 AI 对话(自然语言)和页面列表(可视化)两种操作入口   |
| 代码透明 | 用户可直接查看页面源码，理解 AI 生成结果               |
| 高效管理 | 支持页面重命名、复制、删除等本地操作，无需通过 AI 对话 |
| 层级组织 | 支持虚拟文件夹分组，多页面项目易于管理                 |

## 三、页面操作功能

### 3.1 操作菜单

每个页面右侧的"更多"按钮(⋮)弹出操作菜单：

| 操作         | 图标 | 行为                           | 注意事项                                       |
| :----------- | :--- | :----------------------------- | :--------------------------------------------- |
| **查看代码** | 👁   | 弹窗展示页面完整代码（可编辑） | 支持 index.tsx 和 config.schema.json 切换      |
| **重命名**   | ✏️   | 弹出输入框，用户输入新名称     | 需校验名称唯一性                               |
| **复制到**   | 📋   | 复制页面为"{原名称} - 副本"    | 复制 index.tsx、config.schema.json、.demo.json |
| **移动到**   | 📦   | 选择目标文件夹                 | 子菜单展示文件夹树                             |
| **删除**     | 🗑️   | 弹出确认对话框                 | 允许删除最后一个页面，删除后项目为空           |

### 3.2 文件夹操作菜单

| 操作             | 行为                     | 注意事项                               |
| :--------------- | :----------------------- | :------------------------------------- |
| **重命名**       | 行内编辑文件夹名称       | 需校验名称唯一性                       |
| **新建子文件夹** | 弹出输入框创建嵌套文件夹 | 最大嵌套深度 3 层                      |
| **移动到**       | 选择目标文件夹           | 禁止循环嵌套                           |
| **删除**         | 二次确认删除             | 可选"仅删除文件夹"或"删除文件夹及内容" |

## 五、文件夹层级系统

### 5.1 虚拟文件夹设计

采用**虚拟文件夹**方案，文件夹仅存在于元数据层，物理目录保持扁平：

| 方案                   | 优点                                               | 缺点                                              |
| :--------------------- | :------------------------------------------------- | :------------------------------------------------ |
| **虚拟文件夹（采用）** | 物理目录不变；AI 操作路径不变；保存/恢复逻辑零改动 | 文件夹信息需额外持久化                            |
| 物理目录嵌套           | 目录即真相                                         | 路径解析全部重写；AI 需感知文件夹；保存流程复杂化 |

### 5.2 数据模型

**DemoFolderMeta（文件夹元数据）**：

```typescript
interface DemoFolderMeta {
  id: string; // "folder_{timestamp}_{random6}"
  name: string; // 文件夹显示名称
  parentId: string | null; // 父文件夹 ID，null 表示根级
  order: number; // 同级内的排序（小者在前）
  createdAt: number;
  updatedAt: number;
}
```

**DemoPageMeta 扩展**：

```typescript
interface DemoPageMeta {
  id: string;
  name: string;
  order: number;
  parentId: string | null; // 【新增】所属文件夹 ID，null 表示根级
  createdAt: number;
  updatedAt: number;
}
```

### 5.3 文件系统持久化

| 存储位置                    | 内容                                          |
| :-------------------------- | :-------------------------------------------- |
| `workspace/.folders.json`   | 文件夹元数据列表                              |
| `demos/{demoId}/.demo.json` | 页面元数据（含 parentId 字段）                |
| `project.json`              | 合并后的 demoFolders + demoPages（缓存/镜像） |

**保存时同步流程**：

```
用户点击保存
      │
      ▼
后端扫描 workspace/demos/ 目录 → 读取各 .demo.json → 读取 .folders.json
      │
      ▼
合并生成 project.json 的 demoPages + demoFolders
      │
      ▼
写入 project.json
```

### 5.4 文件夹约束

| 约束               | 值       | 原因                               |
| :----------------- | :------- | :--------------------------------- |
| 最大嵌套深度       | **3 层** | 过深嵌套降低可用性                 |
| 单文件夹内最大项数 | **50**   | 防止列表过长影响性能               |
| 循环嵌套防护       | 禁止     | 文件夹不能成为自身或子文件夹的子级 |

## 六、拖拽排序

### 6.1 技术选型

使用 `@dnd-kit/core` + `@dnd-kit/sortable`，与项目已有的 OrderControl 组件保持一致。

### 6.2 渲染策略

采用**扁平化列表 + 缩进**渲染树形结构：

```typescript
interface FlatTreeItem {
  item: DemoPageItem; // 文件夹或页面
  depth: number; // 缩进深度
  isExpanded: boolean; // 文件夹展开状态
  hasChildren: boolean; // 是否包含子项
}
```

**优势**：

- 与 `SortableContext` 的 `verticalListSortingStrategy` 天然兼容
- 拖拽计算简单（一维数组索引映射）
- 避免嵌套 `DndContext` 的复杂性

### 6.3 拖拽交互规则

| 操作                 | 行为                          | 视觉反馈              |
| :------------------- | :---------------------------- | :-------------------- |
| **同级排序**         | 拖拽页面/文件夹到同级另一位置 | 蓝色插入线            |
| **拖入文件夹**       | 拖拽页面到文件夹项上停留      | 文件夹高亮 + 展开动画 |
| **拖出文件夹**       | 拖拽页面到文件夹外区域        | 目标位置显示插入线    |
| **文件夹嵌套**       | 拖拽文件夹到另一文件夹上      | 父文件夹高亮          |
| **拖入已折叠文件夹** | 停留 500ms 后自动展开         | 文件夹展开动画        |

### 6.4 拖拽判定逻辑

```
拖拽结束（onDragEnd）
      │
      ├─ active 和 over 是同一项？→ 忽略
      │
      ├─ over 是文件夹？→ 将 active 移入该文件夹
      │   └─ parentId = over.id，order 追加到末尾
      │
      └─ over 是页面或空白区域？→ 将 active 插入到 over 旁边
          │
          ├─ 计算目标 parentId（取 over 的 parentId）
          │
          └─ 计算 target order（over 的 order ± 0.5，然后重排）
```

## 七、API 接口

### 7.1 页面管理

| 方法     | 路由                                       | 说明                                      |
| :------- | :----------------------------------------- | :---------------------------------------- |
| `GET`    | `/api/sessions/{sessionId}/files/{pageId}` | 获取页面代码（code + schema）             |
| `POST`   | `/api/projects/{projectId}/demos`          | 创建页面（支持 parentId 和 sourcePageId） |
| `PATCH`  | `/api/projects/{projectId}/demos/{demoId}` | 更新页面（name / order / parentId）       |
| `DELETE` | `/api/projects/{projectId}/demos/{demoId}` | 删除页面                                  |
| `POST`   | `/api/sessions/{sessionId}/files/{pageId}` | 保存页面代码修改                          |

### 7.2 文件夹管理

| 方法     | 路由                                           | 说明                                              |
| :------- | :--------------------------------------------- | :------------------------------------------------ |
| `POST`   | `/api/projects/{projectId}/folders`            | 创建文件夹（需 sessionId）                        |
| `PATCH`  | `/api/projects/{projectId}/folders/{folderId}` | 更新文件夹（name / parentId / order）             |
| `DELETE` | `/api/projects/{projectId}/folders/{folderId}` | 删除文件夹（query: `?deleteContents=true/false`） |

### 7.3 批量排序

| 方法    | 路由                                           | 说明                                    |
| :------ | :--------------------------------------------- | :-------------------------------------- |
| `PATCH` | `/api/projects/{projectId}/demo-pages/reorder` | 批量更新页面和文件夹的 order / parentId |

**请求体**：

```typescript
interface ReorderDemoPagesRequest {
  pages: Array<{
    id: string;
    order: number;
    parentId: string | null;
  }>;
  folders?: Array<{
    id: string;
    order: number;
    parentId: string | null;
  }>;
}
```

## 九、AI 代理适配

### 9.1 核心原则：AI 不感知文件夹

文件夹是 **UI 组织概念**，AI 操作页面时仍使用 `demos/{demoId}/` 路径。

**理由**：

1. 物理目录结构不变，AI 的文件读写路径不变
2. AI 不需要理解文件夹层级来定位页面
3. 减少上下文注入量，降低 AI 心智负担

### 9.2 上下文注入调整

AI 上下文中仅列出页面，不展示文件夹层级：

```
当前项目「我的项目」，包含 5 个页面：
  📄 "首页" → demos/demo_001/
  📄 "引导页" → demos/demo_002/
  📄 "产品列表" → demos/demo_003/
  📄 "产品详情" → demos/demo_004/
  📄 "关于我们" → demos/demo_005/
```

### 9.3 AI 页面管理操作扩展

| 自然语言示例                       | AI 行为                                                          |
| :--------------------------------- | :--------------------------------------------------------------- |
| "新建一个叫'产品中心'的页面"       | 调用 `POST /api/projects/{id}/demos`，不指定 parentId（根级）    |
| "在'产品模块'文件夹下新建一个页面" | 调用 `POST /api/projects/{id}/demos`，parentId 设为对应文件夹 ID |
| "把首页移到'首页模块'文件夹"       | 调用 `PATCH /api/projects/{id}/demos/{demoId}`，更新 parentId    |

AI 需要感知文件夹的场景极少（仅当用户明确提到文件夹时），默认不注入文件夹信息。
