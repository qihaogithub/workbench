# 页面模块 - 需求文档

> 版本：v1.0
> 创建日期：2026-04-06

---

## 一、模块定位

页面模块是 Web 前端系统的核心交互入口，负责为用户提供 Demo 管理和编辑的可视化界面。本模块承载用户与系统交互的全部场景，是用户感知系统价值的主要窗口。

---

## 二、功能描述

### 2.1 首页功能

#### 2.1.1 Demo 列表展示

**用户视角**：
- 用户进入首页后，系统应以卡片形式展示所有已创建的 Demo
- 每个 Demo 卡片应包含：预览图、名称、更新时间
- 卡片应按更新时间倒序排列，最新更新的 Demo 显示在最前面

**交互效果**：
- 当 Demo 列表为空时，系统应显示空状态提示，引导用户创建第一个 Demo
- 列表加载过程中，应显示加载状态

#### 2.1.2 Demo 搜索

**用户视角**：
- 用户在搜索框输入关键词后，系统应实时过滤 Demo 列表
- 搜索应支持按 Demo 名称模糊匹配

**交互效果**：
- 搜索结果为空时，应显示"未找到匹配的 Demo"提示
- 清空搜索框后，应恢复显示全部 Demo

#### 2.1.3 创建 Demo

**用户视角**：
- 用户点击"新建 Demo"按钮后，系统应弹出创建对话框
- 用户在对话框中输入 Demo 名称，点击"创建"后，系统应创建新 Demo 并跳转到编辑页面

**交互效果**：
- 名称输入框不能为空，为空时"创建"按钮应禁用
- 用户可通过按 Enter 键快速创建
- 创建成功后自动跳转到编辑页面

#### 2.1.4 删除 Demo

**用户视角**：
- 用户点击 Demo 卡片上的删除按钮后，系统应弹出确认对话框
- 用户确认删除后，系统应删除该 Demo 并从列表中移除

**交互效果**：
- 删除操作需要二次确认，防止误删
- 确认对话框应显示待删除 Demo 的名称
- 删除成功后，列表应立即更新

#### 2.1.5 进入 Demo 详情

**用户视角**：
- 用户点击 Demo 卡片主体区域后，系统应跳转到该 Demo 的使用页面

**交互效果**：
- 卡片主体区域应有视觉反馈（如悬停效果），提示可点击
- 点击删除按钮区域不应触发跳转

---

### 2.2 Demo 使用页面功能

#### 2.2.1 Demo 预览

**用户视角**：
- 用户进入 Demo 使用页面后，系统应在预览区实时渲染 Demo 组件
- 预览区应占据页面主要空间，提供清晰的预览效果

**交互效果**：
- 预览区应支持响应式布局，适应不同屏幕尺寸
- 编译或运行错误应在预览区友好展示

#### 2.2.2 参数配置

**用户视角**：
- 用户在配置面板修改参数后，预览区应实时反映配置变更
- 配置表单应根据 Demo 的配置定义自动生成

**交互效果**：
- 配置项变更后，预览应在 500ms 内更新
- 配置表单应支持多种输入类型（文本、数字、选择器、颜色选择等）

#### 2.2.3 进入编辑

**用户视角**：
- 用户点击"编辑 Demo"按钮后，系统应跳转到编辑页面

**交互效果**：
- 按钮应位于页面顶部显著位置

---

### 2.3 Demo 编辑页面功能

#### 2.3.0 页面类型

**用户视角**：
- 用户可在同一个项目中管理高保真页和原型页
- 普通创建与 AI 首次生成仍按系统默认页面类型进入，不要求用户预先理解技术差异
- 草图页能力在 SDK 单独开发完成前暂不在创作端开放；用户不应在页面列表或 AI 助手中看到草图页入口

**交互效果**：
- 页面列表的新增入口只开放普通页面、文件夹和 Figma 导入，不展示新建草图页
- 创作端 API 不接受新建草图页或切换为草图页的请求；已有底层草图数据只作为 SDK 研发和兼容数据保留
- AI 助手的工作区上下文不列出草图页，工具能力中也不默认暴露草图页对象级操作

#### 2.3.1 代码编辑

**用户视角**：
- 用户可在代码编辑区直接修改 Demo 的源代码
- 代码修改后，预览区应实时更新

**交互效果**：
- 编辑器应支持语法高亮
- 编辑器应支持 TypeScript 类型提示
- 代码变更应实时同步到预览

#### 2.3.2 AI 对话编辑

**用户视角**：
- 用户可在对话区输入自然语言指令，请求 AI 修改代码
- AI 修改代码后，预览区应实时更新
- 用户可查看 AI 的修改历史

**交互效果**：
- 对话区应支持流式响应，实时显示 AI 输出
- AI 修改过程中应显示处理状态
- 用户可随时中断 AI 操作

#### 2.3.3 配置 Schema 编辑

**用户视角**：
- 用户可在配置区修改 Demo 的配置定义（JSON Schema）
- Schema 修改后，配置表单应实时更新

**交互效果**：
- Schema 编辑器应支持 JSON 格式校验
- 无效 Schema 应显示错误提示

#### 2.3.4 保存与放弃

**用户视角**：
- 用户点击"保存"后，系统应将编辑内容合并到原始 Demo
- 用户点击"放弃"后，系统应丢弃所有编辑内容并返回上一页

**交互效果**：
- 保存前应确认用户意图
- 放弃前应提示未保存的变更将丢失
- 操作完成后应返回首页

## 三、业务规则与约束

### 3.1 命名规则

| 规则 | 说明 |
|:-----|:-----|
| Demo 名称必填 | 创建 Demo 时必须提供名称 |
| 名称长度限制 | 名称长度应在 1-50 个字符之间 |
| 名称唯一性 | 同一项目内 Demo 名称可重复，但建议保持唯一 |

### 3.2 编辑隔离规则

| 规则 | 说明 |
|:-----|:-----|
| 草稿机制 | 编辑操作在独立工作区进行，不影响原始 Demo |
| 自动过期 | 长时间未活动的编辑会话将自动清理 |
| 单会话限制 | 同一用户同时只能编辑一个 Demo |

### 3.3 权限控制

| 规则 | 说明 |
|:-----|:-----|
| AI 操作权限 | 敏感操作（如文件删除）需用户确认 |
| 代码执行限制 | 预览代码在沙箱环境中执行，无法访问用户系统 |
