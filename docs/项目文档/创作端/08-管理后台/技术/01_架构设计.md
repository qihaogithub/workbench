# 管理后台架构设计

## 技术架构概览

管理后台采用三层架构:

```
浏览器访问 → Middleware 鉴权 → 页面渲染 → API 读写配置 → SQLite 存储
                                    ↓
                         推送全局供应商配置到 agent-service
```

**核心技术栈**:

- 存储: SQLite `system_configs` 表(复用 `data/users.db`)
- 鉴权: Admin Secret + httpOnly Cookie(2 小时过期)
- 前端: Next.js App Router + shadcn/ui + `@dnd-kit` 拖拽
- 缓存: 内存缓存(TTL 1 分钟) + 环境变量 fallback

## 数据流转

### 配置读取流程

```
前端组件请求模型列表
  ↓
调用 /api/models/config (公开 API,无需 admin 权限)
  ↓
服务端读取 model-config.ts
  ↓
优先查询 SQLite system_configs 表
  ↓ (如无配置)
Fallback 到环境变量 (向后兼容)
  ↓
返回前端渲染
```

### 模型白名单写入流程

```
管理员修改配置 → 点击保存
  ↓
PUT /api/admin/model-config (需 admin 鉴权)
  ↓
写入 SQLite system_configs 表
  ↓
清除内存缓存
  ↓
前端收到成功响应,自动重新拉取配置
```

### 全局供应商写入流程

```
管理员在 AI 模型管理页编辑供应商
  ↓
填写 providerId / baseURL / API Key / 模型列表 / 默认模型
  ↓
PUT /api/admin/model-config
  ↓
写入 SQLite system_configs 表
  ↓
清除 author-site 模型配置缓存
  ↓
POST /internal/backend-providers 推送到 agent-service
  ↓
agent-service 更新运行时供应商配置
```

全局供应商配置服务于平台默认模型。普通用户如果在账号设置中保存了个人模型 API，该用户的会话会优先使用个人配置；相关链路见 [../05-AI对话/技术/04_用户模型配置.md](../../05-AI对话/技术/04_用户模型配置.md)。

### 模型拉取流程

```
管理后台点击"拉取模型"
  ↓
GET /api/admin/available-models (需 admin 鉴权)
  ↓
代理请求 agent-service /models
  ↓
Pi Agent 查询可用模型
  ↓
返回模型列表供管理后台展示
```

## 模块职责

### 鉴权模块 (`lib/admin-auth.ts`)

负责管理员身份验证:

- `verifyAdminRequest()`: 检查 URL 参数或 Cookie 中的密钥
- URL 参数验证: 直接比对明文 `ADMIN_SECRET`
- Cookie 验证: 比对哈希值(防伪造)
- 验证通过后返回包含哈希值的响应头,由 middleware 设置 Cookie

### 数据库配置模块 (`lib/db-config.ts`)

封装 SQLite 配置读写:

- `readDbConfig(id)`: 根据配置 ID 读取 JSON 配置
- `writeDbConfig(id, config)`: UPSERT 写入配置(INSERT ON CONFLICT DO UPDATE)
- 使用 `getDb()` 单例连接,确保路径正确(通过 `findProjectRoot()` 定位 monorepo 根目录的 `data/users.db`)

### 配置读取层 (`lib/model-config.ts`)

统一配置读取入口:

- `getModelConfig()`: 优先读数据库,fallback 到环境变量
- `normalizeConfig()`: 新旧配置结构双向转换
- 内存缓存机制: 避免频繁查询数据库
- 缓存失效: 保存配置后自动清除
- `backendProviders`: 保存平台级后端供应商列表、激活供应商和激活模型

### 模型过滤层 (`lib/ai-models.ts`)

应用配置到模型列表:

- `applyModelConfigsAsync()`: 通过公开 API 读取完整配置
- `applyModelConfigsWithFullData()`: 支持新结构(`enabledModels` + `autoEnableRules`)
- 精确启用列表模式: 只放行列表中的模型 + 匹配自动规则的新模型
- 自动按 `enabledModels` 顺序排序,自动启用的模型追加末尾

## 关键设计决策

### 1. 客户端模块隔离

**问题**: `ai-models.ts` 动态导入 `model-config.ts` 会导致 Next.js 追踪到 `better-sqlite3` 依赖链,客户端构建失败(`Module not found: Can't resolve 'fs'`)。

**解决方案**:

- 创建公开 API `/api/models/config`,在服务端读取配置
- `applyModelConfigsAsync()` 改用 `fetch()` 调用 API,彻底断开客户端到 `better-sqlite3` 的依赖
- 客户端组件通过 HTTP 请求获取配置,不直接依赖 Node.js 模块

### 2. 配置结构

当前使用 `enabledModels` + `autoEnableRules` 管理前端模型白名单，同时使用 `backendProviders` 管理后端供应商:

- `enabledModels`: 启用并按优先级排序的模型 ID 列表
- `autoEnableRules`: 自动启用规则(分组前缀/名称关键词)
- `backendProviders`: 平台级供应商列表，包含 baseURL、API Key、模型列表和默认模型
- `activeProviderId` / `activeModelId`: 平台默认供应商和默认模型
- `normalizeConfig()`: 负责配置数据的转换和校验

### 3. Agent-service ESM 兼容

**问题**: `@earendil-works/pi-agent-core` 是 ESM-only 包,但 agent-service 使用 CJS(`tsx`),静态导入导致模块解析失败。

**解决方案**:

- `PiAgentBackend` 改为动态 `import()` + try/catch
- 加载失败时优雅降级(不影响 Pi Agent 后端)
- 新增 `GET /models` HTTP 端点,复用 `OpenCodeHttpBackend.getModelInfo()`,不依赖 WebSocket session

## 路由设计

### 页面路由

| 路径                  | 说明             | 鉴权         |
| --------------------- | ---------------- | ------------ |
| `/admin`              | 管理后台概览首页 | Admin Secret |
| `/admin/models`       | AI 模型管理页    | Admin Secret |

### API 路由

| 方法 | 路径                          | 说明                         | 鉴权         |
| ---- | ----------------------------- | ---------------------------- | ------------ |
| GET  | `/api/admin/available-models` | 获取后端可用模型列表         | Admin Secret |
| GET  | `/api/admin/model-config`     | 获取当前模型配置             | Admin Secret |
| PUT  | `/api/admin/model-config`     | 更新模型配置                 | Admin Secret |
| POST | `/api/admin/backend-providers/sync` | 重新推送全局供应商配置到 agent-service | Admin Secret |
| POST | `/api/admin/reload-config`    | 手动清除配置缓存             | Admin Secret |
| GET  | `/api/admin/users`            | 获取所有注册用户列表         | Admin Secret |
| GET  | `/api/models/config`          | 公开模型配置(供前端组件调用) | 登录用户     |

### Agent-service 路由

| 方法 | 路径      | 说明                             |
| ---- | --------- | -------------------------------- |
| GET  | `/models` | 查询可用模型列表(供管理后台代理) |
| POST | `/internal/backend-providers` | 接收 author-site 推送的全局供应商配置 |
| GET  | `/internal/backend-providers` | 获取当前全局供应商配置(调试/验证用，API Key 脱敏) |

## 数据库设计

### system_configs 表

```sql
CREATE TABLE system_configs (
  id TEXT PRIMARY KEY,           -- 配置项唯一标识,如 "model_config"
  config_json TEXT NOT NULL,     -- JSON 格式的配置数据
  updated_at INTEGER NOT NULL,   -- 最后更新时间 (Unix timestamp)
  updated_by TEXT                -- 可选: 修改者标识
);
```

### 配置 JSON 结构

```typescript
{
  backendProviders: {
    providers: Array<{
      id: string;
      name: string;
      baseURL: string;
      apiKey: string;
      models: string[];
      defaultModel?: string;
      enabled?: boolean;
    }>;
    activeProviderId?: string;
    activeModelId?: string;
  },
  frontend: {
    enabledModels: string[];     // 启用并按优先级排序的模型 ID 列表
    autoEnableRules: Array<{     // 自动启用规则
      type: "prefix" | "nameFilter";
      value: string;
    }>;
  },
  multimodalModels: string[];    // 支持图片的模型 ID
}
```

## 缓存策略

### 内存缓存

- 位置: `model-config.ts` 模块级变量 `cachedConfig`
- TTL: 1 分钟(`CACHE_TTL = 60 * 1000`)
- 失效时机:
  - 保存配置后自动清除
  - 调用 `/api/admin/reload-config` 手动清除
  - TTL 过期后自动重新读取

### 多实例部署注意

当前使用内存缓存,多实例部署时需:

- 共享 SQLite 数据库(通过 NFS 或云数据库)
- 或改用 Redis 缓存(需额外实现)

## 安全机制

### Admin Secret 验证

1. URL 参数验证: `?secret=xxx` (首次访问)
2. Cookie 验证: `admin_token` (后续访问)
3. Cookie 属性: httpOnly, 2 小时过期
4. 存储方式: 数据库中存储哈希值,不存储明文

### Middleware 保护

在 `middleware.ts` 中拦截:

- `/admin` 路径: 验证失败重定向到 401 页面
- `/api/admin/` 路径: 验证失败返回 401 JSON 响应
- 验证通过后自动设置 Cookie

### 用户管理 API

`GET /api/admin/users` 返回所有注册用户的基本信息（不含密码等敏感字段），供管理员查看平台用户规模。

实现方式：调用 `lib/user.ts` 的 `listAllUsers()` 函数，从 SQLite `users` 表查询。

### 供应商凭证

管理后台的全局供应商 API Key 存储在 `system_configs` 的 JSON 中，访问入口受 Admin Secret 保护。对外调试读取 agent-service 当前配置时，agent-service 会对 API Key 做脱敏处理。普通用户自己的 API Key 不存在这里，而是存储在用户级配置中，见 [用户模型配置](../../05-AI对话/技术/04_用户模型配置.md)。

## 已知限制

1. **单实例部署**: 多实例需共享 SQLite 或改用 Redis 缓存
2. **配置版本管理**: 暂未支持配置历史与回滚功能
3. **内部同步依赖**: 实时推送依赖 author-site 与 agent-service 共享 `INTERNAL_API_TOKEN`
4. **全局与个人配置边界**: 管理后台只管理平台默认配置，用户个人配置以 session 级配置覆盖
