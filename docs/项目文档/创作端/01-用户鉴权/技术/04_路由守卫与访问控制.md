# 路由守卫与访问控制

## 1. 路由守卫概述

### 1.1 什么是路由守卫

路由守卫是在路由层面拦截用户访问的机制，用于：
- 保护需要登录才能访问的页面
- 防止已登录用户访问登录/注册页
- 实现自动重定向

### 1.2 实现方式

本项目使用 **Next.js Middleware** 实现路由守卫：

| 特性 | 说明 |
|:-----|:-----|
| **中间件位置** | `src/middleware.ts` |
| **拦截时机** | 请求到达页面之前 |
| **拦截范围** | 所有路由（除静态资源外） |
| **处理方式** | 验证身份 → 放行或重定向 |

## 2. 路由分类

### 2.1 受保护路由

需要登录才能访问的路由：

| 路由 | 说明 | 访问权限 |
|:-----|:-----|:---------|
| `/demo` | Demo 编辑页 | 已登录用户 |
| `/projects` | 项目列表页 | 已登录用户 |
| `/api/sessions` | 会话管理 API | 已登录用户 |

**访问控制逻辑**：
- 未登录访问 → 重定向到登录页（携带 redirect 参数）
- 已登录访问 → 正常显示页面

### 2.2 认证路由

仅未登录用户可访问的路由：

| 路由 | 说明 | 访问权限 |
|:-----|:-----|:---------|
| `/login` | 登录页 | 未登录用户 |
| `/register` | 注册页 | 未登录用户 |

**访问控制逻辑**：
- 未登录访问 → 正常显示页面
- 已登录访问 → 重定向到首页

### 2.3 公开路由

所有用户均可访问的路由：

| 路由 | 说明 | 访问权限 |
|:-----|:-----|:---------|
| `/` | 首页 | 所有用户 |
| `/demos/[id]` | Demo 使用页 | 所有用户 |
| 其他静态资源 | 图片、CSS、JS 等 | 所有用户 |

**访问控制逻辑**：
- 无需验证身份，直接访问

## 3. 中间件实现

### 3.1 中间件配置

**匹配规则**：

```
matcher: ['/((?!_next/static|_next/image|favicon.ico).*)']
```

**说明**：
- 匹配所有路由
- 排除 Next.js 内部静态资源（`_next/static`、`_next/image`）
- 排除 favicon.ico

### 3.2 中间件执行流程

```
请求到达
  ↓
中间件拦截
  ↓
从 Cookie 中提取 Token
  ↓
验证 Token（签名、有效期）
  ↓
解析用户身份
  ↓
判断路由类型
  ↓
├─ 受保护路由 + 未登录 → 重定向到登录页
├─ 认证路由 + 已登录 → 重定向到首页
└─ 其他情况 → 放行
```

### 3.3 Token 验证逻辑

**验证步骤**：

```
提取 Cookie 中的 auth_token
  ↓
调用 verifyToken(token)
  ↓
├─ Token 不存在 → 返回 null（未登录）
├─ 签名验证失败 → 返回 null（Token 无效）
├─ Token 已过期 → 返回 null（Token 过期）
└─ 验证成功 → 返回用户信息（已登录）
```

**关键点**：
- Token 验证在中间件中完成，无需查询数据库
- 验证失败不抛出错误，返回 null 表示未登录
- 验证成功返回用户信息，可用于后续处理

### 3.4 重定向逻辑

#### 未登录访问受保护路由

```
用户访问 /demo
  ↓
中间件检测到未登录
  ↓
构造登录页 URL：/login?redirect=/demo
  ↓
重定向到登录页
```

**重定向参数**：
- `redirect` 参数记录用户原始访问路径
- 登录成功后跳转到该路径

#### 已登录访问认证路由

```
用户访问 /login
  ↓
中间件检测到已登录
  ↓
重定向到首页：/
```

**说明**：
- 已登录用户无需再次登录
- 自动跳转到首页，提升用户体验

## 4. 重定向机制

### 4.1 登录后重定向

**场景**：用户访问受保护页面时未登录，登录后应跳转到原页面

**实现流程**：

```
用户访问 /demo（未登录）
  ↓
重定向到 /login?redirect=/demo
  ↓
用户在登录页输入账号密码
  ↓
登录成功
  ↓
检查 URL 中的 redirect 参数
  ↓
├─ 存在 redirect → 跳转到 /demo
└─ 不存在 redirect → 跳转到首页 /
```

**关键代码逻辑**：

```
const redirect = searchParams.get('redirect') || '/'
router.push(redirect)
```

### 4.2 注册后重定向

**场景**：用户注册成功后自动登录并跳转

**实现流程**：

```
用户在注册页提交注册信息
  ↓
注册成功（自动登录）
  ↓
跳转到首页 /
```

**说明**：
- 注册成功后立即设置 Cookie，实现自动登录
- 无需用户再次输入密码登录

### 4.3 登出后重定向

**场景**：用户登出后跳转到登录页

**实现流程**：

```
用户点击登出按钮
  ↓
清除 Cookie
  ↓
跳转到登录页 /login
```

**说明**：
- 登出操作在客户端完成，无需调用后端接口
- 清除 Cookie 后立即跳转

## 5. 认证页面设计

### 5.1 页面布局

认证页面（登录/注册）采用共享布局：

| 组件 | 文件路径 | 说明 |
|:-----|:---------|:-----|
| 布局组件 | `src/app/(auth)/layout.tsx` | 认证页面共享布局 |
| 登录页 | `src/app/(auth)/login/page.tsx` | 登录页面 |
| 注册页 | `src/app/(auth)/register/page.tsx` | 注册页面 |

**布局特点**：
- 居中显示表单
- 简洁的视觉设计
- 响应式布局

### 5.2 登录页实现

**页面结构**：

```
登录页
  ├── 标题："登录"
  ├── 描述："输入您的用户名和密码"
  ├── 登录表单
  │   ├── 用户名输入框
  │   ├── 密码输入框
  │   └── 登录按钮
  └── 注册链接："还没有账号？去注册"
```

**交互逻辑**：

```
用户输入用户名和密码
  ↓
点击登录按钮
  ↓
调用登录 API：POST /api/auth/login
  ↓
├─ 登录成功 → 跳转到目标页面
└─ 登录失败 → 显示错误提示
```

**错误处理**：
- 用户名或密码为空：提示"用户名和密码不能为空"
- 用户名或密码错误：提示"用户名或密码错误"
- 网络错误：提示"登录失败"

### 5.3 注册页实现

**页面结构**：

```
注册页
  ├── 标题："注册账号"
  ├── 描述："创建您的账号以开始使用"
  ├── 注册表单
  │   ├── 用户名输入框
  │   ├── 密码输入框
  │   └── 注册按钮
  └── 登录链接："已有账号？去登录"
```

**交互逻辑**：

```
用户输入用户名和密码
  ↓
点击注册按钮
  ↓
调用注册 API：POST /api/auth/register
  ↓
├─ 注册成功 → 自动登录 → 跳转到首页
└─ 注册失败 → 显示错误提示
```

**错误处理**：
- 用户名格式错误：提示"用户名至少 3 个字符"等
- 密码格式错误：提示"密码至少 6 个字符"
- 用户名已存在：提示"用户名已存在"
- 网络错误：提示"注册失败"

```
  ├── 用户名

  └── 登出按钮（红色）
```

**交互逻辑**：

```

  ↓

  ↓

  ↓

  ↓
```

### 6.3 登出实现

**登出流程**：

```
用户点击登出
  ↓
客户端清除 Cookie
  ↓
显示 Toast 提示："已登出"
  ↓
跳转到登录页
```

**关键代码逻辑**：

```
document.cookie = 'auth_token=; path=/; expires=Thu, 01 Jan 1970 00:00:00 GMT'
toast({ title: "已登出" })
router.push("/login")
```

**说明**：
- 登出操作在客户端完成，无需调用后端接口
- 纯 JWT 方案无法在服务端使 Token 失效
- 登出仅影响当前设备

## 7. API 访问控制

### 7.1 API 路由保护

部分 API 路由需要登录才能访问：

| API 路由 | 说明 | 访问权限 |
|:---------|:-----|:---------|
| `/api/sessions` | 会话管理 API | 已登录用户 |
| `/api/auth/me` | 获取当前用户 | 已登录用户 |

**保护方式**：
- 通过中间件拦截未登录请求
- 返回 401 状态码和错误信息

### 7.2 API 错误响应

**未登录访问受保护 API**：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "未登录"
  }
}
```

**Token 过期**：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "登录已过期"
  }
}
```

## 8. 前端状态管理

### 8.1 用户状态获取

**获取当前用户信息**：

```
前端调用 /api/auth/me
  ↓
API 从 Cookie 中提取 Token
  ↓
验证 Token
  ↓
├─ 验证成功 → 返回用户信息
└─ 验证失败 → 返回 401 错误
```

**使用场景**：
- 页面加载时获取用户信息
- 显示用户菜单中的用户名

### 8.2 状态同步

**多标签页状态同步**：

| 场景 | 处理方式 |
|:-----|:---------|
| 标签页 A 登录 | 标签页 B 刷新后获取最新状态 |
| 标签页 A 登出 | 标签页 B 刷新后跳转到登录页 |

**说明**：
- Cookie 在同一域名下的所有标签页共享
- 每个标签页独立验证身份
- 不支持实时同步（需刷新页面）

## 9. 安全考虑

### 9.1 重定向安全

**防止开放重定向攻击**：

| 措施 | 说明 |
|:-----|:-----|
| 相对路径 | redirect 参数仅接受相对路径（如 `/demo`） |
| 不接受完整 URL | 防止重定向到外部恶意网站 |

### 9.2 Cookie 安全

| 措施 | 说明 |
|:-----|:-----|
| httpOnly | 防止 JavaScript 读取 Cookie |
| sameSite: lax | 防止 CSRF 攻击 |
| secure（生产环境） | 仅通过 HTTPS 传输 |

### 9.3 错误信息

| 措施 | 说明 |
|:-----|:-----|
| 统一错误提示 | 登录失败不暴露具体原因（用户不存在或密码错误） |
| 不泄露系统信息 | 错误提示不包含技术细节 |

## 10. 性能优化

### 10.1 中间件性能

| 优化措施 | 说明 |
|:---------|:-----|
| 快速验证 | Token 验证无需查询数据库，耗时 < 10ms |
| 最小拦截范围 | 排除静态资源，减少不必要的拦截 |
| 边缘计算 | Next.js 中间件可在边缘节点执行 |

### 10.2 缓存策略

| 场景 | 缓存策略 |
|:-----|:---------|
| 认证页面 | 不缓存（动态页面） |
| 受保护页面 | 不缓存（用户相关） |
| 公开页面 | 可缓存（静态内容） |
