## 一、定位

钉钉企业登录用于让企业内部用户通过钉钉身份进入创作端，并把钉钉企业身份绑定到本地用户。该能力不替代原有用户名密码登录，而是作为企业内部部署下的登录入口和后续钉钉文档授权复用基础。

当前实现覆盖第一阶段：企业登录、账号自动创建、账号复用和绑定状态展示。钉钉文档读取、画布插入和文件变更事件同步仍属于后续阶段。

## 二、配置

服务端通过环境变量启用钉钉企业登录：

| 变量 | 说明 |
| :--- | :--- |
| `DINGTALK_LOGIN_ENABLED` | 设置为 `true` 时显式启用钉钉登录；也可通过完整应用配置自动启用 |
| `DINGTALK_CORP_ID` | 钉钉企业 ID，用于前端免登 code 请求和本地身份隔离 |
| `DINGTALK_APP_KEY` | 企业内部应用 AppKey |
| `DINGTALK_APP_SECRET` | 企业内部应用 AppSecret，只允许服务端使用 |
| `DINGTALK_LOGIN_AUTH_URL` | 可选；非钉钉容器中使用的跳转授权地址 |
| `DINGTALK_API_BASE_URL` | 可选；默认 `https://api.dingtalk.com` |
| `DINGTALK_OAPI_BASE_URL` | 可选；默认 `https://oapi.dingtalk.com` |

浏览器端只能读取脱敏后的启用状态、`corpId` 和可选授权跳转地址，不会获得 AppSecret 或 access token。

## 三、登录流程

登录页新增“使用钉钉企业账号登录”入口。

在钉钉容器内打开时，前端调用钉钉 JS 环境中的免登 code 能力，并把 code 提交给 author-site。若当前部署配置了 `DINGTALK_LOGIN_AUTH_URL`，登录页会在非钉钉容器中跳转到该地址。若回调 URL 携带 `dingtalkCode` 或 `authCode`，登录页会自动完成登录。

服务端收到 code 后：

1. 使用企业内部应用配置获取应用 access token。
2. 使用免登 code 换取钉钉用户 ID。
3. 读取钉钉用户详情，尽量获得 `unionId`、名称和头像。
4. 按 `corpId + unionId` 优先查找绑定；没有 `unionId` 时按 `corpId + dingtalkUserId` 查找。
5. 找到绑定则复用本地用户；找不到则创建一个不可密码登录的本地用户并写入绑定。
6. 复用现有 JWT Cookie 登录态，让后续创作端 API 继续使用现有鉴权机制。

## 四、身份存储

本地数据库新增 `user_dingtalk_identities` 表，用来保存钉钉身份与本地用户的绑定关系。

关键约束：

- 同一企业内 `unionId` 唯一绑定到一个本地用户。
- 同一企业内 `dingtalkUserId` 唯一绑定到一个本地用户。
- 删除本地用户时，钉钉身份绑定随用户删除。

自动创建的本地用户会生成随机密码哈希，不向用户展示，也不允许用户用该随机密码登录。用户后续仍可通过原有密码修改能力设置自己的本地密码。

## 五、绑定状态

设置弹窗的外部账号授权页会展示当前账号的钉钉企业身份：

- 已绑定时展示钉钉名称或 userId，以及企业 ID。
- 钉钉企业登录已启用但当前账号未绑定时，提示尚未通过钉钉企业登录绑定。
- 钉钉企业登录未配置时，展示未启用或配置缺失信息。

该状态只用于展示当前本地用户与钉钉企业身份的关系，不包含任何敏感凭据。

## 六、与外部授权的关系

钉钉企业登录解决“用户是谁”的问题；现有用户级外部工具授权解决“AI 能否用该用户权限访问钉钉文档”的问题。

两者的关系如下：

- 企业登录完成后，同一用户后续进入创作端不需要重新识别身份。
- 当前 dws 用户级授权仍按原逻辑保存和注入 AI session。
- 后续可把企业身份、企业应用 access token、用户级 token 或 dws 登录态统一收敛到授权中心。
- 不同用户之间仍不得共享用户级 token 或 dws 配置目录。

## 七、后续扩展

后续钉钉文档同步会复用本轮的身份绑定，并继续补齐：

- 钉钉文档链接解析。
- 文档读取与 Markdown 快照。
- 画布外部文档节点。
- 文件变更事件订阅。
- 后台刷新任务。
- AI 写操作确认与审计。

共享包已预留外部文档引用和快照类型，但当前未接入画布运行链路。
