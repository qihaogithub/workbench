## 一、需要准备的环境变量

企业内部钉钉登录需要以下变量：

| 环境变量 | 必填 | 来源 | 说明 |
| :--- | :--- | :--- | :--- |
| `DINGTALK_LOGIN_ENABLED` | 是 | 本系统配置 | 设置为 `true` 启用钉钉企业登录 |
| `DINGTALK_CORP_ID` | 是 | 钉钉开放平台企业信息 | 企业唯一标识 |
| `DINGTALK_APP_KEY` | 是 | 企业内部应用详情 | 企业内部应用 AppKey |
| `DINGTALK_APP_SECRET` | 是 | 企业内部应用详情 | 企业内部应用 AppSecret，只能放服务端 |
| `DINGTALK_LOGIN_AUTH_URL` | 可选 | 自行拼接或后续登录页 | 非钉钉容器中使用的跳转登录地址 |
| `DINGTALK_API_BASE_URL` | 可选 | 默认值 | 默认 `https://api.dingtalk.com` |
| `DINGTALK_OAPI_BASE_URL` | 可选 | 默认值 | 默认 `https://oapi.dingtalk.com` |

第一版最少只需要配置前四个变量。

## 二、进入钉钉开放平台

1. 打开 [钉钉开放平台](https://open.dingtalk.com/)。
2. 使用企业管理员账号，或已被授予开发者权限的钉钉账号登录。
3. 进入开发者后台。
4. 选择目标企业。

如果当前账号没有企业应用开发权限，需要企业管理员在钉钉管理后台或开放平台中授予开发者权限。

## 三、获取 CorpId

`DINGTALK_CORP_ID` 是企业标识，不是应用标识。

推荐方式：

1. 登录钉钉开发者后台。
2. 进入目标企业。
3. 在首页、企业基本信息或对应应用的开发管理页面查看企业 `CorpId`。
4. 复制到部署环境：

```bash
DINGTALK_CORP_ID=dingxxxxxxxxxxxxxxxx
```

钉钉官方说明中也提到，工作台应用首页地址里的 `$CORPID$` 只有从钉钉工作台打开应用时才会动态解析。因此本系统服务端仍应显式配置固定 `DINGTALK_CORP_ID`。

参考：[获取企业 CorpId](https://open.dingtalk.com/document/development/obtain-enterprise-corpid)

## 四、创建企业内部 H5 应用

1. 在钉钉开发者后台进入“应用开发”。
2. 选择“企业内部应用”。
3. 选择“钉钉应用”或“H5 微应用”。
4. 点击创建应用。
5. 填写应用名称、描述、图标等基础信息。
6. 创建完成后进入应用详情。

注意：

- 应用必须是企业内部应用。
- 开发模式应选择“开发应用”，不要选择只作为跳转入口的快捷链接模式；快捷链接模式可能没有 AppKey 和 AppSecret。

参考：[创建企业内部应用](https://open.dingtalk.com/document/development/create-an-h5-application-for-your-enterprise)

## 五、获取 AppKey 和 AppSecret

`DINGTALK_APP_KEY` 和 `DINGTALK_APP_SECRET` 来自刚创建的企业内部应用。

操作步骤：

1. 打开钉钉开发者后台。
2. 进入“应用开发”。
3. 进入“企业内部应用”。
4. 点击刚创建的应用。
5. 在应用详情的基础信息、凭证与基础信息或开发管理页面中找到 `AppKey`、`AppSecret`。
6. 复制到部署环境：

```bash
DINGTALK_APP_KEY=dingxxxxxxxxxxxxxx
DINGTALK_APP_SECRET=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

安全要求：

- `AppSecret` 只能放在服务端环境变量中。
- 不要提交到 Git。
- 不要写入前端代码。
- 不要在浏览器控制台、日志或错误响应中输出。

参考：[钉钉基础概念：AppKey 与 AppSecret](https://open.dingtalk.com/document/org/basic-concepts)

## 六、配置应用首页地址

在钉钉应用详情中配置应用首页地址，让用户可以从钉钉工作台进入创作端。

建议配置：

```text
https://你的创作端域名/login
```

如果希望登录后直接进入首页，也可以配置：

```text
https://你的创作端域名/
```

但第一版建议先配置 `/login`，因为登录页已经有“使用钉钉企业账号登录”按钮，并能在钉钉容器内拉起免登 code。

如果有 PC 端首页地址，也配置同一个可访问地址。

## 七、配置免登能力

本系统当前前端会优先使用钉钉容器里的 JSAPI：

```text
dd.runtime.permission.requestAuthCode
```

这个能力要求用户从钉钉客户端或钉钉工作台打开应用。前端拿到免登 code 后，会调用：

```text
POST /api/auth/dingtalk/login
```

服务端再用企业内部应用凭证换取钉钉用户身份，并写入本系统 JWT 登录态。

钉钉免登官方入口参考：[网页应用 H5 微应用免登](https://open.dingtalk.com/document/development/enterprise-internal-application-logon-free)

## 八、非钉钉容器登录地址

如果用户可能从普通浏览器打开创作端，可以额外配置：

```bash
DINGTALK_LOGIN_AUTH_URL=https://...
```

这个地址用于登录页在无法调用钉钉 JSAPI 时跳转到钉钉授权页面。

当前代码只读取并跳转该地址，不自动拼接授权 URL。正式生产前需要根据你在钉钉开放平台选择的登录方式确定最终 URL 格式，并保证回调回到：

```text
https://你的创作端域名/login?dingtalkCode=免登码
```

或：

```text
https://你的创作端域名/login?authCode=免登码
```

如果你的使用场景只从钉钉工作台进入，可以暂时不配置 `DINGTALK_LOGIN_AUTH_URL`。

## 九、部署环境示例

本地或 Docker 环境可以这样配置：

```bash
DINGTALK_LOGIN_ENABLED=true
DINGTALK_CORP_ID=dingxxxxxxxxxxxxxxxx
DINGTALK_APP_KEY=dingxxxxxxxxxxxxxx
DINGTALK_APP_SECRET=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

如果需要普通浏览器跳转登录，再增加：

```bash
DINGTALK_LOGIN_AUTH_URL=https://你的钉钉授权地址
```

修改环境变量后需要重启 author-site。

## 十、配置后如何验证

### 验证配置是否生效

登录页会请求：

```text
GET /api/auth/dingtalk/config
```

如果返回 `enabled: true`，说明服务端已读取到必要配置。

### 验证钉钉工作台免登

1. 从钉钉工作台打开应用。
2. 进入创作端登录页。
3. 点击“使用钉钉企业账号登录”。
4. 登录成功后进入创作端。
5. 打开设置弹窗，进入外部账号授权页。
6. 查看“钉钉企业身份”是否显示钉钉用户和企业 ID。

### 常见失败原因

| 现象 | 可能原因 | 处理方式 |
| :--- | :--- | :--- |
| 登录页提示钉钉登录未启用 | 环境变量未配置或服务未重启 | 检查前四个必填变量并重启 |
| 无法拉起钉钉登录 | 不是从钉钉容器打开，且未配置 `DINGTALK_LOGIN_AUTH_URL` | 从钉钉工作台打开，或补充跳转授权地址 |
| 服务端提示无法获取 access token | AppKey/AppSecret 错误，或应用类型不正确 | 回到应用详情核对凭证 |
| 服务端提示无法换取用户信息 | 免登 code 失效、应用未发布或权限不足 | 重新从钉钉打开应用，检查应用发布和权限 |
| 找不到 AppKey/AppSecret | 应用创建成了快捷链接模式 | 创建企业内部 H5 微应用并选择开发应用模式 |
