# CLI 自动维护定时任务需求文档

> 更新日期：2026-06-29

## 模块定位

CLI 自动维护定时任务是创作端 CLI 模块的自动化维护机制。它面向编码代理和项目维护者，用来持续检查 Web 创作端新增或变化的业务能力，并自动推动 CLI 跟进。

它不是新的用户操作入口，而是 CLI 能力长期保持可用、完整和可验证的维护流程。

## 背景与目标

CLI 的目标是让编码代理可以完成 Web 创作端除管理后台外的大部分业务操作。随着 Web 创作端持续迭代，如果每次都靠人工记住补 CLI，长期一定会遗漏。

定时任务的目标是让 CLI 维护从“人工记忆”变成“自动发现、自动补齐、自动验证、按风险处理”。

## 目标用户

| 用户 | 需要什么 |
|:-----|:-----|
| 编码代理 | 需要稳定、完整、机器可读的 CLI 能力 |
| 项目维护者 | 希望普通 CLI 补齐工作自动完成，只审核风险边界 |
| 功能开发者 | 希望 Web 新功能上线时，CLI 是否需要跟进有明确检查 |

## 功能需求

### 自动发现差距

系统需要定期检查 Web 创作端、领域服务和 CLI 命令之间是否存在能力差距。

当 Web 已经支持某个业务操作，而 CLI 没有对应能力时，系统应记录缺口，并给出影响说明。

### 自动维护 CLI

对低风险或规则明确的能力，系统应能自动补齐 CLI 相关内容，包括：

- CLI 命令入口。
- 机器可读输出。
- 命令测试。
- 能力清单。
- CLI 模块文档。

自动维护的前提是：业务规则已经明确，不需要定时任务自行解释产品语义。

### 自动验证结果

每次自动维护后，系统必须运行与改动范围匹配的验证。

验证通过后，系统才能进入合入或待审核阶段。验证失败时，系统应留下失败原因和下一步建议。

### 分级处理风险

系统必须按风险等级处理自动维护结果：

| 等级 | 处理方式 |
|:-----|:-----|
| 低风险 | 可以在验证通过后自动合入 |
| 中风险 | 可以自动完成改动，但需要人工审核 |
| 高风险 | 只能自动生成草案或方案，不允许自动合入 |

删除、回滚、发布、权限、鉴权和管理后台相关能力必须视为高风险或排除范围。

### 保留人工控制点

定时任务可以承担日常维护，但不能替代人工确认业务边界。

以下情况必须等待人工确认：

- 新业务规则不清楚。
- 涉及权限或安全边界。
- 涉及删除、回滚或发布。
- 涉及管理后台。
- 自动任务需要改变既有业务语义。

## 业务规则与约束

- CLI 仍然服务编码代理，不替代 Web 可视化体验。
- 定时任务只维护 CLI 与 Web 业务能力的对齐，不维护纯 UI 状态。
- 管理后台能力默认不纳入自动维护范围。
- 自动任务不能绕过高风险操作的预览、确认和审计规则。
- 自动任务不能自行新增业务规则，只能接入已经明确的业务能力。
- 自动任务必须留下可追溯记录，包括发现的缺口、生成的改动、验证结果和风险等级。
- 自动合入必须默认保守，先从查询类和文档类能力开始。

## 验收标准

- 系统能说明当前 CLI 覆盖了哪些 Web 业务能力。
- 系统能发现 Web 有能力但 CLI 未覆盖的情况。
- 系统能自动补齐低风险 CLI 缺口。
- 系统能给自动改动补测试和文档。
- 系统能在验证通过后处理低风险自动合入。
- 系统能把中高风险改动留给人工审核。
- 系统能说明每次自动维护为什么做、改了什么、验证结果是什么。
