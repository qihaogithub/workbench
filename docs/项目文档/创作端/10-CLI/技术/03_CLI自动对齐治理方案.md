# CLI 自动对齐治理方案

> 更新日期：2026-06-29

## 一句话结论

CLI 的迭代工作可以自动化完成：自动发现 Web 新能力、自动补 CLI 命令、自动补测试、自动更新文档、自动跑验证、自动创建 PR。

但“自动完成”不等于“所有改动都自动合入”。更稳妥的做法是：低风险能力可以自动合入，中高风险能力自动完成草案 PR 后等待审核。

## 目标

这个方案要解决的问题是：以后 Web 创作端新增项目、页面、配置、资产、发布、AI 会话、审计等能力时，CLI 的日常迭代由自动任务承担，不再要求开发者手工逐项维护。

理想状态是：

| 目标 | 通俗说明 |
|:-----|:-----|
| 能发现遗漏 | Web 加了新业务能力，CLI 没跟上时能自动提示 |
| 能自动补齐 | 自动任务可以补命令、测试、文档和能力清单 |
| 能自动验证 | 自动跑 CLI、领域服务和相关文档检查 |
| 能自动合入 | 低风险改动在验证通过后可以自动合入 |
| 能保留边界 | 高风险操作、权限边界和发布链路仍需要人工 review |

## 不追求什么

这个方案不追求把 Web 页面完全复制成命令行版。

CLI 对齐的是“业务能做什么”，不是“页面长什么样”。例如：

| Web 行为 | CLI 是否需要覆盖 | 原因 |
|:-----|:-----|:-----|
| 创建项目 | 需要 | 这是业务操作 |
| 修改页面代码 | 需要 | 代理需要批量维护 |
| 拖拽调整页面顺序 | 需要 | CLI 可以用排序参数表达 |
| 弹窗动画 | 不需要 | 只是视觉表现 |
| 表单展开收起状态 | 不需要 | 只是页面交互状态 |
| 管理后台全局配置 | 暂不纳入 | 当前目标明确排除管理后台 |

## 核心思路

把“自动维护 CLI”拆成六层。

| 层级 | 作用 | 可以自动化到什么程度 |
|:-----|:-----|:-----|
| 能力清单 | 说明 Web 有哪些业务能力 | 可以自动检查 |
| 命令元数据 | 说明 CLI 每条命令覆盖哪个能力 | 可以自动检查 |
| 契约测试 | 验证命令输入输出稳定 | 可以自动运行 |
| 自动实现 | 让代理补齐命令、测试和文档 | 可以自动完成代码改动 |
| 自动验证 | 跑类型检查、单测和对齐检查 | 可以自动决定是否达标 |
| 分级合入 | 按风险等级决定是否合入 | 低风险可自动合入，高风险只开 PR |

可以把它理解成“仓库里的自动维修流水线”。检查任务发现 Web 和 CLI 不一致后，自动代理先修，再跑验收；只有通过规则的低风险维修单才能自动关单。

## 能力对齐清单

CLI 能力对齐清单已经沉淀为 [CLI 能力自动化清单](./05_CLI能力自动化清单.md)，用来记录每个业务能力的状态。

清单至少包含这些列：

| 列 | 说明 |
|:-----|:-----|
| 能力编号 | 稳定标识，例如 `project.create` |
| 所属模块 | 项目、模板、页面、配置、资产、发布、AI、审计等 |
| Web 入口 | 用户在 Web 端从哪里触发 |
| 领域服务 | 业务规则最终由哪里处理 |
| CLI 命令 | 对应的命令 |
| JSON 输出 | 代理依赖的稳定字段 |
| 测试覆盖 | 是否有命令测试和领域服务测试 |
| 风险等级 | 普通、受保护、高风险 |
| 自动化等级 | 可自动报告、可自动 PR、可自动合入或必须人工审核 |
| 当前状态 | 已覆盖、部分覆盖、暂不覆盖、待确认 |

这张清单不只是文档，而是后续自动检查和自动实现的依据。

## Web 新功能的开发规则

以后 Web 新增业务功能时，建议按这个顺序走：

1. 先判断它是不是业务能力。
2. 如果只是 UI 体验，不要求 CLI 对齐。
3. 如果是业务能力，先确认领域服务是否已经承载规则。
4. 更新 CLI 能力对齐清单。
5. 给 CLI 增加或更新命令。
6. 给 CLI JSON 输出补契约测试。
7. 更新 CLI 文档。

在自动化模式下，第 4 到第 7 步可以由自动任务完成。人主要负责确认业务能力是不是要纳入 CLI、风险等级是否正确，以及高风险能力是否允许合入。

这里最重要的是第三步：业务规则要尽量放在领域服务里。Web 和 CLI 都调用同一套规则，CLI 才不会变成另一套影子系统。自动代理只负责把已有规则接到 CLI，不负责另写一套规则。

## 自动化等级

建议把 CLI 迭代自动化分为五级。

| 等级 | 自动化程度 | 适合范围 |
|:-----|:-----|:-----|
| L1 自动发现 | 只生成差距报告 | 所有能力 |
| L2 自动起草 | 自动改代码、补测试、补文档、开草案 PR | 大多数普通能力 |
| L3 自动合入 | 验证全绿后自动合入 | 低风险、只包装已有领域服务的能力 |
| L4 人工审核 | 自动完成 PR，但必须人工确认 | 创建、更新、提交、资产替换等写操作 |
| L5 禁止自动合入 | 只能自动生成方案或草案 | 删除、回滚、发布、权限、鉴权、管理员能力 |

这张分级表是自动任务的刹车系统。它允许 CLI 日常维护自动完成，同时避免删除、发布、权限这类关键链路被静默改动。

## 检查机制

建议增加一个 `check:cli-parity` 检查任务。

这个检查不需要一开始就很聪明，可以分阶段增强。

第一阶段只检查：

| 检查项 | 失败条件 |
|:-----|:-----|
| 能力清单存在 | 找不到清单 |
| CLI 命令元数据完整 | 命令没有声明覆盖能力 |
| 高风险能力有双阶段保护 | 删除、回滚、发布等命令缺少预览或确认说明 |
| JSON 输出测试存在 | 命令没有稳定输出测试 |

第二阶段再检查：

| 检查项 | 失败条件 |
|:-----|:-----|
| Web API 与能力清单匹配 | 新增业务 API 没有登记能力 |
| 领域服务与 CLI 命令匹配 | 新增领域服务方法没有 CLI 决策记录 |
| 文档与命令清单匹配 | 文档列出的能力在 `commands --json` 中找不到 |

## 自动维护任务怎么跑

定时任务建议每天或每周跑一次。具体执行规则见 [CLI 自动维护定时任务执行设计](./04_CLI自动维护定时任务执行设计.md) 和 [CLI 自动维护运行手册](./06_CLI自动维护运行手册.md)。它不是只做报告，而是按固定流水线完成 CLI 迭代：

1. 扫描 Web API、领域服务、CLI 命令元数据和能力清单。
2. 跑 CLI 对齐检查，找出缺口。
3. 判断缺口的自动化等级。
4. 对 L2 到 L4 缺口，启动编码代理补 CLI 命令、测试和文档。
5. 跑对应验证命令。
6. 生成变更报告。
7. L3 低风险改动自动合入。
8. L2 和 L4 改动保留草案 PR 或待审核 PR。
9. L5 改动只输出方案和风险说明。

报告里应该说清楚：

| 内容 | 示例 |
|:-----|:-----|
| 缺了什么 | `asset.replace` 有 Web 能力，但 CLI 未登记 |
| 影响什么 | 代理无法通过 CLI 替换资产 |
| 建议怎么补 | 增加 `asset replace` 命令和 JSON 输出测试 |
| 是否高风险 | 删除、发布、回滚类能力需要人工确认 |
| 自动化等级 | `L4 人工审核`，自动完成 PR 但不合入 |
| 验证结果 | `pnpm check:project-cli` 是否通过 |

## 自动实现的边界

编码代理可以自动生成实现改动，但必须遵守几个边界。

适合自动实现：

| 类型 | 说明 |
|:-----|:-----|
| 命令注册 | 已有领域服务能力，只是 CLI 没包装 |
| JSON 输出测试 | 输出结构明确，可以补契约测试 |
| 文档补齐 | 能力清单、索引和说明文档同步 |
| 普通查询命令 | 列表、详情、状态、摘要类命令 |

可以自动合入：

| 类型 | 条件 |
|:-----|:-----|
| 查询类命令 | 不改变远端数据，输出结构有测试 |
| 命令别名和帮助信息 | 不改变业务语义 |
| JSON 输出补字段 | 字段来自已有领域服务，兼容旧字段 |
| 文档和能力清单补齐 | 不改变业务规则 |

必须人工审核：

| 类型 | 原因 |
|:-----|:-----|
| 删除、回滚、发布 | 高风险操作必须人工 review |
| 权限规则变化 | 容易影响安全边界 |
| 业务语义不清 | 需要产品或开发者确认 |
| 绕过领域服务的实现 | 可能造成 Web 与 CLI 规则不一致 |

禁止自动代理自行决定：

| 类型 | 原因 |
|:-----|:-----|
| 新增业务规则 | 应先进入领域服务设计 |
| 改变权限边界 | 需要明确安全审查 |
| 改变发布产物语义 | 影响线上结果 |
| 跳过预览和确认 | 会削弱高风险操作保护 |

## 分阶段落地

### 第一阶段：先把账本建起来

目标是建立 CLI 能力对齐清单，并把现有命令补上能力编号。

完成标准：

| 检查点 | 标准 |
|:-----|:-----|
| 能力清单 | 覆盖现有 CLI 文档里的能力域 |
| 命令元数据 | `commands --json` 能说明每条命令覆盖什么能力 |
| 文档索引 | CLI 模块索引能找到对齐方案 |

### 第二阶段：让检查能跑起来

目标是增加 `check:cli-parity`，先检查最基础的缺漏。

完成标准：

| 检查点 | 标准 |
|:-----|:-----|
| 本地可运行 | 开发者能手动跑检查 |
| CI 可运行 | PR 中能看到检查结果 |
| 失败可理解 | 输出说明缺哪条能力、缺哪个测试 |

### 第三阶段：接入定时任务

目标是让系统定期发现遗漏，并自动完成低风险和中风险的 CLI 迭代 PR。

完成标准：

| 检查点 | 标准 |
|:-----|:-----|
| 周期执行 | 每天或每周自动跑 |
| 报告稳定 | 报告能列出新增、缺失和风险 |
| 自动实现 | 能自动补 CLI 命令、测试和文档 |
| 分级输出 | L3 自动合入，L4/L5 等待审核 |

### 第四阶段：允许低风险自动合入

目标是让代理处理低风险缺口，并在验证通过后自动合入。

完成标准：

| 检查点 | 标准 |
|:-----|:-----|
| 自动判断等级 | 每个改动都有 L1 到 L5 标记 |
| 自动验证 | PR 附带检查命令结果 |
| 自动合入受限 | 只有 L3 能自动合入 |
| 高风险标记 | 删除、发布、权限类改动必须突出提示 |

### 第五阶段：减少人工维护负担

目标是让人不再维护 CLI 细节，只维护规则和审核边界。

完成标准：

| 检查点 | 标准 |
|:-----|:-----|
| 人不手补普通命令 | 普通能力由自动任务补齐 |
| 人只看风险边界 | 重点审核权限、发布、删除和业务语义 |
| 自动任务可追溯 | 每次自动改动都有报告、测试和能力编号 |

## 验收标准

这个方案落地后，至少应该满足：

- Web 新增业务能力时，有明确位置登记是否需要 CLI。
- CLI 命令清单能被机器读取，不依赖人看帮助文案。
- 关键 CLI 命令都有稳定 JSON 输出测试。
- 检查任务能发现“Web 有能力、CLI 没命令”的情况。
- 定时任务能自动补齐 CLI 命令、测试和文档。
- 低风险改动能在验证通过后自动合入。
- 中高风险改动能自动生成 PR，但等待人工审核。
- 高风险操作仍保留预览、确认和人工 review。

## 风险与处理

| 风险 | 处理方式 |
|:-----|:-----|
| 自动化误判 Web 功能 | 允许在能力清单中标记“暂不覆盖”并说明原因 |
| CLI 复制了 Web 业务规则 | 要求先补领域服务，再补 CLI |
| 定时任务生成错误代码 | 低风险也必须先通过验证；中高风险只允许 PR |
| 能力清单没人维护 | 把清单检查放进 PR 流程 |
| 高风险命令被简化 | 删除、回滚、发布继续要求预览和确认 |
| 自动合入范围扩大过快 | 先只允许 L3 查询类能力自动合入，再逐步放开 |

## 推荐结论

建议采用“契约驱动的 CLI 自动迭代”，而不是完全放任的无人值守改代码。

简单说：让系统自动发现、自动实现、自动验证，并允许低风险改动自动合入；让人负责风险分级、业务边界和高风险审核。这样可以把 CLI 日常维护成本降下来，同时不把发布、权限和删除这类关键链路交给不受控的自动化。
