# 本地项目包与脚手架同步

> 更新日期：2026-07-05

## 入口边界

创作端项目编辑页和使用端项目详情页都不再提供右上角“导出代码”按钮，也不再保留 Web 专用 scaffold 下载接口。本地代码获取统一走 CLI 的 `project pull` 和本地项目包流程，避免 Web 可视化预览页与 CLI 形成重复入口。

## 技术定位

`@workbench/project-scaffold` 是创作端项目在“线上工作区”和“本地工程目录”之间转换的公共层。CLI 拉取项目、提交本地项目、模板本地化都复用这个包。

它的稳定面是项目包协议，而不是某个临时目录细节。线上项目仍由 `project-core` 管理；本地项目包只是一种适合代理开发的表达方式。

## 本地项目包结构

本地项目包以 `workbench.project.json` 为清单入口，包含以下核心信息：

| 字段 | 说明 |
|:-----|:-----|
| `schemaVersion` | 项目包协议版本 |
| `scaffoldVersion` | 脚手架托管文件版本 |
| `projectId` | 远端项目标识 |
| `baseVersion` | 拉取或提交后的远端版本基线 |
| `pages` | 页面列表、运行时类型、入口文件、Schema 文件、原型页/草图页运行时文件、父级和排序 |
| `folders` | 虚拟文件夹树 |
| `appGraph` | 应用逻辑图文件入口 |
| `projectConfig` | 项目级配置 Schema 文件 |
| `assetsDir` | 本地资产目录 |
| `knowledgeDir` | 本地工作空间知识文档目录 |

`.workbench/sync-state.json` 记录拉取时的文件哈希和版本基线，用于后续 diff 与提交后刷新。`.workbench/remote.json` 记录远端身份和来源信息，不保存真实密钥。

## 拉取流程

`project pull` 通过 `project-scaffold` 从 `project-core` 导出项目包条目，再写入本地目录。写出的内容包括项目清单、页面源码、页面 Schema、HTML/CSS 原型页文件、草图页 scene 文件、项目级 Schema、应用逻辑图、资产、工作空间知识文档、脚手架托管脚本和同步状态。

线上工作空间的 `knowledge/` 会映射到本地项目包的 `src/knowledge/`。该目录中的 Markdown 文档和 `manifest.json` 由项目包协议托管，提交后会回写到线上工作空间的 `knowledge/`，模板保存时继续被独立知识库服务读取并进入阅读地图。

Web 端不再把项目包条目打成 zip 响应。需要本地工程目录时，由 CLI 直接调用项目包协议完成拉取、校验和提交。

## 本地校验

`validate` 读取本地项目清单和文件，检查项目包基本形状、页面入口、页面 Schema、项目级 Schema、页面/文件夹层级、应用逻辑图入口和知识文档 manifest。

校验目标是快速发现本地项目包是否还能被创作端理解。通用前端类型检查、单元测试和构建由脚手架自己的脚本负责，CLI 不把这些工具内置成项目协议的一部分。

## Diff 与提交

`diff` 使用 `.workbench/sync-state.json` 里的哈希基线对比当前托管文件，输出新增、修改和删除摘要。代理可以先看 diff，再决定是否提交。

`submit` 会先校验本地项目包，再检查远端当前版本是否仍匹配 `baseVersion`。版本匹配时，它会打开一个 `project-core` 编辑事务，并同步页面新增、页面删除、文件夹变化、页面代码、HTML/CSS 原型页文件、草图页 scene 文件、页面 Schema、页面元信息、项目级 Schema、资产增删改和工作空间知识文档增删改。

提交成功后，`project-scaffold` 刷新本地 `baseVersion` 和 sync-state。这样提交本身不会继续留在下一次业务 diff 里。

## 模板本地化

`template init` 先从线上模板实例化项目，再把项目拉取成本地项目包。代理可以在本地完成开发、校验和预览。

`template submit` 先按普通项目包提交本地变更，再把提交后的项目保存为线上模板快照。模板本地开发因此仍然经过项目事务、版本和审计链路，不直接改模板内部存储；本地 `src/knowledge/` 中的知识文档会随项目提交进入模板快照，并由阅读地图索引为模板知识入口。

## 脚手架托管文件

默认脚手架写出自己的 `package.json` 和 `scripts/dev-server.mjs`。`pnpm dev` 启动轻量本地预览服务，`pnpm build` 做本地项目包可启动性检查。脚手架声明浏览器自动化依赖，用于本地真实截图检查。

本地预览服务读取项目级和页面级 Schema 默认值，按项目级默认值先进入、页面级默认值覆盖的顺序合并。这个方向需要与 [配置合并策略](../../04-配置与预览/技术/04_配置合并策略.md) 保持一致。

脚手架还提供 `pnpm preview:check` 和 `pnpm preview:screenshot`。这两个命令会启动临时本地预览服务并用真实浏览器访问页面路由，生成本地页面效果检查报告；其中 `preview:screenshot` 输出 PNG 截图工件。HTML/CSS 原型页按本地 `prototype.html`、`prototype.css` 和资产目录真实渲染；高保真 React 页和草图页在当前脚手架中以源码视图降级预览，并在报告中标注 `degraded` 与降级原因。

本地预览报告包含页面标识、运行时类型、预览 URL、截图路径、非空检测、失败请求、控制台错误、是否降级以及汇总统计。它用于代理提交前自检和证据留存；正式浏览器渲染、截图缓存和发布前复验仍以 author-site / screenshot-service 链路为准。

`project pull` 返回的 `nextActions` 会按本地开发主路径排序：进入目录、安装依赖、启动本地预览、运行 build、运行 preview 检查、运行项目包校验、查看摘要 diff、提交。代理可以只读取这组动作完成本地项目包开发闭环。

`upgrade` 只更新脚手架托管文件和版本元数据，不重写页面源码、页面 Schema 或资产文件。升级完成后同步托管文件的基线哈希，避免脚手架升级污染业务 diff。

## 失败与恢复

常见失败分为三类：

| 类型 | 处理方式 |
|:-----|:-----|
| 本地结构无效 | 返回校验问题和下一步建议，要求先修复项目包 |
| 远端版本冲突 | 拦截提交，提示重新拉取或处理冲突 |
| 事务中途失败 | 丢弃已打开的编辑事务，返回失败原因和恢复建议 |

CLI 层负责把这些失败转换为稳定 JSON；转换器层负责在失败时尽量避免留下半提交状态。

## 验证

项目包转换器验证优先使用：

```bash
pnpm check:project-scaffold
```

涉及 CLI 命令入口时同步运行：

```bash
pnpm check:project-cli
```

涉及 Web 入口边界时，还需要运行覆盖对应包的类型检查或测试，确认页面不再引用旧下载接口。
