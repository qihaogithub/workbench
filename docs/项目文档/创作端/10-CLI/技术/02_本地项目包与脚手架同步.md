# 本地项目包与脚手架同步

> 更新日期：2026-06-30

## 入口边界

创作端项目编辑页不再提供右上角“导出代码”按钮，也不再保留创作端专用 scaffold 下载接口。本地代码获取统一走 CLI 的 `project pull` 和本地项目包流程，避免 Web 可视化编辑页与 CLI 形成重复入口。

使用端项目详情页的“导出代码”按钮走公开 viewer 下载接口。公开接口只允许已发布且没有未发布改动的项目导出；未发布项目不可下载，已发布后又发生编辑的项目需要先重新发布。这个限制用于保证使用端看到的 demo 与下载到的代码包一致。

## 技术定位

`@opencode-workbench/project-scaffold` 是创作端项目在“线上工作区”和“本地工程目录”之间转换的公共层。CLI 拉取项目、提交本地项目、模板本地化和使用端公开下载能力都复用这个包。

它的稳定面是项目包协议，而不是某个临时目录细节。线上项目仍由 `project-core` 管理；本地项目包只是一种适合代理开发的表达方式。

## 本地项目包结构

本地项目包以 `opencode.project.json` 为清单入口，包含以下核心信息：

| 字段 | 说明 |
|:-----|:-----|
| `schemaVersion` | 项目包协议版本 |
| `scaffoldVersion` | 脚手架托管文件版本 |
| `projectId` | 远端项目标识 |
| `baseVersion` | 拉取或提交后的远端版本基线 |
| `pages` | 页面列表、入口文件、Schema 文件、父级和排序 |
| `folders` | 虚拟文件夹树 |
| `appGraph` | 应用逻辑图文件入口 |
| `projectConfig` | 项目级配置 Schema 文件 |
| `assetsDir` | 本地资产目录 |
| `knowledgeDir` | 本地工作空间知识文档目录 |

`.opencode/sync-state.json` 记录拉取时的文件哈希和版本基线，用于后续 diff 与提交后刷新。`.opencode/remote.json` 记录远端身份和来源信息，不保存真实密钥。

## 拉取流程

`project pull` 通过 `project-scaffold` 从 `project-core` 导出项目包条目，再写入本地目录。写出的内容包括项目清单、页面源码、页面 Schema、项目级 Schema、应用逻辑图、资产、工作空间知识文档、脚手架托管脚本和同步状态。

线上工作空间的 `knowledge/` 会映射到本地项目包的 `src/knowledge/`。该目录中的 Markdown 文档和 `manifest.json` 由项目包协议托管，提交后会回写到线上工作空间的 `knowledge/`，模板保存时继续被独立知识库服务读取并进入阅读地图。

使用端公开脚手架下载接口也调用同一套导出逻辑，只是把文件条目打成 zip 响应。这样 CLI 拉取和公开下载不会形成两套项目包结构。

## 本地校验

`validate` 读取本地项目清单和文件，检查项目包基本形状、页面入口、页面 Schema、项目级 Schema、页面/文件夹层级、应用逻辑图入口和知识文档 manifest。

校验目标是快速发现本地项目包是否还能被创作端理解。通用前端类型检查、单元测试和构建由脚手架自己的脚本负责，CLI 不把这些工具内置成项目协议的一部分。

## Diff 与提交

`diff` 使用 `.opencode/sync-state.json` 里的哈希基线对比当前托管文件，输出新增、修改和删除摘要。代理可以先看 diff，再决定是否提交。

`submit` 会先校验本地项目包，再检查远端当前版本是否仍匹配 `baseVersion`。版本匹配时，它会打开一个 `project-core` 编辑事务，并同步页面新增、页面删除、文件夹变化、页面代码、页面 Schema、页面元信息、项目级 Schema、资产增删改和工作空间知识文档增删改。

提交成功后，`project-scaffold` 刷新本地 `baseVersion` 和 sync-state。这样提交本身不会继续留在下一次业务 diff 里。

## 模板本地化

`template init` 先从线上模板实例化项目，再把项目拉取成本地项目包。代理可以在本地完成开发、校验和预览。

`template submit` 先按普通项目包提交本地变更，再把提交后的项目保存为线上模板快照。模板本地开发因此仍然经过项目事务、版本和审计链路，不直接改模板内部存储；本地 `src/knowledge/` 中的知识文档会随项目提交进入模板快照，并由阅读地图索引为模板知识入口。

## 脚手架托管文件

默认脚手架写出自己的 `package.json` 和 `scripts/dev-server.mjs`。`pnpm dev` 启动轻量本地预览服务，`pnpm build` 做本地项目包可启动性检查。

本地预览服务读取项目级和页面级 Schema 默认值，按项目级默认值先进入、页面级默认值覆盖的顺序合并。这个方向需要与 [配置合并策略](../../04-配置与预览/技术/04_配置合并策略.md) 保持一致。

`upgrade` 只更新脚手架托管文件和版本元数据，不重写页面源码、页面 Schema 或资产文件。升级完成后同步托管文件的基线哈希，避免脚手架升级污染业务 diff。

## 失败与恢复

常见失败分为三类：

| 类型 | 处理方式 |
|:-----|:-----|
| 本地结构无效 | 返回校验问题和下一步建议，要求先修复项目包 |
| 远端版本冲突 | 拦截提交，提示重新拉取或处理冲突 |
| 事务中途失败 | 丢弃已打开的编辑事务，返回失败原因和恢复建议 |

CLI 层负责把这些失败转换为稳定 JSON；转换器层负责在失败时尽量避免留下半提交状态。

## 验证

项目包转换器验证优先使用：

```bash
pnpm check:project-scaffold
```

涉及 CLI 命令入口时同步运行：

```bash
pnpm check:project-cli
```

涉及使用端公开下载脚手架接口时，还需要运行 author-site 对应 route 测试或 `pnpm check:author`。
