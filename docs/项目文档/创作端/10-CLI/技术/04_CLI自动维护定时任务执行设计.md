# CLI 自动维护定时任务执行设计

> 更新日期：2026-06-29

## 技术定位

CLI 自动维护定时任务是围绕 `@opencode-workbench/project-cli` 的自动化维护流水线。它读取能力清单、现有 CLI 命令、领域服务和 Web API 变化，发现 CLI 缺口后启动编码代理完成补齐，再运行验证并按风险等级决定后续处理。

它不是业务规则层。业务规则仍以 `project-core` 和相关模块文档为准；定时任务只负责把已经明确的能力接入 CLI。

## 输入来源

定时任务每次运行时，需要读取以下输入：

| 输入 | 作用 |
|:-----|:-----|
| [CLI 自动维护定时任务需求](../CLI自动维护定时任务_需求文档.md) | 判断任务目标和边界 |
| [CLI 自动对齐治理方案](./03_CLI自动对齐治理方案.md) | 判断自动化等级和治理策略 |
| [CLI 能力自动化清单](./05_CLI能力自动化清单.md) | 作为 Web 与 CLI 对账依据 |
| [CLI 自动维护运行手册](./06_CLI自动维护运行手册.md) | 指导自动代理实际执行 |
| CLI 命令注册入口 | 获取现有命令名称、别名和说明 |
| CLI 全命令测试 | 判断命令是否有基本契约覆盖 |
| 领域服务 | 判断业务能力是否已有确定性承载 |
| Web API 路由 | 判断 Web 是否新增业务入口 |
| 根验证脚本 | 决定本次改动应运行哪些检查 |

## 输出结果

每次定时任务必须产出结构化结果，方便人和代理继续处理。

| 输出 | 内容 |
|:-----|:-----|
| 差距报告 | 哪些能力缺 CLI、缺测试或缺文档 |
| 自动化等级 | 每个缺口对应 L1 到 L5 的处理级别 |
| 变更分支或 PR | 自动补齐后的代码和文档改动 |
| 验证结果 | 实际运行的验证命令、通过情况和失败原因 |
| 审核建议 | 是否可以自动合入，或需要谁确认什么 |

## 执行流程

### 1. 准备阶段

定时任务先确认仓库处于可执行状态，并读取本模块文档。它必须识别当前已有的用户改动，后续自动修改不能回滚或整理无关文件。

如果已有同类自动维护 PR 未处理，定时任务应优先更新已有 PR 或输出冲突说明，避免重复开多个相互覆盖的维护分支。

### 2. 扫描阶段

定时任务扫描四类信息：

| 扫描对象 | 关注点 |
|:-----|:-----|
| Web API | 是否出现新的项目、页面、配置、资产、发布、AI 或审计入口 |
| 领域服务 | 是否出现新的可复用业务方法或类型 |
| CLI 命令 | 命令是否覆盖能力清单中的能力 |
| 测试与文档 | 命令是否有 JSON 契约测试和文档记录 |

扫描结果不直接等于缺口。定时任务需要根据能力清单判断：该能力是已覆盖、待覆盖、暂不覆盖，还是必须人工确认。

### 3. 分级阶段

定时任务按 [CLI 能力自动化清单](./05_CLI能力自动化清单.md) 的自动化等级处理。

| 等级 | 自动任务动作 |
|:-----|:-----|
| L1 | 只生成报告 |
| L2 | 自动生成草案 PR |
| L3 | 自动实现、验证，通过后可自动合入 |
| L4 | 自动实现、验证，但等待人工审核 |
| L5 | 只生成方案和风险说明 |

如果一个改动同时包含多个等级，按最高风险等级处理。例如一个 PR 同时补查询命令和发布回滚能力，整体按 L5 处理。

### 4. 自动实现阶段

自动代理实现时必须遵守以下顺序：

| 步骤 | 要求 |
|:-----|:-----|
| 读取领域语义 | 先确认相关项目文档和领域服务已有能力 |
| 补 CLI 参数适配 | 只做参数读取、调用和 JSON 输出整理 |
| 补命令元数据 | 标记能力编号、风险等级和自动化等级 |
| 补契约测试 | 覆盖成功、失败和高风险保护路径 |
| 补文档 | 更新能力清单、技术文档和模块索引 |

自动代理不能把 Web 页面里的临时逻辑复制到 CLI。若领域服务缺能力，自动任务应先把该缺口标成待人工确认或单独发起领域服务改动。

### 5. 验证阶段

验证选择遵循最小但足够原则。

| 改动范围 | 必跑验证 |
|:-----|:-----|
| 只改 CLI | `corepack pnpm check:project-cli` |
| 改 CLI 和领域服务 | `corepack pnpm check:project-core`、`corepack pnpm check:project-cli` |
| 改本地项目包协议 | `corepack pnpm check:project-scaffold`、`corepack pnpm check:project-cli` |
| 改 Web API 与 CLI 对齐 | `corepack pnpm check:author`、`corepack pnpm check:project-cli` |
| 改共享类型 | `corepack pnpm check:project-core`、`corepack pnpm check:project-scaffold`、`corepack pnpm check:project-cli`、`corepack pnpm check:author` |

验证失败时，自动任务可以尝试修复低风险失败。若连续失败或失败涉及业务语义，必须停止并输出待人工确认事项。

定时任务应使用 `corepack pnpm`，不要直接调用全局 `pnpm`。这样可以固定使用仓库 `packageManager` 声明的 pnpm 版本，避免 Codex 运行环境的全局 pnpm 与锁文件不兼容。

### 6. 合入阶段

合入策略必须保守。

| 条件 | 处理 |
|:-----|:-----|
| L3 且验证全绿 | 可以自动合入 |
| L2 或 L4 | 保留 PR，等待人工确认 |
| L5 | 不合入，只保留方案或草案 |
| 验证失败 | 不合入 |
| 改动包含未知文件 | 不合入 |
| 改动触及密钥或环境变量 | 不合入 |

自动合入前必须再次确认改动范围只包含本次维护需要的文件。

## 失败处理

| 失败类型 | 处理方式 |
|:-----|:-----|
| 扫描无法判断能力 | 标记为待确认，不自动实现 |
| 领域服务缺能力 | 不在 CLI 中绕过实现，生成领域服务缺口说明 |
| 测试失败 | 保留失败日志和修复建议 |
| 文档冲突 | 保留 PR，等待人工处理 |
| 高风险能力误入自动合入 | 阻断合入并提升为 L5 |

## 记录要求

每次运行后，定时任务都要留下运行记录。记录至少包含：

- 运行时间。
- 扫描到的能力差距。
- 自动化等级判断。
- 实际修改文件。
- 运行的验证命令。
- 验证结果。
- 是否创建 PR、是否自动合入。
- 剩余风险和待人工确认项。
