# 02 配置入口治理

## 设计目标

配置入口治理用于减少同一配置在多个文件中重复解析、默认值不一致和环境变量命名漂移。它不要求一次性重构所有历史代码，但要求后续新增或调整配置时有固定入口。

## 分层规则

| 层级 | 职责 |
|---|---|
| 服务内 `env` 或 `config` 文件 | 读取 `process.env`、提供默认值、做格式校验 |
| 共享契约层 | 放跨包常量、API response、事件结构和协议字段 |
| 业务代码 | 只消费已解析配置，不直接散落读取 `process.env` |
| 文档与示例 | 说明变量用途、默认值和是否可暴露到浏览器 |

## 当前优先治理项

| 配置类型 | 风险 | 治理方式 |
|---|---|---|
| Agent 服务地址 | author-site、viewer-site、project-core 可能使用不同变量名或默认值 | author-site 与 project-core 已建立服务内配置入口；viewer-site 保留浏览器公开配置 |
| Session 过期时间 | 多处重复常量容易漂移 | 提取到共享常量或服务配置入口 |
| 模型白名单/黑名单 | 解析逻辑重复会造成 UI 与后端判断不一致 | 保留一个解析函数，其他位置复用 |
| 数据目录 | `DATA_DIR` 影响 `data/` 诊断和测试隔离 | 所有服务读取时都应支持覆盖，并在测试中使用临时目录 |
| 浏览器公开变量 | `NEXT_PUBLIC_*` 会进入前端产物 | 文档中明确是否允许暴露 |

## 已落地入口

| 包 | 配置入口 | 当前覆盖 |
|---|---|---|
| author-site | `packages/author-site/src/lib/runtime-config.ts` | Agent 服务地址、内部 token、截图服务地址、截图代理超时 |
| project-core | `packages/project-core/src/config.ts` | viewer 发布地址、agent-service 地址、screenshot-service 地址 |
| agent-service | `packages/agent-service/src/utils/config.ts` | 服务端口、日志、截图服务地址、Pi Agent 基础配置、限流配置 |
| author-site 模型配置 | `packages/author-site/src/lib/runtime-config.ts`、`packages/author-site/src/lib/model-config.ts` | 浏览器公开模型白名单、黑名单、默认模型和名称过滤器的环境变量 fallback |
| project-core Project Admin | `packages/project-core/src/config.ts` | 数据目录、审计目录、批量上限、CLI mode、默认 actor |

## 新增配置约束

- 新增配置前先确认是否已有同义变量。
- 新增服务端配置时，优先放入服务内配置入口。
- 新增跨包配置时，先判断它是运行时配置还是协议常量；只有协议常量进入共享层。
- 新增 `NEXT_PUBLIC_*` 时必须说明为什么需要暴露给浏览器。
- 测试中需要覆盖环境变量时，必须在测试结束后恢复原值。

## 验证方式

- 配置入口或 package scripts 变更后运行 `corepack pnpm check:repo`。
- 涉及 viewer 数据源时运行 `corepack pnpm check:viewer` 或 `node scripts/check-viewer-contracts.mjs`。
- 涉及 Agent 服务地址、stream event 或 session 字段时运行 `corepack pnpm check:agent`。
