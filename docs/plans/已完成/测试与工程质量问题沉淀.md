## 两处既有验证失败（与 AI 架构统一改造无关，2026-07-21 确认）

- **现象**：`pnpm check:all` 中两项失败：
  1. `check:workspace-authority` 门禁要求测试包含"孤立 Authority state 没有 live Workspace 时**阻止启动**"，但 `workspace-authority-startup-recovery.test.ts` 实际用例名为"…时**自动清理陈旧条目**"——测试语义已改，门禁脚本字符串未同步。
  2. `check:project-cli` 的 `cli.test.ts` 断言 `invalidUpdateRuntime.ok === false`（期望 `react/jsx-runtime` import 被运行时校验拒绝，`AUTHORING_RUNTIME_IMPORT_UNSUPPORTED`），实际返回 `ok: true`；preview-contract 自身同校验码的单测通过。
- **确认方式**：在 HEAD（349dca22）干净 worktree 独立 `pnpm install` 后复跑，两项**同样失败**，确认为既有问题，非 2026-07-21 AI 架构统一改造引入。
- **当前结论**：workspace-authority 一项需 Authority 模块维护者裁决是"门禁字符串过期"还是"测试削弱了启动保障"后同步修正；project-cli 一项需排查 CLI 的 runtime validation 链路为何未触发 import 拒绝。
- **后续事项**：
  - [ ] 裁决并修正 `scripts/check-workspace-authority-guards.mjs` 第 975 行断言与测试用例的语义一致性。
  - [ ] 排查 `packages/project-cli` `page update-code` 的 runtimeValidation 未拒绝 `react/jsx-runtime` 的原因。
- **相关命令**：`pnpm check:workspace-authority`、`pnpm check:project-cli`

## pnpm node_modules 损坏导致 MODULE_NOT_FOUND（2026-07-19）

- **现象**：`pnpm preview:local` 报两个 MODULE_NOT_FOUND：
  1. `packages/preview-contract/src/runtime.ts` 找不到 `typescript` 模块（agent-service 启动时通过 tsx 加载 preview-validation → preview-contract/runtime）
  2. Next.js webpack build 找不到 `./impl` 模块
- **根因**：pnpm 的 `node_modules` 状态损坏。`node_modules/.pnpm/typescript@5.9.3/node_modules/typescript/` 目录存在但为空（只有空目录，无实际文件），导致 Node.js 无法解析 `typescript` 模块。其他包（Next.js 的 webpack `./impl`）也存在类似空目录问题。
- **修复**：`pnpm install --force` 强制重新安装全部依赖（1685 个包），覆盖损坏的目录结构。
- **验证**：
  - ✅ `require.resolve('typescript')` 成功
  - ✅ agent-service typecheck 通过
  - ✅ `build:preview-runtime` 成功生成
  - ✅ `pnpm preview:local --build-only` 构建完全成功（所有路由 + Middleware）
- **后续**：如果再次遇到类似 MODULE_NOT_FOUND 但文件明显存在的情况，优先 `pnpm install --force` 修复 pnpm store 状态，而不是逐个排查模块。
- **相关命令**：`pnpm install --force`
