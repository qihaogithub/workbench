## 一、问题回顾

### 遗留问题（P5）

**问题描述**：AI 在执行过程中修改文件（但未完成回复），前端无法感知，导致预览区和配置面板无法实时更新。

**根因**：
- ✅ 服务端已实现 `file_operation` 事件的实时 WebSocket 推送
- ❌ 前端 `ai-chat.tsx` 仅用 `console.log` 打印，未处理
- ❌ 无累计文件变更状态管理
- ❌ 无实时触发预览刷新的机制

---

## 二、实施方案

### 核心思路

**利用已有的 `file_operation` WebSocket 事件**，在前端实现实时文件变更追踪和自动刷新。

**方案优势**：
- ✅ 无需修改后端代码（推送机制已就绪）
- ✅ 真正实现实时同步（AI 写入文件后立即触发更新）
- ✅ 保留现有的 `finish` 事件兜底方案（双重保障）
- ✅ 支持长时间任务的中间结果展示
- ✅ 防抖处理（100ms 批量通知，避免频繁触发）

## 三、修改内容

### 修改文件：`packages/web/src/components/ai-elements/ai-chat.tsx`

#### 3.1 添加实时文件变更追踪逻辑

**位置**：`handleSend` 函数内部，`stream.on("file_operation", ...)` 监听器

**修改前**（第 277-279 行）：
```typescript
// 监听文件操作(可选,用于日志记录)
stream.on("file_operation", (event: StreamEvent) => {
  console.log("File operation:", event.fileOperation);
});
```

**修改后**（第 277-346 行）：
```typescript
// 监听文件操作（实时文件变更追踪）
const realtimeFilesRef = new Map<
  string,
  { action: string; content?: string }
>();
let fileUpdateTimer: NodeJS.Timeout | null = null;

// 处理实时文件变更的辅助函数
const processRealtimeFiles = () => {
  const files = Array.from(realtimeFilesRef.entries()).map(([path, info]) => ({
    path,
    action: info.action as "created" | "modified" | "deleted",
    content: info.content,
  }));

  // 通知父组件
  if (files.length > 0) {
    onFilesChange?.(files);

    // 实时提取代码和 schema 更新
    for (const file of files) {
      if (
        (file.path.includes("index.tsx") || file.path.includes("index.ts")) &&
        file.content
      ) {
        onCodeUpdate?.(file.content);
      } else if (file.path.includes("config.schema.json") && file.content) {
        onSchemaUpdate?.(file.content);
      }
    }
  }
};

stream.on("file_operation", (event: StreamEvent) => {
  if (event.fileOperation) {
    const { method, path, content } = event.fileOperation;

    // 仅处理文件写入操作
    if (method === "fs/write_text_file" && path) {
      // 更新累计文件变更（去重）
      realtimeFilesRef.set(path, {
        action: "modified",
        content,
      });

      // 防抖：100ms 后批量通知
      if (fileUpdateTimer) {
        clearTimeout(fileUpdateTimer);
      }

      fileUpdateTimer = setTimeout(() => {
        processRealtimeFiles();
        fileUpdateTimer = null;
      }, 100);
    }
  }
});
```

**关键改进**：
1. ✅ 使用 `Map` 数据结构去重（同一文件多次写入仅保留最新内容）
2. ✅ 添加防抖处理（100ms 批量通知，避免频繁触发）
3. ✅ 实时调用 `onCodeUpdate`/`onSchemaUpdate` 触发预览和配置面板刷新
4. ✅ 实时调用 `onFilesChange` 通知父组件

#### 3.2 修改 `finish` 事件处理逻辑

**位置**：`stream.on("finish", ...)` 监听器

**修改前**（第 368-392 行）：
```typescript
stream.close();
streamRef.current = null;

// 处理文件变更
if (event.files && event.files.length > 0) {
  onFilesChange?.(event.files);
  // ... 文件提取逻辑
}
```

**修改后**（第 368-414 行）：
```typescript
stream.close();
streamRef.current = null;

// 清理文件更新定时器
if (fileUpdateTimer) {
  clearTimeout(fileUpdateTimer);
  fileUpdateTimer = null;
}

// 处理文件变更（优先使用 event.files，兜底使用 realtimeFiles）
const finalFiles =
  event.files && event.files.length > 0
    ? event.files
    : Array.from(realtimeFilesRef.entries()).map(([path, info]) => ({
        path,
        action: info.action as "created" | "modified" | "deleted",
        content: info.content,
      }));

if (finalFiles.length > 0) {
  onFilesChange?.(finalFiles);
  // ... 文件提取逻辑
}

// 清空实时文件缓存
realtimeFilesRef.clear();
```

**关键改进**：
1. ✅ 清理防抖定时器，避免内存泄漏
2. ✅ 合并最终文件列表（优先使用 `event.files` 作为权威数据源，兜底使用 `realtimeFiles`）
3. ✅ 清空实时缓存，避免状态残留

## 七、已知限制

### 7.1 非流式模式（HTTP 降级）

**限制**：非流式模式（WebSocket 失败时降级到 HTTP）不支持实时文件同步，因为 HTTP 是请求-响应模式，文件变更会在响应中一次性返回。

**影响**：轻微。非流式模式本身就不支持实时更新，这是预期行为。

**缓解措施**：保留现有的 `finish` 事件处理逻辑，确保最终状态正确。

```typescript
import { create } from "zustand";

interface FileStore {
  files: Map<string, FileChange>;
  updateFile: (path: string, content: string) => void;
  resetFiles: () => void;
}

export const useFileStore = create<FileStore>((set) => ({
  files: new Map(),
  updateFile: (path, content) =>
    set((state) => {
      const newFiles = new Map(state.files);
      newFiles.set(path, { path, content, action: "modified" });
      return { files: newFiles };
    }),
  resetFiles: () => set({ files: new Map() }),
}));
```
