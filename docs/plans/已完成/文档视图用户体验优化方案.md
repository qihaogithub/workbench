# 文档视图用户体验优化方案

## 背景与目标

当前「文档模式」界面存在以下问题：
- AI 记忆文件（memory.md）与普通文档混在一起，重要性不突出
- 文件结构对用户友好但对 Agent 不友好，无法满足不同视角的需求
- 添加文件的操作入口不够清晰

**优化目标**：
- 提升 AI 记忆的可发现性和操作便捷性
- 分离用户视角与 Agent 视角的文件组织逻辑
- 优化文件添加的交互体验

---

## 方案设计

### 1. 顶部「AI记忆」横幅模块

**位置**：文件视图顶部，与文件树分离

**视觉设计**：
- 独立的卡片/横幅区域
- 白色或浅灰色背景，与下方文件树形成视觉分隔
- 图标使用简洁的脑形图标或 AI 相关图标（纯色/灰色调）
- 文字显示「AI记忆」

**交互行为**：
- 点击后直接打开当前项目的 `memory.md` 文件编辑页
- 可考虑增加「最近修改时间」或「字数统计」的轻量信息

### 2. 文件树结构重构

**底部保持当前文件树结构**，分为两个默认文件夹：

#### 2.1 内置知识库

**用途**：存放系统内置的文档，不允许用户随意删除

**内容**：
- `config-system.md`（配置系统参考文档）
- 未来可扩展的其他内置文档

**限制**：
- 文件夹本身不可删除
- 内部文件不可重命名、不可移动

#### 2.2 项目知识库

**用途**：存放用户自己添加的文档

**结构**：
- 支持用户创建子文件夹（虚拟结构，非真实文件系统）
- 支持拖拽排序

**添加文件交互**：
- 文件夹最右侧显示「+」按钮
- 点击「+」弹出气泡菜单，包含：
  - **新建文件**：创建新的 markdown 文件
  - **上传文件**：支持上传 `.md` 格式文件

---

### 3. 代码视图 vs 文档视图的双视角设计

这是核心的架构设计决策：

| 视图 | 定位 | 文件夹性质 | 面向对象 |
|------|------|-----------|---------|
| **文档视图** | 用户友好，方便人类阅读和理解项目知识 | 虚拟结构，可随意组织 | 人类用户 |
| **代码视图** | 真实的目录结构，对 Agent 友好 | 真实文件系统结构 | Agent |

#### 3.1 文档视图特点

- **虚拟文件夹**：文件夹是逻辑上的组织单位，不是真实目录
- **灵活组织**：用户可以自由创建/删除/重命名文件夹，不影响实际文件
- **用户体验优先**：忽略 Agent 的目录偏好（如 `/knowledge/` 前缀）

#### 3.2 代码视图特点

- **真实目录结构**：完全对应 `data/workspaces/{sessionId}/knowledge/` 下的实际文件
- **Agent 友好**：使用扁平化目录结构，如：
  ```
  knowledge/
    memory.md
    config-system.md
    user-doc-1.md
    user-doc-2.md
  ```
- **不考虑人类阅读体验**：不使用嵌套过深的目录，避免特殊命名

## 数据结构设计

### 虚拟文件夹结构（文档视图）

```typescript
interface VirtualFolder {
  id: string;
  name: string;
  type: 'builtin' | 'user';      // 内置 vs 用户创建
  isProtected: boolean;         // 是否受保护（不可删除）
  children: VirtualFile[];      // 直接子文件
  order: number;                 // 排序顺序
}

interface VirtualFile {
  id: string;
  name: string;
  path: string;                  // 真实文件路径
  type: 'builtin' | 'user';      // 内置 vs 用户创建
  isProtected: boolean;         // 是否受保护（不可重命名/移动）
}

interface FileTreeState {
  folders: VirtualFolder[];      // 有序的文件夹列表
  // 约定：第一个是"内置知识库"，第二个是"项目知识库"
}
```

### 真实文件存储结构

```
data/workspaces/{sessionId}/
  knowledge/
    memory.md                    # AI 记忆（自动生成）
    config-system.md             # 内置配置参考
    user-file-1.md               # 用户上传的文件
    user-file-2.md               # 用户创建的文件
```

## 交互流程

### 添加文件流程

```
用户点击「+」按钮
    ↓
弹出气泡菜单
    ├── 新建文件 → 弹出输入框 → 输入文件名 → 创建空白 .md 文件
    └── 上传文件 → 打开文件选择器（仅限 .md）→ 上传 → 保存到 knowledge/
```

### AI记忆访问流程

```
用户点击顶部「AI记忆」横幅
    ↓
打开 memory.md 编辑页面
```

## 迁移说明

### 已有数据的迁移

1. **memory.md**：如果不存在，自动创建
2. **config-system.md**：从现有位置移动到 `knowledge/` 目录
3. **用户文档**：保持现有路径不变

### 用户自定义文件夹的处理

- 如果用户已有自定义文件夹结构，需要：
  1. 将文件移动到 `knowledge/` 根目录
  2. 清空虚拟文件夹数据
  3. 初始化为新的默认结构

## 影响范围

| 模块 | 影响 |
|------|------|
| 文件树组件 | 需要重构，支持虚拟文件夹 + 双视图切换 |
| 文件存储层 | 统一使用 `knowledge/` 目录 |
| AI 记忆模块 | 入口从文件树改为顶部横幅 |
| 编辑器 | 基本不受影响 |

---

## 实现优先级

1. **P0**：顶部「AI记忆」横幅模块
2. **P0**：文件树重构（内置知识库 + 项目知识库）
3. **P1**：添加文件气泡菜单（新建 + 上传）
4. **P2**：代码视图 vs 文档视图切换
