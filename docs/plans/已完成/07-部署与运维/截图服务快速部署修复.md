# 截图服务快速部署修复

## 背景

2026-06-30 尝试通过 `scripts/deploy-fast.sh shot` 单独部署 `screenshot-service`。SSH 与代码同步通过，但同步阶段传输了历史 `packages/web`、`.opencode` 和本地数据快照，随后 Docker 构建卡在 `apt-get update && apt-get install chromium`。已中断残留构建进程，线上旧截图服务仍保持 healthy。

## 目标

- 缩短截图服务单独部署的同步时间。
- 避免业务代码变更导致 Chromium 安装层反复重建。
- 修复后重新部署 `screenshot-service` 并验证健康检查。

## 范围

- `scripts/deploy.sh` 的 rsync 排除规则。
- `docker/screenshot-service/Dockerfile` 的构建层顺序。
- Docker 部署长期技术文档。
- 不调整截图服务业务代码与运行时接口。

## 方案

1. 扩充部署同步排除规则，排除根 `.opencode/`、历史 `packages/web/`、历史 `packages/snapshot-service/`、包内 `.next/`、包内 `data/` 和包内 `.opencode/`。
2. 将 `screenshot-service` 镜像中的 Chromium 与运行时依赖安装步骤放在业务产物复制之前，提升 Docker 层复用概率。
3. 重新运行截图服务单独部署，观察同步规模、Docker 构建是否命中缓存，并检查容器健康状态。
