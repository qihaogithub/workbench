# 实时保存退出同步提示异常排查

> 创建日期：2026-06-30
> 状态：进行中

## 背景

用户在项目编辑页退出时看到“修改仍在同步”弹窗。按 [实时协作保存需求](../../项目文档/创作端/03-项目管理/实时协作保存_需求文档.md)，当内容已自动保存且同步状态正常时，应直接离开；只有连接中、同步中、离线待同步或同步失败时才提示风险。

## 目标

- 找到退出时错误提示“修改仍在同步”的原因。
- 修复实时保存状态、退出前 flush 或画布保存的异常链路。
- 更新相关项目文档，使需求和实现说明与代码行为一致。

## 范围

- 创作端项目编辑页退出逻辑。
- 协同文档自动保存状态聚合。
- 退出前 Workspace flush 与画布强制保存。
- 相关项目管理文档。

## 方案

1. 读取实时协作保存需求和技术文档，确认退出提醒边界。
2. 用 CodeGraph 定位保存状态聚合、退出拦截和 flush 调用链。
3. 修复导致正常自动保存后仍被判定为同步中的实现问题。
4. 补充或调整测试，运行匹配范围的验证命令。
5. 更新项目文档与本任务记录。

## 进度记录

- 2026-06-30：根据截图确认问题表现为退出时仍进入“修改仍在同步”风险弹窗；需求文档要求同步正常时直接离开。
- 2026-06-30：定位到退出前 flush 使用浏览器直连 agent-service 的 `flush-all`，该路径会受浏览器可达地址、跨域和部署环境影响；关键动作已有服务端 flush coordinator，但编辑页退出缺少同源轻量入口。
- 2026-06-30：新增 Session 级 `workspace-flush` 同源 API，并将编辑页退出、命名版本、发布和自动检查点前端预 flush 统一切到该 API。
- 2026-06-30：目标 Jest 测试和 author-site 类型检查通过。
- 2026-06-30：用户复测仍出现弹窗，截图显示顶部为“已自动保存”，判断弹窗来自退出前补 flush 或画布强制保存异常，而不是协同状态风险。先将退出前 flush 改为 best-effort，但随后用户确认“已自动保存”后重新进入仍丢失编辑，说明该方向不足。
- 2026-06-30：用户确认退出后重新进入丢失编辑。定位到编辑页首次加载调用 `/api/sessions` 时传 `forceNew: true`，会归档旧 Session 并从项目 canonical workspace 创建新 Workspace，绕过刚才实时保存的共享 Workspace。已改为首次进入复用活跃 Session；无活跃 Session 且非显式强制新建时，服务端从最近 Workspace 续开。
- 2026-06-30：进一步确认顶部“已自动保存”不能只等同 WebSocket Yjs 同步成功。新增前端 Workspace 待落盘状态：本地编辑后显示“同步中”，同源 flush 成功后才显示“已自动保存”，flush 失败显示“同步失败”并阻止普通退出。
