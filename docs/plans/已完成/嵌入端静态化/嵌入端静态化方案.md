## 一、背景与目标

### 1.1 问题背景

当前「嵌入端」（embed）的三个路由全部依赖 author-site 的 Node.js 运行时：

| 路由 | 作用 | 当前实现 |
|---|---|---|
| `/embed/{demoId}` | 嵌入引导页（展示嵌入代码 + 预览） | 服务端 SSR，`fs.readFileSync` 读 schema |
| `/embed/{demoId}/iframe` | 旧 iframe API（单页） | Route Handler，每次请求 `compileCode()` 实时编译 |
| `/api/embed/{projectId}/iframe` | 新 iframe API（多页） | Route Handler，每次请求 `compileCode()` 实时编译 + merge schema |

**核心问题：每一步都已经在发布时做过，但嵌入时又重复做一遍。**

在使用端完全解耦方案（v2.2，四个阶段全部实施完成）之后：

- viewer-site 已实现纯静态化（`output: "export"`），消费 `data/published/{projectId}/` 下的 `project.json` + `compiled.js` + `schema.json`
- 发布时已经完成了 TSX → JS 编译并生成 `compiled.js`
- Cloudflare 同步脚本已将 `data/published/` 完整部署到 CDN
- 局域网 nginx 已通过共享卷实时读取 `data/published/`

**但嵌入端仍然留在 author-site 上做动态编译。** 第三方站点每次加载嵌入 iframe，都触发一次 author-site 的 `fs.readFileSync` + `compileCode()` + `generateIframeHtml()`。这些工作在发布时已经全部做过，作者发布后嵌入内容不会再变，每次请求重复编译纯粹是资源浪费。

此外，这意味着**嵌入端和 viewer-site 架构不统一**：

- viewer-site = 纯静态（CDN 部署，零依赖 author-site）
- embed 端 = 动态 + 耦合 author-site（必须 author-site 在线）

### 1.2 设计目标

1. **嵌入端也消费「已发布」的静态文件**，与 viewer-site 使用同一份 `data/published/` 数据源
2. **预生成 iframe 完整 HTML**：发布时一次性生成每个 Demo 的 `iframe.html`，第三方嵌入直接引用静态 URL
3. **第三方嵌入零依赖 author-site**：author-site 宕机不影响已发布 Demo 的嵌入展示
4. **与 viewer-site 同生命周期部署**：iframe.html 与 viewer-site 静态产物一起部署到局域网 + Cloudflare，共享缓存策略
5. **兼容现有嵌入场景**：现有的 postMessage `UPDATE_CONFIG` 协议继续工作（iframe.html 内嵌 configData + postMessage 监听器）

### 1.3 成功标准

- 第三方嵌入 `<iframe src="https://viewer.example.com/data/{projectId}/demos/{demoId}/iframe.html">` 完全独立运行，不依赖 author-site
- iframe 内容加载时间 < 500ms（CDN 缓存，静态 HTML + configData 内嵌）
- iframe.html 文件大小 < 50KB（压缩后 < 15KB）
- 发布后 iframe.html 自动生成，零额外配置
- 已有 `postMessage` 控制协议完全兼容（`UPDATE_CONFIG` 可动态修改内嵌组件的 props）

### 2.1 嵌入端当前实现详解

三个路由的调用链完全相同（以 `/embed/{demoId}/iframe` 为例）：

```
请求 → fs 读 index.tsx 源码 → compileCode() 编译 → fs 读 config.schema.json
→ generateIframeHtml({ compiledCode, configData }) → 返回完整 HTML
```

这一步的核心耗时在两个地方：
1. **`compileCode()`**：Sucrase 编译 TSX → JS + 重写 npm imports 为 esm.sh CDN URL。单页约 50ms
2. **`fs.readFileSync`**：读源码 + 读 schema。几乎无性能损耗

但问题是：发布时已经做了 `compileCode()` 并把结果存入 `compiled.js`。发布后嵌入内容不会变，每次请求重新编译是 100% 的无效工作。

### 2.2 发布物已包含的数据

当前 `publish-manager.ts:106-121` 在发布时为每个 Demo 生成了：

```
data/published/{projectId}/
├── project.json              ← 项目元数据 + 页面列表
├── config-schema.json        ← 项目级 Schema
├── thumbnail.png
└── demos/{demoId}/
    ├── compiled.js           ← 已编译的 JS（关键！）
    └── schema.json           ← 页面级 Schema
```

**这正是生成 iframe.html 所需的全部素材：**

- `compiled.js` 作为 `compiledCode` 参数传给 `generateIframeHtml()`
- `schema.json` 提取 `default` 值作为 `configData` 参数
- `config-schema.json` 提取项目级 `default` 值，合并到 `configData`

**`generateIframeHtml` 接口完全匹配：**

```typescript
// packages/shared/src/demo/iframe-template.ts
interface IframeTemplateOptions {
  compiledCode?: string;          // ← compiled.js 的内容
  configData?: Record<string, unknown>; // ← schema.json 的 default 值
  cssImports?: string[];          // 当前项目无 CSS 导入
  cdnBaseUrl?: string;            // 可配置
  supportUrlMode?: boolean;       // 生成双向 postMessage 支持
}
```

### 2.3 核心可复用点

1. **`compileCode()` 无需再调用** —— 发布物中 `compiled.js` 已是编译结果
2. **`generateIframeHtml()` 是纯函数**（`shared/demo/iframe-template.ts`）—— 无任何副作用，可在 publish-manager 中直接调用
3. **Cloudflare 同步脚本已能传播** —— `sync-to-cloudflare.sh` 已将 `data/published/*` 完整复制到部署目录
4. **nginx 配置已就绪** —— 局域网部署的 nginx 已 `/data/` alias 到共享卷

## 三、核心方案

### 3.1 整体架构对比

#### 当前（嵌入时动态编译）

```
第三方站点
└── <iframe src="https://author-site.com/embed/{demoId}/iframe">
    │
    ▼ (每次加载都触发)
author-site 服务（必须在线）
├── fs.readFileSync(codePath)     ← 读 TSX 源码
├── compileCode()                  ← 实时编译（50ms）
├── fs.readFileSync(schemaPath)   ← 读 schema
├── generateIframeHtml()           ← 生成 HTML
└── 返回完整 HTML
```

#### 改造后（发布时预生成 HTML）

```
发布时（一次生成）
~~~~~~~~~
author-site publish-manager
├── compileCode() → compiled.js  ← 已有，不变
├── 读 compiled.js + schema.json
├── generateIframeHtml()           ← 新增：一次性生成
└── 写 demos/{id}/iframe.html     ← 新增：存入发布物

运行时（零开销）
~~~~~~~~
第三方站点
└── <iframe src="https://viewer.example.com/data/{projectId}/demos/{demoId}/iframe.html">
    │
    ▼ (直接命中 CDN 缓存)
Cloudflare Pages / nginx 静态服务器
└── 返回预生成的 iframe.html（纯静态文件，无任何后端逻辑）
```

### 3.2 数据流

```
作者点击"发布"
    │
    ├── 1. publishProject() 开始（现有逻辑不变）
    │   ├── compileCode() → compiled.js
    │   ├── 写 schema.json
    │   └── 写 config-schema.json
    │
    ├── 2. 新增：预生成 iframe.html
    │   │   for each demo:
    │   │     读 compiled.js
    │   │     读 schema.json（提取 default 值作为 configData）
    │   │     读 config-schema.json（提取项目级 default 值，合并到 configData）
    │   │     调 generateIframeHtml({
    │   │       compiledCode: compiledJsContent,
    │   │       configData: mergedDefaults,
    │   │       cssImports: [],
    │   │       cdnBaseUrl: getCdnBaseUrl(),
    │   │     })
    │   │     写 demos/{id}/iframe.html
    │   └── 记录 iframeHtmlPath 到 project.json
    │
    ├── 3. Cloudflare 同步（现有逻辑不变）
    │   └── sync-to-cloudflare.sh：data/published/* → deploy/data/
    │       iframe.html 随 compiled.js/schema.json 一起上传
    │
    └── 4. 局域网部署（现有逻辑不变）
        └── 共享卷：iframe.html 直接可读
```

### 3.3 关键设计决策

#### D1: 每个 Demo 独立 iframe.html

每个 Demo 都生成一份独立的 `iframe.html`，原因：

1. **各 Demo 的 compiled.js 不同**（不同页面不同代码）
2. **各 Demo 的 configData 不同**（不同页面不同 Schema）
3. **URL 语义清晰**：`demos/{demoId}/iframe.html` 直观对应
4. **缓存粒度高**：单页面更新只影响自
己的 `iframe.html`，其他页面不受影响

#### D2: configData 内嵌到 HTML 中

iframe.html 内嵌 configData（compile-time 生成的 merge 后的配置默认值），原因：

1. **iframe 加载即可渲染**，无需额外的数据请求
2. **仍支持运行时覆盖**：`postMessage({type:'UPDATE_CONFIG', configData})` 可以在加载后动态修改（`iframe-template.ts` 已支持）

#### D3: 不再需要 `config-schema.json` 里的全局配置

对于多页面项目，embed 端当前使用 `mergeConfigToProps(projectConfigSchema, pageSchema)` 合并项目配置 + 页面配置。改造后：

- 发布时同样执行 merge，把合并后的值直接内嵌到 iframe.html
- 嵌入第三方后，`configData` 已经是 merge 后的结果，postMessage 也可覆盖。
- 引入 `packages/shared/src/demo/iframe-template.ts` 时不需要 `runtime-props` 依赖。

#### D4: 嵌入代码 URL 指向 viewer-site 域名

已发布 Demo 的嵌入代码指向部署目标：

```
<!-- 局域网 -->
<iframe src="http://192.168.x.x:3300/data/{projectId}/demos/{demoId}/iframe.html" />

<!-- Cloudflare -->
<iframe src="https://viewer.example.com/data/{projectId}/demos/{demoId}/iframe.html" />
```

嵌入代码在 publish-manager 中自动生成并存入 `project.json` 的 `demoPages[].embedCode` 字段，供 author-site 引导页读取。

## 四、技术方案

### 4.1 publish-manager 改造：预生成 iframe.html

**改造文件**：`packages/author-site/src/lib/publish-manager.ts`

在现有 `publishProject()` 的循环中（`publish-manager.ts:98-132`），对每个 Demo 追加一步：

1. 读取已编译的 `compiled.js`
2. 读取页面级 `schema.json`，提取 `default` 值
3. 读取项目级 `config-schema.json`，提取 `default` 值，合并
4. 调用 `generateIframeHtml()` 生成完整 HTML
5. 写入 `demos/{demoId}/iframe.html`
6. 在 `PublishedDemoPage` 结构中记录 `iframeHtmlPath`

**新增 `PublishedDemoPage` 字段**：

```typescript
interface PublishedDemoPage {
  // ... 现有字段不变 ...
  iframeHtmlPath?: string;  // 新增：相对路径 "demos/{id}/iframe.html"
  embedCode?: string;        // 新增：嵌入代码片段
}
```

**configData 生成逻辑**：

```
读页面 schema.json → 提取 {"字段名": default值} (同 getDefaultValues)
读项目 config-schema.json → 提取 {"字段名": default值} (同 getDefaultValues)
合并：{ ...项目默认值, ...页面默认值 }（页面覆盖项目）
→ 传入 generateIframeHtml({ configData: merged, compiledCode, ... })
```

merge 时页面级 schema 不应声明与项目级同名字段（这是 Schema 设计规范决定的），所以「页面覆盖项目」理论上不会触发冲突，但保留此顺序以保证正确性。

### 4.2 嵌入代码生成

**改造文件**：`packages/author-site/src/lib/publish-manager.ts`

发布时为每个 Demo 生成嵌入代码片段，写入 `PublishedDemoPage.embedCode`：

```
<!-- 局域网部署时嵌入代码 -->
<iframe
  src="http://192.168.x.x:3300/data/{projectId}/demos/{demoId}/iframe.html"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>
```

**跨域注意事项**：`sandbox` 属性建议使用 `allow-scripts` 但不建议 `allow-same-origin`（iframe-template.html 不依赖同源策略，postMessage 使用 `"*"` 目标源）。

**host 配置**：嵌入代码中的域名来源：
- 局域网：从 `process.env.VIEWER_LAN_URL` 读取（如 `http://192.168.1.100:3300`）
- Cloudflare：从 `process.env.VIEWER_CLOUDFLARE_URL` 读取（如 `https://viewer.example.com`）
- 如果环境变量未配，嵌入代码中仅写相对路径 `/data/{projectId}/demos/{demoId}/iframe.html`，由第三方自己补充 host。

### 4.3 author-site 嵌入引导页改造

**改造文件**：`packages/author-site/src/app/embed/[demoId]/page.tsx`

当前 embed 引导页的职责：
1. 让作者复制嵌入代码
2. 附带预览 + 发布前配置面板

改造后：
- **嵌入代码从 `data/published/{projectId}/project.json` 读取**（`demoPages[].embedCode`），不再手写 HTML 字符串。
- **预览部分改用 `published` 数据**：已发布项目的 embed 引导页从 `data/published/` 读取数据渲染预览，确保预览效果与第三方实际嵌入一致。
- **未发布项目仍从 workspace 直接读取**：出于实时预览需求。作者发布前仍可预览（只是这时走动态模式）。
- **配置面板保留**：因为这个是引导页内部的功能，不涉及第三方嵌入。

### 4.4 author-site embed 路由处置

两个选择：

| 选项 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A（推荐） | **保留 `/embed/{demoId}` 引导页**，作为"发布前预览 + 发布后获取嵌入代码"的统一入口。删除 `/embed/{demoId}/iframe`（旧 iframe API 已被新 API 覆盖），保留 `/api/embed/{projectId}/iframe` 作为"未发布实时预览"。 | 过渡平滑，已发布指向静态，未发布仍可用 | 保留了两套路径 |
| B | 全部删除，只保留静态化方案 | 干净 | 破坏未发布预览能力 |

**推荐选项 A**：保留引导页入口，发布前后用不同数据源。未发布时走动态（读 workspace），发布后走静态（读 published）。

### 4.5 Cloudflare 同步脚本适配

**改造文件**：`scripts/sync-to-cloudflare.sh`

`sync-to-cloudflare.sh:24` 已经是 `cp -r "$PUBLISHED_DIR"/* "$DEPLOY_DIR/data/"`，会将整个 `data/published/` 目录完整复制，所以 **新增的 `iframe.html` 会自然被带过去**。

只需新增一条 `_headers` 规则：

```
/data/*/demos/*.html
  Cache-Control: public, immutable
  Access-Control-Allow-Origin: *
```

**解释**：iframe.html 内容在发布后不再变化（compiled.js 也在同级 immutable），故使用 `immutable` 缓存策略。如果第三方需要跨域加载 iframe.html，可以加 `Access-Control-Allow-Origin: *`。

### 4.6 局域网 nginx 配置适配

**改造文件**：`docker/viewer-site/nginx.conf`

nginx 已配置 `/data/` alias 到共享卷的 `published/` 目录，新增的 iframe.html 自动可通过 `/data/{projectId}/demos/{demoId}/iframe.html` 访问。

无需额外修改 nginx 配置，`/data/` 路径的现有规则已经覆盖。

### 4.7 兼容性

| 场景 | 是否兼容 | 说明 |
|---|---|---|
| 旧 `<iframe src="/embed/{demoId}/iframe">` | 过渡保留 | 旧路由可不删除，但标注「已废弃，指向 `/api/embed/{projectId}/iframe`」 |
| 旧 `<iframe src="/api/embed/{projectId}/iframe?page=...">` | 过渡保留 | 同上 |
| 新 `<iframe src="/data/{id}/demos/{demoId}/iframe.html">` | ✅ 新增 | 发布后即有 |
| postMessage `UPDATE_CONFIG` | ✅ 完全兼容 | iframe-template.ts 已内置支持 |
| postMessage `UPDATE_CODE` | ❌ 不支持 | 静态化后代码已内嵌，无需运行时换代码 |

**`UPDATE_CONFIG` 仍能用**，因为 `iframe-template.ts:225-226` 的处理逻辑在生成 HTML 时已嵌入。第三方站点可以通过 postMessage 动态修改 props（如换主题、改颜色），这些配置变化只影响组件渲染，不需要重编译。

**`UPDATE_CODE` 不支持**，因为代码已内嵌在 iframe.html 中（不再有 `/api/compile` 端点可用）。这是合理的：嵌入场景
面向「已定稿」的内容，不需要运行时换代码。如需更新，作者重新发布即可（Compiled.js 不变但 configData 可能变了）。

## 六、风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| **iframe.html 体积过大** | CDN 带宽浪费 | `compiled.js` 通常在 5 – 30KB 范围，加上 iframe 模板约 10KB，总体 < 50KB；压缩后 < 15KB |
| **esm.sh CDN 不可用** | iframe 内 React 依赖加载失败 | 发布时已将 esm.sh URL 锁定到 `compiled.js`；极端情况可切换 `CDN_BASE_URL` 后重新编译 |
| **第三方站点 CSP 限制** | iframe 被第三方安全策略阻止 | 嵌入代码文档中附温馨提示：建议将 viewer-site 域名加入 CSP `frame-src` 白名单 |
| **旧嵌入代码失效** | 已发布的旧嵌入代码（`/embed/{demoId}/iframe`）停止工作 | 保留旧路由过渡，标注废弃；下次发布时更新嵌入代码 |
| **并发发布同时生成 iframe.html** | 文件写入竞争 | `publishProject` 本身就是全量覆盖该项目的发布目录，不存在竞争 |
| **configData merge 时出现冲突** | 项目 Config 和页面 Config 同名字段覆盖 | Schema 规范已禁止页面 Schema 存在项目字段，merge 顺序安全 |

## 九、总结

本方案是「使用端完全解耦静态化方案」的延伸补完，专治 embed 端仍然依赖 author-site 服务运行的问题。

**核心思路**：把「生成 iframe HTML」这一步从**请求时**挪到**发布时**。发布物中已有 `compiled.js` + `schema.json`，这些都是生成完整 HTML 的素材。只是在 publish-manager 中多加一个 `generateIframeHtml()` 调用，产出 `iframe.html` 文件，就可以和 viewer-site 一起部署到 CDN + 局域网。

**改动量极小**：
- publish-manager.ts：加约 30 行
- embed 引导页：从 published 数据读嵌入代码
- Cloudflare 同步脚本：加一行缓存规则
- 其余全部复用已有基础设施

**收益**：嵌入端与使用端架构完全统一，第三方的 `<iframe>` 真正实现"无服务端依赖"。
