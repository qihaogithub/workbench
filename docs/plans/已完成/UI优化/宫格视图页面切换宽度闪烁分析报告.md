## 一、问题背景

### 1.1 问题描述

宫格视图下，点击不同页面卡片时，宫格中所有页面宽度会**快速变为等宽**，然后**立即恢复正常比例**。这个过程在视觉上表现为短暂的"闪烁"或"跳动"，影响预览体验。

### 1.2 预期行为

点击不同页面卡片后，宫格布局应平滑过渡到新状态，不应出现等宽闪烁。各卡片应保持其基于 `previewSize` 的宽度比例。

### 1.3 实际行为

```
点击前（页面 A 活跃）        点击瞬间（等宽闪烁）     点击后（恢复正常）
┌──────────┬──────┐         ┌────────┬───────┐       ┌──────┬──────────┐
│  页 A    │ 页 B │         │  页 A  │  页 B │       │ 页 B │   页 A   │
│ 1.333fr  │0.462fr│   →    │   1fr  │  1fr  │   →   │1.333fr│ 0.462fr  │
└──────────┴──────┘         └────────┴───────┘       └──────┴──────────┘
  比例不同                   等宽闪烁（约 200-500ms）    比例恢复
```

## 二、根因分析

### 2.1 核心根因

**`activeDemoId` 与 `previewSize` 状态更新不同步** — 这是一个典型的 **React 状态一致性漏洞**：同步的状态更新与异步的状态更新之间存在时间窗口，导致 `PreviewGrid` 在中间态收到了不一致的 props。

### 2.3 闪烁发生的时间线

假设页面 A 有 `previewSize = {width: 1024, height: 768}`（平板），页面 B 有 `previewSize = {width: 375, height: 812}`（手机）。

| 时间点 | activeDemoId | previewSize (prop) | PreviewGrid 列宽计算 | 视觉效果 |
|:-------|:-------------|:-------------------|:---------------------|:---------|
| T=0 (点击前) | A | 页面 A 的 size | 页 A(active): A的size, 页 B(inactive): B的size → **不同宽** | 正常比例 |
| T=0+ (点击瞬间) | B ✅ (已更新) | 页面 A 的 size ❌ (未更新) | 页 B(active): **A的size**, 页 A(inactive): **A的size** (B的page.previewSize=A的size) → **等宽** ❌ | **闪烁** |
| T=200ms (API 返回) | B | 页面 B 的 size ✅ (已更新) | 页 B(active): B的size, 页 A(inactive): B的size → **不同宽** | 恢复正常 |

### 2.4 等宽现象的本质

两种巧合叠加导致"等宽"：

1. **活跃页面规则**：活跃页面使用 prop `previewSize`，非活跃页面使用自己的 `page.previewSize ?? previewSize`
2. **previewSize 滞后**：新活跃页面（页 B）在 T=0+ 时刻使用的仍是旧 `previewSize`（页 A 的 size），而页 A 作为非活跃页面回退到自己的 `page.previewSize`（恰好是页 A 的 size）
3. **结果**：两个页面都使用了页 A 的 size → `gridTemplateColumns: 0.462fr 0.462fr` → 等宽

### 2.5 证据级别

| 证据 | 内容 | 来源 | 级别 |
|:-----|:-----|:-----|:-----|
| 同步 vs 异步更新 | `setActiveDemoId(pageId)` 同步更新，`setPreviewSize(size)` 在 fetch.then 中异步更新 | [page.tsx#L1091-L1104](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/app/demo/%5Bid%5D/edit/page.tsx#L1091-L1104) | **A** |
| 列宽随 activePageId 切换 | `p.id === activePageId ? previewSize : (p.previewSize ?? previewSize)` | [PreviewGrid.tsx#L393-L395](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/components/demo/PreviewGrid.tsx#L393-L395) | **A** |
| 等宽公式推导 | 两者都使用相同 size → 相同 `ratio`fr → 等宽 | PreviewGrid.tsx#L396 | **A** |

### 2.6 触发条件

此问题在**不同页面的 previewSize 不同**时必然触发。若所有页面的 previewSize 相同（或均为 undefined），则等宽状态与最终状态一致，闪烁不可见。

## 三、解决方案

### 方案一：乐观更新 previewSize（推荐）⭐

**做法**：在 `onCardClick` 中，立即从已有的 `demoPages` 数据中提取新页面的 `previewSize` 并同步更新，无需等待异步 API 返回。

**复杂度**：低。改动范围仅局限在 page.tsx 的一处回调。

**修改位置**：[page.tsx#L1089-L1112](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/web/src/app/demo/%5Bid%5D/edit/page.tsx#L1089-L1112)

```typescript
onCardClick={(pageId) => {
  if (pageId === activeDemoId) return;

  // 立即从 demoPages 中获取新页面的 previewSize，保持与 activeDemoId 同步
  const clickedPage = demoPages.find(p => p.id === pageId);
  if (clickedPage?.previewSize) {
    setPreviewSize(clickedPage.previewSize);
  }

  setActiveDemoId(pageId);

  if (sessionId) {
    fetch(`/api/sessions/${sessionId}/files/${pageId}`)
      .then((res) => res.json())
      .then((data) => {
        if (data.success) {
          setCode(data.data.code);
          setSchema(data.data.schema);
          setEditorContent(buildFigmaText(...));
          setConfigData(defaults);
          // API 返回的 size 可能更精确，可以覆盖乐观值
          setPreviewSize(size);
        }
      });
  }
}}
```

**为何有效**：`demoPages` 数组在页面初始化时已加载了每个页面的 `previewSize`。切换页面时，`activeDemoId` 和 `previewSize` 同时更新，消除了中间态时间窗口。

### 方案二：PreviewGrid 内部消化过渡

**做法**：在 PreviewGrid 中添加 CSS transition 使列宽变化动画化。

```css
transition: grid-template-columns 0.15s ease;
```

**缺点**：只是掩盖而非修复问题，过渡动画本身也是一种视觉变化，可能不符合设计预期。

### 方案三：使用 useMemo 在 PreviewGrid 中推导 previewSize

**做法**：让 PreviewGrid 从 `demoPages` 中自行推导活跃页面的 previewSize，不再依赖父组件的 prop。

**缺点**：破坏了 prop 接口设计，`demoPages` 中的 previewSize 可能不如父组件 `previewSize` 精确（父组件的 size 可能包含实时变更）。

### 方案对比

| 维度 | 方案一（乐观更新） | 方案二（CSS 过渡） | 方案三（内部推导） |
|:-----|:------------------|:------------------|:------------------|
| 解决根因 | ✅ 消除状态时间窗口 | ❌ 只是掩盖 | ✅ 消除状态时间窗口 |
| 复杂度 | 低（3 行代码） | 极低（1 行 CSS） | 中（重构 prop 逻辑） |
| 影响范围 | 仅 page.tsx | 仅 PreviewGrid.tsx | PreviewGrid + types |
| 副作用 | 无 | 所有列宽变化都动画化 | 父组件 previewSize 可能失效 |
| 推荐度 | ⭐⭐⭐ | ⭐⭐ | ⭐ |

## 五、结论

**根本原因**：`activeDemoId` 同步更新后，`previewSize` 需等待异步 API 返回才能更新，两者之间存在 200-500ms 的中间态。在此中间态中，`PreviewGrid` 使用不一致的 props 计算列宽，导致所有卡片使用相同的 `previewSize`，表现为"等宽闪烁"。

**推荐方案**：采用**乐观更新策略**（方案一），在点击页面卡片时立刻从已预加载的 `demoPages` 数据中同步更新 `previewSize`，彻底消除状态不一致的时间窗口。改动量仅 3-4 行代码，无副作用。
