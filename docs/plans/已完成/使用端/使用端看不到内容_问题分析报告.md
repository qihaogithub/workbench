> 分析日期：2026-05-05
> 分析范围：`@opencode-workbench/viewer-site` 使用端无法展示项目列表

## 1. 问题背景

### 问题描述

用户在使用端（`@opencode-workbench/viewer-site`，端口 3300）无法看到任何项目内容。打开使用端首页后，页面显示为空或加载失败状态。

### 预期行为

- 使用端首页（`/`）应展示从 agent-service 获取的项目列表
- 用户点击项目卡片后进入预览页
- 无需登录即可浏览所有项目

### 实际行为

- 使用端看不到任何内容

### 涉及系统组件

| 组件 | 端口 | 职责 |
|:-----|:-----|:-----|
| viewer-site（使用端） | 3300 | 展示项目列表和预览 |
| agent-service | 3201 | 提供项目列表/详情 API |
| web（创作端） | 3200 | 创建项目、嵌入 iframe、编译 |

## 2. 根因分析

### 调查过程

#### 证据 1：项目数据目录为空

**结论**：`packages/agent-service/data/projects/` 目录存在但完全为空，没有任何项目数据。

#### 证据 2：三个服务均未运行

```
$ lsof -i :3201  → 无输出（agent-service 未运行）
$ lsof -i :3200  → 无输出（web 创作端 未运行）
$ lsof -i :3300  → 无输出（viewer-site 未运行）
```

**结论**：所有三个服务当前均未启动。

#### 证据 3：数据存储路径分析

agent-service 的项目数据存储路径由以下代码决定：

```typescript
// packages/agent-service/src/workspace/project-workspace-manager.ts:19
const BASE_DIR = path.resolve(
  process.env.PROJECTS_BASE_DIR || path.join(process.cwd(), 'data')
);
const PROJECTS_DIR = path.join(BASE_DIR, 'projects');
```

当未设置 `PROJECTS_BASE_DIR` 环境变量时，数据存储在 `process.cwd()/data/projects/`，即 agent-service 的工作目录下的 `data/projects/`。

#### 证据 4：使用端 API 调用链路

使用端首页（`page.tsx`）的调用链路：

```
page.tsx (SWR hook)
  → fetcher() → getProjects()
    → fetchApi(AGENT_SERVICE_URL, "/api/projects")
      → fetch("http://localhost:3201/api/projects")
        → agent-service routes/projects.ts
          → projectWorkspaceManager.getProjects()
            → fs.readdir(PROJECTS_DIR)  // 读取 data/projects/ 目录
```

当 `PROJECTS_DIR` 为空时，`getProjects()` 返回 `{ projects: [], total: 0 }`，使用端展示"暂无项目"。

#### 证据 5：创作端项目创建流程

创作端创建项目时的调用链路：

```
web/src/app/projects/new/page.tsx (用户提交表单)
  → projectApiClient.createProject({ name, description, workspacePath })
    → fetch("http://localhost:3201/api/projects", { method: "POST" })
      → agent-service routes/projects.ts
        → projectWorkspaceManager.createProject(request)
          → 在 PROJECTS_DIR 下创建项目目录和 project.json
```

创作端通过 `POST /api/projects` 向 agent-service 创建项目，项目数据写入 `data/projects/{projectId}/` 目录。

#### 证据 6：CORS 配置已包含使用端

**Agent Service（端口 3201）**：

```typescript
// packages/agent-service/src/server.ts:42-44
const allowedOrigins = process.env.CORS_ORIGINS
  ? process.env.CORS_ORIGINS.split(',')
  : ['http://localhost:3200', 'http://127.0.0.1:3200',
     'http://localhost:3300', 'http://127.0.0.1:3300'];
```

**Web 创作端（端口 3200）**：

```typescript
// packages/web/src/middleware.ts
const VIEWER_ORIGINS = [
  'http://localhost:3300',
  'http://127.0.0.1:3300',
];

// 在 middleware 中，对 /api/ 和 /embed/ 路径的请求，
// 如果 origin 在 VIEWER_ORIGINS 白名单中，自动注入 CORS 头
if (origin && isApiOrEmbedRoute && VIEWER_ORIGINS.includes(origin)) {
  response.headers.set('Access-Control-Allow-Origin', origin);
  response.headers.set('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  response.headers.set('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  response.headers.set('Access-Control-Allow-Credentials', 'true');

```

**结论**：两个服务的 CORS 配置均已默认允许来自 3300 端口（使用端）的请求，不存在 CORS 阻断问题。

### 根因定位

**根本原因：Web 创作端和 Agent-Service 使用了两个完全独立的数据目录，项目数据只存在于 Web 端目录中。**

#### 数据目录分裂（核心问题）

| 服务 | 数据目录 | 项目数量 | 配置来源 |
|:-----|:---------|:---------|:---------|
| **Web 创作端首页** (`/`) | `packages/web/data/projects/` | **2 个** | `fs-utils.ts` 的 `DATA_DIR`（默认 `process.cwd()/data`） |
| **Web 创作端项目页** (`/projects`) | 调用 agent-service API | **0 个** | `project-api.ts` 的 `AGENT_SERVICE_URL` |
| **Agent-Service API** | `packages/agent-service/data/projects/` | **0 个** | `project-workspace-manager.ts` 的 `PROJECTS_BASE_DIR`（默认 `process.cwd()/data`） |
| **使用端** | 调用 agent-service API | **0 个** | `api.ts` 的 `AGENT_SERVICE_URL` |

用户在创作端首页看到的 2 个项目（"表单控件合集"、"测试123"）存储在 `packages/web/data/projects/`，但使用端通过 agent-service API 读取的是 `packages/agent-service/data/projects/`，该目录为空。

#### 数据目录路径解析

```typescript
// Web 端 (fs-utils.ts:18) — process.cwd() = packages/web/
const DATA_DIR = process.env.DATA_DIR || path.join(process.cwd(), "data");
// → packages/web/data/

// Agent-Service (project-workspace-manager.ts:19) — process.cwd() = packages/agent-service/
const BASE_DIR = path.resolve(process.env.PROJECTS_BASE_DIR || path.join(process.cwd(), 'data'));
// → packages/agent-service/data/
```

两个服务的 `process.cwd()` 不同，导致默认数据目录不同，项目数据被隔离在两个独立的目录中。

#### Web 端内部的数据源分裂

Web 创作端自身也存在数据源不一致：
- **首页** (`/`) 通过 `fs-utils.ts` 的 `listProjects()` 直接读本地文件 → 看到 2 个项目
- **项目列表页** (`/projects`) 通过 `project-api.ts` 调 agent-service API → 看到 0 个项目

## 3. 解决方案

### 方案 A：统一数据目录（已实施 ✅）

通过设置 `PROJECTS_BASE_DIR` 环境变量，让 agent-service 使用 Web 端的数据目录：

**修改内容**：根目录 `package.json` 的启动脚本

```json
"dev": "concurrently -n web,agent -c blue,green \"pnpm --filter @opencode-workbench/web dev\" \"PROJECTS_BASE_DIR=../web/data pnpm --filter @opencode-workbench/agent-service dev\"",
"dev:agent": "PROJECTS_BASE_DIR=../web/data pnpm --filter @opencode-workbench/agent-service dev",
"dev:all": "concurrently -n web,agent,viewer -c blue,green,cyan \"pnpm --filter @opencode-workbench/web dev\" \"PROJECTS_BASE_DIR=../web/data pnpm --filter @opencode-workbench/agent-service dev\" \"pnpm --filter @opencode-workbench/viewer-site dev\"",
```

**原理**：`PROJECTS_BASE_DIR=../web/data` 是相对于 agent-service 工作目录（`packages/agent-service/`）的路径，`path.resolve()` 会将其解析为 `packages/web/data/`，与 Web 端的 `DATA_DIR` 指向同一目录。

**验证结果**：
- agent-service 启动日志：`baseDir: "/Users/.../packages/web/data"` ✅
- `GET /api/projects` 返回 2 个项目 ✅
- 使用端首页正常显示项目列表 ✅
- 使用端预览页正常工作 ✅

### 方案 B：检查环境变量配置（排查用）

如果启动服务后仍然看不到内容，需检查：

1. **viewer-site 的 `.env` 文件**：当前只有 `.env.example`，没有 `.env` 文件。默认值 `http://localhost:3201` 和 `http://localhost:3200` 应该可以工作，但如果 agent-service 运行在其他端口，需要创建 `.env` 文件：
   ```
   NEXT_PUBLIC_AGENT_SERVICE_URL=http://localhost:3201
   NEXT_PUBLIC_WEB_URL=http://localhost:3200
   ```

2. **agent-service 的 `PROJECTS_BASE_DIR`**：确保该环境变量设置为 `../web/data`（相对于 agent-service 工作目录）
