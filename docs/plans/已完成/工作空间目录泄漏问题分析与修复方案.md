# 工作空间目录（ws-*）泄漏问题分析与修复方案

> 版本：v2.0
> 创建日期：2026-05-04
> 更新日期：2026-05-06
> 状态：✅ 已实施

## 一、背景

用户发现 `packages/web/data/workspaces/` 目录下积累了 23 个 `ws-{timestamp}-{random}` 格式的目录，询问是否应该存在自动清理机制。经排查，**Web 端 session 清理机制存在缺口：清理了 session 元数据文件，但未清理对应的工作空间目录**，导致目录持续累积。

## 三、根因分析

### 3.1 两套独立的 workspace 系统

经过全面代码审查，发现系统中有**两套独立的 workspace 机制**，但它们在生命周期管理上互不知晓：

| 系统 | 路径 | 创建方 | 清理方 | 状态 |
|------|------|--------|--------|------|
| **Web workspace** | `data/workspaces/{userId}/{projectId}/ws-*` | Web 前端 `workspace-manager.ts:createWorkspace()` | ❌ **无** | 泄漏 23 个 |
| **Agent-service workspace** | `/tmp/opencode-workspaces/` | Agent-service `WorkspaceManager.create()` | Agent-service `DELETE /api/agent/:sessionId` | ✅ 正常 |

**关键发现**：Agent-service 在 session 销毁时（`routes/agent.ts:177-199`）会清理自己的临时 workspace，但 Web 前端的 workspace  (`data/workspaces/`) 完全不在它的管理范围内。两者指向的是**不同的文件系统路径** — Agent-service 管理 `/tmp` 下的临时目录，Web 前端管理 `data/workspaces/` 下的持久副本。

### 3.2 创建链路

```
用户打开 AI 对话
  → createEditSession()  [session-manager.ts:L180]
    → createWorkspace()  [workspace-manager.ts:L102]
      → 生成 ws-{timestamp}-{random} ID
      → 复制 project workspace → data/workspaces/{userId}/{projectId}/{workspaceId}/
      → 写入 .workspace.json 元数据
      → 注入 .opencode/ Agent 配置
    → 创建 session 目录 data/sessions/{userId}/{projectId}/{sessionId}/
    → session 元数据中记录 workspaceId 引用
```

### 3.3 保存链路（未完全释放）

```
用户保存 AI 对话
  → saveEditSession()  [session-manager.ts:L297]
    → 将 workspace 内容复制到 project workspace
    → 写入 meta.status = 'saved'
    → 调用 deleteSession() → 删除 session 目录
    → ❌ deleteSession() 未读取 workspaceId → workspace 目录残留
```

### 3.4 会话过期链路（无释放）

```
session 过期（> 2小时）
  → cleanupAllExpiredSessions()  [session-manager.ts:L500]
    → 删除 session 目录（直接 fs.rmSync）
    → ❌ 未清理对应的 workspace 目录
```

### 3.5 丢弃链路（无释放）

```
用户放弃编辑
  → discardEditSession()  [session-manager.ts:L420]
    → 写入 meta.status = 'discarded'
    → 删除 session 目录（直接 fs.rmSync，绕过 deleteSession）
    → ❌ 未清理对应的 workspace 目录
```

### 3.6 被动清理链路（无释放）

```
查找活跃 session 时发现过期 session
  → findActiveSession()  [session-manager.ts:L106]
    → 删除过期 session 目录（直接 fs.rmSync，绕过 deleteSession）
    → ❌ 未清理对应的 workspace 目录
```

### 3.7 API 删除链路（无释放）

```
API DELETE /api/sessions/:sessionId
  → deleteSession()  [fs-utils.ts:L552]
    → 删除 session 目录
    → ❌ 未读取 workspaceId → 未清理 workspace 目录
```

### 3.8 根因总结

**所有 session 清理出口均未处理 workspace 目录的清理。** 核心原因是：

1. **session 和 workspace 是独立的文件系统实体**，但清理逻辑只覆盖了 session 一方
2. **session 删除存在两条路径**，但均未处理 workspace：
   - **路径 A — deleteSession()**：被 `saveEditSession`、`dropEditSession`、API DELETE 三个出口使用，但只删 session 目录，未读取 `.session.json` 中的 `workspaceId` 去同步删除 workspace
   - **路径 B — 直接 fs.rmSync**：被 `discardEditSession`、`cleanupExpiredSessions`、`cleanupAllExpiredSessions`、`findActiveSession` 四个路径使用，绕过了 `deleteSession()`，连修复 `deleteSession()` 也无法覆盖这些路径
3. **没有全局 GC 机制**扫描 workspace 目录中哪些已无关联的 session 引用
4. **Agent-service 有自己的 workspace 清理**（`/tmp/opencode-workspaces/`），但 Web 前端的 `data/workspaces/` 不在其管理范围内

## 四、修复方案

### 4.1 设计原则

**session 是 workspace 的生命周期管理者** — session 创建时 workspace 诞生，session 死亡时 workspace 终结。GC 作为安全网处理极端情况（进程崩溃、数据损坏等）。

### 4.2 三层防护架构

```
┌──────────────────────────────────────────────────────────┐
│  第一层：deleteSession() 内联清理（覆盖 3 个出口）        │
│  删除 session 前读取 workspaceId，同步删除 workspace      │
│  覆盖路径：saveEditSession → deleteSession               │
│           dropEditSession → deleteSession                │
│           API DELETE → deleteSession                     │
├──────────────────────────────────────────────────────────┤
│  第二层：绕过 deleteSession 的路径独立修复（覆盖 4 个出口）│
│  覆盖路径：discardEditSession（直接 fs.rmSync）           │
│           cleanupExpiredSessions（直接 fs.rmSync）        │
│           cleanupAllExpiredSessions（直接 fs.rmSync）     │
│           findActiveSession（直接 fs.rmSync）             │
├──────────────────────────────────────────────────────────┤
│  第三层：后台孤儿 GC 定时器（兜底，防止任何遗漏）          │
│  每 30 分钟扫描 data/workspaces/                          │
│  清理：过期 + 无活跃 session 引用的 workspace 目录        │
│  TTL: 24 小时（允许用户第二天继续对话）                   │
└──────────────────────────────────────────────────────────┘
```

### 4.4 风险与注意事项

| 风险 | 应对 |
|------|------|
| 正在进行的 AI 对话被误删 | GC 通过 session 的 `expiresAt` 判断活跃性 — 未过期的 session 引用的 workspace 不会被清理 |
| 保存中竞态导致数据丢失 | `saveEditSession()` 先复制到 project workspace + 创建 version snapshot，成功后才删除 workspace |
| 元数据损坏导致无法清理 | GC fallback 到目录 mtime 判断，不依赖元数据 |
| `deleteSession` 内部 workspace 删除失败阻断 session 删除 | workspace 删除用 try/catch 包裹，失败只 warn 不阻断 |
| 多用户隔离 | 按 `data/workspaces/{userId}/` 目录结构遍历，天然隔离 |
| Agent-service 仍在引用 | Agent-service 的 workspace 在 `/tmp/` 下，与 `data/workspaces/` 物理隔离 |
| GC TTL 与 session TTL 不匹配 | GC 用 24h（session 过期后用 24h 宽限期），而非 2h（session TTL） |

### 6.3 历史残留清理

一次性清理命令：

```bash
# 查看当前残留
ls -la packages/web/data/workspaces/*/*/ | grep "ws-"

# 手动清理（确认无误后）
find packages/web/data/workspaces -type d -name "ws-*" -exec rm -rf {} +
```

或者等待修复上线后，孤儿 GC 会自动在 24h TTL 后清理这些历史残留。
