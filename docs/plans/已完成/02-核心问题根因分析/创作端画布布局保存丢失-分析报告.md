# 创作端画布布局保存丢失 — 分析报告

## 背景

用户反馈：创作端画布中修改页面位置或尺寸后，点击“保存项目”并不会稳定保存画布布局。

## 目标

- 定位画布布局修改后未随项目保存落盘的根因。
- 修复保存链路，保证保存项目时包含最新画布布局。
- 用最小验证确认 TypeScript 与关键行为不回归。

## 范围

- 创作端编辑页画布工作台。
- 画布布局 session 持久化接口。
- 保存项目时的布局刷新/同步逻辑。

不处理与当前问题无关的 data 目录生成文件、AI 对话展示变更和截图缓存变更。

## 方案

推荐方案：

1. 将 `.canvas-layout.json` 视为 workspace 状态：布局 API 读取时在 workspace/session 候选中选择最新有效文件，兼容旧 session 文件；写入时同时写 workspace 与 session。
2. `useCanvasWorkspace` 暴露 `flushCanvasState()`，取消 pending debounce 并立即保存当前内存布局。
3. 创作端“保存项目”在写页面文件和调用 `/save` 前先 `await flushCanvasState()`。

## 根因分析

### 现象

用户在创作端画布调整页面布局后点击“保存项目”，重新进入项目时布局未保存或刚调整的布局丢失。

### 证据链

| 证据 | 级别 | 来源 | 说明 |
| --- | --- | --- | --- |
| 画布更新通过 `onCanvasStateChange={setCanvasState}` 进入 `useCanvasWorkspace` 管理 | A | `packages/author-site/src/app/demo/[id]/edit/page.tsx` | 页面位置/尺寸变更只更新 React 状态 |
| `useCanvasWorkspace` 通过 `SAVE_DELAY = 700` 的定时器调用 `saveCanvasLayout` | A | `packages/author-site/src/components/demo/useCanvasWorkspace.ts` | 点击保存时可能仍有未落盘的 debounce |
| 布局 API 原先只写 `getCanvasLayoutPath(access.sessionPath)` | A | `packages/author-site/src/app/api/sessions/[sessionId]/canvas-layout/route.ts` | 布局文件落在 session 目录 |
| 项目保存 `saveEditSession` 的 sourcePath 优先使用 `workspaceId` 对应 workspace，并复制该目录到项目 workspace | A | `packages/author-site/src/lib/session-manager.ts` | session 目录下的 `.canvas-layout.json` 不会进入项目 workspace |

### 根本原因

画布布局的持久化位置与项目保存源不一致：布局接口写 session 目录，而项目保存复制 workspace 目录。同时保存按钮没有主动 flush 画布布局 debounce，导致刚拖拽后的最新状态即使在内存中，也可能尚未写入任何持久化文件。

### 代码执行路径

`CanvasPageItem` 拖拽/缩放 → `PreviewCanvas.handleLayoutChange` → `useCanvasWorkspace.setCanvasState` → debounce `saveCanvasLayout` → `POST /api/sessions/[sessionId]/canvas-layout` → 原先写入 session 目录。

保存项目路径：`handleSave` → `PUT /api/sessions/[sessionId]/files/[activeDemoId]` → `POST /api/sessions/[sessionId]/save` → `saveEditSession` → 复制 workspace 到项目 workspace。
