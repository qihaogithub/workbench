### 2.3 根本原因

#### 根因 1: 保存操作的原子性缺陷（保存失败）

**证据**: E1, E3

`saveEditSession` 函数在 L419-423 执行以下操作：

```typescript
fs.rmSync(workspacePath, { recursive: true, force: true });  // 先删除
fs.cpSync(sourcePath, workspacePath, { recursive: true });   // 再复制
```

**问题**: 虽然 L414-416 已做 sourcePath 存在性守卫，但 `cpSync` 在执行过程中仍可能失败（如磁盘空间不足、文件权限变更、IO 错误），此时 `rmSync`（L419）已执行，workspace 已删除但未能恢复，导致：
- workspace 内容为空或不完整
- 后续保存操作因找不到文件而失败
- 项目处于损坏状态

**影响**: 一次保存失败会导致后续所有保存操作都失败，形成"保存常常失败"的现象。

#### 根因 2: 发布状态初始化逻辑错误（未发布显示已发布）

**证据**: E4, E5

当用户**首次创建项目并保存**时：
1. `project.versions` 数组添加第一个版本，如 `"v1"`
2. `project.publishedVersion` 为 `undefined`（从未发布）

但存在一种边界情况：**如果用户曾经发布过，然后继续编辑并保存**：
1. 假设发布时 `publishedVersion = "v3"`, `currentVersion = "v3"`
2. 用户编辑后保存，`currentVersion` 变为 `"v4"`
3. 此时 `publishedVersion ("v3") !== currentVersion ("v4")` → 正确显示"未发布变更"

**然而**，如果用户在保存后**刷新页面**或**重新进入编辑**：
- 页面加载时调用 `getPublishStatus()` (edit/page.tsx:702-708)
- 如果某个历史操作中 `publishedVersion` 被错误设置为最新版本号
- 就会显示"已发布"

**关键发现**: 前端代码在保存成功后**强制设置** `setPublishStatus('unpublished_changes')` (L907)，这表明开发者已经意识到发布状态计算可能不准确，采用了前端覆盖的方式。

**但这只是表面修复**，根本原因尚未在代码中确认。需进一步排查的方向包括：
1. **用户是否实际点击过"发布"按钮** — 发布操作会设置 `publishedVersion`（`publish-manager.ts:286`，这是代码中唯一写入该字段的位置），发布后继续编辑保存就会触发 `publishedVersion !== currentVersion`，此时页面加载 API 会返回 `unpublished_changes`，正确；但若用户发布后**从未再保存**，刷新页面就会显示"已发布"
2. **可能存在前端状态未同步的场景** — 例如用户在多个标签页中打开同一项目，一个标签页发布后，另一个标签页的前端缓存状态过期
3. **`project.json` 文件损坏** — 极端情况下文件系统损坏可能使 `publishedVersion` 字段异常

> ⚠️ **注意**: 全库搜索确认 `publishedVersion` 仅在一处写入（`publish-manager.ts:286`），不存在其他代码路径意外修改该字段。如果用户确认从未执行发布操作但仍显示"已发布"，建议直接检查 `data/projects/{projectId}/project.json` 中 `publishedVersion` 的实际值以确定问题来源。

#### 根因 3: 环境依赖缺失（可能加剧保存失败）

**证据**: E7

Agent Service 未运行（端口 3201 空闲）。虽然保存操作本身不依赖 Agent Service，但如果：
- Session 创建需要 Agent Service
- 或者编辑过程中的某些功能（如代码验证）依赖 Agent Service

那么 Agent Service 缺失可能导致 session 状态异常，间接影响保存流程。

## 3. 解决方案

### 3.1 推荐方案：修复保存操作的原子性

**针对根因 1**

#### 具体做法

将 `saveEditSession` 中的"先删后拷"改为"先拷后删"的原子操作：

```typescript
// 当前代码 (危险) — session-manager.ts L419-423
fs.rmSync(workspacePath, { recursive: true, force: true });
fs.cpSync(sourcePath, workspacePath, {
  recursive: true,
  filter: (src: string) => !src.includes('node_modules'),
});

// 改进方案 (安全) — 替换 L419-432 整段
const tempPath = workspacePath + '.tmp';
try {
  // 1. 先复制到临时目录（保持原始 node_modules 过滤规则）
  fs.cpSync(sourcePath, tempPath, {
    recursive: true,
    filter: (src: string) => !src.includes('node_modules'),
  });

  // 2. 清理临时目录中的元数据文件（替代原始 L425-432 的清理）
  const tempMetaJson = path.join(tempPath, ".session.json");
  if (fs.existsSync(tempMetaJson)) {
    fs.rmSync(tempMetaJson, { force: true });
  }
  const tempWsJson = path.join(tempPath, ".workspace.json");
  if (fs.existsSync(tempWsJson)) {
    fs.rmSync(tempWsJson, { force: true });
  }

  // 3. 原子替换：删除旧目录，重命名新目录
  fs.rmSync(workspacePath, { recursive: true, force: true });
  fs.renameSync(tempPath, workspacePath);
} catch (error) {
  // 4. 失败时仅清理临时目录，旧 workspace 完好无损
  fs.rmSync(tempPath, { recursive: true, force: true });
  throw error;
}
```

> **方案说明**: 使用 `cpSync → renameSync` 替代 `rmSync → cpSync`，
> 确保复制失败时旧 workspace 不受影响。
> 原始代码中 L425-432 的元数据清理逻辑一并移入临时目录阶段。

**为什么有效**: 
- 即使复制失败，旧 workspace 仍然完好
- 使用临时目录 + rename 实现原子替换
- 保留原始 node_modules 过滤规则和元数据清理逻辑

**影响范围**: 仅修改 `session-manager.ts` 的 `saveEditSession` 函数

**风险**: 低 — 仅改变操作顺序，不影响正常流程

**复杂度**: 低 — 约 15 行代码改动

### 3.2 方案二：增加保存失败后的恢复机制

**针对根因 1 的补充**

#### 具体做法

在 `saveEditSession` 的 catch 块中增加恢复逻辑：

```typescript
catch (error) {
  console.error(`[saveEditSession] 保存失败:`, error);
  
  // 尝试从最新快照恢复 workspace
  if (project.versions.length > 0) {
    const latestVersion = project.versions[project.versions.length - 1];
    const latestSnapshot = latestVersion.snapshotPath;
    
    if (fs.existsSync(latestSnapshot)) {
      console.log(`[saveEditSession] 尝试从快照 ${latestVersion.versionId} 恢复`);
      try {
        fs.cpSync(latestSnapshot, workspacePath, { recursive: true });
        console.log(`[saveEditSession] 恢复成功`);
      } catch (restoreError) {
        console.error(`[saveEditSession] 恢复失败:`, restoreError);
      }
    }
  }
  
  if (error instanceof Error) {
    return { success: false, error: `Save failed: ${error.message}` };
  }
  return { success: false, error: 'Save failed' };
}
```

**为什么有效**: 即使保存失败，也能从快照恢复，避免 workspace 损坏

**影响范围**: 修改 `session-manager.ts` 的错误处理逻辑

**风险**: 低 — 仅在失败时触发，不影响正常流程

### 3.3 方案三：修复发布状态计算逻辑

**针对根因 2**

#### 具体做法

在 `getPublishStatus` 中增加更严格的验证：

```typescript
export function getPublishStatus(projectId: string) {
  const project = readProjectMeta(projectId);
  if (!project) {
    throw new Error("PROJECT_NOT_FOUND");
  }

  const currentVersion =
    project.versions.length > 0
      ? project.versions[project.versions.length - 1].versionId
      : undefined;

  // 增加验证：publishedVersion 必须是有效的版本号
  const isValidPublishedVersion = 
    project.publishedVersion && 
    project.versions.some(v => v.versionId === project.publishedVersion);

  const status = !isValidPublishedVersion
    ? "never_published"
    : project.publishedVersion === currentVersion
      ? "published"
      : "unpublished_changes";

  return {
    projectId: project.id,
    publishedVersion: isValidPublishedVersion ? project.publishedVersion : null,
    publishedAt: isValidPublishedVersion ? project.publishedAt : null,
    currentVersion: currentVersion ?? null,
    hasUnpublishedChanges: status === "unpublished_changes",
    status,
  };
}
```

**为什么有效**: 
- 防止 `publishedVersion` 指向不存在的版本号
- 增加数据一致性校验

**影响范围**: 修改 `publish-manager.ts` 的 `getPublishStatus` 函数

**风险**: 极低 — 仅增加验证逻辑

## 5. 总结

### 5.1 核心问题

1. **保存失败根因**: `saveEditSession` 采用"先删后拷"的非原子操作，一旦复制失败导致 workspace 损坏，后续保存全部失败
2. **状态异常根因**: 根因尚未在代码中确认。`publishedVersion` 仅发布操作写入（`publish-manager.ts:286`），无其他代码路径意外修改。但前端保存后强制设置 `unpublished_changes` 是表面修复，真正的触发条件需检查实际的 `project.json` 文件内容定位

### 5.2 修复优先级

1. **P0 (紧急)**: 修复保存操作的原子性（方案一）— 直接影响用户核心功能
2. **P1 (高)**: 增加保存失败恢复机制（方案二）— 提高系统容错能力
3. **P2 (中)**: 修复发布状态计算逻辑（方案三）— 改善用户体验

### 5.3 验证方法

修复后应进行以下测试：

1. **正常保存测试**: 编辑页面 → 保存 → 验证 workspace 内容更新
2. **异常保存测试**: 模拟复制失败（如删除 sourcePath）→ 验证 workspace 未被损坏
3. **发布状态测试**: 
   - 新建项目保存 → 应显示"未发布"
   - 发布后保存 → 应显示"有未发布变更"
   - 刷新页面 → 状态应保持正确
4. **并发保存测试**: 快速连续点击保存 → 验证不会出现竞争条件
