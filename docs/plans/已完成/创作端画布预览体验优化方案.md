# 创作端画布预览体验优化方案

> 创建日期：2026-06-25
> 状态：进行中
> 范围：创作端画布页面选中态、截图与 iframe 切换、活跃/休眠 iframe 调度、页面资源指纹缓存、截图服务延迟降级策略

## 背景

画布工作台用于同时查看多个 Demo 页面。现有实现已经支持截图替代 iframe，能降低多页面同时渲染的性能压力；但选中页面仍可能停留在截图态，用户无法感知页面已经进入可编辑上下文。截图服务延迟时，画布也需要在“永远显示最新效果”和“避免 iframe 过多拖慢浏览器”之间做明确取舍。

## 目标

- 用户在画布中选中页面后，页面边框以白色描边高亮。
- 被选中的页面优先升级为真实 iframe，并在 iframe 页面加载完成后再从截图渐变切换。
- 非选中页面优先使用当前版本截图；截图不可用时使用实时 iframe。
- 当可见区域内需要实时 iframe 的页面过多时，只保留有限数量的活跃 iframe，其余页面显示轻量加载态，避免性能被拖垮。
- 超出活跃预算但近期加载过的页面可进入休眠 iframe，保留 iframe DOM 并暂停更新与测量。
- 页面图片资源应按资源指纹提前预热，降低页面从截图或加载态升级为 iframe 时的图片闪烁。
- 保持“最新页面效果优先”原则：旧截图不能替代当前版本页面。

## 方案

### 1. 选中页面无感升级

选中页无论是否已有截图，都渲染真实 iframe。若已有截图，截图先继续显示，iframe 在背后加载；收到 `PreviewPanel` 的内容加载完成信号后，截图淡出，iframe 淡入。

### 2. 活跃 iframe 预算

画布顶层根据可见页、选中页、截图可用性和页面距离视口中心的距离排序，给需要实时预览的页面分配有限 iframe 预算。默认保留 12 个活跃 iframe；选中页优先进入活跃态。超出活跃预算但近期进入过活跃态的页面最多保留 12 个休眠 iframe，超出后回到加载态。

### 3. 截图优先但不牺牲最新性

截图 URL 仍由 author-site 过滤 hash、loading 和 renderBox 兼容性后传入画布。画布只消费“可用截图”，不自行读取旧截图；没有截图时优先实时 iframe，预算不足时显示加载态。

### 4. 客户端截图评估

客户端截图不作为本轮实现。原因是 iframe 内容截图通常依赖 DOM 克隆或 canvas 绘制，容易受跨域图片、字体、shadow/iframe 边界和样式还原影响；它可以作为“服务端截图不可达且 iframe 预算耗尽”时的实验性兜底，但不应替代服务端 1:1 截图主链路。

### 5. 页面资源指纹缓存

画布对 active 和 sleeping 页面提取配置与代码中的图片 URL，按页面代码、配置图片列表和预览尺寸生成资源指纹。相同指纹只预热一次；图片预热使用浏览器图片缓存和解码能力，失败不阻断页面渲染。

## 风险与待确认事项

- 活跃 iframe 默认预算需要结合真实项目页面数量和机器性能继续调优。
- 休眠 iframe 仍会占用一部分浏览器内存；默认最多 12 个 sleeping，需要结合真实项目继续观察。
- 图片资源预热第一版只处理图片，不处理字体、CSS、JS module 或 iframe HTML blob。
- 客户端截图若后续实施，需要单独验证跨域图片、字体、滚动长页和 iframe 内交互状态一致性。
