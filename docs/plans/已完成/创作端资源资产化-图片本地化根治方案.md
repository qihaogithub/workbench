# 创作端资源资产化-图片本地化根治方案

## 当前状态

P0/P1/P2 已实现，P3 的第一层引用语义已通过项目级资产清单和 `asset list` 引用扫描落地。本文档用于沉淀创作端图片资源资产化的目标方案，覆盖预览区选中图片、右侧属性面板、AI 对话、项目工作区资源、发布本地化和 Figma/R2 导入资源。当前问题的直接现象是：图片在预览区能正常显示，但用户要求“不要远程 URL，下载到本地”时，AI 通过后端 `saveImage(source="url")` 下载远程 URL 超时，无法完成本地化。

已落地的 P0 边界：

- 新增 `POST /api/sessions/{sessionId}/assets/localize`，优先接收浏览器端 `browserBlob`，浏览器不可读时才由 author-site 后端按远程 URL 兜底下载。
- 右侧属性面板的图片属性新增“本地化当前图片”动作，成功后返回 `../../assets/images/...` 并复用现有原型页属性直接写回链路。
- 视觉属性补充说明命中“改为本地图片 / 不要远程 URL / 下载到本地”等意图时，发送给 AI 前先复用同一本地化入口；高保真页 prompt 只携带已生成的 `relativePathFromPage`，HTML/CSS 原型页可直接写回时不再进入 AI。
- 本地化产物写入当前 workspace `assets/images/`，并登记到项目级 `images.json`，可被 `listImages` 看到。
- 远程 URL 兜底保留公网协议、私网地址、content-type、大小和重定向防护；双路径失败时明确提示需要上传原图。

已落地的 P1/P2/P3 基础边界：

- `project-core` 的 `uploadAsset` 统一写入项目级 `images.json`，记录 `contentHash`、`mimeType`、`originalUrl`、`sourceType`、`createdBy` 和创建时间；默认落盘文件名改为内容哈希前缀，显式 `targetPath` 仍保持调用方指定路径。
- `asset list` 合并 workspace 文件扫描、项目资产清单元数据和引用关系；删除资产时同步移除清单条目。
- agent-service `saveImage` 支持 `assetId`、`sessionAsset`、`base64`、`url`，受管资产优先，远程 URL 只作为兜底。
- Project Admin CLI 的 `project import-prototype` / `page update-prototypes` 在写入 HTML/CSS 原型页前本地化远程图片 URL；成功时改写为 `../../assets/images/...`，失败时保留外链并返回导入警告。

## 当前结论

这不是模型能力问题，也不应主要通过延长下载超时、重试或换模型解决。根因是系统把“资源本地化”作为 AI 工具任务处理，而不是作为创作平台的基础资源能力处理。

预览区显示图片和 AI 保存图片是两条不同链路：

- 预览区由浏览器 iframe 加载 `<img src="...">`，可能命中浏览器网络、缓存、CORS 规则或已经完成的 DOM 渲染。
- AI 的 `saveImage(source="url")` 在 agent-service 进程中用 Node `http/https` 重新下载同一个 URL，受生产容器网络、DNS、Cloudflare/R2 可用性、超时和 content-type 校验影响。

因此“用户看得到”不能推导出“后端能下载”。更关键的是，用户看得到的图片二进制已经存在于浏览器可见上下文中，但当前产品没有把这份可见资源先资产化，再把受管资产路径交给 AI。

## 目标原则

目标不是“让 AI 更会下载图片”，而是：

> 图片一进入创作系统，就应被转成项目资产；URL 只是资产的一种来源，不是长期身份。

长期产品约束：

- 用户在预览区能看到的图片，系统应能一键或自动转为项目本地资产。
- AI 不负责抓取浏览器资源、不负责跨域兜底、不负责生产网络诊断，只负责把已登记资产应用到代码或配置。
- 页面源码、配置值、发布产物和 AI 工具返回值使用不同形态的资源 URL，但它们应指向同一个稳定资产身份。
- 外部 URL 只作为资源来源和审计信息保留，不作为项目长期事实。

## 目标架构

### 1. Asset Acquisition 资源获取

负责把各种来源转成二进制输入，并处理获取阶段的不确定性。

来源优先级：

1. 浏览器端 Blob 或 base64：用于“预览区已经显示的图片”，优先从当前页面可见资源获取，避免服务器网络差异。
2. 用户上传文件：沿用属性面板和聊天附件的文件输入能力。
3. Session asset：复用 `/api/sessions/{sessionId}/assets/...` 已上传资源。
4. 工作区本地资源：复用 `assets/images/...` 或页面相对路径。
5. 自家 R2/Worker 资源：可由后端服务端下载或内部复制，作为受控远程来源。
6. 普通 http/https URL：最后兜底，需要协议、私网地址、content-type、大小和重定向校验。

资源获取失败时，应返回具体原因：跨域不可读、远程下载超时、非图片、超限、私网地址拒绝或源资源缺失。不要让 AI 生成“建议用户再上传”的泛化回答，除非所有获取路径都失败。

### 2. Asset Registry 资产登记

引入项目级资产登记层，以 `assetId` 或 `contentHash` 作为稳定身份，记录：

- `assetId`
- `contentHash`
- `workspacePath`
- `mimeType`
- `size`
- `width` / `height`（可选）
- `originalUrl`（可选，仅作来源和审计）
- `sourceType`：`browser_blob`、`upload`、`session_asset`、`workspace_asset`、`r2_worker`、`remote_url`
- `createdBy`：`user`、`ai`、`figma`、`system`
- `createdAt`
- `references`：后续可记录页面、配置字段和发布引用

现有 `saveImage` 的 `images.json` 可以作为早期实现基础，但目标上应升级为统一项目资产清单，而不是仅服务 AI 工具的图片列表。

### 3. Asset Materialization 资产物化

同一个资产在不同场景下生成不同可用路径：

- 工作区落盘路径：`assets/images/{hash}-{filename}`
- 页面内相对路径：`../../assets/images/{hash}-{filename}`
- 编辑态预览 URL：`/api/sessions/{sessionId}/workspace/assets/images/{hash}-{filename}` 或等价 session workspace URL
- 发布态 URL：`/data/{projectId}/assets/images/{contentHash}.{ext}`
- AI 工具返回：同时返回 `workspacePath`、`relativePathFromPage`、`assetId` 和 `contentHash`

页面和配置可以继续存字符串路径，但内部 API 应尽量携带资产元数据，避免把“字符串 URL”误当成唯一事实。

### 4. Reference Rewrite 引用写回

资源资产化完成后，才进入引用写回：

- 目标元素能唯一定位时，优先由 visual-configure 或属性面板直接写回，不进入 AI。
- 目标元素不能唯一定位、需要跨多个文件判断或需要代码结构推理时，才把 `assetId`、页面相对路径和选中元素上下文交给 AI。
- AI 收到的任务应是“把这个已本地化资产应用到这个元素”，而不是“请从这个远程 URL 下载图片”。

## 核心用户流程

### 预览区选中远程图片并要求本地化

1. 用户选中图片元素，属性面板或视觉选择状态拿到 `src`、`currentSrc`、`ow-id`、DOM 路径、页面 ID 和运行时类型。
2. 前端调用 `localizeSelectedImage`，提交选中资源上下文。
3. `localizeSelectedImage` 优先尝试浏览器端读取当前图片 Blob。若可读，直接上传二进制。
4. 若浏览器端因 CORS 或 canvas taint 不可读，调用后端 localizer 以远程 URL 下载或从受控 R2/Worker 复制。
5. 后端写入工作区 `assets/images/`，更新项目资产清单，返回资产物化路径。
6. 若选中元素可唯一定位，系统直接把 `src` 改成页面相对路径并标记 workspace dirty。
7. 若不能唯一定位，把本地资产路径和目标上下文交给 AI 修改。
8. 预览和右侧属性面板刷新为本地资产，不再展示远程 URL。

### 配置项图片上传或替换

1. `format: "image"` 字段继续使用现有上传控件。
2. 上传成功不只返回 `/api/sessions/.../assets/...`，还应进入项目资产登记或具备后续 promotion 信息。
3. 保存、发布或用户选择“固化到项目资源”时，session asset 被提升为 workspace asset。
4. 配置运行值使用可物化的本地项目路径，发布阶段继续按现有发布本地化规则生成 `/data/{projectId}/assets/images/...`。

### AI 聊天中上传图片并要求放入页面

1. 图片附件先进入图片理解链路或文件附件链路。
2. 当用户明确要求“放入页面”“替换图片”“保存到本地”时，后端将附件通过同一 Asset Registry 保存成项目资产。
3. AI 只拿到资产路径和目标说明，不再需要把附件 base64 再手动传给 `saveImage`。

### Figma/R2 导入资源

导入阶段应批量执行资产化。由 Figma 插件、R2 Worker 或导入服务产生的远程 URL 应尽早转为项目资产。页面初始代码应优先引用 `assets/images/...`，而不是长期保留 `r2-asset-worker...`。

发布阶段的外部图片本地化继续保留，但定位为安全网，不作为编辑期主路径。

## 分阶段实施

### P0：修复当前体验缺口

- 新增或扩展前端“本地化选中图片”动作：在用户选中图片并要求下载到本地时，先执行资源资产化，再决定是否交给 AI。
- 对可读图片优先走浏览器 Blob 上传，避免 agent-service 服务器网络差异。
- 对不可读图片返回明确错误原因，提示用户上传原图；不要让 AI 重复尝试同一个远程 URL。
- AI 提示词调整：如果系统已经提供本地资产路径，AI 必须使用该路径，不再调用 `saveImage(source="url")`。

验收标准：

- 复现 `deco-bg` 场景时，用户点击图片并输入“不要远程 URL，下载到本地”，最终 `prototype.html` 中的该图片引用变为 `../../assets/images/...`。
- agent-service 无法访问远程 URL 时，只要浏览器端能读取当前图片 Blob，任务仍能成功。
- 浏览器端也因跨域不可读且后端下载失败时，界面明确提示“需要上传原图”，AI 不输出泛化建议。

### P1：统一项目资产服务

- [x] 在 project-core 中建立统一 Asset Registry 基础能力，收敛 AI saveImage、属性面板本地化、CLI 资产上传和项目资产清单。
- [x] `saveImage` 升级为资产服务客户端，支持 `source=assetId/sessionAsset/base64/url`，其中 URL 是最低优先级。
- [x] 项目资产清单记录 `contentHash`、`originalUrl`、来源、尺寸、创建者和创建时间等元数据；引用关系由 `asset list` 扫描合并。
- [x] 默认资产落盘命名改为内容哈希前缀，相同内容可识别复用；显式 `targetPath` 保持目录导入语义。

验收标准：

- 属性面板上传、AI 保存和选中图片本地化产生的图片都出现在同一项目资产清单。
- 相同图片内容重复本地化只生成一个内容资产或可识别复用。
- `listImages` 能看到属性面板和 localizer 创建的资产，不再只看到 AI `saveImage` 创建的图片。

### P2：导入阶段资产化

- [x] Figma/R2 原型导入项目时批量本地化远程图片。
- [x] 导入后的原型 HTML/CSS、页面 Schema 默认值优先引用项目 workspace assets。
- [x] 对无法本地化的远程资源记录导入警告，并允许保留外链作为兼容降级。

验收标准：

- 新导入项目默认不再大面积保留 `r2-asset-worker...` 作为页面长期图片引用。
- 导入后的项目即使 R2 Worker 临时不可用，已资产化图片仍可在编辑态预览和发布态显示。

### P3：资产引用语义升级

- [x] 配置系统外部仍兼容字符串 URL，项目内部通过 `images.json` 保留资产元数据。
- [x] 页面、配置、发布和 AI 工具之间已可传递 `assetId + materializedUrl`，减少字符串路径歧义。
- [x] 引用关系已由 `asset list` 扫描合并到资产摘要，可用于删除预览、替换影响检查和发布差异审计。

验收标准：

- 删除或替换图片前，系统能知道哪些页面或配置字段引用了该资产。
- 发布包能基于资产身份判断图片是否已本地化，而不是反复扫描和下载同一 URL。

### AI 工具升级

`saveImage` 的目标形态：

```ts
type SaveImageSource =
  | { source: "assetId"; assetId: string }
  | { source: "sessionAsset"; url: string }
  | { source: "base64"; data: string; filename: string }
  | { source: "url"; data: string; filename: string };
```

优先级上，`assetId` 和 `sessionAsset` 是平台受管资源；`url` 是远程兜底，不应作为选中预览图本地化的默认路径。

## 风险

- 浏览器 Blob 读取受 CORS 限制。方案必须允许“前端 Blob 优先，后端下载兜底，全部失败再要求上传原图”，不能承诺所有可见图片都一定能被浏览器直接读取。
- 资产身份升级会触及 session assets、workspace assets、AI 工具、发布本地化和 project-core，必须分阶段兼容旧字符串路径。
- 如果过早把配置字段值改成复杂对象，可能破坏现有预览、发布和使用端配置面板。短期应保持字符串路径兼容，内部补充 metadata。
- 发布阶段外部图片本地化仍需保留，因为历史项目和手写代码仍可能包含外链。
- 远程 URL 下载必须继续保留私网地址防护，避免资产本地化接口变成 SSRF 通道。
