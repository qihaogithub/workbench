# 项目管理方案长期优化评估

> 状态：P0 实施完成并验证通过
> 日期：2026-06-30

## 背景

本记录用于沉淀对当前创作端项目管理方案的架构评估：当前方案是否合理，是否存在更优雅的长期优化方向，以及后续优化应优先收紧哪些边界。

评估范围来自 [项目管理模块](../../项目文档/创作端/03-项目管理/) 及其相关 CLI、协作、模板和源码入口。本文属于任务追踪材料，不替代长期项目知识库；如果后续实施会改变产品行为、接口契约、架构边界或模块职责，需要另行更新 `docs/项目文档/` 下对应模块文档。

## 目标

- 判断当前项目管理方案的长期合理性。
- 记录关键事实依据和相关路径，方便后续继续推进。
- 给出更优雅的长期优化点和建议优先级。
- 明确哪些问题是语义收敛问题，哪些问题是工程可靠性问题。

## 范围

本次评估覆盖：

- Workspace、Session、Project、Version 的职责边界。
- 实时保存与多人协作的当前态模型。
- 项目版本、发布快照、模板快照和恢复快照关系。
- `project-core`、CLI、本地项目包和 Web API 的能力边界。
- 文件系统落盘、事务、快照和未来多实例协作风险。

本次不覆盖：

- 具体 UI 视觉设计。
- 具体代码改造实现。
- 数据库化或对象存储迁移的详细方案。
- 多实例协同的选型和部署细节。

- Web 侧命名版本和自动检查点已有前端 `flush-all` 调用，agent-service 也提供单资源 flush 和 workspace flush-all；P0 实施后，author-site 发布、导出、自动检查点、保存/合并、页面命名版本和页面恢复入口已经补齐后端强制 flush 前置。
- 当前协同服务用内存 Yjs 房间、dirty 标记、debounce 落盘和空闲房间清理前 flush 管理草稿；文件写入使用临时文件加 `rename`，但 Yjs update、presence 和 flush 队列本身不跨实例持久化。
- 模板元信息已经包含 `sourceProjectId`，但缺少来源版本、模板版本、归档状态等更完整 lineage 字段；模板转普通项目和项目复制的临时模板仍采用物理删除。

### P0 实施前识别的不一致信号

- [会话管理需求](../../项目文档/创作端/03-项目管理/会话管理_需求文档.md) 曾保留“原始项目文件直到明确保存才变”的旧心智；P0 已将需求语义调整为“Workspace 当前态持续自动保存，版本用于留档和回退”。
- 文档中仍存在“临时工作空间”“独立工作空间”“正式工作空间”“当前工作空间”等历史说法，P0 已清理项目管理核心需求和协同/版本技术文档，其他历史方案文档后续按需继续治理。
- Web API、author-site 旧文件系统工具、`project-core` 和 CLI 之间仍存在历史双轨路径，长期应继续收敛到领域服务。
- 多人协作的目标语义是多个协作者编辑同一个 Workspace 草稿；P0 已选择项目共享草稿方向，并移除协同写入中的单一 workspace owner 阻断。项目成员/角色模型仍需后续补齐。

## 结论

当前方案方向合理，不建议推倒重来。最核心、最值得保留的设计是：

```text
Session = AI 对话上下文
Workspace = 当前可编辑文件状态
Version = 可回退历史节点
Publish = 对外可见快照
Template = 可复用项目快照
```

这个模型比“每个会话一份临时目录，最后手动保存覆盖正式目录”的旧模型更适合长期发展，因为它能同时支撑：

- AI 对话切换不影响文件状态。
- 多人共同编辑同一个项目草稿。
- 自动保存防丢失。
- 命名版本、发布快照和恢复快照追溯。
- CLI、本地项目包和模板能力复用同一套项目语义。

当前主要问题不是模型方向错误，而是模型演进后仍有旧语义、旧实现路径和身份边界残留。长期最优方向应是“明确状态源、统一关键入口、补齐协作身份语义、强化事务可靠性”。其中跨用户 Workspace 归属和关键动作强制 flush 的 ROI 最高，因为它们直接决定协作与历史快照是否可信。

## 长期优化点

### 1. 统一 Workspace 当前态语义

建议把 “Workspace 当前态” 明确提升为项目编辑的主状态源：

- 自动保存负责把编辑内容持续写入当前编辑 Workspace。
- 命名版本、自动检查点、发布快照和恢复快照都只是从当前态派生出的历史节点。
- 用户离开编辑页时，不再把“没有命名版本”视为未保存风险；只有未 flush、离线、同步失败或权限异常才是风险。
- 需要区分“项目 canonical workspace”和“编辑 workspace”：如果短期保留用户级编辑 workspace，就不要把它描述成项目级共享草稿；如果产品目标是真多人协作，应把项目当前草稿提升为共享状态源。

对应需要清理的表述：

- “原始项目文件应保持不变，直到用户明确保存”。
- “临时工作空间”作为主要心智。
- “保存”同时表示防丢、提交、版本命名和发布前置。

### 2. 继续收敛到 `project-core`

长期应让 Web API、CLI、脚手架同步、模板和发布都通过 `project-core` 处理核心业务规则。

建议边界：

- `project-core`：项目、Workspace、版本、模板、事务、审计、权限、发布状态的唯一业务规则层。
- author-site API：认证、请求适配、响应包装和 Web 交互语义。
- project-cli：命令参数、JSON 输出和代理可读错误建议。
- project-scaffold：项目包协议转换和本地目录读写。

这样可以减少 Web 与 CLI 能力不一致，也降低后续批量治理成本。

短期不要一次性迁移所有 Web API。ROI 更高的路径是先收敛会改变项目正式状态的入口：命名版本、发布前同步、恢复版本、模板转换、Web scaffold 导出和 session 保存。

### 3. 把 flush 变成后端关键动作协议

当前前端已在命名版本、自动检查点等流程调用协同 `flush-all`，但可靠边界不能长期依赖每个调用方记得先 flush。建议把以下规则下沉为统一协议：

- 命名项目版本、页面版本、自动检查点、发布、Web scaffold 导出和恢复前，后端都应先执行 workspace 级 flush。
- flush 应返回明确结果：没有活跃房间、已落盘、落盘失败、会话/权限不匹配。
- `project-core` 不应直接依赖 agent-service 的 HTTP 细节，可通过 author-site 适配层或可替换的 `WorkspaceFlushCoordinator` 接口注入。
- CLI 场景没有协同房间时，flush coordinator 应返回 no-op，但仍保留同一业务前置点。

### 4. 补齐跨用户协作的 Workspace 身份语义

当前代码会用 workspace 元数据里的 `userId` 约束协同写入，这与“多人共同编辑同一个项目草稿”的目标存在差距。长期需要二选一并写入产品语义：

- 若 Workspace 是用户私有草稿：多人协作应限定为同一用户的多个会话/窗口，文档和 UI 不应承诺跨用户共同编辑。
- 若 Workspace 是项目共享草稿：workspace 元数据应从单一 `userId` 改为成员/权限模型，Session 校验应校验项目权限而不是 workspace owner。

从长期价值看，项目共享草稿更符合实时协作、发布前同步和自动保存模型；但它会牵涉权限、presence 展示、工作区清理和冲突策略，适合先做方案和小范围验收。

### 5. 提升提交、恢复和发布的事务可靠性

当前文件系统方案适合快速迭代，但长期应引入事务化落盘协议：

- 先写入 staging 目录。
- 校验 staging 完整性。
- 写入 transaction journal。
- 原子切换正式 workspace 指针或目录。
- 成功后清理旧状态，失败时按 journal 恢复。

这套机制应复用于：

- 编辑事务提交。
- 命名版本。
- 发布前同步。
- 恢复版本。
- 模板生成和模板转普通项目。
- CLI submit。

短期可以先做更小的高 ROI 改进：把 `commitEdit`、`restoreVersion`、`syncEditSessionToProjectWorkspace` 这类正式 workspace 替换统一为“复制到临时目录、校验、rename 替换、失败保留原目录”的 helper；journal 可在文件替换路径收敛后再补。

### 6. 为多实例协同预留抽象边界

[实时保存与协同编辑](../../项目文档/创作端/03-项目管理/技术/11_实时保存与协同编辑.md) 当前明确先面向单实例部署，这是合理阶段选择。

但长期应提前把以下能力抽象为可替换接口：

- 房间定位和房间生命周期。
- Yjs update 存储。
- presence 状态。
- dirty flush 队列。
- workspace 级 flush-all。

短期可以继续用内存实现；关键是不要让内存房间模型成为不可替换的业务核心。

### 7. 快照从全量复制演进到内容寻址

当前版本和模板都以工作空间快照为核心，简单直观。项目规模扩大后，可以逐步演进为内容寻址存储：

- 文件内容按哈希去重。
- 版本记录 tree hash。
- 模板复制和项目复制复用已有 blob。
- 发布快照绑定不可变 tree。

这会降低版本、模板、复制项目和本地包导出的存储成本，也能增强审计和一致性。

### 8. 模板补充 lineage 与归档语义

模板当前是独立快照，不跟随源项目自动更新，这个方向正确。

当前已具备 `sourceProjectId`。长期建议补充：

- `sourceVersionId`
- `templateVersion`
- `createdFromSnapshot`
- `archivedAt`

“模板转普通项目”可以优先考虑归档模板而不是物理删除，保留推荐、审计和历史追溯价值。

### 9. 本地项目包协议成为一等契约

项目包协议已经承担 Web 下载、CLI pull/submit、模板本地化和脚手架同步的稳定面。后续建议补充：

- schema version 迁移策略。
- 兼容矩阵。
- 破坏性变更检测。
- 协议级 fixture 和 golden tests。
- 本地包与远端 Workspace 当前态的冲突合并规则。

## 优先级建议

| 优先级 | 优化项 | 原因 |
|---|---|---|
| P0 | 明确 Workspace 当前态与跨用户协作语义 | 当前文档承诺多人共享草稿，但代码仍有 workspace owner 限制，必须先定边界 |
| P0 | 后端关键动作统一强制 workspace flush | 已有前端和服务端能力，改造成本低，直接防止历史节点、发布产物和导出包落后于协同草稿 |
| P0 | 清理旧“明确保存才改变原始项目文件”表述 | 文档语义和产品心智冲突会继续误导实现 |
| P1 | 会改变正式状态的 Web API 继续收敛到 `project-core` | 先收敛保存、发布、恢复、模板、导出等高风险入口，ROI 高于全量迁移 |
| P1 | 正式 workspace 替换抽取 staging + atomic replace helper | 比完整 journal 更小、更快降低删除后复制失败风险 |
| P1 | CLI submit 与 Web 协同编辑冲突策略 | `baseVersion` 已覆盖项目版本冲突，但协同草稿未命名版本时仍需定义处理 |
| P2 | 协同层抽象多实例边界 | 为未来扩展预留空间，不阻塞当前单实例部署 |
| P2 | 模板 lineage、归档语义与项目包协议兼容矩阵 | 提升审计和演进能力，当前不是最大风险 |
| P3 | 快照内容寻址 | 存储和复制成本扩大后收益明显，当前复杂度高于短期收益 |

## 目标模式执行手册

本节用于支持后续目标模式长期稳定运行。后续 agent 不应把本文当成一次性评估结论，而应把它当作执行队列和接力协议：每次只推进一个可验收切片，完成后同步更新本文进度、相关 `docs/项目文档/` 文档和验证结果。

### 执行原则

- 先定语义，再改实现。Workspace 是项目共享草稿还是用户私有草稿，是所有协作、保存、发布和冲突策略的前置决策。
- 先收紧关键入口，再清理长尾入口。优先处理命名版本、自动检查点、发布、恢复、导出、CLI submit 这类会生成历史节点或影响正式状态的入口。
- 先做小而可逆的可靠性改进，再做完整存储架构。atomic replace helper 的 ROI 高于一开始引入完整 journal 或内容寻址存储。
- `project-core` 保持业务规则中心，但不要让它直接依赖 Web 或 agent-service 的 HTTP 细节。跨服务能力通过接口或 author-site 适配层进入。
- 文档与代码同步推进。只要实施改变产品行为、接口契约、状态流转或模块职责，就必须更新 `docs/项目文档/` 对应需求/技术文档和模块索引。
- 每个切片都要有明确验证命令。文档-only 切片说明不跑测试；代码切片按影响面选择 `check:*` 脚本。

### 分阶段路线

| 阶段 | 目标 | 产出 | 建议验证 |
|---|---|---|---|
| Phase 0 | 固化语义和执行边界 | 确认 Workspace 共享/私有决策；清理旧保存心智；补充项目文档 | 文档复查 |
| Phase 1 | 关键动作强制 flush | 统一 flush coordinator；命名版本、自动检查点、发布、导出前后端都走同一协议 | `pnpm check:author`、`pnpm check:agent` |
| Phase 2 | 协作身份模型 | 根据 Phase 0 决策调整 workspace owner 或项目权限校验；补齐协作者 presence 语义 | `pnpm check:author`、`pnpm check:agent`、专项 E2E |
| Phase 3 | 高风险入口收敛 | Web 保存、发布、恢复、模板转换、导出等入口逐步复用 `project-core` | `pnpm check:author`、`pnpm check:project-core` |
| Phase 4 | 文件替换可靠性 | 抽取 staging + atomic replace helper，替换正式 workspace 覆盖路径 | `pnpm check:project-core`、`pnpm check:author` |
| Phase 5 | 协议与模板治理 | 项目包兼容矩阵、fixture/golden tests、模板 lineage 和归档策略 | `pnpm check:project-scaffold`、`pnpm check:project-cli` |
| Phase 6 | 长期存储演进 | 多实例协同抽象、Yjs update 持久化、内容寻址快照方案 | 方案评审后再定 |

### 优化项执行卡片

#### A. Workspace 当前态与跨用户协作语义

- 当前证据：需求文档描述多人编辑同一个 Workspace；协同持久化实现会校验 workspace owner 与 session user 一致。
- 决策问题：项目是否需要真正跨用户共享同一个编辑 Workspace。
- 主要文档：[实时协作保存需求](../../项目文档/创作端/03-项目管理/实时协作保存_需求文档.md)、[实时保存与协同编辑](../../项目文档/创作端/03-项目管理/技术/11_实时保存与协同编辑.md)、[工作空间与对话解耦](../../项目文档/创作端/03-项目管理/技术/07_工作空间对话解耦.md)。
- 主要代码：`packages/author-site/src/lib/workspace-manager.ts`、`packages/author-site/src/lib/session-manager.ts`、`packages/agent-service/src/collab/workspace-file-persistence.ts`。
- 推荐切片：先写决策记录和验收用例；再调整 workspace 元数据或协同校验；最后更新 UI 文案和协作者展示。
- 验收标准：两个不同用户打开同一项目时，行为与文档一致；若是共享草稿，彼此能进入同一资源协作；若是私有草稿，UI 和需求不再承诺跨用户共同编辑。
- 停止条件：权限模型、用户角色或项目成员关系无法确定时，不改代码，只保留决策待确认。

#### B. 后端关键动作统一强制 flush

- 当前证据：前端版本控制 hook 和编辑页已有 `flush-all`，agent-service 已有 flush 接口；创作端专用导出入口已移除，后续重点收敛发布、保存、checkpoint、命名版本和恢复路径。
- 主要文档：[版本管理](../../项目文档/创作端/03-项目管理/技术/04_版本管理.md)、[实时保存与协同编辑](../../项目文档/创作端/03-项目管理/技术/11_实时保存与协同编辑.md)、[本地项目包与脚手架同步](../../项目文档/创作端/10-CLI/技术/02_本地项目包与脚手架同步.md)。
- 主要代码：`packages/author-site/src/app/demo/[id]/edit/hooks/useVersionControl.ts`、`packages/author-site/src/app/api/projects/[projectId]/publish/route.ts`、`packages/author-site/src/app/api/sessions/[sessionId]/checkpoint/route.ts`、`packages/agent-service/src/routes/collab.ts`。
- 推荐切片：先抽 author-site 内部 flush helper；再让发布、checkpoint、命名版本和恢复调用同一 helper；最后把错误结构和前端提示统一。
- 验收标准：所有生成快照或发布产物的入口，在有活跃协同房间时都会先落盘；flush 失败时阻断关键动作并返回明确错误。
- 停止条件：agent-service 不可达时的降级策略未定，不应静默继续生成快照。

#### C. 旧保存语义清理

- 当前证据：会话管理需求仍保留“原始项目文件直到明确保存才变”；实时协作需求已经把自动保存定义为当前态防丢入口。
- 主要文档：[会话管理需求](../../项目文档/创作端/03-项目管理/会话管理_需求文档.md)、[实时协作保存需求](../../项目文档/创作端/03-项目管理/实时协作保存_需求文档.md)、[版本管理](../../项目文档/创作端/03-项目管理/技术/04_版本管理.md)。
- 推荐切片：先只改文档语义，把“保存”拆为“自动同步当前态”“命名版本”“发布快照”；再检查 UI 文案和 API 命名是否需要兼容层。
- 验收标准：用户离开页面、命名版本、发布、恢复、自动检查点的语义在需求和技术文档中一致。
- 停止条件：产品仍要求“显式保存才影响项目草稿”时，需要重新评估实时协作模型。

#### D. 高风险 Web API 收敛到 `project-core`

- 当前证据：项目/模板/脚手架部分 API 已走 `ProjectAdminService`，但 session 保存、checkpoint、恢复、发布等仍大量依赖 author-site `fs-utils` 和 `session-manager`。
- 主要文档：[CLI 能力层实现设计](../../项目文档/创作端/10-CLI/技术/01_CLI能力层实现设计.md)、[项目模板库](../../项目文档/创作端/03-项目管理/技术/08_项目模板库.md)、[本地项目包与脚手架同步](../../项目文档/创作端/10-CLI/技术/02_本地项目包与脚手架同步.md)。
- 主要代码：`packages/project-core/src/service.ts`、`packages/author-site/src/lib/project-admin-service.ts`、`packages/author-site/src/lib/fs-utils.ts`、`packages/author-site/src/lib/session-manager.ts`。
- 推荐切片：按入口迁移，不做横向大重构。优先级为发布前同步、版本恢复、命名版本、模板转换、Web scaffold 导出。
- 验收标准：同一业务动作在 Web、CLI、本地项目包路径下走同一领域规则，错误码和审计记录一致。
- 停止条件：遇到 UI 专属交互语义时，保留在 author-site API 层，不下沉到 `project-core`。

#### E. 正式 workspace 替换可靠性

- 当前证据：`project-core` 的 `commitEdit` 删除正式 workspace 后复制事务 workspace；author-site 的 `replaceDirectoryWithTemp` 已有重试和 rename 思路，但实现分散。
- 主要代码：`packages/project-core/src/service.ts`、`packages/author-site/src/lib/session-manager.ts`、`packages/author-site/src/lib/workspace-manager.ts`。
- 推荐切片：先在 `project-core` 内抽 helper；覆盖 commit、restore、publish snapshot 相关正式目录替换；再评估是否回收 author-site 的类似逻辑。
- 验收标准：替换失败时原 workspace 不丢失；临时目录可清理；测试覆盖复制失败、目标存在和元数据文件过滤。
- 停止条件：不要在这一阶段引入数据库、对象存储或内容寻址，避免扩大改动面。

#### F. CLI submit 与 Web 协同编辑冲突策略

- 当前证据：本地项目包 submit 已检查远端 `baseVersion`，但 Web 协同草稿若尚未形成新版本，CLI 可能无法感知未命名草稿。
- 主要文档：[本地项目包与脚手架同步](../../项目文档/创作端/10-CLI/技术/02_本地项目包与脚手架同步.md)、[实时保存与协同编辑](../../项目文档/创作端/03-项目管理/技术/11_实时保存与协同编辑.md)。
- 主要代码：`packages/project-scaffold/src/index.ts`、`packages/project-core/src/service.ts`、`packages/author-site/src/lib/session-manager.ts`。
- 推荐切片：先定义冲突检测口径，例如 workspace updatedAt、dirty room、active session 或草稿 revision；再让 CLI submit 在冲突时给出可执行 nextActions。
- 验收标准：CLI submit 不会静默覆盖 Web 未落版本的协同草稿；用户能选择重新拉取、强制提交或先命名 Web 草稿。
- 停止条件：没有统一草稿 revision 时，不要假装已解决冲突，只在文档中标注剩余风险。

#### G. 多实例协同边界

- 当前证据：协同房间、presence、dirty 状态都在 agent-service 进程内；文档已说明当前先面向单实例部署。
- 主要代码：`packages/agent-service/src/collab/collab-room-manager.ts`、`packages/agent-service/src/collab/workspace-file-persistence.ts`。
- 推荐切片：先抽接口和生命周期边界，不急于引入共享存储。候选接口包括 room registry、update store、presence store、flush queue。
- 验收标准：单实例行为不变；新增抽象不让业务逻辑直接依赖内存 Map；未来可以替换实现。
- 停止条件：没有多实例部署计划或压测数据时，不做分布式存储落地。

#### H. 模板 lineage 与归档

- 当前证据：模板已有 `sourceProjectId`，但没有来源版本和归档状态；模板转普通项目会物理删除模板。
- 主要文档：[项目模板库](../../项目文档/创作端/03-项目管理/技术/08_项目模板库.md)、[项目模板需求](../../项目文档/创作端/03-项目管理/项目模板_需求文档.md)。
- 主要代码：`packages/project-core/src/service.ts`、`packages/shared/src/index.ts`、`packages/author-site/src/components/demo/home-page.tsx`。
- 推荐切片：先补类型和读取兼容；再把删除/转换改为归档可选；最后调整推荐和列表过滤。
- 验收标准：历史模板可追溯来源版本；归档模板不参与默认推荐和列表，但审计可查。
- 停止条件：如果产品仍要求模板转项目后完全移除，不引入归档 UI，只记录 lineage。

#### I. 项目包协议与快照内容寻址

- 当前证据：项目包协议已支撑 Web 下载、CLI pull/submit、模板本地化；快照仍为全量复制。
- 主要文档：[本地项目包与脚手架同步](../../项目文档/创作端/10-CLI/技术/02_本地项目包与脚手架同步.md)、[版本管理](../../项目文档/创作端/03-项目管理/技术/04_版本管理.md)。
- 主要代码：`packages/project-scaffold/src/index.ts`、`packages/project-scaffold/src/scaffold.test.ts`、`packages/project-cli/src/index.ts`、`packages/project-core/src/service.ts`。
- 推荐切片：短期只补 schema version 迁移策略、兼容矩阵和 golden tests；内容寻址等项目规模或存储压力出现后再做。
- 验收标准：协议升级不会污染业务 diff；旧本地包能给出明确 upgrade/validate 结果；快照存储演进有兼容方案。
- 停止条件：不要为了理论存储优化提前重写快照层。
