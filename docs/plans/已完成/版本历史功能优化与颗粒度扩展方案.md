# 版本历史功能优化与颗粒度扩展方案

> 版本：v1.2
> 创建日期：2026-06-05
> 校正日期：2026-06-21
> 状态：实施中
> 类型：功能优化与扩展

---

## 一、准确性校正

本方案经现有实现核对后，需做如下校正：

1. `VersionInfo.savedBy` 在共享类型中本就定义为“保存者用户名”，不是缺少字段；列表出现 UUID 的根因应优先检查保存链路是否传入了登录用户名，并在展示层对 UUID 做友好兜底。
2. 当前编辑页实际使用 author-site 本地 Session 保存链路：`/api/sessions/{sessionId}/save`，页面级版本也应优先落在 author-site 本地文件系统与当前 session workspace 上，避免绕到无法感知当前临时编辑状态的 agent-service 项目接口。
3. 项目级版本恢复在不同实现中行为不完全一致：agent-service 恢复会新增恢复后的项目版本；author-site 文件工具中还存在“恢复前自动备份 + 恢复后版本”的双记录逻辑。文案不能笼统写成只有一条记录。
4. 页面级版本若与项目级版本共享 `v1, v2, ...` 序列，版本号生成不能再只使用 `project.versions.length + 1`，必须扫描项目级与页面级历史中的最大版本号。
5. “历史版本运行态预览”涉及编译缓存、隔离 workspace、多页面导航和只读运行环境，风险高于列表与恢复能力。本轮先落地只读代码/Schema 预览，完整渲染预览作为后续增强。

---

## 二、目标

- 历史列表按时间倒序展示一条统一时间线，不让用户切换“项目版本 / 页面版本”。
- UI 不展示版本号、文件数量，也不提供页面/项目筛选。
- 页面历史标题保持简单，例如“修改了首页”；页面历史不显示操作者。
- 项目历史标题保持简单，例如“保存项目”；项目历史显示保存者。
- 保存项目时自动为当前修改页面创建页面历史，再创建项目历史。
- 页面历史可查看页面历史内容，并可仅恢复该页面。
- 页面级恢复只替换目标页面文件，同时新增一条项目级版本记录，用于保留全局时间线。
- 项目级与页面级版本底层仍共用递增版本号序列，但 UI 不暴露版本号。

## 三、业务规则

| 规则 | 说明 |
|:-----|:-----|
| 保存者显示 | 优先显示 `savedBy`；为空或 UUID 形式时显示“未知用户” |
| 历史呈现 | 按时间倒序合并项目历史与页面历史，形成统一时间线 |
| 版本号与文件数量 | UI 不展示 `versionId` 和 `fileCount`，字段继续保留用于内部逻辑 |
| 项目级版本 | 用户点击保存时仍生成完整项目快照 |
| 页面级版本 | 用户点击保存时自动为当前页面创建页面快照；快照来源优先取当前 session workspace |
| 页面记录标题 | 页面变更显示为“修改了{页面名}”，页面恢复显示为“恢复了{页面名}” |
| 项目记录标题 | 项目保存显示为“保存项目”，项目恢复显示为“恢复项目” |
| 操作者显示 | 页面记录不显示操作者，项目记录显示保存者 |
| 页面快照内容 | 仅保存该页面的 `index.tsx` 与 `config.schema.json` |
| 页面级恢复 | 只写回目标页面文件；随后创建完整项目级版本记录 |
| 版本号 | 项目级与页面级版本扫描全局最大 `vN` 后生成下一号，但仅作为内部标识 |
| 预览能力 | 本轮为只读代码/Schema 预览；运行态渲染预览后续扩展 |

## 四、实施范围

### 4.1 本轮实施

- 扩展共享类型：
  - 页面级版本信息。
  - 页面版本历史响应。
  - 页面版本恢复响应。
- 扩展 author-site 文件工具：
  - 页面快照路径。
  - 页面版本历史读取。
  - 页面快照创建。
  - 页面版本文件读取。
  - 页面版本恢复。
- 扩展 API：
  - 查询页面版本历史。
  - 创建页面版本。
  - 读取页面历史版本文件。
  - 恢复页面历史版本。
- 扩展编辑页版本 Tab：
  - 统一历史时间线。
  - 保存时自动创建当前页面历史。
  - 简化历史标题，不展示版本号。
  - 页面历史查看与仅恢复此页面。
  - 项目历史恢复整个项目。
  - 保存者 UUID 兜底显示。

### 4.2 后续增强

- 项目级历史版本运行态预览。
- 页面级历史版本运行态预览。
- 预览窗口中的多页面导航。
- 预览编译缓存复用与隔离 workspace 清理策略。
- 按实际变更页面批量创建页面历史；当前实现先记录保存时的当前页面。
