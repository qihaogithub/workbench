## 二、P0 任务

### 2.1 `/api/generate-schema` 接口

#### 2.1.1 目的

AI 修改组件 props 后，配置面板需要知道新字段，否则配置面板与组件脱节。

#### 2.1.2 方案设计

**接口定义**：
```typescript
// POST /api/generate-schema
// 输入：从草稿工作区读取 index.tsx
{
  sessionId: string
}

// 输出
{
  success: true,
  data: {
    schema: object,           // 生成的 JSON Schema
    updatedAt: string         // 更新时间
  }
}
```

**生成策略**：
1. 解析 `index.tsx` 中的 `Props` 接口或组件函数参数类型
2. 基础类型映射：
   - `string` → `{ type: 'string' }`
   - `number` → `{ type: 'number' }`
   - `boolean` → `{ type: 'boolean' }`
   - 联合类型（如 `'a' | 'b'`）→ `{ type: 'string', enum: ['a', 'b'] }`
3. 保留现有 Schema 中已配置的 `ui:widget`、`default` 等扩展字段（不覆盖）
4. 新增字段使用类型默认值作为 `default`

**与配置面板联动**：
- Schema 更新后，配置面板自动重新生成表单
- 新增字段使用默认值填充，已修改的配置项保持不变
- 删除字段时，对应配置数据自动清理

#### 2.1.3 触发时机

1. AI 完成一次代码修改（`UPDATE_CODE` 成功返回后）
2. 用户手动保存代码时

#### 2.1.4 实现位置

- 新建 `src/app/api/generate-schema/route.ts`

## 三、P1 任务

### 3.1 CDN 资源缓存策略优化

#### 3.1.1 当前状态

- 依赖浏览器自动缓存 esm.sh 资源
- 无服务端缓存层

#### 3.1.2 优化目标

- 服务端对编译结果做缓存（代码 hash → 编译结果）
- 减少重复编译开销

#### 3.1.3 方案设计

```typescript
// 缓存 key 生成
const cacheKey = hash(compileInput); // code + sessionId hash

// 缓存结构
interface CompileCache {
  compiledCode: string;
  dependencies: string[];
  cssImports: string[];
  createdAt: number;
}

// 缓存过期策略
// - 永久缓存（依赖版本锁定后）
// - 或 TTL 24h（未锁定版本，定期失效）
```

### 3.2 iframe 复用优化

#### 3.2.1 当前状态

- 页面切换时创建新的 iframe
- CDN 依赖需要重新加载

#### 3.2.2 优化目标

- 页面切换时复用已有 iframe
- 保持 CDN 依赖缓存有效

#### 3.2.3 方案设计

```typescript
// PreviewPanel 内部维护 iframe ref
// 切换 session 时：
// 1. 复用同一 iframe
// 2. 通过 postMessage 发送新代码
// 3. 依赖已缓存，只需重新执行组件代码
```

**注意**：重新注入组件代码会导致 React 状态完全重置（`useState`、`useRef` 等归零）。这是 iframe 完全隔离的固有代价。

### 3.3 依赖版本锁定持久化

#### 3.3.1 当前状态

- 使用 esm.sh 最新版
- 每次编译可能使用不同版本

#### 3.3.2 优化目标

- 首次编译时锁定依赖版本
- 版本信息持久化到 demo 数据中
- 后续编译复用锁定版本

#### 3.3.3 方案设计

```typescript
// demo 数据结构扩展
interface DemoData {
  id: string;
  // ... 其他字段
  lockedDependencies: Record<string, string>; // package -> version
}

// 锁定逻辑
// 首次编译时：
// 1. 从 esm.sh 获取当前版本（如 date-fns@3.0.0）
// 2. 保存到 demo.lockedDependencies
// 3. 后续编译使用锁定版本
```

### 3.4 Demo 列表页缩略图生成

#### 3.4.1 问题背景

Demo 列表页以卡片形式展示所有 Demo，如果每个卡片都渲染实时 iframe，10 个 Demo 同时加载 → 10 个 iframe × React + Tailwind CDN = 内存和 CPU 灾难。

#### 3.4.2 方案设计

**预览策略**：
| 页面 | 预览方式 | 原因 |
|:-----|:---------|:-----|
| **Demo 列表页** | 静态缩略图（保存时自动生成） | 卡片多、无需交互、性能优先 |
| **Demo 使用页面** | 实时 iframe（`PreviewPanel` + 配置面板） | 需要交互、配置联动 |
| **AI 编辑工作台** | 实时 iframe（`PreviewPanel`） | 需要实时预览代码变更 |
| **跨系统嵌入** | 实时 iframe（独立 URL） | 外部系统需要实时渲染 |

**缩略图生成时机**：
1. 用户点击"保存"时，服务端触发截图（使用 `playwright` 或 `html-to-image`）
2. 截图结果保存为静态图片（如 `/public/thumbnails/{demoId}.png`）
3. 列表页直接加载图片，无 iframe 开销

**fallback**：如果缩略图不存在（新创建未保存），显示占位图或"暂无预览"。

#### 3.4.3 实现位置

- 修改 `src/app/api/save/route.ts`，保存后触发截图
- 新建 `src/app/api/thumbnail/route.ts` 处理截图生成
- 修改列表页组件，使用 `<img>` 而非 iframe

## 四、P2 任务

### 4.1 内网环境支持

#### 4.1.1 问题背景

如果嵌入的系统在内网、无法访问外网 CDN，动态 CDN 方案会遇到限制。

#### 4.1.2 应对策略（按优先级）

**1. 内网搭建轻量级 npm 代理（推荐）**：
- 使用 verdaccio 或 nexus 搭建内网 npm registry
- 或搭建简单的 CDN 代理，缓存外网资源
- 将 esm.sh URL 替换为内网代理地址

**2. 预置常用依赖到本地静态资源**：
- 提前下载 React、Tailwind、lucide-react 等高频依赖
- iframe 通过相对路径加载本地资源
- 业务依赖仍走 CDN（如果允许），或预置常见库

**3. 混合方案**：
- 公共依赖（React、Tailwind）→ 本地预置
- 业务依赖（date-fns、framer-motion）→ CDN 或按需预置

#### 4.1.3 实现方式

```typescript
// compiler.ts 扩展
interface CompileOptions {
  useInternalCdn?: boolean;  // 是否使用内网 CDN
  internalCdnBase?: string;  // 内网 CDN 地址
}

// CDN URL 映射
function getCdnUrl(packageName: string, options: CompileOptions): string {
  if (options.useInternalCdn && options.internalCdnBase) {
    return `${options.internalCdnBase}/${packageName}`;
  }
  return `https://esm.sh/${packageName}`;
}
```
