# 页面列表拖拽排序与文件夹层级优化方案

> 版本：v1.0
> 创建日期：2026-05-05
> 状态：方案设计中

---

## 一、背景与目标

### 1.1 现状问题

当前编辑页左侧「页面」Tab 的页面列表存在以下局限：

- **不支持拖拽排序**：页面顺序仅通过 `order` 字段管理，用户无法直观地调整页面排列
- **不支持文件夹分组**：所有页面平铺展示，项目页面多时难以管理
- **不支持层级组织**：无法按业务模块（如"首页模块"、"产品模块"）归类页面

### 1.2 目标

1. **拖拽排序**：页面列表支持拖拽调整顺序，操作直观高效
2. **文件夹分组**：支持创建文件夹整理页面，文件夹可嵌套
3. **跨层级移动**：支持将页面拖入/拖出文件夹，支持文件夹嵌套

### 1.3 设计原则

| 原则 | 说明 |
|:-----|:-----|
| 虚拟文件夹 | 文件夹仅存在于元数据层，物理 `demos/` 目录保持扁平结构，不引入物理子目录嵌套 |
| 复用现有技术 | 沿用项目已有的 `@dnd-kit/core` + `@dnd-kit/sortable`，参考 `OrderControl` 的拖拽模式 |
| 向后兼容 | `DemoPageMeta` 扩展而非替换，现有数据自动升级 |
| Agent 友好 | 文件夹是 UI 组织概念，AI 操作页面时仍使用 `demos/{demoId}/` 路径，无需感知文件夹 |

## 二、数据模型变更

### 2.1 核心设计：虚拟文件夹

**为什么选择虚拟文件夹而非物理目录嵌套？**

| 方案 | 优点 | 缺点 |
|:-----|:-----|:-----|
| **虚拟文件夹（采用）** | 物理目录不变，不破坏现有路径解析逻辑；AI 操作路径不变；保存/恢复/版本逻辑零改动 | 文件夹信息仅存于元数据，需额外持久化 |
| 物理目录嵌套 | 目录即真相 | 路径解析全部重写；AI prompt 需感知文件夹层级；保存/恢复流程复杂化；跨文件夹移动需物理搬迁文件 |

### 2.2 新增类型：DemoFolderMeta

```typescript
interface DemoFolderMeta {
  id: string            // "folder_{timestamp}_{random6}"
  name: string          // 文件夹显示名称
  parentId: string | null  // 父文件夹 ID，null 表示根级
  order: number         // 同级内的排序（小者在前）
  createdAt: number
  updatedAt: number
}
```

### 2.3 扩展类型：DemoPageMeta

在现有 `DemoPageMeta` 基础上新增 `parentId` 字段：

```typescript
interface DemoPageMeta {
  id: string
  name: string
  order: number
  parentId: string | null  // 【新增】所属文件夹 ID，null 表示根级
  createdAt: number
  updatedAt: number
}
```

### 2.4 统一树节点类型

```typescript
type DemoPageItem = DemoFolderMeta | DemoPageMeta

// 类型守卫
function isDemoFolder(item: DemoPageItem): item is DemoFolderMeta {
  return item.id.startsWith("folder_")
}
```

### 2.5 更新类型：Project

```typescript
interface Project {
  id: string
  name: string
  description?: string
  workspacePath: string
  demoPages: DemoPageMeta[]       // 页面列表（含 parentId）
  demoFolders: DemoFolderMeta[]   // 【新增】文件夹列表
  versions: VersionInfo[]
  createdAt: number
  updatedAt: number
  lockedDependencies?: Record<string, string>
  thumbnail?: string
}
```

### 2.7 数据迁移

项目已上线数据需自动迁移（`parentId` 字段缺失时视为 `null`）：

```typescript
function migrateDemoPages(pages: any[]): DemoPageMeta[] {
  return pages.map(p => ({
    ...p,
    parentId: p.parentId ?? null,

}
```

`readProjectMeta` 读取时自动补全 `parentId: null`，无需一次性迁移脚本。

## 三、文件系统持久化

### 3.1 文件夹元数据存储

文件夹元数据存储在项目级配置文件中，**不创建物理目录**：

| 存储位置 | 内容 |
|:---------|:-----|
| `project.json` 的 `demoFolders` 字段 | 所有文件夹定义 |
| `project.json` 的 `demoPages` 字段 | 页面定义（含 `parentId`） |
| `demos/{demoId}/.demo.json` | 页面元数据（新增 `parentId` 字段） |

### 3.2 .demo.json 扩展

```json
{
  "id": "demo_1714824000000_a1b2c3",
  "name": "首页",
  "parentId": "folder_001",
  "order": 0,
  "createdAt": 1714824000000,
  "updatedAt": 1714824100000
}
```

### 3.3 文件夹元数据独立文件

文件夹信息存储在 `workspace/.folders.json`：

```json
{
  "folders": [
    {
      "id": "folder_001",
      "name": "首页模块",
      "parentId": null,
      "order": 0,
      "createdAt": 1714824000000,
      "updatedAt": 1714824100000
    }
  ]
}
```

**为什么用独立文件而非仅存 project.json？**

与 `.demo.json` 的设计理念一致——元数据靠近数据源，保存时后端扫描 `demos/` 目录 + 读取 `.demo.json` + 读取 `.folders.json` 合并得到完整树结构。`project.json` 中的 `demoFolders` 是合并后的缓存/镜像。

### 3.4 保存时同步流程

```
用户点击保存
      │
      ▼
后端扫描 workspace/demos/ 目录 → 读取各 .demo.json → 读取 .folders.json
      │
      ▼
合并生成 project.json 的 demoPages + demoFolders
      │
      ▼
写入 project.json
```

## 四、拖拽排序设计

### 4.1 技术选型

沿用项目已有的 `@dnd-kit/core` + `@dnd-kit/sortable`，与 `OrderControl` 组件保持一致。

### 4.2 渲染策略：扁平化列表 + 缩进

树形结构渲染为扁平列表，通过缩进表示层级关系：

```
[拖拽手柄] 📁 首页模块                    [▼] [⋯]
[拖拽手柄]     📄 首页                     [⋯]
[拖拽手柄]     📄 引导页                   [⋯]
[拖拽手柄] 📁 产品模块                    [▼] [⋯]
[拖拽手柄]     📄 产品列表                 [⋯]
[拖拽手柄]     📁 详情页组                 [▼] [⋯]
[拖拽手柄]         📄 产品详情             [⋯]
[拖拽手柄]         📄 规格参数             [⋯]
[拖拽手柄]     📄 对比页                   [⋯]
[拖拽手柄] 📄 关于我们                    [⋯]
[拖拽手柄] 📄 联系我们                    [⋯]
```

**扁平化的优势**：
- 与 `SortableContext` 的 `verticalListSortingStrategy` 天然兼容
- 拖拽计算简单（一维数组索引映射）
- 避免嵌套 `DndContext` 的复杂性

### 4.3 拖拽交互规则

| 操作 | 行为 | 视觉反馈 |
|:-----|:-----|:---------|
| **同级排序** | 拖拽页面/文件夹到同级另一位置 | 蓝色插入线 |
| **拖入文件夹** | 拖拽页面到文件夹项上停留 | 文件夹高亮 + 展开动画 |
| **拖出文件夹** | 拖拽页面到文件夹外区域 | 目标位置显示插入线 |
| **文件夹嵌套** | 拖拽文件夹到另一文件夹上 | 父文件夹高亮 |
| **拖入已折叠文件夹** | 停留 500ms 后自动展开 | 文件夹展开动画 |

### 4.4 拖拽判定逻辑

```
拖拽结束（onDragEnd）
      │
      ├─ active 和 over 是同一项？→ 忽略
      │
      ├─ over 是文件夹？→ 将 active 移入该文件夹（parentId = over.id，order 追加到末尾）
      │
      └─ over 是页面或空白区域？→ 将 active 插入到 over 旁边
              │
              ├─ 计算目标 parentId（取 over 的 parentId）
              │
              └─ 计算 target order（over 的 order ± 0.5，然后重排）
```

### 4.5 循环嵌套防护

将文件夹拖入自身或自身子文件夹会导致循环引用。防护措施：

```typescript
function isDescendant(folderId: string, targetParentId: string, folders: DemoFolderMeta[]): boolean {
  let current = folders.find(f => f.id === targetParentId)
  while (current) {
    if (current.id === folderId) return true
    current = folders.find(f => f.id === current!.parentId)
  }
  return false
}

// 拖拽文件夹到另一文件夹前校验
if (isDemoFolder(activeItem) && isDescendant(activeItem.id, overFolder.id, folders)) {
  return // 禁止操作
}
```

### 4.6 排序持久化

拖拽完成后，重新计算受影响层级的 `order` 值：

```typescript
function recalculateOrder(items: DemoPageItem[], targetParentId: string | null): void {
  const siblings = items
    .filter(i => i.parentId === targetParentId)
    .sort((a, b) => a.order - b.order)
  siblings.forEach((item, index) => {
    item.order = index
  })
}
```

批量调用 `PATCH /api/projects/{id}/demos/{demoId}` 或新增批量排序 API 更新。

## 五、文件夹管理

### 5.1 操作清单

| 操作 | 触发方式 | 行为 |
|:-----|:---------|:-----|
| **新建文件夹** | 页面列表顶部「+ 新建文件夹」按钮 | 弹出名称输入框 → 创建根级文件夹 |
| **新建子文件夹** | 文件夹右键菜单「新建子文件夹」 | 弹出名称输入框 → 创建嵌套文件夹 |
| **重命名文件夹** | 文件夹右键菜单「重命名」 | 行内编辑名称 |
| **删除文件夹** | 文件夹右键菜单「删除」 | 二次确认：选择「仅删除文件夹（页面移至上级）」或「删除文件夹及所有页面」 |
| **折叠/展开** | 点击文件夹行 | 切换展开状态 |
| **移动页面到文件夹** | 拖拽 或 右键菜单「移动到」 | 更新页面 parentId |

### 5.2 删除文件夹的两种模式

```
删除文件夹「产品模块」
      │
      ├─→ 模式一：仅删除文件夹
      │     产品模块内的页面 → 移至根级
      │     子文件夹 → 移至根级
      │
      └─→ 模式二：删除文件夹及内容
            产品模块内所有页面 → 删除（需二次确认）
            子文件夹及内含页面 → 递归删除
```

**默认推荐模式一**（安全），模式二需额外确认弹窗。

### 5.3 文件夹展开状态

展开状态（`isExpanded`）为纯 UI 状态，不持久化到元数据：

```typescript
// 组件内状态
const [expandedFolders, setExpandedFolders] = useState<Set<string>>(new Set())

const toggleFolder = (folderId: string) => {
  setExpandedFolders(prev => {
    const next = new Set(prev)
    next.has(folderId) ? next.delete(folderId) : next.add(folderId)
    return next
  })
}
```

### 5.4 文件夹嵌套深度限制

| 限制 | 值 | 原因 |
|:-----|:---|:-----|
| 最大嵌套深度 | **3 层** | 过深嵌套降低可用性；拖拽交互复杂度随深度指数增长 |
| 单文件夹内最大项数 | **50** | 防止列表过长影响性能 |

创建子文件夹时校验深度：

```typescript
function getFolderDepth(folderId: string, folders: DemoFolderMeta[]): number {
  let depth = 0
  let current = folders.find(f => f.id === folderId)
  while (current?.parentId) {
    depth++
    current = folders.find(f => f.id === current!.parentId)
  }
  return depth
}

// 创建子文件夹前校验
if (getFolderDepth(parentFolderId, folders) >= 3) {
  toast.error("文件夹嵌套不能超过 3 层")
  return
}
```

## 七、API 变更

### 7.1 新增路由：文件夹管理

| 方法 | 路由 | 说明 |
|:-----|:-----|:-----|
| `POST` | `/api/projects/{projectId}/folders` | 创建文件夹（需 `sessionId`） |
| `PATCH` | `/api/projects/{projectId}/folders/{folderId}` | 更新文件夹元数据（name / parentId / order） |
| `DELETE` | `/api/projects/{projectId}/folders/{folderId}` | 删除文件夹（query: `?deleteContents=true/false`） |

### 7.2 修改路由：页面管理

| 路由 | 变更 |
|:-----|:-----|
| `PATCH /api/projects/{projectId}/demos/{demoId}` | `PatchDemoPageMetaRequest` 新增 `parentId` 字段 |
| `POST /api/projects/{projectId}/demos` | `CreateDemoPageRequest` 新增可选 `parentId` 字段 |

### 7.3 新增路由：批量排序

| 方法 | 路由 | 说明 |
|:-----|:-----|:-----|
| `PATCH` | `/api/projects/{projectId}/demo-pages/reorder` | 批量更新页面和文件夹的 order / parentId |

**请求体**：

```typescript
interface ReorderDemoPagesRequest {
  pages: Array<{
    id: string
    order: number
    parentId: string | null
  }>
  folders?: Array<{
    id: string
    order: number
    parentId: string | null
  }>
}
```

> **为什么需要批量排序 API？** 拖拽排序后可能同时影响多个项的 order 和 parentId，逐条 PATCH 会导致多次请求和中间状态不一致。批量接口保证原子性。

### 7.4 请求类型更新

```typescript
interface CreateDemoPageRequest {
  name: string
  parentId?: string | null   // 【新增】创建时指定所属文件夹
}

interface PatchDemoPageMetaRequest {
  name?: string
  order?: number
  parentId?: string | null   // 【新增】移动到其他文件夹
}

interface CreateDemoFolderRequest {
  name: string
  parentId?: string | null   // 父文件夹 ID，null 为根级
}

interface PatchDemoFolderRequest {
  name?: string
  parentId?: string | null
  order?: number
}
```

### 7.5 后端校验

| 校验项 | 时机 | 错误码 |
|:-------|:-----|:-------|
| `parentId` 指向的文件夹必须存在 | 创建/移动页面或文件夹时 | `FOLDER_NOT_FOUND` |
| 文件夹嵌套深度 ≤ 3 | 创建子文件夹/移动文件夹时 | `FOLDER_DEPTH_EXCEEDED` |
| 文件夹不能成为自身的子级 | 移动文件夹时 | `CIRCULAR_REFERENCE` |
| 文件夹 ID 格式为 `folder_{ts}_{random}` | 创建文件夹时 | — |

## 八、前端实现方案

### 8.2 树扁平化工具函数

```typescript
interface FlatTreeItem {
  item: DemoPageItem
  depth: number
  isExpanded: boolean
  hasChildren: boolean
}

function flattenTree(
  pages: DemoPageMeta[],
  folders: DemoFolderMeta[],
  expandedFolders: Set<string>,
): FlatTreeItem[] {
  const result: FlatTreeItem[] = []

  function walk(parentId: string | null, depth: number) {
    const childFolders = folders
      .filter(f => f.parentId === parentId)
      .sort((a, b) => a.order - b.order)
    const childPages = pages
      .filter(p => p.parentId === parentId)
      .sort((a, b) => a.order - b.order)

    for (const folder of childFolders) {
      const isExpanded = expandedFolders.has(folder.id)
      const hasChildren = folders.some(f => f.parentId === folder.id)
        || pages.some(p => p.parentId === folder.id)
      result.push({ item: folder, depth, isExpanded, hasChildren })
      if (isExpanded) {
        walk(folder.id, depth + 1)
      }
    }

    for (const page of childPages) {
      result.push({ item: page, depth, isExpanded: false, hasChildren: false })
    }
  }

  walk(null, 0)
  return result
}
```

### 8.3 DndContext 配置

```typescript
function DemoPageTree({ pages, folders, ... }) {
  const [expandedFolders, setExpandedFolders] = useState<Set<string>>(new Set())
  const flatItems = useMemo(
    () => flattenTree(pages, folders, expandedFolders),
    [pages, folders, expandedFolders]
  )

  const sensors = useSensors(
    useSensor(PointerSensor, { activationConstraint: { distance: 5 } }),
    useSensor(KeyboardSensor, { coordinateGetter: sortableKeyboardCoordinates }),
  )

  const handleDragEnd = (event: DragEndEvent) => {
    const { active, over } = event
    if (!over || active.id === over.id) return

    const activeItem = findItemById(active.id, pages, folders)
    const overItem = findItemById(over.id, pages, folders)

    if (!activeItem || !overItem) return

    // 循环嵌套防护
    if (isDemoFolder(activeItem) && isDemoFolder(overItem)) {
      if (isDescendant(activeItem.id, overItem.id, folders)) return
    }

    // 判定：拖入文件夹 vs 同级排序
    if (isDemoFolder(overItem)) {
      // 拖入文件夹：active 的 parentId 设为 over 的 id
      moveItemToFolder(activeItem, overItem.id)
    } else {
      // 同级排序：active 的 parentId 设为 over 的 parentId
      moveItemToPosition(activeItem, overItem)
    }
  }

  return (
    <DndContext
      sensors={sensors}
      collisionDetection={closestCenter}
      onDragEnd={handleDragEnd}
    >
      <SortableContext
        items={flatItems.map(f => f.item.id)}
        strategy={verticalListSortingStrategy}
      >
        {flatItems.map(flat => (
          <DemoPageTreeItem key={flat.item.id} flatItem={flat} ... />
        ))}
      </SortableContext>
    </DndContext>
  )
}
```

### 8.5 拖入文件夹的视觉判定

当拖拽项悬停在文件夹上时，需要区分「拖入文件夹」和「排在文件夹旁边」。方案：

- **文件夹项的上半区域** → 排在文件夹前面（同级）
- **文件夹项的下半区域** → 拖入文件夹内部
- **文件夹项中间区域** → 拖入文件夹内部

简化实现：**悬停在文件夹上即视为拖入文件夹**，因为"排在文件夹旁边"可以通过拖到文件夹上方/下方的页面项实现。这与 macOS Finder 的行为一致。

## 九、AI 代理适配

### 9.1 核心原则：AI 不感知文件夹

文件夹是 **UI 组织概念**，AI 操作页面时仍使用 `demos/{demoId}/` 路径。理由：

1. 物理目录结构不变，AI 的文件读写路径不变
2. AI 不需要理解文件夹层级来定位页面
3. 减少上下文注入量，降低 AI 心智负担

### 9.2 上下文注入调整

AI 上下文中仅列出页面，不展示文件夹层级：

```
当前项目「我的项目」，包含 5 个页面：
  📄 "首页" → demos/demo_001/
  📄 "引导页" → demos/demo_002/
  📄 "产品列表" → demos/demo_003/
  📄 "产品详情" → demos/demo_004/
  📄 "关于我们" → demos/demo_005/
```

### 9.3 AI 页面管理操作扩展

| 自然语言示例 | AI 行为 |
|:------------|:--------|
| "新建一个叫'产品中心'的页面" | 调用 `POST /api/projects/{id}/demos`，不指定 `parentId`（根级） |
| "在'产品模块'文件夹下新建一个页面" | 调用 `POST /api/projects/{id}/demos`，`parentId` 设为对应文件夹 ID |
| "把首页移到'首页模块'文件夹" | 调用 `PATCH /api/projects/{id}/demos/{demoId}`，更新 `parentId` |

AI 需要感知文件夹的场景极少（仅当用户明确提到文件夹时），默认不注入文件夹信息。

## 十一、风险与对策

| 风险 | 影响 | 对策 |
|:-----|:-----|:-----|
| 拖拽到文件夹的判定不够直观 | 用户误操作将页面拖入文件夹 | 悬停文件夹时高亮 + 500ms 延迟触发；提供「移动到」菜单作为精确操作替代 |
| 文件夹删除误操作 | 用户误删含页面的文件夹 | 默认推荐"仅删除文件夹"模式；"删除文件夹及内容"需二次确认 |
| 循环嵌套 | 文件夹 A 的子文件夹是 B，B 的子文件夹是 A | 后端校验 `isDescendant`；前端拖拽时实时拦截 |
| 嵌套过深 | 3 层以上嵌套降低可用性 | 硬限制最大 3 层；创建/移动时校验 |
| 批量排序 API 部分失败 | 某些项更新成功，某些失败 | API 使用事务性写入（先校验全部，再统一写入）；失败时整体回滚 |
| `parentId` 指向已删除文件夹 | 页面成为"孤儿" | 删除文件夹时自动将子项 `parentId` 重置为 `null`（移至根级）或随文件夹一起删除 |
