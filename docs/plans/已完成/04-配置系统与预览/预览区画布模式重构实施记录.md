# 预览区画布模式重构实施记录

> **状态**：Vibe Coding 迭代中 | **日期**：2026-06-04 | **方案**：iframe 内 DOM 采集 + Thumbnail Generator → Thumbnail Renderer | **最新迭代**：2026-06-04（双信号时序 + 编译步骤 + zoom 阈值切换）

---

## 〇、背景：预览区的三种模式

创作端预览区支持三种预览模式，通过 `PreviewMode` 切换：

| 模式 | 值 | 说明 |
|------|-----|------|
| **单页面模式** | `"single"` | 单个 iframe 实时预览当前编辑页 |
| **宫格模式** | `"grid"` | 多页面等比缩略网格展示，懒加载 iframe |
| **画布模式** | `"canvas"` | 无限画布 + 自由拖拽平移/缩放 + 多页面卡片布局 |

本文档聚焦 **画布模式** 的重构。单页面和宫格模式不受影响。

---

## 一、方案演进

### 1.1 原始方案（已废弃）

原方案为 **Puppeteer 截图 + 单 iframe 混合画布**：

```
画布上默认展示 Puppeteer 生成的 PNG 截图（<img>）
用户编辑某页时，该页切换为 <iframe> 实时预览
截图由 agent-service 的 Puppeteer 服务端离线生成
```

### 1.2 废弃原因

- 依赖 Puppeteer/Chromium，增加 Docker 复杂度
- 需要维护截图缓存、版本号、文件存储
- 截图生成异步延迟，画布打开时可能无图
- 快照与实际代码可能不同步

### 1.3 最终方案

**彻底移除截图，改为 iframe 内 DOM 采集 + 结构化缩略图**：

```
Thumbnail 数据由 iframe 内部 DOM 采集生成
画布默认展示 Thumbnail Renderer（结构化缩略图）
当前编辑页才使用真实 iframe 实时预览
不再生成、存储、读取任何截图文件
```

---

## 二、最终架构

三层架构：

```
1. Preview iframe
   负责真实运行 React 页面，内置 DOM 采集能力
   
2. Thumbnail Generator
   负责从 iframe 采集到的 DOM layout evidence 中生成 ThumbnailMeta

3. Thumbnail Renderer
   负责根据 ThumbnailMeta 渲染轻量结构化缩略图
```

画布渲染优先级：

```typescript
if (zoom >= CANVAS_THUMBNAIL_ZOOM_THRESHOLD) {
  return <PreviewPanel iframe />;       // 放大时：真实 iframe
}
if (page.thumbnailMeta) {
  return <ThumbnailRenderer />;         // 有数据：结构化缩略图
}
return <ThumbnailPlaceholder />;        // 无数据：占位
```

其中 `CANVAS_THUMBNAIL_ZOOM_THRESHOLD = 0.55`（可通过 canvas 缩放状态传入）。缩小时全部用轻量缩略图，放大到 0.55× 以上才挂载 iframe。

### 4.2 从单页/宫格切换至画布模式时

```
进入画布模式
  ↓
遍历每个页面：
  - 已有 thumbnailMeta → ThumbnailRenderer 渲染
  - 无 thumbnailMeta → ThumbnailPlaceholder 显示
  ↓
后台低并发（并发 2）隐藏 iframe 队列：
  - 创建隐藏 iframe（position:fixed, top:-9999px）
  - 加载页面代码 → 采集 DOM → 生成 ThumbnailMeta
  - 销毁 iframe → 继续下一个
  ↓
采集超时 5 秒自动放弃
```

### 4.3 编辑某页时

```
用户点击某页「修改配置」按钮
  ↓
该页缩略图 → 切换为 PreviewPanel iframe（仅当 zoom < 0.55 时需切换
    若 zoom ≥ 0.55 则本来就在展示 iframe，无需额外操作）
配置面板打开，实时预览（代码使用 pageCodes[p.id] 确保每页独立）
  ↓
用户保存/关闭编辑
  ↓
重新采集该页 DOM → 更新 thumbnailMeta
切换回缩略图（或保持 iframe 视 zoom 而定）
```

**每页独立代码**：edit page 维护 `pageCodes` 状态，从 `multi.demos` 中提取各页代码。canvas 接收 `code: pageCodes[p.id] || code`，确保不同页展示不同代码。

## 五、Thumbnail Generator 规则

### 5.1 元素过滤

```typescript
function isUsefulRawElement(el): boolean {
  // 排除不可见：display:none, visibility:hidden, opacity:0
  if (rect.width <= 0 || rect.height <= 0) return false;
  // 排除过小：面积 < 24×24
  if (area < 576) return false;
  // 保留有内容的元素：文本/图片/背景色/阴影/边框
  return hasText || hasImage || hasBackground || hasShadow || hasBorder;
}
```

### 5.2 元素分类

```typescript
tag === "img" || attrs.src        → "image"
tag === "button" || role=button   → "button"
input/textarea/select              → "input"
fontSize >= 14px && 有文本         → "text"
hasCardLikeStyle (radius≥8, shadow, solid bg, >80×80)  → "card"
其他                               → "unknown"
```

### 5.3 评分公式

```
score = areaScore × 0.35      // 面积占比（相对于 375×812 视口）
      + semanticScore × 0.25  // 语义价值（image 0.9 > button 0.7 > text 0.6 > card 0.5 > input 0.4）
      + textScore × 0.15      // 文本长度（最多 100 字符满分）
      + centerScore × 0.15    // 距中心距离（越近越高）
      + visualStyleScore × 0.1 // 视觉特征（卡片/阴影/背景图）
```

### 5.4 识别背景

```typescript
宽高比 > 0.8（覆盖视口 80%+）→ 背景
x <= 0 && y <= 0 && 覆盖 90%+ 宽高 → 背景
有背景色 && 覆盖 90%+ 宽度 → 背景
```

### 5.5 页面类型推断

```
有表单输入元素（input/textarea/select）           → "form"
有 ≥2 个按钮元素                                   → "question"
其他                                              → "custom"
```

## 六、Thumbnail Renderer 渲染

Thumbnail Renderer 不读取原始 React 代码，只读取 ThumbnailMeta，做低保真结构预览：

| Block 类型 | 渲染方式 |
|-----------|---------|
| **background** | 纯色 div（z-index:-1） |
| **text** | 灰色横条（高度 = fontSize×1.2，带裁剪） |
| **image** | 浅灰背景 + 图片占位 SVG 图标 |
| **button** | 70%×60% 居中实心圆角矩形 |
| **card** | 半透明白色 + 细边框 |
| **input** | 半透明白底 + 细边框 + 60% 宽度灰色条 |
| **options** | 等间距灰色横条（最多 4 条） |
| **unknown** | 极淡灰背景块 |

## 七、批量生成策略

画布低并发隐藏 iframe 队列：

```typescript
const THUMBNAIL_IFRAME_CONCURRENCY = 2;   // 并发数
const THUMBNAIL_COLLECT_DELAY = 800;      // 渲染稳定后采集延迟（ms），含编译时间
const THUMBNAIL_COLLECT_TIMEOUT = 8000;   // 采集超时（ms），含编译时间
const CANVAS_DEFAULT_ZOOM = 0.5;          // 画布默认缩放，低于 0.55 阈值
const CANVAS_THUMBNAIL_ZOOM_THRESHOLD = 0.55; // zoom ≥ 此值显示真实 iframe
```

流程：
1. 先渲染已有 thumbnailMeta 的页面（默认 zoom=0.5<阈值，全部用缩略图）
2. 缺失页面进入队列，每次并发 2 个隐藏 iframe
3. iframe 渲染前先调 `/api/compile` 编译代码
4. iframe 渲染完成 → 双信号时序采集 DOM → 生成 ThumbnailMeta → 销毁 iframe
5. 通过 `setOnPageThumbnailGenerated()` 回调通知父组件保存

**编译步骤**：`collectThumbnailFromIframe` 先调用 `/api/compile` 将 TSX/JSX 转为带 CDN import 的 JavaScript，再嵌入 iframe。若编译失败，该页缩略图生成立即 reject，保持占位状态。

## 八、Hash 缓存策略

```typescript
thumbnailHash = hash({
  codeHash,        // 代码内容的 hash
  configHash,      // 配置数据的 hash
  previewSize,     // 预览尺寸
  generatorVersion // 生成器版本号
});
```

hash 不变则复用已有 thumbnailMeta，避免重复采集。

## 九、失败兜底

失败场景和应对：

| 场景 | 应对 |
|------|------|
| iframe 渲染失败 | 显示 ThumbnailPlaceholder（页面名称 + 骨架 SVG） |
| 代码编译失败 | 保持占位状态，不创建隐藏 iframe |
| 采集超时（8s） | 清空隐藏 iframe，该页保持占位状态 |
| meaningful block < 3 | confidence 降低至 0.2~0.6，但仍渲染 |
| confidence < 0.4 | isThumbnailReady 返回 false，视为不可用 |

兜底 UI（ThumbnailPlaceholder）：展示浏览器窗口 SVG 图标 + 页面名称，不显示加载 spinner。

## 十一、设计原则总结

```
1. 不做截图            → 零图片文件存储
2. 不存图片            → 只存结构化 JSON（ThumbnailMeta）
3. 不引入 Puppeteer    → 无服务端浏览器依赖
4. 不同时挂载 N 个 iframe → 画布上最多 1 个（编辑页），其余用轻量 div 渲染
5. iframe 仅用于编辑    → 编辑时实时预览，后台低并发采集缩略图数据
6. 画布默认用 Renderer  → 轻量 div，60fps 流畅
7. Renderer 只追求结构  → 不追求像素级还原，关注布局/主色/元素类型
8. Hash 缓存           → 代码/配置不变则复用
9. 失败结构占位兜底     → 不阻塞画布渲染
```

## 十二、最终效果

**画布缩小时**：所有页面显示大致布局（主色背景 + 标题横条 + 图片块 + 按钮块 + 卡片结构），拖拽缩放流畅，无大量 iframe 性能问题，无截图等待。

**编辑某页时**：该页切换为真实 iframe 实时预览，配置面板同步编辑，其余页面保持结构化缩略图不变。编辑页代码使用 `pageCodes` 状态确保每页独立。

**整体**：用轻量结构化缩略图替代截图，牺牲像素级还原，换取零服务端依赖、更轻的实现和更好的交互一致性。

## 十三、Vibe Coding 迭代记录（2026-06-04）

### 13.1 新增：每页独立代码（pageCodes）

**问题**：画布中所有页面共享同一份代码，切换「预览页面类型」时所有页面一起变。

**解决**：edit page 维护 `pageCodes` 状态，从 `multi.demos` 中各页的 `code` 字段提取。canvas 接收 `code: pageCodes[p.id] || code`。用户在配置面板切换页面类型时，仅更改当前页的代码。

**涉及**：`edit/page.tsx` — 新增 `pageCodes` state + `useEffect` 同步

### 13.2 新增：/api/compile 编译步骤

**问题**：iframe 内的 `<script type="module">` 无法直接 import TSX/JSX。需要编译为纯 JavaScript 才能通过 CDN import 加载。

**解决**：`collectThumbnailFromIframe` 在创建隐藏 iframe 前先调 `/api/compile` 将代码编译为带 CDN import 路径的 JavaScript。编译结果（compiledCode + cssImports）传入 `generateIframeHtml`。

**影响**：每页缩略图生成增加一次网络请求；超时从 5s 延长至 8s。

### 13.3 新增：COMPONENT_READY 双信号时序

**问题**：iframe 的 `READY` 消息在代码 `import()` 之前发送，此时 DOM 还是空的。若在 READY 后立即采集（原 300ms），得到的缩略图没有元素。

**解决**：新增 `COMPONENT_READY` 消息（组件首次渲染完成后 setTimeout(fn, 0) 发送）。采集时序改为双信号互斥：
- `READY` 到达 → 排队 800ms 后采集（兜底）
- `COMPONENT_READY` 到达 → 取消前一个计时器，改为 200ms 后采集（Fast Path）

同一时刻只有一次采集请求，避免重复触发。

**涉及**：`PreviewCanvas.tsx` — 消息处理器重写；`iframe-template.ts` — 发送 COMPONENT_READY

### 13.4 新增：zoom 阈值切换（CanvasPageItem）

**问题**：画布放大后缩略图过于粗糙，用户希望看到真实页面。

**解决**：`CanvasPageItem` 接收 `zoom` prop，比较 `CANVAS_THUMBNAIL_ZOOM_THRESHOLD (0.55)`：
- `zoom < 0.55`（缩小状态）→ ThumbnailRenderer / Placeholder
- `zoom >= 0.55`（放大状态）→ PreviewPanel iframe（实时页面）

默认 zoom 0.5 低于阈值，画布打开时全用缩略图。

**涉及**：`CanvasPageItem.tsx` — 新增 zoom prop + 条件渲染；`PreviewCanvas.tsx` — 透传 zoom

### 13.5 改进：ThumbnailBlockView 视觉呈现

**问题**：缩略图中 text 块只是灰色横条，用户无法区分文本/按钮/卡片/输入框。

**解决**：
- text block：显示实际 `contentHint` 文字（最多 40 字符 + 省略号）
- button block：70%×60% 居中实心圆角矩形，显示按钮文本
- card block：半透明白底 + 细边框 + 内部灰色内容预览条
- input block：半透明白底 + 细边框 + 60% 宽度灰色占位条

**涉及**：`ThumbnailBlockView.tsx` — 各类型渲染逻辑重写

### 13.6 修复：面板模式切换按钮

**问题**：进入画布模式后无法切回单页/宫格模式；宫格模式缺少「画布」按钮。

**解决**：
- 画布模式工具栏新增 `showModeToggle` 按钮（单页/宫格）
- `PreviewGrid` 的 mode 切换按钮组新增 canvas 模式

**涉及**：`PreviewCanvas.tsx` — 新增模式切换按钮；`PreviewGrid.tsx` — 新增 canvas 按钮

### 13.7 当前已知问题

- iframe 的 `READY` 在 `import()` 之前发送；`COMPONENT_READY` 在组件渲染后发送。若编译成功但组件渲染抛异常，COMPONENT_READY 不会发送 → 由 READY 的 800ms 兜底采集，此时 DOM 为空 → 缩略图为空/占位
- 编译步骤增加了一次网络往返，缩略图生成总时间约 1~3s（含编译 + CDN 加载 + 渲染 + 采集）
- 默认 zoom 0.5 低于阈值，所有页面默认缩略图 → 用户需主动放大才看到真实页面
