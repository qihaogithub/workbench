## 一、问题背景

### 1.1 现状描述

画布模式（Canvas Mode）存在两类问题：

**功能缺失（核心）**：

1. **页面无法拖动位置**：代码中有拖拽逻辑但存在严重缺陷——`onMouseMove`/`onMouseUp` 仅绑定在页面 DOM 元素上，未使用 `setPointerCapture`，鼠标快速移动或离开元素后偏移追踪中断，拖拽体验断裂。
2. **页面无法缩放大小**：`CanvasPageLayout` 类型定义了 `width`/`height` 字段，但没有任何 UI 交互可以修改它们——无缩放手柄、无缩放事件处理、无单页尺寸变更逻辑。

**性能问题**：

3. **加载慢**：首次进入画布模式时，所有页面 iframe 同时初始化，导致 CDN 依赖下载阻塞
4. **缩放/拖动卡顿**：视口变换时每次事件都触发 React state 更新，导致频繁重渲染
5. **iframe 闪烁**：组件卸载重新挂载时 iframe 完全重建，用户看到内容 → 空白 → 加载中 → 内容淡入的闪烁过程

### 1.2 架构概览

画布模式核心组件位于 `packages/shared/src/demo/`：

| 组件 | 文件 | 职责 |
|------|------|------|
| PreviewCanvas | [PreviewCanvas.tsx](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/PreviewCanvas.tsx) | 画布顶层容器，管理状态和页面布局 |
| CanvasViewport | [CanvasViewport.tsx](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasViewport.tsx) | 视口组件，处理画布平移和缩放 |

## 二、功能缺失分析

### 2.1 缺陷一：页面拖拽不可用

#### 2.1.1 当前实现

[CanvasPageItem.tsx 第 45-76 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasPageItem.tsx#L45-L76)：

```typescript
const handleDragStart = useCallback(
  (e: React.MouseEvent) => {
    if (!editable || isEditing) return;
    e.stopPropagation();
    e.preventDefault();
    setIsDragging(true);
    startPosRef.current = { x: e.clientX, y: e.clientY };
    layoutStartRef.current = { x: layout.x, y: layout.y };
    onDragStart?.();
  },
  [editable, isEditing, layout.x, layout.y, onDragStart],
);

const handleDragMove = useCallback(
  (e: React.MouseEvent) => {
    if (!isDragging) return;
    e.stopPropagation();
    const dx = e.clientX - startPosRef.current.x;
    const dy = e.clientY - startPosRef.current.y;
    onLayoutChange?.(page.id, {
      ...layout,
      x: layoutStartRef.current.x + dx,
      y: layoutStartRef.current.y + dy,
    });
  },
  [isDragging, page.id, layout, onLayoutChange],
);

const handleDragEnd = useCallback(() => {
  setIsDragging(false);
  onDragEnd?.();
}, [onDragEnd]);

// DOM 绑定 (第 103-105 行)
<div
  onMouseDown={handleDragStart}
  onMouseMove={handleDragMove}   // ← 仅在鼠标位于元素上方时触发
  onMouseUp={handleDragEnd}      // ← 仅在鼠标位于元素上方时触发
  ...
>
```

#### 2.1.2 三个致命缺陷

1. **无 pointer capture**：未调用 `(e.target as HTMLElement).setPointerCapture(e.pointerId)`。当鼠标移动速度超过元素边界时，`onMouseMove` 不再触发，拖拽中途断裂。

2. **事件作用域局限**：`onMouseMove` 和 `onMouseUp` 绑定在页面元素自身，而非 `document` 或 `window`。鼠标离开元素后（即使只移动 1px 到元素外），事件追踪立即中断。作为对比，`CanvasViewport` 的 panning 使用了 `setPointerCapture`，工作正常。

3. **`onMouseUp` 丢失**：如果 `onMouseUp` 在元素外触发（鼠标离开后松开），`isDragging` 状态永久卡在 `true`，后续 hover 行为异常。

4. **`handleDragMove` 依赖 `layout`**：[第 70 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasPageItem.tsx#L70) 依赖数组包含 `layout`，每次拖拽移动时 layout 变化 → callback 重建，虽然没有功能 Bug 但增加了不必要的重渲染开销。

#### 2.1.3 修复方向

将拖拽改为 pointer events + `setPointerCapture` 模式（与 CanvasViewport 的 panning 一致）：

- `onPointerDown` → `setPointerCapture`，记录初始位置
- `onPointerMove` → 计算 delta，调用 `onLayoutChange`
- `onPointerUp` / `onLostPointerCapture` → 清理状态

### 2.2 缺陷二：页面缩放完全未实现

#### 2.2.1 当前状态

- `CanvasPageLayout.width`/`height` 仅在 `computeInitialLayout()` 中根据 `previewSize` 配置计算，之后不可变更
- `handleLayoutChange` 可修改任意字段，但 `CanvasPageItem` 只通过拖拽修改 `x`/`y`
- 没有任何缩放手柄 UI、没有任何 resize 事件处理、没有任何光标指示
- 编辑页（`editable=true`）下，页面项仅有 `cursor-move` 样式，无 resize 相关 cursor

#### 2.2.2 需要实现

| 项目 | 描述 |
|------|------|
| 缩放手柄 | 8 个手柄：四角 + 四边中点（nw, n, ne, e, se, s, sw, w） |
| 手柄可见性 | `editable && !isEditing` 时 hover 显示 |
| 光标样式 | 各手柄对应 `nwse-resize` / `nesw-resize` / `ew-resize` / `ns-resize` |
| 缩放交互 | pointer capture 模式，根据拖拽的手柄方向计算新的 width/height/x/y |
| 最小尺寸 | 限制最小 100×100，防止缩放为负或零 |
| 等比缩放 | 可选：按住 Shift 时保持宽高比 |
| 状态更新 | 调用 `onLayoutChange(pageId, { x, y, width, height })` |

#### 2.2.3 缩放方向矩阵

各手柄对 `x, y, width, height` 的影响：

| 手柄 | 需更新的字段 | 说明 |
|------|------------|------|
| nw (左上) | `x, y, width, height` | 右下角固定，左上角移动 |
| n (上中) | `y, height` | 下边固定，上边移动 |
| ne (右上) | `y, width, height` | 左下角固定，右上角移动 |
| e (右中) | `width` | 左边固定，右边移动 |
| se (右下) | `width, height` | 左上角固定，右下角移动 |
| s (下中) | `height` | 上边固定，下边移动 |
| sw (左下) | `x, width, height` | 右上角固定，左下角移动 |
| w (左中) | `x, width` | 右边固定，左边移动 |

## 三、性能瓶颈分析

### 3.1 瓶颈一：无视口裁剪

[PreviewCanvas.tsx 第 132-149 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/PreviewCanvas.tsx#L132-L149)：

```typescript
<CanvasViewport ...>
  {pages.map((page) => (
    <CanvasPageItem  // ← 所有页面无条件渲染，包括视口外的页面
      key={page.id}
      page={page}
      ...
    />
  ))}
</CanvasViewport>
```

**对比宫格模式**：`PreviewGrid` 使用了 `useVisiblePages` hook（IntersectionObserver）实现虚拟滚动，画布模式完全没有。

**影响程度**：与页面数量成正比。10 个页面时影响有限，20+ 页面时显著。因为每个 `CanvasPageItem` 在无截图时会挂载一个 `PreviewPanel`（含 iframe），每个 iframe 独立加载 React + ReactDOM + Tailwind CDN。

### 3.2 瓶颈二：事件处理无节流

[CanvasViewport.tsx 第 46-58 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasViewport.tsx#L46-L58)：

```typescript
const handlePointerMove = useCallback(
  (e: React.PointerEvent) => {
    if (!isPanning) return;
    const dx = e.clientX - startPosRef.current.x;
    const dy = e.clientY - startPosRef.current.y;
    onViewportChange({  // ← 每次 pointermove 都触发 state 更新
      ...viewport,
      x: viewportStartRef.current.x + dx,
      y: viewportStartRef.current.y + dy,
    });
  },
  [isPanning, viewport, onViewportChange],
);
```

每次 `pointermove` 事件都调用 `onViewportChange`，触发 React 重渲染。

**注意**：由于 `handlePointerMove` 的依赖中包含 `viewport`，每次 viewport 更新后 callback 也会重建，形成"更新 → callback 重建 → 下次事件用新 callback"的循环。但由于 `viewportStartRef` 在 pointerDown 时已固定，实际计算结果不受影响，只是每次都创建新对象传给 `onViewportChange`。

**实际影响**：拖动画布时每次 pointermove 触发一次 React 渲染周期。由于 CSS transform 是 GPU 加速的，DOM 操作本身很轻量，瓶颈主要在于：子组件（CanvasPageItem）因父组件 state 变化而全部重渲染。配合方案 4（移除 zoom prop）和方案 1（视口裁剪）后，此问题可大幅缓解。

### 3.3 瓶颈三：组件卸载重建导致 iframe 闪烁

**根因分析**：

iframe 闪烁的真正原因不是 `isUrlMode` 变化（画布模式下 `compiledJsUrl` 始终为空，`isUrlMode` 始终为 false），而是以下场景：

1. **CanvasPageItem 条件渲染切换**：[CanvasPageItem.tsx 第 113-155 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasPageItem.tsx#L113-L155) 中，`isEditing`、`screenshotUrl` 的变化会导致渲染分支切换（iframe ↔ img），React 卸载旧组件、挂载新组件，iframe 被完全重建。

2. **PreviewCanvas 受控状态更新**：[PreviewCanvas.tsx 第 71 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/PreviewCanvas.tsx#L71) 中 `canvasState = externalState || internalState`，两个使用场景（编辑页和查看页）都传入 `canvasState` 和 `onCanvasStateChange`，属于受控模式。每次 `onCanvasStateChange` 调用后，父组件 state 更新 → PreviewCanvas 重渲染 → 所有 CanvasPageItem 重渲染。

3. **PreviewPanel 的 contentLoaded 机制**：[PreviewPanel.tsx 第 511-513 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/PreviewPanel.tsx#L511-L513) 中，`iframeSrcUrl` 变化时 `contentLoaded` 重置为 false，显示 loading spinner。虽然画布模式下 `iframeSrcUrl` 只在组件首次挂载时设置一次（`isUrlMode` 不变），但组件卸载重新挂载时仍会经历完整的初始化流程。

**闪烁场景**：
- 用户点击"修改配置"进入编辑 → `isEditing` 变为 true → 渲染分支从截图/img 切换到 iframe → iframe 从零初始化 → 闪烁
- 用户退出编辑 → `isEditing` 变为 false → iframe 被卸载，截图/img 挂载 → 如果截图尚未生成，又回退到 iframe → 闪烁

### 3.4 瓶颈四：zoom prop 引发不必要重渲染

[CanvasPageItem.tsx 第 30 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/CanvasPageItem.tsx#L30)：

```typescript
zoom = 0,
```

`zoom` prop 从 [PreviewCanvas 第 142 行](file:///e:/重要文件/Programming/1_Work/opencode-workbench/packages/shared/src/demo/PreviewCanvas.tsx#L142) 传入，但实际缩放已通过 CSS transform 在父容器实现。CanvasPageItem 内部未使用 `zoom` 值做任何计算，但每次 zoom 变化都触发所有页面项重渲染。

**实际影响**：每次拖动/缩放时，所有 CanvasPageItem 都会重渲染。虽然 React diff 后 DOM 不变，但组件函数执行和 VNode 比较仍有开销，页面多时累加明显。

## 四、方案设计

### 方案 1：修复页面拖拽 — pointer capture 模式（P0 ★ 功能）

**优先级**：P0
**类型**：功能修复
**预期收益**：页面可在画布中自由拖拽移动，手感流畅
**实施难度**：⭐⭐

#### 4.1.1 方案描述

将 mouse 事件改为 pointer 事件 + `setPointerCapture` 模式（与 CanvasViewport 的 panning 一致）。pointer capture 确保即使指针移出元素边界，事件仍投递给该元素，从根本上解决拖拽断裂问题。

#### 4.1.2 实现位置

`packages/shared/src/demo/CanvasPageItem.tsx`

#### 4.1.3 修改要点

- `onMouseDown` → `onPointerDown`，调用 `setPointerCapture(e.pointerId)`
- `onMouseMove` → `onPointerMove`（在 pointer capture 下持续触发）
- `onMouseUp` → `onPointerUp` + `onLostPointerCapture`，清理拖拽状态
- `handleDragMove` 依赖中移除 `layout`，改用 ref 保存 layout
- 移除 `onDragStart` / `onDragEnd` 回调（内部不需要对外通知拖拽状态，上层只需 `onLayoutChange`）

#### 4.1.4 注意事项

- pointer capture 必须在 `onPointerDown` 中调用，且需要检查 `e.isPrimary`
- 拖拽中应阻止文本选择和默认拖拽行为
- 与缩放手柄的交互互斥：拖拽页面和拖拽缩放不能同时进行

### 方案 2：实现页面缩放 — 缩放手柄（P0 ★ 新功能）

**优先级**：P0
**类型**：新功能
**预期收益**：页面可在画布中自由缩放宽高，灵活布局
**实施难度**：⭐⭐⭐

#### 4.2.1 方案描述

在 `CanvasPageItem` 上添加 8 个缩放手柄（四角 + 四边中点），`editable && !isEditing` 且 hover 时显示。拖拽手柄时使用 pointer capture 模式，根据手柄方向计算新的 `x, y, width, height`，调用 `onLayoutChange` 更新布局。

#### 4.2.2 实现位置

`packages/shared/src/demo/CanvasPageItem.tsx`

#### 4.2.3 手柄定义

```typescript
type ResizeHandle = "nw" | "n" | "ne" | "e" | "se" | "s" | "sw" | "w";

const HANDLE_DEFS: Record<ResizeHandle, {
  cursor: string;
  position: React.CSSProperties;   // 定位到对应边缘/角落
}> = {
  nw:  { cursor: "nwse-resize", position: { top: -4, left: -4 } },
  n:   { cursor: "ns-resize",   position: { top: -4, left: "50%", marginLeft: -4 } },
  ne:  { cursor: "nesw-resize", position: { top: -4, right: -4 } },
  e:   { cursor: "ew-resize",   position: { top: "50%", right: -4, marginTop: -4 } },
  se:  { cursor: "nwse-resize", position: { bottom: -4, right: -4 } },
  s:   { cursor: "ns-resize",   position: { bottom: -4, left: "50%", marginLeft: -4 } },
  sw:  { cursor: "nesw-resize", position: { bottom: -4, left: -4 } },
  w:   { cursor: "ew-resize",   position: { top: "50%", left: -4, marginTop: -4 } },
};
```

每个手柄为 8×8px 的小方块，带白色背景 + 蓝色边框，hover 时显示。

#### 4.2.4 缩放逻辑

以拖拽 `se`（右下角）手柄为例：

```typescript
// dx, dy = 拖拽位移（已考虑视口缩放）
const newWidth = Math.max(MIN_SIZE, startWidth + dx);
const newHeight = Math.max(MIN_SIZE, startHeight + dy);
onLayoutChange?.(page.id, { ...layout, width: newWidth, height: newHeight });
```

各手柄方向的新宽高/x/y 计算公式见 [2.2.3 缩放方向矩阵](#223-缩放方向矩阵)。

**最小尺寸**：`MIN_SIZE = 100`，防止缩放为 0 或负数。

**等比缩放（Shift 键）**：按住 Shift 时锁定宽高比，取 `max(dx, dy) / startWidth` 和 `max(dx, dy) / startHeight` 中主导方向的比例。

#### 4.2.5 视觉设计

- 手柄尺寸：8×8px 方块（可点击区域适当扩大为 12×12px 的命中区）
- 默认隐藏，父元素 hover 时显示（`opacity: 0 → 1` 过渡）
- 手柄自身 hover 时放大到 10×10px（视觉反馈）
- 编辑状态（`isEditing`）下隐藏所有手柄
- 拖拽过程中手柄暂时隐藏，避免干扰

### 方案 3：视口裁剪 — 只渲染可见页面（P0）

**优先级**：P0
**类型**：性能优化
**预期收益**：页面多时显著减少同时存活的 iframe 数量
**实施难度**：⭐⭐

#### 4.3.1 方案描述

画布模式页面位置已知（`CanvasPageLayout.x/y/width/height`），视口范围可计算（`viewport.x/y/zoom` + 容器尺寸）。使用纯数学计算判断哪些页面在视口内，无需 IntersectionObserver。

#### 4.3.2 实现位置

`packages/shared/src/demo/PreviewCanvas.tsx`

#### 4.3.3 代码示例

```typescript
// 新增工具函数
function getVisiblePageIds(
  pages: Record<string, CanvasPageLayout>,
  viewport: CanvasViewportState,
  containerWidth: number,
  containerHeight: number,
  buffer: number = 200, // 缓冲区像素
): Set<string> {
  const visible = new Set<string>();

  // 视口在画布坐标系中的范围
  const vx = -viewport.x / viewport.zoom;
  const vy = -viewport.y / viewport.zoom;
  const vw = containerWidth / viewport.zoom;
  const vh = containerHeight / viewport.zoom;

  for (const [id, layout] of Object.entries(pages)) {
    if (
      layout.x + layout.width + buffer > vx &&
      layout.x - buffer < vx + vw &&
      layout.y + layout.height + buffer > vy &&
      layout.y - buffer < vy + vh
    ) {
      visible.add(id);
    }
  }
  return visible;
}
```

#### 4.3.4 CanvasPageItem 修改

不可见的页面只渲染占位符（保持布局），不挂载 PreviewPanel/iframe：

```typescript
interface CanvasPageItemProps {
  visible?: boolean;  // 新增
  // ...
}

// 在组件内，最前面判断
if (!visible) {
  return (
    <div
      style={{
        position: "absolute",
        left: layout.x,
        top: layout.y,
        width: layout.width,
        height: layout.height,
        backgroundColor: "transparent",
        pointerEvents: "none",
      }}
    />
  );
}
```

#### 4.3.5 注意事项

- `containerSize` 需要通过 ResizeObserver 获取
- 初始加载时（`containerSize` 为 0）应默认所有页面可见，避免首屏空白
- 缓冲区 `buffer` 值需根据实际体验调整，太小会导致页面"突然出现"，太大会削弱裁剪效果

### 方案 4：rAF 节流视口更新（P0）

**优先级**：P0
**类型**：性能优化
**预期收益**：拖拽/缩放画布流畅度显著提升
**实施难度**：⭐⭐

#### 4.4.1 方案描述

使用 `requestAnimationFrame` 合并 pointermove 和 wheel 事件触发的状态更新，确保每帧最多一次渲染。

#### 4.4.2 实现位置

`packages/shared/src/demo/CanvasViewport.tsx`

#### 4.4.3 代码示例

```typescript
export function CanvasViewport({ viewport, onViewportChange, ... }) {
  const rafIdRef = useRef(0);
  const pendingViewportRef = useRef<CanvasViewportState | null>(null);
  const containerRef = useRef<HTMLDivElement>(null);
  const [isPanning, setIsPanning] = useState(false);
  const isInteractingRef = useRef(false);  // 用 ref 而非 state，避免额外渲染
  const [willChangeTransform, setWillChangeTransform] = useState(false);

  const flushUpdate = useCallback(() => {
    if (pendingViewportRef.current) {
      onViewportChange(pendingViewportRef.current);
      pendingViewportRef.current = null;
    }
  }, [onViewportChange]);

  const scheduleUpdate = useCallback((newViewport: CanvasViewportState) => {
    pendingViewportRef.current = newViewport;
    cancelAnimationFrame(rafIdRef.current);
    rafIdRef.current = requestAnimationFrame(flushUpdate);
  }, [flushUpdate]);

  // will-change 管理：交互开始时启用，交互结束后延迟关闭
  const interactTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

  const markInteracting = useCallback(() => {
    isInteractingRef.current = true;
    setWillChangeTransform(true);
    if (interactTimerRef.current) clearTimeout(interactTimerRef.current);
  }, []);

  const markInteractingEnd = useCallback(() => {
    // 延迟关闭 will-change，确保最后一帧渲染完成
    interactTimerRef.current = setTimeout(() => {
      isInteractingRef.current = false;
      setWillChangeTransform(false);
    }, 100);
  }, []);

  const handlePointerMove = useCallback(
    (e: React.PointerEvent) => {
      if (!isPanning) return;
      const dx = e.clientX - startPosRef.current.x;
      const dy = e.clientY - startPosRef.current.y;
      scheduleUpdate({
        x: viewportStartRef.current.x + dx,
        y: viewportStartRef.current.y + dy,
        zoom: viewport.zoom,
      });
    },
    [isPanning, viewport.zoom, scheduleUpdate],
  );

  const handlePointerUp = useCallback(
    (e: React.PointerEvent) => {
      if (!isPanning) return;
      setIsPanning(false);
      markInteractingEnd();
      (e.target as HTMLElement).releasePointerCapture?.(e.pointerId);
      // pendingViewport 已在 flushUpdate 中清空，无需额外处理
      const dx = e.clientX - startPosRef.current.x;
      const dy = e.clientY - startPosRef.current.y;
      if (Math.abs(dx) < 3 && Math.abs(dy) < 3) {
        onCanvasClick?.();
      }
    },
    [isPanning, onCanvasClick, markInteractingEnd],
  );

  const handleWheel = useCallback(
    (e: React.WheelEvent) => {
      if (!editable) return;
      e.preventDefault();
      markInteracting();

      const rect = containerRef.current?.getBoundingClientRect();
      if (!rect) return;

      const mouseX = e.clientX - rect.left;
      const mouseY = e.clientY - rect.top;
      const zoomFactor = e.deltaY > 0 ? 1 / ZOOM_STEP : ZOOM_STEP;
      const newZoom = Math.min(Math.max(viewport.zoom * zoomFactor, MIN_ZOOM), MAX_ZOOM);
      const scale = newZoom / viewport.zoom;

      scheduleUpdate({
        x: mouseX - (mouseX - viewport.x) * scale,
        y: mouseY - (mouseY - viewport.y) * scale,
        zoom: newZoom,
      });

      markInteractingEnd();
    },
    [viewport, editable, scheduleUpdate, markInteracting, markInteractingEnd],
  );

  // 组件卸载时清理
  useEffect(() => {
    return () => {
      cancelAnimationFrame(rafIdRef.current);
      if (interactTimerRef.current) clearTimeout(interactTimerRef.current);
    };
  }, []);

  return (
    <div ...>
      <div
        style={{
          transform: `translate(${viewport.x}px, ${viewport.y}px) scale(${viewport.zoom})`,
          willChange: willChangeTransform ? "transform" : "auto",
        }}
      >
        {children}
      </div>
    </div>
  );
}
```

#### 4.4.4 设计要点

- **`isInteractingRef` 用 ref 而非 state**：避免"标记交互中"本身触发额外渲染
- **`willChangeTransform` 用 state**：因为需要驱动 style 变化，必须触发渲染
- **wheel 事件的交互结束**：wheel 没有对应的"结束"事件，每次 wheel 后重置延迟定时器即可
- **`handlePointerMove` 依赖精简**：移除对 `viewport` 的依赖，只依赖 `viewport.zoom`（panning 时 zoom 不变），减少 callback 重建

### 方案 5：修复 iframe 闪烁 — 避免渲染分支切换时卸载 iframe（P0）

**优先级**：P0
**类型**：性能优化
**预期收益**：消除编辑/非编辑切换时的 iframe 闪烁
**实施难度**：⭐⭐⭐

#### 4.5.1 根因

CanvasPageItem 中 `isEditing` 和 `screenshotUrl` 的变化导致条件渲染分支切换（iframe ↔ img），React 卸载旧组件、挂载新组件，iframe 被完全重建。

#### 4.5.2 方案描述

将条件渲染改为"始终渲染 iframe + 叠加截图层"的模式，避免 iframe 被卸载：

- **编辑状态**：显示 iframe，隐藏截图
- **有截图的非编辑状态**：显示截图覆盖层，iframe 保持在 DOM 中但不可见（`visibility: hidden`）
- **无截图的非编辑状态**：显示 iframe

这样切换编辑状态时 iframe 不会被卸载重建，消除了闪烁。

#### 4.5.3 代码示例

```typescript
function CanvasPageItem({ page, layout, editable, isEditing, screenshotUrl, ... }) {
  // iframe 是否需要保持在 DOM 中
  const showIframe = isEditing || !screenshotUrl;
  // 截图覆盖层是否可见
  const showScreenshotOverlay = !isEditing && !!screenshotUrl;

  return (
    <div
      className={cn("absolute ...", ...)}
      style={{ left: layout.x, top: layout.y, width: layout.width, height: layout.height, ... }}
      ...
    >
      {/* iframe 层：始终存在或按需隐藏 */}
      <div
        className="w-full h-full rounded-lg overflow-hidden"
        style={{
          visibility: showIframe ? "visible" : "hidden",
          position: showScreenshotOverlay ? "absolute" : "relative",
        }}
      >
        <PreviewPanel
          code={page.code}
          sessionId={sessionId}
          demoId={page.id}
          configData={page.configData}
          previewSize={page.previewSize}
          onConsoleEntry={onConsoleEntry}
        />
      </div>

      {/* 截图覆盖层 */}
      {showScreenshotOverlay && (
        <div className="absolute inset-0 rounded-lg overflow-hidden shadow-md">
          <img
            src={screenshotUrl}
            alt={page.name}
            className="w-full h-full object-contain pointer-events-none"
            loading="lazy"
            draggable={false}
          />
          <div className="absolute bottom-0 left-0 right-0 bg-gradient-to-t from-black/60 to-transparent p-2 rounded-b-lg">
            <span className="text-xs text-white font-medium truncate block">
              {page.name}
            </span>
          </div>
        </div>
      )}

      {/* 缩放手柄 ... */}
      {/* hover 按钮和拖拽指示器 ... */}
    </div>
  );
}
```

#### 4.5.4 权衡

- **内存**：有截图时 iframe 仍保留在 DOM 中（`visibility: hidden`），占用内存但不会执行绘制。对于页面数量多的场景，需配合方案 3（视口裁剪）一起使用，视口外的页面直接渲染占位符，不保留 iframe
- **一致性**：iframe 中的组件状态在编辑/非编辑切换时得以保留，用户体验更好

### 方案 6：移除不必要的 zoom prop（P1）

**优先级**：P1
**类型**：性能优化
**预期收益**：减少拖动/缩放时 CanvasPageItem 的不必要重渲染
**实施难度**：⭐

#### 4.6.1 方案描述

`CanvasPageItem` 接收 `zoom` prop 但实际不使用（缩放已通过 CSS transform 在父容器实现）。移除此 prop 避免每次 zoom 变化触发所有页面项重渲染。

#### 4.6.2 实现位置

`packages/shared/src/demo/CanvasPageItem.tsx` — 从 props 接口和解构中移除 `zoom`
`packages/shared/src/demo/PreviewCanvas.tsx` — 移除 `zoom={canvasState.viewport.zoom}` 传参

#### 4.6.3 注意

需确认 `zoom` 确实未被使用。当前代码中 `zoom` 仅在解构时赋默认值 `0`，组件内无任何引用，可安全移除。

### 方案 7：截图优先策略增强（P1）

**优先级**：P1
**类型**：性能优化
**预期收益**：减少同时存活的 iframe 数量，降低内存和 CPU 占用
**实施难度**：⭐⭐

#### 4.7.1 方案描述

1. **初始加载时批量生成所有页面截图**，确保画布模式始终优先显示图片
2. **只在双击进入编辑时才激活 iframe**（已在方案 5 中实现：有截图时 iframe 用 `visibility: hidden` 保留）

#### 4.7.2 与方案 5 的关系

方案 5 已将有截图时的 iframe 设为 `visibility: hidden`，使其不参与绘制但保留在 DOM 中。方案 7 进一步优化：配合视口裁剪（方案 3），视口外页面的 iframe 完全不渲染，只有进入视口且有截图时才以 `visibility: hidden` 保留 iframe。

#### 4.7.3 截图生成时机

当前截图通过 `useScreenshotGeneration` hook 的 `startBatchGeneration` 手动触发。优化方向：

- 在切换到画布模式时自动触发批量截图
- 在代码变更后延迟重新生成对应页面截图（防抖）
