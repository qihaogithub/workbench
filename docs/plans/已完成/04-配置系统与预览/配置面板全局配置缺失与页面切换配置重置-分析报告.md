# 配置面板全局配置缺失与页面切换配置重置 — 分析报告

> 版本：v1.0
> 创建日期：2026-05-17
> 项目：proj_1776526720347（测试123）

## 一、问题背景

### 1.1 问题描述

用户在项目 `proj_1776526720347` 的编辑页面中遇到三个关联问题：

1. **全局配置不可见**：配置面板中只看到了页面配置，没有看到全局配置区域
2. **宫格模式配置串扰**：修改页面标题后，宫格模式预览区所有页面都随之更新（预期：仅当前页面更新）
3. **页面切换配置重置**：切换页面时，之前修改的配置值被重置为默认值

### 1.2 预期行为

| 场景 | 预期行为 |
|:-----|:---------|
| 配置面板展示 | 当项目存在全局配置时，应同时显示全局配置区域和页面配置区域 |
| 修改页面级字段 | 仅影响当前选中页面的预览，其他页面不受影响 |
| 切换页面后切回 | 之前修改的配置值应保留，而非重置为默认值 |

### 1.3 实际行为

| 场景 | 实际行为 |
|:-----|:---------|
| 配置面板展示 | 仅显示页面配置区域，全局配置区域不可见 |
| 修改页面级字段 | 宫格模式下所有页面的相同字段同时更新 |
| 切换页面后切回 | 配置值被重置为 Schema default 值 |

## 二、根因分析

### 根因一：项目工作空间缺少 `project.config.schema.json` 文件

**现象**：配置面板中全局配置区域不显示。

**调查过程**：

编辑页面中全局配置区域的渲染受 `projectConfigSchema` 状态控制：

```tsx
// edit/page.tsx:1165
{projectConfigSchema && (
  <ConfigScopeWrapper scope="project">
    <ConfigForm ... />
  </ConfigScopeWrapper>
)}
```

`projectConfigSchema` 的值来源于多页面数据加载 API 返回的 `multi.projectConfigSchema`，该值由后端 `getProjectConfigSchema(wsPath)` 函数提供：

```typescript
// fs-utils.ts:1223-1229
export function getProjectConfigSchema(workspacePath: string): string | undefined {
  const filePath = getProjectConfigPath(workspacePath);
  if (!fs.existsSync(filePath)) return undefined;
  return fs.readFileSync(filePath, "utf-8");
}
```

该函数读取 `workspace/project.config.schema.json` 文件。经检查，项目工作空间目录下**不存在此文件**：

```
workspace/
├── .opencode/
├── demos/
│   ├── demo_1777965200000_x8k2p9/
│   ├── demo_1778077850198_fjxwmf/
│   └── demo_1778751411983_7h2rpo/
├── .folders.json
├── config.schema.json          ← 旧格式遗留文件，系统不读取
└── index.tsx
```

工作空间根目录下存在 `config.schema.json`，但这是旧单页面格式遗留文件，系统在新多页面格式下不再读取此文件作为全局配置。

**根本原因**：项目从未创建过 `project.config.schema.json` 文件，导致 `getProjectConfigSchema()` 返回 `undefined`，前端 `projectConfigSchema` 状态为 `undefined`，全局配置区域被条件渲染跳过。

**证据级别**：A级（直接代码证据）

**证据链**：

|:-----|:-----|:-----|
| 全局配置区域条件渲染 | [edit/page.tsx:1165](../../../../packages/author-site/src/app/demo/[id]/edit/page.tsx#L1165) | `{projectConfigSchema && (...)}` |

### 根因二：宫格模式下所有页面共享同一个 `configData` 状态

**现象**：修改页面标题后，宫格模式预览区所有页面都随之更新。

**调查过程**：

编辑页面使用单一的 `configData` 状态管理所有配置值：

```typescript
// edit/page.tsx:79
const [configData, setConfigData] = useState<Record<string, unknown>>({});
```

该状态同时包含项目级字段和页面级字段，通过 `mergeConfigToProps()` 合并。两个 `ConfigForm`（全局和页面）的 `onChange` 都合并到同一个 `configData`：

```tsx
// 全局配置 onChange
onChange={(data) => { setConfigData((prev) => ({ ...prev, ...data })); }}

// 页面配置 onChange
const handleConfigChange = useCallback((data) => {
  setConfigData((prev) => ({ ...prev, ...data }));
}, []);
```

在宫格模式中，`PreviewGrid` 组件将同一个 `configData` 传递给所有 `GridIframe`：

```tsx
// PreviewGrid.tsx:472-479
<GridIframe
  sessionId={sessionId}
  page={page}
  visible={visiblePages.has(page.id)}
  hasChanges={changedPageIds?.has(page.id) ?? false}
  configData={configData}    // ← 所有页面接收相同的 configData
  previewSize={effectiveSize}
/>
```

每个 `GridIframe` 在 `configData` 变化时向 iframe 发送 `UPDATE_CONFIG` 消息：

```typescript
// PreviewGrid.tsx:282-288
useEffect(() => {
  if (!iframeReadyRef.current || !blobUrlRef.current) return
  const iframe = iframeRef.current
  if (!iframe?.contentWindow) return
  const resolvedConfig = configData ? resolveImageUrls(configData) : {}
  iframe.contentWindow.postMessage({ type: "UPDATE_CONFIG", configData: resolvedConfig }, "*")
}, [configData])
```

**执行路径**：

```
用户修改"页面标题"字段
  → ConfigForm.onChange({ title: "新标题" })
  → setConfigData(prev => ({ ...prev, title: "新标题" }))
  → React 重渲染
  → PreviewGrid 接收新的 configData
  → 所有 GridIframe 接收新的 configData
  → 每个 GridIframe 发送 UPDATE_CONFIG 到各自的 iframe
  → 所有页面显示"新标题"
```

**根本原因**：系统使用单一扁平 `configData` 对象管理所有配置值，未按页面隔离页面级配置。宫格模式下，页面级字段（如 `title`）的修改会广播到所有页面的 iframe，导致配置串扰。

**证据级别**：A级（直接代码证据）

**证据链**：

| 证据 | 来源 | 说明 |
|:-----|:-----|:-----|
| 单一 configData 状态 | [edit/page.tsx:79](../../../../packages/author-site/src/app/demo/[id]/edit/page.tsx#L79) | `useState<Record<string, unknown>>({})` |
| 两个 ConfigForm 共享 configData | [edit/page.tsx:1170-1186](../../../../packages/author-site/src/app/demo/[id]/edit/page.tsx#L1170) | 全局和页面 onChange 都合并到同一状态 |
| PreviewGrid 传递同一 configData | [PreviewGrid.tsx:477](../../../../packages/author-site/components/demo/PreviewGrid.tsx#L477) | `configData={configData}` 对所有页面相同 |

### 根因三：页面切换时完全重置 `configData`，无按页面缓存机制

**现象**：切换页面时，之前的配置修改被重置为默认值。

**调查过程**：

所有页面切换入口（树形目录点击、下拉选择、宫格卡片点击）都执行相同的逻辑：

```typescript
// edit/page.tsx:874-886（树形目录点击）
onPageSelect={async (pageId) => {
  setActiveDemoId(pageId);
  const res = await fetch(`/api/sessions/${sessionId}/files/${pageId}`);
  const data = await res.json();
  if (data.success) {
    setCode(data.data.code);
    setSchema(data.data.schema);
    setEditorContent(buildFigmaText(data.data.code, data.data.schema));
    const defaults = getSafeMergedDefaults(data.data.schema);
    setConfigData(defaults);    // ← 完全重置为默认值
    // ...
  }
}}
```

`getSafeMergedDefaults` 从 Schema 的 `default` 字段提取默认值：

```typescript
// edit/page.tsx:367-383
const getSafeMergedDefaults = useCallback(
  (pageSchema: string) => {
    try {
      return mergeConfigToProps(projectConfigSchema, pageSchema);
    } catch (err) {
      if (err instanceof SchemaConflictError) { toast(...); }
      return getDefaultValues(pageSchema);
    }
  },
  [projectConfigSchema],
);
```

系统设计文档明确说明不持久化 `configData`：

> 系统不持久化 `configData`，每次进入编辑页都靠 `getDefaultValues(loadedSchema)` 从 Schema 的 `default` 字段实时推导。
> — [项目级与页面级配置设计.md §7](./已完成/项目级与页面级配置设计.md)

**执行路径**：

```
用户在页面A修改了 title = "自定义标题"
  → configData = { title: "自定义标题", subtitle: "...", ... }

用户切换到页面B
  → setActiveDemoId("pageB")
  → fetch 新页面的 code + schema
  → getSafeMergedDefaults(pageBSchema)
  → setConfigData({ title: "✨ 炫彩世界", subtitle: "探索色彩的无限可能", ... })
  → 页面A的自定义配置丢失

用户切换回页面A
  → 同样重置为默认值，"自定义标题"已丢失
```

**根本原因**：系统未实现按页面缓存配置数据的机制。`configData` 是单一状态对象，页面切换时被完全替换为新页面的 Schema 默认值，用户之前的修改无法保留。

**证据级别**：A级（直接代码证据）

**证据链**：

| 证据 | 来源 | 说明 |
|:-----|:-----|:-----|
| 树形目录点击重置 | [edit/page.tsx:885-886](../../../../packages/author-site/src/app/demo/[id]/edit/page.tsx#L885) | `setConfigData(defaults)` |
| 下拉选择重置 | [edit/page.tsx:1056-1057](../../../../packages/author-site/src/app/demo/[id]/edit/page.tsx#L1056) | `setConfigData(defaults)` |

| 设计文档确认不持久化 | [项目级与页面级配置设计.md:247](./已完成/项目级与页面级配置设计.md) | "系统不持久化 configData" |

## 三、根因关系图

三个根因之间存在递进关系：

```
根因一：缺少 project.config.schema.json
  │
  └→ 全局配置区域不显示（独立问题）

根因二：单一 configData 状态，无页面隔离
  │
  ├→ 宫格模式配置串扰（直接表现）
  │
  └→ 根因三：无按页面缓存机制
       │
       └→ 页面切换配置重置（直接表现）
```

根因二和根因三共享同一个底层设计缺陷：**配置数据模型未按页面维度隔离**。当前架构将所有配置值存储在单一扁平对象中，既无法在宫格模式下按页面独立传递配置，也无法在页面切换时按页面缓存/恢复配置。

## 四、解决方案

### 方案一：创建 `project.config.schema.json`（解决根因一）

**做法**：在工作空间目录下创建 `project.config.schema.json` 文件，将需要跨页面共享的配置字段定义在其中。

**为何有效**：`getProjectConfigSchema()` 检测到文件存在后会返回其内容，前端 `projectConfigSchema` 不再为 `undefined`，全局配置区域将被渲染。

**影响范围**：仅影响该项目，需确保项目级字段与页面级字段不重名（系统已有冲突校验）。

**复杂度**：低

**风险**：需手动确保项目级 Schema 中的字段名与所有页面级 Schema 不冲突。

---

### 方案二：按页面隔离 configData（解决根因二和根因三）

**做法**：将单一 `configData` 状态改为按页面 ID 索引的 Map 结构：

```typescript
// 改造前
const [configData, setConfigData] = useState<Record<string, unknown>>({});

// 改造后
const [configDataMap, setConfigDataMap] = useState<Record<string, Record<string, unknown>>>({});
const activeConfigData = configDataMap[activeDemoId] ?? {};
```

关键改造点：

1. **编辑页面**：`configDataMap` 按 `activeDemoId` 索引，页面切换时从 Map 中读取对应页面的缓存数据，而非重置为默认值
2. **PreviewGrid**：每个 `GridIframe` 接收对应页面的 `configData`，而非全局共享的同一对象
3. **ConfigForm onChange**：更新时写入 `configDataMap[activeDemoId]`，而非全局 `configData`
4. **页面切换**：若目标页面在 Map 中有缓存则使用缓存，否则从 Schema 默认值初始化

**为何有效**：
- 宫格模式下每个 iframe 接收独立的配置数据，修改页面A的标题不会影响页面B
- 页面切换时从 Map 缓存恢复配置，而非重新从 Schema 默认值计算

**影响范围**：
- `edit/page.tsx`：configData 状态管理逻辑
- `PreviewGrid.tsx`：configData 传递方式
- `viewer/[projectId]/page.tsx`：同上
- `viewer/[projectId]/[demoId]/page.tsx`：同上

**复杂度**：中

**风险**：
- 项目级配置的修改仍需广播到所有页面（项目级字段应共享，页面级字段应隔离），需在 `onChange` 中区分字段归属
- 需处理项目级 Schema 变更时的缓存失效逻辑

---

### 方案三：持久化配置数据（远期方案，解决根因三的深层问题）

**做法**：在页面目录下新增 `config.data.json` 文件，持久化用户修改的配置值。

**为何有效**：配置值不再仅依赖 Schema default 推导，刷新页面或重新进入编辑页后可恢复用户修改。

**影响范围**：需新增 API 端点和文件读写逻辑，改动较大。

**复杂度**：高

**风险**：Schema 变更时需处理字段增删与持久化数据的兼容性。

**建议**：作为远期优化，当前阶段先实施方案二（内存级缓存）即可解决页面切换重置问题。
