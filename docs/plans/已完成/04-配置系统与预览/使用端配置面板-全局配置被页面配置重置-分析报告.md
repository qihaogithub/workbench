# 使用端配置面板：全局配置被页面配置重置 — 分析报告

> **日期**: 2026-05-26
> **类型**: Bug 分析 / 数据流缺陷
> **状态**: 待修复

---

## 一、问题背景

### 问题描述

在使用端（Viewer）项目详情页的配置面板中，按以下步骤操作会导致全局配置（即项目级配置）的修改丢失：

1. 先修改 **全局配置**（如修改 `bgColor`）→ 预览正常更新
2. 再修改 **页面配置**（如修改 `titleColor`）→ 预览中 `bgColor` 回退到修改前的值

创作端（Author）项目编辑页虽然结构类似，但用户报告不存在此问题。

### 预期行为

全局配置和页面配置的修改应该**互不干扰**。修改页面配置后，全局配置的修改应保持不变。

### 实际行为

页面配置的修改会**连带覆盖**全局配置的值，使全局配置的修改丢失。

### 数据流追踪

#### Viewer 项目页的配置面板渲染（`viewer/[projectId]/page.tsx:459-484`）

```tsx
{/* 全局配置表单 */}
<ConfigForm
  schema={data.projectConfigSchema}
  onChange={handleProjectConfigChange}
  initialData={configData}   // ← 共享同一个 configData
/>

{/* 页面配置表单 */}
<ConfigForm
  schema={activePageSchema}
  onChange={handleConfigChange}
  initialData={configData}   // ← 共享同一个 configData
/>
```

两个表单使用同一个 `configData` state 作为 `initialData`。`configData` 是项目级和页面级配置的**扁平合并**（由 `mergeConfigToProps` 合并，字段名无重叠）。

#### ConfigForm 内部状态管理（`ConfigFormNew.tsx:797-913`）

```tsx
const [formData, setFormData] = useState<Record<string, unknown>>(initialData || {});

// initialData 同步：仅添加新字段，不更新现有字段
useEffect(() => {
  if (initialData && Object.keys(initialData).length > 0) {
    setFormData((prev) => {
      const merged = { ...prev };
      for (const [key, value] of Object.entries(initialData)) {
        if (!(key in merged)) {   // ← 关键：仅当 key 不存在时才添加
          merged[key] = value;
        }
      }
      return merged;
    });
  }
}, [initialData]);

// 每次字段变更时，发送整个 formData 给父组件
const handleFieldChange = (key: string, value: unknown) => {
  const newData = { ...formDataRef.current, [key]: value };
  setFormData(newData);
  onChange(newData);   // ← 发送的是完整的 formData，不是 {key: value}
};
```

关键发现：
1. `handleFieldChange` 发送**完整 formData** 而非仅变更字段（第 908-910 行）
2. `initialData` 同步 effect **只添加新字段**，不更新已有字段（第 854-856 行）

#### Viewer 父组件的事件处理

```tsx
// 页面配置变更处理（line 234-244）
const handleConfigChange = (newData: Record<string, unknown>) => {
  setConfigData((prev) => {
    const merged = { ...prev, ...newData };  // ← 浅合并，会覆盖全局配置
    postOutgoing({ type: "VIEWER_CONFIG_CHANGE", configData: merged });
    return merged;
  });
};

// 全局配置变更处理（line 246-259）
const handleProjectConfigChange = (newData: Record<string, unknown>) => {
  setConfigData((prev) => {
    const merged = { ...prev, ...newData };
    postOutgoing({ type: "VIEWER_CONFIG_CHANGE", configData: merged });
    return merged;
  });
  // 广播到所有页面...
};
```

## 三、根因分析

### 根因：三处缺陷共同作用

#### 缺陷 ①：`handleFieldChange` 发送完整 formData（`ConfigFormNew.tsx:908-910`）

每次字段变更时，`onChange(newData)` 发送的是**整个表单数据**，而非仅变更的 `{ key: value }`。这意味着：

- 页面 ConfigForm 的 `formData` 包含**所有字段**（项目级 + 页面级）
- 当表单有 N 个字段时，发送的是 N 个键值对，而非 1 个

#### 缺陷 ②：`initialData` 同步 effect 不更新已有字段（`ConfigFormNew.tsx:854-856`）

```tsx
if (!(key in merged)) {  // 仅添加新字段
  merged[key] = value;
}
```

当全局配置修改后，`initialData` 更新了，但页面 ConfigForm 的 `formData` 中**项目级字段仍然保持旧值**。

#### 缺陷 ③：父组件的 `handleConfigChange` 做浅合并（`viewer/[projectId]/page.tsx:236`）

```tsx
const merged = { ...prev, ...newData };
```

`newData` 来自页面 ConfigForm 的完整 `formData`，其中项目级字段是旧的默认值，因此浅合并会将全局配置的修改覆盖掉。

### 复现流程（逐步追踪）

假设：
- 项目 schema: `{ bgColor: { type: "string", default: "#fff" } }`
- 页面 schema: `{ titleColor: { type: "string", default: "#000" } }`
- `configData` 初始: `{ bgColor: "#fff", titleColor: "#000" }`

| 步骤 | 操作 | 页面 ConfigForm `formData` | `configData` |
|------|------|---------------------------|-------------|
| 0 | 初始加载 | `{ bgColor: "#fff", titleColor: "#000" }` | `{ bgColor: "#fff", titleColor: "#000" }` |
| 1 | 修改全局 bgColor → "#333" | `{ bgColor: "#fff", titleColor: "#000" }`（未更新！） | `{ bgColor: "#333", titleColor: "#000" }` |
| 2 | 修改页面 titleColor → "#f00" | `onChange({ bgColor: "#fff", titleColor: "#f00" })` | `{ ...prev, ...newData }` = `{ bgColor: **"#fff"**, titleColor: "#f00" }` |

**步骤 2 中，全局修改的 `bgColor: "#333"` 被页面表单的旧值 `bgColor: "#fff"` 覆盖！**

### 为什么创作端不受影响

创作端编辑页（`edit/page.tsx`）不存在 `configData` state，而是从 `configDataMap[activeDemoId]` 派生。但**实际上同样的缺陷在创作端也存在**，区别在于：

1. **创作端不依赖 `configData` 做实时预览** — 预览数据走 `configDataMap`，有页面级隔离
2. **创作端项目配置 onChange 不上传 `configData`** — 仅更新 `configDataMap`
3. 但如果在创作端反复切换页面，`configDataMap[pageId]` 同样会被页面 ConfigForm 的完整 formData 污染

### 证据级别

| 证据 | 类型 | 来源 |
|------|------|------|
| `handleFieldChange` 发送完整 formData | **A** | `ConfigFormNew.tsx:908-910` |
| `initialData` 同步 effect 不更新已有字段 | **A** | `ConfigFormNew.tsx:854-856` |
| Viewer `handleConfigChange` 浅合并覆盖全局值 | **A** | `viewer/[projectId]/page.tsx:236` |
| 页面 ConfigForm 不感知全局配置变更 | **A** | `ConfigFormNew.tsx:854-856` |

## 四、解决方案

### 方案一（推荐）：`handleFieldChange` 仅发送变更字段

**做法**: 将 `handleFieldChange` 的 `onChange` 参数改为仅发送变更的键值对。

```tsx
const handleFieldChange = (key: string, value: unknown) => {
  const newData = { ...formDataRef.current, [key]: value };
  setFormData(newData);
  onChange({ [key]: value });  // ← 改为仅发送变更字段
};
```

**优点**:
- 改动最小，仅 1 行
- 父组件的 merge 逻辑 `{ ...prev, ...newData }` 自然生效
- `initialData` 同步 effect 的 `!key in merged` 逻辑不再有问题
- 影响范围可控，不修改组件接口

**缺点**:
- 改变了 ConfigForm 的 `onChange` 回调语义。其他调用方（创作端、嵌入端等）原本期望 `onChange` 返回完整 formData，需要排查所有 `onChange` 调用方是否依赖完整 formData

**需要排查的调用方**:
- `viewer/[projectId]/page.tsx:464` — `handleProjectConfigChange`（仅依赖变更字段 ✓）
- `viewer/[projectId]/page.tsx:480` — `handleConfigChange`（仅依赖变更字段 ✓）
- `edit/page.tsx:1329` — 项目配置 onChange（`...data` 广播，仅依赖变更字段 ✓）
- `edit/page.tsx:1358` — 页面配置 onChange（仅依赖变更字段 ✓）
- `embed/[demoId]/EmbedConfigPanel.tsx` — 需要验证

### 方案二：`initialData` 同步 effect 改为合并所有字段

**做法**: 将 `initialData` 同步 effect 改为覆盖所有字段，而非仅添加新字段。

```tsx
useEffect(() => {
  if (initialData && Object.keys(initialData).length > 0) {
    setFormData((prev) => ({ ...prev, ...initialData }));
  }
}, [initialData]);
```

**优点**:
- 页面 ConfigForm 的 `formData` 项目级字段始终与 `configData` 同步
- `handleFieldChange` 发送完整 formData 时，项目级字段是最新的

**缺点**:
- 同步 effect 会覆盖用户正在编辑的字段值（如果 `initialData` 在用户编辑过程中被外部修改）
- 可能引入闪烁或输入抖动问题
- 对高频输入场景不够友好

### 方案三：抽离共享的 `configData`，按作用域隔离

**做法**: 在 Viewer 页面中，分别维护项目 `configData` 和页面 `configData`，不再共享扁平化合并对象。

**优点**: 架构上最干净，完全隔离两个作用域
**缺点**: 改动量大，涉及 `PreviewPanel` props、`postMessage` 消息格式等

### 推荐：方案一

在确认所有 `onChange` 回调调用方不依赖完整 formData 后，采用方案一。这是最小的侵入式改动，直接切断根源：页面 ConfigForm 的 `onChange` 不再发送项目级字段，自然不会被错误覆盖。
