## 一、问题背景

### 1.1 现状描述

在 author-site 项目编辑页中,预览区首次加载需要 **1-2 秒** 才能显示预览效果,主要耗时集中在:

- CDN 依赖下载(React、ReactDOM、第三方库等)
- iframe 初始化和模块加载
- 开发环境缓存未命中

### 1.2 架构瓶颈分析

当前架构采用**服务端编译 + iframe CDN 渲染**模式,加载流程为:

```
用户进入编辑页
  ↓
调用 /api/compile (sucrase 编译 TSX → ESM, ~50ms)
  ↓
iframe 加载 Blob URL 模板
  ↓
iframe 发送 READY 消息
  ↓
父窗口发送 UPDATE_CODE 消息
  ↓
iframe dynamic import() 加载编译后模块
  ↓
【瓶颈】从 esm.sh CDN 下载依赖 (200-500ms/个)
  ↓
组件渲染 (~50-100ms)
```

**核心问题**:

1. CDN 依赖下载是串行阻塞操作,首次冷启动无缓存
2. esm.sh 是海外 CDN,国内访问延迟高
3. 开发环境频繁刷新,浏览器缓存命中率低
4. 依赖版本未锁定时可能触发额外的版本解析

### 1.3 预览模式切换性能问题

**新增发现**:不仅在首次加载时存在性能瓶颈,**预览模式切换**(单页面 ↔ 宫格模式)时也会导致预览区刷新,体验不佳。

#### 1.3.1 单页面模式(PreviewPanel)

**组件生命周期**:

- iframe 实例在组件挂载时创建一次(`useEffect([], [])`),生成 Blob URL
- 代码变更通过 `postMessage` 发送 `UPDATE_CODE` 消息,**不重新加载 iframe**
- 配置变更通过 `postMessage` 发送 `UPDATE_CONFIG` 消息,**不重新加载 iframe**

**性能特征**:

- ✅ iframe 实例复用,避免重复初始化
- ✅ 依赖通过浏览器缓存命中,二次加载快
- ⚠️ 首次加载仍需下载所有 CDN 依赖

#### 1.3.2 宫格模式(PreviewGrid)

**组件生命周期**:

- 每个页面卡片对应一个 `GridIframe` 子组件
- iframe 实例在**可见性变化时**重新创建(`useEffect([visible, sessionId, page.id, page.code, snapshotVersion])`)
- 不可见时销毁 Blob URL(`URL.revokeObjectURL`),可见时重新编译和创建

**性能特征**:

- ⚠️ **可见性切换导致 iframe 重建**:从不可见变为可见时,需要重新编译和创建 Blob URL
- ⚠️ **编译缓存有 TTL 限制**:5 分钟过期(`CACHE_TTL = 5 * 60 * 1000`),过期后需重新请求 `/api/compile`
- ⚠️ **Blob URL 不持久化**:每次可见性变化都重新生成,无法复用
- ⚠️ **模式切换时所有卡片重新渲染**:单页面 → 宫格模式切换时,所有卡片的 `visible` 状态重新计算,可能触发批量编译

#### 1.3.3 模式切换性能瓶颈分析

**场景 1:单页面 → 宫格模式**

```
previewMode 从 'single' 变为 'grid'
  ↓
PreviewGrid 组件挂载
  ↓
IntersectionObserver 开始观察所有卡片
  ↓
visiblePages 状态更新(当前视口 + 缓冲区)
  ↓
每个可见卡片的 GridIframe 组件:
  - 检查编译缓存(可能命中或过期)
  - 如果缓存未命中:调用 /api/compile(~50-100ms)
  - 生成新的 Blob URL(~1ms)
  - iframe 加载 Blob URL(~50ms)
  - CDN 依赖下载(首次 200-500ms,缓存 0-50ms)
  ↓
【瓶颈】多个卡片并行加载,总耗时 = max(各卡片耗时)
```

**场景 2:宫格模式 → 单页面模式**

```
previewMode 从 'grid' 变为 'single'
  ↓
PreviewGrid 组件卸载
  ↓
所有 GridIframe 组件卸载
  ↓
清理所有 Blob URL(URL.revokeObjectURL)
  ↓
PreviewPanel 组件挂载(如果是首次进入)
  ↓
创建新的 Blob URL 和 iframe 实例
  ↓
如果编译缓存已过期,需重新编译
  ↓
【瓶颈】单页面模式 iframe 重新初始化
```

**场景 3:宫格模式下滚动**

```
用户滚动宫格容器
  ↓
IntersectionObserver 触发
  ↓
新进入视口的卡片:visible = true
  ↓
GridIframe 组件:
  - 如果编译缓存命中:直接挂载 iframe(~50ms)
  - 如果编译缓存过期:重新编译 + 挂载(~100-600ms)
  ↓
离开视口的卡片:visible = false
  ↓
GridIframe 组件:
  - 销毁 Blob URL
  - 显示占位符
  ↓
【瓶颈】频繁滚动时反复编译和创建/销毁 Blob URL
```

#### 1.3.4 关键性能问题总结

| 问题            | 影响场景           | 耗时            | 根因             |
| --------------- | ------------------ | --------------- | ---------------- |
| CDN 依赖下载    | 所有首次加载       | 200-500ms/依赖  | esm.sh 海外延迟  |
| 编译缓存过期    | 宫格模式可见性切换 | 50-100ms        | 5 分钟 TTL 限制  |
| Blob URL 重建   | 宫格模式可见性切换 | ~50ms           | 不可见时主动销毁 |
| iframe 实例重建 | 模式切换           | 100-200ms       | 组件卸载/挂载    |
| 批量编译阻塞    | 宫格模式首次渲染   | max(各卡片耗时) | 并行编译无节流   |

## 二、优化目标

### 2.1 性能指标

| 场景                  | 当前耗时      | 优化目标    | 说明             |
| --------------------- | ------------- | ----------- | ---------------- |
| 首次冷启动            | 1-2s          | **< 800ms** | 依赖首次下载     |
| 二次加载(缓存命中)    | 500-800ms     | **< 300ms** | 浏览器缓存生效   |
| 代码变更热更新        | 200-500ms     | **< 150ms** | 仅更新编译后代码 |
| **单页面 → 宫格模式** | **200-600ms** | **< 50ms**  | **iframe 复用**  |
| **宫格模式滚动**      | **100-300ms** | **< 10ms**  | **CSS 隐藏**     |

### 2.2 优化原则

- ✅ **不改变现有架构**:保持服务端编译 + iframe 沙箱隔离
- ✅ **渐进式优化**:优先低成本高收益方案
- ✅ **开发体验优先**:不影响开发环境的实时预览体验
- ✅ **向后兼容**:所有优化可降级,不影响生产环境

## 三、优化方案

### 方案 1:切换国内 CDN 镜像(推荐优先实施)

**优先级**:⭐⭐⭐⭐⭐  
**实施难度**:⭐  
**预期收益**:降低 CDN 延迟 50-70%

#### 3.1.1 方案描述

将 `CDN_BASE_URL` 从 `https://esm.sh` 切换为国内镜像源,减少网络延迟。

#### 3.1.2 可选 CDN 源

| CDN 源             | 地址                      | 优势           | 风险                  |
| ------------------ | ------------------------- | -------------- | --------------------- |
| **esm.sh(官方)**   | `https://esm.sh`          | 最新版本、稳定 | 海外节点,延迟高       |
| **unpkg 中国镜像** | `https://unpkg.zhimg.com` | 国内快速       | 不支持 ESM import map |
| **自定义代理**     | 内网部署 esm.sh 代理      | 完全可控       | 需维护基础设施        |
| **Cloudflare CDN** | 通过 CF 加速 esm.sh       | 全球加速       | 需配置 CDN 规则       |

**推荐方案**:在 `.env` 中配置国内可访问的 CDN 代理或镜像

#### 3.1.3 实施步骤

1. **修改环境变量**(`.env` 文件):

   ```bash
   # 使用国内 CDN 镜像(示例)
   CDN_BASE_URL=https://your-cdn-proxy.example.com
   ```

2. **验证兼容性**:
   - 确保 CDN 支持 ESM 格式
   - 验证 `?deps` 参数正常工作
   - 测试 React 单例机制不受影响

3. **回退机制**:
   - 保留默认值 `https://esm.sh` 作为降级方案
   - CDN 不可用时自动回退到官方源

#### 3.1.4 风险评估

- ⚠️ 国内镜像可能版本滞后
- ⚠️ 需定期同步最新版本
- ⚠️ 某些第三方库可能不在镜像中

### 方案 2:预加载核心依赖(高收益)

**优先级**:⭐⭐⭐⭐⭐  
**实施难度**:⭐⭐  
**预期收益**:首次加载提速 40-60%

#### 3.2.1 方案描述

在父窗口(author-site)提前预加载 React、ReactDOM 等核心依赖,利用浏览器 HTTP 缓存,使 iframe 加载时直接从缓存读取。

#### 3.2.2 实施策略

**策略 A:隐式预加载(推荐)**

在编辑页组件挂载时,通过 `<link rel="preload">` 或动态 `<script>` 预取核心依赖:

```typescript
// packages/author-site/components/demo/PreviewPanel.tsx

useEffect(() => {
  const cdnBase = getCdnBaseUrl();
  const coreDeps = [
    `${cdnBase}/react@18.3.1`,
    `${cdnBase}/react-dom@18.3.1/client`,
  ];

  // 预加载脚本到浏览器缓存
  coreDeps.forEach((url) => {
    const link = document.createElement("link");
    link.rel = "preload";
    link.as = "script";
    link.href = url;
    link.crossOrigin = "anonymous";
    document.head.appendChild(link);
  });

  return () => {
    // 清理预加载标签
    coreDeps.forEach((url) => {
      const link = document.querySelector(`link[href="${url}"]`);
      link?.remove();
    });
  };
}, []);
```

**策略 B:Service Worker 缓存**

注册 Service Worker 拦截 CDN 请求,实现持久化缓存:

```javascript
// packages/author-site/public/sw.js
self.addEventListener("fetch", (event) => {
  if (event.request.url.includes("esm.sh")) {
    event.respondWith(
      caches.match(event.request).then((response) => {
        return (
          response ||
          fetch(event.request).then((fetchResponse) => {
            return caches.open("cdn-cache").then((cache) => {
              cache.put(event.request, fetchResponse.clone());
              return fetchResponse;
            });
          })
        );
      }),
    );
  }
});
```

#### 3.2.3 预期效果

| 阶段          | 优化前    | 优化后            |
| ------------- | --------- | ----------------- |
| React 下载    | 200-500ms | **0ms(缓存命中)** |
| ReactDOM 下载 | 200-500ms | **0ms(缓存命中)** |
| 总耗时        | 1-2s      | **400-800ms**     |

#### 3.2.4 风险评估

- ⚠️ 预加载可能浪费带宽(用户未进入编辑页)
- ⚠️ Service Worker 需处理缓存失效逻辑
- ✅ 可通过懒加载策略优化(进入编辑页后再预加载)

### 方案 3:依赖版本锁定优化(中收益)

**优先级**:⭐⭐⭐⭐  
**实施难度**:⭐⭐  
**预期收益**:减少重复解析延迟 100-200ms

#### 3.3.1 问题分析

当前依赖锁定是**异步非阻塞**的,导致:

- 首次编译时依赖未锁定,每次可能解析到不同版本
- `resolveDependencyVersions()` 需要发 HEAD 请求获取重定向 URL
- 锁定完成前的编译无法享受锁定带来的性能优势

#### 3.3.2 优化方案

**方案 A:提前锁定依赖(推荐)**

在项目创建或导入时,预先分析代码中的依赖并锁定:

```typescript
// packages/author-site/src/lib/fs-utils.ts

export async function preLockDependencies(projectId: string, code: string) {
  const dependencies = extractImports(code);
  const npmDeps = dependencies.filter((dep) => isNpmPackage(dep));

  // 同步解析版本(阻塞但只执行一次)
  const lockedDeps = await resolveDependencyVersions(npmDeps);

  const project = readProjectMeta(projectId);
  project.lockedDependencies = {
    ...project.lockedDependencies,
    ...lockedDeps,
  };
  writeProjectMeta(projectId, project);
}
```

**触发时机**:

- 项目首次创建时
- AI 生成代码后
- 用户手动点击"锁定依赖"按钮

**方案 B:缓存依赖解析结果**

将 `resolveDependencyVersions()` 的结果缓存到服务端,避免重复 HEAD 请求:

```typescript
// packages/author-site/src/lib/compiler.ts

const dependencyVersionCache = new Map<string, string>();

export async function resolveDependencyVersion(packageName: string) {
  if (dependencyVersionCache.has(packageName)) {
    return dependencyVersionCache.get(packageName);
  }

  const url = `${ESM_SH_BASE}/${packageName}`;
  const response = await fetch(url, { method: "HEAD", redirect: "manual" });
  const versionUrl = response.headers.get("location") || url;

  dependencyVersionCache.set(packageName, versionUrl);
  return versionUrl;
}
```

#### 3.3.3 预期效果

| 场景             | 优化前               | 优化后              |
| ---------------- | -------------------- | ------------------- |
| 首次编译(未锁定) | 50ms + HEAD 请求延迟 | **50ms + 缓存命中** |
| 二次编译(已锁定) | 50ms                 | **50ms(无变化)**    |
| 依赖解析总耗时   | 100-300ms            | **< 50ms**          |

### 方案 4:宫格模式 iframe 实例复用(高收益)

**优先级**:⭐⭐⭐⭐⭐  
**实施难度**:⭐⭐⭐  
**预期收益**:模式切换和滚动时减少 80-90% 的 iframe 重建开销

#### 3.4.1 问题分析

当前 `GridIframe` 组件在 `visible` 状态变化时会:

1. **销毁阶段**(`visible = false`):

   ```typescript
   if (!visible) {
     if (blobUrlRef.current) {
       URL.revokeObjectURL(blobUrlRef.current); // 销毁 Blob URL
       blobUrlRef.current = null;
     }
     iframeReadyRef.current = false;
     return;
   }
   ```

2. **重建阶段**(`visible = true`):
   - 检查编译缓存(可能过期)
   - 重新调用 `/api/compile`(如果缓存过期)
   - 生成新的 Blob URL
   - 创建新的 iframe 实例
   - 等待 iframe 加载和 CDN 依赖下载

**核心问题**:

- Blob URL 销毁后无法恢复,必须重新生成
- iframe 实例重建导致完整的加载流程
- 频繁滚动时反复创建/销毁,性能开销大

#### 3.4.2 优化方案

**方案 A:隐藏而非销毁(推荐)**

使用 CSS `display: none` 或 `visibility: hidden` 隐藏不可见的 iframe,而非销毁:

```typescript
// packages/author-site/components/demo/PreviewGrid.tsx

function GridIframe({ visible, ... }) {
  const iframeRef = useRef<HTMLIFrameElement>(null);
  const blobUrlRef = useRef<string | null>(null);
  const [isVisible, setIsVisible] = useState(visible);

  // 首次加载时创建 Blob URL(仅一次)
  useEffect(() => {
    if (!blobUrlRef.current) {
      const html = generateIframeHtml({ ... });
      const blob = new Blob([html], { type: 'text/html' });
      blobUrlRef.current = URL.createObjectURL(blob);
    }
  }, []);

  // visible 变化时只切换 CSS,不销毁
  useEffect(() => {
    setIsVisible(visible);
  }, [visible]);

  // 组件卸载时才清理
  useEffect(() => {
    return () => {
      if (blobUrlRef.current) {
        URL.revokeObjectURL(blobUrlRef.current);
        blobUrlRef.current = null;
      }
    };
  }, []);

  if (!blobUrlRef.current) {
    return <div>加载中...</div>;
  }

  return (
    <div style={{ display: isVisible ? 'block' : 'none' }}>
      <iframe src={blobUrlRef.current} ref={iframeRef} />
    </div>
  );
}
```

**优势**:

- ✅ Blob URL 只创建一次,模式切换时复用
- ✅ iframe 实例保持挂载,避免重复加载
- ✅ CDN 依赖通过浏览器缓存命中,无需重新下载
- ✅ 滚动时仅切换 CSS,性能开销极低

**劣势**:

- ⚠️ 不可见的 iframe 仍占用内存(可通过限制可见卡片数量缓解)
- ⚠️ 大量页面时内存占用增加(需评估实际影响)

**方案 B:延迟销毁策略**

引入延迟销毁机制,避免频繁创建/销毁:

```typescript
function GridIframe({ visible, ... }) {
  const destroyTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
  const blobUrlRef = useRef<string | null>(null);

  useEffect(() => {
    if (visible) {
      // 可见时取消销毁计划
      if (destroyTimerRef.current) {
        clearTimeout(destroyTimerRef.current);
        destroyTimerRef.current = null;
      }

      // 如果没有 Blob URL,创建它
      if (!blobUrlRef.current) {
        createBlobUrl();
      }
    } else {
      // 不可见时延迟销毁(例如 30 秒)
      destroyTimerRef.current = setTimeout(() => {
        if (blobUrlRef.current) {
          URL.revokeObjectURL(blobUrlRef.current);
          blobUrlRef.current = null;
        }
      }, 30000); // 30 秒延迟
    }

    return () => {
      if (destroyTimerRef.current) {
        clearTimeout(destroyTimerRef.current);
      }
    };
  }, [visible]);
}
```

**优势**:

- ✅ 短期不可见时保留 Blob URL,避免频繁重建
- ✅ 长期不可见时释放内存,防止内存泄漏
- ✅ 平衡性能和内存占用

**劣势**:

- ⚠️ 延迟时间需要调优(太短无效,太长浪费内存)
- ⚠️ 仍需处理编译缓存过期问题

#### 3.4.3 预期效果

| 场景              | 优化前         | 优化后           |
| ----------------- | -------------- | ---------------- |
| 单页面 → 宫格模式 | 200-600ms      | **< 50ms**       |
| 宫格模式滚动      | 100-300ms/卡片 | **< 10ms**       |
| 宫格 → 单页面     | 100-200ms      | **< 50ms**       |
| 内存占用          | 低             | 中(增加 20-50MB) |

#### 3.4.4 风险评估

- ⚠️ 内存占用增加:每个 iframe 约 5-10MB,10 个页面约 50-100MB
- ⚠️ 需监控内存泄漏:确保组件卸载时正确清理
- ✅ 可添加最大缓存数量限制(例如最多保留 15 个 iframe)

### 方案 5:编译缓存持久化(辅助优化)

**优先级**:⭐⭐⭐  
**实施难度**:⭐⭐  
**预期收益**:减少编译缓存过期后的重新编译延迟

#### 3.5.1 增强客户端编译缓存

当前缓存策略(`compile-cache.ts`):

```typescript
const MAX_CACHE_SIZE = 20;
const CACHE_TTL = 5 * 60 * 1000; // 5 分钟
```

**问题**:

- 缓存仅存在于内存,刷新页面后丢失
- 5 分钟 TTL 过短,宫格模式下容易过期
- 缓存 key 基于 `sessionId:demoId`,相同代码在不同 session 中无法复用

**优化**:

1. **延长 TTL 或取消过期**:

   ```typescript
   const CACHE_TTL = 30 * 60 * 1000; // 延长到 30 分钟
   // 或基于代码 hash 而非时间
   ```

2. **基于代码 hash 的缓存**:

   ```typescript
   // 使用代码内容的 hash 作为缓存 key
   function buildCacheKey(code: string): string {
     return createHash("md5").update(code).digest("hex");
   }

   // 相同代码在不同 session 中共享缓存
   ```

3. **持久化到 localStorage**:

   ```typescript
   export function setCachedCompile(
     sessionId: string,
     demoId: string,
     result: CompileResult,
   ) {
     const key = buildKey(sessionId, demoId);
     compileCache.set(key, { result, timestamp: Date.now() });

     // 同时保存到 localStorage
     try {
       localStorage.setItem(`compile-cache:${key}`, JSON.stringify(result));
     } catch {
       // 忽略存储失败
     }
   }

   export function getCachedCompile(
     sessionId: string,
     demoId: string,
   ): CompileResult | null {
     const key = buildKey(sessionId, demoId);
     const cached = compileCache.get(key);

     if (cached) {
       if (Date.now() - cached.timestamp > CACHE_TTL) {
         compileCache.delete(key);
         return null;
       }
       return cached.result;
     }

     // 尝试从 localStorage 恢复
     try {
       const stored = localStorage.getItem(`compile-cache:${key}`);
       if (stored) {
         const result = JSON.parse(stored);
         compileCache.set(key, { result, timestamp: Date.now() });
         return result;
       }
     } catch {
       // 忽略解析失败
     }

     return null;
   }
   ```

#### 3.5.2 增强服务端编译缓存

当前服务端缓存(`compiler.ts`):

```typescript
const compileCache = new Map<string, CompileResult>();
const MAX_CACHE_SIZE = 100;
```

**优化**:

1. **持久化缓存**:将缓存写入磁盘,重启后保留
2. **跨 Session 共享**:相同代码片段在不同项目中复用编译结果
3. **LRU 淘汰策略**:替换当前的 FIFO 策略

```typescript
import fs from "fs";
import path from "path";

const CACHE_DIR = path.join(process.cwd(), ".compile-cache");

export function getFromCache(codeHash: string): CompileResult | null {
  const cacheFile = path.join(CACHE_DIR, `${codeHash}.json`);
  if (fs.existsSync(cacheFile)) {
    return JSON.parse(fs.readFileSync(cacheFile, "utf-8"));
  }
  return null;
}

export function saveToCache(codeHash: string, result: CompileResult) {
  if (!fs.existsSync(CACHE_DIR)) {
    fs.mkdirSync(CACHE_DIR, { recursive: true });
  }
  const cacheFile = path.join(CACHE_DIR, `${codeHash}.json`);
  fs.writeFileSync(cacheFile, JSON.stringify(result));
}
```

| -------------------- | ----------------------- | ------------------- |
