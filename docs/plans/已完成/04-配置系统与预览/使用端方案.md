# 使用端方案

> 为系统新增"使用者"角色视图。采用**独立站点 + API 调用**的架构，使用端与创作端完全解耦，实现高可维护性，同时便于嵌入其他系统。

> **📝 名称变更（已完成）**：创作端包名由 `packages/web` 更名为 `packages/author-site`，npm 包名由 `@opencode-workbench/web` 更名为 `@opencode-workbench/author-site`。

---

## 一、背景与目标

### 1.1 用户角色定义

| 角色 | 权限 | 对应界面 |
|------|------|---------|
| **编辑者 (Editor)** | 创建项目、AI 对话编辑代码、管理版本 | 现有 Web 端 (`/projects/`) |
| **使用者 (Viewer)** | 浏览项目列表、查看预览、调整配置参数 | **独立的使用端** |
| **开发者 (Developer)** | 系统管理、扩展开发 | 后续规划 |

### 1.2 核心需求

- 使用端是一个**完全独立的 Web 应用**，可独立部署
- 通过 HTTP API 从创作端获取项目数据和编译结果
- 使用端只包含：
  - **项目列表页** — 展示可供使用的项目
  - **预览页** — 预览区 + 配置面板（无 AI 对话、无代码编辑）
- 使用端可轻松嵌入其他系统（iframe 或 API 对接）
- 使用者不需要登录/设置用户名，开箱即用

---

## 二、为什么选择独立站点

### 2.1 架构对比

| 维度 | 同一站点内分角色 | **独立站点（本方案）** |
|------|-----------------|----------------------|
| **耦合度** | 高 — 共用路由、组件、构建产物 | **低 — 仅通过 API 通信** |
| **可维护性** | 编辑者变更可能影响使用者 | **独立迭代，互不影响** |
| **部署灵活性** | 必须一起部署 | **可独立部署到不同域名/服务器** |
| **嵌入其他系统** | 需要额外封装 | **天生适合嵌入 — 无多余 UI 元素** |
| **开发速度** | 需理解整个项目后再修改 | **从零开始，只需对接 API** |
| **安全性** | 共用一个攻击面 | **使用端只读，风险低** |
| **构建产物** | 大（含 AI 对话等重组件） | **轻量（仅预览+配置）** |

### 2.2 适用场景

- 将生成的组件展示嵌入客户官网、产品文档
- 为不同客户提供独立的组件预览门户
- 作为 CMS 或设计系统的展示层
- 与创作端分属不同团队维护

## 三、架构设计

### 3.1 Monorepo 结构

项目采用 pnpm workspaces 管理，包含以下包：

```
packages/
├── studio/                  # 【原 web/】创作端 Web 应用（Next.js，端口 3200）
│   ├── src/
│   │   ├── app/           # App Router（页面和 API 路由）
│   │   │   ├── (auth)/    # 认证相关页面（login/register）
│   │   │   ├── api/       # API 路由
│   │   │   │   ├── embed/[projectId]/iframe/  # 多页面嵌入 iframe API
│   │   │   │   ├── projects/[projectId]/      # 项目管理 API（demos/config/versions）
│   │   │   │   ├── compile/                   # 编译 API
│   │   │   │   └── sessions/                  # 会话管理 API
│   │   │   ├── embed/[demoId]/                # 单页面嵌入（含 iframe 路由和嵌入说明页）
│   │   │   ├── projects/                      # 项目管理页面
│   │   │   └── demo/[id]/edit/                # Demo 编辑页面
│   │   ├── components/    # React 组件
│   │   └── lib/          # 工具函数
│   ├── components/demo/   # Demo 组件（含 ConfigForm、PreviewPanel 等）
│   └── data/              # 项目数据存储
├── shared/                 # 共享类型定义
│   └── src/
│       ├── types.ts       # DemoMeta, DemoFiles, SessionMeta, ApiResponse 等基础类型（旧版，部分类型已迁移至 index.ts）
│       ├── workspace.ts   # Project, DemoPageMeta, ProjectListResponse, DemoPageDetail 等项目类型
│       └── index.ts       # 统一导出（含 MultiDemoFiles, MergedComponentProps, 完整 ErrorCode 常量）
├── agent-service/          # Agent 服务（Fastify，端口 3201）
└── agent-client/           # Agent 客户端 SDK
```

### 3.2 使用端定位

使用端作为 **packages/viewer-site/** 加入 monorepo：

```
packages/

├── viewer-site/            # 使用端（新增）

│   │   └── lib/           # 工具函数
│   └── package.json

```

### 3.3 整体架构

使用端需要对接**两个独立服务**：agent-service（项目管理）和 web（嵌入/编译）。

```
┌──────────────────────┐                        ┌──────────────────────────┐
│                      │  GET /api/projects      │  Agent Service           │
│                      │  GET /api/projects/:id  │  (Fastify, 端口 3201)    │
│                      │  GET /api/projects/:id/ │  项目管理 API            │
│                      │       versions          │                          │
│   使用端              │───────────────────────► │                          │
│   viewer-site        │                        └──────────────────────────┘
│   (独立部署)          │
│                      │                        ┌──────────────────────────┐
│                      │  GET /api/embed/:id/    │  Web 创作端               │
│                      │       iframe?page=      │  (Next.js, 端口 3200)    │
│                      │  GET /api/projects/:id/ │  嵌入/编译/页面管理 API   │
│                      │       demos             │                          │
│                      │  POST /api/compile      │                          │
│                      │───────────────────────► │                          │
└──────────────────────┘                        └──────────────────────────┘
```

> **关键点**：`GET /api/projects` 和 `GET /api/projects/:id` 由 agent-service 提供（端口 3201），而嵌入 iframe、Demo 页面列表、编译等 API 由 web 端提供（端口 3200）。使用端需要分别配置两个服务的地址。

### 3.4 现有 API 盘点

创作端已实现的与使用端相关的 API，按服务来源分组：

**Agent Service（端口 3201）提供的 API：**

| API | 方法 | 说明 | 认证 | 状态 |
|-----|------|------|------|------|
| `GET /api/projects` | 获取项目列表 | 返回 `ProjectListResponse` | 无需 | 已有 |
| `GET /api/projects/:id` | 获取项目详情 | 返回 `ProjectDetailResponse`（含 `demoPages` 列表） | 无需 | 已有 |
| `GET /api/projects/:id/versions` | 获取版本历史 | 返回 `VersionHistoryResponse` | 无需 | 已有 |

**Web 端（端口 3200）提供的 API：**

| API | 方法 | 说明 | 认证 | 状态 |
|-----|------|------|------|------|
| `GET /api/projects/:id/demos` | 获取项目的 Demo 页面列表 | 返回 `{ demoPages: DemoPageMeta[] }` | 无需 | 已有 |
| `GET /api/projects/:id/demos/:demoId` | 获取单个 Demo 页面元信息 | 返回 `DemoPageMeta` | 无需 | 已有 |
| `GET /api/projects/:id/config` | 获取项目级共享配置 Schema | 返回 `{ schema, exists }` | 无需 | 已有 |
| `GET /api/embed/:projectId/iframe?page=:demoId` | 多页面模式：生成 iframe HTML | 合并项目级+页面级 schema | 无需 | 已有 |
| `GET /embed/:demoId/iframe` | 单页面模式：生成 iframe HTML | 仅读取项目级 schema | 无需 | 已有 |
| `POST /api/compile` | 编译代码 | 支持 `code` 或 `sessionId` 模式 | 无需 | 已有 |
| `PUT /api/projects/:id/demos/:demoId/files` | 更新 Demo 页面文件 | 写入需要 sessionId | 需认证 | 已有 |

> **注意**：
> - `GET /api/embed/:projectId/iframe` 需要 `page` 参数指定 demoId，返回合并了项目级和页面级 schema 的 iframe HTML。适用于多页面项目。
> - `GET /embed/:demoId/iframe` 不需要 `page` 参数，直接以 projectId 作为 demoId，仅读取项目级 schema。适用于单页面项目。
> - `GET /api/projects/:id/demos/:demoId/files`（读取代码和 schema）目前**不存在**，仅有 PUT 版本。

### 3.5 使用端结构

使用端是一个**独立的 Next.js 应用**：

```
viewer-site/
├── src/
│   ├── app/
│   │   ├── page.tsx                    # 项目列表页
│   │   ├── layout.tsx                  # 根布局
│   │   ├── globals.css                 # 全局样式
│   │   └── [projectId]/
│   │       ├── page.tsx                # 项目预览页（选择 Demo 页面 + 预览区 + 配置面板）
│   │       └── [demoId]/
│   │           └── page.tsx            # 指定 Demo 页面的预览页（直接嵌入 iframe）
│   ├── components/
│   │   ├── project-list.tsx            # 项目列表组件
│   │   ├── project-preview.tsx         # 预览页主组件
│   │   ├── demo-page-selector.tsx      # Demo 页面选择器（多页面项目）
│   │   ├── config-panel.tsx            # 配置面板（参考 web 端 ConfigForm）
│   │   └── iframe-renderer.tsx         # iframe 渲染器（封装 postMessage 通信）
│   └── lib/
│       ├── api.ts                      # API 客户端（区分 agent-service 和 web 端）
│       └── types.ts                    # 类型定义（从 @opencode-workbench/shared 导入）
├── package.json
├── next.config.js
└── tsconfig.json
```

> **为什么需要 `[projectId]/[demoId]` 路由**：一个项目可以包含多个 Demo 页面（如"首页"、"详情页"），每个页面有独立的代码和配置。使用端需要让使用者选择查看哪个页面，同时支持通过 URL 直接访问特定页面。

### 3.6 路由设计

| 路径 | 说明 |
|------|------|
| `/` | 项目列表页 — 从 agent-service 获取项目列表 |
| `/[projectId]` | 项目预览页 — 展示 Demo 页面选择器 + 默认页面预览 + 配置面板 |
| `/[projectId]/[demoId]` | 指定 Demo 页面预览页 — 直接嵌入该页面的 iframe + 配置面板 |

## 四、关键组件实现思路

### 4.1 项目列表页 (`/`)

```
┌──────────────────────────────────────┐
│  Logo / 站点名称                      │
│                                      │
│  ┌──────────┐  ┌──────────┐          │
│  │ 项目卡片  │  │ 项目卡片  │  ...     │
│  │ 名称/描述 │  │ 名称/描述 │          │
│  │ 页面数量  │  │ 页面数量  │          │
│  │ 查看预览  │  │ 查看预览  │          │
│  └──────────┘  └──────────┘          │
└──────────────────────────────────────┘
```

- 调用 agent-service 的 `GET /api/projects` 获取列表（返回 `ProjectListResponse`，含 `name`、`description`、`demoCount`、`currentVersion` 等）
- 点击项目卡片跳转到 `/{projectId}`
- 无"新建项目"、无用户名选择器
- 可配置只展示特定项目（通过环境变量过滤）

### 4.2 预览页 (`/[projectId]` 和 `/[projectId]/[demoId]`)

```
┌─────────────────────────────────────────────────┐
│  ← 返回列表  │  项目名称  │  [页面1] [页面2] ... │
├──────────────────────┬──────────────────────────┤
│                      │                          │
│    预览区            │    配置面板               │
│    (iframe)          │    (表单控件)              │
│                      │                          │
│                      │                          │
└──────────────────────┴──────────────────────────┘
```

#### 页面加载流程

1. 调用 agent-service 的 `GET /api/projects/:id` 获取项目详情（含 `demoPages` 列表）
2. 如果项目有多个 Demo 页面，展示页面选择器（Tab 或下拉）
3. 选择页面后，调用 web 端的 `GET /api/projects/:id/demos/:demoId` 获取页面元信息
4. 调用 web 端的 `GET /api/projects/:id/config` 获取项目级配置 Schema
5. 嵌入 iframe 并初始化配置面板

#### 预览区实现

根据项目页面数量选择不同的嵌入端点：

**多页面项目**（有 `demos/` 目录）：
```html
<iframe
  src="https://web端域名/api/embed/{projectId}/iframe?page={demoId}"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>
```

**单页面项目**（无 `demos/` 目录，仅有 `index.tsx`）：
```html
<iframe
  src="https://web端域名/embed/{projectId}/iframe"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>
```

创作端负责：
1. 读取项目 workspace 中的代码和 schema
2. 编译代码（使用锁定的依赖版本，`lockedDependencies`）
3. 合并项目级和页面级配置（通过 `mergeConfigToProps()`）
4. 生成完整的 iframe HTML（含 React 运行时 + Tailwind CDN）

#### 配置面板

- 从项目级 Schema（`project.config.schema.json`）和页面级 Schema（`demos/{id}/config.schema.json`）解析字段定义
- 动态生成表单控件（可参考 web 端已有的 `ConfigForm` / `ConfigFormNew` 组件）
- 配置变更通过 `postMessage` 的 `UPDATE_CONFIG` 消息发送到 iframe
- 配置状态保存在本地 state，不持久化

### 4.3 iframe 通信

复用现有 `iframe-template.ts` 生成的 HTML 中的 `postMessage` 通信机制：

#### 已实现的协议

```
父窗口 → iframe:
  { type: 'UPDATE_CONFIG', configData: { ... } }        # 更新配置（重新渲染组件）
  { type: 'UPDATE_CODE', code: '...', configData: {...}, cssImports: [...] }  # 更新代码+配置+CSS

iframe → 父窗口:
  { type: 'READY' }                                     # iframe 脚本已就绪
  { type: 'LOADED' }                                    # 组件模块加载并渲染完成
  { type: 'RUNTIME_ERROR', error: '...', stack?: '...', source?: '...', line?: ... }  # 运行时错误
  { type: 'RESIZE', height: 800 }                       # 内容高度变化
```

> **注意**：
> - `UPDATE_CODE` 用于热更新场景（创作端编辑时实时预览），使用端通常只需使用 `UPDATE_CONFIG`。
> - `READY` 在 iframe 脚本执行后立即发送，此时组件可能尚未加载完成。
> - `LOADED` 在组件模块成功加载并渲染后发送，是更可靠的"就绪"信号。
> - `RUNTIME_ERROR` 包含 `stack`（错误堆栈）、`source`（错误文件）、`line`（行号）等额外字段。

## 五、创作端需新增/调整的 API

### 5.1 获取 Demo 页面代码和 Schema（可选）

当前 embed iframe 端点直接读取文件并编译。如果使用端需要自己处理编译逻辑，需要新增：

```
GET /api/projects/:projectId/demos/:demoId/files
```

返回：

```json
{
  "success": true,
  "data": {
    "code": "import React from 'react'...",
    "schema": "{ \"type\": \"object\", \"properties\": {...} }"
  }
}
```

> **注**：当前 `PUT /api/projects/:projectId/demos/:demoId/files` 已存在，但缺少 GET 版本。GET 版本不需要认证（只读），也不需要 sessionId。

### 5.2 获取项目级配置 Schema 的默认值（建议新增）

当前 `GET /api/projects/:id/config` 仅返回 Schema 字符串和 `exists` 标记。使用端配置面板需要知道每个字段的默认值，建议在返回中增加 `defaults` 字段：

```json
{
  "success": true,
  "data": {
    "schema": "{ \"type\": \"object\", \"properties\": { \"theme\": { \"type\": \"string\", \"default\": \"light\" } } }",
    "exists": true,
    "defaults": { "theme": "light" }
  }
}
```

### 5.3 嵌入端点的 fields 参数（建议新增）

当前 embed iframe 端点返回完整的合并后配置。如果需要选择性暴露字段，建议扩展：

```
/api/embed/:projectId/iframe?page=:demoId&fields=theme,color
```

## 六、嵌入其他系统的方式

### 6.1 iframe 整体嵌入

外部系统直接嵌入使用端的完整预览页（含配置面板）：

```html
<iframe
  src="https://使用端域名/{projectId}"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; height: 800px; border: none;"
/>
```

适用于：使用端不需要任何定制，直接展示创作端的完整 UI。

### 6.2 混合嵌入模式（外部控制 + iframe 预览）⭐ 推荐

外部系统自己实现精简的配置面板，仅展示需要的配置项，同时用 iframe 渲染预览区。创作端提供细粒度的控制能力。

#### 6.2.1 嵌入端点参数

当前端点：

```
/api/embed/{projectId}/iframe?page={demoId}
```

建议新增参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `fields` | string | 逗号分隔的字段名列表，只渲染指定配置项（留空则全部渲染） |
| `externalControl` | boolean | 为 `true` 时，隐藏内置配置面板，由外部完全控制 |
| `config` | JSON string | 初始配置 |

**示例**：

```
/api/embed/demo123/iframe?page=page1&fields=theme,color&config={"theme":"dark"}
```

#### 6.2.2 模式 A：使用 iframe 内置配置面板（可选择字段）

```html
<iframe
  src="/api/embed/{projectId}/iframe?page={demoId}&fields=theme,color"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none; height: 600px;"
/>
```

- 预览区自动调整高度（通过 `RESIZE` postMessage）
- 外部系统只需处理 iframe 容器布局
- 适用于：使用端需要完整的配置体验，但只需要部分字段

#### 6.2.3 模式 B：外部实现配置面板，iframe 仅渲染预览

```html
<!-- 外部系统自己实现的精简配置面板 -->
<div class="config-panel">
  <label>主题：
    <select id="theme-select">
      <option value="light">亮色</option>
      <option value="dark">暗色</option>
    </select>
  </label>
  <button id="apply-btn">应用</button>
</div>

<!-- 预览 iframe（隐藏配置面板） -->
<iframe
  id="preview-frame"
  src="/api/embed/{projectId}/iframe?page={demoId}&externalControl=true"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>

<script>
  const iframe = document.getElementById('preview-frame');
  const applyBtn = document.getElementById('apply-btn');

  applyBtn.addEventListener('click', () => {
    const theme = document.getElementById('theme-select').value;
    iframe.contentWindow.postMessage({
      type: 'UPDATE_CONFIG',
      configData: { theme }
    }, '*');
  });
</script>
```

- 外部系统完全控制配置 UI，只暴露需要的字段
- iframe 内部不渲染配置面板，专注预览渲染
- 通过 `postMessage` 发送配置变更
- 适用于：使用端需要统一风格、或配置项来自外部数据源

#### 6.2.4 postMessage 通信协议

**已实现的消息：**

```
父窗口 → iframe:
  { type: 'UPDATE_CONFIG', configData: { ... } }        # 更新配置
  { type: 'UPDATE_CODE', code: '...', configData: {...}, cssImports: [...] }  # 更新代码+配置

iframe → 父窗口:
  { type: 'READY' }                                     # 脚本已就绪
  { type: 'LOADED' }                                    # 组件已渲染
  { type: 'RUNTIME_ERROR', error: '...', stack?: '...' } # 运行时错误
  { type: 'RESIZE', height: 800 }                       # 高度调整
```

**建议新增的消息（用于外部控制模式）：**

```
父窗口 → iframe:
  { type: 'GET_STATE' }                    # 请求当前状态

iframe → 父窗口:
  { type: 'STATE', state: { config, ... } } # 返回当前状态
```

### 6.3 API 对接（完全自定义渲染）

外部系统直接调用创作端的 API，自己实现数据获取、编译和渲染。注意 API 分布在两个服务上：

```javascript
// 1. 获取项目列表（agent-service，端口 3201）
const projects = await fetch('https://agent-service域名/api/projects');

// 2. 获取项目详情，含 demoPages 列表（agent-service）
const project = await fetch(`https://agent-service域名/api/projects/${projectId}`);

// 3. 获取 Demo 页面列表（web 端，端口 3200）
const demos = await fetch(`https://web端域名/api/projects/${projectId}/demos`);

// 4. 获取项目级配置 Schema（web 端）
const config = await fetch(`https://web端域名/api/projects/${projectId}/config`);

// 5. 获取 Demo 代码和 Schema（需新增 GET 接口，web 端）
const files = await fetch(`https://web端域名/api/projects/${projectId}/demos/${demoId}/files`);

// 6. 编译代码（web 端）
const compiled = await fetch('https://web端域名/api/compile', {
  method: 'POST',
  body: JSON.stringify({ code: files.data.code })
});
```

适用于：使用端有特殊的渲染技术栈要求，或需要完全控制 UI。

## 七、部署方案

### 7.1 部署拓扑

```
Agent Service (packages/agent-service) → 端口 3201
├── GET /api/projects                     # 项目列表
├── GET /api/projects/:id                 # 项目详情
├── GET /api/projects/:id/versions        # 版本历史
└── POST /api/projects                    # 创建项目（编辑者用）

创作端 (packages/studio) → 端口 3200
├── 编辑者界面（/projects/、/demo/）
├── 嵌入 iframe 端点（/api/embed/、/embed/）
├── 编译 API（/api/compile）
├── Demo 页面管理 API（/api/projects/:id/demos/）
└── 项目配置 API（/api/projects/:id/config）

使用端 (packages/viewer-site) → 独立域名/子域名
└── Next.js SSR 或静态导出
```

> **端口说明**：Agent Service 默认端口为 3201（见 `agent-service/src/utils/config.ts`）。`.env.example` 中的端口已校正为 3201。

### 7.2 环境变量

使用端需要配置两个服务的地址：

```env
# Agent Service 地址（项目列表、项目详情、版本历史）
NEXT_PUBLIC_AGENT_SERVICE_URL=https://agent-service域名

# Web 创作端地址（嵌入 iframe、编译、Demo 页面管理、项目配置）
NEXT_PUBLIC_WEB_URL=https://web端域名
```

> **为什么需要两个地址**：当前创作端的 API 分布在两个独立服务上。`GET /api/projects` 等项目管理 API 由 agent-service（Fastify）提供，而嵌入 iframe、编译、Demo 页面等 API 由 web 端（Next.js）提供。两者端口不同，部署时可能在不同域名/服务器上。

### 7.3 CORS 配置

使用端与创作端分属不同域名，需要配置跨域访问：

- **Agent Service**：已使用 `@fastify/cors`，需将使用端域名加入允许列表
- **Web 端（Next.js）**：当前未配置 CORS，需在 `next.config.js` 中添加 CORS 头，或在 API 路由中手动设置 `Access-Control-Allow-Origin`
