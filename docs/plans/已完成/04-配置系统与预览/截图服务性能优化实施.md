# 截图服务性能优化实施

## 背景

项目文档 [截图服务性能优化方案](../../../项目文档/创作端/04-配置与预览/技术/09_截图服务性能优化方案.md) 已明确当前 screenshot-service 具备 Browser 复用、队列限流、同 hash 合并、编译缓存、hash 文件缓存和 renderBox 校验等基础能力。用户要求开始实施并持续推进全部优化计划。

## 目标

- 先落地高 ROI、低风险优化：性能观测、当前可见页优先、后端显式优先级调度、浏览器预热。
- 保持截图一致性规则：不展示旧图，不让快速截图覆盖 immutable 严格截图产物。
- 在每轮实现后同步测试和项目文档，确保后续 agent 能接着推进。

## 范围

- author-site 截图调度、批量请求、状态轮询和测试。
- demo-ui 画布可见性与截图优先级相关能力。
- screenshot-service 路由、队列、浏览器池、缓存指标和测试。
- `docs/项目文档/创作端/04-配置与预览/技术/09_截图服务性能优化方案.md` 以及必要索引。

## 非目标

- 本轮不优先实现 page 池。page 池属于高风险状态复用能力，需在指标证明 `newPage` 是主要瓶颈后再推进。
- 不把浏览器扩展作为主链路优化。
- 不整理当前工作区里与截图优化无关的既有 dirty changes。

## 方案

1. 先检查当前源码，确认已有改动是否已覆盖部分优化。
2. 实施 P0 指标：批量状态携带阶段耗时、缓存命中和优先级完成情况；日志补充缓存和 in-flight 合并信息。
3. 实施 P1 调度：请求字段增加 `priority`，服务端按显式优先级排序；前端在批量生成时按当前页面、可见页、近邻页、后台页排序。
4. 实施低风险浏览器预热：配置开关控制服务启动后预热 Browser，保持默认行为兼容。
5. 补测试覆盖优先级排序、指标字段、旧接口兼容。
6. 更新项目文档和计划进度。
