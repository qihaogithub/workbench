# AI Agent 文档整理工作方法论docs/plans/进行中

> 本文档定义 AI Agent 自动化整理技术文档的标准流程。  
> 适用场景：`docs/plans/已完成/` 等历史文档目录的周期性整理  
> 最后更新：2026-06-03

---

## 🎯 核心原则

### 三原则决策（固定策略）

1. **整理方式**：先生成执行清单报告 → 用户确认 → 再执行文件操作
2. **归档策略**：**激进清理**（直接删除过期/低价值文档，而非仅归档）
3. **经验沉淀**：仅按主题分类整理目录结构（不额外提炼抽象经验知识库）

### 价值判断标准

| 等级              | 特征                                                      | 处理方式       |
| ----------------- | --------------------------------------------------------- | -------------- |
| **⭐⭐⭐ 高价值** | 架构决策、技术选型理由、复杂bug根因、最佳实践、踩坑记录   | 保留并置顶分类 |
| **⭐⭐ 中价值**   | 功能实现方案、API设计、部署流程、问题分析（可能部分过期） | 保留并分类     |
| **⭐ 低价值**     | 临时问题修复、简单UI调整、一次性任务记录、纯过程记录      | **直接删除**   |
| **❌ 过期**       | 已被新方案替代、技术栈已移除、问题不再存在                | **直接删除**   |

## 📋 工作流程（6步法）

### 第一步：文档扫描与审计

**目标**：全面了解文档状况，生成价值评估矩阵

**执行步骤**：

```
1. 列出目标目录所有文件（含子目录）
2. 读取每个文档的前50行，提取关键信息：
   - 文档标题和类型（方案/报告/记录/经验）
   - 是否有版本号、日期、时效性描述
   - 技术栈提及（判断是否过时）
   - 问题/功能是否仍存在
3. 生成四象限评估矩阵：
   - 高价值文档清单（⭐⭐⭐）
   - 中价值文档清单（⭐⭐）
   - 低价值文档清单（⭐）
   - 过期文档清单（❌）
```

**输出物**：`文档审计报告.md`（含每个文档的价值评级和简短理由）

---

### 第二步：重复文档识别

**目标**：找出高度相似或重复的文档组

**识别规则**：

```
✅ 重复判定标准：
- 标题关键词重叠度 > 60%
- 描述同一问题的不同视角
- 同一功能的迭代版本（保留最新版）
- 同一技术方案的方案+实施记录（保留方案版）

⚠️ 需要人工确认：
- 标题相似但可能侧重不同
- 同一主题的不同阶段文档
```

**输出物**：`重复文档组清单.md`（每组标注保留/合并/删除建议）

---

### 第三步：目录结构设计

**目标**：基于文档内容特征，设计合理的分类结构

**设计原则**：

```
1. 按价值分层：高价值文档独立分类，便于快速查找
2. 按主题聚合：相关文档归入同一主题目录
3. 控制层级深度：最多2级目录（分类/文档）
4. 预留归档目录：99-归档/ 存放待删除文档

推荐结构（可根据实际情况调整）：
├── 01-架构决策/       # 技术选型、架构设计
├── 02-核心问题分析/   # 复杂bug根因、生产问题
├── 03-{主题A}/        # 按实际主题命名
├── 04-{主题B}/
├── ...
└── 99-归档/           # 历史/过期/重复
```

**输出物**：`新目录结构方案.md`

---

### 第四步：生成执行清单

**目标**：产出详细的逐文档处理清单，供用户确认

**清单格式要求**：

```markdown
## 逐个文档处理清单

### ✅ 保留并移动（高价值）

| 原文档路径 | 新路径 | 保留理由（1句话） |
| ---------- | ------ | ----------------- |

### ✅ 保留并移动（中价值）

| 原文档路径 | 新路径 | 保留理由（1句话） |
| ---------- | ------ | ----------------- |

### ⚠️ 需要合并后保留

| 原文档组 | 处理方式 | 保留文档 | 删除文档 |
| -------- | -------- | -------- | -------- |

### ❌ 直接删除（低价值/过期）

| 文档路径 | 删除理由（1句话） |
| -------- | ----------------- |

### 📦 暂时保留待确认

| 文档路径 | 待确认原因 |
| -------- | ---------- |
```

**输出物**：`文档整理执行清单.md`（含完整路径映射表）

---

### 第五步：执行文件操作

**前置条件**：用户确认执行清单

**执行顺序**（严格按此顺序）：

```
1. 创建新目录结构
   mkdir -p docs/plans/已完成/{01-架构决策,02-核心问题分析,...}

2. 移动高/中价值文档（按清单逐篇）
   git mv 原文档路径 新路径

3. 合并重复文档
   - 读取多篇文档内容
   - 整合为1篇增强版
   - 删除原始文档

4. 删除低价值/过期文档（按清单逐篇）
   git rm 文档路径

5. 清理空目录
   - 删除原分类目录（文档已移至新目录）
   - 删除空文件夹

6. 创建 README.md 索引文件
   - 目录说明
   - 核心文档推荐
   - 文档维护规范
```

**安全机制**：

```
✅ 必须使用 git mv（保留历史记录）
✅ 每完成一批操作后 git status 确认
✅ 删除操作前再次确认清单
✅ 最后统一 git commit（不中途提交）
```

---

### 第六步：生成整理报告

**目标**：记录整理过程和结果，便于追溯

**报告内容**：

```markdown
# 文档整理完成报告

## 📊 整理统计

- 整理前文档数：XX篇
- 整理后文档数：XX篇
- 删除文档数：XX篇

- 新建分类目录：XX个

## 📂 最终目录结构

（树状图展示）

## 🌟 核心文档推荐

（列出前5篇高价值文档）

## ⚠️ 待确认事项

（列出需要人工判断的文档）

## 📝 后续建议

- 下次整理时间建议
- 文档维护规范建议
- 需要补充的文档类型
```

**输出物**：`文档整理完成报告.md`

## 🔍 价值评估详细标准

### 高价值文档（⭐⭐⭐）特征

```
✅ 包含架构决策和技术选型理由
✅ 记录复杂bug的完整排查路径和根因
✅ 提炼出可复用的最佳实践或模式
✅ 包含性能优化数据和成本节省估算
✅ 解决系统性问题（非单点问题）
✅ 有版本迭代记录（如 v1.0 → v3.2）

示例：
- AI行为约束机制完善方案.md（四层约束架构，token节省~70%）
- AI编辑后预览不实时更新.md（Drain机制设计，时序竞争处理）
- 全面迁移至Pi-Agent方案.md（多维度技术选型对比）
```

### 中价值文档（⭐⭐）特征

```
✅ 完整的功能方案设计文档
✅ 清晰的API设计和数据流分析
✅ 部署流程和运维指南
✅ 问题分析报告（问题仍存在或解决方案通用）
⚠️ 可能随代码迭代部分过期，但思路仍可参考

示例：
- AI对话多模态图片上传方案.md（全链路断裂点分析）
- 找回密码功能方案.md（完整功能设计）
- Docker部署方案.md（部署流程参考）
```

### 低价值文档（⭐）特征

```
❌ 仅修改1-2个参数的简单调整
❌ 特定项目的UI交互问题（无通用性）
❌ 一次性任务记录（如"重命名XX包"）
❌ 简单的需求文档（非技术经验）
❌ 已实施且无后续参考价值的功能方案

示例：
- 三栏布局左右栏等宽固定方案.md（仅改defaultSizes参数）
- 项目封面图设置功能方案.md（简单功能实现）
- Web包重命名为Studio开发计划.md（一次性任务）
```

### 过期文档（❌）特征

```
❌ 技术栈已完全移除（如 OpenCode HTTP 后端）
❌ 功能已被新方案替代（如多后端支持已移除）
❌ 问题不再存在（如特定环境bug已根治）
❌ 文档明确标注"已废弃"或"不再维护"

示例：
- OpenCodeHttpBackend/ 整个目录（已迁移至Pi-Agent）
- opencode不能对话-问题记录.md（opencode后端已移除）
- 模型列表与Agent后端解耦方案.md（多后端支持已移除）
```

## 🛠️ 自动化脚本模板

### 文档扫描脚本（伪代码）

```python
def scan_documents(directory):
    """扫描目录并生成价值评估矩阵"""

    results = {
        'high_value': [],
        'medium_value': [],
        'low_value': [],
        'outdated': []
    }

    for doc in list_files(directory, '*.md'):
        content = read_first_n_lines(doc, 50)

        # 提取关键信息
        has_version = extract_version(content)
        has_date = extract_date(content)
        tech_stack = extract_tech_mentions(content)
        is_architecture = detect_architecture_keywords(content)
        is_bug_fix = detect_bug_fix_keywords(content)

        # 价值评估逻辑
        if is_architecture and has_version:
            results['high_value'].append(doc)
        elif is_bug_fix and has_detailed_analysis(content):
            results['high_value'].append(doc)
        elif is_feature_design(content):
            results['medium_value'].append(doc)
        elif is_simple_fix(content) or is_one_time_task(content):
            results['low_value'].append(doc)
        elif is_tech_removed(tech_stack):
            results['outdated'].append(doc)

    return results
```

### 重复检测脚本（伪代码）

```python
def detect_duplicates(documents):
    """检测重复文档组"""

    groups = []

    for i, doc1 in enumerate(documents):
        for j, doc2 in enumerate(documents[i+1:], i+1):
            similarity = calculate_similarity(doc1.title, doc2.title)

            if similarity > 0.6:
                # 检查是否为同一主题
                if same_topic(doc1.content, doc2.content):
                    groups.append({
                        'docs': [doc1, doc2],
                        'similarity': similarity,
                        'action': recommend_action(doc1, doc2)
                    })

    return groups
```

## ⚠️ 注意事项

### 安全机制

```
1. 删除操作不可逆：
   - 必须先 git add 所有移动操作
   - 确认无误后再 git commit
   - 建议使用 git mv 而非 mv（保留历史记录）

2. 待确认文档：
   - 标记为 ⚠️ 的文档需要人工检查
   - 不要自动删除不确定的文档

3. 引用失效检查：
   - 检查其他文档是否引用了被移动的文档
   - 如有引用，需要更新相对路径

4. Git 历史保留：
   - 使用 git mv 保持文件历史可追溯
   - 提交信息格式：docs: 整理已完成文档目录结构
```

### 质量标准

```
✅ 每次整理必须产出：
   - 文档审计报告
   - 执行清单（含完整路径映射）
   - 整理完成报告
   - README.md 索引文件

✅ 目录结构必须：
   - 按价值分层（高价值置顶）
   - 按主题聚合（相关文档归类）
   - 层级不超过2级

✅ 删除操作必须：
   - 有明确理由（低价值/过期/重复）
   - 记录在清单中
   - 用户确认后执行
```
