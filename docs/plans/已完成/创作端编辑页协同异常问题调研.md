| ----------------------------------- | -------------------------------------------------------------------------- | ------------- | ------------------------------- |
| Yjs 协同 WebSocket                  | `ws://localhost:3201/api/collab/...`                                       | agent-service | 连接失败 → "协同连接失败"       |
| Workspace Authority API（浏览器端） | `http://localhost:3201/api/workspace-authority/...`（经 author-site 代理） | agent-service | 503 → 自动保存失败              |
| Workspace Authority API（服务端）   | 同上                                                                       | agent-service | 503 → 页面/文件夹/资源 API 失败 |

## 六、结论

"协同异常"不是 WMA 改造引入的 bug，而是 **agent-service 未启动** 导致的预期行为。

WMA 架构下，agent-service 成为编辑页的唯一写入权威和协同中心，其不可用会导致：

1. **WebSocket 协同连接失败** → `connection-error` → 资源状态 `error` → 显示 "协同异常"
2. **Authority API 不可达** → 503 → 自动保存失败（但状态机依赖 `navigator.onLine`，不直接感知 Authority）
3. **BackendProviders 推送失败** → 模型配置不同步（不影响协同但影响 AI 对话）

**状态机局限**：当前 `computeSaveStateFromContext` 的 `isConnected = navigator.onLine` 仅反映浏览器网络在线状态，不检测 Authority 是否可达。因此当浏览器在线但 agent-service 未启动时，状态机可能输出 `editing` 或 `autosaved`，但协同状态覆盖机制（第 4.1 节）仍会强制显示 `协同异常`。

## 九、BackendProviders Sync 自动恢复机制（2026-07-13 新增）

**问题**：agent-service 未启动时，BackendProviders Sync 的 8 次自动重试耗尽后不再重试，即使 agent-service 恢复后也需要手动重启 author-site 或手动触发同步。

**修复**：在 `backend-providers-sync.ts` 中增加 agent-service 健康轮询机制：

- 当同步失败且重试耗尽（8 次）后，自动启动 15s 间隔的健康检查。
- 每次检查 agent-service `/health` 端点，检测服务可达性变化。
- 当 agent-service 从不可达变为可达时，自动触发一次 `recovery` 同步。
- 同步成功后重置 `attemptCount`，停止健康监控，恢复正常的重试机制。

**相关代码**：

- `backend-providers-sync.ts`：`recoveryTick`、`startRecoveryMonitoring`、`stopRecoveryMonitoring`、`checkAgentServiceHealth`
- 同步成功后重置 `attemptCount`：`updateStateForResult` 中 `result.ok` 分支
