## P2-1 创作端复杂状态下沉为可测试内核

### 背景

创作端的预览、配置、画布、保存和 AI 对话状态复杂。若规则主要藏在组件和 Hook 生命周期中，Codex 修改时很难判断完整影响范围。

### 目标

- 将核心状态规则提取为纯函数、领域 service 或小型状态机。
- 让主要回归在 Node/Vitest 中完成，而不是依赖人工浏览器验证。

### 范围

- 预览刷新策略。
- 配置合并策略。
- 画布节点布局、选择、多选和对齐状态。
- 自动保存、手动保存、版本历史状态流转。
- AI stream event 聚合和消息展示状态。

### 任务清单

- [ ] 选定一个高频问题域作为第一阶段，例如配置合并或预览刷新。
- [ ] 绘制输入、状态、输出和副作用边界。
- [ ] 提取纯规则函数并补单元测试。
- [ ] 将组件改为消费规则结果，减少组件内分支。
- [ ] 逐步扩展到其他状态域。

### 验收标准

- 关键规则有独立测试覆盖。
- 组件测试不再承担所有状态规则验证。
- 修改单个状态域时能运行小范围测试确认。

## P2-2 拆分 `shared` 包职责

### 背景

`@opencode-workbench/shared` 同时承载类型、常量和复杂 React UI 能力，导致公共包边界不清晰，也让后端依赖受到前端依赖污染。

### 目标

- `shared` 回归纯契约层。
- UI、预览、配置表单和画布组件进入独立前端包。
- author-site 与 viewer-site 复用 UI 包。

### 范围

- `packages/shared/src`
- `packages/author-site`
- `packages/viewer-site`
- workspace 包配置

### 任务清单

- [ ] 盘点 shared 中纯契约、前端 UI、工具函数和历史代码。
- [ ] 设计新包边界，例如 `@opencode-workbench/demo-ui`。
- [ ] 先迁移无状态 UI 和工具，再迁移复杂组件。
- [ ] 调整 author-site 和 viewer-site import。
- [ ] 确认后端包只依赖纯契约层。

### 验收标准

- agent-service、screenshot-service 不因 shared 引入前端 UI 依赖。
- author-site 和 viewer-site 不复制同一 UI 组件。
- shared 导出清单可以被快速理解为协议、类型和常量。

## P2-3 agent-service route/service 分层

### 背景

agent-service 的路由层承载了较多业务编排逻辑。长期看会增加工具调用、会话管理、工作区同步和错误处理的维护成本。

### 目标

- 路由层只负责 HTTP/WebSocket 协议适配。
- 业务规则进入 service 层。
- 错误结构和事件输出保持统一。

### 范围

- agent-service route。
- session、workspace、permission、stream event。
- agent backend 适配层。

### 任务清单

- [ ] 挑选一个低风险 route 作为试点。
- [ ] 提取 service，保持外部 API 不变。
- [ ] 为 service 补单元测试。
- [ ] 梳理错误码和错误响应结构。
- [ ] 再扩展到 WebSocket 和 agent message 主流程。

### 验收标准

- route 文件行数和业务分支减少。
- service 可脱离 Fastify 直接测试。
- 外部接口和事件结构保持兼容。

## P3-1 历史目录、历史脚本、过期方案归档治理

### 背景

历史目录、旧脚本和过期方案对追溯有价值，但不应继续作为默认开发入口。它们如果没有状态标记，会增加后续 Agent 的误读概率。

### 目标

- 历史材料可追溯，但不会误导当前实现。
- 过期方案有明确状态。
- 开发脚本和正式回归测试边界清楚。

### 范围

- `packages/web/`
- legacy snapshot 相关入口。
- `scripts/development/`
- `docs/plans/已完成/` 中已过期或低价值文档。

### 任务清单

- [ ] 为历史目录增加局部说明，标注“不作为默认源码入口”。
- [ ] 盘点 development 脚本，区分诊断脚本和应迁移为正式测试的脚本。
- [ ] 为过期方案文档补状态标记。
- [ ] 将明确无当前参考价值的文档移动到归档目录。
- [ ] 更新已完成文档索引，减少推荐过期文档。

### 验收标准

- Agent 能通过目录说明判断是否应该修改历史目录。
- 诊断脚本不会被误认为长期回归测试。
- 核心推荐文档只指向当前仍有价值的内容。

## 后续执行建议

建议按以下顺序推进：

1. 进入 P2 结构性改造，先做状态内核，再做 shared 拆包。
2. agent-service 工具域配置治理可作为 P2/P3 之间的小任务单独处理。
3. P3 历史归档治理用于继续处理剩余历史 warning 和根目录现场产物。

每个任务都应单独创建进行中文档，避免把架构改造、测试补充和历史清理混在同一次变更中。
