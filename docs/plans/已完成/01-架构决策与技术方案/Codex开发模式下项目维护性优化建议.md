## 文档定位

本文只保留尚未完成的维护性优化任务。已经完成的治理项不在本文重复展开，避免后续 Agent 把历史完成项误判为待办。

后续使用方式：

- 新任务进入实施前，从本文选择一个最小任务块创建 `docs/plans/进行中/` 任务文档。
- 实施完成后，将长期规则同步到 `docs/项目文档/`，过程记录归档到 `docs/plans/已完成/`。
- 若某项任务被证明不再需要，应在本文中标记为“取消”并说明原因，不要静默删除。
- 每完成一个任务块，应从本文删除该任务块，并在“进度总览”中更新剩余任务数量。

## 执行顺序建议

1. 先做 P2-1-1 到 P2-1-4，建立一个可复用的状态内核改造样板。
2. 再做 P2-2-1 和 P2-2-2，确认 `shared` 拆包边界后再迁移代码。
3. P2-3 可与 P2-2 并行，但每次只选一个 route 或一个 service 边界。
4. P3-1 放在结构性改造之后处理，用于继续降低历史上下文噪音。

## 更新规则

- 任务完成后，从本文删除对应任务块，并更新“进度总览”的数量。
- 若任务进行中但未完成，将状态改为“进行中”，并写明当前卡点或下一个最小动作。
- 若任务取消，将状态改为“取消”，补充取消原因和替代方案。
- 不在本文重复记录已完成详情；完成详情只链接到对应归档文档。
