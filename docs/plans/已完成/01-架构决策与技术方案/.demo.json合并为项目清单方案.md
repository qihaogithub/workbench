# .demo.json 与 .folders.json 合并为统一清单方案

> 版本：v2.0 | 更新日期：2026-06-03 | 目标：将每个 demo 页面独立的 `.demo.json` 和项目级的 `.folders.json` 合并为 workspace 根目录的单一 `workspace-tree.json`，同时精简冗余字段，最大化 AI Agent 友好度。

## 一、问题分析

### 1.1 当前架构

```
workspace/
├── .folders.json                          ← 虚拟文件夹清单（合并模式）
├── project.config.schema.json
└── demos/
    ├── demo_xxx/
    │   ├── index.tsx
    │   ├── config.schema.json
    │   └── .demo.json                     ← 页面元数据（逐个文件模式）
    └── demo_yyy/
        ├── index.tsx
        ├── config.schema.json
        └── .demo.json
```

当前项目共有 **20 个** `.demo.json` 文件，每个约 166-178 字节。内容为：

```json
// .demo.json
{
  "id": "demo_1779608460379_a1b2c3",
  "name": "广场页面-手机",
  "order": 1,
  "parentId": null,
  "createdAt": 1779608460379,
  "updatedAt": 1780450209708
}

// .folders.json
{
  "folders": [
    {
      "id": "folder_1776526720347_abc123",
      "name": "首页模块",
      "order": 0,
      "parentId": null,
      "createdAt": 1776526720347,
      "updatedAt": 1776526720347
    }
  ]
}
```

两者类型定义（`packages/shared/src/workspace.ts:60-82`）字段完全一致：`id`、`name`、`order`、`parentId`、`createdAt`、`updatedAt`。

### 1.2 三个核心问题

#### 问题 1：对 AI Agent 不友好（IO 次数）

AI 要了解项目全貌需要：

| 操作 | 文件读取次数 |
|------|-------------|
| 扫描 demos/ 目录 | 1 次 readdir |
| 逐个读 20 个 `.demo.json` | 20 次 readFile |
| 读 `.folders.json` | 1 次 readFile |
| **合计** | **22 次文件 IO** |

合并后只需 **1 次 readFile**。

#### 问题 2：同一棵树拆成两个文件

页面的 `parentId` 指向文件夹 ID，本质是一个数据结构的两个部分。拆开存放导致：

- parentId 变成跨文件引用，可能断裂
- AI 必须读两个文件才能理解完整层级关系
- 两者读写逻辑高度重复（`readFoldersMeta`/`writeFoldersMeta` vs 对应的页面函数族）

#### 问题 3：冗余字段 `createdAt` / `updatedAt`

逐字段评估必要性：

| 字段 | 页面 | 文件夹 | 必要性 | 理由 |
|------|------|--------|--------|------|
| `id` | ✅ | ✅ | **必要** | 唯一标识，目录名即 ID |
| `name` | ✅ | ✅ | **必要** | 显示名称 |
| `order` | ✅ | ✅ | **必要** | 排序依据 |
| `parentId` | ✅ | ✅ | **必要** | 页面归属文件夹、文件夹可嵌套（max 3 层） |
| `createdAt` | ❌ | ❌ | **可删** | 仅作为 `order` 相同时的兜底排序依据，此场景极少发生；可用文件系统 birthtime 替代 |
| `updatedAt` | ❌ | ❌ | **可删** | 系统自动维护但从未对 AI 有用。AI 不需要知道「上次什么时候改的」，它只需要知道「现在是什么」 |

**对 `parentId` 的补充分析**：文件夹的 `parentId` 支持嵌套（已实现 `getFolderDepth` 最大 3 层、`isDescendant` 循环检测），这是合理的 UI 功能。如果未来决定不支持文件夹嵌套，`parentId` 可进一步删减为仅页面有而文件夹无——但当前保留它更安全。

## 二、目标方案

### 2.1 目标文件结构

```
workspace/
├── workspace-tree.json           ← 统一清单：文件夹 + 页面（一棵树）
├── project.config.schema.json
└── demos/
    ├── demo_xxx/
    │   ├── index.tsx             ← 纯代码
    │   └── config.schema.json    ← 纯配置
    └── demo_yyy/
        ├── index.tsx
        └── config.schema.json
```

### 2.2 `workspace-tree.json` 格式（精简后）

```json
{
  "folders": [
    { "id": "folder_xxx", "name": "首页模块",    "order": 0, "parentId": null },
    { "id": "folder_yyy", "name": "弹窗模块",    "order": 1, "parentId": null }
  ],
  "pages": [
    { "id": "demo_aaa", "name": "广场页面-手机", "order": 0, "parentId": "folder_xxx" },
    { "id": "demo_bbb", "name": "通用弹窗",     "order": 0, "parentId": "folder_yyy" },
    { "id": "demo_ccc", "name": "广场页面-PC",  "order": 1, "parentId": "folder_xxx" },
    { "id": "demo_ddd", "name": "全局设置页",   "order": 0, "parentId": null }
  ]
}
```

仅 4 个字段：`id`、`name`、`order`、`parentId`。一个文件 = 一棵完整的项目树。

### 2.3 删减字段的影响分析

#### `createdAt` 的影响

当前唯一用途在 `listDemoPages()` 中：

```typescript
// fs-utils.ts:369-372 当前逻辑
return result.sort((a, b) => {
  if (a.order !== b.order) return a.order - b.order;
  return a.createdAt - b.createdAt;  // ← 唯一使用点
});
```

`order` 相同时用 `createdAt` 兜底。这种场景只发生在新建页面且未显式分配 order 时，极为罕见。替代方案：

1. **创建时强制分配不重复的 order**（当前 `createWorkspaceDemoPage` 已这样做：同级内取 `max(order) + 1`），避免 order 冲突
2. 若仍冲突，按 `id` 字典序兜底（ID 含时间戳，效果等同）

#### `updatedAt` 的影响

当前唯一用途在 `updateWorkspaceDemoFiles()` 中：

```typescript
// fs-utils.ts:1089-1092 当前逻辑
if (meta) {
  writeDemoPageMeta(wsPath, demoId, meta);
} else {
  writeDemoPageMeta(wsPath, demoId, {});  // ← 仅为了刷新 updatedAt
}
```

仅用于「更新时间戳」这个仪式感操作，无任何读取消费方。删减后，`updateWorkspaceDemoFiles` 不再需要调用元数据写入函数。

## 三、改造范围

### 3.1 类型精简（`packages/shared/src/workspace.ts`）

```typescript
// 改造前
export interface DemoPageMeta {
  id: string;
  name: string;
  order: number;
  parentId: string | null;
  createdAt: number;   // 删除
  updatedAt: number;   // 删除
}

export interface DemoFolderMeta {
  id: string;
  name: string;
  parentId: string | null;
  order: number;
  createdAt: number;   // 删除
  updatedAt: number;   // 删除
}

// 改造后
export interface DemoPageMeta {
  id: string;
  name: string;
  order: number;
  parentId: string | null;
}

export interface DemoFolderMeta {
  id: string;
  name: string;
  order: number;
  parentId: string | null;
}
```

### 3.2 新增统一清单读写（`packages/author-site/src/lib/fs-utils.ts`）

```typescript
// 路径工具
function getWorkspaceTreePath(workspacePath: string): string {
  return path.join(workspacePath, "workspace-tree.json");
}

interface WorkspaceTree {
  folders: DemoFolderMeta[];
  pages: DemoPageMeta[];
}

// 读取整棵树
function readWorkspaceTree(workspacePath: string): WorkspaceTree {
  const treePath = getWorkspaceTreePath(workspacePath);
  // 新格式
  if (fs.existsSync(treePath)) {
    try {
      const parsed = JSON.parse(fs.readFileSync(treePath, "utf-8"));
      return {
        folders: Array.isArray(parsed?.folders) ? parsed.folders : [],
        pages: Array.isArray(parsed?.pages) ? parsed.pages : [],
      };
    } catch { /* fall through */ }
  }
  // 兜底：旧格式迁移
  return migrateLegacyToTree(workspacePath);
}

// 写回整棵树
function writeWorkspaceTree(workspacePath: string, tree: WorkspaceTree): void {
  fs.writeFileSync(
    getWorkspaceTreePath(workspacePath),
    JSON.stringify(tree, null, 2),
    "utf-8",
  );
}
```

### 3.3 核心函数改造清单

| 函数 | 改造 |
|------|------|
| `readFoldersMeta()` | → `readWorkspaceTree().folders` |
| `writeFoldersMeta()` | → 读 tree → 改 folders → 写回 tree |
| `readDemoPageMeta()` | → `readWorkspaceTree().pages.find(p => p.id === id)` |
| `writeDemoPageMeta()` | → 读 tree → upsert pages 中对应记录 → 写回 tree |
| `listDemoPages()` | → `readWorkspaceTree().pages`（过滤文件系统不存在的） |
| `createWorkspaceDemoPage()` | → 创建目录 + tree.pages 追加记录 → 写回 |
| `deleteWorkspaceDemoPage()` | → 删除目录 + tree.pages 移除记录 → 写回 |
| `copyWorkspaceDemoPage()` | → 复制目录 + tree.pages 追加记录 → 写回 |
| `reorderDemoPages()` | → 读 tree → 批量更新 pages 的 order/parentId → 写回 |
| `updateWorkspaceDemoFiles()` | → 仅写代码/Schema，**不再触碰元数据**（无 updatedAt 需要维护） |
| `ensureWorkspaceFiles()` | → 创建默认页面 + 初始化 tree |
| `syncProjectDemoPagesFromWorkspace()` | → `listDemoPages()` + `readWorkspaceTree().folders` |
| `createDemoFolder()` / `updateDemoFolder()` / `deleteDemoFolder()` | → 改为通过 tree 读写 |

**关键简化**：合并后不再需要 `readFoldersMeta` / `writeFoldersMeta` / `readDemoPageMeta` / `writeDemoPageMeta` 四个独立函数，统一为 `readWorkspaceTree` / `writeWorkspaceTree` + 调用方自行操作内存中的 tree 对象。

## 四、数据迁移

### 4.1 迁移逻辑（内置兜底）

```typescript
function migrateLegacyToTree(workspacePath: string): WorkspaceTree {
  // 1. 尝试读旧 .folders.json
  const foldersMetaPath = path.join(workspacePath, ".folders.json");
  let folders: DemoFolderMeta[] = [];
  if (fs.existsSync(foldersMetaPath)) {
    try {
      const parsed = JSON.parse(fs.readFileSync(foldersMetaPath, "utf-8"));
      if (Array.isArray(parsed?.folders)) {
        folders = parsed.folders.map(f => ({
          id: f.id, name: f.name, order: f.order, parentId: f.parentId
        }));
      }
    } catch { /* ignore */ }
  }

  // 2. 扫描 demos/ 下所有 .demo.json
  const pages: DemoPageMeta[] = [];
  const demosDir = path.join(workspacePath, "demos");
  if (fs.existsSync(demosDir)) {
    for (const entry of fs.readdirSync(demosDir, { withFileTypes: true })) {
      if (!entry.isDirectory()) continue;
      const metaPath = path.join(demosDir, entry.name, ".demo.json");
      if (fs.existsSync(metaPath)) {
        try {
          const m = JSON.parse(fs.readFileSync(metaPath, "utf-8"));
          pages.push({ id: m.id, name: m.name, order: m.order, parentId: m.parentId });
        } catch { /* ignore */ }
      }
    }
  }

  // 3. 写入新格式 workspace-tree.json
  const tree: WorkspaceTree = { folders, pages };
  writeWorkspaceTree(workspacePath, tree);
  return tree;
}
```

迁移是**读取时自动执行**的（懒迁移），无需单独脚本。首次读取时发现 `workspace-tree.json` 不存在则自动从旧格式生成。

### 4.2 旧文件清理

迁移后可选择保留或删除旧文件：

- 保留：`.folders.json` 和 `demos/*/.demo.json` 保留不动（无副作用，但浪费空间）
- 删除：通过一次性脚本遍历所有 workspace，删除旧文件

建议 **Phase 4 阶段手动执行一次清理**，不需要自动化。

## 五、风险评估

| 风险 | 概率 | 缓解措施 |
|------|------|---------|
| 并发写入 tree 冲突 | 极低 | 当前单用户编辑场景，无并发问题；未来如需可加文件锁 |
| 迁移时数据遗漏 | 极低 | 懒迁移 + 旧文件不删除，可反复安全迁移 |
| AI 仍按旧 prompt 写 `.demo.json` | 中 | Prompt 更新 + `readWorkspaceTree` 保留旧格式兜底扫描 |
| tree 文件损坏 | 低 | 文件系统目录是真值来源，懒迁移逻辑始终可重建 tree |
| 前端依赖 `createdAt`/`updatedAt` | 低 | 需排查是否有组件展示这两个字段，如有则改为不显示或从文件系统读取 |

## 八、不做的事

- ❌ **不合并 `config.schema.json`**：页面级 Schema 与页面代码强绑定，物理相邻有利于理解上下文，且 Schema 可能很大（2000+ 字节），合并后文件膨胀
- ❌ **不合并到全工作空间根目录**：保持在项目 workspace 级别，维护项目边界
- ❌ **不在 tree 中包含 Schema 路径引用**：tree 只做结构清单，Schema 由页面目录自身管理
- ❌ **不删除文件系统扫描兜底**：保留 `demos/*/` 目录扫描作为 tree 损坏时的重建能力
