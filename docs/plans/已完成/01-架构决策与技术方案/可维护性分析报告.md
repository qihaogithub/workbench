## 四、跨包依赖关系

### 4.1 正常依赖

```
author-site ──→ @opencode-workbench/shared (workspace:*)
            ──→ @opencode-workbench/agent-client (workspace:*)

viewer-site ──→ @opencode-workbench/shared (workspace:*)

agent-service ──→ @opencode-workbench/shared (workspace:*)

agent-client ──→ (无内部依赖)

cli-tools ──→ (无内部依赖)
```

### 4.2 架构级循环依赖

```
shared (demo/) ──→ @/components/ui/* ──→ author-site
     ↑                                        │
     └────────────────────────────────────────┘
```

shared 包的 `demo/` 子目录通过 `@/components/ui/*` 路径别名引用 author-site 的 shadcn/ui 组件，形成反向依赖。当前仅因 `shamefully-hoist=true` 和构建时路径解析才能工作。

### 4.3 依赖不合理

- shared 包依赖 `@tiptap/*`(5个)、`@dnd-kit/*`(3个)、`dompurify`、`lucide-react` — 纯后端的 agent-service 被迫安装这些前端依赖
- 15 个 shadcn/ui 组件在 author-site 和 viewer-site 间逐字复制

## 五、各包关键问题

### 5.1 shared 包 — 包身份错位

**核心问题**：名为"shared"的包承载了完整 React 组件库（富文本编辑器、拖拽排序、Canvas 画布、iframe 预览、缩略图生成等），依赖了 `@tiptap/*`、`@dnd-kit/*`、`dompurify` 等前端库。

**次要问题**：
- `src/types.ts` 与 `src/index.ts` 存在**同名不同义**的类型（两个 `DemoMeta` 结构完全不同）
- `ErrorCode` 在两文件中数量不同（25 vs 14）
- 14 处 `as any` 违反项目规范
- `iframe-template.ts`（360 行模板字符串拼接 HTML+JS）无法获得 IDE 支持
- `canvas-utils.ts` 硬编码 API 路径 `/api/sessions/${sessionId}/canvas-layout`
- 零测试覆盖
- `tsconfig.json` 配置矛盾：`declaration: true` + `noEmit: true`

**建议**：拆分为 `@opencode-workbench/shared`（纯类型+常量）和 `@opencode-workbench/demo-ui`（React 组件）。

### 5.2 author-site — 上帝文件

**核心问题**：`fs-utils.ts`（1638 行）承担 8+ 职责：路径管理、项目 CRUD、Session CRUD、Workspace CRUD、Demo 页面 CRUD、文件夹管理、版本管理、资产文件管理、API 响应工具。

**次要问题**：
- `session-manager.ts` 与 `fs-utils.ts` 功能大量重叠（createSession/discardSession 等）
- JSON 文件读写模式重复 24+ 次，目录遍历模式重复 33 次
- ID 生成逻辑（`Math.random().toString(36)...`）在 8 处重复
- 121 处 `console.log/warn/error` 调试语句
- 全部使用同步文件 I/O（`readFileSync/writeFileSync`）
- Viewer CORS origins 硬编码 `["http://localhost:3300"]`
- 环境变量解析逻辑在 `ai-models.ts`、`model-config.ts`、`admin/model-config/route.ts` 三处重复
- `AGENT_SERVICE_URL` 在 4 处重复定义，环境变量名不一致

**建议**：拆分 fs-utils.ts 为 `paths.ts`、`project-repo.ts`、`demo-page-repo.ts`、`version-manager.ts`、`session-assets.ts`、`api-response.ts`。

### 5.3 agent-service — 分层边界模糊

**核心问题**：路由层职责过重，`routes/agent.ts`（557 行）和 `routes/websocket.ts`（625 行）包含业务逻辑（工作空间创建、快照初始化、会话同步），缺少 service 层。

**次要问题**：
- `core/types.ts` 反向依赖 `backends/pi-tools/permissions.ts`（违反分层原则）
- `pi-agent.ts` 大量 `any` 类型（`Agent`、`streamSimple`、`getModel` 等全部声明为 `any`）
- `ErrorCode` 枚举形同虚设，路由层散落 15+ 个未注册错误码
- 用 `throw new Error('PROJECT_NOT_FOUND')` 做业务流控制，路由层字符串匹配捕获
- `events/event-bus.ts` 是死代码（全局定义但从未使用）
- 6 个全局单例获取方式不统一（函数式 / 静态方法 / 直接导出实例混用）
- 双重 logger 实例（pino），生产环境硬编码 `pino-pretty`
- `snapshot-service.ts` 的 `execSync('git ...')` 存在命令注入风险
- `internal-config.ts` 的 token 比较未用时序安全方法
- AGENTS.md 与实际代码结构严重不同步
- `AgentInfo` 接口在 `core/agent.ts` 和 `core/agent-manager.ts` 各定义一份，字段略有不同
- `BackendWithModelSupport` 接口与 `IBackendAdapter` 重复定义可选方法
- `agent-factory.ts` 硬编码 `const type: AgentType = "pi-agent"`，工厂模式名存实亡

### 5.4 viewer-site — 单文件巨石

**核心问题**：`ViewerApp.tsx`（797 行）包含 5 个组件 + 3 个工具函数，`ProjectPreviewPage` 单组件约 350 行、12 个 `useState`。

**次要问题**：
- 15 个 UI 组件与 author-site 完全重复
- 零测试覆盖
- 内联 `<style>` 标签注入 `@keyframes`
- 空 `catch {}` 块静默吞掉错误

### 5.5 agent-client — HTTP 状态码未校验

**核心问题**：`request<T>()` 方法直接 `response.json()` 而不检查 `response.ok`，4xx/5xx 响应被静默传递。

**次要问题**：
- `StreamEvent` 所有字段均为可选，缺乏可辨识联合
- `package.json` 的 `main`（指向 dist）与 `exports`（指向 src）不一致

## 九、Docker 部署审查

### 9.1 镜像构建

| 服务 | 构建阶段 | 运行阶段 | 问题 |
|---|---|---|---|
| agent-service | `node:20-bookworm-slim` | `node:20-bookworm` | 运行镜像比构建镜像更大，应改 slim |
| author-site | `node:20-bookworm-slim` | `node:20-bookworm-slim` | 合理 |
| viewer-site | `node:20-bookworm-slim` | `nginx:alpine` | 合理 |

agent-service 运行阶段用 `npm install` 安装 10 个外部依赖，无 lock 文件固定版本，丧失可复现性。应改为完整 esbuild 打包或复制 node_modules。

### 9.2 安全配置

| 问题 | 严重度 |
|---|---|
| 所有容器以 root 运行 | 严重 |
| agent-service 端口暴露到主机 | 严重 |
| JWT_SECRET/ADMIN_SECRET 弱默认值 | 严重 |
| author-site 健康检查不验证状态码 | 高 |
| viewer-site 无健康检查 | 高 |
| nginx 缺少安全响应头和 gzip | 中 |
| `.env.docker` 未在 `.dockerignore` 中排除 | 中 |

### 9.3 服务编排

- `depends_on` 仅保证容器启动，不等待健康检查通过（应使用 `condition: service_healthy`）
- 两个服务（author-site + agent-service）同时读写同一个 `app-data` 卷，存在并发冲突风险
- 未定义自定义网络，缺少网络隔离

### 9.4 deploy.sh

- 硬编码服务器 IP `10.130.33.131` 和 `root` 用户
- 缺少回滚机制
- 容器名硬编码，项目目录名变化时失效

## 十一、下一步分析方向

1. **运行时分析**：启动 `pnpm dev` 后检查内存泄漏、性能瓶颈、WebSocket 稳定性
2. **构建产物分析**：`next build` 后分析 bundle 大小、tree-shaking 效果、共享代码拆分
3. **依赖版本审计**：切换到官方 npm registry 执行 `pnpm audit`（当前 npmmirror 不支持 audit 端点）
4. **无障碍性审查**：使用 Lighthouse 评估 WCAG 合规性
5. **API 契约测试**：验证 author-site 与 agent-service 的 API 接口一致性
