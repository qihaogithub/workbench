# 配置入口代码迁移

> 最终状态：已完成
> 创建日期：2026-06-28
> 完成日期：2026-06-28

## 背景

[Codex 开发模式下项目维护性剩余优化任务拆分](./Codex开发模式下项目维护性优化建议.md) 将“配置入口代码迁移”列为 P1-4。当前项目已经有配置入口治理原则，但业务代码中仍存在直接读取 `process.env`、重复默认值和服务 URL 散落的问题。

## 目标

- 每个服务优先通过本服务配置入口消费环境变量。
- 先迁移低风险配置：服务 URL、超时、数据目录和浏览器公开配置。
- 不改变外部环境变量名称和默认运行行为。

## 范围

- `packages/author-site/`
- `packages/viewer-site/`
- `packages/agent-service/`
- `packages/screenshot-service/`
- 相关配置治理文档和验收矩阵

## 方案

1. 盘点 `process.env` 读取点并按包分类。
2. 选择已有配置入口或新增轻量入口，不引入新依赖。
3. 第一批只迁移可验证、低风险的读取点，避免一次性重构模型配置或 session 状态。
4. 为新增配置入口补最小单元测试或脚本契约检查。

## 风险与待确认事项

- `NEXT_PUBLIC_*` 变量会进入浏览器产物，迁移时不能误改服务端私有变量的暴露边界。
- agent-service 模型配置影响真实 LLM 调用，本任务第一批不改变变量名、默认值或 provider 语义。
- agent-service 的 web-read、web-search、image-description 配置仍分散在工具和服务内；这部分属于工具域配置治理，后续应独立拆分，避免和服务 URL/数据目录迁移混在一起。
