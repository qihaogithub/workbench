## 一、问题本质

### 1.1 污染链路

当前应用设置了全局深色主题，CSS 变量和 `body` 样式被定义为深色：

```css
/* globals.css */
@layer base {
  :root {
    --background: 222.2 84% 4.9%;    /* 深色 */
    --foreground: 210 40% 98%;       /* 浅色文字 */
    --border: 217.2 32.6% 17.5%;     /* 深色边框 */
    /* ... */
  }
  * {
    @apply border-border;            /* 所有元素的边框颜色 */
  }
  body {
    @apply bg-background text-foreground;  /* 深色背景 + 浅色文字 */
  }
}
```

AI 生成的预览组件直接渲染在主 DOM 树中，遭受三类污染：

| 污染类型 | 影响 | 具体表现 |
|:---------|:-----|:---------|
| **CSS 变量污染** | shadcn/ui 组件使用 `bg-background` / `text-foreground` | 组件显示为深色主题 |
| **继承属性污染** | `color` 从 `body` 继承 | 未显式设色的文字变为浅色，在白色背景上看不见 |
| **全局选择器污染** | `* { border-color: ... }` | 边框颜色异常 |

### 1.2 为什么之前的方案不够

| 方案 | 问题 |
|:-----|:-----|
| `all: initial` CSS Reset | **破坏 Tailwind Preflight**。`h1`/`p`/`ul` 恢复浏览器默认样式（大字号、外边距等），而非 Tailwind 的扁平化样式。 |
| CSS 变量覆盖 (`.preview-sandbox`) | 只覆盖变量，**未解决继承属性**（如 `color`）和 **全局选择器**（`* { border }`）。新增加全局样式时容易遗漏。 |
| Shadow DOM | React 18 不原生支持，社区方案维护不活跃。 |
| iframe 沙箱 | 配置联动延迟 1-3s，与"动态编译 + 直接渲染"架构目标冲突。 |

**核心矛盾**：需要在保留 Tailwind Preflight 的同时，切断全局样式对未显式设置元素的继承污染。AI 生成的 Demo 应当完全自主决定自身主题（浅色、深色、彩色均可），不应受宿主应用全局样式的隐式影响。

## 二、推荐方案：四层防御体系

放弃"单点修复"思路，建立从组件到规范的分层防御：

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 4: 规范与自动化（防止回归）                              │
│  - 开发规范：新增全局样式必须检查隔离影响                        │
│  - 自动化检测：检查 globals.css 修改是否破坏预览                │
├──────────────────────────────────────────────────────────────┤
│  Layer 3: Portal 逃逸处理（边界场景）                           │
│  - Dialog / Dropdown 等组件默认挂载到 body                     │
│  - 通过注入自定义 Portal + CSS 兜底双重保障                     │
├──────────────────────────────────────────────────────────────┤
│  Layer 2: 全局样式防御（globals.css 重构）                     │
│  - 避免使用 `body { color: ... }` 等继承式全局样式              │
│  - 全局样式限定在应用根容器，不污染子树                         │
├──────────────────────────────────────────────────────────────┤
│  Layer 1: 容器级隔离（PreviewScope 组件）⭐ 核心               │
│  - 在预览容器上完整重写 CSS 变量                                │
│  - 显式重置关键继承属性                                         │
│  - 封装为可复用组件，统一所有预览入口                            │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 Layer 1：容器级隔离（核心）

创建 `PreviewScope` 组件，将所有隔离逻辑封装在组件内部。

**组件设计**：

```tsx
// components/demo/PreviewScope.tsx
'use client';

import React from 'react';

interface PreviewScopeProps {
  children: React.ReactNode;
  className?: string;
  theme?: 'light' | 'dark';
}

export function PreviewScope({
  children,
  className,
  theme = 'light',
}: PreviewScopeProps) {
  return (
    <div
      className={`preview-scope preview-scope--${theme}${className ? ` ${className}` : ''}`}
    >
      {children}
    </div>
  );
}
```

**配套 CSS**（追加到 `globals.css`）：

```css
/* ============================================
   预览区样式隔离 (.preview-scope)
   四层防御体系 —— Layer 1
   ============================================ */

.preview-scope {
  /* 1. 安全回退：系统级属性
     不强制 light 或 dark，让 AI Demo 自主决定 */
  color-scheme: normal;

  /* 2. 显式重置关键继承属性
     这些属性可能从 body / * 选择器继承而来 */
  text-align: left;
  line-height: 1.5;
  font-family: ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont,
    "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;

  /* 3. 重写 shadcn/ui 核心 CSS 变量为中性默认值
     作用：当 AI 生成的组件使用 bg-background/text-foreground 等语义类时，
     显示为可读的浅色默认，而非被宿主深色主题覆盖。
     AI 显式使用的具体 Tailwind 类名（如 bg-gray-900）优先级更高，可覆盖 */
  --background: 0 0% 100%;
  --foreground: 222.2 84% 4.9%;
  --card: 0 0% 100%;
  --card-foreground: 222.2 84% 4.9%;
  --popover: 0 0% 100%;
  --popover-foreground: 222.2 84% 4.9%;
  --primary: 222.2 47.4% 11.2%;
  --primary-foreground: 210 40% 98%;
  --secondary: 210 40% 96.1%;
  --secondary-foreground: 222.2 47.4% 11.2%;
  --muted: 210 40% 96.1%;
  --muted-foreground: 215.4 16.3% 46.9%;
  --accent: 210 40% 96.1%;
  --accent-foreground: 222.2 47.4% 11.2%;
  --destructive: 0 84.2% 60.2%;
  --destructive-foreground: 210 40% 98%;
  --border: 214.3 31.8% 91.4%;
  --input: 214.3 31.8% 91.4%;
  --ring: 222.2 84% 4.9%;
  --radius: 0.5rem;

  /* 4. 应用基础颜色（阻断从 body 的继承链）
     子元素显式声明的 Tailwind 类名会覆盖这些值 */
  background-color: hsl(var(--background));
  color: hsl(var(--foreground));
  border-color: hsl(var(--border));

  /* 5. 布局属性 */
  display: block;
  width: 100%;
  height: 100%;
  overflow: hidden;
  position: relative;
}

/* 对伪元素同样生效 */
.preview-scope::before,
.preview-scope::after {
  border-color: hsl(var(--border));
}

/* 子元素继承重置 */
.preview-scope * {
  border-color: hsl(var(--border));
}

.preview-scope *::before,
.preview-scope *::after {
  border-color: hsl(var(--border));
}

/* 深色主题模式（可通过 theme="dark" 启用） */
.preview-scope--dark {
  color-scheme: dark;
  --background: 222.2 84% 4.9%;
  --foreground: 210 40% 98%;
  --card: 222.2 84% 4.9%;
  --card-foreground: 210 40% 98%;
  --popover: 222.2 84% 4.9%;
  --popover-foreground: 210 40% 98%;
  --primary: 217.2 91.2% 59.8%;
  --primary-foreground: 222.2 47.4% 11.2%;
  --secondary: 217.2 32.6% 17.5%;
  --secondary-foreground: 210 40% 98%;
  --muted: 217.2 32.6% 17.5%;
  --muted-foreground: 215 20.2% 65.1%;
  --accent: 217.2 32.6% 17.5%;
  --accent-foreground: 210 40% 98%;
  --destructive: 0 62.8% 30.6%;
  --destructive-foreground: 210 40% 98%;
  --border: 217.2 32.6% 17.5%;
  --input: 217.2 32.6% 17.5%;
  --ring: 224.3 76.3% 48%;
  background-color: hsl(var(--background));
  color: hsl(var(--foreground));
}
```

**为什么这不会破坏 Tailwind Preflight**：

Tailwind Preflight 通过 `*` 选择器重置浏览器默认样式（如 `h1` 的 `margin`、`font-size`）。这些规则作用于**所有元素**，优先级为 `0-0-1`（一个元素选择器）。

`.preview-scope` 的优先级为 `0-1-0`（一个类选择器），高于 `*` 选择器。但 `.preview-scope` **只设置特定属性**（`background-color`、`color`、`border-color` 等），**不会覆盖** Preflight 设置的 `margin`、`font-size`、`font-weight` 等属性。

因此，Tailwind Preflight 对 `h1`/`p`/`ul` 的扁平化样式仍然完全生效。

### 2.2 Layer 2：全局样式防御

**问题**：`globals.css` 中 `body { @apply bg-background text-foreground; }` 使用了继承式全局样式，任何未显式覆盖的子元素都会受到影响。

**改进原则**：全局样式应限定在应用自身的根容器内，不依赖继承污染子树。

**当前 `globals.css` 的问题代码**：

```css
@layer base {
  body {
    @apply bg-background text-foreground;
  }
}
```

**改进方案**（可选，视实施成本决定）：

```css
@layer base {
  /* 不在 body 上设置继承属性，改为限定在应用根容器 */
  #app-root, [data-app-root] {
    @apply bg-background text-foreground;
  }
}
```

**短期妥协**：如果改动 `layout.tsx` 成本高，至少确保新增全局样式时遵循以下原则：

- ❌ 不在 `body` / `*` 上设置 `color`、`font-family` 等继承属性
- ❌ 不在 `*` 上设置可能影响组件的样式（如 `border-color`）
- ✅ 使用类选择器限定范围（如 `.app-dark { color: ... }`）
- ✅ 必须通过 `.preview-scope` 的隔离测试

### 2.3 Layer 3：Portal 逃逸处理

**问题**：shadcn/ui 的 Dialog、DropdownMenu、Popover、Tooltip 等组件使用 Radix UI 的 Portal，默认将内容渲染到 `document.body`，逃出 `.preview-scope` 的隔离范围。

**影响**：如果这些组件在预览区域内触发，弹窗内容会继承全局深色主题，显示异常。

**解决方案**：

#### 3.1 短期：CSS 兜底

在 `globals.css` 中，为从预览区域逃逸的 Portal 内容提供兜底样式：

```css
/* Portal 逃逸兜底：当 Portal 内容包含 preview-scope 内的元素时 */
/* 注意：这只能解决变量问题，无法解决布局问题（如定位基准） */
[data-radix-popper-content-wrapper] {
  /* 至少确保边框和基础变量不会完全错乱 */
  --border: 214.3 31.8% 91.4%;
}
```

> ⚠️ **局限性**：这是兜底方案，无法完全解决 Portal 的颜色主题问题。如果 AI 生成的 Demo 大量使用 Dialog / Dropdown，需要启用方案 3.2。

#### 3.2 长期：自定义 Portal 注入

为 AI 组件提供一个可感知预览容器的 Portal 实现：

```tsx
// lib/preview-portal.tsx
'use client';

import React from 'react';
import * as PortalPrimitive from '@radix-ui/react-portal';

const PreviewPortalContext = React.createContext<HTMLElement | null>(null);

export function PreviewPortalProvider({
  container,
  children,
}: {
  container: HTMLElement;
  children: React.ReactNode;
}) {
  return (
    <PreviewPortalContext.Provider value={container}>
      {children}
    </PreviewPortalContext.Provider>
  );
}

export function PreviewPortal({
  children,
  ...props
}: React.ComponentPropsWithoutRef<typeof PortalPrimitive.Portal>) {
  const container = React.useContext(PreviewPortalContext);
  return (
    <PortalPrimitive.Portal container={container || undefined} {...props}>
      <div className="preview-scope">{children}</div>
    </PortalPrimitive.Portal>
  );
}
```

在 `component-executor.ts` 的 `requireMap` 中注入自定义 Portal：

```typescript
// 需要修改 shadcn/ui 组件以支持自定义 Portal
// 或：在 requireMap 中提供包装过的 Dialog/Dropdown 组件

const previewRequireMap: Record<string, unknown> = {
  // ... 原有映射 ...
  '@radix-ui/react-portal': {
    Portal: PreviewPortal,
    Root: PreviewPortal,
  },
};
```

> ⚠️ **实施复杂度**：高。需要修改所有使用 Portal 的 shadcn/ui 组件（Dialog、DropdownMenu、Popover、Tooltip、Select 等），或提供对应的预览版本。
>
> **建议**：在 MVP 阶段，如果 AI 不生成含复杂弹窗的 Demo，Layer 3 可先不实施，但需在文档中记录此限制。

## 五、源头治理：AI 提示词规范

技术隔离（PreviewScope）是**兜底防线**。如果 AI 生成的代码本身遵循样式规范，可以从**源头减少**污染发生。

在 `demo-generator.md`（AI 代理提示词）中增加以下规则：

```markdown
### 样式隔离规范（必须遵守）

你的 Demo 组件渲染在宿主应用的 DOM 树中，宿主应用有全局样式（如深色主题的 body 背景）。如果不显式声明颜色，未设色的元素会继承宿主样式，导致显示异常。

为避免样式污染，请严格遵守：

1. **根容器必须显式声明背景和文字色**：最外层 `<div>` 必须有明确的背景色和文字色类，不要依赖默认颜色。
   - 浅色主题示例：`className="bg-white text-gray-900"`
   - 深色主题示例：`className="bg-gray-900 text-white"`
   - 彩色主题示例：`className="bg-gradient-to-r from-purple-500 to-pink-500 text-white"`
2. **不使用 shadcn/ui 语义变量类**：禁止使用 `bg-background`、`text-foreground`、`border-border`、`bg-primary`、`text-muted-foreground` 等。这些变量的值由宿主应用全局控制，不受你的组件控制。
3. **使用具体的 Tailwind 颜色类**：如 `bg-white`、`bg-gray-900`、`text-gray-900`、`text-white`、`border-gray-200` 等。不要使用依赖 CSS 变量的语义类。
4. **所有可见文字必须有颜色类**：不要假设文字颜色会正确继承，每个 `<p>`、`<span>`、`<h1>` 等文本元素都要有明确的 `text-*` 类。
5. **如果使用 shadcn/ui 组件**：为其显式传入 `className` 覆盖默认样式，不要依赖组件的默认变量。例如：`<Button className="bg-blue-600 text-white">`。
```

**为什么这是必要的补充**：

| 层面 | 作用 | 局限性 |
|:-----|:-----|:-------|
| **技术隔离（PreviewScope）** | 切断继承链，提供安全默认 | 无法阻止 AI 写出依赖语义变量的代码 |
| **AI 提示词规范** | 教育 AI 显式声明颜色 | 无法 100% 保证 AI 遵守，需要技术兜底 |

两者**互补**：提示词减少问题发生，技术隔离确保即使 AI 犯错也能正常显示。

## 六、与历史方案的区别

| | 历史方案（v1.0） | 本方案（v2.0） |
|:--|:----------------|:--------------|
| **思路** | 单点修复：加个 CSS 类覆盖变量 | 体系化防御：四层架构 |
| **覆盖范围** | 仅 CSS 变量 | 变量 + 继承属性 + 全局选择器 + Portal |

| **主题支持** | 仅浅色 | 浅色/深色可切换 |
| **回归防护** | 无 | 有规范和自动化检查机制 |
| **维护成本** | 低（但容易遗漏） | 中（但系统性强） |

## 九、总结

本方案的核心升级是**从"单点 CSS 覆盖"升级为"体系化防御"**：

1. **PreviewScope 组件**封装隔离逻辑，提供中性安全默认，允许 AI 生成任意主题
2. **显式重置继承属性**，解决 `color`、`border-color` 等继承链污染
3. **AI 提示词规范**从源头教育 AI 显式声明颜色，减少污染发生
4. **预留 Portal 处理机制**，为复杂组件提供解决路径
5. **建立开发规范**，防止未来新增全局样式时破坏隔离

技术隔离兜底 + AI 规范预防，两者互补，构成完整的样式隔离体系。
