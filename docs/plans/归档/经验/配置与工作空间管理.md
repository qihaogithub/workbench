## 一、配置单一来源设计原则

### 1.1 设计原则

配置与代码分离，职责清晰。

### 1.2 预览尺寸配置示例

**唯一配置来源**：`config.schema.json` 中的 `$demo.previewSize`

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$demo": {
    "previewSize": {
      "width": 375,
      "height": 667
    }
  },
  "title": "Banner Demo 配置",
  "type": "object",
  "properties": { ... }
}
```

**优势**：
- 同一个 demo 在不同页面使用时不需重复配置
- 产品/设计人员可直接修改 JSON 调整预览效果
- 避免多处配置导致的不一致

### 1.3 配置优先级

```
config.schema.json 中声明 > 默认值 (375×667)
```

## 二、workingDir 传递链路

### 2.1 完整数据流

```
Demo 编辑页面 (page.tsx)
  ↓ workingDir prop
AIChat 组件 (ai-chat.tsx)
  ↓ stream.send(message, id, { workingDir })
AgentStream (agent-client/src/client.ts)
  ↓ WebSocket 消息
Agent Service WebSocket 路由 (websocket.ts)
  ↓ AgentConfig.workingDir
Agent 实例 (agent-manager)
  ↓ ACP session/new { cwd: ... }
Agent CLI (opencode/claude/codex)
```

### 2.2 常见断点

| 断点位置 | 问题 | 修复 |
|---------|------|------|
| Demo 编辑页面 | `workingDir` 硬编码为 `undefined` | 从 session 创建结果中获取 `tempWorkspace` |
| WebSocket 路由 | `workingDir` 硬编码为 `process.cwd()` | 从 WebSocket 消息中读取 `message.workingDir` |
| AgentStream.send | 未传递 `workingDir` 参数 | 在 `SendMessageOptions` 中添加 `workingDir` 字段 |

### 2.3 排查方法

在关键路径添加日志追踪 `workingDir` 传递：

```typescript
// AIChat 组件
console.log('[AIChat] Sending message with workingDir:', workingDir)

// WebSocket 路由
logger.info({ workingDir: message.workingDir }, 'WebSocket message received')

// Agent Manager
logger.info({ workingDir: config.workingDir }, 'Agent getOrCreate')

// ACP Connection
logger.info({ cwd: this.workingDir }, 'Creating ACP session')
```

## 三、工作空间隔离机制

### 3.1 核心设计

```
┌─────────────────────────────────────────────────────────┐
│ Session Store                                            │
│ • sessionId ↔ workingDir 绑定                            │
│ • customWorkspace 标记（用户指定 vs 系统临时）             │
├─────────────────────────────────────────────────────────┤
│ Workspace Manager                                        │
│ • 创建临时工作空间（{backend}-temp-{timestamp}）           │
│ • 规范化用户指定路径                                      │
├─────────────────────────────────────────────────────────┤
│ Session Guard                                            │
│ • 路径白名单校验                                          │
│ • 文件变更检测（git/snapshot 模式）                       │
├─────────────────────────────────────────────────────────┤
│ Backend Adapter                                          │
│ • 启动时设置 cwd = workingDir                            │
│ • 文件操作限制在 workingDir 内                            │
└─────────────────────────────────────────────────────────┘
```

### 3.2 临时工作空间命名

```typescript
// 格式：{backend}-temp-{timestamp}
const tempDir = path.join(os.tmpdir(), `${backend}-temp-${Date.now()}`);
```

### 3.3 路径校验

```typescript
function validatePath(workingDir: string, targetPath: string): boolean {
  const resolved = path.resolve(workingDir, targetPath);
  return resolved.startsWith(path.resolve(workingDir));
}
```

## 四、实时文件同步架构

### 4.1 双重保障机制

```
Agent 写入文件 
  → file_operation WebSocket 事件（实时）
  → 更新 realtimeFilesRef（去重）
  → 防抖 100ms
  → onCodeUpdate/onSchemaUpdate ✅ 实时触发

AI 回复完成
  → finish 事件
  → 合并最终文件列表（event.files 或 realtimeFiles）
  → onCodeUpdate/onSchemaUpdate ✅ 兜底保障
```

### 4.2 关键实现

```typescript
// 使用 Map 去重（同一文件多次写入仅保留最新内容）
const realtimeFilesRef = new Map<string, { action: string; content?: string }>();

// 防抖处理（100ms 批量通知）
let fileUpdateTimer: NodeJS.Timeout | null = null;

stream.on("file_operation", (event: StreamEvent) => {
  if (event.fileOperation?.method === "fs/write_text_file") {
    realtimeFilesRef.set(path, { action: "modified", content });
    
    if (fileUpdateTimer) clearTimeout(fileUpdateTimer);
    fileUpdateTimer = setTimeout(() => {
      processRealtimeFiles();
      fileUpdateTimer = null;
    }, 100);
  }
});
```
