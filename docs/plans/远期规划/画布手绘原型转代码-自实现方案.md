## 一、决策背景

### 1.1 调研结论：开源项目不适合直接接入

对 tldraw、Excalidraw、OpenPencil 三个主流开源画布项目的调研结论：

| 项目                     | 不适合直接接入的原因                                                                                                                     |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **tldraw**               | v2+ 改为非标准许可，生产环境需购买商业许可（v4 要求 license key）；SDK 体量大，定制成本高                                                |
| **Excalidraw**           | 纯白板工具，无节点模型扩展能力，无代码生成能力，数据模型与项目不兼容                                                                     |
| **OpenPencil (pen-sdk)** | MIT 许可虽友好，但作为独立设计工具（CanvasKit WASM 渲染、Yjs CRDT 协作、Skia 矢量引擎），架构重量远超"嵌入式画布"需求；v0.7.x API 不稳定 |

### 1.2 核心判断

项目需要的画布能力范围有限：

```
需要：画框 + 标注 + 连线 + 结构化数据转代码
不需要：矢量编辑、布尔运算、复杂路径编辑、实时协作、设计变量系统
```

自实现代价远低于集成第三方 SDK，且更可控。

### 1.3 技术路线选择

**结构化数据驱动，而非多模态猜测。**

```
多模态方案（OpenPencil / draw-a-ui）：
  手绘像素 → AI 视觉识别 → 猜测意图 → 生成代码
  不确定性高、成本高、需要 Vision API

结构化数据方案（本项目）：
  用户绘制（矩形/文本/连线） → 结构化 CanvasNode[]
  → AI 直接读取 JSON → 生成代码
  零歧义、零猜测、纯文本 LLM 即可完成
```

### 3.2 连接线设计（ConnectorNode）

```typescript
// 连接线的核心设计：
// 1. 始终引用源/目标节点 ID（而非记录路径点）
// 2. 路径由渲染层根据锚点自动计算
// 3. 源/目标节点移动时连接线自动跟随

// 折线路由算法（orthogonal）：
//   sourceAnchor → 中间点 1 → 中间点 2 → targetAnchor
//   简单策略：先水平（X方向）再垂直（Y方向）
//   改进策略：走"冖"或"凵"字形，避开节点碰撞

function computeOrthogonalPath(
  source: { x: number; y: number; width: number; height: number },
  target: { x: number; y: number; width: number; height: number },
  sourceAnchor: AnchorPosition,
  targetAnchor: AnchorPosition,
): Array<{ x: number; y: number }> {
  // 获取锚点坐标
  const start = getAnchorPoint(source, sourceAnchor);
  const end = getAnchorPoint(target, targetAnchor);

  // L 形路由：先水平到中段，再垂直
  const midX = (start.x + end.x) / 2;
  return [
    start,
    { x: midX, y: start.y }, // 水平段
    { x: midX, y: end.y }, // 垂直段
    end,
  ];
}
```

### 4.3 连接线工具细节

连接线工具是 MVP 中唯一需要特殊处理的：

```
1. 用户选择 connector 工具
2. 鼠标移到节点上 → 显示锚点高亮（top/bottom/left/right/center）
3. 按住拖拽 → 从源锚点画线跟随鼠标（正交折线预览）
4. 鼠标悬停到目标节点 → 锚点高亮，线吸附到最近锚点
5. 松开 → 创建 ConnectorNode
```

锚点位置计算：

```typescript
function getAnchorPoint(
  node: { x: number; y: number; width: number; height: number },
  anchor: AnchorPosition,
): { x: number; y: number } {
  const { x, y, width, height } = node;
  switch (anchor) {
    case "top":
      return { x: x + width / 2, y: y };
    case "bottom":
      return { x: x + width / 2, y: y + height };
    case "left":
      return { x, y: y + height / 2 };
    case "right":
      return { x: x + width, y: y + height / 2 };
    case "center":
      return { x: x + width / 2, y: y + height / 2 };
  }
}
```

## 五、渲染架构

### 5.1 双层渲染

```
CanvasViewport (CSS transform: translate + scale)
  │
  ├── SVG 层（pointer-events: none）       ← 新增
  │     └── 矩形 / 文本 / 连接线 / 自由画笔
  │
  ├── 页面卡片层（DOM）                     ← 已有
  │     └── CanvasPageItem × N
  │
  └── 交互覆盖层（pointer-events: none）   ← 新增
        └── 锚点高亮 / 拖拽预览 / 连接线预览
```

分层原则：

| 层         | 渲染方式        | 交互                                                 | 说明                       |
| ---------- | --------------- | ---------------------------------------------------- | -------------------------- |
| SVG 层     | 手绘元素        | 不响应事件（pointer-events: none），事件由坐标层处理 | 避免 SVG 元素遮挡 DOM 事件 |
| DOM 层     | 已有页面卡片    | 响应鼠标事件                                         | 现有逻辑不变               |
| 交互覆盖层 | 临时 SVG/Canvas | 不响应事件                                           | 仅视觉反馈                 |

### 5.2 SVG 元素渲染

```typescript
// 每个 CanvasNode 类型对应一个 SVG 渲染器

function SvgLayer({ nodes }: { nodes: CanvasNode[] }) {
  // 仅渲染非 page 类型的手绘节点
  const drawNodes = nodes.filter(n => n.type !== 'page' && n.visible);

  return (
    <svg className="absolute inset-0 pointer-events-none" style={{ overflow: 'visible' }}>
      {drawNodes.sort((a, b) => a.zIndex - b.zIndex).map(node => (
        <SvgNode key={node.id} node={node} />
      ))}
    </svg>
  );
}

function SvgNode({ node }: { node: CanvasNode }) {
  switch (node.type) {
    case 'rectangle':
      return (
        <rect
          x={node.x} y={node.y}
          width={node.width} height={node.height}
          rx={node.style.borderRadius}
          fill={node.style.fill}
          stroke={node.style.stroke}
          strokeWidth={node.style.strokeWidth}
          opacity={node.style.opacity}
        />
      );
    case 'text':
      return (
        <text
          x={node.x} y={node.y + node.fontSize}
          fontSize={node.fontSize}
          fontFamily={node.fontFamily}
          fontWeight={node.fontWeight}
          fill={node.color}
          textAnchor={node.textAlign === 'center' ? 'middle' : 'start'}
        >
          {node.content}
        </text>
      );
    case 'connector':
      return <ConnectorPath node={node} nodes={allNodes} />;
    // freehand 等...
  }
}

// 连接线路径渲染（实时计算）
function ConnectorPath({ node, nodes }: { node: ConnectorNode; nodes: CanvasNode[] }) {
  const source = nodes.find(n => n.id === node.sourceId);
  const target = nodes.find(n => n.id === node.targetId);
  if (!source || !target) return null;

  const points = node.lineType === 'orthogonal'
    ? computeOrthogonalPath(source, target, node.sourceAnchor, node.targetAnchor)
    : [getAnchorPoint(source, node.sourceAnchor), getAnchorPoint(target, node.targetAnchor)];

  const pathD = points.map((p, i) => `${i === 0 ? 'M' : 'L'} ${p.x} ${p.y}`).join(' ');

  return (
    <>
      <path d={pathD} fill="none" stroke={node.style.stroke} strokeWidth={node.style.strokeWidth} />
      {node.label && (
        <text x={(points[0].x + points[points.length - 1].x) / 2}
              y={(points[0].y + points[points.length - 1].y) / 2 - 8}
              fontSize={12} textAnchor="middle" fill="#666">
          {node.label}
        </text>
      )}
    </>
  );
}
```

## 六、AI 代码生成管线

### 6.1 核心思路

将 CanvasNode[] 序列化为结构化 JSON 描述，直接作为系统 prompt 交给纯文本 LLM 生成代码。

**绝不使用多模态视觉能力。**

### 6.2 结构化序列化

```typescript
// 将画布节点转为 AI 可读的结构化描述
interface StructuredLayout {
  width: number;
  height: number;
  regions: Array<{
    id: string;
    type: "rectangle" | "text";
    role: string; // 语义角色（来自 semanticRole + label）
    rect: { x: number; y: number; width: number; height: number };
    text?: string; // 文本内容
    style?: Partial<CSSStyleDecleration>;
    children?: string[]; // 包含的子节点 ID（基于空间包含关系推断）
  }>;
  connections: Array<{
    from: string;
    to: string;
    label?: string;
  }>;
}

function serializeToStructuredLayout(nodes: CanvasNode[]): StructuredLayout {
  const rectNodes = nodes.filter(
    (n) => n.type === "rectangle" || n.type === "page",
  );
  const textNodes = nodes.filter((n) => n.type === "text");
  const connectors = nodes.filter(
    (n) => n.type === "connector",
  ) as ConnectorNode[];

  // 1. 计算画布总尺寸
  const bounds = computeContentBounds(nodes);

  // 2. 为每个矩形节点推断子节点（基于空间包含）
  const parentMap = inferParentChild(rectNodes);

  // 3. 找到与矩形关联的文本（基于位置重叠/邻近）
  const textMap = associateTextWithRect(rectNodes, textNodes);

  return {
    width: bounds.width,
    height: bounds.height,
    regions: rectNodes.map((r) => ({
      id: r.id,
      type: "rectangle",
      role:
        r.semanticRole === "other" || !r.semanticRole
          ? r.label || "container"
          : r.semanticRole,
      rect: { x: r.x, y: r.y, width: r.width, height: r.height },
      text: textMap.get(r.id),
      style: extractCSSStyle(r),
      children: parentMap.get(r.id),
    })),
    connections: connectors.map((c) => ({
      from: c.sourceId,
      to: c.targetId,
      label: c.label,
    })),
  };
}
```

### 6.4 代码生成流程

```
用户点击 "转代码"
  │
  ▼
1. 收集当前画布所有节点（nodes）
  │
  ▼
2. serializeToStructuredLayout(nodes)
  │  → 计算布局 → 推断父子关系 → 关联文本 → 拾取样式
  │  → 输出 StructuredLayout JSON
  │
  ▼
3. 调用 LLM（非多模态）
  │  输入：SYSTEM_PROMPT + USER_PROMPT(layout)
  │  输出：React + Tailwind 代码
  │
  ▼
4. 代码写入项目文件
  │  默认路径：src/demo/{pageName}/generated.tsx
  │
  ▼
5. 自动编译 + 预览刷新
  │  复用现有 iframe 通信协议（UPDATE_CODE）
  │
  ▼
6. （可选）在画布上创建一个 page 节点
  │  将生成代码作为一个新页面添加到画布
  │  用户可继续在画布上调整该页面布局
```

### 6.5 与 Figma 代码生成引擎的关系

Figma 插件代码生成引擎处理和渲染精确的设计稿；画布原型转码处理草稿阶段的粗粒度布局。

| 维度     | Figma 引擎               | 画布原型转码                    |
| -------- | ------------------------ | ------------------------------- |
| 输入精度 | 精确像素（Figma 节点树） | 粗略位置（CanvasNode）          |
| 布局引擎 | 基于坐标的绝对定位       | 基于区域语义的 Flexbox          |
| 输出质量 | 像素级还原               | 结构级还原（自动适配）          |
| 适用阶段 | 设计定稿后               | 设计草稿阶段                    |
| 技术共享 | —                        | 复用 AltNode 转换逻辑做样式映射 |

## 九、技术细节

### 9.1 坐标命中检测

手绘节点的交互事件处理策略：

```
SVG 层 pointer-events: none
  → 鼠标事件由 CanvasViewport 统一捕获
  → 通过 CoordinateTransform 转换到画布坐标
  → 遍历 nodes 进行命中检测（矩形范围 / 折线距离容差）
  → 选中/拖拽通过 Store action 更新节点数据
  → SVG 层响应式重渲染
```

连接线命中检测（非矩形，需要额外处理）：

```typescript
function hitTestConnector(
  canvasX: number,
  canvasY: number,
  nodes: CanvasNode[],
  hitRadius = 5,
): string | null {
  const connectors = nodes.filter(
    (n) => n.type === "connector",
  ) as ConnectorNode[];
  for (const conn of connectors) {
    const points = computeOrthogonalPath(/* ... */);
    // 计算点到折线的最小距离
    for (let i = 0; i < points.length - 1; i++) {
      const dist = pointToSegmentDistance(
        canvasX,
        canvasY,
        points[i],
        points[i + 1],
      );
      if (dist < hitRadius) return conn.id;
    }
  }
  return null;
}
```

### 9.2 锚点交互

连接线工具的锚点高亮需要知道连接线路径，这需要实时读取 Store 中的节点数据计算锚点位置。为避免每次鼠标移动都计算全部节点，可用空间索引优化：

```typescript
// 粗略快速过滤：只计算鼠标附近的节点的锚点
function getNearbyAnchorPoints(
  mousePos: { x: number; y: number },
  nodes: CanvasNode[],
  threshold: number,
): Array<{
  nodeId: string;
  anchor: AnchorPosition;
  pos: { x: number; y: number };
}> {
  const nearby = nodes.filter(
    (n) =>
      Math.abs(n.x + n.width / 2 - mousePos.x) < n.width / 2 + threshold &&
      Math.abs(n.y + n.height / 2 - mousePos.y) < n.height / 2 + threshold,
  );

  return nearby.flatMap((n) =>
    (["top", "bottom", "left", "right", "center"] as AnchorPosition[])
      .map((a) => ({
        nodeId: n.id,
        anchor: a,
        pos: getAnchorPoint(n, a),
      }))
      .filter(
        (p) =>
          Math.abs(p.pos.x - mousePos.x) < threshold &&
          Math.abs(p.pos.y - mousePos.y) < threshold,
      ),
  );
}
```

### 9.3 连接线重绘

当源/目标节点被拖拽时，连接线自动更新：

```
用户拖拽节点 → Store: updateNode(id, { x, y })
  → Store change event
  → SVG 层重新计算所有以该节点为 source/target 的连接线路径
  → 仅更新受影响连接线的 <path> 元素

优化：不在 Store 中存储路径点，路径始终实时计算。
     ConnectorNode 只存储 sourceId/targetId/anchors，不存路径。
     这样节点移动后连接线路径"自动"更新。
```

## 十、风险与缓解

| 风险                                     | 概率 | 影响 | 缓解                                                         |
| ---------------------------------------- | ---- | ---- | ------------------------------------------------------------ |
| SVG 在手绘节点 > 500 时性能下降          | 低   | 中   | 原型阶段节点数通常 < 100；可降级为 Canvas 渲染 freehand 节点 |
| 连接线折线路由在复杂布局中路径穿模       | 中   | 低   | MVP 先用简单 L 形路由；后续可加 A\* 避障                     |
| zustand Store 与现有 useState 并存期混乱 | 高   | 中   | 明确规则：新功能用 Store，旧状态逐步迁移，过渡期加 lint 规则 |
| AI 生成的代码质量低（布局失真）          | 中   | 中   | 初版用确定性模板 + AI 填充内容；后续迭代逐步放开 AI 自由度   |
| anchor 命中区域在多节点密集时重叠        | 低   | 低   | 取最近锚点；锚点选择增加偏移优先策略                         |

## 十二、总结

自实现方案的核心设计原则：

1. **结构化数据驱动** — 用户画出的每个元素都是类型明确的节点，AI 直接读取结构而非猜测像素
2. **增量演进** — 不推翻现有画布架构，在已有 DOM 层上叠加 SVG 层，工具系统渐进扩展
3. **低成本验证** — 矩形 + 文本 + 连接线三种节点即可覆盖 80% 的原型场景
4. **与现有能力互补** — 复用 Figma 代码生成引擎的设计理念和部分逻辑，避免重复造轮子

这不是一个要"追赶 OpenPencil"的计划，而是一个精准定位在"嵌入式画布草稿 → 结构化转码"场景的务实方案。
