## 一、目标与愿景

### 1.1 远期目标

在画布中支持用户手绘原型（矩形、线条、文本标注、自由画笔等），并通过 AI 将手绘原型自动转为可运行的 React 代码——类似 OpenPencil 的 "Design-as-Code" 体验。

### 1.2 OpenPencil 的核心启示

OpenPencil 是当前最前沿的开源 AI 原生设计工具（GitHub 4.6K Star），其核心架构决策值得参考：

| 决策 | OpenPencil 方案 | 对我们的启示 |
|---|---|---|
| 渲染引擎 | Skia (CanvasKit WASM) | 手绘/矢量元素需要专业渲染引擎，CSS + DOM 无法胜任 |
| 布局引擎 | Yoga WASM (Flexbox) | Auto-layout 能力是设计工具的标配 |
| 文件格式 | `.op` (人类可读 JSON)，Git-friendly | 画布数据需要结构化、可 diff、可版本管理 |
| AI 集成 | 内置 75 个工具 + MCP 协议 + 并发 Agent 团队 | AI 不只是对话，而是画布的一等公民操作者 |
| 工具系统 | 矩形/椭圆/钢笔/文本/布尔运算 | 画布工具必须可扩展，而非硬编码 |
| 状态管理 | Yjs (CRDT) | 协作和撤销/重做的基础设施 |

### 1.3 本文档定位

**不修改任何代码**，仅分析现状、识别技术债、规划架构演进路径，确保当前画布功能的每一步演进都为手绘原型转代码铺路，而非挖坑。

## 三、技术债清单

以下技术债若不清理，将直接阻碍手绘原型功能的实现。按严重度排序：

### T1. 画布节点模型硬编码为"页面" — 严重度：🔴 阻塞

**现状**：`CanvasPageLayout` 是唯一画布节点类型，只有 `x/y/width/height/zIndex` 五个字段。画布上只能放"页面卡片"。

**问题**：手绘原型需要矩形、线条、文本标注、自由画笔、连接线等多种节点类型。当前模型无法表达。

**目标**：抽象为通用 `CanvasNode` 模型，`CanvasPageLayout` 成为其中一种节点类型。

### T2. 无全局状态管理 — 严重度：🔴 阻塞

**现状**：所有状态（canvasState、canvasEditingPageId、previewMode、demoPages、configDataMap 等 20+ 个 useState）集中在 `edit/page.tsx`，该文件已超 2000 行。

**问题**：
- 添加手绘功能后状态量将翻倍（画笔状态、选中节点、绘制历史等），继续用 useState 将完全失控
- 无法实现撤销/重做（需要统一的状态快照）
- 无法实现跨组件状态共享（手绘工具栏需要读写画布状态）

**目标**：提取画布状态到独立的 Store（推荐 zustand，轻量、无 boilerplate、支持 middleware）。

### T3. 无撤销/重做机制 — 严重度：🔴 阻塞

**现状**：画布操作（拖拽、缩放、布局变更）没有 undo/redo 栈。

**问题**：手绘操作的撤销是基本需求。没有统一的操作历史栈，每次操作都是不可逆的。

**目标**：基于状态管理重构，实现 Command Pattern 或 Immer patches 的操作历史。

### T4. 画布布局不持久化 — 严重度：🔴 阻塞

**现状**：`canvas-utils.ts` 中的 `saveCanvasLayout()` 调用 `POST /api/sessions/:id/canvas-layout`，但该 API 路由**不存在**。每次刷新页面布局重置为 `computeInitialLayout()` 计算的初始网格。

**问题**：手绘元素的位置、样式、层级必须持久化，否则用户工作成果无法保存。

**目标**：实现画布布局持久化 API，扩展为完整的画布数据持久化（包含手绘节点数据）。

### T5. 渲染引擎基于 CSS + DOM — 严重度：🟡 架构限制

**现状**：画布渲染完全依赖 CSS `transform` + `position: absolute`，每个节点是一个 React 组件渲染的 DOM 元素。

**问题**：
- 自由画笔（数千个路径点）用 DOM 渲染性能不可接受
- 布尔运算（union/subtract/intersect）需要矢量渲染能力
- 旋转、非矩形变换在 CSS 中受限
- 大量节点时 DOM 节点数爆炸

**目标**：引入 Canvas/SVG 渲染层，至少对手绘元素使用 Canvas 渲染。可参考 OpenPencil 的 CanvasKit (Skia WASM) 方案，或使用 lighter 的 Fabric.js/Konva.js。

### T6. 工具系统不可扩展 — 严重度：🟡 架构限制

**现状**：`CanvasToolMode` 仅有 `"hand" | "select"` 两个值，硬编码在类型定义中。工具切换逻辑分散在 CanvasViewport 和 CanvasPageItem 中。

**问题**：手绘功能需要矩形工具、画笔工具、文本工具、连接线工具等，当前架构无法扩展。

**目标**：设计可插拔的工具系统，每个工具封装为独立的 Tool 模块（光标、交互逻辑、渲染层）。

### T7. 坐标系无抽象 — 严重度：🟡 架构限制

**现状**：画布坐标直接使用像素值，视口变换通过 CSS `transform` 实现。没有坐标变换层，所有坐标计算散落在各组件中。

**问题**：
- 旋转元素需要更精确的坐标系统
- 手绘元素的命中检测需要坐标变换
- 不同缩放级别下的交互精度不一致

**目标**：抽象坐标变换层，统一处理屏幕坐标 ↔ 画布坐标 ↔ 节点本地坐标的转换。

### T8. 对齐系统仅支持矩形页面 — 严重度：🟢 可演进

**现状**：`computeAlignment()` 只处理 `CanvasPageLayout` 类型的矩形对齐，8 个对齐点（left/right/center-x/top/bottom/center-y）。

**问题**：手绘元素的对齐需求更复杂（线条端点对齐、文本基线对齐、等间距分布等）。

**目标**：对齐系统需要支持任意节点类型的对齐点注册。

### T9. 截图服务与画布耦合 — 严重度：🟢 可演进

**现状**：截图生成逻辑通过 `useScreenshotGeneration` hook 硬编码在编辑页，截图服务独立运行在端口 3202。

**问题**：手绘元素不需要通过 Puppeteer 截图，而是直接从 Canvas 导出。截图服务的触发时机和策略需要与手绘渲染解耦。

**目标**：截图策略抽象为可配置的渲染管线，根据节点类型选择截图方式。

### T10. iframe 通信协议未充分利用 — 严重度：🟢 可演进

**现状**：`COLLECT_THUMBNAIL_LAYOUT` 消息类型可采集 iframe 内 DOM 布局信息（元素位置、样式、文本），但数据仅用于缩略图，未暴露给画布层做结构化分析。

**问题**：手绘原型转代码时，需要理解已有页面的结构来生成匹配的代码。DOM 布局数据是宝贵的结构化信息源。

**目标**：将 DOM 布局采集能力暴露为画布的"结构感知"接口，供 AI 转码时使用。

#### 4.2.3 撤销/重做机制（T3）

**方案选择**：zustand + immer + temporal middleware

```typescript
// zustand 的 temporal middleware 提供开箱即用的 undo/redo
import { temporal } from 'zundo';

const useCanvasStore = create<CanvasStoreState>()(
  temporal(
    immer((set, get) => ({
      // ... state and actions
    })),
    {
      limit: 50,  // 最多 50 步历史
      equality: (a, b) => JSON.stringify(a) === JSON.stringify(b),
    }
  )
);

// 使用

```

**为手绘铺路**：撤销/重做是手绘操作的基本需求。基于 Store 的 temporal middleware 可以统一处理所有画布操作的撤销，包括未来的手绘操作。

#### 4.3.1 通用 CanvasNode 模型（T1）

**核心设计**：将 `CanvasPageLayout` 泛化为 `CanvasNode`，页面卡片成为节点的一种类型。

```typescript

  // 屏幕 → 画布
  screenToCanvas(screenX: number, screenY: number): { x: number; y: number } {
    return {
      x: (screenX - this.viewport.x) / this.viewport.zoom,
      y: (screenY - this.viewport.y) / this.viewport.zoom,
    };
  }

  // 画布 → 屏幕
  canvasToScreen(canvasX: number, canvasY: number): { x: number; y: number } {
    return {
      x: canvasX * this.viewport.zoom + this.viewport.x,
      y: canvasY * this.viewport.zoom + this.viewport.y,
    };
  }

  // 节点本地 → 画布（考虑旋转）
  localToCanvas(node: CanvasNode, localX: number, localY: number): { x: number; y: number } {
    const rotation = node.rotation ?? 0;
    if (rotation === 0) {
      return { x: node.x + localX, y: node.y + localY };
    }
    const rad = (rotation * Math.PI) / 180;
    const cx = node.width / 2;
    const cy = node.height / 2;
    const dx = localX - cx;
    const dy = localY - cy;
    return {
      x: node.x + cx + dx * Math.cos(rad) - dy * Math.sin(rad),
      y: node.y + cy + dx * Math.sin(rad) + dy * Math.cos(rad),
    };
  }

  // 命中检测：屏幕坐标是否在节点内
  hitTest(screenX: number, screenY: number, node: CanvasNode): boolean {
    const canvas = this.screenToCanvas(screenX, screenY);
    // 简化版：不考虑旋转的矩形命中检测
    // 阶段 2 扩展为支持旋转和任意形状
    return (
      canvas.x >= node.x &&
      canvas.x <= node.x + node.width &&
      canvas.y >= node.y &&
      canvas.y <= node.y + node.height
    );
  }

  // 获取视口在画布坐标系中的可见范围
  getVisibleBounds(): { x: number; y: number; width: number; height: number } {
    return {
      x: -this.viewport.x / this.viewport.zoom,
      y: -this.viewport.y / this.viewport.zoom,
      width: this.containerSize.width / this.viewport.zoom,
      height: this.containerSize.height / this.viewport.zoom,
    };
  }
}
```

### 4.4 阶段 2：手绘渲染引擎

#### 4.4.1 双层渲染架构

```
CanvasViewport
  ├── SVG/Canvas 渲染层（手绘元素）    ← 新增
  │     ├── 矩形节点
  │     ├── 椭圆节点
  │     ├── 线条节点
  │     ├── 文本节点
  │     ├── 自由画笔节点
  │     └── 连接线节点
  │
  └── DOM 渲染层（页面卡片）           ← 现有
        └── CanvasPageItem × N
              └── PreviewPanel / <img>
```

**方案选择**：

| 方案 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|
| **SVG** | 声明式、事件友好、与 React 集成好 | 大量节点时性能不如 Canvas | ★★★★ |
| **Canvas 2D** | 高性能、适合大量路径点 | 命中检测需手动实现、无 DOM 事件 | ★★★ |
| **CanvasKit (Skia WASM)** | 专业级渲染、OpenPencil 验证 | 包体积大（~2MB WASM）、学习成本高 | ★★ |
| **Konva.js** | Canvas 封装、事件系统完善 | 额外依赖、与 React 集成需 react-konva | ★★★ |
| **Fabric.js** | Canvas 封装、对象模型成熟 | 较重、与 React 集成需手动桥接 | ★★ |

**推荐方案**：SVG 优先，性能瓶颈时局部切换到 Canvas。

理由：
1. SVG 与 React 天然集成，每个节点是一个 SVG 元素，事件处理零成本
2. 手绘原型阶段节点数量通常 < 200，SVG 完全胜任
3. 需要高性能场景（自由画笔数千点）时，仅将 freehand 节点用 Canvas 渲染
4. 避免引入 CanvasKit 的 2MB WASM 包体积

#### 4.4.2 渲染管线

```typescript
// 画布渲染管线
function CanvasRenderer({ nodes, viewport }: CanvasRendererProps) {
  return (
    <CanvasViewport viewport={viewport}>
      {/* 手绘元素层（SVG） */}
      <svg className="absolute inset-0" style={{ pointerEvents: 'none' }}>
        {nodes
          .filter(n => n.type !== 'page' && n.visible)
          .sort((a, b) => a.zIndex - b.zIndex)
          .map(node => (
            <NodeRenderer key={node.id} node={node} />
          ))}
      </svg>

      {/* 页面卡片层（DOM） */}
      {nodes
        .filter(n => n.type === 'page' && n.visible)
        .sort((a, b) => a.zIndex - b.zIndex)
        .map(node => (
          <PageNodeRenderer key={node.id} node={node} />
        ))}
    </CanvasViewport>
  );
}

// 节点渲染器（按类型分发）
function NodeRenderer({ node }: { node: CanvasNode }) {
  switch (node.type) {
    case 'rectangle':
      return <RectangleNode node={node} />;
    case 'ellipse':
      return <EllipseNode node={node} />;
    case 'line':
      return <LineNode node={node} />;
    case 'text':
      return <TextNode node={node} />;
    case 'freehand':
      return <FreehandNode node={node} />;
    default:
      return null;
  }
}
```

## 五、关键设计决策

### 5.1 为什么不直接引入 OpenPencil？

| 维度 | OpenPencil | 本项目 |
|---|---|---|
| 定位 | 独立设计工具 | 嵌入式画布（编辑器的一部分） |
| 渲染 | CanvasKit (Skia WASM, ~2MB) | 需要轻量，不能拖慢首屏 |
| 运行时 | Tauri 桌面应用 | 浏览器 Web 应用 |
| 数据 | `.op` 独立文件格式 | 嵌入现有项目数据结构 |
| 协作 | WebRTC P2P + Yjs CRDT | 暂无协作需求 |

**结论**：OpenPencil 的架构理念值得借鉴，但不能直接引入。本项目需要的是"画布内嵌手绘能力"，而非"替换为独立设计工具"。

### 5.2 为什么选择 zustand 而非 Redux/Jotai/Valtio？

| 维度 | zustand | Redux | Jotai | Valtio |
|---|---|---|---|---|
| 包体积 | ~1KB | ~7KB | ~3KB | ~3KB |
| Boilerplate | 极少 | 多 | 少 | 少 |
| Undo/Redo | zundo middleware | 手动 | 手动 | 手动 |
| 非 React 使用 | 支持 | 需额外配置 | 不支持 | 支持 |
| 学习成本 | 低 | 中 | 低 | 低 |

**选择 zustand**：最轻量、undo/redo 开箱即用、非 React 组件（如工具系统）可直接读写状态。

### 5.3 为什么选择 SVG 而非 Canvas 作为手绘渲染层？

1. **节点数量可控**：手绘原型阶段通常 < 200 个节点，SVG 完全胜任
2. **事件处理零成本**：每个 SVG 元素天然支持 DOM 事件，命中检测免费
3. **React 集成**：SVG 是声明式的，与 React 的编程模型一致
4. **渐进升级**：性能瓶颈时仅将 freehand 节点切换到 Canvas，无需全量替换
5. **调试友好**：SVG 元素可在 DevTools 中直接检查

### 5.4 持久化格式为什么用 JSON 而非二进制？

1. **Git-friendly**：JSON 可 diff、可 merge，适合版本管理
2. **调试友好**：开发者可直接阅读和编辑
3. **AI 可读**：LLM 可以理解 JSON 结构并生成/修改画布数据
4. **OpenPencil 验证**：OpenPencil 的 `.op` 格式也是人类可读 JSON

## 六、当前画布功能应遵循的约束

在阶段 0 完成之前，当前画布功能的任何修改都应遵循以下约束，避免增加技术债：

### 6.1 数据模型约束

- **不新增 CanvasPageLayout 字段**：任何新字段应放在 `data` 扩展字段中，为后续 CanvasNode 迁移做准备
- **不硬编码节点类型**：所有条件判断使用 `node.type` 而非 `instanceof` 或类型守卫
- **持久化数据加版本号**：所有新增的持久化数据结构必须包含 `version` 字段

### 6.2 状态管理约束

- **不新增 useState**：画布相关状态应逐步迁移到 Store，而非继续在 `edit/page.tsx` 中新增 useState
- **状态变更通过 action**：不直接修改状态对象，通过 Store action 修改，为 undo/redo 做准备
- **状态粒度合理**：避免一个巨型 state 对象，按职责拆分为多个 slice

### 6.3 组件约束

- **不硬编码工具逻辑**：新增交互逻辑时，考虑是否应抽象为 Tool 模块
- **渲染路径可扩展**：CanvasPageItem 的渲染逻辑应保持分支清晰，为新增节点渲染器留位置
- **坐标计算统一**：所有坐标计算应收敛到 CoordinateTransform，不散落在各组件中

### 6.4 API 约束

- **画布数据 API 独立于页面数据 API**：画布布局/节点数据的 CRUD 应有独立端点，不混在 `/api/sessions/:id/files` 中
- **API 响应格式预留扩展**：使用 `{ version, data }` 包装，方便未来数据迁移

## 九、总结

当前画布功能已有坚实的交互和性能基础（视口裁剪、rAF 节流、截图优先、对齐辅助线、边框式缩放、键盘快捷键等），但架构层面存在三个阻塞级技术债：

1. **节点模型硬编码** — 画布只能放页面卡片，无法扩展为手绘元素
2. **无状态管理库** — 画布状态集中在 useState 中，无法支撑撤销/重做和跨组件共享
3. **布局不持久化** — API 不存在，用户工作无法保存

此外还有两个架构限制需要在手绘功能之前解决：
4. **工具系统不可扩展** — 仅 hand/select 两种工具，无法添加手绘工具
5. **坐标系无抽象** — 坐标计算散落各组件，无法支持旋转和精确命中检测

这三个问题必须在引入手绘功能之前解决。本文档规划的演进路线从清债开始，逐步抽象节点模型和工具系统，最终实现手绘原型转代码的完整能力。

每一步都应遵循"不增加技术债"的原则：新代码为未来扩展留接口，旧代码逐步迁移到新架构，而非在旧架构上打补丁。
