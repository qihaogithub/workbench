# 创作端管理后台方案

## 一、需求概述

为创作端( author-site )增加一个管理后台,允许管理员通过特定链接进入,配置以下全局设置:

- **模型供应商配置**: API Key、API Base、供应商名称、模型列表
- **模型白名单**: 允许在前端显示的模型分组前缀
- **模型黑名单**: 禁止在前端显示的完整模型 ID
- **默认模型列表**: 按优先级排列的默认选中模型
- **多模态支持**: 哪些模型支持图片输入
- **名称过滤器**: 分组级别的模型名称关键词过滤

**核心目标**: 无需修改 `.env` 文件或重启服务,通过 Web UI 即可动态调整模型配置。

## 二、架构设计

### 2.1 技术选型

| 层级 | 方案 | 说明 |
|------|------|------|
| **存储** | SQLite 新表 `system_configs` | 与现有 `users.db` 同一数据库,统一管理 |
| **鉴权** | Admin Secret (环境变量) | 独立的 secret key,通过 URL 参数或 Cookie 验证 |
| **前端** | Next.js App Router 页面 | 在 `packages/author-site/src/app/admin/` 下新增 |
| **API** | Next.js Route Handlers | 在 `packages/author-site/src/app/api/admin/` 下新增 |
| **UI 组件** | shadcn/ui + Tailwind CSS | 复用现有组件库 |

### 2.2 数据流

```
用户访问 /admin?secret=xxx
  ↓
middleware.ts 验证 Admin Secret
  ↓
加载管理后台页面 (读取当前配置)
  ↓
修改配置 → 提交到 /api/admin/model-config
  ↓
API 写入 SQLite system_configs 表
  ↓
前端模型配置读取逻辑改为优先读数据库, fallback 到环境变量
```

## 三、数据库设计

### 3.1 新增表: system_configs

在 `data/users.db` 中新增表:

```sql
CREATE TABLE system_configs (
  id TEXT PRIMARY KEY,           -- 配置项唯一标识,如 "model_config"
  config_json TEXT NOT NULL,     -- JSON 格式的配置数据
  updated_at INTEGER NOT NULL,   -- 最后更新时间 (Unix timestamp)
  updated_by TEXT                -- 可选: 修改者标识
);
```

### 3.2 配置 JSON 结构

```typescript
{
  // 供应商配置 (对应用于生成 opencode.json)
  provider: {
    name: string;                // 供应商名称,如 "xjjj"
    apiKey: string;              // API Key
    apiBase: string;             // API 基础地址
    models: string[];            // 模型名称列表,逗号分隔转数组
  },
  
  // 前端模型过滤配置
  frontend: {
    allowedPrefixes: string[];   // 白名单: ["xjjj/", "jojo/"]
    blacklist: string[];         // 黑名单: ["xjjj/old-model"]
    defaultModelIds: string[];   // 默认模型: ["xjjj/deepseek-v4-flash"]
    nameFilters: string[];       // 名称过滤: ["opencode:Free"]
  },
  
  // 多模态模型标记
  multimodalModels: string[];    // 支持图片的模型 ID: ["xjjj/gpt-4-vision"]
  
  // 元数据
  lastSyncedToEnv?: number;      // 最后同步到环境变量的时间 (可选功能)
}
```

## 五、鉴权机制

### 5.1 Admin Secret 配置

在 `.env` 中新增:

```env
# 管理后台访问密钥 (生产环境必须修改)
ADMIN_SECRET=admin-change-this-to-random-string
```

### 5.2 验证流程

1. **首次访问**: 用户访问 `/admin?secret=xxx`
2. **验证**: middleware 检查 `secret` 参数是否匹配 `ADMIN_SECRET`
3. **设置 Cookie**: 验证通过后设置 `admin_token` Cookie (httpOnly, 2 小时过期)
4. **后续访问**: 通过 Cookie 验证,无需重复输入 secret

### 5.3 Middleware 改造

在 `packages/author-site/src/middleware.ts` 中新增:

```typescript
const ADMIN_ROUTES = ['/admin'];
const ADMIN_API_ROUTES = ['/api/admin/'];

// 验证 Admin Secret (参数或 Cookie)
async function verifyAdminSecret(request: NextRequest): Promise<boolean> {
  const secretParam = request.nextUrl.searchParams.get('secret');
  const adminCookie = request.cookies.get('admin_token')?.value;
  
  const expectedSecret = process.env.ADMIN_SECRET || 'admin-change-this-to-random-string';
  
  if (secretParam === expectedSecret) return true;
  if (adminCookie === hashSecret(expectedSecret)) return true;
  
  return false;
}
```

## 七、配置读取逻辑改造

### 7.1 现有逻辑

当前模型配置直接从环境变量读取 (`packages/author-site/src/lib/ai-models.ts`):

```typescript
const raw = process.env.NEXT_PUBLIC_ALLOWED_MODEL_PREFIXES || "";
```

### 7.2 改造方案

新增配置读取层:

```typescript
// packages/author-site/src/lib/model-config.ts

interface CachedConfig {
  data: ModelConfigData;
  lastFetched: number;
}

let cachedConfig: CachedConfig | null = null;
const CACHE_TTL = 60 * 1000; // 1 分钟缓存

export async function getModelConfig(): Promise<ModelConfigData> {
  // 1. 检查缓存
  if (cachedConfig && Date.now() - cachedConfig.lastFetched < CACHE_TTL) {
    return cachedConfig.data;
  }
  
  // 2. 尝试从 API 读取数据库配置
  try {
    const res = await fetch('/api/admin/model-config');
    if (res.ok) {
      const { data } = await res.json();
      cachedConfig = { data, lastFetched: Date.now() };
      return data;
    }
  } catch (e) {
    // API 不可用时 fallback 到环境变量
  }
  
  // 3. Fallback: 从环境变量读取 (保持向后兼容)
  return readFromEnv();
}
```

### 7.3 ai-models.ts 改造

将原有的环境变量读取函数改为调用新的配置层:

```typescript
// 原: const raw = process.env.NEXT_PUBLIC_ALLOWED_MODEL_PREFIXES || "";
// 改: const config = await getModelConfig();
//     const raw = config.frontend.allowedPrefixes.join(',');
```

**注意**: 由于 `ai-models.ts` 部分函数在服务端构建时执行,需要区分服务端/客户端读取逻辑。

## 八、API 实现

### 8.1 获取配置 API

`packages/author-site/src/app/api/admin/model-config/route.ts`:

```typescript
import { NextResponse } from 'next/server';
import { verifyAdminRequest } from '@/lib/admin-auth';
import { readDbConfig } from '@/lib/db-config';

export async function GET(request: Request) {
  if (!await verifyAdminRequest(request)) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  
  const config = await readDbConfig('model_config');
  return NextResponse.json({ success: true, data: config });
}
```

### 8.2 更新配置 API

```typescript
export async function PUT(request: Request) {
  if (!await verifyAdminRequest(request)) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  
  const body = await request.json();
  await writeDbConfig('model_config', body);
  
  // 可选: 触发配置热更新 (清除缓存、通知其他实例)
  
  return NextResponse.json({ success: true, message: '配置已保存' });
}
```

### 8.3 数据库操作封装

`packages/author-site/src/lib/db-config.ts`:

```typescript
import Database from 'better-sqlite3';
import path from 'path';

const DB_PATH = path.join(process.env.DATA_DIR || 'data', 'users.db');

export async function readDbConfig(id: string): Promise<any | null> {
  const db = new Database(DB_PATH);
  const row = db.prepare('SELECT config_json FROM system_configs WHERE id = ?').get(id);
  db.close();
  return row ? JSON.parse(row.config_json) : null;
}

export async function writeDbConfig(id: string, config: any): Promise<void> {
  const db = new Database(DB_PATH);
  db.prepare(`
    INSERT INTO system_configs (id, config_json, updated_at)
    VALUES (?, ?, ?)
    ON CONFLICT(id) DO UPDATE SET config_json = ?, updated_at = ?
  `).run(
    id, 
    JSON.stringify(config), 
    Date.now(),
    JSON.stringify(config),
    Date.now()
  );
  db.close();
}
```

## 九、配置生效机制

### 9.1 实时生效策略

由于 Next.js 服务端渲染的特性,配置更新后需要:

1. **客户端组件**: 通过 SWR/React Query 自动重新获取配置
2. **服务端组件**: 
   - 使用内存缓存 (TTL 1 分钟)
   - 或提供 `/api/admin/reload-config` 端点手动触发缓存失效

### 9.2 缓存失效方案

```typescript
// packages/author-site/src/lib/model-config.ts

let configCache: any = null;

export function invalidateConfigCache() {
  configCache = null;
}

// API 端点: /api/admin/reload-config
export async function POST(request: Request) {
  invalidateConfigCache();
  return NextResponse.json({ success: true });
}
```

### 9.3 环境变量同步 (可选功能)

提供"同步到 .env"按钮,将数据库配置写回环境变量文件:

```typescript
import fs from 'fs';
import dotenv from 'dotenv';

export async function syncToEnvFile(config: any) {
  const envPath = path.join(process.cwd(), '.env');
  const envConfig = dotenv.parse(fs.readFileSync(envPath));
  
  // 更新环境变量
  envConfig.NEXT_PUBLIC_ALLOWED_MODEL_PREFIXES = config.frontend.allowedPrefixes.join(',');
  envConfig.NEXT_PUBLIC_MODEL_BLACKLIST = config.frontend.blacklist.join(',');
  // ... 其他配置
  
  // 写回文件
  fs.writeFileSync(envPath, Object.entries(envConfig)
    .map(([k, v]) => `${k}=${v}`)
    .join('\n'));
}
```

## 十、供应商配置热更新

### 10.1 问题

供应商配置 (API Key, API Base) 修改后,需要更新 `opencode-serve` 容器的 `opencode.json`,这通常需要重启容器。

### 10.2 解决方案

**方案 A: 动态配置 API** (推荐)

`opencode-serve` 提供配置更新端点,无需重启:

```typescript
// 调用 OpenCode Server 的配置更新 API
await fetch('http://opencode-serve:4096/api/admin/reload-config', {
  method: 'POST',
  body: JSON.stringify(newProviderConfig)
});
```

**方案 B: 提示用户重启**

管理后台显示提示:

> "供应商配置已保存,但需要重新构建 opencode-serve 容器才能生效。
> [查看重启命令]"

```bash
docker compose down opencode-serve && docker compose build opencode-serve && docker compose up -d opencode-serve
```

**方案 C: 仅支持前端配置热更新**

管理后台仅管理前端配置 (白名单/黑名单/默认模型),供应商配置仍通过 `.env` 管理。

**推荐**: 先实现方案 C (MVP),后续根据需求支持方案 A。

## 十二、安全考虑

### 12.1 Admin Secret 安全

- **必须**: 生产环境使用强随机字符串 (至少 32 字符)
- **建议**: 支持 IP 白名单限制访问
- **建议**: 记录配置修改日志 (审计)

### 12.2 API Key 保护

- API Key 在数据库中**加密存储** (使用 `crypto` 模块)
- 前端展示时**部分隐藏** (如 `sk-xxxx****xxxx`)
- 传输时使用 HTTPS

### 12.3 CSRF 防护

- 管理后台表单使用 CSRF Token
- API 端点验证 `Content-Type` 和来源

## 十六、风险与限制

### 16.1 已知限制

1. **多实例部署**: 如果有多个 author-site 实例,需要共享 SQLite 或使用 Redis 缓存
2. **供应商配置热更新**: 依赖 `opencode-serve` 是否提供配置重载 API
3. **构建时配置**: 部分 Next.js 优化可能缓存环境变量,需验证

### 16.2 回退方案

- 数据库配置损坏时,自动 fallback 到环境变量
- 提供"重置为默认配置"按钮
- 保留配置历史版本 (可选)
