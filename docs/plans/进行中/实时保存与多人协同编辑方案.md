# 实时保存与多人协同编辑方案

## 背景

创作端项目编辑当前以 Session 绑定 Workspace 的草稿模式为基础，用户手动保存时再生成项目版本。为了支持 2-3 人同时编辑同一项目，需要在不改变版本快照语义的前提下，让草稿工作区可以实时同步与落盘。

## 目标

- 支持单实例部署下多人同时编辑同一 Workspace 中的页面代码、配置文件、富文本内容和知识文档。
- 草稿实时保存到 Workspace，手动保存继续生成版本快照。
- 展示协同连接状态和在线协作者。
- 保持现有 Session、Workspace、版本和发布流程兼容。

## 范围

- 本轮优先覆盖 `demos/*/index.tsx`、`demos/*/config.schema.json`、项目级 Schema、`workspace-tree.json`、画布布局文件、富文本内容和知识文档的协同通道。
- 前端接入 Workspace 文件编辑弹窗、编辑页当前页面代码/Schema、项目级 Schema、页面树、画布布局、富文本/知识文档编辑入口和协作者状态展示；页面树结构操作继续走原 API，并将最新 `workspace-tree.json` 广播到协同通道。
- 富文本/知识文档与代码类资源共用同一协同 WebSocket、房间隔离、权限校验、flush 和落盘机制；编辑器侧按内容形态适配 Yjs 文档模型。
- 不支持多实例横向扩展，不引入 Redis/Postgres 协调层。

## 方案

- 在 agent-service 中新增协同 WebSocket 路由，使用 Yjs 协议维护每个资源的文档状态。
- 每个协同房间按项目、workspace、资源路径隔离，连接时校验 session 与 workspace 归属。
- 服务端从 Workspace 文件加载初始内容，Yjs 更新 debounce 后原子写回磁盘。
- author-site 使用 `y-websocket` 和 `y-codemirror.next` 接入 CodeMirror 协同编辑。
- 富文本/知识文档编辑器复用协同资源协议，按资源路径进入同一房间管理与持久化流程；富文本结构化内容需要在进入 Yjs 文档前后保持序列化格式稳定。
- 保存按钮先 flush 协同草稿，再沿用现有保存/版本流程。

## 风险与待确认事项

- 单实例、小规模使用场景下的剩余风险见 [实时保存与多人协同编辑风险分析](./实时保存与多人协同编辑风险分析.md)。
- 当前实现面向单实例部署，多实例部署需要共享 Yjs update 或外部协调层。
- 页面树排序/重命名和画布结构变更仍以现有 API 为主；协同通道已支持对应资源类型，后续可继续改成细粒度 CRDT 操作。
- 富文本/知识文档需要确认现有编辑器的数据模型、序列化格式和最小变更粒度；若当前内容不是纯文本，需要避免协同更新破坏块结构或知识文档元数据。
- 非协同路径直接写文件时，本轮以重新打开资源同步最新磁盘内容为主，后续可补充冲突提示 UI。
