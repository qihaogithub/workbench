## 一、背景

当前预览系统采用“服务端 sucrase 编译 + iframe 沙箱 + 远程 ESM CDN 运行时”的架构。这个方向在隔离性、实现成本和 AI 实时编辑体验上是正确的：服务端编译很轻，iframe 能隔离不可信页面代码，配置变更也能通过 `UPDATE_CONFIG` 避免重新编译。

真正拖慢预览首屏的不是 TSX 编译，而是 iframe 内部运行时加载链路：

- iframe 模板默认从 `https://esm.sh` 加载 React、ReactDOM 和 `react/jsx-runtime`。
- iframe 模板还从 jsdelivr 加载 Tailwind CDN 脚本。
- 页面模块通过 Blob URL 再动态 import，只有模块 import 成功并发出 `LOADED` 后，父窗口才移除加载遮罩。
- 单页预览已有截图文件时，首屏状态没有主动从本地截图 meta 预填占位图。
- 首次进入编辑页时还可能启动截图批量生成任务，与预览 iframe 争用本机 Chrome、CPU 和网络资源。

历史方案主要围绕“换 CDN、预加载 CDN、延长缓存”优化。这些措施能降低部分等待，但仍把首屏确定性交给外部网络。长期最优方案应把预览运行时从远程 CDN 链路中移出，让预览的关键路径尽量同源、可缓存、可观测、可灰度。

## 二、目标

### 2.1 用户体验目标

- 编辑页单页预览冷启动稳定在可接受范围内，避免长期停留在加载圈。
- 配置变更继续保持近实时响应，不触发重新编译和重新加载运行时。
- 页面切换、单页/画布切换不反复重建完整运行时。
- 首屏优先显示可用旧截图或上一次成功预览，避免空白等待。

### 2.2 架构目标

- 保留 iframe 作为不可信 AI 页面代码的安全边界。
- 保留 sucrase 服务端编译，不引入浏览器端打包。
- 将核心运行时和已登记依赖本地化为同源资源。
- 将编译结果模块内容寻址化，便于 HTTP 缓存、诊断和复用。
- 将截图生成移出首屏关键路径。
- 让预览依赖策略与 `@preview/sdk` 成为长期契约，而不是继续追求任意 npm 的无限兼容。

### 2.3 非目标

- 不恢复 Direct Render，不在 author-site React 树内直接执行 AI 生成代码。
- 不引入 Sandpack、browserify、webpack-in-browser 等浏览器端打包方案。
- 不把“可导入任意 npm 包”作为近期目标；未知依赖仍应通过运行时契约校验失败。
- 不在本方案中改变页面配置 Schema、配置面板、AI 对话协议的产品语义。

## 四、当前约束与关键判断

### 4.1 当前架构中应保留的部分

| 能力 | 判断 | 原因 |
| --- | --- | --- |
| iframe 沙箱 | 保留 | AI 页面代码不可信，iframe 是 DOM、CSS、JS 隔离边界 |
| sucrase 编译 | 保留 | 当前编译成本低，符合实时预览需求 |
| `UPDATE_CODE` / `UPDATE_CONFIG` 双通道 | 保留 | 代码变更与配置变更的成本模型不同 |
| 依赖白名单 | 强化 | 当前已收敛到 react、react-dom、lucide-react、framer-motion、`@preview/sdk`，适合本地化 |
| `@preview/sdk` | 强化 | 它是替代任意 npm 能力膨胀的长期出口 |

### 4.2 当前架构中应替换的部分

| 问题 | 当前表现 | 长期替代 |
| --- | --- | --- |
| 核心运行时依赖远程 CDN | React/ReactDOM 首屏受 esm.sh 影响 | 同源 vendor chunk |
| iframe HTML 每次 Blob 化 | shell 生命周期与页面模块耦合 | 固定 shell URL |
| 编译结果只走 postMessage 字符串 | 难以 HTTP 缓存和定位 | 内容寻址模块 URL |
| 截图任务进入首屏并发 | 页面刚打开就抢资源 | 本地 meta 预填 + idle 后台再生 |
| 加载状态只有 spinner | 没有分阶段诊断 | 编译、shell、模块、资源分段埋点 |

### 4.3 为什么不只是换 CDN

换 CDN 仍然依赖外部 ESM 服务、查询参数兼容、跨域缓存和网络稳定性。它适合作为短期缓解，不适合作为长期最优。当前项目已经有依赖白名单和 `@preview/sdk`，这意味着本地化运行时的边界是清晰的，不需要为“任意 npm”继续牺牲确定性。

## 五、长期最优方案：同源 Preview Runtime + 常驻 Shell

### 5.1 总体架构

```
编辑页
  │
  ├─ 创建或复用 iframe shell
  │    └─ /preview-runtime/shell.html
  │         ├─ 同源加载 React / ReactDOM / jsx-runtime
  │         ├─ 同源加载已登记第三方依赖
  │         ├─ 内置 ErrorBoundary / postMessage 协议
  │         └─ 常驻视觉编辑与批注通信层
  │
  ├─ /api/compile 编译页面代码
  │    ├─ sucrase TSX -> ESM
  │    ├─ 依赖契约校验
  │    ├─ import 重写为同源 runtime specifier 或同源 URL
  │    └─ 返回 moduleHash / moduleUrl / diagnostics
  │
  └─ iframe shell 接收 UPDATE_MODULE
       ├─ import(moduleUrl)
       ├─ render default component
       ├─ 发送 LOADED / RUNTIME_ERROR
       └─ 配置变更只走 UPDATE_CONFIG
```

### 5.2 运行时资源本地化

将当前预览运行时允许的依赖打包为同源静态资源：

| 依赖 | 来源 | 本地化形式 |
| --- | --- | --- |
| `react` | workspace 依赖 | vendor ESM chunk |
| `react-dom/client` | workspace 依赖 | vendor ESM chunk |
| `react/jsx-runtime` | workspace 依赖 | vendor ESM chunk |
| `lucide-react` | 已登记依赖 | vendor ESM chunk |
| `framer-motion` | 已登记依赖 | vendor ESM chunk |
| `@preview/sdk` | 项目内虚拟能力 | runtime SDK chunk |

建议新增一个轻量构建脚本，在 author-site 构建前产出 `public/preview-runtime/manifest.json` 和带 hash 的 vendor 文件。manifest 记录 specifier 到同源 URL 的映射，供服务端编译器和 iframe shell 同时读取。

运行时依赖版本应由 package lock 和 manifest 共同固定，而不是运行时请求 esm.sh 再解析。升级 React、lucide 或 framer 时，通过项目验证矩阵统一升级。

### 5.3 固定 iframe shell

将当前 `generateIframeHtml()` 从“每次生成 Blob HTML”演进为固定 shell：

- shell 位于同源静态路径或 Next route。
- shell 只负责运行时，不包含具体页面代码。
- shell 生命周期尽量长，页面切换只更新 module URL 和 props。
- shell 内继续保留父子窗口来源验证、ErrorBoundary、`RESIZE`、`CONSOLE_LOG`、视觉编辑消息。

短期可以继续由 `generateIframeHtml()` 生成 shell 内容，但输出目标从 Blob 迁移为固定 URL。这样能先减少运行时重复创建，再逐步清理模板生成逻辑。

### 5.4 编译结果内容寻址化

当前 `/api/compile` 返回 `compiledCode` 字符串，父窗口把字符串发给 iframe，iframe 创建 Blob URL 再 import。长期应改为：

- 服务端根据编译后代码内容生成 `moduleHash`。
- 编译结果存入内存 LRU 或 `DATA_DIR/preview-modules/`。
- 新增只读模块路由，例如 `/api/preview-modules/{hash}.js`。
- `/api/compile` 返回 `moduleUrl`、`dependencies`、`cssImports`、`diagnostics`。
- iframe 通过 `import(moduleUrl)` 加载页面模块。

这样做的收益：

- 相同代码天然复用浏览器 HTTP 缓存。
- 模块 URL 可在日志和性能面板中定位。
- 截图服务、author-site、viewer-site 可以复用同一模块交付协议。
- 不再依赖 Blob URL 的缓存行为和调试体验。

迁移阶段可以兼容 `compiledCode` 和 `moduleUrl` 双返回。旧 PreviewPanel 继续消费 `compiledCode`，新 shell 优先消费 `moduleUrl`。

### 5.5 import 重写策略

当前 `rewriteImportsToCdn()` 将裸 specifier 改写为 esm.sh URL。新方案改成同源 runtime 映射：

| import 类型 | 新策略 |
| --- | --- |
| `react` / `react/*` | 重写到 manifest 中的同源 URL |
| `react-dom` / `react-dom/*` | 重写到 manifest 中的同源 URL |
| `lucide-react` | 重写到 manifest 中的同源 URL |
| `framer-motion` | 重写到 manifest 中的同源 URL |
| `@preview/sdk` | 重写到同源 SDK chunk |
| CSS import | 保持现有 CSS 列表机制，后续再纳入同源资产策略 |
| 其他 npm import | 继续契约校验失败，不动态放行 |
| 相对源码 import | 继续禁止，页面保持单文件边界 |

也可以选择“保留裸 specifier + shell import map”模式，但服务端重写为同源 URL 更直接，截图服务和静态发布链路更容易复用。若后续需要支持多版本 runtime，再引入 import map 作为版本路由层。

### 5.6 iframe 池与画布模式

单页预览只需要一个常驻 shell。画布模式不应为所有页面长期保留活跃 iframe，应采用分层渲染：

- 当前可编辑页：活跃 iframe shell。
- 邻近或最近访问页：少量 warm iframe 池。
- 其他页面：截图占位。
- 页面进入编辑态时，从截图切到 warm iframe 或创建新 shell。
- 页面离开活跃范围时，发 `SLEEP`，必要时回收到池中。

这与当前 `sleeping-iframe` 和截图占位方向一致，但长期应把策略提升为明确的资源调度器，而不是散落在页面组件里。

### 5.7 截图从首屏关键路径剥离

截图服务的长期定位是“画布占位、缩略图和历史视觉缓存”，不应与单页预览首屏竞争资源。

调整原则：

- 页面加载时先读取本地截图 meta，立即构造 `placeholderScreenshotUrl`。
- iframe 真实内容加载完成后再淡入替换占位。
- 代码或配置变更后，先标记截图过期，但保留旧截图作为弱占位。
- 截图再生放到 debounce + idle + 服务健康检查之后。
- 单页模式优先保证 iframe 交互，画布模式再批量生成截图。

这样用户看到的是“旧画面快速出现，然后更新”，而不是“空白卡片加加载圈”。

### 5.8 加载态与可观测性

当前 `showPreviewLoading = isCompiling || !contentLoaded` 只暴露一个总加载圈。长期应拆成阶段：

| 阶段 | 事件 | 用户侧表现 | 诊断价值 |
| --- | --- | --- | --- |
| compile | `/api/compile` 开始/结束 | 通常不显示独立状态 | 判断服务端编译是否异常 |
| shell | iframe shell ready | 占位图或轻加载 | 判断 iframe 初始化是否异常 |
| module | 页面模块 import | 小型加载指示 | 判断模块与依赖加载耗时 |
| render | React render 完成 | 淡入真实页面 | 判断页面运行时错误 |
| assets | 图片/字体等资源 | 不阻塞 `LOADED`，但可记录 | 判断视觉完成慢的原因 |

`LOADED` 应代表组件模块已经成功挂载，不应等待所有图片完全加载。图片和外部资源加载可作为单独的 `ASSET_SETTLED` 或性能日志。

## 六、分阶段实施计划

### 阶段 1：本地化核心运行时

- [x] 新增 preview runtime manifest 生成脚本。

## 九、风险与待确认事项

### 9.1 构建复杂度

本地化 runtime 会引入 manifest 生成脚本和 vendor chunk。需要控制它只服务预览 runtime，不要扩散成新的全局构建系统。

### 9.2 多版本兼容

如果历史发布页面依赖旧 CDN URL，不能直接切断旧路径。需要为创作中页面、新发布页面、历史发布页面定义 runtime 版本策略。

### 9.3 第三方依赖边界

本方案默认继续限制 npm 依赖。若未来业务确实需要更多包，应优先通过 `@preview/sdk` 提供语义能力；只有高频、稳定、可验证的依赖才纳入 manifest。

### 9.4 CSS 与 Tailwind

当前 iframe 模板使用 Tailwind CDN 脚本。长期应评估是否改为本地预构建 Tailwind runtime 样式，或约束 AI 页面只使用已知 class 集合。这个问题会影响视觉还原，但不应阻塞 React runtime 本地化。

### 9.5 安全边界

固定 shell 和同源模块路由仍在 iframe sandbox 中运行。模块路由必须只返回服务端编译产物，不能开放任意文件读取，也不能接受可穿越路径。

## 十一、方案结论

长期最优解不是继续围绕远程 CDN 做补丁，而是把预览运行时变成项目自己的同源基础设施：

- iframe 沙箱继续负责安全隔离。
- sucrase 继续负责快速 TSX 编译。
- 依赖白名单和 `@preview/sdk` 继续负责能力边界。
- 同源 runtime 负责稳定、可缓存、可观测的加载体验。
- 内容寻址模块负责代码变更后的高效复用。
- 截图和 iframe 池负责多页面场景的资源调度。

这条路线改动较大，但可以分阶段落地。阶段 0 和阶段 1 就能显著改善当前加载慢问题；阶段 3 之后，预览架构会从“动态 CDN 页面执行器”演进为“项目内置预览运行时平台”。
