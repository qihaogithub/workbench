# 创作端项目编辑页预览区

## 当前状态

本文档专门沉淀创作端项目编辑页预览区问题。当前保留三类仍有接手价值的问题：“原型页接入截图服务优化”、“预览画布页面框尺寸不一致”和“闯关活动项目画布批量预览白屏/空块”。

## 原型页接入截图服务优化

当前状态：已完成实施并通过目标验证。截图服务已从“高保真 React 页截图工具”升级为“运行时无关的页面快照机制”：高保真页和 HTML/CSS 原型页都进入统一 snapshot pipeline，但原型页只在大画布、非编辑、有效快照存在时消费截图；编辑态、小项目和截图失效时继续使用现有 `prototype` 轻量渲染。

### 当前实现结论

- 页面运行时类型已统一为 `prototype-html-css` 与 `high-fidelity-react`；编辑页会把原型页的 `prototypeHtml`、`prototypeCss`、`prototypeMeta`、`configData` 和 `previewSize` 传给 `PreviewCanvas` 与截图批次。
- `PreviewCanvas` 与 `CanvasPageItem` 已有通用截图层，能按 `screenshotUrls` 和 `screenshotRenderBoxes` 展示静态快照并修正画布高度。
- 原型页在小项目和编辑态继续使用 `prototype` 轻量渲染；大项目非编辑原型页有有效截图时可切到 `screenshot`，且不占 iframe 预算。
- 编辑页 `buildScreenshotBatchPages` 已按页面 `runtimeType` 构造 `PageSnapshotInput`；前端截图 hook 与 screenshot-service 请求协议支持 React 与原型页 payload。
- screenshot-service 高保真分支保持 `compileCode(code) -> generateIframeHtml -> Puppeteer`，原型分支使用共享原型渲染核心生成静态 HTML 后进入同一 Puppeteer/cache/write 流程。
- 当前截图机制继续要求 hash 精确匹配、禁止旧图，并以 `renderBox` 校验画布尺寸口径；原型页接入沿用这些约束。

### 长期架构原则

1. 统一快照协议：截图服务不直接关心页面类型，只接收标准化 `PageSnapshotInput`，再按 `runtimeType` 构建可截图 HTML。
2. 共享原型渲染核心：原型页截图必须与 `PrototypePagePreview` 共用清洗、配置绑定、viewport unit 归一化和尺寸规则，避免真实预览与截图漂移。
3. 截图是派生产物：页面事实仍来自当前代码、HTML/CSS、配置和尺寸；任何 hash 不匹配、截图失败、服务离线或尺寸不兼容，都不能展示旧图。
4. 成本模型调度：画布调度器按编辑价值和渲染成本决策，而不是按页面类型硬分叉。高保真 live 成本最高，原型 live 成本较低，snapshot 成本最低；editing/active 必须 live，background 优先 snapshot。

### 关键方案

#### 统一 PageSnapshotInput

截图请求新增运行时无关的页面快照输入，旧 React 请求默认兼容为 `high-fidelity-react`：

```ts
type PageSnapshotInput =
  | {
      runtimeType: "high-fidelity-react";
      code: string;
      configData: Record<string, unknown>;
      previewSize: PreviewSize;
    }
  | {
      runtimeType: "prototype-html-css";
      prototypeHtml: string;
      prototypeCss: string;
      prototypeMeta?: Record<string, unknown>;
      configData: Record<string, unknown>;
      previewSize: PreviewSize;
    };
```

snapshot hash 基于标准化输入计算，至少包含 `runtimeType`、渲染输入、配置、`previewSize`、`fullPage`、截图运行时 identity 和 `snapshotSchemaVersion`。`prototypeMeta` 只应纳入影响渲染的字段，避免无视觉影响的元数据变更触发重截。

#### 共享原型渲染核心

把原型页预览里的纯逻辑抽到共享层，至少包含：

- `sanitizePrototypeHtml`
- `sanitizePrototypeCss`
- `applyPrototypeBindings`
- `normalizePrototypeViewportUnits`
- `buildPrototypePreviewDocumentHtml`

浏览器端 `PrototypePagePreview` 和 screenshot-service 都调用同一套函数。screenshot-service 的原型分支生成完整静态 HTML document，不执行用户脚本、不加载远程脚本、不依赖 React runtime；页面尺寸与 `previewSize` 对齐，长页面继续通过 `fullPage` 和 `renderBox.height` 表达。

#### 画布消费策略

原型页不把截图作为主路径，只在以下条件同时成立时使用截图：

- 页面总数达到截图阈值，沿用当前 `MIN_CANVAS_SCREENSHOT_PAGE_COUNT = 6`。
- 页面不是当前明确编辑页，也不是 active 属性编辑页。
- 存在当前版本有效截图，且 `hash/expectedHash/renderBox` 与当前输入兼容。

其他情况继续使用 `prototype` 轻量渲染。截图服务离线、截图失败、截图缺失、hash 不匹配或 `renderBox` 不兼容时，原型页不进入 iframe 预算，直接回退 `prototype`。

### 分阶段实施方案

第一阶段：协议与缓存闭环。

- 扩展 author-site 代理、`useScreenshotGeneration` 和 screenshot-service 请求 schema，支持 `runtimeType` 与原型页 payload。
- snapshot hash 改为基于 `PageSnapshotInput` 的标准化输入计算；React 页保持旧协议兼容。
- 批量和单页截图状态仍复用现有 `expectedHash`、轮询、取消旧批次、文件读取和 `renderBox` 规则。

第二阶段：原型渲染分支。

- 拆分 screenshot-service 的“构建截图 HTML”和“Puppeteer 渲染/写缓存”职责。
- React 分支保持 `compileCode -> generateIframeHtml`。
- 原型分支调用共享 `buildPrototypePreviewDocumentHtml` 生成静态 HTML，再进入同一 Puppeteer 渲染和缓存写入流程。

第三阶段：画布调度。

- `buildScreenshotBatchPages` 不再过滤 `prototype-html-css`，改为按 `runtimeType` 构造 `PageSnapshotInput`。
- 保存 `prototype.html`、`prototype.css`、影响渲染的 `prototypeMeta`、配置数据或 `previewSize` 变化时，立即失效旧截图并按 debounce 再生。
- 调整 `computeCanvasRenderModes`：小项目和编辑态原型页继续 `prototype`；大项目非编辑原型页有有效截图时进入 `screenshot`；无有效截图时仍为 `prototype`。

第四阶段：文档与旧入口收口。

- 若继续保留 `/api/screenshots/ensure`，必须基于页面 `runtimeType` 读取 `index.tsx` 或 `prototype.html/css/meta`，避免历史入口只覆盖高保真页。
- 代码落地后再更新 `docs/项目文档/创作端/04-配置与预览/技术/07_截图服务与预览快照机制.md`，把长期事实从“编译 React iframe HTML”改为“按页面运行时构建快照 HTML”。

### 验收标准

- 20 个原型页加 2 个高保真页的项目进入画布后，截图批次 payload 同时包含两类页面。
- 非编辑原型页截图完成后可切到 `screenshot` 模式；选中或进入编辑态后仍切回 `prototype`，属性编辑、配置绑定和视觉选择可用。
- 修改 `prototype.html`、`prototype.css`、配置数据或 `previewSize` 后，旧截图立即失效，不展示旧图。
- 截图服务关闭、截图失败或原型页缺少 `prototypeHtml` 时，原型页仍显示 `prototype` 轻量预览。
- React 页截图协议保持兼容，现有高保真页截图、fast/strict variant、`renderBox` 和批量轮询行为不退化。

### 风险

- 如果原型截图模板与 `PrototypePagePreview` 分叉，最容易出现截图和真实预览不一致；必须优先抽共享纯函数。
- 原型页允许远程图片、CSS 背景和本地资源时，截图服务的资源解析必须和 author-site 预览一致，否则图片会在截图里缺失。
- 原型页截图是大画布性能优化，不应反向增加首屏等待；首屏仍应先显示轻量原型预览，再用截图替换非编辑页。
- `prototypeMeta` 当前结构较松，hash 只应包含影响渲染的字段；如果直接全量 hash，会让无视觉影响的 meta 更新触发批量重截。
- 第一阶段不引入智能 DOM 成本评分；只用页面数量、编辑态和截图有效性做确定性调度，避免过早引入难解释策略。

## 闯关活动项目画布批量预览白屏/空块

现象：`proj_1782839405716_tqjl1f` 单页预览可进入，但切换画布模式后页面框长时间显示白/灰色空块，且触发“检测到预览异常，正在自动修复”。

当前结论：直接原因是 `prototype-01` 当前项目工作区代码被重复拼接，导致画布批量预览调用 `/api/compile` 时返回 422。错误为 `DUPLICATE_TOP_LEVEL_DECLARATION` 和 `MULTIPLE_DEFAULT_EXPORTS`，指向 `prototype-01` 的重复 `interface DemoProps` 与 `export default function Demo`。单页模式可能显示旧成功结果或进入自动修复状态，画布批量编译会稳定暴露该坏页。

修复摘要：已清理 `data/projects/proj_1782839405716_tqjl1f/workspace/demos/prototype-01/index.tsx` 中第二段重复模块；浏览器自动修复流程也把 active live workspace 的 `prototype-01` 恢复为单份模块。

验证状态：`node packages/project-cli/bin/ow.mjs project validate-runtime proj_1782839405716_tqjl1f --json` 通过，27 个页面 `issues: []`；Playwright 复现曾捕获画布 `/api/compile` 422 并确认根因，修复后不再继续用浏览器触发自动修复写入。前序 25 秒浏览器采样显示画布冷启动期间大量 iframe 处于编译/加载状态，且当前 fit 后缩放约 7%，页面内容在画布上非常小，后续若用户仍认为“白屏”，优先区分“编译硬错误”与“冷启动/缩放/截图兜底体验问题”。

后续事项：如果需要继续优化该项目画布体验，优先检查批量预览是否应先展示已有截图缓存、限制冷启动 iframe 数量，或在 27 页项目首次进入画布时避免全量 fit 到过小缩放。

## 预览画布页面框尺寸不一致

当前结论：最终根因是 `PreviewPanel` 在画布工作台中读取了被 `CanvasViewport` CSS transform 缩放后的容器尺寸，并把该屏幕尺寸再次用于 iframe 内部缩放。当前 active workspace 中“平板”页面 schema、页面布局和页面代码已经对齐 `1133x749`；截图缓存 meta 与 PNG 尺寸也都是 `1133x749`，因此前序“历史 `.canvas-layout.json` 缺少 `previewSize` 兼容校验”是真实相邻缺陷，但不是当前仍复现的最终根因。

已落地行为：`CanvasPageLayout` 增加 `sizeMode: "preview" | "custom"` 与 `previewSizeKey`；画布布局合成前按当前页面 `previewSize` 归一化历史布局，历史坏尺寸只保留 `x/y/zIndex`；用户手动 resize 写入 `custom`，右键“重置大小”恢复 `preview`；内容高度上报不再使用过期 `layout.width` 推导高度；canvas-layout API、workspace 发布读取和 agent-service `arrangeCanvasPages` 保留新增尺寸语义字段；`PreviewPanel` 初始容器量测优先使用未受 transform 影响的 `clientWidth/clientHeight`，只在无布局尺寸时回退 `getBoundingClientRect()`。

待办：用户在真实浏览器页面确认 bundle 刷新后，画布 zoom 小于 1 时 iframe 内容是否不再二次缩小；若仍异常，优先检查 `PreviewPanel` 容器量测值是否等于布局尺寸 `1133x749`，而不是画布 zoom 后的屏幕尺寸。

验证状态：`corepack pnpm --filter @opencode-workbench/author-site test -- --testPathPatterns=preview-panel-sleep --runInBand` 通过；`corepack pnpm --filter @opencode-workbench/author-site test -- --testPathPatterns=canvas-layout --runInBand` 通过。`corepack pnpm --filter @opencode-workbench/author-site typecheck` 当前失败在既有 `src/app/api/sessions/[sessionId]/workspace-flush/route.test.ts` 的 `TestResponse` 重复标识符，与本次画布尺寸链路无直接关系。前序 `corepack pnpm check:agent` 通过。

风险：当前修复依赖 `clientWidth/clientHeight` 在画布 transform 下返回布局尺寸；若未来容器改成非 HTMLElement 或不可布局节点，需要保留 ResizeObserver/contentRect 作为主路径。
