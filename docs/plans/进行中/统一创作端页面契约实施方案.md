# 统一创作端页面契约实施方案

## 背景

创作端页面源码的可运行规则目前分散在 author-site 编译器、Agent prompt、CLI 静态校验和外部 skill 说明中。CLI 能完成项目事务写入，但无法提前保证页面能通过创作端真实预览链路，导致批量生成页面时容易出现白屏、依赖不兼容或规则漂移。

## 目标

- 建立一份可执行的创作端页面契约，作为 author-site、project-cli、创作端 Agent 和 Codex skill 的共同规则源。
- 让 CLI 在提交前能发现页面源码运行契约问题。
- 让创作端 Agent prompt 从同一份契约生成规则文本，避免 prompt 与运行时校验不一致。
- 保持 author-site 仍是最终预览编译入口。

## 范围

- 新增共享包 `@opencode-workbench/preview-contract`。
- 改造 author-site 编译链路复用共享契约。
- 改造 project-cli 增加运行时契约校验命令，并把预览编译静态校验升级为页面级契约校验。
- 改造创作端 Agent prompt 注入契约生成的规则。
- 新增 Codex skill 作为流程编排说明，不复制底层规则。
- 更新相关长期项目文档。

## 方案

1. 抽出 preview contract 包，集中维护依赖策略、源码契约校验、编译前包装、Sucrase 转换和 Agent 规则文本生成。
2. author-site `compiler.ts` 只保留运行时 URL 解析与编译入口职责，具体源码契约和基础编译逻辑委托给共享包。
3. project-cli 引入共享包，新增页面/项目运行时校验命令；`preview compile` 复用同一套页面级校验输出。
4. Agent 静态 prompt 动态插入共享包生成的创作规则。
5. Codex skill 只描述事务化工作流和必须运行的 CLI 校验命令。

## 风险与待确认事项

- 旧项目可能已经存在不符合新契约的页面；首阶段只对新增或修改页面严格阻断，历史页面以 warning 形式暴露。
- author-site 仍需保留最终编译与 runtime URL 解析职责，不能把浏览器/manifest 相关逻辑迁入 Node-only contract 包。
- 截图缓存与画布低缩放占位属于预览体验链路，不纳入首阶段源码契约。
- Codex skill 文件尚未落盘：当前会话对 `.agents/skills` 只有读权限，且权限审查拒绝了 unsandboxed 写入请求。需要用户显式授权后创建 `opencode-page-authoring` skill。
- 当前 node_modules 是系统 pnpm v11 恢复出的依赖树；`pnpm-lock.yaml` 已恢复为项目要求的 pnpm 8 v6 格式。正式提交前建议在可联网环境运行 `corepack pnpm install --no-frozen-lockfile`，再跑 `corepack pnpm check:project-cli`、`corepack pnpm check:project-core` 和 author-site 定向测试。
