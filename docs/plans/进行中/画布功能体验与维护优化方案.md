# 画布功能体验与维护优化方案

> 创建日期：2026-06-21
> PM 复审日期：2026-06-21
> 状态：进行中（收敛后保留）
> 范围：画布现有体验与维护质量优化，暂不扩展新功能范式

> 复审结论：画布当前最值得投入的方向不是继续加大功能，而是补齐“可保存、可恢复、状态可见、维护边界清楚”的基础体验。保留布局持久化、状态收拢、截图反馈、工具心智和关键测试；删除手绘原型、AI 深度融合、复杂 undo/redo、command/action 模型、待审 patch 汇总等短期价值不清或依赖条件不足的内容。

---

## 一、产品判断

创作端画布已经具备多页预览、缩放平移、页面拖拽缩放、对齐吸附、截图替代 iframe、可见页裁剪等能力。它目前不像“功能缺失”，更像“可信度不足”：用户做了移动、缩放、切换和截图刷新后，系统没有给出足够明确的保存、恢复和失败反馈。

从专业产品经理视角，当前应该优先解决三件事：

| 优先级 | 保留方向 | 产品价值 |
| :--- | :--- | :--- |
| P0 | 画布布局保存与恢复 | 让用户相信画布操作不会丢失，是后续所有画布能力的基础 |
| P0 | 画布状态从编辑页收拢 | 降低维护成本，减少状态联动导致的体验回归 |
| P1 | 页面识别与后台日志 | 让用户清楚识别画布页面，同时把截图、保存等后台状态留给日志排查 |

暂不投入的方向：

| 不做方向 | 删除原因 |
| :--- | :--- |
| 手绘、连接线、自由节点 | 会把画布从多页工作台扩展成白板工具，产品边界过早变宽 |
| 画布内 AI 深度操作流 | 当前更需要先稳定预览、保存和状态反馈，AI 入口依赖批注与截图链路成熟 |
| undo/redo 和 command/action 模型 | 现阶段没有足够复杂的可逆操作，过早抽象会增加工程负担 |
| 画布汇总待审 patch、未处理批注数量 | 依赖批注、AI 修改和审阅流程先成熟，短期不是画布体验的主矛盾 |

| 文件 | 当前职责 |

| `packages/shared/src/demo/PreviewCanvas.tsx` | 画布顶层组件，负责初始布局、可见页计算、对齐吸附、工具栏和页面项编排 |

| `packages/shared/src/demo/CanvasPageItem.tsx` | 单个页面卡片，负责拖拽、缩放、右键菜单、截图/iframe 渲染切换 |

| `packages/shared/src/demo/canvas-utils.ts` | 暴露画布布局保存函数，但当前未形成完整接入 |

已有优势：

- 画布组件已经沉到 shared 包，具备复用基础。
- `PreviewCanvas` 已有可见页裁剪，避免所有页面长期渲染。
- `CanvasPageItem` 支持截图替代 iframe，能降低多页画布的运行成本。
- `CanvasViewport` 已使用 rAF 合并视口更新，并支持常用快捷键。
- 页面拖拽和缩放已有基础吸附线，具备专业工作台雏形。

## 三、保留问题

### P0. 画布布局没有形成可保存闭环

当前 `canvasState` 在编辑页中本地维护，刷新或重新进入后依赖初始布局重新计算。`saveCanvasLayout()` 已存在，但 `/api/sessions/:sessionId/canvas-layout` 没有形成用户可感知、开发者可验证的完整闭环。

影响：

- 用户移动和缩放页面后，容易预期“画布已保存”，但实际可能丢失。
- 后续任何画布能力都依赖稳定的布局数据。
- 半接通的工具函数会误导后续开发者，以为布局保存已经完成。

保留建议：

- 明确画布数据持久化格式，至少包含 `version`、`viewport`、`pages`。
- 进入编辑页时优先加载已保存布局，没有布局时再使用 `computeInitialLayout()`。
- 拖拽、缩放、视口变化结束后 debounce 保存。
- 不在工具栏常驻展示保存状态；保存失败进入日志，避免高频画布操作时产生无效干扰。

### P0. 编辑页承担过多画布状态

画布相关状态包括 `previewMode`、`canvasState`、`canvasEditingPageId`、`focusCanvasPageId`、截图再生计时器和画布页面加载逻辑，均散在编辑页中。编辑页同时还承担 AI 对话、文件弹窗、配置表单、可视化批注等职责。

影响：

- 修改画布交互需要穿过大页面文件，认知成本高。
- 状态联动隐含在多个 effect 和 callback 中，容易出现时序问题。
- 后续增加小型画布操作时，容易继续放大编辑页复杂度。

保留建议：

- 抽出 `useCanvasWorkspace`，集中管理画布模式、选中页、聚焦页、布局状态和截图触发。
- 保持 `PreviewCanvas` 作为工作台组件，编辑页只负责把项目页面数据和回调传入。
- 短期不引入新状态库，避免为了画布局部问题扩大技术栈。

### P1. 工具模式的用户心智不够清晰

当前 hand 模式偏“移动视口”，select 模式偏“移动/缩放页面”。这接近 Figma 类工具习惯，但主要依赖图标和 tooltip 表达，新用户容易不知道为什么拖不动页面，或者为什么点击页面会打开配置面板。

保留建议：

- 工具条保持图标化表达，通过 tooltip 保留快捷键提示。
- 选中页面后显示小型浮动操作条：编辑配置、聚焦此页、重置大小、刷新截图。
- 空白画布点击后隐藏配置面板时，保留明确的“未选中页面”状态。
- 每个页面框外左上角常驻展示页面名，形成类似 Figma 的对象识别体验。

### P1. 截图生成状态不进入画布前台

截图服务已经常驻启用，切换画布时会触发批量截图。截图是性能优化能力，不是用户完成画布整理必须理解的前台状态。

影响：

- 截图服务不可达时，用户只会感觉画布慢或没有更新。
- 开发者难以判断截图、iframe、配置更新之间的真实状态。

保留建议：

- 截图开始、完成、失败和服务不可达信息写入浏览器日志。
- 截图服务不可达时降级为 iframe 渲染，不在画布卡片上叠加状态覆盖层。
- 画布 UI 不展示截图生成状态，避免后台优化细节干扰创作心流。

### P2. 基础布局能力需要变成可控能力

当前已有对齐吸附和辅助线，但用户没有明确的规则入口。此项值得做，但应控制范围，只服务于多页整理效率，不扩展成复杂设计工具。

保留建议：

- 增加“吸附开关”。
- 增加“整理布局”“适应全部”“适应当前页”。
- 暂不做页面分组、自由连接线、白板节点和复杂对齐面板。

### P2. 交互测试覆盖不足

现有测试主要覆盖 `PreviewPanel` 的编译、错误、配置更新等路径。画布本身的缩放、平移、拖拽、截图 fallback、右键菜单等高风险交互缺少测试。

保留建议：

- 为纯函数补单测：初始布局、可见页计算、吸附计算、尺寸解析。
- 为组件补交互测试：工具切换、缩放按钮、页面选择、拖拽更新、重置大小。
- 为截图 hook 补状态测试：批量生成成功、失败、服务不可达、轮询结束。

## 四、删除内容

以下内容从本方案执行范围中删除，不进入近期任务池：

| 已删除内容 | 删除理由 |
| :--- | :--- |
| “为手绘原型转代码奠基”作为目标 | 当前画布问题是基础可信度，不是新范式入口 |
| 阶段四“深度编辑融合” | 依赖批注、AI 修改、审阅流程成熟，短期无法独立验收 |
| “发送给 AI”“待审 patch 汇总” | 需求链路过长，会稀释本方案目标 |
| 画布页卡显示未处理批注数量 | 批注体系尚未成为画布主流程，先不增加信息密度 |
| undo/redo 和正式 command/action 模型 | 当前操作复杂度不足以支撑这类架构投入 |
| 页面尺寸分组、页面分组 | 对当前核心问题帮助有限，容易引入二级规则和边界争议 |
| 手绘、连接线、自由节点 | 会改变画布产品定位，暂不做 |

## 五、优化路线

### 阶段一：可靠性优先

目标：让画布状态可保存、可恢复、可理解。

范围：

- 补齐画布布局加载与保存闭环。
- 补齐后台保存与截图日志。
- 抽出 `useCanvasWorkspace`，收拢编辑页中的画布状态。
- 为布局计算和截图状态补最小测试。

验收标准：

- 刷新页面后，画布中页面位置、尺寸和视口能恢复。
- 截图服务失败时，系统降级为 iframe 预览，开发者可在日志中看到失败原因。
- 编辑页中画布相关状态明显减少，画布逻辑有独立入口。

### 阶段二：工作台体验补齐

目标：让画布从“能预览”升级为“可整理、可定位、可恢复”的多页面工作台。

范围：

- 优化工具栏 tooltip，保留 hand/select 快捷键信息。
- 增加选中页面浮动操作条。
- 增加适应当前页、适应全部、整理布局、吸附开关。
- 在画布页卡外左上角展示页面名称，在页卡上展示基础配置状态。

验收标准：

- 新用户不看文档也能理解“拖动画布”和“移动页面”的区别。
- 多页面项目中能快速定位、整理、编辑指定页面。
- 画布状态变化不会让配置面板和当前页面选择产生意外跳转。

### 阶段三：维护模型固化

目标：让后续画布功能可以稳定迭代。

范围：

- 拆出画布纯函数模块：布局、可见性、吸附、坐标转换。
- 梳理 `PreviewCanvasProps`，区分必需能力和创作端专属能力。
- 建立画布交互测试矩阵。

验收标准：

- 新增一个画布操作时，能明确落在 hook、纯函数、组件或 API 哪一层。
- 核心交互有测试兜底。
- 不需要修改超大编辑页即可扩展多数画布能力。

## 六、设计原则

### 6.1 先补可信度，再谈扩展性

保存、恢复、失败反馈和测试是画布体验的底座。底座没稳之前，不继续叠加手绘、AI、审阅汇总等新能力。

### 6.2 画布是多页面工作台

画布负责多页面总览、整理、定位、状态汇总；单页预览负责批注、inline edit、视觉选区和精细修改。当前方案只优化两者之间必要的入口，不把画布改造成深度编辑器。

### 6.3 高频状态必须轻量可见

页面名、选中页、当前工具、是否吸附等与用户操作直接相关的状态应该轻量可见。保存中、截图中、失败等后台状态不常驻展示，进入日志即可。

### 6.4 复杂交互先抽纯逻辑

布局计算、吸附、可见页判断、坐标转换不应长期留在组件中。纯函数越清晰，交互测试越稳定，后续维护越轻。

## 七、风险与取舍

| 风险 | 影响 | 缓解 |
| :--- | :--- | :--- |
| 先做持久化会牵涉 API 和数据格式 | 短期工作量上升 | 只保存当前必要字段，并加 `version` 预留迁移 |
| 抽 hook 可能引入行为回归 | 编辑页联动变复杂 | 先迁移画布相关状态，不碰 AI 和配置主链路 |
| 截图状态 UI 过多会打扰用户 | 画布变嘈杂 | 仅在页面卡片角落和工具栏汇总展示，不使用弹窗 |
| 阶段二控件过多 | 工具栏复杂度上升 | 只保留整理、吸附、适应视图等高频动作 |

## 九、建议下一步

建议只启动阶段一，并拆成三张任务卡：

1. 画布布局持久化与保存状态。
2. `useCanvasWorkspace` 状态收拢。
3. 截图状态展示与失败降级。

这三项完成后，画布会从“功能可用”推进到“体验可信、开发可控”。阶段二和阶段三等阶段一稳定后再评估，不提前排期。
