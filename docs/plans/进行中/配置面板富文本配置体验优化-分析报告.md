# 配置面板富文本配置体验优化方案

> **问题描述**：配置面板中富文本字段（`ui:widget: "richtext"`）当前使用原生 Textarea 控件，编辑体验不佳，希望集成 Markdown 编辑器提升配置体验。

---

## 一、问题背景

### 1.1 问题现象

在 Demo 编辑页面的配置面板中，当 Schema 定义富文本字段时（通过 `ui:widget: "richtext"` 声明），当前渲染为普通的 `<Textarea>` 多行文本框，存在以下体验问题：

- **缺乏格式化工具栏**：无法快速插入标题、列表、代码块、链接等 Markdown 语法
- **无实时预览能力**：无法即时查看富文本渲染效果
- **语法提示缺失**：用户需手动输入 Markdown 标记符，容易出错
- **与系统其他编辑器不一致**：知识库文档、工作区代码说明等场景已使用 `MemoryMarkdownEditor` 组件

### 1.2 预期行为

富文本配置字段应提供类 Markdown 编辑器的交互体验：

- 可视化工具栏（加粗、标题、列表、代码块、链接等）
- 编辑/预览模式切换
- 与现有 `MemoryMarkdownEditor` 组件一致的交互体验

### 1.3 当前技术栈

| 组件                 | 位置                                                                | 技术               |
| -------------------- | ------------------------------------------------------------------- | ------------------ |
| ConfigForm           | `packages/shared/src/demo/ConfigForm.tsx`                           | 动态表单生成器     |
| Textarea             | `packages/author-site/src/components/ui/textarea.tsx`               | shadcn/ui 基础组件 |
| MemoryMarkdownEditor | `packages/author-site/src/components/demo/MemoryMarkdownEditor.tsx` | TipTap + Markdown  |
| 项目文档             | `docs/项目文档/创作端/04-配置与预览/技术/03_表单生成器.md`          | 表单生成器技术文档 |

## 二、根因分析

### 2.1 调查过程

通过代码搜索和文档分析，追踪了配置面板富文本字段的完整渲染链路：

1. **Schema 解析层**：`parseSchemaToFields()` 函数解析 `config.schema.json`，提取 `ui:widget: "richtext"` 标记
2. **字段渲染层**：`FieldRenderer` 组件根据 `field.uiWidget` 值选择对应控件
3. **控件实现层**：当前 `richtext` 分支直接使用 `<Textarea>` 组件

### 2.2 代码执行路径

```
config.schema.json (ui:widget: "richtext")
  ↓
parseSchemaToFields() → FieldConfig.uiWidget = "richtext"
  ↓
FieldRenderer.renderInput()
  ↓
if (field.uiWidget === "richtext") {
  return <Textarea ... />  ← 当前实现（第 282-292 行）
}
```

**关键代码位置**：`packages/shared/src/demo/ConfigForm.tsx:282-292`

```typescript
if (field.uiWidget === "richtext") {
  return (
    <Textarea
      value={(value as string) || ""}

      rows={5}
      className="resize-y min-h-[100px]"

```

### 2.3 根本原因

**根因**：配置面板的 `richtext` 控件实现停留在基础 Textarea 阶段，未复用系统已有的 `MemoryMarkdownEditor` 组件。

**支撑证据**：

| 问题                            | 影响                                   | 复杂度 |

| 系统内存在两套编辑器实现        | MemoryMarkdownEditor 已成熟但未复用    | 低     |

## 三、解决方案

### 3.1 方案概述

在 ConfigForm 的 `richtext` 字段渲染分支中，将 `<Textarea>` 替换为 `<MemoryMarkdownEditor>` 组件，实现 Markdown 编辑器体验。

### 3.2 方案设计

#### 3.2.1 核心改动

**文件**：`packages/shared/src/demo/ConfigForm.tsx`

**改动位置**：第 282-292 行 `richtext` 分支

**改动内容**：

```typescript
// 当前实现（删除）
if (field.uiWidget === "richtext") {
  return (
    <Textarea
      value={(value as string) || ""}
      onChange={(e) => onChange(e.target.value)}
      placeholder={`请输入${field.title}`}
      rows={5}
      className="resize-y min-h-[100px]"
    />
  );
}

// 新实现（替换）
if (field.uiWidget === "richtext") {
  return (
    <MemoryMarkdownEditor
      value={(value as string) || ""}
      onChange={onChange}
      readOnly={readonly}
    />
  );
}
```

#### 3.2.2 依赖导入

在 ConfigForm.tsx 顶部添加导入：

```typescript
import { MemoryMarkdownEditor } from "./MemoryMarkdownEditor";
```

**注意**：由于 ConfigForm 位于 `packages/shared/src/demo/`，而 MemoryMarkdownEditor 位于 `packages/author-site/src/components/demo/`，存在跨包引用问题。需要评估以下两种策略：

**策略 A：移动 MemoryMarkdownEditor 到 shared 包**

- 将 `MemoryMarkdownEditor.tsx` 移动至 `packages/shared/src/demo/`
- 更新 author-site 中的导入路径
- **优点**：符合 shared 包定位（共享组件）
- **缺点**：MemoryMarkdownEditor 依赖 TipTap、Streamdown 等重型库，会增加 shared 包体积

**策略 B：在 ConfigForm 中通过 props 注入编辑器组件**

- ConfigForm 接受 `customWidgets` props，允许外部注入自定义控件
- 在 author-site 侧传入 MemoryMarkdownEditor
- **优点**：保持 shared 包轻量，解耦依赖
- **缺点**：增加 ConfigForm API 复杂度

**推荐策略**：**策略 A**（移动到 shared 包）

- 理由：MemoryMarkdownEditor 已是多场景复用的共享组件（知识库、工作区说明、配置面板）
- shared 包本身已包含 TipTap 依赖（通过 widgets.tsx 等组件）
- 符合架构文档定义的"共享组件架构"（参见 `05_共享组件架构设计.md`）

#### 3.2.3 Schema 语义优化

当前 Schema 使用 `ui:widget: "richtext"` 标记富文本字段，但实际实现为纯文本框。建议：

**选项 1**：保持 `richtext` 标记，升级实现为 Markdown 编辑器

- **优点**：向后兼容，无需修改现有 Schema
- **缺点**：`richtext` 语义模糊（HTML 富文本 vs Markdown）

**选项 2**：新增 `markdown` 标记，保留 `richtext` 作为降级方案

- **优点**：语义清晰，支持渐进式迁移
- **缺点**：需要维护两套控件实现

**推荐选项**：**选项 1**（保持 `richtext` 标记）

- 理由：项目中 `richtext` 已被文档定义为"编辑格式化文本"（需求文档 254 行）
- Markdown 是富文本的一种实现方式，语义上可接受
- 减少迁移成本

### 3.3 实施步骤

#### Phase 1：组件迁移（1-2 小时）

1. **移动 MemoryMarkdownEditor 到 shared 包**
   - 复制 `packages/author-site/src/components/demo/MemoryMarkdownEditor.tsx`
   - 粘贴至 `packages/shared/src/demo/MemoryMarkdownEditor.tsx`
   - 更新导入路径（`@/lib/utils` → `./utils`）

2. **更新 author-site 中的引用**
   - 搜索所有 `import { MemoryMarkdownEditor } from "./MemoryMarkdownEditor"`
   - 替换为 `import { MemoryMarkdownEditor } from "@opencode-workbench/shared/demo"`

3. **验证依赖完整性**
   - 检查 `packages/shared/package.json` 是否包含 TipTap 相关依赖
   - 缺失则添加：`@tiptap/react`, `@tiptap/starter-kit`, `@tiptap/extension-*` 等

#### Phase 2：ConfigForm 集成（30 分钟）

4. **修改 ConfigForm.tsx**
   - 导入 MemoryMarkdownEditor
   - 替换 `richtext` 分支的 Textarea 实现
   - 传递 `readonly` props 支持只读模式

5. **调整样式适配**
   - MemoryMarkdownEditor 默认 `min-h-[200px]`，可能需要缩小以适配配置面板
   - 通过 `uiOptions.height` 支持自定义高度

#### Phase 3：测试验证（1 小时）

6. **功能测试**
   - 在 Demo 编辑页面配置面板中测试富文本字段
   - 验证工具栏功能（加粗、标题、列表、代码块、链接等）
   - 验证编辑/预览模式切换
   - 验证只读模式（viewer 页面）

7. **兼容性测试**
   - 验证现有 Demo 的 `config.schema.json` 无需修改即可生效
   - 验证配置数据持久化（Markdown 字符串存储）

#### Phase 4：文档更新（30 分钟）

8. **更新技术文档**
   - `03_表单生成器.md`：更新 RichTextWidget 描述为 Markdown 编辑器
   - `配置系统_需求文档.md`：明确富文本编辑器基于 Markdown 实现

### 3.4 风险评估

| 风险                                | 影响         | 概率 | 缓解措施                    |
| ----------------------------------- | ------------ | ---- | --------------------------- |
| TipTap 依赖增加 shared 包体积       | 构建时间增加 | 低   | shared 包已包含 TipTap 依赖 |
| MemoryMarkdownEditor 迁移后路径错误 | 编译失败     | 低   | 系统性更新所有导入路径      |
| 配置面板空间不足导致编辑器体验差    | 用户体验下降 | 中   | 通过 uiOptions 支持高度配置 |
| 只读模式未正确传递                  | 预览页可编辑 | 低   | 明确传递 readonly props     |

### 3.5 替代方案

#### 方案 B：轻量级 Markdown 预览 + Textarea 编辑

不引入 TipTap，使用纯文本 Textarea + Markdown 预览面板：

```typescript
if (field.uiWidget === "richtext") {
  const [preview, setPreview] = useState(false);

  return (
    <div className="border rounded-md">
      <div className="flex gap-2 px-2 py-1 border-b bg-muted/30">
        <Button size="sm" variant={!preview ? "default" : "ghost"}>
          编辑
        </Button>
        <Button size="sm" variant={preview ? "default" : "ghost"}>
          预览
        </Button>
      </div>
      {preview ? (
        <div className="p-3 text-sm">
          <MarkdownRenderer content={value as string} />
        </div>
      ) : (
        <Textarea
          value={(value as string) || ""}
          onChange={(e) => onChange(e.target.value)}
          rows={5}
          className="border-0 resize-y min-h-[100px]"
        />
      )}
    </div>
  );
}
```

**对比分析**：

| 维度       | 方案 A（MemoryMarkdownEditor） | 方案 B（Textarea + 预览） |
| ---------- | ------------------------------ | ------------------------- |
| 开发成本   | 中（需迁移组件）               | 低（新增简单组件）        |
| 用户体验   | 高（所见即所得编辑）           | 中（需切换预览）          |
| 维护成本   | 低（复用现有组件）             | 高（新增独立实现）        |
| 系统一致性 | 高（统一编辑器）               | 低（多套实现）            |
| 依赖增量   | 无（已存在）                   | 低（仅需 markdown-it）    |

**推荐**：**方案 A**，理由：

- 系统已存在成熟的 MemoryMarkdownEditor 组件
- 符合 DRY 原则，避免重复实现
- 提供一致的跨场景编辑体验

## 五、后续建议

### 5.1 增强功能

1. **图片上传支持**
   - 在 MemoryMarkdownEditor 中集成图片上传能力
   - 支持拖拽上传图片到工作区
   - 自动生成图片 URL 并插入 Markdown

2. **表格编辑支持**
   - 添加 TipTap Table 扩展
   - 提供可视化表格编辑工具栏

3. **快捷键支持**
   - `Cmd/Ctrl + B`：加粗
   - `Cmd/Ctrl + I`：斜体
   - `Cmd/Ctrl + K`：插入链接

### 5.2 性能优化

1. **编辑器懒加载**
   - 配置面板中富文本字段可能不常用
   - 使用 `React.lazy()` 延迟加载 MemoryMarkdownEditor
   - 减少初始渲染开销

2. **高度自适应**
   - 根据内容自动调整编辑器高度
   - 避免配置面板出现双重滚动条

### 5.3 用户体验优化

1. **工具栏紧凑模式**
   - 配置面板空间有限，可提供 `compact` 模式
   - 隐藏不常用工具（如任务列表、分隔线）

2. **占位符自定义**
   - 通过 `uiOptions.placeholder` 支持自定义占位符
   - 例如："输入产品描述（支持 Markdown）"

3. **字数统计**
   - MemoryMarkdownEditor 已实现字数统计（底部显示）
   - 可在配置面板中保留此功能

## 六、总结

### 6.1 问题根因

配置面板富文本字段使用基础 Textarea 组件，未复用系统已有的 MemoryMarkdownEditor Markdown 编辑器，导致编辑体验不佳。

### 6.2 推荐方案

将 MemoryMarkdownEditor 组件迁移至 shared 包，并在 ConfigForm 的 `richtext` 分支中集成，实现：

- 可视化工具栏（加粗、标题、列表、代码块、链接等）
- 编辑/预览模式切换
- 与系统其他场景一致的编辑体验

### 6.3 实施成本

- **开发时间**：2-3 小时（含迁移、集成、测试）
- **依赖增量**：无（TipTap 已存在于项目）
- **兼容性**：向后兼容，现有 Schema 无需修改

### 6.4 预期收益

- 配置面板富文本编辑体验显著提升
- 系统编辑器组件统一，降低维护成本
- 符合需求文档定义的"编辑格式化文本"能力
