# 使用者视图方案

> 为系统新增"使用者"角色视图。采用**独立站点 + API 调用**的架构，消费端与生成端完全解耦，实现高可维护性，同时便于嵌入其他系统。

---

## 一、背景与目标

### 1.1 用户角色定义

| 角色 | 权限 | 对应界面 |
|------|------|---------|
| **编辑者 (Editor)** | 创建项目、AI 对话编辑代码、管理版本 | 现有 Web 端 (`/projects/`) |
| **使用者 (Viewer)** | 浏览项目列表、查看预览、调整配置参数 | **独立的新站点** |
| **开发者 (Developer)** | 系统管理、扩展开发 | 后续规划 |

### 1.2 核心需求

- 使用者站点是一个**完全独立的 Web 应用**，可独立部署
- 通过 HTTP API 从生成端获取项目数据和编译结果
- 使用者站点只包含：
  - **项目列表页** — 展示可供使用的项目
  - **预览页** — 预览区 + 配置面板（无 AI 对话、无代码编辑）
- 使用者站点可轻松嵌入其他系统（iframe 或 API 对接）
- 使用者不需要登录/设置用户名，开箱即用

---

## 二、为什么选择独立站点

### 2.1 架构对比

| 维度 | 同一站点内分角色 | **独立站点（本方案）** |
|------|-----------------|----------------------|
| **耦合度** | 高 — 共用路由、组件、构建产物 | **低 — 仅通过 API 通信** |
| **可维护性** | 编辑者变更可能影响使用者 | **独立迭代，互不影响** |
| **部署灵活性** | 必须一起部署 | **可独立部署到不同域名/服务器** |
| **嵌入其他系统** | 需要额外封装 | **天生适合嵌入 — 无多余 UI 元素** |
| **开发速度** | 需理解整个项目后再修改 | **从零开始，只需对接 API** |
| **安全性** | 共用一个攻击面 | **使用者站点只读，风险低** |
| **构建产物** | 大（含 AI 对话等重组件） | **轻量（仅预览+配置）** |

### 2.2 适用场景

- 将生成的组件展示嵌入客户官网、产品文档
- 为不同客户提供独立的组件预览门户
- 作为 CMS 或设计系统的展示层
- 与生成端分属不同团队维护

## 三、架构设计

### 3.1 整体架构

```
┌──────────────────────┐       HTTP API        ┌──────────────────────┐
│                      │ ◄──────────────────── │                      │
│   使用者站点          │       /api/projects    │   生成端（Web 端）     │
│   (独立部署)          │       /api/projects/   │                      │
│                      │       :id/demos        │   含 Agent Service    │
│                      │       /api/compile      │                      │
│                      │ ─────────────────────► │                      │
└──────────────────────┘                        └──────────────────────┘
```

### 3.2 使用者站点结构

使用者站点是一个**独立的 Next.js 应用**，包含：

```
viewer-site/
├── app/
│   ├── page.tsx                    # 项目列表页
│   ├── layout.tsx                  # 根布局
│   ├── globals.css                 # 全局样式
│   └── [projectId]/
│       └── page.tsx                # 预览页（预览区 + 配置面板）
├── components/
│   ├── project-list.tsx            # 项目列表组件
│   ├── project-preview.tsx         # 预览页主组件
│   └── iframe-renderer.tsx         # iframe 渲染器（轻量版 PreviewPanel）
├── lib/
│   ├── api.ts                      # API 客户端（调用生成端接口）
│   └── types.ts                    # 类型定义
├── package.json
├── next.config.js
└── tsconfig.json
```

### 3.3 路由设计

| 路径 | 说明 |
|------|------|
| `/` | 项目列表页 — 从生成端获取项目列表 |
| `/[projectId]` | 项目预览页 — 从生成端获取代码/schema，编译后渲染 |

### 3.4 与生成端的 API 契约

使用者站点通过 HTTP API 从生成端获取数据，需要生成端提供以下接口：

| API | 方法 | 说明 | 当前状态 |
|-----|------|------|---------|
| `GET /api/projects` | 获取项目列表 | 已有（agent-service） |
| `GET /api/projects/:id/demos` | 获取项目的 Demo 页面列表 | 已有（web 端路由） |
| `POST /api/compile` | 编译代码 | 已有（web 端路由） |

> 使用者站点**不需要**直接调用 Agent Service，所有请求通过 Web 端的 Next.js API Route 代理即可。

## 四、关键组件实现思路

### 4.1 项目列表页 (`/`)

```
┌──────────────────────────────────────┐
│  Logo / 站点名称                      │
│                                      │
│  ┌──────────┐  ┌──────────┐          │
│  │ 项目卡片  │  │ 项目卡片  │  ...     │
│  │ 名称/描述 │  │ 名称/描述 │          │
│  │ 查看预览  │  │ 查看预览  │          │
│  └──────────┘  └──────────┘          │
└──────────────────────────────────────┘
```

- 调用 `GET /api/projects` 获取列表
- 点击项目卡片跳转到 `/{projectId}`
- 无"新建项目"、无用户名选择器
- 可配置只展示特定项目（通过环境变量过滤）

### 4.2 预览页 (`/[projectId]`)

```
┌─────────────────────────────────────────────────┐
│  ← 返回列表  │  项目名称                        │
├──────────────────────┬──────────────────────────┤
│                      │                          │
│    预览区            │    配置面板               │
│    (iframe)          │    (表单控件)              │
│                      │                          │
│                      │                          │
└──────────────────────┴──────────────────────────┘
```

#### 预览区

两种实现方式：

**方式 A：直接 iframe 嵌入（推荐）**

直接嵌入 `https://生成端域名/embed/{demoId}/iframe`，由生成端渲染完整 HTML：

```html
<iframe
  src="https://生成端域名/embed/{demoId}/iframe"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>
```

- 优点：实现极简，生成端负责编译和渲染
- 缺点：配置变更需要通过 `postMessage` 通信（已有支持）

**方式 B：API 编译 + 本地 iframe**

调用 `POST /api/compile` 获取编译结果，在本地生成 iframe HTML：

```
1. 调用 GET /api/projects/:id/demos → 获取 demo 列表
2. 读取 demo 的 code 和 schema（需新增 API 或扩展现有接口）
3. 调用 POST /api/compile → 编译代码
4. 本地生成 iframe HTML 并渲染
```

- 优点：独立性强，生成端只提供数据
- 缺点：需要更多 API 调用

#### 配置面板

- 从 schema 解析字段定义，动态生成表单控件
- 配置变更通过 `postMessage` 发送到 iframe
- 配置状态保存在本地 state，不持久化

### 4.3 iframe 通信

复用现有 `iframe-template.ts` 生成的 HTML 中的 `postMessage` 通信机制：

```
父窗口 → iframe:
  { type: 'UPDATE_CONFIG', configData: { ... } }

iframe → 父窗口:
  { type: 'READY' }
  { type: 'LOADED' }
  { type: 'RUNTIME_ERROR', error: '...' }
  { type: 'RESIZE', height: 800 }
```

## 五、生成端需新增/调整的 API

现有 API 基本满足需求，但需要以下补充：

### 5.1 获取 Demo 页面代码和 Schema

当前 `GET /api/projects/:id/demos` 只返回页面元数据列表，不包含 `code` 和 `schema` 内容。

需要新增或在现有接口扩展：

```
GET /api/projects/:id/demos/:demoId/files
```

返回：
```json
{
  "success": true,
  "data": {
    "code": "import React from 'react'...",
    "schema": "{ \"type\": \"object\", \"properties\": {...} }"
  }
}
```

> **注**：查看代码发现 `sessions/[sessionId]/files/[demoId]/route.ts` 已有类似接口，但依赖于 session。需要一个**不依赖 session** 的只读版本。

### 5.2 编译 API 支持直接传 code

现有 `POST /api/compile` 已支持直接传 `code` 参数，无需修改。

## 六、嵌入其他系统的方式

### 6.1 iframe 整体嵌入

外部系统直接嵌入使用者站点的完整预览页（含配置面板）：

```html
<iframe
  src="https://使用者站点域名/{projectId}"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; height: 800px; border: none;"
/>
```

适用于：消费端不需要任何定制，直接展示生成端的完整 UI。

### 6.2 混合嵌入模式（外部控制 + iframe 预览）⭐ 推荐

外部系统自己实现精简的配置面板，仅展示需要的配置项，同时用 iframe 渲染预览区。生成端提供细粒度的控制能力。

#### 6.2.1 embed 端点参数

```
/embed/{demoId}/iframe
```

| 参数 | 类型 | 说明 |
|------|------|------|
| `fields` | string | 逗号分隔的字段名列表，只渲染指定配置项（留空则全部渲染） |
| `externalControl` | boolean | 为 `true` 时，隐藏内置配置面板，由外部完全控制 |
| `config` | JSON string | 初始配置，会被发送到预览 iframe |

**示例**：
```
/embed/demo123/iframe?fields=theme,color&config={"theme":"dark"}
```

#### 6.2.2 模式 A：使用 iframe 内置配置面板（可选择字段）

```html
<iframe
  src="/embed/demo123/iframe?fields=theme,color"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none; height: 600px;"
/>
```

- 预览区自动调整高度（通过 `RESIZE` postMessage）
- 外部系统只需处理 iframe 容器布局
- 适用于：消费端需要完整的配置体验，但只需要部分字段

#### 6.2.3 模式 B：外部实现配置面板，iframe 仅渲染预览

```html
<!-- 外部系统自己实现的精简配置面板 -->
<div class="config-panel">
  <label>主题：
    <select id="theme-select">
      <option value="light">亮色</option>
      <option value="dark">暗色</option>
    </select>
  </label>
  <button id="apply-btn">应用</button>
</div>

<!-- 预览 iframe（隐藏配置面板） -->
<iframe
  id="preview-frame"
  src="/embed/demo123/iframe?externalControl=true"
  sandbox="allow-scripts allow-same-origin"
  style="width: 100%; border: none;"
/>

<script>
  const iframe = document.getElementById('preview-frame');
  const applyBtn = document.getElementById('apply-btn');

  applyBtn.addEventListener('click', () => {
    const theme = document.getElementById('theme-select').value;
    iframe.contentWindow.postMessage({
      type: 'UPDATE_CONFIG',
      configData: { theme }
    }, '*');
  });
</script>
```

- 外部系统完全控制配置 UI，只暴露需要的字段
- iframe 内部不渲染配置面板，专注预览渲染
- 通过 `postMessage` 发送配置变更
- 适用于：消费端需要统一风格、或配置项来自外部数据源

#### 6.2.4 postMessage 通信协议

```
父窗口 → iframe:
  { type: 'UPDATE_CONFIG', configData: { ... } }
  { type: 'GET_STATE' }                    # 请求当前状态

iframe → 父窗口:
  { type: 'READY' }                        # 预览已就绪
  { type: 'LOADED' }                       # 页面加载完成
  { type: 'STATE', state: { config, ... } } # 返回当前状态
  { type: 'RUNTIME_ERROR', error: '...' }   # 运行时错误
  { type: 'RESIZE', height: 800 }          # 高度调整
```

### 6.3 API 对接（完全自定义渲染）

外部系统直接调用生成端的 API，自己实现数据获取、编译和渲染：

```javascript
// 1. 获取项目列表
const projects = await fetch('https://生成端域名/api/projects');

// 2. 获取 Demo 页面
const demos = await fetch(`https://生成端域名/api/projects/${projectId}/demos`);

// 3. 编译代码
const compiled = await fetch('https://生成端域名/api/compile', {
  method: 'POST',
  body: JSON.stringify({ code })
});
```

适用于：消费端有特殊的渲染技术栈要求，或需要完全控制 UI。

## 七、部署方案

### 7.1 部署拓扑

```
生成端 (opencode-workbench)
├── Web 端 (Next.js) → 端口 3200
│   ├── 编辑者界面
│   ├── 编译 API
│   └── 嵌入 iframe 端点
└── Agent Service (Fastify) → 端口 3101

使用者站点 (viewer-site) → 独立域名/子域名
└── Next.js 静态导出或 SSR
```

### 7.2 环境变量

使用者站点通过环境变量配置生成端地址：

```env
NEXT_PUBLIC_API_BASE_URL=https://生成端域名
```

## 八、实施步骤

1. **准备生成端 API** — 新增 `GET /api/projects/:id/demos/:demoId/files` 只读接口
2. **初始化使用者站点** — 新建 `viewer-site/` 目录，配置 Next.js 项目
3. **实现项目列表页** — 调用 API 获取项目列表并展示
4. **实现预览页** — 集成 iframe 嵌入和配置面板
5. **配置面板组件** — 实现基于 JSON Schema 的动态表单（可复用现有 ConfigForm 思路）
6. **嵌入测试** — 验证 iframe 嵌入和 postMessage 通信
7. **部署配置** — 配置独立域名/子域名部署
