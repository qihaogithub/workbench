# 实时保存与多人协同编辑风险分析

## 背景

本文是 [实时保存与多人协同编辑方案](./实时保存与多人协同编辑方案.md) 的补充分析，用于记录在“单实例部署、同时使用人数不超过 50 人”前提下，仍需要后续关注的产品、数据一致性、测试和运维风险。

多实例横向扩展不作为当前阶段风险项；如未来部署形态变化，再重新评估共享协同状态与外部协调层。

## 分析前提

- 当前工具面向内部或小规模创作场景。
- 同时在线使用人数预计不超过 50 人。
- 单个项目内的实际同时编辑人数以 2-3 人为主要目标。
- 草稿实时落盘到 Workspace，手动保存继续生成版本快照。
- 协同能力优先覆盖代码和配置类文本资源，部分结构化操作仍沿用现有 API。

## 主要风险

### 协同粒度不一致

当前文本类资源使用 Yjs 协同编辑，但页面树排序、重命名和画布结构变更仍主要走既有 API。用户可能感知到“代码编辑能实时同步，但结构操作不是同一套协同体验”。

潜在影响：

- 两名用户同时调整页面结构时，后提交的操作可能覆盖先提交的结果。
- 文本内容已经实时同步，但页面树或画布状态需要刷新后才表现一致。
- 保存前 flush 可以降低落盘遗漏，但不能完全表达结构操作之间的业务冲突。

后续建议：

- 为页面树、画布布局等结构化资源补充明确的冲突提示。
- 将高频结构操作逐步改为更细粒度的协同操作，而不是整文件或整对象覆盖。
- 在 UI 上区分“实时协同编辑中”和“该操作将按普通保存流程提交”。

### 非协同写入路径可能造成静默覆盖

如果协同通道之外的 API 或工具直接写入 Workspace 文件，当前方案主要依赖重新打开资源后同步最新磁盘内容。这个策略实现简单，但对用户不够透明。

潜在影响：

- 用户正在编辑的内容可能被外部保存流程影响。
- 同一文件被协同编辑和普通 API 同时写入时，缺少清晰的冲突确认。
- 用户难以判断内容变化来自其他协作者、系统保存，还是外部工具。

后续建议：

- 为直接写文件路径增加版本戳或更新时间检查。
- 当磁盘内容与当前协同文档状态不一致时，提示用户确认覆盖、合并或重新加载。
- 记录关键写入来源，便于排查“内容被谁改了”的问题。

### 跨资源一致性不足

实时保存以单个资源为单位落盘，但项目编辑经常涉及多份文件共同变化，例如页面代码、页面 Schema、项目级 Schema、页面树和画布布局。

潜在影响：

- 某个资源已落盘，另一个相关资源尚未 flush，短时间内可能出现中间状态。
- 手动保存生成版本快照时，如果 flush 覆盖不完整，版本中可能混入部分旧内容。
- 预览、编译或发布流程读取 Workspace 时，可能遇到跨文件状态暂时不一致。

后续建议：

- 保存、发布、预览等关键入口统一执行 workspace 级 flush。
- 为关键资源组定义一致性检查，例如页面代码与对应 Schema 是否同时存在。
- 对跨资源操作引入操作批次或保存事务记录，方便失败后定位。

### 用户保存心智变复杂

方案同时保留“草稿实时保存到 Workspace”和“手动保存生成版本快照”。这有利于兼容现有流程，但用户可能混淆草稿保存、版本保存和发布之间的区别。

潜在影响：

- 用户看到内容已同步后，误以为项目版本也已保存。
- 协作者实时修改了草稿，但项目所有者未手动保存，发布版本仍可能是旧版本。
- 出现问题时，用户不容易判断应该恢复 Workspace 草稿还是恢复历史版本。

后续建议：

- UI 明确展示草稿同步状态、版本保存状态和发布状态。
- 保存按钮文案或提示中说明“保存为版本快照”的含义。
- 为草稿恢复和版本恢复提供不同入口，避免混用。

### 断线重连与浏览器关闭边界需要验证

实时协同依赖 WebSocket 连接和 debounce 落盘。网络抖动、浏览器关闭、标签页休眠等场景会影响最后一段编辑内容是否已经送达服务端并写入磁盘。

潜在影响：

- 用户关闭页面前的最后几次输入可能尚未 flush。
- 断线重连后本地编辑状态与服务端状态合并结果不符合用户预期。
- 多标签页同时打开同一资源时，连接状态和协作者列表可能误导用户。

后续建议：

- 在页面关闭、切换项目、保存和发布前执行显式 flush。
- 增加断线、重连、关闭页面前快速输入等自动化或半自动化测试。
- 在连接异常时降低编辑信任感，例如提示“离线编辑中，尚未同步”。

### 测试覆盖仍偏基础

当前验证重点覆盖类型检查、服务端基础测试和双浏览器手测。多人协同的高风险点主要集中在并发时序和异常路径，单次手测难以覆盖。

潜在影响：

- 某些并发覆盖问题只在真实使用中暴露。
- 断线、重连、快速保存、外部写入等边界缺少回归保障。
- 后续改动容易破坏协同落盘或保存前 flush 语义。

后续建议：

- 补充协同服务端的并发写入、flush、断连清理和重连测试。
- 建立双客户端端到端测试，覆盖同文件编辑、不同文件编辑、保存、发布前 flush。
- 将“非协同路径写入同一资源”的冲突场景作为专项测试。

## 优先级建议

| 优先级 | 风险项 | 处理建议 |
|---|---|---|
| P0 | 保存、发布前 flush 不完整 | 先确保所有关键入口都有 workspace 级 flush，并有测试覆盖 |
| P1 | 非协同写入静默覆盖 | 增加版本戳检查和冲突提示 |
| P1 | 断线重连边界 | 补充重连与关闭页面前 flush 验证 |
| P2 | 页面树和画布结构协同粒度不足 | 按高频操作逐步改为细粒度协同或冲突提示 |
| P2 | 用户保存心智复杂 | 优化 UI 状态和文案区分草稿、版本、发布 |

## 后续观察指标

- 协同连接失败率和重连次数。
- 保存、发布前 flush 失败次数。
- Workspace 文件写入失败或冲突次数。
- 用户反馈的内容丢失、覆盖、状态不同步问题。
- 双浏览器协同测试的通过率。

## 结论

在不考虑横向扩展、并发人数不超过 50 人的前提下，当前方案的主要短板不在容量，而在协同边界和异常路径：结构化操作尚未完全协同、非协同写入缺少冲突感知、跨资源一致性依赖 flush 纪律、用户需要理解草稿与版本的区别。

后续优化应优先保证保存和发布链路的完整 flush，再逐步补充冲突提示、断线重连测试和结构化资源的细粒度协同。
