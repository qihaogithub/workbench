# 部署与运维问题沉淀

## 当前状态

正式环境已从旧 `opencode-workbench-*` 端口服务切换到新 `workbench-*` compose 项目；`author-site`、`agent-service`、`viewer-site` 和 `screenshot-service` 均可访问。2026-07-08 已按用户要求使用本地 `data/` 覆盖正式环境 `/opt/opencode-workbench/data`。截图服务当前使用旧稳定镜像重新标记为 `workbench-screenshot-service:latest`，以规避新构建镜像中的 Chromium 150 启动失败。

## 当前结论

- 2026-07-08 正式部署时，`scripts/deploy.sh` 默认构建的 `agent-service`、`author-site`、`viewer-site` 镜像成功，但远端仍运行旧 `opencode-workbench-*` 容器，占用 3200、3201 和 3300 端口。切换时已停止旧四个端口服务，保留 `opencode-workbench-opencode-serve-1`。
- 局域网服务器部署的常用加速路径是 `scripts/deploy-fast.sh`。该入口默认使用 `DEPLOY_SYNC_MODE=targeted` 和 `DEPLOY_BUILD_MODE=local`，只同步目标服务 Dockerfile 构建所需的 workspace 包，并在本地构建 linux/amd64 镜像后上传到服务器；`scripts/deploy.sh` 默认仍保留完整同步，必要时可通过 `scripts/deploy-fast.sh --full-sync ...` 回退。
- 2026-07-08 `scripts/deploy-fast.sh author viewer` 后出现创作端项目列表为空。排查确认容器挂载 `/opt/opencode-workbench/data:/app/data`，但该 bind 目录在 13:26 被重新创建为空目录；已用本地 `data/` 重新覆盖正式环境并验证恢复。`scripts/deploy.sh` 已改为默认不自动创建缺失的 `APP_DATA_DIR`，避免普通部署把正式环境切到空 data；首次部署如确需创建空目录，必须显式设置 `ALLOW_CREATE_APP_DATA_DIR=true`。
- 2026-07-08 再次执行 `scripts/deploy-fast.sh author viewer` 时，远端卡在 `author-site` 的 `next build`，随后 SSH banner 和 3200/3201/3300 HTTP 均连接后无响应。已中断本地部署 SSH，但远端一段时间内仍不可控；脚本先改为逐服务 `docker compose build` 和 `docker compose up -d --force-recreate --no-deps`，避免 author/viewer 部署时通过 compose 依赖链拉起或重建额外服务。
- 2026-07-09 服务器重启后复盘确认正式机主要风险来自宿主资源争用：`figma_mirror_sync` 容器达到 2GiB cgroup 内存上限并触发 OOM，同时 workbench 远端构建叠加 CPU/内存/IO 压力，导致 SSH、HTTP 和 1Panel 控制面无响应。`scripts/deploy.sh` 已改为默认 `DEPLOY_BUILD_MODE=local`，在本机按 `linux/amd64` 构建镜像并上传，服务器只执行 `docker load` 和 `docker compose up --no-build`；显式 `DEPLOY_BUILD_MODE=remote`/`deploy-fast --remote-build` 才允许服务器构建，并增加远端可用内存与负载预检。
- 新 `workbench-screenshot-service` 镜像通过 Debian apt 安装到 Chromium `150.0.7871.46`，在正式机容器内 headless 启动退出码 133；`/health` 的 HTTP 状态仍为 200，但 `browser.lastError` 会记录 `BROWSER_LAUNCH_ERROR`，不能只用普通健康页判断截图服务可用。
- 旧 `opencode-workbench-screenshot-service` 镜像内 Chromium `149.0.7827.196` 在同一正式机上 headless 探针通过。当前线上临时处置是把旧镜像标记为 `workbench-screenshot-service:latest`，再用新 compose 项目重建截图容器，使它加入 `workbench_default` 网络并继续挂载 `/opt/opencode-workbench/data:/app/data`。
- 2026-07-08 使用 `scripts/deploy-author-with-data.sh --overwrite-data --confirm-overwrite-production-data --verify` 完成生产 data 覆盖：覆盖前备份为 `/opt/workbench-data-backups/20260708-103428-bind-data.tar.gz`，大小 109M，SHA256 `b700eed85415609637530a836428dc101626a9cd8cb3aace885cfe407f676212`。本地 data 上传 staging 后显示 761M、19 个一级条目；覆盖后正式 `/opt/opencode-workbench/data` 同为 761M、19 个一级条目。

## 风险

- 当前截图服务镜像是旧稳定镜像重标记，不是本轮新构建产物；如果后续直接执行 `INCLUDE_SCREENSHOT_SERVICE=true scripts/deploy.sh`，可能再次构建出 Chromium 150 镜像并复现启动失败。
