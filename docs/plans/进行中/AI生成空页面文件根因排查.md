# AI生成空页面文件根因排查

## 背景

创作端项目 `proj_1779608460372` 的页面 `demo_1779608460501_abc123` 曾出现预览不显示。现场证据显示，该页面在部分工作区和历史快照中的 `index.tsx` 与 `config.schema.json` 为 0 字节，预览运行时报出 JSX runtime named export 相关错误。

该项目不是空项目。正式项目工作区的 `workspace-tree.json` 声明有两个页面，`workspace/demos/` 下也应存在两个页面目录及其页面代码和 Schema。因此 0 字节页面文件不是合法空状态，而是工作空间文件状态被破坏。

## 目标

- 确认空页面文件出现在哪些工作区、版本快照和会话链路中。
- 按 [会话管理需求](../../项目文档/创作端/03-项目管理/会话管理_需求文档.md) 校准 Session 与 Workspace 的业务边界。
- 定位哪条链路违反“Session 只切换 AI 上下文、Workspace 承载文件状态”的设计。
- 在根因明确后再设计修复，避免只增加写入拦截和前端兜底。

## 范围

- 创作端编辑页会话创建、对话新建/切换、Workspace 复用与工作区文件读取。
- 实时协同保存、AI 文件写入、自动检查点和项目版本记录只作为传播链路继续核对。
- 不回滚或整理与本次排查无关的工作区改动。

## 当前判断

预览报错不是根因。空文件被当成模块参与预览编译后，才表现为误导性的 JSX runtime named export 错误。

此前提出的“AI 工具空写拒绝、协同落盘空写拒绝、前端收到空协同值时用本地非空缓存回填”只能止血，且会增加系统复杂度。这些改动已经撤回，不再作为当前方案。

2026-06-30 复核左侧栏“文件 → 代码”弹窗后，当前更可信的根因收敛为：可编辑文件弹窗和主编辑器都会连接协同房间；当某个页面代码的协同房间已经在 agent-service 内存中处于空文本状态时，即使 REST 文件读取接口先从 Workspace 读到了非空内容，前端仍会在协同同步后用空协同值覆盖编辑器内容。若该空值继续触发页面状态同步、协同回填或版本检查点，就会把“空协同房间”传播成空工作区文件和空历史快照。

这一链路比单纯的 Session/Workspace 复用问题更能解释用户截图：本地磁盘上的 `data/projects/proj_1779608460375/workspace/demos/demo_1779934698596_dvvk55/index.tsx` 和多个 `data/workspaces/.../proj_1779608460375/.../index.tsx` 当前均为 8KB 左右的非空文件，但左侧栏代码弹窗仍可能显示空内容。

已确认的具体问题点：

- `WorkspaceCodeDialog` 打开可编辑页面文件时，先接收 REST 读取到的文件内容，再通过 `useCollabDocument` 接入协同房间；原实现没有等待协同完成，也没有区分“连接中的默认空值”和“真实远端空值”。
- 主编辑区对页面 Schema 已有“空协同值 + 本地非空 Schema”回填保护，但页面 Code 缺少同类保护。
- agent-service 的协同房间在内存中复用时直接返回既有房间；若房间曾经以空文件创建，之后磁盘文件被修复为非空，只要服务未重启，该房间不会重新从磁盘读取当前内容。

Session/Workspace 解耦仍是需要继续排查的传播条件，但不再作为当前最主要根因。需求规定：

- Workspace 是文件状态载体，同一 Workspace 可被多个 Session 共享。
- Session 只是 AI 交互上下文。
- 新建或切换对话不应改变代码、Schema 和编辑器内容。
- 刷新或重新进入编辑页后，应看到最近一次已同步的 Workspace 内容。
- 打开项目编辑时应创建或复用工作空间。

## 已排除或降级的方向

- 预览 JSX runtime named export 错误：这是空文件进入预览编译后的表象，不是文件变空的原因。
- 只在 AI `writeFile` 上拒绝空内容：只能覆盖模型工具调用入口，不能解释 Workspace 为什么已经处于空文件状态。
- 只在协同落盘上拒绝空内容：只能阻止一次落盘，不能解释协同文档或当前 Workspace 为什么会出现空资源。
- 前端用本地缓存回填空协同值：会把数据修复职责推到 UI 状态层，和会话切换、页面切换、多人协作叠加后复杂度过高。
- 只排查 Session/Workspace 复用：仍有价值，但不能解释当前“磁盘文件非空、文件弹窗显示空”的现场。

## 下一步排查方案

1. 观察修复后同一项目再次打开左侧栏文件弹窗时，空协同房间是否被非空文件内容回填。
2. 继续追踪编辑页首次进入项目时的 `/api/sessions` 请求参数，确认是否无条件 `forceNew` 且未传入已有 `workspaceId`。
3. 追踪 `POST /api/sessions` 中 `archiveActiveSession`、`findActiveSession`、`createEditSession` 和 `createWorkspace` 的实际顺序，确认是否导致活跃 Workspace 无法复用。
4. 核对新建对话和切换对话回调是否严格满足“不改代码、不改 Schema、不重建 Workspace”。
