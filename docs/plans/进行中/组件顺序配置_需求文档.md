# 组件顺序配置 - 需求文档

> 版本：v1.1
> 创建日期：2026-05-02
> 更新日期：2026-05-02

---

## 一、功能背景

当前配置系统支持 AI 生成单个 React 组件，并通过 JSON Schema 驱动配置面板调整组件属性。但在实际场景中，一个 Demo 往往由**多个子组件**组合而成（如：标题 + 描述 + 按钮组 + 图片列表），用户需要调整这些子组件的**排列顺序**来达到最佳布局效果。

核心设计原则：**不是所有组件都支持顺序调整**，而是由 AI 在生成组件时声明哪些子组件可排序，配置面板据此提供排序能力。

---

## 二、功能描述

### 2.1 AI 声明可排序子组件

#### 2.1.1 Schema 扩展

**用户视角**：
- AI 生成 Demo 时，可在 JSON Schema 中声明哪些子组件支持顺序调整
- 未声明的子组件不参与排序，保持固定位置

**交互效果**：
- AI 通过 Schema 中的扩展字段 `$demo.orderable` 声明可排序子组件列表
- 声明后，配置面板自动出现"组件排序"区域

**Schema 示例**：

```json
{
  "type": "object",
  "properties": {
    "title": { "type": "string", "title": "标题" },
    "description": { "type": "string", "title": "描述" },
    "showBadge": { "type": "boolean", "title": "显示徽章" }
  },
  "$demo": {
    "orderable": ["title", "description", "showBadge"]
  }
}
```

#### 2.1.2 声明规则

| 规则 | 说明 |
|:-----|:-----|
| 可选声明 | `orderable` 为可选字段，未声明时配置面板不显示排序功能 |
| 精确控制 | 只有 `orderable` 列表中的属性名对应的子组件才可排序 |
| 属性名对应 | `orderable` 中的名称必须与 `properties` 中的属性名一致 |
| 最低数量 | `orderable` 列表至少包含 2 项，否则无排序意义 |

### 2.2 配置面板排序控件

#### 2.2.1 交互方式选型

排序控件采用**箭头按钮排序**（上下移动），而非拖拽排序。选型理由：

| 对比维度 | 箭头按钮 | 拖拽排序 |
|:---------|:---------|:---------|
| 发现性 | ✅ 一看就懂，无需学习 | ❌ 需要发现拖拽手柄 |
| 操作精确度 | ✅ 点击即移动，不会误操作 | ❌ 窄空间下容易拖错位置 |
| 实现复杂度 | ✅ 无需额外依赖 | ❌ 需引入 dnd-kit 等库 |
| 无障碍性 | ✅ 键盘可操作 | ❌ 键盘难以操作 |
| 适用数量 | ✅ 2-8 项完全够用 | ✅ 适合大量项目 |
| 侧边栏适配 | ✅ 窄空间无影响 | ❌ 拖拽空间受限 |

排序项通常为 2-6 个子组件，箭头按钮在此数量级下效率与拖拽持平，且更符合项目"简洁优先"原则。

#### 2.2.2 箭头按钮排序区域

**用户视角**：
- 配置面板中出现"组件排序"分组，展示可排序的子组件列表
- 用户可通过上下箭头按钮调整子组件的排列顺序
- 点击箭头后预览区实时反映新顺序

**交互效果**：
- 排序区域以独立分组形式出现在配置面板顶部（"组件排序"分组）
- 每个可排序项显示为一行，包含：左侧组件标题、右侧上移 ↑ 和下移 ↓ 按钮
- 第一项的上移按钮禁用，最后一项的下移按钮禁用
- 点击箭头后，对应项与相邻项交换位置，预览区在 16ms 内更新渲染
- 位置交换添加 150ms 过渡动画，让变化可感知

#### 2.2.3 排序状态管理

**用户视角**：
- 排序结果应持久化，切换页面后不丢失
- 可一键恢复默认顺序

**交互效果**：
- 排序结果作为配置数据的一部分保存
- 提供"恢复默认顺序"按钮
- 排序变更实时同步到预览区

---

### 2.3 预览区顺序渲染

#### 2.3.1 按序渲染子组件

**用户视角**：
- 预览区中的子组件按照配置面板设定的顺序渲染
- 顺序变更后，预览区立即更新

**交互效果**：
- 组件代码通过约定的 props 接收排序信息
- 预览区根据排序信息调整子组件渲染顺序
- 排序变更不触发重新编译，仅通过 `UPDATE_CONFIG` 消息更新

#### 2.3.2 排序数据传递

**用户视角**：
- 排序结果作为组件 props 的一部分传递，组件代码据此渲染

**交互效果**：
- 排序数据以 `__order` 属性注入组件 props
- `__order` 为字符串数组，值与 `orderable` 列表中的属性名对应
- 组件代码读取 `__order` 决定子组件渲染顺序

**Props 示例**：

```typescript
// 组件接收到的 props
{
  title: "欢迎使用",
  description: "这是一个示例",
  showBadge: true,
  __order: ["description", "title", "showBadge"]  // 排序信息
}
```

## 三、业务规则与约束

### 3.1 排序范围约束

| 约束 | 说明 |
|:-----|:-----|
| 仅限声明项 | 只有 AI 在 `orderable` 中声明的属性才可排序 |
| 固定项不移动 | 未在 `orderable` 中的子组件保持原始位置，不受排序影响 |
| 全部可排序 | 若 `orderable` 包含所有 properties，则全部子组件可自由排序 |

### 3.2 排序与固定项的混合渲染

当存在固定项和可排序项时，渲染规则如下：

| 场景 | 渲染规则 |
|:-----|:---------|
| 全部可排序 | 按 `__order` 顺序依次渲染 |
| 部分可排序 | 固定项保持原始位置，可排序项在各自相邻的固定项之间按 `__order` 排列 |
| 无可排序项 | 不传递 `__order`，按原始顺序渲染 |

**示例**：假设 properties 定义顺序为 `A, B, C, D, E`，其中 `orderable: ["B", "D", "E"]`

- 默认渲染：A → B → C → D → E
- 调整后渲染（`__order: ["E", "B", "D"]`）：A → E → C → B → D
- A 和 C 是固定项，位置不变；B、D、E 在各自相邻的固定项之间按 `__order` 排列

### 3.3 数据完整性约束

| 约束 | 说明 |
|:-----|:-----|
| `__order` 完整性 | `__order` 必须包含 `orderable` 中的所有项，不能遗漏或多余 |
| 默认值 | 未调整顺序时，`__order` 等于 `orderable` 的原始顺序 |
| 类型安全 | `__order` 为 `string[]` 类型，每个元素是有效的属性名 |

### 3.4 命名约定

| 约定 | 说明 |
|:-----|:-----|
| `__order` | 双下划线前缀表示系统注入属性，避免与用户定义的 props 冲突 |
| `$demo.orderable` | 遵循现有 `$demo` 扩展字段命名规范 |

## 四、配置定义格式规范

### 4.1 Schema 扩展格式

在现有 `$demo` 扩展字段中新增 `orderable`：

```json
{
  "type": "object",
  "properties": {
    "header": { "type": "string", "title": "页头" },
    "content": { "type": "string", "title": "内容" },
    "footer": { "type": "string", "title": "页脚" },
    "theme": { "type": "string", "title": "主题", "enum": ["light", "dark"] }
  },
  "$demo": {
    "previewSize": { "width": "100%", "height": "600px" },
    "orderable": ["header", "content", "footer"]
  }
}
```

### 4.2 配置值格式

排序结果作为配置数据的一部分：

```json
{
  "header": "标题文本",
  "content": "内容文本",
  "footer": "页脚文本",
  "theme": "light",
  "__order": ["content", "header", "footer"]
}
```

## 五、配置面板交互规范

### 5.1 排序控件位置

- 排序控件作为独立分组"组件排序"出现在配置面板**最顶部**
- 仅当 Schema 中存在 `$demo.orderable` 时显示

### 5.2 排序控件样式

| 元素 | 样式说明 |
|:-----|:---------|
| 分组标题 | "组件排序"，带排序图标 |
| 排序项行 | 单行布局，左侧组件标题（使用 Schema 中的 title），右侧上移/下移图标按钮 |
| 箭头按钮 | 小尺寸图标按钮（ChevronUp / ChevronDown），禁用态置灰 |
| 当前位置 | 每行左侧显示序号标记（1、2、3...） |
| 恢复按钮 | 分组标题右侧，点击恢复默认顺序 |

### 5.3 交互行为

| 行为 | 说明 |
|:-----|:-----|
| 上移 | 点击 ↑ 按钮，该项与上方相邻项交换位置 |
| 下移 | 点击 ↓ 按钮，该项与下方相邻项交换位置 |
| 边界禁用 | 第一项 ↑ 禁用，最后一项 ↓ 禁用 |
| 实时预览 | 点击箭头后立即更新预览区 |
| 过渡动画 | 位置交换时 150ms 过渡动画 |
| 恢复默认 | 点击"恢复默认顺序"按钮，回到 Schema 定义顺序 |
| 禁用状态 | `orderable` 不足 2 项时，排序区域不显示 |

### 5.4 未来扩展

v1 采用箭头按钮排序。若用户反馈需要拖拽排序，可在 v2 引入 dnd-kit 库扩展，排序数据结构（`__order`）无需变更。

## 六、AI 生成组件约定

### 6.1 AI 何时声明 orderable

AI 在以下场景应考虑声明 `orderable`：

| 场景 | 示例 |
|:-----|:-----|
| 多区块布局 | 卡片列表、功能模块、步骤条 |
| 可重排内容 | 图文混排、FAQ 列表、特性展示 |
| 灵活组合 | 仪表盘卡片、信息面板、导航项 |

### 6.2 AI 何时不应声明 orderable

| 场景 | 原因 |
|:-----|:-----|
| 语义固定顺序 | 步骤流程（1→2→3）、时间线 |
| 布局依赖顺序 | 表头→表体→表尾、导航→内容→侧栏 |
| 单一组件 | 只有一个子组件，无需排序 |

### 6.3 组件代码约定

AI 生成的组件代码应遵循以下约定：

```typescript
interface DemoProps {
  header: string;
  content: string;
  footer: string;
  __order?: string[];  // 可选，系统注入的排序信息
}

function Demo({ header, content, footer, __order }: DemoProps) {
  const order = __order || ["header", "content", "footer"];

  const sections: Record<string, React.ReactNode> = {
    header: <HeaderSection text={header} />,
    content: <ContentSection text={content} />,
    footer: <FooterSection text={footer} />,
  };

  return (
    <div>
      {order.map((key) => sections[key])}
    </div>
  );
}
```

## 七、与现有系统的集成点

### 7.1 Schema 解析器

- 扩展 `getPreviewSize()` 同级逻辑，新增 `getOrderable()` 函数
- 从 `$demo.orderable` 提取可排序属性列表

### 7.2 配置面板（ConfigForm）

- 在 `parseSchemaToFields()` 中识别 `orderable` 字段
- 新增"组件排序"分组，渲染箭头按钮排序控件
- 排序变更通过 `onChange` 回调更新 `__order` 字段

### 7.3 预览面板（PreviewPanel）

- `__order` 作为 `configData` 的一部分通过 `UPDATE_CONFIG` 消息传递
- 无需修改 iframe 通信机制，现有 `postMessage` 链路已支持

### 7.4 默认值提取

- `getDefaultValues()` 新增逻辑：当 Schema 包含 `orderable` 时，自动生成 `__order` 默认值
- 默认值等于 `orderable` 数组的原始顺序

## 八、验收标准

| 编号 | 验收项 | 预期结果 |
|:-----|:-------|:---------|
| AC-1 | Schema 含 `orderable` | 配置面板顶部出现"组件排序"分组 |
| AC-2 | Schema 不含 `orderable` | 配置面板不显示排序功能，行为与现有一致 |
| AC-3 | 箭头按钮调整顺序 | 点击上移/下移按钮，预览区实时反映新顺序 |
| AC-4 | 恢复默认顺序 | 排序回到 Schema 定义顺序 |
| AC-5 | 部分可排序 | 固定项位置不变，可排序项在固定项之间调整 |
| AC-6 | `__order` 传递 | 组件 props 中包含 `__order` 数组 |
| AC-7 | 配置持久化 | 排序结果随配置数据保存，刷新后不丢失 |
| AC-8 | `orderable` 不足 2 项 | 不显示排序区域 |
