# 创作端画布页面位置保存缺失-分析报告

## 背景

用户反馈：在创作端画布中修改页面位置后，保存按钮没有亮起，页面在画布中的位置信息似乎不会保存。

## 目标

- 定位页面画布位置变更没有触发保存状态的根因。
- 确认保存 payload 是否包含页面位置信息。
- 在最小范围内修复，并补充匹配验证。

## 范围

- 创作端编辑页画布交互、dirty 状态、保存逻辑。
- shared 画布页面节点类型和事件回调。
- 不处理与本问题无关的项目管理、发布、Agent 或使用端功能改动。

## 方案

1. 收集环境与代码路径证据。
2. 追踪页面位置变更从画布组件到编辑页状态的调用链。
3. 修复位置变更未纳入保存状态或保存数据的问题。
4. 运行与创作端相关的最小验证。

## 根因分析

画布页面位置变更的数据流是：`CanvasPageItem` 拖拽/缩放时触发 `PreviewCanvas` 的布局更新，`PreviewCanvas` 再通过 `onCanvasStateChange` 更新 `useCanvasWorkspace` 中的 `canvasState`。后端 `canvas-layout` API 能保存 `CanvasState.pages`，存储路径也包含 workspace 与 session。

问题根因不在后端存储，而在前端 dirty 状态边界：`useCanvasWorkspace` 原本只用 `dirtyRef` 记录画布变更，并通过 debounce 自动保存，但这个 ref 不会触发编辑页重新渲染，也没有返回给顶栏。编辑页保存按钮、返回提醒、发布前保存只判断页面内容的 `hasUnsavedChanges`，因此移动页面位置后顶栏保存按钮不会亮起；如果用户在自动保存 debounce 完成前返回，保存定时器可能被清理，画布位置变更存在丢失风险。

二次修复发现还需要区分两类“保存”：画布自动保存只是把布局写入 session/workspace 的 `.canvas-layout.json`，并不等价于顶栏保存项目、生成版本或更新发布前状态。若自动保存成功后清除顶栏 dirty，用户会看到保存按钮重新变灰，但项目级保存尚未发生。

发布后使用端位置不同的根因是发布数据契约缺字段：创作端画布布局持久化在 workspace `.canvas-layout.json`，但发布流程只把页面编译产物、页面清单、配置 schema 和项目信息写进 `published/{projectId}/project.json`。使用端无法读取 `.canvas-layout.json`，只能拿空 `CanvasState` 初始化，因此由共享 `PreviewCanvas` 的默认布局算法重新排版。

发布数据契约修复后仍不一致的根因是发布元数据缓存：`project.json` URL 不变，而数据路由对 JSON 返回 1 小时缓存。浏览器可继续使用旧 `project.json`，旧数据没有 `canvasState`，因此使用端仍按默认布局渲染。该问题与磁盘上的新发布产物是否正确无关，属于浏览器可见数据未及时失效。

## 实施摘要

- `useCanvasWorkspace` 新增 `hasUnsavedCanvasChanges` 状态，表示项目级画布变更待保存。
- `dirtyRef` 仅表示画布布局文件待写入；自动保存成功只清除 `dirtyRef`，不清除顶栏未保存状态。
- `useCanvasWorkspace` 新增 `markCanvasChangesSaved`，由编辑页项目保存成功后调用。
- 编辑页新增 `hasPendingChanges = hasUnsavedChanges || hasUnsavedCanvasChanges`，用于保存按钮、返回提醒、发布前保存与发布按钮禁用判断。
- 新增 hook 测试，验证画布布局变更会标记项目未保存，手动 flush 会 POST `canvas-layout` 但不清除项目 dirty，自动保存成功后也仍保留顶栏未保存状态。
- 新增 `canvas-layout-file` 读取 helper，复用 `.canvas-layout.json` 解析逻辑。
- 发布流程在 `project.json` 中写入 `canvasState`，实时 viewer API 也返回 `canvasState`。
- viewer-site 与 author-site 内置 viewer 在加载数据后使用保存的 `canvasState` 初始化画布。
- 发布数据路由对 JSON 返回 `Cache-Control: no-store`，保留 JS 产物长期缓存。
- viewer-site 请求发布 JSON 时使用 `fetch(..., { cache: "no-store" })`，双重避免旧发布元数据缓存。
