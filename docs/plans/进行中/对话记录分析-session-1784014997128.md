## 关键发现

### 1. `node -e` 沙箱拦截与错误信息矛盾

`permissions.ts` 将 `node` 列入白名单，但对 `node -e` / `node --eval` 做了特殊拦截（L88-90），直接返回 false。错误信息只输出基础命令名，导致 `node` 同时出现在 "Allowed" 列表和 "not allowed" 结果中。

理论上 `process.mainModule.require` 可以绕过 Node.js 的 require 限制：

```bash
node -e "process.mainModule.require('fs').unlinkSync('path/to/file')"
```

但当前沙箱已在命令层拦截 `node -e`，该绕过方式实际不可触发。

### 2. 原型→高保真升级路径设计缺陷

当前升级路径：

1. AI 创建原型页 → 校验失败 → 要求升级
2. AI 创建高保真页 → 成功
3. AI 尝试清理原型文件 → **全部失败**（无删除工具、bash 只读、清空触发校验）
4. AI 陷入死循环

改进方向：

- 为 AI 工具链增加 `deleteFile` 工具
- 或在 prototype→high-fidelity 升级时由系统自动清理原型文件
- 或采用 `transform: translateZ(0)` 方案，从根源消除升级需求

### 3. 系统自动修改 workspace-tree.json 导致 drift（设计预期）

系统在 AI 写入 `workspace-tree.json` 后自动添加了 `routeKey: "page"` 字段，导致 revision 跳变，触发 EXTERNAL_DRIFT。经调研确认这是设计预期行为：`file-tools.ts` 的 `readFile`/`writeFile`/`listFiles` 均已实现 EXTERNAL_DRIFT 自动重试（捕获后调用 `reconcileAdopt` 对齐 revision），AI 重试即可恢复。

### 4. WMA 底层已支持 `delete_path` 操作

调研发现 `workspace-mutation-authority.ts` L734/L784 已实现 `delete_path` 操作类型，`delete-page-tool.ts` L447-451 已使用该操作删除页面文件。新增 `deleteFile` 工具只需复用同一模式，无需修改 WMA 底层。
