# 数据目录架构优化方案

> **变更概述**：将数据目录从 `packages/web/data/` 迁移到项目根目录 `data/`，统一环境变量配置，实现代码与数据解耦，为 Docker 部署和多 Agent 协作奠定基础。

---

## 一、问题分析

### 1.1 当前架构现状

当前数据存储存在**两套独立实现**：

| 包 | 数据路径配置 | 环境变量 | 默认位置 | 实际位置（dev 模式） |
|---|---|---|---|---|
| `web` | `fs-utils.ts` 中的 `DATA_DIR` | `DATA_DIR` | `path.join(process.cwd(), "data")` → `packages/web/data/` | `packages/web/data/` |
| `agent-service` | `project-workspace-manager.ts` 中的 `BASE_DIR` | `PROJECTS_BASE_DIR` | `path.join(process.cwd(), 'data')` → `packages/agent-service/data/` | `packages/web/data/`（通过脚本环境变量覆盖） |

**关键事实**：
- `web` 包运行时 `process.cwd()` 为 `packages/web/`，所以默认数据在 `packages/web/data/`
- `agent-service` 运行时 `process.cwd()` 为 `packages/agent-service/`，默认数据在 `packages/agent-service/data/`
- **当前 dev 模式下两者已指向同一位置**：根 `package.json` 的 `dev` / `dev:agent` 脚本通过 `cross-env PROJECTS_BASE_DIR=../web/data` 将 agent-service 指向 `packages/web/data/`
- 这意味着当前通过**脚本级环境变量**实现了数据共享，但属于临时方案，不够健壮

**两包的子目录结构差异**：

| 子目录 | `web` 包 | `agent-service` 包 |
|--------|---------|-------------------|
| `projects/` | ✅ 项目元数据 + workspace | ✅ 项目元数据 + workspace |
| `sessions/` | ✅ 会话目录（含工作空间副本） | ✅ 会话元数据（JSON 文件） |
| `workspaces/` | ✅ 用户工作空间 | ❌ 不存在 |
| `snapshots/` | ✅ 版本快照 | ✅ 版本快照 |

**两包的 sessions 结构差异**（重要）：
- `web`：`sessions/{userId}/{projectId}/{sessionId}/` — 目录结构，内含工作空间文件副本和 `.session.json`
- `agent-service`：`sessions/{projectId}/{sessionId}.json` — 扁平 JSON 文件，仅存元数据

### 1.2 核心问题

#### 问题 1：数据共享依赖脚本级环境变量
当前 agent-service 通过 `package.json` 脚本中的 `PROJECTS_BASE_DIR=../web/data` 指向 web 的数据目录：
- 未通过脚本启动时（如生产部署、Docker），两个包默认指向不同位置
- 环境变量名不统一（`DATA_DIR` vs `PROJECTS_BASE_DIR`），概念相同但命名不同
- 相对路径 `../web/data` 依赖特定的目录结构，不够健壮

#### 问题 2：数据耦合在代码包中
`packages/web/data/` 位于 web 包内部，存在风险：
- `pnpm install` / `pnpm build` 可能意外覆盖数据
- Docker 容器重建时数据丢失
- **用户数据被 Git 跟踪**：根 `.gitignore` 无 data 规则，`packages/web/.gitignore` 仅忽略 `*.db` 文件，其他用户数据（项目、会话、快照）会被提交
- 代码迁移时需要单独处理数据

#### 问题 3：路径计算存在硬编码绕过
部分文件绕过了 `fs-utils.ts` 的统一路径配置，直接硬编码路径：
- `packages/web/src/app/api/sessions/project/[projectId]/route.ts`：自行计算 `SESSIONS_DIR`，未考虑 `DATA_DIR` 环境变量
- `packages/web/scripts/migrate-sessions.ts`：硬编码 `path.join(process.cwd(), 'data', 'sessions')`，完全忽略环境变量

#### 问题 4：Docker 部署不友好
当前架构在 Docker 场景下：
- 无法将数据持久化到容器外
- 无法在多容器间共享数据
- 无法方便地备份和迁移

### 1.3 为什么文件系统是正确的选择

对于当前项目特点（局域网内部工具、多 Agent 协作），文件系统是最佳存储方案：

| 考量 | 文件系统 | 数据库 |
|------|---------|--------|
| Agent 友好性 | ✅ 直接操作文件，无需适配层 | ❌ 需要额外接口 |
| 部署复杂度 | ✅ 无依赖 | ❌ 需要数据库服务 |
| 数据规模 | ✅ 适合当前规模 | ⚠️ 过度设计 |
| 并发控制 | ⚠️ 需应用层处理 | ✅ 原生支持 |
| 备份迁移 | ✅ 简单（cp/rsync） | ⚠️ 需要导出导入 |

**结论**：保留文件系统，但需要统一数据目录位置。

## 二、优化目标

### 2.1 目标列表

1. **统一数据位置**：所有包读写同一个 `data/` 目录
2. **环境变量驱动**：通过环境变量配置数据路径，支持外部挂载
3. **代码与数据解耦**：数据不在任何包内部，部署时独立管理
4. **Docker 就绪**：数据目录可作为 volume 持久化
5. **Agent 友好**：Agent 可直接通过文件系统路径操作项目
6. **消除硬编码路径**：所有路径计算统一通过配置模块，消除绕过

### 2.2 目标架构

```
项目根目录/
├── data/                    # ← 统一数据目录（与 packages 平级）
│   ├── projects/            # 项目工作区
│   ├── sessions/           # 临时会话（可自动清理）
│   ├── workspaces/          # 用户工作空间
│   └── snapshots/           # 版本快照
├── packages/
│   ├── web/                 # Studio 前端（无 data/ 子目录）
│   ├── agent-service/       # Agent 服务
│   ├── shared/              # 共享类型
│   ├── agent-client/        # Agent 客户端 SDK
│   └── viewer-site/         # 预览站点
└── docker-compose.yml        # Docker 编排配置
```

## 三、实施方案

### 3.1 环境变量配置统一

将分散的环境变量统一为：

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `DATA_DIR` | 数据根目录 | `path.join(projectRoot, 'data')` |
| `PROJECTS_DIR` | 项目目录 | `path.join(DATA_DIR, 'projects')` |
| `SESSIONS_DIR` | 会话目录 | `path.join(DATA_DIR, 'sessions')` |
| `WORKSPACES_DIR` | 工作空间目录 | `path.join(DATA_DIR, 'workspaces')` |
| `SNAPSHOTS_DIR` | 快照目录 | `path.join(DATA_DIR, 'snapshots')` |

> **注意**：`web` 包已使用上述环境变量名，`agent-service` 需将 `PROJECTS_BASE_DIR` 迁移为 `DATA_DIR`。

**项目根目录计算方式**：

采用**向上查找标记文件**的方式，比字符串匹配更健壮：

```typescript
import fs from 'fs';
import path from 'path';

function findProjectRoot(cwd: string): string {
  let current = path.resolve(cwd);
  while (current !== path.dirname(current)) {
    if (fs.existsSync(path.join(current, 'pnpm-workspace.yaml'))) {
      return current;
    }
    current = path.dirname(current);
  }
  return cwd;
}
```

> **为什么不用 `lastIndexOf('packages')`**：
> 1. 路径中可能出现多个 `packages` 目录（如 `packages/my-packages/`），导致匹配错误
> 2. Windows 路径分隔符 `\` 与 `/` 混合时字符串匹配不可靠
> 3. `cwd.substring(0, packagesIndex - 1)` 假设 `packages` 前总有路径分隔符，边界情况不安全
> 4. 使用 `pnpm-workspace.yaml` 作为标记文件，语义明确且不会误匹配

### 3.2 需要修改的文件

#### 3.2.1 路径配置模块

| 文件 | 修改内容 |
|------|----------|
| `packages/web/src/lib/fs-utils.ts` | 更新默认路径计算逻辑，使用 `findProjectRoot()` 替代 `process.cwd()` |
| `packages/agent-service/src/workspace/project-workspace-manager.ts` | 将 `PROJECTS_BASE_DIR` 迁移为 `DATA_DIR`，使用 `findProjectRoot()` 计算默认值 |
| `packages/web/src/app/api/sessions/project/[projectId]/route.ts` | **消除硬编码路径**，改用 `getSessionsDir()` |
| `packages/web/scripts/migrate-sessions.ts` | **消除硬编码路径**，改用环境变量或导入 `fs-utils` |
| `packages/web/src/lib/db/index.ts` | **消除硬编码路径** `path.join(process.cwd(), 'data', 'users.db')`，改用 `getDataDir()` 计算数据库路径 |

#### 3.2.2 启动脚本

| 文件 | 修改内容 |
|------|----------|
| `package.json`（根目录） | 移除 `dev` / `dev:agent` 脚本中的 `PROJECTS_BASE_DIR=../web/data`，改为依赖默认路径计算 |

当前脚本：
```json
"dev": "concurrently -n web,agent,viewer -c blue,green,cyan \"pnpm --filter @opencode-workbench/web dev\" \"cross-env PROJECTS_BASE_DIR=../web/data pnpm --filter @opencode-workbench/agent-service dev\" \"pnpm --filter @opencode-workbench/viewer-site dev\"",
"dev:agent": "cross-env PROJECTS_BASE_DIR=../web/data pnpm --filter @opencode-workbench/agent-service dev"
```

迁移后脚本：
```json
"dev": "concurrently -n web,agent,viewer -c blue,green,cyan \"pnpm --filter @opencode-workbench/web dev\" \"pnpm --filter @opencode-workbench/agent-service dev\" \"pnpm --filter @opencode-workbench/viewer-site dev\"",
"dev:agent": "pnpm --filter @opencode-workbench/agent-service dev"
```

#### 3.2.3 Docker 配置

创建 `docker-compose.yml`：
```yaml
services:
  web:
    build: ./packages/web
    ports:
      - "3000:3000"
    environment:
      - DATA_DIR=/app/data
    volumes:
      - ./data:/app/data
    depends_on:
      - agent-service

  agent-service:
    build: ./packages/agent-service
    ports:
      - "3101:3101"
    environment:
      - DATA_DIR=/app/data
    volumes:
      - ./data:/app/data
```

> **注意**：已移除 `version: '3.8'`，该字段在 Docker Compose V2 中已废弃。

#### 3.2.4 Git 配置

更新根目录 `.gitignore`，添加：
```gitignore
# 数据目录
/data/
```

更新 `packages/web/.gitignore`，移除旧的 data 相关规则：
```gitignore
# 移除以下行（数据已不在 web 包内）
# data/*.db
# data/*.db-wal
# data/*.db-shm
```

### 3.3 数据迁移步骤

#### 步骤 1：备份现有数据
```bash
cp -r packages/web/data data_backup
```

#### 步骤 2：迁移数据
```bash
# packages/web/data/ 已包含正确的子目录结构，直接移动整个目录
mv packages/web/data data
```

> **注意**：不需要预先 `mkdir -p` 创建子目录，因为 `packages/web/data/` 已包含 `projects/`、`sessions/`、`workspaces/`、`snapshots/` 四个子目录。

#### 步骤 3：修改代码
- 更新 `fs-utils.ts` 和 `project-workspace-manager.ts` 的路径计算逻辑
- 修复 `route.ts` 和 `migrate-sessions.ts` 的硬编码路径
- 更新根 `package.json` 脚本
- 更新 `.gitignore` 配置

#### 步骤 4：验证
- 启动服务，验证所有功能正常
- 确认 Agent 能访问项目文件
- 确认 web 能正常读写数据
- 确认 `DATA_DIR` 环境变量可覆盖默认路径

### 3.4 向后兼容处理

为了平滑迁移，提供兼容处理：

1. **日志提示**：启动时打印数据目录位置，便于排查
2. **环境变量优先**：`DATA_DIR` 环境变量始终优先于默认值
3. **废弃 `PROJECTS_BASE_DIR`**：agent-service 仍支持读取 `PROJECTS_BASE_DIR` 作为 fallback，但打印弃用警告

> **不使用软链接**：Windows 上创建符号链接需要管理员权限或开发者模式，兼容性差。优先使用直接迁移。

## 五、风险与注意事项

| 风险 | 缓解措施 |
|------|----------|
| 迁移过程中数据丢失 | 迁移前完整备份 |
| 路径计算错误 | 添加启动时路径校验日志 |
| 硬编码路径遗漏 | 全局搜索 `process.cwd()` + `"data"` 组合，确保无遗漏 |
| 现有功能回归 | 迁移后全面测试 |
| Git 历史中残留用户数据 | 迁移后检查并清理 Git 历史中的敏感数据 |

## 六、验收标准

1. ✅ `packages/web/` 目录下不再有 `data/` 子目录
2. ✅ 项目根目录 `data/` 包含所有用户数据
3. ✅ `web` 和 `agent-service` 包读写同一数据位置（无需脚本级环境变量覆盖）
4. ✅ Docker Compose 启动后数据持久化到宿主机
5. ✅ Agent 能正常读写项目工作区文件
6. ✅ 所有环境变量支持通过外部配置覆盖
7. ✅ `route.ts` 和 `migrate-sessions.ts` 不再有硬编码路径
8. ✅ 根 `package.json` 脚本不再需要 `PROJECTS_BASE_DIR` 环境变量
9. ✅ 根 `.gitignore` 包含 `/data/` 规则，用户数据不被 Git 跟踪
