# 预览区页面比例统一异常分析报告

> 状态：**分析完成**，根因为 API 层 `extractPreviewSize` 与前端 `getPreviewSize` 返回值结构不一致

---

## 一、问题背景

### 1.1 问题现象

- **单页预览（PreviewPanel）**：无论手机页面（375×812）还是 iPad 页面（1024×768），预览比例都相同
- **宫格预览（PreviewGrid）**：同一行中不同比例的卡片显示为相同比例
- **预期行为**：不同页面应根据各自的 `config.schema.json` 中 `$demo.previewSize` 显示不同比例
- **实际行为**：所有页面显示为统一比例（默认 375/812 手机比例）

### 1.2 复现条件

- 项目包含多个不同比例的页面（如手机和 iPad）
- 在编辑页或 Viewer 页切换不同页面预览
- 宫格视图中同一行放置不同比例的页面

### 2.3 关键函数对比

#### 前端 `getPreviewSize`（lib/validator.ts:328-359）

```typescript
export function getPreviewSize(schema: string): PreviewSize | undefined {
  try {
    const parsed = JSON.parse(schema);
    if (!parsed.$demo?.previewSize) {
      return undefined;
    }
    const preview = parsed.$demo.previewSize;
    const size: PreviewSize = {};
    if (preview.width !== undefined) {
      size.width = preview.width;   // 保持原值（string | number）
    }
    if (preview.height !== undefined) {
      size.height = preview.height; // 保持原值（string | number）
    }
    // ...
    return Object.keys(size).length > 0 ? size : undefined;
  } catch {
    return undefined;
  }
}
```

**返回值示例**：`{ width: "1024px", height: "768px" }`

#### API 层 `extractPreviewSize`（src/lib/preview-size.ts:9-17）

```typescript
export function extractPreviewSize(schemaStr: string): PreviewSize | undefined {
  try {
    const schema = JSON.parse(schemaStr);
    if (schema.$demo?.previewSize) {
      return schema.$demo.previewSize;  // 直接返回原对象
    }
  } catch {}
  return undefined;
}
```

**返回值示例**：`{ width: "1024px", height: "768px" }`

#### 尺寸解析 `parseSizeValue`（PreviewPanel.tsx 和 PreviewGrid.tsx）

```typescript
function parseSizeValue(value: string | number | undefined): number | null {
  if (typeof value === "number") return value;
  if (typeof value === "string") {
    const num = parseFloat(value.replace(/px$/, ""));
    return isNaN(num) ? null : num;
  }
  return null;
}
```

**解析示例**：
- `"1024px"` → `parseFloat("1024")` → `1024` ✅
- `"768px"` → `parseFloat("768")` → `768` ✅

## 三、根因分析

### 3.1 直接原因

**所有页面的 `config.schema.json` 中 `$demo.previewSize` 配置值相同**，导致前端解析出的比例自然相同。

### 3.2 深层原因分析

用户提到"之前是正常的"，说明曾经有不同比例的页面配置。可能的原因：

1. **Schema 被覆盖**：在 AI 生成或编辑过程中，Schema 被统一覆盖为相同的 `previewSize`
2. **复制页面时未修改**：创建新页面时复制了已有页面的 Schema，未调整 `previewSize`
3. **AI 生成代码时固定了比例**：AI 在生成组件代码时，统一使用了 iPad 比例

### 3.3 代码逻辑验证

#### 验证 1：前端解析逻辑是否正确

`getPreviewSize` 和 `extractPreviewSize` 都正确读取 `$demo.previewSize`，逻辑无误。

#### 验证 2：尺寸计算逻辑是否正确

`parseSizeValue("1024px")` → `1024`，`parseSizeValue("768px")` → `768`，宽高比 `1024/768 = 1.333`，计算正确。

#### 验证 3：宫格比例分配是否正确

```typescript
const ratios = row.map((p) => {
  const size = p.id === activePageId ? previewSize : (p.previewSize ?? previewSize)
  return getAspectRatioValue(size)
})
```

如果所有 `previewSize` 都相同，则 `ratios` 数组中所有值相同，Grid 列宽自然相同。

#### 验证 4：PreviewPanel 单页缩放是否正确

```typescript
const aspectRatio = designWidth / designHeight;  // 1024 / 768 = 1.333
```

如果所有页面都传入相同的 `previewSize`，缩放比例自然相同。

## 四、结论

### 4.1 根因定位

**根因：数据层（config.schema.json）中所有页面的 `$demo.previewSize` 配置被统一为相同值（`1024px × 768px`），而非代码逻辑错误。**

前端代码（`getPreviewSize`、`extractPreviewSize`、`parseSizeValue`、`computePreviewScale`、`getAspectRatioValue`）的逻辑都是正确的。问题在于**输入数据本身相同**，导致输出必然相同。

### 4.2 证据链

| 证据 | 级别 | 说明 |
|------|------|------|
| demo_1777965200000_x8k2p9 的 config.schema.json | A | `$demo.previewSize = {width: "1024px", height: "768px"}` |
| demo_1778077850198_fjxwmf 的 config.schema.json | A | `$demo.previewSize = {width: "1024px", height: "768px"}` |
| 前端 `getPreviewSize` 正确解析 schema | B | 逻辑正确，返回值与 schema 一致 |
| 前端 `parseSizeValue` 正确解析 "1024px" | B | 返回 1024，计算比例 1.333 |
| 两个 demo 名称分别为"手机"和"横屏 iPad mini" | B | 名称暗示应有不同比例，但配置相同 |

## 五、解决方案

### 方案 1：修正数据文件（推荐，立即生效）

手动修改各 demo 的 `config.schema.json`，将 `$demo.previewSize` 设置为正确的比例：

- **手机页面**（如 demo_1777965200000_x8k2p9）：
  ```json
  {
    "$demo": {
      "previewSize": {
        "width": "375px",
        "height": "812px"
      }
    }
  }
  ```

- **iPad 横屏页面**（如 demo_1778077850198_fjxwmf）：
  ```json
  {
    "$demo": {
      "previewSize": {
        "width": "1024px",
        "height": "768px"
      }
    }
  }
  ```

**优点**：立即生效，无需改代码  
**缺点**：需要手动修改每个页面的 Schema，且未来可能被再次覆盖

### 方案 2：增强 AI 生成代码时的比例提示（长期）

在 AI 生成组件的 prompt 中，明确告知根据页面类型（手机/平板/桌面）生成对应比例的 `previewSize`。

**优点**：从源头避免问题  
**缺点**：需要调整 AI Agent 的 prompt 模板

### 方案 3：添加页面比例校验/提示（辅助）

在编辑页面的配置面板中，显示当前页面的比例信息，并在比例与页面名称不匹配时给出提示。

**优点**：提升用户体验，帮助用户发现问题  
**缺点**：增加 UI 复杂度
