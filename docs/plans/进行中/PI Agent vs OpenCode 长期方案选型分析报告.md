## 一、背景与目的

### 1.1 项目现状

**Opencode Workbench** 是一个面向组件化开发的 AI 辅助创作与使用平台，其核心架构包括：

| 组件 | 技术栈 | 职责 |
|:-----|:-------|:-----|
| **author-site** | Next.js 14 | 创作端，AI 对话生成/修改组件代码 |
| **viewer-site** | Next.js 14 | 使用端，浏览、调整配置参数 |
| **agent-service** | Fastify + TypeScript | 独立 Agent 服务层，支持多后端切换 |

### 1.2 Agent 方案现状

agent-service 已支持 **13 种后端**：

```
opencode (已废弃) | opencode-http (默认) | claude | codex | gemini
qwen | goose | auggie | kimi | copilot | qoder | vibe | custom
```

**当前默认方案**：`opencode-http`（通过 HTTP 直连 OpenCode Server）

### 1.3 决策背景

1. **Pi Agent 已完成集成**：根据 [Pi Agent 后端集成方案 - 实施记录](..%5C远期规划%5Cpi%20agent%5CPi%20Agent%20后端集成方案%20-%20实施记录.md)，Pi Agent 后端已全部完成集成
2. **长期演进需求**：需要明确哪个方案更适合作为长期主力方案
3. **资源投入考量**：避免在两个方案上分散精力

## 二、方案介绍

### 2.1 OpenCode 方案

#### 2.1.1 简介

OpenCode 是由 **SST 团队**（serverless 框架厂商）开发的开源终端 AI 编程助手，采用 **Go 语言**实现，具备客户端/服务器架构。

#### 2.1.2 核心特性

| 特性 | 说明 |
|:-----|:-----|
| **技术栈** | Go 语言，客户端/服务器架构 |
| **模型支持** | 75+ LLM 提供商（Anthropic、OpenAI、Google、DeepSeek 等） |
| **会话持久化** | 进程常驻，会话跨终端断开保持连接 |
| **LSP 集成** | 语言服务器协议支持，诊断、上下文、错误感知 |
| **许可证** | MIT，开源免费 |
| **GitHub Stars** | 95K+ |
| **最新状态** | 项目已迁移至 **Crush**（Charm 团队接手） |

#### 2.1.3 在项目中的集成方式

```
创作端 ──▶ agent-service ──HTTP/SSE──▶ opencode serve (端口 4096) ──▶ 云端 LLM
```

- 通过 `opencode-http.ts` 后端适配器连接
- OpenCode Server 作为独立进程运行
- Agent 内部处理逻辑对 agent-service 是黑盒

### 2.2 PI Agent 方案

#### 2.2.1 简介

Pi 是由 **Mario Zechner**（libGDX 框架创始人，GitHub 23K+ stars）开发的极简终端编程 Agent，核心理念是「做减法」—— 用最少的工具和提示词实现强大的编码能力。

#### 2.2.2 核心特性

| 特性 | 说明 |
|:-----|:-----|
| **技术栈** | TypeScript/Node.js，可嵌入 SDK 模式 |
| **工具数量** | 核心仅 4 个（read, write, edit, bash） |
| **系统提示词** | 不到 1000 tokens（vs Claude Code 10K+） |
| **模型支持** | 15+ 提供商，支持会话中切换模型 |
| **会话结构** | 树形会话，分支历史，不会丢失工作 |
| **许可证** | MIT，开源免费 |
| **GitHub Stars** | 14.4K |
| **性能表现** | Terminal Bench 2.0 前五名 |
| **生态案例** | OpenClaw（160K+ stars）使用 Pi 作为核心引擎 |

#### 2.2.3 在项目中的集成方式

```
创作端 ──▶ agent-service ──进程内调用──▶ pi-agent-core ──API──▶ 云端 LLM
                   │
              事件转发
              文件同步
```

- 通过 `pi-agent.ts` 后端适配器连接
- pi-agent-core 运行在 agent-service 进程内
- 所有处理逻辑完全透明可控

## 三、多维度对比分析

### 3.1 功能完整性对比

| 维度 | OpenCode | PI Agent | 胜出 |
|:-----|:---------|:---------|:-----|
| **内置工具数量** | 20+ | 4（可扩展） | OpenCode |
| **LSP 集成** | ✅ 原生支持 | ❌ 需自行实现 | OpenCode |
| **代码诊断** | ✅ 内置 | ❌ 需自行实现 | OpenCode |
| **多模型切换** | ✅ 支持 | ✅ 支持 | 平局 |
| **树形会话** | ❌ 不支持 | ✅ 支持 | PI Agent |
| **会话持久化** | ✅ 进程级持久 | ❌ 会话级 | OpenCode |
| **工具定制** | 受限 | 完全自由 | PI Agent |

### 3.2 技术架构对比

| 维度 | OpenCode | PI Agent | 胜出 |
|:-----|:---------|:---------|:-----|
| **架构复杂度** | 高（Go 服务 + 客户端分离） | 低（单进程嵌入式） | PI Agent |
| **依赖管理** | 外部服务进程 | npm 包直接集成 | PI Agent |
| **启动速度** | 较快（Go 编译） | 快（npm 包初始化） | 平局 |
| **内存占用** | 较高（独立进程） | 较低（进程内运行） | PI Agent |
| **进程隔离** | ✅ 完全隔离 | ❌ 共享进程 | OpenCode |
| **部署复杂度** | 高（多服务） | 低（单服务） | PI Agent |

### 3.3 开发与维护对比

| 维度 | OpenCode | PI Agent | 胜出 |
|:-----|:---------|:---------|:-----|
| **代码可读性** | 黑盒（Go 二进制） | 完全透明 | PI Agent |
| **调试能力** | 有限（日志） | 完整（源码调试） | PI Agent |
| **自定义扩展** | 受限（配置文件） | 完全自由（TypeScript） | PI Agent |
| **社区活跃度** | 高（95K stars） | 中（14K stars） | OpenCode |
| **文档完善度** | 高 | 中 | OpenCode |
| **维护稳定性** | 已迁移至 Crush | 活跃开发中 | 待观察 |

### 3.4 性能对比

| 维度 | OpenCode | PI Agent | 胜出 |
|:-----|:---------|:---------|:-----|
| **Terminal Bench** | 前十 | 前五 | PI Agent |
| **Token 效率** | 较低（10K+ prompt） | 高（<1K prompt） | PI Agent |
| **API 调用成本** | 较高（冗余上下文） | 较低（精简提示词） | PI Agent |
| **响应延迟** | 较快 | 较快 | 平局 |

### 3.5 安全对比

| 维度 | OpenCode | PI Agent | 胜出 |
|:-----|:---------|:---------|:-----|
| **文件访问控制** | 基于配置 | 可精确实现白名单 | PI Agent |
| **Shell 命令控制** | 白名单 | 完全自定义 | PI Agent |
| **审计能力** | 有限 | 完全可控 | PI Agent |
| **漏洞响应** | 依赖上游 | 自己修复 | PI Agent |

## 四、优劣势总结

### 4.1 OpenCode 优势

| 优势 | 说明 |
|:-----|:-----|
| **功能开箱即用** | LSP 集成、代码诊断、多工具支持，无需额外开发 |
| **成熟度高** | 95K+ GitHub stars，经过大量用户验证 |
| **会话持久化** | 进程常驻，断开重连可恢复 |
| **社区生态** | 丰富的文档、插件、教程 |
| **企业级特性** | SSO、审计日志、集中管理（企业版） |

### 4.2 OpenCode 劣势

| 劣势 | 说明 |
|:-----|:-----|
| **黑盒问题** | Agent 内部逻辑不可见，异常难以排查 |
| **依赖外部服务** | 需要部署 OpenCode Server（端口 4096） |
| **提示词膨胀** | 系统提示词 10K+ tokens，长期成本高 |
| **定制受限** | 工具集和行为固定，难以深度定制 |
| **维护风险** | 项目已迁移至 Crush，未来走向不确定 |
| **版本不稳定** | 作者反馈：每次发布可能破坏工作流 |

### 4.3 PI Agent 优势

| 优势 | 说明 |
|:-----|:-----|
| **架构透明** | 完全可控，每行代码可调试 |
| **Token 高效** | 提示词不到 1K，长期运营成本低 |
| **高度可扩展** | TypeScript 扩展、Skills、pi packages |
| **SDK 嵌入式** | 可嵌入应用内部，无需独立服务 |
| **树形会话** | 分支历史，不会丢失任何尝试 |
| **性能领先** | Terminal Bench 前五 |
| **Vibe Coding 友好** | 代码量小，AI 可高效理解和修改 |

### 4.4 PI Agent 劣势

| 劣势 | 说明 |
|:-----|:-----|
| **功能需自建** | LSP、诊断等能力需自行实现 |
| **社区较小** | 14K stars，生态不如 OpenCode |
| **集成工作量大** | 需要持续投入完善工具集 |
| **会话不持久化** | 断开后需要手动恢复 |
| **知名度较低** | 品牌影响力不如 OpenCode |
| **部分功能缺失** | 无内置 plan mode、to-do 系统 |

## 六、建议方案

### 6.1 战略建议：**长期以 PI Agent 为主力方案**

**核心理由**：

1. **技术可控性是长期成功的关键**
   - OpenCode 的黑盒特性会导致技术债务积累
   - 每次版本更新可能破坏现有功能
   - PI Agent 的透明架构支持持续优化

2. **成本效益显著**
   - 提示词精简 90%+，长期 API 成本大幅降低
   - 无需维护额外服务进程
   - 开发团队可自主修复问题，无需等待上游

3. **扩展性决定未来天花板**
   - PI Agent 的 SDK 模式支持深度定制
   - 工具集可精确匹配业务需求
   - 树形会话对创意探索价值巨大

4. **Vibe Coding 友好**
   - 代码量小、逻辑透明
   - AI 助手可高效理解和修改框架代码
   - 符合项目「AI 辅助开发」的本质定位

### 6.2 实施路线图

#### 阶段一：稳固 PI Agent 基础（1-2 周）

| 任务 | 优先级 | 说明 |
|:-----|:-------|:-----|
| 完善文件操作工具 | P0 | readFile, writeFile, listFiles 增强 |
| 实现 Shell 沙箱 | P0 | 命令白名单、超时控制 |
| 添加 Schema 校验钩子 | P1 | beforeToolCall, afterToolCall |
| 性能基准测试 | P1 | 对比 OpenCode 的 Token 消耗和响应质量 |

#### 阶段二：增强 PI Agent 能力（3-4 周）

| 任务 | 优先级 | 说明 |
|:-----|:-------|:-----|
| 实现 LSP 集成 | P1 | 依赖 @anthropic-ai/sdk |
| 添加代码诊断工具 | P1 | ESLint, TypeScript 错误检查 |
| 实现会话持久化 | P2 | 基于 SQLite 的会话存储 |
| 树形会话 UI | P2 | 支持分支切换和合并 |

#### 阶段三：OpenCode 作为备选（持续）

| 任务 | 优先级 | 说明 |
|:-----|:-------|:-----|
| 保留 opencode-http 后端 | P0 | 紧急回退使用 |
| 定期验证功能正常 | P2 | 版本更新后快速验证 |
| 监控性能差异 | P2 | 收集用户反馈 |

### 6.3 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|:-----|:-----|:-----|:---------|
| PI Agent 编码质量不足 | 中 | 高 | 对比测试，必要时调整提示词或切换回 OpenCode |
| PI Agent 停止维护 | 低 | 高 | 代码完全可控，可自主维护 |
| LSP 集成复杂度超预期 | 中 | 中 | 分阶段实现，优先支持核心语言 |
| 团队适应成本 | 低 | 中 | 渐进式迁移，充分测试验证 |

## 七、结论

### 7.1 最终建议

**长期主力方案选择：PI Agent**

| 维度 | 结论 |
|:-----|:-----|
| **技术路线** | 嵌入式 SDK，无外部依赖 |
| **成本控制** | Token 效率提升 90%+ |

| **维护策略** | 自主演进，不依赖上游 |
| **迁移计划** | 渐进式，保留 OpenCode 回退能力 |
