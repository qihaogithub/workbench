# 画布模式页面内容错位与缩略图截断 — 问题分析报告

## 1. 问题背景

### 问题描述

创作端画布模式下存在两个相关问题：

1. **页面内容错位**：第三个页面（赛博朋克展示页）显示了第二个页面（pad页面）的内容，而非自身真实内容。保存项目后重新打开可恢复正常。
2. **缩略图截断**：pad页面（平板尺寸 1024×768）的缩略图只显示了半截，不是完整图片。

### 发生场景

- **时间**：当前版本持续存在
- **环境**：开发环境，author-site（Next.js 14）+ screenshot-service（Puppeteer）
- **触发条件**：
  - 问题1：在画布模式下切换页面编辑时
  - 问题2：进入画布模式，pad页面的截图生成后

### 预期行为

1. 每个页面应显示自身对应的代码内容和截图
2. 所有页面的缩略图应完整显示，不被截断

### 实际行为

1. 第三个页面错误显示第二个页面的内容
2. pad页面（1024×768）缩略图被截断，只显示半截

---

## 2. 根因分析

### 调查过程

1. 追踪画布模式渲染链路：`edit/page.tsx` → `PreviewCanvas.tsx` → `CanvasPageItem.tsx`
2. 分析 `pageCodes` 状态管理机制及其与 `activeDemoId`/`code` 的联动关系
3. 检查批量截图生成流程：`useScreenshotGeneration.ts` → `screenshot-service/routes/screenshots.ts`
4. 比对 pad 页面 schema 尺寸（1024×768）与截图服务默认尺寸（375×812）

### 证据链

| 证据 | 级别 | 来源 | 说明 |
|------|------|------|------|
| `code: pageCodes[p.id] \|\| code` 使用了 fallback | A | edit/page.tsx:L1727 | 当 `pageCodes` 无对应页面时，回退到当前编辑器代码 |
| `useEffect` 依赖 `[code, activeDemoId]` 同步 pageCodes | A | edit/page.tsx:L556-563 | `activeDemoId` 变更时用旧 `code` 覆写目标页 |
| `onPageConfigEdit` 先设 activeDemoId 再异步取 code | A | edit/page.tsx:L1740-1766 | 异步 fetch 完成前存在时间窗口 |
| 批量截图未传 width/height | A | edit/page.tsx:L188-202 | 所有页面使用默认 375×812 截图 |
| pad 页面 schema 定义为 1024×768 | A | demo_...d4e5f6/config.schema.json:L4-7 | 平板尺寸，与默认截图尺寸不匹配 |
| `computeInitialLayout` 行间距用固定 `DEFAULT_PAGE_SIZE.height` | A | PreviewCanvas.tsx:L42 | 不同高度页面的布局定位可能重叠 |

### 根本原因

**问题1（页面内容错位）的根因：`pageCodes` 状态同步存在竞态条件**

在 `edit/page.tsx:556-563` 中，有一个 `useEffect` 将当前编辑器代码同步到 `pageCodes[activeDemoId]`：

```js
useEffect(() => {
  if (activeDemoId && code) {
    setPageCodes((prev) => {
      if (prev[activeDemoId] === code) return prev;
      return { ...prev, [activeDemoId]: code };
    });
  }
}, [code, activeDemoId]);
```

当用户在画布中点击第三个页面进行编辑时：
1. `onPageConfigEdit` 立即调用 `setActiveDemoId(page3Id)`（同步）
2. `useEffect` 因 `activeDemoId` 变化而触发，但此时 `code` 仍为第二个页面的代码
3. `pageCodes[page3Id]` 被覆写为第二个页面的代码
4. 随后异步 fetch 完成，`setCode(page3Code)` 才恢复正确值
5. Canvas 渲染使用 `pageCodes[p.id] || code`，因此第三个页面显示了第二个页面的内容

保存后重新加载之所以恢复正常，是因为初始化时从磁盘完整加载了所有页面代码（`edit/page.tsx:518-532`），不受竞态影响。

**问题2（缩略图截断）的根因：批量截图未传递页面尺寸参数**

在 `edit/page.tsx:188-202` 中，批量截图生成的 pages 数据未包含 `width` 和 `height`：

```js
const pages = demoPages
  .filter((p) => pageCodes[p.id] || code)
  .map((p) => ({
    pageId: p.id,
    code: pageCodes[p.id] || code,
    configData: configDataMap[p.id] || {},
    // 缺少: width 和 height
  }));
```

截图服务在 `screenshot-service/routes/screenshots.ts:250-251` 中默认使用 375×812：

```js
const w = page.width || config.viewport.width;  // 375
const h = page.height || config.viewport.height; // 812
```

pad 页面实际尺寸为 1024×768（横屏），但截图以 375×812（竖屏手机）生成，导致内容被截断。手机页面恰好与默认尺寸一致，所以不受影响。

### 代码执行路径

**问题1**：
```
用户点击画布中第三个页面
→ CanvasPageItem.handleConfigClick (CanvasPageItem.tsx:78)
→ onPageConfigEdit(page3Id) (edit/page.tsx:1740)
→ setActiveDemoId(page3Id) (同步)
→ useEffect 触发 (edit/page.tsx:556) ← code 仍为 page2
→ pageCodes[page3Id] = page2 的代码 ← 错误覆写
→ Canvas 重渲染，page3 显示 page2 内容
→ 异步 fetch 完成 → setCode(page3Code) → pageCodes 恢复正确
```

**问题2**：
```
切换到画布模式
→ useEffect 触发批量截图 (edit/page.tsx:188)
→ startBatchGeneration(pages) ← 未传 width/height
→ screenshot-service processBatch (screenshots.ts:229)
→ generateScreenshot(..., 375, 812) ← 默认手机尺寸
→ pad 页面(1024×768)被以 375×812 截图 → 内容截断
```

## 3. 解决方案

### 方案一：修复 pageCodes 竞态 + 补传截图尺寸（推荐）

- **描述**：
  1. 修改 `onPageConfigEdit`，在异步 fetch 完成前不要触发 useEffect 覆写 `pageCodes`
  2. 在批量截图生成时，从 `pagePreviewSizeMap` 读取每个页面的尺寸并传递给截图服务
- **原理**：
  1. 消除 `activeDemoId` 变更与 `code` 更新之间的时间窗口
  2. 截图服务收到正确的 width/height，生成与页面实际尺寸匹配的截图
- **影响范围**：`edit/page.tsx`（两处修改）
- **风险**：低，仅调整状态更新时序和参数传递
- **复杂度**：低

### 方案二：将 pageCodes 的同步逻辑改为仅在 code 变化时更新

- **描述**：将 `useEffect` 的依赖改为仅 `[code]`，并在回调中使用 ref 获取最新的 `activeDemoId`，避免 `activeDemoId` 变化时触发覆写
- **原理**：`pageCodes` 的更新应仅在编辑器内容变化时发生，而非 `activeDemoId` 变化时
- **影响范围**：`edit/page.tsx:556-563`
- **风险**：低
- **复杂度**：低

### 后续建议

- `PreviewCanvas.tsx:computeInitialLayout` 中行间距使用固定 `DEFAULT_PAGE_SIZE.height`（812），对于不同高度的页面（如 pad 的 768）可能导致布局重叠，建议改为动态计算每行最大高度
- 考虑为画布模式增加截图尺寸的单元测试，确保不同尺寸页面都能正确生成缩略图
