# 创作端日志系统方案

## 当前状态

已形成方案，并根据评审建议收敛实施顺序。现有编辑页诊断日志已经具备 `editorSessionId`、`traceId`、JSONL 写入和隐藏导出入口，但主要服务于单次编辑页问题排查；未来需要升级为覆盖创作端关键操作、跨服务关联、可被 Codex 直接查询的结构化日志系统。

## 目标

建立一套低维护、可长期保留、Codex 友好的创作端日志系统，让后续排查能回答这些问题：

- 用户在某个时间段打开了哪个项目、进入了哪个 Session 和 Workspace。
- 用户切换了哪些页面，当前页面的协同文本、磁盘文件和项目基准是否一致。
- 用户发送了哪次 AI 请求，AI 调用了哪些工具，改了哪些文件，改动是否进入协同房间和自动保存链路。
- 预览、编译、运行时、自动修复和自动保存在哪个环节失败。
- 重新进入项目时，首个协同快照、磁盘文件和项目基准分别来自哪里。

本方案不以“记录一切”为目标，而是把关键状态边界和错误上下文结构化，方便 Codex 用固定命令和 SQL 快速复盘。

## 当前结论

推荐采用 **SQLite 事件库作为主查询入口，JSONL 和诊断导出包作为兜底归档**。

实施顺序上，先稳定 **事件契约、旧日志映射、payload 白名单和 CLI 查询输出契约**，再落 SQLite 写入模块。原因是当前代码里的编辑页诊断事件仍使用 `timestamp`、`category`、`name`、`details` 模型，直接落库会让新旧事件长期并存，增加查询 CLI 和后续排查成本。

选择 SQLite 的原因：

- 与当前 `data/` 文件存储体系一致，不引入额外服务和运维面。
- 可以通过索引按 `projectId`、`editorSessionId`、`traceId`、`workspaceId`、`eventType` 查询。
- Codex 可以用 CLI 或只读 SQL 快速定位时间线，不必在多个 JSONL、run log 和服务日志之间手工拼接。
- WAL 模式适合 author-site、agent-service 和辅助脚本并发写入；写入失败可降级到 JSONL，不阻塞业务。

不建议直接上 Postgres、ELK、OpenTelemetry Collector 或外部 SaaS 日志平台。它们能提供更强观测能力，但本项目当前更需要低维护、本地可读、和文件数据目录一致的排查基础设施。

## 总体架构

日志系统分三层：

| 层级 | 职责 | 存储 |
|:-----|:-----|:-----|
| 事件库 | 结构化记录创作端关键操作和跨服务 trace | `data/diagnostics/editor-events.db` |
| 归档文件 | 保留单会话原始诊断事件、导出包和 AI run log 索引 | `data/editor-diagnostics/`、`data/agent-run-logs/` |
| 查询入口 | 为 Codex 和开发者提供固定命令，不要求记住内部路径 | OPS CLI 或项目诊断 CLI |

事件写入必须是 append-only。业务流程不能依赖诊断库写入成功；数据库不可用时，前端和服务端继续写 JSONL 缓冲，并在后续导出包和 CLI 输出中标记诊断库缺失、JSONL 兜底已启用或事件可能存在缺口。

## 事件模型

核心事件必须携带统一关联字段：

| 字段 | 说明 |
|:-----|:-----|
| `schemaVersion` | 事件契约版本，首版为 `1` |
| `ts` | 事件发生时间，使用 ISO 字符串和毫秒精度 |
| `source` | `frontend`、`author-api`、`agent-service`、`preview`、`ai-run`、`cli` |
| `level` | `debug`、`info`、`warn`、`error` |
| `eventGroup` | 事件组，例如 `collab`、`autosave`、`ai`、`preview`、`project`、`publish` |
| `eventType` | 稳定事件名，例如 `project.opened`、`autosave.succeeded` |
| `projectId` | 项目 ID |
| `sessionId` | 编辑 Session ID |
| `workspaceId` | 当前 Workspace ID |
| `editorSessionId` | 单次打开编辑页的浏览器会话 ID |
| `traceId` | 串联一次用户动作或系统动作 |
| `operationId` | 串联一次长事务，例如一次 AI 回复或一次发布 |
| `pageId` | 当前页面 ID，可为空 |
| `resourcePath` | 当前资源路径，可为空 |
| `message` | 面向人读的短描述 |
| `payload` | JSON 摘要，不保存大段源码或敏感内容 |

建议 SQLite 主表：

```sql
CREATE TABLE editor_events (
  id TEXT PRIMARY KEY,
  ts TEXT NOT NULL,
  schema_version INTEGER NOT NULL,
  source TEXT NOT NULL,
  level TEXT NOT NULL,
  event_group TEXT NOT NULL,
  event_type TEXT NOT NULL,
  project_id TEXT,
  session_id TEXT,
  workspace_id TEXT,
  editor_session_id TEXT,
  trace_id TEXT,
  operation_id TEXT,
  page_id TEXT,
  resource_path TEXT,
  message TEXT,
  payload_json TEXT NOT NULL
);
```

建议索引：

```sql
CREATE INDEX idx_editor_events_project_ts ON editor_events(project_id, ts);
CREATE INDEX idx_editor_events_session_ts ON editor_events(session_id, ts);
CREATE INDEX idx_editor_events_editor_session ON editor_events(editor_session_id, ts);
CREATE INDEX idx_editor_events_trace ON editor_events(trace_id, ts);
CREATE INDEX idx_editor_events_operation ON editor_events(operation_id, ts);
CREATE INDEX idx_editor_events_workspace ON editor_events(workspace_id, ts);
CREATE INDEX idx_editor_events_type_ts ON editor_events(event_type, ts);
```

可选辅助表：

| 表 | 用途 |
|:---|:-----|
| `editor_sessions` | 记录一次打开编辑页的摘要，包括开始、结束、项目、Session、Workspace、最后同步状态 |
| `operation_traces` | 记录一次 AI 回复、一次自动保存、一次发布等长事务的开始、结束和最终状态 |
| `diagnostic_artifacts` | 记录 JSONL、导出包、截图、AI run log、预览错误附件等外部文件索引 |

### 旧事件兼容映射

现有编辑页 JSONL 事件不直接废弃，而是作为 SQLite 和导出包的兼容输入：

| 旧字段 | 新字段 |
|:-------|:-------|
| `timestamp` | `ts`，转换为 ISO 字符串 |
| `category` | `eventGroup` |
| `name` | `eventType` |
| `details` | `payload`，进入统一脱敏和白名单过滤 |
| `activePageId` | `pageId` |

兼容映射必须只发生在诊断层，不反向改变业务调用方。后续新增事件优先直接按新模型生成；旧模型保留到现有编辑页诊断入口完成迁移为止。

### payload 治理

payload 默认采用事件类型白名单，而不是仅依赖敏感 key 黑名单。每个 `eventType` 需要定义允许字段集合，通用脱敏函数作为最后兜底。

默认禁止进入长期事件库的字段包括完整 `code`、`content`、`schema`、`prompt`、`response`、`diff`、`toolResult`、`authorization`、`token`、`password`、`secret`、`cookie`、上传文件内容和数据库快照。

如确需关联大对象，只保存路径、hash、长度、mtime、错误码、状态、耗时、计数和摘要。AI 工具调用事件只能保存工具名、状态、耗时、文件路径摘要、变更数量和错误摘要；完整工具入参和结果仍只存在受控的 AI run log 中，并由导出包按索引引用。

### 噪声控制

默认长期写入只记录关键状态边界：开始、成功、失败、版本变化、工作区绑定、文件变更摘要和用户触发的关键操作。

高频事件默认不进入 SQLite，或只在显式诊断模式下写入：

- `autosave.scheduled`
- `resource.hash_changed`
- `workspace_tree.updated`
- `collab.snapshot_received`
- `preview.runtime_diagnostic` 中的重复同类错误

这些事件需要按 `traceId`、`resourcePath`、`eventType` 或 `stackHash` 做合并、采样或只保留最近摘要，避免把事件库变成噪声日志。

## Codex 查询入口

建议提供固定 CLI，不要求 Codex 记住 SQL：

```bash
corepack pnpm diagnostics:recent -- --project <projectId>
corepack pnpm diagnostics:session -- --editor-session <editorSessionId>
corepack pnpm diagnostics:trace -- --trace <traceId>
corepack pnpm diagnostics:operation -- --operation <operationId>
corepack pnpm diagnostics:project -- --project <projectId> --since 24h
corepack pnpm diagnostics:export -- --project <projectId> --since 24h
```

查询输出默认 JSON，适合 Codex 继续分析。人类阅读时可加 `--format text` 输出简短时间线。

CLI 实现建议放在 `OPS/CLI` 或项目诊断 CLI 中，根目录只暴露稳定脚本别名。这样 Codex 使用固定命令，内部实现仍归工程诊断工具维护。

JSON 输出必须包含查询元信息，便于判断结果是否完整：

```json
{
  "success": true,
  "query": {
    "kind": "trace",
    "traceId": "trace-xxx",
    "since": "2026-07-02T00:00:00.000Z"
  },
  "diagnostics": {
    "sqliteUsed": true,
    "jsonlFallbackUsed": false,
    "dbUnavailable": false,
    "eventGapDetected": false,
    "warnings": []
  },
  "events": []
}
```

当 SQLite 不可用、锁定、缺失或疑似漏写时，CLI 必须尝试合并 JSONL 和 AI run log 索引，并在 `diagnostics` 中明确标记，而不是静默返回空事件。

推荐内置问题模板：

| 命令 | 回答的问题 |
|:-----|:-----------|
| `diagnostics:recent --project` | 最近一次编辑项目发生了什么 |
| `diagnostics:trace --trace` | 某次 AI 回复、自动保存或发布完整链路 |
| `diagnostics:operation --operation` | 某次长事务从开始到结束的状态、耗时和失败点 |
| `diagnostics:autosave --project` | 自动保存为什么失败或复原 |
| `diagnostics:collab --workspace` | 协同房间是否覆盖了磁盘文件 |
| `diagnostics:preview --project` | 预览失败来自编译、运行时还是自动修复 |

导出包应包含：

- 事件库中相关事件 JSON。
- 相关 JSONL 原始诊断片段。
- AI run log 索引和工具调用摘要。
- 当前 Workspace 元数据。
- 关键资源的 hash、长度、mtime。
- 预览错误摘要和截图索引。

## 隐私与保留策略

默认不保存：

- 完整页面源码。
- 完整 AI prompt、完整回复和上传文件内容。
- token、cookie、Authorization、密钥、用户密码。
- 数据库完整快照。

默认保存：

- 文件路径、hash、长度、mtime。
- 错误码、错误短消息、stack hash。
- 操作状态、耗时、资源 ID。
- 文件变更数量和 diff 统计。

保留策略：

- 默认保留 30 天。
- 每个项目可设置最大事件数或最大数据库大小。
- 导出包可按需保留更久，但必须显式生成。
- 清理任务只删除诊断数据，不触碰项目数据、Session 数据、Workspace 数据或发布数据。

## 实施阶段

### 第一阶段：统一事件契约和查询契约

- 定义 `schemaVersion`、事件字段、事件类型、事件组和 payload 白名单规则。
- 把现有编辑页 JSONL 事件映射到统一事件模型。
- 明确 `traceId` 和 `operationId` 的生成、传递和继承规则。
- 定义 CLI JSON 输出结构和 SQLite 缺失、JSONL 兜底、事件缺口的状态字段。
- 修正文档和实现中的日志路径命名，统一使用 `data/agent-run-logs/`。

验收标准：

- 编辑页打开、AI 发送、自动保存、预览错误至少能通过同一个 `editorSessionId` 串起。
- 事件 payload 不包含完整源码、密钥或完整 prompt。
- 旧 JSONL 事件能被转换为新事件模型，且不要求业务调用方一次性迁移。
- CLI 输出格式即使在无 SQLite 时也能表达诊断数据来源和缺口状态。

### 第二阶段：SQLite 事件库

- 新增 `data/diagnostics/editor-events.db`。
- author-site 提供诊断事件写入模块。
- agent-service 复用同一事件模型写入协同、工具调用和 AI run 摘要。
- 写入失败降级到 JSONL，不影响业务流程，并记录 `dbUnavailable` 或 `jsonlFallbackUsed` 摘要事件。

验收标准：

- 能按项目、编辑会话、trace、operation 查询事件。
- author-site 与 agent-service 并发写入不破坏数据库。
- 数据库缺失、只读或锁定时主业务不报错，CLI 能从 JSONL 兜底查询并给出明确 warning。
- SQLite writer 开启 WAL、busy timeout，并保证 append-only。

### 第三阶段：Codex 查询 CLI

- 增加最近项目诊断、单 trace 诊断、单 operation 诊断、自动保存诊断、协同诊断、导出包命令。
- 输出 JSON-first，文本格式只作为人工辅助。
- 查询命令写入 OPS 或项目诊断 CLI，并在相关 AGENTS 中说明。
- 查询逻辑合并 SQLite、JSONL、AI run log 索引和诊断 artifact 索引。

验收标准：

- Codex 能用一个命令拿到某项目最近一次编辑时间线。
- Codex 能用一个命令导出复现包，并看到相关 JSONL、AI run log 和资源 hash。
- 当事件库不可用时，Codex 能从输出中明确知道数据来自兜底路径，且不能误判为空事件。

### 第四阶段：真实浏览器流程复验

- 复验 AI 修复后退出再打开项目。
- 复验预览运行时错误触发自动修复。
- 复验发布前 flush 和 checkpoint。
- 复验协同房间外部文件变更命中与未命中。

验收标准：

- 每个流程都能从事件库中还原用户操作、服务端动作和最终状态。
- 排查“AI 已修复但重新打开复原”时，不需要手工翻多个目录。
- 至少覆盖一次 SQLite 不可写或锁定场景，确认业务流程不中断且导出包能标记诊断降级。

## 风险

- 事件过多会产生噪声，必须默认只记录关键状态边界，详细内容放到显式诊断模式。
- payload 若没有按事件类型做白名单，可能把源码、prompt、工具结果或密钥写入长期存储。
- 多进程写 SQLite 需要 WAL、busy timeout 和失败降级，否则诊断系统可能反过来影响编辑体验。
- 如果只建数据库但不提供 Codex CLI，排查成本仍会转移到手写 SQL 上，收益会明显降低。
- 如果 CLI 不显式标记 SQLite 缺失、JSONL 兜底和事件缺口，Codex 可能把“不完整诊断”误判为“没有问题”。
