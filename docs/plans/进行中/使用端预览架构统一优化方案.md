# 使用端预览架构统一优化方案

> 版本：v1.0
> 创建日期：2026-05-10
> 状态：进行中
> 上游方案：[使用端方案.md](./使用端方案.md)、[预览双模式开发文档.md](./预览双模式开发文档.md)

---

## 一、背景

### 1.1 现状问题

viewer-site（使用端）与 author-site（创作端）各自维护了一套预览区 + 配置面板代码，存在以下问题：

**1. 代码重复严重**

| 维度 | viewer-site | author-site | 关系 |
|------|-------------|-------------|------|
| 预览核心组件 | 964 行 | 3248 行 | author-site 是超集 |
| shadcn/ui 副本 | 488 行 | (已有) | 完全重复 |
| **总重复代码** | **~1452 行** | — | viewer-site 中约 2/3 代码是 author-site 的子集 |

具体重复点：

| viewer-site 文件 | author-site 对应 | 重复关系 |
|-----------------|-----------------|---------|
| `iframe-renderer.tsx` (156行) | `PreviewPanel.tsx` (447行) | postMessage 通信机制相同，缩放逻辑相似 |
| `viewer-preview-grid.tsx` (239行) | `PreviewGrid.tsx` (474行) | `useVisiblePages` 几乎一字不差 |
| `config-panel.tsx` (479行) | `ConfigFormNew.tsx` (1035行) | 只读子集，缺少分组/上传/排序 |
| 10 个 shadcn/ui 组件 (488行) | `src/components/ui/` | 标准组件复制 |

**2. 效果不一致**

两套代码在以下维度存在差异，导致同一组件在两个站点上预览效果不同：

- 宫格列宽分配：author-site 按宽高比分配，viewer-site 等宽分配
- 宫格模式配置传递：viewer-site 未传递 `configData`，配置变更不生效
- 缩放逻辑：`transformOrigin`、`minHeight`/`maxHeight` 支持、尺寸解析方式不同
- 配置面板功能：viewer-site 缺少智能分组、文件上传、图片列表、拖拽排序
- iframe 滚动条、对齐模式、错误处理等细节差异

**3. 维护成本持续累积**

每次 author-site 改了预览逻辑或配置面板，都需要手动同步到 viewer-site。长期来看，两套代码的差距只会越来越大。

### 1.2 根因

viewer-site 的 iframe **内部**渲染逻辑（编译、React 运行时、ErrorBoundary、postMessage 协议）已经在 author-site 维护，但 iframe **外部**的包裹层（缩放、消息通信、配置面板、模式切换）却在 viewer-site 又维护了一套。

```
当前架构的问题：

viewer-site                          author-site
┌──────────────────────┐            ┌─────────────────────┐
│ IframeRenderer       │  iframe    │ /api/embed/.../iframe│
│ (自己维护的缩放逻辑)  │──src─────▶│ (服务端编译+生成HTML) │
│ ConfigPanel          │            │                     │
│ (自己维护的表单渲染)  │  postMsg   │ iframe 内的渲染逻辑  │
│ ViewerPreviewGrid    │───────────▶│ (已在 author-site)   │
│ (自己维护的宫格逻辑)  │            │                     │
└──────────────────────┘            └─────────────────────┘
     ↑ 重复维护了 1452 行                  ↑ 已有 3248 行
```

---

## 二、目标

1. **消除代码重复** — viewer-site 不再维护预览区 + 配置面板代码，删除 ~1400 行重复代码
2. **保证效果一致** — 同一套代码渲染，不存在 viewer-site 与 author-site 的显示差异
3. **降低维护成本** — 预览逻辑和配置面板只在 author-site 一处维护
4. **提供灵活配置** — 通过 URL 参数让调用方按需选择显示哪些 UI 元素
5. **向后兼容** — 现有的裸 HTML embed API 保持不变

---

## 三、方案

### 3.1 核心思路

**author-site 新增 `/viewer/[projectId]` 完整嵌入页路由，将预览区 + 配置面板 + 模式切换打包为一个自包含页面，viewer-site 通过一个 iframe 整体嵌入。**

```
目标架构：

viewer-site                          author-site
┌──────────────────────┐            ┌─────────────────────────────┐
│ 项目列表页（自己维护）│            │ /viewer/:projectId?...      │
│                      │  iframe    │ ┌─────────────────────────┐ │
│ 项目预览页           │──src─────▶│ │ 预览区 + 配置面板 +      │ │
│ (仅一个 iframe)      │            │ │ 模式切换 + 工具栏       │ │
│                      │           │ │ 全部由 author-site 渲染  │ │
└──────────────────────┘            │ └─────────────────────────┘ │
     ↑ 仅 ~50 行                         ↑ 复用已有组件          │
                                    └─────────────────────────────┘
```

### 3.2 三层嵌入 API

author-site 提供三个层级的嵌入能力，满足不同场景需求：

| 层级 | 路由 | 返回内容 | 适用场景 |
|------|------|---------|---------|
| **层级 1**（现有） | `/api/embed/:projectId/iframe` | 裸组件 HTML | 需要完全自定义 UI 的调用方 |
| **层级 2**（新增） | `/viewer/:projectId?...` | 完整预览页 | 大多数嵌入场景 |
| **层级 3**（现有） | `/embed/:demoId` | 嵌入引导页 | 面向人工操作 |

### 3.3 URL 配置参数

层级 2 的 `/viewer/:projectId` 支持以下 URL 参数：

#### 基础参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `page` | string | 第一个页面 | 初始显示的页面 ID |
| `mode` | `single` \| `grid` | `single` | 预览模式 |
| `columns` | `2` \| `3` \| `4` | `2` | 宫格列数（仅 mode=grid 时生效） |

#### UI 控制参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `config` | boolean | `true` | 是否显示配置面板 |
| `configWidth` | number | `320` | 配置面板宽度(px) |
| `pages` | boolean | `true`(多页面) / `false`(单页面) | 是否显示页面选择器/目录 |
| `toolbar` | boolean | `true` | 是否显示顶部工具栏 |
| `modeSwitch` | boolean | `true` | 是否显示模式切换按钮 |

#### 主题参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `theme` | `light` \| `dark` \| `auto` | `dark` | UI 主题 |
| `background` | string | `#fff` | 预览区背景色 |

#### 配置注入参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `configData` | JSON string | — | 初始配置数据，覆盖 Schema 默认值 |

#### 参数优先级

```
URL 参数 > 项目 Schema 中的 $demo.viewerDefaults > 系统默认值
```

项目可在 Schema 中预设默认嵌入配置：

```json
{
  "$demo": {
    "viewerDefaults": {
      "mode": "grid",
      "columns": 3,
      "config": true,
      "theme": "dark"
    }
  }
}
```

### 3.4 使用示例

```html
<!-- 最简嵌入：单页预览 + 配置面板 -->
<iframe src="http://localhost:3200/viewer/my-project" />

<!-- 宫格模式，3列，无配置面板 -->
<iframe src="http://localhost:3200/viewer/my-project?mode=grid&columns=3&config=false" />

<!-- 指定页面，纯预览（无工具栏无配置） -->
<iframe src="http://localhost:3200/viewer/my-project?page=detail&toolbar=false&config=false" />

<!-- 暗色主题，自定义配置面板宽度 -->
<iframe src="http://localhost:3200/viewer/my-project?theme=dark&configWidth=400" />

<!-- 注入初始配置值 -->
<iframe src='http://localhost:3200/viewer/my-project?configData={"title":"自定义标题","count":5}' />
```

### 3.5 postMessage 运行时控制

嵌入页支持 `postMessage` 通信，供调用方在运行时动态控制：

```typescript
// 调用方 → 嵌入页
iframe.contentWindow.postMessage({ type: "VIEWER_SET_CONFIG", configData: { title: "新标题" } }, "*");
iframe.contentWindow.postMessage({ type: "VIEWER_SET_MODE", mode: "grid" }, "*");
iframe.contentWindow.postMessage({ type: "VIEWER_SET_PAGE", pageId: "detail" }, "*");

// 嵌入页 → 调用方
window.addEventListener("message", (event) => {
  if (event.data.type === "VIEWER_READY") { /* 嵌入页就绪 */ }
  if (event.data.type === "VIEWER_CONFIG_CHANGE") { /* 配置变更 */ }
  if (event.data.type === "VIEWER_PAGE_CHANGE") { /* 页面切换 */ }
});
```

### 3.6 viewer-site 简化后的代码

项目预览页变为极简实现：

```tsx
// viewer-site/src/app/[projectId]/page.tsx
export default function ProjectPreviewPage() {
  const { projectId } = useParams();
  const viewerUrl = `${WEB_URL}/viewer/${projectId}?mode=grid&config=true&pages=true`;

  return (
    <iframe
      src={viewerUrl}
      className="w-full h-full border-none"
      sandbox="allow-scripts allow-same-origin"
    />
  );
}
```

```tsx
// viewer-site/src/app/[projectId]/[demoId]/page.tsx
export default function DemoPreviewPage() {

  return (
    <iframe
      src={viewerUrl}
      className="w-full h-full border-none"
      sandbox="allow-scripts allow-same-origin"
    />
  );
}
```

## 四、实施步骤

### 步骤 1：author-site 新增 `/viewer/[projectId]` 页面

**新增文件**：`packages/author-site/src/app/viewer/[projectId]/page.tsx`

这是一个客户端组件（`"use client"`），包含：

- 解析 URL 参数（mode、config、pages、toolbar、theme 等）
- 通过 author-site 自己的 API 获取项目数据、Demo 页面列表、Schema
- 渲染顶部工具栏（项目名、模式切换、页面选择器、列数选择、配置面板开关）
- 渲染预览区（复用 `PreviewPanel` + `PreviewGrid`）
- 渲染配置面板（复用 `ConfigFormNew`）
- 根据 URL 参数控制各 UI 元素的显示/隐藏

核心逻辑：将 [DemoEditPage](file:///e:/重要文件/Programming/1_Work/opencode工作台/packages/author-site/src/app/demo/[id]/edit/page.tsx) 的右半部分（预览区 + 配置面板）独立出来，去掉 AI 对话和代码编辑功能。

**预估代码量**：~250 行

### 步骤 2：author-site 新增 `/viewer/[projectId]/[demoId]` 页面

**新增文件**：`packages/author-site/src/app/viewer/[projectId]/[demoId]/page.tsx`

简化版，直接定位到特定 Demo 的单页预览。

**预估代码量**：~80 行

### 步骤 3：author-site middleware 添加 `/viewer/` 路径支持

**修改文件**：`packages/author-site/src/middleware.ts`

在 `isApiOrEmbedRoute` 条件中添加 `/viewer/` 路径，使其享有与 `/embed/` 相同的 CORS 和免认证待遇：

```typescript
const isApiOrEmbedRoute =
  pathname.startsWith('/api/') ||
  pathname.startsWith('/embed/') ||
  pathname.startsWith('/viewer/');
```

同时将 `/viewer/` 排除在认证保护路由之外（与 `/embed/` 同等处理）。

**预估代码量**：修改 ~3 行

### 步骤 4：viewer-site 简化项目预览页

**修改文件**：

- `packages/viewer-site/src/app/[projectId]/page.tsx` — 简化为 iframe 嵌入
- `packages/viewer-site/src/app/[projectId]/[demoId]/page.tsx` — 简化为 iframe 嵌入

**预估代码量**：每个文件 ~30 行

### 步骤 5：删除 viewer-site 中不再需要的组件

**删除文件**：

- `src/components/iframe-renderer.tsx`
- `src/components/viewer-preview-grid.tsx`
- `src/components/config-panel.tsx`
- `src/components/ui/` 下 10 个不再需要的组件文件

**修改文件**：

- `src/lib/api.ts` — 删除 `getEmbedIframeUrl`、`getDemoPages`、`getProjectConfig` 等不再需要的函数，删除 `DemoPageWithSchema`、`PreviewSize` 等类型
- `package.json` — 删除不再需要的 Radix UI 依赖（slider、switch、select、dialog、tooltip、scroll-area、label）

### 步骤 6：验证

- 启动 author-site + viewer-site 开发服务器
- 验证 viewer-site 项目列表页正常

## 五、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 嵌套 iframe 性能 | 低 — viewer-site 的 iframe 直接加载 author-site 预览页，预览页内部用 Blob URL 渲染组件，不存在嵌套 iframe | 无需特殊处理 |
| 跨域限制 | 低 — author-site middleware 已配置 CORS 允许 viewer-site 域名 | 已有基础设施 |
| 部署耦合 | 中 — viewer-site 依赖 author-site 的 /viewer 路由可用 | 这已经是现状（embed API 依赖），不增加新耦合 |
| viewer-site 失去 UI 定制能力 | 低 — 预览区本就应与 author-site 一致 | 顶部导航栏仍由 viewer-site 自己控制；URL 参数提供足够的灵活性 |
| URL 参数安全性 | 低 — configData 参数需防止 XSS | 对 configData 做 JSON.parse 校验，拒绝非法输入 |
