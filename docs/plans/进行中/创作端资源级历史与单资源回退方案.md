# 创作端内容版本图与单资源回退实施方案

## 当前状态

待实施。

本方案用于把创作端历史能力从“项目快照 + 页面快照”的局部机制，升级为可长期扩展的内容版本图。第一轮实施目标是让画布中的页面和知识文档拥有独立历史，并支持单独回退；架构上为后续画布布局、资产、项目配置和发布追溯统一建模。

## 一句话结论

项目历史是不可变提交日志；提交指向一组稳定资源 ID 的不可变版本；工作区只是基于某个提交的草稿；文件目录只是从提交物化出来的缓存。

## 目标

- 页面、知识文档拥有独立对象历史，可从画布或知识库面板单独回退。
- 单资源回退创建新提交，不改写旧历史，也不影响无关资源。
- 项目发布、整体恢复、命名版本和审计统一绑定 `commitId`。
- `project-core` 承载核心语义，author-site、CLI、自动化入口复用同一套能力。
- 兼容现有 `project.json`、`pageVersions`、`workspace/`、`knowledge/manifest.json` 和 `.canvas-layout.json`，通过迁移层逐步切换。

## 非目标

- 首轮不实现多人分支合并 UI。
- 首轮不强制迁移全部历史数据。
- 首轮不删除旧页面版本 API 和 CLI 命令。
- 首轮不实现资产历史和画布布局历史的完整 UI，但数据模型必须预留。
- 首轮不把自动保存的每个 checkpoint 都暴露到普通用户历史列表。

## 核心决策

1. 稳定 ID 优先：页面、知识文档、画布、资产都以稳定 `resourceId` 为身份；标题、路径、文件名只是版本元数据。
2. 删除用 tombstone：删除资源是提交，不是立即清理文件；被删除资源可从历史恢复。
3. 回退用新提交：回退只是把当前提交中的某个资源指针拨回旧资源版本。
4. 提交是事务边界：`headCommitId` 原子推进；物化目录写失败只标记物化失败，不回滚提交。
5. 物化目录是缓存：`workspace/` 继续服务预览、发布、CLI 和旧代码，但不再是唯一事实源。
6. 引用图必须可查询：画布引用页面/文档，页面引用资产，删除和回退前必须做依赖预检。
7. 历史分层展示：普通用户默认看 semantic/protected 历史；draft checkpoint 只在高级恢复或诊断中展示。
8. 运行时身份入史：资源版本记录 schema、runtime、preview-contract、materializer 版本，避免旧历史无法解释。
9. 保留按引用保护：发布、命名版本、模板来源和用户保护版本不能被数量策略清理。

## 术语

| 术语 | 含义 |
| --- | --- |
| Resource | 项目内稳定对象，例如页面、知识文档、画布、资产、项目配置 |
| ResourceVersion | 某个资源的不可变内容版本 |
| ProjectCommit | 一次用户可理解的项目变化，指向一组资源版本 |
| ProjectState | 当前项目 head commit 对应的资源指针集合 |
| DraftChange | 工作区中尚未形成提交的草稿变更 |
| Materialized Workspace | 从 commit 导出的文件目录，用于兼容运行时 |
| Tombstone | 删除标记，表示资源在当前 head 中不可见但历史仍可恢复 |

## 目标数据模型

### ProjectResourceKind

```ts
export type ProjectResourceKind =
  | "page"
  | "knowledge_document"
  | "canvas"
  | "asset"
  | "project_config";
```

首轮实现 `page` 和 `knowledge_document`，但类型和存储结构预留其他资源。

### ResourcePointer

```ts
export interface ResourcePointer {
  kind: ProjectResourceKind;
  resourceId: string;
  versionId?: string;
  deleted?: boolean;
}
```

说明：

- `resourceId` 永远稳定。
- `deleted: true` 表示当前 head 中不可见。
- 恢复被删资源时生成新 commit，把 `deleted` 移除并指向目标版本。

### ResourceVersion

```ts
export interface ResourceVersion {
  id: string;
  projectId: string;
  kind: ProjectResourceKind;
  resourceId: string;
  previousVersionId?: string;
  restoredFromVersionId?: string;
  contentHash: string;
  blobRefs: string[];
  metadata: Record<string, unknown>;
  runtime: {
    schemaVersion: number;
    runtimeType?: string;
    previewContractVersion?: string;
    materializerVersion: string;
    migrationStatus?: "native" | "migrated" | "legacy";
  };
  createdAt: number;
  createdBy: string;
  source: "user" | "ai" | "import" | "restore" | "publish" | "system";
  note?: string;
}
```

### ProjectCommit

```ts
export interface ProjectCommit {
  id: string;
  projectId: string;
  parentCommitId?: string;
  visibility: "draft_checkpoint" | "semantic" | "protected";
  intent: "edit" | "checkpoint" | "restore" | "publish" | "import" | "ai" | "system";
  title: string;
  resourcePointers: ResourcePointer[];
  changedResources: Array<{
    kind: ProjectResourceKind;
    resourceId: string;
    fromVersionId?: string;
    toVersionId?: string;
    deleted?: boolean;
  }>;
  createdAt: number;
  createdBy: string;
  audit: {
    actorType: "user" | "ai" | "system" | "cli";
    sessionId?: string;
    workspaceId?: string;
    bypassedValidation?: boolean;
  };
}
```

### ProjectContentState

```ts
export interface ProjectContentState {
  projectId: string;
  headCommitId: string;
  materializationStatus?: "ready" | "pending" | "failed";
  materializedCommitId?: string;
  updatedAt: number;
}
```

### ResourceReference

```ts
export interface ResourceReference {
  from: { kind: ProjectResourceKind; resourceId: string };
  to: { kind: ProjectResourceKind; resourceId: string };
  reason: "canvas_node" | "asset_usage" | "config_route" | "knowledge_context";
}
```

引用图首轮至少覆盖：

- `canvas:main -> page:*`
- `canvas:main -> knowledge_document:*`
- `page:* -> asset:*`，如果当前页面资产解析能力不足，先提供 best-effort。

## 存储设计

### 新内容图目录

```text
projects/{projectId}/content/
  state.json
  commits/
    {commitId}.json
  resources/
    page/{resourceId}/{versionId}.json
    knowledge_document/{resourceId}/{versionId}.json
    canvas/{resourceId}/{versionId}.json
    asset/{resourceId}/{versionId}.json
    project_config/{resourceId}/{versionId}.json
  blobs/
    {hashPrefix}/{hash}
  refs/
    references.json
  materialization/
    manifest.json
```

### 旧目录兼容

```text
projects/{projectId}/workspace/
projects/{projectId}/project.json
projects/{projectId}/workspace/knowledge/manifest.json
projects/{projectId}/workspace/.canvas-layout.json
```

兼容规则：

- 读：优先读 `content/state.json`；不存在时从旧目录构建 legacy state。
- 写：新增历史和回退只写内容图，再物化到旧目录。
- 旧 `pageVersions` 只读兼容；页面旧 API 调用新资源历史服务。
- `workspace/` 是物化结果；检测到外部文件改动时进入 import/reconcile 流程，不直接推进 head。

## 资源内容闭包

### 页面资源

高保真 React 页面版本包含：

- `index.tsx`
- `config.schema.json`
- 页面名称、排序、父目录、运行时类型、预览尺寸等元数据

HTML/CSS 原型页面版本包含：

- `prototype.html`
- `prototype.css`
- `prototype.meta.json`
- `config.schema.json`
- 页面名称、排序、父目录、运行时类型、预览尺寸等元数据

### 知识文档资源

知识文档版本包含：

- Markdown 正文 blob
- 标题、描述、分类、标签、摘要状态等元数据
- 旧 `fileName` 仅作为兼容元数据，不作为身份

长期物化形态应改为：

```text
workspace/knowledge/{docId}.md
workspace/knowledge/manifest.json
```

迁移阶段可继续物化为旧 `fileName`，但内容图中必须以 `docId` 为准。

### 画布资源

首轮不实现画布历史 UI，但需要预留 `canvas:main`：

- 页面布局
- 文档节点与聚合目录
- 隐藏文档列表
- 自由标注
- 视口状态

画布只引用页面和文档的 `resourceId`，不保存正文副本。

## 核心流程

### 创建提交

1. 校验 actor 权限。
2. 读取当前 `headCommitId` 和资源指针。
3. flush 相关协同房间。
4. 将 draft change 转换为 ResourceVersion。
5. 写入 blob 和 ResourceVersion。
6. 写入 ProjectCommit。
7. 原子推进 `content/state.json.headCommitId`。
8. 物化 `workspace/`。
9. 物化失败时写入 `materializationStatus=failed` 并返回可重试错误。

### 页面回退

1. 用户在页面对象历史选择目标版本。
2. 服务端加载当前 head 和目标 ResourceVersion。
3. 校验目标页面运行时和 Schema。
4. 检查目标页面引用资产是否存在；缺失时提示或阻断。
5. flush 该页面协同房间。
6. 生成回退前 draft checkpoint，默认隐藏。
7. 创建 semantic restore commit，只更新该页面指针。
8. 物化 workspace 中该页面文件和相关 manifest。
9. 刷新预览、截图缓存和页面历史列表。

### 知识文档回退

1. 用户在文档对象历史选择目标版本。
2. 服务端确认文档不是系统只读文档。
3. 检查画布、AI 上下文、模板引用关系。
4. flush `knowledge/{docId}` 或旧 `knowledge/{fileName}` 协同房间。
5. 生成回退前 draft checkpoint，默认隐藏。
6. 创建 semantic restore commit，只更新该文档指针。
7. 物化 Markdown 正文和 manifest。
8. 刷新知识库面板、画布文档节点和聚合目录项。

### 删除资源

1. 依赖预检。
2. 对仍被画布引用的文档，默认执行“从画布隐藏 + 文档进入回收站”。
3. 创建 semantic delete commit，设置 `deleted: true`。
4. 物化时从可见列表移除，但保留内容图历史。
5. 垃圾回收只清理无 commit、无发布、无命名版本引用的 blob。

### 发布

1. 发布前 flush 当前 workspace。
2. 必要时创建 protected publish commit。
3. 发布记录绑定 `commitId`。
4. 发布产物记录资源 hash 清单。
5. 后续可从发布记录进入“恢复此页面到发布时版本”或“恢复整个项目到发布时提交”。

## UI 设计

### 对象历史面板

入口：

- 页面卡片右键或工具栏：`历史`
- 文档卡片右键或工具栏：`历史`
- 聚合文档目录项菜单：`查看此文档历史`
- 知识库面板文档操作菜单：`历史`

展示：

- 版本时间、创建者、来源、备注、是否发布/命名保护
- 默认隐藏 draft checkpoint
- 支持按来源筛选：用户、AI、导入、恢复、发布

恢复确认：

- 页面按钮文案：`恢复此页面`
- 文档按钮文案：`恢复此文档内容`
- 画布按钮文案：`恢复画布布局`
- 项目按钮文案：`恢复整个项目`

### 差异预览

| 资源 | 预览要求 |
| --- | --- |
| 页面 | 视觉预览、代码 diff、配置 diff、运行时校验结果 |
| 文档 | Markdown 正文 diff、标题/描述变化 |
| 画布 | 缩略布局、节点移动/隐藏/合并变化 |
| 资产 | 缩略图、尺寸、hash、引用方 |
| 项目 | 按资源类型分组的变更摘要 |

### 项目时间线

项目时间线展示 ProjectCommit，而不是资源版本。适合回答：

- AI 刚才改了哪些对象？
- 昨天发布前项目是什么状态？
- 哪次导入删除了某些页面？
- 是否只恢复其中一个页面？

| 聚合文档恢复 | 只影响目录项对应文档 |
| 删除恢复 | tombstone 可恢复 |
| 引用预检 | 页面引用缺失资产时提示或阻断 |
| 发布绑定 | 发布记录包含 commitId 和资源 hash |
| 物化失败 | commit 已存在，状态 failed，可重试 |
| 外部改文件 | 不直接推进 head，进入 import/reconcile |
| checkpoint 噪声 | 普通历史默认不展示 draft checkpoint |

## 验收标准

- 页面、知识文档都有对象历史入口。
- 单资源恢复不会修改无关资源。
- 恢复操作生成新的 semantic restore commit。
- 历史记录能区分用户、AI、导入、恢复、发布来源。
- 发布记录绑定 commitId。
- `workspace/` 可从 head commit 重建。
- 旧页面版本 API 和 CLI 无回归。
- 项目重新打开后当前态与 head commit 一致。

## 风险与处理

| 风险 | 处理 |
| --- | --- |
| 内容图和旧目录双写不一致 | 单写内容图，旧目录只物化 |
| 历史列表过噪 | commit visibility 分层，默认隐藏 draft checkpoint |
| 文档重命名破坏引用 | 以 docId 为资源 ID，fileName 只做元数据 |
| 删除资源后无法恢复 | tombstone + protected 引用保护 |
| 旧版本无法当前运行时预览 | 记录 runtime/materializer version，并提供只读查看 |
| 物化失败导致半成功 | commit 原子成功，物化状态 failed 可重试 |
| 实施范围过大 | 按 Phase 1-6 分层，先页面兼容，再文档历史，再 UI |
