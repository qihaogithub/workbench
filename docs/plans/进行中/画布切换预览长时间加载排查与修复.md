# 画布切换预览长时间加载排查与修复

## 背景

用户反馈在项目编辑页中，单页面模式已经能看到预览效果，但切换到画布模式后，画布里的页面卡片长时间停留在加载中。

## 目标

- 切换到画布时，页面卡片能稳定拿到对应页面代码并渲染预览。
- 代码未就绪时不要表现为无限加载，需要有明确的等待或兜底行为。
- 保持单页预览、画布截图生成、画布交互模式的现有行为不被破坏。

## 范围

- 创作端项目编辑页的画布预览入口。
- `@opencode-workbench/demo-ui` 中预览 iframe 的加载态。
- 与本问题直接相关的文档说明和测试。

## 方案

1. 排查单页预览与画布预览传入 `PreviewPanel` 的代码来源差异。
2. 在切换画布时补齐缺失页面代码，避免画布页面收到空代码。
3. 为预览 iframe 增加空代码兜底，避免空代码状态显示为永久加载。
4. 让预览请求生命周期只受代码、模块 URL、session 或页面身份等预览源变化驱动，父级回调重建不能重启已加载预览。
5. 增加或更新最小验证覆盖。

## 进度记录

- 2026-06-30：确认截图中的两个页面数量低于截图渲染阈值，画布应直接使用 iframe；长期加载更可能来自画布传入空代码或 iframe `LOADED` 未回传。
- 2026-06-30：发现 `resolvePreviewPageCode` 会在非当前页未加载时返回空字符串；`PreviewPanel` 对空代码会保持 `contentLoaded=false`，导致加载层持续显示。
- 2026-06-30：已在进入画布时按页面 ID 补齐缺失代码、Schema、默认配置和预览尺寸；`PreviewPanel` 在空代码状态显示“等待页面代码加载”，避免无限 spinner。
- 2026-06-30：确认 `App通用广告弹窗` 项目工作区内两个页面的 `index.tsx` 均有代码；问题不在项目文件丢失，而在编辑页内存态的页面代码缓存需要更稳。初始加载时已改为缓存服务端返回的每个页面代码字段，不再用 truthy 判断过滤。
- 2026-06-30：定位到根因：`PreviewPanel` 在 effect 开始时就推进 requestId，即使随后发现当前没有可发送的代码或模块 URL 也会 return。这会让画布切换期间短暂的空代码过渡占用新请求版本，导致 iframe 针对上一份真实代码返回的 `LOADED` 被父组件判定为过期并丢弃。修复为只有真正会发送或编译预览源时才创建新 request；无预览源时进入 no-request 状态。
- 2026-06-30：用户使用 `proj_1779608460377` 复测后仍出现“先看到预览，后又一直加载中”。再次核查该项目工作区，`pad-popup_a3f2/index.tsx` 和 `phone-popup_k8m2/index.tsx` 均非空，进一步排除项目代码文件未加载。
- 2026-06-30：最终根因收敛为 `PreviewPanel` 的编译 effect 依赖父级回调身份。编辑页中 `handlePreviewError` 会随单页/画布焦点变化重建，`reportTiming` 也会随控制台回调身份变化重建；这些变化并不是代码源变化，却会让 `PreviewPanel` 重新执行编译 effect、把 `contentLoaded` 置回 `false` 并创建新 requestId。旧 iframe 针对已显示内容返回的消息随后被版本过滤，页面就表现为已显示后又回到加载中。
- 2026-06-30：修复为通过 ref 读取最新错误回调和控制台回调，让编译 effect 只响应实际预览源变化；新增“父级回调重建不应重启已加载的预览请求”回归测试。
- 2026-06-30：目标测试和 author-site 类型检查通过；根目录 `pnpm` 命令因当前 runtime 使用 pnpm 11 与仓库 pnpm 8 lockfile 不兼容，未继续让其清理依赖目录。
