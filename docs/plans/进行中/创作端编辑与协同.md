## AI 编辑自动保存重新打开复原

状态：已完成代码级修复，等待真实浏览器流程复验。

当前结论：问题不是自动保存完全没有触发。诊断日志已出现 `autosave.flush_succeeded` 与 `exit_flush_succeeded`，但同一项目重新打开后仍会收到旧的 `snapshot.apply source=collab`。根因收敛到 agent-service 的活跃协同房间：AI 工具写盘后通知 `applyExternalFileChanges` 重载房间，但文件变更有时以当前页相对路径 `index.tsx` / `config.schema.json` 上报；服务端只按 `demos/<pageId>/index.tsx` 这类完整 workspace 路径精确匹配，导致活跃 Y.Doc 未重载。房间在默认空闲 TTL 内复用时，旧协同文本会再次成为前端权威值，后续自动保存又会把同一个重复声明问题带回。

已落地边界：checkpoint 成功前同步 Session Workspace 到项目当前 workspace；普通 workspace 持久化接口覆盖“看到已自动保存后立刻返回首页”的路径；前端 AI 文件变更后只刷新当前页状态、refs 和预览缓存，并显式标记 workspace 待 flush/persist，不再主动把同一份完整文本写入 Yjs；前端当前页相对路径会先归一成 workspace 内页面路径；agent-service 协同层记录文件哈希基线，AI 工具写入后通知活跃房间重载，flush 前发现磁盘已脱离房间基线时以磁盘文件重置房间并拒绝旧文本写回。服务端外部文件变更现在会在完整路径未命中、且 workspace 根部不存在同名文件时，把 `index.tsx` / `config.schema.json` 回退匹配到活跃页面协同房间；根目录同名文件存在时不回退，避免误刷嵌套页面。

2026-07-02 重新定位结论：截图中的 `phoneAssetBase` 不是旧预览缓存误报。`proj_1779608460371` 的项目基准工作区当前仍能被 `project validate-runtime` 检出重复顶层声明；同一时间 active live Workspace 中页面代码已被 AI 修成单份。诊断时间线显示：13:01 左右 AI 修复后代码为单份并成功 flush；13:06 重新打开时第一个 `collab snapshot.apply source=collab` 已经变成双份；13:09 AI 再次修成单份后，自动保存两次失败于 `SESSION_EXPIRED`，因此项目基准工作区没有被推进，重新打开仍可能读到坏基准。

根因拆分：

- 协同层根因：服务端新建 Yjs 房间时把磁盘文件作为初始 `Y.Text` 插入；若客户端本地 Y.Doc 已持有同一份文本，再与新服务端房间同步，Yjs 会把两边的同文本视作两组独立插入并合并为两份，后续可继续倍增为 4 份。
- 持久化根因：编辑页长时间打开后 Session 超过默认 2 小时有效期，前端仍能收到 AI 文件变更并刷新 live Workspace，但 agent-service 在 flush 时拒绝过期 Session，项目基准工作区没有同步到修复后的 live Workspace。

已落地补救：agent-service 协同房间在非服务端更新后，如果发现房间文本等于当前磁盘文件的 2 倍或更多倍完整重复，会立即以磁盘文件重置房间并取消保存；如果重复内容是完整 JSON Schema，会归一化为第一份可解析 JSON 后再落盘，避免坏 Schema 继续阻塞编辑页加载；author-site 的 flush coordinator 在调用 agent-service 前会续期仍处于编辑状态的 Session，避免长时间编辑后修复无法同步项目基准。此前的自动修复诊断指纹限流仍保留，但它只是避免同一错误无限触发 AI，不是本次重复拼接的根因。

2026-07-02 补充：`proj_1779608460371` 的重复 JSON 文件批量修复后，编辑页仍可能停在“选择页面或文档 / 等待页面代码加载”。现场验证显示 `/api/sessions/:sessionId/files` 已能返回页面，磁盘 `workspace-tree.json` 也正常；残留问题来自开发环境里活跃协同房间与前端初始化时序。重启 agent-service 可清空旧 Yjs 房间，服务端会在房间重新同步时归一化重复 JSON；前端已增加兜底：当 session 加载完成、页面列表非空但 `activeDemoId` 仍为空时，自动选择排序第一的页面并加载其 code/schema，避免协同状态先连接成功但页面未选中时卡死。

待办：用真实浏览器流程复验 AI 修复后退出、重新打开同一项目，首个 `snapshot.apply source=collab` 应是修复后的短代码或单份 Schema，而不是旧重复拼接版本；同时确认打开超过 2 小时后自动保存不再因 `SESSION_EXPIRED` 失败。

验证状态：已有定向测试覆盖完整路径命中时会重载旧协同房间、当前页相对路径会回退命中嵌套页面协同房间、根目录同名文件存在时不会误命中嵌套页面、flush 前会拒绝旧协同文本覆盖新文件、重复 Yjs 基线合并会重置为单份内容、重复 JSON Schema 会归一化为单份内容再落盘。`corepack pnpm --filter @opencode-workbench/agent-service test -- tests/unit/collab-room-manager.test.ts` 通过，6 tests；`corepack pnpm --filter @opencode-workbench/agent-service typecheck` 通过；`corepack pnpm --filter @opencode-workbench/author-site typecheck` 通过。当前数据现场中项目级 Schema 与手机版页面 Schema 已由重复 JSON 块恢复为单份可解析 JSON；`node packages/project-cli/bin/ow.mjs project validate-runtime proj_1779608460371 --json` 返回 `ok: true`。真实 Chrome 复验中，“伴读广告页面”编辑页已恢复显示页面“手机”，预览 iframe 能渲染图片内容，且不再出现“加载失败”或“等待页面代码加载”。

## 页面版本恢复后同步失败

状态：已完成代码级修复，等待真实浏览器复验。

当前结论：页面版本恢复会先生成新的项目级恢复版本，再把恢复后的页面文件写回当前 live workspace。此前 live workspace 只更新 `updatedAt`，没有把 `.workspace.json` 的 `baseVersion` 提升到新生成的恢复版本；后续自动保存、退出保存或同步项目当前工作区时，`syncActiveWorkspaceToCanonical` 会把该 workspace 判定为 `WORKSPACE_STALE`，编辑页顶部显示“同步失败”。

已落地边界：新增 workspace 版本基线更新 helper；页面版本恢复接口在写入 Session live workspace 后同步更新 `baseVersion`，再执行项目基准工作区同步；完整项目恢复也会在创建恢复版本后同步 active workspace 基线，避免同类 stale 误判。

待办：用真实浏览器打开“伴读广告页面”，恢复手机页历史版本后等待自动保存，并验证顶部状态回到“已自动保存”；再执行返回首页和重新打开，确认不会再次出现“当前工作区已过期，请刷新项目后重试”。

验证状态：页面版本历史定向 Jest 通过，覆盖恢复页面版本后更新 live workspace 基线并成功同步项目工作区；author-site `tsc --noEmit` 通过。

风险：现有 `detect-sync-status-flap.mjs --flush-only` 只验证 `/workspace-flush`，不能覆盖后续 `/persist-workspace` 的 stale 失败。后续应给脚本增加 persist 探针，报告 `latestVersionId`、`activeWorkspaceId` 和 workspace `baseVersion`。

## 项目级版本创建后 live Workspace 基线未推进

状态：已完成代码级修复，等待真实浏览器复验。

当前结论：截图中的“同步失败”和“命名版本失败 / 合并到 Demo 失败”来自同一类状态不一致：项目最新项目级版本已经被自动检查点推进到新版本，但当前 live Workspace 的 `.workspace.json.baseVersion` 仍停留在上一版本。后续自动保存、命名版本后的保存/合并、发布前同步都会调用项目工作区同步校验，并把这个仍在使用中的 live Workspace 误判为 `WORKSPACE_STALE`。

已落地边界：`createProjectVersionSnapshot` 支持统一推进已同步 active live workspace 的 `baseVersion`；自动检查点、保存/合并 Session 和发布快照都会复用这条规则。`saveEditSession` 的 live workspace 主路径已收敛到项目级版本创建函数，避免手写版本记录时漏掉基线推进。关键同步入口对历史坏状态保留受限恢复，只在最新版本 `sessionId` 与当前请求 Session 完全一致、且 workspace 仍是项目 active workspace 时修复基线并重试，避免旧工作区覆盖其他来源的新版本。

待办：用真实浏览器在“闯关活动”编辑页连续编辑触发自动检查点，确认顶部状态保持“已自动保存”；再点击“命名此版本”和执行发布，确认不再出现“合并到 Demo 失败”或发布后下一次自动保存失败。

验证状态：`workspace-flush.test.ts`、`checkpoint/route.test.ts`、`session-manager.test.ts`、`session-manager-save.test.ts`、`publish/route.test.ts`、`publish-manager-status.test.ts`、`page-version-history.test.ts` 定向测试通过；`corepack pnpm --filter @opencode-workbench/author-site typecheck` 通过。`corepack pnpm check:author` 中 typecheck 通过，全量 Jest 在 `home-page.test.tsx` 和 `preview-canvas-interaction-mode.test.tsx` 两个 UI 用例上并行超时；两个失败文件随后用 `--runInBand` 单独重跑均通过，当前判断为既有并行/定时器噪声。
