# 协同保存与实时协作架构根治方案

## 背景

创作端的实时保存和多人协作功能经过多轮修复（P1 EXTERNAL_DRIFT 自愈、P4 WORKSPACE_RESOURCE_CONFLICT baseline 跟踪、canvas-layout 双写消除等）仍然不稳定。本方案从架构层面分析不稳定的结构性根因，提出根治方案。

## 目标

- 定位"多轮修复仍然不稳定"的架构级根因
- 设计消除竞态条件的根治方案，而非继续在现有架构上打补丁
- 输出可分阶段实施的工程方案

## 范围

- `packages/agent-service/src/collab/` — Yjs 房间管理与落盘
- `packages/agent-service/src/workspace/` — Workspace Mutation Authority
- `packages/author-site/src/hooks/useCollabDocument.ts` — 前端 Yjs 客户端
- `packages/author-site/src/app/demo/[id]/edit/` — 编辑页 autosave 管道
- `packages/author-site/src/app/api/sessions/[sessionId]/canvas-layout/` — HTTP 保存路由
- `packages/author-site/src/lib/workspace-autosave-scheduler.ts` — 前端 autosave 调度器
- `packages/author-site/src/lib/workspace-flush.ts` — 工作区 flush 与 canonical 物化

## 当前架构

### 三层结构

系统当前同时维护两条**独立的内容同步通道**，二者在一处桥接，桥接点是所有不稳定的根源。

```
┌───────────────────────────────────────────────────────────────────────┐
│  通道 A：Yjs CRDT 实时协作                                               │
│  ┌──────────┐     WebSocket      ┌──────────────────────┐              │
│  │ 前端      │ ←────────────────→ │ CollabRoomManager    │              │
│  │ Yjs Doc  │    y-protocols     │ (per-resource Y.Doc) │              │
│  │ CodeMirror│   sync/awareness  │ dirty/saving/baseline│              │
│  └──────────┘                   └──────────┬───────────┘              │
│                                            │ flushRoom()               │
│                                            │ (debounce 1s)             │
│  通道 B：Workspace Mutation Authority       ▼                          │
│  ┌──────────┐     HTTP          ┌──────────────────────┐              │
│  │ 前端      │ ─────────────────→ │ WorkspaceMutation    │              │
│  │ autosave │   commitMutation   │ Authority            │              │
│  │ scheduler│                    │ (serial queue +      │              │
│  └──────────┘                    │  lease + hash check) │              │
│                                  └──────────┬───────────┘              │
│  通道 C：Pi Agent 工具                       │                          │
│  ┌──────────┐     进程内          ┌──────────┴───────────┐              │
│  │ Pi tools │ ─────────────────→ │ resolveLiveWorkspace  │              │
│  │ (AI)     │   authority.mutate │ MutationContext       │              │
│  └──────────┘                    └──────────────────────┘              │
└───────────────────────────────────────────────────────────────────────┘
```

#### 通道 A：Yjs CRDT 实时协作

| 组件 | 文件 | 职责 |
|------|------|------|
| 前端 Yjs 客户端 | `useCollabDocument.ts` | 每个 resource 创建 Y.Doc + WebsocketProvider，处理 sync/awareness |
| 后端 Yjs 房间 | `collab-room-manager.ts` | 每个 resource 一个 Y.Doc，debounce 1s 落盘，baseline 跟踪 |
| 持久化适配器 | `workspace-file-persistence.ts` | 桥接 collab room → Authority |

**一致性模型**：CRDT 最终一致。并发编辑自动合并，无冲突。每个客户端的 Yjs Doc 是内容的权威副本。

#### 通道 B：Workspace Mutation Authority

| 组件 | 文件 | 职责 |
|------|------|------|
| Authority 核心 | `workspace-mutation-authority.ts` | 唯一持久化写者，串行队列 + lease + prepared/receipt 模式 |
| 前端 HTTP 客户端 | `workspace-authority-client.ts` | 封装 `commitWorkspaceMutation`，含 auto-retry |
| 前端 autosave 调度器 | `workspace-autosave-scheduler.ts` | 800ms debounce + 3s max-wait + in-flight barrier |
| 前端 Authority 状态轮询 | `useWorkspaceAuthorityState.ts` | 2s 间隔轮询 committed revision |

**一致性模型**：悲观并发控制。每次 mutation 携带 `expectedHash`（per-resource hash）和 `baseRevision`（全局 revision），通过 `assertExpected()` 校验，不匹配则拒绝（`WORKSPACE_RESOURCE_CONFLICT`）。

#### 通道 C：Pi Agent 工具

| 组件 | 文件 | 职责 |
|------|------|------|
| Pi 工具集 | `backends/pi-tools/*.ts` | AI 文件操作（read/write/edit/delete） |
| 本地 Authority 接入 | `resolveLiveWorkspaceMutationContext()` | 每个 tool call 创建临时 Authority 实例，直接调用 `mutate()` |

**一致性模型**：与通道 B 相同（通过 Authority），但有独立的 `EXTERNAL_DRIFT` auto-retry wrapper。

### 桥接点：flushRoom()

两条通道在 `CollabRoomManager.flushRoom()` 中桥接：

```
Yjs text.toString() → commitResource(expectedHash=room.baselineHash) → authority.mutate()
```

这是所有不稳定的结构性根源。

## 不稳定根因分析

### 根因 1：三路并发写入同一组文件（结构性根因）

三个独立的代码路径同时通过 Authority 写入 workspace 文件，各自维护独立的 baseline：

| 写入方 | baseline 来源 | 路径 |
|--------|---------------|------|
| Collab room autosave | 内存 `room.baselineHash`（由 listener 回调更新） | `flushRoom()` → `commitResource()` → `authority.mutate()` |
| Author-site HTTP route | 文件系统读取（TOCTOU 窗口） | `commitWorkspaceMutation()` → HTTP → `authority.mutate()` |
| Pi Agent tools | 文件系统读取 + Authority state | `resolveLiveWorkspaceMutationContext()` → `authority.mutate()` |

**问题本质**：Authority 的 `assertExpected()` 使用 per-resource hash 做乐观并发控制。当写入方 A 提交后，写入方 B 的 `expectedHash` 立即过期。B 提交时被拒绝，触发 `WORKSPACE_RESOURCE_CONFLICT`。

**已有的缓解措施及不足**：
- `flushDraftsForMutation`：非 collab mutation 前先 flush collab rooms。但 flush 和 mutate 之间仍有时间窗口，新的 dirty 状态可能在 flush 后立即产生。
- `canvas-layout POST 前先 flush`：同上，flush 后用户可能立即拖动画布产生新的 collab dirty。
- baseline listener 回调用 receipt 数据更新：修复了 baseline 永久过期的问题，但 listener 是 fire-and-forget，不阻塞 mutate() 返回，存在窗口期。

### 根因 2：Yjs CRDT 与 Authority 悲观锁的一致性模型冲突（架构级根因）

**Yjs 的保证**：
- 所有副本最终收敛
- 并发编辑自动合并，不丢失
- 无冲突（CRDT 设计保证）

**Authority 的保证**：
- 写入串行化
- hash 不匹配则拒绝
- 拒绝后调用方必须处理冲突

**冲突场景**：
1. 用户 A 和用户 B 同时编辑同一资源的 Yjs room
2. 用户 A 的编辑先 flush，baselineHash 更新为 hash(X)
3. Pi Agent 同时写入该资源，authority revision 推进，文件内容变为 Y
4. 用户 B 的编辑后 flush，`expectedHash` 仍是 hash(X)，但文件已经是 Y
5. Authority 拒绝 → `flushRoom` 触发 self-heal → `replaceRoomText(room, Y)` 
6. **用户 B 的编辑被丢弃**，Yjs text 被替换为文件 Y 的内容
7. 替换触发 Yjs update 广播，用户 A 也收到内容变更

这是 CRDT 与悲观锁的根本矛盾：CRDT 保证合并不丢失编辑，但 Authority 的 self-heal 机制通过全量替换 Yjs text 来"恢复"，反而丢弃了 CRDT 合并的结果。

### 根因 3：多个 Authority 实例与进程内直调

```typescript
// 通道 A：WorkspaceFilePersistence 构造时创建
this.authority = new WorkspaceMutationAuthority({
  dataDir, resolveWorkspacePath: (workspaceId) => this.findWorkspacePath(workspaceId),
});

// 通道 C：每个 pi-tool call 创建临时实例
authority: new WorkspaceMutationAuthority({
  dataDir, resolveWorkspacePath: (id) => id === meta.workspaceId ? workspacePath : null,
}),
```

虽然 static maps（`queues`、`listeners`、`draftProviders`）以 `path.resolve(dataDir)` 为 key 实现了进程内共享，确保串行化，但：

- 每个实例的 `options.resolveWorkspacePath` 回调不同，可能在并发下返回不同路径
- `draftProviders` 注册在 `CollabRoomManager` 构造时，但如果 `CollabRoomManager` 尚未初始化（agent-service 启动顺序），pi-tools 的 `flushDraftsForMutation` 找不到 provider
- HTTP 路由（通道 B）通过 HTTP API 调用 agent-service，不经过进程内 Authority 实例，而是通过 Fastify route 到达另一个 Authority 实例

### 根因 4：Yjs 全量替换破坏 CRDT 语义

```typescript
private replaceRoomText(room: CollabRoom, content: string): void {
  room.doc.transact(() => {
    if (room.text.length > 0) {
      room.text.delete(0, room.text.length);  // ← 删除全部
    }
    if (content) {
      room.text.insert(0, content);  // ← 插入新内容
    }
  }, SERVER_RELOAD_ORIGIN);
}
```

当 self-heal 触发时：
1. 删除整个 Yjs text（所有客户端的未提交编辑丢失）
2. 插入文件系统读取的内容
3. 广播 update 到所有连接的客户端
4. 客户端看到内容"闪烁"（旧内容消失，新内容出现）

**这破坏了 CRDT 的核心保证**：CRDT 设计上不应该有"全量替换"操作。正确的做法是发送 Yjs update（增量 delta），让 CRDT 自动合并。但 Authority 写入的是完整文件内容，不是 Yjs delta，所以只能做全量替换。

### 根因 5：canvas-layout 双写无法彻底消除

canvas-layout 同时由两个写入方维护：

1. **Collab room**（kind=`canvas-layout`）：前端画布拖拽 → Yjs text 变更 → debounce flush
2. **Author-site HTTP route**（`canvas-layout/route.ts`）：前端 `useCanvasWorkspace` 节流保存 → POST → Authority

即使 POST 前先 flush collab room，也无法消除竞态：
- flush 完成后、POST 读文件前，用户可能拖动画布 → collab room dirty → 新的 debounce timer 启动
- POST 读文件得到 flush 后的内容，生成 `expectedHash`
- 但 collab room 的 debounce flush 可能在 POST 的 `commitWorkspaceMutation` 之前提交
- POST 的 `expectedHash` 再次过期

**根因**：canvas-layout 有两个独立的"变更源"（Yjs room 和 HTTP route），它们不应该同时存在。

### 根因 6：baseRevision 语义模糊

```typescript
// workspace-mutation-authority.ts:749
if (request.baseRevision > state.revision)
  throw new WorkspaceMutationAuthorityError("WORKSPACE_RESOURCE_CONFLICT");
```

`baseRevision` 只在**大于**当前 revision 时拒绝（不可能的场景）。当 `baseRevision` 小于当前 revision 时，不做任何检查——真正的冲突检测完全依赖 `expectedHash`。

但 collab room 的 `commitResource` 传入 `baseRevision: room.baselineRevision`，这个值可能是旧的（listener 尚未更新）。虽然旧 `baseRevision` 不会导致拒绝（因为 `< state.revision`），但它也没有提供任何保护——它既不是乐观锁，也不是版本检查，只是一个几乎无效的字段。

### 根因 7：前端保存状态与后端 Authority 状态脱节

前端有两套独立的"保存状态"：

| 状态来源 | 机制 | 更新频率 |
|----------|------|----------|
| `useCollabDocument` 的 `status` | WebSocket 连接状态 | 实时（WebSocket event） |
| `useWorkspaceAuthorityState` 的 `committedRevision` | HTTP 轮询 | 2s 间隔 |
| `hasPendingWorkspaceFlush` | `markWorkspaceChanged()` 递增 | 用户操作时 |
| `workspaceFlushRevision` | 同上 | 同上 |

这四套状态没有统一的仲裁者。当 collab room flush 成功但 Authority 轮询尚未更新时，前端可能同时显示"已自动保存"和"保存中"。当 collab room 冲突 self-heal 时，前端 `status` 可能保持 `synced`（因为 WebSocket 连接正常），但实际内容已被替换。

## 已有修复的局限性

| 已有修复 | 解决的问题 | 未解决的结构性根因 |
|----------|-----------|-------------------|
| P1: Pi-tools EXTERNAL_DRIFT auto-retry | 磁盘文件被外部修改后 AI 操作阻断 | 未消除"多路写入导致 drift"的根因 |
| P4: Bulletproof baseline tracking | Listener 回调不更新 baseline 导致 13 分钟循环冲突 | 未消除"三路写入导致 baseline 频繁过期"的根因 |
| P4: Canvas-layout POST 前先 flush | POST 与 collab autosave 的 TOCTOU 竞态 | 未消除"canvas-layout 有两个独立变更源"的根因 |
| P4: Self-heal guard 更新 baseline | Guard 跳过导致 baseline 永久过期 | 未消除"self-heal 全量替换破坏 CRDT"的根因 |
| Conflict backoff | 冲突后立即重试导致雪崩 | 未消除冲突本身 |

**总结**：已有修复都是在"三路写入 + CRDT/Authority 桥接"的现有架构上打补丁。每次修复解决一个具体症状，但架构层面的竞态条件仍然存在，只是以新的表现形式出现。

## 根治方案

### 方案对比

| 方案 | 核心思路 | 优点 | 缺点 | 改动量 |
|------|----------|------|------|--------|
| **A: Yjs-First 单写者** | Yjs room 是唯一内容写入者，Authority 退化为持久化后端 | 保留 CRDT 合并语义；消除多路写入 | 改动面大；需要重构 Pi tools 和 HTTP routes 的写入路径 | 大 |
| **B: Authority-First 去Yjs内容** | Authority 是唯一内容通道，Yjs 仅保留 awareness/presence | 架构简单；无 CRDT/锁冲突 | 丢失 CRDT 自动合并；需要手动处理并发编辑 | 中 |
| **C: 写入代理统一** | 所有非 collab 写入通过 collab room 的 Yjs text，而非直接到 Authority | 最小化改动；保留 CRDT | Pi tools 需要 WebSocket 接入；HTTP routes 需要改造 | 中 |

### 推荐方案：A — Yjs-First 单写者模式

#### 设计原则

1. **Yjs room 是每个 resource 的唯一内容权威**：所有内容变更（用户编辑、AI 修改、HTTP 保存）都通过 Yjs text 的 `insert`/`delete` 操作完成
2. **Authority 退化为持久化后端**：不再做 `expectedHash` 冲突检测，只负责将 Yjs text 的快照原子写入文件系统
3. **消除三路写入**：Pi tools 和 HTTP routes 不再直接调用 `authority.mutate()`，而是通过 Yjs room 写入内容
4. **消除 self-heal 全量替换**：由于 Yjs 是唯一内容源，不存在"外部修改"导致的内容不一致

#### 架构变更

```
┌─────────────────────────────────────────────────────────────────┐
│  统一内容通道：Yjs CRDT                                          │
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                    │
│  │ 前端编辑器 │   │ Pi Agent  │   │ HTTP API  │                    │
│  │ (用户)    │   │ (AI)      │   │ (画布等)   │                    │
│  └─────┬────┘   └─────┬────┘   └─────┬────┘                    │
│        │              │              │                          │
│        ▼              ▼              ▼                          │
│  ┌─────────────────────────────────────────┐                    │
│  │         Yjs Room (per resource)         │                    │
│  │  Y.Doc → Y.Text (CRDT merge)           │                    │
│  └──────────────────┬────────────────────┘                    │
│                     │ debounce flush                            │
│                     ▼                                          │
│  ┌─────────────────────────────────────────┐                    │
│  │    Authority (持久化后端)                │                    │
│  │  flush Yjs text → atomic write          │                    │
│  │  revision tracking (no hash check)      │                    │
│  │  crash recovery (prepared/receipt)      │                    │
│  └─────────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
```

#### 关键改动

##### 改动 1：Pi tools 写入路径改为通过 Yjs room

**当前**：`resolveLiveWorkspaceMutationContext()` → `authority.mutate()` → 直接写文件

**改为**：Pi tools 通过 agent-service 内部 API 将内容写入 Yjs room

```typescript
// 新增：agent-service 内部 API
async function writeToCollabRoom(
  workspaceId: string,
  resourcePath: string,
  content: string,
): Promise<void> {
  const room = await collabRoomManager.getOrCreateRoom(
    descriptor, workspacePath,
  );
  room.doc.transact(() => {
    room.text.delete(0, room.text.length);
    room.text.insert(0, content);
  });
  // 立即 flush（AI 写入不需要 debounce）
  await collabRoomManager.flushRoom(room);
}
```

**效果**：Pi tools 不再是 Authority 的独立写入方，而是通过 Yjs room 写入。Yjs CRDT 自动处理与用户并发编辑的合并。

##### 改动 2：HTTP routes 写入路径改为通过 Yjs room

**当前**：`canvas-layout/route.ts` POST → `commitWorkspaceMutation()` → HTTP → `authority.mutate()`

**改为**：POST → agent-service 内部 API → 写入 Yjs room → flush

```typescript
// canvas-layout/route.ts
if (isLiveWorkspace(access.workspaceId)) {
  // 不再直接调 commitWorkspaceMutation
  // 改为通过 agent-service API 写入 collab room
  await fetch(`${agentServiceUrl}/api/collab/.../write`, {
    method: "POST",
    body: JSON.stringify({ resourcePath, content, kind }),
  });
}
```

**效果**：消除 canvas-layout 双写。画布布局只有 Yjs room 一个变更源，HTTP route 通过 Yjs room 写入，不再有 TOCTOU 窗口。

##### 改动 3：Authority 移除 expectedHash 冲突检测

**当前**：`assertExpected()` 检查 `expectedHash` vs 文件当前 hash，不匹配则拒绝

**改为**：Authority 只做 revision 追踪和原子写入，不再检查 per-resource hash

```typescript
// 简化后的 mutate()
async mutate(request: WorkspaceMutationRequest): Promise<WorkspaceMutationReceipt> {
  // 不再调用 assertExpected()
  // 直接 prepare → apply → commit
  // revision 递增，rootHash 仅用于 drift 检测（不作为冲突拒绝依据）
}
```

**效果**：消除所有 `WORKSPACE_RESOURCE_CONFLICT`。Authority 变成纯粹的持久化层，不再做乐观并发控制。

**风险控制**：保留 `WORKSPACE_EXTERNAL_DRIFT` 检测（rootHash 不匹配时标记 drift，但不拒绝写入——改为 `reconcileAdopt` 自动采纳），避免外部文件系统修改被静默覆盖。

##### 改动 4：消除 self-heal 全量替换

**当前**：冲突时 `replaceRoomText(room, fileContent)` 全量替换 Yjs text

**改为**：由于 Yjs room 是唯一内容源，不再需要 self-heal。外部写入已经通过 Yjs room 完成，Yjs text 始终是最新的。

**效果**：消除"用户编辑被全量替换丢弃"的问题。

##### 改动 5：前端 autosave 简化

**当前**：前端有 autosave scheduler（通道 B）+ collab room flush（通道 A）双轨保存

**改为**：前端编辑只通过 Yjs（通道 A），不再有独立的 autosave scheduler。画布布局等也通过 Yjs room 同步。

**效果**：前端保存状态只有一套来源（Yjs room 的 dirty/saving/baseline），消除状态不一致。

#### 被否决的替代方案

| 方案 | 否决原因 |
|------|----------|
| B: Authority-First 去 Yjs 内容 | 丢失 CRDT 自动合并能力，多人并发编辑时需要手动处理冲突，用户体验降级 |
| C: 写入代理统一（不改 Authority） | 保留了 `expectedHash` 冲突检测，Pi tools 写入 Yjs room 后 flush 仍可能与 HTTP route flush 冲突 |
| 保持现状 + 增强重试 | 根因未消除，每次新场景都需要新的补丁 |

## 风险与待确认事项

1. **Pi tools 写入 Yjs room 的延迟**：AI 写入需要经过 Yjs room + flush，比直接调 Authority 多一跳。预估延迟增加 5-20ms（Yjs transact + flush），对 AI 操作可接受。
2. **Yjs room 未激活时的写入**：如果 collab room 不存在（用户未打开编辑器），Pi tools 需要临时创建 room 并 flush。需要确保 room 创建路径轻量化。
3. **Authority 移除 expectedHash 后的数据一致性**：由于 Yjs room 是唯一写入方，同一 resource 的写入被 Yjs 串行化（flush 是串行的），不会出现并发写入。但需要验证 crash recovery 路径（prepared mutation 恢复）在无 expectedHash 时的行为。
4. **前端 autosave scheduler 移除的影响**：部分场景可能依赖 scheduler 的 3s max-wait 语义（持续编辑时强制保存）。Yjs room 的 debounce（1s）+ flush 是等价的，但需要验证是否所有调用点都被覆盖。
5. **非 live workspace 的兼容性**：branch workspace 不经过 Authority，直接写文件。改动需要保持非 live 路径不变。

### Hocuspocus Authority 持久化 Extension 设计

Yjs-First 架构下，Hocuspocus 的 `onStoreDocument` 成为唯一持久化入口，Authority 退化为纯文件写入器：

```typescript
import { Extension } from "@hocuspocus/extension";

class AuthorityPersistenceExtension implements Extension {
  constructor(private authority: WorkspaceMutationAuthority) {}

  async onStoreDocument({ document, documentName }) {
    // Yjs-First: 这是唯一持久化点
    // Authority 已无 expectedHash 冲突检测，直接写入
    const text = document.getText("content").toString();
    const { projectId, workspaceId, resourcePath } = parseDocumentName(documentName);
    await this.authority.mutate({
      mutationId: crypto.randomUUID(),
      projectId,
      workspaceId,
      actor: "collab",
      reason: "collab_autosave",
      operations: [{ type: "put_text", path: resourcePath, content: text }],
    });
  }
}
```

对比当前 `flushRoom` + `commitResource` + `onMutationCommitted` listener + `BASELINE_RESET` 的复杂链路，Hocuspocus 方案将持久化逻辑收敛到一个 hook 函数。

### 迁移收益

1. **消除 ~500 行自研代码**：room 管理、flush、listener、awareness、cleanup 全部由 Hocuspocus 内置
2. **社区验证的健壮性**：WebSocket 重连、Yjs GC、并发连接处理经过大量生产验证
3. **水平扩展能力**：通过 `@hocuspocus/extension-redis` 实现多实例部署，为未来多 agent-service 节点做准备
4. **扩展生态**：`onAuthenticate`（JWT）、`onChange`（变更通知）、`onDisconnect`（连接清理）等 hook 覆盖常见需求
5. **降低维护负担**：自研代码中的 edge case（room idle TTL、connection limit、conflict backoff）由框架处理

### 迁移风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| Hocuspocus 的 debounce 语义与当前不同 | 落盘频率变化 | `onStoreDocument` 支持 `debounce` 配置，可对齐当前 1s |
| Hocuspocus 的 documentName 编码方式不同 | 前端连接参数变化 | 使用 Hocuspocus 的 `documentName` + `token` 参数体系，前端改动量小 |
| Hocuspocus 与 agent-service Fastify 的集成 | 需要确认 HTTP/WebSocket 共存 | Hocuspocus 可作为 Fastify 插件或独立 WebSocket server 挂载 |
| 非直播 workspace 不走 Authority | Hocuspocus 需要区分 live/branch | `onStoreDocument` 内判断 workspace scope，branch 直接写文件 |
