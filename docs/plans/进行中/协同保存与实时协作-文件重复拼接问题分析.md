# 协同保存与实时协作 — 文件内容重复拼接问题分析

## 当前状态

**根因已定死，修复已实施并通过真实浏览器验证。**

三层修复已落地：
1. 消除客户端 "connected" 事件过早设置 status="synced"
2. `replaceCollabText` 用 Yjs transaction 包裹 delete + insert
3. 服务端 `onStoreDocument` 加内容去重守卫（defense-in-depth）

## 根因

`useCollabDocument.ts` 的 HocuspocusProvider "status" 事件处理器在 `event.status === "connected"` 时**过早设置 status="synced"**。

"connected" 只表示 WebSocket 连接已建立，Yjs sync step 1/2 尚未完成。此时 ytext 仍为空，但 page.tsx 的 useEffect 看到 status="synced" 且 value="" 会触发 `replaceCollabText(ytext, codeRef.current)`，在本地 ytext 插入编辑器内容。

与此同时，服务端 `onLoadDocument` 也会向自己的 ytext 插入磁盘内容。两个 insert 在 Yjs CRDT 中各自独立，sync 完成后合并为两份内容拼接 → 重复。

诊断日志实证（浏览器打开-关闭循环）：
```
[DIAG] code-effect: pushing local to empty ytext, codeRef.len=3560
[DIAG] replaceCollabText called, ytext.beforeLen=0, value.len=3560, deleteLen=0
```
确认 useEffect 在 ytext 为空时执行了 insert。由于竞态条件（服务端 ytext 是否已加载磁盘内容），重复不一定每次都发生，但代码路径明确存在。

## 修复方案

### 修复 1：消除过早 "synced" 状态（根因修复）

`packages/author-site/src/hooks/useCollabDocument.ts` L260-268：

"connected" 事件处理器不再设置 status="synced"，仅清除离线计时器。正确的 "synced" 状态由 "synced" 事件处理器（sync step 2 完成后触发）设置。

### 修复 2：replaceCollabText 原子化（防御性修复）

`packages/author-site/src/app/demo/[id]/edit/page.tsx` L523-544：

用 `ytext.doc.transact()` 包裹 delete + insert，确保作为单个 Yjs update 传播，避免中间空态在 sync 过程中被服务端误判。

### 修复 3：onStoreDocument 内容去重守卫（defense-in-depth）

`packages/agent-service/src/collab/extensions/authority-persistence.ts` L65-128：

`onStoreDocument` 写入前检测 room 内容是否为自拼接重复（`isContentDuplicated`）。如果是，截断为前半段并同步修正 Yjs room 内容，确保重复内容不会落盘。
