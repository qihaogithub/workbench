# 协同保存与实时协作 — 文件内容重复拼接问题分析

## 当前状态

已确认磁盘文件存在重复拼接，**根因机制尚未实证定死**。需加诊断日志做一次打开-关闭循环才能定位。

## 现象

- `pad-popup_a3f2/index.tsx` 文件内容被拼接为 2 份完全相同的 84 行模块（总 168 行）
- 用户报告曾达到 9 份完全相同的模块拼接
- **即使不做任何修改**，仅打开-关闭编辑页就会导致重复次数增加
- AI auto-repair 修复后退出再打开，修复不持久（被重复内容覆盖）

## 已排除的可能性

1. **文件写入不是追加**：`writeTextAtomic` = `writeFileSync(tmp)` + `renameSync(tmp, file)`，是原子替换
2. **`onStoreDocument` 不是追加**：读取 `text.toString()` 后通过 `commitResource` → `put_text` → `writeTextAtomic` 整体替换
3. **PUT handler 不是追加**：`persistActivePageToSession` → Authority `put_text` → `writeTextAtomic` 整体替换
4. **所有已知写入路径都是替换操作，理论上不可能累积**

## 已识别的可疑点（未实证）

### 可疑点 A：`replaceCollabText` 非原子操作

`page.tsx` 中的 `replaceCollabText`（L523-534）：

```typescript
function replaceCollabText(ytext, value) {
    if (!ytext || ytext.toString() === value) return;
    ytext.delete(0, ytext.toString().length);  // 操作 1
    if (value) ytext.insert(0, value);          // 操作 2
}
```

**问题**：delete + insert 不在 Yjs transaction 中，是两个独立的 Yjs update。通过 HocuspocusProvider sync 协议分别发送到服务端。中间状态下服务端 Yjs text 为空。

**未验证**：这个中间状态如何导致服务端 Yjs text 内容重复。Yjs CRDT 理论上是幂等的，重复应用同一操作不应累积。

### 可疑点 B：`flushBeforeExit` 双重 sync

`flushBeforeExit`（page.tsx L4978-5044）：

```typescript
await flushSyncWorkspaceRef.current();  // 内部调用 syncWorkspaceToProject
if (hasPendingWorkspaceFlush) {
    await syncWorkspaceToProject();      // 又调用了一次
} else if (shouldPersistWorkspace) {
    await runWorkspaceSyncStep("persist-workspace", persistWorkspaceToProject);
}
```

`syncWorkspaceToProject` 执行三步：`persistActivePageToSession`（PUT 写文件）→ `flushWorkspaceCollab`（Yjs flush）→ `persistWorkspaceToProject`（canonical 物化）。

双重 sync 实际会发生：`flushSyncWorkspace` 内部 `setHasPendingWorkspaceFlush(false)` 是异步 React 状态更新，不改变 `flushBeforeExit` 闭包中的旧值；`syncInFlightRef` 守卫只防并发不防顺序调用。因此 `hasPendingWorkspaceFlush` 在闭包中仍为 true，触发第二次 `syncWorkspaceToProject()`。

**未验证**：PUT 写入和 Yjs flush 之间的交互是否导致内容重复。

### 可疑点 C：`onLoadDocument` 非事务性插入

`AuthorityPersistenceExtension.onLoadDocument`：

```typescript
if (text.length > 0) return;
const state = this.persistence.readResourceState(...);
if (state.content) {
    text.insert(0, state.content);  // 不在 transaction 中
}
```

**问题**：插入不在 transaction 中，触发 `doc.on("update")` → Hocuspocus 可能调度 `onStoreDocument`。此时如果文件内容本身已重复（上次退出写入的），重复会被再次加载。

**未验证**：首次重复是如何产生的（鸡生蛋问题）。

## 诊断日志方案

需要在以下位置加 `[DIAG]` 日志：

1. **服务端 `onLoadDocument`**：记录从磁盘读取的文件行数、是否已重复、Yjs text 插入前行数
2. **服务端 `onStoreDocument`**：记录 Yjs room 内容行数、磁盘文件行数、是否将写入、Yjs room 内容是否重复
3. **客户端 `replaceCollabText`**：记录 ytext 当前行数、value 行数、是否在 transaction 中
4. **`persistActivePageToSession`**：记录 PUT body 中 code 的行数

关键判断逻辑：
- 如果 `onLoadDocument` 报 `duplicated: true` → 文件在加载前已重复 → 问题在上次退出写入
- 如果 `onStoreDocument` 报 `roomDuplicated: true` → Yjs room 内存中已重复 → 需要追溯哪一步操作导致
- 如果 `replaceCollabText` 被触发且 `hasTransaction: false` → 确认非原子操作是触发因素
