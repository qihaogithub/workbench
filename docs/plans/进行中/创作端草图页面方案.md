## 当前结论

创作端当前已经有两类页面：

| 页面类型 | 系统类型 | 人类编辑方式 | AI 编辑方式 | 主要价值 |
| :--- | :--- | :--- | :--- | :--- |
| 原型页 | `prototype-html-css` | 可通过属性面板编辑局部文本、样式和配置项，但不适合自由拖拽绘制 | AI 直接改写 HTML/CSS | 低运行时成本表达任意静态页面布局 |
| 高保真页 | `high-fidelity-react` | 可通过属性面板表达修改意图，复杂变更仍交给 AI/代码 | AI 改写 React/TSX | 承载真实运行时、状态、交互和发布验收 |
| 草图页 | 暂定 `sketch-scene` | 人类可直接拖拽、绘制、排版、编辑文本和形状 | AI 读写同一份结构化场景文档 | 让人类和 AI 在同一可编辑画板上共同塑形页面 |

草图页面不是“低配原型页”，也不是预览画布里的批注层。它应作为正式页面运行时参与页面树、单页预览、画布工作台、版本、协同、AI 工具、CLI、截图和发布流程。核心差异在于页面源文件不是代码或 HTML，而是一份结构化 scene 文档。

草图页面的第一原则是“人类可直接编辑”。用户应能在固定页面尺寸内直接添加文字、形状、图片、箭头、便签、分组和基础组件占位；AI 应能读取这些对象的语义、位置、样式和关系，并对同一 scene 做结构化增删改，而不是只能看截图后重新生成代码。

## 产品定位

### 适用场景

- 早期信息架构、页面草稿、流程说明和低成本视觉沟通。
- 用户希望直接拖拽排版，而不是每次都向 AI 描述“把按钮往右挪一点”。
- AI 需要基于用户手绘结构继续补全布局、文案、页面层级或转成原型页/高保真页。
- 多人围绕同一个页面草案协作，且希望所有修改都能被 AI 稳定理解。

### 不适用场景

- 需要真实交互、状态联动、接口请求、复杂组件生命周期或运行时校验的页面，应升级为高保真页。
- 需要完整视觉还原和发布验收的静态页面，应转为原型页或高保真页。
- 仅用于项目说明、评审标注或流程图的内容，优先放在画布工作台的语义文本、文档或未来白板节点，不应误建为页面。

### 与现有画布工作台的边界

现有画布工作台是项目级编排空间，用于摆放多个页面、文档、图片和文本节点；草图页面是页面级内容本身。二者都可能使用画板交互，但数据边界不同：

| 能力 | 画布工作台 | 草图页面 |
| :--- | :--- | :--- |
| 保存对象 | 页面位置、文档节点、图片节点、文本语义节点 | 单个页面内部的图形、文本、图片、分组和组件占位 |
| 文件位置 | Workspace 级 `.canvas-layout.json` | 页面目录内的草图 scene 文件 |
| AI 语义 | 项目组织、说明和页面关系 | 当前页面的布局、内容和视觉结构 |
| 发布语义 | 不直接作为用户访问页面 | 可被发布为静态草图页，或转为原型/高保真后发布 |

## 用户体验

### 创建入口

普通项目生成仍默认走原型页，不把页面类型选择前置给用户。草图页面应通过明确入口创建：

- 页面列表或单页预览的“新建页面”扩展入口：新建草图页。
- AI 对话中用户明确表达“先画草图”“做一个可拖拽编辑的线框草稿”“我想手动画一下”。
- 原型页或高保真页的转换入口：转为草图页，作为静态结构复刻或重新整理，不承诺无损。

### 编辑体验

草图页只在单页面预览模式下支持编辑。默认状态仍是预览态，和原型页、高保真页一样用于查看当前页面效果；用户点击预览区顶部的“编辑”按钮后，进入草图编辑模式。编辑模式下，预览区底部额外出现草图工具栏，承载绘制、选择和对象编辑工具。退出编辑模式后，底部工具栏收起，预览区回到普通页面预览。

草图页编辑不进入项目级画布工作台。画布工作台对所有页面保持同一套交互：页面卡片可选中、移动、缩放、聚焦和配置，不因为页面是草图页而挂载草图编辑器或展示页面内部绘制工具。

草图页编辑模式聚焦固定尺寸页面编辑，不需要无限画布、画布拖拽平移、画布缩放、缩略导航或跨页面大画板能力，但页面内部的绘图工具应尽量丰富，避免草图页只能做少量线框。

MVP 基础工具：

- 选择、框选、对象拖拽、对象缩放、旋转、对齐、分布、层级调整。
- 矩形、圆形、线条、箭头、自由线、文本、图片、便签、基础按钮/输入框/卡片占位。
- 多选、分组、锁定、复制、粘贴、撤销、重做。
- 对象属性编辑：位置、尺寸、填充、描边、圆角、透明度、字号、字重、对齐、层级。
- 页面编辑区域继承 `previewSize` 语义，草图对象坐标以该固定页面尺寸为基准；用户编辑的是页面内部对象，不编辑一个可无限延展的大画布。

增强绘图工具建议进入 Phase 1.5 或 Phase 2，不阻塞首版草图页运行时，但选型时需要确认底层引擎支持：

| 工具类目 | 能力 |
| :--- | :--- |
| 基础图形 | 矩形、圆形/椭圆、三角形、多边形、星形、菱形、圆角矩形 |
| 线条连接 | 直线、折线、箭头、双向箭头、曲线、连接线吸附 |
| 钢笔路径 | 点击添加锚点、拖拽控制手柄、直线/曲线段混排、闭合路径、锚点增删 |
| 自由绘制 | 铅笔、马克笔、荧光笔、橡皮擦、笔刷颜色和粗细 |
| 文本 | 普通文本、文本框、标题样式、列表文本、文字对齐 |
| 图片与资源 | 图片插入、裁剪、圆角、透明度、替换资源 |
| 组件占位 | 按钮、输入框、卡片、导航栏、标签、头像、图标占位 |
| 排版辅助 | 吸附线、等距分布、网格吸附、复制样式、组合/取消组合 |
| 标注 | 便签、气泡、编号标记、高亮框、局部放大框 |

底部工具栏应采用分组工具，而不是把所有工具一次性铺开。建议首屏展示选择、图形、线条、画笔、文本、图片、组件占位和更多；“更多”中再承载多边形、星形、标注、连接线等低频工具。

Figma 式钢笔工具可以支持，但不作为首版最小闭环的硬依赖。首版可以先支持“路径对象可创建、可移动、可缩放、可改描边/填充”；后续再补齐锚点级编辑。完整钢笔工具至少包括：

- 点击创建直线段，拖拽创建 Bezier 曲线段。
- 路径上显示锚点和控制手柄，支持拖拽调整。
- 支持开放路径和闭合路径。
- 支持添加、删除、移动锚点。
- 支持直角点、平滑点和镜像手柄。
- 支持路径填充、描边、端点样式和箭头样式。

实现上应把路径保存为项目自有 `path` 节点，记录标准化 path commands 或分段模型，避免把引擎私有 path 字符串作为唯一权威。底层引擎可负责渲染和命中测试，锚点/手柄编辑器由草图页实现。

### AI 协作体验

AI 和用户编辑同一份 scene 文档。用户不需要把草图改完“发给 AI”，系统应自动把当前 scene 摘要作为 AI 上下文的一部分。

- 用户选中对象后询问 AI，AI 应拿到选区对象 ID、类型、文本、样式、位置和邻近关系。
- 用户发送“把这里改成支付流程三步卡片”，AI 应对 scene 做结构化 patch，而不是重写整页截图。
- AI 修改后，用户应看到对象级 diff 或轻量变更摘要，并可撤销。
- AI 生成草图时，应优先使用语义对象和基础组件占位，避免把文字烘焙进图片。

### 配置项体验

草图页也应支持配置项，但配置范围可以比原型页和高保真页窄。草图页配置项的定位不是完整业务参数系统，而是让用户和 AI 能把草图中的关键文案、图片、颜色和简单状态暴露成可复用变量。

MVP 支持类目：

| 类目 | 绑定对象 | 示例 |
| :--- | :--- | :--- |
| 文本 | 文本对象、按钮占位、卡片标题 | 标题、副标题、按钮文案 |
| 图片 | 图片对象、头像占位、封面占位 | 主视觉、头像、图标资源 |
| 颜色 | 填充色、描边色、文本颜色 | 品牌色、强调色、背景色 |
| 显示状态 | 任意对象或分组 | 是否展示角标、提示区、空状态 |
| 简单枚举 | 分组或组件占位的变体 | 选中/未选中、轻/深色、横排/竖排 |

暂不作为 MVP 的类目：

- 复杂布局规则，例如自动栅格、响应式断点、条件布局。
- 运行时交互状态，例如表单联动、异步数据、点击后状态机。
- 复杂数组数据，例如动态列表、表格数据源、嵌套对象集合。
- 高级样式系统，例如完整 typography token、组件 variant 矩阵。

配置项添加入口应和原型页/高保真页保持一致：用户选中对象后，可在属性面板将可配置属性标记为配置项。草图页写入 `config.schema.json`，同时在 `sketch.scene.json` 的对象属性上记录绑定关系；配置变更时，草图渲染器只替换绑定属性，不改变对象几何结构。

## 数据与系统设计

### 技术栈选型结论

草图页当前需求已经收敛为“固定页面尺寸内的对象编辑器”，不是无限白板。因此技术栈确定为项目自有 scene 协议 + SVG/DOM 渲染 + Moveable/Selecto 交互层，不采用 Excalidraw、tldraw 或 Drawnix 这类完整白板产品作为主方案。

确定主栈：

| 层级 | 推荐 | 作用 |
| :--- | :--- | :--- |
| 场景协议 | 自定义 `SketchSceneDocument` | 项目长期数据契约，供 AI、配置项、版本、截图和转换复用 |
| 渲染层 | SVG + HTML overlay | 固定页面内渲染图形、路径、图片、文本和组件占位 |
| 变换交互 | `react-moveable` / `moveable` | 拖拽、缩放、旋转、组合、吸附和编辑控制框 |
| 框选选择 | `react-selecto` / `selecto` | 多选、框选、选择状态管理 |
| 状态管理 | React state + reducer / Zustand 局部 store | 对象级 patch、撤销重做、AI 操作合并 |
| 路径工具 | 自研 SVG path 编辑器 | Figma 式钢笔锚点、手柄和 path commands |
| 截图/只读 | 同一 SVG/DOM renderer | author-site、viewer-site、screenshot-service 复用只读渲染 |

选择理由：

- 固定页面不需要白板库的 viewport、无限画布、缩略导航和协同白板模型。
- SVG path 天然适合钢笔工具、锚点、Bezier 曲线、箭头、填充和描边，也方便 AI 读取标准化 path commands。
- DOM/SVG 对文本、图片、配置项绑定、对象语义、无障碍名称和调试更友好，避免把所有语义藏进 canvas 私有对象。
- 只读渲染可以在浏览器、viewer 和截图服务中复用同一套组件，减少“编辑器渲染”和“发布渲染”不一致。
- `moveable` 解决通用变换控制框，不需要从零实现拖拽、缩放、旋转和吸附。

备选方案仅作为风险兜底或局部能力补充，不作为当前实施主线：

| 方案 | 适用情况 | 不作为首选的原因 |
| :--- | :--- | :--- |
| Fabric.js | SVG/DOM 自由绘制或图片处理成本明显失控时，可作为局部绘图内核兜底 | 数据和交互容易被 Fabric 私有模型牵引，AI/config 绑定需要额外适配；Figma 式钢笔仍要自研 |
| Konva / react-konva | 需要更强 canvas 性能或复杂动画 | 编辑器能力要自组，文本与配置项语义不如 DOM/SVG 直接 |
| Excalidraw | 想最快验证白板草图 | 白板模型和固定页面编辑不完全匹配，配置项和页面组件语义较弱 |
| tldraw | 接受授权且想使用完整白板 SDK | 内部生产使用仍需确认 license，且能力重心偏无限画布/白板 |

实施边界：业务代码只读写 `SketchSceneDocument`，不得直接依赖 Moveable/Selecto 的运行时状态，也不得引入 Fabric/Konva/Excalidraw/tldraw 的私有 JSON 作为长期存储格式。Moveable、Selecto 和 SVG path 编辑器只是编辑交互层；只读预览、截图、AI 工具和转换流程都消费同一份 scene 协议。

### 页面运行时类型

建议把共享运行时扩展为三类，命名后续实施前可调整：

```ts
type DemoPageRuntimeType =
  | "prototype-html-css"
  | "high-fidelity-react"
  | "sketch-scene";
```

当前代码里存在多个“非 `prototype-html-css` 即 `high-fidelity-react`”的兼容分支，实施时必须统一改成显式三分支，避免草图页被误送入 iframe 编译、高保真转换或原型清洗逻辑。

### 页面文件协议

草图页建议使用页面目录内的结构化文件：

| 文件 | 用途 | 默认展示 |
| :--- | :--- | :--- |
| `sketch.scene.json` | 草图对象、画板、图层、资源引用和绑定信息 | 是 |
| `sketch.meta.json` | 引擎版本、生成来源、最近转换信息、性能统计 | 否 |
| `config.schema.json` | 可配置字段；只有存在配置项时展示 | 有内容时展示 |

`sketch.scene.json` 不应只是某个第三方库的私有 dump。即使底层选用现成绘图或白板库，也需要在项目层定义稳定的 `SketchSceneDocument` 适配协议，至少包含：

- `version`：项目协议版本。
- `pageSize`：固定页面宽高，与预览尺寸对齐。
- `nodes`：对象列表或对象树，包含稳定 ID、类型、几何、样式、文本、资源引用。
- `bindings`：箭头连接、配置项绑定、对象关系。
- `assets`：图片和外部资源引用，必须走工作区资源协议。
- `metadata`：AI 生成来源、最近编辑者、可选对象语义标签。

配置项绑定建议进入 scene 文档中的对象级属性，而不是把配置逻辑塞进对象名称或自由文本。例如文本对象可以记录 `bindings.text = "heroTitle"`，图片对象可以记录 `bindings.src = "coverImage"`，颜色属性可以记录 `bindings.fill = "brandColor"`。这样 AI、配置面板和只读渲染器都能稳定理解同一份绑定关系。

### 协同保存

草图页不能复用 `page-code` 文本房间简单整文件覆盖，否则多人移动或调整对象会造成大 JSON 冲突和丢改。建议新增协同资源类型，或在现有协同层下扩展专用资源：

| 资源 | 建议 |
| :--- | :--- |
| 草图 scene | 新增 `page-sketch-scene`，初期可按 JSON 文本协同，正式版应评估对象级 CRDT |
| 配置 Schema | 继续使用 `page-schema` |
| 画布工作台布局 | 继续使用 `canvas-layout`，不与草图 scene 混用 |

MVP 可以先使用成熟画板库自带的协同模型或 JSON patch 队列，但必须保留服务端 flush、Workspace 基准同步、版本快照和外部 AI 写入后重载房间的权威边界。

### 截图与预览

草图页应进入同一截图与画布调度体系：

- 单页预览：默认使用只读草图渲染；点击顶部“编辑”按钮后挂载草图编辑器，并在预览区底部展示草图工具栏。
- 画布工作台：与所有页面保持一致，优先展示截图；截图不可用时使用轻量只读渲染或加载态，不挂载草图编辑器，也不显示草图工具栏。
- 截图服务：新增 `PageSnapshotInput` 分支，输入 scene 文档和资源映射，渲染确定性的只读草图页面。
- viewer-site：至少支持只读草图渲染；是否允许对外直接发布，需要由发布策略决定。

### AI 工具协议

AI 不应通过普通 `writeFile` 盲写整份 `sketch.scene.json` 作为主要路径。建议新增面向草图的工具层：

- `read_sketch_scene(pageId)`：返回压缩后的对象树、选区、页面尺寸和资源清单。
- `patch_sketch_scene(pageId, operations)`：按对象级操作增删改，服务端校验 ID、资源和几何范围。
- `create_sketch_nodes(pageId, nodes, placement)`：批量创建语义对象。
- `bind_sketch_config(pageId, nodeId, property, field)`：把对象属性绑定到页面配置项，并同步更新 scene 与 schema。
- `convert_sketch_page(pageId, targetRuntimeType)`：转原型页或高保真页。

对象级 patch 应支持 dry-run 和诊断输出，失败时返回可给 AI 修复的结构化错误，而不是只返回 JSON 写入失败。

## 转换策略

### 草图页转原型页

目标是把结构化 scene 转为可预览、可发布的 HTML/CSS 静态页面。适用条件：

- 页面以静态布局、文本、图片和基础视觉元素为主。
- 不要求保留自由线条的逐点编辑能力。
- 用户希望从草稿进入更接近真实页面的视觉表达。

转换流程：

1. 读取 scene 对象树和资源清单。
2. 将基础对象映射为语义 HTML，形状和自由线可转为 SVG 或 CSS 装饰。
3. 生成 scoped CSS，并按页面设计尺寸保持布局。
4. 运行原型页清洗和 `prototypeGate`。
5. 通过后切换为 `prototype-html-css`，保留草图来源快照用于回退。

### 草图页转高保真页

适用于用户要求真实交互、状态、组件化或直接进入开发验收。流程与原型升级高保真类似，但 AI 输入应优先使用 scene 语义摘要，而不是只看截图。

### 原型页/高保真页转草图页

该转换不承诺无损，定位是“生成可继续手工编辑的结构草稿”：

- 原型页转草图页：解析 DOM/CSS，抽取主要文本、图片、卡片、按钮、列表和布局块，生成可编辑对象。
- 高保真页转草图页：优先基于当前渲染 DOM 和截图辅助抽取，复杂交互只保留静态状态。
- 转换前应提示用户：草图页会保留视觉结构和内容层级，但运行时交互、复杂样式和组件状态可能丢失。

## 验收标准

### 产品验收

- 用户能创建草图页，在单页面预览点击顶部“编辑”按钮后绘制、拖拽、编辑文本、插入图片和调整样式。
- 草图编辑模式下，预览区底部展示草图工具栏；退出编辑后工具栏消失。
- 画布工作台对草图页、原型页和高保真页保持一致交互，不提供页面内部草图编辑能力。
- 用户不需要把草图手动“发给 AI”；AI 能基于当前 scene 和选区直接修改草图。
- 草图页在页面列表、单页预览、画布工作台、版本历史和导出包中都表现为正式页面。
- 草图页可把选中对象的文本、图片、颜色、显示状态和简单枚举标记为配置项，并在配置面板实时生效。
- 草图页与原型页、高保真页的差异文案保持中性，不暗示哪一种更高级。

### 技术验收

- `DemoPageRuntimeType`、`CanvasPageRuntimeType`、`PageSnapshotInput`、runtime validation 和文件树展示都显式支持第三类运行时。
- 草图 scene 文件通过共享校验入口写入，不能绕过路径安全、资源白名单和文件大小限制。
- 草图配置项同时更新 `config.schema.json` 和 `sketch.scene.json` 对象绑定，刷新后绑定关系不丢失。
- 草图页不会被错误送入 React 编译、iframe runtime 或 HTML/CSS 原型清洗路径。
- 截图服务和 viewer-site 能使用同一只读渲染协议展示草图页。
- 协同 flush、版本、发布和 AI 外部写入不会互相覆盖草图 scene。

### AI 验收

- AI 能读取草图对象树，而不是只依赖截图理解页面。
- AI 修改草图优先输出对象级操作，避免整文件重写造成冲突。
- AI 能识别并维护草图对象上的配置项绑定，不应在改写对象时静默移除绑定。
- AI 可以把草图转换为原型页或高保真页，并在失败时收到结构化诊断。
- AI 不应把草图页当作批注层或临时图片，也不应在用户要求草图时默认生成 React/HTML。

## 风险

- 画板引擎如果直接把私有数据结构作为项目协议，后续更换引擎或升级版本会很痛苦；必须有项目级稳定适配层。
- 草图 scene 如果按整文件文本协同，AI 和多人编辑容易互相覆盖；对象级 patch 或 CRDT 是长期方向。
- 草图页容易和项目画布工作台混淆，需要在数据边界和 UI 文案上明确“页面内部草图”和“项目组织画布”。
- 草图页若直接发布，用户可能误以为它具备真实交互；发布态需要清晰的只读/静态语义。
- 转换为原型页或高保真页不可能完全保留草图编辑能力，应保留来源快照和回退路径。
- 新增第三类运行时会放大当前二分支假设，必须先做共享类型和 runtime 分发梳理，再接 UI。
