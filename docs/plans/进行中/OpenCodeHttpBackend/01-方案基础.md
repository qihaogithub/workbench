## 一、背景与目标

### 1.1 项目背景

当前系统中，Agent 服务层通过 ACP 协议（Agent Client Protocol）对接 OpenCode CLI，存在以下问题：

| 问题 | 影响 |
|:---|:---|
| **部署复杂** | 需要在服务器安装 OpenCode CLI，并管理子进程生命周期 |
| **多用户支持弱** | ACP 协议设计为单用户本地使用，不适合多用户场景 |
| **edit 工具通知丢失** | OpenCode 的 edit 工具不触发 ACP 通知，需要兜底处理（已在 `opencode-acp.ts:195-234` 实现） |
| **进程管理复杂** | 子进程可能僵死或异常退出，资源泄漏风险 |

**决策**：不废弃 ACP，而是新增 HTTP API 作为更优的默认选项，ACP 继续保留作为兼容备选。

### 1.2 目标

基于企业实际需求，扩展 Agent 服务层架构：

| 目标 | 说明 |
|:---|:---|
| **双轨支持** | 同时支持 HTTP API 和 ACP 协议，用户可按需选择 |
| **默认 HTTP** | HTTP API 作为默认后端，简化部署和运维 |
| **保留 ACP** | ACP 继续保留，兼容现有生态（Zed IDE 等） |
| **多用户支持** | HTTP 后端支持多用户场景，ACP 用于单用户/本地开发 |
| **统一 API 配置** | HTTP 后端：所有用户共享系统级配置；ACP 后端：保持现有模式 |
| **渐进迁移** | 不破坏现有功能，逐步完善 HTTP 后端能力 |

### 1.3 使用场景

```
✅ 企业内部工具（低并发、多用户）
✅ 所有用户使用相同的大模型 API（统一配置）
✅ 降低开发维护成本
✅ 不希望自研 Agent 能力
```

## 二、技术选型

### 2.1 为什么选择 OpenCode HTTP API？

| 对比项 | ACP 协议方式 | HTTP API 方式 |
|:---|:---|:---|
| **部署方式** | 需要安装 CLI，管理子进程 | 仅需启动 HTTP Server |
| **进程管理** | 复杂（spawn/kill/僵死） | 无，纯 HTTP 调用 |
| **多用户支持** | 不适合 | 适合（会话隔离） |
| **流式响应** | 支持 | 支持（SSE） |
| **文件操作通知** | 支持但有缺陷 | 支持 |
| **维护成本** | 高 | 低 |

### 2.2 OpenCode HTTP API 能力

OpenCode 提供完整的 HTTP API，支持：

| 能力 | 支持情况 | 当前实现状态 |
|:---|:---|:---|
| 会话管理（创建/删除/查询） | ✅ 完整支持 | `opencode-http.ts:48-67` |
| 消息发送与接收 | ✅ 支持同步模式（已实现）/ 异步模式（待实现） | `opencode-http.ts:69-118` |
| SSE 流式响应 | ✅ `/event` 端点（待实现） | 待实现（Phase 2） |
| Provider 配置 | ✅ 系统级配置一次 | 通过环境变量 `OPENCODE_SERVER_URL` |
| 模型指定 | ✅ `POST /session` 创建时传入 `model` 参数 | `opencode-http.ts:52-55` |
| 文件操作 | ✅ 读写/搜索/状态 | 待实现（Phase 2） |
| 权限审批 | ✅ `/session/:id/permissions/:permissionID` | 待实现（Phase 2） |
| MCP 集成 | ✅ 动态添加 MCP Server | 待实现（Phase 3） |

### 2.3 与闭源方案对比

| 方案 | 优点 | 缺点 |
|:---|:---|:---|
| **OpenAI Codex** | 能力最强 | 闭源，按量付费 |
| **Claude Code API** | Claude 原生 | 闭源，绑定 Claude |
| **OpenCode HTTP** | 开源可控，成本低 | 非厂商原生 |

## 三、架构设计

### 3.2 多用户支持机制

#### 核心思路

```
所有用户共享同一套 API 配置（管理员统一配置）
    ↓
用户级别隔离靠 Session 实现
    ↓
每个用户选择自己想用的模型
    ↓
OpenCode Server 单实例，无需多实例隔离
```

#### 用户配置 vs 系统配置

| 配置项 | 级别 | 说明 |
|:---|:---|:---|
| API 端点 | **系统级** | 管理员配置一次，所有用户共用 |
| API Key | **系统级** | 管理员配置一次，所有用户共用 |
| 可用模型列表 | **系统级** | 管理员配置一次 |
| 用户选择模型 | **用户级** | 每个用户选择自己想要的模型 |
| 用户会话 | **用户级** | 每个用户独立的会话 |

#### 请求流程

```
用户 A 发送消息
    ↓
前端 → agent-service → { sessionId: "xxx", message: "帮我写代码" }
    ↓
agent-service 查找会话绑定的模型：
{
  sessionId: "xxx",
  model: "claude-3-5-sonnet-20240620"
}
    ↓
发送消息时指定模型：
POST /session/:id/message
{
  "model": "claude-3-5-sonnet-20240620",
  "parts": [{ "type": "text", "text": "帮我写代码" }]
}
    ↓
OpenCode 使用系统级配置的 API Key 调用对应模型
```

### 3.3 分层职责

| 层级 | 组件 | 职责 |
|:---|:---|:---|
| **消费层** | Web 前端 | 用户交互、后端选择、模型选择、会话管理 UI |
| **网关层** | agent-service | 会话生命周期管理、后端路由（HTTP/ACP）、模型参数注入 |
| **Agent 层 (HTTP)** | OpenCode Server | LLM 调用（系统级配置）、工具执行、多用户隔离 |
| **Agent 层 (ACP)** | OpenCode CLI | LLM 调用（用户级配置）、本地开发、IDE 集成 |

> **当前实现状态**：
> - `OpenCodeHttpBackend` 已实现基础功能（`opencode-http.ts:1-150`）
> - **未在 `server.ts` 中注册**（需 Phase 1 补充）
> - 仅支持同步消息，缺少 SSE 流式、文件追踪、权限审批（需 Phase 2 补充）

### 3.4 后端选择策略

| 场景 | 推荐后端 | 理由 |
|:---|:---|:---|
| **生产环境（多用户）** | `opencode-http` | 无状态、易部署、系统级配置 |
| **本地开发** | `opencode` (ACP) | 轻量、无需启动 Server |
| **IDE 插件集成** | `opencode` (ACP) | 兼容 ACP 生态（Zed 等） |
| **调试/测试** | `opencode` (ACP) | 进程内调试更方便 |
| **高可用部署** | `opencode-http` | 支持负载均衡、健康检查 |

## 四、数据模型

### 4.1 系统 LLM 配置

> **⚠️ 当前系统未实现此类型**。系统级 LLM 配置由 OpenCode Server 环境变量管理。

```typescript
interface SystemLLMConfig {
  providerType: 'anthropic' | 'openai' | 'gemini' | 'custom';
  providerId: string;           // 固定为 "enterprise"
  apiKey: string;                // 加密存储
  apiBase: string;               // API 端点
  models: string[];              // 可用模型列表
  updatedAt: number;
}
```

### 4.2 Session 元数据（扩展）

> **当前实现**：`session-store.ts:4-18` 已定义 `SessionMeta` 接口，包含 `opencodeSessionId` 字段。

```typescript
// 当前实现（session-store.ts）
export interface SessionMeta {
  sessionId: string;             // agent-service 会话 ID
  demoId?: string;               // 绑定的演示 ID
  backend: string;               // 后端类型："opencode" | "opencode-http" | ...
  workingDir: string;            // 工作目录
  customWorkspace: boolean;      // 是否自定义工作空间
  workspaceType: 'user' | 'temp'; // 工作空间类型
  snapshotMode: 'git-repo' | 'snapshot' | null;
  snapshotBranch: string | null;
  status: AgentStatus;
  createdAt: number;
  updatedAt: number;
  messageCount: number;
  opencodeSessionId?: string;    // OpenCode Server 会话 ID（HTTP 后端使用）
}
```

> **注意**：当前 `SessionMeta` 不包含 `modelName` 字段。模型信息通过 `AgentConfig.model` 传递（`core/types.ts:34`），在创建会话时传入 `OpenCodeHttpBackend`（`opencode-http.ts:52-55`）。
>
> **是否需要扩展**：如需记录用户选择的模型，可在 `SessionMeta` 中添加 `modelName?: string` 字段，并在 `session-store.ts:44-57` 中初始化。

### 4.3 存储方案

| 数据 | 存储位置 | 说明 |
|:---|:---|:---|
| 系统 LLM 配置 | OpenCode Server 环境变量/配置文件 | API Key 由 OpenCode 管理 |
| Session 元数据 | agent-service MemorySessionStore | 包含模型和 OpenCode 会话关联 |
| OpenCode 会话 | OpenCode Server 内存 | 由 `opencodeSessionId` 关联 |

> **当前实现**：`MemorySessionStore` 位于 `session-store.ts:38-98`，使用内存 Map 存储。
>
> **注意**：当前 agent-service 使用内存存储（`MemorySessionStore`），重启后会话丢失。生产环境应考虑持久化存储。
>
> **HTTP 后端关联**：`opencode-http.ts:65` 创建 OpenCode 会话后，`sessionId` 存储在 `OpenCodeHttpBackend` 实例中，但**未同步到 `SessionMeta.opencodeSessionId`**。需在 `agent.ts` 或 `backend-agent.ts` 中补充此逻辑。

## 五、方案优势

| 优势 | 说明 |
|:---|:---|
| **极低成本** | OpenCode 免费，仅需 LLM API 费用 |
| **配置简单** | 管理员配置一次，无需逐用户配置 |
| **快速落地** | 开发量小，3 人日可完成 |
| **低维护** | OpenCode 官方维护，单实例部署 |
| **安全可控** | API Key 统一管理，用户不接触 |

