# 对话历史持久化与切换

> 状态：进行中 | 日期：2026-06-04

## 一、背景与问题

当前编辑页（`/demo/[id]/edit`）中，用户与 AI 的对话仅在当前浏览器会话中保留。每次重新打开项目编辑页时，系统总是创建全新会话（Session），之前的对话记录虽然已保存到磁盘（`.messages.json`），但缺少有效的历史管理机制。用户无法回顾或切换到之前的对话。

## 二、需求概述

| 需求 | 说明 |
|---|---|
| 打开项目 | 总是新建对话（保持当前行为） |
| 历史保留 | 每个项目最多保留最近 5 个对话，超出自动删除最旧的 |
| 切换对话 | 可从历史列表切换到之前对话，仅恢复消息内容，不恢复代码/编辑状态 |
| 入口位置 | 聊天消息框底部的「历史对话」按钮（保持现有位置） |

### 3.3 缺失的功能

**唯一缺失**：每个项目下的 Session 数量无上限，不会被自动清理。需要在创建新 Session 时，若项目下 Session 超过 5 个，按创建时间升序删除最旧的。

## 四、方案设计

### 4.1 选定方案：后端清理

在 `POST /api/sessions` 创建新 Session 后，检查 `{userId}/{projectId}/` 目录下 Session 数量。若超过 5 个，按 `createdAt` 排序，删除最旧的（保留最新的 5 个）。

### 4.2 清理时机

每次创建新 Session 时触发（覆盖所有入口）：
- 用户打开编辑页 → `useEffect` 调用 `POST /api/sessions`
- 用户点击「新建对话」→ `onNewSession` 调用 `POST /api/sessions {forceNew: true}`

### 4.3 清理流程

```
POST /api/sessions 创建 Session
        │
        ▼
createEditSession() 成功
        │
        ▼
🆕 enforceSessionLimit(userId, projectId, maxCount=5)
        │
        ├── 读取 sessions/{userId}/{projectId}/ 下所有子目录
        │
        ├── 解析每个 .session.json，收集 {sessionId, createdAt}
        │
        ├── 按 createdAt 升序排序
        │
        ├── 若 count > 5，裁剪前 (count - 5) 个最旧的
        │
        └── 逐个删除旧 Session 目录（含所有文件）
```

### 4.4 删除操作复用现有函数

`deleteSession(sessionId)` 已存在于 `lib/fs-utils.ts:628`，会删除整个 Session 目录。清理逻辑直接复用此函数。

## 五、实施步骤

### Step 1：在 `session-manager.ts` 中添加清理函数

```typescript
// 新增函数
export function enforceSessionLimit(userId: string, projectId: string, maxCount = 5): number {
  const projectSessionsDir = path.join(getSessionsDir(), userId, projectId);
  if (!fs.existsSync(projectSessionsDir)) return 0;

  const sessions: Array<{ sessionId: string; createdAt: number }> = [];
  const entries = fs.readdirSync(projectSessionsDir, { withFileTypes: true });

  for (const entry of entries) {
    if (!entry.isDirectory()) continue;
    const metaPath = path.join(projectSessionsDir, entry.name, '.session.json');
    if (!fs.existsSync(metaPath)) continue;
    try {
      const meta = JSON.parse(fs.readFileSync(metaPath, 'utf-8'));
      sessions.push({ sessionId: meta.sessionId, createdAt: meta.createdAt || 0 });
    } catch { /* skip */ }
  }

  if (sessions.length <= maxCount) return 0;

  // 按创建时间升序，删除最旧的
  sessions.sort((a, b) => a.createdAt - b.createdAt);
  const toDelete = sessions.slice(0, sessions.length - maxCount);

  let deletedCount = 0;
  for (const s of toDelete) {
    try {
      deleteSession(s.sessionId);
      deletedCount++;
    } catch (err) {
      console.error(`Failed to delete old session ${s.sessionId}:`, err);
    }
  }

  return deletedCount;
}
```

## 八、风险与回滚

- **风险极低**：清理逻辑仅在有 6+ Session 时触发，且删除的是最旧的
- **回滚方式**：注释掉 `enforceSessionLimit` 调用即可
- **数据安全**：Session 本身是临时工作副本，删除不影响项目原始数据
