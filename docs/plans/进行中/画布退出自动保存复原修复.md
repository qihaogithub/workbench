# 画布退出自动保存复原修复

## 背景

用户在创作端画布中移动页面位置、添加文本节点后直接退出，重新打开项目发现画布改动复原，说明退出前的画布自动保存没有稳定落盘。

## 目标

- 画布页面位置、文本节点等布局状态在直接退出前必须被保存。
- 重新打开项目后读取到退出前的最新画布状态。
- 保持现有协同保存、自动检查点和历史版本边界不变。

## 范围

- 创作端编辑页退出前 flush 行为。
- `useCanvasWorkspace` 画布状态保存来源。
- 既有 Playwright 回归测试和相关项目文档。

## 方案

- 使用 ref 持有最新 `CanvasState`，避免快速退出时 `flushCanvasState` 读取旧 React state 闭包。
- 编辑页返回首页前先调用画布 flush，由 hook 判断是否存在需要保存的差异；非画布内容存在未保存改动时才用协同 Workspace flush 阻塞退出。
- `/api/sessions/:sessionId/canvas-layout` 保存时同步写入 Session、当前 Workspace 和项目 canonical Workspace，避免直接退出后新建编辑 Session 只从项目 Workspace 拷贝而丢失画布布局。
- 增加 hook 单测覆盖“画布状态更新后立刻 flush”的持久化请求内容。

## 进度记录

- 2026-06-28：确认画布保存同时经过 `.canvas-layout.json` 协同房间和 `/api/sessions/:sessionId/canvas-layout`；退出前已有 flush，但 hook 的强制 flush 以 React state 闭包为保存源，快速退出可能落盘上一版状态。
- 2026-06-28：已将 `useCanvasWorkspace` 的定时保存和强制保存改为读取最新状态 ref，并让编辑页退出前始终调用画布 flush。
- 2026-06-28：目标 E2E 跑到返回步骤时发现协同 flush 异常会先弹出“修改仍在同步”，导致画布 flush 被阻断；已调整为画布 flush 优先，非画布改动才阻塞协同 flush。
- 2026-06-28：继续验证发现画布协同远端布局应用会调用非画布的 `markWorkspaceChanged()`，导致画布改动被误判为工作区文件未保存；已移除该误标记，让画布脏状态只由 `hasUnsavedCanvasChanges` 管理。
- 2026-06-28：Playwright 回归显示布局已写入当次临时 Workspace，但重新打开项目的新 Session 仍读不到；根因是直接退出没有把 `.canvas-layout.json` 同步到项目 canonical Workspace。已将画布布局 API 扩展为同时写项目 Workspace，并增强回归脚本从实际保存响应识别当前 Session，避免多次 Session 创建导致误判。
