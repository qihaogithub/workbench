# 创作端项目编辑页预览区问题沉淀

## 当前状态

2026-07-21 已完成两项结构性修复：统一 React/原型页面在编辑预览与截图链路中的本地图片解析；为实时预览补齐请求生命周期、分阶段超时和普通错误终态。单元测试与类型检查已通过，真实浏览器、Docker 截图和生产诊断时间线仍待专项验收，因此本文继续保留在"进行中"。

## 当前结论

### 问题 A：图片相对路径不会被重写（设计缺陷）

#### 证据

**1. 编译器只处理 import 语句，不处理字符串字面量**

`packages/preview-contract/src/compiler.ts:38-52`：
```
sucrase.transform(TSX → JS)
  → extractImportDeclarations()  // TypeScript AST 只解析 import 声明
  → rewriteImportsWithResolver()  // 只替换 import 的 npm 包路径
```

JSX 属性 `src="../../assets/images/xxx.png"` 是普通字符串字面量，不经过 import 声明提取，原封不动保留在编译产物中。

**2. 编译 API 中有空的路径重写占位**

`packages/author-site/src/app/api/compile/route.ts:81-83`：
```ts
if (result && sessionId && typeof sessionId === 'string') {
  // 图片已通过图床绝对 URL 直接访问，无需路径重写
}
```
这个 if 块体为空，注释说明系统设计期望是图床绝对 URL，但实际 AI 生成代码使用的是相对路径。

**3. 路径重写函数已实现但未接入**

`packages/author-site/src/lib/rewrite-local-paths.ts` 实现了 `rewriteLocalAssetPaths(code, basePath, sessionId)`，覆盖单引号、双引号、模板字符串、CSS url() 等所有路径模式，有 14 个单元测试。grep 全仓确认：生产代码中无任何调用点。

**4. saveImage 工具是指令来源：主动引导 AI 使用相对路径**

`packages/agent-service/src/backends/pi-tools/save-image-tool.ts` 是 AI 获取图片的唯一入口。工具在保存图片后，返回给 AI 的引用格式为 `../../assets/images/{hash}-{filename}`：

```ts
// save-image-tool.ts:74
function getDemoRelativeAssetPath(storedFilename: string): string {
  return `../../assets/images/${storedFilename}`;
}

// 工具返回消息（line 722-723）:
"Image saved locally: assets/images/xxx.png. From page files use ../../assets/images/xxx.png"
```

工具描述 `line 514-515` 同样指示：`Use ../../assets/images/{hash}-{filename} from files inside demos/{pageId}/`。

同时，工具持有 `config.sessionId`（line 708、648、359），完全有能力返回可直接访问的工作空间 API URL（如 `/api/sessions/{sid}/workspace/assets/images/{filename}`），但当前未提供此选项。

这意味着：**即使编译管线修复了路径重写，saveImage 工具仍在主动生成无法在浏览器中直接解析的路径格式**。理想的根治应同时更新工具，使其返回可解析路径，编译管线重写作为兜底。

**5. 截图服务的编译客户端未传递 demoId**

`packages/screenshot-service/src/utils/compile-client.ts:34-49` 发送 `{code, sessionId}` 但未携带 `demoId`。截图路由 `screenshots.ts` 在调用时持有 `pageId`（即 demoId），但未传入编译客户端。

**6. 浏览器端路径解析必然 404**

编译产物保留 `../../assets/images/xxx.png`。浏览器解析：

| 场景 | 解析结果 | 对端 |
|------|---------|------|
| 编辑态预览（无 `<base>`） | `../../` → `/assets/images/xxx.png` | 无此路由 |
| 截图预览（有 `<base href="author-site">`） | `../../` → `author-site/assets/images/xxx.png` | 无此路由 |

实际图片服务：`/api/sessions/{sessionId}/workspace/assets/images/xxx.png`（`[...path]/route.ts`，有三级回退查找）。

#### 根因

编译链路缺少对 JSX/HTML 中图片相对路径的字符串字面量替换。`rewriteLocalAssetPaths` 已完备但从未接入编译管线。截图服务侧缺少 `demoId` 导致无法进行上下文感知的路径解析。

### 问题 B：预览模块 fetch 失败（运行时）

#### 证据

`captureScreenshot` 多次返回：
```
dependency_import: Failed to fetch dynamically imported module:
http://author-site:3200/api/preview-modules/{hash}.js
```

修改源码后 hash 会变化，证明编译成功且 module 被注册。但 Puppeteer 浏览器 fetch 时 module 不可达。

#### 分析

截图服务调用 `POST /api/compile {code, sessionId}` → 命中 inline-code 分支 → `compileCode()` → `registerPreviewModule(hash, code)` 写入内存 Map + 文件系统。Puppeteer 通过 Docker 网络请求同一模块。可能原因：Docker Compose 多副本时编译和注册在实例 A，Puppeteer 请求命中实例 B；或 module TTL 过期（30 分钟）。

### 根治方案

#### 设计原则

1. **Agent 无感**：Agent 继续使用 `../../assets/images/{hash}-{filename}`，系统自动解析
2. **页面类型无关**：React 和 HTML 原型页使用同一套路径解析
3. **编译时重写**：在编译管线中完成路径替换，编译产物包含浏览器可直接访问的绝对 API 路径
4. **缓存安全**：`{ ...result }` 展开避免污染 `compileCode` 的 Map 缓存

#### 改动 1：编译 API 接入路径重写（核心）

**文件**：`packages/author-site/src/app/api/compile/route.ts`

替换空 if 块（line 81-83）：

```ts
import { rewriteLocalAssetPaths } from '@/lib/rewrite-local-paths';

// 替换原 line 81-83
if (result && sessionId && typeof sessionId === 'string' && demoId && typeof demoId === 'string') {
  const rewritten = rewriteLocalAssetPaths(
    result.compiledCode,
    `demos/${demoId}/`,
    sessionId,
  );
  if (rewritten !== result.compiledCode) {
    result = {
      ...result,
      compiledCode: rewritten,
      moduleHash: createHash('sha256').update(rewritten).digest('hex'),
    };
    result.moduleUrl = `/api/preview-modules/${result.moduleHash}.js`;
  }
}

registerPreviewModule(result.moduleHash, result.compiledCode);
```

`compileCode()` 内部有 `compileCache`（Map），返回的是缓存引用。用 `{ ...result }` 展开新对象确保不污染缓存。sucrase 编译开销由缓存节省，路径重写是纯字符串 replace，开销可忽略。

#### 改动 2：截图服务编译客户端传递 demoId

**文件**：`packages/screenshot-service/src/utils/compile-client.ts`

```ts
export async function compileCode(
  code: string,
  sessionId?: string,
  demoId?: string,  // 新增
): Promise<CompileResult> {
  const body: Record<string, unknown> = { code };
  if (sessionId) body.sessionId = sessionId;
  if (demoId) body.demoId = demoId;  // 新增
  // ...
}
```

#### 改动 3：截图路由传入 pageId

**文件**：`packages/screenshot-service/src/routes/screenshots.ts`

调用编译客户端时传入 `pageId`：

```ts
const compileResult = await compileCode(
  snapshotInput.code,
  snapshotInput.sessionId,
  snapshotInput.pageId,  // 新增
);
```

#### 不改动的部分

- **AI 提示词**：保持 `../../assets/images/{hash}-{filename}` 模式
- **HTML 原型页**：`rewritePrototypeAssetUrls` 已正确处理 `../../` 路径
- **`screenshot-tool.ts` 的 base64 内联**：保留作为截图服务独立进程兜底
- **`rewrite-local-paths.ts`**：复用现有实现，无需修改

#### 补充：saveImage 工具可同步增强

`saveImage` 工具（`save-image-tool.ts:74`）当前只返回 `../../assets/images/{filename}` 相对路径。工具持有 `config.sessionId`，可同步返回 `/api/sessions/{sid}/workspace/assets/images/{filename}` 作为备选引用格式，让 AI 在代码中直接使用可解析路径。这在编译管线重写之外提供第二层保障，但非必需——编译时重写已覆盖所有来源的相对路径。

#### 兼容性

| 场景 | 改动前 | 改动后 |
|------|--------|--------|
| React 编辑预览 | `../../assets/...` → 浏览器 404 | → `/api/sessions/{sid}/workspace/...` ✅ |
| React 截图预览 | 同上 404 | 路径正确 + demoId 传递 ✅ |
| HTML 原型截图 | `rewritePrototypeAssetUrls` 已处理 | 无变化 ✅ |
| 编译缓存 | 路径重写在缓存返回后执行，展开避免污染 | 缓存仍节省 sucrase 开销 ✅ |

#### 剩余风险

1. **多实例部署**：Docker 多副本时 `preview-module-store` 文件系统持久化可兜底内存 Map，需确认 `data/preview-modules/` 多容器共享
2. **`demoId` 缺失降级**：请求中无 `demoId` 时路径不重写，图片仍 404
3. **类型检查**：`compile-client.ts` 增加参数后调用方需同步更新，TS 类型检查会捕获

#### 验证

1. React 编辑预览：图片正常显示
2. React 截图：`captureScreenshot` 返回正常 PNG
3. HTML 原型截图：回归确认不受影响
4. 编译缓存：连续两次编译同一页面，hash 一致
5. 路径格式：`curl` 验证重写后的 workspace 端点可获取图片
