# 创作端项目编辑页预览区问题沉淀

## 当前状态

历史补丁已经消除“收到明确编译/运行错误后仍显示 loading”的一个症状，但问题尚未完整关闭。当前仍缺少统一的预览请求终态、无回执超时、普通错误展示、最近成功预览可见性和单页/画布一致性，因此本条从“已完成”恢复为进行中。

## P4：预览失败后的加载态与错误终态

### 现象与复现页

- 复现项目：`proj_1783334663364_30hsrv`。
- `product-showcase_a3f2` 曾因顶层声明 `features` 重复而无法通过页面运行契约校验；同项目的正常页面可继续预览。
- 用户侧曾长期看到“正在加载页面预览”，无法判断系统是在编译、等待 iframe、自动修复，还是已经失败。

### 诊断证据

2026-07-21 使用结构化诊断入口复核该项目，结果为：

- SQLite 主账本可用；同时读取到 JSONL fallback/spool；未检测到 event gap。
- 2026-07-19 的事件明确记录 `preview.error`，阶段为 `module_parse`，错误为顶层声明 `features` 重复，随后记录 `compile_error`。
- 同一时段也有正常的 `compile_done`、`iframe_ready` 和 `preview.content_loaded`，说明编译、iframe shell 和页面加载是可区分的阶段，不应合并成单个 `loading` 布尔值。

复现页无法预览的页面内容根因，是页面源码被重复拼接；完整链路见 [创作端编辑页-AI自动修复循环与WMA外部漂移](../已完成/创作端编辑页-AI自动修复循环与WMA外部漂移.md)。预览区永久 loading/空白则是独立的状态呈现缺陷，不能把两者写成同一个根因。

### 已确认的直接缺陷

`packages/demo-ui/src/PreviewPanel.tsx` 当前同时维护 `isCompiling`、`contentLoaded`、`compileError`、`runtimeError`、`lastSuccessfulResult`、`pendingCompileResult` 等互相耦合的状态。

历史实现用“有源码，并且正在编译或尚未收到 `LOADED`”推导 loading。编译失败时源码仍存在、`contentLoaded` 仍为 false，因此 spinner 不会结束。2026-07-14 的补丁 `0f6dc28b` 为 loading 条件增加了 `compileError` / `runtimeError` 排除项，明确错误回执到达后 spinner 会消失。

这个改动是有效的症状级修复，但不是完整根因修复：

1. `PreviewPanel` 普通错误状态没有独立错误卡片。只有父页面处于 `isAutoRepairing` 时，组件才显示“正在修复预览”。所以历史文档中的“loading 遮挡错误信息”不准确；实际情况是错误信息从未在预览区渲染，移除 spinner 后可能只剩空白。
2. 每次新源码请求开始都会先把 `contentLoaded` 清为 false。组件虽然保留 `lastSuccessfulResult`，iframe 也可能仍承载上一次成功内容，但渲染层用 `contentLoaded` 控制 iframe 透明度，失败后最近成功预览并不保证可见。
3. 当前没有覆盖编译请求、shell `READY` 或页面 `LOADED` 的终态超时。如果请求挂起、shell 未启动、消息丢失或运行时没有返回明确错误，`compileError` / `runtimeError` 都不会出现，spinner 仍可能永久显示。
4. 画布还在 `CanvasPageItem` 中维护截图加载、iframe 加载和 render mode。单页内部 loading 修复不能自动保证画布外层截图/iframe/loading 切换具有相同终态。
5. 有截图占位时，内部 loading 本来就不会显示；因此“所有编译失败页面、单页和画布均以相同方式受影响”的旧结论过宽。截图是否可作为兜底，还必须受当前内容 hash 一致性约束。
6. 共享诊断契约已经声明编译开始、成功、失败和运行时错误等事件，但当前页面主要生产通用 `preview.runtime_event` 与 `preview.error`。错误事件缺少稳定的 requestId 关联，尚不能直接验证“一次请求恰好有一个终态”。

### 根因重定位

长期根因不是某一个条件少了 `!compileError`，而是预览请求生命周期和用户可见内容没有明确建模：

- `contentLoaded` 同时承担“当前请求是否完成”和“iframe 是否可见”两个职责。
- loading 由多个布尔值反向推导，没有“失败”“超时”“正在修复”“展示最近成功结果”等互斥终态。
- 当前请求状态、可见兜底产物、AI 修复状态分散在 `PreviewPanel`、编辑页和 `CanvasPageItem`，各层可独立进入不一致状态。
- 异步流程只依赖成功/显式失败回执，没有保证每个请求最终进入成功、失败、取消或超时之一。
- 诊断事件命名与实际生产路径没有收敛到同一请求生命周期，导致运行日志能看到阶段片段，却难以自动判定缺失的终态。

因此，布尔条件补丁只能处理已经收到错误回执的一条路径，无法从结构上消除永久 loading 和错误后空白。

## 长期方案

### 1. 分离三个正交状态

预览状态应由一个纯 reducer 统一管理，而不是继续叠加布尔条件：

| 状态维度 | 建议状态 | 职责 |
| --- | --- | --- |
| 当前请求生命周期 | `idle`、`compiling`、`waiting-shell`、`rendering`、`ready`、`failed`、`timed-out`、`cancelled` | 描述当前 requestId 的真实进度与唯一终态 |
| 可见产物 | 当前实时预览、同页最近成功预览、当前 hash 有效截图、无兜底 | 决定用户此刻看什么，不再由 `contentLoaded` 间接控制透明度 |
| 修复生命周期 | `idle`、`scheduled`、`running`、`succeeded`、`exhausted` | 只描述 AI 修复，不冒充预览请求状态 |

所有编译、iframe 和修复事件都必须携带页面 ID 与 requestId。reducer 只接受当前请求事件，旧请求晚到的 `LOADED`、错误和尺寸消息统一丢弃。

### 2. 以呈现选择器统一 UI

由单一 presentation selector 决定单页和画布显示内容：

- 当前请求 `ready`：显示当前实时预览。
- 请求进行中且有同页最近成功预览或当前 hash 有效截图：保持兜底可见，仅显示非阻塞的“正在更新”状态。
- 请求失败且自动修复运行中：保持兜底可见；没有兜底时显示友好的修复中面板。
- 请求失败且修复未触发、失败或预算耗尽：显示可操作的错误终态，至少提供摘要、重试和查看诊断入口；不能显示 spinner 或空白。
- 没有源码：显示明确空状态，不创建请求，也不占用 requestId。

最近成功预览必须按页面保存，并与“当前请求是否加载完成”解耦。截图只能在内容 hash 与当前页面输入一致时作为兜底，禁止用过期截图掩盖当前内容。

### 3. 为异步边界补齐终态协议

为三段流程分别设置可观测的 deadline：

- 编译阶段：请求可取消；超时后进入 `timed-out`，记录编译服务阶段和 requestId。
- shell 阶段：iframe 创建后未收到 `READY`，进入 shell 超时，而不是继续等待页面 `LOADED`。
- 渲染阶段：发送模块后未收到 `LOADED` 或 `RUNTIME_ERROR`，进入 render 超时，并保留最后一个 timing stage 作为诊断依据。

超时不是立即判定页面代码错误；它应生成结构化诊断，区分服务、shell、模块网络和页面运行时，交给现有有界自动修复/人工重试流程。

### 4. 收敛组件职责

- `PreviewPanel` 负责请求 reducer、iframe 协议和可见产物选择，不负责决定 AI 是否继续修复。
- 编辑页负责诊断落账、修复预算和对话任务，把修复状态显式传回预览组件。
- `CanvasPageItem` 复用同一 presentation model，只负责画布调度和尺寸，不再另建一套成功/失败判断。
- 诊断事件补齐 requestId、页面 ID、开始阶段、终态、兜底类型和 deadline，确保 `diagnostics:preview` 能还原一次请求为何结束。

### 5. 分阶段落地

1. 先提取纯状态模型和 presentation selector，用表驱动测试覆盖所有状态组合，不改变 UI。
2. 将 `PreviewPanel` 的编译、`READY`、`LOADED`、`RUNTIME_ERROR` 和取消事件接入 reducer，补齐三段超时。
3. 让 iframe 可见性只由 selector 控制，真正实现最近成功预览/有效截图兜底和终态错误面板。
4. 让 `CanvasPageItem` 消费同一模型，消除单页与画布的重复 loading 判定。
5. 最后接入结构化诊断与浏览器 E2E，验证自动修复成功、预算耗尽、网络挂起和快速切页。

## 验收矩阵

| 场景 | 预期结果 |
| --- | --- |
| 首次编译成功 | loading 最终切换到当前实时预览 |
| 首次编译明确失败 | 显示修复中或可操作错误终态，不空白、不永久 loading |
| 已有成功预览后新代码失败 | 最近成功预览继续可见，并明确提示当前更新失败 |
| shell 不返回 `READY` | deadline 后进入 shell 超时并记录结构化诊断 |
| 模块不返回 `LOADED` / `RUNTIME_ERROR` | deadline 后进入 render 超时，不永久 loading |
| 自动修复成功 | 新 requestId 成功后原子切换到当前预览 |
| 自动修复预算耗尽 | 显示终态错误和人工重试入口 |
| 快速切页、旧消息晚到 | 旧 requestId 不改变当前页面状态或可见内容 |
| 当前 hash 有效截图存在 | 加载/失败期间可作为兜底；hash 不一致时不得显示 |
| 画布休眠/唤醒 | 与单页使用相同终态语义，不产生第二套永久 loading |
