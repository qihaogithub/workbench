## 发现的问题

### 🔴 P0：saveImage 工具返回结果极度冗余

**现象**：每个 saveImage 调用返回约 80-100 行 JSON，其中包含完整的 `receipt`（mutationId、rootHash、baseRevision、revision、actor、resources[]、committedAt）、`sha256`、`absolutePath` 等元数据。且 `details` 对象在 `result.details` 和顶层 `details` 中被**完整复制了两份**。

**影响**：本次对话约 15 次 saveImage 调用，产生约 1200+ 行 / 数万 token 的纯冗余上下文。模型实际只需要 `relativePathFromPage` 来更新 HTML。

**证据**：
- 行 121-200：单次 saveImage 结果占 80 行，其中有效信息仅 `relativePathFromPage: "../../assets/images/e3d84d41a35b-kingkong-1.png"` 一行
- 所有 saveImage 结果中 `result.details` 和顶层 `details` 内容完全一致，100% 重复

**建议**：
1. 工具返回给模型的结果精简为 `{ relativePathFromPage, filename, size, error? }` 即可
2. 完整 receipt/mutation 数据只保留在内部日志和事件流中，不回传给模型
3. 去掉 `result.details` 和顶层 `details` 的重复结构

### 🔴 P0：大文件仍使用 writeFile 完整输出

**现象**：AI 使用 `writeFile` 完整输出了 prototype.html（18KB）和 prototype.css（10KB），逐字搬运了整个文件内容。项目已明确 editFile 范式是既定策略，writeFile 搬运模式是已修复的拼接污染根因。

**影响**：
- 输出 token 极度浪费（28KB 文件内容搬运）
- 存在拼接污染回退风险（虽然本次未出错，但模式本身就是风险）

**证据**：
- 行 1112-1198：writeFile prototype.html，content 参数 18236 字节
- 行 1240-1327：writeFile prototype.css，content 参数 10255 字节

**建议**：
1. 对大文件（>2KB）在系统提示或工具层面引导/强制使用 editFile
2. 本次实际只需 10-15 处局部替换（占位 div → img 标签），editFile 的 old_string/new_string 模式完全适用
3. 可在 writeFile 工具返回中增加 warning 提示模型优先使用 editFile

### 🔴 P1：readFile 返回了含重复内容的完整大文件

**现象**：prototype.html 存在内容重复（整个 HTML 出现两遍，共 34KB），readFile 将全部内容返回给模型。模型在 reasoning 中注意到了重复并做了清理，但读取时已浪费大量输入 token。

**影响**：34KB 的输入中约 17KB 是重复内容，模型额外消耗 token 处理重复数据。

**证据**：
- 行 1084-1096：readFile 返回 34014 字节，HTML 内容从 `<!DOCTYPE html>` 开始出现了两次
- prototype.css 同样存在重复（19762 字节，内容出现两遍）

**建议**：
1. readFile 工具对超大文件（>10KB）增加截断或摘要机制
2. 可检测文件内容重复并在返回结果中提示模型
3. 或在返回大文件时只返回前 N 行 + 总行数提示，让模型按需读取

### 🟡 P2：同一图片 URL 重复下载

**现象**：`h_b90b54ac.png` 第一次下载失败（TLS 错误），重试时成功保存为 kingkong-5.png。但 `be63de2fde89-kingkong-7.png`（仅 149 字节，极可能是 1×1 占位图或无效图片）被复用给了 4 个不同的金刚区图标（周周分享、福利社区、直播预约、许个愿望），视觉效果可能不符合预期。

**影响**：
- 浪费一次失败的网络请求
- 4 个不同功能图标使用同一张图，可能不是用户期望的效果

**建议**：
1. saveImage 工具应检测相同 URL 的重复下载，直接返回已保存路径
2. 对极小文件（<500 字节）返回 warning 提示可能是占位图

### 🟡 P2：碎片化 reasoning 浪费 token

**现象**：模型有 7-8 个连续的碎片化 reasoning 步骤，每个只有 1-3 句话，完全可以合并。

**证据**：
- 行 43-119：URL 提取被拆成 3 段 reasoning（"Let me first read..." → URL 列表第一段 → URL 列表第二段）
- 行 1052-1073：更新 HTML 的计划被拆成 4 段 reasoning

**建议**：模型或系统层面应鼓励合并相关推理步骤，减少 turn 内的 reasoning 碎片。

### 🟢 P3：无意义的验证性读取

**现象**：对话末尾分别读取了 HTML 和 CSS 的前 5 行，仅为了"验证文件写入正确"。writeFile 已经返回了成功状态和 runtimeValidation。

**建议**：在系统提示中引导模型信任工具返回的成功状态，避免冗余验证。
