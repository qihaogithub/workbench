# 预览区不更新（闪烁后回退）：根因分析与根治方案

> 创建时间：2026-06-03
> 分析范围：AI 完成 → 代码/schema 更新 → PreviewPanel 编译 → iframe 渲染的完整链路
> 当前阶段：根因分析完成，根治方案已出，待实现

---

## 1. 问题描述

AI 编辑代码文件后，预览区**先闪烁一下新内容，然后变回旧内容**。必须「保存后重新打开页面」才能看到正确的效果。

此问题已反复出现、修复多次，但始终未根除。

---

## 2. 完整数据流梳理

### 2.1 文件变更的三条路径

AI 编辑后，代码/schema 可以从**三条独立路径**到达 React 状态：

```
路径 A: 实时流式文件 → onFileOperation → realtimeFilesRef → processRealtimeFiles() → onCodeUpdate
路径 B: 后端 finish 事件 → result.files → extractCodeAndSchemaUpdates() → onCodeUpdate
路径 C: HTTP 兜底 → fetchSessionFiles() → onCodeUpdate
```

这三条路径的**时序和内容不一致**，是根本问题所在。

### 2.2 onFinish 完整执行序列

```
use-chat-stream.ts:onFinish (async)
│
├─ ① setMessages / setCurrentMessage / setStreamContent
│
├─ ② await persistMessages(sessionId, ...)   ← 异步断点 1
├─ ③ await updateSessionTitle(sessionId, ...) ← 异步断点 2
│
├─ ④ processRealtimeFiles()
│     → onCodeUpdate(code_from_realtime)      ← 路径 A
│     → applyDemoSnapshot → setCode + snapshotVersion++
│
├─ ⑤ extractCodeAndSchemaUpdates(finalFiles)
│     → onCodeUpdate(code_from_final)         ← 路径 B
│     → 返回 { codeUpdated, schemaUpdated }
│
├─ ⑥ setIsStreaming(false)
│
├─ ⑦ if (!codeUpdated || !schemaUpdated)
│     → await fetchSessionFiles(...)          ← 异步断点 3
│     → onCodeUpdate(code_from_disk)          ← 路径 C ⚠️
│
├─ ⑧ onSnapshotReady()
│     → setSnapshotVersion(v => v + 1)
│
└─ ⑨ realtimeFilesRef.clear()
```

### 2.3 PreviewPanel 编译触发条件

```typescript
// PreviewPanel.tsx:263-348
useEffect(() => {
    const body = sessionId
        ? { sessionId, code }        // ← 发送 code 到 compile API
        : { code };

    fetch("/api/compile", {
        method: "POST",
        body: JSON.stringify(body),
    });
    // ...
}, [code, sessionId, demoId, snapshotVersion, /* ... */]);
```

依赖数组中同时有 `code` 和 `snapshotVersion`。这意味着**每次 `snapshotVersion` 自增，即使 code 不变，也会触发一次编译**。

## 3. 「闪烁后回退」的精确时序分析

用户描述的症状：**预览区先短暂出现新内容（闪烁），然后恢复为旧内容（回退）**。

这个精确症状揭示了以下执行时序：

### 阶段1：闪烁（流式 debounce 期间）

```
AI 流式写文件 →
  T1: onFileOperation → realtimeFilesRef.set("index.tsx", code_final)
  T2: 300ms debounce 到期 → processRealtimeFiles()
      → onCodeUpdate(code_final) → applyDemoSnapshot({code: code_final})
      → setCode(code_final), setSnapshotVersion(v→v+1)
      → React 渲染 → PreviewPanel 编译 code_final → iframe 显示新内容 ✓👁
```

预览区已正确显示了最终代码（但因为 `result.files` 可能不完整，后面会被覆盖）。

### 阶段2：回退（onFinish 中 HTTP 兜底）

```
  T3: onFinish 触发
  T4: await persistMessages + updateSessionTitle  (异步断点)
  T5: processRealtimeFiles() → onCodeUpdate(code_final)  (冗余，代码不变)
  T6: finalFiles = result.files (后端 finish 事件文件列表)
      → result.files 不包含代码文件（例如仅含 schema）
      → extractCodeAndSchemaUpdates(finalFiles) → codeUpdated = false

  T7: !codeUpdated → HTTP 兜底触发
      → await fetchSessionFiles(sessionId, demoId)
      → 从 Workspace 文件系统读取
      → 如果 agent 的 finish 事件与文件写入之间存在时序间隙
        （或 agent 写入路径与 session 读取路径不同）
      → 读取到旧代码 code_old
      → onCodeUpdate(code_old)
      → setCode(code_old), setSnapshotVersion(v→v+1)
      → React 渲染 → PreviewPanel 编译 code_old → iframe 显示旧内容 ✗👁
```

### 为什么 `result.files` 会缺少代码文件？

后端 agent 会在 finish 事件中发送 `files` 数组。但 finish 事件的 `files` 与流式推送的 `onFileOperation` 事件的覆盖范围**可能不同**：
- 流式推送捕获**所有**文件写入操作（通过 WebSocket 实时推送）
- finish 事件的 `files` 可能只包含**本轮修改**的文件
- 如果代码文件在之前的流式批处理中已推送，finish 事件可能**不再包含**它
- 这导致 `extractCodeAndSchemaUpdates(finalFiles)` 返回 `codeUpdated: false`，触发 HTTP 兜底

## 4. 根因分析

### 4.1 根因 A（P0）：三条独立路径导致「新代码被旧数据覆盖」← 主罪

**核心问题：** `onFinish` 中存在**三条独立的代码更新路径**，彼此无感知、时序不同、数据可能不一致：

| 路径 | 来源 | 触发位置 | 数据新鲜度 |
|------|------|----------|-----------|
| A | `realtimeFilesRef` | `processRealtimeFiles()` | ✅ 流式实时捕获 |
| B | `result.files` | `extractCodeAndSchemaUpdates()` | ⚠️ 可能不完整 |
| C | `fetchSessionFiles()` | HTTP 兜底 | ❌ 最不可靠 |

**问题机制：**
1. 路径 A 已正确更新代码（`codeUpdated` 来自实时流）
2. 路径 B 处理 `result.files`，若不含代码则 `codeUpdated = false`
3. 路径 C **无条件信任**文件系统数据，且 `!codeUpdated` 条件未考虑路径 A 已完成的更新
4. 路径 C 从 `fetchSessionFiles` 读取到旧代码 → **覆盖 React 状态中的新代码** → 预览回退

**关键代码：**
- `use-chat-stream.ts:363-387` — `!codeUpdated || !schemaUpdated` 条件 + HTTP 兜底
- `use-chat-stream.ts:325-330` — `processRealtimeFiles()` 与 `extractCodeAndSchemaUpdates()` 独立调用
- `message-service.ts:46-76` — `fetchSessionFiles` 依赖 Session API 文件同步

### 4.2 根因 B（P0）：snapshotVersion 被连加 2~4 次 → ConfigForm 强制重建

**触发路径：**
1. `processRealtimeFiles()` → `applyDemoSnapshot()` → `snapshotVersion +1`
2. `extractCodeAndSchemaUpdates()` → `applyDemoSnapshot()` → `snapshotVersion +1`
3. HTTP 兜底 → `applyDemoSnapshot()` → `snapshotVersion +1`
4. `onSnapshotReady()` → `snapshotVersion +1`

**影响：**

1. **PreviewPanel 多次编译：** 每次 `snapshotVersion` 变化（即使 code 不变），依赖数组 `[code, ..., snapshotVersion]` 触发了 PreviewPanel 编译 effect。一次 AI 完成触发 2~4 次编译，前几次被 cancelled 清理。

2. **ConfigForm 强制重建：** ConfigForm 的 key 为 `${activeDemoId}-${snapshotVersion}`（`page.tsx:1615` 和 `page.tsx:1630`）。每次 `snapshotVersion` 变化都销毁并重建 ConfigForm 组件。虽然 ConfigForm 在 mount 时不会主动调用 `onChange`（不会直接导致数据回写），但会造成：
   - 用户正在编辑的配置值丢失（表单重置为 `initialData`）
   - 折叠/展开状态全部重置
   - 用户正在输入的内容被清空

**代码位置：**
- `page.tsx:232` — `applyDemoSnapshot` 内部 `snapshotVersion++`
- `page.tsx:931` — `onSnapshotReady` 额外 `snapshotVersion++`
- `page.tsx:1606` — ConfigForm key = `${activeDemoId}-${snapshotVersion}`

### 4.3 根因 C（P1）：实时文件与 final 文件重复处理

即使代码内容完全相同，步骤 ④ 和步骤 ⑤ 各调用一次 `onCodeUpdate`。

- 步骤 ④ 处理 `realtimeFilesRef`（流式会话期间累积的文件）
- 步骤 ⑤ 处理 `finalFiles`（`result.files` 或 fallback 到 `realtimeFilesRef`）

当 `result.files` 为空时：
- `finalFiles` fallback 到 `realtimeFilesRef`
- 第 ④、⑤ 两步处理**完全相同的文件列表**
- 重复调用 `onCodeUpdate`，代码 setter 使用 functional updater 避免重复写入，但 `snapshotVersion` 被额外 +1

**代码位置：** `use-chat-stream.ts:332-342`

### 4.4 根因 D（P2）：applyDemoSnapshot 依赖缺失 activeDemoId

```typescript
// page.tsx:188-243
const applyDemoSnapshot = useCallback(
    (params) => {
        if (sessionId && activeDemoId) {     // ← 闭包中的 activeDemoId
            invalidateCompileCache(sessionId, activeDemoId);
        }
        // ...
    },
    [code, schema, sessionId],               // ← 缺少 activeDemoId
);
```

当用户在 AI 请求进行期间切换了页面，`activeDemoId` 已变化，但 `applyDemoSnapshot` 闭包中的 `activeDemoId` 仍是旧值。`invalidateCompileCache` 清理的是旧页面的编译缓存，新页面的缓存未被清理。

同时，`configDataMap` 更新时使用了 `activeDemoIdRef.current`（行 210），这个 ref 总是最新的，与 `invalidateCompileCache` 中使用的闭包 `activeDemoId` 不一致。

### 4.5 根因 E（P1）：mergeConfigWithUserValues 遍历范围缺失

原报告已详述。当 AI 修改 schema 后，用户的非默认字段值可能丢失，导致配置面板回传的 config 与预期不同。这间接影响预览表现。

### 4.6 根因 F（P0）：「代码」Tab 编辑保存后未更新预览状态

**触发路径：** 用户在「代码」Tab 中直接编辑 demo 页面代码文件（`index.tsx`）或 schema 文件（`config.schema.json`），点击保存后，内容写入了文件系统，但 React 状态（`code` / `schema` / `snapshotVersion`）完全未更新。

**当前 `onSave` 回调**（`page.tsx:1673-1687`）：

```typescript
onSave={async (content) => {
    // 写入文件系统 ✅
    const res = await fetch(`/api/sessions/${sessionId}/workspace/files/...`, {
        method: "PUT",
        body: JSON.stringify({ content }),
    });
    // ❌ 缺少：更新 React 状态 (code / schema / snapshotVersion)
    // ❌ 缺少：通知 PreviewPanel 重新编译
}}
```

**对比 AI 路径**（正常工作的路径）：

```typescript
handleCodeUpdate(newCode) → applyDemoSnapshot({
    code: newCode,           // → setCode() → PreviewPanel 重新编译
    source: "ai-realtime"
})
```

**为什么预览不更新：**

1. PreviewPanel 编译 effect 依赖 `[code, ..., snapshotVersion]`
2. 代码 Tab 保存只写了文件系统，React 状态中的 `code` / `schema` / `snapshotVersion` 均未变化
3. PreviewPanel 的 props 无变化 → 编译 effect 不触发 → 预览区保持旧内容

**可编辑文件白名单**（`workspace-file-utils.ts:10-15`）：
- `demos/{demoId}/index.tsx` — Demo 页面组件代码
- `demos/{demoId}/config.schema.json` — Demo 页面配置 Schema
- `project.config.schema.json` — 项目级配置
- `memory.md` — AI 记忆文件

其中前两项的修改**必须同步更新预览状态**。

**代码位置：**
- `page.tsx:1673-1687` — `onSave` 回调（问题位置）
- `page.tsx:753-758` — `handleCodeUpdate`（对比：正常工作的路径）
- `page.tsx:188-243` — `applyDemoSnapshot`（应使用的统一入口）
- `workspace-file-utils.ts:10-15` — 可编辑文件白名单

## 5. 为什么反复出现

### 5.1 修复了「症状」而非「根因」

历史修复（推测）可能是：
- 加了 HTTP 兜底（`fetchSessionFiles`）→ 解决了「代码没更新」但引入了「覆盖新代码」
- 加了 `snapshotVersion` → 解决了「编译不触发」但引入了「多次编译+ConfigForm 重建」
- 加了 `activeDemoIdRef` → 解决了部分闭包问题但没统一所有引用

每次修复都在**打补丁**，而非重新审视架构。

### 5.2 缺少单一真源（Single Source of Truth）

当前架构下，代码更新有**三个来源**，且彼此独立、时序未知：
- 路径 A：realtimeFilesRef（流式期间捕获）
- 路径 B：result.files（后端 finish 事件）
- 路径 C：fetchSessionFiles（文件系统）

任何一个来源的数据偏离都会导致不一致。正确做法是**只有一个来源**。

### 5.3 snapshotVersion 语义过载

一个计数器驱动了两个不同关注点：
1. PreviewPanel 编译重触发
2. ConfigForm 强制重建

这两者应该解耦。

## 5½. 现行方案的批判性审视

以下为对上述根治方案的自我批判，确保方案具有长期可维护性。

### 5½.1 方案六：onSave 回调承载了过多业务逻辑

当前提案在 `WorkspaceCodeDialog` 的 `onSave` 回调中直接写路径匹配和 `applyDemoSnapshot` 调用：

```typescript
onSave={async (content) => {
    // 写文件系统...
    const normalizedPath = filePath.replace(/^\/+/, "");
    if (normalizedPath === `demos/${activeDemoId}/index.tsx` || ...) {
        applyDemoSnapshot({ ... });
    }
}}
```

**问题：** `WorkspaceCodeDialog` 是一个通用文件编辑组件，它的 `onSave` 回调只应该表达「文件保存完毕」。现在它知道了 demo 路径格式、`activeDemoId`、`applyDemoSnapshot`——变成了迷你控制器。

**改进方向：** `onSave` 只负责文件系统写入 + 返回元数据（`{ path, content, demoId?, fileType? }`），由调用方 page.tsx 在 `onSave` **完成之后**，通过一个独立的 `handleWorkspaceFileSaved` 函数决策是否需要更新预览。这样：

```typescript
// 改进后：onSave 保持简洁
onSave={async (content) => {
    // 只做文件系统写入
    await saveWorkspaceFile(sessionId, filePath, content);
}}

// page.tsx 中用独立的 handler 响应保存完成
const handleWorkspaceFileSaved = useCallback((filePath: string, content: string) => {
    const demoId = extractDemoIdFromPath(filePath);
    const fileType = extractFileTypeFromPath(filePath);
    if (demoId === activeDemoId) {
        applyDemoSnapshot({
            [fileType === "code" ? "code" : "schema"]: content,
            source: "manual-load",
        });
    }
}, [activeDemoId, applyDemoSnapshot]);
```

分离关注点后，`onSave` 回调只做文件 I/O，`handleWorkspaceFileSaved` 只做状态同步，各自职责清晰。

### 5½.2 方案一改动范围过大，应拆分为两步

方案一要同时改动 `processRealtimeFiles`、`extractCodeAndSchemaUpdates`、HTTP 兜底逻辑、timer 管理。一个 `onFinish` 函数变成 5 步以上的复杂编排。如果引入新 bug，难以二分定位。

**建议拆分：**
- **方案一-A（最小版）：** 只改 HTTP 兜底触发条件，把 `if (!codeUpdated || !schemaUpdated)` 改为 `if (!codeUpdated && !schemaUpdated)`。一行改动，消除「当 code 已由实时流更新时，schema 未更新也会触发 HTTP 覆盖」的核心问题。
- **方案一-B（完整版）：** 在方案一-A 验证稳定后，再合并重复的文件处理 + 清理 timer 逻辑。

### 5½.3 方案间存在执行顺序依赖（未标注）

部分方案的执行顺序不能随意调换：

| 依赖关系 | 说明 |
|----------|------|
| 方案二（幂等化）→ 方案三（移除 snapshotVersion） | setCode 不幂等时，移除 snapshotVersion 可能导致 code 值相同时仍触发重编译 |
| 方案二（幂等化）→ 方案四（ConfigForm key） | 移除 snapshotVersion 后，ConfigForm 需要其他更新触发源（方案二控制 snapshotVersion 增量时机） |
| 方案二（幂等化）→ 方案六（代码Tab 同步） | `applyDemoSnapshot` 的 `source: "manual-load"` 依赖幂等逻辑不意外取消流式 debounce |
| 方案三（编译去 snapshotVersion）→ 方案四（ConfigForm key） | ConfigForm 和 PreviewPanel 共享 snapshotVersion，必须先解耦消费端再修改生产端 |

**结论：方案二是所有其他方案的前置条件，必须最先实施。**

### 5½.4 方案六遗漏：文件树缓存未更新

`WorkspaceFileTree` 内部维护了文件内容的本地缓存。`onSave` 写入文件系统成功后，如果不更新该缓存，后续从文件树重新打开同一文件时，显示的仍是旧内容。

**建议：** `onSave` 成功后，通知 `WorkspaceFileTree` 更新缓存（或让文件树在下次展开该文件时重新 fetch）。

### 5½.5 applyDemoSnapshot 的 source 参数滥用

```typescript
// page.tsx:753-758
const handleCodeUpdate = useCallback(
    (newCode: string) => {
        applyDemoSnapshot({ code: newCode, source: "ai-realtime" });  // ← 总是 "ai-realtime"
    },
    [applyDemoSnapshot],
);
```

`handleCodeUpdate` 在两个场景被调用：
1. 流式 debounce → 正确，应为 `"ai-realtime"`
2. `onFinish` 中 `extractCodeAndSchemaUpdates` → **错误**，此时已非实时阶段

`source` 参数用于控制 schema 自动重生成 timer 的取消行为（只应在 `"ai-realtime"` / `"ai-finish"` 时取消）。错误标记可能导致非实时阶段的调用意外取消 timer。

**改进：** `handleCodeUpdate` 应接受 `source` 参数，或 `extractCodeAndSchemaUpdates` 在 `onFinish` 中使用不同来源标记。

### 5½.6 applyDemoSnapshot 依赖 activeDemoId 的权衡

将 `activeDemoId` 加入 `useCallback` deps 数组后，每次页面切换都会改变 `applyDemoSnapshot` 的函数引用，导致所有依赖它的回调（`handleCodeUpdate`、`handleSchemaUpdate`、`handleWorkspaceFileSaved`）也重新创建。这在页面频繁切换时会增加子组件的 re-render。

**权衡：** 这是正确性的必要代价。当前 `activeDemoIdRef` 方案（用 ref 绕过 deps）正是导致根因 D 的原因。稳定引用的代价是闭包过期——接受引用变化是更安全的选择。

如果 re-render 成为实际性能问题，可考虑用 `useRef` + `useEffect` 按需同步，而非直接加入 deps。但目前在缺乏性能数据的情况下，**正确性优先**。

## 6. 根治方案

### 6.1 核心设计原则

> **单一真源 + 幂等交付 + 不变式保护**

1. **AI 完成后的代码更新必须来自唯一来源**
2. **代码状态更新必须幂等（同值不触发副作用）**
3. **编译触发只依赖 code 变化，不再使用 snapshotVersion 作为编译触发器**
4. **ConfigForm 不应因 AI 完成而强制重建**

### 6.2 方案一-A（最小版）：修复 HTTP 兜底条件（一行改动）

**目标：** 消除根因 A——「HTTP 兜底用旧数据覆盖新代码」

**当前问题：**
```typescript
// use-chat-stream.ts:363
if (!codeUpdated || !schemaUpdated) {  // ← OR：任一未更新都触发 HTTP
    await fetchSessionFiles(sessionId, demoId);
    // ...
}
```

当 `codeUpdated = true`（来自实时流）但 `schemaUpdated = false`（result.files 不含 schema）时，HTTP 兜底仍会触发，并从文件系统读取旧 code → 覆盖新 code。

**改动（一行）：**

```typescript
// 改前：
if (!codeUpdated || !schemaUpdated) {

// 改后：
if (!codeUpdated && !schemaUpdated) {  // ← AND：两者都未更新才触发 HTTP
```

**理由：**
- 如果 `codeUpdated` 或 `schemaUpdated` 中至少一个为 true，说明 `allFiles` 中包含有效数据，即使另一个未被更新，也不应冒风险从文件系统读取可能过时的数据
- 兜底的本意是「完全没有文件变更时从磁盘恢复」，而非「补充不完整的变更」

**风险：** 极低。仅改变触发条件，不改变逻辑。如果 `allFiles` 包含 code 但不含 schema，schema 不会被 HTTP 更新——但 schema 应已在 React 状态中（初始加载时写入），无需覆盖。

### 6.3 方案一-B（完整版）：在 onFinish 中统一文件处理

**目标：** 消除重复处理 + 彻底消除根因 A、C（旧代码覆盖 + 重复处理）

**前置条件：** 方案一-A 已实施并验证稳定。

**具体改动：**

在 `use-chat-stream.ts` 的 `onFinish` 中：

```typescript
onFinish: async (result) => {
    // ... 消息处理保持不变 ...

    await persistMessages(sessionId, updatedMessages);
    await updateSessionTitle(sessionId, userMessage, ...);

    // ── 改动开始 ──

    // Step A: 收集所有文件（最终态）
    // 规则：优先使用 result.files；如果 result.files 为空，使用 realtimeFilesRef
    const allFiles: FileChangeEntry[] = (
        result.files && result.files.length > 0
            ? result.files
            : Array.from(realtimeFilesRef.entries()).map(([path, info]) => ({
                  path,
                  action: info.action as "created" | "modified" | "deleted",
                  content: info.content,
              }))
    );

    // Step B: 清除待处理的实时文件定时器（不再调用 processRealtimeFiles）
    if (fileUpdateTimer) {
        clearTimeout(fileUpdateTimer);
        fileUpdateTimer = null;
    }

    // Step C: 统一应用所有文件变更（仅一次）
    if (allFiles.length > 0) {
        for (const f of allFiles) {
            if (f.path && f.path.endsWith(".md")) {
                memoryFilePathsRef.current.add(f.path);
            }
        }
        onFilesChange?.(allFiles);
        extractCodeAndSchemaUpdates(allFiles, {
            onCodeUpdate,      // ← 只调用一次
            onSchemaUpdate,    // ← 只调用一次
        });
    }

    setIsStreaming(false);

    // Step D: HTTP 兜底改为「仅在 code/schema 都没被任何步骤更新时才触发」
    //      extractCodeAndSchemaUpdates 返回 { codeUpdated, schemaUpdated }
    //      如果任一已更新 → 跳过兜底
    //      如果都没更新 → 尝试从文件系统读取（防御性，但不应覆盖已更新值）
    const { codeUpdated, schemaUpdated } = extractCodeAndSchemaUpdates(allFiles, { ... });
    if (!codeUpdated && !schemaUpdated) {  // ← 改为 AND 而非 OR
        const filesData = await fetchSessionFiles(sessionId, demoId);
        if (filesData) {
            if (filesData.code) onCodeUpdate?.(filesData.code);
            if (filesData.schema) onSchemaUpdate?.(filesData.schema);
        }
    }

    // Step E: 通知完成（不再额外 +1 snapshotVersion）
    //         onSnapshotReady 改为不做任何事（由上游 applyDemoSnapshot 内部的 +1 驱动）
    onSnapshotReady?.();

    realtimeFilesRef.clear();
};
```

**同时**，移除 `processRealtimeFiles` 函数中的双调用，以及 `onFileOperation` 中 debounce 定时器的 `processRealtimeFiles` 调用——或至少确保实时推送时不调用 `onCodeUpdate`（仅更新 `realtimeFilesRef`），等到 `onFinish` 统一处理。

### 6.8 方案六：「代码」Tab 保存后同步预览状态

**目标：** 用户在「代码」Tab 编辑 demo 代码/schema 文件后，预览区立即更新

**问题位置：** `page.tsx:1673-1687` — `onSave` 回调只写文件系统，不更新 React 状态

**设计原则：** `onSave` 保持简洁（仅文件 I/O），状态同步由独立的 handler 负责，遵循关注点分离。

**具体改动：**

```typescript
// page.tsx — 新增独立的 handler 函数

const handleWorkspaceFileSaved = useCallback(
    (filePath: string, content: string) => {
        const normalizedPath = filePath.replace(/^\/+/, "");
        const demoId = extractDemoIdFromPath(normalizedPath);
        const fileType = normalizedPath.endsWith("index.tsx") ? "code" : "schema";

        // 只有当前激活 demo 的文件才更新预览
        if (demoId && demoId === activeDemoId) {
            applyDemoSnapshot({
                [fileType === "code" ? "code" : "schema"]: content,
                source: "manual-load",
            });
        }

        // 同时通知文件树刷新缓存
        workspaceFileTreeRef.current?.invalidatePath(filePath);
    },
    [activeDemoId, applyDemoSnapshot],
);

// 辅助函数：从路径提取 demoId
function extractDemoIdFromPath(normalizedPath: string): string | null {
    const match = normalizedPath.match(/^demos\/([^/]+)\//);
    return match ? match[1] : null;
}

// ── WorkspaceCodeDialog 的 onSave 保持简洁 ──
<WorkspaceCodeDialog
    open={wsCodeDialogOpen}
    onOpenChange={setWsCodeDialogOpen}
    filePath={wsCodeDialogData.filePath}
    content={wsCodeDialogData.content}
    editable={wsCodeDialogData.editable}
    onSave={async (content) => {
        if (!sessionId) return;
        const res = await fetch(
            `/api/sessions/${sessionId}/workspace/files/${encodeURIComponent(wsCodeDialogData.filePath)}`,
            {
                method: "PUT",
                headers: { "Content-Type": "application/json" },
                body: JSON.stringify({ content }),
            },
        );
        const data = await res.json();
        if (!data.success) {
            throw new Error(data.error?.message || "保存失败");
        }
        // ← 不再在这里调用 applyDemoSnapshot
    }}
    onSaved={({ filePath, content }) => {  // ← 新增 onSaved 回调
        handleWorkspaceFileSaved(filePath, content);
    }}
/>
```

**组件接口变更：**
- `WorkspaceCodeDialog` 新增 `onSaved` prop：在文件系统写入成功后触发，传递 `{ filePath, content }` 元数据
- `onSave` 继续保持原有语义：执行保存操作（可抛异常表示失败），成功后触发 `onSaved`

**设计要点：**
- `onSave` 只负责数据持久化（单一职责），不知道任何业务逻辑
- `handleWorkspaceFileSaved` 负责状态同步，路径 → demoId 转换封装在辅助函数中
- 正则提取 `demoId` 替代硬编码路径模板，更健壮
- 同时通知 `WorkspaceFileTree` 刷新缓存（通过 ref），解决根因「文件树显示过期内容」
- 使用 `activeDemoId` 匹配确保只更新当前页面的预览
- 如果未来支持跨 demo 页面编辑，只需扩展 `handleWorkspaceFileSaved` 逻辑
