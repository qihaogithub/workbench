## 1. 问题描述

AI 编辑代码文件后，预览区**先闪烁一下新内容，然后变回旧内容**。必须「保存后重新打开页面」才能看到正确的效果。

此问题已反复出现、修复多次，但始终未根除。

## 2. 完整数据流梳理

### 2.1 文件变更的三条路径（当前代码，use-chat-stream.ts:295-394）

AI 编辑后，代码/schema 可以从**三条独立路径**到达 React 状态：

```
路径 A: 实时流式文件 → onFileOperation → realtimeFilesRef → processRealtimeFiles() → onCodeUpdate("ai-realtime")
路径 B: 后端 finish 事件 → result.files → extractCodeAndSchemaUpdates() → onCodeUpdate("ai-finish")
路径 C: HTTP 兜底 → fetchSessionFiles() → onCodeUpdate("ai-finish")
```

### 2.2 onFinish 完整执行序列（当前代码）

```typescript
// use-chat-stream.ts:295-394
onFinish: async (result) => {
    // ① 消息持久化
    await persistMessages(sessionId, updatedMessages);
    await updateSessionTitle(...);

    // ② Flush pending realtime files (路径 A 二次触发)
    if (fileUpdateTimer) {
        clearTimeout(fileUpdateTimer);
        fileUpdateTimer = null;
        processRealtimeFiles();                    // ← ⚠️ 路径 A 重复触发
    }

    // ③ 构建最终文件列表：优先用 result.files（⚠️ 数据优先级颠倒）
    const finalFiles = result.files?.length > 0
        ? result.files                            // ← 后端数据优先
        : Array.from(realtimeFilesRef.entries());// ← 实时流数据作后备

    // ④ 通知文件树
    onFilesChange?.(finalFiles);
    setIsStreaming(false);

    // ⑤ 路径 B：从 finalFiles 提取 code/schema 更新
    const { codeUpdated, schemaUpdated } =
        extractCodeAndSchemaUpdates(finalFiles, {
            onCodeUpdate: (code) => onCodeUpdate?.(code, "ai-finish"),
            onSchemaUpdate: (schema) => onSchemaUpdate?.(schema, "ai-finish"),
        });

    // ⑥ 路径 C：HTTP 兜底（条件已改为 && ）
    if (!codeUpdated && !schemaUpdated) {
        const filesData = await fetchSessionFiles(sessionId, demoId);
        // ...
    }

    // ⑦ 通知快照就绪
    onSnapshotReady?.();                          // → invalidateCompileCache()

    realtimeFilesRef.clear();
};
```

## 4. 当前仍存在的根因分析

> 以下为基于当前代码（2026-06-04）的深度分析。所有代码行号均为实际行号。

### 4.1 根因 A（P0 CRITICAL）：数据优先级颠倒 — result.files 优先于 realtimeFilesRef

**关键代码（use-chat-stream.ts:333-342）**：

```typescript
const finalFiles: FileChangeEntry[] =
    result.files && result.files.length > 0
        ? result.files                    // ← ⚠️ 后端数据优先
        : Array.from(realtimeFilesRef.entries()).map(...); // ← 实时流数据作后备
```

**问题**：`realtimeFilesRef` 包含的是 AI 写文件时的**原始内容**（通过 WebSocket 实时推送），是最权威的数据源。而 `result.files` 是后端在 finish 事件中构建的文件列表，可能包含：
- 后端自己的缓存（未及时 flush）
- 文件系统的旧快照
- 后端 API 返回与前端不匹配的数据

**时序分析**：

```
T0: AI 写入 index.tsx（v_final）
T1: onFileOperation → realtimeFilesRef.set("index.tsx", v_final)
T2: 300ms debounce → processRealtimeFiles() → applyDemoSnapshot({ code: v_final }) → setCode(v_final) ✓

T3: onFinish 触发
T4: finalFiles = result.files（包含 index.tsx 但内容是 v_old）
T5: extractCodeAndSchemaUpdates(finalFiles, ...)
    → onCodeUpdate(v_old, "ai-finish")
    → applyDemoSnapshot({ code: v_old })
    → setCode((prev) => prev === v_old ? prev : v_old)
    → prev = v_final (来自 T2), v_final !== v_old
    → setCode(v_old) ← ⚠️ 旧值覆盖新值！预览回退！
```

**证据级别：A（直接代码证据）**
- `use-chat-stream.ts:333-342` — 数据优先级颠倒
- `use-chat-stream.ts:356-362` — 路径 B 用旧数据调用 onCodeUpdate
- `page.tsx:210` — 幂等检查 `prev === newCode ? prev : newCode` 不防旧值覆盖

### 4.2 根因 B（P0 CRITICAL）：onFinish 中路径 A 的重复触发

**关键代码（use-chat-stream.ts:326-330）**：

```typescript
if (fileUpdateTimer) {
    clearTimeout(fileUpdateTimer);
    fileUpdateTimer = null;
    processRealtimeFiles();    // ← ⚠️ 路径 A 在 onFinish 中再次触发
}
```

**时序分析**：

```
T0: AI 写入 index.tsx（v_final）
T1: onFileOperation → realtimeFilesRef.set("index.tsx", v_final)
    → 启动 300ms debounce timer

T2: 300ms debounce 到期 → processRealtimeFiles()
    → applyDemoSnapshot({ code: v_final }) → setCode(v_final) ✓

T3: AI 继续，又写入文件 → 更新 realtimeFilesRef
    → 重启 300ms debounce timer

T4: onFinish 到达
    → fileUpdateTimer 存在（T3 的新 timer）
    → clearTimeout + processRealtimeFiles()  ← ⚠️ 路径 A 再次执行
    → applyDemoSnapshot({ code: v_intermediate }) → setCode(v_intermediate)

T5: finalFiles = result.files（包含 v_old 或 v_final）
    → extractCodeAndSchemaUpdates(finalFiles)
    → applyDemoSnapshot({ code: result_version }) → setCode(result_version)
```

在这条路径中，`processRealtimeFiles()` 在 T4 执行时使用的是 `realtimeFilesRef` 中**最后一个状态**，可能是 v_intermediate（AI 修改了两次，debounce 只捕获了第一次）。然后 T5 又用 result.files 覆盖。**三次写入，至少两次可能产生竞争。**

**证据级别：A（直接代码证据）**
- `use-chat-stream.ts:288-291` — debounce timer 重置逻辑
- `use-chat-stream.ts:326-330` — onFinish 中 processRealtimeFiles 调用

### 4.3 根因 C（P0 CRITICAL）：applyDemoSnapshot 闭包陷阱

**关键代码（page.tsx:237-252）**：


**问题机制**：

1. `handleCodeUpdate` 只传 `code`，不传 `schema`：

```typescript

        applyDemoSnapshot({ code: newCode, source });  // ← schema 为 undefined
```

## 5. 为什么反复出现 — 根源诊断

### 5.1 原有修复的致命盲区

之前所有修复（幂等 setCode、HTTP 兜底条件、去 snapshotVersion）都只解决了**表层症状**。真正的问题在于：

| 修复 | 解决了 | 未解决 |
|------|--------|--------|
| setCode 幂等 | 相同值不触发渲染 | **旧值覆盖新值不防** |
| HTTP 兜底 `\|\|` → `&&` | 部分更新不触发兜底 | **路径 B 仍可覆盖路径 A** |
| 去 snapshotVersion | 多余编译 | **数据竞争未解决** |
| ConfigForm key 优化 | 表单重建 | **闭包 fallback 未解决** |

### 5.2 根源架构缺陷

**缺陷 1：多数据源无优先级机制**

当前 `finalFiles` 构建逻辑将后端 `result.files` 置于优先位置，而实时流数据 `realtimeFilesRef` 只是备用。但实际上实时流数据才是**最权威的**——它是 AI 写文件的原始内容，没有经过后端缓存、序列化等可能引入延迟的环节。

**缺陷 2：多数据源并发写入无协调**

路径 A（实时流）和路径 B（finish 事件）都在同一次 onFinish 中调用 `onCodeUpdate`，但没有机制确保**新值不会被旧值覆盖**。`setCode` 的幂等检查只防止相同值重复渲染，不防止旧值覆盖新值。

**缺陷 3：闭包捕获的过期状态**

`applyDemoSnapshot` 的 `useCallback` deps 包含 `code` 和 `schema`，这导致：
- 函数重建发生在 React 渲染后，但在同一微任务中的多次调用之间，闭包值不会更新
- `setEditorContent` 和 `mergeConfigWithUserValues` 中的 fallback 值（`code`/`schema`）在连续调用中可能是过期的

## 6. 根治方案

### 6.1 核心设计原则

> **实时流优先 + 单一写入点 + ref 修复闭包 + 冗余清除**

1. 实时流数据优先于后端数据（数据新鲜度保证）
2. AI 完成后只从一个来源写入代码/schema（消除数据竞争）
3. 使用 ref 替代闭包捕获（消除过期状态）
4. 清除所有冗余的缓存清理调用

### 6.2 方案 R-A（P0 根治）：实时流数据优先 + 去重写入

**目标：** 消除路径 B 用旧数据覆盖路径 A 新值的问题

**核心改动 1：finalFiles 构建逻辑 — 实时流优先（use-chat-stream.ts:333-342）**

```typescript
// 改前：result.files 优先
const finalFiles: FileChangeEntry[] =
    result.files && result.files.length > 0
        ? result.files
        : Array.from(realtimeFilesRef.entries()).map(...);

// 改后：realtimeFilesRef 优先，result.files 补充缺失文件
const realtimeFileMap = new Map<string, FileChangeEntry>();
for (const [path, info] of realtimeFilesRef.entries()) {
    realtimeFileMap.set(path, {
        path,
        action: info.action as "created" | "modified" | "deleted",
        content: info.content,
    });
}

// result.files 中的文件仅在 realtimeFilesRef 中不存在时才加入
if (result.files && result.files.length > 0) {
    for (const f of result.files) {
        if (!realtimeFileMap.has(f.path)) {
            realtimeFileMap.set(f.path, f);
        }
    }
}

const finalFiles = Array.from(realtimeFileMap.values());
```

**为什么这样做：**
- `realtimeFilesRef` 包含 AI 写文件的原始内容，是最权威的数据源
- `result.files` 只补充实时流中没有的文件（如后端额外生成的配置文件）
- 对于代码/schema 文件，实时流中一定有最新版本（如果 AI 修改了它们）

**核心改动 2：移除 onFinish 中 processRealtimeFiles 的冗余调用（use-chat-stream.ts:326-330）**

```typescript
// 改前：
if (fileUpdateTimer) {
    clearTimeout(fileUpdateTimer);
    fileUpdateTimer = null;
    processRealtimeFiles();  // ← 冗余调用，与步骤⑤冲突
}

// 改后：只清除 timer，不重复处理
// 实时流期间的 debounce 更新已经在流式阶段完成
// onFinish 会通过 finalFiles 统一处理所有文件（包括未 debounce 的部分）
if (fileUpdateTimer) {
    clearTimeout(fileUpdateTimer);
    fileUpdateTimer = null;
    // 不再调用 processRealtimeFiles()
    // 原因：realtimeFilesRef 的内容已合并到 finalFiles 中
    //       避免路径 A 与路径 B 的数据竞争
}
```

**为什么安全：**
- 如果 debounce timer 已触发（T2 时刻），processRealtimeFiles 已经执行过，onCodeUpdate 已经调用 → 无需在 onFinish 中重调
- 如果 debounce timer 未触发，说明有未处理的实时文件 → 这些文件的内容已在 realtimeFilesRef 中，通过改动 1 合并到 finalFiles → 统一由步骤⑤处理
- 流式阶段已执行过的 onCodeUpdate 更新了 React 状态（`code` 为最新值），步骤⑤的 extractCodeAndSchemaUpdates 如果也包含相同文件（且内容相同），`setCode` 会幂等跳过

**但等等**——如果 debounce timer 未触发（即流式阶段的最后一波更新还没被处理），那些更新只在 `realtimeFilesRef` 中，没有通过 `processRealtimeFiles` 触发 `onCodeUpdate`。按改动 1 的逻辑，这些文件会被合并到 `finalFiles` 中，然后在步骤⑤通过 `extractCodeAndSchemaUpdates` 统一处理。这意味着：

- 最终代码/schema 会被正确更新（只是延迟到了 onFinish，而不是在流式阶段实时更新）
- 这是可接受的权衡——用户在 AI 完成的瞬间看到最终结果，而非在过程中看到中间更新

**如果需要保持流式阶段的实时预览体验**（推荐），可以在清除 timer 时手动 flush 一次：

```typescript
if (fileUpdateTimer) {
    clearTimeout(fileUpdateTimer);
    fileUpdateTimer = null;
    // 确保 pending 的实时更新被应用（保持流式预览体验）
    // 但注意：这次调用使用的回调是 onCodeUpdate("ai-realtime")
    processRealtimeFiles();
}
```

如果保留这行调用，需要确保步骤⑤的 `extractCodeAndSchemaUpdates` **不会覆盖已通过实时流更新的数据**。这就是核心改动 3 的作用。

**核心改动 3：onFinish 中跳过已被实时流更新的数据类型**

在步骤⑤中，对于已经通过 `processRealtimeFiles()` 更新过的数据类型，不应再从 `finalFiles` 中提取并覆盖。

```typescript
// 追踪实时流已更新的数据类型
let realtimeUpdatedCode = false;
let realtimeUpdatedSchema = false;

// 改造 processRealtimeFiles 以追踪更新
const processRealtimeFiles = () => {
    const files = Array.from(realtimeFilesRef.entries()).map(
        ([path, info]) => ({
            path,
            action: info.action as "created" | "modified" | "deleted",
            content: info.content,
        }),
    );

    if (files.length > 0) {
        // 检测是否有代码/schema更新
        for (const f of files) {
            if (isCodeFile(f.path) && f.content) realtimeUpdatedCode = true;
            if (isSchemaFile(f.path) && f.content) realtimeUpdatedSchema = true;
        }
        processFileChanges(files, {
            onCodeUpdate,
            onSchemaUpdate,
            onFilesChange,
        });
    }
};

// 在 onFinish 的步骤⑤中：
const { codeUpdated, schemaUpdated } =
    finalFiles.length > 0
        ? extractCodeAndSchemaUpdates(finalFiles, {
            // 如果实时流已更新代码，跳过 result.files 中可能的旧版本
            onCodeUpdate: realtimeUpdatedCode
                ? undefined
                : (code) => onCodeUpdate?.(code, "ai-finish"),
            onSchemaUpdate: realtimeUpdatedSchema
                ? undefined
                : (schema) => onSchemaUpdate?.(schema, "ai-finish"),
        })
        : { codeUpdated: false, schemaUpdated: false };

// 但仍需要正确设置 codeUpdated/schemaUpdated 用于 HTTP 兜底判断
// 如果 realtimeUpdatedCode/codeUpdated 为 true，则已更新
const effectiveCodeUpdated = realtimeUpdatedCode || codeUpdated;
const effectiveSchemaUpdated = realtimeUpdatedSchema || schemaUpdated;

// HTTP 兜底
if (!effectiveCodeUpdated && !effectiveSchemaUpdated) {
    const filesData = await fetchSessionFiles(sessionId, demoId);
    // ...
}
```

**核心改动 4：完整的 onFinish 重写**

```typescript
onFinish: async (result) => {
    streamService.stopKeepalive();
    stopSilenceTracking();
    const currentMsg = currentMessageRef.current;
    const assistantMessage: ChatMessage = { ... };
    const updatedMessages = [...messagesRef.current, assistantMessage];
    setMessages(updatedMessages);
    setCurrentMessage({ role: "assistant", content: "", parts: [] });
    setStreamContent("");

    await persistMessages(sessionId, updatedMessages);
    await updateSessionTitle(sessionId, userMessage, messagesRef.current.length === 0);

    // ── 追踪实时流更新状态 ──
    let realtimeUpdatedCode = false;
    let realtimeUpdatedSchema = false;

    // ── Flush pending realtime files（保留流式预览体验） ──
    if (fileUpdateTimer) {
        clearTimeout(fileUpdateTimer);
        fileUpdateTimer = null;
    }

    // 从 realtimeFilesRef 检测哪些数据类型已通过实时流更新
    for (const [path, info] of realtimeFilesRef.entries()) {
        const normalizedPath = normalizePath(path);
        if (isCodeFile(normalizedPath) && info.content) realtimeUpdatedCode = true;
        if (isSchemaFile(normalizedPath) && info.content) realtimeUpdatedSchema = true;
    }

    // 如果有 pending 的实时文件，先 flush（保持流式预览的完整性）
    const pendingRealtimeFiles = Array.from(realtimeFilesRef.entries()).map(
        ([path, info]) => ({
            path,
            action: info.action as "created" | "modified" | "deleted",
            content: info.content,
        }),
    );
    if (pendingRealtimeFiles.length > 0) {
        processFileChanges(pendingRealtimeFiles, {
            onCodeUpdate,
            onSchemaUpdate,
            onFilesChange,
        });
    }

    // ── 构建最终文件列表：realtimeFilesRef 优先 ──
    const realtimeFileMap = new Map<string, FileChangeEntry>();
    for (const [path, info] of realtimeFilesRef.entries()) {
        realtimeFileMap.set(path, {
            path,
            action: info.action as "created" | "modified" | "deleted",
            content: info.content,
        });
    }
    if (result.files && result.files.length > 0) {
        for (const f of result.files) {
            if (!realtimeFileMap.has(f.path)) {
                realtimeFileMap.set(f.path, f);
            }
        }
    }
    const finalFiles = Array.from(realtimeFileMap.values());

    if (finalFiles.length > 0) {
        for (const f of finalFiles) {
            if (f.path && f.path.endsWith(".md")) {
                memoryFilePathsRef.current.add(f.path);
            }
        }
        onFilesChange?.(finalFiles);
    }

    setIsStreaming(false);

    // ── 从 finalFiles 提取更新，但跳过实时流已更新的数据类型 ──
    const { codeUpdated, schemaUpdated } =
        finalFiles.length > 0
            ? extractCodeAndSchemaUpdates(finalFiles, {
                onCodeUpdate: realtimeUpdatedCode
                    ? undefined  // 实时流已更新，跳过（避免旧值覆盖）
                    : (code) => onCodeUpdate?.(code, "ai-finish"),
                onSchemaUpdate: realtimeUpdatedSchema
                    ? undefined  // 实时流已更新，跳过
                    : (schema) => onSchemaUpdate?.(schema, "ai-finish"),
            })
            : { codeUpdated: false, schemaUpdated: false };

    // ── HTTP 兜底：仅在完全没有数据更新时触发 ──
    const effectiveCodeUpdated = realtimeUpdatedCode || codeUpdated;
    const effectiveSchemaUpdated = realtimeUpdatedSchema || schemaUpdated;

    if (!effectiveCodeUpdated && !effectiveSchemaUpdated) {
        const filesData = await fetchSessionFiles(sessionId, demoId);
        if (filesData) {
            const { code, schema } = filesData;
            if (code) onCodeUpdate?.(code, "ai-finish");
            if (schema) onSchemaUpdate?.(schema, "ai-finish");

            const fetchedFiles: FileChangeEntry[] = [];
            if (code) fetchedFiles.push({ path: "index.tsx", action: "modified", content: code });
            if (schema) fetchedFiles.push({ path: "config.schema.json", action: "modified", content: schema });
            if (fetchedFiles.length > 0) onFilesChange?.(fetchedFiles);
        }
    }

    onSnapshotReady?.();
    realtimeFilesRef.clear();
};
```

**目标：** 消除 `setEditorContent` 和 `setConfigDataMap` 中的过期闭包值 fallback

**具体改动（page.tsx）：**

**步骤 1：添加 codeRef 和 schemaRef**

**步骤 2：修改 applyDemoSnapshot**

```typescript
const applyDemoSnapshot = useCallback(
    (params: {
        code?: string;
        schema?: string;
        source: "ai-realtime" | "ai-finish" | "manual-load" | "page-switch";
    }) => {
        const { code: newCode, schema: newSchema, source } = params;

        if (newCode !== undefined) {
            setCode((prev) => (prev === newCode ? prev : newCode));
            // 同步更新 ref，供 setEditorContent 和 setConfigDataMap 使用
            codeRef.current = newCode;
            if (sessionId && activeDemoId) {
                invalidateCompileCache(sessionId, activeDemoId);
            }
        }

        if (newSchema !== undefined) {
            setSchema(newSchema);
            // 同步更新 ref
            schemaRef.current = newSchema;
            setPreviewSize(getPreviewSize(newSchema));

            try {
                setConfigDataMap((prev) => {
                    const current = prev[activeDemoIdRef.current] ?? {};
                    const merged = mergeConfigWithUserValues(
                        current,
                        newSchema,
                        schemaRef.current,  // ← 使用 ref 而非闭包 schema
                    );
                    return { ...prev, [activeDemoIdRef.current]: merged };
                });
            } catch (e) {
                console.warn("[DemoEditPage] Failed to merge schema defaults:", e);
            }
        }

        // 使用 ref 替代闭包中的 code 和 schema
        setEditorContent((prev) => {
            const currentCode = newCode ?? extractCodeFromFigma(prev) ?? codeRef.current;    // ← ref
            const currentSchema = newSchema ?? extractSchemaFromFigma(prev) ?? schemaRef.current; // ← ref
            return buildFigmaText(currentCode, currentSchema);
        });

        if (source === "ai-realtime" || source === "ai-finish") {
            if (schemaRegenerateTimerRef.current) {
                clearTimeout(schemaRegenerateTimerRef.current);
                schemaRegenerateTimerRef.current = null;
            }
        }
    },
    [sessionId, activeDemoId],  // ← 移除 code 和 schema
);
```

**关键改动：**
1. `setEditorContent` 中的 fallback 从闭包 `code`/`schema` 改为 `codeRef.current`/`schemaRef.current`
2. `mergeConfigWithUserValues` 中的第三个参数从闭包 `schema` 改为 `schemaRef.current`
3. 新增 `newCode`/`newSchema` 更新时同步更新 ref
4. 从 `useCallback` deps 中移除 `code` 和 `schema`

**为什么安全：**
- `codeRef.current` 和 `schemaRef.current` 始终持有最新的 `code` 和 `schema` 值
- 移除 `code`/`schema` 从 deps 后，`applyDemoSnapshot` 不会因 `code`/`schema` 变化而重建，保持了稳定的回调引用
- `handleCodeUpdate` 和 `handleSchemaUpdate` 依赖 `applyDemoSnapshot`，deps 稳定后它们也更稳定
- `invalidateCompileCache` 使用 `sessionId` 和 `activeDemoId`，已是 ref（`activeDemoIdRef`）或 dep

**性能影响：**
- `applyDemoSnapshot` 的引用稳定性提高（不再因 `code`/`schema` 变化而重建）
- 依赖它的所有回调（`handleCodeUpdate`、`handleSchemaUpdate`、`handleWorkspaceFileSaved`）的引用也更稳定
- 减少了子组件的不必要 re-render

**风险：** 低。仅增加 200ms 延迟，不影响用户体验。

### 7.2 优先级表格

|------|------|--------|----------|------|------|
| **1** | R-A：实时流数据优先 + 去重写入 | 中 | 根因 A、B、D（数据竞争、旧值覆盖） | 低 | 无 |

**注意：R-A 和 R-B 可以同时实施，它们不互为依赖。**

### 7.3 R-A 方案的 `processRealtimeFiles` 保留 vs 移除决策

R-A 方案中有一个关键决策：**在 onFinish 中是否保留 `processRealtimeFiles` 调用？**

| 策略 | 优点 | 缺点 |
|------|------|------|
| **保留调用** | 保持流式预览的实时性；AI 写文件过程中用户能看到中间更新 | 可能产生路径 A + 路径 B 的重复调用（但 R-A 的标记法可避免覆盖） |
| **移除调用** | 彻底消除路径 A 在 onFinish 中的触发；逻辑更简洁 | 如果有 pending 的实时文件（debounce 未到期），这些更新会延迟到 onFinish 统一处理，流式阶段的最后一次更新可能丢失 |

**推荐：保留调用 + 标记法（R-A 中的核心改动 3）**。这样既保持了流式预览体验，又通过 `realtimeUpdatedCode`/`realtimeUpdatedSchema` 标记避免了路径 B 用旧值覆盖路径 A 新值。

## 10. 2026-06-04 第三轮分析：技术细节补充

### 10.1 幂等 setCode 的局限性

当前 `setCode((prev) => prev === newCode ? prev : newCode)` 只防止**相同值**的重复渲染。它**不防止旧值覆盖新值**：

```
// 时序：
setCode(v_current) // T1: 设置当前最新值
setCode(v_old)      // T2: 设置旧值
// functional updater 执行：
//   T1: prev = v_previous, v_previous !== v_current → 返回 v_current ✓
//   T2: prev = v_current,  v_current !== v_old     → 返回 v_old ❌
```

**结论**：幂等比较是优化手段，不是防竞争机制。根治必须从数据流层面保证**旧值不会到达 setCode**。

### 10.2 setEditorContent 的 fallback 问题更隐蔽

`setEditorContent` 的 fallback `newCode ?? extractCodeFromFigma(prev) ?? code` 中：

- `newCode`：当 `handleCodeUpdate` 调用时传入，值正确
- `extractCodeFromFigma(prev)`：从编辑器内容中解析，可能正确
- `code`：**闭包捕获的值，在同批更新中可能是过期的**
  - 当 `handleSchemaUpdate` 调用 `applyDemoSnapshot({ schema: s })` 时，`newCode` 为 `undefined`
  - fallback 到 `extractCodeFromFigma(prev)` 或闭包 `code`
  - 如果编辑器内容也过期（React 批处理），则会使用闭包中的旧 `code`

**R-B 方案通过 `codeRef.current` 替代闭包 `code`，解决了这个问题。**

### 10.3 完整的「闪烁后回退」时序（根因 A 综合分析）

```
T0: AI 流式写入 index.tsx → v_final
T1: onFileOperation → realtimeFilesRef.set("index.tsx", v_final)
T2: 300ms debounce → processRealtimeFiles()
    → applyDemoSnapshot({ code: v_final, source: "ai-realtime" })
    → setCode(v_final) ✓
    → setEditorContent → 使用 newCode=v_final ✓

T3: onFinish 回调开始
T4: await persistMessages + updateSessionTitle

T5: fileUpdateTimer 清除 + processRealtimeFiles()  ← 路径 A 二次触发
    → applyDemoSnapshot({ code: v_final, source: "ai-realtime" })
    → setCode((prev) => prev === v_final ? prev : v_final) → 幂等跳过 ✓
    → setEditorContent → newCode=v_final, 闭包 code 可能仍为旧值 ⚠️

T6: finalFiles 构建：result.files 优先
    → result.files 包含 index.tsx → 使用 result.files 中的 content
    → 如果 result.files 中的 content 是 v_old（后端缓存） ← ⚠️ 数据优先级颠倒

T7: extractCodeAndSchemaUpdates(finalFiles)
    → onCodeUpdate(v_old, "ai-finish")
    → applyDemoSnapshot({ code: v_old, source: "ai-finish" })
    → setCode((prev) => prev === v_old ? prev : v_old)
    → prev = v_final (来自 T2), v_final !== v_old → setCode(v_old) ❌ 覆盖！

T8: PreviewPanel 编译 v_old → iframe 显示旧内容 ❌
```

**R-A 方案如何修复：**
- 改动 1（数据优先级）：finalFiles 构建时 realtimeFilesRef 优先 → v_final 覆盖 v_old ✓
- 改动 2（去重写入）：如果保留 processRealtimeFiles，realtimeUpdatedCode 标记为 true → extractCodeAndSchemaUpdates 跳过 onCodeUpdate ✓
- 改动 3（闭包修复）：setEditorContent 用 codeRef.current 替代闭包 code ✓

**三个改动缺一不可**：
- 只改数据优先级：如果 result.files 中没有 index.tsx（不在 finalFiles 中），则 T7 不执行，没问题。但如果 result.files 中有旧版本，realtimeFilesRef 覆盖后 content 是 v_final → 正确 ✓
- 只改闭包、只改去重写入：如果 result.files 仍优先于 realtimeFilesRef，则 finalFiles 中的 content 仍是 v_old → T7 仍执行 → 覆盖 ❌
