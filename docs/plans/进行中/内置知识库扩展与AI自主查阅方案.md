# 知识库独立子项目与多端集成方案

## 背景

此前方案从“内置知识库扩展”和“AI 自主查阅”出发，重点讨论系统规则知识、模板项目知识、当前项目知识、异步摘要索引和图书馆管理员式检索报告。随着讨论深入，知识库的职责已经明显超过创作端的一个局部功能：

- 创作端需要维护当前项目知识、模板项目知识、模板阅读地图和 AI 对话上下文。
- 使用端未来可能需要面向访问者、运营人员或终端用户提供 AI 问答和内容解释。
- Agent Service 需要获取结构化知识报告，而不是把大量检索噪音塞进主 AI 上下文。
- 模板推荐、项目诊断、配置解释、页面生成和业务规则校验都可能复用同一套知识索引。
- LightRAG 或其他 RAG/图谱检索能力只解决“怎么检索”，不能替代项目自己的知识来源、权限、可信等级和报告协议，近期不作为实施依赖。

因此，知识库不应继续作为 `author-site` 内部功能自然膨胀，而应规划为独立子项目，作为可被创作端、使用端、Agent Service 和未来其他系统复用的 AI 可读知识基础设施。

## 核心判断

知识库子项目的核心价值不是文档 CRUD，而是把项目、模板、系统规则和多端上下文整理成 AI 可检索、可解释、可追溯、可分层阅读的知识资产。

独立子项目需要定制自己的领域模型和服务边界。v1 采用“自研轻量知识索引 + LLM 异步整理 + 知识库管理员报告”的路线：先用结构化索引、标签、关键词、基础全文检索和轻量关系索引完成候选召回，再用充足的 LLM 额度做离线整理、候选重排、原文压缩和报告生成。LightRAG、MiniRAG、向量数据库或图数据库只作为远期增强方向，不应直接定义业务协议。项目必须自己掌握：

- 知识来源分层。
- 当前项目事实、模板建议、其他模板参考样本、系统硬约束的优先级。
- 多端权限和可见范围。
- 阅读地图、异步索引、任务导向入口和知识报告格式。
- 主 AI 与知识库管理员之间的上下文隔离规则。
- 摘要导航与原文确认的协作约束。

产品评审后的修订判断：v1 的价值成立，且当前项目尚未上线，不应为了降低短期实现量而砍掉长期一定需要的核心能力。v1 可以激进，但必须把“激进”落在清晰的服务边界、权限接口、索引状态和验收指标上，而不是把所有未来能力一次性做成重部署架构。也就是说，v1 应完整覆盖知识来源分层、权限过滤、阅读地图、异步索引和知识报告，但部署形态可以先保持轻量。

## 目标

- 将知识库规划为独立子项目，供创作端、使用端、Agent Service 和未来系统集成。
- 定义统一的知识来源、索引、阅读地图、报告和任务状态模型。
- 支持当前项目知识库、模板项目知识库、系统规则知识和全量模板知识的统一管理。
- 支持模板保存后异步生成模板阅读地图、页面摘要、配置摘要和知识库摘要。
- 提供 `knowledge.search`、`knowledge.read`、`knowledge.related`、`knowledge.report` 等能力。
- 支持“知识库管理员”子智能体式检索报告，隔离无用检索上下文。
- 使用 LLM 额度增强离线索引和运行时报告质量，而不是在 v1 引入重型 RAG 框架。
- 预留检索后端扩展边界，但近期不考虑接入 LightRAG。
- 保持 v1 实现可控，避免过早引入过重存储、队列和模型部署复杂度。

## 范围

本方案覆盖：

- 知识库独立子项目的职责边界。
- `knowledge-core`、`knowledge-service`、`knowledge-worker` 的分层规划。
- 创作端、使用端、Agent Service 和未来系统的集成方式。
- 知识来源、索引产物、阅读地图和报告模型。
- 异步索引整理、stale 状态、报告缓存和原文确认规则。
- 远期增强检索能力的预留边界。
- 分阶段落地路线、验证方式和风险事项。

本方案暂不覆盖：

- 具体数据库表结构。
- 具体 API TypeScript 类型定义。
- LightRAG 或其他增强检索后端的生产部署参数。
- 多租户计费、组织级知识市场和外部知识交易。
- 具体 UI 设计。

## 子项目定位

建议拆分为三个层次：

| 子项目 | 类型 | 职责 |
|---|---|---|
| `@opencode-workbench/knowledge-core` | 共享领域包 | 定义知识条目、来源、阅读地图、报告、任务状态、可信等级、权限过滤和来源优先级等纯模型与纯函数 |
| `@opencode-workbench/knowledge-service` | 服务包 | 提供搜索、读取、关联推荐、轻量关系检索、报告生成、索引管理、权限校验和跨端 API |
| `@opencode-workbench/knowledge-worker` | 后台任务 | 处理模板保存后的异步索引、AI 摘要、stale 刷新和深度报告 |

v1 可以先让 `knowledge-service` 以内嵌包或单独进程两种方式运行，但代码边界应按独立子项目设计。不要让 `author-site` 继续直接拥有知识库的核心领域逻辑。

## 集成方

| 集成方 | 使用知识库的目的 |
|---|---|
| 创作端 `author-site` | 维护项目知识，保存模板后触发索引，向创作端 AI 提供报告和可读上下文 |
| 使用端 `viewer-site` | 未来提供使用端 AI 问答、页面解释、活动规则解释和用户侧帮助 |
| `agent-service` | 在主 AI 修改项目、解释配置或生成页面前获取知识报告 |
| 模板推荐 | 使用模板阅读地图、标签、摘要和任务导向入口做候选收敛 |
| 管理/运维系统 | 查看索引健康度、任务失败、stale 状态和知识来源覆盖情况 |
| 未来子系统 | 通过统一 API 获取知识搜索、读取和报告能力 |

## 知识来源分层

| 来源 | 维护者 | 主要载体 | 可信等级 |
|---|---|---|---|
| 系统规则知识 | 开发者 | 仓库内系统文档、配置规则、AI 操作边界 | 硬约束 |
| 当前项目知识 | 当前项目作者 | 当前项目 `knowledge/`、项目配置、页面结构、资源说明 | 当前事实 |
| 关联模板知识 | 模板作者 | 模板 `knowledge/`、模板页面、模板配置、模板阅读地图 | 默认参考 |
| 全量模板知识 | 模板作者/平台 | 所有模板的摘要、标签、阅读地图和任务入口 | 参考样本 |
| 会话上下文 | 当前用户 | 用户消息、当前任务、已读报告、已读文件 | 临时事实 |

知识库服务必须在报告中明确区分这些来源。其他模板的资料只能作为参考样本，不能覆盖当前项目事实；系统规则知识优先级最高。

## 核心模型

### 知识条目

知识条目是最小可检索对象，可以来自文档、页面、配置、资源、模板摘要或系统规则。核心字段包括：

- `id`：稳定标识。
- `sourceType`：系统规则、当前项目、关联模板、模板库、会话。
- `sourceId`：项目 ID、模板 ID 或系统来源 ID。
- `kind`：知识文档、页面、配置、资源、业务规则、设计规范、操作说明、索引产物。
- `title`：人和 AI 都能理解的标题。
- `summary`：说明该条目解决什么问题。
- `tags` / `keywords`：用于粗筛。
- `relations`：关联页面、配置项、组件、资源、模板或原文路径。
- `trustLevel`：硬约束、当前事实、默认参考、参考样本、AI 辅助摘要。
- `visibility`：可见范围，例如仅创作端、已发布使用端、Agent 内部、系统内部。
- `permissions`：声明该条目允许哪些主体执行搜索、读取、关联、报告引用和原文读取。
- `derivedFrom`：若条目由 LLM 摘要、模板快照或系统文档整理生成，记录来源条目和生成任务。
- `version` / `updatedAt`：用于 stale 判断。
- `readPath`：供 AI 继续读取原文或摘要的虚拟路径。

### 阅读地图

阅读地图是 AI 的导航入口，不是正文替代物。一个项目或模板的阅读地图至少包括：

- 总览：项目/模板适用场景、主要页面、核心业务对象、关键配置和更新时间。
- 结构索引：页面、配置、知识文档、资源和模板来源。
- 局部摘要：单页、单配置模块、单知识文档的用途和常见修改点。
- 任务导向入口：按修改页面、修改字段、生成页面、调整文案、排查异常等任务给出推荐阅读路径。
- 原文入口：指向具体文档、页面、配置或资源的可读路径。

### 知识报告

知识报告是主 AI 与知识库服务之间的主要协作产物。建议固定结构：

```text
结论摘要
相关资料
来源路径
可信等级
适用范围
建议主 AI 继续读取的原文
未找到的信息
风险提示
```

报告不是最终产品决策，只是经过检索、阅读和压缩后的高质量上下文。主 AI 仍负责结合用户任务完成修改、解释或生成。

## 服务能力

### 基础工具

- `knowledge.search`：按关键词、任务类型、来源、模板、页面、配置、标签搜索。
- `knowledge.read`：读取知识条目、阅读地图、模板页面摘要、配置摘要或原文。
- `knowledge.related`：根据当前页面、配置项、资源、模板或知识条目寻找相关资料。
- `knowledge.report`：围绕一个问题完成搜索、读取、筛选和压缩，返回结构化报告。

### v1 检索策略

v1 不追求完整 Graph RAG，而是用轻量结构达到足够好的检索质量：

- 结构化索引：按来源、类型、项目、模板、页面、配置、标签和可信等级过滤。
- 全文检索：对标题、摘要、关键词和正文片段做基础全文召回。
- 轻量关系索引：维护节点和边，例如页面关联配置、知识文档解释字段、模板包含页面、资源被页面使用。
- LLM 候选重排：对召回结果做语义重排和去噪。
- LLM 报告生成：读取少量高价值原文，压缩成带来源、可信等级和风险提示的报告。

这条路线利用 LLM 额度提升体验，但避免在 v1 引入 LightRAG、MiniRAG 或复杂图数据库。轻量关系索引只服务“找得到、说得清、可追溯”，不追求自动构建完整知识图谱。

### API 形态

后续实现可暴露 HTTP API 和 Agent 工具两种形态：

- `POST /knowledge/search`
- `GET /knowledge/items/{id}`
- `POST /knowledge/related`
- `POST /knowledge/report`
- `GET /knowledge/reports/{reportId}`
- `POST /knowledge/index-jobs`
- `GET /knowledge/index-jobs/{jobId}`

Agent 工具不应直接暴露底层检索后端细节。主 AI 只需要知道搜索、读取、关联和报告这四类能力。

### 权限边界作为服务能力

权限边界不应写死在创作端、使用端或 Agent Service 的调用方逻辑里，而应成为知识库服务的接口能力。所有调用方只声明“谁在什么场景下要做什么”，由知识库服务统一完成可见性过滤、来源降级和审计记录。

建议 v1 在 `knowledge-core` 中定义统一的访问上下文：

- `principalType`：调用主体类型，例如作者、访问者、创作端 AI、使用端 AI、系统任务、运维管理员。
- `principalId`：用户、会话、系统任务或服务账号标识。
- `tenantScope`：项目、模板、组织或系统范围；v1 可先支持项目和模板，预留组织字段。
- `surface`：调用入口，例如 author、viewer、agent-service、worker、admin。
- `purpose`：调用目的，例如编辑辅助、只读问答、模板推荐、项目诊断、索引生成、运维排查。
- `capabilities`：本次请求希望使用的能力，例如 search、readSummary、readOriginal、related、report、manageIndexJob。

知识库服务按访问上下文输出过滤后的结果，而不是让调用方自行过滤：

- `knowledge.search` 只返回当前主体可发现的条目。
- `knowledge.read` 在没有原文权限时可以返回摘要、拒绝原因或脱敏内容。
- `knowledge.related` 不应通过关系边泄露不可见条目的标题、路径或存在性。
- `knowledge.report` 必须只引用当前主体可用资料，并在报告中标注因权限不可见而无法确认的信息。
- `knowledge.index-jobs` 只允许系统任务、创作端作者或管理员创建和重试，使用端不可直接触发。

可见范围建议使用可组合策略，而不是单个布尔字段：

| 可见范围 | 含义 | 典型来源 |
|---|---|---|
| `system-internal` | 仅系统、管理员和 Agent 内部硬约束可见 | 系统规则、AI 操作边界 |
| `author-private` | 仅项目作者和创作端 AI 可见 | 草稿知识、未发布项目资料 |
| `project-agent` | 创作端 Agent 可读，但不向使用端展示 | 配置规则、开发说明、编辑约束 |
| `published-viewer` | 发布后使用端 AI 和访问者可用 | 活动规则、页面解释、公开帮助 |
| `template-library` | 模板推荐和模板实例化可用 | 模板阅读地图、模板摘要 |
| `ops-admin` | 运维和管理后台可见 | 索引健康、失败原因、审计记录 |

这套权限模型必须同时影响搜索、读取、报告和缓存。报告缓存键应包含访问上下文摘要，避免把创作端可见报告复用给使用端。

## 异步索引流程

### 模板保存后

1. 创作端保存项目为模板。
2. 模板快照和基础元数据立即写入。
3. 知识库服务创建异步索引任务，状态为 `pending`。
4. Worker 读取模板 workspace、页面、配置、资源清单和模板 `knowledge/`。
5. Worker 生成模板阅读地图、页面摘要、配置摘要、知识库摘要、标签、关键词和任务导向入口。
6. 任务成功后状态为 `ready`。
7. 任务失败或内容过大时状态为 `failed` 或 `partial`，保留基础索引兜底。
8. 模板内容更新后状态标记为 `stale`，后台重新整理。

### 当前项目更新后

当前项目不一定每次保存都做重型索引。v1 可采用轻量策略：

- 知识文档新增或更新后刷新该文档摘要和 manifest。
- 页面保存后更新页面摘要和最近修改标记。
- 配置结构变化后刷新配置摘要。
- AI 对话需要深度理解时再创建临时报告任务。

### 系统规则更新后

系统规则由开发期文件维护，发布时生成只读索引。涉及配置系统、Schema、写保护和 AI 操作边界时，知识报告必须提示主 AI 读取原文确认。

## 知识库管理员

知识库管理员是 `knowledge.report` 的高级形态，可以实现为专门检索的子智能体。它负责检索和报告，不直接修改项目，不替主 AI 做最终决策。

管理员内部可以执行多轮搜索、原文读取、候选排序和摘要压缩，但这些过程不直接进入主 AI 上下文。主 AI 只接收最终报告。

管理员必须遵守：

- 报告必须带来源路径。
- 报告必须标注可信等级。
- 未找到的信息必须明确列出。
- 涉及配置、Schema、系统规则和业务规则时，必须提示原文确认。
- 其他模板内容不能被表述为当前项目事实。
- 复杂检索可以异步生成深度报告，并缓存到会话或项目范围。

管理员子智能体可使用的内部工具包括：

- 搜索工具：从结构化索引、全文索引和轻量关系索引召回候选。
- 阅读工具：读取阅读地图、摘要和必要原文。
- 关联工具：沿页面、配置、字段、模板、知识文档之间的关系扩展候选。
- 重排工具：调用 LLM 对候选资料做排序、去重和适用性判断。
- 报告工具：把检索结果压缩为主 AI 可直接使用的结构化报告。

这种方式可以在不引入 LightRAG 的情况下接近“智能检索”的体验：无关搜索过程留在管理员内部，主 AI 只接收精简报告和必要引用。

## 远期增强检索规划

近期不考虑接入 LightRAG。当前阶段的重点是把知识库服务的领域模型、权限边界、阅读地图、异步索引、报告协议和 Basic 检索能力做稳。

远期如果模板库规模、跨模板关系检索和复杂知识报告需求明显增长，并且自研轻量索引与管理员报告无法满足召回质量，再重新评估 LightRAG、MiniRAG、向量数据库或图谱检索后端。它们只应作为检索增强模块，不应成为知识库子项目本身。

远期适合评估增强检索的场景：

- 跨模板、跨文档、跨页面的关系检索。
- 从模板知识库、页面摘要和配置摘要中抽取实体与关系。
- 生成复杂问题的深度报告。
- 需要同时利用图谱关系和向量语义的检索任务。

不应交给外部检索后端定义的内容：

- 知识来源分层。
- 权限和可见范围。
- 当前事实与模板参考的优先级。
- 报告格式。
- stale 状态。
- 主 AI 与知识库管理员的协作规则。

近期可以只预留一个轻量 `RetrievalBackend` 边界，但 v1 只实现 Basic 后端：

- `BasicRetrievalBackend`：结构化索引、标签、关键词、全文检索。
- 远期 `EmbeddingRetrievalBackend`：向量检索和 rerank。
- 远期 `GraphRetrievalBackend`：LightRAG 或其他图谱检索后端的适配层。

近期不做 LightRAG POC，也不设计 LightRAG 专用数据结构。所有索引产物先按项目自有模型设计，保证未来替换或接入任何检索后端时，业务 API 不需要变化。

## 存储与部署路线

### v1：激进功能闭环，轻量部署边界

- 新增 `knowledge-core`。
- `knowledge-service` 可先在现有 Node/Fastify 体系内运行。
- 存储使用项目现有数据目录或 SQLite/文件索引。
- 检索使用结构化索引、标签、关键词、简单全文检索和轻量关系索引。
- LLM 用于异步摘要、候选重排和知识报告生成。
- 异步任务可先复用轻量队列或文件状态。

v1 不再以“最小功能”作为目标，而以“完整知识协作闭环”作为目标。v1 应一次性打通：

- 当前项目知识、系统规则知识、模板知识和会话上下文的统一来源模型。
- 权限上下文和可见范围过滤，确保未来系统接入时不需要重写权限逻辑。
- 模板保存后的基础阅读地图和异步索引任务。
- 当前项目的轻量索引刷新和按需深度报告。
- Agent Service 的 `knowledge.report` 调用协议。
- 创作端可观察的索引状态、失败重试和 stale 提示。
- 使用端只读问答的权限过滤接口预留；是否首期接入 UI 可按排期决定，但服务能力不应缺失。

v1 可以暂不做的不是领域能力，而是重型基础设施：不独立部署 worker、不引入向量库、不做 LightRAG POC、不建设完整运维看板。服务边界和接口能力必须先做对。

### v2：独立服务化

- `knowledge-service` 独立进程。
- `knowledge-worker` 独立后台任务。
- 提供 HTTP API 给创作端、使用端和 Agent Service。
- 引入报告缓存、任务状态和 stale 刷新。
- 优化 LLM 异步整理、管理员子智能体和报告复用机制。

### v3：多端知识基础设施

- 使用端 AI、模板推荐、项目诊断和运营工具统一调用知识库服务。
- 增加审计、权限、组织隔离和索引健康看板。
- 根据规模再考虑更强的图数据库、向量库、多模态解析或 LightRAG 类增强后端。

## 多端集成方式

### 创作端

- 维护当前项目知识库。
- 保存模板后触发索引任务。
- 在 AI 对话中调用 `knowledge.report`。
- 展示索引状态、摘要生成状态和失败重试入口。

### 使用端

- 只读取发布后允许公开的知识。
- 不暴露创作端内部草稿、私有模板和开发者系统规则全文。
- 可用于页面解释、活动规则问答、用户帮助和公开内容检索。

### Agent Service

- 在修改配置、页面或项目结构前请求知识报告。
- 需要硬约束时读取系统规则原文。
- 通过报告减少主 AI 的无关上下文。

### 未来系统

- 通过统一 API 接入，不直接读取项目文件系统。
- 由知识库服务负责权限过滤、来源标注和报告生成。

## 迁移策略

1. 保留现有项目 `knowledge/manifest.json` 兼容。
2. 把现有系统规则知识纳入只读系统来源。
3. 为模板快照补充基础阅读地图，不立即要求 AI 摘要全部完成。
4. 新增知识报告协议，先给 Agent Service 使用。
5. 再逐步让创作端 UI、模板推荐和使用端接入。
6. 远期再评估是否需要向量检索、图谱检索或 LightRAG 等增强后端。

迁移时应把现有 `system` / `user` 来源映射到新领域模型，而不是直接废弃：

| 现有来源 | 新 `sourceType` | 默认 `trustLevel` | 默认 `visibility` |
|---|---|---|---|
| `system` 系统知识 | 系统规则知识 | 硬约束或默认参考，按文档类型区分 | `system-internal` 或 `project-agent` |
| `user` 工作空间知识 | 当前项目知识 | 当前事实 | 默认 `author-private`，发布时可显式标记为 `published-viewer` |
| 模板快照页面/配置 | 关联模板知识或模板库 | 默认参考或参考样本 | `template-library` |
| LLM 摘要产物 | 索引产物 | AI 辅助摘要 | 继承原文可见范围，不得扩大权限 |

## 风险与待确认事项

- 过早服务化会增加部署、队列、存储和权限复杂度，应先抽领域边界。
- 如果知识库服务只做文件 CRUD，会失去独立子项目的价值。
- 如果权限过滤只写在调用方，未来接入使用端、运营系统或外部系统时会重复实现并产生泄露风险；v1 必须把权限判断收敛到知识库服务。
- 报告缓存若不包含访问上下文，可能把高权限报告错误复用给低权限主体。
- LightRAG 能力强但部署链路较重，近期不接入；过早引入会分散领域建模和服务边界的工作重点。
- 自研轻量关系索引不要演变成复杂图数据库，v1 应控制在可维护的节点、边和关系类型内。
- LLM 额度充足能提升摘要和报告质量，但仍需缓存、stale 和原文确认，不能每次都实时深度检索。
- AI 摘要可能遗漏细节，报告必须保留原文读取入口。
- 多端集成需要清晰权限边界，尤其是使用端不能看到创作端私有知识。
- 模板知识、当前项目知识和系统规则的优先级必须稳定，否则主 AI 容易混淆事实与参考。
- 异步索引任务失败时必须有基础索引兜底。
- 报告缓存需要版本和 stale 机制，否则可能把过期知识继续提供给 AI。
