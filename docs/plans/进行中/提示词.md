## 系统提示词的问题分析

我仔细重读了系统提示词，找到了**至少 3 处导致 AI 私自加配置项的漏洞**：

---

### 问题 1：「加个配置」示例模糊了"用户明确指示"的门槛

**位置**：「页面内容编辑」章节

> "给详情页加个配置" → `demos/{详情页 demoId}/`

这里把「加个配置」作为自然语言匹配的示例，而后面**重要约束**又说：

> 只有当用户**明确说**「加一个配置」「删除这个字段」等**才可操作**

**冲突**：用户说「加个图片展示区域」—— AI 可能理解为"要给组件加一个图片"（样式修改，不该加配置），也可能理解为"加一个可配置的图片"（配置项修改，该加配置）。这两个规则打架，AI 没有明确的标准来判断边界。

**根因**：缺少「什么算配置请求，什么算样式请求」的判断标准。

---

### 问题 2：「文件修改决策规则」第 2 条和第 3 条的边界模糊

```
2. 配置项修改 → 修改 config.schema.json
3. 组件结构修改 → 修改 index.tsx
```

这两个规则**没有互斥性说明**。当用户说「加一个可以自定义颜色的标题」时：

- 规则 2 说「配置项修改」→ 要在 config 加 color 字段
- 规则 3 说「组件结构修改」→ 要在 index.tsx 加标题组件

AI **可能同时执行两者**——在 index.tsx 加了标题组件，顺手在 config.schema.json 加了个 color 字段，以为这是「配套的」。但用户只要求了「加标题」，没有要求「加配置」。

**根因**：规则没有说「配置字段必须有且仅有用户明确提到的那些」。

---

### 问题 3：DemoProps 接口示例产生了「配置项暗示」

```tsx
interface DemoProps {
  title: string;
  description?: string;
  showBadge?: boolean;
}
```

这个示例展示了**配置驱动**的模式。当 AI 写代码时，看到页面里用到一个动态文本，会**自然地**认为「这个应该有对应的配置字段」，然后主动在 config.schema.json 补上。

**根因**：示例中没有说明「哪些字段不需要进配置，直接硬编码即可」，AI 会默认「所有 props 里的动态数据都需要配置」。

---

### 建议的修复方向

| 问题 | 建议修复 |
|------|---------|
| 1 | 删除「加个配置」这个模棱两可的示例，或明确补充「只有用户使用"配置"二字才触发」 |
| 2 | 在规则 2 和 3 之间加一句：「两个规则互斥：配置项修改只修改 schema，不修改组件；组件结构修改只修改 tsx，不修改 schema」 |
| 3 | 在 DemoProps 示例旁加注释：「仅声明页面级配置字段，硬编码数据直接从组件内部写死，不需要加配置项」 |
