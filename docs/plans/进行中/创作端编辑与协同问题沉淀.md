# 创作端编辑与协同问题沉淀

## 当前状态

正式环境 auto-checkpoint 流程中断，页面编辑内容实际已保存，但版本快照创建失败，导致 UI 协同状态异常。具体校验失败类型待诊断。

## 问题 P1：auto-checkpoint Schema 校验失败

### 现象

1. 正式环境部署后，编辑页面时自动保存（auto-checkpoint）失败
2. 浏览器控制台报错（见下方日志）
3. 编辑内容**实际已持久化**——返回首页重新打开项目，修改仍在
4. 右上角协同状态持续显示"连接中"
5. 点击返回首页时弹出"修改仍在同步"对话框

### 日志输出

```
[auto-checkpoint] failed: ProjectApiError: 页面 Schema 校验失败
    at ri.createPageVersion (page-3c160adfdfb4df2b.js:1:171546)
    at async page-3c160adfdfb4df2b.js:1:341635
```

控制台无其他输出。当前 `console.warn` 对 `Error` 对象仅打印 `name: message`，不会展开 `details` 属性，因此无法从现有日志判断具体校验失败类型。

### 发生时机

- 仅在正式环境部署后复现
- 本地开发环境未复现

### 影响范围

- 所有正式环境项目的 auto-checkpoint 流程
- 不影响页面文件的实际保存（PUT 文件写入成功）
- 影响版本历史快照的创建
- 影响 UI 协同状态显示和退出流程

`createAutoCheckpoint` 函数位于 `packages/author-site/src/app/demo/[id]/edit/hooks/useVersionControl.ts` L435-518，按顺序执行以下步骤：

| ① | L448 | `flushWorkspaceCollab()` — 协同状态落盘 | ✅ 成功 |
| ② | L449 | `flushCanvasState()` — 画布状态落盘 | ✅ 成功 |
| ③ | L463-473 | `PUT /api/sessions/${sessionId}/files/${activeDemoId}` — 页面文件写入 | ✅ 成功（证据：编辑内容已持久化） |

| ⑤ | L492-500 | `POST /api/sessions/${sessionId}/checkpoint` — 检查点记录 | ⏭️ 未执行（步骤④抛异常） |

### 步骤④的失败链路

**前端 API 客户端**：`packages/author-site/src/lib/project-api.ts` L155-179

```typescript
async createPageVersion(projectId, demoId, request?) {
  const response = await this.localRequest(
    `/api/projects/${projectId}/resources/page/${demoId}/versions`,
    { method: 'POST', body: JSON.stringify(request ?? {}) },
  );
  if (!response.success || !response.data) {
    throw new ProjectApiError(
      response.error?.code || 'FILE_WRITE_ERROR',
      response.error?.message || '创建页面版本失败',
      response.error?.details,  // ← 包含 validation.issues，但 console.warn 不展开
    );
  }
}
```

**API Route Handler**：`packages/author-site/src/app/api/projects/[projectId]/resources/[kind]/[resourceId]/versions/route.ts` L112-216

- L141-177：当 `kind === "page"` 且携带 `sessionId` 时，先通过 `flushAndSyncProjectWorkspace` 将 live workspace 同步到 canonical
- L179-198：调用 `projectService().resourceVersionCreate()`
- L199-213：错误处理，将 `VALIDATION_BLOCKED` 映射为 `VALIDATION_ERROR`（HTTP 422），并将 `validation` 对象放入响应 `details`

**核心校验逻辑**：`packages/project-core/src/service.ts`

`createPageVersion`（L2604-2644）在创建版本前执行 Schema 校验：

```typescript
// L2639-2644
const validation = this.validateSchemaPair(
  this.readProjectConfig(sourceWorkspacePath),  // 读取 canonical workspace 的 project.config.schema.json
  files.schema,                                   // 读取页面的 config.schema.json
);
if (!validation.ok)
  return fail("VALIDATION_BLOCKED", "页面 Schema 校验失败", { validation });
```

`validateSchemaPair`（L5882-5915）执行以下检查：

1. 解析项目级 Schema（`project.config.schema.json`），提取 `properties` 键名集合
2. 解析页面级 Schema（`config.schema.json`），提取 `properties` 键名
3. 如果两级 Schema 存在同名键 → 产生 `SCHEMA_CONFLICT` blocking issue

`parseSchema`（L5936-5975）可能产生的 blocking issue 类型：

| 错误码 | 触发条件 | 含义 |
|--------|----------|------|
| `INVALID_JSON` | `JSON.parse(schema)` 抛异常 | Schema 文件不是合法 JSON（含空文件） |
| `INVALID_SCHEMA` | `properties` 字段不是对象类型 | Schema 结构不合法 |
| `SCHEMA_CONFLICT` | 项目级和页面级 `properties` 存在同名键 | 两级配置字段冲突 |

### UI 异常链路

**"连接中"显示逻辑**：`packages/author-site/src/app/demo/[id]/edit/page.tsx` L5942-5947

```typescript
if (!workspaceFlushError && !hasPendingWorkspaceFlush) {
  if (collabStatuses.includes("error")) {
    collabStatusLabel = "协同异常";
  } else if (collabStatuses.includes("connecting")) {
    collabStatusLabel = "连接中";
  }
}
```

`collabStatuses`（L5905-5911）聚合了 5 个协同房间的状态：`activePageCollabStatuses`、`activeSchemaCollab`、`projectSchemaCollab`、`workspaceTreeCollab`、`canvasLayoutCollab`。任一房间处于 `"connecting"` 即显示"连接中"。

**退出弹窗触发逻辑**：`page.tsx` L4974-4986

```typescript
const hasExitSyncRisk =
  (hasPendingWorkspaceFlush && !authoritySynced) ||
  workspaceFlushError !== null ||
  syncInFlightRef.current ||
  syncDebounceRef.current !== null ||
  (hasUnsavedChanges &&
    exitSyncStatuses.some(
      (status) => status === "error" || status === "offline" ||
                  status === "saving" || status === "connecting",
    ));
```

当 `hasUnsavedChanges` 为 `true`（因步骤⑥未执行）且任一协同房间处于 `"connecting"` 时，`hasExitSyncRisk` 为 `true`，触发"修改仍在同步"弹窗。
