## P1：EXTERNAL_DRIFT 阻断 AI 全部操作 ✅

### 现象

AI 的所有文件操作（readFile、editFile、writeFile 等）全部返回 `WORKSPACE_EXTERNAL_DRIFT`，一旦 drift 发生，后续操作持续失败，无自愈能力。

### 根因

**agent-service pi-tools 直接实例化 `WorkspaceMutationAuthority`，绕过了 author-site 的 auto-retry 包装。**

- `WorkspaceMutationAuthority.getSnapshot()` 和 `mutate()` 都会检查磁盘哈希与 state rootHash 是否一致，不一致则抛 `WORKSPACE_EXTERNAL_DRIFT`
- author-site 的 `commitWorkspaceMutation()` 有 auto-retry：捕获 DRIFT → `reconcileAdopt()` → 重试 1 次。但 pi-tools 直接调用 Authority 本地实例，不在 HTTP 调用路径上，auto-retry 完全无效
- `getSnapshot()` 连读操作也检查磁盘一致性，导致 DRIFT 时读写全部阻断

### 影响范围

所有通过 AI 对话对 live workspace 执行的文件操作（读+写），bash 工具返回 `WORKSPACE_AUTHORITY_REQUIRED` 是独立的设计行为（live workspace 限制 bash 为只读）。

### 修复方案

**推荐方案：pi-tools 层添加 auto-retry wrapper**

捕获 `WORKSPACE_EXTERNAL_DRIFT` → 调用 `authority.reconcileAdopt()` → 重试原操作（读写均覆盖）。与 author-site 层机制对齐。

关键代码入口：

- `reconcileAdopt()` 已在 `workspace-mutation-authority.ts:287` 实现
- HTTP API `POST /api/workspace-authority/.../reconcile/adopt` 已暴露
- 需要 wrapper 的 pi-tools：`file-tools.ts`（readFile/writeFile）、`edit-file-tool.ts`、`read-file-lines-tool.ts`

### 验证状态

- ✅ 已修复，pnpm check:agent 通过
- 改动文件：`file-tools.ts`、`edit-file-tool.ts`、`read-file-lines-tool.ts`
- 修复方式：在每个 pi-tool 的 `getSnapshot()` 和 `mutate()` 调用处添加 EXTERNAL_DRIFT 捕获 → `reconcileAdopt()` → 重试

## P2：页面删除后 `workspace/demos/` 残留空目录 ✅

### 现象

通过编辑器 UI 或 AI 对话删除页面后，`workspace/demos/{pageId}/` 空目录残留。用户报告空目录会导致创建同名页面失败。

### 根因

Authority 的 `delete_path` 操作只删除单个文件，不清理父目录：

1. `createManagedPageDeleteOperations`（`demos/[demoId]/route.ts:78-111`）只遍历文件生成 `delete_path`
2. Authority `apply`（`workspace-mutation-authority.ts:784-787`）对 `delete_path` 执行 `fs.rmSync(filePath, { force: true })`，不带 `recursive`
3. 非 live 路径的 `deleteWorkspaceDemoPage()` 使用 `fs.rmSync(dir, { recursive: true, force: true })`，行为正确

### 受影响入口（4 处）

| #   | 文件                                          | 函数                         | 场景              |
| --- | --------------------------------------------- | ---------------------------- | ----------------- |
| 1   | `author-site/.../demos/[demoId]/route.ts`     | DELETE handler (L697-719)    | UI 单页删除       |
| 2   | `author-site/.../folders/[folderId]/route.ts` | DELETE handler (L481-504)    | UI 文件夹连同删除 |
| 3   | `agent-service/.../delete-page-tool.ts`       | `deleteOnePage` (L433-459)   | Agent 单页删除    |
| 4   | `agent-service/.../delete-page-tool.ts`       | `deletePageBatch` (L543-573) | Agent 批量删除    |

### 修复方案

在每个入口的 `commitWorkspaceMutation` / `authority.mutate` 成功后，追加 `fs.rmSync(pageDir, { recursive: true, force: true })`：

1. `demos/[demoId]/route.ts` L718 后：清理 `getDemoDirPath(workspacePath, demoId)`
2. `folders/[folderId]/route.ts` L504 后：遍历 `deletedPageIds` 清理各页面目录
3. `deleteOnePage` L459 后：清理 `pageDir`
4. `deletePageBatch` L573 后：遍历 `pageIds` 清理各页面目录

### 验证状态

- ✅ 已修复，pnpm check:author + pnpm check:agent 通过
- 改动文件：`demos/[demoId]/route.ts`、`folders/[folderId]/route.ts`、`delete-page-tool.ts`
- 修复方式：在每个入口的 mutation 成功后追加 `fs.rmSync(pageDir, { recursive: true, force: true })`

## P3：AI 自动修复循环重复拼接 ✅

### 现象

- 页面文件被 AI 完整重复拼接两次（如 158 行 → 316 行），包含两个 `interface DemoProps`、两个 `export default function`
- 预览编译失败，报 `DUPLICATE_TOP_LEVEL_DECLARATION`
- AI 自动修复反复触发，循环不终止

### 根因摘要

**拼接发生在 AI 模型生成内容层面，不在服务端代码路径。** 所有 writeFile/editFile/WMA 路径均为纯覆盖写入，不存在拼接逻辑。

**7 个系统级诱因：**

1. **Pi Agent 会话历史累积**：每次 readFile/writeFile 的完整内容永久留在对话历史，AI 处理多版本时混淆
2. **Auto-repair prompt 缺少「先读后写」指令**：AI 从对话历史中「回忆」可能过时的版本
3. **前端对话历史前缀截断**：截断到 2000 字符，AI 看到不完整片段后在此基础上生成
4. **writeFile 工具描述笼统**：未强调覆盖语义，无拼接防护警告
5. **写后验证反馈措辞**：「继续修复」指令让 AI 在已污染文件上再次尝试
6. **editFile 多重匹配拒绝**：文件被拼接后 editFile 必然失败，迫使 AI 使用更易拼接的 writeFile
7. **验证在写入之后执行**：文件先被污染，验证才触发

**4 个防线失效：**

1. **Per-fingerprint 计数器**：fingerprint 含 `codeHash`，每次修复内容变化 → fingerprint 变化 → 计数器重置
2. **Page-level 预算**：存储在 localStorage，清除浏览器数据即重置
3. **重复错误指纹拦截**：`lastPreviewErrorFingerprintRef` 是内存 ref，页面刷新即重置
4. **editFile 多重匹配检测**：间接恶化，迫使 AI 使用 writeFile

### 修复方向（推荐实施优先级）

**批次一（源头防护）：方向 A + D + F + G**

- **方向 A（写前拼接检测）**：`file-tools.ts` writeFile 的 execute 中，写入前对比 `args.content` 与 `existing`。若 `args.content.length > existing.length * 1.5` 且 `args.content.includes(existing)`，返回错误指令要求 AI 先 readFile 再只输出新内容
- **方向 D（验证前回滚）**：`file-tools.ts` writeFile 改为先验证 `args.content`，通过后再写入；或验证失败时回滚到 `existing`
- **方向 F（先读后写指令）**：`edit/page.tsx` L2153-2167 的 auto-repair prompt 增加「修复前必须先用 readFile 读取当前文件内容」
- **方向 G（工具描述增强）**：`file-tools.ts` L135 改为明确覆盖语义：「Write the complete new content of a file, replacing the entire existing content.」

**批次二（循环终止）：方向 B**

- **方向 B（fingerprint 修复）**：`auto-preview-repair-guard.ts` L147-173 的 `buildAutoPreviewRepairFingerprint` 去掉 `codeHash`，改用稳定标识（`projectId::pageId::diagnosticCode`）

**批次三（长期增强，可选）：方向 C + E**

- **方向 C（服务端预算）**：将 page-level 预算移到 workspace authority state 或 session metadata
- **方向 E（自动截断）**：检测到文件被完整重复拼接时自动截断为前半部分

## 已解决归档

### AI 编辑后"保存中"永久卡住 ✅

**根因**：`onError` 未清 pending 标志 + 状态机未感知 Authority committed revision。

**修复**：`onError` 回调添加 `setHasPendingWorkspaceFlush(false)`；编辑页接入 `useWorkspaceAuthorityState` hook。

**可复用经验**：

- 保存状态卡死排查入口：先看 `hasPendingWorkspaceFlush` 的所有清除路径（仅 3 处），再看 `markWorkspaceChanged()` 的所有调用点（9 处）
- Authority 状态感知模式：前端已有 `useWorkspaceAuthorityState` hook 轮询 server committed revision
- `onError` 兜底原则：异步流水线的 `onError` 必须清除所有 in-flight 标志
