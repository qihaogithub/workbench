## 问题一：AI 编辑后"保存中"永久卡住

### 现象

- 用户通过 AI 对话让 AI 创建/编辑页面后，编辑器右上角始终显示"保存中…"
- 退出编辑页时弹出对话框："最新修改尚未确认同步完成。网络恢复或同步完成后可安全退出。"
- 用户需要点击"仍然退出"才能离开

### 影响范围

- 所有通过 AI 对话创建或编辑页面文件的操作
- 原型页面（prototype-html-css）和 React 页面均受影响
- 每次 AI 完成编辑后必现

### 复现路径

对话记录样本：`session-1783948316496-lwigvfylf`（2026-07-13）

1. 进入编辑页，新建或打开一个项目
2. 在 AI 对话框中发送消息（如"随便新建一个原型页面"）
3. AI 通过 `writeFile` 工具创建文件（prototype.html、prototype.css、config.schema.json、workspace-tree.json）
4. AI 回复完成后，右上角状态从"AI 正在更新"变为"保存中…"，且不再变化
5. 点击返回按钮，弹出"修改仍在同步"对话框

### 根因分析

#### 1. 数据流全链路追踪

**AI 侧（agent-service）：**

```
AI writeFile → agent-service workspace authority route → 提交 mutation → 返回 receipt
                                                                          ↓
                                                        广播 workspace_mutation_committed 事件
                                                        （通过 WS 流，revision 15→16→17→18）
```

AI 的每次 `writeFile` 都通过 agent-service 的 workspace authority 路由提交，返回包含 `committed: true` 的持久化回执。所有变更已在 server 端完成持久化。

**前端（author-site edit page）：**

```
AI stream finish → event.files 包含变更文件列表
                   ↓
onFilesChange 回调 → handleAiFilesChange()
                     ↓
  ① handleWorkspaceTreeChanged() → markWorkspaceChanged()  ← 设置 hasPendingWorkspaceFlush=true
     + syncWorkspaceFileToCollab("workspace-tree.json")     ← 异步刷新 Yjs 文档
  ② fetch /api/sessions/${sessionId}/files                  ← 重新拉取所有页面文件
  ③ 更新 demoPages, prototypes 等 state
  ④ markWorkspaceChanged()                                  ← 再次设置 hasPendingWorkspaceFlush=true
                   ↓
WorkspaceAutosaveScheduler debounce 800ms 后触发
                   ↓
commitFn → syncWorkspaceToProject()
  ├─ persistActivePageToSession()   ← API 写入活动页面
  ├─ flushWorkspaceCollab()         ← API 将 Yjs 内存态 flush 到磁盘
  └─ persistWorkspaceToProject()    ← API 同步 workspace 到项目
                   ↓
成功 → onCommitted → setHasPendingWorkspaceFlush(false)
失败 → onError → setWorkspaceFlushError(message)
```

#### 2. 核心断裂点

**问题 A：前端保存状态机不知道 AI 已提交了 mutation**

- `saveState` 由 `computeSaveStateFromContext()` 计算，关键输入是 `hasPendingWorkspaceFlush` 和 `workspaceFlushRevision`
- 当 `hasPendingWorkspaceFlush=true && workspaceFlushRevision>0` 时，`isMutationInFlight=true` → 状态为 `"saving"`（"保存中…"）
- AI 的 mutation 在 server 端已经 committed，但前端仍将其视为"本地待保存变更"
- `useWorkspaceAuthorityState` hook 已实现轮询 authority 事件并更新 `committedRevision`，但 **edit page 完全没有使用该 hook**（grep 确认 0 处引用）

**问题 B：`handleAiFilesChange` 将 server-side commit 误标为本地脏状态**

- `handleAiFilesChange` 在处理 AI 文件变更时调用 `markWorkspaceChanged()` 至少两次
- 每次调用都设置 `hasPendingWorkspaceFlush=true` 并递增 `workspaceFlushRevision`
- 这触发 `WorkspaceAutosaveScheduler` 尝试将"变更"重新同步到 server
- 但这些变更已经在 server 上了（AI 通过 authority 直接提交的）

**问题 C：syncWorkspaceToProject 流水线可能在 AI 编辑后失败**

`syncWorkspaceToProject` 执行三步流水线：

1. `persistActivePageToSession()` — 将当前编辑器代码写入 session
2. `flushWorkspaceCollab()` — 将 Yjs 内存文档 flush 到磁盘
3. `persistWorkspaceToProject()` — 将 workspace 同步到项目

AI 直接通过 authority 写入磁盘文件，**绕过了 Yjs 协同层**。这导致：

- 前端 Yjs 文档（workspace-tree.json 等）持有旧内容
- `syncWorkspaceFileToCollab` 尝试从磁盘读取并更新 Yjs，但存在时序竞争
- `flushWorkspaceCollab` 可能将 Yjs 旧内容写回磁盘，与 AI 的新内容冲突
- 如果流水线任何步骤失败，`onCommitted` 不被调用，`hasPendingWorkspaceFlush` 永远不会清除

#### 3. 状态机逻辑确认

```
computeSaveStateFromContext:
  isMutationInFlight = hasPendingWorkspaceFlush && workspaceFlushRevision > 0
  → return "saving"  (显示"保存中…")

hasExitSyncRisk:
  hasPendingWorkspaceFlush           ← true（被 markWorkspaceChanged 设置后未清除）
  || workspaceFlushError !== null    ← 如果 sync 失败则有值
  || (hasUnsavedChanges && collab 异常)
  → true → 弹出退出对话框
```

## 问题二：WORKSPACE_EXTERNAL_DRIFT 阻断 AI 全部操作

### 现象

- 用户在 AI 对话中发送编辑请求后，AI 的所有文件操作（readFile、readFileWithLines、editFile、writeFile、bash）全部失败
- 错误信息统一为 `WORKSPACE_EXTERNAL_DRIFT`（工作区外部漂移）
- AI 无法完成任何文件读写，只能告知用户"稍后刷新页面重试"
- bash 命令返回不同的错误 `WORKSPACE_AUTHORITY_REQUIRED`（WMA 架构设计禁止 bash 直接写 live workspace）

### 影响范围

- 所有通过 AI 对话对 live workspace 执行的文件操作
- 读操作和写操作均受影响（`getSnapshot()` 本身就会检查磁盘一致性）
- 一旦 drift 发生，AI 的所有后续操作持续失败，无自愈能力

### 复现路径

对话记录样本：`session-1783948316496-lwigvfylf`（2026-07-13）

1. 进入编辑页，打开项目 `proj_1783585261167_7h7prw`，workspace `live-1783654372084-5b2qhswm4`
2. 在 AI 对话框中发送"标题改为蓝色"
3. AI 调用 `readFileWithLines` 读取 `prototype.css` → 返回 `WORKSPACE_EXTERNAL_DRIFT`
4. AI 依次尝试 `readFile`、`bash`、再次 `readFile`、`readFileWithLines` → 全部失败
5. AI 尝试 `editFile` 和 `writeFile` → 同样返回 `WORKSPACE_EXTERNAL_DRIFT`
6. AI 放弃操作，告知用户手动修改

### 根因分析

#### 1. EXTERNAL_DRIFT 触发机制

`WorkspaceMutationAuthority`（`packages/agent-service/src/workspace/workspace-mutation-authority.ts`）在以下位置抛出 WORKSPACE_EXTERNAL_DRIFT：

| 方法 | 行号 | 触发条件 |
|------|------|----------|
| `getSnapshot()` | L226-227 | `rootHash(磁盘实际哈希) !== state.rootHash`（**读操作时触发**） |
| `mutate()` | L514-515 | 同上，在 prepare 之前检查（**写操作时触发**） |

核心检测逻辑：将 live workspace 磁盘上所有受管文件的 SHA-256 哈希计算 rootHash，与 authority state JSON 中记录的 rootHash 比较。任何不一致（即使一个文件一个字符差异）都会触发 DRIFT。

#### 2. agent-service pi-tools 调用链路

```
AI 工具调用 (pi-tools)
  │
  ├─ resolveLiveWorkspaceMutationContext(workingDir)
  │     └─ new WorkspaceMutationAuthority({ dataDir, resolveWorkspacePath })
  │
  ├─ [读操作] authority.getSnapshot(projectId, workspaceId)
  │     ├─ ensureBootstrap() → 读取/创建 authority state
  │     ├─ readResourceHashes(磁盘路径) → 计算 actual rootHash
  │     └─ 比较 actual vs state.rootHash → 不等则抛 EXTERNAL_DRIFT ★
  │
  └─ [写操作] authority.mutate(request)
        ├─ ensureBootstrap()
        ├─ readResourceHashes → 比较 → 不等则抛 EXTERNAL_DRIFT ★
        ├─ prepare() → 乐观锁校验
        └─ apply() → 写磁盘 + 更新 state
```

所有 pi-tools（readFile、readFileWithLines、editFile、writeFile、listFiles）都经过 `getSnapshot()` 或 `mutate()`，因此 DRIFT 会阻断一切操作。

#### 3. auto-retry 覆盖缺口

| 层级 | 是否有 auto-retry | 覆盖范围 |
|------|-------------------|----------|
| **author-site** | 有（`workspace-authority-client.ts` L162-184） | 仅覆盖通过 `commitWorkspaceMutation()` 的**写操作** |
| **agent-service pi-tools** | **无** | AI 工具调用完全暴露，读操作和写操作均无保护 |

author-site 的 auto-retry 实现：捕获 `WORKSPACE_EXTERNAL_DRIFT` → 调用 `reconcileWorkspaceAuthority({ mode: "adopt" })` → 重试原请求（仅 1 次）。

agent-service 的 pi-tools 直接实例化 `WorkspaceMutationAuthority` 类（本地进程内调用），不经过 author-site HTTP API，因此 author-site 的 auto-retry 包装完全不在调用路径上。

#### 4. reconcile/adopt 可用性

- `WorkspaceMutationAuthority` 类本身有 `reconcileAdopt()` 方法（L287-307）：读取磁盘哈希 → 更新 state JSON → 递增 revision
- agent-service 已暴露 HTTP API：`POST /api/workspace-authority/projects/:pid/workspaces/:wid/reconcile/adopt?sessionId=...`
- pi-tools 已通过 `resolveLiveWorkspaceMutationContext()` 获取了 Authority 实例，理论上可直接调用 `authority.reconcileAdopt()`

#### 5. bash 工具的不同错误

bash 工具（`bash-tool.ts` L20-36）返回 `WORKSPACE_AUTHORITY_REQUIRED` 是 WMA 架构的**设计行为**：live workspace 的 bash 命令被限制为只读，写操作必须通过 editFile/writeFile 走 Authority 路径。这与 EXTERNAL_DRIFT 是独立的两个问题。

### 修复方案建议

| 方案 | 描述 | 评估 |
|------|------|------|
| A. pi-tools 层添加 auto-retry wrapper | 捕获 DRIFT → `reconcileAdopt()` → 重试（读+写均覆盖） | **推荐** — 与 author-site 层机制对齐 |
| B. 读操作绕过 drift 检查 | `getSnapshot()` 检测到 drift 时返回磁盘实际内容而非抛错 | 短期缓解 — 至少读操作不被阻断 |
| C. Authority 启动时自动 reconcile | `ensureBootstrap()` 检测到 drift 时自动 adopt | 防御增强 — 但静默采纳有数据安全风险 |

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| `flushWorkspaceCollab` 用 Yjs 旧内容覆盖 AI 新文件 | 数据回退 | 批次二的 authority revision 守卫可避免冗余 flush |
| sync 流水线静默失败时 `hasPendingWorkspaceFlush` 永不清除 | 永久卡住 | 批次一的 `onError` 修复直接解决 |
| stale revision 导致 `onCommitted` 被跳过 | saving 状态卡住 | `commitNow` 改为成功即通知，仅 `appliedRevision` 单调递增 |
| collab `flush()` 网络异常未捕获 | collab status 永久 saving | `try-catch` 包裹 fetch，网络错误转 error 状态 |
| 2s 轮询延迟导致 "保存中…" 短暂闪烁 | UX 瑕疵 | 可接受（save 确实 pending confirmation），批次三 WebSocket 可消除 |
| `workspaceFlushRevisionRef` 与 server revision 不对齐时误清 pending | 数据丢失 | 批次二使用 `>=` 比较，仅 server 追上时才清除；新本地编辑会递增 ref 使其高于 server |
| Authority 轮询 API 失败 | 状态不准 | hook 自带错误处理和重连；批次一 `onError` 修复作为独立兜底 |

## 被拒绝的替代方案

| 方案 | 拒绝原因 |
|------|----------|
| 新建 WebSocket `useAuthorityStream` hook（批次三方案）提前到批次一/二 | 已有 server-side WebSocket `/stream` 端点，但当前 2s HTTP 轮询已足够解决问题。WebSocket 增加复杂度（重连、回退），ROI 不高，可后续按需升级 |
| `handleAiFilesChange` 中跳过 `markWorkspaceChanged()` | 风险较高：`markWorkspaceChanged` 同时更新 scheduler 和触发 collab 同步，移除可能导致 Yjs 文档不一致。用 authority revision 守卫更安全 |
| 修改 `computeSaveStateFromContext` 函数签名 | 不需要：通过在调用方构造正确的 context（加入 `authoritySynced`），即可达到目的，不改底层 API |
