# 首页加载缓慢与卡片点击延迟问题分析报告

> 分析日期：2026-05-02

---

## 一、问题背景

### 问题描述

用户访问首页（`/`）后，遇到两个关联的性能问题：

| 问题 | 预期行为 | 实际行为 |
|------|---------|---------|
| 首页加载缓慢 | 打开首页后，项目卡片应在 1-2 秒内显示 | 信息需要加载很久才出现，用户长时间看到骨架屏占位动画 |
| 卡片点击无效 | 点击项目卡片应立即跳转到编辑页 | 一开始点击卡片无任何反应，很久之后点击才生效 |

### 发生场景

- 每次打开首页时稳定复现
- 两个问题存在因果关系：卡片点击无效是首页加载缓慢的直接后果

### 涉及组件

- 首页页面入口、首页内容组件、项目卡片组件、数据获取 Hook、后端 API 路由、文件系统工具函数

## 二、根因分析

### 根因一（核心根因）：首页采用纯客户端渲染（CSR），无服务端数据预取

**这是导致两个问题的根本原因。**

#### 证据 1：页面组件架构导致数据只能在客户端获取

首页的组件层级如下：

```
page.tsx（服务端组件）
  └── MainLayout（服务端组件）
        └── HomePage（'use client' 客户端组件）
              └── useDemos() → SWR → fetch('/api/demos')
```

- [page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/app/page.tsx) 是服务端组件，但它只做布局组合，不获取数据
- [home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx) 标记为 `'use client'`，数据获取逻辑（`useDemos()`）完全在客户端执行

#### 证据 2：SWR 没有使用 fallbackData 承接服务端数据

在 [api.ts:L11-L24](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/api.ts#L11-L24)，`useDemos()` 的 SWR 配置中：

```typescript
const { data, error, isLoading, mutate: revalidate } = useSWR(
  '/api/demos',
  () => fetcher<DemoMeta[]>('/api/demos'),
  {
    revalidateOnFocus: false,
  }
)
```

没有配置 `fallbackData` 参数。这意味着即使服务端组件可以预取数据，也没有机制将数据传递给 SWR 作为初始值。

#### 证据 3：加载时序导致用户长时间看到不可交互的骨架屏

在 [home-page.tsx:L103-L111](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx#L103-L111)：

```typescript
{isLoading ? (
  <div className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6">
    {Array.from({ length: 4 }).map((_, i) => (
      <div key={i} className="h-[280px] rounded-lg bg-muted animate-pulse" />
    ))}
  </div>
) : ...}
```

加载期间渲染的是纯 `<div>` 骨架屏，不是 `<Link>` 组件，因此**完全不可点击**。

#### 完整执行流程

```
用户打开 / 
  → 服务端渲染 HTML（含骨架屏占位）
  → 浏览器接收 HTML，显示骨架屏动画
  → 浏览器下载 JS Bundle
  → React 水合（hydration）客户端组件
  → SWR 发起 GET /api/demos 请求
  → 服务端处理请求（文件系统 I/O）
  → 返回 JSON 数据
  → SWR 更新状态，isLoading → false
  → React 重新渲染，骨架屏替换为真实 DemoCard（含 <Link>）
  → 用户现在可以点击卡片
```

**从用户看到骨架屏到卡片可点击，中间经历了：JS 下载 + 水合 + API 请求 + 服务端 I/O + 重新渲染**。在开发模式下或网络较慢时，这个过程可能需要数秒甚至更久。

---

### 根因二（加剧因素）：API 路由存在不必要的同步文件系统开销

#### 证据 1：每次请求都调用 ensureDirsExist()

在 [fs-utils.ts:L163-L164](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L163-L164)，`listProjects()` 的第一行：

```typescript
export function listProjects(): DemoMeta[] {
  ensureDirsExist();  // ← 每次请求都执行
  // ...
}
```

`ensureDirsExist()` 在 [fs-utils.ts:L42-L53](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L42-L53) 执行 4 次 `fs.existsSync()` + 可能 4 次 `fs.mkdirSync()`。目录只需在服务启动时确保一次，每次 API 请求都检查是浪费。

#### 证据 2：每个项目执行两次同步文件系统调用

在 [fs-utils.ts:L166-L181](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L166-L181)：

```typescript
for (const entry of entries) {
  // ...
  const stats = fs.statSync(projectPath);           // 同步 I/O 调用 1
  const project = readProjectMeta(entry.name);       // 同步 I/O 调用 2（读+解析 project.json）
  // ...
}
```

每个项目产生 2 次同步文件系统调用。同步调用会阻塞 Node.js 事件循环，当项目数量较多时（如 50+），累积延迟显著。

#### 证据 3：无内存缓存

`listProjects()` 没有任何缓存机制。每次请求都重新遍历目录、读取所有 `project.json` 文件。项目列表是相对静态的数据，完全可以在内存中缓存并设置合理的失效时间。

---

### 根因三（直接原因）：骨架屏与真实卡片的视觉相似性造成用户困惑

骨架屏使用了与真实卡片相同的网格布局（`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6`）和相似的高度（`h-[280px]`），加上 `animate-pulse` 动画，让用户误以为卡片已经加载完成。用户尝试点击这些"看起来像卡片"的骨架屏，自然不会有任何反应。

---

### 根因总结

```
┌──────────────────────────────────────────────────┐
│              核心根因：纯客户端渲染                  │
│  服务端不预取数据 → 客户端等 JS+API → 长时间骨架屏    │
└────────────────────┬─────────────────────────────┘
                     │
          ┌──────────┴──────────┐
          ▼                     ▼
┌─────────────────┐   ┌─────────────────────┐
│  加剧因素：       │   │  直接表现：           │
│  API 同步 I/O 开销 │   │  骨架屏不可点击       │
│  无缓存机制       │   │  用户误以为已就绪      │
└─────────────────┘   └─────────────────────┘
```

## 三、解决方案

### 长期最优根治方案：RSC 数据获取 + Suspense 流式渲染 + 数据层缓存

#### 3.1 方案总览

本方案不是单一改动，而是一套**架构层面的组合优化**，从数据获取模式、渲染策略、缓存机制三个维度同时发力，彻底消除问题并建立长期可持续的性能基础。

```
┌─────────────────────────────────────────────────────────────┐
│                    长期最优根治方案架构                          │
├─────────────────────────────────────────────────────────────┤
│  ① RSC 数据获取    →  服务端直接读取数据，HTML 自带完整内容       │
│  ② Suspense 流式   →  页面渐进式呈现，不等待全部数据              │
│  ③ 内存缓存层      →  减少文件 I/O，加速后续请求                  │
│  ④ 启动时初始化    →  消除每次请求的冗余目录检查                  │
│  ⑤ 乐观 UI 更新    →  创建/删除操作即时反馈，无需等待 API          │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2 核心改动一：将数据获取提升到服务端组件（RSC 模式）

**当前问题**：`page.tsx` 是服务端组件但只做布局组合，真正的数据获取在 `HomePage`（`'use client'`）中通过 SWR 客户端发起。

**目标架构**：

```
page.tsx（服务端组件）← 在此直接调用 listProjects()
  │
  ├── 数据在服务端渲染时已就绪
  │
  └── Suspense fallback={加载中提示}
        └── HomePage（客户端组件）
              └── 接收 initialDemos prop 作为 SWR fallbackData
```

**具体做法**：

1. **`page.tsx`** 改为 async 服务端组件，直接调用 `listProjects()` 获取项目列表
2. 用 `<Suspense>` 包裹 `HomePage`，提供优雅的加载降级（而非空白骨架屏）
3. 将获取到的数据通过 props 传入 `HomePage`

**为什么这是长期最优**：
- 服务端渲染的 HTML **直接包含项目数据**，浏览器接收到的就是完整页面
- 无需等待 JS 下载、水合、API 请求——首屏即是终态
- 这是 Next.js App Router 的**设计初衷**，符合框架最佳实践
- 骨架屏仅作为 Suspense 的 fallback 在极端情况下出现，正常访问不会触发

**影响文件**：
- [page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/app/page.tsx) — 改为 async，添加数据获取 + Suspense
- [home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx) — 新增 `initialDemos` prop
- [api.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/api.ts) — `useDemos()` 支持 `fallbackData` 参数

#### 3.3 核心改动二：建立数据访问缓存层

**当前问题**：`listProjects()` 每次调用都遍历文件系统，无任何缓存。

**目标设计**：

在 `fs-utils.ts` 中新增一个轻量级内存缓存层，专门服务于项目列表查询：

```
请求 → 缓存层（检查是否有效）
         ├── 命中 → 直接返回（零 I/O）
         └── 未命中 → listProjects() → 写入缓存 → 返回
```

**缓存策略**：
- **缓存键**：基于项目目录的 `mtime`（最后修改时间），目录无变化则缓存有效
- **失效机制**：创建/删除/重命名项目时主动清除缓存（通过 `mutate` 调用）
- **TTL 兜底**：设置 30 秒最大有效期，防止极端情况下的脏读

**为什么这是长期最优**：
- 项目列表是**读多写少**的数据，缓存命中率极高
- 基于 `mtime` 的失效策略比纯 TTL 更精准，不会返回过期数据
- 缓存层独立封装，未来可平滑替换为 Redis 等分布式缓存（如果需要）

**影响文件**：
- [fs-utils.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts) — 新增缓存层函数，修改 `listProjects()`

---

#### 3.4 核心改动三：启动时一次性初始化

**当前问题**：`ensureDirsExist()` 在每次 `listProjects()` 调用时执行。

**目标设计**：

将目录确保逻辑移至 [instrumentation.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/instrumentation.ts) 的服务启动钩子中：

```
服务启动 → register() → ensureDirsExist()（仅一次）
                              ↓
                         后续所有请求不再检查目录
```

**为什么这是长期最优**：
- 目录结构在服务运行期间不会变化，启动时确保一次即可
- 消除每次请求的 4 次 `existsSync` + 可能的 4 次 `mkdirSync`
- 改动极小，零风险

**影响文件**：
- [instrumentation.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/instrumentation.ts) — 添加 `ensureDirsExist()` 调用
- [fs-utils.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts) — 从 `listProjects()` 移除 `ensureDirsExist()`

---

#### 3.5 辅助优化：乐观 UI 更新

**当前问题**：创建/删除项目后需要等待 API 响应 + SWR 重新验证，操作反馈有延迟。

**目标设计**：

在 `HomePage` 中，创建和删除操作采用**乐观更新**策略：

- **创建**：调用 API 的同时，立即在本地状态中添加临时项目卡片（带"创建中…"标识），API 成功后替换为真实数据
- **删除**：点击删除后立即从列表中移除卡片，API 失败时恢复并提示错误

**为什么这是长期最优**：
- 用户操作即时响应，感知延迟降为零
- 这是现代 Web 应用的**标准交互模式**
- 与 SWR 的 `mutate` + `optimisticData` 天然契合

**影响文件**：
- [home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx) — 添加乐观更新逻辑

```
Phase 1（基础 — 必须最先做）
  ├── 改动三：启动时初始化（独立，无依赖）
  └── 改动二：缓存层（独立，无依赖）

        ↓

  └── 改动四：乐观 UI 更新
```

**Phase 1 和 Phase 2 完成后，问题即被根治。** Phase 3 是锦上添花的体验优化。

---

#### 3.7 方案对比：为什么这是长期最优

| 维度 | 当前状态 | 根治后 |
|------|---------|--------|
| 首屏数据 | 客户端 API 请求获取 | 服务端渲染 HTML 自带 |
| 用户等待 | JS 下载 + 水合 + API + 渲染 | 零等待（首屏即终态） |
| 骨架屏 | 每次必现，不可点击 | 仅极端情况出现（Suspense fallback） |
| 文件 I/O | 每次请求遍历全部目录 | 缓存命中时零 I/O |
| 操作反馈 | 等待 API 响应 | 即时乐观更新 |
| 架构合规 | 不符合 RSC 最佳实践 | 完全符合 Next.js App Router 设计 |
| 可扩展性 | 项目增多时线性变慢 | 缓存层保证恒定性能 |

---

#### 3.8 未来演进方向（可选，非必须）

如果未来项目规模持续增长，可考虑以下进一步优化：

1. **项目元数据入 SQLite**：当前项目已有 SQLite（用于用户认证），可将项目元数据（id、name、createdAt、updatedAt、thumbnail）迁移到 SQLite 表，利用数据库索引加速查询。文件内容（代码、schema）保留在文件系统。这需要新增 `projects` 表和数据迁移脚本。

2. **ISR（增量静态再生成）**：如果首页访问量很大，可对首页启用 ISR，定时在服务端重新生成静态 HTML，进一步减少服务端渲染开销。

但就当前阶段而言，**3.2-3.5 的方案组合已足够根治问题**，无需过度设计。
