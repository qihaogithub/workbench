# 首页加载缓慢与卡片点击延迟问题分析报告

> 分析日期：2026-05-02

---

## 一、问题背景

### 问题描述

用户访问首页（`/`）后，遇到两个关联的性能问题：

| 问题 | 预期行为 | 实际行为 |
|------|---------|---------|
| 首页加载缓慢 | 打开首页后，项目卡片应在 1-2 秒内显示 | 信息需要加载很久才出现，用户长时间看到骨架屏占位动画 |
| 卡片点击无效 | 点击项目卡片应立即跳转到编辑页 | 一开始点击卡片无任何反应，很久之后点击才生效 |

### 发生场景

- 每次打开首页时稳定复现
- 两个问题存在因果关系：卡片点击无效是首页加载缓慢的直接后果

### 涉及组件

- 首页页面入口、首页内容组件、项目卡片组件、数据获取 Hook、后端 API 路由、文件系统工具函数

---

## 二、根因分析

### 根因一（核心根因）：首页采用纯客户端渲染（CSR），无服务端数据预取

**这是导致两个问题的根本原因。**

#### 证据 1：页面组件架构导致数据只能在客户端获取

首页的组件层级如下：

```
page.tsx（服务端组件）
  └── MainLayout（服务端组件）
        └── HomePage（'use client' 客户端组件）
              └── useDemos() → SWR → fetch('/api/demos')
```

- [page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/app/page.tsx) 是服务端组件，但它只做布局组合，不获取数据
- [home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx) 标记为 `'use client'`，数据获取逻辑（`useDemos()`）完全在客户端执行

#### 证据 2：SWR 没有使用 fallbackData 承接服务端数据

在 [api.ts:L11-L24](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/api.ts#L11-L24)，`useDemos()` 的 SWR 配置中：

```typescript
const { data, error, isLoading, mutate: revalidate } = useSWR(
  '/api/demos',
  () => fetcher<DemoMeta[]>('/api/demos'),
  {
    revalidateOnFocus: false,
  }
)
```

没有配置 `fallbackData` 参数。这意味着即使服务端组件可以预取数据，也没有机制将数据传递给 SWR 作为初始值。

#### 证据 3：加载时序导致用户长时间看到不可交互的骨架屏

在 [home-page.tsx:L103-L111](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx#L103-L111)：

```typescript
{isLoading ? (
  <div className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6">
    {Array.from({ length: 4 }).map((_, i) => (
      <div key={i} className="h-[280px] rounded-lg bg-muted animate-pulse" />
    ))}
  </div>
) : ...}
```

加载期间渲染的是纯 `<div>` 骨架屏，不是 `<Link>` 组件，因此**完全不可点击**。

#### 完整执行流程

```
用户打开 / 
  → 服务端渲染 HTML（含骨架屏占位）
  → 浏览器接收 HTML，显示骨架屏动画
  → 浏览器下载 JS Bundle
  → React 水合（hydration）客户端组件
  → SWR 发起 GET /api/demos 请求
  → 服务端处理请求（文件系统 I/O）
  → 返回 JSON 数据
  → SWR 更新状态，isLoading → false
  → React 重新渲染，骨架屏替换为真实 DemoCard（含 <Link>）
  → 用户现在可以点击卡片
```

**从用户看到骨架屏到卡片可点击，中间经历了：JS 下载 + 水合 + API 请求 + 服务端 I/O + 重新渲染**。在开发模式下或网络较慢时，这个过程可能需要数秒甚至更久。

---

### 根因二（加剧因素）：API 路由存在不必要的同步文件系统开销

#### 证据 1：每次请求都调用 ensureDirsExist()

在 [fs-utils.ts:L163-L164](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L163-L164)，`listProjects()` 的第一行：

```typescript
export function listProjects(): DemoMeta[] {
  ensureDirsExist();  // ← 每次请求都执行
  // ...
}
```

`ensureDirsExist()` 在 [fs-utils.ts:L42-L53](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L42-L53) 执行 4 次 `fs.existsSync()` + 可能 4 次 `fs.mkdirSync()`。目录只需在服务启动时确保一次，每次 API 请求都检查是浪费。

#### 证据 2：每个项目执行两次同步文件系统调用

在 [fs-utils.ts:L166-L181](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L166-L181)：

```typescript
for (const entry of entries) {
  // ...
  const stats = fs.statSync(projectPath);           // 同步 I/O 调用 1
  const project = readProjectMeta(entry.name);       // 同步 I/O 调用 2（读+解析 project.json）
  // ...
}
```

每个项目产生 2 次同步文件系统调用。同步调用会阻塞 Node.js 事件循环，当项目数量较多时（如 50+），累积延迟显著。

#### 证据 3：无内存缓存

`listProjects()` 没有任何缓存机制。每次请求都重新遍历目录、读取所有 `project.json` 文件。项目列表是相对静态的数据，完全可以在内存中缓存并设置合理的失效时间。

---

### 根因三（直接原因）：骨架屏与真实卡片的视觉相似性造成用户困惑

骨架屏使用了与真实卡片相同的网格布局（`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6`）和相似的高度（`h-[280px]`），加上 `animate-pulse` 动画，让用户误以为卡片已经加载完成。用户尝试点击这些"看起来像卡片"的骨架屏，自然不会有任何反应。

---

### 根因总结

```
┌──────────────────────────────────────────────────┐
│              核心根因：纯客户端渲染                  │
│  服务端不预取数据 → 客户端等 JS+API → 长时间骨架屏    │
└────────────────────┬─────────────────────────────┘
                     │
          ┌──────────┴──────────┐
          ▼                     ▼
┌─────────────────┐   ┌─────────────────────┐
│  加剧因素：       │   │  直接表现：           │
│  API 同步 I/O 开销 │   │  骨架屏不可点击       │
│  无缓存机制       │   │  用户误以为已就绪      │
└─────────────────┘   └─────────────────────┘
```

## 三、解决方案

### 方案一（推荐）：服务端预取数据 + SWR fallbackData

**原理**：在服务端组件 `page.tsx` 中直接调用 `listProjects()` 获取数据，通过 props 传递给 `HomePage`，`HomePage` 将其作为 SWR 的 `fallbackData`。

**优点**：
- 从根本上解决问题，服务端渲染的 HTML 直接包含真实数据
- 用户看到的首屏即是完整内容，无骨架屏闪烁
- 对用户体验改善最大
- 实现简单，改动范围小

**缺点**：
- 服务端渲染时间略微增加（但远小于客户端等待时间）
- 需要调整 `HomePage` 的 props 接口

**影响范围**：
- `src/app/page.tsx` — 添加数据获取
- `src/components/demo/home-page.tsx` — 接收 fallbackData prop
- `src/lib/api.ts` — `useDemos` 支持 fallbackData 参数

---

### 方案二：API 层添加内存缓存

**原理**：在 `listProjects()` 中添加内存缓存（如使用 `Map` + 时间戳），设置 5-10 秒的缓存有效期，避免每次请求都遍历文件系统。

**优点**：
- 减少服务端 I/O 开销
- 实现简单，仅修改 `fs-utils.ts`
- 对高并发场景有明显改善

**缺点**：
- 不解决客户端渲染等待的根本问题
- 缓存失效时间内可能返回过期数据
- 需要处理缓存失效逻辑

**影响范围**：
- `src/lib/fs-utils.ts` — `listProjects()` 函数

---

### 方案三：将 ensureDirsExist() 移至服务启动时执行

**原理**：在 `instrumentation.ts` 的服务启动钩子中调用 `ensureDirsExist()`，从 `listProjects()` 中移除该调用。

**优点**：
- 消除每次 API 请求的冗余检查
- 零风险改动

**缺点**：
- 改善幅度有限，不能单独解决问题

**影响范围**：
- `src/instrumentation.ts` — 添加启动时目录确保
- `src/lib/fs-utils.ts` — 从 `listProjects()` 移除 `ensureDirsExist()`

---

### 方案四：骨架屏添加视觉提示

**原理**：在骨架屏上添加文字提示（如"加载中..."），让用户明确知道内容尚未就绪。

**优点**：
- 改动最小
- 改善用户感知

**缺点**：
- 治标不治本，不解决加载慢的问题
- 用户仍需等待

---

### 推荐实施策略

**短期（立即）**：方案一 + 方案三，从根本上解决客户端渲染等待问题，同时消除冗余 I/O。

**中期（优化）**：方案二，进一步减少服务端负载。

**可选（体验）**：方案四，作为辅助改善。
