# 首页加载缓慢与卡片点击延迟问题分析报告

> 分析日期：2026-05-02
> 修订日期：2026-05-03（根据代码核查与简洁原则精简方案）

---

## 一、问题背景

### 问题描述

用户访问首页（`/`）后，遇到两个关联的性能问题：

| 问题 | 预期行为 | 实际行为 |
|------|---------|---------|
| 首页加载缓慢 | 打开首页后，项目卡片应在 1-2 秒内显示 | 信息需要加载很久才出现，用户长时间看到骨架屏占位动画 |
| 卡片点击无效 | 点击项目卡片应立即跳转到编辑页 | 一开始点击卡片无任何反应，很久之后点击才生效 |

### 发生场景

- 每次打开首页时稳定复现
- 两个问题存在因果关系：卡片点击无效是首页加载缓慢的直接后果

### 涉及组件

- 首页页面入口、首页内容组件、项目卡片组件、数据获取 Hook、后端 API 路由、文件系统工具函数

## 二、根因分析

### 根因（核心根因）：首页采用纯客户端渲染（CSR），无服务端数据预取

**这是导致两个问题的根本原因。**

#### 证据 1：页面组件架构导致数据只能在客户端获取

首页的组件层级如下：

```
page.tsx（服务端组件）
  └── MainLayout（服务端组件）
        └── HomePage（'use client' 客户端组件）
              └── useDemos() → SWR → fetch('/api/demos')
```

- [page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/app/page.tsx) 是服务端组件，但它只做布局组合，不获取数据
- [home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx) 标记为 `'use client'`，数据获取逻辑（`useDemos()`）完全在客户端执行

#### 证据 2：SWR 没有使用 fallbackData 承接服务端数据

在 [api.ts:11-17](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/api.ts#L11)，`useDemos()` 的 SWR 配置中：

```typescript
const { data, error, isLoading, mutate: revalidate } = useSWR(
  '/api/demos',
  () => fetcher<DemoMeta[]>('/api/demos'),
  {
    revalidateOnFocus: false,
  }
)
```

没有配置 `fallbackData` 参数。这意味着即使服务端组件可以预取数据，也没有机制将数据传递给 SWR 作为初始值。

#### 证据 3：加载时序导致用户长时间看到不可交互的骨架屏

在 [home-page.tsx:105-113](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx#L105)：

```typescript
{isLoading ? (
  <div className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6">
    {Array.from({ length: 4 }).map((_, i) => (
      <div key={i} className="h-[280px] rounded-lg bg-muted animate-pulse" />
    ))}
  </div>
) : ...}
```

加载期间渲染的是纯 `<div>` 骨架屏，不是 `<Link>` 组件，因此**完全不可点击**。

#### 证据 4：骨架屏与真实卡片的视觉相似性造成用户困惑

骨架屏使用了与真实卡片相同的网格布局（`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6`）和相似的高度（`h-[280px]`），加上 `animate-pulse` 动画，让用户误以为卡片已经加载完成。用户尝试点击这些"看起来像卡片"的骨架屏，自然不会有任何反应。

#### 完整执行流程

```
用户打开 /
  → 服务端渲染 HTML（含骨架屏占位）
  → 浏览器接收 HTML，显示骨架屏动画
  → 浏览器下载 JS Bundle
  → React 水合（hydration）客户端组件
  → SWR 发起 GET /api/demos 请求
  → 服务端处理请求（文件系统 I/O）
  → 返回 JSON 数据
  → SWR 更新状态，isLoading → false
  → React 重新渲染，骨架屏替换为真实 DemoCard（含 <Link>）
  → 用户现在可以点击卡片
```

**从用户看到骨架屏到卡片可点击，中间经历了：JS 下载 + 水合 + API 请求 + 服务端 I/O + 重新渲染**。在开发模式下或网络较慢时，这个过程可能需要数秒甚至更久。

### 次要因素（理论上存在，当前未触发）

#### API 路由的同步 I/O 调用

在 [fs-utils.ts:170-194](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/fs-utils.ts#L170) 的 `listProjects()`：

- 第一行调用 `ensureDirsExist()`，重复 4 次 `fs.existsSync()`
- 每个项目执行 `fs.statSync()` + `readProjectMeta()`（含 `readFileSync` + `JSON.parse`）

**实测当前 `data/projects/` 目录为空（0 个项目）**，循环根本不执行，单次调用耗时 < 1ms。

> 当项目数量增长到数百个量级时，这部分开销才会显著。当前规模下不构成瓶颈，**本报告不在此处优化**。

#### 客户端 Bundle 体积与组件树深度

报告核查时观察到的潜在加剧因素（不在本次根治范围内，需独立专项处理）：

- `.next` 缓存目录已达 1.5GB，依赖含 `streamdown`、`@streamdown/mermaid`、`@rjsf/*`、`ai`、`@ai-sdk/react`、十多个 `@radix-ui/*` 等重量级包
- `layout.tsx` 注入 `ThemeProvider` + `TooltipProvider` + `ToastProviderWrapper` 三层 Provider，全部 `'use client'`，整页接近 100% 客户端渲染
- 缺少 `app/loading.tsx`，无路由级 Suspense fallback
- 存在 `next.config.js` 与 `next.config.ts` 双配置文件，行为可能不一致

这些是 dev 模式下加载慢的潜在主因，但**与本报告"根因→根治"主线独立**，不应混入主线方案。

## 三、解决方案（精简版）

> **设计原则**（遵循 [AGENTS.md](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/AGENTS.md)）：
> - 用最少的代码解决问题，不为不可能出现的场景编写异常处理
> - 这是一个局域网内使用的公司内部工具，避免过度工程化设计
> - 单次使用的代码不做抽象封装

### 3.1 方案总览

**核心方案：把数据获取从客户端 SWR 提升到服务端组件预取，通过 props + SWR `fallbackData` 把数据带到客户端**。

```
┌─────────────────────────────────────────────────────────────┐
│   page.tsx（async 服务端组件）                                 │
│     └─ const initialDemos = listProjects()                  │
│         └─ <HomePage initialDemos={initialDemos} />         │
│              ├─ useDemos({ fallbackData: initialDemos })    │
│              └─ isLoading 永远为 false（首屏）                │
│                  └─ 直接渲染 DemoCard（<Link> 立即可点击）     │
└─────────────────────────────────────────────────────────────┘
```

完成后：首屏 HTML 已包含完整的 `<Link>` 卡片，骨架屏不再出现，"卡片不可点击"问题彻底消失。

### 3.2 改动清单

#### 改动 1：扩展 `useDemos` 接受 `fallbackData`

**文件**：[packages/web/src/lib/api.ts](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/lib/api.ts)

```typescript
export function useDemos(options?: { fallbackData?: DemoMeta[] }) {
  const { data, error, isLoading, mutate: revalidate } = useSWR(
    '/api/demos',
    () => fetcher<DemoMeta[]>('/api/demos'),
    {
      revalidateOnFocus: false,
      fallbackData: options?.fallbackData
        ? { success: true as const, data: options.fallbackData }
        : undefined,
    }
  )
  // 其余逻辑不变
}
```

要点：`fallbackData` 必须包装为 `ApiResponse<DemoMeta[]>` 形态，与 `fetcher` 返回类型一致，否则后续 `data?.success` 判断会出错。

#### 改动 2：`HomePage` 接收 `initialDemos` prop 并透传给 `useDemos`

**文件**：[packages/web/src/components/demo/home-page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/components/demo/home-page.tsx)

```typescript
export function HomePage({ initialDemos }: { initialDemos: DemoMeta[] }) {
  const router = useRouter()
  const { demos, error, revalidate } = useDemos({
    fallbackData: initialDemos,
  })
  // 其余逻辑不变；移除 isLoading 解构
}
```

**同时移除骨架屏分支**——既然 SSR 阶段一定有 `initialDemos`，直接以 `filteredDemos.length === 0` 判断是否显示"暂无 Demo"即可，无需 `isLoading` 中间态。

> ⚠️ 实测踩坑：SWR v2 配置 `fallbackData` 后，**`data` 在 SSR 阶段确实立即可用**，但 `isLoading` 在 SSR 输出 HTML 中仍可能为 `true`。
> 如果保留 `{isLoading ? <Skeleton /> : ...}` 分支，SSR 渲染出的 HTML 仍是骨架屏，本次改造效果会被这一行抹掉。
> **正确做法**：直接删除 `isLoading` 分支，让 `demos.length` 决定渲染。

#### 改动 3：`page.tsx` 改为 async 服务端组件，预取数据

**文件**：[packages/web/src/app/page.tsx](file:///Users/qh2/Documents/PGM/1·Work/opencode-workbench/packages/web/src/app/page.tsx)

```typescript
import { MainLayout } from '@/components/layout/main-layout'
import { HomePage } from '@/components/demo/home-page'
import { listProjects } from '@/lib/fs-utils'

export const dynamic = 'force-dynamic'

export default async function Page() {
  const initialDemos = listProjects()
  return (
    <MainLayout>
      <HomePage initialDemos={initialDemos} />
    </MainLayout>
  )
}
```

要点：
- `export const dynamic = 'force-dynamic'`：禁用静态化，确保每次请求都拿到最新项目列表
- 直接调用 `listProjects()`：服务端组件中可以直接使用 Node 文件系统 API
- 服务端渲染产出的 HTML 已包含完整 `<Link>` 卡片

#### 改动 4：新建 `app/loading.tsx`（兜底 Suspense fallback）

**文件**：`packages/web/src/app/loading.tsx`（新建）

```typescript
export default function Loading() {
  return (
    <div className="container py-6 px-4 text-muted-foreground">
      加载中…
    </div>
  )
}
```

正常情况下 `listProjects()` < 5ms，用户基本看不到这个 fallback；它仅在极端情况（首次冷启动 + 大量项目）作为兜底。

### 3.3 不做的事（明确排除）

为遵循「避免过度工程化」原则，以下改动**明确不做**：

| 排除项 | 理由 |
|--------|------|
| `mtime` 缓存层 + TTL 兜底 | 当前 0 项目，listProjects() < 1ms；mtime 不反映 `project.json` 内容修改，缓存策略本身有缺陷 |
| 启动时 `ensureDirsExist`（移出 `listProjects`） | 4 次 `existsSync` 是 nanosecond 级，改动收益微小 |
| 创建/删除的乐观 UI 更新 | 内部工具操作不频繁，现有 `mutate('/api/demos')` 已够用；改 RSC 后，可改为 `router.refresh()` 一行替代 |
| 项目元数据 SQLite 化 | 与当前规模差三个数量级，纯过度设计 |
| Bundle 体积优化 | 与本根治主线独立，需另开专项任务 |

### 3.4 实施顺序

```
1. 改 api.ts（扩展 useDemos 签名，向后兼容——options 可选）
2. 改 home-page.tsx（增加 props，透传 fallbackData）
3. 改 page.tsx（改 async + 调 listProjects + 传 props）
4. 新建 loading.tsx
5. 运行 pnpm typecheck 验证
6. 启动 dev 服务，浏览器实测首页加载与卡片点击
```

依赖关系：1 → 2 → 3，必须按顺序；4 独立。
