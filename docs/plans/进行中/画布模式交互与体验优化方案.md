## 一、问题概述

当前画布模式在交互体验上存在多个明显短板，核心问题集中在三个方面：

1. **页面缩放体验差**：缩放手柄仅 8 个离散点、尺寸过小（12×12px），与 Figma 等设计工具的"边框四周任意拖拽缩放"体验差距大
2. **页面外框与内容对不齐**：iframe 内容区域与页面容器尺寸存在偏差，导致视觉上边框和内容不匹配
3. **无用 UI 干扰**：页面右上角"修改配置"按钮功能薄弱，点击后仅切换编辑页面，无实际配置编辑能力

除上述用户反馈的问题外，代码审查中还发现以下值得优化的点：

4. 缺少对齐辅助（吸附线/网格）
5. 工具栏功能过于简陋
6. 键盘快捷键缺失
7. 右键菜单缺失
8. 布局持久化 API 未实现

### 2.2 当前缩放手柄实现

```
CanvasPageItem.tsx 第 28-40 行：

HANDLE_DEFS 定义了 8 个方向的手柄：
nw  n  ne
w       e
sw  s  se

每个手柄：12×12px 白色方块 + 蓝色边框
显示条件：hover + 可编辑 + 非编辑中 + 非拖拽中 + 非缩放中
```

**问题**：
- 手柄仅 12×12px，在低缩放比例下极难点击
- 手柄是离散的 8 个点，不是连续的边框
- 手柄 hover 时才出现，用户难以发现缩放能力

### 2.3 当前页面外框与内容对齐

```
CanvasPageItem.tsx 第 248-286 行：

页面内容结构：
<div absolute, left/top/width/height=layout>     ← 外框容器
  <div w-full h-full rounded-lg overflow-hidden>  ← 内容包裹层
    <PreviewPanel fillContainer />                 ← iframe（100% 填充）
  </div>
  <img screenshot overlay />                       ← 截图叠加层
</div>
```

**对不齐的根因**：
1. `PreviewPanel` 在 `fillContainer` 模式下，iframe 设置为 `width: 100%; height: 100%`，但 iframe 内部内容有自身的 `designWidth/designHeight`（默认 375×812），当页面容器尺寸与设计尺寸比例不一致时，内容会被拉伸或压缩
2. 截图叠加层使用 `object-contain`，会保持原始宽高比，而 iframe 内容直接 100% 填充不保持比例，两者视觉不一致
3. `rounded-lg` 和 `overflow-hidden` 应用在内容层而非外框层，导致边框圆角裁切与外框不完全对齐

### 2.4 当前"修改配置"按钮

```
CanvasPageItem.tsx 第 329-337 行：

显示条件：hover + 非编辑 + 有 onConfigEdit 回调
点击行为：onConfigEdit(pageId) → 在 edit/page.tsx 中：
  1. setCanvasEditingPageId(pageId)
  2. setActiveDemoId(pageId)
  3. fetch 文件内容 → 更新 code/schema/editorContent/configDataMap
```

**问题**：按钮名为"修改配置"但实际行为是"切换到该页面编辑"，与用户预期不符。用户已有双击进入编辑的交互方式，此按钮冗余且误导。

### 2.5 当前 iframe 加载机制

**加载流程**（`PreviewPanel.tsx`）：

```
组件挂载 → generateIframeHtml() → 创建 Blob URL → iframe.src = blobUrl
       → 浏览器加载 HTML → 下载 esm.sh 依赖（React/ReactDOM）→ 执行 JS
       → iframe 发送 READY → 父窗口发送 UPDATE_CODE(postMessage) → React 渲染组件
```

**当前问题**：

1. **每挂载一次就新建 Blob URL**：`generateIframeHtml()` 生成的 HTML 骨架对所有页面完全相同（仅包含 importmap、ErrorBoundary、消息处理，不含用户代码），但每次调用都创建新的 Blob URL

2. **非编辑页的 iframe 永不卸载**：
   ```
   CanvasPageItem.tsx 第 245-284 行：
   
   showIframe = isEditing || !screenshotUrl           // 编辑中 或 无截图
   showScreenshotOverlay = !isEditing && !!screenshotUrl  // 非编辑且有截图
   
   // 当有截图 overlay 时，iframe 设为 visibility:hidden，仍留在 DOM 中
   ```
   即使用户看到的是截图，底层的 iframe 仍然挂在 DOM 中，占用内存和 CPU

3. **不可见页面卸载 iframe**：`!visible` 时返回空 div（第 229-243 行），滚回时重新挂载 → 重新走完整加载流程

4. **截图仅进入画布时才生成**：`useScreenshotGeneration` 的 `enabled: previewMode === "canvas"`，平时编辑页面不会预先生成截图，导致进入画布模式后需要等待批量截图完成

5. **编译缓存容量小**：`compile-cache.ts` 仅 20 条、5 分钟 TTL，且 `PreviewPanel` 不直接使用该缓存

## 三、优化方案

### 方案 1：边框式缩放交互（P0）

**目标**：将离散的 8 个缩放小方块改为 Figma 风格的边框式缩放，四周任意位置均可拖拽缩放。

**设计**：

```
┌─────────────────────────┐
│                         │  ← 上边：任意位置拖拽，上下缩放
│                         │
│ ← 左边拖拽，左右缩放    │  → 右边拖拽，左右缩放
│                         │
│                         │
└─────────────────────────┘  ← 下边：任意位置拖拽，上下缩放
  四角：拖拽时同时缩放宽高
```

**实现要点**：

1. **边框热区**：用 4 个透明的边条 div 覆盖页面四周，宽度 6-8px，作为缩放热区
   - 上边条：`absolute top-0 left-0 right-0 h-2`，cursor: `ns-resize`
   - 下边条：`absolute bottom-0 left-0 right-0 h-2`，cursor: `ns-resize`
   - 左边条：`absolute top-0 left-0 bottom-0 w-2`，cursor: `ew-resize`
   - 右边条：`absolute top-0 right-0 bottom-0 w-2`，cursor: `ew-resize`
   - 四角区域由相邻边条重叠自然形成，通过坐标判断确定缩放方向

2. **角点手柄**：保留四角的小方块作为视觉指示器（非交互必需），尺寸增大到 8×8px，颜色改为更醒目的样式
   - 角点手柄仅作为视觉提示，实际缩放交互由边框热区驱动
   - 角点手柄 hover 时放大 + 高亮

3. **缩放方向判定**：
   - 鼠标按下时，根据按下位置确定缩放方向
   - 距离四角 < 16px → 对角缩放（nw/ne/sw/se）
   - 在边条范围内 → 单方向缩放（n/s/e/w）

4. **最小尺寸**：保持 `MIN_SIZE = 100`，但增加最大尺寸限制（如 2000px），防止误操作产生超大页面

5. **等比缩放**：按住 Shift 时，角点缩放保持宽高比

**修改文件**：`CanvasPageItem.tsx`

**与现有方案的差异**：现有优化方案（方案 2）仅优化了 8 个手柄的 pointer capture，本方案从根本上改变交互模式为边框式缩放。

### 方案 2：修复页面外框与内容对齐（P0）

**目标**：确保页面外框边界与 iframe 内容/截图内容视觉一致。

**根因分析与修复**：

1. **iframe 内容保持宽高比**：
   - 当前 `fillContainer` 模式下 iframe 为 `width:100%; height:100%`，内容被拉伸
   - 修改为：iframe 内部保持 `designWidth × designHeight` 的原始比例，通过 CSS `transform: scale()` 适配容器
   - 当容器宽高比与设计宽高比不一致时，内容居中显示，空白区域填充背景色

2. **截图叠加层与 iframe 一致**：
   - 截图当前使用 `object-contain`，改为与 iframe 相同的缩放逻辑
   - 或统一使用 `object-fill` + 保持容器宽高比与设计尺寸一致

3. **圆角统一**：
   - 将 `rounded-lg overflow-hidden` 从内容层移到外框容器层
   - 确保边框、内容、截图三层共享同一圆角裁切

4. **边框指示器对齐**：
   - 拖拽/缩放时的蓝色边框指示器（`border-2 border-blue-500`）应紧贴外框容器边缘
   - 当前 `absolute inset-0` 可能因内层 padding 偏移

**修改文件**：`CanvasPageItem.tsx`、`PreviewPanel.tsx`

### 方案 3：移除"修改配置"按钮（P0）

**目标**：移除页面右上角的"修改配置"按钮，减少视觉干扰。

**实现**：
- 删除 `CanvasPageItem.tsx` 第 329-337 行的按钮渲染代码
- 保留 `onConfigEdit` prop 接口（viewer 模式可能需要），但不在 UI 上渲染按钮
- 如未来需要配置编辑功能，应设计独立的配置面板而非页面内按钮

**修改文件**：`CanvasPageItem.tsx`

### 方案 4：对齐辅助线（P1）

**目标**：拖拽/缩放页面时显示对齐辅助线，帮助用户对齐多个页面。

**设计**：

```
        │ 中心对齐线
        │
  ┌─────┼─────┐
  │     │     │
  │  页面 A   │
  │     │     │
  └─────┼─────┘
        │
  ┌─────┼─────┐
  │     │     │
  │  页面 B   │
  │     │     │
  └─────┼─────┘
```

**实现要点**：

1. **对齐规则**：
   - 页面中心 X/Y 对齐
   - 页面边缘对齐（左对左、右对右、上对上、下对下）
   - 页面等间距对齐

2. **吸附阈值**：8px，当页面边缘/中心距离 < 8px 时自动吸附

3. **辅助线渲染**：在 `PreviewCanvas` 层级渲染，避免被页面遮挡
   - 红色虚线，1px 宽
   - 仅在拖拽/缩放过程中显示

4. **数据来源**：遍历其他页面的 `CanvasPageLayout` 计算对齐点

**修改文件**：`PreviewCanvas.tsx`（新增对齐计算逻辑）、`CanvasPageItem.tsx`（触发吸附）

### 方案 5：工具栏功能增强（P1）

**目标**：增强底部工具栏，提供更多视口控制能力。

**当前工具栏**：仅 缩小 | 百分比 | 放大 | 重置

**增强后**：

```
[适应屏幕] [缩小] [50% ▾] [放大] [1:1] [重置]
```

新增功能：

1. **适应屏幕**（Fit to Screen）：自动调整 zoom 和 pan，使所有页面完整显示在视口中
   - 计算所有页面的包围盒（bounding box）
   - `zoom = min(containerW / bboxW, containerH / bboxH) * 0.9`（留 10% 边距）
   - `pan` 使包围盒居中

2. **1:1 按钮**：将 zoom 设置为 1.0，保持当前中心点不变

3. **缩放下拉菜单**：点击百分比数字弹出预设缩放比例
   - 预设值：25%、50%、75%、100%、150%、200%
   - 支持手动输入自定义百分比

**修改文件**：`CanvasToolbar.tsx`、`PreviewCanvas.tsx`（新增 fitToScreen 逻辑）

### 方案 8：布局持久化实现（P1）

**目标**：实现画布布局的保存和恢复，用户重新打开项目时保持上次的布局状态。

**当前状态**：
- `canvas-utils.ts` 已实现 `saveCanvasLayout` 函数（debounce 500ms）
- 但 API 路由 `POST /api/sessions/:sessionId/canvas-layout` **不存在**
- 每次进入画布模式都从 `computeInitialLayout` 重新计算，用户手动调整的布局丢失

**实现**：

1. **API 路由**：在 `packages/author-site/src/app/api/sessions/[sessionId]/canvas-layout/route.ts` 新增
   - `POST`：保存布局到 `data/sessions/{sessionId}/canvas-layout.json`
   - `GET`：读取布局

2. **加载时机**：进入画布模式时，先尝试从 API 加载已保存的布局，无则使用 `computeInitialLayout`

3. **保存时机**：
   - 页面拖拽/缩放结束后（已有 debounce）
   - 视口变化结束后（新增）
   - 页面删除/新增后

**修改文件**：新增 API 路由，修改 `PreviewCanvas.tsx`（加载逻辑）

### 方案 9：页面拖拽/缩放与视口平移的冲突优化（P1）

**目标**：解决页面拖拽/缩放操作与画布平移操作的冲突。

**当前问题**：
- 页面拖拽使用 `pointerdown` 在 `CanvasPageItem` 上，`stopPropagation` 阻止冒泡
- 画布平移使用 `pointerdown` 在 `CanvasViewport` 上
- 但当用户在页面上按下鼠标后移出页面区域，`pointerCapture` 保证页面拖拽继续，此时画布无法平移
- 缩放手柄同理

**优化方案**：

1. **Space 键切换模式**：按住 Space 时，页面上的 pointer 事件不拦截，统一由画布处理平移
2. **中键平移**：中键（button=1）始终触发画布平移，即使在页面上
3. **Esc 取消**：拖拽/缩放过程中按 Esc 取消操作，恢复原始位置

**修改文件**：`CanvasPageItem.tsx`、`CanvasViewport.tsx`

### 方案 10：iframe 加载性能优化 — 截图优先 + 单 iframe 槽位（P0）

**目标**：画布模式下支持几十个页面同时流畅显示，消除 iframe 重复加载的等待时间。

**设计**：

```
画布模式 = 编辑页(实时 iframe) + 非编辑页(截图; 无截图时临时 iframe → 截图就绪后无缝替换)
         + 单页面配置面板(全局+页面配置合并显示)
```

方案拆分为四个子项：

---

#### 10a. 截图常驻生成

将截图从"按需生成（进入画布时）"改为"持续生成（代码变更即生成）"。

**改动**：

- `useScreenshotGeneration.ts`：删除 `enabled: previewMode === "canvas"` 门控，改为始终启用
- 代码变更时 debounce 3s 自动触发 `regeneratePage()`
- 页面配置变更时：编辑页实时 iframe 立即更新，后台 debounce 3s 生成新截图
- 项目配置变更时：编辑页 iframe 立即更新，后台触发所有页面的截图再生
- 截图长期存储于 `data/screenshots/`，截图服务后端已有内容哈希缓存，同内容不重复渲染

**修改文件**：`useScreenshotGeneration.ts`、`edit/page.tsx`

---

#### 10b. CanvasPageItem 渲染逻辑重构

将 iframe 挂载范围从"所有有内容的页面"缩减为"仅编辑页 + 无截图页面"。

**四种渲染路径**：

| 状态 | 渲染内容 |
|------|---------|
| `isEditing` | 始终渲染 PreviewPanel（实时 iframe），无论是否有截图 |
| `!isEditing && 有截图` | **仅**渲染 `<img>` 截图，不挂载 PreviewPanel |
| `!isEditing && 无截图` | 渲染 PreviewPanel（无截图可用）→ 截图到达后替换为 `<img>` |
| `!visible && !isEditing` | 空 div（占位） |

**无缝替换机制**：

当前代码中截图 overlay 通过 `visibility:hidden` + `position:absolute` 叠加在 iframe 上方实现无感切换（第 250-283 行），但 iframe 不会被卸载。优化为：

```
截图到达 → 同时渲染 iframe(隐藏) + img(加载中)
        → img.onLoad → 卸载 iframe，仅保留 img
```

过渡期间两者同时存在于 DOM（~300ms），随后 iframe 被销毁释放内存。用户感知内容无变化（截图为 iframe 的快照）。

**修改文件**：`CanvasPageItem.tsx`

---

#### 10c. 配置面板合并（canvas 模式专用）

当前 canvas 模式下配置面板显示两个分隔区域：

```

改为 canvas 模式中移除 `ConfigScopeWrapper` 标题和分隔线，两个 `ConfigForm` 堆叠呈现为一个连续表单：

┌─ 当前页面配置 ──────┐
│ theme: [dark  v]    │  ← 项目字段（修改后传播到所有页面）
│ brandColor: [#ff0]  │  ← 项目字段
│ buttonText: [提交]  │  ← 页面字段（仅影响当前页）
│ showHeader: [✓]     │  ← 页面字段
└─────────────────────┘
```

底层行为不变：
- 项目字段 onChange → 通过 `setConfigDataMap` 传播到所有页面（现有逻辑第 1956-1968 行）
- 页面字段 onChange → 通过 `handleConfigChange` 仅更新当前页面
- 切换编辑页时，项目字段值自动保留（已传播到所有页面）

## 六、与性能优化方案的关系

本方案聚焦**交互与体验**优化，与已有的 [画布模式性能优化方案.md](./画布模式性能优化方案.md)（聚焦**性能**优化）互补：

| 维度 | 性能优化方案 | 本方案 |
|------|------------|--------|
| 页面拖拽 | 修复 pointer capture 缺陷 | 边框式缩放、冲突优化 |
| 页面缩放 | 新增 8 方向手柄 | 改为边框式缩放（覆盖） |
| 视口性能 | rAF 节流 + will-change | 键盘快捷键、工具栏增强 |
| iframe 加载性能 | 避免卸载重建 | 方案 10：截图优先 + 单 iframe 槽位 + 编译缓存（全面覆盖） |
| 视口裁剪 | 只渲染可见页面 | — |
| 截图策略 | 截图优先 | 方案 10a：截图常驻生成 + 方案 10b：截图替换 iframe |

**建议**：先实施性能优化方案中的方案 1（修复拖拽 pointer capture）和方案 4（rAF 节流），再实施本方案的阶段一。性能优化方案中的方案 2（8 方向手柄）被本方案 1（边框式缩放）覆盖，无需重复实施。
