# 创作端 Agent 友好画布语义层方案

## 背景

创作端画布的首要价值不是成为白板，而是让多页面 Demo 的状态、布局和文字说明成为 Agent 可以稳定读取、引用和改写的工作上下文。

当前方向进一步收敛：画笔、箭头、形状等视觉批注不进入当前阶段；它们可以作为未来“批注模式”的计划，但不是当前画布核心能力。当前阶段只保留页面管理、文档说明、图片素材和文本语义节点。

文本工具不应被视为批注层能力。文字天然适合 Agent 读取、搜索、总结和改写，因此文本应归入 Agent 可读写的语义层。

## 目标

- 优先保证页面预览画布稳定、流畅、可理解。
- 保留现有 `arrangeCanvasPages`，让 Agent 能整理页面位置和尺寸。
- 让 Agent 能读取和改写画布中的文本语义节点。
- 让 Agent 能读取画布中的文档说明。
- 删除当前已经实现的画笔和箭头工具，避免当前阶段走向白板化。
- 将画笔、箭头、形状、高亮等视觉批注能力放入未来计划。

## 非目标

- 不把创作端画布改造成完整白板产品。
- 不在当前阶段提供画笔、箭头、形状或高亮工具。
- 不在当前阶段建设待办、任务、分组、评审结论等画布功能。
- 不让 Agent 依赖图像、笔迹或自由箭头来理解项目语义。
- 不在主画布中嵌入 Excalidraw、React Flow、tldraw 或类似第二套画布。

## 当前边界

### 当前保留

- 页面卡片：位置、尺寸、选中、拖拽、缩放、自动整理。
- 文档节点：用于较长说明和 Markdown 内容。
- 图片节点：用于素材参考。
- 文本语义节点：用于短说明、备注、修改意图和 Agent 可读写文本。

### 当前删除

- 箭头工具。
- 画笔工具。
- 箭头节点创建链路。
- 画笔节点创建链路。
- 当前正式状态中的 `arrow/draw` 工具态。

### 未来计划

未来可以新增“批注模式”，但它必须是独立的视觉说明层：

- 进入批注模式后才显示画笔、箭头、形状、高亮等工具。
- 批注层只用于人类视觉表达。
- Agent 默认不写批注层。
- Agent 如需理解批注，只通过画布截图作为视觉参考。
- 批注层不得接管页面拖拽、页面配置入口、页面预览和 Agent 语义层。

## 现状实现

### 已有 Agent 画布控制能力

当前 Agent Service 已提供 `arrangeCanvasPages` 工具，入口位于 `packages/agent-service/src/backends/pi-tools/canvas-layout-tool.ts`，并在 `createWorkbenchTools` 中注册。工具能力列表由 `getWorkbenchToolCapabilities()` 暴露。

`arrangeCanvasPages` 当前支持：

- `mode: preserveGroups | grid`：保留当前大致行列关系，或重排为规则网格。
- `orderBy: currentPosition | pageOrder`：按当前画布位置或页面树顺序排列。
- `sizeMode: preserve | preview`：保留当前画布尺寸，或按页面 `$demo.previewSize` 复原。
- `pageIds`：只整理指定页面。
- `gap`、`columns`、`viewportWidth`、`viewportHeight`：控制间距、网格列数和保存后的视口。

该工具读取 `workspace-tree.json` 和每个页面的 `config.schema.json`，写入工作区根目录 `.canvas-layout.json`，返回被整理页面的布局摘要和结构化 details。

### 已有 Prompt 约束

`packages/author-site/src/lib/agent/system-prompt.ts` 会根据工具能力动态生成画布管理规则：

- 如果存在 `arrangeCanvasPages`，用户要求整理画布、排列画布页面、调整画布中页面位置或尺寸时，Agent 必须使用该工具。
- 明确禁止 Agent 用 `writeFile`、`editFile`、`bash` 或 `node` 直接创建、修改或覆盖 `.canvas-layout.json`。
- 如果工具不存在，Agent 必须说明当前 Agent Service 版本过旧或工具未加载。

### 前端画布同步链路

前端画布状态由 `useCanvasWorkspace` 管理：

- 通过 `loadCanvasLayout()` 从 `/api/sessions/:sessionId/canvas-layout` 加载。
- 通过 `saveCanvasLayout()` debounce 保存。
- 通过 `flushCanvasState()` 在版本保存等流程前强制落盘。
- 通过 `applyRemoteCanvasState()` 应用远端协同状态。

编辑页还把 `.canvas-layout.json` 接入协同文档：

- `useCollabDocument({ resourcePath: ".canvas-layout.json", kind: "canvas-layout" })` 监听工作区文件。
- 当 Agent 工具写入 `.canvas-layout.json` 后，协同层同步文本内容。
- 编辑页解析远端 layout 后调用 `applyRemoteCanvasState()`，画布随之刷新。
- 本地画布交互产生未保存变更时，会把当前 `canvasState` 序列化回协同文档。

因此，当前 Agent 对画布的控制链路是：`arrangeCanvasPages` 写工作区文件 -> 协同文档同步 -> 编辑页应用远端状态 -> `PreviewCanvas` 重新渲染。

## 核心原则

### 1. 页面画布优先

页面卡片、截图/iframe 调度、页面选择、页面拖拽、页面尺寸、自动排版、聚焦和回到单页编辑始终是核心能力。

### 2. 文本是语义，不是批注

画布文本节点必须作为 Agent 可读写语义对象，而不是截图像素或自由批注。Agent 应能直接读取文本内容、标题、位置和关联页面；后续也应能通过命令创建、修改和删除文本节点。

### 3. 当前不做复杂画布功能

当前阶段不做待办、任务、分组、评审、关系图等画布功能。需要表达意图时，优先使用文本语义节点或文档节点。

### 4. 命令式改写

Agent 修改画布必须通过工具或命令入口完成。已经存在的 `arrangeCanvasPages` 是第一条画布命令，后续文本语义节点也应走命令，而不是让 Agent 直接写 `.canvas-layout.json`。

### 5. 未来批注层不影响核心画布

未来批注层只是一种视觉说明模式。它可以给人画圈、画箭头、画形状，也可以被 Agent 通过截图看见，但不承担 Agent 主数据层职责。

## 数据模型建议

当前阶段只保留必要结构：

```text
canvasState
  viewport
  pages
  nodes
    document nodes
    image nodes
    text semantic nodes
  layers
    annotations
      nodes              # 当前仅兼容文档、图片、文本；未来批注层不得混入当前语义文本
    documents
      nodes
    semantics
      textNodes          # 后续可将文本语义节点迁移到这里
```

迁移策略：

- 旧 `nodes` 字段继续兼容当前文档、图片、文本节点。
- 保存时继续写 `nodes`，同时写 `layers.annotations.nodes`。
- 当前不再新增 arrow/drawing 节点。
- 后续如果加入批注层，应使用独立 `layers.visualAnnotations`，不要混入文本语义层。
- Agent 工具更新 `pages` 或 `viewport` 时必须保留 `nodes`、`layers` 和未来语义字段。

## 实施路径

### 阶段一：撤回白板化和绘图能力

状态：进行中。

任务：

- 移除正式工具栏中的结构化图层和 Excalidraw 验证入口。
- 移除未进入当前正式路径的图编辑/白板依赖。
- 移除 `graph/excalidraw` 工具态、graph 默认数据和 graph 保存解析。
- 删除当前已经实现的箭头工具和画笔工具。
- 删除箭头/画笔创建链路和当前正式类型。
- 保留文本工具、文档节点、图片节点。

验收：

- 画布工具栏不再显示箭头和画笔。
- 点击或拖拽画布不会创建箭头或画笔节点。
- 页面预览、页面拖拽、缩放、自动排版不受影响。
- 文本节点仍可创建、编辑、移动和保存。

### 阶段二：文本语义节点 Agent 可读

目标：让 Agent 能稳定读取画布文本，而不是靠截图 OCR。

任务：

- 定义文本语义节点摘要：id、text、title、layout、关联页面线索、updatedAt。
- 在 `CanvasAgentContext` 中注入文本节点摘要。
- 控制上下文大小：短文本完整注入，长文本截断并保留节点 id。
- 文本节点应能被 Agent 引用，例如“修改画布上的说明文字”。

验收：

- Agent 不需要截图也能知道画布文本节点内容。
- 文本节点内容变化后，Agent 上下文能反映最新内容。

### 阶段三：文本语义节点 Agent 可写

目标：让 Agent 能通过命令修改文本节点。

任务：

- 新增文本节点命令，例如 `createCanvasText`、`updateCanvasText`、`deleteCanvasText`。
- 命令必须保留页面布局、文档节点、图片节点和未知合法扩展字段。
- 命令返回变更摘要，用于聊天区展示。
- Prompt 中明确：修改画布文本必须走文本节点命令，不能直接写 `.canvas-layout.json`。

验收：

- Agent 可以新增、修改、删除画布文本节点。
- Agent 文本命令不会破坏页面布局。
- 用户能看懂 Agent 改了哪条文本。

### 阶段四：加固现有 `arrangeCanvasPages`

目标：确保现有 Agent 页面布局命令不会破坏文本语义节点。

任务：

- 扩展 agent-service 内部 `CanvasState` 解析，使其保留 `nodes`、`layers` 和未来语义字段。
- `arrangeCanvasPages` 只改 `state.pages` 和 `state.viewport`，不得丢弃文本、文档或图片节点。
- 将 author-site 与 agent-service 的画布状态解析规则对齐。
- 增加 agent-service 测试：已有 `.canvas-layout.json` 包含文本节点时，调用 `arrangeCanvasPages` 后文本节点仍保留。

验收：

- Agent 整理画布不会删除文本、文档或图片节点。
- 前端协同同步后能正确刷新页面布局。

### 阶段五：未来批注层评估

目标：在核心画布和文本语义层稳定后，再评估视觉批注模式。

可选能力：

- 画笔。
- 箭头。
- 矩形/圆形。
- 高亮。

边界：

- 批注层默认只给人看。
- Agent 默认不可写批注层。
- Agent 可通过画布截图把批注作为视觉参考。
- 批注层不能影响文本语义节点、页面布局和页面预览。

## 验证策略

- shared：覆盖工具栏不显示箭头/画笔、文本节点仍可创建。
- author-site API：覆盖 arrow/drawing 节点不再作为合法当前节点保存。
- agent-service：后续覆盖 `arrangeCanvasPages` 保留文本节点。
- 必要时运行 `corepack pnpm --filter @opencode-workbench/author-site typecheck` 和相关 Jest 用例。

## 风险与待确认事项

- 如果用户已有历史箭头/画笔节点，当前删除正式支持后需要决定是忽略、迁移还是清理。
- 文本节点从自由节点迁移到更明确的 `layers.semantics.textNodes` 时，需要保持旧数据兼容。
- 未来批注层如果只靠截图给 Agent 看，理解精度会低于结构化文本。
