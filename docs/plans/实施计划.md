## Phase 1: 基础设施（优先级 P0）

### 1.1 修复 AI 对话区核心 Bug

**问题：** 当前 `/api/ai/chat` 使用错误的 SSE 事件流解析方式，导致 AI 回复始终为空。

**改动文件：**
- `packages/web/src/app/api/ai/chat/route.ts` — 重写
- `packages/web/src/app/demo/[id]/edit/page.tsx` — 简化前端调用

**实施内容：**
- [ ] 移除 `readSSEStream` 函数和 SSE 事件流逻辑
- [ ] 改为同步调用 `POST /session/{id}/message`，直接获取 `{ info, parts }` 响应
- [ ] 从 `parts` 中提取 `type === 'text'` 的文本内容作为 AI 回复
- [ ] 前端 `handleAiSend` 改为普通 HTTP 调用，不再解析 SSE
- [ ] 添加错误处理和日志输出

**验收标准：**
- 发送 AI 消息后能正确收到 AI 回复文本
- 浏览器控制台无 SSE 相关错误

### 1.2 Session 管理服务

**新增文件：**
- `packages/web/lib/session-manager.ts` — Session 生命周期管理
- `packages/web/lib/session-guard.ts` — 文件变更校验（兜底防护）
- `packages/web/lib/opencode-client.ts` — opencode API 封装

**实施内容：**

#### 1.2.1 `session-manager.ts`

```typescript
// 核心函数
createEditSession(demoId: string)     // Clone + Inject → 返回 sessionId
getEditSession(sessionId: string)     // 获取 session 信息
saveEditSession(sessionId: string)    // Merge → demos, 清理 session
dropEditSession(sessionId: string)    // 删除 session 目录
cleanupExpiredSessions()              // 超时清理（2 小时）
```

**工作流程：**
1. `createEditSession`:
   - `fs.cpSync(demoDir, sessionDir, { recursive: true })`
   - 写入 `opencode.json`（权限白名单配置）
   - 写入 `AGENTS.md`（行为约束）
   - 记录 session 元数据（demoId、创建时间、用户）
2. `saveEditSession`:
   - 后端校验文件变更合法性
   - `fs.cpSync(sessionDir, demoDir, { recursive: true })`
   - `fs.rmSync(sessionDir, { recursive: true })`
3. `dropEditSession`:
   - `fs.rmSync(sessionDir, { recursive: true })`

#### 1.2.2 `session-guard.ts`

```typescript
// 核心函数
validateFileChanges(sessionId, changedFiles)  // 校验变更白名单
rollbackIllegalChanges(sessionId, violations) // 自动回滚
```

**校验逻辑：**
- 白名单文件：`index.tsx`、`config.schema.json`、`AGENTS.md`
- 非法新文件 → 删除
- 非法修改 → 从原始 Demo 恢复

#### 1.2.3 `opencode-client.ts`

```typescript
// 核心函数
createOpencodeSession(sessionDir)    // POST /session
sendAiMessage(sessionId, message)    // POST /session/{id}/message
readFile(sessionId, path)            // GET /file/content?path=
getFileStatus(sessionId)             // GET /file/status
```

**关键修正：**
- 使用 `POST /session/{id}/message` 同步调用（非 SSE）
- 请求体：`{ parts: [{ type: 'text', text: message }] }`
- 响应：`{ info: Message, parts: Part[] }`
- Session 复用：首次创建后缓存 sessionId

**改动文件：**

**API 设计：**

| 路由 | 方法 | 功能 | 请求体 | 响应 |
|------|------|------|--------|------|
| `/api/sessions` | POST | 创建编辑会话 | `{ demoId }` | `{ sessionId, code, schema }` |
| `/api/sessions` | GET | 列出活跃会话 | - | `{ sessions[] }` |
| `/api/sessions/[id]` | GET | 获取会话信息 | - | `{ sessionId, demoId, ... }` |
| `/api/sessions/[id]` | DELETE | 放弃编辑 | - | `{ success: true }` |
| `/api/sessions/[id]/files` | GET | 读取文件 | `?path=index.tsx` | `{ content }` |
| `/api/sessions/[id]/save` | POST | 保存会话 | - | `{ success: true }` |
| `/api/ai/chat` | POST | AI 对话 | `{ sessionId, message }` | `{ aiReply, code, schema }` |

## Phase 2: 前端重构（优先级 P0）

### 2.1 AI 编辑工作台页面重构

**改动文件：**
- `packages/web/src/app/demo/[id]/edit/page.tsx` — 大幅重写

**改动内容：**

#### 2.1.1 简化 AI 对话逻辑

```typescript
// Before: SSE 流式解析 + 手动维护消息历史
// After: 普通 HTTP 调用 + opencode 自动维护历史

const handleAiSend = async () => {
  const response = await fetch('/api/ai/chat', {
    method: 'POST',
    body: JSON.stringify({ sessionId, message: userInput }),
  });
  const result = await response.json();
  
  // 更新 AI 对话历史
  setAiMessages(prev => [...prev,
    { role: 'user', content: userInput },
    { role: 'assistant', content: result.data.aiReply },
  ]);
  
  // 更新预览
  setCode(result.data.code);
  setSchema(result.data.schema);
};
```

#### 2.1.2 Session 生命周期管理

```typescript
// 进入编辑页
useEffect(() => {
  const init = async () => {
    const res = await fetch('/api/sessions', {
      method: 'POST',
      body: JSON.stringify({ demoId }),
    });
    const { sessionId, code, schema } = (await res.json()).data;
    setSessionId(sessionId);
    setCode(code);
    setSchema(schema);
  };
  init();
  
  // 离开时自动清理
  return () => {
    fetch(`/api/sessions/${sessionId}`, { method: 'DELETE' });
  };
}, [demoId]);
```

#### 2.1.3 保存/放弃按钮逻辑

```typescript
const handleSave = async () => {
  await fetch(`/api/sessions/${sessionId}/save`, { method: 'POST' });
  router.push('/');
};

const handleDiscard = async () => {
  await fetch(`/api/sessions/${sessionId}`, { method: 'DELETE' });
  router.push('/');
};
```

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| opencode 权限系统行为与预期不符 | 中 | 高 | Phase 1 先手动测试权限配置，确认行为后再开发 |
| `POST /session/{id}/message` 返回格式变化 | 低 | 高 | 先用 curl 手动测试确认格式 |
| 单进程下多 Session 并发冲突 | 低 | 中 | Session 目录隔离 + 权限白名单，理论上不会冲突 |
| opencode server 崩溃影响所有编辑 | 低 | 高 | 监控 + 自动重启机制 |

## 关键验证步骤（开发前必须执行）

在开始编码前，必须先验证以下假设：

```bash
# 1. 确认 opencode serve 能以 sessions 为工作目录启动
cd sessions
opencode serve --hostname 127.0.0.1 --port 4096

# 2. 确认 POST /session 能创建 session
curl -X POST http://127.0.0.1:4096/session \
  -H "Content-Type: application/json" \
  -d '{"title":"test"}'

# 3. 确认 POST /session/{id}/message 同步返回
curl -X POST http://127.0.0.1:4096/session/{id}/message \
  -H "Content-Type: application/json" \
  -d '{"parts":[{"type":"text","text":"hello"}]}'

# 4. 确认 opencode.json 权限配置生效
# 在 session 目录中写入测试用 opencode.json
# 验证 AI 尝试编辑非白名单文件时被拦截

# 5. 确认 GET /file/content 能读取文件
curl "http://127.0.0.1:4096/file/content?path=index.tsx"
```
