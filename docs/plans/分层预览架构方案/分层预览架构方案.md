# 分层预览架构方案

> **版本**: v2.3
> **更新**: 2026-04-19
> **前提**: AI 生成的代码即为最终发布代码，不存在人工审核。系统首要目标是稳定、省心，让 AI 可以自由发挥使用任意 npm 依赖。开发成本可控。

---

## 一、问题诊断

### 1.1 当前方案的本质缺陷

`PreviewScope` 通过**白名单式 CSS 属性重置**隔离污染。但全局样式和 AI 生成代码的组合是无限的：

| 已踩过的坑 | 根因 |
|:-----------|:-----|
| 文字色继承为浅色 | `body { color: ... }` 污染 |
| 边框色异常 | `* { border-color: ... }` 污染 |
| 背景色无法生效 | `.preview-scope` 的 `background-color` 覆盖了 AI 的 Tailwind 类 |
| 字体大小异常 | `font-size` 继承链污染 |
| 未来未知 | 新的全局样式、新的 CSS 变量、新的继承属性 |

**核心矛盾**：CSS Reset 是在跟无限打地鼠。

### 1.2 场景特征

系统只有两类场景，核心诉求相同——**完全隔离 AI 生成的未知代码**：

| 维度 | AI 创作/调试 | 跨系统嵌入 |
|:-----|:-------------|:-----------|
| **代码来源** | AI 实时生成 | AI 生成的同一份代码 |
| **代码可控性** | 不可控（AI 任意生成） | 不可控（AI 任意生成） |
| **运行环境** | 本应用（全局样式复杂） | 外部系统（环境完全未知） |
| **首要目标** | 安全隔离、不崩页面 | 安全隔离、不崩宿主页面 |
| **可接受延迟** | 1-2s | 1-2s（更低当然更好） |
| **出错容忍** | 必须隔离 | 必须隔离 |

**关键认知**：两个场景的本质需求完全相同——**完全隔离**。不存在"已验证代码可以放心直接渲染"的情况。

## 二、推荐方案：统一 iframe 沙箱 + 动态 CDN 依赖

### 2.1 架构决策

> **两个场景统一使用 iframe 完全隔离。**
>
> 不维护 Direct 模式。AI 生成的代码即最终代码，没有"已验证可以放心直接渲染"的中间态。
>
> AI 可以使用任意 npm 依赖，通过动态 CDN（esm.sh / unpkg）按需加载，无需白名单限制。

| 场景 | 渲染模式 | 代码注入方式 | 依赖加载方式 | 隔离性 |
|:-----|:---------|:-------------|:-------------|:-------|
| AI 创作/调试 | iframe 沙箱 | `postMessage` 动态注入编译后的代码 | 首次按需从 CDN 下载，浏览器缓存 | DOM/JS/CSS 完全隔离 |
| 跨系统嵌入 | iframe 沙箱 | 静态 iframe URL 加载预编译完整 HTML | 首次按需从 CDN 下载，浏览器缓存 | DOM/JS/CSS 完全隔离 |

### 2.2 为什么不用 Sandpack

| 对比 | Sandpack | iframe + 动态 CDN |
|:-----|:---------|:------------------|
| **隔离性** | ✅ iframe | ✅ iframe |
| **依赖支持** | ✅ 任意 npm | ✅ 任意 npm |
| **冷启动** | ❌ 2-5s（浏览器端打包） | ✅ ~1-2s（只下载，不打包） |
| **代码变更后** | ❌ 重新打包 2-5s | ✅ 依赖已缓存，只重新执行组件代码 |
| **包体积** | ❌ ~500KB | ✅ ~0KB |

**Sandpack 的 2-5s 冷启动在 AI 实时调试场景不可接受**——每次 AI 修改代码后用户都要等 2-5s 才能看到效果。

动态 CDN 方案的核心优势：
- **不打包**：浏览器直接通过 ES Module 加载已打包好的 CDN 资源，省去浏览器端打包的时间
- **缓存复用**：React、Tailwind 等公共依赖首次下载后缓存，后续代码变更无需重新下载
- **省心**：AI 可以任意 `import`，系统自动解析并映射到 CDN

### 2.3 为什么不保留 Direct 模式

| 论点 | 结论 |
|:-----|:-----|
| "同一应用内性能更好" | 同一应用内展示场景不考虑，且 iframe 的 1-2s 延迟可接受 |
| "AI 代码在本应用运行，环境已知" | AI 代码不可控，本应用的全局样式同样不可控（Tailwind preflight、主题变量、第三方库样式） |
| "维护两套模式更灵活" | 增加复杂度，且 Direct 模式在 AI 代码即最终代码的前提下没有安全使用场景 |

**单模式策略的好处**：
- 开发成本最低：只维护一套渲染系统
- 稳定性最高：iframe 完全隔离，任何场景的 AI 代码都不会破坏宿主
- 逻辑最清晰：不存在"什么场景用什么模式"的判断负担

## 三、iframe 沙箱实现

### 3.1 核心机制：服务端编译 + iframe 动态加载

**服务端（`/api/compile`）职责**：
1. 使用 sucrase 编译用户代码（~50ms）
2. **分析代码中的 `import` 语句，提取依赖列表**（使用 sucrase 的 AST 输出，而非正则，避免注释/字符串误判）
3. 生成 **Import Map**，将每个依赖映射到 CDN URL
4. 返回：编译后的代码 + Import Map

**接口输入**（推荐传 `sessionId`，让服务端直接读取草稿文件）：
```typescript
// POST /api/compile
{
  sessionId: string  // 服务端从 session/{sessionId}/draft/index.tsx 读取代码
}
```

**为什么传 `sessionId` 而非 `code: string`**：
- 减少一次前端读取文件再传输的网络开销
- 编译接口天然知道去哪里找 `config.schema.json`，便于后续扩展
- 便于在服务端做缓存（以文件内容 hash 为 key）

**错误处理与降级策略**：
- **编译失败**（语法错误、sucrase 不支持语法）→ 返回 `success: false`，附带具体错误信息
  - **降级**：iframe 保持上一个成功渲染的结果，叠加错误提示层（不直接清空），直到用户修复错误或重新生成
- **代码包含 React 渲染错误** → 由 iframe 内的 Error Boundary 捕获，通过 `postMessage` 回传错误详情到父窗口展示
- **CDN 资源加载失败**（404/500）→ iframe 内捕获 `error` 事件，回传失败依赖名称，UI 提示"依赖加载失败，AI 可能使用了不存在的包"
  - **降级**：跳过该依赖继续渲染（可能组件部分功能缺失），或完全失败并提示

**iframe 内职责**：
1. 通过 `importmap` 加载依赖（浏览器原生 ES Module）
2. 执行编译后的组件代码
3. **接收配置数据**：通过 `postMessage` 接收 `configData`，作为 props 传入 React 组件
4. 渲染到 DOM

**iframe 内引导代码（HTML 模板中的常驻脚本）**：
iframe 加载时包含一段固定引导代码，负责挂载 React 应用和监听 `postMessage`：
```typescript
// iframe HTML 模板中的常驻脚本
let currentRoot: ReactDOM.Root | null = null
let currentConfig: Record<string, unknown> = {}

window.addEventListener('message', (event) => {
  // 来源验证：只接受父窗口的消息
  if (event.source !== window.parent) return

  const { type, code, importMap, configData } = event.data

  if (type === 'UPDATE_CODE') {
    // 动态注入 Import Map
    const importMapEl = document.getElementById('importmap') as HTMLScriptElement
    importMapEl.textContent = JSON.stringify(importMap)

    // 创建 blob URL 加载编译后的 ES Module 代码
    const blob = new Blob([code], { type: 'application/javascript' })
    const moduleUrl = URL.createObjectURL(blob)

    import(moduleUrl).then((module) => {
      const Component = module.default
      if (!currentRoot) {
        currentRoot = ReactDOM.createRoot(document.getElementById('root')!)
      }
      currentRoot.render(React.createElement(Component, currentConfig))
    }).catch((err) => {
      window.parent.postMessage({ type: 'RUNTIME_ERROR', error: err.message }, '*')
    })
  }

  if (type === 'UPDATE_CONFIG') {
    currentConfig = configData || {}
    if (currentRoot) {
      // 触发重新渲染，使用新的 props
      // 组件引用需要从缓存中获取，这里简化示意
    }
  }
})
```

**注意**：使用 React 18 的 `createRoot` API，废弃 `ReactDOM.render()`。

**配置数据传递机制**（支撑配置面板实时联动）：
iframe 内维护一个 `currentConfig` 状态，支持两种消息类型：
```typescript
// 代码变更 → 重新编译执行组件
{ type: 'UPDATE_CODE', code: '...', importMap: {...} }

// 配置变更 → 只更新 props，不重新编译，不重新加载依赖
{ type: 'UPDATE_CONFIG', configData: {...} }
```
收到 `UPDATE_CONFIG` 时，iframe 内调用 `currentRoot.render(React.createElement(Component, configData))`，实现配置变更到预览更新的一帧级响应（~16ms）。

**iframe sandbox 安全属性**：
```html
<iframe sandbox="allow-scripts allow-same-origin" />
```
- `allow-scripts`：必须，否则 JS 不执行
- `allow-same-origin`：需要，否则 `srcdoc` 模式下的 ES Module + Import Map 行为异常
- **来源验证**：父窗口通过 `event.source === iframe.contentWindow` 验证消息来源；iframe 内通过 `event.source === window.parent` 验证，防止恶意页面伪造消息

**为什么浏览器端不需要打包**：
- 现代浏览器原生支持 ES Module + Import Map
- CDN（如 esm.sh）已经提供了打包好的、可直接 import 的 ES Module
- 浏览器只需"下载"，不需要"打包"，这是冷启动快 10 倍的核心原因

### 3.2 依赖加载流程

```
AI 生成代码
    │
    ▼
import { format } from 'date-fns'
import { motion } from 'framer-motion'
    │
    ▼
服务端编译（sucrase）
    │
    ├── 编译代码 → 编译后的 JS
    ├── 分析 import → ['date-fns', 'framer-motion']
    └── 生成 Import Map:
        {
          "imports": {
            "date-fns": "https://esm.sh/date-fns",
            "framer-motion": "https://esm.sh/framer-motion"
          }
        }
    │
    ▼
发送到 iframe
    │
    ▼
iframe 内执行：
    1. 注入 Import Map
    2. 浏览器自动从 esm.sh 下载 date-fns、framer-motion
    3. 下载完成后执行编译代码
    4. 渲染组件
```

**缓存机制**：
- 浏览器会自动缓存 `esm.sh` 的资源
- 后续代码变更（如 AI 修改了组件逻辑），依赖不需要重新下载，只需重新执行组件代码
- 这带来的体验是：首次 1-2s，后续代码变更几乎是瞬时的

### 3.3 iframe srcdoc 内容

iframe 的 HTML 模板包含：
- 基础样式重置（独立的，不受宿主影响）
- Tailwind CSS（CDN 或本地）
- React / ReactDOM（CDN）
- Import Map 占位符（运行时由父窗口或服务端填充）
- CSS `<link>` 占位符（用于动态注入 AI 代码中的 `import 'xxx.css'` 对应的 CDN 链接）
- **Error Boundary**：包裹根组件，捕获 React 渲染错误并通过 `postMessage` 回传
- **ResizeObserver**：监听 `document.body` 高度变化，通过 `postMessage` 回传到父窗口实现高度自适应
- 组件渲染逻辑

**CDN 选择**：
- **esm.sh**：推荐，自动 tree-shaking、依赖解析、TypeScript 类型支持
- **unpkg / jsdelivr**：备选，如果 esm.sh 不稳定可切换
- **自建 CDN**：如果内网环境无法访问外网，需要自建 npm 包代理

### 3.4 场景一：AI 创作/调试（草稿工作区模式）

**代码来源**：草稿工作区（`session/{sessionId}/draft/index.tsx`），而非原始 Demo 文件。AI 编辑不直接修改原文件，所有变更在草稿副本中进行。

**通信流程（双通道协议）**：

**通道 A：代码变更（低频，AI 生成/用户直接编辑时触发）**
1. 父窗口调用 `/api/compile` 编译草稿工作区的代码
2. 服务端返回：编译后的代码 + Import Map
3. 父窗口创建/复用 iframe，通过 `postMessage` 发送：
   ```typescript
   { type: 'UPDATE_CODE', code: '...', importMap: {...} }
   ```
4. iframe 内重新注入 Import Map，下载新依赖（如需要），执行代码，渲染组件

**通道 B：配置变更（高频，用户在配置面板操作时触发）**
1. 用户在配置面板修改某项配置（如替换图片 URL、修改文案）
2. 父窗口直接通过 `postMessage` 发送：
   ```typescript
   { type: 'UPDATE_CONFIG', configData: { title: '新标题', image: '...' } }
   ```
3. iframe 内仅更新 `currentConfig` 状态，调用 `currentRoot.render(React.createElement(Component, configData))`
4. **不重新编译，不重新加载依赖，一帧级响应（~16ms）**

**两条通道的对比**：
| 通道 | 触发时机 | 耗时 | 是否重新编译 | 是否重置状态 |
|:---|:---|:---|:---|:---|
| `UPDATE_CODE` | AI 修改代码、用户直接编辑代码 | ~200ms（依赖已缓存）| ✅ | ✅ |
| `UPDATE_CONFIG` | 用户在配置面板修改配置 | ~16ms | ❌ | ❌ |

**iframe 复用优化**：
- AI 连续生成多版代码时，复用同一个 iframe，避免重复创建和 CDN 重新加载
- 只有代码发生变更（`UPDATE_CODE`）时才重新注入，依赖缓存保持有效
- **注意**：重新注入组件代码会导致 React 状态完全重置（`useState`、`useRef` 等归零）。这是 iframe 完全隔离的固有代价，预期管理：AI 实时调试时，状态变更不会跨代码版本保留

**保存操作与预览的关系**：
1. AI 编辑过程中，iframe 加载的是**草稿工作区**的代码
2. 用户点击"保存"时：
   - 将草稿内容复制到原文件（`projects/{projectId}/index.tsx`）
   - 触发 `/api/generate-schema` 更新 `config.schema.json`
   - 生成缩略图（供列表页使用）
3. 保存**不影响**当前正在预览的 iframe（继续显示草稿内容）
4. 用户刷新页面后，Demo 使用页面加载的是已保存的原文件

**iframe 高度自适应**：
- AI 生成的组件高度往往是动态的（列表展开/收起、内容自适应）
- iframe 内通过 `ResizeObserver` 监听 `document.body` 高度变化
- 高度变化时通过 `postMessage` 回传实际高度到父窗口
- 父窗口动态调整 iframe 的 `height` CSS 属性，避免内容截断或大量空白
- 跨系统嵌入场景同样适用：外部系统无需手动计算高度，iframe 自动适配

### 3.5 场景二：跨系统嵌入（外部系统）

**问题**：如果要求嵌入方先调用 `/api/compile` 再通过 `postMessage` 注入代码，集成成本过高。

**方案**：提供**独立的静态 iframe URL**，服务端直接返回一个完整的、已编译的 HTML 文档。

嵌入方只需一行代码：
```html
<iframe 
  src="https://workbench.company.com/embed/demo-123/iframe" 
  sandbox="allow-scripts"
/>
```

这个 URL 返回的 HTML 包含：
- 基础样式重置
- React、Tailwind（CDN）
- **内联的 Import Map**（服务端已根据代码分析生成）
- **已编译的组件代码**（内联在 `<script>` 中）
- 内联的默认配置数据

**好处**：
- 嵌入方零集成成本，不需要知道任何 postMessage 协议
- 编译和依赖分析在服务端完成，iframe 只负责下载依赖并渲染
- 完全隔离样式和依赖

**本系统内的 Demo 使用页面**：不通过独立 iframe URL，而是直接使用 `PreviewPanel` 组件 + 配置面板组件组合。`PreviewPanel` 内部复用同一套 iframe 机制，只是代码通过 `/api/compile` 实时编译注入，而非服务端预生成完整 HTML。

### 3.6 延迟分析

| 阶段 | 耗时 | 说明 |
|:-----|:-----|:-----|
| 服务端编译（sucrase） | ~50ms | 代码越短越快 |
| 依赖分析 + Import Map 生成 | ~10ms | AST 提取 import 语句，精确可靠 |
| iframe 创建 | ~50ms | 可复用已有 iframe |
| CDN 下载公共依赖（React、Tailwind） | ~200-500ms | 首次，后续浏览器缓存 |
| CDN 下载业务依赖（date-fns 等） | ~100-300ms/个 | 首次，后续浏览器缓存 |
| 代码执行 + 渲染 | ~50-100ms | 取决于组件复杂度 |
| **总延迟（首次冷启动）** | **~1-2s** | 可接受 |
| **总延迟（代码变更，依赖已缓存）** | **~200ms** | 几乎瞬时 |

**关键体验**：AI 第一次生成代码时等待 1-2s，后续 AI 修改代码（如调整颜色、文案）几乎秒出，因为公共依赖已缓存。

### 3.7 为什么不内联 React（CDN vs 本地）

| 方案 | 优点 | 缺点 |
|:-----|:-----|:-----|
| CDN 加载 React | iframe 完全独立，版本可控，浏览器缓存共享 | 首次需下载 |
| 本地打包 React 进 iframe | 无网络依赖 | 每个 iframe 都包含一份 React，浪费带宽 |

**推荐 CDN**：React、Tailwind 等公共依赖通过 CDN 加载，浏览器缓存后多 iframe 共享。

### 3.8 config.schema.json 同步更新机制

**问题**：AI 修改组件 props 后，配置面板需要知道新字段，否则配置面板与组件脱节。

**方案**：新增 `/api/generate-schema` 接口，根据组件 TypeScript 接口自动生成 JSON Schema。

**触发时机**：
1. AI 完成一次代码修改（`UPDATE_CODE` 成功返回后）
2. 用户手动保存代码时

**接口输入**：
```typescript
// POST /api/generate-schema
{
  sessionId: string  // 从草稿工作区读取 index.tsx
}
```

**接口输出**：
```typescript
{
  success: true,
  data: {
    schema: object,           // 生成的 JSON Schema
    updatedAt: string         // 更新时间
  }
}
```

**生成策略**：
- 解析 `index.tsx` 中的 `Props` 接口或组件函数参数类型
- 基础类型映射：
  - `string` → `{ type: 'string' }`
  - `number` → `{ type: 'number' }`
  - `boolean` → `{ type: 'boolean' }`
  - 联合类型（如 `'a' | 'b'`）→ `{ type: 'string', enum: ['a', 'b'] }`
- 保留现有 Schema 中已配置的 `ui:widget`、`default` 等扩展字段（不覆盖）
- 新增字段使用类型默认值作为 `default`

**与配置面板联动**：
- Schema 更新后，配置面板自动重新生成表单
- 新增字段使用默认值填充，已修改的配置项保持不变
- 删除字段时，对应配置数据自动清理

### 3.9 Demo 列表页预览策略

**问题**：Demo 列表页以卡片形式展示所有 Demo，如果每个卡片都渲染实时 iframe，10 个 Demo 同时加载 → 10 个 iframe × React + Tailwind CDN = 内存和 CPU 灾难。

**方案**：列表页不使用实时 iframe，采用**静态缩略图**策略。

| 页面 | 预览方式 | 原因 |
|:---|:---|:---|
| **Demo 列表页** | 静态缩略图（保存时自动生成） | 卡片多、无需交互、性能优先 |
| **Demo 使用页面** | 实时 iframe（`PreviewPanel` + 配置面板） | 需要交互、配置联动 |
| **AI 编辑工作台** | 实时 iframe（`PreviewPanel`） | 需要实时预览代码变更 |
| **跨系统嵌入** | 实时 iframe（独立 URL） | 外部系统需要实时渲染 |

**缩略图生成时机**：
1. 用户点击"保存"时，服务端触发截图（使用 `playwright` 或 `html-to-image`）
2. 截图结果保存为静态图片（如 `/public/thumbnails/{demoId}.png`）
3. 列表页直接加载图片，无 iframe 开销

**fallback**：如果缩略图不存在（新创建未保存），显示占位图或"暂无预览"。

## 四、依赖解析的边界情况

### 4.1 内网环境限制

**如果嵌入的系统在内网、无法访问外网 CDN**：

动态 CDN 方案会遇到限制，因为需要从 esm.sh / unpkg 下载 npm 包。

**应对策略**（按优先级）：

1. **内网搭建轻量级 npm 代理**（推荐）：
   - 使用 verdaccio 或 nexus 搭建内网 npm registry
   - 或搭建简单的 CDN 代理，缓存外网资源
   - 将 esm.sh URL 替换为内网代理地址

2. **预置常用依赖到本地静态资源**：
   - 提前下载 React、Tailwind、lucide-react 等高频依赖
   - iframe 通过相对路径加载本地资源
   - 业务依赖仍走 CDN（如果允许），或预置常见库

3. **混合方案**：
   - 公共依赖（React、Tailwind）→ 本地预置
   - 业务依赖（date-fns、framer-motion）→ CDN 或按需预置

### 4.2 AI 使用深路径导入

AI 可能写：
```typescript
import { format } from 'date-fns/format'
```

Import Map 需要支持子路径映射：
```json
{
  "imports": {
    "date-fns/format": "https://esm.sh/date-fns/format"
  }
}
```

实现方式：服务端分析 import 路径时保留完整路径，直接映射到对应 CDN URL。

### 4.3 CSS 导入（P0 必须支持）

AI 可能写：
```typescript
import 'some-lib/dist/style.css'
```

**为什么 P0 必须支持**：`antd`、`@mui/material`、`react-datepicker` 等主流 UI 库都依赖独立的 CSS 文件。如果 P0 不支持 CSS 导入，这些库的预览会呈现"裸奔"状态（无样式），直接影响可用性。

实现方式：
- 服务端检测到 `.css` import，在 iframe HTML 中注入对应的 `<link>` 标签（推荐，浏览器并行下载）
- 或通过 `fetch` 在 iframe 内动态加载 CSS 并注入 `<style>`（备选）

### 4.4 多文件约束（MVP 阶段）

AI 可能将组件拆分为多个文件：
```typescript
// index.tsx
import { Banner } from './components/Banner'

// components/Banner.tsx
export function Banner({ title }) { ... }
```

**MVP 阶段约束**：限制 AI 只能生成**单文件组件**。所有逻辑必须内联在 `index.tsx` 中，禁止 `import './xxx'` 形式的相对路径模块导入。

**原因**：
- `/api/compile` 当前只接收单字符串输入，多文件需要传递文件树，增加复杂度
- iframe 内动态注入多文件需要构建虚拟文件系统，超出 MVP 范围

**提示词约束**：在 `demo-generator.md` 中明确要求 AI：
- 所有组件代码写在单一文件中
- 如有复用逻辑，以内联函数形式实现
- 图片等资源使用绝对 URL 或 base64

**后续迭代**：如需支持多文件，可引入虚拟文件系统（如 `@webcontainers/api` 轻量方案），但冷启动时间会增加。

### 4.5 资源路径约束

AI 代码可能引用本地资源：
```tsx
<img src="/images/banner.png" />
```

在 `srcdoc` 或 `about:blank` 的 iframe 中，相对路径指向 `about:blank` → **404**。

**约束**：在 `demo-generator.md` 中要求 AI 使用绝对 URL 或 base64：
```tsx
// ✅ 正确
<img src="https://cdn.example.com/banner.png" />
<img src="data:image/png;base64,iVBORw0K..." />

// ❌ 错误
<img src="/images/banner.png" />
<img src="./assets/logo.png" />
```

### 4.6 依赖版本锁定

AI 代码可能依赖特定版本：
```typescript
import { format } from 'date-fns'  // 未指定版本 → 首次编译时锁定当前最新版
```

**策略**（避免版本漂移导致的运行时错误）：
- 未指定版本 → **首次编译时锁定为当前最新版**（如 `https://esm.sh/date-fns@3.0.0`），并将版本信息持久化到 demo 数据中
- 后续重新编译同一份代码时，复用已锁定的版本，避免"今天生成能用，下周生成报错"
- 如需升级版本 → 删除锁定记录后重新编译，或 AI 在 import 中显式指定版本
- 版本问题导致的不兼容 → AI 重新生成（不存在人工审核，AI 自己修）

### 4.7 浏览器兼容性

Import Map + ES Module 需要现代浏览器支持：

| 浏览器 | 最低版本 | 支持 Import Map |
|:---|:---|:---|
| Chrome | 89+ | ✅ |
| Safari | 15.4+ | ✅ |
| Firefox | 108+ | ✅ |
| Edge | 89+ | ✅ |

**判断**：本项目面向公司内部工具，Chrome/Edge 为主流浏览器，无需兼容旧版 IE/Safari。

**如需要兼容旧版浏览器**：
- 备选方案：使用 `SystemJS` 替代原生 ES Module（增加复杂度，不推荐）
- 或明确声明"不支持 IE 及 Safari 15 以下版本"

## 五、PreviewPanel 组件

### 5.1 接口

```tsx
interface PreviewPanelProps {
  code: string              // 编译前的原始组件代码
  configData?: Record<string, unknown>  // 配置数据，通过 postMessage 注入 iframe 作为组件 props
  className?: string
  previewSize?: PreviewSize // 从 config.schema.json 的 previewSize 字段读取
  onError?: (error: Error) => void      // 编译错误 or iframe 运行时错误
}
```

**设计要点**：
- **无需 `mode` 参数**：统一使用 iframe sandboxed，不存在 Direct 模式
- **`configData` 实时联动**：组件内部通过 `postMessage` 向 iframe 发送 `UPDATE_CONFIG` 消息，实现配置面板修改后预览区一帧级更新
- **`previewSize` 来源**：由父组件从 `config.schema.json` 的 `previewSize` 字段读取后传入，控制 iframe 容器尺寸
- **`onError` 统一收口**：编译错误（sucrase 报错）、iframe 内 React 渲染错误、CDN 加载失败，全部收敛到此回调

### 5.2 配置面板与 PreviewPanel 的数据流

**推荐模式：父组件统一管理 `configData`**

三栏布局（用户交互区 | 预览区 | 配置面板）的数据流：

```
配置面板（右）         父组件 state          PreviewPanel（中）
    │                        │                       │
    │── 用户修改配置 ────────▶│── configData ────────▶│── postMessage
    │                        │                       │   (UPDATE_CONFIG)
    │                        │                       │
    │◀── 表单重新生成 ◀──────│◀── schema 更新 ◀──────│
```

**初始配置数据初始化**：
1. 打开 Demo 使用页面时，从 `config.schema.json` 读取各字段的 `default` 值
2. 组装成初始 `configData` 对象
3. 同时传给配置面板（表单初始值）和 `PreviewPanel`（iframe 初始 props）
4. 用户修改配置后，父组件更新 `configData` state，双向同步

### 5.3 跨系统嵌入的独立路由

| 路由 | 用途 | 说明 |
|:-----|:-----|:-----|
| `/embed/[demoId]` | 嵌入指引页 | 提供 iframe 嵌入代码示例、尺寸说明 |
| `/embed/[demoId]/iframe` | 实际 iframe 内容 | 返回完整的预编译 HTML，供外部系统直接嵌入 |

## 九、核心决策

> **两个场景统一使用 iframe 完全隔离 + 动态 CDN 加载任意依赖。**
>
> AI 可以任意使用 npm 包，系统自动解析并映射到 CDN，无需白名单约束。
>
> 不维护 Direct 模式。AI 生成的代码即最终代码，iframe 是唯一可靠的隔离边界。
>
> 开发成本最小化：只维护一套渲染系统，两个场景的差异仅在代码注入方式（动态 postMessage vs 静态 URL）。

### 10.2 实现细节调整（与方案差异）

**Import Map 方案调整为 import 路径直接替换**：

- **方案原设计**：服务端生成 Import Map，iframe 内通过 `<script type="importmap">` 映射 npm 包到 CDN URL
- **实际实现**：服务端直接替换编译后代码中的 `from 'package'` 为 `from 'https://esm.sh/package'`，iframe 无需 Import Map 支持
- **原因**：sucrase 编译后代码中的 import 路径可被直接字符串替换，比 Import Map 更简洁，兼容性更好（旧版浏览器不支持 Import Map 的场景由方案本身的现代浏览器要求覆盖）

**依赖分析方式**：

- **方案原设计**：使用 sucrase AST 提取 import
- **实际实现**：使用正则提取（先移除注释），因为 sucrase 不提供 AST 输出 API，正则已足够可靠（MVP 阶段无复杂边界情况）
