# 编辑页保存失败问题分析报告

## 问题描述

用户在编辑页点击保存时，显示"保存失败"。

## 问题定位过程

### 第一阶段：初始诊断 (2026-04-18)

**客户端日志**：

```
[loadDemo] 开始加载 demo: proj_1776522616396
[loadDemo] Session API 响应: 201; JSON: Object; 创建成功, sessionId: session-1776522617027-54kzi464b
[loadDemo] Files API 响应状态: 200
[handleSave] 开始保存, sessionId: "session-1776522617027-54kzi464b"
[handleSave] 发送 PUT 请求到 /api/sessions/session-1776522617027-54kzi464b/files
POST /api/sessions/session-1776522617027-54kzi464b/merge 500 (Internal Server Error)
```

**结论**：
- ✅ PUT `/api/sessions/files` 成功
- ❌ POST `/api/sessions/merge` 失败，返回 500

**服务器端日志**：

```
[mergeSession] workspacePath: ...\data\projects\proj_1776522616396\workspace
[mergeSession] 创建快照: ...\data\snapshots\proj_1776522616396\v1
[mergeSession] 文件操作失败: Error: ENOENT: no such file or directory, lstat '...\workspace'
```

**初步判断**：项目目录下缺少 `workspace` 子目录。

### 第二阶段：深入调查

**问题1：项目是通过正常流程创建的，为何 workspace 会缺失？**

通过检查文件系统发现：
- 项目目录 `proj_1776522616396/` 下**只有** `project.json`，**没有** `workspace` 子目录
- session 目录 `session-1776522617027-54kzi464b` 下存在 `index.tsx` 和 `config.schema.json`
- 文件修改时间 `22:45` 表明 session 创建时 workspace 是存在的

**结论**：workspace 是在 session 创建后被删除的（可能原因：之前的失败操作残留、或外部因素）。

**问题2：为什么 `createEditSession` 创建 session 时没有报错？**

实际上，在后续复现时发现了更根本的问题——当 workspace 缺失时，`POST /api/sessions` 也会返回 500：

```
POST /api/sessions 500 (Internal Server Error)
[handleSave] sessionId 为空!
```

`createEditSession` 在 `fs.cpSync(workspacePath, sessionPath)` 时同样依赖 workspace 存在，缺失时会直接抛出 `ENOENT`。

**问题3：两套不兼容的 Session 管理系统**

| 系统 | 文件 | 创建 Session | 保存/合并 | 特性 |
|------|------|-------------|-----------|------|
| **旧系统** | `fs-utils.ts` | `createSession` | `mergeSession` | 无 `status` 管理，不检查 workspace |
| **新系统** | `session-manager.ts` | `createEditSession` | `saveEditSession` | 有 `status` 管理，缺失 workspace 时可恢复 |

**编辑页实际流程**：
- 打开编辑页：`POST /api/sessions` → 调用**新系统** `createEditSession`
- 点击保存：`POST /api/sessions/merge` → **调用旧系统** `mergeSession`

两套系统混用，旧系统 `mergeSession` 对缺失 workspace 没有任何容错，这是问题的核心。

### 第三阶段：确定根因

**真正根因**：项目 workspace 目录缺失后，多处代码直接崩溃，而非自动恢复。

具体表现在两个层面：

1. **创建 session 时**：`createEditSession` 执行 `fs.cpSync(workspacePath, sessionPath)`，workspace 缺失 → `ENOENT` → 500 错误
2. **保存时**：`mergeSession` 执行 `fs.cpSync(workspacePath, snapshotPath)`，workspace 缺失 → `ENOENT` → 500 错误

此外，旧 session 元数据缺少 `status` 字段，`saveEditSession` 的严格检查 `status !== 'editing'` 会拒绝这些 session。

## 修复方案

### 核心思路

1. **统一使用新系统**：编辑页保存从 `/merge` 改为 `/save`，统一调用 `saveEditSession`
2. **自动恢复 workspace**：所有依赖 workspace 的操作前，自动确保 workspace 存在且有默认内容
3. **兼容旧数据**：放宽 `status` 检查，缺失时默认为 `'editing'`

### 实施步骤

#### 步骤1：统一保存入口

**文件**：`packages/web/src/app/demo/[id]/edit/page.tsx`

```typescript
// 修改前
const mergeRes = await fetch(`/api/sessions/${sessionId}/merge`, {
  method: "POST",
});

// 修改后
const saveRes2 = await fetch(`/api/sessions/${sessionId}/save`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({}),
});
```

#### 步骤2：兼容旧 session 元数据

**文件**：`packages/web/src/lib/session-manager.ts`

```typescript
// 修改前
if (sessionMeta.status !== 'editing') {
  return { success: false, error: 'Session not in editing status' };
}

// 修改后
const status = sessionMeta.status || 'editing';
if (status !== 'editing') {
  return { success: false, error: 'Session not in editing status' };
}
```

#### 步骤3：统一 `/merge` 路由行为

**文件**：`packages/web/src/app/api/sessions/[sessionId]/merge/route.ts`

重写为调用 `saveEditSession`，与 `/save` 路由行为一致，保留 `/merge` 端点作为向后兼容。

#### 步骤4：删除旧 `mergeSession` 函数

**文件**：`packages/web/src/lib/fs-utils.ts`、`packages/web/src/lib/api.ts`

删除旧的 `mergeSession` 函数和对应的 API 客户端函数，避免未来被误用。

#### 步骤5：自动恢复 workspace

**文件**：`packages/web/src/lib/fs-utils.ts`

提取 `ensureWorkspaceFiles` 函数：

```typescript
export function ensureWorkspaceFiles(workspacePath: string): void {
  if (!fs.existsSync(workspacePath)) {
    fs.mkdirSync(workspacePath, { recursive: true });
  }

  const codePath = path.join(workspacePath, "index.tsx");
  const schemaPath = path.join(workspacePath, "config.schema.json");

  if (!fs.existsSync(codePath)) {
    fs.writeFileSync(codePath, DEFAULT_DEMO_CODE, "utf-8");
  }
  if (!fs.existsSync(schemaPath)) {
    fs.writeFileSync(schemaPath, DEFAULT_DEMO_SCHEMA, "utf-8");
  }
}
```

在以下三处调用：

| 位置 | 场景 |
|------|------|
| `createProject` | 创建新项目时 |
| `createSession` (旧系统) | 旧系统创建 session 时 |
| `createEditSession` (新系统) | 新系统创建 session 时 |

## 经验总结

1. **两套系统混用是隐患**：项目中存在新旧两套 Session 管理系统，调用链路不一致导致问题。应逐步废弃旧系统，统一到新系统。

2. **防御性编程**：任何依赖文件系统目录的操作，都应先确保目录存在，而非假设它一定存在。

3. **数据兼容性**：系统升级时，旧数据（如缺少新字段的元数据）需要兼容处理，不能严格校验新字段。

4. **日志驱动调试**：详细的日志是定位问题的关键。`[mergeSession]`、`[createEditSession]` 等日志标记帮助快速定位到具体失败函数。
