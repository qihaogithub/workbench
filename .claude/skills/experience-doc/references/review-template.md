```markdown
---

**解决的最核心问题**：{一两句话点明该机制/模块解决的最核心问题}

---

## 二、架构机制与正确方向

{准确描述当前的运作机制：调度策略、资源隔离方案、并发推导公式、数据流向、状态机流转等}

```

## 格式规范

### 必须包含的板块
1. **核心定位与设计哲学** — 一两句话点明核心问题和铁律
2. **架构机制与正确方向** — 当前运作机制的精确描述
3. **反模式与历史避坑** — 废弃方案、痛点、为什么绝对不能这么做
4. **核心指标与安全边界** — 极限容量、监控指标、降级阈值

### 可选板块
5. **关键教训** — 高度浓缩的一句话总结

### 严禁
- 伪代码或大段代码摘抄
- 函数签名、if/else 逻辑
- 开发过程记录
- 实施步骤清单

### 允许的代码形式
- 关键系统常量及其推导依据（如 `MAX_TOTAL_AREA = 35,000,000`）
- 极简的接口/类型定义（仅用于说明数据结构边界）
- 配置示例（JSON/YAML，仅用于说明配置策略）

### 目录与命名规则
- 按功能模块或业务边界建子目录：`docs/复盘文档/{模块名}/`
- 文件名直击技术主题，禁止套话（如"开发实战指南"、"XXX优化复盘"）
- 单个文档不超过 200 行，超过时按子机制拆分
