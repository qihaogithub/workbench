# 架构复盘文档生成

从代码变更、Bug 修复、技术决策和历史文档中提取架构级经验，按功能模块拆分为结构化文档，写入 `docs/复盘文档/`。

## 工作流程

### 1. 去重检查

读取 `references/existing-index.md`，确认是否已有同主题文档：
- 同主题 → 追加到已有文档的新章节
- 不同主题 → 新建文档

### 2. 主题归类与拆分

按功能模块或业务边界确定子目录和文件名：
- 子目录：`docs/复盘文档/{模块名}/`
- 文件名直击技术主题，禁止套话（如"开发实战指南"、"XXX优化复盘"）
- 严禁将所有内容堆砌在一个大文件中，按子机制拆分为独立文档

### 3. 按模板生成

读取 `references/review-template.md`，严格遵循四板块结构：
1. **核心定位与设计哲学** — 核心问题 + 铁律
2. **架构机制与正确方向** — 调度策略、资源隔离、并发公式、数据流向等
3. **反模式与历史避坑** — 废弃方案、痛点、为什么绝对不能这么做
4. **核心指标与安全边界** — 极限容量、监控指标、降级阈值

### 4. 写入文件

以多文件代码块形式输出，每个文件开头标注保存路径：
**文件路径**：`docs/复盘文档/{模块名}/{具体技术主题}.md`

### 5. 更新索引

写入完成后，更新 `references/existing-index.md` 中的文档索引。

## 表达维度

- **只谈机制与方向**：阐述系统边界、数据流向、状态机流转、组件交互规则或关键配置策略
- **严禁代码/伪代码**：绝对不要输出函数签名、if/else 逻辑或伪代码实现，用架构描述、业务规则或决策树替代
- **精确但非细节**：可指出关键系统常量及其推导依据，但不展示声明或调用位置

## 质量红线

1. 禁止输出多余的寒暄、审计过程或废话
2. 禁止出现伪代码或大段代码摘抄，必须用系统设计语言（如"引入令牌桶与亲和性路由"、"采用三阶段任务拆分隔离内存"）进行总结
3. 把未来开发者当作聪明的工程师，给架构方向、设计红线和踩过的坑，而不是手把手教写代码
4. 禁止输出开发过程记录、实施步骤清单、测试验证报告
