# 经验沉淀

从代码变更、Bug 修复、技术决策和历史文档中提取结构化技术经验，写入项目文档体系。

## 文档层级判定

收到经验沉淀请求时，先判定输出层级：

| 判定条件 | 输出层级 | 目标目录 |
|---------|---------|---------|
| 单一技术点的正确/错误写法对比 | 代码级 | `docs/plans/归档/经验/` |
| 第三方库的 API 陷阱或配置踩坑 | 代码级 | `docs/plans/归档/经验/` |
| 跨模块的架构决策与机制规则 | 架构级 | `docs/复盘文档/` |
| 系统边界、数据流向、状态机流转 | 架构级 | `docs/复盘文档/` |
| 组件交互规则与关键配置策略 | 架构级 | `docs/复盘文档/` |

**混合情况**：同一主题同时涉及代码陷阱和架构机制时，优先输出架构级文档，代码级细节作为"反模式"板块内的示例。

## 工作流程

### 1. 去重检查

读取 `references/existing-index.md`，确认是否已有同主题文档。若存在：
- 同层级同主题 → 追加到已有文档的新章节
- 不同层级同主题 → 在对应层级新建，交叉引用

### 2. 主题归类

根据经验所属技术领域确定文件名和目录：

**代码级归类**（`docs/plans/归档/经验/`）：
- 已有分类直接追加，不新建文件
- 全新领域新建 `{领域}经验.md`

**架构级归类**（`docs/复盘文档/`）：
- 按功能模块或业务边界建子目录
- 文件名直击技术主题，禁止套话命名

### 3. 按模板生成

根据判定层级读取对应模板：
- 代码级 → 读取 `references/experience-template.md`
- 架构级 → 读取 `references/review-template.md`

### 4. 写入文件

严格遵守模板格式，内容要求：
- 代码级：必须含 ❌ 错误写法 + ✅ 正确写法 + 根因 + 核心原则
- 架构级：必须含核心定位 + 架构机制 + 反模式 + 核心指标，严禁伪代码
- 单个文档不超过 200 行，超过时拆分

### 5. 更新索引

写入完成后，更新 `references/existing-index.md` 中的文档索引。

## 质量红线

- 禁止输出开发过程记录、实施步骤清单、测试验证报告
- 禁止在架构级文档中出现伪代码或大段代码摘抄
- 代码级文档的代码示例必须可直接复制使用
- 文档顶部保留 `> 从...中提取的...经验` 说明行
