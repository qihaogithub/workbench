# 问题分析报告模板

> 结构化问题分析报告的标准格式与示例

---

## 报告结构

每份问题分析报告必须包含以下四个部分，按顺序呈现：

1. 问题背景
2. 根因分析
3. 解决方案
4. 相关代码路径

## 报告示例

```markdown
# WebSocket 连接频繁断开 — 问题分析报告

## 1. 问题背景

### 问题描述
用户在使用聊天功能时，WebSocket 连接每隔约 30 秒自动断开并重连，
导致消息发送延迟和重复接收。

### 发生场景
- **时间**：2025-04-20 起持续出现
- **环境**：生产环境，Chrome 120+ 浏览器
- **触发条件**：用户进入聊天页面后，空闲约 30 秒即触发

### 预期行为
WebSocket 连接应保持稳定，仅在用户离开页面或网络中断时断开。

### 实际行为
连接在空闲约 30 秒后自动断开，客户端自动重连，期间消息丢失或重复。

### 错误信息
```
WebSocket connection to 'wss://api.example.com/ws' failed: 
Heartbeat timeout — no pong received within 25000ms
```

---

## 2. 根因分析

### 调查过程
1. 搜索 WebSocket 相关代码，定位心跳机制实现
2. 检查服务端心跳配置与客户端响应逻辑
3. 对比服务端超时设置与客户端发送间隔

### 证据链
| 证据 | 级别 | 来源 | 说明 |
|------|------|------|------|
| 服务端心跳超时设为 25s | A | server.ts:L45 | 超时阈值过短 |
| 客户端心跳间隔设为 30s | A | ws-client.ts:L78 | 间隔大于服务端超时 |
| 无 pong 响应时服务端主动断开 | A | heartbeat.ts:L32 | 断开逻辑正常 |

### 根本原因
客户端心跳发送间隔（30s）大于服务端心跳超时阈值（25s），
导致服务端在等待 pong 响应超时后主动断开连接。

**根因说明**：表面现象是"连接断开"，实际根因是心跳参数配置不一致。
服务端超时阈值应大于客户端发送间隔，当前配置恰好相反。

### 代码执行路径
客户端启动 → ws-client.ts 建立连接 → 服务端 heartbeat.ts 启动心跳检测
→ 客户端每 30s 发 ping → 服务端 25s 超时未收到 pong → 主动断开

---

## 3. 解决方案

### 方案一：调整服务端超时阈值（推荐）
- **描述**：将服务端心跳超时从 25s 调整为 35s（客户端间隔 30s + 5s 缓冲）
- **原理**：确保服务端超时阈值大于客户端发送间隔，避免误判超时
- **影响范围**：仅修改 server.ts 一行配置
- **风险**：极低，仅延长超时等待时间
- **复杂度**：低

### 方案二：缩短客户端心跳间隔
- **描述**：将客户端心跳间隔从 30s 缩短为 15s
- **原理**：客户端更频繁发送 ping，确保在服务端超时前收到 pong
- **影响范围**：修改 ws-client.ts 配置，增加网络流量
- **风险**：增加服务器负载（每连接每分钟多一次心跳）
- **复杂度**：低

### 后续建议
- 将心跳参数统一到共享配置包，避免两端配置不一致
- 添加心跳参数的启动时一致性校验

---

## 4. 相关代码路径

### 涉及文件
| 文件路径 | 行号 | 说明 |
|---------|------|------|
| packages/agent-service/src/server.ts | L45 | 服务端心跳超时配置 |
| packages/web/src/lib/ws-client.ts | L78 | 客户端心跳间隔配置 |
| packages/agent-service/src/utils/heartbeat.ts | L32 | 心跳检测与断开逻辑 |

### 调用链
[ws-client.ts:connect()] → [server.ts:handleUpgrade()] → [heartbeat.ts:startMonitor()] → [heartbeat.ts:timeout → disconnect]

### 相关配置
- `HEARTBEAT_TIMEOUT=25000`（服务端环境变量）
- `REACT_APP_WS_PING_INTERVAL=30000`（客户端环境变量）
```
