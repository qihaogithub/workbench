- 每个写端点先通过 `getAuthCookie` + `verifyToken` 校验登录态，再从 URL query 或 JSON body 中取 `sessionId`，校验其归属用户、过期状态及与 `projectId` 的一致性后才继续处理。
- 对 live workspace 的写操作一律构造 `WorkspaceMutationOperation[]`，使用 `hashText(fs.readFileSync(...))` 计算 `expectedHash` 实现乐观锁，再通过 `commitWorkspaceMutation` 提交；branch 工作空间则直接调用 `fs-utils` 的同步写入函数。
- 所有响应统一通过 `createApiSuccess` / `createApiError` 包装，错误码采用大写常量（如 `PROJECT_NOT_FOUND`、`SESSION_EXPIRED`、`SCHEMA_CONFLICT`），并在 catch 块中将 `WorkspaceAuthorityClientError` 映射回 `FILE_WRITE_ERROR`。
- 涉及 schema 修改的路径在落盘前先遍历当前项目所有页面的 `config.schema.json`，调用 `validateNoSchemaConflictFromStrings` 检查字段冲突，返回 `SCHEMA_CONFLICT` 错误。
- 路由参数类型通过 `{ params: { projectId: string; demoId?: string } }` 显式声明，请求体通过 `request.json().catch(() => ({}))` 安全解析后再做字段存在性与类型断言。
