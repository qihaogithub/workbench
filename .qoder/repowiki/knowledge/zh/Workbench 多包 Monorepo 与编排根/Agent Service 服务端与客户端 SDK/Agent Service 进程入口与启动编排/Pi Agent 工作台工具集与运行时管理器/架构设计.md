模块由两个并列子目录组成，围绕 `@earendil-works/pi-agent-core` 的 `AgentTool` 协议展开：

- `pi-tools/`：每个 `.ts` 文件对应一个独立工具，统一通过 `createXxxTool(config, ...)` 工厂函数返回 `AgentTool` 对象（含 `name`/`label`/`description`/`parameters`(TypeBox)/`execute`）。`index.ts` 暴露 `createWorkbenchTools()` 作为装配入口，按 `mode`（workbench / viewer-readonly）、环境变量（`PI_AGENT_SKETCH_TOOLS_ENABLED`）和可选 handler（`PlanApprovalHandler`、`UserChoiceHandler`、`SubagentRunner`、`PermissionHandler`）动态拼装工具列表，并导出 `getWorkbenchToolCapabilities()` 用于能力探测。
- `managers/`：面向工具执行生命周期的横切关注点。
  - `tool-hook-manager.ts`：注册到 harness 的 `tool_call`/`tool_result` 钩子，拦截路径权限、解析 Workspace Mutation Receipt 以生成 `FileChange[]`、提取 `updatePlan` 结果并记录知识库读取。
  - `permission-manager.ts`：集中实现路径白名单校验、knowledge/ 写保护、以及基于 `permission_request` 事件的异步用户确认（deletePage / plan approval），通过 `resolvePermission` 被外部回调唤醒。
  - `assistant-text-utils.ts`、`event-mapper.ts`：工具输入/结果的结构化抽取辅助。
  - `model-manager.ts`、`user-interaction-manager.ts`：模型选择与会话交互封装。
  - `pi-agent-deps.ts`：对 `@earendil-works/pi-agent-core` 与 `@earendil-works/pi-ai` 的懒加载门面，避免顶层静态依赖导致冷启动开销。

依赖方向：`pi-tools/*` → `permissions.ts` / `virtual-knowledge.ts` / `preview-validation.ts` / `workspace-mutation-authority`；`managers/*` → `pi-tools/*` 中的类型（如 `PermissionHandler`、`PlanApprovalRequest`）但不反向依赖具体工具实现，形成“工具实现 ↔ 管理器”的双向仅类型耦合。
