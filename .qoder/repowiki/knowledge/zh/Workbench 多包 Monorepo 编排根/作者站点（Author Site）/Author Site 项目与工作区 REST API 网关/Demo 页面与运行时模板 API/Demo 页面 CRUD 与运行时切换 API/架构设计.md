模块基于 Next.js App Router 的 `app/api/projects/[projectId]/demos/` 和 `demo-pages/reorder/` 目录组织，每个子目录对应一个资源端点：
- `demos/route.ts`：按 `GET` 列出页面（可选 `includeSchema`）、`POST` 创建或复制页面；
- `demos/[demoId]/route.ts`：`GET/PATCH/DELETE` 单个页面元信息，以及通过 `POST {action:"restoreDeletedSnapshot"}` 恢复已删除页面快照；
- `demos/[demoId]/files/route.ts`：`PUT` 批量写入 `index.tsx` / `config.schema.json` / sketch 相关文件；
- `demos/[demoId]/runtime/route.ts`：`PUT` 在 `prototype-html-css` / `high-fidelity-react` / `sketch-scene` 三种运行时之间切换并校验目标文件结构；
- `demo-pages/reorder/route.ts`：`PATCH` 批量重排页面与文件夹的 order/parentId。

所有写操作遵循统一的两条路径分支：当 `isLiveWorkspacePath(wsPath)` 为真时，通过 `commitWorkspaceMutation` 提交 `WorkspaceMutationOperation[]`（`put_text`/`delete_path`）经 Workspace Authority 落盘，并使用 `expectedHash` 做乐观锁冲突检测；否则走 branch/non-live 工作空间，直接调用 `@/lib/fs-utils` 中的 `createWorkspaceDemoPage` / `updateWorkspaceDemoFiles` / `reorderDemoPages` 等函数进行本地文件写入。鉴权由 `getAuthCookie` + `verifyToken` 完成，并通过 `sessionId` 绑定到具体 workspace 与用户，防止越权。Schema 变更在进入文件系统前经由 `validateNoSchemaConflictFromStrings` 做跨页面字段冲突校验，运行时切换再经 `project-admin-service.validateDemoPageFilesRuntime` 做二次校验。
