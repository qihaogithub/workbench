模块由两个 Next.js App Router route.ts 组成，分别对应 REST 路径 `/projects/[projectId]/resources/[kind]/[resourceId]/versions` 与 `.../versions/[versionId]`。
- 认证与鉴权：每个 handler 通过 `getAuthCookie` + `verifyToken` 解析 JWT，失败返回 401；所有对 `ProjectAdminService` 的调用统一附带 `{ id, name, role: 'creator', source: 'author-site' }` actor 上下文。
- 参数校验：`normalizeKind` 白名单限定 `page | knowledge_document | canvas | asset | project_config`，非法直接 400。
- 业务层：所有持久化操作委托给 `@workbench/project-core` 的 `ProjectAdminService`（构造时注入 `dataDir`），API 仅做请求包装与错误映射。
- Session/Live Workspace 集成：当 kind=page 且携带 sessionId 时，先经 `sessionExists` / `isSessionExpired` 校验，再通过 `flushAndSyncProjectWorkspace` 或 `flushWorkspaceBeforeCriticalAction` 将 live workspace 落盘并获取 `workspaceProof`（revision/rootHash），随后写入版本或提交 `commitWorkspaceMutation`。
- 版本恢复分支：`POST .../versions/[versionId]` 对 page 类型走特殊分支——若目标为 live workspace 则先做 schema 冲突检测 (`validateNoSchemaConflictFromStrings`)，再构造 `put_text` 操作序列通过 `commitWorkspaceMutation` 写回文件，否则走 `restorePageVersion`/`resourceRestore` 通用路径并同步更新工作区 demo 文件与版本基线标记。
- 响应封装：统一使用 `createApiSuccess` / `createApiError` 包裹数据，错误码映射到 HTTP 状态（如 FORBIDDEN→403、SESSION_EXPIRED→410）。
