根目录作为 monorepo 编排层，不承载业务逻辑，而是通过以下机制把各子包串起来：
- 构建与依赖：`pnpm-workspace.yaml` 声明 packages/* 下的所有子包（author-site、viewer-site、agent-service、screenshot-service、knowledge-*、sketch-*、project-*、shared、preview-contract 等），由 `turbo.json` 统一调度 build/dev/test；`package.json` 的 root scripts 提供跨包的一键入口。
- 运行时编排：`docker-compose.yml` 启动 author-site、viewer-site、agent-service、screenshot-service、knowledge-service 等服务，配合 `.env.docker` / `.env.example` 注入配置，使本地 `docker compose up` 即可运行全栈。
- 契约与类型共享：`packages/shared` 提供跨进程共享的类型与工具函数，`packages/preview-contract` 定义预览引擎与宿主之间的协议，被 author-site、viewer-site、screenshot-service 共同消费，确保前后端与子服务间的接口一致性。
- 数据持久化：`data/` 目录是 SQLite (`users.db`)、项目工作区 (`workspaces/`, `projects/`)、已发布产物 (`published/`)、截图与快照 (`screenshots/`, `snapshots/`)、AI 会话日志 (`agent-run-logs/`)、审计日志 (`audit/`) 等的统一落盘位置，被多个服务通过文件路径约定访问。
- 测试与验证：`scripts/check-*.mjs` 在 CI 中校验契约、部署预检与工作区权限守卫；`.github/workflows/check.yml` 触发 Turborepo 任务；Playwright 用例位于 `test/` 下，结果沉淀到 `test-results/`。
- 运维与自动化：`OPS/CLI` 提供项目管理 CLI，`OPS/automations` 用 registry 脚本驱动日常巡检、问题分诊与回归报告生成，`scripts/deploy*.sh` 封装生产/快速部署流程。
